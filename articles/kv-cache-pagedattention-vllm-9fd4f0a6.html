執行摘要

KV Cache（Key-Value Cache）是 Transformer 推理加速的核心優化技術，透過緩存注意力層的中間狀態避免重複計算，可實現 5-24 倍吞吐量提升。然而，傳統 KV Cache 管理存在 60-80% GPU 記憶體浪費問題（碎片化與過度預留）。

PagedAttention 受作業系統虛擬記憶體與分頁技術啟發，將 KV Cache 分割為固定大小的 blocks，透過 block tables 映射非連續物理記憶體，實現近零記憶體浪費（<4%）與 Copy-on-Write 共享語義。vLLM 基於 PagedAttention 實現的推理引擎，相比 HuggingFace Transformers 提升 24 倍吞吐量，相比前代最佳系統（TGI）提升 3.5 倍。

2025-2026 年，KV Cache 優化進入新階段：Entropy-Guided Caching（根據注意力熵分配快取預算）、Low-Dimensional Key Selection（僅用 O(log N) 維度做選擇）、Frequency-Domain Filtering（DCT 低通濾波保留關鍵 tokens）等技術，進一步突破記憶體與頻寬瓶頸。

一、KV Cache 基礎原理

1.1 核心問題：自迴歸生成的重複計算

在 Transformer 的自迴歸文本生成中，模型每生成一個新 token 都需要查看所有先前的 tokens（因果注意力遮罩）。標準推理流程：


Token 1 生成：計算 Q1, K1, V1 → Attention(Q1, K1, V1) → Output1
Token 2 生成：計算 Q2, K1, V1, K2, V2 → Attention(Q2, [K1,K2], [V1,V2]) → Output2
Token 3 生成：計算 Q3, K1, V1, K2, V2, K3, V3 → Attention(Q3, [K1,K2,K3], [V1,V2,V3]) → Output3

問題：K1、V1 在每次推理中都被重複計算（Token 2、3、4... 都會重算一次），造成計算量隨序列長度二次增長（O(n²)）。

1.2 KV Cache 解決方案

核心思想：將已計算的 Key 和 Value 矩陣緩存起來，每次只計算新 token 的 K、V，並與緩存拼接。


class KVCache:
    def __init__(self):
        self.cache = {"key": None, "value": None}

    def update(self, key, value):
        if self.cache["key"] is None:
            self.cache["key"] = key
            self.cache["value"] = value
        else:
            # 沿序列維度連接新的 K、V
            self.cache["key"] = torch.cat([self.cache["key"], key], dim=1)
            self.cache["value"] = torch.cat([self.cache["value"], value], dim=1)

優化效果：

  • 計算複雜度：從 O(n) 降為 O(1)（每步只計算 1 個新 token 的 K、V）
  • 實測加速：5.21 倍（T4 GPU，SmolLM2-1.7B，標準推理 61 秒 vs KV Caching 11.7 秒）[Hugging Face, 2026]
  • 長文本友好：序列越長，加速效果越顯著

1.3 記憶體權衡

優勢：避免重複計算，大幅加速推理

代價：需要額外記憶體存儲緩存（每層、每個 attention head 都需要存 K、V）

記憶體佔用估算（以 LLaMA-7B 為例）：

  • 模型參數：7B × 2 bytes (FP16) = 14 GB
  • KV Cache（序列長度 2048）：32 layers × 2 (K+V) × 32 heads × 128 dim × 2048 tokens × 2 bytes ≈ 1 GB per request
  • Batch size = 8 → 額外 8 GB KV Cache 記憶體

這導致傳統方法的兩大問題：

  1. 碎片化：序列長度不可預測，預分配空間會浪費
  2. 過度預留：為最壞情況（max_seq_len）預留記憶體，實際使用率低

二、PagedAttention 機制深度解析

2.1 靈感來源：作業系統虛擬記憶體

PagedAttention 借鑑作業系統的虛擬記憶體與分頁技術，解決 KV Cache 記憶體浪費問題：

| OS 概念 | PagedAttention 對應 |

|--------|-------------------|

| Process（進程） | Sequence（序列） |

| Page（頁） | Block（固定大小的 KV 區塊） |

| Virtual Address（虛擬位址） | Logical Block（邏輯區塊） |

| Physical Address（實體位址） | Physical Block（GPU 記憶體實體區塊） |

| Page Table（頁表） | Block Table（區塊映射表） |

2.2 核心機制：Block-Based KV Cache

分割策略：

  • 將 KV Cache 分割為固定大小的 blocks（如 16 tokens/block）
  • 每個 block 包含連續 16 個 tokens 的 Keys 和 Values
  • Block Table 維護邏輯 block ID → 物理 block ID 的映射

範例（序列長度 41 tokens，block size = 16）：


Sequence: [Token 0-15] [Token 16-31] [Token 32-40] [padding]
          ↓            ↓            ↓
Logical:  Block 0      Block 1      Block 2
          ↓            ↓            ↓
Physical: Block 7      Block 1      Block 3  （非連續映射）

Block Table：


Logical Block | Physical Block
-------------|---------------
     0       |       7
     1       |       1
     2       |       3

2.3 按需分配（On-Demand Allocation）

傳統方法：啟動時預分配 max_seq_len 的完整空間（如 2048 tokens）

PagedAttention：

  1. 初始只分配 1 個 block（16 tokens）
  2. 生成新 token 時，若當前 block 滿了，再分配新 block
  3. 序列結束時，立即釋放所有 blocks（可被其他序列重用）

記憶體效率：

  • 傳統方法浪費：60-80%（碎片化 + 過度預留）
  • PagedAttention 浪費：<4%（僅最後一個 block 可能未滿）

2.4 Copy-on-Write 共享語義

應用場景：Beam Search、Parallel Sampling（同一 prompt 生成多個輸出）

機制：

  1. 多個輸出序列共享相同的 prompt blocks（映射到同一物理 blocks）
  2. 當某個輸出開始生成不同內容時，才複製該 block（Copy-on-Write）
  3. 減少重複存儲 prompt 的記憶體開銷

實測效果：

  • 複雜採樣（beam search）記憶體開銷減少 55%
  • Beam search 吞吐量提升 2.2 倍 [vLLM Blog, 2023]

2.5 vLLM 架構實現

核心組件：

  1. Block Manager：

- 維護 free block pool（空閒區塊池）

- 處理 block 分配與回收

- 管理 block tables（每個序列一個）

  1. Attention Kernel：

- 改寫的 CUDA kernel，支援非連續的 block 訪問

- 透過 block table 查找實際的 K、V 記憶體位址

- 與標準 attention 計算結果完全一致（數學等價）

  1. Scheduler：

- 根據可用 blocks 動態調整 batch size

- 優先排程短序列（減少碎片）

- 支援 preemption（優先序列可搶佔資源）

記憶體佈局（Logical vs Physical）：


Logical View (Sequence A):
[Block 0] [Block 1] [Block 2] ...

Physical GPU Memory:
| Free | Block 7 | Free | Block 1 | Free | Block 3 | Free |
        ↑               ↑               ↑
        Seq A Block 0   Seq A Block 1   Seq A Block 2

三、2026 年最新進展

3.1 Entropy-Guided KV Caching

論文：_Entropy-Guided KV Caching for Efficient LLM Inference_ (MDPI Mathematics, 2026)

核心思想：

  • 計算每個 attention head 的注意力熵（entropy of attention weights）
  • 高熵層（broad attention dispersion）→ 分配更大 KV cache 預算
  • 低熵層（sink-like attention）→ 分配較小 cache 預算

數學形式：


H(layer_i) = -Σ p_ij log(p_ij)  （p_ij 為 attention weights）
Cache_budget(layer_i) = Total_budget × (H(layer_i) / Σ H(all_layers))

效益：

  • 在相同記憶體預算下，保留更關鍵的 KV pairs
  • 適用於長上下文場景（128K+ tokens）

3.2 Low-Dimensional Key Selection

論文：_Thin Keys, Full Values: Reducing KV Cache via Low-Dimensional Attention Selection_ (arXiv 2603.04427, 2026)

理論基礎：

  • Selection（Q × K^T）：產生標量注意力權重，僅需 O(log N) 維度區分 N 個 tokens
  • Value Transfer（Attention × V）：需要完整維度傳遞豐富表示

方法：

  • Keys 降維：從 128-dim 降至 32-dim（4 倍壓縮）
  • Values 保持：維持原始維度（128-dim）
  • Queries 同步降維：與 Keys 維度匹配

實測效果：

  • KV Cache 記憶體減少 25-30%（Key 佔一半，Values 不變）
  • 精度損失 <1%（MMLU, HellaSwag 基準測試）

3.3 Frequency-Domain Filtering (FlashCache)

核心技術：DCT（離散餘弦變換）低通濾波

流程：

  1. 對 KV tensors 應用 DCT
  2. 識別高頻異常 tokens（能量偏離主要低頻趨勢）
  3. 保留這些 outliers，丟棄低頻常規 tokens
  4. 無需重新計算 attention matrix（直接在頻域判斷）

優勢：

  • 計算成本低（DCT 是 O(n log n)）
  • 可與 PagedAttention 結合（動態標記 outlier blocks 為"必須保留"）

3.4 Grouped-Query Attention (GQA) 與 Multi-Query Attention (MQA)

標準 Multi-Head Attention：

  • 32 heads，每個 head 獨立的 K、V
  • KV Cache = 32 × (K + V)

Multi-Query Attention (MQA)：

  • 32 heads 共享 1 組 K、V
  • KV Cache = 1 × (K + V)
  • 記憶體減少 32 倍，但精度可能下降

Grouped-Query Attention (GQA)：

  • 32 heads 分成 8 組，每組共享 K、V
  • KV Cache = 8 × (K + V)
  • 記憶體減少 4 倍，精度損失小於 MQA

採用現況：

  • LLaMA 2：GQA（8 組）
  • Gemini 1.5：MQA（極端壓縮）
  • GPT-4：標準 MHA（優先精度）

四、實務應用建議

4.1 框架選擇

| 框架 | PagedAttention 支援 | GQA/MQA | Speculative Decoding | 適用場景 |

|------|------------------|---------|---------------------|---------|

| vLLM | ✅ 原生支援 | ✅ | ✅ | 生產級高吞吐服務 |

| SGLang | ✅ 優化版 | ✅ | ✅ | 複雜 prompt 管線 |

| TensorRT-LLM | ✅ | ✅ | ✅ | NVIDIA GPU 極致優化 |

| HF Transformers | ❌ 標準 KV Cache | ⚠️ 部分模型 | ❌ | 快速原型開發 |

4.2 記憶體優化策略組合

推薦配置（24 GB GPU，LLaMA-7B）：

  1. 基礎層：PagedAttention（減少浪費 60% → <4%）
  2. 模型層：GQA 或 4-bit 量化（AWQ/GPTQ）
  3. 長上下文：Entropy-Guided Caching（128K+ tokens）
  4. 加速層：Speculative Decoding（draft model: 1.3B）

預期效果：

  • 吞吐量：24 倍（vs 標準 HF）
  • 記憶體：支援 batch size 32（vs 標準 4-8）
  • 延遲：保持 <100ms/token（P99）

4.3 部署檢查清單

✅ 確認 GPU 記憶體分配：nvidia-smi 監控 KV Cache 實際使用率

✅ 設定 block size：16-32 tokens（trade-off 碎片 vs 靈活性）

✅ 啟用 tensor parallelism：多 GPU 時分散 KV Cache（每張卡存一部分）

✅ 監控 cache hit rate：CoW 共享是否生效（beam search 場景）

✅ 設定 max_num_seqs：根據可用 blocks 動態調整（避免 OOM）

4.4 常見陷阱

❌ 過度量化 KV Cache：4-bit 量化 Keys 可能破壞 attention 精度（建議僅量化 Values 或用 8-bit）

❌ 忽略 block size 調參：太小（4 tokens）→ 碎片多；太大（64 tokens）→ 浪費增加

❌ 未啟用 CoW：beam search 時每個 beam 獨立分配 prompt blocks → 記憶體爆炸

❌ 混用不同框架：PagedAttention 與標準 KV Cache 不相容，轉換需重新加載模型

五、與其他推理優化技術的協同

5.1 PagedAttention + Speculative Decoding

協同效果：

  • PagedAttention：優化記憶體，提升 batch size（3-4 倍）
  • Speculative Decoding：加速生成，提升吞吐量（2-3 倍）
  • 組合：6-12 倍整體提升（兩者互補，非相互替代）

實作要點：

  • Draft model 與 target model 共享 prompt blocks（CoW）
  • Verification 階段可批量處理（PagedAttention 的 batching 優勢）

5.2 PagedAttention + FlashAttention

誤解澄清：

  • FlashAttention：優化 attention 計算本身（IO-aware，減少 HBM 訪問）
  • PagedAttention：優化 KV Cache 記憶體管理（分頁，減少浪費）
  • 兩者可同時使用：vLLM 內部的 attention kernel 整合了 FlashAttention 的 IO 優化

5.3 與量化技術結合

推薦組合：

  • 模型權重：AWQ 4-bit 量化（減少參數記憶體 70%）
  • KV Cache：8-bit 量化（減少 cache 記憶體 50%）
  • PagedAttention：進一步減少浪費（<4%）

注意：KV Cache 量化需謹慎，過度壓縮（4-bit）會顯著影響長上下文精度。

六、未來方向與開放問題

6.1 跨請求 KV Cache 共享

問題：不同用戶的請求可能有相同 prefix（如系統 prompt）

方案：Prefix Caching（SGLang RadixAttention）— 自動偵測並共享相同 prefix blocks

挑戰：安全隔離（避免洩漏其他用戶的 cache）、失效策略（cache 何時過期）

6.2 異構記憶體層級

趨勢：GPU HBM（高速）+ CPU RAM（大容量）+ SSD（超大容量）

目標：將不常用的 KV blocks 換出到 CPU/SSD，需要時再換入

技術：類似 OS 的 swap，需要預測哪些 blocks 即將被訪問（attention pattern 預測）

6.3 動態 Block Size

現狀：vLLM 使用固定 block size（如 16 tokens）

改進：根據序列特性動態調整（短序列用小 blocks，長序列用大 blocks）

挑戰：block table 管理複雜度增加
