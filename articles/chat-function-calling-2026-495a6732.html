一、系統概述與目標

1.1 系統定位

建構一套輕量級、可擴展的 AI Chat 對話系統，具備優質 RESTful API 與現代化前端介面。系統核心特色是內建 Function Calling（工具調用） 能力，使 LLM 不僅能回答文字問題，還能透過結構化的工具調用執行實際操作（如查詢天氣、搜尋資料庫、呼叫外部 API）。

1.2 目標用戶

  • 開發者：需要一個可自訂工具的 Chat 平台來整合內部服務
  • 個人用戶：希望透過自然語言操控各種工具（知識庫查詢、日程管理、資料分析等）
  • 團隊：需要共享對話歷史與工具配置的協作環境

1.3 核心功能

| 功能 | 描述 | 優先級 |

|------|------|--------|

| 即時對話 | 支援 SSE Token Streaming 的即時打字效果 | P0 |

| Function Calling | LLM 自動判斷並調用已註冊工具 | P0 |

| 工具註冊 | 透過 JSON Schema 定義並動態註冊工具 | P0 |

| 對話管理 | 建立、列表、搜尋、刪除對話 | P0 |

| 前端介面 | 現代化 React Chat UI，含工具調用視覺化 | P0 |

| 多模型支援 | 支援 OpenAI / Anthropic / 本地模型切換 | P1 |

| 對話匯出 | 匯出 Markdown / JSON 格式 | P2 |

| MCP 相容 | 工具系統可升級為 MCP 協議 | P2 |

3.2 SSE 串流格式


Event Stream Format:

event: token
data: {"content": "Hello"}

event: token
data: {"content": " world"}

event: tool_call_start
data: {"tool_name": "get_weather", "tool_call_id": "tc_001", "arguments": {"location": "Taipei"}}

event: tool_call_result
data: {"tool_call_id": "tc_001", "result": {"temp": 25, "condition": "sunny"}, "execution_time_ms": 150}

event: token
data: {"content": "The weather in Taipei is..."}

event: done
data: {"message_id": "uuid", "usage": {"input_tokens": 150, "output_tokens": 45}}

event: error
data: {"code": "TOOL_TIMEOUT", "message": "Tool execution timed out after 30s"}

3.3 認證機制

JWT Bearer Token 認證：

  1. 登入：POST /api/v1/auth/login -> 回傳 access_token（15 分鐘）+ refresh_token（7 天）
  2. 請求認證：Authorization: Bearer
  3. Token 刷新：POST /api/v1/auth/refresh -> 新的 access_token
  4. 登出：POST /api/v1/auth/logout -> 將 refresh_token 加入黑名單

權限模型：

  • user：基本對話、使用已啟用工具
  • admin：管理工具定義、查看系統統計、管理用戶

3.4 錯誤回應格式（統一）


{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid tool parameters",
    "details": [
      {"field": "parameters.location", "issue": "required field missing"}
    ]
  }
}

| HTTP Status | Error Code | 描述 |

|-------------|------------|------|

四、Function Calling 機制

4.1 工具定義格式（JSON Schema）

每個工具透過 JSON Schema 定義其輸入參數規格：


{
  "name": "search_knowledge_base",
  "description": "搜尋個人知識庫中的筆記，回傳最相關的結果",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜尋關鍵字或自然語言查詢"
      },
      "top_k": {
        "type": "integer",
        "description": "回傳結果數量",
        "default": 5,
        "minimum": 1,
        "maximum": 20
      },
      "search_type": {
        "type": "string",
        "enum": ["hybrid", "keyword", "semantic"],
        "default": "hybrid"
      }
    },
    "required": ["query"]
  },
  "handler": {
    "type": "http",
    "method": "POST",
    "url": "http://localhost:3000/api/search/hybrid",
    "headers": {"Content-Type": "application/json"},
    "body_template": {"query": "{{query}}", "topK": "{{top_k}}"}
  },
  "timeout_ms": 10000,
  "permission_level": "read"
}

4.2 調用流程（6 步驟）


Step 1: 用戶輸入
  |  "幫我查一下知識庫裡關於 React Hooks 的筆記"
  v
Step 2: 建構 LLM 請求
  |  messages=[用戶訊息] + tools=[已註冊工具定義]
  v
Step 3: LLM 判斷（模型決策）
  |  模型分析意圖，決定是否需要工具調用
  |  回傳：finish_reason="tool_calls"
  |  tool_calls=[{name: "search_knowledge_base", arguments: {query: "React Hooks", top_k: 5}}]
  v
Step 4: 安全驗證
  |  - 檢查工具是否存在且已啟用
  |  - 驗證參數符合 JSON Schema
  |  - 檢查用戶對該工具的權限
  |  - 檢查速率限制（每分鐘調用上限）
  v
Step 5: 執行工具
  |  - 在隔離環境中執行（timeout 控制）
  |  - 記錄調用日誌（工具名、參數、耗時、結果摘要）
  |  - 將結果以 tool_result 角色附加到對話
  v
Step 6: 模型整合回覆
  |  將工具結果 + 對話歷史再送給 LLM
  |  模型根據工具結果生成自然語言回覆
  v
最終回覆呈現給用戶

4.3 多輪工具調用（遞迴機制）

模型可能需要連續調用多個工具。系統設定最大迭代上限為 5 次，防止無限迴圈：


async def process_with_tools(messages, tools, max_iterations=5):
    for i in range(max_iterations):
        response = await llm_client.chat(messages, tools)
        
        if response.finish_reason != "tool_calls":
            return response  # 最終回覆
        
        for tool_call in response.tool_calls:
            result = await execute_tool(tool_call)
            messages.append({"role": "tool", "content": result, "tool_call_id": tool_call.id})
    
    # 超過上限，強制回覆
    return await llm_client.chat(messages, tools=None)

4.4 內建工具清單建議

| 工具名稱 | 描述 | 權限 | 類型 |

|---------|------|------|------|

| get_current_time | 取得當前時間與時區資訊 | read | 內建 |

| calculate | 安全的數學運算（支援基本算術與科學計算） | read | 內建 |

| search_web | 網路搜尋（整合搜尋引擎 API） | read | HTTP |

| search_knowledge_base | 查詢個人知識庫 | read | HTTP |

| create_note | 建立知識庫筆記 | write | HTTP |

| get_weather | 取得天氣資訊 | read | HTTP |

| run_code | 執行 Python 程式碼（沙箱環境） | privileged | Script |

| fetch_url | 擷取網頁內容並摘要 | read | HTTP |

4.5 安全沙箱設計

三層防護：

  1. 參數驗證層：JSON Schema 驗證所有輸入參數，拒絕不符合規格的調用
  2. 權限控制層：

- read：只讀操作，無需額外確認

- write：寫入操作，記錄審計日誌

- privileged：高風險操作（如執行程式碼），需用戶即時確認（Human-in-the-Loop）

  1. 執行隔離層：

- 每次工具調用設定 timeout（預設 30 秒）

- Script 類型工具在受限環境中執行（限制網路存取、檔案系統存取、記憶體上限 256MB）

- 結果大小限制（截斷超過 10KB 的回傳，避免撐爆 context window）

速率限制：

  • 全域：每分鐘最多 60 次工具調用
  • 每用戶：每分鐘最多 20 次
  • 每工具：每分鐘最多 30 次（可個別配置）

七、安全考量

7.1 認證與授權

  • JWT Token：access_token 短效（15 分鐘）、refresh_token 長效（7 天）
  • 密碼儲存：bcrypt（cost factor = 12）
  • Token 黑名單：登出時將 refresh_token 加入 Redis 黑名單
  • RBAC：基於角色的存取控制（user / admin）

7.2 輸入驗證

  • 所有 API 端點：Pydantic Model 驗證請求體（型別、長度、格式）
  • 工具參數：JSON Schema 二次驗證（防止 LLM 產生超出規格的參數）
  • SQL Injection 防護：SQLAlchemy ORM 參數化查詢（不使用原生 SQL 拼接）
  • 訊息長度限制：單則訊息最多 32,000 字元

7.3 XSS / CSRF 防護

  • XSS：前端使用 React 預設的 JSX 自動轉義；Markdown 渲染使用 sanitize 處理（DOMPurify）
  • CSRF：SameSite Cookie 屬性 + 前後端分離（JWT 不受 CSRF 影響）
  • Content Security Policy：設定嚴格的 CSP header

7.4 Function Calling 權限控制

  • 工具分級：read（無需確認）/ write（記錄日誌）/ privileged（用戶確認）
  • Human-in-the-Loop：privileged 工具調用前，前端彈出確認對話框
  • 審計日誌：所有工具調用記錄到 ToolInvocation 表（不可刪除）
  • 結果消毒：工具回傳的 HTML/Script 內容一律消毒後才傳給 LLM

7.5 速率限制（Rate Limiting）

| 層級 | 限制 | 機制 |

|------|------|------|

| API 全域 | 1000 req/min | FastAPI middleware + 滑動窗口 |

| 每用戶 | 60 req/min | JWT 識別 + Redis 計數 |

| LLM 調用 | 20 req/min/user | 防止 Token 濫用 |

| 工具調用 | 20 calls/min/user | 防止工具濫用 |

| SSE 連線 | 5 concurrent/user | 防止連線耗盡 |

7.6 LLM API Key 保護

  • API Key 存於伺服器端環境變數，前端永不接觸
  • 使用 .env 管理（不納入版本控制）
  • 支援 Key 輪替（多 Key 負載均衡 + 故障轉移）

九、技術風險與緩解

| 風險 | 可能性 | 影響 | 緩解措施 |

|------|--------|------|----------|

| LLM API 不穩定 / 限流 | 高 | 中 | 多供應商 fallback + 指數退避重試 + 快取常見回覆 |

| Function Calling 產生非法參數 | 中 | 中 | JSON Schema 嚴格驗證 + 模型 prompt 優化 + 參數消毒 |

| 工具執行超時或失敗 | 中 | 低 | timeout 強制終止 + graceful degradation + 告知用戶 |

| SSE 連線中斷 | 中 | 低 | 前端 EventSource 自動重連 + 斷點續傳（message_id 追蹤） |

| Token 用量爆增（成本風險） | 中 | 高 | 每用戶每日 Token 上限 + 對話歷史自動裁剪 + 工具結果摘要化 |

| 安全漏洞（Prompt Injection） | 中 | 高 | 系統提示詞隔離 + 工具結果消毒 + 輸出監控 |

| 並發連線過多（DoS） | 低 | 高 | 速率限制 + 連線數上限 + 反向代理（Nginx） |

| 資料庫效能瓶頸 | 低 | 中 | 適當索引 + 分頁查詢 + 對話歷史定期歸檔 |

附錄 B：關鍵技術決策記錄

| 決策 | 選擇 | 替代方案 | 理由 |

|------|------|---------|------|

| 傳輸層 | SSE（Token Streaming）+ HTTP REST | WebSocket | SSE 以 10% 複雜度提供 90% 效果；雙向需求低（用戶訊息用 POST） |

| 狀態管理 | Zustand | Redux / Jotai | 輕量（< 1KB）、無 boilerplate、支援 middleware |

| ORM | SQLAlchemy 2.0 | Tortoise ORM | async 支援成熟、社群大、遷移工具（Alembic）完善 |

| 認證 | JWT | Session Cookie | 前後端分離友好、無需伺服器端 session 儲存 |

| 工具系統 | 自建 Registry + MCP 相容 | 純 LangChain Tools | 保持協議無關性、避免框架鎖定 |

| 部署 | FastAPI 托管前端靜態檔 | 分離部署 | 簡化部署（單一進程）、適合個人/小團隊使用 |
