缺口分析

產出物

1. config/schemas/task-result-error.schema.json

統一錯誤物件格式（RFC 7807 Problem Details 風格），包含必填欄位：

  • error.type：錯誤類型識別碼（必須在 taxonomy 中定義）
  • error.title：錯誤簡短標題（10-50 字）
  • error.detail：錯誤詳細描述（含上下文與診斷資訊）
  • error.retryable：是否可重試（布林值）
  • error.instance：錯誤實例唯一識別碼（task_key + timestamp + phase）

選填欄位：status（HTTP 狀態碼）、task_key、phase、extensions（任務特定診斷資訊）。

2. config/error-taxonomy.yaml

定義 8 種錯誤類型（version 1）：

  1. timeout（執行逾時）- retryable: true
  2. dependency_unhealthy（依賴服務不可用）- retryable: false
  3. rate_limited（API 請求頻率超限）- retryable: true
  4. phase_error（Multi-Phase 執行錯誤）- retryable: false
  5. validation_failed（輸出驗證失敗）- retryable: false
  6. api_error（API 呼叫錯誤）- retryable: true
  7. filesystem_error（檔案系統錯誤）- retryable: false
  8. network_error（網路錯誤）- retryable: true

每種錯誤類型包含：category、title、description、typical_causes、suggested_actions。

3. workflows/error-object-validation-checklist.md

驗證清單包含 5 個步驟：

  1. Schema 語法驗證
  2. Taxonomy 完整性檢查
  3. Prompt 範例抽樣驗證（抽樣 3 個 prompt）
  4. 實際執行驗證（可選）
  5. system-insight 整合驗證

設計理念

為何選擇 RFC 7807 Problem Details

  • 業界標準，提供 type/title/detail/instance 等結構化欄位
  • 易於機器解析與人類閱讀
  • 支援擴充欄位（extensions）以適應任務特定需求

為何外部化 Taxonomy

  • 將 error.type 列表外部化至 config/error-taxonomy.yaml
  • 避免 schema 頻繁變更
  • 支援動態擴充新錯誤類型

為何加入 retryable 欄位

  • system-insight 與 self-heal 可據此決策是否重試或標記需人工介入
  • 減少無意義重試（如 validation_failed 不應重試）
  • 提升自動修復成功率

向後相容性

  • 所有新欄位皆為 optional（除 5 個必填欄位外）
  • 現有結果 JSON 的 error.message 欄位仍可用（已標記 deprecated）
  • prompt 範例需逐步更新，但不影響現有任務執行

整合點

  1. 35 個自動任務 prompt：更新錯誤輸出範例（逐步進行）
  2. context/system-insight.json：新增 error_type_distribution 欄位，統計各類錯誤分布
  3. tools/collect_system_data.py：新增 error.type 統計邏輯
  4. self-heal Skill：讀取 error.retryable 判斷可修復性
  5. arch-evolution Skill：根據 error.type 分布決策架構改善方向
