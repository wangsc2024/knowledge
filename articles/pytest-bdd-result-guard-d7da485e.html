pytest-bdd 契約測試與 result guard 驗證實踐

執行摘要（200-400 字）

pytest-bdd 是一個將行為驅動開發（BDD）整合到 pytest 測試框架的 Python 工具，透過 Gherkin 語法讓非技術利害關係人也能理解測試場景。與傳統 BDD 工具不同，pytest-bdd 直接整合 pytest，無需獨立 runner，可重用既有 pytest fixtures [來源：pytest-bdd Documentation, 2026]。

契約測試（Contract Testing）是一種驗證服務間通訊介面的測試方法，特別適用於微服務架構。相較於端到端測試，契約測試速度更快、可維護性更高，並能在本機開發階段提早發現整合問題 [來源：Pactflow, n.d.]。

本研究探討如何將 pytest-bdd 應用於 result guard 契約測試場景（如 results/todoist-auto-*.json 的 schema 驗證），並整合到 GitHub Actions CI/CD 管線。核心發現包括：(1) pytest-bdd 的 Scenario Outline 支援參數化測試，可有效測試多組 JSON schema 變體；(2) 透過 pytest fixtures 可重用既有驗證邏輯（如 tools/validate_results.py）；(3) BDD 場景可作為「可執行規格」，確保 Phase 2/3 交接契約不漂移。

適用場景：結果檔 schema 驗證、workflow YAML 規則測試、去重策略契約、Phase 間交接格式驗證。

---

研究背景

問題脈絡

daily-digest-prompt 專案採用三階段管線（Phase 1 查詢 → Phase 2 執行 → Phase 3 組裝），各階段透過 JSON 檔案交接狀態。當前挑戰：

  1. 交接格式漂移：results/todoist-auto-*.json 的必填欄位（agent、status、summary）可能在不同 prompt 實作中不一致
  2. 驗證邏輯分散：tools/validate_results.py（Python）、tools/ps/atomic-result-json.ps1（PowerShell）、workflows/*.yaml（規格）三處重複定義規則
  3. 回歸測試缺口：修改 prompt 或 schema 後，缺乏自動化測試確保交接契約不破壞

BDD 與契約測試的價值

行為驅動開發（BDD）透過 Given-When-Then 語法，將測試場景寫成「可執行規格」[來源：Virtuoso QA, 2024]。契約測試則專注於驗證通訊介面格式，避免端到端測試的維護成本 [來源：Pactflow Blog, n.d.]。兩者結合可將「結果檔應包含哪些欄位」這類規則，從隱性知識轉為自動驗證的 feature 檔案。

核心發現

1. pytest-bdd 核心機制

#### 整合模式

pytest-bdd 實作 Gherkin 語言子集，但與 Cucumber 等傳統工具的關鍵差異在於：直接整合 pytest，無需獨立 runner [來源：pytest-bdd Documentation, 2026; TestMu AI Learning Hub, n.d.]。

這帶來三個優勢：

  • 可重用所有 pytest 既有功能（fixtures、markers、plugins、並行執行）
  • 統一單元測試與 BDD 測試的執行入口
  • 降低工具鏈複雜度（不需學習 Cucumber/behave 的獨立 CLI）

#### Feature 檔案結構

Gherkin 語法範例（features/result_validation.feature）：


Feature: Result file validation
  確保 Phase 2 產出的 JSON 符合交接契約

  Scenario: 必填欄位完整性
    Given 一個新建的結果檔 "results/todoist-auto-test.json"
    When 寫入包含 agent, status, summary 的 JSON
    Then 驗證器應該通過
    And 不應有 missing_fields 錯誤

  Scenario: 缺少 summary 欄位
    Given 一個新建的結果檔 "results/todoist-auto-test.json"
    When 寫入缺少 summary 的 JSON
    Then 驗證器應該失敗
    And 錯誤訊息應包含 "summary"

#### Step 定義與 Fixtures

對應的 Python 實作（tests/test_result_validation.py）：


from pytest_bdd import scenario, given, when, then, parsers
import json
from pathlib import Path
from tools.validate_results import validate_result_file  # 重用既有驗證器

@scenario('features/result_validation.feature', '必填欄位完整性')
def test_required_fields():
    pass

@given(parsers.parse('一個新建的結果檔 "{filepath}"'), target_fixture="result_file")
def create_result_file(filepath, tmp_path):
    """使用 pytest 的 tmp_path fixture"""
    file_path = tmp_path / filepath
    file_path.parent.mkdir(parents=True, exist_ok=True)
    return file_path

@when(parsers.parse('寫入包含 agent, status, summary 的 JSON'))
def write_valid_json(result_file):
    result_file.write_text(json.dumps({
        "agent": "todoist-auto-test",
        "status": "success",
        "summary": "測試完成"
    }))

@then('驗證器應該通過')
def validator_passes(result_file):
    result = validate_result_file(result_file)
    assert result["valid"] is True

關鍵機制：target_fixture 參數讓 @given 步驟產生的物件（如 result_file）可被後續步驟重用 [來源：pytest-bdd Documentation, 2026]。這避免了傳統 BDD 工具的全域 context 狀態問題。

#### 參數化測試（Scenario Outline）

測試多組輸入變體：


Scenario Outline: 不同 status 值的驗證
  Given 一個新建的結果檔 "results/test.json"
  When 寫入 status 為 "<status>" 的 JSON
  Then 驗證結果應為 "<expected>"

  Examples:
    | status  | expected |
    | success | valid    |
    | failed  | valid    |
    | invalid | invalid  |

pytest-bdd 會自動展開為 3 個獨立測試，每個測試對應一列 Examples [來源：pytest-bdd Documentation, 2026; Pytest with Eric, 2025]。

2. 契約測試原理與適用場景

#### 定義與核心價值

契約測試是「確保兩個獨立系統（如兩個微服務）相容且能互相通訊的方法論」[來源：Pactflow, n.d.]。與端到端測試的差異：

| 測試類型 | 執行環境 | 速度 | 維護成本 | 除錯難度 |

|---------|---------|------|---------|----------|

| 端到端測試 | 需部署所有服務 | 慢（分鐘級） | 高（需了解整個生態系） | 高（多系統交互） |

| 契約測試 | 本機獨立執行 | 快（秒級） | 低（專注單一介面） | 低（問題隔離到具體元件） |

[來源：Pactflow, n.d.; Should You Use Contract Testing?, Alex Romanov, 2021]

#### 雙向驗證

契約測試是雙向的：既驗證消費端（consumer）的請求符合契約，也驗證提供端（provider）的回應符合契約 [來源：Essential Microservice Testing, Fernando Villalba, Medium]。這與 mock 測試的單向驗證不同。

#### 何時使用契約測試

適用場景 [來源：Pactflow, n.d.; Contract Testing: How and When?, Michał Koźmiński, Medium]：

  • 微服務架構中，不同團隊維護不同服務
  • 需在 provider 完成前開始開發 consumer
  • 希望在本機開發階段提早發現整合問題

不適用場景 [來源：Should You Use Contract Testing?, Alex Romanov, 2021]：

  • 單體應用（內部 API 呼叫）
  • 服務完全由同一團隊維護
  • 整合點非常少（< 3 個）

#### 對 daily-digest-prompt 的應用

本專案的「Phase 間交接」可視為微服務間的契約：

  • Consumer：Phase 3 組裝邏輯（讀取 results/*.json）
  • Provider：Phase 2 各 prompt（寫入 results/*.json）
  • 契約：config/schemas/results-auto-task-schema.json

契約測試可確保：

  1. Phase 2 的 29 個 prompt 都產出符合 schema 的 JSON
  2. Phase 3 的組裝邏輯能正確解析所有變體（success/failed/partial）
  3. 新增 prompt 時，測試會強制執行 schema 合規性

[推論；依據：KB 既有 BDD mechanism 筆記, 2026-04-10]

3. pytest-bdd 與契約測試結合模式

#### 將 schema 規則寫成 Feature


Feature: Result JSON schema contract
  確保所有自動任務產出的 JSON 符合統一 schema

  Background:
    Given result schema 定義在 "config/schemas/results-auto-task-schema.json"

  Scenario: 成功任務的最小必填欄位
    When 任務狀態為 "success"
    Then JSON 必須包含 "agent", "status", "summary"
    And "summary" 長度至少 20 字元
    And "status" 只能是 "success" | "failed" | "partial"

  Scenario Outline: 不同任務類型的擴展欄位
    When 任務類型為 "<task_type>"
    Then JSON 可包含 "<optional_fields>" 欄位

    Examples:
      | task_type       | optional_fields           |
      | tech_research   | topic, kb_imported        |
      | arch_evolution  | adr_changes, decisions_made |
      | self_heal       | fixed_count, skipped_count  |

#### Step 實作重用既有驗證器


from pytest_bdd import given, when, then, parsers
from tools.validate_results import validate_result_file
from jsonschema import validate, ValidationError
import json

@given(parsers.parse('result schema 定義在 "{schema_path}"'), target_fixture="schema")
def load_schema(schema_path):
    return json.loads(Path(schema_path).read_text())

@when(parsers.parse('任務狀態為 "{status}"'), target_fixture="test_json")
def create_test_json(status):
    return {
        "agent": "todoist-auto-test",
        "status": status,
        "summary": "測試摘要，長度足夠通過驗證"
    }

@then(parsers.parse('JSON 必須包含 "{fields}"'))
def verify_required_fields(test_json, fields):
    required = [f.strip() for f in fields.split(',')]
    missing = [f for f in required if f not in test_json]
    assert not missing, f"缺少必填欄位：{missing}"

@then(parsers.parse('"summary" 長度至少 {min_len:d} 字元'))
def verify_summary_length(test_json, min_len):
    assert len(test_json["summary"]) >= min_len

#### 整合既有 Python 驗證工具

優勢：pytest-bdd 可直接呼叫既有的 tools/validate_results.py，無需重寫驗證邏輯 [來源：pytest-bdd Documentation, 2026]。


@then('驗證器應該通過')
def validator_passes(test_json, tmp_path):
    # 寫入暫存檔
    test_file = tmp_path / "test_result.json"
    test_file.write_text(json.dumps(test_json))

    # 呼叫既有驗證器
    result = validate_result_file(test_file)
    assert result["valid"], f"驗證失敗：{result.get('errors')}"

4. CI/CD 整合實踐

#### GitHub Actions 工作流配置

pytest-bdd 可無縫整合到 GitHub Actions [來源：Pytest with Eric, 2025; Pytest Tutorial - GitHub Action, Nao, Medium, 2024]。

基本配置 (.github/workflows/contract-tests.yml)：


name: Contract Tests

on: [push, pull_request]

jobs:
  contract-validation:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.10'

      - name: Install dependencies
        run: |
          pip install pytest pytest-bdd jsonschema
          pip install -r requirements.txt

      - name: Run contract tests
        run: |
          pytest tests/contract/ -v --gherkin-terminal-reporter

      - name: Upload test report
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: contract-test-failures
          path: test-results/

關鍵參數：

  • --gherkin-terminal-reporter：在 CI 日誌中顯示 Gherkin 場景，提升可讀性 [來源：pytest-bdd Documentation, 2026]
  • if: failure()：僅在測試失敗時上傳報告，節省 artifact 空間

#### 並行執行加速

pytest-bdd 支援 pytest-xdist 並行執行 [來源：TestMu AI Learning Hub, n.d.]：


- name: Run contract tests (parallel)
  run: |
    pytest tests/contract/ -n 4 --gherkin-terminal-reporter

-n 4 會在 4 個 CPU 核心並行執行測試，適用於大量 Scenario Outline 變體的場景。

#### 報告格式整合

生成 Cucumber JSON 格式供 CI 視覺化工具使用：


pytest --cucumberjson=reports/cucumber.json

可與 Jenkins Cucumber Plugin、GitLab CI 的 Cucumber Reports 整合 [來源：pytest-bdd Documentation, 2026]。

5. 與本專案現有測試框架配合

#### 現有測試架構

daily-digest-prompt 目前有 1620 個測試 [來源：KB memory, 2026-03-28]：

  • tests/hooks/：Hook 規則測試（529 個）
  • tests/skills/：Skill 邏輯測試（27 個）
  • tests/tools/：工具函數測試（300 個）

#### pytest-bdd 的互補定位

| 測試類型 | 既有方式 | pytest-bdd 補強 |

|---------|---------|----------------|

| Hook 攔截規則 | test_pre_bash_guard.py 單元測試 | Feature: nul 禁令場景（可讀性提升） |

| 結果檔 schema | test_validate_results.py 單元測試 | Feature: Phase 2/3 交接契約（規格驅動） |

| workflow YAML 規則 | 手動驗證 | Feature: timeout 必填、output_schema 一致性（自動化） |

| 去重策略 | test_research_series_advisor.py 單元測試 | Feature: topic 冷卻期、飽和閾值（BDD 場景） |

#### 漸進式導入策略

Phase A（0-1 週）：選一個高價值場景試點

  • 對象：results/todoist-auto-*.json 的必填欄位驗證
  • 產出：1 個 features/result_validation.feature + 對應 step 定義
  • 目標：驗證 pytest-bdd 與既有 validate_results.py 整合可行性

Phase B（1-2 週）：擴展到 workflow 契約

  • 對象：workflows/*.yaml 的 output_schema 與實際 schema 一致性
  • 產出：features/workflow_contract.feature
  • 整合：pre-commit hook 或 GitHub Actions PR 檢查

Phase C（長期）：將分散規則統一為 BDD 場景

  • 對象：config/hook-rules.yaml、config/dedup-policy.yaml 等配置檔規則
  • 方式：將「禁止 nul」「topic 冷卻期 3 天」等規則寫成 feature
  • 價值：規則可被非工程人員閱讀，降低隱性知識依賴

實務應用建議

1. 結果檔 schema 驗證場景

具體實作：


# features/result_json_contract.feature
Feature: Phase 2 結果 JSON 契約
  所有 todoist-auto-*.json 必須符合統一 schema

  Scenario: tech_research 成功路徑
    Given 一個 tech_research 任務
    When 任務完成並寫入結果
    Then JSON 必須包含 "topic", "kb_imported", "done_cert"
    And "status" 應為 "success"
    And "kb_imported" 應為 true

  Scenario Outline: 不同任務的 status 驗證
    Given 一個 <task_type> 任務
    When 任務狀態為 <status>
    Then 必填欄位 "agent, status, summary" 必須存在
    And "summary" 長度 >= 20 字元

    Examples:
      | task_type       | status  |
      | tech_research   | success |
      | arch_evolution  | partial |
      | self_heal       | failed  |

價值：每次新增自動任務時，必須通過這些場景，確保交接契約不破壞。

2. workflow YAML 規則測試

當前痛點：workflows/.yaml 定義的 output_schema 與實際 config/schemas/.json 可能不一致。

BDD 解決方案：


Feature: Workflow 與 Schema 一致性

  Scenario: arch_evolution workflow 宣告的 schema 實際存在
    Given workflow "workflows/ooda-decide-arch-evolution-result-schema.json"
    When 讀取其 output_schema 欄位
    Then 對應的 schema 檔案必須存在
    And schema 必須是有效的 JSON Schema Draft 2020-12

  Scenario: 所有 workflow 的 task_types 對應有效 task_key
    Given 所有 "workflows/*.yaml" 檔案
    When 收集其 task_types 欄位
    Then 每個 task_type 必須在 "config/frequency-limits.yaml" 中定義

3. 去重策略契約測試


Feature: 研究去重策略契約

  Scenario: topic 冷卻期強制執行
    Given research-registry.json 中有主題 "pytest-bdd" 在 2026-04-10
    When 今天是 2026-04-11（1 天後）
    And 嘗試研究主題 "pytest-bdd"
    Then 去重檢查應該失敗
    And 錯誤訊息應包含 "3 天冷卻期"

  Scenario: 飽和閾值觸發換方向建議
    Given KB 中有 3 筆關於 "AI Agent" 的筆記
    When 嘗試研究 "AI Agent 相關主題"
    Then 去重檢查應該建議換方向
    And 建議訊息應包含 "saturation_threshold"

pre-commit 階段：

知識缺口與未來方向

1. 尚未驗證的假設

假設：pytest-bdd 的 step 定義與 PowerShell 腳本（如 atomic-result-json.ps1）可透過 JSON schema 共同驗證邏輯。

驗證方式：實作一個 PoC，讓 PowerShell 與 Python 都呼叫同一份 config/schemas/results-auto-task-schema.json，確保雙方驗證規則一致。

2. 效能影響評估

問題：加入 BDD 契約測試後，CI 管線會增加多少時間？

下一步研究：

  • 測量現有 pytest 測試套件執行時間（基準線）
  • 加入 10-20 個 Scenario Outline 後重新測量
  • 若增加 > 30 秒，考慮分層執行（快速契約測試 vs 完整測試）

3. 與 JSON Schema 驗證的分工

當前困惑：既有 tools/validate_results.py 已用 JSON Schema 驗證，BDD 是否重複？

建議研究方向：

  • JSON Schema：定義結構規則（必填欄位、型別、格式）
  • pytest-bdd：定義業務邏輯規則（如「KB 不可用時 kb_imported 必為 false」）
  • 兩者互補，schema 驗證結構，BDD 驗證語義

4. Step 定義的重用與維護

風險：隨著場景增加，step 定義可能碎片化（如 10 個 feature 檔案共用 50 個 step 函數）。

待研究：

  • Step 定義的組織模式（按領域分檔 vs 按步驟類型分檔）
  • pytest-bdd 的 step library 最佳實踐
  • 如何避免 step 定義命名衝突
