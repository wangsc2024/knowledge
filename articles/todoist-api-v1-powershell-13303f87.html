Todoist API v1 整合實務指南（PowerShell、分頁、Webhook 與 MCP）

技術概述

Todoist 在 2026 年的主軸不是再維護舊版 REST v1，而是以新的 Todoist API v1 統一先前的 Sync API v9 與 REST API v2。對 daily-digest-prompt 這類需要穩定擷取「今日已完成任務」的自動化系統，最重要的不是只會呼叫 tasks，而是理解哪個端點負責歷史資料、哪個機制負責增量同步，以及 webhook 與 MCP 各自適合做什麼。官方文件也顯示 Todoist 已把 AI 整合推到第一線，提供官方託管的 MCP 與 todoist-ai 工具庫；但對批次、可重跑、可驗證的排程工作而言，HTTP API 仍然是最可控的資料平面。

核心概念與原理

1. API v1 是「統一層」，不是舊 REST v1 的延伸

新 API v1 把 Sync API v9 與 REST API v2 合併，統一錯誤格式、物件命名與端點路徑，並明確要求使用 api.todoist.com 與全小寫 endpoint。這點對既有整合很重要，因為舊資料模型中的 items、notes 已對應到新的 tasks、comments。如果專案裡還混用舊命名，後續維護會很痛。

2. 抓「已完成任務」應優先用 completed 系列端點

如果需求是像本任務一樣分析今日已完成任務，最直接的是 GET /api/v1/tasks/completed/by_completion_date。官方文件指出它以完成時間區間查詢，單次預設 50 筆，並透過 next_cursor 做 cursor pagination。這比先抓 active tasks 再自行比對狀態可靠，因為完成後任務會移入歷史。

3. Sync API 仍有價值，但定位不同

Sync 端點仍保留，且官方文件明說它是例外區塊，保留部分 legacy naming。它適合做全量/增量同步、快取初始化與批次 command；但若只是查某天完成紀錄，直接用 completed endpoint 成本更低。換言之：completed/by_completion_date 解決查詢，/sync 解決狀態鏡像。

4. Webhook 是觸發器，不是權威資料源

Todoist 提供 item:completed 等 webhook 事件，但 webhook 的正確用途是「收到事件後喚醒流程」，不是直接把 payload 當唯一真相。較穩妥的作法是：Webhook 只負責通知，真正的資料仍回頭呼叫 API v1 查詢完成任務，這樣才能處理重送、漏事件與排程補跑。

5. MCP/AI 工具是控制平面，不取代 ETL

官方文件顯示 Todoist 已提供託管 MCP，且舊 todoist-mcp 倉庫已轉移到 todoist-ai。這很適合讓 Claude、Cursor、ChatGPT 直接操作 Todoist；但在 daily-digest-prompt，MCP 更適合作為互動查詢或人工介面，正式排程仍應維持 HTTP API，因為比較容易做重試、日誌、驗證與限流控制。

最佳實踐與常見陷阱

最佳實踐

  • 以 tasks/completed/by_completion_date 當作每日已完成任務的主來源，並持久化最後查詢區間與 next_cursor。
  • 對寫入型操作帶 X-Request-Id，把重試設計成冪等流程，避免重複建立任務。
  • 把 webhook 視為「增量提示」，收到 item:completed 後再回查 API，確認最終資料。
  • 如果未來要讓 AI 助手直接讀寫 Todoist，優先接官方 todoist-ai 或託管 MCP；排程批次工作則繼續走 HTTP API。
  • 在 PowerShell 中固定封裝 Authorization: Bearer ...、錯誤解析、cursor 分頁與 429/backoff 處理，避免每支腳本各自拼裝。

常見陷阱

  • 把舊 REST API v1 誤認為現行主版本。官方舊頁面已標示 deprecated，真正該用的是新的 Todoist API v1。
  • 用 active tasks 端點推測完成狀態，會漏掉已移入歷史的任務。
  • 忽略 opaque string ID。API v1 已把 ID 視為不透明字串，不能再假設是數字。
  • 把 webhook payload 直接落庫當真相，會讓補償與重播很難做。
  • 把 MCP 當成背景批次管線；MCP 比較像 agent 介面層，不是穩定 ETL 替代品。

與本專案的關聯

daily-digest-prompt 目前要分析「今日 Todoist 已完成任務所需的技術」，其第一步其實就是先把 completed tasks 的抓取方式做對。建議專案採兩層設計：

  1. 資料平面：PowerShell 直接呼叫 GET /api/v1/tasks/completed/by_completion_date，按日期區間抓取今日完成任務，處理 cursor 與重試。
  2. 控制平面：若未來要讓 Claude Code 或其他 AI 助手主動查任務、補任務、做互動式回顧，再接官方 MCP / @doist/todoist-ai。

這樣可避免把互動式 AI 協定混進每日排程主流程，也能讓結果 JSON、registry 與 KB 匯入維持可重現。

程式碼範例


function Get-TodoistCompletedTasks {
    param(
        [Parameter(Mandatory)] [string]$ApiToken,
        [Parameter(Mandatory)] [datetime]$Since,
        [Parameter(Mandatory)] [datetime]$Until
    )

    $headers = @{ Authorization = "Bearer $ApiToken" }
    $baseUrl = 'https://api.todoist.com/api/v1/tasks/completed/by_completion_date'
    $cursor = $null
    $items = @()

    do {
        $query = @{
            since = $Since.ToUniversalTime().ToString('o')
            until = $Until.ToUniversalTime().ToString('o')
            limit = 200
        }
        if ($cursor) { $query.cursor = $cursor }

        $uri = $baseUrl + '?' + (($query.GetEnumerator() | ForEach-Object {
            '{0}={1}' -f [uri]::EscapeDataString($_.Key), [uri]::EscapeDataString([string]$_.Value)
        }) -join '&')

        $response = Invoke-RestMethod -Method Get -Uri $uri -Headers $headers
        $items += $response.items
        $cursor = $response.next_cursor
    } while ($cursor)

    return $items
}

$todayStart = [datetime]::Parse('2026-03-11T00:00:00+08:00')
$todayEnd = [datetime]::Parse('2026-03-12T00:00:00+08:00').AddSeconds(-1)
$done = Get-TodoistCompletedTasks -ApiToken $env:TODOIST_API_TOKEN -Since $todayStart -Until $todayEnd
$done | Select-Object id, content, completed_at, project_id
