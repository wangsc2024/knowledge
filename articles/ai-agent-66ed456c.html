AI Agent 記憶檢索品質與錯租戶污染評測：深度技術研究報告

執行摘要

本研究檢視 AI Agent 長期記憶的兩個核心問題：檢索品質如何量化，以及多租戶環境如何避免錯租戶污染。多個 benchmark 與官方文件一致顯示，production memory 不能只看命中率，還要量測更新、衝突解決、恢復力與 abstention；而真正的隔離必須落在 namespace、index、account 等資料邊界，並搭配可回溯的寫入與檢索觀測。安全研究另指出，memory poisoning 會跨 session 持續擴散，因此記憶寫入面需要 baseline、快照與回滾。對本專案而言，最小可行方案是建立多維評測、tenant scope 測試、污染回放與事件級 observability，而不是只擴大 context window。

研究背景與動機

2026 年的 Agent 系統已逐漸把長期記憶視為基本能力。問題在於，只要系統從「單輪對話」走向「跨 session 記憶」，原本隱性的品質與安全問題就會浮上來。AWS 在 AgentCore 的官方文件與部落格都把短期事件、長期記憶、namespace、存取控制與 focused retrieval 明確列為核心設計點，這代表記憶層已從附屬功能變成一個獨立的基礎設施面 [來源：Amazon Bedrock AgentCore Memory: Building context-aware agents, 2025][來源：Memory organization in AgentCore Memory, 2026]。

同時，學術與社群 benchmark 也在修正過去「只要能從大量上下文答對問題，就代表記憶做得好」的假設。LongMemEval 顯示長期互動中的品質會明顯下滑；MemoryAgentBench 把 accurate retrieval 之外的 test-time learning、long-range understanding 與 conflict resolution 拉成獨立能力；LoCoMo-Plus 和 ShiftBench 更直接指出 latent constraints 與 distribution shift recovery 不能被傳統平均分數掩蓋 [來源：LongMemEval: Benchmarking Chat Assistants on Long-Term Interactive Memory, 2024][來源：Evaluating Memory in LLM Agents via Incremental Multi-Turn Interactions, 2025][來源：Locomo-Plus: Beyond-Factual Cognitive Memory Evaluation Framework for LLM Agents, 2026][來源：ShiftBench: Measuring Recovery of Agent Memory Under Distribution Shift, 2026]。

對本專案而言，這個主題的重要性更高。系統本身已同時存在 continuity、results、KB 與 registry 等多層記憶，若沒有明確的租戶邊界、寫入審核與檢索觀測，錯租戶污染不一定會以明顯 crash 出現，而更可能表現為「看似合理、實則拿錯上下文」的隱性失真。這種錯誤最難除錯，也最容易長期累積。

核心技術解析

第一個核心結論是：記憶檢索品質是多維問題，不是單一排序問題。LongMemEval 把長期記憶拆成資訊抽取、多 session 推理、時間推理、知識更新與 abstention；MemoryAgentBench 再加入衝突解決；LoCoMo-Plus 則指出真實 agent 還需要維持對隱含約束的穩定遵守。換句話說，系統不只要「撈到對的片段」，還要知道何時更新、何時拒答、何時覆寫舊記憶，以及在使用者偏好改變時能否恢復正確行為 [來源：LongMemEval: Benchmarking Chat Assistants on Long-Term Interactive Memory, 2024][來源：Evaluating Memory in LLM Agents via Incremental Multi-Turn Interactions, 2025][來源：Locomo-Plus: Beyond-Factual Cognitive Memory Evaluation Framework for LLM Agents, 2026]。

第二個核心結論是：租戶隔離不能只靠應用層條件判斷。AWS AgentCore 把 actorId、sessionId 與 namespace 當成記憶結構的基本識別；Azure AI Search 與 Azure Cosmos DB 則把 per-index、per-service、per-account 等模式當成不同隔離強度的正式設計選項。這些文件的共識是，真正的隔離一定要落在資料邊界與權限控制，而不是只在 prompt 裡補一句「僅取本使用者資料」[來源：Memory organization in AgentCore Memory, 2026][來源：Amazon Bedrock AgentCore Memory: Building context-aware agents, 2025][來源：Design patterns for multitenant SaaS applications and Azure AI Search, 2026][來源：Azure Cosmos DB 中向量搜尋的多租用戶管理, 2026]。若底層仍是共享索引或共享帳戶，應用層的租戶條件只是次級防線。

第三個核心結論是：observability 不是附加功能，而是品質與安全評估的必要前提。AgentCore 官方文件已經把 latency、errors、throttles、CreateEvent、RetrieveMemoryRecords、extract 與 consolidate 等訊號納入預設觀測資料。這意味著 production memory 若無法把每次寫入、檢索與整併追到 actor、session 與 namespace，就很難知道錯誤究竟發生在召回、抽取、權限還是資料污染 [來源：AgentCore generate memory observability data, 2026][來源：Memory organization in AgentCore Memory, 2026]。ShiftBench 進一步提醒，即使 aggregate accuracy 看起來正常，session interruption 後的恢復失敗仍可能非常嚴重，因此觀測資料必須保留足夠的回放能力 [來源：ShiftBench: Measuring Recovery of Agent Memory Under Distribution Shift, 2026]。

第四個核心結論是：長期記憶已成為持久攻擊面。OWASP Agent Memory Guard 直接把 memory poisoning 當成實務風險，並主張以 hash baseline、snapshot、rollback 與政策化 read/write gate 來保護記憶層 [來源：OWASP Agent Memory Guard, 2026]。DSRM 論文則展示了攻擊者如何把經過語義偽裝的內容注入知識庫，使 agent 在後續任務中穩定取回被污染的歷史知識；MemoryGraft 更把問題推進一步，指出少量毒化經驗就可能沿著語義相似任務長期擴散，形成跨 session 的行為漂移 [來源：Memory poisoning attacks on retrieval-augmented Large Language Model agents via deceptive semantic reasoning, 2026][來源：MemoryGraft: Persistent Compromise of LLM Agents via Poisoned Experience Retrieval, 2025]。這代表安全模型必須擴充到「寫入前審核、寫入後驗證、檢索時 provenance」三層，而不是只做輸入 prompt 過濾。

最後，Mem0 與 AgentCore 所代表的工程方向都顯示，結構化長期記憶比一味擴張 context 更適合 production。Mem0 在 LoCoMo 類任務上展示出更好的品質/延遲/成本平衡，AWS 也把策略化抽取與 focused retrieval 作為預設能力 [來源：Mem0: Building Production-Ready AI Agents with Scalable Long-Term Memory, 2025][來源：Amazon Bedrock AgentCore Memory: Building context-aware agents, 2025]。這不是說長 context 無用，而是說只靠它不足以解決跨 session 更新、衝突覆寫與租戶隔離這些真正麻煩的問題。

關鍵洞見

  • ✅ 洞見 1：production memory 的最低評測集合應至少包含 retrieval precision、更新成功率、衝突覆寫率、跨 session 恢復率與 abstention 正確率。支持來源數：5。
  • ✅ 洞見 2：錯租戶污染通常是 namespace、索引分區、權限與檢索範圍同時鬆動後的系統性失敗，不是單點 bug。支持來源數：5。
  • ✅ 洞見 3：觀測資料若不能關聯到 actor、session、namespace，就無法做有效污染鑑識。支持來源數：4。
  • ✅ 洞見 4：記憶寫入面必須被視為高風險 API，至少要具備 baseline、snapshot 與 rollback。支持來源數：3。
  • ✅ 洞見 5：結構化長期記憶比單純拉大 context 更適合線上系統的成本與治理需求。支持來源數：3。
  • ✅ 洞見 6：本專案的最小可行審計流程應先做 tenant scope fixture、污染回放 fixture 與 shift recovery fixture 三組測試。支持來源數：4。

實務應用建議

  1. 為每次記憶寫入加入明確的 tenant_id / actor_id / namespace 契約，並把這些欄位納入檢索 API 的必填條件；若呼叫端未提供，寧可拒絕而不是猜測。
  2. 在評測層建立三種固定回歸集：正常檢索、錯租戶污染、session 中斷後恢復。前兩者測安全邊界，後者測穩定性。
  3. 為記憶層新增事件級觀測欄位，至少追蹤 create、retrieve、extract、consolidate、rollback 五類事件，並保留可回放的 trace id。
  4. 對長期記憶寫入做兩段式治理：寫入前做政策檢查與 hash baseline，比對通過後再寫入；寫入後做抽樣驗證與異常告警。
  5. 若目前資源有限，先從「每租戶 namespace + 每次檢索 scope 檢查 + 污染回放測試」三件事開始，這是成本最低但收益最高的第一批控制點。

與現有知識的連結

本研究與 2026-04-10 已完成的「Agentic Memory × RAG 的混合架構：2026 AI Agent 長期記憶、租戶隔離與推理基礎設施」直接相連。前一份研究比較偏架構層，強調短期狀態、RAG 檢索與長期記憶應拆成三層；本次則把焦點收斂到其中最容易在 production 出問題的兩件事：如何量化記憶品質，以及如何證明沒有把別的租戶資料混進來。

因此，兩份研究合起來可以形成一條更完整的落地路線：先用前一份研究回答「記憶系統該怎麼分層」，再用本次研究回答「分層之後要怎麼量測、怎麼審計、怎麼防污染」。對 daily-digest-prompt 而言，下一步不是再新增一種記憶類型，而是先把既有 continuity、KB 與 registry 的邊界與評測補齊。

未來展望

短期內，記憶 benchmark 會持續從「答對多少題」轉向「在變動環境裡是否仍守住約束」。LoCoMo-Plus 與 ShiftBench 已經顯示，latent constraints 與 post-shift recovery 會成為下一波基準的主軸，這和真實產品場景更接近，因為使用者偏好、權限與任務脈絡本來就會變 [來源：Locomo-Plus: Beyond-Factual Cognitive Memory Evaluation Framework for LLM Agents, 2026][來源：ShiftBench: Measuring Recovery of Agent Memory Under Distribution Shift, 2026]。

中期來看，安全治理會從 prompt injection 防護擴展到 memory governance。也就是說，未來成熟的 Agent 平台不只要管模型輸入，還要管歷史記憶的 provenance、簽章、版本、回滾與租戶隔離證據。誰能把這些能力做成標準化審計介面，誰就更可能把長期記憶真正帶進企業生產環境。
