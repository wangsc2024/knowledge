四、深度問題分析

4.1 Critical 問題：語義快取形同虛設

現象：

  • kb_semantic_cache.py 工具完善（支援 lookup/store/stats）
  • 但 18 個 hybrid 查詢任務直接 curl 呼叫，完全繞過快取
  • 僅 1 個任務（kb_insight_evaluation）正確使用快取流程

根因：

  1. 模板未標準化：各任務 prompt 獨立撰寫，未統一引用快取步驟
  2. 缺少強制規範：knowledge-query SKILL.md 將快取標記為「可選」（步驟 0.5）
  3. 開發者不感知：新任務複製舊任務 prompt 時，繼承了「無快取」模式

影響：

  • KB API 負載增加：相同語義查詢重複執行（如「RAG 優化」每日查 5+ 次）
  • 延遲未優化：混合搜尋平均 800ms，快取可降至 50ms（16x 提速）
  • Token 浪費：快取 miss 導致多次 LLM 調用（語義相似查詢）

4.2 P1 問題：網路超時無防護

現象：

所有 KB 查詢的 curl 命令無 --connect-timeout 和 --max-time 參數。

風險場景：

  1. 知識庫服務卡住：curl hang 30+ 秒，自動任務超時失敗
  2. 網路不穩定：連線延遲 10+ 秒，整體 Phase 2 延遲
  3. 無 fail-fast：長時間等待而非快速降級

建議修復：


# 當前（無防護）
curl -s -X POST "http://localhost:3000/api/search/hybrid" -d @query.json

# 修復後（有防護）
curl -s --connect-timeout 10 --max-time 30 \
  -X POST "http://localhost:3000/api/search/hybrid" -d @query.json

def hybrid_search_with_cache(query: str, topK: int = 5, namespace: str = "kb_query"):
    """混合搜尋 + 自動快取封裝"""
    # 1. 嘗試快取
    cache_result = lookup_cache(namespace, f"hybrid:topK={topK}", query)
    if cache_result["hit"]:
        return json.loads(cache_result["response"])

    # 2. 呼叫 API
    response = requests.post(f"{KB_BASE}/api/search/hybrid",
                            json={"query": query, "topK": topK},
                            timeout=(10, 30))  # connect/read timeout

    # 3. 存快取
    store_cache(namespace, f"hybrid:topK={topK}", query, response.text)

    return response.json()

Step 1：創建 Chunk 預處理工具


# tools/kb_chunk_preprocessor.py

def semantic_chunking(text: str, max_chunk_size: int = 2000):
    """語義分段：依 embedding 聚類分段"""
    # 1. 依段落初步分割
    paragraphs = text.split("\n\n")

    # 2. 計算每段 embedding
    embeddings = [get_embedding(p) for p in paragraphs]

    # 3. 聚類（cosine similarity < threshold 則斷開）
    chunks = []
    current_chunk = []

    for i, para in enumerate(paragraphs):
        current_chunk.append(para)

        # 達到最大長度或語義跳轉
        if (len("\n\n".join(current_chunk)) >= max_chunk_size or
            (i + 1 < len(paragraphs) and
             cosine_sim(embeddings[i], embeddings[i+1]) < 0.85)):
            chunks.append("\n\n".join(current_chunk))
            current_chunk = []

    return chunks

def enrich_chunk_with_context(chunk: str, doc_title: str, doc_summary: str):
    """Contextual Retrieval：加文件級上下文"""
    prefix = f"【文件：{doc_title}】{doc_summary}\n\n"
    return prefix + chunk

# tools/kb_reranker.py

from flashrank import Ranker, RerankRequest

def rerank_results(query: str, results: list, top_n: int = 5):
    """混合搜尋結果 Reranking"""
    ranker = Ranker(model_name="ms-marco-MiniLM-L-12-v2")

    # 構造 passages
    passages = [{
        "id": i,
        "text": r["contentText"][:500],  # 取前 500 字
        "meta": {"noteId": r["noteId"], "title": r["title"]}
    } for i, r in enumerate(results)]

    # Rerank
    rerank_request = RerankRequest(query=query, passages=passages)
    reranked = ranker.rerank(rerank_request)

    # 返回 top_n
    return [results[p["id"]] for p in reranked[:top_n]]

Step 3：整合到混合搜尋


# tools/kb_query_with_cache.py 擴展

def hybrid_search_with_rerank(query: str, topK: int = 10, rerank_top: int = 5):
    # 1. 混合搜尋（topK = 10，多取一些候選）
    results = hybrid_search_with_cache(query, topK=topK)

    # 2. Rerank 精排（保留 top 5）
    reranked = rerank_results(query, results["results"], top_n=rerank_top)

    return {"query": query, "results": reranked, "reranked": True}

九、關鍵學習與洞見

9.1 為何語義快取採用率如此低？

根因分析：

  1. 開發模式問題：新任務複製舊任務 prompt → 繼承「無快取」模式
  2. SKILL.md 標記錯誤：將快取標為「可選」而非「強制」
  3. 無強制規範：pre_bash_guard 未檢查 KB 查詢是否經快取

深度洞見：

技術實現 ≠ 實際採用。工具再完善，若無「強制使用」機制，採用率仍會趨近 0。

對比案例：

  • Fail-Safe 佔坑：強制規範 → 100% 執行
  • 語義快取：可選建議 → 4.3% 執行

啟示：未來優化應：

  1. 在 Hook 層強制檢查（如 pre_bash_guard 檢測 curl KB 必經快取）
  2. 模板提供「已完成範例」而非「待實現步驟」
  3. check-health.ps1 監控採用率，低於 80% 時告警

9.2 RAG 管線的「最後一哩」問題

觀察：

  • 知識庫服務完善（8 個端點）
  • 應用層深度整合（37/47 任務）
  • 但「連接層」效率極低（快取 4.3%、超時無防護）

類比：

RAG 管線就像高速公路：兩端（服務端/應用端）都完善，但「交流道」（快取/路由/容錯）卻未優化，導致整體效率低落。

深度洞見：

系統優化的 Pareto 原則：80% 效益來自 20% 的「連接層」優化（快取/路由/容錯），而非兩端能力提升。

9.3 知識庫驅動開發（KDD）的威力

本次優化流程：


查詢 KB（RAG 優化歷史）
  ↓
發現 3 個已實施優化 + 2 個未完成優化
  ↓
深度分析 37 個任務的實際使用情況
  ↓
發現 Critical 問題（快取覆蓋率 4.3%）
  ↓
制定 4 階段優化方案（P0-P3）
  ↓
預估效益（延遲 -93.75%、MRR +9.3pp）

時間對比：

  • 無 KB：從零開始調研 RAG 優化技術 → 需 4-6 小時
  • 有 KB：直接讀取歷史優化報告 → 僅需 30 分鐘

深度洞見：

知識庫不僅是「記憶體」，更是「加速器」。每次優化都是下次優化的基礎，形成「知識複利」效應。
