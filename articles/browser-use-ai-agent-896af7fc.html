一、專案概述與市場定位

Browser Use（browser-use/browser-use）是由 Browser Use 團隊開發的開源 AI 瀏覽器自動化框架，以 Python 為主語言、Playwright 為底層引擎，讓 AI Agent 能透過自然語言指令自主操作網頁。截至 2026 年 4 月已累積 85,900+ Stars、9,900+ Forks，是 GitHub 上 AI 瀏覽器自動化領域星數最高的專案 [來源：GitHub Repo, 2026]。

AI 瀏覽器市場預計從 2024 年的 45 億美元成長至 2034 年的 768 億美元（CAGR 32.8%），Browser Use 正處於這波浪潮的核心位置 [來源：Web Search, 2026]。三大趨勢使瀏覽器 Agent 在 2026 年成為可行方案：LLM 推理能力突破（GPT-4o、Claude 4、Gemini 2.5 能準確解讀網頁結構）、開源生態成熟、以及企業自動化需求爆發。

二、核心架構：三層組件設計

Browser Use 的架構圍繞三個整合系統運作 [來源：Labellerr, 2026；GitHub Repo, 2026]：

2.1 LLM 整合層（Control Plane）

LLM 作為 Agent 的控制平面，負責任務分解、行動規劃與決策。框架是模型不可知的（model-agnostic），透過 LiteLLM 統一介面支援 OpenAI、Anthropic、Google、以及本地 Ollama 模型 [來源：Firecrawl, 2026；GitHub Repo, 2026]。此外，團隊推出專門優化的 ChatBrowserUse 模型，宣稱完成任務速度比通用模型快 3-5 倍 [來源：GitHub Repo, 2026]。

2.2 瀏覽器控制引擎（Actuator）

底層使用 Playwright 透過 WebSocket 與 Chromium 通訊，執行點擊、輸入、導航等操作。Playwright 扮演 Agent 的「執行器」——收到明確指令後以可預測行為執行 [來源：NxCode, 2026；Labellerr, 2026]。

2.3 視覺理解系統（Perception）

這是 Browser Use 的關鍵區分特徵：它不只讀取 DOM 程式碼，還會同時截取螢幕截圖進行視覺分析，形成 DOM + Vision 的混合感知模式 [來源：Labellerr, 2026]。這種雙重感知使系統對網站設計變更具有天然韌性。

三、核心技術深度分析

3.1 Perception-Action Loop（感知-行動迴圈）✅

這是 Browser Use 的核心運作機制，模擬人類瀏覽行為的迭代循環 [來源：Labellerr, 2026；Firecrawl, 2026；NxCode, 2026]：

  1. 使用者提供任務指令 → Agent 接收
  2. 同步捕捉頁面狀態：DOM 提取 + 螢幕截圖
  3. LLM 分析：結合結構化與視覺資料理解頁面
  4. LLM 規劃下一步行動（點擊 / 輸入 / 導航）
  5. Playwright 執行行動
  6. 觀察結果，將新狀態回饋給 LLM
  7. 循環直到任務完成

這個迴圈的設計使 Agent 能處理動態頁面、AJAX 載入、以及非預期的彈窗干擾——因為每一步都重新感知頁面狀態。

3.2 DOM Distillation（DOM 蒸餾）✅

Browser Use 不將原始 HTML 或完整截圖餵給 LLM，而是採用 DOM 蒸餾技術：將頁面精簡為核心互動元素（按鈕、連結、表單欄位），大幅降低 token 消耗 [來源：Firecrawl, 2026]。

蒸餾後的元素以索引編號標記，LLM 只需指定「點擊元素 #7」即可執行操作。這比傳統 CSS 選擇器或 XPath 方法更具韌性——當網站改版導致選擇器失效時，DOM 蒸餾仍能正確辨識互動元素。

與傳統方法的對比：

  • 傳統 Playwright：依賴 CSS/XPath 選擇器，網站改版後 30 天內有 15-25% 故障率 [來源：NxCode, 2026]
  • Browser Use：透過 DOM 蒸餾 + 視覺辨識，prompt 調整需求 < 5% [來源：NxCode, 2026]

3.3 Multi-Tab 狀態管理 ✅

Browser Use 原生支援多分頁操作——Agent 可同時在多個瀏覽器分頁中工作。系統維護跨導航步驟的對話歷史與頁面上下文，實現需要「記憶」先前互動的複雜有狀態工作流 [來源：Firecrawl, 2026；Labellerr, 2026]。

3.4 自訂工具擴展 ✅

使用者可透過裝飾器模式（decorator pattern）擴展 Agent 功能，整合外部系統與資料持久化。自訂工具無縫接入 Agent 的決策流程 [來源：Labellerr, 2026]。

五、對 Daily-Digest-Prompt 系統的五大借鏡啟示

啟示 1：Perception-Action Loop → Agent 決策模式優化

Browser Use 的感知-行動迴圈與本專案的 OODA 閉環（Observe→Orient→Decide→Act）高度同構。借鏡方向：在 Todoist Agent 的任務執行步驟中引入「執行後感知」機制——每個 Phase 完成後捕捉狀態快照（類似 DOM snapshot），讓下一步驟能根據實際結果調整策略，而非假設前一步成功。

啟示 2：DOM Distillation → Context 壓縮策略

DOM 蒸餾的核心思想——從龐大資訊中萃取核心互動元素——直接對應本專案的 Context 壓縮需求（ADR-036）。借鏡方向：對大型 state 檔案（research-registry.json、research-series.json）實施「蒸餾式讀取」，只提取當次決策所需的欄位子集，降低 token 消耗。

啟示 3：Model-Agnostic 設計 → LLM Router 強化

Browser Use 透過 LiteLLM 實現模型不可知，與本專案的 tools/llm_router.py 理念一致。借鏡方向：參考 Browser Use 的 ChatBrowserUse 專用模型策略，考慮為特定任務類型（如新聞摘要、研究分析）調配不同模型參數（temperature、max_tokens），而非全部使用相同配置。

啟示 4：自訂工具擴展模式 → Skill 系統增強

Browser Use 的裝飾器模式自訂工具，讓開發者能以最小侵入性擴展 Agent 功能。借鏡方向：參考此模式設計 Skill 的 pre/post hook 機制——讓 Skill 能在執行前後自動注入預處理或後處理邏輯（如快取檢查、結果驗證），無需修改 Skill 核心程式碼。

啟示 5：自修復韌性 → Hook 系統進化

Browser Use 對網站變更的 < 5% 調整需求（vs Playwright 的 15-25%）來自其語義理解能力。借鏡方向：在 pre_bash_guard.py 中引入「語義匹配」層——當 API endpoint 微調（如 /v1/ → /v2/）時，不直接硬性阻擋，而是嘗試智慧映射到新路徑，降低假陽性攔截。

六、知識缺口與未來研究方向

  1. DOM distillation 演算法細節：官方未開源蒸餾演算法的具體實作，需深入原始碼分析
  2. ChatBrowserUse 專用模型：宣稱 3-5x 加速的微調策略細節不明
  3. 多 Agent 協作：Browser Use 支援平行 Agent，但跨 Agent 狀態同步機制待研究
  4. 安全模型：Agent 自主操作網頁的安全邊界（如避免意外購買、資料洩漏）
  5. WebVoyager 基準差異：官方 89.1% vs 獨立 72-78% 的落差需進一步釐清
