端雲協同與 NPU 感知的多模態 Agent 部署：小模型在裝置端與雲端之間的協作架構

執行摘要

本研究聚焦多模態 Agent 在裝置端與雲端之間的協作架構，核心結論是：端側小模型不應被視為雲端大模型的廉價替代，而應被設計為一個具備明確分工的前線執行層，負責低延遲感知、隱私敏感資料處理、局部工具調用與離線回應；雲端則承接長上下文推理、跨系統協調、較高成本的多步規劃與集中治理 [來源：Gemini Nano, 2026] [來源：On-device small language models with multimodality, RAG, and Function Calling, 2025] [來源：LLM Inference guide, 2025]。研究同時顯示，若要讓端雲協同真正可產品化，僅有模型壓縮還不夠，還必須把 runtime、量化、NPU/CPU/GPU 路由、快取策略、RAG 索引生命週期與觀測資料結構整合成同一套工程協議 [來源：ONNX Runtime — Mobile 教學, 2025] [來源：ExecuTorch Overview, 2026] [來源：OpenTelemetry — Gen AI spans, 2025]。

就三個研究問題而言，答案可概括如下。第一，多模態 Agent 的合理切分方式，是把感知、局部檢索、低風險 function calling 與失敗預檢放在端側，把高成本推理、跨裝置協作、跨租戶知識整合與長鏈路工具編排保留在雲端 [來源：On-Device Function Calling in Google AI Edge Gallery, 2026] [來源：AI Edge RAG guide, 2025] [來源：Gemma 3, 2025]。第二，NPU-aware runtime 與量化不是單純追求 benchmark，而是決定哪些路徑能在可接受功耗與記憶體預算下常駐裝置端；因此部署策略應先以延遲、功耗與熱限制建立服務等級目標，再回推模型大小、delegate 與 fallback 設計 [來源：ExecuTorch Edge, 2026] [來源：ONNX Runtime — Mobile 教學, 2025]。第三，在端雲協同場景中，RAG、prompt/prefix caching、semantic caching 與 tracing 必須共用版本與 partition 邊界，否則會出現命中不一致、回退不可追與跨情境污染 [來源：Anthropic — Prompt caching, 2025] [來源：Redis — What is semantic caching?, 2024] [來源：Semantic Caching for Low-Cost LLM Serving, 2024]。整體而言，最務實的架構方向不是追求「全離線」或「全雲端」，而是建立可觀測、可回退、可分區治理的分層 Agent 系統。

背景與動機

2025 到 2026 年間，端側生成式 AI 的產品敘事從單純「在手機上跑模型」快速轉向「在裝置上完成有用工作」。Gemma 3、Gemma 3n、Gemini Nano、FunctionGemma 與 AI Edge Gallery 的公開材料顯示，產業關注點已從模型是否能執行，轉為它能否在有限硬體上完成多模態理解、工具調用、檢索增強與互動式工作流 [來源：Gemma 3, 2025] [來源：FunctionGemma: Bringing bespoke function calling to the edge, 2025] [來源：On-Device Function Calling in Google AI Edge Gallery, 2026]。

這個轉變對 daily-digest-prompt 這類 Agent 系統尤其重要。專案本身已經碰到快取治理、回退路由、知識庫同步與觀測一致性等問題；若未來要把部分能力推向端側，例如離線摘要、私有筆記檢索、裝置內工具調用或本地預判，就不能只把「換成小模型」當成策略，而必須重畫整個資料流、控制流與遙測面 [來源：OpenTelemetry — Gen AI spans, 2025] [來源：LangChain — cache（含 semantic cache 層）, 2026]。

因此，本研究的動機有兩層。其一，是釐清端側多模態小模型與雲端大模型之間的功能分工，避免把所有能力都推到一端，導致體驗、成本與可靠性同時惡化。其二，是把 NPU-aware runtime、RAG、caching 與 observability 放回同一張架構圖中檢視，因為真實產品失敗往往不是模型能力不足，而是系統邊界定義錯誤。

核心技術解析

多模態 Agent 的第一個關鍵，不是模型本身，而是推理邊界。Google 在 Gemini Nano 與 AI Edge LLM Inference 的文件中反覆強調，裝置端模型適合處理低延遲、低成本、隱私優先的任務，並透過系統層服務與裝置硬體取得穩定推理能力 [來源：Gemini Nano, 2026] [來源：LLM Inference guide, 2025]。這代表端側的最佳用途通常是「先看、先篩、先做」而不是「全權決策」：例如把語音、圖片、短文件先轉成結構化候選，再根據複雜度決定是否上雲。這一點與 Google 2025 年發布的 AI Edge SLM/RAG/Function Calling 路線一致，該路線把多模態輸入、RAG 與 function calling 明確視為可在 edge 端組裝的能力積木，而非單一模型的神奇屬性 [來源：On-device small language models with multimodality, RAG, and Function Calling, 2025]。這個主張已有至少三個獨立來源支持，屬於已驗證共識 ✅。

第二個關鍵是 function calling 的部署位置。FunctionGemma 與 2026 年的 AI Edge Gallery 更新說明了一個重要方向：把工具調用本身做成小型專用模型，讓自然語言到函式呼叫的轉譯可在裝置端即時完成 [來源：FunctionGemma: Bringing bespoke function calling to the edge, 2025] [來源：On-Device Function Calling in Google AI Edge Gallery, 2026]。這種設計的價值不只是快，而是讓低風險、明確 schema 的動作可以不經網路就完成，例如開 app、調整設定、建立行事曆項目。從架構角度看，這意味著 Agent 應把工具分級：本地白名單工具由端側 function-calling 模型直接處理，高風險或跨系統工具再交由雲端代理。這種「工具風險分層」與專案內既有 guardrails 思路可以直接對接，屬於已驗證洞見 ✅，目前至少有 Google 的兩篇官方資料與 Gemini/Gemma 產品說明間接支持。

第三個關鍵是 runtime 與硬體路由。ONNX Runtime Mobile 與 ExecuTorch 都把 edge 部署問題描述為「如何在多種 backend 與加速器之間，把模型轉成可執行、可維護、可剖析的形式」 [來源：ONNX Runtime — Mobile 教學, 2025] [來源：ExecuTorch Overview, 2026] [來源：ExecuTorch Edge, 2026]。這些文件共同指出，實際部署不只是選一個模型檔，而是選一條 execution path：CPU、GPU、NPU、DSP 或混合委派，各自對延遲、電量、記憶體與支援算子有不同影響。因此，所謂 NPU-aware deployment 的真正意義，是讓部署決策由服務目標反推。例如，若互動場景要求 P95 延遲小於 300ms、連續使用 10 分鐘不降頻、記憶體占用不超過某個門檻，那麼模型量化位寬、KV cache 大小、分塊推理與 delegate 選擇都必須服務於這些 SLO，而不是相反。這是由多來源共同支持的 ✅ 結論。

第四個關鍵是 RAG 與 cache 的治理。Google AI Edge RAG guide 已把端側 RAG 管線拆成匯入、切塊、向量化、索引與檢索生成等步驟 [來源：AI Edge RAG guide, 2025]。但如果把這個流程與 Anthropic 的 prompt caching、Redis 的 semantic caching、Microsoft Research 的 semantic caching 論文放在一起看，會得到更完整的系統圖：RAG 負責把知識拉近模型，prefix/prompt cache 重用穩定前綴，semantic cache 重用「近似意圖」的既有答案，而這三者都可能在端與雲兩邊各自存在 [來源：Anthropic — Prompt caching, 2025] [來源：Redis — What is semantic caching?, 2024] [來源：Semantic Caching for Low-Cost LLM Serving, 2024]。因此，若 partition key 沒有綁定 task_key、model_id、kb_snapshot、prompt_version 與租戶資訊，就會產生跨版本命中、過期知識回覆與錯誤回退。這是本研究最重要的已驗證洞見之一 ✅。

第五個關鍵是 observability。OpenTelemetry 的 GenAI spans 規範提供了端雲協作最缺的一塊拼圖：如何把 prompt、completion、tool call、usage 與錯誤路徑用一致的 span 結構表示 [來源：OpenTelemetry — Gen AI spans, 2025]。對端雲 Agent 而言，若沒有統一 trace 邊界，就無法回答「慢的是本地推理、向量檢索、網路往返還是雲端 fallback」，也無法建立穩定的 burn-rate 或故障歸因。這點與專案當前對 JSONL、OTLP 與 phase-based tracing 的需要高度一致，且有至少三個來源支持，因此可標記為 ✅。

關鍵洞見

  1. ✅ 已驗證（4 來源） 端側與雲端最穩健的分工，不是「小事在端上，大事上雲」這種模糊原則，而是明確按資料敏感度、延遲預算、工具風險與上下文長度切分。可在端上完成的應是感知前處理、輕量檢索、白名單工具與離線回應；雲端保留長鏈規劃與跨系統協調 [來源：Gemini Nano, 2026] [來源：LLM Inference guide, 2025] [來源：On-device small language models with multimodality, RAG, and Function Calling, 2025] [來源：On-Device Function Calling in Google AI Edge Gallery, 2026]。
  1. ✅ 已驗證（4 來源） RAG、prompt cache 與 semantic cache 必須共用版本治理，不可被當成三個孤立優化器。對 Agent 系統而言，它們本質上都是「降低重算與降低未知」的檢索/重用層，只是作用位置不同 [來源：AI Edge RAG guide, 2025] [來源：Anthropic — Prompt caching, 2025] [來源：Redis — What is semantic caching?, 2024] [來源：Semantic Caching for Low-Cost LLM Serving, 2024]。
  1. ✅ 已驗證（3 來源） NPU-aware runtime 的成功條件不是追求單一 benchmark，而是把 latency、power、thermal 與 memory budget 前置成 SLO，再用 runtime delegate、量化與模型尺寸去滿足它 [來源：ONNX Runtime — Mobile 教學, 2025] [來源：ExecuTorch Overview, 2026] [來源：ExecuTorch Edge, 2026]。
  1. ✅ 已驗證（3 來源） 端雲回退鏈若沒有標準化 span 與結構化失敗原因，將很難在真實系統中調優。每次 fallback 都應記錄觸發條件，例如超時、置信度不足、工具不可用或資料版本不匹配 [來源：OpenTelemetry — Gen AI spans, 2025] [來源：LangChain — cache（含 semantic cache 層）, 2026] [來源：Anthropic — Prompt caching, 2025]。
  1. ⚠️ 待驗證（2 來源） 對多模態 Agent 而言，最佳策略可能是先在端側完成感知與輕量語義壓縮，再將壓縮後的中介表徵送往雲端，而不是把原始多媒體資料直接上傳。這有助於隱私與成本，但目前公開文件多為產品敘事，缺乏一致 benchmark，仍需更多一手實測 [來源：Gemma 3, 2025] [來源：On-device small language models with multimodality, RAG, and Function Calling, 2025]。
  1. ⚠️ 待驗證（2 來源） 不同硬體平台對 per-layer NPU 切分與 delegate 支援差異極大，公開文件通常只到框架層，真正可移植的最佳化策略仍需以裝置實測決定 [來源：ExecuTorch Edge, 2026] [來源：ONNX Runtime — Mobile 教學, 2025]。

實務應用建議

若把這份研究落到 daily-digest-prompt 或個人專案，最直接的做法不是立刻做完整 mobile app，而是先在現有 Agent 架構中引入「端雲路由思維」。第一步，可以把目前的工作流節點分成三類：禁止快取、可前綴快取、可語意快取；再加上一個欄位標記「可否離線執行」。這樣即使還沒有真正的端側模型，也能先把未來需要的 partition key 和 observability 契約固定下來。

第二步，是把 fallback 從隱性行為改成顯性資料。每一個從本地模型切到雲端、從快取切到重算、從 RAG 命中切到一般推理的轉換，都應生成結構化事件或 trace span。對 daily-digest-prompt 而言，這可以直接延伸既有的 phase、hook 與 JSONL/OTLP 管線，未來若接入本地模型或離線知識索引，才不會失去可除錯性。

第三步，是把 KB snapshot 與 prompt version 納入快取鍵。專案已經明確知道某些任務不適合快取，例如即時 Todoist 狀態與當日研究；下一步應把這種規則升級為系統欄位，而不是分散在 prompt 文字中。這樣未來即使擴展到端雲 Agent，也能避免端側命中舊資料、雲端回答新資料的分裂狀態。

第四步，是在真正碰硬體前先用 SLO 驅動假設。與其先決定「我要用哪個 NPU backend」，不如先定義幾個可驗證目標，例如單輪本地摘要 P95 < 500ms、回退率 < 10%、本地索引同步延遲 < 5 分鐘、熱節流下功能降級但不中斷。這樣之後評估 Gemini Nano、Gemma 3n、ExecuTorch 或 ONNX Runtime 時，才有穩定尺規。

與現有知識的連結

本研究與專案近兩日的幾個主題呈現高度連續性。首先，它延伸了 semantic caching 的研究，但把焦點從單一服務成本優化，推進到端雲 Agent 的知識一致性與回退治理。其次，它與 Coding Agent observability / SLO 研究互補，指出觀測不該只停留在雲端 LLM 呼叫，而要延伸到本地推理、向量檢索與 fallback 路徑。第三，它也與 tool-use guardrails 研究相連，因為 function calling 一旦下放到裝置端，工具白名單、權限邊界與結構化事件紀錄就變成核心控制面，而非附屬功能。

換言之，這份研究不是孤立的「行動端 AI 筆記」，而是對整個專案未來架構的一次前瞻補圖：如果系統未來要兼容本地模型、私有資料與多層快取，就必須提前把路由、快取、RAG、trace 與回退當成同一套協議設計。

未來展望

下一步最值得深化的方向是「端雲協同與 NPU 部署實測」。目前公開文件已足夠支持宏觀架構判斷，但在以下問題上仍缺實測資料：不同裝置上 Gemma 3n / Gemini Nano 的真實延遲與熱表現、端側 RAG 的索引大小上限、function calling 在本地 schema 複雜化後的準確率、以及多種 runtime 在同一任務上的 delegate 差異。若後續研究能加入裝置實測矩陣，這條研究線就能從架構建議升級為部署手冊。

另一個值得追蹤的方向，是把端側多模態中介表徵與隱私治理結合。若端側能先把語音、影像或文件壓縮為低敏感度特徵，再送雲端做高階規劃，將可同時改善成本、延遲與法規風險。但目前這部分公開證據仍不足，應列為後續 ⚠️ 驗證主題。
