執行摘要

PETsARD（發音 /pəˈtɑrd/，全名 Privacy Enhancing Technologies Analysis, Research, and Development）是由台灣國家資通安全研究院（NICS-CAPE）主導開發、數位發展部（moda）委託支持的開源 Python 程式庫，專注於合成表格資料的生成與多維度評估 [1][2]。

本報告的核心發現如下：

  1. 模組化管線設計：PETsARD 採用宣告式 YAML 配置驅動的八模組管線（Loader→Splitter→Preprocessor→Synthesizer→Postprocessor→Constrainer→Evaluator→Reporter），架構清晰、可拆可組。
  1. 多元合成演算法：同時提供自研高斯 Copula（Numba JIT + PyTorch GPU 加速）及 SDV 整合方案（CTGAN、TVAE、CopulaGAN、GaussianCopula），並支援自訂演算法接口。
  1. 領先的評估框架：整合 GDPR 三維隱私風險（Anonymeter：Singling Out / Linkability / Inference）、SDMetrics 保真度與效用評估（XGBoost / K-means），並自研實驗性 mpUCCs（Maximal Partial Unique Column Combinations）演算法——聲稱比 Anonymeter 快 16 倍且為完整評估而非抽樣。
  1. 政府級應用定位：目標受眾明確包含金融保險業、法律業及政府機關，提供離線部署方案，適合企業內網環境。
  1. 成熟度現況：截至 2026 年 6 月，已釋出 v1.10.2（累計 42 版），PyPI 狀態為 Production/Stable，但仍有若干限制（Python 3.10–3.11 嚴格限定、多表支援缺失、mpUCCs 仍為實驗性功能）。

1. 引言與背景

1.1 政策驅動背景

隨著台灣《個人資料保護法》修法討論與歐盟 GDPR 影響擴散，政府與醫療、金融、法律等受監管產業對資料共享的合規需求急速升高。合成資料（Synthetic Data）作為一種隱私強化技術（Privacy Enhancing Technology, PET），已被新加坡 PDPC [3]、英國 ICO、歐盟 ENISA 等多個監管機構認可為資料開放的可行路徑。

台灣數位發展部（moda）於 2022 年後積極佈局數位主權與資料治理，國家資通安全研究院（NICS）旗下前瞻研究籌獲中心（CAPE）獲委託開發一套標準化、可驗證、開源的合成資料工具鏈，PETsARD 即是此政策目標的具體產物 [1]。

1.2 命名與定位

PETsARD 字面意涵為「隱私強化技術分析、研究與開發」，以此凸顯其不僅是單一合成工具，而是涵蓋生成→評估→報告完整研究管線的平台。名稱本身也是一個雙關：法語 pétard（爆竹）的轉寫，暗示對隱私威脅的「爆破式」應對。

1.3 版本演進

  • 初期版本（moda-gov-tw/PETsARD）：數位部 GitHub 組織，已於 2025-06-05 封存，為早期試作版。
  • 現役版（nics-dp/petsard）：NICS-CAPE 主要維護，v1.3.1 起（2025-05-08）至今，現為 v1.10.2（2026-06-22）。
  • 鏡像版（nics-tw/petsard）：nics-tw 組織的同步鏡像，顯示 6 stars、42 個版本 [4]。

2. 架構設計

2.1 整體管線

PETsARD 採取八模組管線，以 YAML 或 Python API 驅動：


Loader → Splitter → Preprocessor → Synthesizer → Postprocessor
                                                       ↓
                                             Constrainer → Describer
                                                       ↓
                                             Evaluator → Reporter

頂層 Executor 統一協調整個管線，提供 run()、get_result()、get_timing()、is_execution_completed()、get_inferred_schema() 等方法。

2.2 各模組職責

| 模組 | 核心功能 |

|------|---------|

| Loader | 從 CSV / Excel 載入資料；支援 benchmark:// 協議從 AWS S3（petsard-benchmark）直接下載基準資料集 |

| Splitter | 預設 80/20 訓練測試分割；支援 custom_data 注入外部合成資料 |

| Preprocessor | 缺失值與離群值處理；Encoding（類別型）、Scaling（數值型）、Discretizing（分箱） |

| Synthesizer | 合成資料生成，見第 3 節 |

| Postprocessor | 逆轉換，還原至原始格式 |

| Constrainer | 業務邏輯約束：NaN groups、欄位約束、欄位比例；resample_until_satisfy() 迭代直到滿足約束 |

| Describer | 欄位統計描述與 Schema 文件化 |

| Evaluator | 多維度評估，見第 4 節 |

| Reporter | 儲存合成資料、評估報告、Schema、時間記錄；支援 global / columnwise / pairwise / details 四粒度 |

2.3 Schema YAML：設計的單一真相來源

PETsARD 以獨立的 Schema YAML 管理欄位定義，包含：

  • 資料型別（數值型、類別型、日期型等）
  • 數值精度（整數位、小數位）
  • 邏輯型別（email、phone 等語義型別）
  • 統計資訊（範圍、基數）

這種設計使欄位的業務語義與合成流程解耦，便於在多次實驗中重複使用同一份欄位定義。

2.4 YAML 工作流配置範例


Executor:
  Loader:
    filepath: data/adult-income.csv
  Splitter:
    test_split_ratio: 0.2
  Synthesizer:
    method: petsard-gaussian-copula
  Evaluator:
    - method: anonymeter-singlingout
    - method: anonymeter-linkability
    - method: sdmetrics-qualityreport
    - method: mlutility-classification
  Reporter:
    output_dir: output/

3. 合成演算法（Synthesizer）

PETsARD 提供四個層次的合成選項：

3.1 自研：PETsARD Gaussian Copula（預設方法）

以高斯 Copula 理論為核心，實作五步合成流程：

  1. 邊際轉換：將各欄位映射至均勻分佈
  2. 高斯化：透過逆正態 CDF 轉換至標準常態
  3. 相關結構學習：估計 Pearson 相關矩陣
  4. 聯合取樣：從多變量常態分佈抽樣
  5. 逆轉換：還原至原始分佈

效能優化：

  • Numba JIT 編譯：首次執行需編譯（數十秒），後續執行加速 2–3 倍，極端場景可達 10–100 倍加速
  • PyTorch GPU 支援：use_gpu: auto 預設於 50,000 行以上啟動 GPU（透過 PyTorch ≥2.8）

已知限制：僅接受數值型與已編碼類別型；線性相關假設不適合高度非線性關係；欄位超過 1000 個時相關矩陣 O(n²) 記憶體需求急劇增加。

3.2 SDV 整合：四種主流演算法

需另安裝 pip install "petsard[all]" 方可使用：

| 方法名稱 | YAML key | 特性 | 預設超參數 |

|---------|---------|------|-----------|

| GaussianCopula | sdv-single_table-gaussiancopula | 快速、統計穩健、適合大型資料集 | — |

| CTGAN | sdv-single_table-ctgan | GAN-based，擅長複雜多峰分佈 | epochs=300, batch=500 |

| CopulaGAN | sdv-single_table-copulagan | 混合 Copula + GAN，適合混合型資料 | epochs=300, batch=500, dist=beta |

| TVAE | sdv-single_table-tvae | VAE-based，訓練穩定、收斂快 | epochs=300, batch=500 |

SDV 整合提供兩模式：built-in（直接套用預設）與 custom_method（可調整 epochs、batch_size 等超參數）。

授權注意：SDV 套件採用 Business Source License 1.1（BUSL 1.1），對商業競爭性使用有限制，PETsARD 整體雖為 MIT，但引入 SDV 時需遵守 BUSL 1.1 條款 [5]。

3.3 自訂合成器接口

支援 custom_method，透過指定 module_path 與 class_name 接入任何第三方或自研演算法。亦可 custom_data 載入外部已生成的合成資料，純做評估流程。

3.4 資料屬性調優技術（Data Property Adjustment）

PETsARD 提供四種迭代調優策略，用於提升生成品質：

| 技術 | 適用場景 |

|-----|---------|

| Split-Synthesize-Merge | 資料量龐大或欄位數多時，分段合成再合併 |

| Uniform Encoding | 類別欄位基數高、分佈嚴重不均時 |

| Logarithmic Transformation | 長尾分佈（如收入、交易金額）前處理 |

| Time Anchoring Scaling | 時間欄位（如出生年月）的錨定縮放，避免時間外推問題 |

4. 評估框架（Evaluator）

PETsARD 的評估框架是其最具競爭力的差異化特色，涵蓋四個維度加一個實驗性模組。

4.1 資料有效性診斷

| 方法 | 工具 | 關鍵指標 | 達標門檻 |

|-----|------|---------|---------|

| sdmetrics-diagnosticreport | SDMetrics | 診斷分數（Schema Validity, Referential Integrity） | ≈ 1.0 |

用於快速確認合成資料是否符合原始資料的基本約束（欄位型別、取值範圍等），作為後續評估的前提條件。

4.2 隱私保護評估（Anonymeter 框架）

採用 Anonymeter（Giomi et al., 2023）框架 [6]，對應 GDPR 及歐盟 Article 29 WP29 2014 定義的三類去識別化風險，2023 年獲法國 CNIL 背書：

| 風險類型 | 方法名稱 | 量測內容 | 達標門檻 |

|---------|---------|---------|---------|

| Singling Out | anonymeter-singlingout | 能否以單一或多欄位組合唯一辨識個人 | < 0.09 |

| Linkability | anonymeter-linkability | 能否跨資料集連結同一個人（Gower's Distance） | < 0.09 |

| Inference | anonymeter-inference | 能否從輔助欄位推導敏感欄位 | < 0.09 |

風險分數公式（控制基準攻擊率，避免低估）：


risk = (主攻擊率 - 基準攻擊率) / (1 - 基準攻擊率)

門檻來源：0.09 門檻參照新加坡 PDPC 2023 合成資料指引 [3]。Singling Out 為強制評估，Linkability 和 Inference 視具體情境決定是否進行。

主要配置參數：

  • n_attacks：攻擊次數（建議 100–2,000，越多越穩健）
  • max_attempts：單次攻擊最大嘗試次數（建議 1,000–n/10）
  • aux_cols（Linkability）：兩組非重疊輔助欄位集
  • secret（Inference）：目標敏感欄位名稱

4.3 保真度評估（Fidelity）

| 方法 | 工具 | 量測內容 | 達標門檻 |

|-----|------|---------|---------|

| sdmetrics-qualityreport | SDMetrics | Column Shapes（各欄分佈相似度）+ Column Pair Trends（欄位對相關性） | ≥ 0.75 |

SDMetrics 的品質分數為 0–1 之間的綜合指標，0.75 以上表示合成資料在統計屬性上與原始資料高度相似。

4.4 效用評估（Utility）

採用 Train on Synthetic, Test on Real（TSTR） 典範，以 XGBoost 作為下游任務模型：

| 任務類型 | 方法名稱 | 關鍵指標 | 達標門檻 |

|---------|---------|---------|---------|

| 分類 | mlutility-classification | F1（≥0.7）、ROC AUC（≥0.8）、MCC（≥0.5） | F1 ≥ 0.7 |

| 迴歸 | mlutility-regression | R²、RMSE | R² ≥ 0.7 |

| 分群 | mlutility-clustering | Silhouette 係數 | ≥ 0.5 |

4.5 實驗性：mpUCCs 自研隱私風險演算法

Maximal Partial Unique Column Combinations（mpUCCs） 是 PETsARD 自主研發的隱私風險評估演算法，目前仍標記為實驗性功能。

技術原理：

  • 系統性搜尋合成資料中能唯一辨識原始記錄的欄位組合
  • 採貪婪基數優先（greedy cardinality-first）策略縮小搜尋空間
  • 輔以決策樹剪枝排除無效組合
  • 以 Rényi 熵（α=2，即碰撞熵）量化組合的辨識風險

輸出指標：

  • Privacy Risk Score：0（安全）→ 1（危險）
  • Overall / Main / Baseline Protection：整體、主要、基準防護層
  • Identification Rate：可辨識率

效能聲稱：在 Adult Income 資料集上，mpUCCs 僅需 44 秒，相比 Anonymeter 的 12 分鐘快約 16 倍，且為完整評估（非抽樣）。

建議門檻（視情境而定）：

| 情境 | 建議 Risk Score |

|-----|--------------|

| 公開釋出 | < 0.05 |

| 研究共享 | < 0.10 |

| 高敏感（醫療/金融） | < 0.01 |

現況說明：mpUCCs API 仍可能調整，風險加權函數仍在研究中，文件明確建議不應將其作為正式合規用途的唯一依據。

5. 與同類工具比較

5.1 功能維度對照

| 面向 | PETsARD | SDV | synthpop（R）| Synthea | DataSynthesizer |

|-----|---------|-----|-------------|---------|----------------|

| 語言 | Python | Python | R | Java | Python |

| 核心演算法 | 自研 GC + SDV 整合 | CTGAN, TVAE, CopulaGAN, GC, PAR | CART, parametric | 規則式醫療 | PrivBayes (DP) |

| GPU 加速 | ✅（Numba + PyTorch） | 部分（GAN類） | ✗ | ✗ | ✗ |

| 差分隱私（DP） | ✗（整合外部 DP 合成器） | ✗（本身無） | ✗ | ✗ | ✅（PrivBayes DP） |

| 內建隱私評估 | ✅（Anonymeter + mpUCCs） | ✗ | ✗ | ✗ | ✗ |

| 內建效用評估 | ✅（XGBoost, K-means） | 部分（sdmetrics） | 需另外實作 | ✗ | ✗ |

| 多表支援 | ✗（建議展平） | ✅（HMA 階層多表） | 有限 | ✅（醫療模型） | ✗ |

| YAML 配置 | ✅ | ✗ | ✗ | ✗ | ✗ |

| 政府/政策目標 | ✅ | ✗ | 部分（ONS） | ✗ | ✗ |

| 授權 | MIT（SDV 整合部分 BUSL） | BUSL 1.1 | GPL | Apache 2.0 | MIT |

5.2 競爭優勢

PETsARD 的最大差異化在於其端到端的完整管線——大多數競爭工具只提供合成生成，而評估部分留給使用者自行組合。PETsARD 將生成與多維評估整合為一個可重複執行的實驗框架，特別適合需要嚴格文件化的監管環境。

另一個優勢是其政策定位：作為政府委託開發的工具，PETsARD 的評估門檻直接參照新加坡 PDPC 等監管機構的指引，對台灣政府機關及受監管產業而言具有天然的合規說服力。

5.3 主要弱點

  1. Python 版本嚴格限定（3.10–3.11）：不支援 Python 3.12+，對已升版的環境造成障礙
  2. 多表合成缺失：官方明確指出現有開源多表合成方案（含 SDV HMA）的限制（≤5 表、僅單層父子關係、品質偏低），但 PETsARD 本身未提供替代方案，僅建議 denormalize 展平
  3. 差分隱私（DP）未內建：本身不提供 DP 合成器，需透過 custom_method 接入外部 DP 工具（如 PrivBayes、OpenDP）
  4. mpUCCs 仍為實驗性：不宜作為正式合規依據
  5. 社群規模小：GitHub 6 stars、1 fork（2026 年 6 月），與 SDV（3,000+ stars）相比生態系差距明顯

7. 應用場景與限制

7.1 主要應用場景

醫療資料開放：醫療資料因高度敏感，傳統方法難以共享。合成電子病歷可讓研究機構在不觸碰真實患者資料的前提下開展分析，Anonymeter 的 GDPR 框架也提供合規依據。

金融徵信與詐騙偵測：金融機構欲訓練詐騙偵測模型，但詐騙樣本天然稀少且共享受監管。PETsARD 可生成統計相似的詐騙事件合成資料，擴充訓練集。

政府資料開放（Open Data）：地方政府欲公開社會經濟調查資料，但須確保個人隱私。以 PETsARD 生成合成版本，並提供 Anonymeter 評估報告作為去識別化聲明佐證。

AI 模型測試資料：開發者需要大量真實分佈的測試資料，但生產資料無法直接使用。合成資料可在開發環境中模擬真實行為分佈。

法規遵循演練：企業透過 PETsARD 評估其資料集的去識別化程度，作為個資保護實踐的內部審計工具。

7.2 已知限制與注意事項

  1. Python 版本鎖定：嚴格要求 Python 3.10 或 3.11，3.12 以上不支援，升版成本需評估
  1. 多表關係：PETsARD 不直接處理多表合成；若資料為標準化多表結構，須先手動 denormalize 展平
  1. 差分隱私缺口：未內建 DP 合成器，對於需要「ε-DP 保證」的合規場景，須自行接入 OpenDP、PrivBayes 等工具
  1. 高維欄位瓶頸：自研 Gaussian Copula 相關矩陣學習的時間與記憶體複雜度為 O(n²)，超過 1,000 欄時效能明顯下降
  1. 合成品質非保證：評估分數僅反映統計相似性與隱私風險的量化結果，並不等同於法律上的去識別化認定
  1. mpUCCs 穩定性：實驗性功能，不建議作為唯一合規評估依據
  1. 社群生態仍小：相比 SDV 的商業化生態，PETsARD 的第三方教學、插件與社群答疑資源相對有限

---

8. 綜合洞察

8.1 PETsARD 在全球合成資料工具生態的定位

全球合成資料工具市場分為三個流派：

  • 學術流派（SDV 系、Synthpop）：強調演算法豐富性，評估由使用者自組合
  • 商業流派（Gretel.ai、Mostly AI）：強調易用性與雲端服務，但評估可能缺乏透明度
  • 政策流派（PETsARD、英國 ONS 的 Synthpop 使用）：強調監管對齊與可審計性

PETsARD 明確屬於第三流派，其評估門檻直接引用新加坡 PDPC 指引，這在政策層面是有意義的選擇——使用者可在合規報告中直接引用評估結果，說明其符合已被特定監管機構認可的門檻。

8.2 mpUCCs 的潛在學術貢獻

如果 mpUCCs 的 16x 速度聲稱在更多資料集上得到驗證，且能以學術論文形式提出完整的理論保證，它將填補現有隱私評估工具中一個重要缺口——Anonymeter 基於抽樣，而 mpUCCs 提供完整評估，這在高精度合規場景中具有顯著優勢。目前仍為實驗性功能，建議追蹤後續發表。

8.3 差分隱私整合的戰略缺口

PETsARD 現階段的隱私評估框架基於攻擊模型（Anonymeter），而非形式化隱私保證（如差分隱私）。對於需要向監管機構提交 ε-DP 證明的場景，PETsARD 目前無法獨立勝任。若未來能整合 OpenDP 或 Google 的 DP 庫作為 Synthesizer 選項，並在評估報告中呈現 ε 值，將大幅提升合規用途的覆蓋範圍。

8.4 台灣資料治理生態系的角色

PETsARD 是台灣政府「資料空間」政策的基礎設施之一。隨著台灣加速推動「健保資料集」「戶政資料」等政府開放資料計畫，PETsARD 若能成為政府機關進行隱私影響評估（PIA）的標準工具，其社群規模與影響力將快速擴大。這是一個值得密切觀察的發展軌跡。

9. 建議

給資料科學從業者

  1. 評估現有工具是否夠用前先試 PETsARD：若你的需求是表格資料合成 + 隱私評估一體化，PETsARD 是目前開源世界中整合最完整的選項之一
  2. 先以 petsard-gaussian-copula 起步：速度快且結果可解釋；若品質不足再換 CTGAN/TVAE
  3. 必跑 Anonymeter Singling Out：這是唯一強制評估項，也是監管機構最容易理解的隱私指標

給資安/法規合規人員

  1. 以 Anonymeter 評估結果作為隱私評估報告的核心依據：GDPR 三維框架在法規圈已有認知基礎（CNIL 背書）
  2. mpUCCs 可輔助參考，但不應作為正式合規文件唯一來源
  3. 追蹤 PETsARD 差分隱私整合計畫：若未來提供內建 DP 選項，合規覆蓋範圍將大幅提升

給政策制定者

  1. 考慮將 PETsARD 評估框架納入政府資料開放標準：其評估門檻已對齊 PDPC，具有監管一致性
  2. 投入資源讓 mpUCCs 完成學術發表：使其成為可被引用、可被審查的正式評估方法，才能在政策文件中合法引用
  3. 解除 Python 版本限制：目前 3.10–3.11 限制可能成為企業導入障礙，建議優先解決

給研究人員

  1. PETsARD 作為隱私評估研究的基準環境值得探索：其內建的基準資料集（AWS S3 petsard-benchmark）可為比較研究提供標準化測試集
  2. mpUCCs 演算法值得獨立驗證與引用評估：若能確認其完整性保證與速度聲稱，這將是重要的貢獻

---

10. 限制與警語

本報告的資訊主要來源為 GitHub 存儲庫、PyPI 頁面與官方文件站，缺乏：

  • 獨立第三方的實測基準數據
  • 學術同儕審查論文（PETsARD 目前無已知 arXiv 論文）
  • 大規模部署的生產環境報告

mpUCCs 的效能聲稱（16x 加速）僅來自開發團隊文件，尚未見到獨立驗證。授權相容性（MIT + SDV BUSL 1.1）的法律影響亦建議與法律顧問確認。
