BDD／Gherkin mechanism 深化：語法邊界、pytest-bdd／behave 執行模型與 step 設計

研究日期：2026-04-10
任務類型：tech_research
系列：ai-driven-requirements-bdd（mechanism）

技術概述

今日已完成任務涉及的技術包括：Workflow YAML、Hook 守衛、結果檔 Schema、Open WebUI、InvokeAI、operational health rollup CLI，以及 AI 驅動需求工程：使用者故事、驗收條件與 BDD／Gherkin 自動化工作流。其中 BDD／Gherkin 最值得深化，因為知識庫目前只有 foundation 概論，尚未整理語法邊界、Python runner 執行模型與 step 設計原則。這些機制層知識直接決定 AI 生成的 .feature 檔會是可維護的活文件，還是退化成脆弱的 UI 腳本。[來源：Cucumber Gherkin, 2024；Cucumber Anti-patterns, 2024；Pytest-BDD 7.3.0 Docs, 2025]

核心發現

核心主張 1：Gherkin 是結構化規格語言，不只是自然語言測試描述

Gherkin 以 Feature、Scenario、Background、Scenario Outline、Examples 與 step keywords 建立可解析的規格結構，而不是隨意的敘述文字；每個 .feature 檔只應有一個 Feature，Background 用來表達跨 scenario 的共同前置條件，Scenario Outline 則將案例表與步驟模板結合。[來源：Cucumber Gherkin, 2024；Pytest-BDD 7.3.0 Docs, 2025；SmartBear TestComplete Gherkin Syntax, 2026]

這代表把需求轉成 Gherkin 時，重點不是「把需求寫得像句子」，而是把行為切成能被 parser 與 runner 穩定解譯的結構。Martin Fowler 將 Given/When/Then 對應為 precondition、behavior、observable outcome，說明它本質上是規格與測試之間的橋梁，而不是僅供閱讀的文件格式。[來源：Martin Fowler Given-When-Then, 2013；Cucumber Gherkin, 2024；SmartBear Agile Testing, 2026]

核心主張 2：可維護的 BDD 場景必須避免 feature-coupled 與 conjunction steps

Cucumber 官方把 feature-coupled step definitions 列為反模式，因為這會導致 step 不能跨 feature 重用、步驟檔數量暴增、維護成本升高。[來源：Cucumber Anti-patterns, 2024；Cucumber Gherkin, 2024；SmartBear Agile Testing, 2026]

同一份官方指南也明確指出，不應把多個動作塞進同一步，像是「Given 我有 A 並且有 B」這種 conjunction step 會讓步驟變得過度專用；較好的做法是拆成原子步驟，必要時用語言本身的 helper method 做組合，而不是在 step definition 中互相呼叫步驟。[來源：Cucumber Anti-patterns, 2024；Martin Fowler Given-When-Then, 2013；SmartBear TestComplete Gherkin Syntax, 2026]

核心主張 3：`pytest-bdd` 與 `behave` 都支援 Gherkin，但執行模型不同

behave 採用標準 features/ 目錄、features/steps/ 與 features/environment.py 生命週期鉤子；before_step、before_scenario、before_feature 等函式提供跨步驟與跨情境控制，狀態主要透過 context 物件傳遞。[來源：behave Tutorial, 2026；behave Tutorial Stable, 2025；behave API Reference, 2025]

pytest-bdd 則把 Gherkin 綁到 pytest 生態，透過 pytest fixture、scenario decorator、background 與 fixture reuse 來組織測試。這表示若專案本來已使用 pytest，pytest-bdd 更容易復用既有 fixture 與測試工具鏈；若需要典型 BDD 目錄結構與 context hook，behave 的模型則更直接。[來源：Pytest-BDD 7.3.0 Docs, 2025；behave Tutorial, 2026；Cucumber Gherkin, 2024]

核心主張 4：近期 `pytest-bdd` 的重要語意變更會影響 AI 生成 feature/step 的正確性

pytest-bdd 7.x 文件明確指出， 樣式只會在 Scenario Outline 中解析，不再對普通 Scenario 生效；同時 step parsed arguments 不再自動成為 fixture，若要建立 fixture 必須顯式使用 target_fixture。[來源：Pytest-BDD 7.3.0 Docs, 2025；Pytest-BDD 6.0.1 Docs, 2025；Pytest-BDD 7.2.0 Docs, 2024]

這對 AI 生成很關鍵，因為許多舊範例或模型記憶仍會把 填進一般 scenario，或預設 Given 參數會自動成為 fixture。若不根據新語義調整，生成的測試可能在 parser 或 binding 階段直接失敗。[來源：Pytest-BDD 7.3.0 Docs, 2025；Pytest-BDD 6.0.1 Docs, 2025；Cucumber Gherkin, 2024]

最佳實踐與常見陷阱

最佳實踐是把 .feature 檔當成「需求契約」，只描述前置條件、行為與可觀察結果，不寫頁面座標、CSS selector、資料表欄位細節。step definition 檔案應依 domain concept 命名，而不是依 feature 檔名命名，這樣同一組步驟才能跨場景重用。[來源：Cucumber Anti-patterns, 2024；Martin Fowler Given-When-Then, 2013；SmartBear Agile Testing, 2026]

Background 應只放共享前置條件，不應塞入行為或驗證；Scenario Outline 適合表驅動案例，但不要再與額外的 pytest parametrization 疊加，否則來源真相會分裂，難以理解與維護。[來源：Pytest-BDD 7.3.0 Docs, 2025；Pytest-BDD 6.0.1 Docs, 2025；Cucumber Gherkin, 2024]

在 behave 中，應把跨情境控制集中到 environment.py 與 fixtures，而不是把 setup/teardown 塞進 Given step；在 pytest-bdd 中，則應利用 pytest fixture 與 target_fixture 管理依賴，避免把資料傳遞全部塞進全域變數。[來源：behave Tutorial, 2026；Pytest-BDD 7.3.0 Docs, 2025；behave API Reference, 2025]

與本專案的關聯

daily-digest-prompt 很適合把 BDD 用在「契約測試」而不是 UI 自動化。具體做法是把 workflow/result guard 的預期行為寫成 feature，例如：

  • 給定一個缺 summary 的結果 JSON，當驗證器執行時，則應被標記為 schema incomplete。
  • 給定 KB 不可用的情境，當 tech_research 任務完成時，則結果仍應是 success 且 kb_imported=false。
  • 給定已存在 3 天內相同 topic 的 registry，當選題執行時，則必須改選其他主題。

這種寫法能把目前分散在 prompt、workflow YAML、Python 工具與 PowerShell 腳本中的行為規則拉成共同語言，並讓 AI 之後生成測試時有明確的語法與 runner 邊界可遵守。這一段是根據本專案現況做的工程推論。[推論；依據：Cucumber Gherkin, 2024；Pytest-BDD 7.3.0 Docs, 2025；behave Tutorial, 2026]

程式碼範例


Feature: Result file validation

  Background:
    Given 一個可被解析的結果 JSON 基底

  Scenario Outline: 缺少必填欄位時應回報 schema 錯誤
    Given 結果 JSON 缺少 "<field>"
    When 執行 validate_results
    Then 驗證結果應包含 "missing field: <field>"

    Examples:
      | field   |
      | summary |
      | status  |


from pytest_bdd import given, when, then, parsers, scenario

@scenario("result_validation.feature", "缺少必填欄位時應回報 schema 錯誤")
def test_missing_required_field():
    pass

@given("一個可被解析的結果 JSON 基底", target_fixture="result_json")
def base_result():
    return {"agent": "todoist-auto-tech_research", "status": "success", "summary": "ok"}

@given(parsers.parse('結果 JSON 缺少 "{field}"'))
def remove_field(result_json, field):
    result_json.pop(field, None)

@when("執行 validate_results", target_fixture="errors")
def validate(result_json):
    required = ["agent", "status", "summary"]
    return [f"missing field: {k}" for k in required if k not in result_json]

@then(parsers.parse('驗證結果應包含 "missing field: {field}"'))
def assert_error(errors, field):
    assert f"missing field: {field}" in errors
