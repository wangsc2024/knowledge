Temporal 深度研究報告

概述

Temporal（temporalio/temporal）是 Uber Cadence 的繼承者，由原 Cadence 核心開發者於 Temporal Technologies 建立，定位為「持久執行（Durable Execution）平台」。與 n8n/Flowise 等視覺化 No-code 平台不同，Temporal 面向工程師，解決的核心問題是：在分散式系統中，如何讓長時間執行、多步驟的業務邏輯在遭遇伺服器崩潰、網路中斷、部署更新時，仍能保證執行至完成。2024-2026 年間，Temporal 從傳統業務流程協調器大幅擴展至 AI Agent 編排領域，成為「程式碼驅動型 AI 工作流」的生產標準之一。OpenAI Codex 即是以 Temporal 為底層執行每日數百萬 AI 請求的真實案例。

技術架構

Temporal 採用事件溯源（Event Sourcing）作為核心持久化機制。每個 Workflow Execution 都維護一條 append-only 事件歷史（Event History），任何時刻都可透過重播（replay）此歷史重建完整執行狀態。架構由三個核心元件組成：

  • Temporal Cluster：包含 History Service（分片儲存 Workflow 狀態）、Matching Service（管理 Task Queue）、Frontend Service（gRPC 入口）
  • Worker Process：用戶端的執行環境，透過 SDK 長輪詢 Task Queue，執行 Workflow 和 Activity 程式碼
  • SDK 層：支援 Python、Go、Java、TypeScript、.NET（Beta）、Ruby（Pre-release）

Workflow vs Activity 設計分離是最關鍵的架構原則：

  • Workflow：純確定性邏輯（orchestration），禁止 I/O 或副作用，負責狀態管理與決策流
  • Activity：所有非確定性操作（LLM 呼叫、HTTP API、資料庫寫入），自動具備 RetryPolicy 能力

這個分離讓系統在 Worker 崩潰後，能夠從 Event History 重播 Workflow 並跳過已完成的 Activity，完全避免重複執行副作用。

工作流核心能力

  • 自動重試與退避策略：Activity 層可設定 RetryPolicy（最大重試次數、退避倍率、錯誤類型白名單），適配 LLM rate limit 和 transient failure
  • Signal（訊號機制）：外部系統或人類可透過 Signal 向運行中的 Workflow 注入狀態變更，實現 Human-in-the-Loop（HITL）
  • Continue-As-New：超長鏈式 Workflow（如持續對話 Agent）可用此機制重置 Event History 防止記憶體過載
  • Temporal Nexus（2026 GA）：跨命名空間工作流連接，支援多租戶 Agent 系統
  • Multi-Region Replication（2026 GA）：99.99% SLA，跨區域高可用

AI 整合機制

Temporal 針對 AI Agent 場景的核心整合模式為三層 Activity 分工：


@activity.defn
async def agent_tool_planner(prompt: str) -> AgentDecision:
    response = await openai_client.chat.completions.create(...)
    return parse_json_response(response)

@activity.defn
async def dynamic_tool_activity(tool_name: str, args: dict) -> ToolResult:
    return await tool_registry[tool_name](**args)

@workflow.defn
class AgentWorkflow:
    @workflow.run
    async def run(self, initial_prompt: str):
        history = ConversationHistory()
        while True:
            decision = await workflow.execute_activity(agent_tool_planner, ...)
            if decision.next == "done":
                break
            result = await workflow.execute_activity(dynamic_tool_activity, ...)
            history.append(result)

2026 年整合生態圈：OpenAI Agents SDK（Public Preview）、Pydantic AI、Vercel AI SDK、Google ADK、Langfuse（可觀測性）、Braintrust（AI 監控）均已有官方 Temporal 整合。

與 LangGraph 的互補關係是業界共識：LangGraph 處理 Agent 推理邏輯與工具路由，Temporal 提供持久執行保障。生產架構模式：工作流超過 30 秒或涉及 3+ 外部系統時，LangGraph Agent 包裹在 Temporal Workflow 內執行。

部署與維運

  • 自託管：temporalite（本機開發）或 docker-compose（含 PostgreSQL/Cassandra 後端）
  • Temporal Cloud：完全託管，支援 AWS 和 Google Cloud（2026 新增）
  • 可觀測性：原生 Web UI 提供工作流執行歷史視覺化、Activity 重試記錄、時間軸除錯；Langfuse 整合提供 LLM 層級追蹤
  • 版本管理：透過 workflow.patched() API 實現 Workflow 代碼的滾動升級（不中斷在途執行）

與 daily-digest-prompt 的連結

  1. 殭屍鎖問題：Temporal 的 Worker 心跳（Heartbeat）機制可作為進程存活監控替代方案，消除 179.7min 殭屍鎖事故類型
  2. FSM zombie on system restart：Temporal 的 Workflow Execution State 由伺服器維護，Worker 重啟後自動恢復，避免 FSM 殘留 running 狀態
  3. 自動任務 timeout 管理：RetryPolicy + Activity timeout 可取代現有 timeouts.yaml 手動調整，透過 exponential backoff 自動適應
  4. HITL 需求：Todoist 任務執行中需人工確認的場景，可透過 Temporal Signal 原生實現，取代現有的 ntfy 通知 + 手動重啟流程

優缺點評估

優點：

  • 持久執行保障（Event History replay）是業界最成熟實作，生產穩定性遠超自建 FSM
  • Workflow-as-code 讓狀態邏輯在版本控制中可讀、可測試、可回溯
  • 原生 HITL（Signal）+ 多語言 SDK + 豐富 AI 生態系整合
  • 2026 年 Nexus + Multi-Region 讓企業級部署門檻大幅降低
  • MIT 授權，可完全自託管

缺點 / 限制：

  • 引入額外基礎設施（Temporal Cluster），對現有 Windows + pwsh 環境有遷移摩擦
  • Workflow 確定性限制（禁止隨機、系統時間等）需要開發者改變思維模式
  • Event History 有 50MB 上限，超長 Agent 對話需搭配 Continue-As-New
  • 與現有 PowerShell + Phase 管線整合需要 Python SDK 橋接，非零遷移成本
