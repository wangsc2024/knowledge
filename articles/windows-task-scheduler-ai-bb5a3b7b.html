Windows Task Scheduler AI Agent 場景進階實踐 2026

單實例鎖定、殭屍進程偵測、FSM 狀態機整合與跨任務並行防護模式

---

技術概述

Windows Task Scheduler 是 AI Agent 自動化系統在 Windows 上的核心基礎設施。每日摘要、Todoist 任務自動化、知識庫更新、健康稽核等排程任務，都依賴它定時觸發。然而，在 AI Agent 場景中，Task Scheduler 面臨的挑戰遠超傳統批次作業：Agent 任務執行時間不確定（可能 1 分鐘也可能 3 小時）；Agent 會派生子進程（LLM API 呼叫、Python 工具）；外部系統重啟會讓 FSM 狀態殘留；任務失敗可能靜默（exit code 0 但邏輯未完成）。

本文聚焦三個在既有基礎 Task Scheduler 筆記中未覆蓋的核心問題：（1）單實例防護的真實邊界（Task Scheduler 原生設定的盲區）、（2）殭屍進程的成因與偵測清除模式、（3）外部 FSM 狀態機與排程系統的整合陷阱。這三個問題都曾導致本專案生產故障，值得深入理解 [來源：daily-digest-prompt MEMORY，2026-04-29]。

替代方案比較：Azure Durable Functions 提供原生狀態機排程，但需 Azure 環境；Prefect/Airflow 有完整 DAG 狀態追蹤，但部署複雜度高；Windows Task Scheduler 勝在零外部依賴、OS 原生、啟動快（<1s），適合本機 AI Agent 自動化場景。

---

核心概念與原理

1. 單實例防護的三層架構

Task Scheduler 的「不啟動新實例」設定（TASK_INSTANCES_IGNORE_NEW）存在關鍵盲區 [來源：Microsoft Q&A - Do not start a new instance issue, 2023]：該設定僅攔截排程觸發，不攔截手動執行。當管理員從 UI 或 Start-ScheduledTask 手動觸發時，新實例一定會啟動，即使舊實例正在執行。Microsoft 官方說明確認這是「by design」行為。

因此，生產環境需要三層防護：

Layer 1：Task Scheduler 設定（防排程觸發重疊）


任務屬性 → 設定 → 如果工作已在執行，套用下列規則：
  ✓ 不啟動新的執行個體（TASK_INSTANCES_IGNORE_NEW）
如果工作執行時間超過此時間，停止工作：
  ✓ 2 小時（依預期執行時間 × 1.5 設定）

Layer 2：具名 Mutex 跨 Session 保護（防任何路徑的重複執行）

Global\ 前綴的 Named Mutex 在 Windows 中跨所有 Session 可見，包含 SYSTEM Session 0 和所有用戶 Session。這是唯一能防止「手動觸發 + 排程觸發」同時進入的方法 [來源：Microsoft Learn - Mutexes, 2024]：


# Global\ 前綴 = 跨所有 Terminal Server sessions 可見
$mutexName = "Global\DailyDigest-$([System.IO.Path]::GetFileNameWithoutExtension($MyInvocation.MyCommand.Path))"
$mutex = New-Object System.Threading.Mutex($false, $mutexName)

$mutexAcquired = $mutex.WaitOne(0, $false)  # 非阻塞嘗試獲取
if (-not $mutexAcquired) {
    Write-Host "[GUARD] 另一個執行個體正在執行中，退出。"
    exit 1
}
try {
    # 主要腳本邏輯
} finally {
    $mutex.ReleaseMutex()
    $mutex.Dispose()
}

Mutex 優於 Lock File 的關鍵：Mutex 在程式 crash 或 PowerShell 進程被強制終止時，OS 會自動釋放（Abandoned Mutex）；Lock File 則需要手動清理，靜默 crash 會留下殭屍鎖 [來源：codegenes.net - Single Instance PowerShell, 2024]。

Layer 3：PID-Aware Lock File（補強 FSM 整合場景）

當任務使用外部 JSON FSM 狀態檔時，鎖定檔應記錄 PID，以便下次啟動時驗證 PID 是否仍存活：


{ "status": "running", "pid": 12345, "started_at": "2026-04-29T10:00:00" }

啟動時檢查：若 PID 不存在（Get-Process -Id $pid -ErrorAction SilentlyContinue），視為殭屍鎖並清除。

2. 殭屍進程（0x41301）的三類根因

0x41301（The task is currently running）是 Task Scheduler 最常見的生產故障碼，表示 Scheduler 認為任務仍在執行，但實際進程可能已消失 [來源：copyprogramming.com - 0x41301 Complete Fix Guide, 2026]：

根因 A：子進程孤兒化

Task Scheduler 只追蹤直接啟動的進程（parent process）。當 PowerShell 腳本用 Start-Process、Start-Job 或 Python subprocess 派生子進程時，即使父進程退出，Scheduler 仍視任務為「執行中」，直到所有子進程也退出。AI Agent 場景中，claude -p 本身就是子進程，若父腳本提前退出，Scheduler 狀態殘留。

修正：父腳本必須等待所有子進程完成（Wait-Job、$proc.WaitForExit()），或使用 Job Object 確保子進程樹跟隨父進程終止。

根因 B：SYSTEM Session 0 隔離造成的靜默卡死

以 SYSTEM 帳戶執行的任務在 Session 0（非互動式桌面）中執行。若腳本嘗試顯示 GUI 對話框（如 MsgBox）、讀取用戶家目錄的相對路徑（~\），或依賴用戶環境變數，會靜默卡住而非拋出錯誤 [來源：woshub.com - Run PowerShell Task Scheduler, 2024]。

診斷：Session 0 卡死不輸出到標準輸出；必須配置日誌路徑為絕對路徑，並加上 -NonInteractive -WindowStyle Hidden -NoProfile。

根因 C：「停止工作若執行超時」設定失效

Task Scheduler 的超時設定對「由 PowerShell 腳本派生的子進程」無效——它只能終止 PowerShell 父進程，子進程（Claude CLI、Python 工具）繼續孤兒化執行。這是本專案「殭屍鎖 PID 7860 卡 179.7 分鐘」事故的根本原因 [來源：daily-digest-prompt MEMORY - 殭屍鎖根因, 2026-04-29]。

3. FSM 狀態機與 Task Scheduler 整合：殘留態問題

Task Scheduler 本身無狀態機概念，生產 AI Agent 系統通常使用外部 JSON 檔（如 state/workflow-state.json）實作 FSM。這引發一類特殊故障：系統重啟後 FSM 狀態殘留。

故障場景：

  1. Task Scheduler 觸發任務 → FSM 寫入 status: "running", pid: 31372
  2. 系統強制重啟（如 Windows Update）→ Task Scheduler 重新啟動服務
  3. 新任務觸發 → 讀到 FSM status: "running" → 認為舊任務仍在執行 → 兩輪排程被阻擋

根本缺陷：FSM 狀態寫入了 PID，但重啟後 PID 空間重置，同一 PID 可能被新進程佔用（PID 重用）。

解法（F-02 Guard PID-aware）[來源：daily-digest-prompt MEMORY - FSM zombie on system restart, 2026]：


# 讀取 FSM 狀態
$fsmState = Get-Content $FsmStatePath | ConvertFrom-Json
if ($fsmState.status -eq 'running') {
    $pid = $fsmState.pid
    $proc = Get-Process -Id $pid -ErrorAction SilentlyContinue
    if ($null -eq $proc) {
        # PID 不存在 → 殭屍狀態，清除
        Write-Warning "[FSM] PID $pid 已不存在，清除殭屍狀態"
        $fsmState.status = 'idle'
        $fsmState | ConvertTo-Json | Set-Content $FsmStatePath
    } elseif ($proc.StartTime -lt (Get-Date).AddHours(-2)) {
        # PID 存在但超過 2 小時 → 視為超時，終止並清除
        Stop-Process -Id $pid -Force
        $fsmState.status = 'timeout'
        $fsmState | ConvertTo-Json | Set-Content $FsmStatePath
    }
}

重要補充：PID 重用問題。新進程可能佔用已終止進程的 PID（Windows PID 範圍 4-65535）。應同時比對 StartTime：若 PID 存在但 StartTime 晚於 FSM 記錄的 started_at，仍視為殭屍。

最佳實踐

實踐 1：雙重守護（Mutex + Task Scheduler 設定並用）

情境：任何可能被手動觸發或多工作流觸發的 AI Agent 任務。

建議：Task Scheduler 設定「不啟動新實例」+ 腳本開頭加 Global\ Named Mutex。兩者各有盲區，並用才完整。

反例：只靠 Task Scheduler 設定 → 管理員手動觸發後兩個 Agent 並行呼叫 LLM，token 雙倍消耗，ntfy 重複推送。只靠 Lock File → 腳本 crash 後殭屍鎖，下次正常觸發被擋。

實踐 2：超時必須同時設定任務層與腳本層

情境：執行 LLM API 呼叫的任務（外部 API 可能卡住）。

建議：

  • Task Scheduler 任務超時：預期時間 × 2（Task Scheduler 層，殺父進程）
  • PowerShell -Timeout 或 Python subprocess.run(timeout=X)：預期時間 × 1.5（腳本層，殺子進程）
  • 確保子進程有獨立超時，不依賴父超時傳遞

依據：Task Scheduler 超時僅殺父 PowerShell 進程，子進程孤兒化繼續執行 [來源：Liebensraum, 2017]。

實踐 3：FSM 狀態記錄 PID + 啟動時間

情境：使用外部 JSON FSM 的多工作流 AI Agent 系統。

建議：FSM 的 running 狀態必須記錄 { pid, started_at, task_name }。每次新任務啟動前，執行 F-02 Guard（PID 存活 + StartTime 比對）。

情境閾值：started_at 超過預期最長執行時間的 1.5 倍 → 即使 PID 存活也視為超時（防止真正無限迴圈）。

實踐 4：Watchdog 任務監控所有 AI 任務的 LastRunTime

情境：部署了 10+ 個 AI Agent 排程任務的生產環境。

建議：建立獨立的「watchdog」任務，每 15 分鐘執行一次，掃描所有 Claude_* 任務：


$tasks = Get-ScheduledTask -TaskPath '\DailyDigest\'
foreach ($task in $tasks) {
    $info = Get-ScheduledTaskInfo -TaskName $task.TaskName -TaskPath $task.TaskPath
    if ($info.LastTaskResult -eq 0x41301) {
        Stop-ScheduledTask -TaskName $task.TaskName -TaskPath $task.TaskPath
        # 發送 ntfy 告警
    }
}

[來源：Microsoft Learn - Stop-ScheduledTask, 2024][來源：GitHub - SammyKrosoft Get-ScheduledTaskInfo, 2022]

實踐 5：以絕對路徑配置所有 SYSTEM 任務

情境：以 SYSTEM 帳戶執行的任務。

建議：所有日誌路徑、設定檔路徑、Python 工具路徑使用絕對路徑（如 D:\Source\daily-digest-prompt\logs\）。pwsh 引數加 -NonInteractive -WindowStyle Hidden -NoProfile。工作目錄明確設定到腳本所在目錄。

反例：使用 ~\logs\ → SYSTEM 帳戶的 ~ 指向 C:\Windows\System32\config\systemprofile，日誌寫入系統目錄，ACL 可能拒絕存取 [來源：woshub.com, 2024]。

陷阱與反例

陷阱 1：「不啟動新實例」設定被手動觸發繞過，產生雙重 Agent 黑洞

成因：TASK_INSTANCES_IGNORE_NEW 僅對排程觸發有效，手動觸發（UI / Start-ScheduledTask）直接繞過此設定 [來源：Microsoft Q&A - 1159691, 2023]。生產事故：管理員看到任務「沒執行」就手動觸發，與排程實例並行，兩個 Claude CLI 同時呼叫同一個 LLM 工作流，FSM 狀態互相覆寫。

規避：腳本開頭加 Global\ Named Mutex。即使手動觸發，第二個實例無法獲取 Mutex 就退出。

陷阱 2：Task Scheduler 超時設定不能殺子進程，導致殭屍進程群

成因：「如果工作執行時間超過此時間，停止工作」只終止 Task Scheduler 直接啟動的進程（父 PowerShell）。Claude CLI（claude -p）、Python uv 工具、TTS 進程等子進程不受控制，繼續孤兒化執行 [來源：Liebensraum, 2017]。

本專案實例：Phase 0a/Snapshot 子進程無 timeout + Task Scheduler「不啟新實例」串成 3 小時黑洞（PID 7860 卡 179.7 分鐘）[來源：MEMORY - 殭屍鎖根因, 2026]。

規避：子進程使用 Start-Process 啟動時傳入 Job Object，確保 Job Object 設定 KillOnJobClose；或腳本中明確 Wait-Job（設超時）+ Stop-Job。

陷阱 3：系統重啟清空 PID 空間，FSM 殘留態阻擋後續排程

成因：系統重啟後新進程 PID 可能與舊 FSM 記錄的 PID 相同（PID 重用），導致 F-02 Guard 誤判「進程仍存活」，FSM 永久卡在 running。

本專案實例：2026-04-25 使用者手動重啟，殺掉 PID 31372，FSM 殘留 running 擋兩輪排程 [來源：MEMORY - FSM zombie on system restart, 2026]。

規避：同時比對 PID 的 StartTime（$proc.StartTime）與 FSM 記錄的 started_at。若 StartTime 晚於 started_at，表示 PID 已被重用，仍視為殭屍。

陷阱 4：`0x41301` 狀態下 `Stop-ScheduledTask` 無法清除殘留

成因：若導致 0x41301 的根本進程已消失（Task Scheduler 的進程 handle 丟失），Stop-ScheduledTask 有時無法清除「Running」狀態。

根本解法：刪除並重建任務（Task Scheduler 狀態儲存在 %SystemRoot%\System32\Tasks\ XML 中；重建即清除所有執行狀態）[來源：Microsoft Q&A - Can't get Task Scheduler to end a task 0x41301, 2025]。

與本專案關聯

落地場景 1：run-todoist-agent-team.ps1 的殭屍鎖修復

本專案 run-todoist-agent-team.ps1 已實作 FSM 殭屍偵測（殭屍閾值 90→60min）。根據本研究，應補強：

  1. FSM 的 running 狀態加入 started_at 欄位（目前已有 PID，但需同時比對 StartTime）
  2. 腳本開頭加 Global\ Named Mutex 防止 Check-Todoist-Lock.ps1 手動觸發與排程並行
  3. Phase 0a/Snapshot 子進程加獨立超時（-Timeout 120），不依賴父超時

落地場景 2：check-health.ps1 整合 0x41301 掃描

check-health.ps1 目前已有多個健康區塊。可新增「[排程健康]」區塊，掃描所有 Claude_* 任務是否有 LastTaskResult=0x41301 的殭屍狀態，並推送 ntfy warning。


# 在 check-health.ps1 新增
$zombieTasks = Get-ScheduledTask -TaskPath '\' | Where-Object { $_.TaskName -like 'Claude_*' } | ForEach-Object {
    $info = Get-ScheduledTaskInfo -TaskName $_.TaskName -TaskPath $_.TaskPath -ErrorAction SilentlyContinue
    if ($info -and $info.LastTaskResult -eq 0x41301) { $_.TaskName }
}
if ($zombieTasks.Count -gt 0) {
    Write-Warning "[排程健康] 偵測到 $($zombieTasks.Count) 個殭屍任務：$($zombieTasks -join ', ')"
}

程式碼範例：完整單實例防護腳本模版

以下為生產級 AI Agent 腳本開頭標準模版（官方文件驗證）：


#Requires -Version 7.0
# 腳本：run-agent-task.ps1
# 用途：AI Agent 任務啟動點，含三層單實例防護

param(
    [string]$TaskKey = "unknown",
    [int]$MaxRunMinutes = 120
)

$scriptDir = Split-Path -Parent $MyInvocation.MyCommand.Path
$logPath = Join-Path $scriptDir "logs\$(Get-Date -Format 'yyyyMMdd_HHmmss')_${TaskKey}.log"

# ── Layer 1: Task Scheduler 設定（需在 Task Properties 手動設定）
# "Do not start a new instance" + "Stop if runs longer than 2 hours"

# ── Layer 2: Global Named Mutex（跨 session 單實例）
$mutexName = "Global\AgentTask-$TaskKey"
$mutex = New-Object System.Threading.Mutex($false, $mutexName)
$mutexAcquired = $mutex.WaitOne(0, $false)
if (-not $mutexAcquired) {
    "[GUARD] $TaskKey 已有執行個體，退出" | Out-File $logPath -Append
    $mutex.Dispose()
    exit 1
}

# ── Layer 3: PID-Aware FSM 殭屍清除
$fsmPath = Join-Path $scriptDir "state\fsm-$TaskKey.json"
if (Test-Path $fsmPath) {
    $fsm = Get-Content $fsmPath | ConvertFrom-Json
    if ($fsm.status -eq 'running') {
        $existingProc = Get-Process -Id $fsm.pid -ErrorAction SilentlyContinue
        $isZombie = ($null -eq $existingProc) -or
                    ($existingProc.StartTime -lt [DateTime]$fsm.started_at)
        if ($isZombie) {
            "[FSM] 清除殭屍狀態 PID=$($fsm.pid)" | Out-File $logPath -Append
            $fsm.status = 'idle'
            $fsm | ConvertTo-Json | Set-Content $fsmPath -Encoding UTF8
        } else {
            "[FSM] $TaskKey 仍在執行中（PID=$($fsm.pid)），退出" | Out-File $logPath -Append
            $mutex.ReleaseMutex(); $mutex.Dispose()
            exit 1
        }
    }
}

# ── 更新 FSM 為 running
@{ status = 'running'; pid = $PID; started_at = (Get-Date -Format 'o'); task_key = $TaskKey } |
    ConvertTo-Json | Set-Content $fsmPath -Encoding UTF8

try {
    # ── 主要 Agent 邏輯
    # ... 呼叫 claude -p 或 Python 工具 ...
    @{ status = 'idle' } | ConvertTo-Json | Set-Content $fsmPath -Encoding UTF8
} catch {
    @{ status = 'failed'; error = $_.ToString() } | ConvertTo-Json | Set-Content $fsmPath -Encoding UTF8
} finally {
    $mutex.ReleaseMutex()
    $mutex.Dispose()
}
