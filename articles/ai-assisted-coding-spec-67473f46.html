AI-Assisted Coding 實作指南：Spec-Driven Development 與 Prompt 契約工作流

研究日期：2026-03-08
研究類型：ai_sysdev
系列：ai-assisted-coding-workflows（mechanism 階段）
交叉參照：承接既有「AI 自主編碼 Agent 深度研究」與「AI 驅動的需求工程」基礎，本文聚焦 repo 內可落地的 AI-Assisted Coding 工作流。

技術概述

AI-Assisted Coding 在 2026 年的重心，已從「把 prompt 寫得更花俏」轉向「把規格、上下文、工具與驗證串成穩定工作流」。核心不是一次把所有需求塞進超長 prompt，而是先用 spec 定義意圖，再用 plan/tasks 拆出可執行步驟，讓 agent 在 gather context -> take action -> verify work -> repeat 的迴圈中逐步完成實作。這代表程式開發 Prompt 不再只是文字技巧，而是帶有輸入輸出契約、上下文邊界、失敗回退與驗證關卡的工程介面。

核心方法與工具

1. Spec-Driven Development（SDD）

GitHub 的 Spec Kit 把流程拆成 /speckit.specify -> /speckit.plan -> /speckit.tasks -> /speckit.implement。其核心思想是先定義「what / why」，再定義技術實作與任務分解，避免一開始就讓 AI 直接寫碼。這種方法特別適合需要可審計變更、多人協作與反覆驗證的 repo。

2. Prompt Contract

Prompt contract 指的是把一次 AI coding 任務的輸入輸出明文化。最低限度應包含：

  • 任務目標：要改什麼，不改什麼
  • 依據規格：PRD、issue、ADR、測試需求
  • 上下文範圍：允許讀哪些檔案、哪些資料夾先看
  • 驗證要求：必跑哪些測試、lint、型別檢查
  • 回報格式：plan、變更摘要、風險、未完成項

3. Context Engineering

Anthropic 指出，context 是有限且會出現 context rot 的資源，因此要追求「最小但高訊號」上下文。實務上不應把整個 repo 全塞進模型，而是提供：

  • 高階規則：AGENTS.md、CLAUDE.md、架構原則
  • 目標檔案路徑與相鄰模組
  • 現有測試與失敗訊息
  • 必要時用 grep、rg、head、tail 做 just-in-time 擷取

4. Verify Loop

Claude Agent SDK 把 agent loop 描述為 gather context -> take action -> verify work -> repeat。這個模式對 coding 特別重要，因為 AI 產出的價值不在第一次就完全正確，而在每一輪都有可驗證輸出：測試結果、diff、lint、log、失敗原因。

5. Subagents / Parallelization

當任務包含多個搜尋或分析面向時，可用 subagents 平行蒐集資訊，再只把濃縮結果回傳主 agent，降低上下文污染。這適合大型 repo 的「先調查再實作」場景，例如先讓一個子代理找資料流、另一個找測試入口、第三個找相依設定。

實際工作流程（step-by-step）

  1. 建立 feature spec

先用自然語言寫清楚需求、限制、驗收條件，只談 what/why，不急著談技術棧細節。

  1. 產生 implementation plan

補上架構、模組邊界、資料流、API 或 UI 約束，讓 AI 知道要在哪些層動手。

  1. 切成 tasks

把大需求拆成小任務，每個 task 都要能獨立驗證，最好附檔案路徑與依賴順序。

  1. 定義 prompt contract

明確限制本輪 agent 的工作範圍，例如只允許修改 src/ 與 tests/，且必須先讀 AGENTS.md、README.md、相關測試。

  1. 讓 agent 做 just-in-time context retrieval

不要一次灌完整 repo；先讀規則檔、目標模組與測試，再按需要擴充搜尋範圍。

  1. 執行 patch 與驗證

每完成一段修改就跑對應測試、lint、typecheck；失敗時把錯誤訊息納入下一輪上下文，而不是要求整段重寫。

  1. 產出可審計回報

要求 AI 回報：變更檔案、測試結果、尚存風險、未處理 edge cases，讓人類 reviewer 快速決策。

  1. 收斂到 merge-ready 狀態

當 spec、實作與驗證結果一致，再進入 PR 或 merge 流程，避免「能跑但不符合原意」的隱性偏差。

程式碼 / 配置範例

範例 1：Prompt Contract 模板


# Task Contract

## Goal
實作 `results/` 寫入流程的 schema 驗證，不改動既有 CLI 介面。

## Source Of Truth
- spec/results-schema.md
- AGENTS.md
- tests/results_schema.test.ts

## Allowed Scope
- src/results/**
- tests/**
- 不得修改 docs/generated/**

## Required Checks
- pnpm test -- results_schema
- pnpm lint

## Deliverables
1. 先提供 3-5 步 plan
2. 完成後列出修改檔案
3. 回報測試結果與剩餘風險

範例 3：daily-digest-prompt 可用的工作流提示


你是 repo coding agent。
先讀 AGENTS.md、templates/shared/preamble.md、目標模組與相鄰測試。
遵守以下順序：
1. 總結需求與不做事項
2. 給出最小可行 plan
3. 只讀必要檔案，不要一次展開整個專案
4. 每次修改後立即驗證
5. 若測試失敗，先分析 root cause，再繼續 patch
6. 最後輸出：變更摘要、驗證結果、風險

效果量化（生產力提升數據，若有）

  • GitHub Spec Kit 的官方流程強調從 spec 到 implement 的完整鏈條，但未提供公開、可重現的統一效能 benchmark。
  • Anthropic 兩篇官方文章主要提供工作流原理與設計準則，也未給出通用的 productivity 百分比。
  • 社群實作 cc-sdd 在其 README 中宣稱「Hours instead of weeks」，並展示一個 photo albums 範例可在約 10 分鐘內產生 requirements.md、design.md、tasks.md。這屬工具維護者主張，可作實務訊號，但不宜視為跨團隊通用基準。

與本專案的應用場景（daily-digest-prompt 如何受益）

1. 把 prompt 任務改成 spec 任務

目前專案有大量模板與 Skill，容易讓 AI 在多個規則間飄移。把新增功能先寫成 spec，可讓 agent 不必每次重新理解全域規則。

2. 為每種任務建立 contract 模板

例如 todoist research、kb import、result json writeback 都可以有各自的 prompt contract，降低 agent 在跨檔修改時的失誤率。

3. 強化 just-in-time context

agent 執行前先讀 AGENTS.md、共享 preamble、目標模板與相鄰測試，而不是把整個 repo 長文一次塞入，可減少 context rot。

4. 建立 verify gate

把 結果 JSON schema、registry 更新、KB import 成功回應 視為標準驗證關卡，讓 agent 每輪都可審計。

5. 導入調查型 subagent 模式

對較大任務，可先分派「找規則」「找相鄰檔」「找測試入口」三個子任務，最後再由主 agent 整合 patch，減少主上下文被搜尋細節污染。

風險與注意事項

  • 規格不足時，AI 只會把模糊需求高速放大；spec 品質差，產出只會更快出錯。
  • 工具太多或邊界重疊，會讓 agent 無法判斷該用哪個工具，反而增加錯誤路徑。
  • 長上下文不是免費午餐；上下文越長，越要注意 context rot 與 irrelevant token 污染。
  • 若沒有 verify loop，AI coding 很容易停在「看起來合理」而非「被證明正確」。
  • Subagent 雖能降噪，但若摘要品質差，主 agent 仍可能建立在錯誤結論上。
  • 生產力數據目前多來自工具商或社群專案宣稱，導入時應自行做 repo-level baseline 比較。
