一、專案概述

OpenHands（原 OpenDevin）是一個專注於 AI 驅動軟體開發的開源平台，目標是讓 AI Agent 像人類開發者一樣完成軟體工程任務——修改程式碼、執行命令、瀏覽網頁、呼叫 API。

解決的問題

傳統 AI 編程工具（Copilot、Cursor）主要是「建議式」——人類仍需手動審閱、測試、部署。OpenHands 提升到「自主式」層級：給定一個 Issue，Agent 可以自主閱讀程式碼、規劃方案、跨檔案修改、執行測試、提交 PR，全程無需人類介入。

二、技術架構：四層模組化設計

OpenHands V1 從 V0 的單體架構重構為清晰的四層模組化體系，解決了 V0 時代的三大痛點：剛性沙盒假設、膨脹的可變配置、研究與生產的緊耦合。

2.1 四個核心套件


openhands.sdk        → 核心抽象層（Agent, Conversation, LLM, Tool, MCP）
openhands.tools      → 具體工具實作（Bash, FileEdit, Browser, MCP 整合）
openhands.workspace  → 執行環境（Docker, 託管 API, 本地檔案系統）
openhands.agent_server → REST/WebSocket API 伺服器（遠端執行）

2.2 四大設計原則

  1. 可選隔離（Optional Isolation）：Agent 預設在本地執行，需要安全控制時可透明切換到 Docker 沙盒
  2. 無狀態元件 + 單一狀態源：所有 Agent 和 Tool 是不可變的 Pydantic 模型，只有 ConversationState 是可變的——實現確定性重播和可靠的 Session 恢復
  3. 嚴格關注點分離：Agent 核心與應用層解耦，CLI/Web UI/GitHub App 共用同一函式庫
  4. 雙層可組合性：四個模組化套件可自由組合，每個元件都是型別安全的可擴展點

三、核心技術特色

3.1 Event-Sourced 狀態管理

這是 OpenHands 最獨特的架構決策。所有 Agent 互動記錄在不可變的事件日誌中：

  • LLMConvertibleEvent：對 LLM 可見（MessageEvent, ActionEvent, ObservationEvent）
  • InternalEvent：僅用於狀態管理（ConversationStateUpdateEvent, CondensationRequest）

ConversationState 維護兩種狀態：(1) 直接存儲的可變 metadata；(2) append-only EventLog。這讓 Session 可以被暫停、序列化、遷移到另一台機器、再恢復執行。

3.2 Action–Execution–Observation 工具模式

統一的工具抽象模式：

  • Action：帶 Pydantic 驗證的輸入 schema
  • ToolExecutor：實作工具邏輯
  • Observation：帶 LLM 轉換的輸出 schema

原生工具和 MCP 工具透過同一抽象統一，工具以輕量 JSON 可序列化規格表示，適合分散式執行。

3.3 多 LLM 路由

透過 LiteLLM 支援 100+ 提供者，加上 RouterLLM 支援基於內容、成本或自訂邏輯的模型選擇。獨特功能：

  • 原生推理支援：ThinkingBlock（Anthropic）、ReasoningItemModel（OpenAI）
  • 非 Function-Calling 回退：對不支援工具呼叫的模型，自動降級為文字 prompt + 正則提取

3.4 安全架構

兩層整合設計：

  1. SecurityAnalyzer：LLM 驅動的風險評估（LOW/MEDIUM/HIGH）
  2. ConfirmationPolicy：條件式核准機制

超過風險閾值的操作觸發 WAITING_FOR_CONFIRMATION 狀態。另有 Secret Registry：工具僅在執行時存取密鑰，所有輸出中的密鑰值自動遮蔽。

3.5 生命週期控制

  • 暫停/恢復：從 checkpoint 狀態恢復執行
  • Context Condensation：LLMSummarizingCondenser 降低 API 成本最高 2×，無效能損失
  • 子 Agent 委派：透過標準工具抽象實現階層式協調
  • Stuck Detection：自動偵測無限迴圈和冗餘工具呼叫

四、產品形態與使用方式

4.1 三種介面

| 介面 | 適用場景 | 特色 |

|------|---------|------|

| CLI | 快速開發、腳本化 | 類似 Claude Code 體驗，支援任何 LLM |

4.2 快速上手


# 安裝 CLI
pip install openhands

# 設定 LLM（支援 Claude/GPT/開源模型）
export ANTHROPIC_API_KEY=sk-xxx

# 啟動互動式 Agent
openhands chat

# 或直接指定任務
openhands resolve --issue-url https://github.com/my-org/repo/issues/123

4.3 企業版功能

  • RBAC 細粒度存取控制

  • 自託管或私有雲部署（Kubernetes）
  • 完整審計與治理
  • GitHub/GitLab/Slack/Jira/Linear 整合

八、對 Daily-Digest-Prompt 專案的啟發

8.1 Event-Sourced 狀態管理

現狀：我們的 digest-memory.json 和 scheduler-state.json 是簡單 JSON 覆寫。

啟發：OpenHands 的 Event-Sourced 模式（append-only EventLog + 確定性重播）提供更可靠的狀態管理。可考慮將 JSONL 結構化日誌進一步演化為事件源，支援：

  • Session 恢復（Agent 異常終止後從最後事件繼續）
  • 狀態審計（完整操作軌跡）
  • 時間旅行除錯（回溯到任意時間點）

8.2 不可變配置模式

現狀：我們的 config/*.yaml 已實現配置外部化。

啟發：OpenHands 所有元件（Agent, Tool, LLM）都是不可變 Pydantic 模型，建構時驗證。我們可將 validate_config.py 從「事後驗證」升級為「建構時驗證」——config 載入即驗證，無效直接拒絕。

8.3 安全分層設計

現狀：我們有 3 個 Guard Hook（pre_bash/pre_write/pre_read）+ 規則外部化。

啟發：OpenHands 的 SecurityAnalyzer + ConfirmationPolicy 雙層設計更精細——不是二元的「允許/阻擋」，而是「風險評估 + 條件核准」。高風險操作暫停等待確認，而非直接阻擋。我們的 Hook 系統可借鑑此模式，增加「暫停等待」的中間狀態。

8.4 Context Condensation

現狀：我們的 Agent 在長 Session 中面臨 Context Window 壓力。

啟發：OpenHands 的 LLMSummarizingCondenser 在不損失效能的前提下將 API 成本降低 2×。我們的長 Session 自動任務（如系統審查、研究任務）可考慮引入類似的摘要壓縮機制。

九、優缺點評估

優點

  • 真正開源（MIT），不被供應商鎖定
  • 架構先進（Event-Sourced + 不可變配置 + 模組化）
  • 多 LLM 支援廣泛（100+ 提供者）
  • 企業級安全（沙盒 + 密鑰遮蔽 + 風險評估）
  • 活躍社群（474 contributors，TikTok/Netflix 等企業用戶）
  • 學術背書（arXiv 論文 + OpenHands Index 評估框架）

缺點

  • 需要 Docker 環境（對輕量部署不友善）
  • 企業版功能需商業授權
  • V0→V1 遷移中，部分文件可能過時
  • 自主編程仍需人類監督（hallucination 風險）
  • 基準測試得分因模型而異，非 OpenHands 本身能力
