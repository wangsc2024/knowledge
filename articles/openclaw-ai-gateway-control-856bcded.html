一、專案概述

OpenClaw（openclaw/openclaw）是由 Peter Steinberger 發起的開源個人 AI 助手平台，定位為「Your own personal AI assistant — Any OS, Any Platform」。截至 2026 年 3 月已累積 247,000+ Stars、47,700+ Forks，採用 MIT 授權，是 GitHub 史上成長最快的開源專案——60 天內從 9,000 飆升至 188,000 Stars。

核心數據

| 指標 | 數值 |

|------|------|

| GitHub Stars | 247,000+ |

| Forks | 47,700+ |

| 授權 | MIT |

| 語言 | TypeScript（Node.js 22+） |

| 支援通道 | 23+（WhatsApp、Telegram、Discord、Slack、Signal、iMessage、Teams 等） |

| 社群 Skills | 100+（ClawHub 註冊表） |

| 記憶體後端 | SQLite + sqlite-vec 向量擴充 |

| 部署方式 | 本機 / VPS / Docker / Fly.io |

為何爆紅

OpenClaw 填補了一個市場空缺：將 AI Agent 能力注入用戶已在使用的通訊平台，而非要求用戶切換到新介面。它不是又一個 ChatGPT 替代品，而是一個自託管的 AI 閘道器（Gateway），讓任何 LLM（Claude、GPT、Gemini、本地模型）都能透過你慣用的聊天 App 與你互動、替你執行任務。2026 年 2 月 14 日，創辦人 Steinberger 宣布加入 OpenAI，專案將移交開源基金會管理。

二、核心架構：六角架構的 AI Agent 實踐

OpenClaw 的架構本質是一個 Hexagonal Architecture（六角架構 / Ports & Adapters），將「AI 推理核心」與「外部通道」完全解耦。

2.1 架構四大層


┌─────────────────────────────────────────────┐
│  Channel Adapters（23+ 平台介面卡）          │
│  WhatsApp │ Telegram │ Discord │ iMessage... │
├─────────────────────────────────────────────┤
│  Gateway Control Plane（WebSocket 中樞）     │
│  路由 │ Session │ ACL │ Health │ Cron        │
├─────────────────────────────────────────────┤
│  Agent Runtime（PiEmbeddedRunner）           │
│  Context Assembly │ LLM │ Tool Exec │ State │
├─────────────────────────────────────────────┤
│  Storage Layer                               │
│  Sessions(JSON) │ Memory(SQLite) │ Creds     │
└─────────────────────────────────────────────┘

#### 層一：Channel Adapters（通道介面卡）

每個通訊平台有獨立介面卡，負責：

  • 認證：WhatsApp 用 QR Code（Baileys）、Telegram/Discord 用 Bot Token、iMessage 用原生整合
  • 訊息正規化：文字提取、媒體附件、執行緒上下文、表情反應
  • 存取控制：白名單、DM 配對策略（pairing/open/disabled）、群組 @mention 要求
  • 輸出格式化：Markdown 轉換、訊息分割、媒體上傳、打字指示器

#### 層二：Gateway Control Plane

核心中樞（src/gateway/server.ts），Node.js 22+ 運行，預設綁定 127.0.0.1:18789：

  • Typed WebSocket 協議：所有通訊框架經 JSON Schema 驗證
  • 事件驅動：訂閱 agent、presence、health、tick 四類事件
  • 冪等性保證：所有副作用操作需攜帶 idempotency key
  • Session 路由：DM → main（完整權限）、Channel DM → dm::（沙箱）、群組 → group::（沙箱）

#### 層三：Agent Runtime

PiEmbeddedRunner（src/agents/piembeddedrunner.ts）執行 AI 迴圈：

  1. Context Assembly：載入 session JSON → 組合動態 System Prompt → 注入相關 Skills + 記憶搜尋結果 → 自動壓縮舊 turns
  2. System Prompt 組合順序：AGENTS.md（核心基線）→ SOUL.md（個性語調）→ TOOLS.md（工具慣例）→ skills//SKILL.md（動態選擇）→ Memory 搜尋結果 → Tool 定義
  3. 串流執行：逐 token 串流 → 即時攔截 tool calls → Docker 沙箱執行 → 將 tool 結果串流回生成中的回應

#### 層四：Storage Layer


~/.openclaw/
├── openclaw.json          # 全域配置（JSON5）
├── sessions/              # Session 狀態（append-only event log）
├── memory/<agentId>.sqlite # 記憶（向量嵌入 + FTS5）
├── credentials/           # 憑證（0600 權限）
└── agents/<agentId>/
    ├── MEMORY.md           # 長期記憶（策展事實）
    └── memory/YYYY-MM-DD.md # 每日活動日誌

2.2 六階段訊息處理管線

| 階段 | 名稱 | 延遲 |

|------|------|------|

| Phase 1 | Ingestion：Baileys/grammY/discord.js 接收平台事件，介面卡提取文字與元資料 | — |

| Phase 2 | Access Control：白名單檢查、DM 配對驗證、Session 路由 | <10ms |

| Phase 3 | Context Assembly：載入 Session、組合 Prompt、查詢記憶搜尋 | <100ms |

| Phase 4 | Model Invocation：串流 context 至 LLM 提供者（Claude/GPT/Gemini/本地） | 200-500ms |

| Phase 5 | Tool Execution：攔截並執行 bash/browser/file 操作，處理 Docker 沙箱 | 100ms-3s |

| Phase 6 | Response Delivery：依平台限制格式化回應，持久化 Session 狀態 | — |

三、三大技術亮點深度剖析

3.1 動態 Skill 注入（Smart Skill Selection）

OpenClaw 不會將所有 Skill 一次性注入 prompt。而是：

  1. Runtime 分析當前 turn 的語義意圖
  2. 從 Skill 註冊表中選擇性注入僅相關的 Skill
  3. 避免 prompt 膨脹導致模型效能下降

這解決了 Agent 系統的經典難題：Skill 數量增長 vs. Context Window 有限。當 Skill 從 10 個成長到 100+ 個時，全量注入會耗盡 context 並降低推理品質。OpenClaw 的動態注入策略讓 Skill 數量可以無限擴展而不犧牲效能。

3.2 Hybrid Memory 架構

OpenClaw 的記憶系統結合三種儲存機制：

| 機制 | 格式 | 用途 |

|------|------|------|

| Event Log | JSONL | 每個事件的完整審計軌跡 |

| Long-term Knowledge | Markdown | 策展後的長期知識（MEMORY.md + 每日日誌） |

| Vector + FTS | SQLite + sqlite-vec + FTS5 | 混合搜尋（語義 + 關鍵字） |

Hybrid Search 權重配置：

  • 向量搜尋（預設 70%）：cosine similarity，處理語義相近但措辭不同的查詢
  • BM25 關鍵字搜尋（預設 30%）：FTS5 全文搜尋，擅長錯誤碼、函式名稱、唯一識別符

QMD 本地推理引擎：透過 Bun + node-llama-cpp 全本地運行，自動從 HuggingFace 下載 GGUF 模型。每個 Agent 在 ~/.openclaw/agents//qmd/ 下有自己的 XDG home。

3.3 三層沙箱信任邊界


信任等級高 ────────────────────────── 信任等級低
  main（操作者）  →  dm（已配對聯絡人）  →  group（群組成員）
  完整主機存取      Docker 沙箱           Docker 沙箱
  所有工具          受限工具              受限工具

沙箱模式三級：

  1. Off：無容器隔離
  2. Non-main（預設）：群組和次要 session 在隔離容器中執行，主 session 保留主機存取
  3. All：所有 tool 執行皆容器化

Tool Policy 優先級鏈（7 層）：

Tool Profile → Provider Profile → Global Policy → Provider Policy → Agent Policy → Group Policy → Sandbox Policy

Network 安全：

  • 預設只綁定 localhost（127.0.0.1）
  • 遠端存取建議 SSH Tunnel 或 Tailscale Serve
  • 非 loopback 連線需 challenge-response 簽章 + 明確核准
  • DM 配對需一次性確認碼

四、對 Daily-Digest-Prompt 系統的六大借鑑啟示

啟示 1：Gateway Control Plane → 統一調度中樞

OpenClaw 做法：單一 Gateway 管理所有通道路由、Session 生命週期、健康監控

我們可借鏡：目前 run-agent-team.ps1、run-todoist-agent-team.ps1、run-system-audit-team.ps1 各自獨立，缺乏統一的 Control Plane。可考慮抽象出一個「Gateway 調度層」，統一管理：

  • Phase 0 預計算
  • 子 Agent 生命週期
  • 健康監控與斷路器
  • 結果收集與組裝

預期效益：減少 3 個 team 腳本的重複程式碼（目前 Set-FsmState、circuit-breaker 邏輯重複）。

啟示 2：動態 Skill 注入 → 智慧 Skill 路由

OpenClaw 做法：根據當前 turn 語義，只注入相關 Skill 的 SKILL.md

我們可借鏡：目前 Skill-First 策略要求「先讀 SKILL_INDEX.md」，但所有 Skill 資訊都載入 context。可改為：

  1. SKILL_INDEX 只載入觸發關鍵字索引（輕量）
  2. 根據任務 labels/keywords 動態決定載入哪些 SKILL.md
  3. 特別適用於 Todoist 路由：目前 routing.yaml 已有模板選擇邏輯，可進一步延伸到 Skill 層面

預期效益：減少每次執行的 context 消耗（目前 26 個專案 Skill 全量載入相當浪費）。

啟示 3：Hybrid Memory → 增強跨次記憶

OpenClaw 做法：JSONL event log + Markdown 長期知識 + SQLite 向量/FTS5 混合搜尋（70/30 權重）

我們可借鏡：目前系統的記憶分散在多處：

  • context/digest-memory.json（結構化 JSON）
  • context/research-registry.json（研究註冊表）
  • 知識庫 localhost:3000（SQLite + 向量搜尋）
  • state/*.json（各種狀態檔）

可考慮引入 統一記憶層，將各 context JSON 的查詢統一為語義搜尋介面，讓 Agent 不需知道具體檔案位置就能檢索相關記憶。

預期效益：降低 prompt 中「必須 Read 特定檔案」的硬編碼依賴。

啟示 4：三層信任邊界 → Hook Security Preset 進化

OpenClaw 做法：main/dm/group 三層，每層有獨立的 Tool Policy 和沙箱配置

我們可借鏡：目前 hook-rules.yaml 有 strict/normal/permissive 三個 preset，但切換是手動的（環境變數 HOOK_SECURITY_PRESET）。可改為：

  1. 排程任務（無人值守）→ 自動 strict
  2. 互動模式（用戶在線）→ normal
  3. 開發模式（本地測試）→ permissive

自動根據執行環境決定信任等級，而非依賴手動設定。

預期效益：消除排程任務忘記設定 strict preset 的風險。

啟示 5：Typed Protocol + Idempotency → 可靠性提升

OpenClaw 做法：所有 WebSocket 框架經 JSON Schema 驗證，副作用操作需 idempotency key

我們可借鏡：目前 results/*.json 的寫入沒有 schema 驗證，偶爾出現格式不一致。可引入：

  1. 結果檔案 JSON Schema（config/result-schema.json）
  2. 寫入前驗證，不合格則拒絕
  3. Phase 間交接的 idempotency 保證（防止重複執行同一任務）

預期效益：減少 Phase 3 組裝失敗的根因之一（結果檔格式不符）。

啟示 6：Plugin 熱載入 → Skill 即插即用

OpenClaw 做法：extensions/ 目錄 + package.json 的 openclaw.extensions 欄位，runtime 掃描並熱載入

我們可借鏡：目前新增 Skill 需要：修改 skills/SKILL_INDEX.md → 修改 config/routing.yaml → 修改 config/frequency-limits.yaml → 可能修改 team prompt。可改為：

  1. 每個 Skill 目錄下的 SKILL.md frontmatter 就是完整的註冊資訊
  2. 建立 Skill Discovery 機制，自動掃描 skills/*/SKILL.md 建立索引
  3. 新增 Skill 只需放置目錄，無需修改其他配置

預期效益：從「新增 Skill 要改 4 個地方」降為「放好 SKILL.md 就生效」。
