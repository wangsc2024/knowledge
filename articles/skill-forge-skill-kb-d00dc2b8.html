研究日期：2026-03-22
研究系列：skill-forge-system（foundation 階段）
研究層級：Standard（Deep Research 協議）

---

執行摘要

skill-forge 是 daily-digest-prompt 專案的元認知 Skill，透過分析系統三大上下文（improvement-backlog、adr-registry、system-insight），自動識別能力缺口並生成完整可用的 SKILL.md。本研究建立 skill-forge 的基礎概念與核心機制，包含八階段執行管線、四維度評分算法（滿分 100）、三層版本策略（v1.0.0/v0.5.0/v0.1.0）、五維度 LLM 自評分品質閘門（≥7.0/10 通過）。研究發現 skill-forge 與學術界的 SkillRL、ToolFactory、MARS 等框架在「從需求自動生成能力」的理念上高度一致，但 skill-forge 獨特之處在於深度整合知識庫（KB）進行版本決策，以及多層驗證閘門確保輸出品質。

---

一、技術概述

1.1 什麼是 skill-forge？

skill-forge 是一個元認知 Skill（meta-cognitive skill）— 不執行具體領域任務，而是分析系統自身的能力空白，自動製造新的 Skill 來填補 [來源：daily-digest-prompt SKILL.md, 2026]。

其設計哲學源自三個核心問題：

  1. 觀測：系統當前缺少什麼能力？
  2. 決策：哪個能力缺口最值得優先填補？
  3. 生成：如何將缺口轉化為可執行的 Skill？

這種「AI 製造 AI 能力」的範式在 2026 年的學術與工業界中快速演進。Meta 的 MARS（Metacognitive Agent with Reflective Self-improvement）框架展示了 AI Agent 如何分析自身失敗，合成改進的 prompts [來源：Medium RAKTIM SINGH, 2026]。SkillRL 則透過強化學習，讓 Agent 從過去經驗中萃取可重用的高層次行為模式 [來源：GitHub aiming-lab/SkillRL, 2026]。

1.2 核心價值

skill-forge 解決了傳統「手動 Skill 開發」的三大痛點：

| 痛點 | 傳統做法 | skill-forge 解法 |

|------|---------|-----------------|

| 能力缺口識別 | 人工掃描系統日誌、Issue、Backlog | 自動並行讀取三大上下文，量化評分 |

| 知識基礎薄弱 | 憑經驗撰寫，缺乏文獻支持 | 呼叫 kb-research-strategist，依 KB 深度決定版本 |

| 品質不一致 | 格式、步驟完整性靠自律 | 五維度 LLM 自評分 + skill-scanner 安全掃描 |

二、八階段執行管線

skill-forge 的執行鏈如下：


觀測（三大上下文）→ 識別（優先級矩陣 + 開創性門檻）→ 去重（Explore 子 Agent）
→ 研究（kb-research-strategist）→ 生成（SKILL.md）→ 驗證（格式 + 字數 + LLM 自評分）
→ 安全掃描（skill-scanner）→ 整合（SKILL_INDEX.md）→ 通知

步驟 1：三大上下文並行讀取

skill-forge 同時讀取三個信號來源，提取「Skill 化信號」清單：

1a. improvement-backlog.json

  • 篩選條件：effort = low 或 medium，且描述未被現有 Skill 直接覆蓋
  • 重點關注：含「缺乏工具」「無法自動化」「需要 Agent 能力」的項目

1b. adr-registry.json

  • 篩選條件：implementation_status = pending 或 partial
  • 判斷：ADR 的 decision 欄位是否暗示需要新 Skill（而非只是修改配置）

1c. system-insight.json

  • 提取：alerts[]（critical/warning）、recommendations[]、skill_heatmap（top 5）
  • 識別：高頻 Skill 的未覆蓋子需求、critical alert 指向的功能缺口

這種多源信號整合策略與 ToolFactory 的 REST API 自動提取理念一致 [來源：arXiv ToolFactory, 2025]，但 skill-forge 進一步將系統內部的「改進意圖」（improvement-backlog）與「架構決策」（ADR）作為第一手信號來源。

步驟 2：優先級矩陣評分（滿分 100 分）

對每個候選 Skill 計算分數，選取最高分 1 個：


信號強度評分（0-40 分）：
  system-insight critical alert       → 40 分
  improvement-backlog P1 項目         → 35 分
  ADR pending + effort=medium         → 25 分
  improvement-backlog P2 項目         → 20 分
  system-insight warning              → 20 分
  skill_heatmap top-3 的子需求        → 15 分

技術可行性評分（0-30 分）：
  只需 Read/Write/Bash/Grep/Glob      → 30 分
  需外部 API（可快取）                → 20 分
  需 WebSearch                        → 15 分
  需安裝新工具                        → 5 分

KB 支援度評分（0-30 分）：
  KB 有 3+ 篇相關筆記                 → 30 分
  KB 有 1-2 篇相關筆記                → 20 分
  KB 無相關筆記                       → 5 分

去重罰分（-30 至 0 分）：
  現有 Skill 80%+ 覆蓋 → -30 分（排除，選次高分）
  現有 Skill 50-80% 覆蓋 → -15 分
  現有 Skill <50% 覆蓋  → 0 分

設計原理：信號強度佔比最高（40%），確保優先解決系統最迫切的問題；技術可行性次之（30%），避免生成無法執行的空殼 Skill；KB 支援度保障知識基礎（30%），這是 skill-forge 的獨特優勢。

步驟 2b：開創性門檻（必要條件）

對評分最高的候選，進行開創性評估：

「這個 Skill 的能力，能否透過組合現有 2-3 個 Skill 完全達成？」

通過判斷標準（任一即可）：

  • 新的 API 整合或資料來源
  • 新的決策邏輯（不是現有邏輯的組合）
  • 現有 Skill 沒有的輸出格式或儲存目標
  • 跨系統整合（需要兩個以上外部系統交互）

不通過判斷標準：本質上是「先呼叫 A Skill，再呼叫 B Skill」的流程包裝

這與學術界的「組合技能合成」（Compositional Skill Synthesis）研究方向一致 [來源：arXiv Agent Skills, 2026]，但 skill-forge 明確將「非組合性」作為品質閘門，避免 Skill 膨脹。

步驟 3：去重確認（委派 Explore 子 Agent）

保護主 Context Window：30 個 SKILL.md 合計 >100KB，不直接讀取。

委派 Explore 子 Agent 掃描 skills/ 目錄，僅讀取每個 SKILL.md 的前 30 行（frontmatter 區段），回傳摘要 JSON（≤6KB）。主 Agent 計算候選 Skill 與現有 Skill 的關鍵詞重疊率，> 60% 則排除。

步驟 4：KB 深研（呼叫 kb-research-strategist）

skill-forge 的核心創新在於知識庫驅動的版本決策。執行流程：

  1. 呼叫 kb-research-strategist Skill，以候選 Skill 主題為研究主題
  2. 讀取輸出的 context/kb-research-brief.json
  3. 依 kb_foundation.notes_analyzed 決定版本：

若 recommendation == "skip_kb_down"
   → v0.1.0（KB 不可用，草稿版）
否則讀取 kb_foundation.notes_analyzed
   → notes_analyzed ≥ 3 且 recommendation 為 deepen/series_continue → v1.0.0
   → notes_analyzed 1-2 → v0.5.0
   → notes_analyzed 0 → v0.1.0

| 版本 | KB 筆記數 | SKILL.md 標記 |

|------|----------|--------------|

| v1.0.0 | ≥ 3 | 正式版，無額外標記 |

| v0.5.0 | 1-2 | description 末尾加「⚠️ 知識基礎薄弱，建議透過 skill-audit 補強」 |

| v0.1.0 | 0 或 KB 不可用 | description 開頭加「⚠️ 草稿版（KB 不可用時生成）」 |

這種「知識驅動版本分級」策略在學術界尚未見到類似案例，是 skill-forge 的獨特貢獻。

步驟 5：生成 SKILL.md

依以下輸入生成完整的 SKILL.md：

  • 步驟 1 的信號上下文（what/why/signal_source）
  • context/kb-research-brief.json（how 的知識基礎）
  • 步驟 3 子 Agent 回傳中最相似的現有 Skill（格式參考）

生成約束（嚴格遵守，每條違反均導致步驟 6 失敗）：

  1. Frontmatter 完整性：必含 name、version、description（含 "Use when"）、allowed-tools、cache-ttl、triggers ≥ 3 個、depends-on
  2. 步驟結構：步驟有編號，每步說明輸入/輸出/操作
  3. 降級章節：必含「降級處理」或「錯誤處理」段落
  4. 無佔位符：禁止「此處填入你的邏輯」「TODO」「」
  5. Windows 相容性：curl POST 必須用 Write 建立 JSON 檔，Python 用 uv run python，禁止 > nul
  6. 本體字數 ≥ 600 字元（frontmatter 以外）

步驟 6：格式驗證 + 內容字數下限

用 Python 腳本驗證：

  • MISSING: []（必填欄位齊全）
  • HAS_USE_WHEN: True
  • TRIGGERS_COUNT: ≥ 3
  • BODY_CHARS: ≥ 600

若任一不通過，用 Edit 工具修正，最多重試 2 輪。2 輪後仍失敗 → 記錄 status: "format_failed"，跳至步驟 10（不整合）。

步驟 6.5：LLM 自評分（Reflexion 品質閘門）

格式驗證通過後，切換為批評者角色重新審閱剛生成的 SKILL.md，對 5 個維度各打 1-10 分：

| 維度 | 評分標準（10 分 = 最佳） |

|------|----------------------|

| 完整性 | 步驟涵蓋完整執行流程？是否有降級、輸出格式說明？ |

| 可執行性 | 每步驟有具體指令/命令/JSON 格式？還是只有抽象描述？ |

| 系統價值 | 真實解決 improvement-backlog 或 system-insight 的問題？ |

| 開創性 | 提供現有 30 個 Skill 組合無法達成的新能力？ |

| Windows 相容性 | curl 用 Write 建 JSON 檔？避免 > nul？用 uv run python？ |

執行流程（防止評分造假）：

  1. 將評分寫入 context/skill_forge_score.json
  2. 用 Python 強制計算均值（LLM 不得自行計算）
  3. 依 Python 輸出的 PASS 決定行動：

- PASS: True（平均分 ≥ 7.0）→ 繼續步驟 7

- PASS: False 且 AVERAGE ≥ 5.0 → 依最低分維度修改，重新評分（最多 1 次）

- PASS: False 且 AVERAGE < 5.0 → 記錄 status: "quality_rejected"，跳至步驟 10

這種「LLM 自評分 + Python 強制驗證」的雙層機制，類似學術界的 Self-Reflection 研究 [來源：npj Artificial Intelligence Fast, slow, and metacognitive thinking, 2025]，但 skill-forge 進一步引入外部計算防止 LLM 偏差。

步驟 7：skill-scanner 安全掃描

呼叫 skill-scanner Skill 進行安全掃描：

| 掃描結果 | status | integration_status | 處理方式 |

|---------|----------|---------------------|---------|

| clean / warning（低嚴重度） | success | integrated | 繼續步驟 8 |

| medium / high finding | partial | held_for_review | 不更新 SKILL_INDEX.md；發送 ntfy 告警；跳至步驟 10 |

| scanner 不存在 | success | integrated | 標記 scanner_skipped: true；繼續步驟 8 |

步驟 8：SKILL_INDEX.md 整合

前提：步驟 6 格式驗證通過、步驟 6.5 品質評分 ≥ 7.0、步驟 7 掃描結果為 clean/warning。

在 skills/SKILL_INDEX.md 的「工具 Skill」表格末尾追加一列，更新 Skill 總數。

步驟 9：匯入知識庫

建立生成報告並匯入 KB，包含信號來源、候選評分、KB 研究結果、生成版本、品質評分、整合結果。

步驟 10：發送通知 + 寫入結果 JSON

依 status 發送 ntfy 通知，並寫入 results/todoist-auto-skill_forge.json，包含 generated_skill、kb_research、quality_score、scanner_result、integration_status 等完整資訊。

三、核心機制深度分析

3.1 四維度評分算法的權重設計

skill-forge 的評分算法採用信號強度優先、可行性與知識並重的策略：

  • 信號強度（40%）：確保優先解決系統最迫切的問題（critical alert 40 分 vs warning 20 分）
  • 技術可行性（30%）：避免生成無法執行的空殼（只需基礎工具 30 分 vs 需安裝新工具 5 分）
  • KB 支援度（30%）：保障知識基礎（3+ 筆 30 分 vs 0 筆 5 分）
  • 去重罰分（-30~0）：防止 Skill 膨脹（80%+ 覆蓋直接排除）

這種多維度量化評分策略與業界的 AI Agent 評估框架一致 [來源：OpenAI Developers Eval Skills, 2026]，但 skill-forge 加入了「KB 支援度」這一獨特維度。

3.2 知識庫驅動的版本決策

skill-forge 的版本策略基於知識深度分級：


v1.0.0（正式版）：KB 有 3+ 篇相關筆記，知識基礎充足
v0.5.0（可用但需審查）：KB 有 1-2 篇筆記，知識薄弱
v0.1.0（草稿版）：KB 無筆記或不可用，僅依 backlog 描述生成

這種策略確保：

  • 品質透明化：使用者看到版本號即知道知識基礎強度
  • 漸進式改進：v0.1.0 可在後續 KB 補強後升級為 v0.5.0/v1.0.0
  • 降級容錯：KB 不可用時仍能生成草稿版，不中斷執行

3.3 五維度 LLM 自評分品質閘門

skill-forge 的品質閘門採用批評者角色 + Python 強制驗證雙層機制：

  1. 批評者角色：切換 LLM 角色為批評者，對 5 個維度打分（完整性、可執行性、系統價值、開創性、Windows 相容性）
  2. Python 強制計算：禁止 LLM 自行計算平均分，由 Python 腳本強制計算並判定通過與否
  3. 修正循環：平均分 5.0-6.9 時，依最低分維度修改後重新評分（最多 1 次）
  4. 拒絕門檻：平均分 < 5.0 直接拒絕，不整合

這種設計借鑑了 Self-Reflection in LLM Agents 的研究成果 [來源：Microsoft AI Agents for Beginners Metacognition, 2026]，但加入了外部驗證層防止 LLM 自評偏差。

四、與本專案的整合模式

4.1 排程執行

skill-forge 作為自動任務（todoist-auto-skill_forge），由 Windows Task Scheduler 排程觸發，每日或每週執行一次，自動檢查系統能力缺口並生成新 Skill。

4.2 信號來源更新

三大上下文由其他 Skill 更新：

  • improvement-backlog.json：由 arch-evolution Skill 更新
  • adr-registry.json：由 arch-evolution Skill 維護
  • system-insight.json：由 system-insight Skill 每日生成

4.3 Skill 生態循環


system-insight 識別問題
  ↓
improvement-backlog 記錄改進項
  ↓
arch-evolution 提出 ADR
  ↓
skill-forge 生成 Skill
  ↓
新 Skill 執行，產生新的 insight
  ↓
循環

這種自我改進循環（self-improvement loop）是 Meta-Cognitive AI 的核心特徵 [來源：Nature Fast, slow, and metacognitive thinking, 2025]。

五、與學術界/業界的對比

5.1 與 SkillRL 的比較

| 維度 | SkillRL | skill-forge |

|------|---------|-------------|

| Skill 來源 | 從過去 Agent 執行軌跡中萃取 | 從系統上下文（backlog/ADR/insight）識別 |

| 學習方式 | 強化學習 + Skill Libraries | 知識庫驅動 + LLM 生成 |

| 適用場景 | 連續決策任務（遊戲、機器人） | 系統能力擴充（DevOps、自動化） |

| 品質保障 | 基於獎勵函數 | 五維度 LLM 自評分 + scanner 掃描 |

5.2 與 ToolFactory 的比較

| 維度 | ToolFactory | skill-forge |

|------|-------------|-------------|

| 輸入 | REST API 文件 | 系統上下文（backlog/ADR/insight） |

| 輸出 | Python 函數（LangChain 相容） | 完整 SKILL.md（含步驟、降級、格式） |

| 知識基礎 | 依賴 API 文件品質 | 依賴知識庫深度（KB-driven versioning） |

| 整合方式 | 自動整合至 LangChain | 手動整合至 SKILL_INDEX.md（待自動化） |

5.3 與 Spring AI Agent Skills 的比較

| 維度 | Spring AI | skill-forge |

|------|-----------|-------------|

| Skill 結構 | 模組化資料夾 + on-demand 載入 | SKILL.md + frontmatter + 步驟式指令 |

| 發現機制 | 自動掃描 skills 目錄 | 依 SKILL_INDEX.md 的 triggers 匹配 |

| 生成方式 | 手動撰寫 | 自動生成（skill-forge）|

| 版本策略 | 無明確版本分級 | KB 驅動的三層版本（v1.0.0/v0.5.0/v0.1.0） |

skill-forge 的獨特優勢在於：

  1. 知識庫驅動的版本決策（其他框架未見）
  2. 多層品質閘門（格式驗證 + LLM 自評分 + scanner 掃描）
  3. 系統內部信號整合（backlog/ADR/insight）

六、實務應用建議

6.1 適用場景

skill-forge 最適合以下場景：

  • DevOps 自動化：系統監控發現重複性手動操作，自動生成對應 Skill
  • 能力缺口填補：ADR 提出新技術方案，skill-forge 自動生成實作 Skill
  • 知識庫擴充：研究系列完成後，自動生成應用 Skill（如完成 Todoist API 研究後，生成 todoist-sync Skill）

6.2 最佳實踐

  1. 定期執行：每週執行一次 skill-forge，確保系統能力持續擴充
  2. KB 先行：在生成新 Skill 前，先執行 kb-research-strategist 補強知識庫
  3. 版本升級：v0.1.0/v0.5.0 的 Skill 在 KB 補強後，重新執行 skill-forge 升級版本
  4. scanner 審查：被 scanner 攔截的 Skill（held_for_review）需人工審查後才能整合

6.3 常見陷阱

  1. 過度生成：不應無限生成 Skill，應依開創性門檻篩選（避免組合性 Skill）
  2. KB 依賴：KB 不可用時生成的 v0.1.0 草稿版品質較低，需後續補強
  3. 評分偏差：LLM 自評分可能過於樂觀，Python 強制驗證確保客觀性

七、知識缺口與未來方向

7.1 當前研究中的知識缺口

  1. 評分權重最佳化：四維度評分的權重（40%/30%/30%）是否為最優？需實證研究
  2. Skill 生命週期管理：Skill 生成後如何追蹤使用率、效能、過時風險？
  3. 跨專案 Skill 遷移：如何將 daily-digest-prompt 的 Skill 遷移到其他專案？
