AI-Ready 文件管線 2026：OpenAPI 單一真相、PR/Release 自動摘要、MCP 與 llms.txt 對外化

技術概述

2026 年文件自動化的重點，已從「讓 LLM 幫你寫幾段說明」轉成「把文件變成可被人與 AI 同時消費的持續交付資產」。實務上，核心不是單點生成，而是建立一條以 OpenAPI 為單一真相來源（single source of truth）的文件管線：PR 階段先生成摘要，release 階段自動彙整變更，API 描述則保留機器可讀結構，再透過 MCP 與 llms.txt 對外暴露給代理與 IDE。這條路線的價值在於減少手寫重複、降低規格漂移，並讓 AI 助手直接讀規格、查文件、執行工具，而不是反覆吃長篇 Markdown。研究上，DocAgent 與 RepoAgent 顯示多代理與 repo-level context 對文件完整性與真實性有明顯幫助；LAPIS 更進一步指出，把 API 規格壓縮成 LLM 友善格式，可顯著降低 token 成本。

核心方法與工具

  1. 契約先行：OpenAPI Initiative 明確建議 design-first，並把 OAD 放入版本控制與 CI，避免程式實作與文件分岔。
  2. 文件雙用途：OpenAPI summary / description / examples 既供人讀，也供 Swagger UI、ReadMe、Mintlify 等自動產生參考文件。
  3. PR 級自動摘要：GitHub Copilot 可直接生成 PR 摘要；GitHub 文件明示應從空白描述開始，避免既有文字污染摘要。
  4. Release 級自動變更日誌：GitHub 自動 release notes 會彙整 merged PR、contributors 與 full changelog；若要更細分，可配 Release Drafter 依 label 分類與解版號。
  5. AI 原生文件平台：ReadMe 提供 Agent、Linter、Docs Audit、MCP Server；Mintlify 則提供 Agent Job、搜尋、頁面內容擷取與 llms.txt，可把 docs 變成可程式化維護的系統。
  6. LLM 友善規格層：LAPIS 實測對 5 組 production API 規格，相較 OpenAPI YAML 平均減少 85.5% token、相較 JSON 減少 88.6%，適合作為 agent runtime 的壓縮表示。
  7. 生成品質守門：DocAgent 採 Reader / Searcher / Writer / Verifier / Orchestrator 五角分工；ReadMe Linter 則把風格規範、錯字、broken links 檢查搬進文件工作流。

實際工作流程

  1. 在 repo 維護 openapi.yaml，規定 API 變更先改規格，再改程式與 SDK。
  2. PR 建立時，讓 Copilot 先產生變更摘要；作者只補「為什麼改」與風險說明。
  3. CI 依 OpenAPI 重新生成或同步 API docs，並執行 style/lint 檢查。
  4. 若使用 ReadMe，將 OpenAPI 與 Markdown 同步進 docs hub，再跑 Linter 或 Agent 修正文案與連結。
  5. 若使用 Mintlify，可在 CI 或排程中呼叫 Agent Job/trigger update，批次修訂頁面與重新部署。
  6. Release 前由 GitHub 自動 release notes 或 Release Drafter 聚合 PR，輸出 changelog 初稿。
  7. 對外提供 llms.txt 與 MCP server，讓 IDE/agent 可直接 search/read/spec-execute，而不是只看靜態頁面。
  8. 若要給 agent runtime 更小上下文，將 OpenAPI 轉成 LAPIS 或其他壓縮視圖，只保留操作、參數、錯誤、rate limit 與 flow 宣告。
  9. 對 repo-level 文件生成需求，採 DocAgent/RepoAgent 類型架構，讓 verifier 負責 truthfulness，避免單輪摘要捏造。

程式碼/配置範例

1. OpenAPI 片段


openapi: 3.1.0
info:
  title: Digest KB API
  version: 1.0.0
paths:
  /api/import:
    post:
      summary: 匯入研究筆記
      description: |
        接收 Markdown 研究筆記並寫入知識庫。

        - `contentText` 為全文索引來源
        - `tags` 用於後續檢索與去重
      requestBody:
        required: true
      responses:
        "200":
          description: 匯入成功

2. GitHub Actions 自動草擬 release notes


name: Release Drafter

on:
  push:
    branches:
      - main

permissions:
  contents: write
  pull-requests: read

jobs:
  update_release_draft:
    runs-on: ubuntu-latest
    steps:
      - uses: release-drafter/release-drafter@v7
        with:
          config-name: release-drafter.yml


# .github/release-drafter.yml
name-template: "v$RESOLVED_VERSION"
tag-template: "v$RESOLVED_VERSION"
categories:
  - title: "Features"
    labels: ["feature", "enhancement"]
  - title: "Fixes"
    labels: ["fix", "bug"]
change-template: "- $TITLE @$AUTHOR (#$NUMBER)"
template: |
  ## Changes

  $CHANGES

3. Mintlify Agent Job / 搜尋導向的文件維護思路


curl -X POST https://www.mintlify.com/docs/api/... \
  -H "Authorization: Bearer $MINT_ADMIN_KEY" \
  -H "Content-Type: application/json"

用途：

  • Create agent job：批次改寫文件
  • Search documentation：給內部 agent 檢索現有頁面
  • Get page content：拉回全文做差異分析

4. 本專案可用的最小 PR / Changelog 規則


pr_summary_template:
  what_changed: required
  why_changed: required
  risk: optional
  rollback: optional

release_categories:
  - feature
  - fix
  - refactor
  - docs
  - infra

效果量化

  • DocAgent：ACL 2025 論文指出其在 Completeness、Helpfulness、Truthfulness 三面向穩定優於 baseline；關鍵做法是拓樸化程式碼處理順序與 Verifier。
  • RepoAgent：repo-level 文件生成與維護在定性與定量評估中優於一般生成方式，重點是主動維護與跨檔脈絡。
  • LAPIS：對 GitHub、Twilio、DigitalOcean 等 5 組 production API，平均比 OpenAPI YAML 少 85.5% token、比 JSON 少 88.6%。
  • GitHub 自動 release notes：可直接從 merged PR 與 contributor 產生初稿，節省人工整理版本說明時間；Release Drafter 再進一步把 PR 依 label 類別化，減少人工編輯。
  • ReadMe Linter：可把 style guide、錯字、broken links 與弱語氣檢查前移到寫作當下；官方明示 issue 可再交由 Agent 自動修正。

與本專案應用場景

  1. results/ 與 KB 匯入可共用「PR 摘要 → release 摘要 → note 摘要」模板，讓自動任務輸出更一致。
  2. 可為 http://localhost:3000 KB API 維護 OpenAPI 檔，避免 knowledge-query、curl body 與後端實作漂移。
  3. tools/validate_results.py、research_saturation_analyzer.py 等工具可補一層文件 lint 規則，檢查 README/Skill 範例是否與實際參數一致。
  4. 若未來要讓外部 agent 直接調用 KB，可從 OpenAPI 生成 MCP server 或至少整理成 LAPIS/精簡規格，降低上下文負載。
  5. 可把 context/research-registry.json 與 KB note metadata 自動轉成 changelog 式研究月報，減少人工整理。

風險與注意事項

  • AI 生成 PR/Release 摘要不等於可直接發布；GitHub 與 GitHub Docs 都要求人工覆核內容。
  • ReadMe 官方明示 Ask AI / Agent 會把 Markdown 與 API definitions 傳給 OpenAI，且保留 30 天 logs；Linter / Docs Audit 則使用 Gemini 2.5 Flash。敏感規格需先分級或遮罩。
  • llms.txt、MCP、公開搜尋端點會放大文件洩漏面；公開頁與內部頁需分層。
  • 單純從程式碼反推 OpenAPI 容易產生漂移。OAI 的 design-first 與 single source of truth 建議應視為主線，不要讓生成器與手改檔並存太久。
  • 若直接把完整 OpenAPI 丟給 agent，token 量與噪音都高；大 API 應切片或轉 LAPIS 類壓縮格式。
