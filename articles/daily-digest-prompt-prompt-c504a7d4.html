二、核心問題：六大重複熱點

2.1 重複段落跨檔案擴散統計

| 重複段落 | 出現次數 | 估計重複行數 | 嚴重度 |

|---------|---------|------------|-------|

| Skill-First 宣告 | 60 個檔案 | ~180 行 | 高 |

| localhost:3000 API 呼叫 | 35 個檔案 | ~210 行 | 高 |

| 品質自評迴圈 | 30 個檔案 | ~150 行 | 高 |

| DONE_CERT 輸出 | 26 個檔案 | ~104 行 | 中 |

| 研究註冊表檢查 | 24 個檔案 | ~288 行 | 高（含不一致） |

| 更新研究註冊表 | 23 個檔案 | ~368 行 | 高（含不一致） |

| 估計總重複 | — | ~1,300 行 | — |

2.2 雙重維護負擔

每個自動任務存在兩份近似檔案：

  • templates/auto-tasks/.md（單一模式用）
  • prompts/team/todoist-auto-.md（團隊模式用）

兩者差異僅在外層包裝（角色宣告、輸出路徑），核心邏輯（去重、研究、匯入、註冊表更新）幾乎相同。維護時容易只改一處忘記另一處。

2.3 一致性漂移實例

發現 1：ai-sysdev-research.md 使用舊版 registry 格式

  • 舊版：{"version":1,"entries":[]} — 無 summary 欄位
  • 新版（其他 10+ 模板）：{"version":1,"summary":{...},"entries":[]} — 有 summary
  • 影響：ai-sysdev 任務每次都會讀完整 entries 做去重，而非利用 summary 快速判定

發現 2：部分模板的判定邏輯差異

  • tech-research.md：summary.by_type.tech_research（新版 summary 路徑）
  • ai-sysdev-research.md：讀完整 entries 後手動計算（舊版方式）
  • 這種不一致來自不同時期的修改，難以靠人工審查發現

發現 3：品質自評問題不同但位置相同

  • tech-research：「是否聚焦於今日任務實際所需？」「是否提供可操作的最佳實踐？」
  • ai-sysdev：「是否包含可執行的工作流程？」「是否有程式碼或配置範例？」
  • 這是合理的差異化（不同任務有不同品質標準），但應顯式參數化而非隱性散落

三、優化建議：三層治理策略

3.1 P0：共用片段抽取（Extract Shared Fragments）

目標：消除 ~1,300 行重複代碼，建立「單一定義處」原則

方案：在 templates/shared/ 新增可參數化的共用片段


templates/shared/
  preamble.md                 # 已有：Skill-First + nul 禁令
  done-cert.md                # 已有：DONE_CERT 格式
  quality-gate.md             # 已有：品質閘門
  kb-depth-check.md           # 已有：KB 深度分析
  registry-check.md           # 新增：研究註冊表檢查（參數：task_type）
  registry-update.md          # 新增：研究註冊表更新（參數：task_type, tags）
  kb-import.md                # 新增：知識庫匯入標準流程
  quality-self-review.md      # 新增：品質自評（參數：criteria 列表）
  dedup-search.md             # 新增：去重搜尋兩階段流程

引用方式（在各模板中）：


## 第零步：研究註冊表檢查
讀取 `templates/shared/registry-check.md`，以 task_type=`tech_research` 執行。

預估收益：

  • 每個研究模板從 ~120 行減至 ~50 行（僅保留任務特定邏輯）
  • 修改去重邏輯只需改 1 個檔案（目前需改 24 個）
  • 消除一致性漂移的根本原因

3.2 P1：雙軌模板合併（Single Source of Truth）

問題：templates/auto-tasks/ 和 prompts/team/todoist-auto-* 的雙重維護

方案 A（推薦）：團隊模式 prompt 改為動態引用模板

團隊模式 prompt 精簡為 ~15 行，僅含角色宣告、模板引用和輸出路徑設定。

方案 B：建構時生成

  • 寫一個 generate-team-prompts.ps1 腳本
  • 從 templates/auto-tasks/.md + 團隊模式 wrapper 模板自動生成 prompts/team/todoist-auto-.md
  • 修改源模板後執行一次生成即可

預估收益：

  • 消除 ~2,000 行重複（19 個自動任務 x ~100 行重複部分）
  • 維護點從 2N 降為 N+1（N 個源模板 + 1 個 wrapper）

3.3 P2：模板健康度量化（Template Health Metrics）

在 check-health.ps1 中新增模板健康指標：

  • template_total_lines（警戒閾值 > 8,000）
  • template_count（警戒閾值 > 75）
  • avg_lines_per_template（警戒閾值 > 120）
  • shared_fragment_ratio（警戒閾值 < 15%）
  • duplication_hotspot_count（警戒閾值 > 5）

四、實施路線圖

Phase 1（P0 緊急修復，1-2 小時）

  1. 修復 ai-sysdev-research.md 的 registry 格式（對齊新版 summary 方案）
  2. 建立 templates/shared/registry-check.md 和 registry-update.md
  3. 在 2-3 個模板中試行引用，驗證效果

  1. 建立 templates/shared/kb-import.md 和 dedup-search.md
  2. 所有研究模板改為引用共用片段
  3. 選擇方案 A 或 B 合併團隊模式 prompt

  1. check-health.ps1 新增模板健康指標
  2. 設定基線值和警戒閾值

五、風險評估

  1. 共用片段的 Token 成本：LLM 每次需額外 Read 共用檔案，增加 ~5 個工具呼叫。但消除重複帶來的維護收益遠大於此。
  2. 間接引用的可理解性：模板不再自包含，需要跨檔案閱讀。建議在每個共用片段中加入一行用途說明。
  3. 向後相容：單一模式和團隊模式的讀取路徑不同，合併需確保兩條路徑都能正確解析。

---

七、結論

Daily-Digest-Prompt 的「文件驅動架構」理念正確，但模板層在 20+ 個自動任務的擴張下出現了「模板膨脹」（Template Bloat）— 7,924 行中估計 ~1,300 行是純重複。這不僅增加維護成本，更導致一致性漂移（如 ai-sysdev 的 registry 格式落後）。

核心解法是共用片段抽取（Extract Shared Fragments），遵循已在 preamble.md 和 hook-rules.yaml 驗證成功的「單一定義處」原則。預計可將模板總量減少 30%，將修改一個通用邏輯的成本從修改 24 個檔案降為修改 1 個檔案。
