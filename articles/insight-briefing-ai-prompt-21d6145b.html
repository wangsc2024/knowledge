認知負荷理論與 AI Prompt 設計：資訊分塊、CoT工作記憶與注意力配置最佳化（2026）

研究日期：2026-06-05 | 類別：認知科學/心理學 | 來源數：5

---

執行摘要

認知負荷理論（Cognitive Load Theory, CLT）由 Sweller 提出，原是教育心理學核心架構，2024-2026 年間被系統性地移植至 LLM Prompt 設計與 Agent 架構研究。核心命題是：LLM 的推論能力受限於其處理「有用資訊」的預算（germane computational load），而非模型規模本身。設計不良的提示注入冗餘負擔（extraneous load），消耗了應用於真實推理的計算資源。本期簡報彙整五篇前沿研究，提煉出五大核心洞見：CLT 三維度量化映射、CLAI-Prompt 零樣本優化框架、多代理工作記憶瓶頸、外部記憶四層架構，以及從 Prompt Engineering 到 Agent Engineering 的範式轉移。這些洞見對本系統的 Harness 設計、Prompt 瘦身與多代理協調具有直接應用價值。

---

核心洞見

1. CLT 三維度可量化映射到 LLM Token Economy

洞見：傳統 CLT 的三種負荷類型——內在負荷（ICL，問題本身的複雜度）、外在負荷（ECL，冗餘/無關資訊的計算消耗）、生成性負荷（GCL，用於真實推理的生產性計算）——可被精確量化為 LLM 的 token 使用分布。CLAI-Prompt 框架以此為基礎，透過結構化元提示（meta-prompt）引導 LLM 自我評估問題複雜度，動態分配推理預算。

支撐證據：arXiv 2507.00653「Cognitive Load-Aware Inference」在 GSM8K 與 MATH 基準上達成超過 35% 的 token 減少，且準確率基本不變；在長文本 QA 任務上優於 LLMLingua 等專用壓縮工具。

實踐建議：在本系統的長提示中導入「複雜度自評」前置步驟：指示 LLM 先評估任務複雜度（簡單/中等/複雜），再動態選擇回應深度。這可減少「把簡單任務複雜化」的 token 浪費。

---

2. 多代理協作存在工作記憶上限，代理數≠效能

洞見：多代理系統中，每增加一個代理即增加每個 LLM 的認知負荷（tracking other agents' states, resolving conflicts）。研究實驗顯示代理數量與協調效能呈反比，高認知負荷下模型易出現冗餘回應、協調失敗與語意漂移。與人類工作記憶限制（Miller's Law: 7±2 chunks）類比，LLM 也存在「協調瓶頸」。

支撐證據：arXiv 2506.06843「United Minds or Isolated Agents?」使用注意力熵作為認知負荷代理指標，實證代理數上升時每個 LLM 的注意力分散程度顯著增加，系統整體推理品質下降。

實踐建議：本系統的多代理編排（如 Todoist Agent Team 四輪並行）應設計明確的角色邊界與通訊協議，避免代理間狀態全量共享。每輪應只傳遞「決策所需的最小上下文」，而非完整 run 記錄。

---

3. 外部記憶四層架構是降低模型認知負荷的根本方案

洞見：arXiv 2604.08224 提出 LLM Agent 的外部化框架：將認知負擔從模型內部轉移到四層可檢查的外部結構——(1)工作上下文（任務中即時狀態）、(2)情節經驗（先前運行記錄）、(3)語義知識（跨情節的抽象知識）、(4)個人化記憶（用戶/環境特定資訊）。Harness Engineering 作為協調層，整合這四層記憶並提供沙箱、可觀測性與治理機制。

支撐證據：論文直接呼應本系統的架構——state/（情節經驗）、context/（語義知識）、KB 知識庫（個人化記憶）、results/（工作上下文）的分層恰與此框架對應。

實踐建議：可依此框架審查本系統的 Harness 資訊注入策略：Phase 0 應只注入「工作上下文」（當日摘要快照），而非每次都重載「情節經驗」（歷史 continuity）。情節記憶應改為按需查詢（via KB）而非批量注入。

---

4. 認知負荷感知的動態稀疏激活可提速 18-25%

洞見：arXiv 2502.19078（CLADA 框架）發現 LLM 在處理高認知負荷 token（高驚奇度/高熵的詞）時激活更多神經元；低負荷 token 的激活模式高度重複。利用此規律，可在不重新訓練模型的前提下，對低負荷 token 套用靜態稀疏掩碼，對高負荷 token 動態擴展激活，實現推論加速。

支撐證據：6 個 LLM × 9 個基準，平均 18-25% 推論加速，精度損失 <2%。首次建立神經語言學的事件相關電位（ERP）與 LLM 推論效率的正式連結。

實踐建議：對本系統中使用本地 LLM（llama_cli_runner）的任務，可評估 CLADA 類型的稀疏激活插件（如 llama.cpp 的 speculative decoding）作為效能提升選項。結構化/短回應任務（如分類、摘要）加速效益最大。

---

5. 從 Prompt Engineering 到 Agent Engineering：設計認知架構而非調整語言

洞見：PMC 論文明確劃分兩個時代：Prompt Engineering 依賴靜態、一次性指令（本質是「凍結的請求」）；Agent Engineering 設計的是認知架構本身——包含代理規格、協調機制、評估體系與治理框架。核心轉變是「人類-AI 關係從命令-回應到夥伴-共發現」。設計重心從「如何表述問題」轉移到「如何組織解決問題的認知流程」。

支撐證據：「設計認知架構本身，而非調整語言表述」（PMC/12613637）。類比「組建人類專家團隊」，強調專業化分工（文獻代理、因果推論代理、驗證代理等）而非一個模型做全部。

實踐建議：本系統的 Todoist Agent 設計已體現此精神（Phase 0-1-2-3 分層），但 Phase 2 個別 prompt 仍偏向「Prompt Engineering」風格（線性指令清單）。可嘗試在複雜任務（如 insight_briefing）中增加「任務拆解與角色分配」前置段，讓 LLM 自行規劃子代理工作流。

---

建議行動

  1. 引入 CLT 複雜度自評：在本系統的長 prompt（>2000 token）頭部加入一行：「先評估本任務複雜度（簡單/中等/複雜），再決定回應深度。」預計降低簡單任務的 token 消耗 20-35%。
  1. 多代理協調瘦身：Todoist Agent Team 代理間資訊傳遞改採「最小決策上下文」原則，每輪只傳 context diff 而非全量 Phase 1 結果。實作位置：run-todoist-agent-team.ps1 的 Phase 2 dispatch 組裝段。
  1. 記憶層分類標籤：在 config/agent-memory-layers.yaml（如不存在則新增）中顯式標籤現有資料路徑對應的記憶層級（工作/情節/語義/個人化），作為未來 Harness refactor 的基礎。
  1. KB 按需查詢替代批量注入：Phase 0 snapshot 不應包含歷史研究系列全文，改為只在 snapshot 中記錄系列標題+最新主題，具體內容在 Phase 2 按需從 KB 查詢。
  1. 本地 LLM 稀疏激活評估：評估 llama.cpp v0.9+ 的 speculative decoding 對本系統 tools/local_model_lock.py 管理的推論任務的適用性，目標為縮短 zen_koan/thought_distillation 的本地推論時間。

---

後續研究方向

  1. CLAI-Prompt 在本地 LLM 的可移植性：CLAI-Prompt 為零樣本框架，理論上可直接用於 Gemma/Llama 等本地模型，值得以 zen_koan 任務做 A/B 測試驗證。
  1. 多代理認知負荷量化工具：開發本系統的代理間通訊負荷指標（如 context entropy per round），建立 baseline 並追蹤優化效果。
  1. 外部記憶架構的完整性審查：對照論文四層分類，審查本系統所有 state/、context/、results/ 路徑，識別未被正確分類或跨層混用的情況。
