  2. 可執行性高（35 分）：僅需建立診斷 workflow，不修改 PS 核心邏輯
  3. 信號強度強（25 分）：來自 improvement-backlog critical 項目
  4. 開創性：補齊結果檔診斷標準化流程的空白

產出內容

1. 工作流 YAML

路徑：workflows/result-file-diagnostic-workflow.yaml

定義 6 步驟診斷流程：

  1. 檔案存在性檢查：驗證 results/todoist-auto-{task_key}.json 是否存在
  2. JSON 有效性驗證：解析 JSON，確認非損壞
  3. 必填欄位驗證：確認 agent、task_key、status、summary 欄位存在
  4. 診斷欄位驗證：確認 cli_exit_note_zh、result_guard_zh 欄位存在且非空
  5. 根因分類：依前 4 步結果，分類為 4 種根因
  6. 修復建議：依根因提供可執行修復步驟

2. 根因分類（4 種）

| 根因 | 條件 | 證據 |

|------|------|------|

| backend_execution_failed | result_file_missing + backend_status != ok | cursor_cli stdout 無 Write 指令、backend_status 為 disabled/quota_exceeded |

| file_overwritten | result_file_missing + backend_status = ok + stdout 有 Write | Phase 2 有多個任務寫同一 task_key（衝突）、Repair/備援流程覆寫原始結果 |

| diagnostic_not_applied | diagnostics_missing + JSON valid + schema incomplete | PS 流程 try-catch 異常提早返回、佔位 JSON 跳過補全 |

| json_corruption | json_invalid | 檔案尾缺 } 或 ]、timeout 或 Ctrl+C 中斷寫入 |

3. 追蹤機制

診斷結果寫入 state/result-file-diagnostics.json（保留 7 天），schema 包含：

  • task_key：自動任務識別碼
  • timestamp：診斷執行時間（ISO 8601）
  • diagnostic_result：ok | missing | invalid | incomplete | diagnostics_missing
  • root_cause：根因分類（若 ok 則為 null）
  • repair_suggestion：修復建議（若 ok 則為 null）
  • evidence：診斷證據清單

4. 整合路徑

觸發點：run-todoist-agent-team.ps1 Phase 2 迴圈結束後，呼叫 Invoke-ResultFileDiagnostic 函數。

本 workflow 定義流程規範，實際執行邏輯仍在 PowerShell 腳本中。
