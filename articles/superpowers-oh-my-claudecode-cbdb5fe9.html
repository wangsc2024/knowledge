執行摘要

2026 年 Agent Skill 生態系統迎來爆發期，Anthropic 開放 Skill 標準後，GitHub 前十大新專案中有五個與 Skill 相關。本文深度分析兩個代表性框架：Superpowers（obra/superpowers，137K Stars）與 oh-my-claudecode（Yeachan-Heo/oh-my-claudecode，24.9K Stars），提煉可借鏡的架構模式與技術項目。

一、Superpowers — 方法論驅動的 Skill 框架

1.1 專案概覽

| 維度 | 數據 |

|------|------|

| Stars | 137K（3 個月增長 27K，月均 9K） |

| Forks | 11.6K |

| 作者 | Jesse Vincent（Prime Radiant） |

| 授權 | MIT |

| 最新版本 | v5.0.7（2026-03） |

| 技術棧 | Shell 58.8%、JavaScript 29.6%、Python 3.7% |

| 平台支援 | Claude Code、Cursor、Codex、OpenCode、Gemini CLI、GitHub Copilot CLI |

1.2 架構設計：七階段開發管線

Superpowers 的核心理念是「方法論即 Skill」，將軟體開發最佳實踐封裝為自動觸發的 Skill 模組：

  1. Brainstorming — 蘇格拉底式提問精鍊需求，產出設計文件
  2. Git Worktrees — 隔離開發分支，驗證測試基線
  3. Planning — 拆解為 2-5 分鐘可驗證任務（含檔案路徑、程式碼、驗證步驟）
  4. Execution — 每個任務派發子 Agent，兩階段審查（合規 + 品質）
  5. TDD — 強制 RED-GREEN-REFACTOR 循環
  6. Code Review — 對照計畫審查，阻斷關鍵問題
  7. Branch Completion — 處理 merge/PR 與清理

1.3 Skill 系統設計（核心亮點）

SKILL.md 格式：


---
name: skill-name
description: Use when <觸發條件描述>
---

觸發機制：「如果你認為某個 Skill 有哪怕 1% 的可能適用，你必須調用它。」這是最激進的 Skill-First 策略，比本系統的「能用 Skill 就用 Skill」更強。

Skill 分類：

  • Rigid Skills（TDD、Debugging）：嚴格遵循，不可調整
  • Flexible Skills（模式類）：依上下文自適應

自動發現：從 /skills/ 目錄層級發現，session 啟動時載入 metadata，按需載入完整內容。

平台適配：跨 6 個 AI 編輯器平台統一 Skill 介面，透過映射檔適配不同工具名稱。

1.4 並行 Agent 派發模式

dispatching-parallel-agents Skill 的設計精華：

  • 領域分組：按問題域分配獨立 Agent（非按檔案）
  • 隔離上下文：子 Agent 不繼承主 session 的歷史
  • 精確範圍：每個 Agent 收到具體 scope + 成功標準 + 約束 + 輸出格式
  • 結果驗證：讀取摘要 → 衝突檢查 → 全套件測試 → 抽樣驗證
  • 反模式警告：禁止「修復所有測試」等寬泛指令

二、oh-my-claudecode — 團隊優先的多 Agent 編排

2.1 專案概覽

| 維度 | 數據 |

|------|------|

| Stars | 24.9K |

| Forks | 2.3K |

| 作者 | Yeachan Heo |

| 授權 | MIT |

| 提交數 | 2,276 |

| 技術棧 | TypeScript |

| 安裝方式 | Plugin Marketplace / npm |

2.2 團隊編排架構（Team Pipeline）


team-plan → team-prd → team-exec → team-verify → team-fix (loop)

這是一個完整的團隊協作流水線，與本系統的 Phase 0→1→2→3 異曲同工，但更強調持續驗證迴圈（team-verify → team-fix 循環直到通過）。

2.3 五種執行模式

| 模式 | 用途 | 對應本系統 |

|------|------|------------|

| Team | 分階段協調管線 | run-todoist-agent-team.ps1 Phase 模式 |

| Autopilot | 單 Agent 自主執行 | run-todoist-agent.ps1 單一模式 |

| Ultrawork | 最大並行（5 workers） | 無對應，可借鏡 |

| Ralph | 持久模式 + verify/fix 迴圈 | 無對應，可借鏡 |

| Pipeline | 順序多階段處理 | Phase 0→1→2 管線 |

2.4 智慧模型路由

自動依任務複雜度選擇模型：Haiku 處理簡單任務、Opus 處理複雜推理。這與本系統的 llm-router.yaml mapping 式路由異曲同工。

2.5 CLI Worker 整合（v4.4.0+）

基於 tmux 的按需 worker 生成：

  • omc team N:codex — Codex CLI 實例
  • omc team N:gemini — Gemini CLI 實例
  • omc team N:claude — Claude CLI workers

任務完成後自動終止，消除閒置資源消耗。這是跨 LLM 後端混合編排的創新實踐。

2.6 可觀測性

  • HUD statusline：即時編排指標
  • Session summaries：.omc/sessions/*.json
  • Replay logs：.omc/state/agent-replay-*.jsonl
  • Stop callbacks：Telegram、Discord、Slack 通知

四、架構比較分析

| 維度 | Superpowers | oh-my-claudecode | 本系統（Daily Digest） |

|------|------------|-----------------|----------------------|

| 核心理念 | 方法論驅動 | 團隊編排驅動 | 文件驅動 + Skill-First |

| Skill 格式 | YAML frontmatter + MD | 自訂格式 + auto-inject | YAML frontmatter + MD（相似） |

| 觸發機制 | 1% 可能性即觸發 | 關鍵字 + auto-inject | 觸發關鍵字 + SKILL_INDEX |

| 並行模式 | 子 Agent 派發 | tmux worker pool | PowerShell Start-Job |

| 品質閘門 | 兩階段審查（合規+品質） | verify/fix 迴圈 | Draft-Verify + quality_gate |

| 可觀測性 | 無（依賴平台） | HUD + replay logs | JSONL hooks + query_logs |

| 自癒能力 | 無 | Ralph 持久模式 | OODA 閉環 + Harness Gap Loop |

| LLM 路由 | 無（單一模型） | 智慧模型路由 | llm-router.yaml mapping |

關鍵洞察

  1. Superpowers 的方法論封裝是最大創新：把 TDD、Code Review、Planning 等軟體工程最佳實踐直接變成 Skill，使 Agent「自帶紀律」。本系統的 test-driven-development 等 Skill 已有類似實踐，但分類（Rigid/Flexible）與觸發激進度可加強。
  1. oh-my-claudecode 的混合後端編排是差異化亮點：同一團隊中混合 Claude/Codex/Gemini worker 是首見，利用不同模型的強項分工。本系統已有 Groq relay 但僅用於翻譯/分類，可評估擴展至任務執行。
  1. 兩者共同驗證了本系統的方向正確性：文件驅動、Skill-First、並行 Agent、Phase 管線 — 這些核心設計選擇都在頂尖開源專案中獲得驗證。
