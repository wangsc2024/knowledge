二、三檔案極簡架構

Autoresearch 的設計哲學是「一個 GPU、一個檔案、一個指標」（One GPU, One File, One Metric）：

2.1 檔案結構


autoresearch/
  prepare.py    # 固定常數 + 一次性資料準備（下載訓練資料、訓練 BPE tokenizer）
  train.py      # Agent 唯一可編輯的檔案（GPT 模型 + Muon/AdamW 優化器 + 訓練迴圈）
  program.md    # Agent 的指令手冊（Markdown 格式，人類撰寫研究策略）

2.2 各檔案職責

  • prepare.py（不可變層）：下載訓練資料、訓練 BPE tokenizer、提供 dataloader 和 evaluation 工具。Agent 不可修改此檔案。
  • train.py（可變層，~630 行）：包含完整 GPT 模型定義、Muon + AdamW 混合優化器、訓練迴圈。這是 Agent 唯一能修改的檔案。
  • program.md（指令層）：以 Markdown 撰寫的研究策略文件，描述 Agent 應如何執行實驗、探索哪些類型的改動。人類撰寫，90% 內容由 AI 輔助生成。

2.3 架構設計洞察

此三層分離與 Daily-Digest-Prompt 的架構有深度同構：

| Autoresearch | Daily-Digest-Prompt | 角色 |

|-------------|---------------------|------|

| prepare.py | PowerShell 基礎設施（run-agent-team.ps1） | 不可變基礎層 |

| train.py | Agent 可修改的執行邏輯 | 可變實驗層 |

| program.md | prompts/ 模板 + config/ YAML | 指令/策略層 |

三、自主實驗迴圈（Agent Loop）

3.1 迴圈機制


讀取 program.md 指令
  ↓
分析 train.py 現有程式碼
  ↓
形成改進假設（如：調整學習率、修改架構深度）
  ↓
修改 train.py
  ↓
執行 5 分鐘固定時間訓練
  ↓
評估 val_bpb（validation bits-per-byte）
  ↓
若改進 → Git commit 到 feature branch
若未改進 → 還原修改
  ↓
重複

3.2 關鍵設計決策

  1. 固定 5 分鐘時間預算：無論硬體規格，每次實驗固定 5 分鐘壁鐘時間。這確保：

- 每小時約 12 個實驗

- 一晚（8 小時）約 100 個實驗

- 實驗成本完全可預測

  1. 單一明確指標（val_bpb）：validation bits-per-byte 是詞彙量無關的評估指標，越低越好。消除了多指標權衡的複雜性。
  1. Git 分支累積式改進：Agent 在 feature branch 上工作，每次找到更好的設定就 commit。累積式設計意味著改進是可追溯、可回滾的。
  1. 修改範圍：Agent 可調整神經網路架構、優化器設定、所有超參數（batch size、warmup/warmdown、window pattern、learning rate、RoPE frequency 等）。

四、實驗成果

4.1 單 GPU 基準測試（83 實驗）

| 指標 | 數值 |

|------|------|

| 總實驗數 | 83 |

| 保留的改進 | 15 |

| 丟棄率 | ~82% |

| 最終 val_bpb | 0.9697 |

4.2 生產環境測試（8×H100，276 實驗）

| 指標 | 數值 |

|------|------|

| 總實驗數 | 276 |

| 保留的改進 | 29 |

| 丟棄率 | ~89% |

4.3 長期自主運行（depth=12 模型，2 天）

  • Agent 自主處理約 700 次程式碼修改
  • 發現約 20 個可累加的改進
  • 關鍵發現：小模型上的改進完美遷移至大模型（scale transfer）

主要改進類別：batch size 優化、warmup/warmdown 調整、window pattern 修改、learning rate 精煉、RoPE frequency 修改。

六、SETI@home 願景：分散式協作實驗

Karpathy 提出更宏大的願景：大規模非同步協作 AI Agent，類似 SETI@home 的分散式運算模式。

核心概念：

  • 數千個 Agent 各自在不同 GPU 上跑實驗
  • 共享實驗結果（成功的 commit）到公共池
  • 自動合併相容的改進
  • 24/7 持續優化，無需人類介入

Karpathy 的名言："The best labs won't just have the most compute. They'll have the best program.md."（最好的實驗室不再是擁有最多算力的，而是擁有最佳 program.md 的。）

七、對 Daily-Digest-Prompt 系統的借鑑啟示

7.1 固定時間預算模式

autoresearch 的 5 分鐘固定預算與本專案 config/timeouts.yaml 的超時配置理念一致。可借鑑：

  • 為每個自動任務設定固定執行預算（非僅超時上限）
  • 固定預算讓排程更可預測，避免長尾延遲

7.2 單一指標驅動

val_bpb 的成功在於其明確性。本專案可借鑑：

  • 每個自動任務定義單一成功指標（如 KB 匯入成功數、研究品質分數）
  • 避免多指標模糊化判定

7.3 Git 累積式改進

Agent 在 feature branch 上 commit 的設計可借鑑：

  • 自動任務的改進（如 YAML 配置調整）可自動 commit 到專用 branch
  • 透過 PR review 機制人工審核 Agent 的改動

7.4 program.md 作為策略層

program.md 的角色與本專案的 templates/ 目錄完全對應。啟示：

  • 「贏在 program.md」= 我們的「Prompt 品質決定產出品質」
  • 策略層的持續精煉比增加算力更有效

---

八、限制與挑戰

  1. 單一 NVIDIA GPU 依賴：目前僅支援 NVIDIA GPU（CUDA），不支援 AMD/Apple Silicon
  2. 固定 5 分鐘可能不足：某些實驗需要更長的訓練時間才能顯現差異
  3. 指標單一性：val_bpb 無法捕捉所有面向（如生成品質、安全性）
  4. Agent 品質依賴：實驗品質高度依賴所使用的 LLM Agent 能力
  5. Scale Transfer 風險：小模型上的改進不保證在所有情況下都能遷移
