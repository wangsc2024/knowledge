Daily-Digest-Prompt PowerShell 編排層架構債務與模組化治理

38 腳本清點、500 行重複函式萃取、巨型腳本拆解與單模式遺孤處理策略

研究日期：2026-03-14

類型：專案優化 / 深度思維 / 知識庫

適用對象：daily-digest-prompt 專案

方法：RAG 知識庫交叉比對 + 子 Agent 靜態分析 + Web 最佳實踐研究

---

一、問題本質：PowerShell 是專案的隱形骨架

1.1 現況概覽

daily-digest-prompt 專案的 Python hooks（529 測試）、Skill 生態（34 個）、YAML 配置（25 個）都已有完善治理。但作為編排層骨架的 PowerShell 腳本長期缺乏結構化管理：

| 指標 | 數值 | 風險等級 |

|------|------|----------|

| 根目錄 .ps1 檔案數 | 35 個 | 高（無清單管理） |

| 總行數 | ~10,000 行 | 中（最大單檔 1,731 行） |

| 共用模組 (.psm1) | 0 個 | 極高（零模組化） |

| 函式重複定義次數 | 5 函式 × 2-7 份 | 極高（500+ 行重複） |

| 單模式遺孤腳本 | 3 個 | 中（維護成本） |

1.2 為何此前未被關注

既有 KB 研究聚焦在 Python hooks、YAML 配置、JSON 狀態檔、Skill 依賴圖、Token 效率、可觀測性等面向。PowerShell 作為「膠水層」被視為理所當然，但實際上承擔了：

  • 狀態機控制（Set-FsmState，100+ 行/份 × 3 份）
  • 遙測追蹤（Write-Span，36 行/份 × 3 份）
  • 故障統計（Update-FailureStats，複雜 JSON 操作）
  • 排程編排（Phase 0→1→2→3 全流程）

二、五大架構債務深度分析

2.1 債務一：DRY 違反 — 500+ 行重複函式（CRITICAL）

子 Agent 靜態分析發現以下函式在多個腳本中完全或近似重複定義：

| 函式名 | 出現腳本數 | 每份行數 | 總重複行數 | 風險 |

|--------|-----------|---------|-----------|------|

| Set-FsmState | 3 | 100+ | 300+ | Bug 修正需改 3 處 |

| Update-State | 5 | 40-50 | 200+ | JSON 持久化邏輯 5 份 |

| Write-Log | 7+ | 4-6 | 28+ | 日誌格式不一致風險 |

| Write-Span | 3 | 36 | 72 | 遙測格式漂移 |

| Remove-StderrIfBenign | 3 | 22 | 44 | 過濾規則分歧 |

真實風險案例：2026-02-24 FSM 殭屍偵測修復需同時修改 run-todoist-agent-team.ps1 和 run-agent-team.ps1 的 Set-FsmState，若漏改一處即產生行為不一致。

業界最佳實踐對照：

  • Microsoft PowerShell Style Guide 建議「頻繁複製貼上的函式應萃取為模組」
  • PowerShell Practice and Style Guide 指出「可重用工具應具備高復用性，僅透過參數接受輸入」

2.2 債務二：零模組化 — 無 .psm1、無 lib/ 目錄

現況：35 個根目錄腳本中，僅有 1 個 dot-source 引用（test-precheck.ps1 引用 circuit-breaker-utils.ps1），其餘完全獨立、自成一體。

影響：

  1. 測試困難：無法對共用函式進行單元測試（Pester），因為函式散落在各腳本中
  2. 版本漂移：同名函式在不同腳本中逐漸產生微妙差異（參數名、錯誤處理、日誌格式）
  3. 認知負擔：新維護者需逐一閱讀腳本才能理解「Set-FsmState 到底定義在哪」

建議架構：


lib/
  DailyDigest.psm1          # 主模組（Export-ModuleMember 控制）
  DailyDigest.psd1          # 模組清單（版本、依賴）
  Public/
    Set-FsmState.ps1        # 狀態機控制
    Update-State.ps1        # JSON 持久化
    Write-Span.ps1          # 遙測追蹤
  Private/
    Write-Log.ps1           # 內部日誌
    Remove-StderrIfBenign.ps1  # 過濾工具

各腳本頂部改為：


Import-Module "$PSScriptRoot/lib/DailyDigest.psm1" -Force

預估收益：減少 500+ 行重複、統一日誌格式、啟用 Pester 模組測試。

2.3 債務三：巨型腳本 — 1,727 行的 run-todoist-agent-team.ps1

分析：

  • 13 個函式定義（~700 行純函式）
  • ~600 行樣板/重複程式碼
  • ~400 行主流程邏輯
  • 職責混合：FSM 管理 + 失敗統計 + 告警推送 + YAML 解析 + 多後端作業啟動

拆解建議：

| 責任區塊 | 目標位置 | 行數 |

|----------|---------|------|

| FSM + 狀態管理 | lib/Public/Set-FsmState.ps1 | ~120 |

| 失敗統計 + 告警 | lib/Public/Update-FailureStats.ps1 | ~80 |

| 遙測 Span | lib/Public/Write-Span.ps1 | ~40 |

| YAML 解析 | lib/Private/ConvertFrom-YamlViapy.ps1 | ~30 |

| 多後端作業 | lib/Public/Start-AgentJob.ps1 | ~60 |

| 主流程 | run-todoist-agent-team.ps1（精簡版） | ~400 |

精簡後主腳本從 1,727 行降至 ~400 行，專注於 Phase 編排邏輯。

2.4 債務四：單模式遺孤腳本

以下腳本有 Team 模式等價物，但仍保留在專案中：

| 單模式腳本 | Team 等價物 | 行數 | 建議 |

|-----------|------------|------|------|

| run-agent.ps1 | run-agent-team.ps1 | 180 | 標記 deprecated 或刪除 |

| run-todoist-agent.ps1 | run-todoist-agent-team.ps1 | 199 | HEARTBEAT.md 仍引用，保留但精簡 |

| run-system-audit.ps1 | run-system-audit-team.ps1 | 174 | 標記 deprecated |

注意：run-todoist-agent.ps1 在 HEARTBEAT.md 排程中仍被「每小時整點」觸發，不可貿然刪除。建議改為 thin wrapper 呼叫 team 腳本的子集功能。

2.5 債務五：腳本清單缺失

35 個根目錄腳本無統一清單（manifest），導致：

  • 無法快速判斷哪些是「生產用」、「開發工具」、「已棄用」
  • check-health.ps1 無法驗證「是否有腳本被意外刪除」

建議：建立 docs/SCRIPTS_MANIFEST.md，列出每個腳本的類型（生產/工具/遺孤）、狀態（Active/Deprecated）、排程觸發時機、一句話說明。

三、實施優先級與路線圖

3.1 優先級矩陣

| 優先級 | 項目 | 影響 | 工作量 | ROI |

|--------|------|------|--------|-----|

| P0 | 萃取 5 函式到 lib/ 模組 | 消除 500 行重複 | 4h | 極高 |

| P1 | 精簡 run-todoist-agent-team.ps1 | 降至 400 行 | 2h | 高 |

| P2 | 建立 SCRIPTS_MANIFEST.md | 可維護性提升 | 0.5h | 中 |

| P3 | 處理單模式遺孤 | 減少困惑 | 1h | 中 |

| P4 | 新增 Pester 模組測試 | 防回歸 | 3h | 高（長期） |

3.2 漸進式遷移策略（非 Big Bang）

根據技術債管理最佳實踐（vFunction, Atlassian），採用增量現代化方式：

Step 1（P0）：建立 lib/DailyDigest.psm1，先萃取 Write-Log 和 Remove-StderrIfBenign（最小風險）

Step 2（P0）：萃取 Set-FsmState（最高價值），三個腳本改為 Import-Module

Step 3（P1）：重構 run-todoist-agent-team.ps1，將函式外移後驗證 Phase 0-3 完整流程

Step 4（P2-P3）：建立清單 + 標記遺孤

Step 5（P4）：為 lib/ 中 5 個公開函式編寫 Pester 測試

每步完成後執行完整測試套件（912+ tests）確認無回歸。

四、風險評估與緩解

| 風險 | 機率 | 影響 | 緩解 |

|------|------|------|------|

| Import-Module 路徑在排程器中不正確 | 中 | 全面故障 | 使用 $PSScriptRoot 相對路徑 |

| 函式簽名微妙差異導致回歸 | 中 | 部分故障 | 萃取前先做 diff 對齊 |

| Pester 測試環境配置困難 | 低 | 延遲 | 先用 -WhatIf 乾跑驗證 |

| 排程器引用舊腳本路徑 | 低 | 部分故障 | setup-scheduler.ps1 同步更新 |

六、量化預期收益

| 指標 | 現況 | 目標 | 改善幅度 |

|------|------|------|----------|

| 重複程式碼行數 | 500+ | ~50（import 語句） | -90% |

| 最大腳本行數 | 1,727 | ~400 | -77% |

| Bug 修正觸及點 | 3-7 處/函式 | 1 處 | -67~86% |

| 新手理解時間（估） | 2h+（逐一閱讀） | 30min（讀模組+清單） | -75% |

| Pester 可測函式 | 0 | 5+ | 從無到有 |

---

七、結論

PowerShell 編排層是 daily-digest-prompt 專案的「隱形骨架」，支撐著 Phase 編排、狀態機、遙測、告警等核心功能。目前的零模組化狀態導致 500+ 行重複程式碼、3× 維護成本、零單元測試覆蓋。建議以 P0 優先級萃取 5 個共用函式到 lib/DailyDigest.psm1，採用漸進式遷移避免 Big Bang 風險，預期可削減 90% 重複並將最大腳本從 1,727 行降至 400 行。
