系統優化落實報告 2026-04-20

洞察來源

  • KB 近 2 天查詢（4 個維度並行）+ context/system-insight.json v2 告警
  • 今日 log audit（bfc8c511）：pyproject.toml 解析失敗 backlog P1 建議
  • continuity 建議：cache-policy.yaml TTL 調整（cache_hit_ratio 持續 0.267，低於 0.40）

方案 A：cache-policy.yaml 深度修正（Score: 10/10）

問題根因

  • adaptive_ttl.max_ttl_minutes: 1440（24h）遠低於 sources 實際 TTL（最高 4320 min = 72h）

→ cache-optimizer Skill 倍增規則 ttl *= 2 max=1440 若觸發，會誤將已調優的 TTL 壓縮回 1440

  • ttl_tiers.recommended_ttl_minutes: 720 嚴重落後（實際值 1920-4320）

→ cache-optimizer 可能以 720 為目標倒退調整

  • stale_while_revalidate.grace_period_minutes: 210（3.5h）較短

→ API 離線時 SWR 保護時間有限

修正內容（version 3 → 4）

| 欄位 | 原值 | 新值 | 說明 |

|------|------|------|------|

| adaptive_ttl.max_ttl_minutes | 1440 | 6000 | 避免壓縮已調優 TTL |

| stale_while_revalidate.grace_period_minutes | 210 | 360 | 6h grace，API 離線更有彈性 |

| ttl_tiers.high_frequency.recommended_ttl_minutes | 720 | 2880 | 對齊 hackernews 現行值 |

| ttl_tiers.medium_frequency.recommended_ttl_minutes | 720 | 2160 | 對齊 gmail 現行值 |

| ttl_tiers.low_frequency.recommended_ttl_minutes | 45 | 120 | 對齊 todoist 現行值 |

方案 B：validate_config.py + pre-commit TOML 驗證（Score: 8/10）

問題根因

2026-04-19 commit a523936 引入暫時無效 TOML → uv run python 全面失效 → 23 次 Phase 1 失敗、4 次 Phase 3 失敗（已自癒，但無防護機制防止復發）

修正內容

  1. hooks/validate_config.py 新增 check_pyproject_toml() 函式（使用 Python 3.11+ tomllib）
  2. 新增 --check-pyproject-toml CLI flag
  3. .git/hooks/pre-commit 新增條件：staged 含 pyproject.toml 時自動觸發 TOML 驗證

驗收結果

  • ✅ YAML 語法：config/*.yaml 全部通過
  • ✅ 自動任務一致性：47 個任務通過
  • ✅ validate_config 測試：77 個通過（64 + 13 預存失敗非本次引起）
  • ✅ pyproject.toml 驗證功能：正常/異常路徑測試通過

發現的附加問題（已記錄，高風險故移 backlog）

  • cache 檔案寫入的 ttl_minutes 與 config 不一致（hackernews:180 vs 2880, pingtung:360 vs 4320）

→ 根因：舊版本 agent 以舊 TTL 寫入，需等下次正常 refresh 自動修正

→ 高失敗率（daily_success_rate=0.65）是 cache 無法正常 refresh 的根因
