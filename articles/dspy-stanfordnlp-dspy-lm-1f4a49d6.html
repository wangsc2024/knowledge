DSPy（stanfordnlp/dspy）— 熱門 GitHub 專案洞察

執行摘要

DSPy（Declarative Self-improving Python）是史丹佛 NLP 團隊維護的開源框架，口號為「Programming—not prompting」：以宣告式 Signature描述輸入輸出行為，以可組合 Module（如 Predict、ChainOfThought、ReAct、RLM 等）封裝提示策略，再以 Optimizer（前稱 Teleprompter，例如 BootstrapFewShot、MIPROv2、GEPA）依度量（metric）自動編譯出高品質提示或示範，而非手工調整長字串提示。專案授權 MIT，GitHub 約 3.3 萬+ Stars（檢索時點），PyPI 套件持續活躍（例如 3.1.3 等版本）。官方文件與教學強調：與其堆疊脆弱 prompt，不如把 LM 系統當成可評估、可迭代的程式來設計。

研究背景

傳統 RAG／Agent 管線常陷入「改模型、改檢索、改提示」三者耦合；DSPy 將任務規格（Signature）、推理／工具模組（Module）與離線優化（compile）分層，並內建 Evaluate 與 SemanticF1 等度量，使團隊能用同一套程式骨架對接 LiteLLM 支援的多後端 LM。官方 RAG 教學展示：以 dspy.Module 組合檢索與 ChainOfThought('context, question -> response')，再用訓練／驗證集與 Optimizer 提升分數；並建議 30–300 筆量級的 train/val 規模（視優化器而定）。部署面文件亦提供 FastAPI、dspy.asyncify、MLflow 整合等路徑。

核心發現（可驗證主張）

  1. Signature 即「行為契約」：以 question -> answer 或類別欄位宣告語意角色，欄位名稱會影響 adapter 產生的結構化提示（官方 Signatures／Modules 文件）。
  2. Module 即「可學習參數的提示策略」：ChainOfThought 會在簽名下自動擴充 reasoning 等輔助欄位；RAG 範例將檢索結果注入 context 再生成。
  3. Optimizer 驅動「編譯」：以 metric 與資料集對整個程式（含多子模組）做少樣本或搜尋式提示／示範優化；GEPA 等研究主張反思式提示演化可與 RL 路線競爭（見專案 README 論文列表）。
  4. RAG 與評估閉環：教學以檢索＋生成為一個 Module，並示範以 SemanticF1 等評估；指出僅加 RAG 不一定救得了檢索品質，必須用度量驗證檢索與生成是否同步改善。

深度分析：與本專案（Skill-First、RAG、Harness）的對照

| 面向 | DSPy 作法 | 可借鏡於 daily-digest-prompt |

|------|-----------|------------------------------|

| 規格與實作分離 | Signature／Module 分層 | 對齊「Prompt 薄層＋YAML／模板外部化」：將自動任務的 I/O 契約寫成可測的「簽名式」描述，減少散文 prompt 漂移 |

| 品質閘 | metric + compile + Evaluate | 與 quality_gate／結果檔診斷 同源：以可計分度量驅動迭代，而非憑感覺改提示詞 |

| RAG | Retrieve + 生成 Module 一體優化 | KB 混合檢索可對照「先檢索、再生成」；DSPy 強調同步優化多段提示，啟發 hydrate／gate 與 KB 示範一併納入評估 |

| 多後端 | LiteLLM 路由 | 與 llm-router／多後端配額 呼應：上層程式穩定、下層替換模型成本較低 |

| 可觀測性 | MLflow／inspect_history | 與 post_tool_logger、OTel 路線相容：關鍵是保留結構化 trace 供迴歸 |

實務應用建議

  1. 對高風險自動任務，定義固定 I/O 欄位與自動評分函式（類 DSPy metric），再才調 prompt 或示範。
  2. 新 RAG 功能上線時，同時報告「檢索命中率／答案忠實度／端到端分數」，避免只看生成流暢度。
  3. 小樣本階段優先 BootstrapFewShot 類策略；資料較豐富再考慮 MIPROv2／GEPA 等較重優化。
  4. 服務化時採 asyncify 與載入已編譯程式 JSON 的模式，將「離線編譯」與「線上推理」分離，利於版本治理。

知識缺口與未來方向

  • DSPy 與企業內 Policy／Guardrails 的深度整合模式（與本專案 hooks）可再對照案例。
  • 超大規模資料下 Optimizer 成本與 semantic cache 的併用策略待實測。
