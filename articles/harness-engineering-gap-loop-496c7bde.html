一、核心概念：Harness 回饋迴路的四階段閉環

根據 OpenAI Cookbook Self-Evolving Agents 與 Philipp Schmid 的 Agent Harness 研究，現代 Harness Engineering 的回饋迴路由四個階段組成：

1.1 四階段閉環模型

| 階段 | 名稱 | 核心動作 | 產出 |

|------|------|---------|------|

| S1 | Baseline Execution | Agent 在生產環境執行任務 | 原始輸出 + 執行軌跡（trajectories） |

| S2 | Feedback Collection | 人工審查或 LLM-as-Judge 自動評分 | 結構化回饋（thumbs-up/down + 評語） |

| S3 | Eval & Scoring | 多維度評分匯總 | 加權品質分數 + 問題分類 |

| S4 | Prompt/Constraint Refinement | 依回饋調整 prompt 或約束 | 新版本 prompt + 版本審計軌跡 |

關鍵洞見（Philipp Schmid）：「競爭優勢不再是 prompt 本身，而是你的 Harness 所捕獲的軌跡資料。」

1.2 評分策略：Multi-Grader 架構

OpenAI Cookbook 示範的多評分器策略平衡了規則式與語義式檢查：

| 評分器 | 類型 | 用途 |

|--------|------|------|

| 關鍵實體保留 | Python 邏輯 | 確保領域關鍵術語未被遺漏 |

| 長度偏差 | Python 邏輯 | 維持簡潔性紀律 |

| 餘弦相似度 | 文本指標 | 錨定摘要與源內容的相關性 |

| LLM 評審 | 模型評估 | 捕捉細微品質訊號 |

通過閾值：75% 評分器通過 或 平均分 ≥ 0.85 即為「寬鬆通過」。

二、Gap Loop 機制：從失敗到嵌入的閉環

2.1 Gap Loop 五步模型

Gap Loop 是 Harness 系統從失敗中學習的核心機制：


Production Failure → Gap Detection → Gap Classification → Architecture Review → Constraint Embedding
     ↑                                                                              |
     └──────────────────── 回饋閉環 ────────────────────────────────────────────────┘
  1. 失敗偵測：透過結構化日誌（JSONL）與 OpenTelemetry 追蹤異常模式
  2. Gap 分類：依失敗分類學（failure taxonomy）歸類為已知模式或新模式
  3. 影響評估：計算 Gap 對 SLI/SLO 的影響程度
  4. 架構審查：判斷是否需要新約束或調整現有約束
  5. 約束嵌入：將修復方案寫入 Harness 配置（constraint evolution）

2.2 自動修復與升級策略

根據 Algomox 的自癒基礎設施研究，Agent 系統的自動修復分三層：

| 層級 | 策略 | 適用場景 |

|------|------|----------|

| L1 即時修復 | 自動重試 + 回滾 | 已知模式的暫時性故障 |

| L2 適應性修復 | 調整參數 + 替代路徑 | 環境變化導致的持續性問題 |

| L3 結構性修復 | 約束重寫 + 架構調整 | 系統設計層面的根本問題 |

關鍵指標：機器學習回饋迴路分析每次修復動作的四個維度 — 成功率、影響效果、資源效率、非預期後果。

三、補償面（Compensation Surface）自適應

3.1 補償面的概念模型

補償面是 Harness 系統中介入來源（intervention sources）與系統行為之間的映射關係。每個介入點可以被視為多維空間中的一個「面」，系統透過調整面的參數來補償 Agent 行為的偏差。

3.2 三層優先順序


runtime_override > daily_limit > component_flags
  • runtime_override：即時覆寫，用於緊急修復（如 Circuit Breaker 觸發後的降級）
  • daily_limit：每日限額控制，防止資源耗盡（如 Token 預算、API 配額）
  • component_flags：元件旗標，長期行為調控（如功能開關、A/B 測試分組）

3.3 A/B Toggle 機制

結合 Feature Flag 最佳實踐，Harness 的 A/B Toggle 分為兩類用途：

  1. 工程驗證（Engineering Verification）：在生產環境安全測試約束變更的效能影響
  2. 行為分析（Behavioral Analytics）：比較不同約束配置下 Agent 的任務完成率

旗標生命週期管理：

  • 新旗標必須有明確的實驗期限（如 7 天）
  • 根據不活躍指標自動識別待退役的旗標
  • 與問題追蹤系統整合，追蹤清理狀態

四、持續優化的三大模式

4.1 Entropy Management（熵管理）

透過排程任務定期掃描：

  • 配置不一致性
  • 死代碼（Dead Code）
  • 文件漂移（Documentation Drift）

4.2 Dynamic Resource Allocation（動態資源分配）

根據上下文在正確時機載入正確資訊：

  • 按需載入 Skill/模板（非預載）
  • 快取 TTL 預計算（PowerShell Phase 0 模式）
  • Token 預算動態分配

4.3 Regression Detection（回歸偵測）

標記 Agent 行為退化：

  • 基準線效能追蹤（benchmark.yaml fitness_functions）
  • 趨勢分析（entropy-trend.json 時間序列）
  • SLO 燃燒率告警（多視窗模式）

六、對 Daily-Digest-Prompt 系統的優化建議

基於上述研究，結合本專案 B0-B5 Harness 分層架構，提出三項具體優化方向：

6.1 強化 B4-A Gap Loop 的自動分類

目前 harness_gap_tracker.py 偵測 gap 後需人工判斷分類。建議：

  • 引入 failure taxonomy 自動分類（對齊 post_tool_logger.py 的失敗標籤）
  • 分類結果直接映射到 arch-evolution 審查佇列
  • 新增「gap → constraint proposal」自動轉化路徑

6.2 導入 Multi-Grader 品質評分

目前 quality_gate.py 以單一確定性門檻為主。建議：

  • 增加餘弦相似度評分器（錨定任務輸出與預期格式）
  • B2-C LLM 評估層增加「寬鬆通過」邏輯（75% pass 或 avg ≥ 0.85）
  • 評分結果回饋至 scoring.yaml 調整權重

6.3 補償面 A/B Toggle 實驗框架

目前 component_flags 為靜態配置。建議：

  • 為高影響約束變更建立 A/B 實驗期（7 天）
  • 實驗結果寫入 state/compensation-experiment.json
  • 月度 Harness 審查（B5-B）納入實驗結果分析
