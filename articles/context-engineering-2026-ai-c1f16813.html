Context Engineering（上下文工程）— 2026 年 AI Agent 開發核心範式

主題概述

Context Engineering（上下文工程）是 2026 年 AI 領域最重要的範式轉移之一。由 Shopify CEO Tobi Lütke 提出、前 OpenAI 研究員 Andrej Karpathy 背書推廣，其核心主張是：從「如何問問題」（Prompt Engineering）轉向「如何系統性地管理模型接收的所有資訊」。

Karpathy 的經典定義：「Context Engineering 是填充上下文窗口的精緻藝術與科學——在每一步放入恰到好處的資訊。」

根據 LangChain 2025 State of Agent Engineering 報告，57% 的組織已在生產環境部署 AI Agent，但 32% 的品質問題根因並非 LLM 能力不足，而是上下文管理不當。

---

核心概念：Context Engineering vs Prompt Engineering

| 維度 | Prompt Engineering | Context Engineering |

|------|-------------------|--------------------|

| 關注點 | 如何措辭指令 | 如何設計資訊架構 |

| 範圍 | 單一提示語 | 完整資訊生態系統 |

| 資料來源 | 靜態文字 | 多源動態組合（RAG、工具、記憶、歷史） |

| 最佳化目標 | 單次回應品質 | 系統性任務完成率 |

| 類比 | 問一個好問題 | 寫一部完整劇本 |

關鍵洞見：Prompt Engineering 聚焦「問什麼」，Context Engineering 聚焦「模型看到什麼」。在生產級 AI 系統中，模型最終看到的 context window 內容來自多個來源的精心編排。

---

四大核心模式（LangChain 框架）

模式一：Write Context（寫入持久化）

  • Scratchpads（暫存區）：在活躍 session 中，透過工具呼叫或狀態物件將資訊存到上下文窗口外
  • Memories（記憶）：跨 session 生成並儲存洞見，使用反思與綜合機制
  • 實作方式：檔案系統、狀態欄位、資料庫
  • 實例：Claude Code 的 CLAUDE.md 作為程序性記憶

模式二：Select Context（選擇檢索）

  • 每一步拉取相關資訊進入上下文窗口
  • 記憶選擇：使用 embedding + 知識圖譜進行語義檢索
  • 工具選擇：對工具描述套用 RAG，研究顯示可提升 3 倍準確率
  • 知識檢索：結合 grep、AST 解析、embedding 搜尋、re-ranking

模式三：Compress Context（壓縮最佳化）

  • 摘要化：遞迴或階層式方法壓縮 Agent 軌跡
  • 裁剪：硬編碼啟發法移除舊訊息，或訓練上下文修剪器（如 Provence）
  • 實例：Claude Code 在 95% 窗口容量時自動 compact

模式四：Isolate Context（隔離分離）

  • 多 Agent：將任務拆分到具有獨立上下文的子 Agent（Anthropic 驗證此法優於單 Agent）
  • 沙箱化：在隔離環境（E2B、Pyodide）執行程式碼，token 密集物件存放在 LLM 外
  • 狀態 Schema：設計結構化狀態物件，每個 turn 只暴露必要欄位

---

六支柱架構（Weaviate 框架）

1. Agent 作為編排器

Agent 同時是上下文的消費者和架構師，包含四個元件：LLM（推理）、Tools（外部功能）、Memory（短期+長期）、Observation（觀察與推理）。

2. 查詢增強（Query Augmentation）

將自然語言輸入轉換為下游任務最佳格式。Weaviate Query Agent 根據叢集結構知識，將使用者提示轉換為最佳化資料庫查詢。

3. 檢索最佳化

關鍵的分塊（Chunking）權衡：

  • 小塊：精確但缺乏上下文
  • 大塊：上下文豐富但 embedding 變「雜訊化」
  • 需要針對具體用例找到甜蜜點

4. 提示技術作為控制層

從 Chain-of-Thought 到 ReAct 框架，提示作為指導模型如何使用檢索資訊的控制機制。

5. 三層記憶架構

  • 短期記憶：活躍上下文窗口（近期對話、工具輸出、檢索文件）
  • 工作記憶：任務特定的暫存空間
  • 長期記憶：外部向量資料庫（情節性、語義性、程序性資料）

關鍵原則：有效的 Agent 是選擇性的——過濾哪些互動被提升到長期儲存，通常讓模型反思事件並分配重要性分數。維護策略包括定期修剪、重複合併、逐字稿替換為摘要。

6. 工具整合與 MCP

工具編排涵蓋：發現→選擇→參數組裝→反思四步驟。Model Context Protocol（MCP） 作為「AI 的 USB-C」，標準化 Agent 與外部服務的連接。

四大上下文失敗模式

| 失敗模式 | 描述 | 影響 |

|---------|------|------|

| Context Poisoning（中毒） | 幻覺資訊進入上下文後被反覆引用 | 錯誤複合放大 |

| Context Distraction（分心） | 過多歷史記錄淹沒推理能力 | 注意力稀釋 |

| Context Confusion（混淆） | 無關資訊誤導工具選擇 | 任務偏離 |

| Context Clash（衝突） | 矛盾資訊造成決策癱瘓 | 輸出不一致 |

---

2026 年前沿趨勢

1. 階層式記憶架構

分層的短期、工作、長期記憶系統，使模型能在延伸互動中處理和記住大量資訊。

2. Agentic Context Engineering（ACE）

Agent 主動重寫自身上下文——上下文像「自我更新的劇本」，根據模型表現回饋自動演化。

3. 多 Agent 上下文隔離

Anthropic 驗證多 Agent 方法優於單 Agent，但 token 成本可達 15 倍。需要精心設計上下文邊界。

4. 上下文窗口作為系統 RAM

將 context window 視為稀缺資源進行工程管理——不是「越大越好」，而是「放對東西」。

商業影響與數據

  • 投資穩健上下文架構的組織：回應時間改善 50%、輸出品質提升 40%
  • 根本洞見：更大的模型不一定創造最好的 AI 系統，更好的工程才會

與本專案的關聯

本 Daily Digest Prompt 專案本身就是 Context Engineering 的實踐案例：

  • Write Context：CLAUDE.md、context/digest-memory.json 作為跨 session 記憶
  • Select Context：Skill-First 策略、knowledge-query KB 檢索
  • Compress Context：cache/status.json 預計算、快取 TTL 管理
  • Isolate Context：Agent Team 多 Agent 並行，每個子 Agent 獨立上下文窗口
