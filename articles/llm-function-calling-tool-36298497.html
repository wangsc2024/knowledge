LLM Function Calling 與 Tool Use 架構設計完整指南

概述

Function Calling（函式呼叫）是 LLM 應用中最重要的能力擴展機制之一。它讓語言模型能夠「決定」何時呼叫外部工具，並以結構化 JSON 格式輸出函式名稱與參數，由應用層負責實際執行。OpenAI 稱之為「Function Calling」，Anthropic 稱為「Tool Use」，Google 稱為「Function Calling」，但核心機制一致：LLM 不直接執行函式，而是生成描述函式呼叫的結構化資料。這種「決策與執行分離」的設計是整個 Agent 架構的基石。

2026 年，隨著 MCP（Model Context Protocol）、Structured Outputs、Server Tools 等標準的成熟，Tool Calling 已從簡單的 API 擴展演進為完整的 Agent 互操作生態系統。

---

一、核心架構：三層分離模型

1.1 決策層（LLM）

LLM 負責理解使用者意圖，從可用工具列表中選擇適當工具，並產生符合 JSON Schema 的參數。關鍵設計原則：

  • 工具定義是 Prompt 的一部分：每個工具的 name、description、input_schema 都消耗 input tokens。Anthropic 內部測試顯示 58 個工具約消耗 55,000 tokens
  • 工具數量影響選擇準確度：隨著工具數增加，模型選擇正確工具的能力下降
  • 描述品質決定呼叫品質：詳細的描述和範例（input_examples）顯著提升準確率

1.2 路由層（Application）

應用層負責：

  • 解析 LLM 回應中的 tool_use 區塊
  • 驗證參數是否合理（Schema 驗證 + 業務邏輯驗證）
  • 將呼叫分派到對應的執行器
  • 將執行結果封裝為 tool_result 回傳給 LLM

1.3 執行層（Tools/Services）

實際的函式執行、API 呼叫、資料庫查詢等。執行層與 LLM 完全解耦。

二、Agentic Loop（代理迴圈）— 核心執行模式

2.1 基本四步驟（Claude Tool Use 官方流程）

  1. 提供工具定義與使用者提示：在 API 請求中定義工具的 name、description、input_schema
  2. Claude 決定使用工具：回應的 stop_reason 為 tool_use，包含 tool_use content block
  3. 執行工具並回傳結果：從 Claude 請求中提取工具名稱和輸入，執行後以 tool_result content block 回傳
  4. Claude 利用結果產生回應：Claude 分析工具結果，生成最終回覆

2.2 進階六步驟流程（2026 生產環境）

現代生產環境在基本四步驟上增加：

  • Step 0：工具發現（Tool Discovery）：透過 MCP 或向量存儲查詢工具注冊表，根據使用者意圖動態載入相關工具定義，避免 Context Window 飽和
  • Step 5：結果評估與迴圈控制：評估是否需要進一步的工具呼叫（多步驟推理），或已可產生最終回應

2.3 Agentic Loop 實作範式


def agentic_loop(user_message, tools, max_iterations=10):
    messages = [{"role": "user", "content": user_message}]
    
    for _ in range(max_iterations):
        response = client.messages.create(
            model="claude-opus-4-6",
            max_tokens=4096,
            tools=tools,
            messages=messages
        )
        
        # 檢查是否需要使用工具
        if response.stop_reason == "end_turn":
            return extract_text(response)  # 完成
        
        # 處理工具呼叫
        messages.append({"role": "assistant", "content": response.content})
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": str(result)
                })
        messages.append({"role": "user", "content": tool_results})
    
    return "達到最大迭代次數"

三、跨平台 API 規範對比

3.1 工具定義格式

| 特性 | OpenAI | Anthropic (Claude) | Google (Gemini) |

|------|--------|-------------------|----------------|

| 參數名 | functions / tools | tools | tools |

| Schema 格式 | JSON Schema | JSON Schema (input_schema) | JSON Schema |

| 停止原因 | tool_calls | tool_use | TOOL_CALL |

| 回傳格式 | tool_calls 陣列 | tool_use content block | functionCall |

| 結果回傳 | tool role message | tool_result content block | functionResponse |

| 並行呼叫 | 支援 | 支援 | 支援 |

| 強制使用 | tool_choice: required | tool_choice: {type: "any"} | REQUIRED |

| 結構化輸出 | strict: true | strict: true (Structured Outputs) | N/A |

3.2 Anthropic 特有功能

  • Server Tools：在 Anthropic 伺服器端執行的工具（如 web_search、web_fetch），無需客戶端實作
  • MCP Connector：直接從 Messages API 連接遠端 MCP 伺服器，無需建構 MCP 客戶端
  • pause_turn 機制：Server-side loop 達到 10 次迭代上限時回傳，客戶端需續傳對話讓 Claude 完成處理
  • Structured Outputs：strict: true 保證工具輸入完全符合 Schema，消除型別不匹配或缺失欄位問題

四、工具定義最佳實踐

4.1 描述的藝術


{
  "name": "search_products",
  "description": "Search for products in the e-commerce catalog. Use this when the user wants to find products by keyword, category, or price range. Returns a list of matching products with name, price, and availability.",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "Search keywords. Can include product name, brand, or category. Example: 'wireless bluetooth headphones'"
      },
      "max_price": {
        "type": "number",
        "description": "Maximum price filter in USD. Optional."
      }
    },
    "required": ["query"]
  }
}

關鍵原則：

  • 描述要說明「何時使用」而非僅「做什麼」
  • 參數描述要包含格式範例
  • 使用 input_examples 提供具體輸入範例（Claude 特有）
  • 必要參數用 required，可選參數明確標註

4.2 Context Window 優化

兩階段選擇法：

  1. 決策階段：模型只看工具名稱和短描述
  2. 參數生成階段：僅載入被選中工具的完整 Schema

這種方法大幅減少 token 消耗，在工具數 > 20 時效果顯著。

4.3 工具數量控制

  • 理想範圍：5-15 個工具
  • 上限建議：40-50 個（超過後準確率明顯下降）
  • 解決方案：Tool Discovery + 動態載入（MCP 模式）

五、錯誤處理與韌性設計

5.1 錯誤回傳模式


# Anthropic Claude - 錯誤回傳
tool_result = {
    "type": "tool_result",
    "tool_use_id": tool_use_id,
    "content": "Error: Product ID 'xyz' not found in database",
    "is_error": True  # Claude 特有，幫助模型理解失敗原因
}

5.2 回退策略層級

  1. 重試：網路超時或暫時性錯誤 → 自動重試 1-2 次
  2. 澄清：參數不明確 → 呼叫 Clarify 類工具向使用者提問
  3. 降級：工具不可用 → 使用模型內建知識回答（標註為「基於一般知識」）
  4. 拒絕：安全檢查未通過 → 明確拒絕並說明原因

5.3 迴圈保護

  • 最大迭代次數：設定 max_iterations（建議 10-15）
  • 相同工具重複檢測：連續 3 次呼叫同一工具且參數相同 → 中斷迴圈
  • Token 預算控制：累計 token 接近上限時強制結束

六、安全設計

6.1 三大威脅模型

| 威脅 | 說明 | 防禦 |

|------|------|------|

| Prompt Injection | 使用者輸入中注入惡意指令 | 輸入清理 + 模式比對 + 隔離執行 |

| Tool Misuse | LLM 被誘導呼叫危險工具 | 白名單 + 權限分級 + 人工確認 |

| Data Exfiltration | 透過工具將敏感資料外洩 | 輸出過濾 + 資料脫敏 + 審計日誌 |

6.2 防禦策略

動作空間限制：使用顯式條件邏輯分派工具呼叫，永遠不要使用 eval() 動態執行


# 正確：顯式分派
def execute_tool(name, params):
    if name == "search":
        return search_handler(params)
    elif name == "get_details":
        return details_handler(params)
    else:
        raise ValueError(f"Unknown tool: {name}")

關鍵操作人工審核：刪除、付款、發送郵件等高風險操作，需使用者二次確認

輸入消毒：


suspicious_patterns = [
    "ignore previous instructions",
    "system prompt",
    "act as",
    "ignore all previous commands"
]

七、MCP（Model Context Protocol）整合

7.1 MCP 的角色

MCP 是 Anthropic 提出的開放協議，標準化 LLM 應用與外部服務的互動方式。在 Tool Calling 架構中，MCP 解決的核心問題是 工具的動態發現與標準化互操作。

7.2 MCP 工具轉換

MCP 工具定義使用 inputSchema，轉為 Claude 格式只需重命名為 input_schema：


async def get_claude_tools(mcp_session):
    mcp_tools = await mcp_session.list_tools()
    return [
        {
            "name": tool.name,
            "description": tool.description or "",
            "input_schema": tool.inputSchema
        }
        for tool in mcp_tools.tools
    ]

7.3 MCP vs Function Calling 選擇指引

| 場景 | 推薦 | 原因 |

|------|------|------|

| 固定工具集、嚴格驗證 | OpenAI Function Calling | Schema 強制 + 型別安全 |

| 動態工具發現、混合推理 | MCP + Claude Tool Use | 彈性高 + inline chaining |

| 跨 LLM 相容性 | MCP | 協議無關性 |

---

八、Structured Outputs vs Function Calling

兩者常混淆，但用途不同：

| 特性 | Function Calling | Structured Outputs |

|------|-----------------|-------------------|

| 用途 | 呼叫外部工具 | 結構化模型回應 |

| 觸發 | stop_reason: tool_use | stop_reason: end_turn |

| 輸出 | 工具名稱 + 參數 JSON | 符合 Schema 的 JSON |

| 執行 | 需要應用層執行 | 直接作為回應 |

互補使用：可在 Function Calling 中啟用 strict: true，確保工具參數符合 Schema（Anthropic Structured Outputs for Tools）。

九、生產系統設計模式

9.1 DAG 執行模式

進階 Agent 使用有向無環圖（DAG）排程工具呼叫：

  • Planner 產生任務 DAG
  • Task Fetching Unit 監控依賴關係
  • 依賴滿足時排程並行執行
  • 可達到 3.6x 加速（相比序列執行）

9.2 可觀測性

  • 每次工具呼叫記錄 trace（工具名、參數、耗時、成功/失敗）
  • 使用 Langfuse 等觀測平台追蹤完整 Agent 執行鏈
  • 定義 SLI/SLO：工具呼叫成功率 > 95%、P99 延遲 < 5s

9.3 漸進式部署策略

Martin Fowler 文章建議：「從低風險操作開始啟用 Function Calling，逐步擴展至關鍵操作，同時隨著安全機制的成熟不斷完善。」

十、與 Daily-Digest 系統的關聯

本專案的 Claude Code CLI 架構本身就是 Tool Use 的實踐：

  • Claude Agent 透過 allowed-tools 定義可用工具（Read, Write, Bash, WebSearch...）
  • Hook 系統（pre_bash_guard, pre_write_guard）相當於「工具執行前的安全驗證層」
  • MCP 整合（屏東新聞 MCP 服務）展示了動態工具發現的實際應用
  • Agentic Loop 的迴圈控制（max_iterations、LoopDetector）直接對應本研究的韌性設計原則
