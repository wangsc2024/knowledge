一、專案概述

Agency-Agents（msitarzewski/agency-agents）是 2026 年 3 月 GitHub Trending 連續兩週榜首的開源 AI Agent 框架，由 Michael Sitarzewski 創建。專案定位為「一個完整的 AI 機構，就在你的指尖」，核心理念是將 AI Agent 組織成一間虛擬公司，每個 Agent 都是具備獨立人格、流程與可量化交付物的專業角色。

二、核心設計理念：Agent-as-Company

2.1 問題定義

傳統 AI 編程助手（Copilot、Claude Code、Cursor）使用通用 LLM 產出程式碼時存在三大痛點：

  1. 泛化幻覺：LLM 缺乏領域聚焦，產出的程式碼常混合不同範式 [YUV.AI Blog, 2026]
  2. 一致性缺失：團隊成員使用相同工具卻產出風格迥異的程式碼
  3. 品質不穩定：無明確的交付標準與驗收指標

2.2 解法：結構化角色專業化

Agency-Agents 的核心假設是：縮窄 LLM 的上下文範圍到特定領域，能大幅降低幻覺並強制最佳實踐 [YUV.AI Blog, 2026]。

每個 Agent 不是一個「prompt」，而是一個完整的角色定義，包含：

  • 身份與人格（Identity）：Agent 的思考方式與溝通風格
  • 核心使命（Mission）：主要職責範圍
  • 不可違反規則（Critical Rules）：領域特定的最佳實踐
  • 技術交付物（Deliverables）：可量化的輸出成果
  • 成功指標（Success Metrics）：績效衡量標準

三、架構分析

3.1 Agent 定義格式

每個 Agent 以 Markdown + YAML frontmatter 格式定義：


---
name: backend-architect
description: 後端架構設計專家
color: "#4A90D9"
emoji: "🏗️"
vibe: "methodical and precise"
services: [PostgreSQL, Redis, Docker]
---

# Identity & Memory
你是後端架構師，專注於可擴展系統設計...

# Core Mission
設計並實現健壯的後端架構...

# Critical Rules
- 所有 API 必須有版本控制
- 資料庫查詢必須有索引
...

# Technical Deliverables
- REST API 設計文件
- 資料庫 Schema
...

# Success Metrics
- API 回應時間 < 200ms
- 測試覆蓋率 > 80%

3.3 轉換管線（convert.sh）

convert.sh 腳本實現四種轉換策略，將 Markdown Agent 定義適配到不同平台：

| 策略 | 目標平台 | 方式 |

|------|---------|------|

| Identity Copy | Claude Code、Copilot、OpenCode、Cursor、Qwen | 直接複製 .md 檔案 |

| Prefix/Namespace | Antigravity、Gemini CLI | 加 agency- 前綴包裝 |

| Semantic Split | OpenClaw | 拆分為 SOUL.md / IDENTITY.md / AGENTS.md 三件套 |

| Consolidation | Aider、Windsurf | 合併所有 Agent 為單一 CONVENTIONS.md |

3.4 安裝與啟動


# 步驟 1：轉換 Agent 定義為平台格式
./scripts/convert.sh

# 步驟 2：安裝到已偵測的平台
./scripts/install.sh          # 互動式（checkbox UI）
./scripts/install.sh --tool claude-code  # 指定平台

安裝器自動偵測本機已安裝的工具（檢查 ~/.claude/、~/.cursor/ 等目錄），提供三種模式：互動式、指定工具、自動偵測。

平台整合路徑：

  • Claude Code / Copilot → ~/.claude/agents/、~/.github/agents/（全域）
  • Cursor → .cursor/rules/（專案級）
  • Aider → CONVENTIONS.md（專案級，自動載入）
  • OpenClaw → ~/.openclaw/agency-agents/（需 openclaw gateway restart）

四、核心發現（已驗證）

發現 1：Markdown-as-Agent-Definition 成為事實標準 ✅

Agency-Agents、obra/superpowers、garrytan/gstack 三個 2026 年 3 月最熱門 Agent 框架都採用 Markdown + YAML frontmatter 作為 Agent 定義格式 [ShareUHack Weekly 2026-03-18]。這印證了 Agent 定義正從程式碼驅動轉向文件驅動——與本專案（Daily-Digest-Prompt）的「文件驅動架構」（ADR-002）高度一致。

佐證來源：

  • Agency-Agents：147 個 .md 檔案 [GitHub README, 2026]
  • obra/superpowers：SKILL.md 格式 [GitHub, 92K Stars]
  • gstack：Claude Code 配置 [Garry Tan, 23K Stars]

發現 2：Agent 角色專業化顯著降低幻覺 ✅

多個來源證實，將 LLM 上下文縮窄到特定領域後，產出品質大幅提升 [YUV.AI Blog, 2026]。Backend Architect Agent 產出的 REST API 設計包含正確的 HTTP 狀態碼、資料庫 Schema、認證/限流考量與效能優化——這些在通用模式下經常缺失。

發現 3：「公司組織」比喻有效解決 Agent 協作問題 ✅

將 Agent 組織為 12 個部門、每個角色有明確職責邊界的做法，解決了多 Agent 協作中常見的「角色模糊」問題。Nexus Spatial Discovery Exercise 展示了 8 個 Agent 同時協作（API 設計 + 安全審查 + 資料庫優化）的能力 [YUV.AI Blog, 2026]。

六、對 Daily-Digest-Prompt 的借鑑

6.1 Agent 定義格式對齊

本專案的 SKILL.md frontmatter（name/version/description/allowed-tools/triggers）與 Agency-Agents 的 frontmatter（name/description/color/emoji/vibe/services）在結構上高度相似。可考慮：

  • 新增 vibe 或 communication_style 欄位，讓 Agent 行為更一致
  • 新增 success_metrics 段落，量化每個 Skill 的預期產出

6.2 部門化思維

本專案目前的 Agent 角色（fetcher/analyst/assembler/auditor）已有分工，但缺乏 Agency-Agents 的「部門」層級。可參考其 12 部門架構，將 27 個 Skill 依功能分組管理。

6.3 多平台轉換管線

Agency-Agents 的 convert.sh 四策略轉換管線（Identity Copy / Prefix / Semantic Split / Consolidation）是跨平台 Skill 復用的好範本。本專案若需支援 Cursor 或其他 IDE，可參考此設計。

---

七、知識缺口與未來方向

  1. Agent 間通訊協議：目前 Agency-Agents 的多 Agent 協作依賴 IDE 的 context 窗口拼接，缺乏正式的 Agent 間通訊機制（如 MCP A2A）
  2. 品質度量：Success Metrics 定義在 Markdown 中但無自動化驗證機制
  3. 動態角色切換：目前需手動啟動 Agent，缺乏根據任務自動路由到最佳 Agent 的機制
