一、Policy-as-Code 在 Agent 約束管理中的應用

1.1 為何需要 Policy-as-Code

2026 年統計顯示，81% 的 AI Agent 已進入生產運行，但僅 14.4% 取得完整安全審批，88% 的組織曾發生 AI Agent 安全事件。這個落差凸顯出：約束不能靠 Prompt 自律，必須有機器強制的驗證層。

Policy-as-Code 將治理規則從自然語言轉化為可執行、可版本化、可測試的程式碼，確保每一次 Agent 操作都經過自動化驗證。

1.2 OPA vs Cedar：兩大 Policy Engine 比較

| 維度 | OPA (Open Policy Agent) | Cedar (AWS) |

|------|------------------------|-------------|

| 語言 | Rego（邏輯式） | Cedar（宣告式） |

| 效能 | 毫秒級 | 亞毫秒級（Rust 實作，40-60x 快於 Rego） |

| 驗證 | 單元測試 + 整合測試 | 形式化驗證（數學證明合規性） |

| 適用場景 | Kubernetes 准入控制、API 閘道 | 細粒度授權、合規自動化 |

| Agent 整合 | MCP Gateway + OPA 架構成熟 | EU AI Act 合規自動化首選 |

關鍵洞見：Cedar 的形式化驗證能力（Automated Reasoning）允許企業向審計方數學證明合規性，這是 OPA 無法達到的保證等級。

1.3 MCP + OPA 閘道架構

來自 InfoQ 的生產案例展示了一個完整的 Agent 約束閘道：


Agent 請求 → MCP 路由（工具發現）→ OPA 授權（政策驗證）→ 佇列 → 隔離執行 → 可觀測性

核心設計原則：

  • Agent 被視為不受信任的請求者，每次操作都必須通過閘道驗證
  • MCP 負責「哪些工具存在」（discovery），OPA 負責「誰在什麼條件下可用哪些工具」（authorization）
  • 分離關注點：政策變更無需重新訓練 Agent

OPA 同時強制多重防護：

  1. RBAC：sre-bot 有完整存取權；deploy-bot 限制為非生產環境
  2. 完整性：計畫雜湊必須匹配已註冊的產出物
  3. 安全性：以 -destroy.plan 結尾的計畫被阻擋
  4. 變更管理：部署僅允許在週一至週五 09:00-17:00 UTC

SLO 目標：

  • 政策決策：< 100ms
  • Runner 啟動：< 2s（dev）/ < 5s（staging）
  • 被拒絕操作：≤ 2%
  • 沙箱清理：< 30s

二、LLM Guardrails 工程化：NeMo Guardrails 與 Guardrails AI

2.1 五類護欄（Rails）

NVIDIA NeMo Guardrails 定義了五類護欄：

  1. Input Rails：輸入驗證（注入偵測、PII 過濾）
  2. Dialog Rails：對話流程控制（主題引導、禁止話題）
  3. Retrieval Rails：RAG 檢索結果過濾
  4. Output Rails：輸出內容審核（事實查核、毒性評分）
  5. Execution Rails：工具調用約束（最小權限、參數驗證）

2.2 NeMo vs Guardrails AI 選擇指南

| 需求 | 推薦框架 |

|------|----------|

| 對話流程控制、主題引導 | NeMo Guardrails（Colang 語言） |

| 可組合的輸出驗證 | Guardrails AI（Hub 生態） |

| 結構化資料抽取驗證 | Guardrails AI |

| 快速覆蓋 80% 常見需求 | NeMo Guardrails（配置驅動） |

| 兩者結合 | NeMo 處理對話層 + Guardrails AI 處理 I/O 驗證 |

2.3 生產環境四層防禦架構


Layer 1（輸入）→ Layer 2（模型）→ Layer 3（輸出）→ Layer 4（監控）
  • Layer 1：Regex 注入偵測 + PII 偵測（Microsoft Presidio）+ Schema 驗證 + 長度檢查
  • Layer 2：安全導向 System Prompt + 保守 Temperature 設定
  • Layer 3：毒性評分 + PII 遮蔽 + 事實查核斷言 + 自訂領域驗證器
  • Layer 4：即時指標追蹤 + 自動斷路器 + 人工審查佇列

每層增加 10-50ms 延遲，累計約 100ms，對聊天應用可接受。

生產案例：10 萬次請求中攔截 847 個惡意輸入、標記 89 個問題輸出，達成 95% 安全覆蓋率。誤報率從 12% 調降至 3%。

三、約束驗證自動化模式

3.1 混合驗證策略

Harness Engineering 的核心是確定性工具 + AI 驅動驗證的混合：

  • 確定性層：Linter、型別檢查、Schema 驗證、單元測試
  • AI 層：Agent 互審（Agent reviews Agent's output）、LLM-based 語義檢查
  • 迴路：失敗原因回饋給模型重試（closed-loop verification）

LangChain Deep Agents 案例：透過系統性失敗追蹤，GPT-5.2-Codex 分數從 52.8% 提升至 66.5%，方法是「追蹤失敗 → 聚類錯誤模式 → 將修正回饋至 Harness」，無需模型重訓。

3.2 一次性修復轉化為永久約束

「Turn one-off fixes into reusable constraints (e.g., new lint rules or sub-agents).」

這是 Harness Engineering 的核心原則：每次生產失敗不只是修復問題，而是將修復轉化為新的約束規則，形成增強迴路。

3.3 短暫執行器（Ephemeral Runners）模式

隔離執行是約束驗證的物理保障：

  1. 產生唯一命名空間（run-UUID）
  2. 在隔離環境中執行基礎設施變更
  3. 無論成敗，一律銷毀命名空間

此模式限制爆炸半徑，防止憑證洩漏或孤立資源。

3.4 斷路器監控

  • 封鎖率告警：被封鎖請求超過 5% 時觸發
  • 警告率告警：警告超過 15% 時標記
  • 時間窗口指標：15 分鐘滑動視窗計算比率
  • 嚴重性路由：分數 ≥ 0.8 自動排入人工審查佇列

四、對本專案 Harness 框架的映射與優化建議

4.1 現有架構對應

| 本專案元件 | 業界對應 | 成熟度 |

|-----------|---------|--------|

| B0-E risk_tiers | OPA RBAC 政策 | ★★★ 已建立 |

| B1-A constraint_evolve | Policy-as-Code 演化 | ★★★ 已建立 |

| B2-A quality_gate deterministic | NeMo Input/Output Rails | ★★★ 已建立 |

| B2-C validate_llm | AI 驗證層 | ★★☆ 旗標控制中 |

| B4-A harness_gap_tracker | 一次性修復→永久約束 | ★★★ 已建立 |

| hooks/pre_bash_guard | Input Rails（指令層） | ★★★ 已建立 |

| hooks/pre_write_guard | Output Rails（檔案層） | ★★★ 已建立 |

4.2 可借鑑的優化方向

  1. 形式化驗證探索：Cedar 的 Automated Reasoning 理念可啟發 constraint_evolve.py 在約束提案階段加入更嚴格的邏輯驗證（非僅人工審查）
  2. SLO 化護欄指標：將 Hook 攔截率、品質門檻通過率設為可追蹤 SLO（如政策決策 < 100ms、封鎖率 ≤ 2%）
  3. 短暫執行器模式：Phase 2 自動任務可考慮更強的隔離（目前依賴 Hook 攔截，可加入命名空間式隔離）
  4. 誤報率追蹤：Hook 系統已有日誌，可新增誤報率指標（false positive rate），持續調降至 < 5%
  5. MCP + Policy 分離：目前 Skill 路由與約束驗證耦合在 routing.yaml，可考慮分離為獨立 policy layer

---

五、關鍵結論

  1. 約束必須機器強制：81% Agent 已投產但僅 14.4% 有完整安全審批 — Prompt 自律不夠
  2. 混合驗證是最佳實踐：確定性規則 + AI 語義檢查，兩者缺一不可
  3. Cedar > OPA 在合規場景：形式化驗證提供數學等級保證
  4. 四層防禦是生產標準：Input → Model → Output → Monitoring
  5. 每次失敗都是約束來源：Gap Loop → 永久規則的閉環是 Harness 的核心價值
  6. 護欄效能可量化：95% 安全覆蓋率、誤報率 3%、封鎖率 ≤ 2% 是可達標的 SLO
