Claude Code Plugin 開發工具鏈與 7 組件架構模型

主題概述

Claude Code 的 Plugin 系統在 2026 年初已從單純的 Slash Command 擴充演進為完整的 7 組件架構模型。每個 Plugin 可包含 Commands、Agents、Skills、Hooks、MCP Servers、Settings 和 Manifest 七大組件，透過 .claude-plugin/plugin.json 統一宣告。官方開源了 14 個 Plugin，涵蓋從安全治理（security-guidance）到全流程功能開發（feature-dev）的完整光譜。本文深入解析 Plugin 系統的架構設計、開發工具鏈、自動化規則引擎，以及從 14 個官方 Plugin 中萃取的設計模式。

一、Plugin 7 組件架構模型

1.1 標準目錄結構


plugin-name/
  .claude-plugin/
    plugin.json          # Plugin 宣告清單（名稱、版本、描述、組件列表）
  commands/              # Slash Commands（Markdown 格式，YAML frontmatter）
  agents/                # 自治 Agent 定義（Markdown + YAML frontmatter）
  skills/                # Agent Skills（SKILL.md + 漸進式資源揭露）
  hooks/                 # 事件處理器（Shell/Python 腳本或 Prompt-based）
  .mcp.json              # MCP 伺服器配置（stdio/SSE/HTTP/WebSocket）
  README.md              # Plugin 文件

1.2 七大組件職責

| 組件 | 格式 | 載入時機 | 職責 |

|------|------|---------|------|

| Commands | Markdown + YAML frontmatter | 使用者鍵入 / 觸發 | 封裝可重複工作流（如 /commit、/code-review） |

| Agents | Markdown + YAML frontmatter | 由 Command 或 Claude 自動啟動 | 自治執行專門任務（探索、架構、審查） |

| Skills | SKILL.md + resources/ | 觸發關鍵字匹配時載入 | 領域知識與指引（漸進式揭露：metadata → SKILL.md → references） |

| Hooks | Shell 腳本或 Prompt-based | 事件驅動（PreToolUse、PostToolUse、Stop 等） | 策略強制、驗證、自動化 |

| MCP Servers | .mcp.json（JSON） | 啟動時連線 | 外部服務整合（資料庫、API、第三方工具） |

| Settings | .claude/plugin-name.local.md | 會話開始時讀取 | 每專案配置（YAML frontmatter + Markdown body） |

| Manifest | plugin.json | 安裝/掃描時解析 | 元資料宣告（名稱、版本、相容性、組件清單） |

1.3 漸進式揭露三層機制

Plugin 的 Skill 組件採用三層揭露策略以控制 Token 消耗：

  1. Metadata 層（永遠載入）：Skill 名稱、觸發關鍵字、簡短描述（< 100 tokens）
  2. Core SKILL.md 層（觸發時載入）：核心 API 參考（約 1,500-2,000 字）
  3. References/Examples 層（按需載入）：詳細指南、模式、可運行程式碼

這種設計使 Plugin 系統的 Token 消耗控制在可管理範圍內，即使安裝多個 Plugin 也不會撐爆 context window。

2.2 四種架構模式

從 14 個官方 Plugin 中，可以歸納出四種設計模式：

模式 A：多 Agent 扇出（Fan-out）

  • 代表：code-review（5 Agent）、pr-review-toolkit（6 Agent）、feature-dev（3 Agent）
  • 模式：1 個 Command 同時啟動 N 個專門 Agent 並行工作
  • 特點：每個 Agent 聚焦單一面向，結果由 Command 彙整
  • code-review 的 5 路 Agent：CLAUDE.md 合規、Bug 偵測、歷史脈絡、PR 歷史、程式碼評論

模式 B：事件驅動策略（Event-driven Policy）

  • 代表：security-guidance、hookify、explanatory-output-style
  • 模式：Hook 監聽特定事件，執行策略邏輯
  • security-guidance 監控 9 種安全模式：命令注入、XSS、eval、危險 HTML、pickle 反序列化、os.system

模式 C：結構化工作流（Structured Workflow）

  • 代表：feature-dev（7 階段）、plugin-dev（8 階段）
  • 模式：線性但可互動的多階段流程，每階段可暫停等待使用者輸入
  • feature-dev 7 階段：Discovery → Codebase Exploration → Clarifying Questions → Architecture Design → Implementation → Quality Review → Completion

模式 D：自指涉迴圈（Self-referential Loop）

  • 代表：ralph-wiggum
  • 模式：Stop Hook 攔截退出，持續迭代直到任務完成
  • 實現 Agent 的持續自主改進能力

三、Hookify：自動化規則引擎

3.1 核心創新

Hookify 是最具創新性的官方 Plugin，它將 Hook 規則的建立從「手動編輯 JSON」簡化為「自然語言描述」：


/hookify Don't use console.log in TypeScript files

這會自動產生 .claude/hookify.warn-debug-code.local.md 規則檔案。

3.2 規則格式


---
name: block-dangerous-rm
enabled: true
event: bash
pattern: rm\s+-rf
action: block
---

⚠️ **Dangerous rm command detected!**

3.3 規則能力矩陣

| 面向 | 支援 |

|------|------|

| 事件類型 | bash、file、stop、prompt、all |

| 動作 | warn（警告但允許）、block（阻止執行） |

| 模式匹配 | Python regex 語法 |

| 多條件 | conditions 陣列（AND 邏輯）|

| 對話分析 | /hookify（無參數）自動分析近期對話找出問題模式 |

| 即時生效 | 無需重啟，下次工具呼叫立即生效 |

3.4 與 Hook 原生 API 的差異

Hookify 建立在 Claude Code Hook 原生事件系統之上（PreToolUse、PostToolUse、Stop、SessionStart 等 17 個事件），但提供了更高層的抽象：

  • 原生 Hook：寫 Shell/Python 腳本 + 手動編輯 settings.json
  • Hookify：自然語言描述 → 自動生成 Markdown 規則檔 → Regex 匹配引擎

四、Plugin-Dev：8 階段建構工作流

4.1 開發工具鏈架構

Plugin-Dev 是官方提供的「建構 Plugin 的 Plugin」，包含 7 個專門 Skill：

  1. Hook Development -- Hook API 與事件驅動自動化
  2. MCP Integration -- Model Context Protocol 伺服器整合
  3. Plugin Structure -- 目錄組織與 plugin.json 配置
  4. Plugin Settings -- .claude/plugin-name.local.md 配置模式
  5. Command Development -- Slash Command 建構與 frontmatter
  6. Agent Development -- 自治 Agent 建構與 AI 輔助生成
  7. Skill Development -- Skill 建構與漸進式揭露設計

4.2 `/plugin-dev:create-plugin` 8 階段流程

| 階段 | 名稱 | 動作 |

|------|------|------|

| 1 | Discovery | 理解 Plugin 目的與需求 |

| 2 | Component Planning | 決定需要哪些組件 |

| 3 | Detailed Design | 規格化每個組件，解決模糊點 |

| 4 | Structure Creation | 建立目錄結構與 Manifest |

| 5 | Component Implementation | 使用 AI 輔助 Agent 實作各組件 |

| 6 | Validation | 執行 plugin-validator 與各組件檢查 |

| 7 | Testing | 在 Claude Code 中驗證功能 |

| 8 | Documentation | 完成 README 與發佈準備 |

4.3 AI 輔助 Agent 生成

Plugin-Dev 內建 agent-creator Agent，能根據描述自動生成 Agent 定義：

  • 分析需求 → 產生 YAML frontmatter（name、description、model、allowed-tools）
  • 自動寫入系統 prompt（含 區塊用於可靠觸發）
  • validate-agent.sh 驗證產出格式正確性

五、Feature-Dev：7 階段功能開發典範

5.1 核心理念

Feature-Dev Plugin 體現了「理解優先、實作其次」的開發哲學：

  1. Discovery：釐清需求與限制
  2. Codebase Exploration：2-3 個 code-explorer Agent 並行探索程式碼
  3. Clarifying Questions：識別所有模糊點並等待使用者回答
  4. Architecture Design：2-3 個 code-architect Agent 設計不同方案
  5. Implementation：經使用者批准後才開始實作
  6. Quality Review：3 個 code-reviewer Agent 並行審查（簡潔性、正確性、慣例）
  7. Completion：收尾與文件

5.2 多 Agent 協作模式

Feature-Dev 展示了 Claude Code 的核心並行能力：

  • Phase 2：同時啟動 2-3 個探索 Agent（各探索不同面向）
  • Phase 4：同時啟動 2-3 個架構 Agent（各提出不同方案）
  • Phase 6：同時啟動 3 個審查 Agent（各審查不同面向）

六、最新版本更新亮點（v2.1.47 -- v2.1.63）

6.1 Plugin 系統增強

| 版本 | 更新 |

|------|------|

| v2.1.63 | 新增 /simplify 和 /batch 內建指令；HTTP Hooks 支援（POST JSON → URL） |

| v2.1.51 | Plugin 商城 Git timeout 從 30s 提升至 120s；支援自訂 npm registry 安裝 Plugin |

| v2.1.50 | Agent 定義支援 isolation: worktree（宣告式工作樹隔離）；Plugin 可附帶 settings.json |

| v2.1.49 | --worktree 啟動旗標；Agent 定義支援 background: true；Plugin 可附帶設定檔 |

| v2.1.47 | ConfigChange Hook 事件（企業安全審計）；Managed Settings 支援 Windows Registry |

6.2 記憶體優化

v2.1.47 至 v2.1.63 之間大量記憶體洩漏修復，包括：

  • Agent Teams 完成的 Teammate 任務未回收
  • MCP Server 快取重連時未清理
  • 長時間 Session 的 Git Root 偵測快取無限增長
  • Tree-sitter Parser WASM 記憶體無法收縮
  • JSON 解析快取無限增長

6.3 HTTP Hooks（v2.1.63 新功能）

新增 HTTP Hooks 機制，可將 Hook 事件以 JSON POST 發送到 URL 並接收 JSON 回應，替代傳統的 Shell 命令執行方式。這使得 Hook 邏輯可以託管在遠端服務，適合企業級集中式策略管理。
