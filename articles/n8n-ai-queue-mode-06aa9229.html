一、專案概述

n8n（讀作 "nodemation"）是 2026 年最受歡迎的開源工作流自動化平台之一，以 177K Stars 穩居 GitHub 自動化領域前列。由 Jan Oberhauser 於 2019 年創建，n8n 採用 fair-code 授權模式（Sustainable Use License），在開源與商業可持續性之間取得獨特平衡。

核心定位

  • 技術團隊的自動化平台：結合 no-code 視覺化拖拉 + code-first 彈性（JavaScript/Python 節點）
  • 自託管優先：所有工作流資料留在組織基礎設施內，滿足資料主權需求
  • 原生 AI 能力：2026 年全面整合 LangChain，支援 6 種 AI Agent 架構

二、架構設計亮點

2.1 前後端分離架構

n8n 採用經典的前後端分離設計：

| 層級 | 技術 | 職責 |

|------|------|------|

| 前端（Visual Editor） | Vue.js | 視覺化工作流設計，將流程轉為 JSON 定義 |

| 後端（Execution Engine） | Node.js + TypeScript | 載入 JSON 定義，順序執行節點，處理資料流轉 |

| 資料層 | SQLite（開發）/ PostgreSQL（生產） | 工作流定義、憑證、執行日誌、使用者資料 |

架構洞察：工作流定義以 JSON 格式儲存，實現了「定義」與「執行」的完全解耦。這與聲明式架構（Declarative Architecture）的核心思想一致 — 使用者描述「做什麼」，引擎負責「怎麼做」。

2.2 節點系統（Node System）

n8n 的核心抽象是「節點」，分為三類：

  1. 觸發節點（Trigger Nodes）：Webhook、排程器、應用事件（如新郵件）
  2. 動作節點（Action Nodes）：HTTP 請求、資料轉換、API 呼叫（500+ 服務）
  3. 核心節點（Core Nodes）：Code、Filter、Switch、Merge 等流程控制

每個節點接收上游資料、處理後輸出，形成 DAG（有向無環圖）執行模型。

2.3 Queue Mode 水平擴展

n8n 提供兩種執行模式：

  • Default Mode：單進程執行，適合開發與小規模部署
  • Queue Mode：透過 Redis 將任務分發到多個 Worker 進程，實現水平擴展

[Main Process] → Redis Queue → [Worker 1] [Worker 2] [Worker N]
     ↓                              ↓          ↓          ↓
  排程/觸發              工作流執行    工作流執行    工作流執行

效能考量：Queue Mode 下，Main Process 僅負責觸發與排程，Worker 負責實際執行，併發量由 Worker 數量線性擴展。最低硬體需求僅 2 核 CPU + 4GB RAM。

三、AI Agent 架構（2026 核心創新）

3.1 LangChain 原生整合

n8n 的 AI 能力建構在 LangChain JavaScript 框架之上，透過階層式節點系統實現：

  1. AI Agent 節點（編排層）：6 種 Agent 架構

- Conversational Agent（多輪對話）

- OpenAI Functions Agent（函數呼叫）

- Plan and Execute Agent（任務分解執行）

- ReAct Agent（推理 + 行動循環）

- SQL Agent（資料庫查詢生成）

- Tools Agent（通用工具編排）

  1. 子節點（能力層）：

- LLM 連接：OpenAI、Anthropic、Google Gemini、Mistral、Ollama

- 記憶管理：Simple Memory、MongoDB、Redis、Postgres、Zep

- 向量儲存：Pinecone、Qdrant、Chroma、Weaviate

- 工具集：Calculator、Wikipedia、SerpApi、自定義程式碼

3.2 多 Agent 協作

n8n 支援在視覺化工作流中同時運行多個 AI Agent，每個 Agent 有獨立的工具集與職責，由工作流引擎協調資料流轉。這實現了「Agent 即節點」的設計模式 — 將複雜的 AI 系統拆解為可組合的視覺化元件。

四、插件生態系統

n8n 的擴展性分為五個層級：

| 層級 | 方式 | 適用場景 |

|------|------|----------|

| L1 | Function Code Node | 快速原型、一次性客製化（支援 npm require） |

| L2 | HTTP/API Nodes | 連接任何外部 API（內建 HTTP Request + Webhook） |

| L3 | Custom Node Modules | 可重用插件（n8n-nodes-starter 模板，支援 JSON 宣告式 + TypeScript 程式式） |

| L4 | Community Nodes | npm 發佈的社群節點（官方 UI 安裝，未驗證節點限自託管） |

| L5 | Trigger Extensions | 自定義觸發器（activate/deactivate 生命週期鉤子） |

生態洞察：n8n 的插件架構實現了「漸進式擴展」— 從零程式碼（拖拉節點）到少量程式碼（Function Node）到完整插件（Custom Module），降低了貢獻門檻，也是社群快速成長到 400+ 節點的關鍵。

五、可借鏡的技術項目（與 daily-digest-prompt 對映分析）

5.1 工作流 JSON 定義 → Prompt YAML 配置

  • n8n 做法：工作流以 JSON 格式儲存，引擎解譯執行
  • 本專案對映：已採用類似思路（pipeline.yaml + routing.yaml），但可進一步將 prompt 步驟骨架也 JSON/YAML 化，實現更精確的步驟控制與條件分支

5.2 Queue Mode → PowerShell Job 佇列化

  • n8n 做法：Redis + Worker 進程池，任務自動分發
  • 可借鏡：目前 Start-Job 是直接啟動，可考慮引入簡易佇列機制（如檔案佇列或 SQLite 佇列），避免同時啟動過多 Agent 造成 API 限流

5.3 節點生命週期鉤子 → Hook 系統強化

  • n8n 做法：Trigger 節點有 activate/deactivate 鉤子，Custom Node 有完整生命週期
  • 可借鏡：本專案的 Hook 系統（PreToolUse/PostToolUse/Stop）已相當完整，可參考 n8n 加入 PreWorkflow/PostWorkflow 層級的鉤子（整個工作流開始/結束時的統一處理）

5.4 AI Agent 類型分類 → Todoist 任務路由優化

  • n8n 做法：6 種 Agent 架構，依任務性質選擇最適合的 Agent
  • 可借鏡：目前 routing.yaml 已有三層模板選擇，可參考 n8n 的 Agent 類型思維，為不同任務性質（研究型、執行型、創意型）設計專屬的 Agent 配置檔

5.5 Community Node 發佈流程 → Skill 貢獻機制

  • n8n 做法：npm 發佈 + 官方 UI 安裝 + 驗證機制
  • 可借鏡：Skill 目前是手動複製到 skills/ 目錄，可設計類似的 Skill 發佈/安裝/驗證流程

5.6 執行日誌與錯誤追蹤 → 可觀測性增強

  • n8n 做法：每次執行完整記錄（輸入/輸出/耗時/錯誤），支援重放與除錯
  • 可借鏡：post_tool_logger 已記錄 JSONL 日誌，可參考 n8n 加入「執行重放」能力 — 從日誌還原並重新執行失敗的工作流
