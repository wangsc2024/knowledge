排程與 API 重試節流：指數退避、Jitter 與熔斷邊界（2026）

技術研究重試與指數退避daily-digest

技術概述

在 daily-digest-prompt 這類「多排程、多子 Agent、頻繁呼叫外部 API」的系統中，單點失敗幾乎不可避免：Todoist、Groq、本機知識庫與 ntfy 都可能短暫不可用。若所有節點在同一時間窗內以固定間隔重試，容易造成同步重試尖峰（thundering herd），反而延長故障回復時間。產業共識是：以指數退避拉開重試間隔，並以 jitter 打亂相位，使多實例、多排程的重試時間分散；同時用上限延遲、最大次數與熔斷／節流避免系統在錯誤狀態下失控放大負載。下列整理以雲端架構與 HTTP 標準文獻為三角佐證，並對照本專案腳本式管線可落地的實務切點。

核心概念與原理（含來源標註）

1. 指數退避與重試策略

主張：重試間隔應隨失敗次數遞增，而非固定常數，以降低對已過載服務的疊加壓力。[來源：Azure Architecture Patterns — Retry, 微軟]

主張：AWS 文件將 SDK 標準重試行為描述為「可配置、具上限」的自動重試，並強調針對節流類回應（例如限流）需要更長的等待與節制。[來源：AWS General Reference — AWS SDKs retry behavior]

主張：HTTP 語意層上，伺服器可透過 Retry-After 告知客戶端最短等待時間；客戶端應尊重此標頭以避免違反服務意圖。[來源：RFC 9110 HTTP Semantics（取代 RFC 7231 系列）]

2. Jitter（隨機化）與「全抖動」直覺

主張：在指數退避之上加入隨機抖動，可讓多個並發客戶端不易在同一瞬間再次衝擊後端；此作法在公開架構指引中常被建議，用於緩解集體同步。[來源：Azure Architecture Patterns — Retry]

主張：雲端 SDK 普遍採取「指數 + 抖動」組合；實作細節依語言與服務而異，但相位分散是共通目標。[來源：AWS General Reference — API retries]

3. 過載治理與「用重試反制過載」的悖論

主張：在服務已過載時，若客戶端仍以高頻或同步模式重試，會使過載惡化；SRE 文獻討論負載拒絕、排隊與退化等策略，核心訊息是把壓力留在邊界並盡量早失敗，而非無限堆疊重試。[來源：Google SRE Book — Handling Overload]

4. 可重試條件與幂等邊界（工程化判斷）

主張：網路逾時、連線重置與部分 5xx / 429 往往被視為暫時性；但 4xx 中的語意錯誤（例如驗證失敗）通常不應盲目重試。[來源：RFC 9110 — 方法安全與幂等性語意，搭配 Azure Retry 模式的情境敘述]

與 daily-digest-prompt 的對照：本專案 scheduler-state.json 已記錄排程執行狀態；對於「同一小時內 Phase 2 再跑一次」這類設計，重試邏輯應區分：讀取型步驟（如 hybrid 搜尋）可較積極退避，寫入型步驟（如 KB import、Todoist 狀態變更）須檢查幂等或可安全重放的條件，避免雙寫。

最佳實踐與常見陷阱

| 作法 | 說明 |

|------|------|

| 指數退避 + 上限 | 每次重試加倍等待，但設 cap（例如最多 60–120 秒）避免單次任務拖過排程窗 |

| 完整 jitter | 在 [0, delay] 均勻取樣（概念上稱「full jitter」），或於基底延遲上加隨機偏移 |

| 尊重 Retry-After | 429/503 若有標頭，優先採用伺服器建議 |

| 最大次數 + 熔斷| 連續失敗達閾值改走降級路徑（本專案已有 KB 不可用時跳過匯入的成功路徑，即屬一種降級） |

| 陷阱：固定 1s 重試 N 次 | 多機用同一节奏會放大尖峰；亦不利長尾尾延 |

| 陷阱：對非幂等寫入盲目重試 | 可能造成重複通知、重複匯入或重複 API 呼叫副作用 |

與本專案管線的應用建議

  1. PowerShell 排程層：Task Scheduler 觸發間隔固定時，腳本內部對外部 API 的重試應自管退避，避免「整點同時踩」localhost:3000 與雲端服務。
  2. Phase 0 / Phase 2 銜接：若 Codex / Claude 子流程逾時，外層重試間隔應高於一般 API（例如 2^n * 基數秒），並寫入 state/scheduler-state.json 供診斷。
  3. ntfy / JSON 檔推播：短暫網路失敗可退避重送；重複推播若要避免，需在內容層做去重鍵或單次 token（視 ntfy topic 策略而定）。

程式碼範例（PowerShell 7：指數退避 + jitter）


function Get-RetryDelaySeconds {
  param([int]$Attempt, [int]$Base = 2, [int]$Cap = 120)
  # 指數部分
  $exp = [Math]::Min([Math]::Pow($Base, $Attempt), $Cap)
  # 全抖動：0..exp 均匀随机
  $j = Get-Random -Minimum 0 -Maximum ([int][Math]::Ceiling($exp) + 1)
  return [Math]::Min($j, $Cap)
}

for ($i = 0; $i -lt 5; $i++) {
  try {
    # Invoke-RestMethod ...
    break
  } catch {
    Start-Sleep -Seconds (Get-RetryDelaySeconds -Attempt $i)
  }
}
