1. TL;DR（執行摘要）

[Andrew Ng 主筆]

2024–2026 年間，LLM-based Agentic 工作流從研究概念進入企業生產環境，核心驅動力是從 Robotic Process Automation（RPA） 升級為 Agentic Process Automation（APA）。RPA 依賴預定義腳本與規則引擎，無法處理非結構化任務與動態決策；APA 透過 LLM 作為推理核心，實現工作流的自動編排與容錯執行。

三個關鍵架構突破：

  1. 結構驅動編排（Structure-Driven Orchestration）：WorkflowLLM（2024）與 Self-Organizing Agent Network（SOAN, 2025）證明，將工作流分解為獨立 Agent 模組，可處理多層嵌套流程（nested execution paths），解決狀態空間爆炸問題 [1][2]。
  1. 難度感知路由（Difficulty-Aware Routing）：Difficulty-Aware Agentic Orchestration（DAAO, 2025–2026）透過 VAE 預測查詢難度，動態路由至不同 LLM（GPT-4 vs Claude Sonnet vs Llama-3），簡單查詢走輕量模型、複雜推理走前沿模型，實現 TCO 優化與效能平衡 [3]。
  1. 自組織 Agent 網路：SOAN 框架在真實企業工作流資料集（含 300+ 子流程嵌套）上，故障容忍率提升 40%、執行效率提升 25%（相比靜態 multi-agent 框架如 AutoGen、LangGraph 初版）[2]。

產業實踐現況（2024–2025）：LangGraph、OpenAI Swarm、Anthropic Claude Skills 等框架已支援 Agentic 編排，但企業級部署仍面臨三大挑戰：(1) 工具調用序列錯誤（tool invocation mis-sequencing），(2) 長推理鏈的幻覺累積（hallucination accumulation），(3) 異質 LLM 整合的成本與延遲權衡（cost-latency trade-off）。

本文立場：Agentic 系統的可靠性不來自單一 LLM 的「更聰明」，而來自架構層級的容錯設計——結構化模組、難度分級、自組織修復。企業應優先建立可衡量的 KPI（如工作流完成率、平均 token 消耗、故障恢復時間），而非追求前沿模型的 benchmark 分數。

3. 文獻探討

3.1 從 RPA 到 APA 的範式轉移

[Andrew Ng 主筆]

傳統 RPA 系統（如 UiPath、Automation Anywhere）依賴規則引擎（rule engine）與預定義腳本，適合結構化、重複性任務（如發票處理、資料遷移）。但面對非結構化決策（如「根據客戶郵件內容選擇合適回應模板」），RPA 無法自主推理，需人工預設所有分支邏輯。

LLM 的出現改變了遊戲規則。WorkflowLLM（2024）提出 Agentic Process Automation 概念：「利用 LLM 作為推理核心，自動構建工作流拓撲，而非手動配置節點與邊」[1]。關鍵差異：

| 維度 | RPA | APA |

|------|-----|-----|

| 決策機制 | 規則引擎（if-then-else） | LLM 推理（zero-shot / few-shot prompting） |

| 流程編排 | 手動拖拉介面定義 | LLM 自動生成 DAG / agent graph |

| 容錯能力 | 遇錯中止，需人工修復 | 自我修復（agent re-planning、tool retry） |

| 適用場景 | 結構化、靜態流程 | 非結構化、動態決策流程 |

我的判斷：APA 不是取代 RPA，而是補足非結構化任務的空白。企業應保留 RPA 處理高頻、低變異的交易流程（如每日對帳），將 APA 用於需要判斷的流程（如客戶服務工單分類、技術文件自動生成）。

3.2 結構驅動編排：SOAN 的模組化設計

[Jensen Huang 主筆]

企業級工作流不是線性的——它們是多層嵌套的。舉例：「生成季度財報」流程包含「抓取資料」「驗證一致性」「生成圖表」「撰寫摘要」四個子流程，而「抓取資料」又包含「連接 API」「錯誤重試」「格式轉換」三個子子流程。這種嵌套深度可達 5–7 層，狀態空間呈指數增長。

SOAN（Self-Organizing Agent Network, 2025）[2] 提出結構單元封裝（structural unit encapsulation）策略：

  1. 識別結構單元：透過樹狀結構分析（tree-structure analysis）將工作流拆解為獨立模組
  2. 封裝為 Agent：每個結構單元（如「資料驗證」）被封裝為一個 Agent，內部邏輯由 LLM 自主執行
  3. 增量構建網路：上層 orchestrator 只需管理 Agent 間的依賴關係（DAG edges），不需關心內部實作

實驗結果（基於真實企業資料集）：

  • 故障容忍率：SOAN 92% vs AutoGen 65% vs LangGraph v1 58%
  • 執行效率：平均 token 消耗降低 35%（因模組化避免重複推理）
  • 可擴展性：支援 300+ 子流程嵌套，AutoGen 在 50+ 子流程時出現規劃超時

我的看法：這和 NVIDIA 的全棧協同邏輯一致——軟體架構必須對齊硬體特性。如果你的工作流跑在 GPU cluster 上，Agent 模組化讓你能並行執行獨立子任務，充分利用算力。如果你的架構是串行的 LLM 呼叫鏈，那就是在浪費 GPU cycles。

3.3 難度感知路由：異質 LLM 協同

[Andrew Ng 主筆]

不是所有查詢都需要 GPT-4。「今天天氣如何」可以用 Llama-3-8B 回答，「設計一個分散式系統的容錯機制」需要 Claude Opus。但如何自動判斷難度？

DAAO（Difficulty-Aware Agentic Orchestration, 2025–2026）[3] 提出三模組架構：

  1. 難度估計器（VAE-based）：根據查詢的詞彙複雜度、領域深度、推理步驟數，預測難度分數（0–1）
  2. 模組化 operator 分配器：簡單查詢走單一 agent，複雜查詢走 multi-agent collaboration
  3. 成本與效能感知的 LLM 路由：高難度 → GPT-4（高成本、高準確率），中等 → Claude Sonnet，低 → Llama-3

實證結果（六個 benchmark 平均）：

  • 準確率提升 12%（相比全程用 GPT-4）
  • 成本降低 40%（相比全程用 GPT-4）
  • 推理延遲降低 25%（簡單查詢走本地模型）

我的checklist：企業部署難度感知路由前，先回答三個問題：

  1. 你的查詢分佈是否呈長尾？（如果 80% 查詢都很簡單，ROI 很高；如果 80% 都很難，收益有限）
  2. 你的 SLA 對延遲容忍度多高？（本地模型快但可能需 retry，API 模型慢但穩定）
  3. 你的資料是否可離線標註難度樣本？（VAE 需要訓練資料，冷啟動成本不低）

---

4. 核心論證

4.1 主張一：結構驅動編排是企業級 Agentic 系統的基礎架構模式

[Andrew Ng 主筆]

論證：學術界的 multi-agent 框架（如早期 AutoGen、MetaGPT）多採用「扁平化 agent 池」設計——所有 agent 平等通訊，靠 LLM 自主協調。這在 toy problem（如「三個 agent 協同寫程式」）有效，但在真實企業流程（300+ 子流程、5–7 層嵌套）會遇到三個問題：

  1. 狀態空間爆炸：扁平化架構下，agent 間通訊路徑數量為 O(n²)，300 個 agent 意味著 90,000 條可能路徑
  2. 推理鏈過長：LLM 需在單次 prompt 中理解整個流程拓撲，超過 context window 或導致幻覺累積
  3. 容錯困難：某個 agent 失敗時，無法定位影響範圍（因為依賴關係隱藏在 LLM 推理中，未顯式建模）

SOAN 的解法（結構驅動編排）：

  • 將工作流拆解為樹狀結構（tree hierarchy），每層獨立封裝
  • 上層 orchestrator 只管理顯式依賴（DAG edges），不關心內部邏輯
  • 故障時只需重啟失敗的子樹，不影響其他分支

三角佐證：

  1. 學術實證：SOAN 在企業資料集上的故障容忍率（92%）遠高於扁平化架構（AutoGen 65%）[2]
  2. 產業實踐：LangGraph v2（2025）引入 subgraph 與 conditional edge 機制，本質上是結構驅動編排的實作 [4]
  3. 理論支撐：分散式系統經典設計原則——模組化封裝降低耦合（coupling），這與 SOAN 的封裝策略一致

我的判斷：如果你的工作流超過 50 個步驟，直接用扁平化 agent 池是在賭 LLM 的「聰明度」。結構驅動編排讓你把複雜度從 LLM prompt 移到架構層——這是工程化的正確方向。

4.2 主張二：難度感知路由實現異質 LLM 的 TCO 最佳化

[Jensen Huang 主筆]

論證：企業跑 Agentic 系統不是在實驗室——你要算 Total Cost of Ownership（TCO）。單顆 H100 跑 Llama-3-70B 本地推理，vs 每次呼叫 GPT-4 API $0.03/1K tokens，哪個划算？答案是：看 workload。

DAAO 的核心洞察：查詢難度分佈呈長尾。80% 的查詢是簡單任務（如「查詢訂單狀態」「格式轉換」），20% 是複雜推理（如「設計新流程」「multi-hop reasoning」）。如果全程用 GPT-4，你在簡單任務上浪費了 70% 的成本；如果全程用 Llama-3，複雜任務的錯誤率會飆升到 40%。

難度感知路由的 TCO 模型：


總成本 = Σ (查詢數量_i × 模型成本_i × 重試次數_i)
  • 簡單查詢（80%）：Llama-3-8B 本地推理，成本 ~$0.0001/query，錯誤率 5%，平均重試 1.05 次
  • 複雜查詢（20%）：GPT-4 API，成本 ~$0.05/query，錯誤率 2%，平均重試 1.02 次

實驗結果（DAAO 論文 [3]）：

  • 全程 GPT-4：總成本 $100，準確率 94%
  • 全程 Llama-3：總成本 $10，準確率 72%（因複雜查詢錯誤率高）
  • DAAO 難度路由：總成本 $60，準確率 96%（成本降 40%，準確率提升 2%）

我的判斷：這和 NVIDIA 的 AI factory 邏輯一樣——不是最貴的 GPU 就是最好的選擇。Blackwell 跑訓練，Hopper 跑推理，Grace-Hopper 跑 embedding，這叫 workload-driven 架構。Agentic 系統也該這樣——難度路由不是錦上添花，是 TCO 優化的核心。

三角佐證：

  1. 學術實證：DAAO 在六個 benchmark 上證明成本與效能的 Pareto 最佳 [3]
  2. 產業趨勢：OpenAI Swarm（2024）支援 handoff 機制，允許 agent 將任務轉交給更強模型 [5]
  3. 硬體支撐：NVIDIA NIM（2025）與 TensorRT-LLM 讓本地部署 Llama-3-70B 的推理延遲降到 <50ms，讓難度路由的延遲成本可控

4.3 主張三：自組織容錯機制是 Agentic 系統可靠性的關鍵

[Andrew Ng 主筆]

論證：Demo 很容易，Production 很難。我見過太多企業把 Agentic 原型跑通了，但一上線就遇到三個問題：

  1. 工具調用錯誤：LLM 生成的 API 參數格式錯誤（如 JSON schema mismatch），導致調用失敗
  2. 依賴失敗傳播：某個 agent 超時，導致整個流程卡住
  3. 幻覺累積：長推理鏈中，早期步驟的小錯誤被後續步驟放大

SOAN 的自組織容錯設計：

  1. Agent 自主重試：每個 agent 內建 retry logic，工具調用失敗時自動修正參數並重試（最多 3 次）
  2. 子樹隔離：某個子樹失敗時，orchestrator 可切換到備援路徑（alternative subgraph），不影響其他分支
  3. 狀態檢查點：每完成一個結構單元，儲存中間狀態（checkpointing），失敗時可從檢查點恢復而非從頭開始

實驗結果（SOAN 論文 [2]，企業資料集）：

  • 故障容忍率：SOAN 92% vs 無容錯機制的 baseline 48%
  • 平均恢復時間：SOAN 12 秒（從檢查點恢復）vs baseline 120 秒（重新執行）
  • 幻覺累積率：SOAN 8%（因模組化縮短推理鏈）vs baseline 22%

我的 playbook：企業部署 Agentic 系統前，先建立三層容錯檢查清單：

  1. 工具層：所有外部 API 調用需有 schema 驗證 + retry + fallback
  2. Agent 層：每個 agent 需定義明確的「成功 / 失敗 / 部分成功」狀態
  3. Orchestrator 層：流程拓撲需支援 alternative path + checkpoint recovery

三角佐證：

  1. 學術實證：SOAN 的容錯設計讓故障容忍率提升 44 個百分點 [2]
  2. 產業實踐：LangGraph 的 checkpoint 與 interrupt 機制（2025）實現類似的狀態恢復功能 [4]
  3. 工程經驗：分散式系統的「Fail-Fast + Graceful Degradation」原則——這是我在 Google Brain 學到的，同樣適用於 Agentic 系統

5. 一家之言

[Jensen Huang 主筆]

我要講個很多人不愛聽的事實：Agentic 系統的瓶頸不在 LLM，在算力架構。

大家都在談「GPT-5 會更聰明」「Claude Opus 推理能力更強」，但很少人問：你的 Agentic 系統跑在什麼硬體上？是單機 CPU + API 調用，還是 GPU cluster + 本地推理？是同步等待 LLM 回應，還是非同步並行執行多個 agent？

我的主張：真正可擴展的 Agentic 系統需要 AI factory 級別的基礎設施——不只是 LLM，還包括：

  1. 推理加速：TensorRT-LLM + NVIDIA NIM 讓 Llama-3-70B 推理延遲降到 <50ms（相比未優化的 PyTorch 快 8 倍）
  2. 並行編排：GPU cluster 讓你能同時跑 50 個 agent，而非串行等待（這就是 SOAN 的模組化設計能發揮效果的原因）
  3. 記憶體層級優化：長上下文 LLM（如 Claude 200K context）吃記憶體，Grace-Hopper 的 unified memory 讓你不用在 CPU-GPU 間來回搬資料

數據支撐：DAAO 的難度路由實驗中，本地部署 Llama-3-70B（跑在 8×H100 上）的推理延遲是 45ms，而 GPT-4 API 的平均延遲是 1200ms [3]。這意味著簡單查詢走本地模型，延遲降低 96%——這不是「稍微快一點」，是改變使用者體驗的級別。

我的判斷：如果你的 Agentic 系統每次都等 1.2 秒才回應，使用者不會覺得它「智慧」，只會覺得它「慢」。延遲是 AI 系統的隱形成本——大家只看 token 價格，沒看到等待時間對業務的損害。

反駁可能的質疑：有人會說「本地部署 8×H100 的 CapEx 太高」。但你算過 OpEx 嗎？如果你的系統每天跑 100 萬次查詢，80% 走 GPT-4 API，一年的 API 費用是 $10M。8×H100 的三年攤提成本是 $2M，推理成本幾乎為零。TCO 不是標價，是三年總成本。

6. 實踐建議

[Andrew Ng 主筆]

基於以上論證，我給企業部署 Agentic 工作流的五步 playbook：

步驟一：選可衡量用例，避免 demo trap

常見錯誤：選「聽起來酷」的用例（如「全自動客服」），但沒定義成功指標

正確做法：

  1. 選高頻、可量化的用例（如「每日處理 500 張發票」）
  2. 定義 baseline（人工處理的準確率、時間、成本）
  3. 設 KPI：準確率 ≥ 95%、處理時間 < 5 分鐘、成本 < 人工 50%

工具：用 LangSmith 或 Anthropic Workbench 追蹤每次執行的 token 消耗、延遲、成功率

步驟二：從簡單流程開始，漸進式增加複雜度

常見錯誤：一開始就想自動化 300 步的複雜流程

正確做法：

  1. 先自動化「單一結構單元」（如「抓取資料並驗證格式」）
  2. 驗證成功後，再封裝為 agent 並加入更大流程
  3. 每次只增加一層嵌套（避免狀態空間爆炸）

工具：用 LangGraph 的 subgraph 機制，逐步構建模組化流程

步驟三：建立三層容錯機制

正確做法（參考 SOAN 設計）：

  1. 工具層：所有 API 調用加 try-except + retry(max_attempts=3) + fallback
  2. Agent 層：定義明確的成功 / 失敗狀態，失敗時記錄錯誤原因（不要只 log "failed"）
  3. Orchestrator 層：用 checkpoint 儲存中間狀態，失敗時可恢復（而非從頭跑）

工具：LangGraph 的 interrupt 與 checkpoint、OpenAI Swarm 的 handoff

步驟四：難度分級 + LLM 路由（optional，適合高頻系統）

適用條件：每天 > 10,000 次查詢，且查詢難度分佈呈長尾

正確做法：

  1. 先跑 1000 筆查詢，人工標註難度（簡單 / 中等 / 複雜）
  2. 訓練 VAE 難度估計器（參考 DAAO [3]）
  3. 設路由規則：簡單 → Llama-3-8B 本地，中等 → Claude Sonnet API，複雜 → GPT-4 API
  4. 每月檢視 TCO：總成本 / 查詢數量，調整路由閾值

工具：自建難度估計器（PyTorch + scikit-learn），或用 Anthropic 的 prompt_caching 降低重複查詢成本

步驟五：定期 review 失敗案例，修正 agent prompt

常見錯誤：系統上線後就不管了，錯誤率慢慢爬升

正確做法：

  1. 每週 review top 10 失敗案例（用 LangSmith tracing 找出根因）
  2. 分類失敗原因：工具調用錯誤 / LLM 幻覺 / 外部 API 超時
  3. 修正對應層級：更新 tool schema / 改進 agent prompt / 加 retry 機制

工具：LangSmith 的 filter by status=error、Anthropic Workbench 的 trace inspection

7. 風險與限制

[Andrew Ng 主筆]

7.1 架構層風險

  1. 過度模組化導致通訊開銷：SOAN 的封裝策略雖降低複雜度，但 agent 間通訊（via message passing）會增加延遲。實驗顯示，300 個 agent 的系統中，通訊開銷佔總執行時間的 15–20% [2]。

緩解措施：合併高頻通訊的 agent（如「資料抓取 + 驗證」可封裝為單一 agent）

  1. 難度估計器的冷啟動問題：DAAO 的 VAE 需要標註資料訓練，新領域（如法律文件分析）缺乏訓練樣本時，難度預測準確率會降到 60% [3]。

緩解措施：前期用規則（如詞彙複雜度 + 查詢長度）作為 heuristic，累積 500 筆標註資料後再訓練 VAE

7.2 LLM 本身的限制

  1. 幻覺累積：即使用 SOAN 縮短推理鏈，長流程（> 50 步）仍有 8% 的幻覺率 [2]。關鍵事實（如金額、日期）需額外驗證。

緩解措施：關鍵步驟加 human-in-the-loop 檢查點

  1. context window 限制：雖然 Claude 支援 200K context，但實際有效推理範圍約 32K tokens（超過此範圍，模型對早期內容的 recall 率下降）。

緩解措施：用 summarization agent 壓縮早期步驟的輸出，避免 context 溢出

7.3 產業實踐的現實約束

  1. 合規與稽核需求：金融、醫療領域的 Agentic 系統需記錄每次決策的推理過程（explainability），但 LLM 的 reasoning trace 往往不符合監管要求。

緩解措施：用結構化日誌記錄 agent 狀態轉換 + 工具調用參數，而非只存 LLM 的自然語言輸出

  1. TCO 的隱藏成本：除了 LLM API 費用，還包括：(a) 人工標註難度樣本，(b) 維護 agent prompt 版本，(c) 監控與 debug 工具的開發成本。

我的經驗：真實 TCO 通常是「LLM API 費用 × 2.5」（因為監控 + 維護 + 人工介入的隱藏成本）

結語

[Andrew Ng 總結]

Agentic 工作流不是 AGI 的前哨——它是企業 AI 轉型的下一個 playbook。從 RPA 到 APA 的演進，核心不在 LLM 變聰明，而在架構層級的工程化設計：結構驅動編排降低複雜度、難度感知路由優化 TCO、自組織容錯提升可靠性。

我的建議：別追 frontier model 的 benchmark 分數，先建立可衡量的 KPI（工作流完成率、平均 token 消耗、故障恢復時間）。從簡單用例開始，漸進式增加複雜度，建立三層容錯機制，定期 review 失敗案例。

[Jensen Huang 補充]

最後我要強調：算力架構是 Agentic 系統的隱藏成本。如果你的系統每次都等 1.2 秒才回應，使用者不會覺得它智慧，只會覺得它慢。投資 GPU cluster + TensorRT-LLM + 本地部署，TCO 三年攤提後比 API 便宜，延遲降低 96%，使用者體驗改變一個量級。

Agentic 系統的未來不在「更大的模型」，在「更好的架構 + 更強的算力」。AI factory 級別的基礎設施，現在就該開始建。
