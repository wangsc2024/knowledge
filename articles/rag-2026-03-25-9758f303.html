一、執行摘要（300 字）

本次對 daily-digest-prompt 專案進行排程遷移後可靠性與效能診斷。最近一次重大架構變更為排程系統從 Windows Task Scheduler 全面遷移至 Claude Code Scheduled-Tasks + PM2（commit c9716c1），同時實施了排程防護機制（968ff5e）防止 parse error 無聲失敗。

診斷發現三大結構性問題：（1）Phase Failure 累計 56 次，其中 03-20 單日爆發 21 次、03-13 和 03-18 各有 10/6 次，顯示排程密度與 Agent 資源競爭是主要肇因；（2）State 檔案持續膨脹，autonomous-recovery-queue.json（148KB）和 context-usage.json（112KB）已超越前次治理的預期上限，loop-state 檔案再次累積至 24 個；（3）Token 消耗波動劇烈，03-21 達到 2,053 萬 tokens（7,407 次 tool calls），較平均值高出 60%，暗示某些自動任務的 Context Window 保護機制失效。

本報告提出三組優化建議，涵蓋 Phase Failure 分級告警、State 清理自動化排程、以及 Token 預算 per-task 上限機制。

二、排程遷移影響評估

2.1 遷移前後架構比較

| 維度 | Task Scheduler（舊） | PM2 + Claude Code（新） |

|------|---------------------|------------------------|

| 觸發機制 | cron + schtasks | PM2 cron_restart + HEARTBEAT.md |

| 錯誤恢復 | retry=1 靜默重試 | 排程防護機制（parse error 攔截） |

| 日誌 | 分散在 logs/ | 結構化 JSONL + PM2 log |

| 並行控制 | 無（依賴 OS 排程） | PM2 instance 管理 |

| 監控 | check-health.ps1 | check-health.ps1 + ntfy 告警 |

2.2 遷移後穩定性指標

排程遷移後（03-25）目前紀錄：

  • scheduler-heartbeat: running（PID 549772，12:06 更新）
  • phase_failure: 2 次（正常範圍）
  • timeout: 0 次
  • api_error: 0 次

初步評估：遷移後穩定性良好，但需持續觀察 3-5 天確認基線。

| 03-25 | 2 | 0 | 遷移後首日 |

3.2 根因歸類（基於 failure_taxonomy）

phase_failure（56 次）的描述為「Phase 執行失敗（LLM 錯誤、結果檔缺失）」，結合歷史記憶分析：

  1. 結果檔案缺失（已知根因，2026-02-24 修復）：$dedicatedPrompts 改為動態掃描已解決靜態路徑問題
  2. LLM 輸出格式異常：Agent 產出非 JSON 結構的結果，Phase 3 組裝無法解析
  3. 資源競爭：03-20 的 21 次爆發可能與多個 todoist-team 30 分鐘排程密集執行有關
  4. PowerShell 變數解析錯誤：f18eac7 修復了導致 todoist-team 排程全面癱瘓的問題

3.3 建議：Phase Failure 分級告警


# 建議新增至 config/notification.yaml
phase_failure_alert:
  thresholds:
    warning: 5   # 單日 >= 5 次 → ntfy 告警
    critical: 10  # 單日 >= 10 次 → ntfy 緊急 + 暫停非必要排程
  cooldown_minutes: 60

| context-usage.json | 112KB | 累積所有 session 用量，無滾動清理 |

| state/ 總計 | 1.4MB | 較 03-21 持平，但三大檔案仍在增長 |

4.2 建議：排程化 State 清理

建議在 kb-backup-all（每日 00:15）之後新增清理步驟：

  • 移除 > 48 小時的 loop-state-*.json
  • autonomous-recovery-queue.json 保留最近 50 筆
  • context-usage.json 保留最近 7 天

五、Token 消耗與 Context Window 治理

5.1 近期 Token 使用趨勢

| 日期 | 預估 Tokens | Tool Calls | 輸出字元 |

|------|-----------|------------|--------|

| 03-18 | 12.6M | 4,557 | 42.4M |

| 03-19 | 10.7M | 3,964 | 36.0M |

| 03-20 | 14.2M | 3,579 | 48.2M |

| 03-21 | 20.5M | 7,407 | 68.5M |

| 03-22 | 10.9M | 5,497 | 35.3M |

03-21 的 20.5M tokens 是近期峰值（較均值高 60%），對應 7,407 次 tool calls。

5.2 根因推測

  • 03-21 的 Phase 2 自動任務（phase2-auto）消耗 1.96M tokens，Phase 3 組裝 1.1M tokens
  • 高 tool call 數量暗示部分 Agent 進入重試迴圈或大量檔案讀取
  • 知識庫關聯：03-21 正好是「State 膨脹治理與配置碎片化統一」研究日，深度研究本身就需要大量 context

5.3 建議：per-task Token 預算

per_task_limits:

  tech_research: 800000
