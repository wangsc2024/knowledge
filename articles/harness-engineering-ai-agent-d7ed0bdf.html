Harness Engineering 深度研究：AI Agent 治理框架與實踐指南（2026）

研究摘要

本研究深入探討 Harness Engineering 作為 AI 驅動軟體工程治理框架的核心概念、實踐原則與實施方法。研究發現，在 2026 年，軟體工程的競爭優勢已從「擁有最大模型」轉向「擁有最有效的 Harness 系統」。

---

一、核心定義

1.1 什麼是 Harness Engineering？

Harness Engineering 是一種新興學科，專注於設計約束、工具、反饋循環、文件和驗證系統，以引導強大但不可預測的 AI Agent 產生可靠、可維護和可擴展的軟體輸出。

核心公式：


Agent = Model + Harness

其中 Harness 包含：

  • Context（上下文）：Agent 所需的宣告式和程序性知識
  • Constraint（約束）：管理 Agent 輸出的規則
  • Convergence（收斂）：迭代評估約束、識別差距、精煉規則的過程

1.2 為何需要 Harness？

在 AI Agent 時代，軟體工程的主戰場正從「編寫代碼」轉移到「設計環境、約束、反饋循環和治理機制」。Harness 是確保自主 Agent 在生產環境中可靠運作的關鍵。

---

二、核心原則

2.1 Context 管理（最被低估的表面）

"Context is the most underappreciated surface in agentic systems."

問題：大多數實踐者花數小時調整 prompt 措辭，卻只花數分鐘在 context 結構上。這是顛倒的。

正確做法：

  • 模型「知道什麼」與「何時知道」的結構，比任何單一指令更能決定長期任務的連貫性
  • 使用 Architecture Decision Records (ADR)、Schema Validators 等機器可讀的 artifacts
  • 按需載入（load-on-demand）而非預先載入所有上下文

2.2 架構約束（Machine-Enforceable）

將架構約束嵌入開發工具鏈，透過機器強制執行：

  • 模組邊界：透過 fitness functions 強制執行分層架構
  • 穩定數據結構：Schema 驗證防止非預期變更
  • 領域特定規則：DSL（Domain-Specific Language）確保業務邏輯一致性

2.3 關注點分離（Specialized Agents）

將工作拆分到專門的 Agent，而非過載單一 Agent：

| 角色 | 職責 |

|------|------|

| Planner | 規劃任務分解與執行順序 |

| Generator/Builder | 生成代碼、配置、文件 |

| Evaluator/QA | 測試運行應用、評分輸出 |

三、Fitness Functions：架構的自動化單元測試

3.1 定義

"Fitness functions are automated (unit) tests for your architecture."

Fitness function 是一種測試，用於評估實施與設計目標的接近程度。它將架構決策轉化為可執行的約束。

3.2 左移治理（Shift-Left Governance）

傳統模式：季度架構審查 → 反饋週期長、發現問題晚

Fitness Functions 模式：


每次 commit → Pre-commit hook → CI/CD pipeline → 即時反饋

好處：

  • 開發團隊在編碼時即獲得快速反饋
  • 防止違規合併到主分支
  • 縮短反饋循環從「季度」到「秒級」

3.3 實施層級

按照反饋速度排序（快 → 慢）：

| 層級 | 工具 | 執行時機 | 示例 |

|------|------|---------|------|

| L0 | PostToolUse Hook | Agent 每次工具呼叫後 | 格式化、簡單驗證 |

| L1 | Pre-commit Hook | git commit 前 | Linter、Type checker |

| L2 | CI Pipeline | Pull Request 時 | 單元測試、整合測試 |

| L3 | CD Pipeline | 部署前 | 端到端測試、效能測試 |

最佳實踐：盡可能將檢查推到更快的層級（L0/L1）。

3.4 工具與範例

ArchUnit（Java/.NET）：


@Test

    classes()
        .that().resideInAPackage("..service..")

四、最佳實踐

4.1 快速反饋循環（Fast Feedback Loops）

將測試、Linter、型別檢查器、UI 自動化作為「背壓」（backpressure）機制：


代碼生成 → 即時驗證 → 發現問題 → 立即修正 → 再次驗證

4.2 人類引導，Agent 執行（Humans Steer, Agents Execute）

工程師的角色轉變：

  • 設計環境與約束
  • 明確指定意圖
  • 建立反饋循環

Agent 的職責：

  • 編寫代碼
  • 執行測試
  • 應用修正

4.3 機器可讀的 Artifacts

優先使用結構化、可解析的格式：

| Artifact 類型 | 格式 | 用途 |

|--------------|------|------|

| ADR | YAML/Markdown + Frontmatter | 架構決策追蹤 |

| Schema | JSON Schema / OpenAPI | API 契約驗證 |

| Fitness Functions | Unit Test Framework | 架構約束執行 |

| Policy | OPA Rego / YAML | 安全與合規政策 |

4.4 CI/CD 整合

將 Fitness Functions 嵌入 CI/CD pipeline：


# .github/workflows/fitness-functions.yml
name: Architecture Fitness
on: [push, pull_request]
jobs:
  fitness:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run Fitness Functions
        run: |
          pytest tests/fitness/
          # 失敗則 build 失敗

五、2026 年趨勢與洞見

5.1 競爭優勢轉移

"By 2026, competitive advantage belongs to those with the most effective and efficient harness rather than those with the largest model."

5.2 供應鏈安全強化

Harness（CI/CD 平台）於 2026 年 2 月推出 Artifact Registry，強化軟體供應鏈安全與治理。

5.3 政策感知 AI（Policy-Aware AI）

新一代 Harness 系統整合：

  • 可審計性：所有 Agent 行為可追溯
  • RBAC 對齊：Agent 權限與人類角色一致
  • 政策引擎：自動執行安全與合規規則

六、實踐建議：Daily Digest Prompt 專案優化方案

6.1 Context 管理優化

現狀問題：

  • 部分 prompt 模板過長（>1000 行）
  • Context 預載入過多（CLAUDE.md + AGENTS.md + SKILL.md 同時載入）

優化方案：

  1. 按需載入：SKILL.md 分段讀取（已實施 offset+limit）
  2. Context 分層：

- L0（必讀）：preamble.md、角色宣告

- L1（條件）：對應的 SKILL.md

- L2（延遲）：參考文件、詳細範例

  1. 摘要快取：將大型 JSON（如 adr-registry.json）預處理為 snapshot

6.2 Fitness Functions 實施

建議新增的 Fitness Functions：


# tests/fitness/test_architecture_fitness.py

def test_no_prompt_exceeds_2000_lines():
    """所有 prompt 模板不得超過 2000 行"""
    for prompt in glob.glob("prompts/**/*.md"):
        line_count = len(open(prompt).readlines())
        assert line_count <= 2000, f"{prompt} 超過限制：{line_count} 行"

def test_all_skills_have_frontmatter():
    """所有 SKILL.md 必須有合法 frontmatter"""
    for skill in glob.glob("skills/*/SKILL.md"):
        with open(skill) as f:
            content = f.read()
            assert content.startswith("---"), f"{skill} 缺少 frontmatter"
            # 驗證必填欄位：name, version, description

def test_results_json_schema_compliance():
    """所有 results/*.json 必須符合 schema"""
    schema = json.load(open("config/schemas/result.schema.json"))
    for result in glob.glob("results/*.json"):
        data = json.load(open(result))
        validate(data, schema)  # jsonschema.validate

def test_no_circular_skill_dependencies():
    """Skills 不得有循環依賴"""
    graph = build_skill_dependency_graph()
    assert not has_cycles(graph), "發現 Skill 循環依賴"

6.3 Hook 層級優化（左移治理）

當前實施：

  • ✅ L0：PostToolUse Hook（agent_guardian.py）
  • ✅ L1：Pre-commit Hook（validate_config.py）

建議新增：

  • L0.5：Pre-write Hook 擴展：

- 檢查同檔 Edit 次數（已達 3 次 → 警告改用 Write）

- 驗證 JSON 寫入前符合 schema

  • L1.5：Pre-push Hook：

- 執行所有 Fitness Functions

- 防止不符架構約束的代碼推送

6.4 Context Compression 自動化

當前機制：ADR-036 Context 壓縮感知

優化方案：

  1. 預測性壓縮：當 token 使用率 >60% 時主動壓縮，而非等到 65-80%
  2. 智能保留：

- 保留最近 N 次關鍵決策（ADR、重要錯誤）

- 壓縮重複模式（如相同 API 呼叫的多次日誌）

  1. 分層快取：

- Hot：當前任務相關（不壓縮）

- Warm：最近 3 次執行（輕度壓縮）

- Cold：歷史記錄（重度壓縮或歸檔）

6.5 Agent 專門化（Specialized Agents）

當前狀態：

  • ✅ 已實施：Fetch、Analyst、Assembler 角色分工
  • ⚠️ 可改進：部分 Agent 職責重疊

優化建議：

| Agent 類型 | 當前 | 優化後 |

|-----------|------|-------|

| todoist-query | Fetch + 初步分析 | 純 Fetch，不做評分 |

| task-optimizer | 分析 + 評分 + 模板選擇 | 專注評分與優化，模板選擇交給 Router |

| task-executor | 執行 + 報告 | 純執行，報告交給 Reporter Agent |

八、關鍵指標（KPIs）

8.1 治理效能

  • 違規檢測時間：從季度審查（~90 天）→ Pre-commit（<1 秒）
  • 架構偏離率：每週違規次數 < 5 次
  • 自動修復率：80% 的違規由 Hook 自動攔截

8.2 Agent 效能

  • Context 利用率：Token cache hit rate > 70%
  • 任務成功率：95% 以上（現有 ~92%）
  • 平均執行時間：單任務 < 2 分鐘（現有 ~3 分鐘）

8.3 代碼品質

  • 測試覆蓋率：> 80%（已達成：1620 tests）
  • 架構一致性分數：100% 通過 Fitness Functions
  • 技術債務趨勢：每月減少 10%

十、結論

Harness Engineering 代表軟體工程範式的根本轉變：從「編寫代碼」到「設計約束與治理系統」。在 AI Agent 時代，最有效的團隊不是擁有最大模型的團隊，而是擁有最精密 Harness 系統的團隊。

核心啟示：

  1. Context 結構 > Prompt 措辭：花 80% 時間設計 context，20% 調整指令
  2. 快速反饋 > 定期審查：從季度審查轉向秒級反饋（Pre-commit Hooks）
  3. 機器執行 > 人工檢查：將架構決策轉化為 Fitness Functions
  4. 專門化 Agent > 全能 Agent：關注點分離，每個 Agent 只做一件事

對於 Daily Digest Prompt 專案，建議優先實施：

  1. 建立 Fitness Functions 測試套件（Phase 1）
  2. 優化 Context 分層載入（Phase 2）
  3. 持續監控與演進（Phase 4）

最終目標：建立自我修復、自我優化的 AI Agent 系統，透過 Harness 確保長期可靠性與可維護性。
