一、專案概述

Mastra（mastra-ai/mastra）是由 Gatsby 創始團隊打造的 TypeScript AI Agent 框架，定位為「The TypeScript AI Framework for building AI-powered applications and agents」。截至 2026 年 3 月已累積 19,400+ Stars、每週 300,000+ npm 下載量，採用 Apache-2.0 授權。

核心數據

| 指標 | 數值 |

|------|------|

| GitHub Stars | 19,400+ |

| npm 週下載量 | 300,000+ |

| 授權 | Apache-2.0 |

| 語言 | TypeScript |

| 創始團隊 | Gatsby 原班人馬 |

| Y Combinator | 已獲投資 |

| 支援 LLM 供應商 | 40+ |

為何選擇研究 Mastra

  1. 記憶系統直接對標：Mastra 的 Observational Memory 與本系統的 context/digest-memory.json 跨次記憶機制解決相同問題——如何在有限 context window 中保持長期記憶
  2. 工作流架構可比較：Mastra 的 DAG 工作流 vs 本系統的 Phase 0→1→2 管線式架構
  3. Skill/Tool 生態：Mastra 的 MCP Client + Tool 系統 vs 本系統的 Skill-First 策略
  4. 團隊血統：Gatsby 團隊有豐富的開發者工具經驗，設計決策值得深入分析

二、核心架構——四大支柱

Mastra 的架構圍繞四個核心原語（Primitives）建構：

2.1 Agents（智能代理）

  • 定義方式：model + system instructions + tools
  • 自主推理：Agent 自行決定使用哪些工具、何時停止
  • Supervisor 模式：支援主管代理委派子代理，可用 hooks 控制委派行為
  • 記憶整合：每個 Agent 可獨立配置記憶範圍（thread-scoped / resource-scoped）

const agent = new Agent({
  model: 'anthropic/claude-haiku-4-5',
  instructions: '你是一個研究助手...',
  tools: [searchTool, analyzeTool],
  memory: new Memory({ observationalMemory: true })
})

與本系統對照：本系統的 Agent 由 Prompt 文件定義（prompts/team/*.md），Mastra 用 TypeScript 物件。Mastra 的 Supervisor 模式類似本系統的 run-todoist-agent-team.ps1 Phase 分層調度。

2.2 Workflows（工作流）

  • 圖形狀態機：步驟組成有向圖（DAG），支援分支、迴圈、條件判斷
  • 暫停與恢復：工作流可在任意步驟暫停，等待外部輸入後恢復
  • 串流結果：支援步驟級別的即時結果串流
  • 強型別：完整的 TypeScript 型別推論，input/output schema 自動推導

與本系統對照：本系統的 config/pipeline.yaml 定義線性管線，Mastra 支援更靈活的 DAG。本系統的 FSM（state/run-fsm.json）是簡化版狀態機，Mastra 的工作流系統更為通用。

2.3 RAG（檢索增強生成）

  • 完整管線：文件 → 分塊 → 嵌入 → 向量儲存 → 查詢時檢索
  • 內建支援：無需外部依賴即可實現完整 RAG 流程

2.4 Evals（評估）

  • 內建評估框架：測試 Agent 效能、準確度、一致性
  • 可量化品質：為 Agent 行為提供可衡量的品質指標

三、Observational Memory（OM）——革命性記憶架構（核心洞察）

OM 是 Mastra 最具突破性的技術創新，也是本次研究的核心焦點。

3.1 設計哲學

OM 的核心理念：讓小 context window 表現得像大 context window。

傳統方法的問題：

  • RAG 記憶：需要向量資料庫，檢索結果不穩定（每次不同），破壞 prompt cache
  • 完整歷史：token 成本爆炸，長對話必然超出視窗
  • 截斷歷史：丟失早期重要資訊

OM 的解法：以觀察日誌取代原始訊息歷史，實現 5-40 倍壓縮率，同時保持 prompt 可快取性。

3.2 三層壓縮架構


┌─────────────────────────────────────┐
│ Tier 1: Recent Messages（原始訊息） │ ← 持續累積
│ 閾值：30,000 tokens               │
├─────────────────────────────────────┤
│ Tier 2: Observations（觀察日誌）    │ ← Observer Agent 壓縮
│ 閾值：40,000 tokens               │
├─────────────────────────────────────┤
│ Tier 3: Reflections（反思精華）     │ ← Reflector Agent 精煉
│ 垃圾回收不相關觀察                 │
└─────────────────────────────────────┘

運作機制：

  1. 累積階段：新訊息持續附加到 Recent Messages 區塊
  2. 觀察觸發：當訊息 token 達到 30K，Observer Agent 將訊息壓縮為結構化觀察日誌
  3. 反思觸發：當觀察日誌達到 40K token，Reflector Agent 進行垃圾回收，移除不相關觀察

3.3 觀察日誌格式

觀察採用文本式日誌格式（非結構化物件或知識圖譜），具備三個關鍵特徵：

  • 三日期模型：觀察日期、引用日期、相對日期（支援時間推理）
  • 優先級標記：🔴 關鍵 / 🟡 中等 / 🟢 資訊性
  • 時序排列：觀察按時間順序附加，非動態檢索

Date: 2026-01-15
- 🔴 12:10 User building Next.js app with Supabase auth, due January 22nd
  - 🔴 12:10 App uses server components with client-side hydration
- 🟡 14:30 User prefers functional components over class components

3.4 非同步緩衝機制

| 參數 | 預設值 | 用途 |

|------|--------|------|

| bufferTokens | 0.2（20%） | 緩衝間隔（~6K tokens） |

| bufferActivation | 0.8（80%） | 清除訊息時的保留比例 |

| blockAfter | 1.2（120%） | 安全閾值（36K 強制同步） |

3.5 Benchmark 表現

LongMemEval 基準測試（記憶持久性評估）：

| 系統 | 模型 | 分數 |

|------|------|------|

| Mastra OM | gpt-5-mini | 94.87% |

| Mastra OM | gemini-3-pro-preview | 93.27% |

| Mastra OM | gemini-3-flash-preview | 89.20% |

| Mastra OM | gpt-4o | 84.23% |

| Supermemory | gpt-4o | 81.63% |

| Full Context Baseline | gpt-4o | 60.20% |

  • 以 gpt-5-mini 達到 94.87%，超越所有先前紀錄 3+ 百分點
  • 以 gpt-4o 達到 84.23%，超越 Supermemory 同模型 2.6 百分點
  • 超越全量 context baseline（60.20%）達 24 百分點

3.6 Prompt Cache 友善設計

這是 OM 最精妙的設計：觀察日誌隨時間附加而非動態檢索，意味著 prompt 前綴保持穩定——每次對話回合都能獲得完整的 cache hit。僅在反思觸發時（不頻繁）才會使整個 cache 失效。

對比 RAG 記憶：每次檢索不同文檔片段，prompt 前綴不斷變化，無法有效利用 prompt cache。

四、四層記憶系統全景

| 層 | 名稱 | 用途 | 範圍 |

|---|------|------|------|

| 1 | Message History | 當前對話的原始訊息 | Thread-scoped |

| 2 | Observational Memory | 壓縮後的觀察日誌 | Thread / Resource-scoped |

| 3 | Working Memory | 持久化結構資料（姓名、偏好、目標） | Resource-scoped |

| 4 | Semantic Recall | 基於語義相似度的關聯回憶 | Cross-thread |

與本系統對照：

  • 本系統的 context/digest-memory.json ≈ Working Memory（持久化結構資料）
  • 本系統的 context/research-registry.json ≈ 簡化版 Observational Memory（研究主題追蹤）
  • 本系統缺乏 Semantic Recall 層（可透過知識庫 hybrid search 部分替代）
  • 本系統缺乏動態壓縮機制（context 超限時直接截斷而非壓縮）

六、對 Daily-Digest-Prompt 系統的六大借鑑啟示

啟示 1：觀察式記憶壓縮（P1，高價值）

現狀：本系統的 context/digest-memory.json 採用手動結構化格式，每次 session 由 Agent 自行判斷更新。

借鑑：引入「觀察日誌」概念，讓 on_stop_alert.py 在 session 結束時自動將本次互動壓縮為結構化觀察，附加到 context/session-observations.json。

具體方案：

  • 新增 hooks/session_observer.py，攔截 Stop 事件
  • 讀取本次 session 的 JSONL 日誌（logs/structured/），提取關鍵操作
  • 用優先級標記（🔴🟡🟢）分類觀察結果
  • 附加到觀察日誌，保留最近 30 天
  • 目標檔案：hooks/session_observer.py、context/session-observations.json

啟示 2：Prompt Cache 友善的記憶設計（P1，低成本高效益）

現狀：本系統的 prompt 每次都完整載入所有 YAML 配置和模板，前綴不穩定。

借鑑：將不變的配置內容固定在 prompt 最前端（前綴區），動態內容放在最後端。

具體方案：

  • 重新排列 prompt 結構：靜態規則（preamble.md）→ 靜態配置 → 動態上下文（放最後）
  • 確保 prompt 前綴在多次執行間保持一致
  • 預期效益：Anthropic prompt cache 命中率顯著提升，降低 token 成本

啟示 3：Working Memory 結構化（P2，中等工作量）

現狀：digest-memory.json 混合了狀態追蹤（連續天數）和偏好資料（用戶習慣）。

借鑑：參照 Mastra 將 Working Memory 獨立為「持久化用戶資料」，與 session 狀態分離。

具體方案：

  • 將 digest-memory.json 拆分為 context/user-profile.json（偏好、習慣）和 context/session-state.json（連續天數、完成率）
  • 不同更新頻率：profile 低頻更新，state 每次 session 更新

啟示 4：DAG 工作流取代線性管線（P2，需評估）

現狀：config/pipeline.yaml 定義線性 Phase 0→1→2 管線。

借鑑：Mastra 的 DAG 工作流支援分支、條件、暫停/恢復，更靈活處理異常路徑。

評估重點：

  • 本系統的線性管線足夠應對目前需求嗎？
  • 若引入 DAG，PowerShell 腳本的複雜度是否可控？
  • 建議：先在 run-todoist-agent-team.ps1 中試驗條件分支（如：Phase 2 任務依類型分流）

啟示 5：記憶可觀測性（P1，低成本）

現狀：本系統缺乏記憶使用追蹤，無法知道 Agent 實際參考了哪些記憶內容。

借鑑：Mastra 的 Tracing 功能顯示每次請求使用了哪些記憶。

具體方案：

  • 在 post_tool_logger.py 中記錄每次 Read context/*.json 的行為
  • 在 check-health.ps1 中新增「記憶使用頻率」報表
  • 追蹤哪些記憶檔案被頻繁讀取、哪些被忽略

啟示 6：Eval 整合——Agent 品質量化（P3，長期投資）

現狀：本系統依賴 done_cert 自評和 audit-scoring.yaml 人工審查。

借鑑：Mastra 的內建 Eval 框架可自動化測試 Agent 品質。

具體方案：

  • 建立 tests/evals/ 目錄，定義 Agent 輸出品質測試
  • 用 LongMemEval 等標準基準測試記憶系統效能
  • 整合到 run-system-audit-team.ps1 的審查流程中

七、技術決策比較矩陣

| 設計面向 | Mastra | Daily-Digest-Prompt | 啟示 |

|---------|--------|---------------------|------|

| 記憶壓縮 | Observer/Reflector 自動壓縮 | 手動結構化 JSON | 可引入自動壓縮 |

| Prompt Cache | 穩定前綴設計 | 動態載入配置 | 應重排 prompt 結構 |

| 工作流 | DAG 圖形狀態機 | 線性 Phase 管線 | 評估 DAG 必要性 |

| 記憶範圍 | Thread/Resource 雙範圍 | 全域單一 JSON | 可分離 profile/state |

| 工具整合 | MCP Client + TypeScript | Skill-First + Bash/PS | 設計理念相似 |

| 評估 | 內建 Eval 框架 | done_cert 自評 | 可引入自動化品質測試 |

| 型別安全 | 完整 TypeScript | YAML + Markdown | 各有優勢 |

| 可觀測性 | Tracing 顯示記憶使用 | JSONL 日誌 | 可增加記憶追蹤 |

八、關鍵數字摘要

  • 5-40x 訊息壓縮率（尤其 Playwright MCP 場景 50K→幾百 tokens）

  • 4 層 記憶系統（Message → Observation → Working → Semantic）
  • 3 種 儲存後端（PostgreSQL、MongoDB、libSQL）
