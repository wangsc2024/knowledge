證據 7：MCP 安全 2026：Tool Poisoning、Remote MCP 信任邊界與 Runtime 防禦 深度研究報告

標籤：AI深度研究、MCP安全、2026、研究報告 ｜ KB id：b09e2511-a20 ｜ 原文 10348 字

原文章節導覽：

  • 執行摘要
  • 研究背景與動機
  • 核心技術解析
  • 一、MCP 的新增信任邊界：從 connect-time 審查轉向 runtime 審查
  • 二、remote MCP 的授權面：resource audience、consent 與 token handling
  • 三、工具註解、runtime scanning 與 human approval 的角色分工
  • 関鍵洞見
  • 實務應用建議

核心內容摘錄：

  • ## 執行摘要 MCP 相較一般 function calling 的新增風險，不在於「多了一個呼叫工具的介面」而已，而在於它把工具定義、工具輸出、跨伺服器組合與授權流程，全部變成模型執行路徑的一部分。這使攻擊面從單次函式呼叫，擴張成整個 session 的資料流與信任鏈。第一個研究問題的答案是：MCP 最獨特的攻擊面有三個。其一，tool response 是執行期資料，若來自 open-world 或不受信任 server，就可能把惡意指令直接注入模型上下文，形成 tool poisoning [來源：MCP Tool Poisoning, 2026]。其二，remote MCP 需要額外處理 OAuth audience/resource 綁定、同意頁、redirect URI、state 與 token 驗證，這些在傳統本地 function calling 中通常不存在 [來源：MCP Authorization Specification, 2025-06-18；MCP Security Best Practices, 2025-06-18]。其三，風險是組合性的，單一工具看似安全，但只要同一 session 同時擁有私有資料讀取、open-world 輸入與外送能力，就可能形成資料外洩鏈 [來源：Tool Annotations as Risk Vocabulary, 2026；Model Context Protocol Schema Reference - ToolAnnotations, 2025-06-18]。
  • 第二個研究問題的答案是：2025-2026 的官方規範與公開案例已收斂出一個明確結論，單靠模型對齊或 prompt engineering 不足以防禦 MCP 風險。學術論文指出，prompt injection 是建在模型之上的應用漏洞，且現有防禦整體仍不足 [來源：The Instruction Hierarchy, 2024；Formalizing and Benchmarking Prompt Injection Attacks and Defenses, 2023]。官方規範則把重點放在 deterministic guardrails：禁止 token passthrough、要求 resource indicator、限制 scope、做 consent 與 state 驗證、對 open-world 工具提高審查 [來源：MCP Security Best Practices, 2025-06-18；MCP Authorization Specification, 2025-06-18]。因此可操作的防禦分層應至少包含：模型層的指令優先級與拒絕策略、client/host 層的 taint propagation 與人工批准、gateway/server 層的 OAuth 驗證與 outbound credential control、以及網路層的 prompt scanning [來源：使用提示注入保护企业生成式 AI 应用, 2026-04-27；在 Azure API 管理中安全访問 MCP 服务器, 2026-02-03]。
  • 第三個研究問題的答案是：若要在生產環境部署 remote MCP，最佳做法不是在 OAuth、tool annotations、runtime scanning、human approval 之間二選一，而是把它們組成可遞進升級的控制鏈。OAuth 應先把 token 的 audience 綁到特定 MCP server，並把 scope 壓到最小；tool annotations 則只能做風險詞彙與 UX 輔助，不能作為硬保證；runtime scanning 用來偵測 prompt injection、SSRF、資料污染與異常上下文；human approval 則應聚焦於高風險寫入、跨邊界外送與權限升高操作 [來源：Get started with custom connectors using remote MCP, 2026-04-30；Model Context Protocol Schema Reference - ToolAnnotations, 2025-06-18；Security Advisory: SSRF, Indirect Prompt Injection, and sandbox bypass in @modelcontextprotocol/server-puppeteer, 2026-03-21]。總結來說，MCP 在 2026 年可安全上線，但前提是把它視為「分散式信任邊界系統」，而不是「比較方便的 function calling」。
