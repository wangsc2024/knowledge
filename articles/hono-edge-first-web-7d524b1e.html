一、專案概述

Hono（日語「炎 🔥」）是由 Yusuke Wada 創建的超輕量 Web 框架，建構於 Web Standards（Request/Response/fetch API）之上，實現「寫一次、跑所有 JavaScript Runtime」的跨平台願景。支援 Cloudflare Workers、Deno、Bun、Node.js、AWS Lambda、Fastly Compute、Vercel Edge 等 8+ 執行環境。

二、核心架構創新

2.1 四層路由器架構（Router Hierarchy）

Hono 最獨特的技術創新是其四種可切換路由器，針對不同場景最佳化：

| 路由器 | 原理 | 適用場景 | 效能特徵 |

|--------|------|----------|----------|

| RegExpRouter | 將所有路由編譯為單一大型正則表達式 | 路由固定的生產環境 | O(1) 匹配，最速 |

| SmartRouter | 自動選擇最佳路由策略 | 預設路由器（推薦） | 智能調度 |

| LinearRouter | 線性註冊、快速啟動 | 每次請求重新初始化的 Serverless | 啟動速度最快 |

| PatternRouter | 純模式匹配 | Bundle 大小極度敏感 | 最小體積 |

RegExpRouter 技術深度：傳統路由器逐條比對路由表（O(n)），RegExpRouter 在啟動時將全部路由編譯成一個 Trie 結構的正則表達式，透過 JavaScript 引擎的正則最佳化實現 O(1) 路由匹配。這是 Hono 在 Cloudflare Workers 上達到 402,820 ops/sec 的核心祕密——比 itty-router（212,598）快 1.9 倍，比 worktop（197,345）快 2.0 倍。

2.2 Web Standards 統一層

Hono 不使用 Node.js 的 http.createServer 或 Deno 的 Deno.serve 等 Runtime 專屬 API，而是完全基於 WHATWG Fetch API 標準：


// 同一份程式碼，跑在任何 Runtime
export default {
  fetch(request: Request): Response {
    const app = new Hono()
    app.get('/', (c) => c.text('Hello Hono!'))
    return app.fetch(request)
  }
}

架構啟示：Web Standards 作為「最大公約數」統一 API，讓框架完全脫離 Runtime 耦合。這與 daily-digest-prompt 的「Skill-First 策略」異曲同工——技能定義與執行環境解耦。

2.3 洋蔥模型中介層（Onion Middleware）


           請求 →
     ┌─────────────────┐
     │  Middleware A    │
     │  ┌─────────────┐ │
     │  │ Middleware B │ │
     │  │ ┌─────────┐ │ │
     │  │ │ Handler │ │ │
     │  │ └─────────┘ │ │
     │  └─────────────┘ │
     └─────────────────┘
           ← 回應

中介層執行順序：A(before) → B(before) → Handler → B(after) → A(after)。透過 c.set() / c.get() 在中介層間傳遞型別安全的上下文資料：


type Env = { Variables: { user: User } }
const authMiddleware = createMiddleware<Env>(async (c, next) => {
  const user = await verifyToken(c.req.header('Authorization'))
  c.set('user', user)  // 型別安全！
  await next()
})

三、RPC 端對端型別安全（零程式碼生成）

Hono 最具革命性的功能是不需要 Code Generation 即可實現前後端完整型別推論：


// 後端：定義 API 並匯出型別
const app = new Hono()
  .get('/users/:id', zValidator('param', z.object({ id: z.string() })), (c) => {
    return c.json({ name: 'John', age: 30 })
  })
export type AppType = typeof app

// 前端：完整型別推論，零生成
import { hc } from 'hono/client'
const client = hc<AppType>('http://localhost:3000')
const res = await client.users[':id'].$get({ param: { id: '1' } })
// res 的型別自動推論為 { name: string, age: number }

技術關鍵：透過 TypeScript 的 Template Literal Types + Conditional Types，Hono 在編譯時從路由定義中萃取路徑參數、請求 body schema、回應型別，不需要 OpenAPI spec 或 protobuf 定義檔。

與 tRPC 的差異：tRPC 需要專屬 client/server 協議；Hono RPC 底層就是標準 HTTP，可搭配任何 HTTP client 使用，同時額外提供型別安全的 hc client。

五、最佳實踐（官方推薦）

5.1 避免 Rails 風格的 Controller


// ❌ 不推薦：分離 Controller 會斷失型別推論
class UserController {
  static getUser = (c) => c.json({ name: 'John' })
}
app.get('/users/:id', UserController.getUser)

// ✅ 推薦：Handler 直接跟路由寫在一起
app.get('/users/:id', (c) => {
  const id = c.req.param('id')  // 型別自動推論為 string
  return c.json({ name: 'John' })
})

5.2 大型應用的模組拆分


// authors.ts
const authors = new Hono()
  .get('/', (c) => c.json([]))
  .post('/', (c) => c.json({ created: true }))
export default authors

// app.ts
const app = new Hono()
  .route('/authors', authors)
  .route('/books', books)
export type AppType = typeof app

5.3 Cloudflare Workers 環境綁定


type Bindings = {

  DB: D1Database

const app = new Hono<{ Bindings: Bindings }>()
app.get('/data', (c) => {

六、可借鏡的技術項目（對 daily-digest-prompt 的啟示）

6.1 RegExpRouter 的「編譯期最佳化」思維

啟示：Hono 在啟動時將動態路由表預編譯為靜態正則表達式，避免執行期的線性搜尋。本專案的 config/routing.yaml Todoist 三層路由目前在每次請求時逐條比對，可借鏡 RegExpRouter 的「一次編譯、多次匹配」策略——在 Phase 0 將路由規則預編譯為決策樹，Phase 1 直接查表。

6.2 中介層組合模式 → Hook 系統演進

啟示：Hono 的洋蔥模型中介層支援 c.set()/c.get() 型別安全上下文傳遞。本專案的 Hook 系統（pre_bash_guard → post_tool_logger）目前透過 stdin JSON 傳遞決策，缺乏跨 Hook 的型別安全上下文鏈。可借鏡 Hono 的 Variables 泛型模式，在 Hook 間建立型別安全的上下文傳遞協議。

6.3 四種路由器可切換策略 → Skill 路由分層

啟示：Hono 根據場景提供四種路由器（速度優先/體積優先/啟動速度優先/智能選擇）。本專案的 Skill 路由目前只有一種策略，可借鏡引入分層路由——「快速精確匹配」（exact label match）→「模糊語義匹配」（keyword scoring）→「預設 fallback」。

6.4 Web Standards 作為統一抽象層

啟示：Hono 用 Web Standards（Request/Response）作為跨 Runtime 的統一介面。本專案的 Skill 系統在不同執行模式（single/team/auto-task）間缺乏統一介面定義。可借鏡定義 Skill 的「標準輸入/輸出介面」（如 SkillRequest/SkillResponse），讓同一個 Skill 在不同模式下皆可重用。

6.5 RPC 型別安全（零程式碼生成）→ 結果檔案 Schema 治理

啟示：Hono 用 TypeScript 型別推論取代 Code Generation 實現端對端型別安全。本專案目前以 JSON 檔案在 Phase 間傳遞結果，缺乏 Schema 驗證。可借鏡 Hono 的 Zod Validator 模式，在 validate_config.py 中加入結果檔案的 Zod-like Schema 驗證。
