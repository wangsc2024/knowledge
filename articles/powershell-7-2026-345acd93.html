PowerShell 7 生產環境錯誤處理與自動恢復設計模式：結構化例外、重試策略與有限狀態機實踐（2026）

PowerShell 7 生產環境錯誤處理與自動恢復設計模式：結構化例外、重試策略與有限狀態機實踐（2026）

技術概述

PowerShell 7 的錯誤處理架構結合 .NET 例外系統與 PowerShell 特有的 ErrorRecord 模型，提供三層防護：結構化例外捕捉（try/catch/finally）、外部程序退出碼整合（$PSNativeCommandUseErrorActionPreference，7.4+ 穩定版）、以及有限狀態機（FSM）工作流程狀態追蹤。掌握這三層機制是建立生產可靠度的基礎，特別適用於以 pwsh 7 為執行環境的自動化管線（如 Windows Task Scheduler + Claude Code CLI 架構）。

核心概念與原理

1. 三類錯誤分類

PowerShell 7 將錯誤分為三種類型 [來源: Microsoft Learn, 2026]：

| 類型 | 行為 | 觸發來源 |

|------|------|----------|

| Non-terminating | 輸出到錯誤流，腳本繼續 | Write-Error（預設）|

| Statement-terminating | 停止當前陳述式，腳本繼續下一條 | 引擎錯誤、$PSCmdlet.ThrowTerminatingError()、.NET method exception |

| Script-terminating | 展開完整 call stack，腳本停止 | throw 關鍵字、parse error、-ErrorAction Stop 升格 |

關鍵陷阱⚠️：try/catch 只能捕捉 terminating errors。若要捕捉 non-terminating 錯誤（如 Write-Error），必須搭配 -ErrorAction Stop。

2. ErrorRecord 結構

catch 區塊內的 $PSItem（即 $_）是 ErrorRecord 物件，核心屬性 [來源: Microsoft Learn, KevinMarquette, 2026]：


catch {
    $_.Exception.Message                   # 例外訊息字串
    $_.Exception.InnerException            # 內層例外（包裝例外時）
    $_.InvocationInfo.ScriptName           # 發生錯誤的腳本路徑
    $_.InvocationInfo.ScriptLineNumber     # 錯誤行號
    $_.ScriptStackTrace                    # 函式呼叫堆疊（PS 層）
    $_.Exception.StackTrace                # .NET managed code 堆疊
}

3. 類型化例外與多重 Catch

優先使用類型化例外，讓 catch 精確比對；子類型必須排在父類型之前 [來源: Microsoft Learn, 2026]：


try {
    Get-Item $path -ErrorAction Stop
}
catch [System.IO.FileNotFoundException] {
    Write-Warning "找不到檔案: $path"
}
catch [System.IO.IOException] {
    Write-Warning "IO 錯誤: $path"
}
catch {
    # 未預期例外：記錄後重拋，保留 call stack
    Write-Error "未預期例外: $($_.Exception.Message)"
    throw  # 重拋保留原始 InvocationInfo
}

⚠️ 避免空白 catch（swallowing exceptions）：靜默吞掉例外是最常見的除錯地獄根源。

4. $PSNativeCommandUseErrorActionPreference（PowerShell 7.4+）

外部程序（native commands：claude.exe、curl、python）不參與 PowerShell 的例外系統，只透過 $LASTEXITCODE 傳遞退出碼。PowerShell 7.4 穩定化了 $PSNativeCommandUseErrorActionPreference [來源: GitHub PR #18695; Matt Horgan's Blog]：


# 放在腳本頂部（需 PowerShell 7.4+）
$PSNativeCommandUseErrorActionPreference = $true
$ErrorActionPreference = 'Stop'

try {
    # non-zero exit code 會觸發 NativeCommandExitException
    claude -p "task prompt" --output-format json
}
catch [System.Management.Automation.NativeCommandExitException] {
    $exitCode = $_.Exception.ExitCode
    Write-Error "Claude CLI 失敗，exit code=$exitCode: $($_.Exception.Message)"
}

7.3 vs 7.4 差異：7.3 為實驗性功能（需手動開啟），7.4 正式穩定，預設值仍為 $false（向後相容）。

5. $PSCmdlet.ThrowTerminatingError() — 指向函式而非 throw 行

在 Advanced Function 內，使用 $PSCmdlet.ThrowTerminatingError() 而非裸 throw，錯誤訊息會指向函式名稱而非 throw 行號，提供更有意義的錯誤定位 [來源: Microsoft Learn, Kirk Munro 建議, 2026]：


function Invoke-CriticalOperation {
    [CmdletBinding()]
    param([string]$Path)
    process {
        try {
            # ... 業務邏輯
        }
        catch {
            # 清理內部細節，讓呼叫者看到函式名稱而非行號
            $PSCmdlet.ThrowTerminatingError($PSItem)
        }
    }
}

Kirk Munro 建議模式：每個 Advanced Function 的 begin/process/end 區塊都包一層 try/catch。

最佳實踐與陷阱

重試策略：指數退避 + Jitter

不是所有錯誤都應重試。可重試類型：網路錯誤、速率限制（429）、服務暫時不可用（503）。不可重試類型：認證失敗（401/403）、資源不存在（404）、業務邏輯錯誤 [來源: DEV Community, Retry Patterns 2026]。

生產級重試函式（帶指數退避 + Jitter）[來源: Daniel Schroeder's Blog; DEV Community 2026]：


function Invoke-WithRetry {
    [CmdletBinding()]
    param(
        [scriptblock] $ScriptBlock,
        [int] $MaxAttempts = 5,
        [int] $BaseDelayMs = 1000,
        [double] $BackoffMultiplier = 2.0,
        [int] $MaxDelayMs = 30000,
        [string[]] $NonRetryableErrors = @()
    )
    $attempt = 0
    while ($true) {
        try {
            $attempt++
            return (& $ScriptBlock)
        }
        catch {
            $errorMsg = $_.Exception.Message
            # 不可重試錯誤直接拋出
            foreach ($pattern in $NonRetryableErrors) {
                if ($errorMsg -match $pattern) { throw }
            }
            if ($attempt -ge $MaxAttempts) {
                Write-Error "已達最大重試次數 ($MaxAttempts)，最後錯誤: $errorMsg"
                throw
            }
            # 指數退避 + Jitter（防止多實例重試碰撞）
            $delay = [Math]::Min($BaseDelayMs * [Math]::Pow($BackoffMultiplier, $attempt - 1), $MaxDelayMs)
            $jitter = Get-Random -Minimum 0 -Maximum ([int]($delay * 0.2))
            Start-Sleep -Milliseconds ([int]($delay + $jitter))
        }
    }
}

# 使用範例：Todoist API 呼叫
Invoke-WithRetry -ScriptBlock {
    Invoke-RestMethod -Uri 'https://api.todoist.com/api/v1/tasks/filter?query=today' \
        -Headers @{ Authorization = "Bearer $env:TODOIST_API_TOKEN" } -ErrorAction Stop
} -MaxAttempts 5 -NonRetryableErrors @('401', '403', 'Unauthorized')

Jitter 的重要性：若多個 Agent 實例同時失敗後以相同延遲重試，會形成 retry storm（重試風暴）。Jitter 在延遲基礎上加入 ±20% 隨機量，分散重試時機。[來源: DEV Community, Retry Patterns 2026]

作用域管理：全局 vs 每命令

⚠️ 陷阱：在腳本頂部設 $ErrorActionPreference = 'Stop' 影響所有命令，包含 Write-Error 和第三方 cmdlet，可能造成意外中斷。

推薦做法 [來源: Fortra Blog, 2025]：

  • 重要操作用 -ErrorAction Stop（per-command）
  • 全局設定改用 $ErrorActionPreference = 'Continue'（預設值）
  • 只在 CI/CD 腳本中用全局 Stop，此時「中止 = 預期行為」

常見陷阱清單

  1. ❌ 空白 catch 吞掉例外 → ✅ 至少 log 後重拋
  2. ❌ 忽略 $LASTEXITCODE（外部程序失敗靜默繼續）→ ✅ 使用 $PSNativeCommandUseErrorActionPreference=$true
  3. ❌ catch 順序錯誤（父類型在前）→ ✅ 子類型優先
  4. ❌ 全局 Stop 加 third-party cmdlet → ✅ per-command -ErrorAction Stop
  5. ❌ 重試認證失敗 → ✅ NonRetryableErrors 過濾 401/403

有限狀態機（FSM）設計模式

PowerShell 7 不支援 PS5.1 的舊版 Workflow，但可用 Enum + Hashtable 轉換表 模式在純腳本環境實作 FSM [來源: Microsoft Learn hashtables, 4sysops enums]。

核心結構


# 狀態定義（enum 提供型別安全與 IDE 自動補全）
enum PipelineState { Idle; Fetching; Processing; Assembling; Done; Failed }

# 轉換表：(CurrentState_Event) → NextState
$transitions = @{
    'Idle_start'           = [PipelineState]::Fetching
    'Fetching_complete'    = [PipelineState]::Processing
    'Fetching_error'       = [PipelineState]::Failed
    'Processing_complete'  = [PipelineState]::Assembling
    'Processing_error'     = [PipelineState]::Failed
    'Assembling_complete'  = [PipelineState]::Done
    'Assembling_error'     = [PipelineState]::Failed
}

function Set-FsmState {
    param(
        [ref] $CurrentState,
        [string] $Event,
        [hashtable] $TransitionTable,
        [string] $StateFile = $null
    )
    $key = "$($CurrentState.Value)_$Event"
    $nextState = $TransitionTable[$key]
    if ($null -eq $nextState) {
        throw "非法 FSM 轉換: $key（當前=$($CurrentState.Value), 事件=$Event）"
    }
    Write-Verbose "FSM: $($CurrentState.Value) --[$Event]--> $nextState"
    $CurrentState.Value = $nextState
    # 持久化（防 Agent crash 後需殭屍偵測）
    if ($StateFile) {
        @{ state = "$nextState"; timestamp = (Get-Date -Format 'o') } |
            ConvertTo-Json | Set-Content $StateFile -Encoding utf8
    }
}

殭屍偵測（Zombie Detection）

長時間停在 running 狀態通常代表 Agent crash 或掛起 [來源: 本專案 run-todoist-agent-team.ps1 Set-FsmState 設計]：


function Test-ZombieState {
    param([string]$StateFile, [int]$TimeoutMinutes = 120)
    if (-not (Test-Path $StateFile)) { return $false }
    $d = Get-Content $StateFile | ConvertFrom-Json
    $elapsed = ([DateTimeOffset]::Now - [DateTimeOffset]::Parse($d.timestamp)).TotalMinutes
    return ($d.state -eq 'running' -and $elapsed -gt $TimeoutMinutes)
}

# 使用：排程任務啟動前檢查
if (Test-ZombieState -StateFile 'state\pipeline.json' -TimeoutMinutes 120) {
    Write-Warning '殭屍狀態偵測：強制重置'
    Remove-Item 'state\pipeline.json' -Force
}

FSM 執行迴圈（整合 try/catch）


$state = [PipelineState]::Idle
$stateRef = [ref] $state

try {
    Set-FsmState -CurrentState $stateRef -Event 'start' -TransitionTable $transitions -StateFile 'state\pipeline.json'
    
    # Phase 1: Fetching（帶重試）
    try {
        $data = Invoke-WithRetry { Invoke-RestMethod $todoistUri -Headers $headers -ErrorAction Stop }
        Set-FsmState -CurrentState $stateRef -Event 'complete' -TransitionTable $transitions -StateFile 'state\pipeline.json'
    } catch {
        Set-FsmState -CurrentState $stateRef -Event 'error' -TransitionTable $transitions -StateFile 'state\pipeline.json'
        throw
    }
    # Phase 2, 3...
}
catch {
    Write-Error "管線失敗: $($_.Exception.Message)，最終狀態: $($stateRef.Value)"
}

與本專案關聯

  1. run-todoist-agent-team.ps1 的 Set-FsmState：已採用殭屍偵測（running>2h→timeout）+ 選擇性清理模式，直接對應上述 FSM 設計。
  2. $PSNativeCommandUseErrorActionPreference：Invoke-LlmGate / Start-CursorCliJob 呼叫 claude -p 時使用 7.4+ 特性可統一捕捉 claude CLI 非零退出碼。
  3. Invoke-WithRetry：可應用於 ntfy.sh 推送、Todoist API（429 速率限制）、KB import，搭配 NonRetryableErrors 過濾認證錯誤。
  4. 跨語言可觀測性：post_tool_logger.py 的 JSONL 日誌 + PowerShell $_.InvocationInfo 組成跨 PS + Python 的完整 call stack 追蹤鏈。

程式碼範例：完整管線守衛模板


#Requires -Version 7.4
Set-StrictMode -Version Latest
$PSNativeCommandUseErrorActionPreference = $true

function Invoke-PipelineWithGuard {
    [CmdletBinding()]
    param([string] $TaskKey, [scriptblock] $Pipeline)
    
    $startTime = Get-Date
    $result = @{ task_key=$TaskKey; status='failed'; error='unexpected' }
    
    try {
        $ErrorActionPreference = 'Stop'
        & $Pipeline
        $result.status = 'success'
        $result.Remove('error')
    }
    catch [System.Management.Automation.NativeCommandExitException] {
        $result.error = "外部命令失敗 exit=$($_.Exception.ExitCode): $($_.Exception.Message)"
    }
    catch [System.IO.IOException] {
        $result.error = "IO 錯誤: $($_.Exception.Message)"
    }
    catch {
        $result.error = "$($_.Exception.GetType().Name): $($_.Exception.Message)"
    }
    finally {
        $result.duration_seconds = [int](((Get-Date) - $startTime).TotalSeconds)
        $result | ConvertTo-Json | Set-Content "results\todoist-auto-$TaskKey.json" -Encoding utf8
    }
}
