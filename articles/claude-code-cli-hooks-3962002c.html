Claude Code CLI hooks、slash commands 與 skills 的擴充架構：以 daily-digest-prompt 為例

技術概述

今天已完成任務可辨識的技術主題包括：Claude Code、Agent Skills、hooks、slash commands、subagents、Superpowers、oh-my-claudecode。其中最值得延伸的是 Claude Code 的擴充架構，因為它直接關係到本專案如何把既有 skills/、workflows/、自動任務與守衛腳本整合成更原生的 Claude Code 能力層。官方文件顯示，Claude Code 已把舊式 custom commands 併入 skills，並讓 skills、hooks、subagents 與 plugins 形成一套可組合的擴充面；第三方框架如 Superpowers 也採用了 commands/、hooks/、skills/、agents/ 並列的 repo 結構，證明這個分層在真實團隊流程中可行 [來源：Extend Claude with skills, 2026] [來源：Hooks reference, 2026] [來源：Superpowers README, 2026]。

核心主張與原理

核心主張 1：在 2026 的 Claude Code 中，skills 已成為主要擴充單位，舊的 custom commands 只是相容入口，不再是獨立能力層。 ✅

Claude Code 官方明確說明 custom commands 已併入 skills；.claude/commands/deploy.md 與 .claude/skills/deploy/SKILL.md 都能建立同名 /deploy，但 skills 額外支援 supporting files、invocation control、subagent execution 與 dynamic context injection，因此新設計應優先以 skill directory 為中心，而不是再新增大量單檔 command [來源：Extend Claude with skills, 2026]。Slash commands 文件則補充，custom slash commands 仍以 Markdown file 暴露，但可被 SlashCommand tool 程式化觸發，並能透過 disable-model-invocation: true 從模型可見上下文中移除 [來源：Slash commands, 2026]。Anthropic 的 Skill authoring best practices 進一步指出，Skill metadata 會先被預載、正文按需載入，這解釋了為什麼把邏輯搬進 skill 而不是散落在大量 prompt/command 檔，更有利於可發現性與 context 控制 [來源：Skill authoring best practices, 2026]。

核心主張 2：hooks 適合做決定性、事件驅動的 guardrail 與自動化，而不適合承擔需要長推理與大量上下文的工作。 ✅

官方 hooks reference 把 hooks 定義為在 Claude Code lifecycle 指定節點自動執行的 shell command、HTTP endpoint 或 prompt/agent handler，並列出 PreToolUse、PostToolUse、SubagentStart、TaskCreated 等大量事件 [來源：Hooks reference, 2026]。同一份文件展示了 PreToolUse 阻擋 rm -rf 的實例，重點是 matcher、if 規則與 JSON 決策回傳；這種模式本質上是 policy enforcement，而不是開放式任務執行 [來源：Hooks reference, 2026]。此外，hooks 可設 async: true 背景執行，也可在 skill 與 agent frontmatter 內做生命週期綁定，表示它最適合「驗證、記錄、阻擋、收尾」這類 deterministic control plane，而不是像研究、摘要、跨檔分析這種需要主代理或 subagent 的資料平面工作 [來源：Hooks reference, 2026] [來源：Create custom subagents, 2026] [來源：Claude Code Hooks 進階實戰指南, 2026]。

核心主張 3：subagents 與 `context: fork` skill 是處理高 I/O 與多步驟工作的主要隔離手段，應用來承接研究、規劃與大型檢查。 ✅

官方 subagents 文件指出，每個 subagent 都在獨立 context window 中運作，擁有自訂 system prompt、工具限制與獨立權限，Claude 會根據 description 自動委派 [來源：Create custom subagents, 2026]。另一方面，skills 文件說明只要在 skill frontmatter 加上 context: fork 與 agent，該 skill 就會在隔離的 subagent 中執行，且 skill content 直接成為 subagent prompt；對研究型技能，官方甚至以 deep-research + Explore 當例子 [來源：Extend Claude with skills, 2026]。這與 Anthropic 的最佳實踐相符：Skill 應保持簡潔、透過 progressive disclosure 載入額外檔案，把重內容與腳本延後到真正需要時再讀，避免主上下文膨脹 [來源：Skill authoring best practices, 2026]。

核心主張 4：實務上，最穩定的架構不是只用單一擴充點，而是把 slash command、skill、hook、subagent 各自放到不同責任層。 ✅

Slash commands 官方定位是互動期常用模板與快捷入口，例如參數化命令、預先執行 ! shell 取得上下文、或由 SlashCommand tool 觸發重複性工作 [來源：Slash commands, 2026]。Skills 官方定位則偏向多檔、多步驟、可攜帶腳本與 supporting files 的能力包，甚至可直接作為 /skill-name 命令使用 [來源：Extend Claude with skills, 2026]。Hooks 是 lifecycle 自動化與 guardrails；subagents 則負責 context isolation、read-only exploration 或多步驟執行 [來源：Hooks reference, 2026] [來源：Create custom subagents, 2026]。Superpowers 的 repo 也把 commands/、hooks/、skills/、agents/ 並列維護，說明成熟框架傾向把「入口」、「能力」、「事件控制」、「委派執行」拆開，而不是全部塞進單一 prompt 層 [來源：Superpowers README, 2026]。

最佳實踐與常見陷阱

實作上，建議把責任切成四層：

  1. slash command：給人直接呼叫的短任務入口，例如 /run-daily-agent、/review-results。需要副作用時加 disable-model-invocation: true，避免模型自行觸發部署或批次修改 [來源：Extend Claude with skills, 2026] [來源：Slash commands, 2026]。
  2. skill：封裝多步驟工作流、 supporting files、腳本與參數。SKILL.md 只保留導覽與決策規則，大型參考資料拆到 reference.md 或 scripts/，符合 progressive disclosure 原則 [來源：Skill authoring best practices, 2026] [來源：Extend Claude with skills, 2026]。
  3. hook：做 preflight 驗證、結果檔檢查、寫檔後 lint、通知與 policy block。PreToolUse 適合阻擋危險命令，PostToolUse 適合做格式驗證，SubagentStart/Stop 適合執行準備與清理 [來源：Hooks reference, 2026]。
  4. subagent 或 context: fork skill：處理研究、repo 探索、批次審查、報告彙整等高 I/O 任務，避免主對話塞滿原始搜尋與中間分析 [來源：Create custom subagents, 2026] [來源：Extend Claude with skills, 2026]。

常見陷阱有四個。第一，把 hooks 當成研究引擎，會造成超時與除錯困難，因為 hooks 的設計中心是事件攔截與簡短決策，不是長推理。第二，把所有邏輯都留在單一 SKILL.md，違反 Anthropic 建議的簡潔與分檔原則，會增加 token 壓力 [來源：Skill authoring best practices, 2026]。第三，在 Windows 環境忽略 shell 差異；skills 的 shell: powershell 需要 CLAUDE_CODE_USE_POWERSHELL_TOOL=1，但 hooks 的 shell: "powershell" 是直接由 hooks 執行器啟動，行為不同，實作時必須分開測 [來源：Extend Claude with skills, 2026] [來源：Hooks reference, 2026]。第四，把 command、skill、subagent 命名得太模糊，會降低自動發現與委派準確度；官方建議 description 要具體且第三人稱，名稱需一致且可搜尋 [來源：Skill authoring best practices, 2026]。

與本專案的關聯

對 daily-digest-prompt，最合理的映射如下：

  • 把目前 prompts/team/todoist-auto-*.md 常用入口，逐步收斂成可人工啟動的 project skills，例如 run-tech-research、import-kb-note、validate-result-json。
  • 把 results/、context/continuity/、research-registry 驗證改成 hooks：PostToolUse 驗證 JSON、Stop 或 TaskCompleted 觸發摘要清理。
  • 把大量探索工作改成 context: fork skills 或專用 subagents，例如 kb-researcher、registry-checker、result-validator，降低主對話 I/O。
  • 保留少量 slash commands 給開發者手動使用，例如 /daily-debug、/regen-result、/kb-healthcheck；它們應是入口，不應承載全部邏輯。

這樣可以讓目前專案已經存在的 skills/、workflows/、PowerShell 腳本與結果 JSON 守衛，逐步對齊 Claude Code 原生的擴充模型，而不是持續依賴大型 prompt 模板手工拼裝。

程式碼範例


---
name: run-tech-research
description: 執行技術研究工作流並輸出結果 JSON。Use when running the daily tech research task.
disable-model-invocation: true
context: fork
agent: general-purpose
allowed-tools: Read Write Bash WebSearch WebFetch
hooks:
  PostToolUse:
    - matcher: "Write"
      hooks:
        - type: command
          command: "${CLAUDE_PROJECT_DIR}/.claude/hooks/validate-result-json.ps1"
          shell: "powershell"
---

1. 讀取今日任務與 research registry。
2. 研究主題去重。
3. 執行深入研究並寫入結果。

上例展示了一個更適合本專案的 pattern：用 skill 當工作流入口、用 context: fork 隔離研究上下文、用 hook 在寫檔後驗證結果 JSON，而不是把驗證與研究都塞在同一段 prompt 中 [來源：Extend Claude with skills, 2026] [來源：Hooks reference, 2026]。
