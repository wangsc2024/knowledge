QA System安全優化測試修復品質改善里程碑

QA System 安全優化報告 #16

日期：2026-02-26 | 品質分數：5/5 | 第十六次掃描

優化方向

修復 test_create_question_api 和 test_close_question_api 既有測試失敗（持續 11 次迭代的技術債）

已修復問題（2 個測試，6 個根因）

test_create_question_api（3 個根因）

  1. 部門代碼不符端點要求：測試建立代碼 1001/1002，但 create_question 端點檢查 code.endswith('00')（處層級部門）→ 改為 1000/2000
  2. 回應欄位斷言錯誤：測試預期 res_data['title'] 和 res_data['id']，但端點實際返回 {'success': True, 'question_id': ...} → 修正斷言

test_close_question_api（3 個根因）

  1. HTTP 方法錯誤：測試用 client.post() 但端點是 @router.put() → 改為 client.put()
  2. 缺少 JSON body：端點用 await request.json() 讀取 summary，測試未發送 → 加 json={'summary': '...'}
  3. ORM Enum 不相容：端點用 raw SQL 寫入字串 'closed'，db_session.refresh(q) 觸發 SQLAlchemy Enum KeyError → 改用 raw SQL SELECT 驗證狀態

測試結果

  • 修復前：103 測試中 101 通過、2 失敗（從 #5 到 #15 持續 11 次迭代）
  • 修復後：103/103 全數通過（零失敗）

安全評估

全綠燈：SECRET_KEY 環境變數化、SQL 注入防護、XSS 防護、CORS 限制、Cookie 安全、JWT 集中解碼、CVE 修復、錯誤訊息零洩漏

剩餘議題

  • zeep cgi 模組棄用警告（Python 3.13 將移除）
  • questions.py 進一步模組拆分（仍 884 行）
  • close_question 端點 raw SQL 與 ORM Enum 不一致
