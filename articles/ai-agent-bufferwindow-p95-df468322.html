AI Agent 記憶壓縮實作模式：BufferWindow 三層架構與 P95 監控閾值設計

日期：2026-05-24

類型：insight_briefing

主題 slug：ai-agent-memory-compression-bufferwindow-2026

研究策略：延續 2026-05-23 I/O 預算工程報告，深化記憶架構具體實作

---

摘要

隨著 LLM Agent 部署規模擴大，記憶體管理成為成本與效能的核心瓶頸。2025–2026 年學術與工程界同步提出多種壓縮策略：從 LangGraph checkpointing 取代 BufferWindow、Active Context Compression 鋸齒形壓縮、兩層語義記憶到 P95 監控告警體系。本次研究梳理 4 個主要實作方向，整合指標數字，提出適用於本系統（avg_io_per_call=20,109、cache_hit_ratio=0.283）的優先改善路徑。

---

核心洞見

洞見 1：LangChain 官方已廢棄 BufferWindow，遷移至 LangGraph Checkpointing

LangChain v0.3.1 起，ConversationBufferWindowMemory、ConversationBufferMemory、ConversationSummaryMemory 全部標記為 deprecated。官方推薦改用 LangGraph checkpointing，原因：

  • 內建多執行緒持久化（跨 session 不需手動序列化）
  • 支援 time travel（可回退至特定檢查點）
  • 與 LangGraph StateGraph 深度整合，無需另外管理 memory 物件

對本系統影響：若任何 workflow 仍使用 ConversationBufferWindowMemory，需優先遷移；否則 v0.3.x 升版後靜默 fallback 到無記憶模式，導致 quality 分數下降而難以察覺。

---

洞見 2：Active Context Compression（Focus 架構）—— 鋸齒形壓縮降低 22.7% token

Focus 架構（Active Context Compression，arXiv 2601.07190）採用兩個核心原語：

  • start_focus：標記探索起點
  • complete_focus：生成學習總結 + 刪除原始交互，僅保留「知識區塊」

效果（SWE-bench 基準）：

| 指標 | 數字 |

|------|------|

| 總 token 削減 | 22.7%（1,490 萬 → 1,150 萬）|

| 最高單任務壓縮率 | 57% |

| 平均每任務壓縮次數 | 6.0 次 |

| 準確率影響 | 0%（60% → 60%，無犧牲）|

關鍵設計原則：每 10–15 次工具呼叫強制壓縮一次（積極提示策略）。探索密集型任務效益最大；反覆迭代型任務可能增加額外開銷，需按任務類型分類應用。

---

洞見 3：兩層記憶架構 vs 全文脈——P95 延遲降幅 92%，準確率反升

LoCoMo benchmark 多 session 回憶測試數據：

| 架構 | 準確率 | P95 延遲 | 每查詢 token 數 |

|------|--------|----------|----------------|

| 全文脈（Full-context）| 72.9% | 17.12 秒 | 26,000 |

| 兩層記憶架構 | 91.6% | 1.44 秒 | 6,956 |

兩層架構設計：

  1. 工作記憶（短期）：最近 K 次交互的滑窗，直接放入 prompt
  2. 知識記憶（長期）：語義壓縮後的事實庫，透過向量相似度按需檢索

反直覺結論：壓縮後準確率反而更高，原因是全文脈中大量低熵噪聲（重複日誌、非任務對話）會干擾模型注意力機制。

---

洞見 4：SimpleMem 語義壓縮——事實化去重解決多 session 參照漂移

SimpleMem（arXiv 2601.02553）的三步壓縮流水線：

  1. 事實化（Factification）：將對話片段轉為獨立、自含事實（消除「它」「上一個」等指代歧義）
  2. 時間正規化：統一時態，避免「明天」「昨天」等相對時間在後續 session 產生誤判
  3. 冗餘過濾：語義 + 詞彙 + 符號三訊號聯合去重，保留高資訊密度內容

適用場景：跨 session 長期對話（如本系統的 continuity 記憶機制），比 SummaryMemory 更精準，因為保留了獨立事實粒度而非連貫敘事。

---

洞見 5：Token 預算監控告警體系——62% 支出來自重複發送

根據 LeanOps 2026 報告（實測 production Agent 成本分析）：

  • 62% 支出用於重新發送相同內容（系統提示 + 工具定義 + 歷史對話）
  • 5 步 Agent 迴圈成本 = 單次呼叫的 3.2 倍
  • P10-P90 成本差距達 20 倍（取決於模型選擇與快取命中率）

推薦監控告警閾值體系：

| 層級 | 條件 | 動作 |

|------|------|------|

| 軟告警 | 日支出 > $50 或 avg_io > 15,000 | ntfy priority=3，記錄 backlog |

| 硬限制 | 日支出 > $100 或 avg_io > 20,000 | 強制切換至較小模型 |

| 月上限 | 月支出 > $1,000 | 需人工批准繼續 |

| 單 session | 單 session > $20 或 > 50,000 tokens | 即時觸發壓縮 |

對本系統直接映射：avg_io_per_call=20,109 已觸達「硬限制」區間（閾值 20,000），cache_hit_ratio=0.283 遠低於 40% 目標 → P0 優先治理。

建議行動

  1. P0（立即）：確認所有 LangGraph workflow 已遷移 BufferWindow → Checkpointing；若有 ConversationBufferWindowMemory 殘留，開啟 deprecation warning 觀察
  2. P0（本週）：avg_io_per_call 超 20,000 觸發硬限制 → 在 tools/context_budget_monitor.py 或 config/benchmark.yaml 補入 hard_cap: 20000，超標自動切換 haiku 系列
  3. P1（本週）：在 check-health.ps1 加入 P95 延遲監控區塊（per task_key，目標 P95 < 5 秒）
  4. P1（下週）：對 research 類任務試點 Active Context Compression 的「每 10–15 工具呼叫強制壓縮」策略，預期削減 token 22–57%
  5. P2（月底）：評估 SimpleMem 事實化機制應用於 context/continuity/ 記憶層，解決跨 session 指代漂移問題
