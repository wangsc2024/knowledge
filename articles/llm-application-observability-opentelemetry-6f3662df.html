LLM Application 可觀測性（Observability）研究報告

研究日期：2026-04-17 | 覆蓋範圍：2025-2026 年最新實踐

---

一、技術概述

LLM Application 可觀測性（LLM Observability）是 2025-2026 年生產 AI 系統的核心基礎建設。不同於傳統應用監控只關注基礎設施健康，LLM 可觀測性還必須追蹤非決定性輸出的「品質」維度：語意相關性、幻覺率、安全違規、Token 成本。分散式追蹤（Distributed Tracing）是現代 LLM 可觀測性的骨幹，OpenTelemetry GenAI 語義慣例（Semantic Conventions）正快速成為工業標準，將 prompt、response、token 用量、model 元資料統一為可互通的 gen_ai.* 屬性命名空間。2025 年起，分散式追蹤、Token 計量、自動化評估與人類回饋迴圈已從「進階功能」變為生產部署的基線要求。

三、三層日誌架構：Request / Model / Application

Layer 1 — Request 層（入口邊界）

捕捉每筆進入 LLM 的原始請求。最低日誌欄位：


{
  "timestamp": "2026-04-17T03:00:00Z",
  "trace_id": "abc123",
  "span_id": "span001",
  "user_id": "u_9847",
  "session_id": "sess_xyz",
  "prompt_template_version": "v2.3.1",
  "prompt_length_tokens": 412,
  "endpoint": "/api/v1/chat",
  "source": "todoist-agent-team"
}

關鍵點：PII 欄位（prompt 原文）在此層進行遮罩或雜湊後才落庫，符合 GDPR/CCPA。

Layer 2 — Model 層（LLM 呼叫邊界）

追蹤 LLM API 呼叫本身，對應 OTel gen_ai.* 屬性：


{
  "gen_ai.system": "anthropic",
  "gen_ai.request.model": "claude-sonnet-4-6",
  "gen_ai.request.max_tokens": 2048,
  "gen_ai.usage.input_tokens": 412,
  "gen_ai.usage.output_tokens": 318,
  "gen_ai.response.finish_reason": "end_turn",
  "latency_ms": 1842,
  "latency_ttft_ms": 380,
  "cost_usd": 0.00147
}

Layer 3 — Application 層（業務品質邊界）

追蹤輸出品質與業務結果：


{
  "eval.relevance_score": 4.2,
  "eval.hallucination_detected": false,
  "eval.judge_model": "claude-haiku-3-5",
  "task_completed": true,
  "user_feedback": "thumbs_up",
  "ntfy_sent": true,
  "downstream_action": "todoist_task_created"
}

四、OpenTelemetry + LLM 整合 Step-by-Step 工作流

Step 1：安裝依賴


uv add opentelemetry-sdk opentelemetry-exporter-otlp opentelemetry-instrumentation-openai
# 或使用 OpenLLMetry（涵蓋多框架）
uv add traceloop-sdk

Step 2：初始化 Tracer Provider


from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

provider = TracerProvider()
exporter = OTLPSpanExporter(endpoint="http://localhost:4317")
provider.add_span_processor(BatchSpanProcessor(exporter))
trace.set_tracer_provider(provider)
tracer = trace.get_tracer(__name__)

Step 3：手動 Instrumentation


with tracer.start_as_current_span("llm_call") as span:
    span.set_attribute("gen_ai.system", "anthropic")
    span.set_attribute("gen_ai.request.model", "claude-sonnet-4-6")
    span.set_attribute("gen_ai.usage.input_tokens", token_count)
    response = call_llm(prompt)
    span.set_attribute("gen_ai.usage.output_tokens", response.usage.output_tokens)
    span.set_attribute("gen_ai.response.finish_reason", response.stop_reason)

Step 4：自動 Instrumentation


# Traceloop 全自動（含 Anthropic / LangChain / CrewAI）
from traceloop.sdk import Traceloop
Traceloop.init(app_name="daily-digest-agent")

Step 5：OTel Collector 配置（otel-collector.yaml）


receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
exporters:
  otlphttp/langfuse:
    endpoint: https://cloud.langfuse.com/api/public/otel
    headers:
      Authorization: "Basic ${LANGFUSE_API_KEY}"
service:
  pipelines:
    traces:
      receivers: [otlp]
      exporters: [otlphttp/langfuse]

Step 6：Grafana 告警規則


groups:
  - name: llm_alerts
    rules:
      - alert: HighLatency
        expr: llm_request_latency_p99 > 5000
        for: 2m
        labels:
          severity: warning
      - alert: ErrorRateHigh
        expr: rate(llm_errors_total[5m]) / rate(llm_requests_total[5m]) > 0.02
        for: 1m
        labels:
          severity: critical
      - alert: CostSpike
        expr: sum(llm_cost_usd_total) by (agent) > 5.0
        annotations:
          summary: "單日 LLM 成本超過 $5 USD"

五、Python 生產級最小可觀測性包裝器


# tools/llm_observability.py
import time, json, logging
from contextlib import contextmanager
from opentelemetry import trace
from opentelemetry.trace import Status, StatusCode

logger = logging.getLogger(__name__)
tracer = trace.get_tracer("daily-digest-agent")

@contextmanager
def observe_llm_call(task_type: str, model: str, prompt: str):
    ctx = {}
    start = time.monotonic()
    with tracer.start_as_current_span(f"llm.{task_type}") as span:
        span.set_attribute("gen_ai.system", "anthropic")
        span.set_attribute("gen_ai.request.model", model)
        span.set_attribute("task_type", task_type)
        logger.info(json.dumps({"event": "llm_call_start", "task_type": task_type, "model": model}))
        try:
            yield ctx
            elapsed_ms = (time.monotonic() - start) * 1000
            span.set_attribute("gen_ai.usage.input_tokens", ctx.get("input_tokens", 0))
            span.set_attribute("gen_ai.usage.output_tokens", ctx.get("output_tokens", 0))
            span.set_attribute("latency_ms", elapsed_ms)
            span.set_attribute("eval.quality_score", ctx.get("quality_score", -1))
            span.set_status(Status(StatusCode.OK))
        except Exception as e:
            span.set_status(Status(StatusCode.ERROR, str(e)))
            logger.error(json.dumps({"event": "llm_call_error", "error": str(e)}))
            raise

六、效果量化數據

| 指標 | 基準值 | 導入可觀測性後 |

|---|---|---|

| 問題偵測時間（MTTD） | 數小時至數天 | < 5 分鐘（即時告警） |

| 幻覺率基線建立時間 | 無法量化 | 2-4 週可建立基線 |

| Token 成本浪費（prompt 膨脹） | 無監控時 15-30% 浪費 | 監控後減少 20-40% |

| 高流量採樣策略 | 全量追蹤 | 10-20% 全追蹤 + 100% 基礎指標 |

---

七、與 daily-digest-prompt 的應用場景

  1. Phase 0-3 追蹤：為每個 Phase 建立 OTel span，trace_id 注入 state/workflow-state.json，實現跨 Phase 根因追蹤
  2. Token 效率排程整合：anthropic_usage_calibrator.py 已採集 Token 用量，加掛 gen_ai.usage.* 屬性可直接輸出 OTel 格式
  3. ntfy 告警閾值自動化：Grafana AlertManager 自動偵測 state/token-usage-real-history.jsonl 的 cost spike 並推播
  4. Todoist team 品質監控：Phase 3 assemble 結果加掛 eval.quality_score，長期追蹤 Haiku vs Sonnet 品質差距
  5. ADR 自動觸發：品質分數連續 3 天低於基線時，自動觸發 tools/validate_adr.py 建立新 ADR

---

八、風險與注意事項

| 風險 | 緩解措施 |

|---|---|

| PII 外洩 | 落庫前 mask/hash，僅保留 token count |

| 觀測成本膨脹 | 高流量時採樣 10-20%，基礎指標 100% |

| 告警疲勞 | 每條 alert 必須對應 runbook，設定靜默期 |

| OTel GenAI 慣例仍在 RC 階段 | 鎖定版本，不依賴 breaking change |

| Windows 環境限制 | 採用 Python 層採集，規避 pwsh UTF-8 問題 |
