uv Workspace 管理、Script 模式（PEP 723）、Python 版本管理與 Windows CI 整合 — Astral uv 2026 進階實踐深度研究

1. 技術概述

uv 是由 Astral（維護 ruff 的團隊）以 Rust 撰寫的現代 Python 套件管理器，2023 年底發布，2025-2026 年已成為生產環境主流選擇之一 [darkthread.net, 2025]。相較於傳統工具鏈（pip + virtualenv + pyenv + poetry），uv 整合了七大功能：套件安裝、依賴管理、虛擬環境建立、Python 版本管理、Workspace 多套件管理、PEP 723 Script 模式、以及工具隔離執行（uvx）[docs.astral.sh, 2026]。

uv 解決的核心問題是 Python 工具鏈碎片化：過去開發者需要多個工具各司其職，版本矩陣複雜、CI 配置冗長。uv 的效能提升顯著，套件安裝與依賴解析速度比 pip 快 10-100 倍 [darkthread.net, 2025]，得益於單一靜態 Rust 二進位、平行下載、全域快取與硬連結去重複技術。

本次研究聚焦三個進階面向（前一份筆記已涵蓋鎖定檔與快取基礎）：① Workspace 多套件管理（Monorepo 架構），② PEP 723 Script 模式（自帶依賴的單一腳本），③ Python 版本管理與 Windows CI 整合。

---

2. 核心概念與原理

2.1 uv Workspace 架構

Workspace 是 uv 從 Rust Cargo 借鑑的 Monorepo 管理機制，允許多個相關 Python 套件共享單一 uv.lock 鎖定檔與 .venv 虛擬環境 [來源：docs.astral.sh/uv/concepts/projects/workspaces/, 2026]。

核心三要素：

  1. Workspace Root：根目錄的 pyproject.toml 定義 [tool.uv.workspace] 表格，含 members（必要，glob patterns）與 exclude（可選）欄位。
  2. 單一 uv.lock：整個 Workspace 共用一個鎖定檔，放在 workspace root，確保所有成員使用版本一致的依賴，防止版本漂移 [來源：pydevtools.com, 2025]。
  3. 成員間依賴聲明：透過 [tool.uv.sources] 中 { workspace = true } 宣告解析本地套件，以可編輯模式（editable）安裝，修改立即生效 [來源：pydevtools.com, 2025]。

成員初始化：


uv init monorepo && cd monorepo
uv init --lib packages/shared-lib
uv init --app packages/worker-app

當 uv 偵測到父層 pyproject.toml 時，自動登錄新套件為 workspace 成員 [來源：pydevtools.com, 2025]。

Python 版本交集計算： uv 自動計算所有成員 requires-python 的交集。若 shared-lib 要求 >=3.10、worker-app 要求 >=3.12，有效約束為較嚴格的 >=3.12 [來源：pydevtools.com, 2025]。

2.2 PEP 723 Script 模式（自帶依賴腳本）

PEP 723 標準化了「單一腳本內嵌元資料」格式，允許腳本在 TOML 格式的 # /// script 注釋塊中宣告依賴與 Python 版本需求 [來源：docs.astral.sh/uv/guides/scripts/, 2026]。

執行機制： uv run example.py 解析 # /// script 塊，建立隔離快取環境並解析安裝依賴。後續執行若環境已快取幾乎瞬間完成。重要行為：存在 inline metadata 時，當前專案的 dependencies 會被完全忽略，無需加 --no-project 旗標 [來源：docs.astral.sh, 2026]。

Script 鎖定機制（生產安全）： uv lock --script example.py 產生 example.py.lock，包含所有依賴的固定版本與 SHA256 雜湊 [來源：lincolnloop.com, 2025]。

  • --locked：本地開發用，鎖定檔缺失或過期時失敗
  • --frozen：CI 用，直接套用鎖定檔不進行解析（零網路呼叫，最高重現性）[來源：lincolnloop.com, 2025]

exclude-newer 時間鎖： 在 [tool.uv] 設定 exclude-newer = "2026-01-01T00:00:00Z"，限制套件解析到指定日期前的版本，適用於需要歷史重現的研究腳本 [來源：docs.astral.sh, 2026]。

2.3 Python 版本管理原理

uv 維護雙層版本聲明機制 [來源：docs.astral.sh/uv/concepts/python-versions/, 2026]：

| 層級 | 位置 | 語意 |

|------|------|------|

| .python-version | 專案根目錄 | 開發環境使用的確切版本（如 3.12） |

| requires-python | pyproject.toml | 套件支援的版本範圍（如 >=3.11） |

.python-version 控制開發時實際使用哪個直譯器，requires-python 是發布套件時的相容性聲明。執行 uv python pin 3.12 更新 .python-version；若指定版本未安裝，uv 在 uv sync 時自動下載 [來源：docs.astral.sh, 2026]。

3. 最佳實踐

3.1 Workspace 成員命名（Virtual Root 陷阱規避）

Virtual Root（package = false）仍需在 [project] 中設定不與任何成員重複的唯一名稱。推薦：根目錄命名 {project-name}-workspace（例：my-app-workspace）。開發依賴放在 [dependency-groups]（PEP 735）而非 [project.dependencies] [來源：dev.to/aws, 2025]。

3.2 pytest 多套件設定

Workspace Monorepo 中多套件包含同名測試檔時，pytest 預設 prepend import mode 會衝突。必要設定：addopts = "--import-mode=importlib" 在根目錄 pyproject.toml。同時避免在測試目錄加 __init__.py [來源：dev.to/aws, 2025]。

3.3 CI 快取最佳化（GitHub Actions）

使用 astral-sh/setup-uv@v7 官方 Action 搭配 enable-cache: true + cache-dependency-glob: "**/uv.lock"。避免用 actions/cache 自行管理（容易跨 PR 快取碎片化）。部署用 uv sync --locked --no-dev [來源：docs.astral.sh, 2026]。

3.4 uv link-mode（Windows）

Windows 保持預設 hardlink。避免使用 symlink：uv cache clean 會破壞 symlink 安裝 [來源：copdips.com, 2025]。

3.5 生產效能：Bytecode 編譯

uv 預設跳過 bytecode 編譯。生產環境設定 UV_COMPILE_BYTECODE=1 縮短後續啟動時間，代價是首次建置時間增加 [來源：copdips.com, 2025]。

4. 陷阱與反例

陷阱 1：Workspace 沒有 Runtime 隔離（最常見誤解）

Python 本身不提供執行時依賴隔離，uv workspace 的單一 .venv 意味著成員 A 可以 import 只有成員 B 宣告的依賴，而不會在執行時報錯 [來源：docs.astral.sh, 2026]。只有在 CI 以 uv sync --package member-A 分別安裝、分別測試時才能暴露隱藏的越界 import。規避：使用靜態分析工具（gauge.sh validate-workspace-deps）[來源：gauge.sh, 2024]。

陷阱 2：Workspace sources 宣告遺漏（靜默解析錯誤）

成員間互依時，[project.dependencies] 要加依賴名，[tool.uv.sources] 也必須補上 { workspace = true } 宣告。漏掉後者，uv 不報錯但從 PyPI 拉取同名套件（若存在）[來源：dev.to/aws, 2025]。規避：一律用 uv add --package member-B member-A 讓 CLI 自動補齊兩處。

陷阱 3：版本衝突在 Monorepo 遷移時才爆發

pip 的 per-venv 方案隱藏了不同套件間的 transitive dependency 衝突；遷移到 uv workspace 時統一浮出 [來源：dev.to/jimmyyeung, 2025]。規避：遷移前用 pip check 掃描衝突，先解決再切換。torch 相依套件（flash-attn、deepspeed）有特殊建置隔離需求，不適合 workspace。

陷阱 4：Script CI 未加 --frozen 導致非確定性

PEP 723 Script 在 CI 只用 uv run example.py 而未加 --frozen，每次執行可能因 upstream 新版本導致依賴變化 [來源：lincolnloop.com, 2025]。生產 CI 必須先產生鎖定檔（uv lock --script）並提交版本控制，CI 使用 --frozen 執行。

5. 與本專案關聯

場景 1：Script 模式應用於一次性工具

tools/apply_series_update.py 等工具若改用 PEP 723 Script 模式（inline metadata），在乾淨環境無需 uv sync 直接執行。對 AI 代理子任務的隔離執行尤其有用。

場景 2：Windows Task Scheduler 啟動效能優化

在 run-token-efficiency.ps1 等排程腳本加入 $env:UV_COMPILE_BYTECODE = '1'，可改善高頻 uv run python -m tools.xxx 呼叫的後續啟動延遲（2小時輪詢場景有實際效益）。

場景 3：Workspace 評估門檻

本專案目前單一套件架構已足夠，但若 hooks/、tools/、skills/ 依賴樹膨脹需獨立測試環境時，可考慮 workspace 拆分（packages/hooks/、packages/tools/），需先確認無版本衝突且接受 Runtime 無隔離限制。

6. 程式碼範例

PEP 723 自帶依賴維運腳本


#!/usr/bin/env -S uv run --script
# /// script
# requires-python = ">=3.11"
# dependencies = ["requests>=2.31", "pyyaml>=6.0"]
# [tool.uv]
# exclude-newer = "2026-01-01T00:00:00Z"
# ///
import requests, yaml
def main():
    resp = requests.get("http://localhost:3000/api/health", timeout=5)
    print(yaml.dump(resp.json()))
if __name__ == "__main__":
    main()

生成鎖定: uv lock --script check_api.py；CI 執行: uv run --frozen --script check_api.py

來源：改編自官方文件 [docs.astral.sh/uv/guides/scripts/, 2026]

Workspace 根目錄 pyproject.toml 最小配置


[project]
name = "my-project-workspace"
package = false

[tool.uv.workspace]
members = ["packages/*"]

[dependency-groups]
dev = ["pytest>=8.0"]

[tool.pytest.ini_options]
addopts = "--import-mode=importlib"

來源：整合自 docs.astral.sh [2026] + dev.to/aws [2025]

Windows Task Scheduler 啟用 Bytecode 編譯


$env:UV_COMPILE_BYTECODE = '1'
uv run python tools/anthropic_usage_calibrator.py

來源：自寫，依據 copdips.com [2025]
