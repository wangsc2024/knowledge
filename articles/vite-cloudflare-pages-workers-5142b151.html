Vite 前端建構系統與 Cloudflare Pages/Workers 部署整合完整指南

主題概述

Vite 是由 Vue.js 作者尤雨溪建立的新世代前端建構工具，以極速開發伺服器啟動和熱模組替換（HMR）著稱。Cloudflare Pages 與 Workers 則是 Cloudflare 提供的邊緣運算與靜態網站託管平台。兩者的整合在 2025-2026 年達到成熟階段，Cloudflare 推出官方 Vite Plugin v1.0（GA），實現了前端、後端、資料庫在同一 Worker 中運作的全端開發模式。本文系統性整理從 Vite 建構配置到 Cloudflare 部署的完整工作流。

---

一、Vite 建構系統核心架構

1.1 核心設計理念

  • 開發階段：利用瀏覽器原生 ESM（ES Modules）按需載入，不預先打包全部模組，實現毫秒級啟動
  • 生產階段：基於 Rollup 進行最佳化打包，支援 tree-shaking、code splitting、CSS 程式碼分割
  • 插件架構：相容 Rollup 插件介面，加上 Vite 特有的生命週期鉤子

1.2 Vite 6 新特性摘要

  • Environment API：支援多環境建構（客戶端 + 伺服器端 SSR），統一 dev/build 行為
  • 改進的 SSR 支援：更小的打包尺寸與更快的載入速度
  • 即時伺服器啟動：無論應用大小，dev server 啟動速度不受影響
  • 原生 TypeScript/JSX/CSS 支援：開箱即用，無需額外配置

1.3 建構配置關鍵選項


// vite.config.js
import { defineConfig } from 'vite'

export default defineConfig({
  base: '/',           // 公共基礎路徑（子目錄部署需調整）
  build: {
    outDir: 'dist',    // 輸出目錄
    target: 'es2015',  // 瀏覽器相容性目標
    cssCodeSplit: true, // CSS 程式碼分割（推薦開啟）
    rollupOptions: {
      output: {
        manualChunks: {
          // 手動分塊策略：將大型依賴抽離
          vendor: ['react', 'react-dom'],
        }
      }
    }
  }
})

關鍵配置說明：

  • base：部署到子路徑時需設定（如 /my-app/），所有資源路徑會自動改寫
  • build.target：預設對齊 Baseline Widely Available（Chrome≥107, Firefox≥104, Safari≥16）
  • build.cssCodeSplit：啟用後每個非同步 chunk 只載入自己的 CSS
  • import.meta.env.BASE_URL：在程式碼中動態取得 base 路徑

1.4 插件系統

Vite 插件在開發和建構兩個階段運行，攔截以下生命週期事件：

  • resolveId：決定模組 ID 的解析方式
  • load：載入模組內容
  • transform：轉換模組內容（如 JSX → JS）
  • buildStart/End：建構開始/結束時執行
  • generateBundle：修改最終輸出的 bundle

1.5 多頁面應用（MPA）


build: {
  rollupOptions: {
    input: {
      main: 'index.html',
      about: 'about.html',
      games: 'games/index.html'
    }
  }
}

1.6 效能最佳化策略

| 策略 | 說明 |

|------|------|

| 動態 import | const Foo = () => import('./Foo.vue') 按需載入 |

| CSS Code Splitting | build.cssCodeSplit: true 每個 chunk 只載入對應 CSS |

| ES 模組優先 | 使用 lodash-es/date-fns 而非 CommonJS 版本 |

| 圖片格式 | 偏好 WebP/AVIF 取代 PNG/JPEG |

| Legacy 瀏覽器 | @vitejs/plugin-legacy 自動產生 polyfill chunks |

二、Cloudflare Pages 部署方式

2.1 Git 整合部署（推薦）

Cloudflare Pages 原生支援 Vite 專案。透過 Git 整合，每次 push 自動觸發建構和部署。

設定步驟：

  1. 將專案推送至 GitHub/GitLab
  2. 登入 Cloudflare Dashboard → Workers & Pages → Create application
  3. 選擇 Connect to Git → 選取 repository
  4. 配置建構設定：

- Build command: npm run build

- Build output directory: dist

  1. Save and Deploy

自動部署特性：

  • 每次 commit push 自動重建並部署
  • 部署完成後可透過 .pages.dev 存取
  • Pull Request 自動產生預覽 URL
  • 非 production 分支自動建立 preview deployment

注意：使用 Git 整合後無法切換為 Direct Upload 模式。

2.2 Direct Upload 部署


# 本地建構後直接上傳
npm run build
npx wrangler pages deploy dist/

適合 CI/CD 自訂管線或不想連結 Git 倉庫的場景。

2.3 GitHub Actions CI/CD


# .github/workflows/deploy.yml
name: Deploy to Cloudflare Pages
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          command: pages deploy dist/ --project-name=my-project

三、Cloudflare Workers Vite Plugin（全端整合）

3.1 平台策略轉變

2025-2026 年 Cloudflare 明確宣示：所有投資和功能開發將聚焦在 Workers。Workers 現在支援靜態資源服務和 SSR，成為統一的全端平台。新專案建議直接使用 Workers 而非 Pages。

3.2 Vite Plugin v1.0（GA）


npm install @cloudflare/vite-plugin


// vite.config.js
import { defineConfig } from 'vite'
import { cloudflare } from '@cloudflare/vite-plugin'

export default defineConfig({
  plugins: [cloudflare()]
})

核心能力：

  • 開發伺服器在 workerd（Workers 運行時）中執行，行為與生產一致
  • 支援 Workers 獨有 API：KV、R2、D1、Durable Objects、Queues
  • HMR 即時更新，同時保有 Workers 運行時特性
  • vite build 一次性產出客戶端和伺服器端
  • vite preview 在 Workers 運行時中預覽建構結果
  • wrangler deploy 直接部署 Vite 建構輸出

3.3 Wrangler 配置


// wrangler.jsonc
{
  "name": "my-app",
  "main": "./src/index.ts",
  "compatibility_date": "2025-01-01",
  "assets": {
    "directory": "./dist/client",
    "binding": "ASSETS"
  }
}

3.4 全端架構：前端 + 後端 + 資料庫


┌─────────────────────────────────────┐
│     Cloudflare Worker（單一部署）      │
├──────────┬──────────┬───────────────┤
│ 靜態資源  │ 後端 API  │ SSR 渲染       │
│ (dist/)  │ (Hono/   │ (React Router │
│          │ Express) │  v7 / Astro)  │
├──────────┴──────────┴───────────────┤
│     Cloudflare 儲存服務               │
│ KV / R2 / D1 / Durable Objects      │
│ Hyperdrive（PostgreSQL/MySQL 橋接）   │
└─────────────────────────────────────┘

單一部署優勢：前端和後端同時部署，無需分別管理 Pages 和 Workers。

3.5 Wrangler vs Vite Plugin 選型

| 面向 | Wrangler | Vite Plugin |

|------|----------|-------------|

| 適用場景 | 後端/API/無伺服器函數 | 前端為主的全端應用 |

| HMR | 不支援 | 支援 |

| 框架支援 | 通用 Workers | React/Vue/Svelte/Solid/React Router v7 |

| 遠端開發 | 支援 --remote | 不支援 |

| 建構最佳化 | 基礎 | 進階（code splitting、tree shaking） |

| 插件生態 | 無 | Vite/Rollup 插件生態系 |

四、SPA 模式配置（實用場景）

4.1 純靜態 SPA（Cloudflare Pages）


// vite.config.js — React SPA
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  build: {
    outDir: 'dist',
    sourcemap: false,
  }
})

搭配 Cloudflare Pages 的 SPA 模式，所有非資源請求自動導向 index.html。

4.2 SPA + API（Workers）


// wrangler.jsonc
{
  "assets": {
    "directory": "./dist",
    "binding": "ASSETS",
    "html_handling": "single-page-application"
  }
}

html_handling: "single-page-application" 讓導航請求返回 index.html，API 請求交給 Worker 處理。

五、部署前檢查清單

  • [ ] vite build 本地建構成功，dist/ 目錄產出正確
  • [ ] base 路徑設定正確（子目錄部署需特別注意）
  • [ ] 環境變數已在 Cloudflare Dashboard 或 wrangler.jsonc 配置

六、常見問題與排除

6.1 部署後路由 404

原因：SPA 使用 client-side routing，但伺服器不知道將未知路徑導向 index.html。

解法：

  • Cloudflare Pages：自動處理（原生 SPA 支援）
  • Workers：設定 html_handling: "single-page-application"

6.2 CSS/資源路徑錯誤

原因：base 設定與實際部署路徑不一致。

解法：確保 vite.config.js 的 base 與部署子路徑一致。

6.3 載入錯誤處理

Vite 提供 vite:preloadError 事件處理動態 import 失敗（常見於部署後舊版本資源被刪除）：


window.addEventListener('vite:preloadError', (event) => {
  event.preventDefault()
  window.location.reload()
})

6.4 Legacy 瀏覽器支援


npm install @vitejs/plugin-legacy


import legacy from '@vitejs/plugin-legacy'

export default defineConfig({
  plugins: [legacy({ targets: ['defaults', 'not IE 11'] })]
})
