Google ADK（Agent Development Kit）深度研究報告

一、專案概述

Google ADK（Agent Development Kit）是 Google 於 2025 年底正式開源的 AI Agent 開發框架，採用 Apache 2.0 授權。截至 2026 年 3 月，Python SDK（adk-python）已獲得 18.6K GitHub Stars、3.1K Forks，累計 2,368 次提交，最新版本 v1.27.4（2026-03-24）。

核心定位：讓 Agent 開發像軟體工程一樣——code-first、可測試、可部署、可組合。雖然針對 Gemini 優化，但設計為模型無關（model-agnostic）、部署無關（deployment-agnostic）的通用框架。

解決的問題：

  • Agent 開發缺乏工程化標準（與傳統軟體開發的差距）
  • 多 Agent 協作缺乏結構化編排機制
  • 從原型到生產部署的斷裂（local 到 cloud 的無縫過渡）
  • 跨語言 Agent 開發的碎片化

二、技術架構

2.1 多語言 SDK 矩陣

| 語言 | 套件名稱 | 成熟度 |

|------|---------|--------|

| Python | google-adk | 最成熟（v1.27.4） |

| TypeScript | @google/adk | 穩定 |

| Go | google.golang.org/adk | 穩定 |

| Java | Maven/Gradle（v0.6.0） | 追趕中 |

2.2 Agent 類型體系

LLM Agent：以大型語言模型為核心的動態路由 Agent，根據上下文自主決定工具呼叫與子 Agent 委派。

Workflow Agent（結構化編排）：

  • Sequential Agent：有序管線執行（A 到 B 到 C）
  • Parallel Agent：並行任務執行（A 與 B 與 C 同時）
  • Loop Agent：迭代處理（while/for 模式）

ADK 2.0 Alpha — 圖式工作流：

  • Graph Routes（條件路由邏輯）
  • 資料處理（Agent 間通訊）
  • Human Input 整合
  • 協作式 Agent 模式
  • 動態工作流執行

2.3 工具生態系統

| 工具類型 | 說明 |

|---------|------|

| Function Tools | 自定義 Python/TS 函數，支援效能優化與動作確認 |

| MCP Tools | Model Context Protocol 整合（Anthropic 標準） |

| OpenAPI Tools | 透過 OpenAPI 規格檔消費第三方 API |

| Pre-built Tools | Google Search、Code Execution 等內建工具 |

| Agent-as-Tool | 將其他 Agent 作為工具呼叫（巢狀組合） |

| LangchainTool | 整合 LangChain 生態元件 |

| CrewaiTool | 整合 CrewAI 生態元件 |

2.4 記憶與狀態管理

  • Context System：含快取機制與壓縮策略
  • Sessions：狀態持久化，支援 rewind（回溯到呼叫前狀態）與 migration
  • Memory Layers：多層記憶（對話歷史加狀態）
  • State Management：持久與瞬態狀態處理

2.5 A2A 協議（Agent-to-Agent）

跨 Agent 遠端通訊標準：

  • 暴露 Agent 能力（Python/Go）
  • 消費外部 Agent（Python/Go）
  • A2A Extensions 自訂協議

三、功能特色與差異化

3.1 vs LangChain（126K Stars）

| 維度 | Google ADK | LangChain |

|------|-----------|----------|

| 設計理念 | 端到端生命週期 | 模組化鏈式組合 |

| 工作流 | 內建 Sequential/Parallel/Loop 加 Graph | 需搭配 LangGraph |

| 部署 | 原生 Vertex AI / Cloud Run | 需自行搭建 |

| 生態 | 較新但快速成長 | 最成熟且整合最多 |

| 多語言 | Python/TS/Go/Java | Python/JS |

3.2 vs CrewAI（44K Stars）

| 維度 | Google ADK | CrewAI |

|------|-----------|--------|

| 多 Agent 模式 | 階層式 sub_agents | 角色驅動 Crew |

| 工作流 | 圖式加結構化 | 結構化任務委派 |

| 企業整合 | Google Cloud 原生 | Oracle/Deloitte 採用 |

| 學習曲線 | 中等 | 較低 |

3.3 獨特優勢

  1. 圖式工作流（ADK 2.0）：條件路由加動態執行，超越線性管線
  2. A2A 協議：跨服務 Agent 通訊標準，非僅框架內協作
  3. 內建評測框架：adk eval 命令支援標準化評估、使用者模擬、軌跡分析
  4. Streaming 能力：Gemini Live API 支援即時音訊/影像/視訊處理
  5. Session Rewind：可回溯 Agent 狀態到任意呼叫前節點
  6. No-Code Agent Config：無需寫程式碼即可建構 Agent

四、快速上手

安裝


pip install google-adk

基本 Agent 建立


from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService

agent = Agent(
    name="my_agent",
    model="gemini-2.5-flash",
    instruction="你是一個有用的助手。",
    tools=[my_custom_tool],
    sub_agents=[specialist_agent]
)

開發 UI


adk web  # 啟動圖形化開發介面
adk eval  # 執行評測
adk deploy cloud-run --project my-project --region us-central1  # 部署

六、潛在應用場景

對 daily-digest-prompt 專案的借鏡

  1. Workflow Agent 模式：目前專案用 PowerShell 腳本編排 Phase 0 到 1 到 2，可借鏡 ADK 的 Sequential/Parallel Agent 實現更優雅的管線控制
  2. Agent-as-Tool 模式：將現有 28 個 Skill 封裝為 ADK Tool，透過 Agent 動態路由取代靜態 SKILL_INDEX 查找
  3. Session Rewind：Agent 執行失敗時可回溯狀態，比目前的 recovery-queue 機制更精確
  4. 內建評測框架：可替代自建的品質閘門（quality gate），用 adk eval 標準化測試 Agent 輸出品質
  5. A2A 協議：若未來需要跨機器 Agent 協作（如 LINE Bot 到本地 Agent），A2A 提供標準通訊層
  6. MCP 整合：已原生支援 MCP，與專案已使用的 MCP 工具（如 pingtung-news MCP）無縫銜接

七、優缺點評估

優點

  • Google 官方維護，長期支援可期
  • 圖式工作流（ADK 2.0）是目前最先進的 Agent 編排模式
  • 多語言 SDK 覆蓋 Python/TS/Go/Java，團隊選型彈性大
  • 模型無關設計，支援 Gemini/Claude/Ollama/vLLM 等
  • 內建評測、開發 UI、部署工具，端到端體驗完整
  • Apache 2.0 開源，無授權費用

缺點

  • 生態成熟度不如 LangChain（社群資源、第三方整合較少）
  • Java/Go SDK 成熟度落後 Python
  • 部署體驗高度依賴 Google Cloud（Vertex AI、Cloud Run），非 GCP 使用者需額外配置
  • 文件雖完整但部分 API 變動頻繁（175+ 版本發布顯示迭代快速）
  • 圖式工作流仍為 Alpha 階段，生產使用需謹慎
