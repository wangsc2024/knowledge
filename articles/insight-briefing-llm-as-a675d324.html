深度研究洞察簡報：AI Agent 評估與 LLM-as-a-Judge 生產管線（2026-05-03）

摘要

本簡報彙整 LLM-as-a-Judge 在代理系統生產評估的角色：以 Rubrics 將主觀品質轉成可批次重現的評分契約，並以 人類校準／雙層抽樣 控制成本；同時對照 SWE-bench Verified 作為「可執行、可自動判讀」的軟體代理基準，納入 OpenAI 2026 對該基準飽和與題目瑕疵的公開結論，避免將單一分數誤讀為真實工程能力。與本 repo 的銜接點在於：results/todoist-auto-*.json 已由 config/schemas/results-auto-task-schema.json 承載 done_cert 等品質欄位，可與 judge 管線對齊為「同一契約、雙軌驗證」（schema 守衛 + LLM rubric）。

關鍵洞見

  1. Rubrics 是生產 Judge 的唯一契約

明確分數錨點、反例與邊界案例，比「請幫我打分」更能降低跨模型、跨版本的漂移；生產管線常搭配 temperature=0 與固定輸出 schema（見業界實作討論：learnwithparam；Gemini Lab 生產管線指南）。

  1. Judge 可信度三支柱：對齊人類、內部自洽、對變體穩健

Microsoft Azure AI Foundry 部落格指出代理評估中 LLM judge 的可信任度問題（格式微調、改寫、篇幅變化可能影響分數）；設計上需預留人機一致性量測與對抗式測試集。

  1. 學術／產業文獻強調：沒有「萬能 Judge」

arXiv:2603.05399（檢索標題：Judge reliability / LLM-as-judge 2026）類工作顯示不同 judge 在不同基準上表現不一致；實務上應以多 judge 仲裁或分域 rubric 降低單點失效。

  1. SWE-bench Verified：方法學仍有效，但「前沿排名」語意已變

OpenAI 說明該子集經人工審閱、Docker 隔離與明確 pass/fail；惟 2026 年 OpenAI 亦公開停止以 Verified 作為前沿能力指標（題目測試瑕疵比例、訓練資料污染疑慮），並建議遷移至 SWE-bench Pro 等後續基準。

  1. 與本專案自動任務的對齊

tools/validate_results.py + results-auto-task-schema.json 提供機械式「及格線」；LLM-as-a-Judge 適合覆蓋 summary 語意、architecture_repair_hint 可執行性、ntfy 可讀性 等 schema 難以表達的維度。兩者並行才符合 skills/insight-briefing/SKILL.md 所描述的「機械優先、LLM 補敘事」順序（亦見 CLAUDE.md 工具 CLI 化原則）。

建議行動

  • 在 skills/task-quality-gate/SKILL.md 或對應 harness 中，將 rubric 版本 與 judge 模型 id 寫入結果 JSON 的延伸欄位（僅在 schema additionalProperties 允許範圍內），便於回溯評分漂移。
  • 為 coding-agent 類研究建立雙指標：自動基準（SWE 家族）+ 管線內 done_cert / validate_results 成功率，避免單一 leaderboard 驅動架構決策。
  • 對外推播與 KB 匯入維持 skills/knowledge-query/SKILL.md 的 contentText 全文索引慣例，利於後續 hybrid 去重（config/dedup-policy.yaml）。
