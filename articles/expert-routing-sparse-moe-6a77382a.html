Expert Routing 與 Sparse MoE 路由機制深度研究

執行摘要

今天已完成的 Todoist 任務雖然表面上分屬 AI 最新技術研究、RAG 洞見報告與專案優化，但它們共通的技術核心都是「如何把請求分派給最適合的專家」。這正是 Sparse Mixture-of-Experts（MoE）中的 expert routing 問題。MoE 的關鍵價值在於：模型不必讓所有參數同時參與每個 token 的計算，而是由 router 只啟用少數專家，因此能在控制單次計算量的同時擴大總參數規模 [來源：Outrageously Large Neural Networks, 2017] [來源：GShard, 2020] [來源：Switch Transformers, 2022]。但真正困難的不是「能不能路由」，而是「能否穩定、均衡、低成本地路由」；負載不均、token drop、專家飢餓與跨裝置通訊，都是 MoE 成敗的主因 [來源：Switch Transformers, 2022] [來源：Mixture-of-Experts with Expert Choice Routing, 2022] [來源：Dense Backpropagation Improves Routing for Sparsely-Gated Mixture-of-Experts, 2024]。

對 daily-digest-prompt 而言，這題的價值不在於直接訓練 LLM，而在於借用 MoE 的設計原理改善 Skill-First 路由：把不同研究類型視為 experts、把 task metadata 視為 router features、把高頻任務壅塞視為 load imbalance、把 fallback / retry 視為 capacity 與 overflow control。這種轉譯是根據上述 MoE 與路由論文對「專家選擇」「負載平衡」「容量治理」的共同描述所做的工程推論。

研究背景

Sparse MoE 的思想可以追溯到把輸入交給多個專家，再由 gating network 做軟或硬選擇的經典 mixture-of-experts 架構 [來源：Adaptive Mixtures of Local Experts, 1991]。到了大型 Transformer 時代，Sparsely-Gated MoE 把這個概念放大到超大模型，證明稀疏啟用可以顯著增加參數量而不把每 token FLOPs 等比例拉高 [來源：Outrageously Large Neural Networks, 2017]。其後 GShard、GLaM、Switch Transformer 等工作，進一步把路由、分片、負載平衡與大規模分散式訓練串成可落地的系統 [來源：GShard, 2020] [來源：GLaM, 2022] [來源：Switch Transformers, 2022]。

核心發現

1. Sparse MoE 的本質是「參數規模擴大」，不是「每次計算更重」

MoE 透過 router 為每個 token 挑選少數專家，讓大多數參數在單次前向中保持閒置，因此總參數可以遠大於每 token 實際啟用參數 [來源：Outrageously Large Neural Networks, 2017] [來源：GShard, 2020] [來源：Switch Transformers, 2022] [來源：GLaM, 2022]。DeepSeekMoE 與 DeepSeek-V3 都延續這個思路，前者強調更細粒度的專家切分與 shared experts，後者則用更大的總參數與較小的 activated parameters per token 追求效率 [來源：DeepSeekMoE GitHub README, 2024] [來源：DeepSeek-V3 README mirror, 2024]。

2. 真正的技術瓶頸不是「選專家」本身，而是路由失衡與容量治理

當 router 偏好少數專家時，會出現熱門專家過載、冷門專家學不到東西、token 被丟棄或被迫降級的問題；這也是 Switch Transformer 必須把路由簡化為單一路由專家、並明確引入 load balancing 機制的重要背景 [來源：Switch Transformers, 2022]。Expert Choice Routing 直接把選擇方向改成「專家挑 token」，目的就是改善負載均衡與訓練穩定性 [來源：Mixture-of-Experts with Expert Choice Routing, 2022]。2024 的 dense backprop router 研究則指出，傳統 sparse router 只拿到稀疏梯度，這本身就會惡化平衡與學習品質，因此改進 router 的梯度訊號是重要方向 [來源：Dense Backpropagation Improves Routing for Sparsely-Gated Mixture-of-Experts, 2024]。

3. MoE 的成敗高度依賴系統工程，而不只是模型公式

GShard 明確把 conditional computation 與 automatic sharding 綁在一起處理，說明如果沒有分片與並行策略，MoE 很難真正擴展 [來源：GShard, 2020]。GLaM 與 Switch Transformer 也都把效率敘事建立在「稀疏啟用 + 分散式訓練」的組合上，而不是單靠 router 數學 [來源：GLaM, 2022] [來源：Switch Transformers, 2022]。DeepSeek-V3 延續這條路線，將 MoE 架構與其他效率設計一起包裝成訓練與推論成本控制方案 [來源：DeepSeek-V3 README mirror, 2024]。

4. 最新路由改進方向集中在三件事：更穩定的平衡、更強的專家特化、較少的副作用

Expert Choice 試圖從分配機制上重做平衡 [來源：Mixture-of-Experts with Expert Choice Routing, 2022]；Dense Backprop 研究把注意力放到 router 訓練訊號不足 [來源：Dense Backpropagation Improves Routing for Sparsely-Gated Mixture-of-Experts, 2024]；DeepSeekMoE 則以 fine-grained expert segmentation 與 shared experts isolation 追求更高專家特化與較低冗餘 [來源：DeepSeekMoE GitHub README, 2024]；DeepSeek-V3 再把 auxiliary-loss-free load balancing 納入實務敘事，顯示近年的重點已不是只有「做大 MoE」，而是降低 balancing tricks 對品質與訓練穩定性的副作用 [來源：DeepSeek-V3 README mirror, 2024]。

深度分析

從工程觀點看，MoE router 至少同時解四個問題。第一是 relevance：哪個專家最適合目前 token。第二是 balance：不能讓所有 token 都湧向同一批專家。第三是 capacity：每個專家一次能吃下多少 token。第四是 communication cost：即使理論上只啟用少量專家，跨 GPU / TPU 的 token dispatch 仍可能成為瓶頸 [來源：GShard, 2020] [來源：Switch Transformers, 2022] [來源：GLaM, 2022]。

這四個問題與本專案的多 Skill 路由很像。若把 tech_research、ai_deep_research、github_scout、system_insight 視為專家，今天的 completed tasks 就是在向系統提出不同型態的工作負載。若所有研究都被送進最強但最慢的 agent，會像熱門專家過載；若某些 skill 長期拿不到任務，就像專家飢餓；若沒有 daily cap、fairness、Phase timeout，就等同沒有 capacity factor。這一段是我根據 MoE 論文對負載平衡與容量治理的描述，轉譯到 daily-digest-prompt 的架構推論。

另一個值得借鏡的點是 shared experts。DeepSeekMoE 強調 shared experts isolation，代表某些能力不應完全交給動態路由，而要作為共用基礎能力常駐 [來源：DeepSeekMoE GitHub README, 2024]。轉到本專案，就是把 results validation、memory policy、KB health check 這類共用步驟視為 shared experts，而不是讓 task routing 任意跳過。這可降低因動態選路過度自由而造成的品質波動。

實務應用建議

對本專案最可操作的做法有四項。第一，將 task metadata 結構化成 router features，例如 task_type、requires_web、kb_required、latency_budget、source_count_target，不要只靠 prompt 文字。第二，保留 shared path：像 preamble、dedup、results validation、continuity write-back 應固定執行，不應被路由省略。第三，對高成本 skill 設 capacity 與 overflow policy，例如同時只允許 1 個 heavy research 進行，超出就降級到較輕流程。第四，建立 load-balance 指標，例如每 7 天各 skill 被選中的次數、成功率與平均耗時，這等同於對專家利用率做觀測。

常見陷阱

最常見的錯誤是把 routing 當成單純分類問題，只追求「選得準」，卻忽略專家飽和與系統成本。Switch、Expert Choice 與 Dense Backprop 其實都在提醒同一件事：routing policy 若不處理負載與學習訊號，很快會失衡 [來源：Switch Transformers, 2022] [來源：Mixture-of-Experts with Expert Choice Routing, 2022] [來源：Dense Backpropagation Improves Routing for Sparsely-Gated Mixture-of-Experts, 2024]。第二個陷阱是把所有能力都做成動態路由，結果共用品質閘消失；shared experts 的思路正好可用來反制。第三個陷阱是只看理論 FLOPs，不看實際 dispatch / all-to-all / synchronization 成本，這會讓 MoE 在系統層面失去預期效益 [來源：GShard, 2020] [來源：GLaM, 2022] [來源：Switch Transformers, 2022]。

與本專案的關聯

今天完成的 Todoist 任務可抽出四組技術需求：RAG / 知識庫檢索、Skill 或專家路由、AI 技術追蹤、專案優化治理。其中最值得沉澱的共通底層技術就是 expert routing。未來若要把 daily-digest-prompt 做得更穩定，可考慮把排程與 skill 選擇設計成「共享基礎步驟 + 動態專家步驟」的雙層路由；這不是直接套用 MoE 模型，而是借用其設計原則。

程式碼範例


// 以結構化 feature 做 Skill 路由，shared steps 不經路由
const sharedSteps = ["preamble", "dedup", "results_validation", "continuity_writeback"];

function routeTask(task) {
  const features = {
    taskType: task.taskType,
    requiresWeb: task.requiresWeb,
    kbRequired: task.kbRequired,
    latencyBudget: task.latencyBudget,
    evidenceLevel: task.evidenceLevel,
  };

  if (features.taskType === "tech_research" && features.requiresWeb) return "deep_research";
  if (features.taskType === "project_optimize" && features.kbRequired) return "kb_insight";
  return "default_agent";
}

知識缺口與未來方向

查無可以直接證明「MoE router 最佳實踐可無縫套用到 Agent Skill routing」的單一權威來源，因此這部分只能作為工程推論，而非已驗證定理。下一步較適合進入 mechanism 階段：比較 top-k routing、expert-choice routing、aux-loss-free balancing 在數學目標與系統成本上的差異，並設計一份 daily-digest-prompt 的 routing metrics schema。
