MCP vs A2A — AI Agent 通訊協議雙標準完整指南

概述

2026 年，AI Agent 生態正式進入「協議標準化」時代。兩大核心協議 — MCP（Model Context Protocol） 與 A2A（Agent-to-Agent Protocol） — 共同構成 AI Agent 通訊的基礎設施。兩者並非競爭關係，而是互補的三層架構：MCP 處理 Agent 與工具的連接（Agent-to-Tool），A2A 處理 Agent 之間的協作（Agent-to-Agent）。

2025 年 12 月，Linux Foundation 成立 Agentic AI Foundation（AAIF），由 Anthropic、OpenAI、Google、Microsoft、AWS、Block 共同創立，作為 MCP 和 A2A 的永久治理機構，標誌著產業對 Agent 互操作性標準達成共識。

---

一、MCP（Model Context Protocol）

1.1 核心定位

MCP 是 Anthropic 於 2024 年 11 月開源的協議，標準化 AI Agent 與外部工具、資料來源的連接方式。核心理念：一次整合，處處可用。

1.2 架構設計

  • 通訊模型：Client-Server，使用 JSON-RPC 2.0
  • Client：AI Agent 或宿主應用程式
  • Server：工具提供者，暴露標準化能力
  • 傳輸層：stdio（本地工具）、SSE（遠端伺服器）、Streamable HTTP（生產部署）

1.3 四大能力類型

| 能力 | 說明 |

|------|------|

| Resources | 唯讀資料存取（檔案、資料庫記錄） |

| Tools | 可執行操作（API 呼叫、計算） |

| Prompts | 可重用的提示模板 |

| Sampling | 反向 LLM 呼叫（工具請求模型推理） |

1.4 2026 年採用數據

  • 9,700 萬月下載量（Python + TypeScript SDK 合計）
  • 5,800+ 公開 MCP Server
  • 所有主要 AI 供應商均已採用：Anthropic、OpenAI、Google、Microsoft、Amazon
  • 內建於：Claude Desktop、VS Code、Cursor、Windsurf、Zed、JetBrains IDE

---

二、A2A（Agent-to-Agent Protocol）

2.1 核心定位

A2A 是 Google 於 2025 年 4 月發布的協議，專注於 AI Agent 之間的任務協調、能力發現與協作通訊。IBM 的 Agent Communication Protocol（ACP）於 2025 年 8 月合併入 A2A。

2.2 架構設計

  • 通訊模型：Client-Remote，使用 HTTP/JSON
  • Agent Card：JSON 元資料文件，發布於 /.well-known/agent.json，描述 Agent 能力
  • Task：工作單元，具有明確狀態機（submitted → working → input-required → completed/failed）
  • Messages：通訊載體，包含 Parts（文字、檔案、資料）
  • Artifacts：任務交付物
  • 串流：透過 Server-Sent Events 實現即時更新

2.3 狀態管理（A2A 關鍵優勢）

A2A 是有狀態的協議，支援長時間執行的多步驟任務。Task 物件的狀態機讓 Client 和 Server 對任務進度有一致的理解。相比之下，MCP 是無狀態的，專注於即時工具呼叫與回應。

2.4 Agent 發現機制

Agent Card 提供三種發現路徑：

  1. Well-known URI：存取 Agent 網域的 /.well-known/agent.json
  2. 註冊表/目錄：企業、公開或領域專用目錄
  3. 直接配置：Client 端手動配置

這被視為 AI 能力的「App Store 時刻」。

三、MCP vs A2A 技術比較

| 面向 | MCP | A2A |

|------|-----|-----|

| 定位 | Agent ↔ Tool | Agent ↔ Agent |

| 通訊協議 | JSON-RPC 2.0 | HTTP/JSON + JSON-RPC 2.0 |

| 狀態管理 | 無狀態（工具回應隱含） | 有狀態（明確狀態機） |

| 發現機制 | 手動配置 | 自動（Agent Card） |

| 非同步處理 | 工具特定 | 標準化推送通知 |

| 串流支援 | SSE 可用 | 內建 SSE |

| 安全模型 | 由傳輸層處理 | Agent Card 包含認證資訊 |

| 適用場景 | 工具整合、資料存取 | 多 Agent 協作、任務委派 |

---

四、三層互補架構（2026 新興模式）


Layer 3: A2A    → Agent-to-Agent 編排與協調
Layer 2: MCP    → Agent-to-Tool 能力連接
Layer 1: WebMCP → 結構化網頁存取（機器可讀內容）

實際案例 — 多 Agent PR Review Pipeline：

  1. 編排者透過 Agent Card 發現專業 Agent（A2A）
  2. 平行發送審查任務給安全、效能、風格 Agent（A2A）
  3. 各 Agent 內部使用 MCP Server 存取 GitHub、Lint 工具、分析平台
  4. 結果匯聚為完整的程式碼審查報告

五、Agentic AI Foundation（AAIF）

5.1 成立背景

2025 年 12 月，Linux Foundation 宣布成立 AAIF，統一治理 AI Agent 標準。

5.2 創始專案

| 專案 | 捐贈者 | 用途 |

|------|--------|------|

| MCP | Anthropic | Agent-to-Tool 標準協議 |

| goose | Block | 開源、本地優先 AI Agent 框架 |

| AGENTS.md | OpenAI | 專案級 AI Agent 指引標準 |

5.3 成員結構

  • Platinum：AWS、Anthropic、Block、Bloomberg、Cloudflare、Google、Microsoft、OpenAI
  • Gold：Cisco、Datadog、Docker、IBM、JetBrains、Okta、Oracle、Salesforce、SAP、Shopify、Snowflake 等 18 家
  • Silver：Hugging Face、Pydantic、Uber、Zapier 等 23 家

5.4 治理意義

  • 確保 MCP 和 A2A 保持開放、中立、社群驅動
  • 統一治理避免標準分裂
  • 為企業採用提供法律與治理保障

---

六、對系統架構的啟示

6.1 現有 Agent 團隊可演進方向

目前多 Agent 系統常用自定義 JSON 檔案交接（如 results/*.json），未來可標準化為：

  • Agent 間通訊：採用 A2A 的 Task 狀態機，取代自定義 FSM
  • 工具整合：已使用 MCP 的系統可受益於標準化發現與安全模型
  • Agent 發現：Agent Card 可取代 hardcoded Agent 清單

6.2 何時用 MCP、何時用 A2A

| 場景 | 建議協議 |

|------|----------|

| Agent 需要查詢資料庫 | MCP |

| Agent 需要呼叫外部 API | MCP |

| 多個 Agent 協作完成任務 | A2A |

| Agent 需要委派子任務 | A2A |

| Agent 內部使用工具執行子任務 | MCP（A2A Server 內部） |

6.3 實施路線建議

  1. 短期：維持 MCP 整合，標準化 Agent Card 描述
  2. 中期：導入 A2A Task 狀態機取代自定義狀態追蹤
  3. 長期：實現完整三層架構，Agent 可跨平台互操作

---

七、2026 年路線圖展望

  • MCP 2026 路線圖：擴展允許 MCP Server 本身作為 Agent 行動（如「旅行 Agent」MCP Server 可自主與「訂票 Agent」MCP Server 協商）
  • A2A 演進：更豐富的任務生命週期管理、跨組織信任鏈、Agent 市場
  • AAIF 統籌：兩協議將在 AAIF 下進一步收斂，可能出現統一的「Agent 互操作性規範」
