Claude Code Multi-Agent Teams 架構深度研究

一、概述

Claude Code 自 v2.1.32（2026-01 前後）起推出「Agent Teams」研究預覽功能，標誌著從單一 Agent 執行模式正式邁入多 Agent 協作時代。此功能允許多個 Claude Code Agent 實例在同一專案中並行協作，透過結構化的訊息傳遞協定（SendMessage）實現任務分配、狀態同步與結果組裝。截至 v2.1.63（2026-02-28），Agent Teams 已經歷多輪穩定性與記憶體管理優化，成為企業級 AI 輔助開發的核心架構基礎。

二、核心架構元件

2.1 啟用方式與前置條件

Agent Teams 目前仍為實驗性功能（research preview），需要透過環境變數啟用：


{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

此設定置於 ~/.claude/settings.json 中。啟用後，Claude Code 會在 tmux session 中衍生（spawn）多個 Agent 實例作為隊友（teammates），各自擁有獨立的 context window 與工具權限。

2.2 SendMessage 通訊協定

Agent Teams 的核心通訊機制是 SendMessage 工具，支援以下訊息類型：

  • message：點對點直接訊息，指定 recipient（隊友名稱），附帶 summary 作為 UI 預覽
  • broadcast：廣播訊息，同時發送給所有隊友（成本高，每個隊友一份獨立投遞）
  • shutdown_request / shutdown_response：優雅關閉協定，隊友可 approve 或 reject
  • plan_approval_request / plan_approval_response：計畫審批機制，支援 plan_mode_required 的隊友

關鍵設計原則：隊友的純文字輸出不可見於 Team Lead 或其他隊友，所有跨 Agent 通訊必須透過 SendMessage 工具。這確保了訊息的可追蹤性與結構化。

2.3 tmux 隔離執行

Agent Teams 的實體隔離層建立在 tmux 之上：

  • 每個隊友在獨立的 tmux pane 中運行
  • 環境變數（API provider credentials）會自動傳播到 tmux 衍生的子進程
  • v2.1.45 修復了 Bedrock、Vertex、Foundry 等第三方 API provider 在 tmux 環境中的傳播問題
  • 隊友之間透過 tmux 的 IPC 機制交換結構化訊息

2.4 Agent 定義與自訂

Agent 定義檔案位於 .claude/agents/*.md，支援以下 frontmatter 欄位：

  • model：指定此 Agent 使用的模型（v2.1.47 修復了自訂 model 欄位在 team spawn 時被忽略的問題）
  • isolation: worktree：宣告式 Git Worktree 隔離（v2.1.50 新增），Agent 在獨立工作樹中操作
  • background: true：始終作為背景任務執行（v2.1.49 新增）
  • memory：持久化記憶，支援 user、project、local 三種作用域（v2.1.33 新增）
  • tools：限制可用工具清單，支援 Task(agent_type) 語法限制子 Agent 生成（v2.1.33 新增）

claude agents 命令可列出所有已設定的 Agent（v2.1.50 新增 CLI 命令）。

三、Hook 事件與多 Agent 生命週期

3.1 多 Agent 專用 Hook 事件

Claude Code Hooks 系統為 Agent Teams 提供了完整的生命週期監控：

  • TeammateIdle（v2.1.33 新增）：隊友閒置時觸發，可用 exit 2 阻止閒置並提供回饋
  • TaskCompleted（v2.1.33 新增）：任務完成前品質閘門，exit 2 可阻止標記完成
  • SubagentStart / SubagentStop：子 Agent 生命週期開始/結束事件
  • WorktreeCreate / WorktreeRemove（v2.1.50 新增）：工作樹建立/移除事件
  • ConfigChange（v2.1.49 新增）：設定檔變更審計事件

3.2 HTTP Hooks（v2.1.63 新增）

除了傳統的 shell command hooks，v2.1.63 新增了 HTTP hooks，可將 JSON POST 到指定 URL 並接收 JSON 回應，適合與外部微服務整合的多 Agent 場景。

四、記憶體管理與長時間運行優化

多 Agent 場景的最大挑戰是記憶體膨脹。Claude Code 團隊在近期版本中進行了大量優化：

v2.1.63 關鍵修復

  • 修復長時間運行的隊友即使在 conversation compaction 後仍保留所有訊息於 AppState 的記憶體洩漏
  • 改善多 Agent session 中的記憶體使用，釋放已完成子 Agent 的任務狀態

v2.1.50 關鍵修復

  • 修復 Agent Teams 中已完成隊友任務未被垃圾回收的記憶體洩漏
  • 修復已完成任務狀態物件未從 AppState 移除的問題
  • 改善長 session 中的記憶體使用：清除大型工具結果、caps file history snapshots、釋放 LSP 診斷資料

v2.1.49 關鍵修復

  • 改善多 Agent session 中子 Agent 的記憶體使用，壓縮期間剝離重型 progress message payloads
  • 修復 Yoga WASM 線性記憶體無限成長的問題
  • 修復 tree-sitter parser 的 WASM 記憶體成長，定期重置

五、社群生態系統：oh-my-claudecode

社群專案 oh-my-claudecode（8,064 Stars）提供了 Agent Teams 之上的高階編排層：

5.1 Team 模式管線

Team 是 OMC v4.1.7+ 的標準編排介面，以分階段管線執行：

team-plan -> team-prd -> team-exec -> team-verify -> team-fix (loop)

  • team-plan：需求分析與任務規劃
  • team-prd：產品需求文件生成
  • team-exec：N 個並行 Agent 執行任務
  • team-verify：自動驗證結果
  • team-fix：修復未通過驗證的項目（迴圈直至通過）

5.2 跨模型編排

OMC v4.4.0+ 支援在 tmux 中混合使用不同 CLI 工具：

  • /omc-teams N:codex：N 個 OpenAI Codex CLI panes
  • /omc-teams N:gemini：N 個 Google Gemini CLI panes
  • /omc-teams N:claude：N 個 Claude CLI panes
  • /ccg：三模型並行（Codex 分析 + Gemini 設計 + Claude 合成）

5.3 32 個專業化 Agent

OMC 內建 32 個專業化 Agent（架構、研究、設計、測試、資料科學等），配合智慧模型路由（Haiku 處理簡單任務、Opus 處理複雜推理）。

六、企業級實踐模式

6.1 PowerShell 外部編排模式

本專案（daily-digest-prompt）採用 PowerShell 7 的 Start-Job 並行啟動多個 claude -p 實例，實現外部編排的 Agent Teams：

  • Phase 1：5 路並行擷取（Todoist + 新聞 + HN + Gmail + 安全審查）
  • Phase 2：單一組裝 Agent 讀取 results/*.json 合成最終輸出
  • Phase 3（Todoist 模式）：組裝結果 + 關閉任務 + 推播通知

此模式不依賴 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS，透過檔案系統（results/*.json）實現 Agent 間的資料交換，適合需要精確控制並行度與 timeout 的生產環境。

6.2 原生 Agent Teams vs 外部編排比較

| 維度 | 原生 Agent Teams | 外部編排（PowerShell） |

|------|-----------------|---------------------|

| 通訊方式 | SendMessage 工具 | 檔案系統 |

| 隔離層 | tmux panes | Start-Job 背景作業 |

| 任務分配 | LLM 自主決策 | 腳本預定義 |

| 記憶體管理 | 內建 GC 與 compaction | 各進程獨立 |

| 錯誤處理 | TeammateIdle / TaskCompleted hooks | Phase timeout + 重試 |

| 適用場景 | 互動式開發、複雜推理 | 排程任務、批次處理 |

6.3 安全治理

多 Agent 場景的安全考量：

  • Hook 強制層：所有 Agent 實例共享 .claude/settings.json 的 Hook 規則
  • Sandbox 模式：v2.1.38 阻止寫入 .claude/skills 目錄
  • managed settings：v2.1.33 修復 disableAllHooks 設定必須尊重 managed settings 階層
  • Worktree 隔離：每個 Agent 可在獨立 Git 工作樹中操作，避免並發修改衝突

七、最新進展與展望（2026 Q1）

7.1 v2.1.63（2026-02-28）亮點

  • HTTP Hooks 支援（JSON POST/Response）
  • claude.ai MCP connectors 可在 Claude Code 中使用
  • 跨 Git Worktree 共享 Project configs 與 auto memory
  • 大量記憶體洩漏修復（隊友 AppState、MCP server fetch cache、JSON parsing cache 等）

7.2 發展趨勢

  1. 從實驗性到穩定：Agent Teams 仍需 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1，預計後續版本將正式 GA
  2. 跨模型協作：oh-my-claudecode 已展示 Claude + Codex + Gemini 三模型協作的可行性
  3. 記憶體效率持續改進：每個版本都帶來新的記憶體洩漏修復，表明長時間多 Agent session 是核心優化方向
  4. Hook 生態完善：TeammateIdle、TaskCompleted 等事件讓外部系統可精細控制 Agent 生命週期
  5. Worktree 隔離成熟：從 v2.1.49 的 --worktree flag 到 v2.1.50 的宣告式 isolation: worktree，隔離機制日趨完善
