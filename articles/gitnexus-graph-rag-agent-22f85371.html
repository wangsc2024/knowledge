一、專案概述

GitNexus（abhigyanpatwari/GitNexus）是一個零伺服器的代碼智能引擎，定位為「The Zero-Server Code Intelligence Engine」。它能將任意程式碼庫索引為知識圖譜——映射每一個依賴、呼叫鏈、功能叢集與執行流——然後透過內建的 Graph RAG Agent 和 7 個 MCP 工具暴露給 AI 助手使用。

核心數據

| 指標 | 數值 |

|------|------|

| GitHub Stars | 17,500+（2026-03） |

| 授權 | PolyForm Noncommercial 1.0.0 |

| 語言支援 | 13 種（TypeScript, JavaScript, Python, Java, Kotlin, C#, Go, Rust, PHP, Ruby, Swift, C, C++） |

| 部署模式 | CLI + MCP Server / 瀏覽器 Web UI（WebAssembly） |

| 核心技術 | Tree-sitter AST + KuzuDB + Leiden 演算法 + BM25/語義混合搜尋 |

兩種運行模式

  1. CLI + MCP（推薦）：本地索引 + MCP Server，整合 Cursor、Claude Code、Windsurf 等編輯器
  2. Web UI：完全在瀏覽器中運行（KuzuDB WASM + Tree-sitter WASM），無需伺服器，資料不離開本機

---

二、核心創新：預計算關係智能（Precomputed Relational Intelligence）

這是 GitNexus 區別於傳統 Graph RAG 的關鍵創新。傳統方法需要 LLM 執行多次查詢來探索圖譜結構，而 GitNexus 在索引時預先計算結構——叢集化、執行流追蹤、影響力評分——使得工具呼叫能在單次回應中返回完整上下文。

三大優勢

| 優勢 | 說明 |

|------|------|

| 可靠性 | 單次工具呼叫即返回完整上下文，無需多輪查詢 |

| Token 效率 | 消除多查詢鏈的 Token 浪費 |

| 模型民主化 | 較小的 LLM 也能透過結構化工具回應有效運作 |

圖譜查詢 vs 向量搜尋

代碼關係是結構性的，而非語義性的。「哪些類別繼承自 X？」這類問題有精確、確定性的答案。向量搜尋給出「看起來相關」的結果；圖譜遍歷給出實際答案。這就是為什麼基於 AST 分析的知識圖譜與基於嵌入的 RAG 系統有本質區別——圖譜不猜測。

三、六階段索引管線（Indexing Pipeline）

管線架構


Structure → Parsing → Resolution → Clustering → Processes → Search

| 階段 | 名稱 | 功能 |

|------|------|------|

| 1 | Structure | 檔案系統樹映射 |

| 2 | Parsing | 透過 Tree-sitter 提取完整 AST，識別每個符號（函數、類別、變數） |

| 3 | Resolution | 跨檔案解析 import、函數呼叫與型別引用 |

| 4 | Clustering | 使用 Leiden 演算法進行功能社群偵測（cohesion scoring） |

| 5 | Processes | 從進入點追蹤執行流 |

| 6 | Search | 建立混合索引（BM25 + 語義搜尋 + RRF 排名） |

技術棧

  • 解析器：Tree-sitter（原生 + WASM 雙模式）
  • 圖資料庫：KuzuDB（原生 + WASM，支援嵌入式/客戶端）
  • 搜尋引擎：BM25 + 語義向量 + Reciprocal Rank Fusion
  • 圖查詢：Cypher 查詢語言
  • 叢集演算法：Leiden（功能社群偵測）
  • 前端：React Web UI
  • 後端：Node.js/TypeScript CLI

四、7 個 MCP 工具（Agent 介面）

| # | 工具名稱 | 功能 | 使用場景 |

|---|---------|------|----------|

| 1 | list_repos | 發現已索引的倉庫 | Agent 初始化時列出可用代碼庫 |

| 2 | query | 混合搜尋 + 執行流分組 | 搜尋特定功能或概念 |

| 3 | context | 360 度符號視圖 | 深入理解特定函數/類別的全貌 |

| 4 | impact | 爆炸半徑分析（含信心評分） | 修改前評估影響範圍 |

| 5 | detect_changes | Git diff 影響映射 | 程式碼審查、PR 分析 |

| 6 | rename | 多檔案協調重命名 | 安全的跨檔案重構 |

| 7 | cypher | 原始圖查詢 | 進階自定義查詢 |

Graph RAG 檢索流程

向量檢索返回 top-k 區塊後，GitNexus 會走訪圖譜拉入相關符號：

  • 若檢索到的函數呼叫了工具函數 → 自動加入工具函數
  • 若類別繼承自基底類別 → 自動加入基底類別定義
  • 遍歷深度通常限制在 1-2 跳（hops）

---

五、多倉庫架構

  • 全域註冊表：~/.gitnexus/registry.json 追蹤所有已索引倉庫
  • 單一 MCP Server 透過連接池服務多個專案
  • 索引儲存在每個倉庫的 .gitnexus/ 目錄（可攜式，已 gitignore）

快速啟動


npx gitnexus analyze          # 索引當前倉庫
npx gitnexus setup            # 配置 MCP 連接到編輯器
npx gitnexus mcp              # 啟動 MCP Server
gitnexus serve                # 啟動 HTTP Server（Web UI）

七、關鍵洞察與可借鑑技術

洞察 1：代碼理解比代碼生成更重要

Glen Rhodes 指出：「代碼理解是比代碼生成更困難且更被忽視的問題。」AI 開發工具過度聚焦生成，卻忽略了開發者加入團隊、面對遺留代碼、導航陌生 monorepo 時的真正痛點——理解必須先於修改。

洞察 2：預計算結構是 Graph RAG 的關鍵突破

傳統 RAG 讓 LLM 在推理時探索圖譜，導致多輪查詢和不穩定結果。GitNexus 的「索引時預計算」模式讓較小的 LLM 也能獲得完整上下文，這對降低推理成本和提高一致性至關重要。

洞察 3：Cypher 查詢取代向量搜尋處理結構性問題

代碼的依賴關係、繼承層級、呼叫鏈是結構性問題，有確定性答案。向量搜尋適合語義相似度，圖查詢適合結構精確性。兩者互補而非替代。

洞察 4：MCP 成為 AI Agent 工具整合的事實標準

GitNexus 選擇 MCP 作為 Agent 介面而非自定義 API，反映了 2026 年 AI 工具生態的趨勢——MCP 正成為連接 LLM 與外部工具的通用協議。

洞察 5：客戶端優先架構保障資料主權

透過 WebAssembly 將 KuzuDB 和 Tree-sitter 搬到瀏覽器，GitNexus 實現了「零伺服器」承諾。這對處理專有代碼庫的企業和個人開發者都極具吸引力。

九、限制與風險

  1. PolyForm Noncommercial 授權：限制企業商用，可能影響長期採用
  2. 語言支援差異：13 種語言的解析能力不均（import 解析、型別推斷、框架偵測等能力因語言而異）
  3. 大型倉庫效能：客戶端模式處理超大型 monorepo 可能遇到記憶體瓶頸
  4. 社群成熟度：2026-02 才上 Trending，生態尚在早期階段
