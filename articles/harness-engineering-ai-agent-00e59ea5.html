Harness Engineering：AI Agent 系統優化完整指南（2026）

研究摘要

研究日期：2026-04-22

研究主題：Harness Engineering 核心概念、實作方法與系統優化方案

資料來源：Martin Fowler、LangChain、HumanLayer、NxCode 等權威來源

核心發現：Harness Engineering 是 2026 年 AI Agent 領域最重要的突破，證實「Agent = Model + Harness」，且 Harness 的影響力遠超模型本身。LangChain 實證顯示，透過改進 Harness（而非升級模型），coding agent 性能從 52.8% 提升至 66.5%（+13.7%），躍升至 Terminal Bench 2.0 排行榜前 5 名。

---

一、核心概念

1.1 定義

Harness Engineering 是設計環境、約束、反饋迴圈與基礎設施，使 AI Agent 在規模化下可靠運作的工程實踐。

公式：Agent = Model + Harness

  • Model：LLM 推理引擎（Claude、GPT、Gemini 等）
  • Harness：模型之外的所有支持系統（文件、工具、約束、驗證、反饋）

核心洞察：底層模型的選擇重要性遠低於圍繞它的系統設計。

1.2 四大組件

| 組件 | 功能 | 實作範例 |

|------|------|----------|

| Constraints | 限制 Agent 可執行的操作 | 架構邊界、依賴規則、linters、type checkers |

| Context | 告知 Agent 應該做什麼 | CLAUDE.md、SKILL.md、API 文件、系統提示 |

| Verification | 驗證 Agent 是否正確執行 | 測試套件、靜態分析、CI 檢查 |

| Correction | Agent 出錯時自我修正 | 反饋迴圈、錯誤重試、自我修復機制 |

---

二、控制機制：Guides vs Sensors

2.1 Feedforward（Guides）：預防性控制

在 Agent 執行前介入，引導行為方向：

| 類型 | 子類別 | 範例 | 特性 |

|------|--------|------|------|

| Computational | 確定性工具 | LSP、linters、type checkers、scripts | 毫秒級、確定性 |

| Inferential | 語義化指引 | CLAUDE.md、SKILL.md、設計原則 | 較慢、非確定性 |

優勢：防患於未然，避免錯誤發生

風險：過度約束可能限制 Agent 創造力

2.2 Feedback（Sensors）：修正性控制

在 Agent 執行後檢測，觸發修正：

| 類型 | 子類別 | 範例 | 特性 |

|------|--------|------|------|

| Computational | 靜態分析 | 測試失敗、lint 錯誤、結構驗證器 | 快速、可重複 |

| Inferential | 語義判斷 | Code review agents、AI judges | 慢、依賴模型 |

優勢：捕捉實際執行中的問題

風險：若無 Guides，Agent 可能重複相同錯誤

最佳實踐：同時使用 Guides 和 Sensors，形成「預防+修正」的雙重保障。

---

三、三大調控類別與成熟度

| 類別 | 關注點 | 成熟度 | 挑戰 |

|------|--------|--------|------|

| Maintainability | 程式碼品質、風格、複雜度 | 高（成熟工具鏈） | 語義判斷缺口 |

| Architecture Fitness | 效能、可觀測性、結構約束 | 中等 | 整合複雜度 |

| Behavior | 功能正確性 | 低（未成熟） | 測試品質不穩定 |

關鍵挑戰：Behavior Harness 仍高度依賴 AI 生成的測試覆蓋率與人工驗證，尚未形成可靠的自動化流程。

四、LangChain 實踐案例：52.8% → 66.5%

4.1 成果

  • 基準測試：Terminal Bench 2.0
  • 提升幅度：+13.7 百分點（52.8% → 66.5%）
  • 排名躍升：Top 30 → Top 5
  • 關鍵：未更換模型，純 Harness 改進

4.2 四大改進措施

#### (1) Build & Self-Verify 迴圈

實現「規劃 → 建構 → 驗證 → 修復」四階段流程：

  • 強化系統提示，引導 agent 進行測試而非僅自我審視
  • 添加 PreCompletionChecklistMiddleware，在 agent 結束前強制執行驗證
  • 鼓勵編寫可測試的程式碼，說明「工作會受程式測試評估」

#### (2) 環境上下文注入

  • LocalContextMiddleware 在啟動時映射目錄結構和可用工具
  • 注入時間預算警告，幫助 agent 在有限時間內完成工作
  • 提供關於評估標準和約束條件的明確指導

#### (3) 迴圈檢測與恢復

LoopDetectionMiddleware 追蹤單一檔案編輯次數，在過度重複後提示 agent「考慮重新評估方法」，防止 agent 陷入無限修復迴圈。

#### (4) 推理預算最適化

採用「reasoning sandwich」策略：

  • 規劃階段：高推理級別（xhigh）
  • 實施階段：較低級別（high）
  • 驗證階段：高推理級別（xhigh）

平衡計算成本與準確性。

4.3 驗證機制

自動化痕跡分析：

  1. 從 LangSmith 收集實驗痕跡
  2. 並行執行錯誤分析 agent，綜合主 agent 提供建議
  3. 將此流程製成可重複使用的 Agent Skill

類似機器學習的提升演算法，專注於前次執行的失敗案例。

---

五、約束演化管線（Constraint Evolution Pipeline）

5.1 核心理念

Reactive Constraint Engineering：從生產環境的失敗中學習，而非預先過度優化。

5.2 四階段流程


Start Minimal → Deploy & Observe → Engineer Solutions → Iterate Deliberately

| 階段 | 行動 | 原則 |

|------|------|------|

| Start Minimal | 避免預先優化 harness 配置 | 不猜測可能的失敗模式 |

| Deploy & Observe | 讓 agents 自然失敗 | 收集真實失敗案例 |

| Engineer Solutions | 針對特定失敗模式調整配置 | 每次改進都要可驗證 |

| Iterate Deliberately | 測試變更並捨棄無效修改 | 保持 Harness 精簡 |

5.3 實作模式

  1. 發現錯誤 → Agent 在生產環境中犯錯
  2. 分析根因 → 確定錯誤是結構性問題（非偶發）
  3. 工程化解決方案 → 新增約束/驗證/文件
  4. 驗證有效性 → 確保同類錯誤不再發生
  5. 記錄決策 → 更新 ADR（架構決策記錄）

六、Skills-Based Harness Engineering

6.1 Progressive Disclosure（漸進式揭露）

問題：將所有指引塞入系統提示會導致：

  • Context window 膨脹
  • 「指令預算」在工作前就耗盡
  • Agent 難以找到相關指引

解決方案：Skills 模式

  • 將可重複使用的知識打包在 SKILL.md 檔案中
  • 按需載入（agent 只在需要時讀取對應 Skill）
  • 包含補充 markdown 檔案供 agent 參考

6.2 工具分發策略

避免：直接在 prompt 中嵌入工具程式碼

推薦：透過 CLI、executables 或 NPM packages 分發工具

6.3 Back-Pressure 系統（反壓機制）

建立 Agent 自我修正的反饋迴圈：

  1. 驗證層：typecheck、build、unit tests 在 agent 變更後執行
  2. 僅報告錯誤：成功保持靜默，避免污染 context
  3. Hooks 驗證：「每次 agent 停止時執行 typecheck，將錯誤浮現給 agent」，使其能迭代修復直到通過

關鍵洞察：Agent 在能夠驗證自己工作時表現最佳。成功是靜默的；失敗觸發重新介入直到解決。

---

七、Timing：Shift Quality Left

將控制點分佈在開發生命週期的不同階段，依成本和速度排序：

| 階段 | 控制類型 | 範例 |

|------|---------|------|

| Pre-commit | 快速 linters、基本 review agents | ESLint、Prettier |

| Pre-integration | 全面測試套件 | Unit tests、Integration tests |

| Post-integration | 昂貴 sensors | Mutation testing、詳細 code review |

| Continuous monitoring | 漂移偵測、執行時反饋 | APM、Error tracking |

原則：越早發現問題，修復成本越低。

八、針對 Daily-Digest-Prompt 專案的優化建議

8.1 當前 Harness 盤點

優勢：

  1. ✅ 已有完善的 CLAUDE.md + SKILL.md 文件（Context 組件成熟）
  2. ✅ Hooks 系統提供 Constraints 層（pre_bash_guard、pre_write_guard 等）
  3. ✅ 自動任務結果驗證（validate_results.py）
  4. ✅ ADR 架構決策追蹤

待強化：

  1. ⚠️ Behavior Harness 未成熟（自動任務成功率追蹤不足）
  2. ⚠️ 迴圈檢測機制存在但未與 Agent 形成閉環反饋
  3. ⚠️ 缺乏「Plan-Execute-Verify」三階段強制驗證

8.2 優化方案（按優先級排序）

#### P0：Build & Self-Verify 迴圈（參考 LangChain）

實作步驟：

  1. 在所有 todoist-auto-*.md 模板中，強制加入「步驟 N：Self-Verify」
  2. 要求 Agent 在完成主要任務後，執行：

- 讀取 results/todoist-auto-{task_key}.json

- 確認必填欄位完整

- 驗證 summary 非占位符

- 若失敗，觸發一次修正機會（最多 1 輪）

  1. 新增 PreCompletionChecklistMiddleware 概念到 templates/shared/draft-verify.md

預期成效：減少結果檔案格式錯誤率（當前 result_file_missing_or_invalid 失敗類型）

#### P1：迴圈檢測與 Agent 反饋閉環

實作步驟：

  1. 擴展 tools/runaway_session_detector.py，新增「單檔編輯次數」偵測
  2. 當偵測到同一檔案被 Edit ≥3 次時：

- 寫入 state/loop-warning-{session_id}.txt

- 內容：「檔案 X 已被編輯 N 次，建議改用 Read → Write 一次覆寫」

  1. Agent prompt 開頭加入「檢查是否存在 loop-warning 檔案」步驟

預期成效：降低 token 浪費（參考 2026-04-14 log_audit 單次 46 次 Edit 導致 45 次 loop-suspected）

#### P2：推理預算最適化（Reasoning Sandwich）

實作步驟：

  1. 在 config/llm-gate.yaml 新增 reasoning_budget 段落
  2. 定義任務階段與推理級別對應：

```yaml

reasoning_budget:

planning: high # 規劃階段

execution: medium # 執行階段

verification: high # 驗證階段

```

  1. 修改 tools/llm_router.py，依任務階段動態調整 reasoning effort

預期成效：在保持品質的前提下，降低 15-20% token 成本

#### P3：環境上下文注入（LocalContextMiddleware）

實作步驟：

  1. 新增 tools/generate_context_map.py，啟動時掃描：

- 可用 Skills（從 skills/SKILL_INDEX.md）

- 可用工具（從 tools/ 目錄）

- 當前系統狀態（從 state/ 關鍵檔案）

  1. 產出 context/runtime-context-map.json
  2. 所有 Agent prompt 開頭讀取此檔案，獲得「可用資源清單」

預期成效：減少 Agent「不知道有哪些工具可用」的試探性搜尋

#### P4：Fitness Function 量化（Architecture Fitness）

實作步驟：

  1. 新增 config/fitness-functions.yaml：

```yaml

fitness_functions:

- name: "Hook 覆蓋率"

type: "coverage"

threshold: 0.8 # 80%

measurement: "pytest --cov=hooks"

- name: "自動任務成功率"

type: "reliability"

threshold: 0.75 # 75%

measurement: "uv run python tools/task_success_rate.py"

- name: "結果檔案格式合規率"

type: "quality"

threshold: 0.95 # 95%

measurement: "uv run python tools/result_format_compliance.py"

```

  1. 新增 tools/fitness_function_runner.py，定期執行並記錄趨勢
  2. 低於門檻時觸發 ntfy 告警

預期成效：將 Harness 健康度從「主觀判斷」提升為「量化指標」

#### P5：約束演化自動化

實作步驟：

  1. 新增 state/constraint-proposals.jsonl，記錄：

- 失敗類型（error_code）

- 發生頻率

- 建議約束（proposed_constraint）

  1. 新增 tools/constraint_evolution.py，每週分析：

- 哪些失敗類型出現 ≥3 次

- 自動生成約束提案（例如：新增 Hook 規則、更新 SKILL.md）

  1. 產出 docs/constraint-evolution-report.md，人工審核後落地

預期成效：將「人工發現問題→修 prompt」的流程自動化

十、關鍵要點（TL;DR）

  1. Harness > Model：改進 Harness 的投資報酬率遠高於升級模型（LangChain +13.7% 實證）
  2. Guides + Sensors：同時使用預防性控制與修正性控制，形成雙重保障
  3. Reactive > Proactive：從真實失敗中學習，而非猜測可能的問題
  4. Progressive Disclosure：Skills 模式按需載入知識，避免 context bloat
  5. Self-Verify 是關鍵：Agent 能驗證自己工作時表現最佳
  6. 量化 Fitness：將 Harness 健康度從主觀判斷提升為可量測指標
  7. Shift Quality Left：越早發現問題，修復成本越低
