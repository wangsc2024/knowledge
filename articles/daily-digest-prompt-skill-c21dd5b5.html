Skill 系統 Token 經濟學與漸進式上下文載入優化方案

研究日期：2026-02-16
研究動機：daily-digest-prompt 專案的 15 個 Skill 合計 84,567 bytes SKILL.md，加上 SKILL_INDEX（9,595 bytes）、配置文件（37,011 bytes）、模板（71,693 bytes），Context Window 壓力日益增大。本文從 Token 經濟學角度提出系統性優化方案。

---

一、現況分析：Token 消耗地圖

1.1 各層 Token 消耗估算

以 1 token 約 4 bytes（英文）或 2-3 bytes（中文）估算：

| 層級 | 檔案數 | 總大小 | 估算 Token 數 | 佔比 |

|------|--------|--------|-------------|------|

| SKILL.md（15 個） | 15 | 84,567 B | ~35,000 | 37% |

| 模板（sub-agent + auto-tasks） | 19 | 71,693 B | ~30,000 | 32% |

| 配置（pipeline/routing/等） | 7+ | ~37,000 B | ~15,000 | 16% |

| SKILL_INDEX.md | 1 | 9,595 B | ~4,000 | 4% |

| 主 Prompt（daily/todoist） | 2 | 13,864 B | ~5,500 | 6% |

| 共用前言 | 1 | 1,180 B | ~500 | 1% |

| 合計 | | ~218 KB | ~90,000 | 100% |

1.2 SKILL.md 大小分布（高低差 6.3 倍）

| 排名 | Skill | 大小 | 特徵 |

|------|-------|------|------|

| 1 | task-manager | 11,600 B | 最大，含完整 CRUD 操作流程 |

| 2 | ntfy-notify | 8,875 B | 多平台通知格式 |

| 3 | gmail | 8,875 → 10,019 B | OAuth2 流程複雜 |

| 4 | todoist | 7,879 B | API v1 遷移後指引 |

| 5 | api-cache | 6,414 B | 快取/降級邏輯 |

| ... | ... | ... | ... |

| 14 | hackernews-ai-digest | 1,848 B | 最精簡 |

1.3 關鍵發現

  1. 一次載入全部 Skill 是不必要的：daily-digest 每次實際使用 11/15 個 Skill；Todoist 單次執行通常只用 3-5 個
  2. SKILL_INDEX 每次全量載入：9,595 bytes 的索引包含完整路由決策樹和鏈式組合模式，但大部分執行路徑只用到其中一小段
  3. 配置文件非按需載入：pipeline.yaml + routing.yaml 在啟動時全量讀入，但 Todoist Agent 不需要 pipeline.yaml
  4. 模板重複度高：19 個模板中有大量結構相同的段落（研究註冊表檢查、去重查詢、品質自評迴圈、DONE 認證）

二、漸進式上下文揭露（Progressive Context Disclosure）

2.1 核心概念

借鑑 antigravity-skills 的三級揭露策略：

| 階段 | Token 預算 | 載入內容 | 時機 |

|------|-----------|---------|------|

| 啟動 | ~100 tokens | 角色宣告 + 能力摘要 | Agent 初始化 |

| 激活 | <5,000 tokens | 核心操作指引 | Skill 被觸發時 |

| 深入 | 按需 | 完整 API 參考、邊界案例 | 遇到特定場景時 |

2.2 對 daily-digest-prompt 的具體應用

#### 層級 1：SKILL_INDEX 精簡版（~1,000 tokens → 節省 ~3,000 tokens）

目前 SKILL_INDEX.md 包含：速查表 + 強制規則 + 標籤路由 + 決策樹 + 鏈式組合 + 外部服務對應 + 團隊模式 + 禁止行為，共 ~4,000 tokens。

建議：拆分為兩層

  • SKILL_INDEX_LITE.md（~1,000 tokens）：僅含速查表（名稱 + 觸發關鍵字）+ 禁止行為
  • SKILL_INDEX_FULL.md（保留原檔）：在需要路由決策或團隊協調時才載入

#### 層級 2：SKILL.md 分層結構

每個 SKILL.md 重構為三段：


---
name: todoist
version: 1.0.0
description: 一句話描述
summary: |
  核心操作：查詢(GET /api/v1/tasks/filter)、關閉(POST /api/v1/tasks/{id}/close)、更新(POST /api/v1/tasks/{id})
  注意：filter 端點為 /tasks/filter?query=today，非 /tasks?filter=
---
  • frontmatter（~100 tokens）：啟動時載入，提供能力概覽
  • ## 核心操作（~300-500 tokens）：激活時載入，80% 場景夠用
  • ## 進階用法 / 邊界案例（剩餘）：僅在遇到錯誤或特殊場景時載入

#### 層級 3：配置文件按需載入

| 配置 | 現況 | 建議 |

|------|------|------|

| pipeline.yaml | daily-digest 啟動時全量載入 | 保持（daily-digest 專用） |

| routing.yaml | Todoist 啟動時全量載入 | 拆分為 routing-lite.yaml（標籤映射表）+ routing-rules.yaml（規則說明） |

| frequency-limits.yaml | 啟動時載入 | 改為「進入步驟 2.5 時才載入」 |

| scoring.yaml | 啟動時載入 | 改為「進入步驟 3 時才載入」 |

2.3 預估節省

| 場景 | 現有 Token 消耗 | 優化後消耗 | 節省 |

|------|---------------|-----------|------|

| Daily Digest（典型） | ~45,000 | ~28,000 | ~38% |

| Todoist（有任務） | ~35,000 | ~22,000 | ~37% |

| Todoist（觸發自動任務） | ~40,000 | ~25,000 | ~38% |

| 子 Agent（研究任務） | ~12,000 | ~7,000 | ~42% |

三、模板去重與共用片段抽取

3.1 高重複片段識別

分析 19 個模板，發現以下段落在 5+ 個模板中近乎相同：

| 片段 | 出現次數 | 大小 | 建議 |

|------|---------|------|------|

| 研究註冊表檢查 | 10 個模板 | ~800 B/個 | 抽為 templates/shared/registry-check.md |

| 去重查詢（兩階段） | 10 個模板 | ~1,200 B/個 | 抽為 templates/shared/dedup-search.md |

| 品質自評迴圈 | 10+ 個模板 | ~600 B/個 | 已有 quality-gate.md，但多數模板仍內嵌 |

| DONE 認證格式 | 10+ 個模板 | ~400 B/個 | 已有 done-cert.md，但多數模板仍內嵌 |

| KB 匯入流程 | 8 個模板 | ~500 B/個 | 抽為 templates/shared/kb-import.md |

合計重複量：~3,500 B x 10 個模板 = ~35,000 B（~14,000 tokens 的重複浪費）

3.2 建議的共用片段架構


templates/shared/
  preamble.md          # 已有：共用前言
  done-cert.md         # 已有：DONE 認證
  quality-gate.md      # 已有：品質閘門
  registry-check.md    # 新增：研究註冊表檢查
  dedup-search.md      # 新增：去重查詢
  kb-import.md         # 新增：KB 匯入流程

模板改為引用方式：


## 研究註冊表檢查
讀取 `templates/shared/registry-check.md` 並依指示執行。

四、SKILL_INDEX 平面化 JSON 索引

4.1 靈感來源

antigravity-skills 專案使用 skills_index.json（平面化 JSON）取代 Markdown 索引。LLM 解析 JSON 的速度和準確度優於 Markdown 表格。

4.2 建議格式


{
  "version": 2,
  "skills": [
    {
      "name": "todoist",
      "triggers": ["待辦", "任務", "todo"],
      "summary": "待辦事項查詢/新增/完成",
      "path": "skills/todoist/SKILL.md",
      "size_tokens": 3200,
      "chains_with": ["api-cache", "ntfy-notify"],
      "required_by": ["daily-digest", "todoist-agent"]
    }
  ]
}

4.3 優勢

  1. 精確解析：JSON 結構比 Markdown 表格更易被 LLM 精確理解
  2. 可搜尋：支援程式化查詢（配合 jq 或 Python）
  3. 附帶 Token 估算：size_tokens 讓 Agent 決定是否值得載入
  4. 依賴關係明確：chains_with 和 required_by 支援智慧預載

---

五、Token 預算管理機制

5.1 概念

為每次 Agent 執行設定 Token 預算上限，並在執行過程中追蹤消耗：


# config/token-budget.yaml
budgets:
  daily-digest:
    total: 50000
    init: 5000      # preamble + index + memory
    per_step: 8000   # 每個 pipeline step
    finalize: 5000   # notification + memory update
  todoist-single:
    total: 40000
    init: 6000
    per_task: 10000
    auto_task: 8000
  todoist-team:
    phase1: 8000     # 查詢 Agent
    phase2: 12000    # 每個執行 Agent
    phase3: 5000     # 組裝 Agent

5.2 實施方式

不需要精確的 Token 計數器（LLM 無法精確自測），而是用啟發式規則：

  1. SKILL.md 大小分級：小（<3KB，直接載入）、中（3-6KB，載入核心段）、大（>6KB，分層載入）
  2. 配置文件延遲載入：只在進入對應步驟時才 Read
  3. 模板引用取代內嵌：用 Read templates/shared/xxx.md 取代重複段落

---

六、具體實施路線圖

Phase 1：低成本高收益（預計 2 小時）

| 項目 | 動作 | 預估節省 |

|------|------|----------|

| 模板共用片段抽取 | 建立 3 個 shared 檔案，10 個模板改為引用 | ~14,000 tokens/次 |

| 配置延遲載入 | frequency-limits + scoring 改為按需讀取 | ~3,000 tokens/次 |

| preamble + SKILL_INDEX 讀取順序優化 | 確保不重複載入 | ~500 tokens/次 |

Phase 2：中等投入（預計 4 小時）

| 項目 | 動作 | 預估節省 |

|------|------|----------|

| SKILL.md 三層重構 | 15 個 Skill 加入 summary frontmatter | ~8,000 tokens/次 |

| SKILL_INDEX 精簡版 | 建立 LITE 版，啟動時僅載入 LITE | ~3,000 tokens/次 |

| routing.yaml 拆分 | 映射表與規則說明分離 | ~2,000 tokens/次 |

Phase 3：進階優化（預計 6 小時）

| 項目 | 動作 | 預估節省 |

|------|------|----------|

| 平面化 JSON 索引 | 建立 skills_index.json + 程式化路由 | 精確度提升 |

| Token 預算配置 | 建立 token-budget.yaml + 啟發式追蹤 | 可觀測性 |

| Skill 按需載入驗證 | 追蹤各執行路徑實際載入的 Skill 數量 | 數據驅動優化 |

---

七、風險評估

| 風險 | 影響 | 緩解措施 |

|------|------|----------|

| 分層過多增加 Read 呼叫次數 | 延遲增加 | 控制在 3 層以內，常用路徑不超過 5 次 Read |

| 精簡版遺漏關鍵資訊 | 功能異常 | 保留完整版作為 fallback，精簡版含「若不確定，讀取完整版」提示 |

| JSON 索引與 Markdown 索引不同步 | 路由錯誤 | 用 CI 腳本自動從 SKILL.md frontmatter 生成 JSON |

| 模板引用增加間接性 | 除錯困難 | 共用片段加入版本號，變更時記錄 changelog |

---

八、與現有架構的關係

本優化方案完全符合專案的「文件驅動架構」原則：

  1. Prompt 是薄層調度器 — 進一步精簡 Prompt 層
  2. 按需載入 — 將「按需」從模板層擴展到配置層和 Skill 層
  3. 改配置不改 prompt — token-budget.yaml 作為新增配置
  4. 單一定義處 — 共用片段消除重複定義

與 MEMORY.md 中的 antigravity-skills 參考對照

| antigravity-skills 特性 | daily-digest-prompt 現況 | 本方案建議 |

|------------------------|------------------------|----------|

| 漸進式上下文揭露（3 級） | 部分（模板按需載入） | 擴展到 Skill + 配置層 |

| 平面化 JSON 索引 | 無（Markdown 表格） | Phase 3 建立 |

| Token 預算（80% 方差來源） | 無追蹤 | Phase 3 配置 |

| Evidence-Based 驗證 | 已有（pipeline.yaml finalize） | 保持 |

---

九、結論

daily-digest-prompt 專案的 Skill 系統已具備良好的模組化基礎，但在 Token 經濟學方面仍有顯著優化空間。核心問題是「一次載入太多、重複載入太多」。

透過三階段實施（模板去重 → Skill 分層 → JSON 索引），預估可將單次執行的 Context Window 消耗降低 35-42%，相當於每次執行節省 ~15,000-20,000 tokens。這不僅降低 API 成本，更重要的是為未來新增 Skill 和更複雜的任務鏈保留了 Context Window 空間。
