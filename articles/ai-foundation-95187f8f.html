一、技術概述（100-200 字）

AI 輔助除錯是利用大型語言模型（LLM）和機器學習技術自動識別、診斷和修復軟體錯誤的新興領域。與傳統除錯依賴 stack traces 和 error codes 不同，AI 除錯工具能夠處理語義層級的失敗——例如 AI Agent 檢索到錯誤文件、選擇錯誤工具或傳遞錯誤參數，但仍回傳成功狀態碼。2026 年，這項技術已從實驗室進入生產環境，Notion、Stripe、Cloudflare 等企業已將其整合到 CI/CD 管線中。自動根因分析（Automated RCA）能在數分鐘內完成原本需要數小時的人工分析，將 MTTR（平均修復時間）減少 50-65%。

技術原理：

  • Agentic AI：多步驟分析計畫，橫跨基礎設施驗證並收斂到可辯護的根因
  • 自我修復基礎設施：MTTR 從數小時降至數分鐘，釋放資深工程師

2.3 系統化除錯框架：AgentRx（Microsoft Research, 2026）

設計理念：

  • 將 agent 執行視為系統 trace，需自動化驗證
  • 不依賴 LLM「猜測」錯誤，而是透過可執行約束進行結構化檢查

四階段管線：

  1. Trajectory Normalization（軌跡正規化）：將異質 logs 轉為統一表示
  2. Constraint Synthesis（約束合成）：從 tool schemas 和 domain policies 自動生成可執行約束
  3. Guarded Evaluation（守衛式評估）：逐步檢查約束，產生可審計的驗證日誌
  4. LLM-based Judging（LLM 判定）：使用 9 類失敗分類法識別 Critical Failure Step（第一個不可恢復的錯誤）

效能提升：

  • 失敗定位準確率提升 +23.6%（相較於 prompting baselines）
  • 基準測試：115 個手動標註的失敗軌跡（τ-bench、Flash、Magentic-One）

失敗分類法（9 類）：

  • Plan Adherence Failure（計畫遵循失敗）
  • Hallucination（幻覺）
  • Invalid Invocation（無效調用）
  • [其他 6 類...]

範例 3：三層日誌架構實作


import structlog

# 配置結構化日誌
logger = structlog.get_logger()

class AgentLogger:
    def log_execution(self, step, action, result):
        """Execution logs: 發生了什麼"""
        logger.info("execution",
            step=step,
            action=action,
            result=result,
            layer="execution"
        )

    def log_reasoning(self, step, thought, decision):
        """Reasoning logs: 為什麼發生"""
        logger.info("reasoning",
            step=step,
            thought=thought,
            decision=decision,
            layer="reasoning"
        )

    def log_performance(self, step, latency_ms, cost_usd, tokens):
        """Performance logs: 效能如何"""
        logger.info("performance",
            step=step,
            latency_ms=latency_ms,
            cost_usd=cost_usd,
            tokens=tokens,
            layer="performance"
        )

# 使用範例
agent_logger = AgentLogger()

# Step 1: Tool invocation
agent_logger.log_execution(1, "search_docs", {"docs": 5})
agent_logger.log_reasoning(1, "User needs password reset info", "Use search_docs")
agent_logger.log_performance(1, 250, 0.002, 150)

六、與本專案的應用場景（daily-digest-prompt）

場景 1：自動任務失敗根因分析

本專案現有 systematic-debugging Skill，可整合 AI RCA：


# tools/ai_rca_analyzer.py
def analyze_auto_task_failure(task_key, error_log):
    """
    整合 Energent.ai 或本地 LLM 進行根因分析
    """
    # 讀取失敗日誌
    with open(f"state/failed-auto-tasks.json") as f:
        failures = json.load(f)

    # 提取相關失敗模式
    task_failures = [f for f in failures if f["task_key"] == task_key]

    # 呼叫 AI RCA
    rca_result = llm_analyze(
        prompt=f"分析任務 {task_key} 的失敗根因",
        context={"failures": task_failures, "error_log": error_log}
    )

    return {
        "root_cause": rca_result.root_cause,
        "fix_suggestions": rca_result.suggestions,
        "confidence": rca_result.confidence
    }

  • check-health.ps1 [根因分析] 區塊

七、風險與注意事項

7.1 技術風險

  1. AI 除錯的局限性

- LLM 可能產生幻覺修復建議（需人工審查）

- 無法處理所有類型的語義失敗

- 需大量高品質訓練數據（失敗案例）

  1. 成本考量

- Reasoning logs 需額外 LLM 呼叫（成本 +30% ~ +50%）

- Enterprise RCA 平台授權費用（Dynatrace、Splunk）

- 開源替代方案（Langfuse、Arize Phoenix）需自行維運

  1. 整合複雜度

- 需從設計階段內建可觀測性（非事後補救）

- 多框架環境需 OpenTelemetry 標準化

- Legacy 系統整合成本高

7.2 操作風險

  1. 過度依賴自動化

- 關鍵系統仍需人工審查修復建議

- AI 可能誤判 Critical Failure Step（AgentRx 準確率 ~78%）

  1. 資料隱私

- 上傳 logs 到雲端 RCA 服務需注意敏感資料

- 建議優先選擇可自架方案（Langfuse、Arize Phoenix）

  1. Alert Fatigue 轉移

- 避免「AI 建議疲勞」（過多未經過濾的建議）

- 需設定信心度門檻（confidence threshold ≥ 0.8）

7.3 最佳實踐建議

  1. Evaluation-First 文化

- 每個生產失敗必須轉為永久測試案例（Braintrust 模式）

- CI/CD 整合品質門檻（通過率 ≥ 80%）

  1. 三層日誌必須從 Day 1 實施

- Execution + Reasoning + Performance

- 結構化日誌（JSON Lines 格式）

  1. 段階式導入

- Phase 1: MLflow tracing（低成本）

- Phase 2: 開源可觀測性（Langfuse）

- Phase 3: Enterprise RCA（高流量場景）
