一、技術概述

需求工程（Requirements Engineering）是軟體開發生命週期的起點，直接影響後續設計與實作品質。2026 年，生成式 AI（特別是大型語言模型 LLM）正在重塑需求工程的核心流程：從需求擷取（Requirement Elicitation）到自然語言轉規格化（NL to Formal Spec）、從使用者故事自動生成到驗收條件自動化（Acceptance Criteria Automation），再到需求追溯性管理（Requirements Traceability）。

AI 在需求工程的應用解決了傳統痛點：

  • 需求擷取：人工訪談耗時且易遺漏，AI 聊天機器人可 24/7 對話式擷取需求
  • 規格化：自然語言需求模糊不清，AI 可轉為形式化規格（FOL、LTL、DSL）
  • 需求衝突：手動檢查易疏漏，AI 語義分析可自動偵測衝突（精確度 83.88%）
  • 使用者故事：撰寫耗時且格式不一，AI 可自動生成標準化故事（87.5% 符合品質標準）
  • 追溯性：手動維護 RTM 繁瑣，AI 可自動化建立需求追溯矩陣並偵測 gap

二、核心機制與工具

2.1 需求擷取（Requirement Elicitation with AI Chatbot）

工作原理：

AI 聊天機器人使用自然語言處理（NLP）與對話式 AI，與利害關係人進行結構化對話，擷取並分類需求。

關鍵技術：

  • 對話式需求擷取：聊天機器人以開放式問題引導使用者表達需求
  • 意圖識別（Intent Recognition）：從對話中識別功能需求與非功能需求
  • 自動分類：NLP 技術自動將需求分類為功能性（Functional）與非功能性（Non-functional）

實證效果：

  • 需求擷取效率提升：比傳統訪談更快速
  • 需求品質提升：結構化對話減少遺漏與模糊性
  • 錯誤減少：自動分類降低人工疏失

參考文獻：

  • Intelligent Chatbot for Requirements Elicitation and Classification
  • Framework for automating requirement elicitation using a chatbot

2.2 自然語言轉規格化（NL to Formal Spec Transformation）

工作原理：

使用 LLM 將非形式化的自然語言需求轉換為可執行的形式化規格，包括：

  • 正規表示式（Regular Expressions, Regex）
  • 一階邏輯（First-Order Logic, FOL）
  • 線性時序邏輯（Linear Temporal Logic, LTL）
  • 領域特定語言（Domain-Specific Language, DSL）

核心方法：

#### (1) nl2postcond 方法

將自然語言意圖轉為程式斷言（Postconditions）：

  • 使用 LLM 生成形式化的程式斷言
  • 精確度高到可偵測工業級專案的歷史 bug（64/70 buggy methods 被偵測）
  • 適用於軟體驗證與定理證明

#### (2) NL-to-DSL 管線

整合三層機制：

  • LLM 驅動解析：將自然語言轉為 DSL 草稿
  • 檢索增強生成（RAG）：從知識庫擷取相關範例增強轉換精確度
  • 符號後驗證（Symbolic Postvalidation）：驗證生成的 DSL 是否符合語法與語義

#### (3) NL2TL 框架

將英文指令轉為時序邏輯（Temporal Logic）：

  • 建立 28K NL-TL 配對資料集（結合 LLM 與人工標註）
  • 適用於硬體規格語言與工業驗證

實證效果：

  • 自動化程度高：減少手動撰寫形式化規格的時間
  • 可追溯性：保留句子級別的追溯性（sentence-level traceability）
  • 可驗證性：生成的規格可直接用於自動化驗證

參考文獻：

  • Can Large Language Models Transform Natural Language Intent into Formal Method Postconditions?
  • Formal Specifications from Natural Language
  • NL2TL: Transforming Natural Languages to Temporal Logics using Large Language Models

2.3 需求衝突偵測（Requirement Conflict Detection）

工作原理：

使用 AI 自動偵測需求間的邏輯衝突、重複或不一致。

核心方法：

#### (1) FSARC（Finer Semantic Analysis-based Requirements Conflict Detector）

  • 語義組成分析：分析自然語言功能需求的細粒度語義
  • 性能指標：召回率近 100%，精確度 83.88%
  • 適用範圍：自動偵測自然語言功能需求間的衝突

#### (2) SR-BERT（Transformer-based Model）

  • 架構：使用 Sentence-BERT 與 Bi-encoders
  • 多階段微調：在需求資料集上微調 Transformer 模型
  • 優勢：語義理解能力強，適合處理複雜需求文件

#### (3) LLM-based Conflict Detection Tool

  • 實作案例：使用 Gemini API（gemini-1.5-flash）分析需求對
  • 工作流程：從 CSV 讀取需求 → 成對比較 → 偵測衝突 → 分類衝突類型
  • 應用領域：軟體工程、航太、汽車系統

實證效果：

  • 早期偵測：在開發前期發現衝突，降低後期修正成本
  • 自動化程度高：減少人工審查負擔
  • 精確度高：FSARC 精確度 83.88%，召回率近 100%

參考文獻：

  • Automatically detecting the conflicts between software requirements based on finer semantic analysis
  • Automatic conflict detection and resolution in digital service network requirement models using large language models
  • GitHub - Ai-Requirement-Analysis

2.4 使用者故事自動生成（Automated User Story Generation）

工作原理：

從需求文件自動生成結構化的使用者故事（User Story），格式為：「As a [role], I want [feature], so that [benefit]」。

核心工具與方法：

#### (1) GeneUS（基於 GPT-4.0）

  • 輸入：需求文件（Requirements Engineering 階段產出）
  • 輸出：結構化使用者故事
  • 策略：RaT（Refine and Thought）prompting，減少 LLM 在複雜輸入時的幻覺問題

#### (2) US-Prompt 方法

  • 品質指標：87.5% 的使用者故事符合 >75% 的品質標準
  • 品質標準：包含角色、功能、效益三要素，語義清晰，可測試

#### (3) 商業工具

  • ChatGPT、Gemini、Claude AI：通用 LLM，需設計良好的 prompt
  • ProductGo、Jira AI Assistant：整合至專案管理工具的 AI 輔助
  • Miro AI、Leiga：視覺化協作平台的 AI 故事生成

最佳實踐：

  1. 清晰的 prompt：避免模糊性，明確指定輸出格式
  2. 人工審查：AI 生成後需 Product Owner 驗證與調整
  3. 一致性格式：AI 確保所有故事遵循 Agile 格式

實證效果：

  • 時間節省：數分鐘完成原本需數小時的工作
  • 格式一致性：自動化確保格式統一
  • 覆蓋率提升：AI 可從多種文件類型（會議紀錄、回饋、需求文件）生成故事

參考文獻：

  • Automated User Story Generation with Test Case Specification Using Large Language Model
  • Take Loads Off Your Developers: Automated User Story Generation using Large Language Model

2.5 驗收條件自動化（Acceptance Criteria Automation）

工作原理：

LLM 將使用者故事的驗收條件（Acceptance Criteria, AC）轉換為可執行的測試案例。

核心工具：

#### (1) AutoUAT & Test Flow（基於 GPT-4 Turbo）

  • AutoUAT：生成驗收測試情境（Acceptance Test Scenarios），95% 使用者認為有用
  • Test Flow：生成驗收測試案例（Acceptance Test Cases），92% 有用率（60% 可直接使用，8% 需微調，24% 需重新生成）
  • 整合：已整合至工業合作夥伴的工作流程

#### (2) Jira MCP + LLM 整合

  • 輸入：Jira ticket 中的驗收條件
  • 輸出：秒級生成測試案例，自動回寫至 ticket
  • 效益：減少 QA 團隊最耗時的手動測試案例撰寫工作

#### (3) TestGen-LLM（Meta 開源實作）

  • 功能：將 AC 轉為單元測試程式碼
  • 適用範圍：自動化單元測試生成，包含邊緣案例

實證效果：

  • 時間節省：秒級生成測試案例（原需數小時）
  • 覆蓋率提升：涵蓋更廣泛的情境，包含邊緣案例與角落案例
  • 品質提升：減少人工疏漏，提升測試完整性

重要注意事項：

  • 人工監督：需驗證 AI 生成的測試案例是否符合業務邏輯
  • 整合至工作流：最有效的應用是整合至現有 CI/CD 流程

參考文獻：

  • Acceptance Test Generation with Large Language Models: An Industrial Case Study
  • Automating Manual Test Cases Generation With JIRA MCP & LLM

2.6 需求追溯性管理（Requirements Traceability with AI）

工作原理：

AI 自動化建立並維護需求追溯矩陣（Requirements Traceability Matrix, RTM），連結需求、設計、程式碼、測試案例。

核心功能：

#### (1) 自動化 RTM 建立

  • 連結建立：自動將需求連結至對應的測試案例、設計文件、程式碼
  • 即時更新：當需求變更時，自動更新追溯關係
  • 覆蓋率分析：即時顯示需求覆蓋率，確保無遺漏

#### (2) 變更影響分析（Change Impact Analysis）

  • 問題：需求變更時，手動評估影響範圍耗時
  • 解決：AI 快速分析變更的下游影響（哪些測試、程式碼需調整）
  • 效益：數秒內完成原本需數小時的影響分析

#### (3) Gap 偵測（Gap Detection）

  • 偵測類型：

- 業務需求未對應功能需求

- 功能需求未對應測試案例

- 測試案例未連結至需求（孤兒測試）

  • 效益：自動化偵測，避免人工遺漏

商業工具：

  • Trace.Space：AI-Native 需求與追溯性平台
  • Visure Solutions：AI 驅動的需求追溯性工具，加速測試與認證
  • Modern Requirements：整合 AI 的追溯性工具

實證效果：

  • 效率提升：自動化建立追溯矩陣，節省大量人力
  • 即時可見性：專案團隊與利害關係人可即時查看需求狀態
  • 認證加速：在航太、汽車等需認證的領域，AI 追溯性可加速認證流程

重要注意事項：

  • 人工監督必要性：AI 不理解業務邏輯，需 Business Analyst 驗證 AI 生成的連結
  • 避免偏見：需人工審查 AI 推薦的連結，避免錯誤關聯

參考文獻：

  • TOP 11 Best Practices for Requirement Traceability with AI
  • Trace.Space: AI-Native Requirements & Traceability Platform
  • AI-Driven Requirements Traceability for Faster Testing and Certification

三、完整工作流程（End-to-End AI Requirements Engineering Pipeline）

將上述機制整合為端到端的 AI 需求工程管線：


1. 需求擷取（Elicitation）
   ↓ AI 聊天機器人對話式擷取
   ↓ 自動分類（功能性/非功能性）
   ↓
2. 需求規格化（Formalization）
   ↓ LLM 將自然語言轉為形式化規格（FOL/LTL/DSL）
   ↓ 符號驗證確保語法正確
   ↓
3. 需求衝突偵測（Conflict Detection）
   ↓ 語義分析偵測衝突（FSARC/SR-BERT）
   ↓ 人工審查與解決衝突
   ↓
4. 使用者故事生成（User Story Generation）
   ↓ LLM 從需求文件生成結構化故事（GeneUS/GPT-4）
   ↓ Product Owner 審查與調整
   ↓
5. 驗收條件自動化（Acceptance Criteria Automation）
   ↓ LLM 將 AC 轉為測試案例（AutoUAT/Test Flow）
   ↓ 自動生成可執行測試程式碼
   ↓
6. 需求追溯性管理（Traceability）
   ↓ AI 自動建立 RTM，連結需求-設計-程式碼-測試
   ↓ 變更影響分析與 gap 偵測
   ↓
7. 持續維護與更新
   ↓ 需求變更時，AI 自動更新追溯矩陣
   ↓ 重新驗證衝突與覆蓋率

4.2 最佳實踐

  1. 人機協作模式：AI 負責自動化產出，人類負責驗證與調整（不是完全替代）
  2. 清晰的 Prompt 設計：避免模糊性，明確指定輸出格式與品質標準

  4. 多階段驗證：生成 → 符號驗證 → 人工審查 → 測試驗證
  5. 追溯性優先：確保 AI 生成的所有工件保留追溯性（sentence-level traceability）
  6. 持續訓練與微調：在專案特定資料集上微調 LLM，提升精確度

五、與 daily-digest-prompt 專案的應用場景

本專案可受益於 AI 需求工程機制：

  1. 自動任務需求規格化：29 個自動任務的需求可用 AI 轉為形式化規格，確保語義一致性
  2. Skill 驗收條件自動化：26 個 Skill 的驗收標準可自動轉為測試案例（整合至現有的 556 個測試套件）
  3. 需求追溯性：建立 ADR（架構決策記錄）→ 實作 → 測試 的自動追溯矩陣

  5. 衝突偵測：多個 Skill 的需求可能衝突（如快取策略不一致），AI 可自動偵測

六、風險與注意事項

  1. LLM 幻覺問題：需人工驗證 AI 生成的規格與測試案例
  2. 業務邏輯理解：AI 不理解特定領域的業務邏輯，需 BA/PO 監督
  3. 過度依賴風險：完全自動化可能導致人類失去對需求的深度理解
  4. 資料隱私：使用雲端 LLM（如 GPT-4）需注意敏感需求的資料隱私
  5. 成本考量：大規模使用 LLM API 有成本，需評估 ROI

---

七、未來展望

  1. 多模態需求擷取：結合文字、語音、圖片、影片的需求輸入
  2. 主動式需求發現：AI 主動從使用者行為日誌中發現潛在需求
  3. 跨專案需求重用：AI 從歷史專案中推薦可重用的需求模式
  4. 即時需求驗證：在需求擷取過程中即時偵測衝突與不一致
  5. 需求品質預測：AI 預測需求的實作難度與變更風險
