NanoBot — 港大超輕量 AI Agent 運行時深度研究報告

執行摘要

本文主張：NanoBot（HKUDS/nanobot）以約 4,000 行 Python 核心程式碼，在保留多平台聊天、MCP 工具、長期記憶與定時任務等完整 Agent 閉環的前提下，將 OpenClaw 級個人助手能力壓縮至可一個下午通讀的可審計框架，是 2026 年 AI Agent「做減法」設計哲學的代表作 [來源:HKUDS/nanobot README,2026][來源:OpenClaw vs Nanobot Comparison,2026]。截至 2026-06-01，GitHub Stars 43,447、Forks 7,675、Open Issues 934，最後推送 2026-05-31，社群活躍度極高 [來源:GitHub API HKUDS/nanobot,2026-06-01]。

---

一、專案概述

NanoBot 由香港大學數據智能實驗室（HKUDS）於 2026-02-01 開源，MIT 授權，定位為「可自託管的超輕量 Agent 運行時」——讓開發者、研究者與隱私敏感用戶能在本機或低規格 VPS 上部署 24/7 個人 AI 助手 [來源:3.5 個月 4.3 万 Star：nanobot,2026][來源:HKUDS/nanobot,2026]。

解決的問題：OpenClaw 等全功能平台雖功能完整（210K+ Stars、ClawHub 數千技能），但程式碼量達 43 萬行以上、記憶體占用 >1GB、啟動慢，且 Karpathy 等開發者對其龐大攻擊面與 prompt injection 風險公開質疑 [來源:The Six Claws Field Guide,2026][來源:Is OpenClaw Overhyped,2026]。NanoBot 針對「想理解並掌控 Agent 核心迴圈」的用戶，提供可讀、可改、可審計的替代方案。

目標使用者：個人開發者、AI 研究者、需要接入微信/飛書/QQ/釘釘等亞太通訊平台的用戶、以及希望在 1C2G VPS 或樹莓派上跑 Agent 的場景 [來源:OpenClaw vs nanobot Asia Channels,2026][來源:Nanobot 超轻量个人 AI 智能体,2026]。

核心價值主張：用「輕量、可讀、可擴展」三詞概括——核心 Agent Loop 約 100 行、整體核心約 4,000 行，卻支援 14+ 聊天平台、25+ LLM 供應商、MCP 多伺服器、五層記憶 + Dream 長期蒸餾、自然語言 Cron 與內建 WebUI [來源:博客园 nanobot 深度剖析,2026][來源:SegmentFault 港大 nanobot 教程,2026]。

二、技術架構

NanoBot 採單進程、事件驅動、以 Agent Loop 為中心的分層架構，刻意避免 LangGraph/AutoGen 式的 DAG 編排或多進程 Gateway/Node 拓撲 [來源:Viblo nanobot Build Guide,2026][來源:Jose Nobile Nanobot Guide,2026]。

2.1 分層模組

| 層級 | 職責 | 技術要點 |

|------|------|----------|

| Channels | 多平台訊息接入 | Telegram、Discord、Slack、WhatsApp、Feishu、QQ、Email、WeChat 等；統一匯入 Message Bus |

| Agent Loop | 核心推理迴圈 | ~100 行 ReAct 式迭代：收訊息 → LLM 決策 → 調工具 → 寫回；無獨立 Planner/Executor 分裂 |

| Providers | LLM 後端抽象 | 統一 chat() 介面；支援 OpenAI/Anthropic/DeepSeek/Groq/Moonshot/智譜/vLLM 等 25+ 供應商與 fallback |

| Tools | 工具執行 | Python Tool 子類 + MCP Stdio/HTTP 包裝 + Sub-agent spawn |

| Memory | 持久化狀態 | 檔案型 MEMORY.md/SOUL.md/USER.md/history.jsonl + Dream 兩階段長期記憶 |

| Cron/Heartbeat | 定時觸發 | 自然語言排程描述 → 轉 cron；Heartbeat 以 inbound message 餵回 Loop |

| Security | 沙箱與存取控制 | restrictToWorkspace、allowFrom 白名單、bwrap 沙箱、workspace confinement |

| API/WebUI | 人機介面 | OpenAI 相容 API、WebSocket 多會話、10 種語言 UI |

[來源:Viblo nanobot Build Guide,2026][來源:andrew.ooo Nanobot Review,2026][來源:Clawdbook OpenClaw vs Nanobot,2026]

2.2 Agent Loop 設計決策

與 OpenClaw 的 Gateway→Node 分散式模型不同，NanoBot 以單 Python 進程處理 LLM 互動、通道適配、記憶、排程與工具執行，部署路徑為 pip install nanobot-ai && nanobot onboard && nanobot gateway [來源:Jose Nobile Nanobot Guide,2026]。這換取的是部署極簡（<1 秒啟動 vs OpenClaw 數十秒）與程式碼可讀性，代價是缺乏跨機器分散式節點能力 [來源:searchmytool OpenClaw vs Nanobot vs PicoClaw,2026]。

MCP-first 擴展：核心保持薄 orchestrator，能力透過 MCP 伺服器外掛（web search、檔案操作、程式碼執行、圖像生成），啟動時 auto-discover 註冊工具——新增能力 = 接新 MCP server，而非改核心程式碼 [來源:ibl.ai Six Claws Field Guide,2026][來源:Medium What is NanoBot,2026]。

2.3 Dream 記憶系統（v0.1.5+）

兩階段設計取代「全塞 prompt」或「強制向量 DB」：

  1. Consolidator（短期）：滾動整理當前會話 history，控制 context 窗口
  2. Dream（長期）：週期性將重要互動蒸餾寫入 markdown 檔案，供後續 session 檢索

[來源:andrew.ooo Nanobot Review,2026][來源:GitHub nanobot discussions/566,2026]

架構圖（概念）：


[Chat Apps] → [Message Bus] → [Agent Loop] ←→ [Providers/LLM]
                    ↑                ↓
              [Cron/Heartbeat]   [Tools/MCP]
                    ↑                ↓
              [Sessions JSON]  [Memory/Dream]
                    ↑
              [WebUI/API]

四、使用方式

4.1 快速上手


# 安裝（Python ≥3.11）
pip install nanobot-ai
# 或
uv tool install nanobot-ai

# 初始化配置（~/.nanobot/config.json）
nanobot onboard

# 啟動 Gateway（WebUI + 通道 + Agent）
nanobot gateway

# Docker 部署
docker compose up -d

[來源:HKUDS/nanobot README,2026][來源:Clawdbook Comparison,2026]

4.2 配置 LLM Provider 範例


{
  "providers": {
    "default": "openai",
    "openai": {
      "api_key": "sk-...",
      "model": "gpt-4o"
    },
    "deepseek": {
      "api_key": "...",
      "model": "deepseek-chat"
    }
  }
}

支援 provider pinning（特定任務綁定特定模型）與 fallback 鏈 [來源:Medium What is NanoBot,2026]。

4.3 接入 MCP 工具

啟動時在 config 中宣告 MCP servers（Stdio 或 HTTP），Agent Loop 自動 discover 並註冊為可調用工具——與 daily-digest-prompt 現有 MCP 生態（n8n-mcp、playwright 等）對齊 [來源:ibl.ai Six Claws,2026]。

六、潛在應用（daily-digest-prompt 落地場景）

場景 1：ntfy HITL 檢查點輕量 Bot

問題：現有 HITL sweep 需 Phase 2 Agent 全量啟動，成本高。

落地：以 NanoBot + MCP（讀 state/human-review-queue.json、發 ntfy）做常駐輕量監聽 Bot；自然語言 Cron「每 30 分鐘檢查待審佇列」。

步驟：① Docker 部署 nanobot gateway；② 註冊 readonly MCP 讀 state；③ 配置 ntfy topic；④ allowFrom 白名單限制觸發源。

場景 2：Todoist 自動任務失敗即時告警通道

問題：Phase 2 失敗目前依排程下一輪才發現。

落地：NanoBot 監聽 results/todoist-auto-*.json 變更（filesystem MCP），失敗時推 Telegram/微信摘要。

步驟：① workspace restrictToWorkspace 指向 repo；② MCP file watcher；③ Provider fallback（DeepSeek→Groq）；④ Dream 記憶累積常見 RCA 模式。

場景 3：研究 Agent 原型驗證沙盒

問題：daily-digest 的 ai_github_research 管線迭代成本高。

落地：用 NanoBot 核心 Loop（~100 行）作為「最小可跑 research agent」原型，驗證新 prompt/工具組合後再移植回 run-todoist-agent-team.ps1。

步驟：① fork nanobot 核心；② 替換 Tools 為 WebSearch/KB import MCP；③ A/B 比較 token 消耗與完成率；④ 成熟後提取 pattern 到 planner_executor。

七、優缺點評估

優點

  1. 極致可讀性：~4,000 行核心，一個下午通讀，安全審計成本低 [來源:腾讯云 Nanobot 4000行,2026]
  2. 資源效率高：RAM ~100–191 MB，1C2G VPS 可 24/7 運行 [來源:Jose Nobile Guide,2026][來源:Jishuzhan 深度剖析,2026]
  3. 亞太平台覆蓋：微信/飛書/QQ/釘釘原生，國內場景優於 OpenClaw [來源:Clawrapid Comparison,2026]
  4. MCP-first 擴展：新增能力不改核心，與 AAIF 標準對齊 [來源:ibl.ai Six Claws,2026]
  5. MIT 真開源：可自由修改、分發、商用 [來源:博客园 4.3万 Star,2026]

缺點

  1. 功能完整性不及 OpenClaw：無 iOS/Android 伴侶 App、無 ClawHub 級技能市場（931 open issues 反映功能請求積壓）[來源:GitHub API,2026-06-01][來源:Clawrapid Comparison,2026]
  2. 無瀏覽器自動化內建：需靠 MCP 外掛，不如 OpenClaw 原生 browser control [來源:博客园 blfbuaa 4000行,2026]
  3. 單進程限制：無法像 OpenClaw Gateway/Node 分散式部署 [來源:Jose Nobile Guide,2026]
  4. Issue 回應壓力：934 open issues、社群規模仍為 OpenClaw 1/5，production SLA 無保證 [來源:GitHub API,2026-06-01]
  5. 安全仍依賴配置：bwrap/whitelist 需使用者正確設定，API key 洩漏風險與 OpenClaw 同類 [來源:Clawdbook Comparison,2026]

不適用情境

  • 需要企業級合規審計、托管 SaaS、ClawHub 現成技能的一站式方案 → 選 OpenClaw Launch 或商業平台
  • 需要原生瀏覽器自動化、視覺工作區、macOS/iOS 深度整合 → OpenClaw
  • 嵌入式 <10MB RAM（RISC-V 單板）→ 考慮 PicoClaw/ZeroClaw [來源:searchmytool Comparison,2026]

---

八、一家之言

NanoBot 的 4.3 萬 Star 不僅是「OpenClaw 平替」 hype，更反映 2026 年 Agent 工程師對可審計最小核心的剛性需求。對 daily-digest-prompt 這類已有完整 Harness（Phase 0→2、llm-gate、MCP 工具鏈）的專案，NanoBot 的價值不在「替換現有管線」，而在提供可讀的 Agent Loop 參考實作——尤其是 Dream 記憶、自然語言 Cron、MCP-first 工具註冊三個 pattern，可直接啟發 context/continuity/ 與 autonomous_runtime_policy.py 的下一版設計。若要做 production HITL Bot，建議先用 NanoBot 原型驗證，而非 fork OpenClaw 430K 行程式碼。
