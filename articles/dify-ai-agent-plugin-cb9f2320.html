二、架構剖析：三層微服務 + Plugin 解耦

2.1 核心服務架構

Dify 採用三層微服務架構，將關注點明確分離：

  • Frontend (Next.js)：Visual Canvas、Prompt IDE、RAG Config、LLMOps
  • API Service (Flask + Python)：Workflow Engine、Agent Framework、Model Router、RAG Pipeline、Auth/RBAC
  • Celery Worker (Python)：文件索引、工作流執行
  • Plugin Daemon (Go)：Model calls、Tool exec、Extensions
  • Data Layer：PostgreSQL、Redis、Vector DB

關鍵設計決策：

  1. Flask + Celery 分離：API 處理同步請求，Celery 處理耗時的文件索引、向量化和工作流執行。兩者共享程式碼庫但獨立部署，透過自訂 FlaskTask 類別注入 Flask 應用上下文。
  1. Redis 三重角色：同時作為快取層、Celery 訊息佇列、以及 API 與 Worker 之間的 Pub/Sub 通道。Workflow 執行事件透過 Redis Pub/Sub 從 Celery Worker 串流回 API 進程。
  1. 向量資料庫可插拔：支援 Weaviate、Qdrant、pgvector 等多種後端，透過統一的 Retriever 介面抽象化差異。

2.2 Plugin 生態系統架構

Dify v1.0 引入的 Plugin 系統是其最具前瞻性的架構決策之一。

五大 Plugin 類型：

  • Models：AI 模型接入（OpenAI o1、Gemini 2.0、Claude）
  • Tools：外部工具整合（Perplexity、Discord、Slack、Firecrawler）
  • Agent Strategies：推理策略（ReAct、Function Calling）
  • Extensions：平台整合（Slack 通知、Webhook）
  • Bundles：多個 Plugin 打包分發

安全模型：每個 Plugin 在 Go daemon 中獨立隔離運行，與主 Python API 進程完全分離。嚴格的權限宣告機制，Marketplace 上架前經過嚴格程式碼審查，支援離線安裝。

2.3 工作流引擎

節點類型：LLM Node、Knowledge Retrieval、Agent Node（自主推理）、Code Execution、HTTP Request、Conditional Branch、Loop/Iteration、Variable Aggregation、Template Transform。

執行模型：串流執行走 Celery Worker（workflow_based_app_execution 佇列），非串流在 API 進程內完成，暫停/恢復透過 Celery，節點間事件透過 Redis Pub/Sub。

效能基準：每 Pod（1 CPU / 2 GB RAM）約 10 QPS；8 核 / 16 GB RAM x 2 Pods 可達 11 QPS（無模型）、6 QPS（含模型）。已知瓶頸：每 Workflow 節點獨立查詢資料庫。

三、RAG 管線設計

3.1 文件處理管線

文件上傳 → 格式解析（PDF/PPT/DOCX/HTML）→ 文本分塊 → 向量嵌入 → 索引建立（Vector DB + PostgreSQL FTS）

3.2 檢索策略

| 策略 | 說明 | 適用場景 |

|------|------|----------|

| 語義搜尋 | 向量相似度匹配 | 概念性問題 |

| 關鍵字搜尋 | 全文檢索（BM25） | 精確詞彙 |

| 混合搜尋 | 語義+關鍵字加權融合 | 通用場景（推薦） |

| Agentic RAG | Agent 反覆分析意圖、改寫查詢、評估證據 | 複雜多步推理 |

3.3 Agentic RAG 的創新

傳統 RAG 是「一次檢索→生成」的線性流程。Dify 的 Agent Node 引入 Agentic RAG：

  1. 意圖分析：Agent 先理解使用者真正的問題
  2. 工具選擇：從多個知識庫和工具中選擇最適合的來源
  3. 查詢改寫：根據上下文自動最佳化檢索查詢
  4. 證據評估：檢索結果不足時自動重試或切換策略
  5. 回退機制：所有來源失敗時的降級處理

五、可借鑑技術項目（七大借鑑）

借鑑 1：Plugin 解耦架構 → Skill 系統升級

Plugin 在獨立 Go daemon 中運行，與核心完全解耦。本專案 26 個 Skill 可借鑑此思路，將描述層（SKILL.md）與執行層進一步分離，高頻 Skill 的核心邏輯封裝為可獨立測試的 PowerShell 模組。

借鑑 2：Agentic RAG 反覆檢索 → 知識庫查詢品質提升

本專案可在 research-task.md 中加入檢索品質評估步驟：若 hybrid search 結果 < 3 筆或最高分 < 0.5，自動改用不同關鍵字重試。

借鑑 3：Celery Worker + Redis Pub/Sub → 工作流可觀測性

可引入 Phase 心跳檔（state/phase-heartbeat.json），各 Agent 定期更新狀態，取代被動的 Wait-Job -Timeout。

借鑑 4：視覺化 Workflow Canvas → 配置檔可視化

為系統審查產出 Mermaid 流程圖，展示 Phase 之間的資料流和依賴關係。

借鑑 5：Marketplace 審查機制 → Skill 品質治理

定義標準化品質基線：完整性（frontmatter 100%）、安全性（0 硬編碼金鑰）、可測試性（≥1 test）、相容性（SKILL_INDEX 同步 100%）。

借鑑 6：LLMOps 監控 → 結構化日誌視覺化

在 check-health.ps1 輸出中加入 API 呼叫/成功率的 7 天趨勢 ASCII 圖表。

借鑑 7：Trigger 系統 → 事件驅動自動任務

在 on_stop_alert.py 中擴展：偵測連續 API 錯誤時寫入 state/event-trigger.json，下次 Agent 優先處理。

六、風險與限制分析

| 面向 | 風險/限制 | 影響 |

|------|----------|------|

| 效能瓶頸 | 每節點獨立 DB 查詢 | 不適合低延遲場景 |

| 授權疑慮 | Apache 2.0 附加條款 | 商業使用需注意 |

| Plugin 安全 | Go daemon 非容器級隔離 | 安全邊界有限 |

| 資源消耗 | 推薦 8 核 / 16 GB | 個人門檻較高 |

---

七、結論

Dify 代表 2026 年開源 AI 平台的演進方向：從框架走向平台，從程式碼走向視覺化，從單體走向可插拔生態系統。131K+ Stars 驗證了市場需求。

對 Daily-Digest-Prompt 系統的核心啟示：

  1. 解耦是進化的前提（Plugin/Skill 分離）
  2. 可觀測性是品質的基礎（LLMOps → 主動分析）
  3. 漸進式自動化優於一步到位（Trigger → 事件驅動）
