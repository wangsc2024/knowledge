對話腳本

曉晨（解說）：今天我們要聊的主題是 AI Agent 斷路器要義，探討如何提升系統韌性。

云哲（提問）：聽起來很重要！傳統的斷路器不是只有三種狀態嗎？

曉晨（解說）：是的，過去的三態斷路器只有 CLOSED、OPEN、HALF‑OPEN，但在 AI Agent 中還需要偵測語義失敗，於是引入了第四種 DEGRADED 狀態。

云哲（提問）：什麼是語義失敗？為什麼會發生？

曉晨（解說）：語義失敗是指模型回應在 HTTP 200 下仍然錯誤，例如幻覺或上下文盲點。研究列出五類：Context Blindness、Rogue Actions、Silent Degradation、Memory Corruption、Runaway Execution。

云哲（提問）：報告說 88% 的失敗源自基礎設施，具體是哪些問題？

曉晨（解說）：主要缺乏守衛、權限管理與即時監控，導致速率限制或資源耗盡時系統仍繼續呼叫模型，結果是大量錯誤。

云哲（提問）：重複失敗後的重試迴圈會帶來什麼成本？

曉晨（解說）：建議在連續 3 次失敗後開啟斷路器，保持 30 秒，同時使用 Redis 共享多副本狀態，避免不可逆操作的 fail‑open。這樣能大幅降低重試帶來的延遲與資源消耗。

云哲（提問）：多 Agent 系統間的協調開銷如何？

曉晨（解說）：每次交接大約 100‑500 毫秒，累計 10 次會增加 1‑5 秒；同時 Token 消耗提升約 3.5 倍，可靠度也從 99.5% 下降到 97%。只有平行化率超過 75% 才算划算。

云哲（提問）：那我們應該從哪些具體行動開始落實？

曉晨（解說）：重點建議：P0：在 tools/agent_guardian.py 的 CircuitBreaker 加入 DEGRADED 狀態；P0：將 embedded_ratio 從 42.9% 提升至 80%；P1：在 config/timeouts.yaml 新增 max_retries_per_task；P1：為 29 個自動任務定義 failure_mode；P2：在 Phase 2 量化多 Agent 任務的平行化率。

云哲（提問）：合規方面有什麼新要求需要配合？

曉晨（解說）：EU AI Act 於 2026 年 8 月生效，規定必須文件化 failure mode 並提供可稽核的 FAILURE.md，確保錯誤處理可追蹤。

云哲（提問）：了解了，這些措施聽起來相當具體且可操作。

曉晨（解說）：總結來說，AI Agent 需要四態斷路器、加強基礎設施守衛、控制重試迴圈、量化多 Agent 成本，同時符合 EU AI Act 的合規要求，才能在生產環境顯著提升成功率。
