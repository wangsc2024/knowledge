httpx 傳輸層重試與連線池：以長生命週期 AsyncClient 取代腳本 curl 的生產模式（2026）

任務脈絡：接續 context/continuity/auto-task-tech_research.json 建議的「RetryTransport + 統一 HTTP 工廠」；對齊本日 todoist-history 中 kb_system_optimize（Test-NetConnection preflight）、log_audit／system_insight 對外部 HTTP 與 KB 管線可靠度需求。

技術概述

本主題處理的是 Python 非同步管線如何把「可重試的網路失敗」與「可預期的連線資源上限」拆開建模。httpx 同時提供高階 AsyncClient API 與底層 HTTPTransport／AsyncHTTPTransport；官方明確區分「連線層重試」與「讀到 HTTP 回應後的語意重試」責任邊界 [來源: Transports - HTTPX, 2026 文件抓取日]。在 daily-digest-prompt 這類 PowerShell 7 先 orchestrate、uv run python 再呼叫本機 KB（localhost:3000）與 ntfy 的架構中，常見反模式是每支工具各自 subprocess/curl：每次冷啟動 TLS、難以集中設定逾時與重試、也不利於把 429/503 與 ConnectTimeout 分類處理。改為單例或長生命週期 AsyncClient + 明確 httpx.Limits，可把「連線池內排隊」變成可觀測的 PoolTimeout，而不是隨機的 shell 失敗碼。若還需要對 HTTP 狀態碼 做退避，官方建議補上泛用重試工具（如 tenacity）或第三方 httpx-retries 的 RetryTransport [來源: Transports - HTTPX, 2026；API - HTTPX Retries, 2026]。

核心概念與原理

1) 兩種「重試」不可混用

HTTPTransport(retries=N)（非同步則 AsyncHTTPTransport(retries=N)）只在 httpx.ConnectError 或 httpx.ConnectTimeout 時重送請求；一旦已經拿到回應（含 503），就不屬於這條路徑 [來源: Transports - HTTPX, 2026]。Stack Overflow 上常見的踩雷是誤用同步 HTTPTransport 搭配 AsyncClient（缺少 async context），以及以為 retries=3 會自動重試 POST 的 5xx——實際上必須改用 AsyncHTTPTransport 並理解其僅涵蓋連線層 [來源: Stack Overflow #77694820, 2023–2025]。若要預設重試 429/502/503/504 並尊重 Retry-After，httpx-retries 的 Retry 預設 status_forcelist 與 respect_retry_after_header=True 即對齊此缺口 [來源: API - HTTPX Retries, 2026]。

2) 連線池參數是「反壓力閥」不是越大越好

httpx.Limits 三個核心欄位：max_keepalive_connections（預設 20）、max_connections（預設 100）、keepalive_expiry（預設 5 秒）——來自官方 Resource Limits 頁 [來源: Resource Limits - HTTPX, 2026]。StatusCodeFYI 的連線池指南補充工程直覺：冷啟動 TLS 可能佔 約 80–220ms（DNS+TCP+TLS 1.3），因此 pool 的價值在「重用」；並建議 idle timeout 設得比上游 keep-alive 更短（範例：ALB 60s → client 55s），避免「伺服器已關閉閒置連線、客戶端池仍握著半死 socket」[來源: Connection Pooling Guide, StatusCodeFYI, 2026]。中文工程筆記亦歸納：PoolTimeout 多半來自 max_connections 上限或連線未釋放（未消耗 response body），解法通常是調整 Limits 或修正 async with 生命週期 [來源: 突破Python HTTP瓶颈：HTTPX连接池与资源管理实战指南, 2026]。

3) 多主機場景：維持單一 AsyncClient 是維護者建議

encode/httpx Discussion #3671 中，維護者對「執行期才知道的 ~10 個 host、每個 host 大量請求」場景的回答是：用單一 client；若撞到連線上限會以 timeout 形式顯性化 [來源: GitHub Discussion #3671, 2025]。這與「每個 host 一支 curl 子行程」相反：後者把池化責任丟給 OS 行程與檔案描述符上限，難以在 Python 層做一致的 limits/event_hooks 監測。

4) 0.28 版遷移對「工具 CLI + Agent」的實際影響

httpx 0.28.0（2024-11-28）收斂 SSL 參數（verify 字串路徑、cert 等 deprecated/removed）、移除 proxies／app 等舊參數，並修正 params={} 與既有 query string 的合併語意 [來源: encode/httpx Release 0.28.0, 2024]。對本 repo 這種大量 tools/*.py CLI 的專案，代表：升級 httpx 必須跑完整測試矩陣（uv run pytest），而不是只依賴「import 成功」。

5) `RetryTransport` 預設與 idempotent 方法

httpx-retries 預設僅對 HEAD/GET/PUT/DELETE/OPTIONS/TRACE 啟用重試迴圈；若要把 POST 納入，必須顯式擴充 allowed_methods，否則仍是一次送達 [來源: API - HTTPX Retries, 2026]。這與 REST 安全語意一致：避免對非冪等寫入自動重放。對 KB POST /api/import 這類寫入，應優先依賴 externalKey／merge 策略（見 skills/knowledge-query/SKILL.md）搭配「應用層只重試可判定為安全」的條件，而不是盲目把 POST 加進 retry list。

最佳實踐

  1. 長生命週期 client：在 async CLI 入口建立一次 AsyncClient(limits=..., timeout=httpx.Timeout(...))，用 async with 包住整段 batch，而不是「每個 API 函式內 async with AsyncClient()」——後者等於放棄連線複用（延遲與檔案描述符成本與官方多主機建議相衝）[來源: GitHub Discussion #3671, 2025；Resource Limits, 2026]。
  2. 分層重試：底層 AsyncHTTPTransport(retries=1|2) 處理 flaky network；上層用 RetryTransport 或 tenacity 處理 429/503 + Retry-After 與業務可重試條件 [來源: Transports - HTTPX, 2026；API - HTTPX Retries, 2026]。
  3. keepalive_expiry 對齊上游：內網 KB、本機 ntfy 若走反向代理，請對照代理 idle timeout；參考「client idle < server idle」原則 [來源: StatusCodeFYI Connection Pooling Guide, 2026]。
  4. 逾時四象限：對 KB POST /api/import 這類「寫入＋JSON 序列化」請求，將 read 設得比 connect 寬鬆，避免把慢速 payload 誤判成連線問題；pool 獨立設定可在高併發時更快暴露 PoolTimeout 而非掛死 [來源: GitCode 博客 HTTPX 超时控制表, 2026]。
  5. 觀測：啟用 httpx logging 或包一層 BaseTransport 以記錄每次請求耗時與 status（官方 Transport 包裝範例見 Transports 文件）[來源: Transports - HTTPX, 2026]。

陷阱與反例

  • 誤以為 HTTPTransport(retries) 會重試 5xx：官方與社群答案一致——拿到 response 後不會因 503 自動重試；需要 tenacity 或 httpx-retries [來源: Transports - HTTPX, 2026；Stack Overflow #77694820, 2025 回答]。
  • 同步 Transport 配 AsyncClient：SO 題主最初組合會炸在非 async context；正解是 AsyncHTTPTransport [來源: Stack Overflow #77694820, 2023 已採納答案]。
  • 高併發 + 大 body 的 ReadError／ReadTimeout：GitHub discussions 有多則非決定性案例（大 POST、並發下 ReadTimeout 升級後出現）；此類屬 待驗證⚠️ 的環境相依問題，落地時應以「固定 workload 基準＋semaphore 限流」驗證，不宜直接把 httpx 換成 aiohttp 僅憑單一第三方 benchmark 宣稱（社群基準專案存在，但測試條件與本 repo 差異大）[來源: encode/httpx Discussions #2932/#3067 等索引, 2024–2025；gfx/benchmark-python-httpx-vs-aiohttp 專案存在性]。

程式碼範例


# 節錄自官方文件與 httpx-retries 文件，組合示意（非本 repo 現況程式碼）
import httpx
from httpx_retries import RetryTransport, Retry

retry = Retry(total=5, backoff_factor=0.5, respect_retry_after_header=True)
transport = RetryTransport(
    transport=httpx.AsyncHTTPTransport(retries=1),
    retry=retry,
)
timeout = httpx.Timeout(10.0, connect=5.0, pool=5.0)
limits = httpx.Limits(max_connections=50, max_keepalive_connections=10, keepalive_expiry=55.0)

async def main():
    async with httpx.AsyncClient(transport=transport, timeout=timeout, limits=limits) as client:
        r = await client.post("http://localhost:3000/api/search/hybrid", json={"query": "health", "topK": 3})
        r.raise_for_status()
        return r.json()
  • 來源合成：AsyncHTTPTransport(retries=...) 語意見 [Transports - HTTPX, 2026]；RetryTransport/Retry 見 [API - HTTPX Retries, 2026]。
