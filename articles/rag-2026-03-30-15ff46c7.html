概述

針對 daily-digest-prompt 專案的 RAG（Retrieval-Augmented Generation）管線進行深度分析，結合 2026 年最新 RAG 優化實踐，提出五大優化方向：語義查詢快取層、查詢路由智慧化、Chunk 策略升級、輕量 Reranking 引入、查詢改寫機制。本次聚焦於「檢索品質提升」而非安全性或程式碼品質（前兩次已涵蓋）。

---

一、專案 RAG 管線現況分析

1.1 架構概況

本專案採用本機知識庫（localhost:3000）+ Qdrant 向量引擎 + HTTP 多層快取降級的混合 RAG 架構：

  • 知識庫服務：Node.js + Qdrant（向量索引）+ SQLite FTS5（全文索引）
  • 搜尋 API：hybrid（RRF 融合）、semantic（純向量）、keyword（BM25）、retrieve（格式化上下文）
  • 快取策略：API-level TTL + SWR 寬限期（config/cache-policy.yaml）
  • 去重防護：三層機制（registry + KB 搜尋 + 冷卻期）

1.2 關鍵配置現值

| 配置項 | 目前值 | 來源 |

|--------|--------|------|

| Knowledge TTL | 480 分鐘（8h） | cache-policy.yaml |

| SWR Grace Period | 120 分鐘 | cache-policy.yaml |

| 預設 topK | 15 | dedup-policy.yaml |

| 快取命中率 | 25.5%（目標 40%+） | 2026-03-29 調整後 |

| 混合搜尋權重 | 系統預設（α 未外部化） | KB 服務內建 |

1.3 使用頻率分析

29 個自動任務中至少 8+ 個任務觸發 KB hybrid search（每日 47-57 次執行，估計 15-20 次 KB 查詢）。搜尋場景分布：

  • 去重查詢（80%）：score > 0.85 判定重複，0.4-0.85 深度分析
  • 知識策略分析（15%）：kb-research-strategist 五階段評估
  • RAG 回答（5%）：POST /api/search/retrieve 取 formattedContext

二、五大優化建議（基於 2026 最新實踐）

優化 1：語義查詢快取層（P0 — 預計降低 25-40% KB 查詢延遲）

問題：目前快取僅在 API response level（cache/knowledge.json，TTL=480min），相同語義但不同措辭的查詢無法命中快取。

建議方案：引入 L1 語義查詢快取，使用 cosine similarity > 0.98 判定相同查詢。

實作路徑：

  1. 在 skills/api-cache/SKILL.md 新增語義快取層說明
  2. KB 服務端（localhost:3000）新增 /api/cache/semantic 端點，或在 Agent 端用 Python 實現 embedding 快取
  3. 快取結構：{query_embedding_hash: {results, cached_at, hit_count}}
  4. TTL 建議：60 分鐘（比 API 快取短，因查詢語義可能演變）

預期效果：

  • 去重查詢場景：語義相近的研究主題查詢可複用結果，減少 50% 重複查詢
  • RAGFlow v3.2 實測 embedding 快取可降低 25% 延遲 [來源：dasroot.net, 2026]
  • Redis v7.0 多層快取從 2.5s 降至 400ms [來源：dasroot.net, 2026]

優化 2：查詢路由智慧化（P1 — 提升檢索精準度 15-20%）

問題：目前模板多數硬編碼使用 hybrid search，未根據查詢意圖動態選擇最佳搜尋方式。

建議方案：在 config/ 新增 kb-query-router.yaml，定義查詢意圖→搜尋方式映射。

路由規則：

| 查詢意圖 | 推薦搜尋方式 | 理由 |

|----------|-------------|------|

| 精確名稱/ID/代碼 | keyword（BM25） | 向量搜尋會稀釋精確 token |

| 概念模糊/探索性 | semantic | 捕捉同義詞與語義近似 |

| 通用去重/一般查詢 | hybrid（預設） | 兼顧精確與語義 |

| 事實查詢 | retrieve + 快取 | 不需完整 pipeline |

實作路徑：

  1. 新建 config/kb-query-router.yaml（路由規則外部化）
  2. 在去重查詢前加入輕量意圖分類（可用 Groq 快速推理或規則式）
  3. LangChain v0.26 的意圖分類器準確率 92% [來源：dasroot.net, 2026]

優化 3：Chunk 策略升級（P1 — 匯入品質提升）

問題：目前 KB 匯入使用 contentText 整段寫入，未做語義分段。長文筆記的向量化效果可能因內容混雜而降低。

建議方案：匯入時採用 Semantic Chunking + Metadata Enrichment。

關鍵技術：

  • Semantic Chunking：依語義相似度分段（embedding 聚類），比 fixed-size 提升 70% 準確率 [來源：Weaviate, 2026]
  • Adaptive Chunking：依內容結構動態調整大小，臨床決策場景達 87% 準確率 vs baseline 50% [來源：PMC, 2026]
  • Contextual Retrieval（Anthropic 提出）：每個 chunk 前加入文件級上下文前綴
  • Metadata Enrichment：每個 chunk 附加來源、標籤、日期等 metadata

實作路徑：

  1. 在 KB 服務端擴充 /api/import 支援 chunkStrategy 參數
  2. 或在 Agent 端用 Python 預處理：tools/kb_chunk_preprocessor.py
  3. 長文（>2000 字）自動啟用 semantic chunking，短文維持整段

優化 4：輕量 Reranking 引入（P2 — MRR 提升 +9.3pp）

問題：目前 hybrid search 回傳結果的排序僅依靠 RRF 融合分數，無二階段精排。

建議方案：在 KB 查詢後加入輕量 cross-encoder reranking。

候選方案：

| Reranker | 延遲 | 精度 | 部署方式 |

|----------|------|------|----------|

| FlashRank | 極低 | 中高 | 本地 Python |

| bge-reranker-v2 | 中等 | 高 | 本地 Python |

| RAGatouille v2.1 | 15ms/query | MAP 93.2% | 本地 Python |

預期效果：

  • 混合搜尋 + Reranking MRR 從 56.72% 提升至 66.43%（+9.3pp）[來源：Medium, 2026]
  • 去重判定更精準：減少 false positive（非重複被判為重複）

實作路徑：

  1. pip install flashrank 或 bge-reranker-v2
  2. 在 tools/kb_reranker.py 實現 rerank wrapper
  3. 修改 knowledge-query SKILL.md，查詢後加 rerank 步驟

優化 5：查詢改寫機制（P2 — 提升召回率 15-25%）

問題：去重查詢的搜尋詞由 LLM 直接生成，可能與 KB 中實際用語不匹配。

建議方案：對複雜查詢啟用 Query Expansion（多查詢擴展）。

技術選項：

  • Multi-query Expansion：為同一問題生成 2-3 個變體查詢，合併結果（最簡單有效）
  • HyDE：先生成假想答案再搜尋（概念型查詢效果佳，但增加一次 LLM 呼叫）
  • Step-back Prompting：退一步生成更廣泛問題（需要背景知識的深度問題）

實作路徑：

  1. 在 templates/shared/kb-depth-check.md 的 Phase A 加入 multi-query 步驟
  2. 對去重查詢生成 2 個同義變體，合併 topK 結果後去重
  3. 簡單查詢跳過改寫（用意圖分類器判斷），避免增加不必要延遲
