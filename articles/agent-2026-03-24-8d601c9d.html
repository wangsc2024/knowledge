Agent 執行品質與成功率提升策略 — 從失敗模式分類到品質閘門的系統性改善路徑

研究日期：2026-03-24

研究類型：Optimization（系統改善與最佳實踐）

研究深度：Thorough（17 來源，10 A 級，5 B 級，1 C 級）

信心水平：High（5 項交叉驗證共識點）

---

執行摘要（約 200 字）

本研究針對 daily-digest-prompt 專案的執行成功率偏低問題（目前 87.5%，目標 95%），彙整業界失敗模式分類、品質閘門設計與自動恢復策略三大面向的最佳實踐。核心發現包括：(1) Multi-Agent 系統失敗可系統化分為 14 種模式與 3 大類（系統設計/Agent 失調/任務驗證），(2) 品質閘門應採 Pre-Flight/In-Flight/Post-Flight 三階段漸進式設計，(3) Retry + Circuit Breaker 必須協同而非單選，(4) 自適應閾值（AI/ML 驅動）可取代固定閾值降低誤報，(5) 所有策略的基礎是可觀測性（Tracing/Logging/Metrics）。本研究提供可直接應用於專案的失敗分類框架、品質閘門設計清單與恢復策略決策樹，預期可將成功率提升至 95% 並建立回歸偵測機制。

關鍵洞見（含證據要點）

洞見 1：Multi-Agent 失敗模式三層分類法

來源：arxiv "Why Do Multi-Agent LLM Systems Fail?" (A 級)

內容：

Multi-Agent 系統的失敗可系統化分為 14 個失敗模式，歸類為 3 大類：

  1. 系統設計問題（System Design Failures）

- Ineffective orchestration（無效編排）：Agent 間協調機制不良

- Insufficient task decomposition（任務拆解不足）：任務未妥善拆分為可管理組件

- Inadequate tool/API integration（工具整合不當）：外部資源使用不當

  1. Agent 間失調（Inter-Agent Misalignment）

- Communication breakdowns（溝通中斷）：Agent 間資訊交換失效

- Role confusion（角色混淆）：職責不明或衝突

- Conflicting objectives（目標衝突）：Agent 目標互相對立

- Context loss（上下文丟失）：關鍵資訊未跨 Agent 傳遞

  1. 任務驗證失敗（Task Verification Failures）

- Missing output validation（輸出驗證缺失）：缺乏驗證機制

- Inadequate error handling（錯誤處理不當）：恢復機制不足

- Poor monitoring and logging（監控日誌不良）：系統行為可見性低

- Incomplete task verification（任務驗證不完整）：未確認任務完成

證據要點：

  • 研究分析了 1,642 個 MAS 執行追蹤（execution traces）
  • 建立 MAST 數據集（Multi-Agent System Taxonomy）記錄真實失敗場景
  • 發現 Multi-Agent 系統需要健全的監督機制、明確角色定義與系統性驗證方法

對專案的應用：

  • daily-digest-prompt 的 3 階段管線（Query→Execute→Assemble）可映射到此分類法
  • 高失敗時段（7 點、13 點）可能源於 orchestration 問題（資源競爭）或 context loss（Phase 間交接）
  • 建議建立失敗分類標籤系統，每次失敗自動標記為 14 種模式之一

洞見 2：品質閘門三階段設計（Pre-Flight / In-Flight / Post-Flight）

來源：InfoQ "Pipeline Quality Gates" (A 級) + Sonar "Quality Gate" (A 級)

共識點：

品質閘門應分為三階段漸進式設計，而非單一檢查點：

Pre-Flight Checks（執行前檢查）

  1. Linting & Code Standards（程式碼規範）：在 checkout 時驗證，違規即拒絕
  2. Unit Testing Requirements（單元測試）：要求 100% 測試完成 + 90% 程式碼覆蓋率（InfoQ 具體閾值）
  3. Environment Readiness（環境健康）：資料庫連線檢查、版本相容性、伺服器資源利用率

In-Flight Checks（執行中檢查）

  1. Security Integration（安全掃描）：檢查漏洞、過期套件、程式碼簽章
  2. Dynamic Code Analysis（動態分析）：執行時行為驗證
  3. Performance Validation（效能驗證）：回應時間、吞吐量、資源使用

Post-Flight Checks（執行後檢查）

  1. Deployment Validation（部署驗證）：確認部署環境符合預期
  2. Health Checks（健康檢查）：定期重新評估所有閘門直到同時成功
  3. Rollback Triggers（回滾觸發）：失敗時自動回滾

證據要點：

  • Sonar 強調「Define clear, measurable pass/fail criteria aligned with organizational standards」
  • InfoQ 提出「Periodically re-evaluate all gates until they succeed simultaneously to prevent flaky deployments」
  • Sonar 建議「Start with essential metrics and expand gradually to avoid overwhelming teams」

對專案的應用：

  • 當前專案缺乏 Pre-Flight Checks（如外部 API 健康探測）
  • 建議在 run-todoist-agent-team.ps1 的 Phase 0 加入環境健康檢查
  • Post-Flight Checks 可用於偵測 Phase 間交接格式漂移

洞見 3：Retry + Circuit Breaker 協同機制（Integration Pattern）

來源：Temporal "Error handling in distributed systems" (A 級) + Microsoft Azure "Circuit Breaker Pattern" (A 級)

關鍵發現：

Retry Pattern 與 Circuit Breaker Pattern 必須配合使用，而非二選一：

Retry Pattern（處理瞬態問題）

  • 適用於：網路閃斷、暫時性資源不足、API 限流
  • 策略：指數退避（Exponential Backoff）+ Jitter（隨機延遲）
  • 關鍵警告（Temporal 獨家）：「Never, ever retry non-idempotent operations. Charging a credit card twice because of a timeout isn't a 'transient error,' it's a lawsuit waiting to happen.」

Circuit Breaker Pattern（處理持久問題）

  • 適用於：服務崩潰、持續性錯誤、資源耗盡
  • 三狀態機制：

1. Closed（正常）：請求正常通過，失敗計數器累積

2. Open（快速失敗）：立即拒絕請求，啟動 timeout 計時器

3. Half-Open（謹慎測試）：允許限量請求測試恢復，成功則 → Closed，失敗則 → Open

協同配置範例（來源：Temporal）：

  • Retry：最多 4 次，指數退避 + jitter
  • Circuit Breaker：連續 3 次失敗則 trip，Open 狀態持續 1 分鐘

證據要點：

  • Temporal：「Retries handle transient blips, while circuit breakers handle persistent problems」
  • Azure：「Half-Open state helps prevent a recovering service from suddenly being flooded with requests」
  • Azure：「Failure counter for Closed state is time based. It automatically resets at periodic intervals.」

對專案的應用：

  • 當前專案缺乏 Circuit Breaker 機制（API 呼叫失敗後持續重試）
  • 建議在 api-cache Skill 中加入 Circuit Breaker 狀態追蹤
  • self-heal Skill 的 Tier 1 重試需確認冪等性（KB import、ntfy 通知皆為冪等操作）

洞見 4：自適應閾值與 AI 驅動監控（Adaptive Thresholds）

來源：Microsoft Azure "Circuit Breaker Pattern" (A 級)

獨特點：

傳統 Circuit Breaker 依賴預配置閾值（如「5 次失敗則 trip」），導致次優行為：

  • 流量尖峰時過早觸發（誤報）
  • 真正故障時延遲觸發（漏報）

自適應方法（AI/ML 驅動）：

根據即時流量模式、異常偵測與歷史失敗率動態調整閾值。

證據要點：

"Traditionally, circuit breakers relied on preconfigured thresholds, such as failure count and time-out duration. This approach resulted in a deterministic but sometimes suboptimal behavior. Adaptive techniques that use AI and machine learning can dynamically adjust thresholds based on real-time traffic patterns, anomalies, and historical failure rates. This approach improves resiliency and efficiency."

對專案的應用：

  • system-insight.json 已追蹤 7 天失敗率趨勢，可用於訓練自適應模型
  • failure-hour-analyzer Skill 可擴展為自適應閾值調整器
  • 初期可用簡單規則（如「高峰時段閾值 × 1.5」）逼近自適應效果

洞見 5：觀測性是所有策略的基礎（Observability Foundation）

來源：所有 5 個來源的共識點

證據：

  • Sonar：「Provide developers with clear feedback on gate failures」
  • Temporal：「Track execution through correlation IDs and execution histories」
  • Azure：「Provide clear observability into both failed and successful requests」
  • InfoQ：「Use tools that provide actionable insights, not just pass/fail signals」
  • arxiv：「Poor monitoring and logging」被列為失敗模式之一

共識要點：

  1. Correlation IDs：每次執行必須有唯一 trace_id，可跨 Phase/Agent 追蹤
  2. Execution Histories：記錄每個決策點（為何 skip、為何 retry、為何 fallback）
  3. Actionable Insights：不只記錄「失敗」，要記錄「為何失敗」與「如何修復」
  4. Real-time Visibility：運維團隊需即時儀表板，非事後日誌分析

對專案的應用：

  • 當前專案已有 post_tool_logger（JSONL 日誌）但缺乏即時可視化
  • execution-journal Skill（ADR-027）可擴展為 OpenTelemetry 整合
  • system-insight 可新增「失敗根因排名」區塊（Top 5 failure reasons）

建議行動（對應 daily-digest-prompt 專案）

短期行動（1-2 週內實施）

  1. 建立失敗分類標籤系統（對應洞見 1）

- 在 hooks/post_tool_logger.py 新增 failure_mode 欄位

- 每次失敗自動分類為 14 種模式之一（可用規則式或簡單 LLM 分類）

- system-insight 新增「失敗模式分佈」統計區塊

  1. 實作 Pre-Flight Check（對應洞見 2）

- 在 run-todoist-agent-team.ps1 Phase 0 新增環境健康檢查

- 檢查項：KB 連線（curl health check）、ntfy 可達性、Groq relay 啟動狀態

- 失敗則跳過執行，避免級聯失敗

  1. 為 api-cache Skill 加入 Circuit Breaker（對應洞見 3）

- 追蹤每個端點的連續失敗次數（閾值：3 次）

- Open 狀態持續 5 分鐘（可配置於 config/cache-policy.yaml）

- Half-Open 狀態允許 1 個測試請求

中期行動（1 個月內實施）

  1. 實作自適應閾值原型（對應洞見 4）

- 利用 system-insight 的 7 天數據訓練簡單模型

- 初期可用規則式（如「高峰時段閾值 × 1.5」）

- 進階可用 sklearn 的 IsolationForest 進行異常偵測

  1. 建立執行品質儀表板（對應洞見 5）

- 整合 post_tool_logger 的 JSONL 日誌到即時儀表板（可用 Grafana 或簡單 HTML）

- 展示：成功率趨勢、失敗模式分佈、高失敗時段、Correlation ID 追蹤

  1. 強化 Phase 間交接驗證（對應洞見 2 的 Post-Flight Checks）

- 在 Phase 2→3 交接處加入 Schema 驗證（agent-result-validator Skill）

- 格式漂移自動告警並阻擋 Phase 3 執行

長期行動（2-3 個月內實施）

  1. 實作 Embedded Durability（checkpoint 機制）（對應獨家洞見 1）

- Phase 1/2/3 各自儲存 checkpoint（檔案或記憶體快取）

- 失敗後從 checkpoint 續跑而非整個重跑

  1. 建立品質回歸偵測管線（對應洞見 5）

- 每日自動比對成功率、失敗模式分佈、平均執行時間

- 若有統計顯著性回歸則觸發告警

  1. 多重審批緊急修復機制（對應獨家洞見 2）

- self-heal Skill 的 Tier 3/4 修復需人工確認

- 建立審批 workflow（可用 GitHub Issues + Webhook）
