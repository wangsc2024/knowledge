Harness Engineering 持續效能基準與回歸防護

optimization 階段：從約束設計到量測閉環

本文為 harness-engineering 系列第 optimization 階段，承接 foundation（學科定義與四大組件）和 application（約束演化管線與自癒機制），深入探討如何量化 Harness 效能、偵測約束漂移並自動校準效能門檻。

---

一、Fitness Function 在 Harness Engineering 的角色

1.1 定義與核心概念

Fitness Function 源自演化式架構（Evolutionary Architecture），定義為「對架構特性的客觀完整性評估」。在 Harness Engineering 語境中，Fitness Function 是衡量 Agent 約束系統有效性的自動化指標：

  • 原子型（Atomic）：評估單一面向，如約束覆蓋率、單次任務成功率
  • 整體型（Holistic）：評估複合特性，如安全性 × 效能 × 成本的平衡
  • 觸發型（Triggered）：在特定事件（部署、PR 合併）時執行
  • 持續型（Continual）：透過生產監控持續驗證，不需排程

1.2 三步驟實作模式

| 步驟 | 說明 | 實例 |

|------|------|------|

| Selection | 識別與架構目標對齊的非功能需求 | Agent 任務成功率、約束違規率 |

| Definition | 建立基準線、目標值與失敗閾值 | 成功率 baseline=85%, target=92%, fail<80% |

| Frequency | 決定量測頻率（依自動化程度） | CI/CD 每次部署、生產環境每小時 |

1.3 治理實踐

建議從 3 個 Fitness Function 開始，成熟團隊擴展至 5-6 個。每週審查追蹤狀態，失敗時需產出修復計畫恢復合規。

二、Agent 約束漂移（Constraint Drift）偵測

2.1 漂移四大類型

研究顯示 91% 的 ML 系統未主動干預時會經歷效能降級。在 Harness Engineering 中，漂移分為：

| 類型 | 特徵 | 偵測方法 |

|------|------|----------|

| 概念漂移（Concept Drift） | 輸入與目標變數關係改變 | 任務成功率趨勢、錯誤頻率監控 |

| 資料漂移（Data Drift） | 輸入資料分佈改變 | KS 檢驗、PSI 指數、KL 散度 |

| Prompt 漂移 | 指令模板不一致或演變 | 版本控制、部署變量管理 |

| 上下文漂移（Context Drift） | RAG 資料特徵變化 | 知識庫過期偵測、向量嵌入偏移監控 |

2.2 早期預警信號

  • 效能降級：成功率下降、錯誤頻率增加、回應不一致
  • 效率異常：回應時間變長、Token 使用量激增
  • 統計偏差：分佈參數超過閾值（PSI > 0.2 為顯著漂移）

2.3 偵測實作模式


持續監控 → 統計檢定（KS/PSI/KL）→ 閾值判定 → 告警/自動修復

三、效能門檻自動校準策略

3.1 動態閾值 vs 靜態閾值

傳統 Fitness Function 採靜態閾值（pass/fail），但 Agent 系統需要動態調整：

| 策略 | 適用情境 | 優勢 |

|------|---------|------|

| 靜態閾值 | 安全性底線、合規要求 | 簡單明確、不可妥協 |

| 滑動窗口 | 效能指標、成功率 | 容許季節性波動 |

| 百分位基準 | 延遲、Token 用量 | 基於歷史分佈自動調整 |

| 趨勢基準 | 長期品質演化 | 持續改進而非固定標準 |

3.2 校準迴圈


收集基準數據（30 天）
  → 計算統計參數（mean, p95, std）
  → 設定初始閾值（mean + 2σ 為警告、mean + 3σ 為失敗）
  → 每週重新計算滑動窗口
  → 檢測顯著偏移（t-test, p<0.05）
  → 觸發校準或告警

3.3 Harness 特有的 Fitness Function 建議

| 指標 | 量測方式 | 建議閾值 |

|------|---------|----------|

| 約束覆蓋率 | 已定義約束 / 可約束操作 | ≥ 85% |

| 約束違規率 | 違規次數 / 總操作次數 | ≤ 2% |

| 自癒成功率 | 自動修復成功 / 觸發次數 | ≥ 70% |

| 回歸逃逸率 | 生產環境發現的已知問題 | ≤ 1/sprint |

| 漂移偵測延遲 | 從漂移發生到偵測的時間 | ≤ 24h |

| Fitness Function 通過率 | 通過數 / 總數 | ≥ 95% |

四、生產環境持續基準追蹤

4.1 分散式追蹤整合

  • OpenTelemetry 標準化遙測，Span 級延遲分析
  • 完整執行路徑可視化，成本歸因追蹤
  • 機器學習異常偵測演算法（Isolation Forest、DBSCAN）

4.2 30-60-90 天路線圖

| 階段 | 重點 | 產出 |

|------|------|------|

| 30 天 | 建立基準線 + 3 個核心 Fitness Function | 效能 dashboard |

| 60 天 | 自動漂移偵測 + 告警管線 | 告警規則 + 回應 SOP |

| 90 天 | 自動校準 + 自癒整合 | 閉環治理系統 |

4.3 關鍵原則

Epsilla 提出的 Harness Engineering 核心原則：「先達成可偵測性，再授予自治權；先建立約束，再優化吞吐量」（achieve detectability before granting autonomy; establish constraints before optimizing for throughput）。

五、對 Daily-Digest-Prompt 專案的應用映射

本專案已有的 Harness 基礎設施可直接對應 Fitness Function 體系：

| 現有機制 | 可轉化的 Fitness Function |

|---------|------------------------|

| config/benchmark.yaml fitness_functions | 直接擴展漂移偵測指標 |

| check-health.ps1 健康檢查 | 轉化為觸發型 Fitness Function |

| hooks/post_tool_logger.py JSONL 日誌 | 提供持續監控數據源 |

| tools/trace_analyzer.py 根因分析 | 整合漂移偵測告警 |

| state/quality-trend.json 品質趨勢 | 滑動窗口基準計算基礎 |

| state/entropy-trend.json 熵趨勢 | 約束漂移的代理指標 |

建議下一步：在 config/benchmark.yaml 新增 drift_detection 段落，定義 3 個核心漂移指標（成功率、Token 用量、約束違規率）的滑動窗口閾值與校準頻率。
