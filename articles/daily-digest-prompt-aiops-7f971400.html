研究日期：2026-03-03

類型：專案優化 / 深度思維

適用對象：daily-digest-prompt 專案

---

一、概述

本文針對 daily-digest-prompt 專案的可觀測性（Observability）現況進行深度診斷，提出四階段成熟度演進路線圖。專案目前的可觀測性基礎已相當紮實（結構化 JSONL 日誌、自動標籤分類、Session 摘要、ntfy 告警），但在指標時序聚合、跨 Phase 因果關聯、預測性異常偵測、SLO/Error Budget 治理四個維度仍有明顯的成長空間。

可觀測性（Observability）的業界定義是「從系統外部輸出推斷系統內部狀態的能力」（Charity Majors, Honeycomb），由三大支柱組成：Logs（日誌）、Metrics（指標）、Traces（追蹤）。本專案已在 Logs 支柱建立深厚根基，但 Metrics 和 Traces 兩個支柱的成熟度尚處於萌芽階段。

2.2 識別的缺口（Gap Analysis）

| 缺口 | 嚴重度 | 說明 |

|------|--------|------|

| G1：指標時序聚合缺失 | 高 | 健康評分是「快照式」（每次查詢即時計算），無歷史時序資料，無法做趨勢比較或回歸偵測 |

| G2：跨 Phase 因果關聯薄弱 | 高 | trace_id 已傳播，但無 span 概念（Phase 起止時間、父子關係），無法回答「Phase 1 哪個 Agent 拖慢了整體耗時」 |

| G3：告警缺乏上下文 | 中 | ntfy 告警僅含統計數字，缺少「為什麼異常」的根因推斷，需人工查日誌 |

| G4：無 SLO/Error Budget 定義 | 中 | benchmark.yaml 定義了目標門檻但非真正的 SLO；無 Error Budget 消耗追蹤，無法區分「偶發波動」與「系統性劣化」 |

| G5：被動式監控 | 中 | 所有異常偵測都是「事後回顧」（Session 結束時或手動查詢），無法在執行中即時預警 |

| G6：Token 經濟追蹤碎片化 | 低 | token-usage.json 與 behavior-patterns.json 獨立追蹤，兩者未交叉分析 |

| G7：FSM 狀態無歷史 | 低 | run-fsm.json 只保留最近的 run，歷史 Phase 耗時無法回溯比較 |

三、四階段成熟度演進模型

Level 1：結構化日誌（Structured Logging）— 已達成

核心特徵：每個工具呼叫產生結構化記錄，支援事後查詢。post_tool_logger.py 已是業界最佳實踐水準（自動標籤、Session 隔離、磁碟保護、50MB 輪轉）。

Level 2：指標聚合與時序追蹤（Metrics & Time Series）— 建議實施

核心目標：將離散的 JSONL 記錄聚合為可比較的時序指標，實現趨勢分析與回歸偵測。

具體建議：

  1. 新增 state/metrics-daily.json：每日聚合當日指標（total_sessions、success_rate、avg_phase1_duration_s、p95_total_duration_s、cache_hit_rate、error_count、token_estimate 等），並按 agent 類型分組。由 on_stop_alert.py 的最後一個 Session 觸發聚合。
  2. 在 query-logs.ps1 新增 -Mode regression：比較最近 7 天與前 7 天基線，自動偵測顯著偏移（如成功率下降超過 10%、平均耗時增加超過 30%）。
  3. 在 check-health.ps1 加入趨勢箭頭：顯示各指標的 7 天趨勢方向（上升/持平/下降）。

預估工作量：3-4 小時實作 + 1 小時測試

Level 3：分散式追蹤與因果關聯（Distributed Tracing）— 建議實施

核心目標：為每個 Phase 和子 Agent 建立 Span（時間跨度），實現跨 Phase 因果關聯和瓶頸定位。

具體建議：

  1. 擴展 JSONL 日誌格式：在 Phase 起止點記錄 span_start/span_end 事件（含 span_id、parent_span、duration_ms）。
  2. 在 PowerShell 團隊腳本中記錄 Span：每個 Start-Job 前後寫入時間戳，計算各 Phase 和各 Agent 的實際耗時。
  3. 新增 query-logs.ps1 -Mode waterfall：以瀑布圖格式顯示單次執行的 Span 時序，自動標註瓶頸。範例輸出：
    4. 
trace: abc123 (daily-digest, 2026-03-03 08:00)
|-- phase1 [0-65s]
|   |-- fetch-todoist   [0-12s]  OK
|   |-- fetch-news      [0-18s]  OK
|   |-- fetch-hackernews [0-25s]  OK
|   |-- fetch-gmail     [0-30s]  OK
|   |-- fetch-security  [0-65s]  SLOW!
|-- phase2 [65-98s]
    |-- assemble-digest [65-98s]  OK
Total: 98s | Bottleneck: fetch-security (65s, 66% of total)
  4. 在 on_stop_alert.py 中加入 Slow Span 偵測：超過歷史 P95 的 1.5 倍則自動告警。

預估工作量：5-6 小時實作 + 2 小時測試

Level 4：預測性 AIOps 與 SLO 治理（Predictive AIOps）— 長期目標

核心目標：從被動監控演進到主動預測，建立 SLO/Error Budget 量化治理。

具體建議：

  1. 定義 SLO：在 config/slo.yaml 中定義服務水準目標（每日摘要可用性 99%、Phase 1 耗時 P95 <= 90s、ntfy 通知送達率 99.5%）。
  2. Error Budget 追蹤：在 state/error-budget.json 中每日更新。剩餘預算 > 50% 可進行風險較高的優化；20-50% 僅允許保守修改；< 20% 凍結變更，專注穩定性。
  3. 簡易異常預測：利用 14 天歷史資料計算移動平均和標準差，偏離 2 個標準差時觸發預警。
  4. Groq 輔助根因分析：將異常日誌送至 Groq Relay 做快速根因摘要。

預估工作量：8-10 小時實作 + 3 小時測試

五、現有基礎的槓桿點分析

本專案有幾個獨特的槓桿點（Leverage Points），使得可觀測性升級的成本遠低於一般專案：

  1. JSONL 日誌已包含 trace_id 和 phase 欄位（post_tool_logger.py 第 439-440 行），Span 聚合只需在 PS1 腳本中正確設定環境變數，無需修改日誌格式。
  2. 三個團隊模式腳本已傳播 trace_id，只需額外記錄 Phase 起止時間戳即可構建完整的 Span 時間線。
  3. on_stop_alert.py 已讀取 session-summary.jsonl，每日指標聚合可自然嵌入最後一個 Session 處理邏輯。
  4. query-logs.ps1 已有 timeline 模式，升級為 Waterfall 只需加入 Span 層級和對齊格式。
  5. run-fsm.json 已追蹤 Phase 狀態和 updated 時間戳，可作為 Span 耗時計算的備用來源。

六、架構決策建議

ADR-009 提案：可觀測性成熟度演進策略

  • 決策：採用四階段漸進式升級（Level 1→2→3→4），優先實施指標聚合（Level 2），再推進 Span 追蹤（Level 3）
  • 根本原因：目前可觀測性停留在 Level 1，無法回答「系統是否在劣化？」和「哪裡是瓶頸？」
  • 替代方案 A：引入 OpenTelemetry SDK — 過重，本專案非 Web 服務，Agent CLI 環境不適合
  • 替代方案 B：引入 Prometheus + Grafana — 過重，需額外服務，違背「不使用 Docker」原則
  • 方案 C（推薦）：自建輕量級指標聚合 + Span 記錄，零外部依賴，符合文件驅動架構
  • 風險：自建方案缺乏社群維護；緩解：保持格式與 OpenTelemetry 語義相容，未來可低成本遷移
