修正技術細節

1. allowed-tools 設定合理性

| Skill | allowed-tools | 理由 |

|-------|--------------|------|

| writing-masters | Read, Write | 知識型 Skill，僅需讀取參考資料、輸出建議，無需執行指令 |

| writing-plans | Read, Write, Bash | 需執行測試驗證指令（pytest、git commit），必須包含 Bash |

2. cache-ttl: N/A 語義

這兩個 Skill 屬於互動式工具，輸出高度依賴：

  • 使用者的具體寫作/規劃情境
  • 當前專案的程式碼庫狀態
  • 即時的需求釐清對話

因此不適合快取，設為 N/A 符合語義。

3. YAML frontmatter 驗證

修正後用 Python yaml.safe_load() 驗證：

發現的系統性問題

問題 1：全域 Skills 規範滯後

根因：

  • 專案內 Skills（skills/*/SKILL.md）在 2026-02-15 標準化 frontmatter
  • 全域 Cursor Skills（C:\Users\user\.cursor\skills\）未同步更新
  • 導致這兩個全域 Skill 缺少 allowed-tools、cache-ttl 欄位

風險：

  • 全域 Skills 可能被複製到其他專案，傳播不規範格式
  • validate_config.py 若新增 Skill frontmatter 檢查，會檢測出大量違規

建議：

  • 建立全域 Skills 同步機制（定期審查 + 規範對齊）
  • 或將常用全域 Skill 複製到專案內，納入版本控制

問題 2：語言一致性挑戰

發現：

  • writing-plans 的 description 用英文（"Create detailed implementation plans..."）
  • 可能來自外部範本（Cursor 官方範例或社群共享）

影響：

  • 違反專案規範（全程使用正體中文）
  • 觸發條件 "Use when" 中英混雜，降低可讀性

建議：

  • 全域 Skills 匯入時強制本地化審查（description 必須中文化）
  • 在 pre_write_guard.py 新增規則：寫入 skills/*/SKILL.md 時檢查 description 是否含大量英文

問題 3：version 欄位順序不統一

統計：

  • 30 個 Skills 中，28 個採用 name → version → description 順序
  • 這兩個全域 Skill 採用 name → description → version 順序

技術影響：

  • YAML 規範允許任意順序，但統一順序提升可維護性
  • 手動編輯時容易漏填（version 夾在中間不顯眼）

建議：

  • validate_config.py 新增 --check-skill-frontmatter-order 驗證
  • Git pre-commit hook 攔截順序異常
