方案 A：Phase Failure 分級告警（總分 10）

背景

從 RAG優化日報 2026-03-25 識別：Phase Failure 累計 56 次，03-20 單日爆發 21 次，需要自動告警機制。

實施策略

新增 config/notification.yaml 的 phase_failure_alert 段落：

  • warning 閾值：單日 ≥ 5 次 → ntfy 告警（priority: 3）
  • critical 閾值：單日 ≥ 10 次 → ntfy 緊急（priority: 5）+ 暫停非必要排程
  • 冷卻機制：60 分鐘冷卻期，防止告警風暴

技術細節

  • 配置檔案：config/notification.yaml（YAML 格式驗證通過）
  • Schema 更新：config/schemas/notification.schema.json 新增 phase_failure_alert 定義
  • 驗收條件：✅ YAML 語法通過 / ✅ 閾值定義完整 / ✅ 冷卻機制設定 / ✅ JSON Schema 驗證通過

成果

  • 新增配置段落共 10 行（含注釋）
  • 測試通過：notification.yaml 的 JSON Schema 驗證錯誤已消除
  • 無需修改程式碼，配置即可生效（讀取配置的邏輯由未來實施者實作）

方案 B：State 清理自動化排程（總分 9）

背景

State 檔案持續膨脹至 1.4MB：

  • autonomous-recovery-queue.json：148KB（418 筆記錄）
  • context-usage.json：112KB（累積所有 session）
  • loop-state-*.json：24 個檔案（無自動清理）

實施策略

建立自動化清理腳本，每日 00:20 執行（於 kb-backup-all 之後）：

  1. 移除 > 48 小時的 loop-state-*.json
  2. autonomous-recovery-queue.json 保留最近 50 筆
  3. context-usage.json 保留最近 7 天

技術細節

  • 清理腳本：tools/cleanup-state-files.ps1（135 行，含 dry-run 模式）
  • 排程定義：HEARTBEAT.md frontmatter 新增 state-cleanup 排程
  • 執行時間：每日 00:20（cron: 20 0 *）
  • 超時設定：120 秒（2 分鐘）

Dry-Run 測試結果

  • ✅ loop-state-*.json：無過期檔案（< 48h）
  • ✅ autonomous-recovery-queue.json：識別 418 筆記錄，將保留最近 50 筆（刪除 368 筆）
  • ✅ context-usage.json：所有 sessions 均在 7 天內，無需清理

驗收條件

  • ✅ 腳本執行無錯誤
  • ✅ Dry-run 輸出正確識別過期檔案
  • ✅ HEARTBEAT.md 已加入排程定義
  • ✅ YAML frontmatter 格式驗證通過

方案 C：Token 預算 per-task 上限（總分 8，風險等級 2）
