Mem0 — AI Agent 通用記憶層深度研究報告

執行摘要

本文主張：Mem0（mem0ai/mem0）以「LLM 驅動的 ADD-only 記憶管線 + 多信號混合檢索」成為 2026 年 AI Agent 長期記憶基礎設施的預設選項，在互動式場景以 p95 0.20s 級延遲與 LOCOMO 91.6 分（2026 新演算法）領先同類；但對 daily-digest-prompt 這類排程 Agent，應限於「使用者偏好／跨輪決策」子集，不可取代現有 context/ JSON 狀態與 KB 結構化知識 [來源:GitHub mem0ai/mem0,2026][來源:Memory Evaluation Mem0 Docs,2026][來源:Atlan LangMem Guide,2026]。

---

一、專案概述

Mem0（讀作 mem-zero）是專為 AI Agent 與助手設計的通用記憶層（Universal Memory Layer），解決 LLM 無法跨 session 保留使用者偏好、歷史決策與上下文的根本限制 [來源:GitHub mem0ai/mem0,2026]。目標使用者包括：建構個人化聊天機器人、多 Agent 編排系統、企業客服與自主 Agent 的後端工程師；核心價值主張是「用結構化記憶取代每次 prompt 塞滿全歷史」，在準確率提升的同時降低 Token 成本與延遲 [來源:HN Show HN Mem0,2024][來源:Mem0 Platform Overview,2026]。

截至 2026-06-03（GitHub API 實測）：Stars 57,452、Forks 6,565、Open Issues 415、pushed_at 2026-06-02，近 90 天持續活躍；主要語言 Python（80%+），Apache-2.0 授權，並提供託管 Platform 與自架 OSS 雙模式 [來源:GitHub API mem0ai/mem0,2026-06-03]。

二、技術架構

2.1 分層與記憶作用域

Mem0 將記憶分為三個作用域：User Memory（跨所有對話持久）、Session Memory（單次對話）、Agent Memory（特定 Agent 實例） [來源:Zylos Mem0 Research,2026]。這與 daily-digest-prompt 的 context/continuity/*.json（任務級連續記憶）與 state/（排程狀態，Agent 唯讀）在概念上對齊，但 Mem0 面向對話型事實抽取，而非管線 JSON schema [來源:Walking in Code Mem0 源碼解析,2026]。

2.2 2026 新演算法：ADD-only 與三層儲存

2026 年 4 月 Mem0 重設核心演算法（v3），架構變更如下 [來源:GitHub mem0ai/mem0 README,2026][來源:Mem0 OSS v2→v3 Migration,2026][來源:Memory Evaluation Docs,2026]：

寫入管線（5 步，非同步）：

  1. Store New Memories — Agent 回應後非同步進管線
  2. Context Lookup — 查既有記憶避免重複
  3. Distill Memories — 單次 LLM 呼叫，ADD-only 抽取事實（不再 UPDATE/DELETE 覆寫）
  4. Deduplicate + Embed — 雜湊去重後向量化
  5. Entity Linking — 抽取實體並跨記憶連結

三層儲存：

| 儲存層 | 內容 | 用途 |

|--------|------|------|

| Vector DB | 記憶文本、embedding、metadata | 語義檢索主體 |

| Entity Store | 實體 embedding + 連結 memory ID | 實體匹配加分 |

| SQL DB | ADD 事件歷史 + 滾動訊息窗 | 稽核與抽取上下文 |

關鍵設計決策：ADD-only 使舊事實與新事實並存，由檢索時的時間感知排序決定「當前狀態」，避免過早合併造成資訊遺失 [來源:Memory Evaluation Docs,2026]。官方宣稱 LoCoMo 71.4→91.6（+20）、LongMemEval 67.8→94.8（+27）、BEAM(1M) 64.1 [來源:GitHub README,2026]。

2.3 多信號混合檢索

查詢時並行計分後融合 [來源:Memory Evaluation Docs,2026]：

  1. Semantic Search — 向量相似度
  2. Keyword Search — BM25 + 詞形還原
  3. Entity Search — 查詢實體匹配已連結記憶

此設計使 Mem0 在「關鍵字精確匹配 + 語義模糊匹配」混合場景優於純向量 RAG [來源:53AI Mem0 深度解析,2026]。

2.4 Provider 抽象

源碼採工廠 + 策略模式，支援 16+ LLM、16+ 向量庫（Qdrant 推薦自架）、3 種圖資料庫 [來源:Walking in Code,2026]。預設開發可用 SQLite；生產通常接 Qdrant/Pinecone/Chroma [來源:Zylos Research,2026]。

│ Agent/Chat  │────▶│ Mem0 Memory API  │────▶│ Vector DB   │

三、功能特色與競品比較

| 維度 | Mem0 | Zep (Graphiti) | LangMem | Letta (MemGPT) |

|------|------|----------------|---------|----------------|

| 核心模型 | 向量+實體+ADD-only 事實 | 時序知識圖譜 | LangGraph Store SDK | OS 式分頁記憶 |

| p95 檢索延遲 | ~0.20s [來源:Atlan,2026] | ~200ms [來源:AgentMarketCap,2026] | ~59.8s（不適互動）[來源:Atlan,2026] | 未公布 |

| LOCOMO / LongMemEval | 91.6 / 94.8（新演算法）[來源:GitHub,2026] | 94.8% LongMemEval [來源:AgentMarketCap,2026] | 58.1% LOCOMO [來源:Atlan,2026] | 93.4%（MemGPT 時代） |

| 框架整合 | 21+（LangChain/CrewAI/AWS SDK）[來源:weavai.app,2026] | 雲端為主 | LangGraph 專用 | 自建 runtime |

| 圖/時序推理 | 實體連結；時序檢索已加入但弱於 Zep [來源:EverMind,2026] | valid_at/invalid_at 原生 [來源:EverMind,2026] | 無 | 中 |

| 自架 | 是（MIT/Apache OSS） | 雲端為主 | 是 | 是 |

| 中文實體抽取 | 預設英文 spaCy，中文需改造 [來源:53AI,2026] | 中 | 中 | 中 |

差異化總結：Mem0 強在速度、生態整合、開箱即用；Zep 強在事實如何隨時間變化的合規/CRM 查詢；LangMem 強在 LangGraph 程序性記憶（改寫 system prompt） 但延遲不適即時；Letta 強在自主 Agent 長期自管記憶 [來源:AgentMarketCap 2026-04,2026][來源:EverMind Zep Alternative,2026]。

四、使用方式

4.1 快速上手（Python OSS）


from mem0 import Memory

m = Memory()  # 需 OPENAI_API_KEY；向量庫可配置 Qdrant

# 寫入對話記憶
m.add(
    messages=[
        {"role": "user", "content": "我偏好正體中文摘要，ntfy topic 是 wangsc2025"},
        {"role": "assistant", "content": "已記錄您的通知偏好。"}
    ],
    user_id="digest-user-1"
)

# 檢索相關記憶注入 prompt
results = m.search(query="通知設定是什麼？", user_id="digest-user-1")
for mem in results["results"]:
    print(mem["memory"], mem.get("score"))

4.2 Platform 託管


from mem0 import MemoryClient
client = MemoryClient(api_key="...")
client.add(messages=[...], user_id="u1")
client.search(query="...", user_id="u1")

Platform 提供 Webhook、多模態、稽核日誌；適合不想自管 Qdrant 的團隊 [來源:Mem0 Platform Overview,2026]。

4.3 整合 LangChain / CrewAI

官方文件提供 mem0 作為 LangChain memory backend 與 CrewAI tool 的範例；AWS Agent SDK 亦有獨家深整合 [來源:weavai.app,2026][來源:InfoQ 火山 Mem0 QCon,2026]。

六、潛在應用（daily-digest-prompt）

場景 1：跨次 Todoist 任務的使用者偏好記憶

問題：context/continuity/auto-task-*.json 只存最近 5 次 runs 的結構化摘要，無法累積「使用者常改的配置偏好」（如 digest 格式、通知時段）。

落地步驟：

  1. 在 Phase 3 組裝完成後，將 summary + 使用者透過 Todoist 留言的偏好以 mem0.add() 非同步寫入
  2. Phase 0 讀取 mem0.search("使用者偏好 通知 digest") 注入 assemble-digest prompt 前 200 token
  3. 保留現有 continuity JSON 作為機械真相，Mem0 僅作軟性偏好層

場景 2：ai_github_research 去重輔助

問題：research-registry.json 以 topic 字串去重，語義相近但命名不同的專案仍可能重複研究。

落地步驟：

  1. 每次研究完成後 add() 專案名 + 核心架構標籤
  2. 步驟 2 發現熱門專案前 search() 比對語義相似度 >0.85 的既有記憶
  3. 與 KB hybrid search 並行，Mem0 管「研究過的事實」，KB 管「報告全文」

場景 3：KB 與 Mem0 分層（互補而非取代）

問題：KB 適合結構化長文；Mem0 適合對話型原子事實。混用會造成雙寫。

落地步驟：

  1. 定義邊界：KB = 研究報告/Skill 文件；Mem0 = 執行期學到的事實（「上次 LiteLLM 研究建議對照 Portkey」）
  2. next_suggested_angle 改由 Mem0 search 動態生成候選
  3. 監控 Mem0 API 成本（每 turn 抽取 ~$0.001–0.01）[來源:Zylos,2026]

七、優缺點評估

優點

  1. 延遲與準確率平衡最佳：p95 ~0.20s + LOCOMO 91.6，適合即時 Agent [來源:Atlan,2026][來源:GitHub,2026]
  2. 生態整合廣：21+ 框架、AWS SDK 背書，降低接入成本 [來源:weavai.app,2026]
  3. ADD-only 避免覆寫遺失：時序查詢「以前 vs 現在」有據可查 [來源:Memory Evaluation Docs,2026]
  4. 雙模式部署：OSS 自架 + Platform 託管，符合 ADR-006 狀態邊界思維 [來源:Mem0 Platform,2026]
  5. 中文 LLM 抽取可用：Phase 2 有中文 prompt（ADDITIVE_EXTRACTION_PROMPT_ZH）[來源:53AI,2026]

缺點

  1. 中文實體連結預設失效：spaCy 英文模型導致 Entity Boost 對中文無效，需 LLM-NER 或 zh 模型 [來源:53AI,2026]
  2. 抽取成本與延遲：圖模式下一輪對話 ~10s（社群實測），不適同步 hot path [來源:微信 Mem0 實測,2026]
  3. 記憶衝突不穩定：本地模型對矛盾事實可能同時 ADD 兩條衝突記憶 [來源:微信實測,2026]
  4. 批評性基準存在反例：MemBench 上 Mem0 比全上下文貴 14–77×、召回低 31–33%（特定設定）[來源:HN Universal LLM Memory,2026]
  5. 非結構化狀態不適排程真相：不可用它取代 state/scheduler-state.json（Agent 唯讀）[來源:本專案 AGENTS.md,2026]

不適用情境

  • 需要 valid_at/invalid_at 合規級時序圖譜 → 選 Zep
  • LangGraph 程序性記憶（自我改 prompt） → 選 LangMem
  • 純排程 FSM / schema 固定狀態 → 維持 JSON + PS1 管線
  • 零外部依賴離線環境 → Mem0 仍需 LLM + 向量庫

---

八、一家之言

Mem0 在 2026 年的定位已从「RAG 加料」升级为 Agent 基础设施中的 Working Memory 外包层。对 daily-digest-prompt 而言，最高杠杆不是全面替换 continuity/KB，而是在 Phase 0 注入 3–5 条用户偏好事实 与 研究去重语义层 两处做 POC；同时必须接受：在中文正体环境下默认配置下，Entity Linking 收益有限，应优先启用纯向量+BM25 路径，待替换 NER 后再开图增强。若下一跳要深化，建议对标 Portkey/LiteLLM 网关层 做「记忆写入成本 vs token-efficiency 管線」的 TCO 实验，而非再堆一个全量 Agent 框架。
