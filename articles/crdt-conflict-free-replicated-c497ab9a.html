CRDT（Conflict-free Replicated Data Types）完整指南

技術概述

CRDT（Conflict-free Replicated Data Types，無衝突複製資料型別）是一類專為分散式系統設計的資料結構，允許多個副本（replica）獨立且並發地更新，無需中央協調或鎖機制，並保證所有副本最終收斂到一致狀態。CRDT 的數學基礎建立在半格（join-semilattice）之上：只要合併操作具備結合律（associative）、交換律（commutative）與冪等律（idempotent），即可保證最終一致性。

2025-2026 年，CRDT 從學術研究走向工業規模應用：Figma 的即時協作、AppFlowy 的 Yrs CRDT 引擎、Redis Enterprise Active-Active 地理分散式資料庫、以及 Loro 框架的 Movable Tree + 富文本 CRDT 均採用 CRDT 作為核心。Local-First Software 運動的興起更讓 CRDT 成為離線優先應用的首選同步機制。

核心概念與原理

兩大 CRDT 類型

| 類型 | 同步機制 | 網路需求 | 優勢 | 劣勢 |

|------|---------|---------|------|------|

| State-based（CvRDT） | 傳送完整狀態，接收方合併 | 最終送達即可 | 實作簡單，容錯性高 | 頻寬消耗大（傳送完整狀態） |

| Operation-based（CmRDT） | 傳送操作（op），接收方重播 | 需因果順序保證 | 頻寬高效（僅傳操作） | 需可靠傳輸；實作較複雜 |

| Delta-state CRDT | 只傳差異（delta） | 最終送達即可 | 結合兩者優點 | 需追蹤已同步的狀態 |

CRDT vs OT（Operational Transformation）

| 面向 | CRDT | OT |

|------|------|-----|

| 架構 | 去中心化（P2P 可用） | 需中央伺服器 |

| 離線支援 | 原生支援（合併即可） | 需伺服器在線轉換 |

| 複雜度 | 資料結構設計複雜 | 轉換函式設計複雜 |

| 意圖保留 | 較弱（依資料結構語意） | 較強（顯式轉換） |

| 效能 | 元資料開銷較大 | 轉換計算開銷較大 |

| 主流使用者 | Figma、AppFlowy、Loro | Google Docs、SharePoint |

| 2026 趨勢 | 快速增長（Local-First 推動） | 成熟穩定（30+ 年歷史） |

關鍵差異：OT 透過「轉換」操作確保意圖一致，CRDT 透過「合併」狀態確保最終一致。OT 較適合需要嚴格意圖保留的場景（如純文字協作編輯），CRDT 較適合離線優先與 P2P 場景。

十大 CRDT 資料結構速查表

1. G-Counter（Grow-Only Counter）

  • 原理：每個副本維護自己的計數，總值 = 所有副本計數之和。合併取各副本的 max。
  • 用途：頁面瀏覽數、按讚數等遞增指標
  • 限制：不支援遞減；空間隨副本數線性增長

2. PN-Counter（Positive-Negative Counter）

  • 原理：組合兩個 G-Counter — 一個記遞增、一個記遞減。值 = 遞增總和 - 遞減總和
  • 用途：庫存數量、資源池
  • 限制：雙倍空間；無法原子重置

3. G-Set（Grow-Only Set）

  • 原理：只支援新增的集合。合併 = 聯集（union）
  • 用途：事件日誌、標籤收集、不可變登錄
  • 限制：無法移除元素

4. 2P-Set（Two-Phase Set）

  • 原理：維護 add-set 和 remove-set。成員 = 在 add-set 且不在 remove-set
  • 用途：任務完成追蹤、權限撤銷
  • 限制：一旦移除永遠無法重新加入

5. OR-Set（Observed-Remove Set）

  • 原理：每次新增賦予唯一標籤（tag）。移除只刪已觀察到的標籤。元素存在 = 仍有存活標籤
  • 語意：Add-wins — 並發的新增和刪除，新增勝出
  • 用途：協作購物車、完整的增刪操作
  • 限制：較大空間開銷（每次新增一個唯一標籤）

6. LWW-Register（Last-Write-Wins Register）

  • 原理：值 + 時間戳。合併取較新者
  • 用途：使用者偏好設定、快取值
  • 限制：並發更新丟失；依賴時鐘同步

7. MV-Register（Multi-Value Register）

  • 原理：保留所有因果並發的值。讀取回傳值集合
  • 用途：需要衝突感知的設定欄位
  • 限制：應用層須處理多值衝突

8. LWW-Element-Set

  • 原理：為每個元素的新增/移除附上時間戳，最新操作決定成員資格
  • 用途：功能旗標、寬鬆一致性的快取集合
  • 限制：需時鐘同步；並發操作可能丟失資料

9. RGA（Replicated Growable Array）

  • 原理：每個元素有唯一 ID 和父參照。順序由因果插入關係決定，非索引
  • 用途：協作文字編輯、共享列表
  • 限制：實作複雜；墓碑累積影響效能

10. Movable Tree

  • 原理：支援節點移動的樹狀 CRDT，基於 Kleppmann 等人的高可用性移動操作算法
  • 用途：檔案系統、組織架構、大綱編輯器
  • 限制：並發移動可能產生循環，需特殊處理

主流 CRDT 框架比較（2026）

| 框架 | 語言 | 特色 | 適用場景 |

|------|------|------|----------|

| Yjs | JavaScript | 最流行的文字協作 CRDT；平面列表結構（非樹）；豐富的編輯器整合（ProseMirror、TipTap、Monaco） | 前端即時協作 |

| Yrs | Rust | Yjs 的 Rust 移植；相同算法、更高效能；WASM/Python/C++ 綁定 | 需要跨語言的後端整合（如 AppFlowy） |

| Automerge | Rust+JS | Martin Kleppmann 主導；JSON-like 文件模型；內建持久化與壓縮 | 通用 Local-First 應用 |

| Loro | Rust+JS+Swift | 新一代高效能框架；Movable Tree + Fugue 文字算法 + Peritext 富文本；時間旅行與 Undo/Redo | 高效能 Local-First 應用 |

| Diamond Types | Rust | 極致效能優化；Fugue 算法實作 | 效能敏感的純文字協作 |

| Redis CRDT | C（Redis 模組） | Active-Active 地理分散式；內建於 Redis Enterprise | 全球分散式快取與會話管理 |

Loro 框架亮點（2025-2026 新星）

  • Replayable Event Graph：基於可重播事件圖的核心架構
  • Fugue 算法：最小化並發文字編輯的交錯異常
  • Peritext 富文本：支援粗體、斜體等格式標記的 CRDT 語意
  • MovableList + MovableTree：支援元素重排和節點移動
  • Fractional Index：保證兄弟節點排序穩定性
  • 時間旅行：可回溯到任意歷史版本

生產環境最佳實踐

1. 選擇正確的 CRDT 類型


需要計數器？
├── 只增 → G-Counter
└── 增減 → PN-Counter

需要集合？
├── 只增 → G-Set
├── 增刪但不重加 → 2P-Set
└── 完整增刪重加 → OR-Set（推薦）

需要單值？
├── 允許丟失並發 → LWW-Register
└── 需保留衝突 → MV-Register

需要序列/文字？
└── RGA 或 Yjs/Automerge/Loro 等框架

需要樹狀結構？
└── Movable Tree（Loro 推薦）

2. 垃圾回收策略

墓碑（tombstone）累積是 CRDT 最大的實務挑戰：

  • 時間基準清除：超過 TTL 的墓碑可安全移除（需確保所有副本已同步）
  • 版本向量協調：確認所有副本已觀察到刪除後清除
  • 有限結構 + 降級語意：限制最大元素數，超限時降級為 LWW
  • 週期性全量同步：定期執行 state-based 合併壓縮元資料

3. 網路分區處理

  • CRDT 在分區期間持續運作（CAP 中的 AP）
  • 重連後自動合併，結果確定性（可測試）
  • 測試重點：模擬分區場景，驗證合併結果正確性

4. 效能優化

  • 使用 Delta-state CRDT 減少同步頻寬
  • 大型文件考慮分片（chunking），每片獨立 CRDT
  • 版本向量的壓縮與裁剪
  • Rust 實作（Yrs/Loro）比 JavaScript（Yjs）快 10-100 倍

程式碼範例：JavaScript 中的 G-Counter


// G-Counter CRDT 實作
class GCounter {
  constructor(replicaId) {
    this.replicaId = replicaId;
    this.counts = {}; // Map<replicaId, number>
  }

  increment() {
    this.counts[this.replicaId] = (this.counts[this.replicaId] || 0) + 1;
  }

  value() {
    return Object.values(this.counts).reduce((sum, n) => sum + n, 0);
  }

  merge(other) {
    for (const [id, count] of Object.entries(other.counts)) {
      this.counts[id] = Math.max(this.counts[id] || 0, count);
    }
  }
}

// 使用範例：兩個副本獨立遞增後合併
const replica1 = new GCounter('agent-1');
const replica2 = new GCounter('agent-2');

replica1.increment(); // agent-1: 1
replica1.increment(); // agent-1: 2
replica2.increment(); // agent-2: 1

replica1.merge(replica2);
console.log(replica1.value()); // 3（正確合併）

程式碼範例：Python 中的 OR-Set


import uuid
from typing import Dict, Set, Any

class ORSet:
    """Observed-Remove Set CRDT — Add-wins 語意"""

    def __init__(self):
        self.elements: Dict[Any, Set[str]] = {}

    def add(self, element):
        tag = str(uuid.uuid4())
        self.elements.setdefault(element, set()).add(tag)
        return tag

    def remove(self, element):
        self.elements.pop(element, None)

    def contains(self, element) -> bool:
        return element in self.elements and len(self.elements[element]) > 0

    def merge(self, other):
        all_elements = set(self.elements) | set(other.elements)
        for elem in all_elements:
            my_tags = self.elements.get(elem, set())
            their_tags = other.elements.get(elem, set())
            merged = my_tags | their_tags
            if merged:
                self.elements[elem] = merged
            elif elem in self.elements:
                del self.elements[elem]

    def value(self) -> set:
        return {e for e, tags in self.elements.items() if tags}

常見陷阱

  1. 時鐘漂移：LWW 系列 CRDT 依賴時間戳排序，不同機器的時鐘漂移會導致非預期結果。使用 Lamport 時鐘或混合邏輯時鐘（HLC）替代物理時鐘。
  1. 墓碑爆炸：頻繁刪除的 OR-Set/RGA 會累積大量墓碑。需定期垃圾回收，但清除時必須確保所有副本已同步。
  1. 記憶體膨脹：CRDT 元資料（版本向量、唯一標籤）隨副本數和操作數增長。Delta-state 和壓縮策略是必要的。
  1. 交錯異常：傳統 RGA 在並發文字插入時可能產生交錯（interleaving）。Fugue 算法（Loro 採用）專門解決此問題。
  1. 不適用交易場景：CRDT 計數器不能用於銀行帳戶餘額（合併可能允許負值）。需要交易語意的場景仍需要共識協議。
