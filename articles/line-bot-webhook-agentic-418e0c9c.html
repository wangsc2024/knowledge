LINE Bot Webhook 驅動的 Agentic 任務管線設計

從訊息解析到「研究→優化→審查→實作→通知」自動化閉環架構

研究日期：2026-03-19

任務 ID：6gC584Jf9VMc4925

研究動機：讓 Daily-Digest-Prompt 系統支援透過 LINE 傳送自然語言指令，自動解析並執行多階段 Agentic 任務管線（深度研究→方案優化→反覆審查→實作→通知），實現端到端自動化。

---

一、架構概述：五階段 Agentic Pipeline

1.1 端到端流程


LINE 使用者訊息
  → LINE Webhook (HTTPS POST)
    → 意圖解析 (Groq classifier)
      → 任務排入 Todoist + 觸發 Agentic Pipeline
        → Phase 1: 深度研究 (Deep Research Agent)
        → Phase 2: 方案生成與優化 (Optimizer Agent)
        → Phase 3: 反覆審查迴圈 (Maker-Checker Loop)
        → Phase 4: 實作執行 (Implementation Agent)
        → Phase 5: 結果通知 (ntfy + LINE pushMessage)

1.2 設計原則

| 原則 | 說明 |

|------|------|

| Sequential Pipeline | 各階段有明確線性依賴，後階段建立在前階段輸出之上（Microsoft Azure Architecture 推薦模式） |

| Quality Gates | 每個階段間設置品質閘門，不合格則回退重做（Maker-Checker Loop） |

| 自癒目標 | 方案必須通過「自主管理門檻」審查才能進入實作，確保系統可持續自我修復 |

| 事件驅動 | Webhook 即時推送事件，零延遲觸發工作流（非輪詢模式） |

二、LINE Webhook 接收與意圖解析

2.1 Webhook 事件結構（LINE Messaging API）

LINE Platform 發送 HTTPS POST 請求，payload 包含：

  • destination：Bot user ID
  • events[]：事件陣列，每個事件含 type、message、source、replyToken
  • webhookEventId：唯一識別碼（用於去重）
  • timestamp：事件時間戳

安全驗證：必須驗證請求 header 中的簽名（HMAC-SHA256），防止惡意請求。

2.2 現有 classifier.js 擴充方案

現有系統已透過 classifier.js 使用 Groq API 進行意圖分類，支援 is_periodic、is_scheduled、is_workflow、task_type 等欄位。

新增 is_agentic_pipeline 意圖：


// classifier.js 擴充
const AGENTIC_PIPELINE_PATTERNS = [

  /研究.+(?:審查|檢討).+(?:實作|執行).+(?:通知)/,
  /(?:自癒|自主管理).+(?:研究|分析)/
];

// 分類結果新增欄位

輸入：「深度研究 agent harness 後提出系統現有 harness 優化方案，反覆以落實系統自癒為目標對方案進行審查與優化，直至方案能讓系統自主管理後再進行實作，完成實作後通知 wangsc2025」

  • 實作觸發：審查通過後

三、五階段 Pipeline 詳細設計

Phase 1：深度研究（Deep Research Agent）

職責：針對指定主題進行全面研究，產出結構化研究報告。

執行方式：

  1. 讀取 skills/web-research/SKILL.md 遵循標準研究框架
  2. 使用 WebSearch 多角度搜尋（概念性 + 技術性 + 比較性）
  3. WebFetch 深度閱讀 A/B 級來源
  4. 產出研究報告 JSON（含來源品質分級、交叉驗證事實）
  5. 匯入知識庫（KB）作為後續階段的知識基礎

輸出 Schema：


{
  "phase": "research",
  "topic": "agent harness",
  "findings": [...],
  "sources": [{"url": "...", "grade": "A", "summary": "..."}],
  "kb_note_id": "uuid",
  "quality_score": 4
}

品質閘門：research_depth >= "adequate"（來源 >= 3 且 A/B 級 >= 1）

Phase 2：方案生成與優化（Optimizer Agent）

職責：基於 Phase 1 研究成果，提出針對性優化方案。

執行方式：

  1. 讀取 Phase 1 輸出 + 系統現有架構（docs/ARCHITECTURE.md、context/ 目錄）
  2. 分析現狀與研究發現的差距（Gap Analysis）
  3. 產出優化方案（含優先級、影響範圍、風險評估）
  4. 方案必須包含「自主管理」可驗證指標

輸出 Schema：


{
  "phase": "optimize",
  "proposals": [
    {
      "id": "P1",
      "title": "...",
      "priority": "P0",
      "impact": "high",
      "self_healing_score": 0.8,
      "verification_criteria": ["..."]
    }
  ]
}

Phase 3：反覆審查迴圈（Maker-Checker Loop）

職責：以「自癒 / 自主管理」為目標，反覆審查並優化方案。

設計模式：採用 Microsoft Azure Architecture Center 推薦的 Maker-Checker Loop（又稱 Evaluator-Optimizer / Generator-Verifier / Critic Loop / Reflection Loop）。

執行流程：


Optimizer Agent (Maker) → 產出/修訂方案
  ↓
Reviewer Agent (Checker) → 依品質閘門評估
  ↓
通過？ → 進入 Phase 4
未通過？ → 回饋具體改進建議 → Maker 修訂 → 重新評估
  ↓
最大迭代次數（3 次）→ 人工升級或取最佳版本

品質閘門（自主管理門檻）：

| 維度 | 門檻 | 說明 |

|------|------|------|

| 自動偵測能力 | >= 0.7 | 方案能自動偵測異常的覆蓋率 |

| 自動修復能力 | >= 0.6 | 方案能自動修復的故障類型比例 |

| 人工介入需求 | <= 0.3 | 需要人工介入的場景比例 |

| 回歸防護 | >= 0.8 | 修復後不引入新問題的信心度 |

迭代控制：

  • max_iterations: 3（防止無限迴圈）
  • 每次迭代記錄改進差異（delta）
  • 若連續 2 次 delta < 5% → 提前結束（收斂判定）

Phase 4：實作執行（Implementation Agent）

職責：依據審查通過的方案執行實作。

執行方式：

  1. 讀取 Phase 3 最終方案
  2. 拆解為可執行任務清單（TDD 粒度）
  3. 依序執行：寫測試 → 最小實作 → 驗證 → 重構
  4. 每個子任務產出 evidence（證據鏈）

安全機制：

  • 實作前自動建立 git branch
  • 每步驟可回滾
  • 禁止直接修改 main branch

Phase 5：結果通知（Notification Agent）

職責：透過 ntfy 和 LINE pushMessage 通知使用者。

通知內容：


✅ Agentic Pipeline 完成
📋 主題：agent harness 優化
🔬 研究：5 個來源，depth=thorough
💡 方案：3 個提案，審查通過（2 輪迭代）
🔧 實作：8 個任務完成，測試全綠
📊 自癒評分：0.85/1.0

4.3 Pipeline Executor 核心邏輯


// pipeline-executor.js（概念設計）
async function executePipeline(config) {
  const { research_topic, optimization_target, quality_gate, notify_topic } = config;
  
  // Phase 1: Research
  const researchResult = await runAgent('deep-research', { topic: research_topic });
  if (!passQualityGate(researchResult, 'research')) throw new PipelineError('research_failed');
  
  // Phase 2: Optimize
  const optimizeResult = await runAgent('optimizer', { 
    research: researchResult, target: optimization_target 
  });
  
  // Phase 3: Maker-Checker Loop
  let proposal = optimizeResult;
  for (let i = 0; i < MAX_ITERATIONS; i++) {
    const review = await runAgent('reviewer', { proposal, gate: quality_gate });
    if (review.passed) break;
    if (review.delta < CONVERGENCE_THRESHOLD) break;
    proposal = await runAgent('optimizer', { feedback: review.feedback, proposal });
  }
  
  // Phase 4: Implement
  const implResult = await runAgent('implementer', { proposal: proposal.final });
  
  // Phase 5: Notify
  await notify(notify_topic, summarize(researchResult, proposal, implResult));
}

五、品質閘門與自癒機制

5.1 跨階段品質閘門

| 閘門位置 | 檢查項目 | 不合格處理 |

|----------|----------|------------|

| Research → Optimize | depth >= adequate, sources >= 3 | 補充搜尋後重試 |

| Optimize → Review | 方案含可驗證指標 | 回退要求補充指標 |

| Review → Implement | 自主管理門檻 4 維度全通過 | Maker-Checker 迴圈 |

| Implement → Notify | 測試全綠, 無 lint 錯誤 | 修復後重跑 |

5.2 自癒評分模型

自癒評分 = weighted_average(

自動偵測能力 × 0.3,

自動修復能力 × 0.3,

(1 - 人工介入需求) × 0.2,

回歸防護 × 0.2

)

門檻：>= 0.7 才允許進入實作階段。

六、與 Todoist 整合

6.1 任務建立

Pipeline 啟動時自動在 Todoist 建立追蹤任務：

  • 標題：[Pipeline] {research_topic} 研究→優化→實作
  • 標籤：[研究, 深度思維, 自動化, Pipeline]
  • 描述：含 pipeline ID、各階段狀態連結

6.2 狀態同步

每個 Phase 完成時更新 Todoist 任務描述，記錄進度。Pipeline 全部完成後自動關閉任務。

七、錯誤處理與降級

| 錯誤場景 | 處理方式 |

|----------|----------|

| Groq 分類失敗 | 降級為通用任務，不啟動 pipeline |

| 研究階段來源不足 | 降級為 KB 已有內容，標記 partial |

| 審查迴圈達上限 | 取最佳版本 + 品質警告通知使用者 |

| 實作階段測試失敗 | 回滾 git branch，通知使用者手動介入 |

| ntfy 通知失敗 | 重試 1 次，仍失敗則記錄日誌 |
