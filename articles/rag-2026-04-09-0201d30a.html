深度思維 RAG 非佛學洞見報告（2026-04-09〜04-11）

概述

本報告分析近三天（2026-04-09 至 2026-04-11）知識庫新增的 47 篇非佛學筆記，涵蓋 AI Agent 架構（10 篇）、軟體工程（7 篇）、認知科學（5 篇）、開源專案（8 篇）及系統營運（17 篇）五大類別。通過跨領域交叉分析，萃取出三個核心洞見模式。

---

一、核心洞見：分層治理契約化

1.1 治理三角（Governance Triangle）

來源：Harness Engineering 治理三角優化（04-11）

三條獨立的治理軸線——路徑風險分級（risk_tiers）、人工審核觸發（human_review_gates）、執行契約（execution_contracts）——應收斂為統一的「治理三角」。借鏡 Google SRE 錯誤預算（Error Budget）機制，以 Phase2 完成率等代理指標作為「可靠性預算」，預算見底時自動收緊變更策略。

關鍵概念：

  • OPA 風險式存取控制
  • IMDA Agentic AI 治理框架
  • component_flags 三層優先序（runtime_override > daily_limit > component_flags）
  • 冪等鍵與 trace_id 一致性

1.2 Reliability Surface R(k,ε,λ)

來源：Harness Engineering 生產級可靠度優化（04-10）

以三維曲面（取樣一致性 × 輸入擾動穩健度 × 故障注入容錯）作為 Harness 的動態適應度函數。Agent 專用 Chaos 工具（Flakestorm/agent-chaos）補足傳統基礎設施 Chaos 無法涵蓋的語意/工具失敗。

1.3 JSONL/OTel 語意對齊

來源：JSONL Hook metadata.source 與 OTel GenAI 對齊（04-10）

結構化日誌的 metadata.source 應對齊 OpenTelemetry GenAI 的 gen_ai.data_source.id 語意，實現分 API 快取 TTL 的精準治理。這是從「全域平均命中率」下鑽到「依端點」調優的前置條件。

二、核心洞見：認知卸載工程（Cognitive Offloading Engineering）

2.1 執行功能三成分

來源：執行功能與待辦事項完成（04-10）

待辦完成與執行功能（EFs）三成分——更新（updating）、轉換（shifting）、抑制（inhibition）——密切相關。關鍵設計啟示：

  • 用可信外部系統承載目標（不依賴人腦記憶）
  • 限制 WIP 上限 3-5
  • 切換前寫「停車句」（parking sentence）

2.2 實施意圖（Implementation Intentions）

來源：Sheeran et al.（2025）642 項後設分析（04-11）

642 項獨立檢驗確認 if-then 計畫對認知、情感與行為結果皆有效（d = 0.27-0.66）。then 子句應對應「可完成的極小步驟」以降低再決策成本。

2.3 記憶三層架構

來源：Agentic Memory × RAG 混合架構（04-10）

Agent 記憶應明確拆分為：

  1. 對話內短期狀態（session context）
  2. 外部知識檢索（RAG/cache）
  3. 跨 session 長期記憶（persistent store）

更大的 context window 不能取代短期記憶治理。

三、跨領域交叉模式

模式 A：分層 + 外部化 + 可量測

無論是技術系統（Harness 治理三角、記憶三層、JSONL 語意分層）或人類行為系統（執行功能三成分、實施意圖的 if-then 格式化），共同指向同一設計原則：

將混為一談的概念拆分為明確層級，各層有獨立職責與介面，並以可量化指標驅動治理決策。

這與第一性原理思考（First Principles Thinking, 04-10）的方法論高度吻合——「拆解到基本命題，再從根基重建」。

模式 B：降級路徑穩健性

Agent Chaos 注入測試系統在故障下的行為不變式，與執行功能中的「轉換成本」和實施意圖的「動機調節變項」形成對照：無論技術系統或人類，都需要在「正常路徑」之外驗證「降級路徑」的穩健性。

模式 C：工作流引擎收斂

Prefect（04-11）的 @flow/@task 裝飾子模式 + Dify（04-10）的 Agentic Workflow + DSPy（04-09）的 Signature/Module 編譯——三個不同層次的工作流框架都在收斂到「宣告式定義 + 可觀測執行 + 自動重試」的共同範式。

四、系統優化建議

建議 1（P0）：治理三角 SLI 定義

目標：在 autonomous-harness.yaml 中新增 governance_triangle 段落，定義 3 條 SLI：

  • Phase2 完成率（7 天滾動）
  • 自動任務成功率
  • 結果檔品質分數中位數

落地位置：config/autonomous-harness.yaml 新增 governance_triangle 區塊

建議 2（P1）：實施意圖模式引入任務排序

目標：在 scoring.yaml 中加入 if-then 格式偵測加分因子，鼓勵任務描述包含具體行動線索（時間/地點/事件觸發）。

落地位置：config/scoring.yaml 新增 implementation_intention_bonus 欄位

建議 3（P1）：metadata.source 標準化

目標：post_tool_logger.py 在 api-call/cache-read/cache-write 事件一律寫入可解析的 source 欄位，對齊 OTel gen_ai.data_source.id。

落地位置：hooks/post_tool_logger.py 的 api-call 標籤增強

建議 4（P2）：記憶分層檢查點

目標：明確區分 digest-memory.json（長期）、cache/（RAG）、continuity/（跨次短期），在 check-health.ps1 新增記憶層健康檢查。

落地位置：check-health.ps1 新增 [記憶分層] 區塊

建議 5（P2）：Phase2 SLO 曲面切片

目標：在 benchmark.yaml 中定義 Phase2 的 3 個 SLO 切片（可用性 ≥95%、延遲 P95 ≤120s、品質分數 ≥3.5），搭配 Chaos 最小 profile 驗證。

落地位置：config/benchmark.yaml 新增 phase2_slo 段落
