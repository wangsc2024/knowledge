二、解決的核心問題：Agent-Software Gap

2.1 現有方案的三大痛點

| 方案 | 痛點 | 後果 |

|------|------|------|

| GUI 自動化（RPA/截圖） | 依賴像素匹配、截圖解析 | 脆弱、不確定性高、解析度依賴 |

| 受限 API | 只覆蓋部分功能、需逐一整合 | 能力折損、開發成本高 |

| 自訂包裝器 | 每個軟體都要寫一套 | 維護成本爆炸、難以規模化 |

2.2 CLI-Anything 的解法

CLI 是人類和 AI Agent 的共同通用介面：

  • 結構化文字命令：天然適合 LLM 推理鏈
  • 可組合性：管道（pipe）、重導向、腳本串接
  • 自描述性：--help 旗標讓 Agent 自動發現能力
  • 確定性輸出：--json 旗標回傳結構化 JSON，消除解析複雜度

關鍵洞見：不是重新發明介面，而是回到 Unix 哲學 — CLI 本身就是最佳的 Agent 介面。

三、7 階段自動化管線（核心技術）

| 階段 | 名稱 | 輸入 | 輸出 | 關鍵動作 |

|------|------|------|------|----------|

| 1 | Analyze | 軟體原始碼 | 功能映射表 | 掃描 GUI 操作，映射到底層 API |

| 2 | Design | 功能映射表 | 命令架構 + 狀態模型 | 設計命令群組、子命令結構、狀態管理 |

| 3 | Implement | 命令架構 | Click CLI + REPL | 建構 Click CLI，含互動式 REPL（ReplSkin） |

| 4 | Plan Tests | CLI 實作 | TEST.md | 建立全面性測試策略 |

| 5 | Write Tests | TEST.md | pytest 測試套件 | 實作單元測試 + E2E 測試 |

| 6 | Document | 測試結果 | 更新 TEST.md | 以實際執行結果更新文件 |

| 7 | Publish | 完整 CLI | setup.py + PATH 安裝 | 建立 pip 可安裝套件，安裝到系統 PATH |

3.1 Analyze 階段

  • 對軟體原始碼進行靜態分析
  • 將 GUI 操作映射到對應的 API 呼叫
  • 建立「GUI 動作 → API 函式」的完整映射表

3.2 Implement 階段架構

  • 命名空間：cli_anything.* 統一命名，避免衝突
  • 雙模式輸出：JSON（Agent）+ 人類可讀（除錯）
  • Session 管理：持久化狀態 + undo/redo
  • REPL（ReplSkin）：每個 CLI 自帶互動式 shell

3.3 測試雙層架構

  • 單元測試（~1,011 項）：個別命令功能驗證
  • E2E 測試（~425 項）：CLI 到軟體後端完整流程
  • 子程序驗證：CLI 作為獨立程序的行為一致性

七、對 Agent 生態的深層影響

7.1 範式轉移：API 經濟 → CLI 經濟

  • 被動等待 API → 主動從原始碼生成 CLI
  • 片段整合 → 完整後端存取
  • 人工包裝 → 自動化管線

7.2 Unix 哲學的 AI 時代復興

CLI-Anything 將 Unix 哲學延伸到 Agent 時代：小工具、管道組合、文字流，--help 等同 MCP Tool Schema。

7.3 與 MCP 互補

  • MCP 定義通訊協議；CLI-Anything 解決工具端問題
  • 組合路徑：CLI-Anything 生成 CLI → MCP 封裝 → Agent 呼叫

7.4 對 Daily-Digest-Prompt 的借鑑

  1. 自描述工具模式（--help ≈ SKILL.md frontmatter）
  2. 7 階段管線（更細粒度的 Phase 自動化）
  3. 雙層測試策略（單元 + E2E）
  4. REPL 互動式除錯

---

八、總結

CLI-Anything 代表 Agent-Software 整合的範式級解法：

  1. 讓任何軟體自動生成 Agent 介面
  2. 回歸 CLI 本質，文字命令是人機互動最佳公約數
  3. 學術 + 工程兼具（港大研究室，生產級品質）
  4. 與 MCP、OpenClaw 互補而非競爭
