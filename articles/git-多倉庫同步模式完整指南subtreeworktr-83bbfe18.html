Git 多倉庫同步模式完整指南

研究背景

在多專案開發中（如 game → game_web → GitHub 部署），經常需要在多個 Git 倉庫之間同步程式碼。本研究比較三種 Git 原生的多倉庫管理方案——Subtree、Worktree、Submodule，並提供 PowerShell 自動化腳本實踐。

一、三種模式比較

1.1 Git Submodule（子模組）

將外部倉庫作為子目錄嵌入，維持獨立的版本控制。

核心指令：

# 新增子模組
git submodule add https://github.com/user/lib.git lib/external

# 初始化（clone 後必須）
git submodule init
git submodule update --recursive

# 更新到最新
git submodule update --remote --merge

優點：

  • 子倉庫完全獨立，可推送變更回上游
  • 精確鎖定版本（commit hash），可重現性高
  • 適合「共用元件庫」場景

缺點：

  • clone 後需額外 submodule init/update 步驟
  • 團隊成員容易忘記更新子模組
  • .gitmodules 管理增加複雜度
  • CI/CD 管線需額外配置

1.2 Git Subtree（子樹）

將外部倉庫的程式碼直接合併進主倉庫的子目錄。

核心指令：

# 新增子樹
git subtree add --prefix=lib/external \
  https://github.com/user/lib.git main --squash

# 拉取更新
git subtree pull --prefix=lib/external \
  https://github.com/user/lib.git main --squash

# 推送變更回上游
git subtree push --prefix=lib/external \
  https://github.com/user/lib.git main

# 分割子樹為獨立分支（Mono-as-Poly 模式）
git subtree split --prefix=lib/external -b lib-external-release

優點：

  • clone 後即可使用，無額外步驟
  • 所有程式碼在同一倉庫，搜尋與重構方便
  • 不需要團隊學習額外工具
  • --squash 可壓縮外部歷史，保持主倉庫整潔

缺點：

  • 倉庫體積較大（含外部程式碼完整歷史）
  • 推送回上游操作較複雜
  • 不適合需要獨立版本號的元件

1.3 Git Worktree（工作樹）

從同一個倉庫建立多個工作目錄，同時操作不同分支。

核心指令：

# 建立工作樹（已有分支）
git worktree add ../project-feature feature-branch

# 建立工作樹（新分支）
git worktree add ../project-hotfix -b hotfix/urgent

# 列出所有工作樹
git worktree list

# 移除工作樹
git worktree remove ../project-feature

# 清理已失效的工作樹
git worktree prune

優點：

  • 無需多次 clone，共享 .git 物件庫
  • 可同時在多個分支上工作（如：main + feature + hotfix）
  • fetch/pull 自動同步到所有工作樹
  • 適合「並行開發 + 快速切換」場景

缺點：

  • 佔用額外磁碟空間（每個工作樹一份檔案副本）
  • 子模組支援仍不完整
  • 不適合跨倉庫場景（僅限同一倉庫的不同分支）

三、Monorepo vs Polyrepo 架構選擇

3.1 Monorepo（單一倉庫）

所有專案放在同一個倉庫中。

適用場景：

  • 專案間緊密耦合，經常需要跨專案修改
  • 團隊規模較小（< 50人）
  • 需要原子提交（一個 PR 修改多個套件）

知名使用者： Google、Facebook、Twitter、Uber、Microsoft

工具支援： Nx、Turborepo、Bazel、Lerna、pnpm workspace

3.2 Polyrepo（多倉庫）

每個專案/服務獨立一個倉庫。

適用場景：

  • 團隊自治性高，各自負責獨立服務
  • 需要精細的存取權限控制
  • 獨立部署週期

知名使用者： Netflix、Amazon、Lyft

3.3 混合模式

  • Poly-as-Mono：多個倉庫用 Meta 等工具統一管理，像 Monorepo 一樣操作
  • Mono-as-Poly：Monorepo 開發，用 git subtree split 分割為獨立倉庫發布

四、PowerShell 自動化多倉庫同步

4.1 批次更新所有本地倉庫

# sync-all-repos.ps1
param(
    [string]$RootPath = "D:\Source",
    [string[]]$ExcludeRepos = @()
)

Get-ChildItem -Path $RootPath -Directory | Where-Object {
    (Test-Path (Join-Path $_.FullName ".git")) -and
    $_.Name -notin $ExcludeRepos
} | ForEach-Object -Parallel {
    $repo = $_.FullName
    $name = $_.Name
    try {
        Push-Location $repo
        $branch = git branch --show-current
        git fetch --all --prune 2>&1 | Out-Null
        $behind = (git rev-list --count "HEAD..origin/$branch" 2>$null)
        if ($behind -gt 0) {
            git pull --rebase 2>&1 | Out-Null
            Write-Output "[UPDATED] $name ($branch, $behind commits)"
        } else {
            Write-Output "[OK] $name ($branch, up to date)"
        }
    } catch {
        Write-Output "[ERROR] $name: $_"
    } finally {
        Pop-Location
    }
} -ThrottleLimit 4

4.2 遊戲專案多倉庫同步腳本

# sync-game-repos.ps1
# 場景：game（開發）→ game_web（部署構建）→ GitHub（發布）
param(
    [string]$GameDir = "D:\Source\game",
    [string]$WebDir = "D:\Source\game_web",
    [string]$GameName = "samsara-tower"
)

# 步驟 1：複製遊戲檔案到 web 部署目錄
$srcPath = Join-Path $GameDir $GameName
$destPath = Join-Path $WebDir $GameName

if (-not (Test-Path $destPath)) {
    New-Item -ItemType Directory -Path $destPath -Force | Out-Null
}

# 使用 robocopy 增量同步（僅複製變更檔案）
robocopy $srcPath $destPath /MIR /XD .git node_modules /XF .gitignore /NFL /NDL /NP

# 步驟 2：在 web 目錄提交並推送
Push-Location $WebDir
try {
    git add $GameName
    $status = git status --porcelain
    if ($status) {
        git commit -m "sync: update $GameName from game repo"
        git push origin main
        Write-Output "[SUCCESS] $GameName synced to game_web and pushed"
    } else {
        Write-Output "[SKIP] $GameName: no changes to sync"
    }
} finally {
    Pop-Location
}

4.3 Robocopy 增量同步參數說明

# /MIR  - 鏡像同步（新增+更新+刪除多餘檔案）
# /XD   - 排除目錄（.git, node_modules）
# /XF   - 排除檔案（.gitignore）
# /NFL  - 不列出檔案名（安靜模式）
# /NDL  - 不列出目錄名
# /NP   - 不顯示進度百分比
# /LOG  - 輸出日誌到檔案

五、常見陷阱與最佳實踐

5.1 常見陷阱

  1. Submodule 版本漂移：團隊成員忘記 git submodule update，導致使用過時版本

- 解法：CI 管線加入 git submodule status --recursive 檢查

  1. Subtree 合併衝突：git subtree pull 遇到衝突時難以解決

- 解法：使用 --squash 壓縮外部歷史，減少衝突機會

  1. Worktree 殘留：工作樹刪除目錄後忘記 git worktree prune

- 解法：定期執行清理，或在 CI 中加入 worktree prune

  1. 跨倉庫提交不同步：多倉庫修改忘記全部推送

- 解法：用自動化腳本確保原子性同步

5.2 最佳實踐

  1. 選擇最簡單的方案：如果手動複製 + 腳本能滿足需求，不必引入 Subtree/Submodule
  2. 文件化同步流程：在 README 或 CLAUDE.md 中記錄多倉庫關係與同步方式
  3. CI/CD 整合：自動化測試必須覆蓋跨倉庫的整合場景
  4. 版本標籤策略：多倉庫使用相同版本標籤（如 v2.0.0）確保一致性
  5. 權限分離：不同倉庫可設定不同的 collaborator 權限

六、與本專案的關聯

daily-digest-prompt 的多倉庫場景

本專案（daily-digest-prompt）在自動任務中涉及多倉庫操作：

  1. 遊戲同步：D:\Source\game（開發倉庫）→ D:\Source\game_web（部署倉庫）→ GitHub Pages

- 採用「腳本同步 + robocopy 增量複製」模式 - 自動任務 git-push.md 負責定時推送

  1. 知識庫同步：knowledge 專案推送至 GitHub

- 直接 git push，無跨倉庫需求

  1. Skills 複製：D:\Source\skills → daily-digest-prompt/skills/

- 手動或腳本複製，確保自包含

最佳化建議

  • 遊戲同步可考慮用 Git Subtree 取代 robocopy，獲得更好的歷史追蹤
  • PowerShell -Parallel 可加速多倉庫操作
  • 建立統一的 sync-all.ps1 腳本覆蓋所有同步需求
