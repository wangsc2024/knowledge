系統優化落實報告 2026-03-27：arch_evolution I/O 膨脹壓縮

系統優化kb-system-optimizedaily-digest-promptI/O 壓縮arch_evolution落實報告

優化背景

系統洞察分析顯示 avg_io_per_call 達 24,408 chars，超標 103%（門檻 12,000），其中 arch_evolution 任務的 avg_io 高達 66,910 chars，位居第一。

根因診斷

經分析發現，arch_evolution 雖已使用 Bash 萃取替代 Read 整檔，但萃取邏輯存在以下問題：

  1. improvement-backlog.json（186 KB）：萃取全部 124 items，無數量限制、無狀態篩選
  2. arch-decision.json（74 KB）：萃取全部 120 decisions，無數量限制
  3. 即使萃取必要欄位，大量 items 仍導致輸出超大（~40 KB）

實施方案

修正 1：improvement-backlog 萃取優化

  • 只萃取 status='pending' 的 items（跳過已完成的）
  • 限制最多 20 項（前 20 個待處理項目足夠決策）
  • description 欄位截斷至 150 字元（保留關鍵資訊）
  • 新增 total_items 和 pending_count 欄位（完整資訊透明化）

修正 2：arch-decision 萃取優化

  • 只保留 execution_status != 'success' 的條目（成功的跳過）
  • 限制最多 30 項
  • 新增 total_decisions 和 non_success_count 欄位

測試驗證


✓ improvement-backlog：25 KB → 9.4 KB（-62%）
✓ arch-decision：15 KB → 6.9 KB（-54%）
✓ 總共減少 23.7 KB I/O
✓ 配置驗證通過
✓ 自動任務一致性檢查通過（35/35）

預期效果

  • arch_evolution 的 avg_io 預計從 66,910 降至 ~43,000（-36%）
  • 雖未達 < 20,000 目標，但已大幅改善
  • 後續優化方向：分析輸出（Write 操作）大小、檢查重複讀取

技術細節

萃取邏輯關鍵改善：


# 原邏輯（無限制）
for item in d.get('items', [])

pending = [item for item in all_items if item.get('status') == 'pending'][:20]

知識點

  • 大檔處理三原則：篩選（只取需要的）、限制（數量上限）、截斷（長欄位壓縮）
  • Context 保護：即使用 Bash 萃取，也要控制輸出大小
  • 資訊透明化：新增 total_*/count 欄位，讓 Agent 知道完整資訊而不失去可觀測性
