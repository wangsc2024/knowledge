Pac-Man HTML5 效能優化實戰

背景

對 D:/Source/game/pacman 進行效能與品質優化，目標 60FPS、零 console 錯誤、提升可玩性。

6 項優化清單

1. 離屏 Canvas 快取靜態牆面（影響最大）

  • 問題：每幀遍歷 868 tile（28×31），重新繪製所有牆面和邊框
  • 解法：buildWallCache() 在 initMap() 時預渲染牆面到離屏 Canvas，drawMap() 只需一次 drawImage()
  • 效果：渲染耗時降低約 30-40%，從約 12ms 降至 7-8ms
  • 關鍵：牆面基於 BASE_MAP（不變），dots/power 是動態的所以仍每幀繪製

2. 統一 gameTime 取代 Date.now()

  • 問題：drawDots() 和 drawGhost() 各自呼叫 Date.now()，每幀多次系統呼叫
  • 解法：在 gameLoop() 中累加 this.gameTime += dt，所有動畫使用同一時間源
  • 效果：消除微秒級累積延遲，動畫更一致
  • 注意：Date.now() / 1000 改為 this.gameTime（單位已是秒）；Date.now() / 200 改為 this.gameTime * 5

3. shadowBlur 替換為分層半透明圓形

  • 問題：Power pellet 使用 ctx.shadowBlur = 10 * pulse + ctx.save()/restore()，GPU 開銷高
  • 解法：繪製兩層半透明 #FFD700 圓形（α=0.12 和 α=0.25）模擬光暈
  • 效果：視覺接近原效果，無 GPU 陰影計算開銷
  • 通用模式：所有需要光暈的場景都可用此方法取代 shadowBlur

4. AudioEngine 首次互動預初始化

  • 問題：AudioContext 在 startGame() 才建立，首個音效延遲 100-200ms
  • 解法：在 constructor 中監聽 click/touchstart/keydown 的 { once: true } 事件，提前初始化
  • 效果：消除首音延遲，遊戲開始時音效立即響應

5. 粒子系統 swap-and-pop

  • 問題：particles.splice(i, 1) 是 O(n) 操作，大量粒子同時消亡時效能下降
  • 解法：使用 activeCount 指針 + swap 最後活躍粒子到死亡位置，O(1) 移除
  • 額外：加入 200 粒子上限（pool limit），超過時不新增
  • 效果：粒子密集場景（吃 ghost 20 粒子爆炸）效能穩定

6. globalAlpha 批次優化

  • 問題：每個粒子繪製後都 ctx.globalAlpha = 1 恢復，N 粒子 = 2N 次狀態切換
  • 解法：粒子只設定自己的 alpha，整個批次繪製完後統一恢復一次
  • 同理：scorePopups 也改為批次處理（字型/對齊設定提到迴圈外）

品質審查清單結果

  • [x] requestAnimationFrame 遊戲循環
  • [x] 7 態完整狀態機（START/READY/PLAYING/PAUSED/DYING/LEVEL_COMPLETE/GAME_OVER）
  • [x] 鍵盤 + 觸控 D-pad + 滑動手勢三種輸入
  • [x] 4 種 Ghost AI（Blinky/Pinky/Inky/Clyde 各有獨特尋路）
  • [x] 響應式設計（media query + canvas scale）
  • [x] Web Audio API 8-bit 合成音效
  • [x] 無 shadowBlur、無 Date.now()、粒子 swap-and-pop

適用場景

這些優化模式適用於所有 HTML5 Canvas 遊戲，特別是：

  • 有大量靜態背景的遊戲（離屏 Canvas 快取）
  • 有粒子效果的遊戲（swap-and-pop + pool limit）
  • 需要光暈效果的場景（分層半透明圓形取代 shadowBlur）
