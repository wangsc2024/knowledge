  • 影響範圍（40 分）：所有 44 個自動任務的系統性失敗分析
  • 可執行性（33 分）：整合現有 failed-auto-tasks.json、system-insight、improvement-backlog，僅需新增 workflow YAML，不修改核心 config

工作流設計

觸發條件（三選一）

  1. system-insight 檢測到 daily_success_rate < 0.9
  2. failed-auto-tasks 新增 entry 且 consecutive_count ≥ 2
  3. 手動執行系統健康審查

步驟概覽（6 步驟）

#### 步驟 0：前置條件檢查

  • 讀取 context/system-insight.json
  • 確認 daily_success_rate < 0.9 且存在 critical alert
  • 若 ≥ 0.9，terminate_with_success（健康狀態跳過不算失敗）

#### 步驟 1：失敗叢聚分析

三類叢聚：

  1. 任務叢聚：同一任務連續失敗 ≥3 次
  2. 原因叢聚：同一失敗原因出現 ≥5 次
  3. 時段叢聚：high_failure_hours 與 failed_at 交叉分析

輸入：

  • state/failed-auto-tasks.json（task_key, reason, last_failed_at, consecutive_count）
  • context/system-insight.json（high_failure_hours）

輸出：

  • task_clusters（陣列）
  • reason_clusters（陣列）
  • time_clusters（陣列）

#### 步驟 2：根因定位

reason 映射到根因類型：

| reason | root_cause_type | 診斷步驟 |

|--------|----------------|----------|

| fsm_phase_timeout | timeout_insufficient | 讀取 timeouts.yaml，建議值 = 實際執行時間 × 1.2 |

| result_file_missing_or_invalid | result_json_issue | 執行 result-file-diagnostic-workflow.yaml |

| format_failed | schema_mismatch | 檢查 schema 與 prompt 必填欄位一致性 |

| task_did_not_complete_or_timeout | execution_interrupted | 檢查 Cursor CLI / Codex 日誌，確認 fail-safe 占位 |

輸出：root_causes 陣列（含 cluster_id、root_cause_type、evidence、affected_scope）

#### 步驟 3：修復建議生成

四類修復建議（SMART 原則）：

  1. schema_fix：

- 標題：修正 {task_key} 結果 JSON schema 不符問題

- 步驟：讀取 schema → 補強 prompt 必填欄位檢查 → validate_results.py 驗證

- 優先級：high

- 預估影響：消除 format_failed，success_rate +3-5%

  1. timeout_adjustment：

- 標題：調整 {task_key} timeout 至 {recommended_value}s

- 步驟：寫入 improvement-backlog（不直接修改 timeouts.yaml）

- 優先級：medium

- 預估影響：減少 fsm_phase_timeout，success_rate +2-4%

  1. prompt_fix：

- 標題：補強 {task_key} prompt 的 fail-safe 占位與驗證步驟

- 步驟：檢查 prompt 是否含 fail-safe + validate_results.py 步驟

- 優先級：high

- 預估影響：消除 task_did_not_complete_or_timeout，success_rate +5-8%

  1. tool_fix：

- 標題：修復 {tool_name} 邏輯缺陷

- 步驟：定位工具問題 → 寫入 backlog

- 優先級：high

- 預估影響：修復工具層問題，success_rate +3-6%

輸出：recommendations 陣列（含 id、type、title、action_steps、priority、estimated_impact、affected_tasks）

#### 步驟 4：整合至 improvement-backlog

優先級映射：

  • high：連續失敗 ≥5 次 或 影響 ≥3 個任務
  • medium：連續失敗 3-4 次 或 影響 2 個任務
  • low：單一任務偶發失敗（consecutive_count ≤ 2）

backlog entry schema：


{
  "id": "backlog_sr_{YYYYMMDD}_{序號}",
  "source": "system_success_rate_workflow",
  "title": "{recommendation.title}",
  "description": "{action_steps + evidence}",
  "priority": "high|medium|low",
  "effort": "low|medium|high",
  "created_at": "ISO-8601",
  "status": "pending",
  "tags": ["system_success_rate", "auto_detected", "workflow_forge"],
  "related_file": "{prompt 檔案路徑}"
}

容錯：backlog 寫入失敗不中斷 workflow（記錄警告）

#### 步驟 5：追蹤驗證設定

建立 context/success-rate-tracking.json：


{
  "version": 1,
  "baseline_success_rate": 0.724,
  "target_success_rate": 0.9,
  "analysis_date": "2026-04-15",
  "repair_items": ["backlog_sr_20260415_001", "backlog_sr_20260415_002"],
  "next_verification_date": "2026-04-22",
  "notes": "執行 workflow 的觸發條件與叢聚摘要"
}

驗證規則：7 天後執行 system-insight，比對 daily_success_rate 是否提升 ≥5%

成功標準：

  • 短期（7 天內）：success_rate ≥ 85%
  • 長期（14 天內）：success_rate ≥ 90%
