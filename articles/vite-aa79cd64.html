本次研究主題：Vite 靜態知識庫同步的內容指紋、快取失效與增量部署策略（因知識庫已有 Vite 同步概論，但尚無針對 hash、manifest、CDN 失效機制的深入研究）

本文為 vite-knowledge-sync-pipeline 系列第 mechanism 階段，承接既有的 build-time data 與 Cloudflare 發布概論，深入探討內容指紋、manifest 映射、CDN revalidation 與增量部署決策。

技術概述

Vite 在 production build 時，會把被引用的靜態資產納入 build asset graph，並輸出帶有內容指紋的檔名。這件事的工程意義不只是「防快取」，而是把每次部署轉成一組可追蹤、可比對、可回滾的 immutable artifacts。對知識庫型網站而言，真正會頻繁變動的是 HTML、索引資料與 manifest；JS/CSS chunk 多半只在版型或程式邏輯變更時才更新。因此最穩定的做法不是讓所有檔案用同一套 TTL，而是把 HTML/JSON 視為短生命週期資源，把 hash 後的 assets 視為長生命週期資源，並讓增量同步流程只重建受影響的內容與對應索引。這樣可以同時降低 origin 壓力、避免 stale HTML 指向舊資產，也讓 GitHub/Cloudflare Pages 發布更可預測。

核心概念與原理

1. Vite 的內容指紋不是附屬功能，而是部署契約

Vite 官方文件指出，被匯入的靜態資產會進入 build assets graph，並輸出帶 hash 的檔名。這代表資產 URL 本身就是版本號；只要內容不變，URL 就不應變；只要內容改變，URL 就必須變。對 CDN 與瀏覽器而言，這讓 Cache-Control: max-age=31536000, immutable 變得安全，因為同一 URL 不再承擔「內容可能偷偷變了」的風險。

2. manifest 是 HTML 與 hashed assets 之間的映射層

Vite 的 backend integration 文件說明，開啟 build.manifest 後，manifest 會記錄 entry chunk、dynamic chunk、CSS、asset 與 imports 關係。對知識庫同步站來說，manifest 的價值在於：

  • HTML 不必硬寫實際檔名，只要依 entry 查 manifest。
  • 若只更新內容資料而未改動前端 bundle，manifest 與 hashed assets 都可保持不變。
  • 若 bundle 改版，HTML 只要換到新 manifest 對應即可，不需要手動處理 preload/link 順序。

3. 快取策略必須分層，而不是全站同一 TTL

web.dev 與 Cloudflare 文件都指向同一原則：

  • 帶內容指紋的 JS/CSS/字型/圖片：可長期快取，因 URL 會隨內容變更。
  • HTML：不能長期 cache，因它引用的是某一批特定 hashed assets，若 HTML 舊了，就會造成整頁停留在舊版本。
  • 資料快照 JSON / 搜尋索引：通常介於兩者之間，適合短 TTL + revalidation。

Cloudflare 在 2026-02-26 將 stale-while-revalidate 改為 fully asynchronous。這對知識庫站很重要：快取過期後，第一個訪客不必等 origin，Cloudflare 會先回 stale 內容並背景更新。這可以降低發布瞬間的尖峰延遲，但前提是你清楚哪些資源允許 stale、哪些不允許。若把 HTML 也放太長的 stale window，使用者可能長時間拿到仍指向舊索引的頁面。

4. 增量部署的真正單位是「受影響資源集合」

今天專案裡的 tools/kb-sync-incremental.ps1 已示範「以上次同步時間為增量起點」的思路：先讀 state/records.json，再只抓 updated_after 之後的筆記，最後更新 snapshot。若把這個想法推進到 Vite/Pages 部署，最佳單位應是：

  • 變動筆記對應的 HTML 或資料頁
  • 受變動筆記影響的列表頁、tag 頁、搜尋索引
  • 若前端程式未改動，則 bundle 檔與 manifest 不應重新命名

也就是說，增量同步不是「少抓資料」而已，而是「盡量不讓無關資產失效」。

最佳實踐與常見陷阱

最佳實踐

  1. 讓 HTML 與資料索引短 TTL、可 revalidate；讓 hashed assets 長 TTL、immutable。
  2. 開啟 build.manifest，把 HTML 模板、部署檢查、回滾流程都綁在 manifest 上，而不是直接猜測 dist/assets 檔名。
  3. 將同步流程拆成三層：內容擷取、影響分析、部署輸出。只有第三層真的需要發佈。
  4. 發布後用 smoke check 驗證：首頁 HTML、manifest、至少一個 JS chunk、至少一個資料 JSON 是否彼此對得上。
  5. 若使用 Cloudflare，針對 HTML/JSON 啟用明確的 Cache-Control 與 revalidation 規則，而不是依賴預設值。

常見陷阱

  1. 把 HTML 也設成一年快取：結果是 HTML 指向舊資產或舊資料快照。
  2. 每次內容更新都重建全部 bundle：快取命中率大幅下降，Pages 發布成本變高。
  3. 只看檔名變化，不看 manifest 關聯：CSS 或 dynamic import 變了，頁面仍可能載到錯誤組合。
  4. 把 stale-while-revalidate 用在不該 stale 的資源：會讓發布後一段時間內仍持續看到舊內容。
  5. 沒有回滾指標：若新 manifest 發出，但關聯 chunk 缺失，網站會直接白屏。

與本專案的關聯

本專案今天的已完成任務包含「知識庫同步更新 Vite 網站並推送 GitHub」與「專案優化(4項+38測試)」。現有流程在 templates/auto-tasks/git-push.md 中要求先同步知識庫，再 build，再推送；而 tools/kb-sync-incremental.ps1 已具備增量抓取的起點。下一步可直接落地三件事：

  1. 將同步輸出分成 content snapshot、search/tag index、frontend bundle 三類，分別決定是否重建。
  2. 在部署後保留前一版 manifest 與關鍵資產清單，讓回滾是「切回前一版 manifest/HTML」而不是重新手工 build。
  3. 把 smoke test 納入 git push 或 Pages 發布後檢查，驗證 HTML 連到的 manifest entry 與實際 chunk 檔案存在。

程式碼範例


// vite.config.ts
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    manifest: true,
    rollupOptions: {
      input: {
        app: 'index.html'
      }
    }
  }
})


// deploy-check.ts
import fs from 'node:fs'

const manifest = JSON.parse(fs.readFileSync('dist/.vite/manifest.json', 'utf8'))
const entry = manifest['index.html'] || manifest['app.js']

if (!entry?.file) {
  throw new Error('manifest 缺少主入口')
}

for (const css of entry.css || []) {
  if (!fs.existsSync(`dist/${css}`)) {
    throw new Error(`缺少 CSS: ${css}`)
  }
}

if (!fs.existsSync(`dist/${entry.file}`)) {
  throw new Error(`缺少 JS chunk: ${entry.file}`)
}


建議快取分層
- HTML: Cache-Control: no-cache, must-revalidate
- 搜尋索引 / snapshot JSON: Cache-Control: public, max-age=300, stale-while-revalidate=30
- hashed assets: Cache-Control: public, max-age=31536000, immutable
