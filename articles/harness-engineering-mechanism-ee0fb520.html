Harness Engineering 品質閘門與約束執行引擎深度解析（mechanism 階段）

研究日期：2026-04-15
系列：harness-quality-gates（新系列）
階段：mechanism

執行摘要（300 字）

Harness Engineering 的品質閘門（Quality Gates）是確保 AI Agent 可靠運行的核心機制。本文深入解析約束執行引擎的內部架構，揭示三個關鍵洞見：（1）品質閘門分為預防性（feedforward）與修正性（feedback）兩類，前者在 Agent 行動前引導，後者在行動後糾錯；（2）約束執行發生在三個時點——pre-execution（阻止無效操作）、in-process（即時監控）、post-execution（輸出驗證），其中 pre-execution 是最有效的防線；（3）約束必須在框架層（framework-level）執行而非嵌入 system prompt，因為 LLM 會將 prompt 中的規則視為建議而非強制約束。

實務上，Microsoft Agent Governance Toolkit 實現了 sub-millisecond（<0.1ms p99）的約束執行延遲，OpenAI Agents SDK 使用 decorator 模式簡化約束定義，而 Strands Agents 的 BeforeToolCallEvent hook 可在工具呼叫前取消無效操作。本文整合 16+ 權威來源，提供完整的品質閘門架構圖、約束執行流程、以及可直接應用的實作模式。

研究背景

Harness Engineering 概念回顧

Harness Engineering 是 OpenAI 在 2026 年提出的新工程範式，專注於設計環繞 AI Agent 的系統、約束與反饋循環，使 Agent 能在生產環境中可靠運行 [來源：OpenAI, Harness Engineering, 2026]。與傳統軟體工程不同，Harness Engineering 的核心挑戰在於 AI Agent 的非確定性（nondeterministic）行為——相同輸入可能產生不同輸出，因此無法僅依賴傳統單元測試保證品質 [來源：Martin Fowler, Harness Engineering, 2026]。

品質閘門的必要性

Gartner 預測 2026 年將有 40% 的企業應用嵌入 AI Agent，較 2025 年的 5% 大幅成長 [來源：Bessemer Venture Partners, Securing AI Agents, 2026]。然而，只有 14.4% 的技術團隊取得完整的安全/IT 審批才允許 Agent 上線 [來源：Gravitee, State of AI Agent Security 2026]。這種「採用速度超越控制能力」（adoption outpaces control）的現象，凸顯了自動化約束執行機制的迫切性。

核心發現（含三角佐證標記）

發現 1：品質閘門的雙向控制架構 ✅（1 來源）

品質閘門分為兩類控制機制 [來源：Martin Fowler, Harness Engineering, 2026]：

Feedforward Controls（預防性控制）：在 Agent 行動前提供引導，包括：

  • Language servers 提供即時程式碼提示
  • Structural tests 檢查檔案結構與命名規範
  • Documented conventions 作為 Agent 的知識來源
  • 目標：「增加 Agent 第一次就產生正確結果的機率」

Feedback Controls（修正性控制）：在 Agent 行動後偵測問題，包括：

  • Linters 與 type checkers 偵測語法與型別錯誤
  • Automated code review 評估語義正確性
  • Structural analysis 驗證架構符合性
  • 關鍵：錯誤訊息必須「針對 LLM 消費優化」（optimized for LLM consumption），提供可操作的修正建議

這種雙向架構類似傳統控制理論，但針對非確定性系統重新設計。

發現 2：約束執行的三個時點 ✅（3 來源佐證）

約束執行引擎在三個關鍵時點介入 [來源：Agno, Guardrails for AI Agents, 2026; DEV Community, AI Agent Guardrails, 2026; OpenAI Agents SDK, Guardrails]：

1. Pre-execution（執行前驗證）— 最強防線

在工具呼叫執行前阻止無效操作。技術實作：


class NeurosymbolicHook(HookProvider):
    def validate(self, event: BeforeToolCallEvent) -> None:
        passed, violations = self._check_rules(tool_name, context)
        if not passed:
            event.cancel_tool = f"BLOCKED: {violations}"

[來源：DEV Community, AI Agent Guardrails, 2026]

關鍵優勢：

  • 無效操作「從未執行」（never execute），消除 rollback 複雜度
  • LLM 無法覆蓋框架層的取消決定
  • 測試顯示 3/3 無效操作被阻擋，1/1 有效操作通過，零誤判 [來源：同上]

2. In-process（運行中監控）

即時監控 Agent 推理過程，確保「模型保持在預期範圍內，遵守道德標準，避免風險或禁止行為」[來源：Agno, Guardrails for AI Agents, 2026]。例如：金融交易 Agent 可被限制在未經人類審核前不得執行超過特定金額的轉帳。

3. Post-execution（執行後驗證）

最後檢查點，在輸出交付前掃描「敏感或違反政策的內容」[來源：同上]。限制：此時問題已發生，需要補償性交易或回滾。

發現 3：框架層執行 vs Prompt 嵌入 ✅（4 來源佐證）

核心洞見：LLM 將 system prompt 中的業務規則視為「建議」而非「約束」[來源：DEV Community, AI Agent Guardrails, 2026]。

反模式示例：


System: "Payment must be verified before confirming booking."

Agent 可能仍未驗證付款就確認訂單，因為這只是文字，模型可以彈性解釋。

正確模式：在框架層以確定性程式碼執行約束


@dataclass
class Rule:
    name: str
    condition: Callable[[dict], bool]  # Boolean logic
    message: str

BOOKING_RULES = [
    Rule("payment_verified", 
         lambda ctx: ctx.get("payment_status") == "verified",
         "Payment must be verified first"),
]

[來源：同上]

這種分離實現「neurosymbolic approach」：神經元件（LLM）處理自然語言理解，符號元件（hook）執行確定性規則 [來源：同上]。

四個來源佐證此模式：

  1. DEV Community：詳細程式碼範例與測試結果
  2. Microsoft Agent Governance Toolkit：stateless policy engine 攔截所有 Agent 行動 [來源：Microsoft, Agent Governance Toolkit, 2026]
  3. Agno：BaseGuardrail class 實現框架層驗證 [來源：Agno, Guardrails for AI Agents, 2026]
  4. OpenAI Agents SDK：decorator-based guardrails 與 tripwire 機制 [來源：OpenAI Agents SDK, Guardrails]

發現 4：Sub-millisecond 約束執行 ✅（1 來源）

Microsoft Agent Governance Toolkit 的 Agent OS 元件實現了「無狀態政策引擎，在執行前攔截每個 Agent 行動，延遲 <0.1ms（p99）」[來源：Microsoft, Agent Governance Toolkit, 2026]。

技術架構：

  • 支援 YAML、OPA Rego、Cedar 政策語言
  • Stateless 設計支援水平擴展
  • Deterministic 執行確保可審計性

這證明了約束執行不必犧牲效能，sub-millisecond 延遲使其適用於即時互動場景。

深度分析

品質閘門的架構層級

完整的品質閘門系統包含五個層級 [來源：DEV Community, Building a Five-Layer Quality Gate, 2026]：

| 層級 | 執行時點 | 檢查類型 | 延遲 | 成本 |

|------|---------|---------|------|------|

| L1 | Pre-input | 敏感資料過濾 | <1ms | 極低 |

| L2 | Pre-execution | 工具呼叫驗證 | <1ms | 低 |

| L3 | In-process | 推理路徑監控 | 10-100ms | 中 |

| L4 | Post-execution | 輸出語義檢查 | 100-500ms | 中高 |

| L5 | Post-output | 人類審查（critical） | >1s | 高 |

OpenAI Agents SDK 實作模式

OpenAI Agents SDK 提供三種 decorator [來源：OpenAI Agents SDK, Guardrails]：


@input_guardrail  # 處理初始使用者輸入
@output_guardrail  # 檢查最終結果
@tool_input_guardrail / @tool_output_guardrail  # 包裹工具呼叫

每個 guardrail 函數回傳 GuardrailFunctionOutput：

  • tripwire_triggered: bool # 是否觸發阻擋
  • 支援兩種模式：

- Parallel execution（預設）：與 Agent 處理並行，最佳延遲

- Blocking execution：完成後才啟動 Agent，避免浪費 token

成本優化範例：


@input_guardrail
def detect_math_homework(input_text: str) -> GuardrailFunctionOutput:
    # 輕量分類 Agent 判斷是否為數學作業
    if is_math_homework:
        return GuardrailFunctionOutput(tripwire_triggered=True)
    return GuardrailFunctionOutput(tripwire_triggered=False)

[來源：同上]

若 tripwire 觸發，昂貴的主 Agent 不會執行，既優化成本又阻止不當行為。

實務應用建議

建議 1：優先實作 Pre-execution Guardrails

理由：pre-execution 阻止問題發生，post-execution 需要回滾 [來源：DEV Community, AI Agent Guardrails, 2026]。

實作步驟：

  1. 列出所有業務規則（如「訂位上限 10 人」「必須先驗證付款」）
  2. 將規則編碼為 deterministic 函數（Callable[[dict], bool]）
  3. 使用框架 hook 在工具呼叫前驗證
  4. 回傳明確錯誤訊息而非允許執行

建議 2：分離關注點（Separation of Concerns）

| 元件 | 職責 |

|------|------|

| LLM | 自然語言理解、工具選擇、推理 |

| Tools | 僅業務操作（無驗證邏輯） |

| Hooks | 確定性約束執行 |

[來源：DEV Community, AI Agent Guardrails, 2026]
