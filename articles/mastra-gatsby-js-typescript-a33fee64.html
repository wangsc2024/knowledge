一、專案概述

Mastra（mastra-ai/mastra）是由 Gatsby.js 創始團隊打造的開源 TypeScript AI Agent 框架，於 2026 年 1 月 21 日正式發布，定位為「Python trains, TypeScript ships」——為 TypeScript/Next.js 開發者提供一站式 AI Agent 開發工具鏈。獲得 Y Combinator W25 批次支持。

二、核心架構分析

2.1 六大核心模組

│            Mastra Instance              │
│  ┌───────┐ ┌──────────┐ ┌───────────┐  │
│  │Agents │ │Workflows │ │  Memory   │  │
│  │(LLM+  │ │(Graph+   │ │(Obs.Mem + │  │
│  │ Tools)│ │ Steps)   │ │ Semantic) │  │
│  └───┬───┘ └────┬─────┘ └─────┬─────┘  │
│      │          │             │         │
│  ┌───┴───┐ ┌────┴─────┐ ┌────┴──────┐  │
│  │  MCP  │ │Guardrails│ │  Evals +  │  │

│  └───────┘ └──────────┘ └───────────┘  │

  1. Agents：自主型 LLM + Tools 執行單元，支援 40+ 模型提供者、動態模型路由（依 requestContext 切換模型）、子 Agent 階層委派、structured output（Zod schema）
  2. Workflows：Graph-based 工作流引擎，.then() 鏈式組合 + createStep() 原子步驟，支援 Suspend/Resume（Human-in-the-loop）、stateSchema 跨步驟狀態共享
  3. Memory：三種記憶系統——對話記憶（短期）、工作記憶（跨輪次）、語義記憶（RAG 向量檢索）+ Observational Memory（獨創）
  4. MCP：Model Context Protocol 整合，支援 stdio/SSE/Streamable HTTP 三種傳輸、mcpClients 持久化儲存（LibSQL/PostgreSQL/MongoDB）
  5. Guardrails：輸入/輸出處理器（Processors），支援攔截、轉換、重試、短路執行，可用 Workflow 原語組合複雜審核管線
  6. Evals + Observability：版本化 Datasets + 可執行 Experiments（SCD-2 風格版本追蹤），OpenTelemetry 原生追蹤，REST API 支援 CI 整合

三、Observational Memory（OM）— 核心創新深度解析

3.1 問題背景：Context Rot 與 Context Waste

傳統 LLM Agent 記憶方案面臨兩大挑戰：

  • Context Rot：對話越長，模型表現越差（關鍵資訊被噪音淹沒）
  • Context Waste：tool call 回傳的大量原始數據（如 Playwright 頁面快照 50,000+ tokens）佔據上下文窗口

3.2 三層壓縮架構


原始對話歷史（50,000+ tokens）
       │
       ▼  Observer Agent（閾值：30,000 tokens）
觀察筆記 Observations（壓縮 5-40x）
       │
       ▼  Reflector Agent（閾值：40,000 tokens）
反思摘要 Reflections（進一步壓縮）
       │
       ▼
穩定的長期記憶上下文
  • 第一層（Messages）：保留最近對話用於當前任務
  • 第二層（Observations）：Observer 背景 Agent 監控對話，當 token 超過 messageTokens 閾值（預設 30,000）時，將歷史壓縮為簡潔觀察筆記
  • 第三層（Reflections）：當觀察筆記超過 observationTokens 閾值（預設 40,000）時，Reflector Agent 合併相關項目並萃取模式，形成長期記憶

3.3 關鍵設計決策

| 參數 | 預設值 | 說明 |

|------|--------|------|

| messageTokens | 30,000 | 觸發觀察壓縮的閾值 |

| observationTokens | 40,000 | 觸發反思壓縮的閾值 |

| bufferTokens | 0.2（20%） | 上下文窗口緩衝比例 |

| 預設模型 | google/gemini-2.5-flash | OM 壓縮用的 LLM |

| 儲存 | @mastra/pg, libsql, mongodb | 持久化後端 |

3.4 兩種作用域

  • Thread Scope（預設）：每個對話線程獨立觀察，安全且成熟
  • Resource Scope（實驗性）：跨對話線程共享觀察（如同一用戶的所有對話共享記憶），適合個人助理場景，但有任務黏性不確定性

3.5 與傳統方案的差異

| 方案 | Context Rot | Token 效率 | 長期記憶 | 快取友善 |

|------|------------|-----------|---------|----------|

| 固定窗口截斷 | 嚴重 | 差 | 無 | 否 |

| RAG 動態檢索 | 中等 | 中等 | 有 | 否（每輪不同） |

| Observational Memory | 低 | 高（5-40x） | 有 | 是（穩定上下文） |

四、技術棧與生態系統

4.1 Monorepo 結構

  • Build：Turborepo + pnpm workspace
  • Testing：Vitest
  • Deploy：多 server adapter（Node.js、Vercel、Cloudflare、Netlify）
  • Storage：PostgreSQL、LibSQL（SQLite）、MongoDB

4.2 整合生態

| 類別 | 整合 |

|------|------|

| LLM 提供者 | OpenAI, Anthropic, Google Gemini, DeepSeek, Groq, Mistral 等 40+ |

| Web 框架 | Next.js, Express, Hono |

| 工具協議 | MCP（stdio/SSE/Streamable HTTP） |

| 觀測 | OpenTelemetry 原生 |

| 語音 | Voice capabilities 模組 |

| Auth | 內建認證模組 |

4.3 開發者體驗亮點

  1. Developer Studio：視覺化 Playground，即時觀看 Agent 思考過程、逐步除錯 Workflow
  2. 自動 OpenAPI 文檔：每個 Agent 自動生成 Swagger 介面
  3. 型別安全：Zod schema 貫穿 Agent → Tool → Workflow → Output
  4. API 自動生成：指向任何 API，自動生成 TypeScript 型別安全套件

六、對 Daily-Digest-Prompt 系統的五大借鑑啟示

啟示 1：三層記憶壓縮 → digest-memory.json 優化

Mastra 做法：Messages → Observations → Reflections 三層自動壓縮

可借鑑：當前 context/digest-memory.json 是扁平結構，可引入「觀察層」——將每日執行結果壓縮為觀察筆記，跨週再壓縮為反思摘要，降低 LLM 讀取 memory 的 token 開銷

啟示 2：Suspend/Resume 工作流 → PowerShell Phase 中斷恢復

Mastra 做法：Workflow 支援 suspend() 暫停 + restart() 恢復，狀態自動持久化

可借鑑：當前 run-todoist-agent-team.ps1 的 Phase 失敗後重試機制可參考此模式——將 Phase 狀態序列化至 state/run-fsm.json，失敗後從上次 Phase 精確恢復而非完全重跑

啟示 3：Guardrails 組合管線 → Hook 管線增強

Mastra 做法：Processors 可用 Workflow 原語組合（並行驗證、複雜審核管線）

可借鑑：當前 Hooks 是獨立的 pre/post 攔截器，可考慮引入組合模式——例如 pre_bash_guard + pre_write_guard 共享判定結果，避免重複計算

啟示 4：版本化評估系統 → 執行品質追蹤

Mastra 做法：Datasets（SCD-2 版本化）+ Experiments（跑分追蹤）

可借鑑：當前 state/research-quality.json 和 state/kb-note-scores.json 可升級為版本化追蹤——每次評估保留歷史版本，便於趨勢分析和回歸偵測

啟示 5：MCP 持久化客戶端 → Skill 連線管理

Mastra 做法：mcpClients 配置持久化儲存，支援多伺服器獨立工具選擇

可借鑑：當前 Skills 依賴各自的 API 連線，可引入統一的「連線管理器」——將 API 端點、認證、健康狀態統一管理（類似 cache/api-health.json 的擴展版）

七、可直接應用的技術項目

7.1 高優先（與現有架構直接對齊）

  1. Observational Memory 模式應用於 digest-memory.json

- 實作 context/digest-observations.json（每日觀察）+ context/digest-reflections.json（每週反思）

- 預期效果：Phase 1 Agent 讀取記憶的 token 降低 3-5 倍

  1. Guardrails 組合模式應用於 Hook 系統

- 將 hook-rules.yaml 的規則支援管線組合（pipe operator）

- 預期效果：減少重複計算，提升 Hook 執行效率

7.2 中優先（需額外開發）

  1. Workflow Suspend/Resume 應用於 FSM

- state/run-fsm.json 增加 suspend_payload 欄位

- Phase 失敗時自動保存中斷點，下次執行精確恢復

  1. 版本化評估系統

- state/research-quality.json 改為 state/research-quality-history.json（保留最近 30 天）

7.3 低優先（長期願景）

  1. Developer Studio 風格的執行視覺化

- 為 query-logs.ps1 增加 Web 視覺化介面
