AI Agent 架構的編排層抽象：從單體推理到多代理協作的四層堆疊與信任模型

TL;DR（執行摘要）

本文主張：多代理編排並非簡單的「多個 agent 並行」，而是需要專門協議層（MCP/A2A/TEA）與四層堆疊抽象（context/agent/data/planner）的架構躍遷。基於 12 篇論文與官方文檔的三角佐證，本研究論證三個核心觀點：(1) 協議標準化取代硬編碼工具呼叫是企業級編排的必要條件，提供版本化生命週期管理與跨系統可追溯性；(2) registry-based routing 相比硬編碼提供治理彈性，但需承擔額外開銷，中小團隊可能不適用；(3) LLM-driven scheduler 可動態分配資源（1–10+ 子代理依複雜度），在並行任務上達 90% 時間減少，但序列任務會退化 70%，選擇閾值為專門化需求、政策隔離、提示清晰度、trace 可讀性四維度。研究涵蓋 2024–2026 年前沿進展，分析 Anthropic/OpenAI/LangChain 實踐與 arXiv 最新架構提案，為開發者提供可操作的編排模式選型指南。

研究問題與範疇

研究目標

本研究聚焦於「學習者能理解編排層抽象的必要性、掌握四層堆疊的職責分離、知道如何選用單體 vs 多代理架構」三個核心目標，採深入學習導向而非學術爭議梳理。

子問題

  1. 多代理編排為何需要專門的協議層（MCP, registry）而非直接工具呼叫？
  2. 四層堆疊（context／agent／data／planner）如何解決企業級知識孤島與權限邊界問題？
  3. 從 ReAct 到 Gradientsys 的演進中，調度器的 LLM-driven 設計帶來哪些 trade-off？

成功標準

讀者讀完後能：理解編排層抽象的必要性（非「為何要多代理」而是「為何要協議層」）、掌握四層堆疊的職責分離（能畫出 context/agent/data/planner 邊界）、知道如何選用單體 vs 多代理架構（依四維度判準）。

文獻探討

協議標準化路線（2024–2026）

MCP (Model Context Protocol) 與 A2A (Agent2Agent) 雙協議設計

arXiv 論文 "The Orchestration of Multi-Agent Systems: Architectures, Protocols, and Enterprise Adoption" (2601.13671, 2026) 提出統一架構框架，整合規劃、政策執行、狀態管理與品質運營為連貫的編排層 [arXiv, 2026]。該研究定義兩種互補協議：MCP 標準化自主代理存取外部工具與上下文數據的方式，A2A 治理對等協調、協商與任務委派 [arXiv, 2026]。這兩種協議共同構成「可互操作的通訊基底，支持可擴展、可審計且符合政策的分散式代理集體推理」 [arXiv, 2026]。

TEA (Tool-Environment-Agent) Protocol 版本化生命週期

AgentOrchestra 框架（arXiv 2506.12508, 2026）引入 TEA Protocol，將工具、環境與代理建模為「一等公民的版本化資源，具備明確生命週期」 [arXiv, 2026]。該設計相比前身協議（A2A / MCP）提供結構化版本控制與生命週期管理，改善複雜多代理場景的可追溯性與可重現性 [arXiv, 2026]。框架在 GAIA Test 基準達 89.04% 準確率，處理專家級任務與科學數學推理挑戰 [arXiv, 2026]。

產業實踐：LangChain Tool Registry 與 OpenAI Agents SDK

LangChain 實作 centralized tool registry，所有連接工具存在單一統一註冊表，可跨工作區存取 [LangChain, 2026]。平台整合 Slack / Gmail / Linear / Pylon 與遠端 MCP server 於單一位置，代理建立時自動存取「工作區中的每個工具」而非個別配置 [LangChain, 2026]。存取控制由工作區管理員層級操作，防止未授權工具增生，同時簡化標準使用者的代理開發 [LangChain, 2026]。

OpenAI Agents SDK 提供兩種編排模式：handoff（控制權完全轉移給專家代理）與 agents-as-tools（管理者保持擁有權並調用專家執行特定任務如摘要或分類）[OpenAI, 2026]。文檔建議從單一代理開始，僅當專家「實質改善能力隔離、政策隔離、提示清晰度或追蹤可讀性」時才新增專家 [OpenAI, 2026]。

編排架構實證（Anthropic / AgentOrchestra）

Orchestrator-Worker Pattern 與動態資源分配

Anthropic 實作 orchestrator-worker 設計，lead agent 作為協調者並委派專門研究任務給並行運作的子代理 [Anthropic, 2026]。Lead agent 執行策略規劃（使用延伸思考評估任務複雜度並確定最優子代理分配）、任務分解（將複雜查詢拆為具明確輸出格式的特定研究目標）、資源分配（簡單事實查詢使用 1 個代理，複雜研究涉及 10+ 專門子代理）與結果綜合（編譯發現並確定是否需額外研究迭代）[Anthropic, 2026]。

並行執行效能實測

系統透過兩種並行化策略達顯著效能提升：同時建立 3–5 個子代理（而非序列）與子代理並行調用 3+ 搜尋工具 [Anthropic, 2026]。Anthropic 報告這些改變「將複雜查詢的研究時間減少達 90%」，評估數據顯示「使用 Claude Opus 4 作為 lead agent 與 Claude Sonnet 4 子代理的多代理系統，勝過單一代理 Claude Opus 4 達 90.2%」 [Anthropic, 2026]。

Google Research 的代理擴展研究發現，多代理協調在可並行任務上提供 +81% 改善，但在序列任務上導致最高 70% 退化 [Google Research, 2024]。獨立多代理系統（並行代理無通訊，輸出由綜合聚合）達平均效能 0.370，但經歷 17.2 倍 trace 級錯誤放大，因代理複製錯誤而無修正 [Medium, 2026]。

單體 vs 多代理決策框架

四維度判準（Microsoft Azure CAF）

Microsoft Azure Cloud Adoption Framework 提供選擇標準：僅當全部四條件成立時採用多代理 —— (1) 任務分解為半獨立子問題，(2) 綜合步驟增值而非增加開銷，(3) 單一代理基線已充分優化且達瓶頸，(4) 增量商業收益超過完整協調、治理與基礎設施成本 [Microsoft, 2026]。

單一代理適用於聚焦、線性任務，涉及一個系統或領域，決策不依賴多個專門角色 [Microsoft, 2026]。多代理考量組織原因：安全或合規邊界要求知識領域隔離、不同團隊擁有不同代理、可審計性需要明確 handoff 日誌 [Microsoft, 2026]。當法規或政策強制嚴格數據隔離時，建立多代理，因不同安全分類需要單一代理無法提供的獨立處理環境 [Microsoft, 2026]。

專門化 Trade-off

更多專門化改善有界子任務的適配性，但增加編排負擔，每個權衡有依任務結構而定的交叉點 [Medium, 2026]。區分角色不自動證成分立代理：首先驗證單一代理能否透過提示、檢索與政策控制模擬專門化，再拆分為多代理 [Medium, 2026]。

核心技術區別為上下文管理：單一代理系統中所有狀態累積於一個視窗，多代理系統中每個子代理在自己隔離上下文中運作 [Dataiku, 2026]。

核心論證

協議層抽象的必要性

標準化存取 vs 硬編碼工具呼叫

直接工具呼叫（function calling）要求每個代理實作自己的工具綁定、認證與錯誤處理邏輯，導致跨系統複用困難與版本飄移風險。MCP / TEA 協議提供統一介面，將工具建模為版本化資源，支持生命週期管理（註冊→啟用→棄用→退役）與跨代理可追溯性 [arXiv 2601.13671; arXiv 2506.12508]。

LangChain Tool Registry 實證顯示，centralized governance 可在新工具連接後立即對所有代理可用，無需個別重新認證步驟，減少運營摩擦同時維持安全協議 [LangChain, 2026]。管理員集中管理憑證，當工具需要重新認證時獲得立即可見性，防止分散式憑證管理的安全漏洞。

協議標準化的未收斂困境

當前 MCP（Anthropic 主導）、A2A（多方提案）、TEA（AgentOrchestra 特定）三足鼎立，尚未出現統一標準。每種協議解決部分問題：MCP 強調外部工具存取標準化，A2A 聚焦代理間協商，TEA 提供版本化生命週期。這種碎片化增加開發者學習成本與跨框架遷移難度，統一工作面臨組織政治阻力（各框架維護者爭奪協議制定權）。

四層堆疊的職責分離

Context 層：知識與權限邊界

arXiv 2601.13671 提出的架構將 context 層定義為「規劃、政策執行、狀態管理與品質運營」的整合層 [arXiv, 2026]。這層處理知識來源路由（哪些代理可存取哪些數據）與權限邊界執行（RBAC / ACL 規則）。

企業場景常見挑戰為知識孤島：不同部門數據有不同安全分類（公開/內部/機密），單一代理累積所有狀態會違反最小權限原則。Context 層透過 scoped retrieval 確保每個代理僅存取授權範圍內的知識，同時維持跨代理 handoff 時的 audit trail [Microsoft, 2026]。

Agent 層：專門化策略與政策隔離

專門化不僅是技術分工，更是政策隔離的必要條件。OpenAI Agents SDK 文檔說明，handoff 適用於「不同分支需要不同指令、工具或政策」的情境，控制權移轉給專家代理 [OpenAI, 2026]。例如：退款代理需遵守財務合規規則（最高退款額度、審計日誌），計費代理需存取支付閘道，兩者政策空間不相容，強制在單一 prompt 中共存會導致指令衝突與錯誤回應。

Agent 層提供 prompt 隔離（每個專家有自己系統提示）、工具隔離（僅授權所需工具存取）與記憶隔離（對話歷史不跨代理洩漏）。

Data 層：Registry-Based Routing 治理

傳統硬編碼路由（if task_type == "refund" then call refund_agent）在小型團隊快速但缺乏治理：新增代理需修改路由邏輯、無版本控制、難以 A/B 測試不同路由策略。

Registry-based routing 將代理註冊為可發現服務，metadata 包含能力描述（capabilities）、輸入 schema、SLA 承諾與版本標籤 [LangChain, 2026]。路由器（可為規則式或 LLM-driven）查詢 registry 選擇適配代理，支持動態路由（運行時決定）與金絲雀部署（新代理版本先路由 5% 流量驗證）。

Trade-off 為額外開銷：registry 維護、metadata 管理、服務發現延遲。中小團隊（< 5 個代理）硬編碼仍優，治理收益不足以抵銷複雜度。

Planner 層：LLM-Driven vs 規則式調度

Anthropic lead agent 實作 LLM-driven planner：根據查詢複雜度動態決定子代理數量（簡單事實查詢 1 個，複雜研究 10+）與並行策略 [Anthropic, 2026]。這種設計適合研究場景，where 任務結構事先未知且需自適應分解。

規則式調度（DAG-based workflows like Airflow / Prefect）提供可預測執行路徑、成本預估與除錯簡易性，適合生產環境已知工作流。Trade-off 為靈活性 vs 可控性：LLM planner 可處理新穎任務但可能過度分配資源（成本超支），規則式調度無法適應未見任務結構但執行可重現。

當前生產環境主流仍為規則式，LLM planner 用於探索性任務（客服對話路由、研究查詢分解）而非關鍵路徑批次作業。

編排模式的 Trade-off

並行執行效能邊界

Anthropic 報告的 90% 時間減少適用於可並行分解任務（多角度研究、多來源驗證），但 Google Research 發現序列任務退化 70% [Google Research, 2024]。原因為協調開銷：並行代理完成後需等待最慢者，再由綜合步驟整合結果，總時間為 max(agent_times) + synthesis_time。若任務本質序列（步驟 B 依賴步驟 A 輸出），強制並行會因空等與重複工作導致效能退化。

實務判斷：若任務可拆為 3+ 個互不依賴子問題且綜合步驟 < 10% 總時間，並行收益顯著；否則單一代理序列執行更優。

狀態持久化與檢查點機制

Anthropic 報告代理維持對話狀態跨越潛在數百輪，需耐久錯誤恢復與檢查點系統而非完全重啟 [Anthropic, 2026]。長時執行研究任務（30 分鐘+）若遇網路中斷或代理錯誤，從頭重跑成本高。檢查點機制（LangGraph 提供 stateful orchestration with checkpointing）允許從最近狀態恢復，但增加狀態管理複雜度（檢查點存儲、版本控制、冪等性保證）。

生產部署策略為「rainbow deployments」：逐步轉移流量於版本間，防止活躍代理中途研究中斷 [Anthropic, 2026]。這需要狀態序列化格式向後相容，確保舊檢查點可被新版本代理讀取。

單體瓶頸識別：四維度判準實務應用

Microsoft CAF 的四條件提供決策框架，實務應用需量化：

  1. 半獨立子問題：若任務分解後子問題間依賴邊 < 20%（DAG 中非序列路徑比例），滿足條件；否則序列依賴過強，拆分無益。
  1. 綜合增值：綜合步驟應產生「1+1>2」洞察（交叉驗證、多角度論證、矛盾調和），而非僅串接輸出。測試：人工綜合 vs 自動拼接，前者品質顯著優則滿足。
  1. 單一代理瓶頸：已優化 prompt、採用 RAG、調整 retrieval strategy，效能仍達瓶頸（準確率 plateau / latency 無法降低）。需基線測試驗證。
  1. 成本收益：估算編排基礎設施成本（registry 維護、檢查點存儲、監控儀表板）+ 治理開銷（政策更新、審計追蹤），與業務收益（準確率提升 → 用戶留存 / latency 降低 → 轉化率）比較。

一家之言

基於前述證據，本研究提出三個獨立判斷：

1. 協議標準化的政治經濟學困境

MCP / A2A / TEA 三足鼎立並非技術能力問題，而是組織政治博弈。Anthropic 主導 MCP 搭配自家 Claude 生態，OpenAI 推動 Agents SDK 與自己 API 深度綁定，開源社群（AgentOrchestra 等）提出第三選擇避免商業鎖定。統一標準需要一方讓渡協議制定權，這在商業競爭環境中不太可能發生。

預測：未來 2–3 年將出現「協議適配層」（protocol adapters）：開發者寫一次代理邏輯，透過適配層同時支持 MCP / A2A / TEA，類似 ORM 抽象不同資料庫方言。這增加抽象成本但避免協議鎖定。

2. Registry 是治理必要而非效能優勢

Registry-based routing 的主要價值不在效能（反而增加服務發現延遲），而在治理可見性：哪個代理被調用多少次、失敗率多少、平均延遲幾秒、是否遵守 SLA。這些 metrics 支持容量規劃、異常檢測與成本歸因，對企業 IT 部門是硬需求。

但中小團隊（< 10 人）硬編碼仍優：開發速度快、除錯直接、無 registry 維護負擔。建議閾值為「5+ 個代理且需跨團隊協作」時才引入 registry，否則過早抽象會拖累迭代速度。

3. LLM Planner 適合探索、規則式調度仍主流生產

Anthropic 的 LLM-driven lead agent 在研究場景展現價值：任務結構未知、需自適應分解。但生產環境（訂單處理、客服路由、批次 ETL）仍以規則式調度為主，原因為：

  • 成本可預測性：規則式調度事先知道會調用幾次 LLM、總 token 成本，LLM planner 可能過度分配（複雜度估計偏高導致建立過多子代理）。
  • 除錯與監控：DAG workflow 失敗時直接看哪個節點錯誤，LLM planner 決策過程為黑箱（「為何建立 7 個子代理而非 3 個」難以解釋）。
  • 合規審計：金融/醫療等監管產業需要決策可解釋性，規則式調度的 if-then 邏輯可被審計，LLM 決策難以滿足合規要求。

預測：LLM planner 會長期存在於探索性任務（新客服查詢、研究報告），成熟後轉為規則式（提煉為固定 DAG）部署生產。混合模式（LLM 提案 + 規則驗證）可能成為中間路徑。

實踐建議

1. 先優化單體 Agent，Plateau 後再拆分

依 Microsoft CAF 判準，僅當單一代理基線已充分優化（prompt engineering、RAG 調校、retrieval strategy）且達效能瓶頸時，才考慮多代理。過早拆分會將簡單問題複雜化，增加協調開銷而無對應收益。

實作檢查清單：

  • [ ] 已測試 5+ 種 prompt 變體並選最優
  • [ ] 已採用 RAG 且調整 top-k / chunk size
  • [ ] 已量測單一代理準確率與 latency 基線
  • [ ] 基線指標達平台期（連續 3 次迭代無顯著改善）

2. 權限邊界 > 專門化需求作為拆分第一判準

技術專門化（「這個代理擅長摘要，那個擅長分類」）可能可用單一代理 + 不同 prompt 模擬，但權限邊界（「這個代理只能存取公開數據，那個可存取機密」）強制分離。優先依安全/合規需求拆分，技術專門化為次要考量。

範例：客服系統拆分為「一般查詢代理」（存取 FAQ）與「帳戶代理」（存取 PII）時，驅動因素應為「帳戶代理需通過額外認證且 audit log 需標記 PII 存取」，而非「帳戶查詢比一般查詢複雜」。

3. 採 Registry 前評估治理開銷 vs 團隊規模

小於 5 個代理時，硬編碼路由（switch-case 或 if-elif）開發速度快且除錯直接。引入 registry 需投入：

  • 初期建置：registry schema 設計、metadata 標準化、服務發現邏輯
  • 持續維護：metadata 更新（代理能力變更時）、registry 高可用（單點故障風險）、監控儀表板（observability）

成本收益臨界點約為「5+ 代理且跨 2+ 團隊協作」或「需要金絲雀部署 / A/B 測試路由策略」時，治理價值才超過開銷。

4. 並行執行需 Task Decomposition 能力驗證

並行收益前提為任務可分解為互不依賴子問題。驗證方法：人工將任務拆為 3+ 子任務，檢查依賴邊比例（DAG 中序列路徑佔比）。若 < 20% 序列依賴，並行有利；若 > 50%，強制並行會因等待與重複工作導致退化。

測試流程：

  1. 選代表性任務 10 個，人工分解為子任務並標記依賴
  2. 計算平均序列依賴比例
  3. 實作單一代理基線 vs 並行編排，量測 wall-clock time
  4. 若並行版本 < 1.5× 單一代理時間，視為成功；否則任務結構不適合並行

風險與限制

1. 研究時效性

本研究截至 2026-06-08，TEA Protocol (AgentOrchestra) 尚未有大規模生產案例驗證，性能與穩定性數據主要來自論文 benchmark（GAIA Test 89.04%）。實際企業部署可能遇論文未涵蓋問題（長時狀態管理、檢查點恢復失敗、版本升級向後相容性）。

MCP / A2A 標準化進展可能快速變化，6 個月後可能出現新版本協議或社群統一提案，影響架構選型。建議讀者追蹤 Anthropic / OpenAI 官方部落格與 arXiv cs.AI 最新提交。

2. 效能數據可比性

Anthropic 報告的 90% 時間減少、Google Research 的 +81% / -70% 數據皆來自 internal benchmark，任務構成、代理配置、基礎模型版本未完全公開。跨組織可比性受限，不同團隊複現可能得到不同結果。

獨立 benchmark（如 GAIA Test）提供標準化評估，但覆蓋任務類型有限（主要為問答與推理），企業實際工作流（文檔處理、客服路由、數據管道）可能表現不同。

3. 協議標準化的不確定性

MCP / A2A / TEA 三足鼎立局面可能持續 1–2 年，開發者選擇協議面臨鎖定風險。若投入 MCP 深度整合後，社群統一到 A2A，遷移成本高。建議策略為：

  • 使用協議適配層（如 LangChain Agents 支持多種後端）降低鎖定
  • 優先選擇有大廠背書協議（MCP 有 Anthropic、Agents SDK 有 OpenAI）降低棄用風險
  • 關注開源替代方案（AgentOrchestra 等）作為備選路徑

4. 成本模型的黑箱問題

LLM-driven planner 的 token 成本難以事先估算：lead agent 可能依查詢複雜度建立 1–10+ 子代理，若複雜度估計演算法未公開（Anthropic 使用 extended thinking 但細節未披露），成本預算風險高。生產環境需設置 hard cap（最大子代理數、總 token 上限）防止成本失控，但這又限制 LLM planner 的自適應能力。

規則式調度的固定 DAG 雖無黑箱，但無法處理新穎任務結構。Trade-off 為可預測性 vs 靈活性，目前無完美解決方案。
