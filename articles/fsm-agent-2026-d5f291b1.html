有限狀態機（FSM）設計模式完整指南

技術概述

有限狀態機（Finite State Machine, FSM）是一種數學計算模型，描述一個系統在任何時刻只能處於有限數量的「狀態」之一，並透過「事件」驅動的「轉換」在狀態之間切換。FSM 是軟體工程中最基礎且強大的行為建模工具——從嵌入式系統、遊戲引擎、UI 元件到 AI Agent 工作流編排，FSM 提供了可預測、可視覺化、可測試的狀態管理框架。

2025-2026 年，隨著 AI Agent 系統的爆發式增長，FSM 正在經歷文藝復興——XState 的 Actor Model 整合、LangGraph 的狀態機驅動 Agent 編排、以及 Python python-statemachine 3.0 的 Statechart 支援，都標誌著 FSM 從「教科書概念」走向「生產級架構核心」的轉變。

核心概念與原理

1. FSM 四大組件

| 組件 | 說明 | 範例 |

|------|------|------|

| 狀態（States） | 系統在某時刻的離散條件 | Draft、Published、Archived |

| 事件（Events） | 觸發狀態轉換的外部輸入 | submit、approve、archive |

| 轉換（Transitions） | 從一個狀態到另一個狀態的路徑 | Draft →(submit)→ Review |

| 動作（Actions） | 轉換時執行的副作用 | 發送通知、寫入日誌 |

2. FSM vs Statechart

| 特性 | 傳統 FSM | Statechart（Harel Statechart） |

|------|---------|------------------------------|

| 層級結構 | 扁平 | 支援父狀態/子狀態（Compound States） |

| 並行區域 | 不支援 | 支援（Parallel States） |

| 歷史狀態 | 不支援 | 支援（淺歷史/深歷史） |

| Guard 條件 | 有限 | 完整支援條件路由 |

| 狀態數量爆炸 | 容易 | 透過層級+並行大幅減少 |

3. 狀態轉換的數學形式


FSM = (Q, Σ, δ, q₀, F)
- Q: 有限狀態集合
- Σ: 輸入字母表（事件集合）
- δ: 轉換函數 Q × Σ → Q
- q₀: 初始狀態
- F: 終止狀態集合

實作模式（六種層級）

模式 1：Enum + Switch（最簡單）


from enum import Enum

class TaskState(Enum):
    PENDING = "pending"
    RUNNING = "running"
    SUCCESS = "success"
    FAILED = "failed"

def handle_event(state, event):
    match (state, event):
        case (TaskState.PENDING, "start"):
            return TaskState.RUNNING
        case (TaskState.RUNNING, "complete"):
            return TaskState.SUCCESS
        case (TaskState.RUNNING, "error"):
            return TaskState.FAILED
        case _:
            return state  # 忽略無效轉換

優點：零依賴、一目了然

缺點：缺乏進入/退出動作、Guard 條件、無法擴展

適用：狀態 ≤5 個的簡單場景

模式 2：轉換表驅動（Table-Driven）


TRANSITION_TABLE = {
    ("pending", "start"): {"next": "running", "action": start_task},
    ("running", "complete"): {"next": "success", "action": finalize},
    ("running", "error"): {"next": "failed", "action": log_error},
    ("failed", "retry"): {"next": "pending", "action": reset},
}

def transition(state, event):
    key = (state, event)
    if key in TRANSITION_TABLE:
        entry = TRANSITION_TABLE[key]
        entry["action"]()
        return entry["next"]
    return state

優點：資料與邏輯分離、可序列化為 JSON/YAML

適用：配置驅動的工作流系統

模式 3：GoF State 設計模式


from abc import ABC, abstractmethod

class State(ABC):
    @abstractmethod
    def handle(self, context, event): ...
    def enter(self, context): pass
    def exit(self, context): pass

class DraftState(State):
    def handle(self, context, event):
        if event == "submit":
            context.transition_to(ReviewState())
    def enter(self, context):
        print("進入草稿狀態")

class Context:
    def __init__(self):
        self._state = DraftState()
        self._state.enter(self)
    
    def transition_to(self, state):
        self._state.exit(self)
        self._state = state
        self._state.enter(self)
    
    def handle(self, event):
        self._state.handle(self, event)

優點：符合 SRP 與 OCP、每個狀態自成一類

缺點：類別數量隨狀態增長

適用：中等複雜度（5-15 個狀態）

模式 4：python-statemachine 宣告式


from statemachine import StateMachine, State

class TaskLifecycle(StateMachine):
    pending = State(initial=True)
    running = State()
    success = State(final=True)
    failed = State()
    
    start = pending.to(running)
    complete = running.to(success)
    error = running.to(failed)
    retry = failed.to(pending)
    
    def on_enter_running(self):
        print("任務開始執行")
    
    def on_exit_running(self):
        print("任務結束")
    
    def before_error(self):
        self.error_count = getattr(self, 'error_count', 0) + 1

優點：宣告式 API、自動生成狀態圖、支援 async

適用：Python 生產系統

模式 5：Pushdown Automata（堆疊式）


class PushdownFSM:
    def __init__(self, initial):
        self._stack = [initial]
    
    @property
    def current(self):
        return self._stack[-1]
    
    def push(self, state):
        self._stack.append(state)
    
    def pop(self):
        if len(self._stack) > 1:
            return self._stack.pop()

適用：需要「暫停當前狀態→處理插入事件→恢復」的場景（如遊戲中角色開火後回到先前動作）

模式 6：Hierarchical FSM（階層式）

子狀態繼承父狀態的行為，未處理的事件向上冒泡。例如「地面」超狀態包含「站立」「蹲下」子狀態，「跳躍」事件在超狀態層級統一處理。

進階主題

Guard 條件（條件式轉換）


class OrderMachine(StateMachine):
    pending = State(initial=True)
    paid = State()
    rejected = State()
    
    process = (
        pending.to(paid, cond="is_payment_valid")
        | pending.to(rejected)  # fallback
    )
    
    def is_payment_valid(self):
        return self.amount > 0 and self.card_verified

並行狀態（Orthogonal Regions）

獨立的狀態機同時運行，避免 n×m 狀態爆炸。例如 Agent 的「執行狀態」（idle/running/done）與「健康狀態」（healthy/degraded/failed）可獨立追蹤。

狀態持久化（State Persistence）


{
  "current_state": "running",
  "state_data": {"start_time": "2026-02-28T10:00:00", "retries": 2},
  "history": ["pending", "running"],
  "version": 1
}

將狀態序列化為 JSON 使 FSM 可跨進程、跨重啟恢復——這是生產系統的關鍵需求。

與本專案的關聯

1. run-fsm.json（執行流程狀態機）

本專案新增的 state/run-fsm.json 正是 FSM 持久化的典型應用——追蹤 Agent 執行的生命週期狀態（init → fetching → processing → assembling → notifying → done），確保中斷恢復與狀態可觀測。

2. Circuit Breaker 三態機

agent_guardian.py 的 CircuitBreaker 已是 FSM 的實例：

  • Closed（正常）→ Open（熔斷）→ Half-Open（試探）
  • 事件：failure_count 超過閾值、cooldown 期滿、試探成功/失敗

3. Todoist 團隊模式 Phase 狀態

run-todoist-agent-team.ps1 的三階段執行流程本質上是一個線性 FSM：

  • Phase 1（Query）→ Phase 2（Execute）→ Phase 3（Assemble）
  • 失敗重試是 Phase 3 → Phase 3 的自環轉換

4. 自動任務頻率限制

config/frequency-limits.yaml 中每個任務的執行計數器可建模為 FSM：

  • available（可執行）→ cooling_down（冷卻中）→ exhausted（達上限）
  • 每日重置事件將所有任務轉回 available

5. 建議優化方向

  • 統一 FSM 基礎類別：用 python-statemachine 取代散落的 if/else 狀態管理
  • 狀態圖自動生成：pip install python-statemachine[diagrams] 可自動產生 .svg 狀態流程圖
  • FSM 事件日誌：在 post_tool_logger.py 加入 fsm-transition 標籤追蹤狀態轉換

AI Agent 工作流中的 FSM 應用

XState + AI Agent（@statelyai/agent）

XState v5 將 Actor Model 與狀態機結合，Stately Expert 框架讓 AI Agent 由狀態機引導決策：

  • Agent 不純粹依賴 LLM 回應，而是在狀態機定義的合法轉換空間內行動
  • 結構化的觀察、回饋與洞見累積使 Agent 可學習改進

LangGraph 狀態機驅動 Agent

LangGraph 用 FSM 描述 Agent 的任務流程，每個節點（狀態）執行特定工具或 LLM 呼叫，邊（轉換）定義條件式路由。這與本專案的 Todoist 路由機制（routing.yaml 的三層路由規則）有異曲同工之妙。

常見陷阱與最佳實踐

陷阱

  1. 狀態爆炸：獨立關注點用並行狀態機而非笛卡爾積
  2. 隱式狀態：布林旗標（isRunning && !isPaused && isConnected）是反模式，應明確定義為離散狀態
  3. God State Machine：單一 FSM 管理所有邏輯 → 拆分為多個專責 FSM
  4. 忽略終止狀態：生產系統必須定義 final state 以確保資源清理

最佳實踐

  1. 狀態命名：使用名詞或形容詞（idle、running），不用動詞（start、stop）
  2. 事件命名：使用動詞或過去式（submit、completed）
  3. 進入/退出動作：初始化放 on_enter、清理放 on_exit，確保不管從哪個狀態轉入都能正確設定
  4. Guard 優先於多重轉換：用條件式守衛而非多條同事件轉換
  5. 持久化：生產 FSM 必須可序列化，支援跨進程恢復
  6. 可視覺化：利用工具自動產生狀態圖，作為活文件
  7. 測試策略：對每條轉換路徑寫測試，確保覆蓋所有合法+非法轉換
