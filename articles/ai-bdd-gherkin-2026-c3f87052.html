研究日期：2026-04-10 · 任務類型：ai_sysdev · 系列：ai-driven-requirements-bdd（foundation）

>

與 KB 既有筆記的關係：承接 Spec Kit／SDD 與 Agent Tool Use 守衛等「文件與工具邊界」主題；本文補齊需求層→可執行 Feature 檔的管線，避免只停在 PRD 文字而無法進 CI。

一、技術概述（約 150 字）

行為驅動開發（BDD）把利害關係人語言（使用者故事、驗收條件）收斂成可執行規格：以 Gherkin 的 Feature／Scenario 結構描述系統行為，並由 Cucumber 生態系（或其他 BDD 執行器）驅動自動化測試。導入 LLM 後，瓶頸從「會不會寫測試」轉為語意對齊（故事是否可測）、語法可解析（Gherkin 合法）、維護成本（步驟是否過細或重複）。實務上宜採「人機協作」：模型產出初稿，開發／QA 做結構與邊界案例審閘，再由 CI 執行 lint 與測試選擇，形成閉環。

二、核心方法與工具

| 層級 | 方法／工具 | 用途 |

|------|------------|------|

| 需求 | INVEST 使用者故事、驗收條件（AC）清單、Example Mapping | 先固定「可測試意圖」再寫 Gherkin |

| 規格語言 | Gherkin（Feature、Scenario、Given／When／Then、Scenario Outline、Examples） | 與 Cucumber 相容之結構化行為描述 |

| 執行 | Cucumber、pytest-bdd、SpecFlow（.NET）等 | 將 .feature 綁定到步驟定義與自動化 |

| 品質閘 | gherkin-lint、formatter、標籤（@smoke） | 靜態檢查與測試子集／平行分片 |

| AI | 指令模板＋範例 few-shot＋輸出 schema 驗證 | 降低幻覺步驟與非法關鍵字 |

三、實際工作流程（step-by-step）

  1. 產品／BA 產出：1 則使用者故事 + 3–7 條驗收條件（含負向與邊界）。
  2. Example Mapping（可選）：以規則／例子／問題便利貼對齊「未知」與測試資料。
  3. LLM 初稿：提示詞要求只輸出 Gherkin，禁止解釋文字；指定關鍵字語言（# language: zh-TW 等）與標籤規範。
  4. 靜態檢查：執行 gherkin-lint／formatter；失敗則把 stderr 貼回模型做單輪修正（避免無限循環）。
  5. 步驟定義對齊：每個 When／Then 對應程式碼；若無法實作則刪或改寫步驟（避免「空洞 Gherkin」）。
  6. CI：lint job → bdd job（依 @smoke 或路徑分片）；與既有 pytest／hooks 並行。
  7. 回寫：將通過的 Feature 合併；registry／研究註冊表記錄主題與日期，利於去重。

四、程式碼／配置範例

最小 Feature（Cucumber 相容骨架）


# language: zh-TW
@smoke @billing
功能: 逾期帳單提醒
  為了降低呆帳
  身為財務人員
  我想要在帳單逾期後收到通知

  場景: 逾期一天觸發通知
    假如 客戶 "A001" 有一筆應於昨天到期的帳單
    當 排程工作 "daily-billing" 執行完成
    那麼 系統應已送出 1 封逾期通知給 "A001"

GitHub Actions 片段（概念）


jobs:
  gherkin-lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npx gherkin-lint features/**/*.feature
  bdd-smoke:
    needs: gherkin-lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test -- --tags @smoke

LLM 系統提示（摘要）


你只輸出合法 Gherkin。必含：功能／場景／假如-當-那麼。
禁止 Markdown 围栏。每步一行。必要時用場景大綱+例子表。

五、效果量化（公開報告方向）

  • 企業導入 BDD 常見報告面向：缺陷左移、回歸覆蓋與溝通回合數下降；具體百分比依團隊與領域差異大，宜以修復前後缺陷密度／發版前驗收輪次做內部 A/B，而非單一通用數字。
  • LLM 輔助寫測試：產業案例多描述「初稿時間縮短」，仍需搭配 lint／CI 才能轉成可靠證據鏈。

六、與 daily-digest-prompt 的應用場景

  • 自動任務結果檔：可把 results/todoist-auto-*.json 的 summary／status 規則改寫成 Gherkin 子集，由 pytest-bdd 或自訂步驟驗證 schema。
  • Hooks／守衛：既有 pre_bash_guard 等可對照「場景：禁止 nul 重導向」類規則，讓非工程利害關係人也能讀 Feature 理解决策。
  • Phase2／多後端：對「成功／partial／failed」狀態轉移寫場景，利於回歸與文件同步。
  • Spec Kit 對齊：AC 與 specs//tasks.md 可一對一連到 Scenario，避免規格與測試漂移。

七、風險與注意事項

  1. 語言關鍵字：中文 Gherkin 需整專案一致；混用英文關鍵字與中文描述時注意 parser 設定。
  2. 過度細分步驟：導致步驟定義爆炸；宜以「業務可讀」為上，技術細節留在底層 helper。
  3. LLM 幻覺：可能發明不存在的 Then；必須紅燈測試與人工審閘。
  4. 維護成本：Scenario Outline 與資料表可降低重複，但需治理測試資料生命週期。
  5. 與 E2E 混淆：BDD 層級應與 UI 脆性平衡；@smoke 宜少而穩。
