二、問題分析：五大結構性挑戰

2.1 標籤爆炸（Tag Explosion）— 最嚴重

問題：1063 個唯一標籤中，765 個（72%）僅被使用一次。這意味著標籤系統失去了「分類」功能，變成了「描述」功能。

根因：

  • 各自動任務（18 個）獨立生成標籤，無統一標籤詞典
  • 研究模板（research-task.md）鼓勵子 Agent 自由添加標籤
  • 近義詞未合併：「AI工具」vs「AI 工具」、「遊戲開發」vs「遊戲設計」、「daily-digest」vs「daily-digest-prompt」
  • 日期標籤（「2026-02-14」等）和版本標籤（「v1.0」等）增加噪音

影響：

  • Hybrid search 的標籤維度匹配精度下降
  • 用戶瀏覽時無法有效按標籤導航
  • kb-curator 的主題分佈分析失真

2.2 主題集中度失衡（Topic Imbalance）

問題：佛學（25.5%）+ AI（19.8%）= 45.3% 佔據近半數內容。其他重要面向（遊戲開發 5.2%、專案優化 3.6%、安全 2.5%）嚴重不足。

根因：

  • frequency-limits.yaml 佛學群組 12 次/日（26.7%），AI/技術群組 17 次/日（37.8%）
  • 研究任務（Todoist）多為研究類型，也集中在 AI/佛學
  • 實際動手的任務（遊戲優化、專案開發）產出的知識庫筆記較少

影響：

  • 知識庫作為「第二大腦」的覆蓋度不夠均衡
  • 某些實作經驗（如 Hook 開發、PowerShell 排程調校）未被記錄

2.3 缺乏內容老化機制（Content Aging）

問題：目前沒有任何機制標記或處理過時內容。技術研究（尤其是 AI 領域）的半衰期約 6-12 個月。

現況：

  • kb-curator Skill 有「過期清理建議」模組，但門檻為 180 天
  • 系統才運行 16 天，此機制尚未觸發
  • 無「內容時效性標記」機制（如：永恆知識 vs 時效性資訊）

預期風險：

  • 3-6 個月後，AI 相關筆記中可能有 30-50% 資訊過時
  • 佛學筆記（永恆知識）與技術筆記（時效性資訊）混在一起，無法區分

2.4 去重機制覆蓋盲區（Dedup Gaps）

問題：現有三層去重（registry + KB search + dedup-policy）僅覆蓋研究模板（research-task.md），不覆蓋以下場景：

  • 子 Agent 模板（skill-task.md、code-task.md、game-task.md）產出的 KB 筆記
  • 每日洞見報告（固定格式但內容不同）
  • 遊戲創建心得（同類遊戲不同版本）

影響：

  • 可能出現「打磚塊 v1.0 創建心得」和「打磚塊 v1.1 優化心得」高度重疊
  • 每日洞見報告的結構性重複（框架相同，數據不同）

2.5 缺乏品質回饋迴路（Quality Feedback Loop）

問題：筆記匯入後沒有「被使用」的追蹤機制。無法知道哪些筆記被查詢最多、哪些從未被引用。

影響：

  • 無法基於使用頻率優化 embedding 權重
  • 無法識別「高價值筆記」vs「低價值筆記」
  • 自動任務無法根據 KB 的實際利用率調整研究方向

三、優化方案（按優先級排序）

方案 A：標籤正規化系統（Priority: P0，預期收益最高）

目標：將 1063 個標籤收斂至 150-200 個有意義的類別。

實施步驟：

  1. 建立標籤詞典（config/tag-dictionary.yaml）：

```yaml

# 標準化映射：近義詞 → 正式名稱

aliases:

"AI工具": "AI 工具"

"daily-digest": "daily-digest-prompt"

"遊戲設計": "遊戲開發"

"Game Loop": "遊戲迴圈"

# 層級化標籤體系

hierarchy:

佛學:

- 楞嚴經

- 教觀綱宗

- 法華經

- 淨土宗

AI:

- LLM

- Agent

- RAG

- 微調

# 禁用標籤（純噪音）

suppress:

- "2026" # 日期標籤無分類意義

- "v1.0" # 版本標籤

```

  1. 修改研究模板：匯入前自動比對標籤詞典，將近義詞合併
  1. 一次性批量正規化：用 kb-curator 的去重掃描 + 標籤映射，對現有 364 筆做清洗

預期效益：標籤從 1063 → 200 以下，hybrid search 精度提升 20-30%

方案 B：內容時效性分級（Priority: P1）

目標：區分永恆知識與時效性資訊，為未來清理提供依據。

實施步驟：

  1. 在匯入時添加 content_class 標籤：

- 永恆知識：佛學、哲學、方法論、設計模式

- 技術參考：框架指南、API 文件、最佳實踐（半衰期 12-24 月）

- 技術快訊：趨勢報告、版本更新、GitHub 熱門（半衰期 3-6 月）

- 專案紀錄：優化報告、創建心得、洞見分析（專案存續期間有效）

  1. 修改 kb-curator 的過期清理規則：依 content_class 設定不同 TTL

- 永恆知識：永不過期

- 技術參考：18 個月

- 技術快訊：6 個月

- 專案紀錄：12 個月

預期效益：6 個月後可自動標記 30-50 筆過時內容，避免 RAG 回傳過時資訊

方案 C：主題均衡控制器（Priority: P1）

目標：調整自動任務頻率，確保知識庫覆蓋均衡。

實施步驟：

  1. 在 frequency-limits.yaml 加入 KB 回饋：

```yaml

rebalance_policy:

check_interval: weekly

target_distribution:

佛學: 20-30%

AI/技術: 20-30%

遊戲: 10-15%

專案: 10-15%

方法論/思維: 10-15%

其他: 10-20%

action: "偏離 target > 10% 時，降低過多類型的 daily_limit 並提高不足類型"

```

  1. system-insight 自動任務：每週一次 KB 分佈檢查，產出再平衡建議

預期效益：3 個月後主題覆蓋更均衡，佛學佔比降至 20-25%

方案 D：擴展去重覆蓋範圍（Priority: P2）

目標：將去重機制從研究模板擴展到所有 KB 寫入路徑。

實施步驟：

  1. 建立統一匯入閘門（hooks/pre_kb_import.py 或在 post_tool_logger 中攔截）：

- 攔截所有 curl POST localhost:3000/api/import 呼叫

- 自動做 hybrid search 去重（score > 0.85 → 警告）

- 自動套用標籤詞典正規化

  1. game-task.md、code-task.md 模板加入去重步驟

預期效益：消除跨任務類型的內容重複

方案 E：使用頻率追蹤（Priority: P3，中長期）

目標：追蹤哪些筆記被 hybrid search 命中最多，建立品質回饋迴路。

實施步驟：

  1. 在 post_tool_logger.py 中記錄 KB 查詢命中的 noteId
  2. 每週由 system-insight 彙總「熱門筆記 Top 20」和「零命中筆記」
  3. 零命中筆記 > 90 天 → 標記為候選清理

預期效益：基於數據驅動的 KB 品質優化

六、風險評估

  1. 標籤正規化可能破壞現有搜尋：建議分批執行，先處理 Top 50 高頻標籤
  2. 過度限制自動標籤可能降低描述力：保留「自由標籤」空間，僅強制正規化核心標籤
  3. 主題均衡可能壓縮佛學研究深度：設定下限（不低於 15%），保護系列研究的連貫性
