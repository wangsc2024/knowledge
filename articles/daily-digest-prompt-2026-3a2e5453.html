五、技術洞察

1. Phase 1 失敗模式的啟示

發現：Agent 執行完成 ≠ 產出物正確寫入

啟示：

  • LLM Agent 可能產生「對話性回應」而非直接執行
  • 必須明確禁止對話性回應，強制工具呼叫
  • Plan 檔案寫入應加入時間戳驗證
  • Retry 機制需要更智能的失敗判斷

最佳實踐：


# Prompt 設計模式
1. 明確禁止開場白和確認對話
2. 第一句話就說明「立即執行步驟 X」
3. 最後必須以 Write 工具寫入結果檔案
4. 加入時間戳驗證機制

2. 快取管理的平衡

發現：過期快取會干擾系統判斷

啟示：

  • 快取不只是效能優化，也關係到系統邏輯正確性
  • 過期快取應主動清理，而非等待覆寫
  • TTL 設定需考慮資料更新頻率與系統負載平衡

最佳實踐：

  • 定期清理（每日 00:01）
  • 快取檔案加入版本號
  • 讀取快取前先驗證時間戳

3. 系統健康度的多維度評估

發現：單一指標（如成功率）無法反映真實健康度

啟示：

  • 需要綜合評估：成功率 + 審查評分 + 技術債 + 快取效益
  • 56.3% 成功率看似低，但系統審查 88.99/A 說明架構健康
  • 失敗可能是暫時性問題（如 Phase 1），非系統性缺陷

最佳實踐：

  • 建立多維度儀表板
  • 趨勢比絕對值更重要
  • 異常值需要深入分析（如本次 Phase 1 失敗）
