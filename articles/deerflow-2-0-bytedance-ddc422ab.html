一、專案概述

DeerFlow（Deep Exploration and Efficient Research Flow）是由字節跳動（ByteDance）開發的開源 SuperAgent 運行時框架，定位為「研究、編碼、創造的自主 Agent 基礎設施」。截至 2026 年 3 月已累積 28,648 Stars，本月增長 +8,378（GitHub Trending 月榜前五），採用 MIT 授權。

核心定位

| 維度 | 說明 |

|------|------|

| 定位 | SuperAgent Harness（運行時基礎設施，非高層抽象框架） |

| 版本 | 2.0（2026-02-28 發佈，完全重寫，與 v1 無共用程式碼） |

| 技術棧 | Python（LangGraph + LangChain）+ Next.js 前端 |

| 授權 | MIT |

| 模型支援 | 模型無關（OpenAI 相容 API），支援 GPT-4、Claude、Gemini、DeepSeek、Ollama 本地模型 |

| 核心能力 | 子 Agent 編排、沙箱執行、持久記憶、可擴展 Skill、多頻道整合 |

與競品的關鍵區別

DeerFlow 自我定位為基礎設施層而非應用層：

| 框架 | 定位 | 擅長 |

|------|------|------|

| LangChain | 對話鏈抽象 | 聊天互動、RAG |

| CrewAI | 角色協作 | 角色扮演式多 Agent 協作 |

| AutoGen | 對話式多 Agent | Agent 間對話編排 |

| DeerFlow | 運行時基礎設施 | 長時間自主任務、沙箱執行、持久狀態 |

二、系統架構

2.1 四層服務拓撲

DeerFlow 以 Nginx（port 2026）為反向代理，背後運行四個網路服務：

| 服務 | 埠號 | 技術 | 職責 |

|------|------|------|------|

| Frontend | :3000 | Next.js + React | 使用者介面 |

| Agent Runtime | :2024 | LangGraph Server | Agent 執行引擎（make_lead_agent 工廠） |

| Gateway | :8001 | FastAPI | REST API（配置、檔案、模型管理） |

| Provisioner | :8002 | 選用 | Kubernetes 沙箱分配 |

2.2 11 層中介軟體鏈（Middleware Chain）

make_lead_agent 工廠建立的 Lead Agent 透過固定順序的 11 層中介軟體處理每個請求：

| 順序 | 名稱 | 職責 |

|------|------|------|

| 1 | ThreadDataMiddleware | 每執行緒建立獨立目錄 |

| 2 | UploadsMiddleware | 注入使用者上傳的檔案 |

| 3 | SandboxMiddleware | 取得/釋放沙箱資源 |

| 4 | DanglingToolCallMiddleware | 補充遺漏的工具呼叫結果（佔位訊息） |

| 5 | SummarizationMiddleware | Token 閾值管理，超限時壓縮上下文 |

| 6 | TodoListMiddleware | 規劃支援（任務清單追蹤） |

| 7 | TitleMiddleware | 自動生成對話標題 |

| 8 | MemoryMiddleware | 異步記憶抽取排隊 |

| 9 | ViewImageMiddleware | 視覺模型支援 |

| 10 | SubagentLimitMiddleware | 強制最大 3 個並行子 Agent |

| 11 | ClarificationMiddleware | 中斷處理（向使用者請求澄清） |

設計洞察：每層中介軟體提供四個鉤點（before_agent、before_model、after_model、wrap_model_call），與 ASP.NET Core 的 Middleware Pipeline 或 Express.js 中介軟體模式同構。

2.3 工具系統（六來源聚合）

get_available_tools() 從六個來源動態載入工具：

| 來源 | 工具範例 | 載入方式 |

|------|---------|----------|

| Sandbox Tools | bash, ls, read_file, write_file, str_replace | 沙箱核心 |

| Built-in Tools | present_files, ask_clarification, view_image | 內建 |

| MCP Tools | 任意 MCP 伺服器提供 | 延遲載入（MultiServerMCPClient） |

| Config-defined | config.yaml 定義 | 反射載入（resolve_variable()） |

| Vision Tools | 條件載入（依模型能力） | 條件性 |

| Task Tool | 子 Agent 委派（task()） | 核心 |

反射系統：resolve_variable() 支援動態模組載入，缺少 provider 時提供可操作的錯誤訊息（如「請執行 uv add langchain-google-genai 安裝」）。

三、子 Agent 編排機制

3.1 雙線程池架構

Lead Agent 透過 task() 工具委派子 Agent，採用雙線程池設計：

| 線程池 | Worker 數 | 職責 |

|--------|-----------|------|

| Scheduler Pool | 3 | 任務排程與分配 |

| Execution Pool | 3 | 實際任務執行 |

| 約束 | 值 | 說明 |

|------|-----|------|

| 最大並行數 | 3/回應 | SubagentLimitMiddleware 強制 |

| 單任務超時 | 15 分鐘 | 防止無限執行 |

| 內建類型 | general-purpose, bash | 全工具集 vs 純 Shell |

3.2 與 Daily-Digest-Prompt 團隊模式對比

| 維度 | DeerFlow 2.0 | Daily-Digest-Prompt |

|------|-------------|---------------------|

| 並行策略 | 固定上限（max 3） | 動態（依任務數決定） |

| 排程模式 | 雙線程池（排程+執行） | Phase 分階段（查詢→N並行→組裝） |

| 子 Agent 隔離 | Docker 容器 / K8s Pod | 獨立 Claude 實例（claude -p） |

| 結果交接 | 結構化回傳至 Lead Agent | results/*.json 檔案交接 |

| 超時控制 | 固定 15 分鐘 | 動態按任務類型（config/timeouts.yaml） |

四、記憶系統深度分析

4.1 異步 LLM 記憶抽取管線

DeerFlow 的記憶系統是其最具創新性的元件之一：


對話訊息 → MemoryMiddleware（過濾：僅保留 user + 最終 AI 回應）
         → MemoryQueue（30 秒去抖動）
         → MemoryUpdater（LLM 抽取事實 + 信心分數）
         → memory.json（原子寫入：暫存檔 + 重命名）
         → 下次呼叫注入 Top 15 事實至 System Prompt

4.2 關鍵設計決策

| 決策 | 選擇 | 理由 |

|------|------|------|

| 去抖動時間 | 30 秒 | 平衡即時性與 LLM API 成本 |

| 事實注入數 | Top 15 | 控制 System Prompt 長度 |

| 寫入策略 | 原子寫入（temp → rename） | 防止並行讀寫損壞 |

| 過濾策略 | 僅 user + 最終回應 | 排除中間推理雜訊 |

4.3 與 Daily-Digest-Prompt 記憶系統對比

| 維度 | DeerFlow | Daily-Digest-Prompt |

|------|----------|---------------------|

| 抽取方式 | LLM 自動抽取（智能但昂貴） | 結構化 JSON 直接寫入（精確但手動） |

| 儲存格式 | 事實+信心分數（memory.json） | 結構化欄位（context/digest-memory.json） |

| 注入策略 | Top 15 事實→System Prompt | 全量讀取→Agent 按需使用 |

| 去抖動 | 30 秒 | 無（即時寫入） |

| 跨 Session | 是（永久） | 是（跨次記憶） |

五、沙箱與虛擬路徑系統

5.1 虛擬路徑翻譯機制

Agent 看到的是虛擬路徑，系統透明重寫為實體路徑：

| 虛擬路徑 | 實體路徑 | 用途 |

|---------|---------|------|

| /mnt/user-data/workspace/ | backend/.deer-flow/threads/{thread_id}/user-data/workspace/ | 工作區 |

| /mnt/user-data/uploads/ | ...uploads/ | 使用者上傳 |

| /mnt/user-data/outputs/ | ...outputs/ | 輸出產物 |

| /mnt/skills/ | 技能模組路徑 | Skill 載入 |

replace_virtual_path() 函數在沙箱工具執行時處理路徑轉換，Agent 無需感知底層儲存結構。

5.2 三種執行模式

| 模式 | 隔離度 | 適用場景 |

|------|--------|----------|

| Local | 低（直接檔案系統） | 開發測試 |

| Docker/AIO | 中（容器隔離） | 生產部署 |

| Kubernetes | 高（Per-Thread Pod） | 企業級多租戶 |

六、配置系統設計

6.1 雙配置模式

DeerFlow 採用雙配置檔案 + 優先級解析的設計：

| 配置檔 | 格式 | 內容 | 更新頻率 |

|--------|------|------|----------|

| config.yaml | YAML | 模型、工具、沙箱設定、Skill 路徑 | 低（手動） |

| extensions_config.json | JSON | MCP 伺服器、Skill 啟停狀態 | 高（Gateway API 動態寫入） |

Singleton + mtime 失效：配置載入後快取為 Singleton，透過檔案修改時間（mtime）偵測變更並自動重載 — 零重啟熱更新。

解析優先級：

  1. 明確路徑參數
  2. 環境變數（DEER_FLOW_CONFIG_PATH）
  3. 當前目錄
  4. 父目錄（推薦）

七、對 Daily-Digest-Prompt 系統的七大借鑑啟示

啟示 1：中介軟體鏈模式可強化 Hooks 架構

DeerFlow 做法：11 層中介軟體，每層四鉤點（before_agent/before_model/after_model/wrap_model_call）

我們現狀：6 個 Hooks（PreToolUse 3 + PostToolUse 2 + Stop 1），作用於工具層級

借鑑方向：引入 Agent 層級的 Middleware（如 SummarizationMiddleware 的 Token 閾值管理），現有 Hooks 可視為工具層中介軟體

啟示 2：LLM 記憶抽取 + 信心分數

DeerFlow 做法：用 LLM 從對話中抽取結構化事實 + 信心分數，異步去抖動

我們現狀：digest-memory.json 手動結構化寫入

借鑑方向：對 Todoist 任務執行後的經驗教訓，可用 LLM 自動抽取並標註信心分數，提升跨次學習效率

啟示 3：虛擬路徑翻譯實現沙箱透明化

DeerFlow 做法：Agent 只看到 /mnt/ 虛擬路徑，底層透明重寫

我們現狀：Agent 直接操作實體路徑，透過 pre_write_guard 做路徑遍歷攔截

借鑑方向：若未來引入多租戶或更強隔離，可參考虛擬路徑模式。當前 Hook 攔截方案已足夠

啟示 4：配置 Singleton + mtime 熱重載

DeerFlow 做法：配置 Singleton 快取 + mtime 偵測變更 → 零重啟更新

我們現狀：YAML 配置每次讀取，無快取機制

借鑑方向：hook_utils.py 的 load_yaml_rules() 可引入 mtime 快取，避免每次 Hook 觸發都重讀 YAML（高頻場景下有效能收益）

啟示 5：上下文壓縮（SummarizationMiddleware）

DeerFlow 做法：Token 超過閾值時自動壓縮上下文，中間結果卸載至檔案系統

我們現狀：依賴 Claude Code 內建的自動壓縮機制

借鑑方向：長時間 Todoist 團隊任務可主動壓縮子 Agent 結果（目前已透過 results/*.json 部分實現）

啟示 6：雙線程池分離排程與執行

DeerFlow 做法：Scheduler Pool（3）+ Execution Pool（3）分離關注點

我們現狀：Phase 式管線（查詢→並行→組裝），排程與執行混合

借鑑方向：若 run-todoist-agent-team.ps1 的並行數從目前最多 ~5 提高，可考慮引入獨立排程器管理任務佇列

啟示 7：多頻道整合模式

DeerFlow 做法：Telegram（Long-polling）、Slack（Socket Mode）、飛書（WebSocket），無需公網 IP

我們現狀：ntfy.sh 單向推播

借鑑方向：若未來需要雙向互動（如 Telegram Bot 觸發 Agent 任務），DeerFlow 的頻道抽象層是良好參考
