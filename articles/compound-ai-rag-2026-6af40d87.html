一、核心理論與系統架構

1.1 METIS: Fast Quality-Aware RAG Systems (Microsoft SOSP 2025)

論文來源: Microsoft Research

發表會議: ACM SOSP, October 2025

#### 核心創新

METIS 是首個聯合調度查詢與調整 RAG 配置的系統，解決 RAG 系統的根本挑戰：使用更多外部知識會導致更高的回應延遲。

#### 技術機制

  1. Query Profiling

使用獨立的 LLM 估計查詢的 profile，包括：

- 需要多少資訊片段來回答查詢

- 這些片段之間是否需要聯合推理

  1. Configuration Space Pruning

將大量配置空間修剪為較小但有希望的子集，包括：

- 檢索的文本塊數量 (topK)

- 合成方法 (synthesis method)

  1. Joint Scheduling

同時調度查詢並為每個查詢調整配置，而非對所有請求應用統一設定。

#### 效能指標

在四個流行的 RAG 問答數據集上測試：

  • 延遲降低: 1.64 − 2.54×
  • 品質: 不犧牲生成品質
  • 改進: 相較於現有最先進的 RAG 優化方案

#### 關鍵洞察

"Unlike prior work that typically focuses on either latency reduction or quality maximization independently, METIS jointly schedules queries and adapts configurations, representing a more holistic approach to the quality-speed tradeoff inherent in RAG systems."

METIS 代表從單一指標優化轉向系統性權衡管理的範式轉移。

1.2 OPTIMAS: Compound AI Systems with Globally Aligned Local Rewards

論文來源: arXiv:2507.03041

發表時間: July 2025 (latest version February 2026)

#### 核心挑戰

Compound AI 系統整合多個組件（LLMs、專業工具、傳統 ML 模型），其優化挑戰在於：

  • 非可微結構: 組件間連接不可微分
  • 異質配置類型: 文本 vs. 數值、連續 vs. 離散
  • 全局獎勵: 需要基於系統輸入和使用者定義的全局獎勵函數找到最優配置策略

#### Local Reward Functions (LRF) 機制

核心思想是為每個組件維護一個 Local Reward Function (LRF)，每個 LRF 滿足局部-全局對齊屬性（local-global alignment property）：

"Each component's local reward correlates with the global system performance."

在每次迭代中，OPTIMAS：

  1. 高效適應 LRF 以維持此屬性
  2. 同時最大化每個組件的局部獎勵
  3. 使用指定的優化方法獨立更新異質配置
  4. 確保局部改進持續導致效能提升

#### 效能表現

  • 平均改善: 11.92% (相較於強基準)
  • 迭代效率: 在少量迭代內達成驗證集 41.7% 的全局獎勵改善
  • 通用性: 跨 5 個真實世界 Compound AI 系統

#### 評估任務

  1. Amazon 產品推薦: 下一個物品預測
  2. PubMedQA: 生物醫學文獻的臨床問答
  3. STaRK: 半結構化知識庫檢索
  4. HotpotQA: 多跳推理任務
  5. BigCodeBench: 從自然語言生成驗證代碼

1.3 SmartRAG: Joint Learning of RAG-Related Tasks (ICLR 2025)

論文來源: ICLR 2025 Proceedings

#### 核心理念

"A system like RAG that incorporates multiple modules should be jointly optimized to achieve optimal performance."

SmartRAG 強調聯合優化的重要性：當聯合優化時，所有模組可以感知其他模組的工作方式，從而找到作為完整系統協同工作的最佳方式。

#### 實證結果

聯合優化的 SmartRAG 可以實現比分別優化的對手更好的效能。

1.4 Multi-Objective Hyperparameter Optimization (arXiv 2502.18635)

論文來源: arXiv:2502.18635

標題: "Faster, Cheaper, Better: Multi-Objective Hyperparameter Optimization for LLM and RAG Systems"

#### 研究貢獻

這是首個針對整個 LLM 和 RAG 系統的多目標參數優化方法，優化目標包括：

  • Cost (成本)
  • Latency (延遲)
  • Safety (安全性)
  • Alignment (對齊)

#### qLogNEHVI 演算法

qLogNEHVI 擴展了 qLogEHVI，使用新的獲取函數（acquisition function），該函數在嘈雜目標評估（noisy objective evaluations）的設定中，理論上和實證上都優於基準方法。

"The probabilistic nature of LLMs can cause noisy objective function evaluations."

#### 關鍵發現

  1. Pareto 前緣: qLogNEHVI BayesOpt 獲得優越的 Pareto 前緣，解決方案集中在高安全性、高對齊、低成本和低延遲
  2. 權衡: 提高安全性可能以增加成本和延遲為代價
  3. 任務特定: 一個任務的最佳配置可能不適合另一個任務，建議超參數調整應為任務特定

#### 實作工具

使用 Ax 和 BoTorch 運行和管理實驗，採用 Bayesian Optimization 與 Gaussian Process 回歸來高效導航 RAG 系統的複雜參數空間。

二、生產環境路由策略

2.1 LiteLLM Router 最佳實踐

文件來源: LiteLLM Docs - Routing

#### 路由策略選擇

  • 生產推薦: simple-shuffle (預設)

- 最佳性能，適用於高流量場景

- Python-based router 增加 3-5ms 開銷

  • 不推薦生產: usage-based routing

- 由於 Redis 操作（tracking TPM/RPM）造成顯著延遲

- 跨多個部署的 async redis calls (redis.incr, redis.mget)

#### 可靠性邏輯

基本可靠性邏輯包括：

  • Cooldowns (冷卻期)
  • Fallbacks (後備)
  • Timeouts (超時)
  • Retries (重試): fixed + exponential backoff

---

2.2 Semantic Router 實作

專案來源: GitHub - aurelio-labs/semantic-router

#### 核心概念

Semantic Router 是一個超快速決策層，用於 LLMs 和 agents：

  • 使用語義向量空間進行路由決策
  • 而非等待緩慢的 LLM 生成來做工具使用決策

#### 生產實踐建議

  1. Semantic Router 套件: 提供良好起點，但需注意生產環境限制
  2. Fine-tuning 專用模型: 最有效的方法是微調自己的專業分類模型
  3. Logistic Regression Layer: 在預訓練 embedding 模型上添加 logistic regression 層，利用模型的預學習語言理解，同時針對特定領域優化

---

2.3 Tiered Routing 策略

來源: Building a RAG Router in 2025 (Medium)

#### 三層路由架構

  1. Tier 1 - Keyword Matching (便宜、快速)

- 捕捉明顯案例（如含 "translate" 的查詢 → 翻譯模型）

  1. Tier 2 - Semantic Classification (中等成本)

- 處理模糊查詢

  1. Tier 3 - LLM-Assisted Routing (高成本、高準確度)

- 最複雜的決策

#### 成本效益

採用動態路由的公司在 RAG 設置中報告成本降低 27-55%，透過根據複雜度將查詢導向適當的模型。

三、RAG 配置優化實務

3.1 Chunk Size 最佳化

來源: LlamaIndex - Chunk Size Guide

#### 基準建議

  • 起始點: 512 tokens chunk size + 50-100 tokens overlap
  • 通用範圍: 200-500 tokens，10-20% overlap
  • 生產優化: 1,800-character chunks + top-50 retrieval 顯著提升效能

#### Chunk Size 與 TopK 的關係

  • 小 chunk (128): similarity_top_k=2 時可能遺漏重要資訊
  • 中 chunk (512): 可能在 top chunks 中涵蓋所有必要資訊

---

3.2 Retrieval 參數調整

#### Hybrid Search

結合向量相似度與關鍵字搜尋可顯著改善檢索品質：

  • Vector search: 捕捉語義意義
  • BM25: 捕捉精確詞彙匹配

#### Reranking

添加 reranker 使 top-k 結果更強：

  1. 初步檢索 (如 top-50)
  2. Rerank 到最終 top-k (如 top-5)
  3. 建立基準並測量改善

---

3.3 Relevance vs. Quality Tradeoff

來源: Relevance Isn't All You Need (arXiv 2504.07104)

#### 關鍵洞察

"In standard RAG pipelines, maximizing for context relevance alone can degrade downstream response quality."

僅優化相關性的方法可能提升 context relevance，但降低生成答案的整體品質。

#### Multi-Criteria Reranking

測量除相關性之外的品質，透過動態兩輪策略（dynamic two-turn strategy）適應個別查詢，推斷查詢特定準則。

五、Daily Digest 專案應用建議

5.1 P0 級優化（立即實施）

#### P0-A: Groq 預分類時輸出建議 KB topK

實施位置: run-todoist-agent-team.ps1 Phase 1a

機制:


# 在 Phase 1 Groq 預分類時同步輸出
query_profile:
  complexity: simple | moderate | complex
  kb_topK_suggestion: 5 | 10 | 15
  reasoning_type: single_hop | multi_hop | joint_reasoning

預期效益:

  • 降低重試率 15-25%
  • 壓縮 auto-task 平均延遲 10-20%
  • 避免因 RAG 深度不足觸發重試

參考: METIS query profiling mechanism

---

#### P0-B: 建立 Query Profile Filter

實施位置: config/kb-query-router.yaml

配置範例:


query_profiles:
  simple:
    topK: 5
    max_depth: 1
    rerank: false
    synthesis_method: "concat"

  moderate:
    topK: 10
    max_depth: 2
    rerank: true
    synthesis_method: "weighted"

  complex:
    topK: 15
    max_depth: 3
    rerank: true
    synthesis_method: "hierarchical"

過濾不良組合:

  • Simple query + topK=15 → 浪費 token
  • Complex query + topK=5 → 深度不足

5.2 P1 級優化（短期實施）

#### P1-A: Tiered Routing for Auto-Tasks

實施位置: 新建 tools/task_router.py

三層路由:


# Tier 1: Keyword Rules (O(1))
if "研究" in task_type or "深度思維" in task_type:
    return "research-task.md"

# Tier 2: Semantic Classification (Groq classify)
semantic_score = groq_classify(task_description)
if semantic_score["research"] > 0.7:
    return "research-task.md"

# Tier 3: LLM-Assisted (GPT-4 fallback)
return llm_route(task_description)

預期效益:

  • 路由決策時間降低 40-60%
  • 成本降低 20-30% (減少 Tier 3 調用)

#### P1-B: Local Reward Functions for Skill Chain

實施位置: context/skill-performance.json

機制 (參考 OPTIMAS):

九、結論

Compound AI 系統的路由與 RAG 配置聯合優化代表 2025-2026 年 AI 工程的核心趨勢。METIS、OPTIMAS、qLogNEHVI 等前沿研究提供了理論基礎與可量化的效能提升證據，而生產環境的 Tiered Routing、Hybrid Search、Multi-Objective Optimization 則提供了可立即實施的最佳實踐。

對於 Daily Digest 專案，建議採用三階段漸進式實施：

  1. Phase 1 (P0): Query Profile-driven KB topK (METIS 啟發)
  2. Phase 2 (P1): Tiered Routing + Local Reward Functions (OPTIMAS 啟發)
  3. Phase 3 (P2): Multi-Objective Pareto Tracking (qLogNEHVI 啟發)

預期整體效能提升：

  • 延遲降低: 10-25%
  • 成本降低: 20-35%
  • 品質提升: 5-15%
  • 重試率降低: 15-25%

此研究為系統優化提供了清晰的技術路徑與可量化的改進目標。
