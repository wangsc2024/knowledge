n8n — 開源 AI 工作流自動化平台深度洞察（182.8K Stars）

執行摘要

n8n 是 2026 年最受矚目的開源工作流自動化平台之一，在 GitHub 累積 182.8K Stars、56.5K Forks。以 TypeScript 開發，採用 Fair-code 授權模式，結合視覺化節點編排與程式碼彈性，提供 400+ 原生整合。2025 年底發布 2.0 版後，其 AI Agent 原生能力與雙向 MCP 整合成為 Agent 生態系統的重要基礎設施。

---

一、專案核心架構

1.1 JSON 原生資料模型

n8n 的所有節點間資料傳遞皆以結構化 JSON 格式進行，每一步驟都可檢視（inspectable）。這與本專案的 JSON 結果檔（results/*.json）設計理念高度吻合——資料透明化是可觀測性的基礎。

1.2 節點式工作流引擎

  • 觸發器多樣性：Webhook 即時觸發、Cron 排程、輪詢觸發、手動觸發、事件觸發
  • 子工作流：可複用的自動化元件，可從其他工作流呼叫（類似本專案的 sub-agent 委派）
  • 錯誤處理：Try-catch 節點、可配置重試邏輯、失敗時觸發錯誤工作流、完整執行歷史記錄

1.3 佇列模式（Queue Mode）擴展架構

  • 單實例標準模式：最多 100 個並行工作流執行
  • 佇列模式（3 workers on C5.large）：約 72 req/s、延遲 < 3 秒
  • 這是典型的 Worker Pool 架構，與本專案的 Agent Pool（tools/agent_pool/coordinator.py）設計異曲同工

二、AI Agent 原生能力

2.1 內建 AI 節點

| 節點類型 | 功能 |

|---------|------|

| AI Agent Node | 中央控制器，管理 LLM 模型與記憶系統 |

| LLM Chain Node | 鏈式 LLM 呼叫，支援多模型串接 |

| Vector Store Node | 向量資料庫整合（RAG 基礎） |

| Embeddings Node | 嵌入向量生成 |

| Memory Node | 對話式記憶管理（上下文感知） |

| Document Loader | PDF、網頁等文件擷取 |

2.2 支援模型生態

原生連接 OpenAI、Anthropic、Google、Hugging Face，並支援 Ollama 本地模型。

2.3 Orchestrator-Agent-Worker 模式

2026 年最有效的新興模式：

  1. Orchestrator（n8n 工作流）：啟動流程、收集上下文
  2. Agent（AI Agent Node）：自主推理、選擇工具、迭代執行
  3. Worker（外部 MCP 服務或子工作流）：專責執行特定任務

這三層架構與本專案的 Phase 0（PowerShell 預計算）→ Phase 1（Agent 執行）→ Phase 2（子 Agent 並行）結構高度對應。

---

三、雙向 MCP Hub（核心創新）

3.1 MCP Client（消費端）

  • MCP Client Tool Node 可連接外部 MCP 伺服器
  • AI Agent 在推理過程中動態發現並調用 MCP 工具
  • 支援選擇性工具暴露（Tool Filtering），避免 Tool Sprawl

3.2 MCP Server（供應端）

  • MCP Server Trigger Node 將 n8n 工作流暴露為 MCP 工具
  • 外部 AI Agent（如 Claude Code、Cursor）可直接呼叫 n8n 工作流
  • 支援 Streamable HTTP 傳輸協定

3.3 設計哲學：「內部複雜、外部簡單」

Rather than exposing raw system access, n8n enables building complex capabilities internally and exposing them as simple tools externally.

這與本專案的「Prompt 是薄層調度器」原則完全一致——將複雜邏輯封裝在 Skill/工具內，對外只暴露簡潔介面。

五、生產部署經驗與警示

n8n 文件明確警告的反模式，對本專案同樣適用：

  1. Tool Sprawl（工具膨脹）：Agent 在聚焦的工具集上表現更好（品質優於數量，改善 Token 效率）
  2. 延遲預算：多工具鏈的網路往返延遲會複合累積
  3. Token 成本：迭代推理會放大使用費用
  4. 可觀測性缺口：多系統 Agent 工作流需要完整記錄所有工具調用

七、關鍵洞見

  1. 工作流即 Agent 骨架：n8n 證明了「工作流引擎 + AI Agent 節點」的組合比純 Agent 框架更適合生產環境——確定性流程（觸發、分派、錯誤處理）不應委託給 LLM，LLM 只負責推理決策。
  1. MCP 是新的 API Gateway：雙向 MCP 使 n8n 從「自動化工具」升級為「Agent 基礎設施」。這預示著未來的 Agent 生態系統將以 MCP 為中間層，取代傳統 REST API 整合。
  1. Fair-code 商業模式創新：n8n 的 Sustainable Use License 在開源與商業間找到平衡——核心開源吸引社群，企業功能付費維持營運。這對開源 Agent 工具的長期永續發展具有參考價值。
