二、P0-A：npm 供應鏈安全風險

問題描述

bot/package.json 中 12 個依賴有 11 個使用 ^（caret）版本範圍：


@line/bot-sdk: "^10.6.0"     → 允許 10.6.0 ~ 10.999.999
express: "^4.18.2"            → 允許 4.18.2 ~ 4.999.999
helmet: "^8.0.0"              → 允許 8.0.0 ~ 8.999.999
express-rate-limit: "^7.5.0"  → 允許 7.5.0 ~ 7.999.999
groq-sdk: "^0.20.0"          → 0.x 版本 ^ 等同 ~，但語義不明確

僅 pino: "10.3.1" 使用嚴格版本鎖定。

風險分析

  • CISA 2025-09 官方警告：npm 生態系廣泛供應鏈攻擊事件，建議組織固定版本並輪換開發者憑證
  • OpenSSF 建議：檢入 lockfile、CI 中使用 npm ci（而非 npm install）
  • groq-sdk: "^0.20.0" 屬 0.x 版本，API 穩定性無保證，次要版本更新可能包含破壞性變更

修復方案

  1. 立即：將安全關鍵依賴改為精確版本（helmet、express-rate-limit、@line/bot-sdk）
  2. 短期：全部依賴改為 ~（tilde，僅允許 patch 更新）或精確版本
  3. 長期：導入 npm audit 到 pre-push hook，設定 Dependabot/Renovate 自動更新
  4. 對比：Python 側（pyproject.toml）已做到 5/7 嚴格 == 鎖定，可作為範本

三、P0-B：io-guard-state 孤兒檔無限膨脹

問題描述

state/ 目錄累積 138 個 io-guard-state-{session_id}.json 檔案，無任何清理機制。

根因追蹤：

  • 產生者：hooks/post_tool_logger.py（816-852 行）每個 session 建立一個檔案追蹤 I/O 用量
  • 現有清理：on_stop_alert.py（349-374 行）清理 loop-state-.json 和 stop-alert-.json（7 天），但遺漏 io-guard-state
  • cleanup-state-files.ps1：清理 loop-state（48h）、autonomous-recovery-queue（50 條）、context-usage（7 天），同樣遺漏 io-guard-state

影響評估

  • 目前：138 檔 × ~280 bytes ≈ 40 KB（尚小）
  • 趨勢：每日 ~44 次排程執行，每月新增 ~1,320 個檔案
  • 6 個月後：~8,000 個檔案，目錄列舉效能下降

修復方案


# on_stop_alert.py 新增清理規則（349-374 行區段）
io_guard_pattern = r"^io-guard-state-[0-9a-f]+\.json$"
# 保留期：7 天（與 loop-state 一致）

同步在 cleanup-state-files.ps1 新增：


Get-ChildItem state/io-guard-state-*.json |
  Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-7) } |
  Remove-Item -Force

四、P1-A：測試覆蓋率盲區（27 個未測試工具）

現況分析

| 類別 | 已測試 | 未測試 | 覆蓋率 |

|------|--------|--------|--------|

| hooks/ | 12/12 | 0 | 100% |

| skills/ | 4/29 | 25 | 14% |

| tools/ | 46/73 | 27 | 63% |

未測試工具分類（27 個）

| 類別 | 工具 |

|------|------|

| 媒體/音訊 | generate_tts.py, generate_podcast_audio.py, concat_audio.py, compose_video.py |

| 資料處理 | create_import_json.py, create_import_dibashi.py, resolve_podcast_series.py |

| 報告生成 | generate_report_pdf.py, changelog_generator.py, generate_adr_batch.py |

| 分析工具 | cache_analyzer.py, peak_hour_analyzer.py, cache_eviction_benchmark.py |

| 維護工具 | cleanup_state.py, run_podcast_create.py, add_prompt_versions.py |

| 特殊功能 | groq_direct.py, groq_quota_guard.py, skill_anomaly_detector.py |

關鍵盲區

  • pyproject.toml 的 [tool.coverage.run] source = ["hooks", "skills"] 排除了 tools/
  • 即使 tools 有 46 個測試檔案，其覆蓋率不計入 CI/CD 指標
  • 當前 fail_under = 44 僅反映 hooks + skills，掩蓋了 tools/ 的真實品質

修復方案

  1. 短期：pyproject.toml 的 source 加入 "tools"，分階段提升覆蓋率
  2. 優先補測試：groq_quota_guard.py（預算控制）、cleanup_state.py（資料清理）、security_policy_engine 的 SQL/XSS 測試
  3. 中期：媒體工具可用 mock 測試（不需實際音訊生成）

六、P2-A：跨配置檔一致性漂移

問題描述

目前 validate_config.py 僅驗證 frequency-limits.yaml 的 auto-task 一致性，但以下跨檔案關係無自動驗證：

| 來源 A | 來源 B | 關係 |

|--------|--------|------|

| frequency-limits.yaml task_key | todoist-auto-{key}.md 檔名 | 必須一致 |

| routing.yaml label 映射 | Todoist 實際 labels | 應一致 |

| llm-router.yaml task_type | timeouts.yaml keys | 應涵蓋 |

| hook-rules.yaml patterns | 實際 hooks 程式碼 | 應同步 |

修復方案

新增 tools/validate_config_consistency.py：

  • 比對 frequency-limits 與 prompt 檔案名
  • 比對 llm-router task_type 與 timeouts keys
  • 整合到 pre-push hook

七、P2-B：FSM 狀態膨脹

問題描述

run-fsm.json 記錄 Agent 執行狀態，但無過期清理機制。

修復方案

在 run-todoist-agent-team.ps1 的 FSM 讀取後（1337-1359 行）加入 retention 邏輯：

  • 自動刪除 >30 天的條目
