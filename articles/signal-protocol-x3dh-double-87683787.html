Signal Protocol 與端到端加密聊天協定研究

研究日期：2026-03-01

研究動機：今日開發 Gun.js P2P 加密聊天室（多用戶 sharedSecrets、SEA 加密任務指派），深入研究 E2EE 聊天協定的密碼學基礎，為未來升級聊天室加密機制提供理論支撐。

---

一、技術概述

端到端加密（E2EE）是確保只有通訊雙方能讀取訊息的加密方式，伺服器僅作為密文中繼節點。Signal Protocol 是目前業界公認的 E2EE 黃金標準，被 WhatsApp（超過 10 億用戶）、Meta Messenger、Google Messages（RCS）等廣泛採用。其核心由三個元件組成：X3DH 金鑰協商、Double Ratchet 持續金鑰更新、以及 PreKey Bundle 非同步建立連線。2025 年 10 月 Signal 更引入 SPQR（Sparse Post-Quantum Ratchet）升級為 Triple Ratchet，抵禦量子電腦威脅。

---

二、核心概念與原理

2.1 X3DH（Extended Triple Diffie-Hellman）金鑰協商

X3DH 解決「雙方不需同時在線即可建立共享密鑰」的問題：

金鑰類型：

  • Identity Key (IK)：長期身份金鑰（Curve25519），用於身份驗證
  • Signed Pre-Key (SPK)：中期金鑰，定期輪換，附帶身份金鑰簽名
  • One-Time Pre-Key (OPK)：一次性金鑰，用後即棄，每個只用一次
  • Ephemeral Key (EK)：臨時金鑰，發送方每次建立連線時生成

協商流程：

  1. Bob 預先上傳 PreKey Bundle（IK_B, SPK_B, OPK_B）到伺服器
  2. Alice 下載 Bob 的 Bundle，進行三次（或四次）DH 計算：

- DH1 = DH(IK_A, SPK_B) — 身份驗證

- DH2 = DH(EK_A, IK_B) — 前向保密

- DH3 = DH(EK_A, SPK_B) — 臨時保護

- DH4 = DH(EK_A, OPK_B) — 額外保護（若有 OPK）

  1. 將 DH1~DH4 透過 HKDF 導出共享密鑰 SK
  2. Alice 用 SK 初始化 Double Ratchet，發送首條加密訊息

安全特性：即使伺服器被入侵，攻擊者無法從 PreKey Bundle 推導出 SK（需要私鑰）。

2.2 Double Ratchet 演算法

Double Ratchet 是持續金鑰管理的核心，結合兩種棘輪機制確保每條訊息使用獨立金鑰：

KDF Chain（金鑰衍生鏈）：

  • 接受一個秘密 KDF Key 和輸入資料，產生不可逆的輸出
  • 三大安全屬性：韌性（Resilience）、前向安全（Forward Security）、入侵恢復（Break-in Recovery）

對稱金鑰棘輪（Symmetric-Key Ratchet）：

  • 每條訊息從 Sending Chain 或 Receiving Chain 衍生獨立的 Message Key
  • 用完即刪，確保單條訊息洩露不影響其他訊息
  • KDF_CK 使用 HMAC-SHA256，常數 0x01（Message Key）和 0x02（Chain Key）

DH 棘輪（Diffie-Hellman Ratchet）：

  • 每方維護一個 Ratchet Key Pair
  • 收到對方新的 Ratchet Public Key 時觸發 DH 棘輪步進
  • 計算新的 DH 輸出，透過 Root KDF 衍生新的 Chain Keys
  • 產生全新的 Ratchet Key Pair 取代舊的
  • 乒乓式交替確保被洩露的金鑰最終被替換

雙棘輪結合：


每條訊息 = 對稱棘輪步進 → 產生 Message Key → 加密明文
每次收到新 DH Public Key = DH 棘輪步進 → 更新 Root Key + Chain Keys

Header 結構：每條訊息 Header 包含發送方的 Ratchet Public Key 和 Chain 索引（用於亂序訊息處理）。

2.3 密碼學原語

| 元件 | 推薦實作 |

|------|----------|

| DH 函數 | Curve25519（X25519）或 Curve448（X448） |

| KDF_RK（Root KDF） | HKDF-SHA256/SHA512 |

| KDF_CK（Chain KDF） | HMAC-SHA256/SHA512 |

| 加密 | AES-256-CBC + HMAC（AEAD）或 AES-256-GCM |

| 簽名 | Ed25519 |

| 後量子擴展 | ML-KEM（CRYSTALS-Kyber）Braid Protocol |

2.4 MLS（Messaging Layer Security）— 群組加密新標準

IETF RFC 9420 標準化的 MLS 協定，解決 Double Ratchet 在大群組中的效能瓶頸：

相較 Olm/Megolm（Matrix 舊方案）的改進：

  • 群組可擴展性：透過演算法改進，大規模群組加密效能顯著提升
  • 密碼學驗證：群組成員可密碼學驗證訊息接收者身份
  • 隱藏通訊圖譜：不像 Olm/Megolm 會洩露「誰和誰通訊、哪些裝置參與」的 metadata

去中心化 MLS 挑戰：

  • 傳統 MLS 假設中心化基礎設施，不適合聯邦式系統
  • Matrix 團隊自 2020 年起開發去中心化 MLS 擴展
  • 德國聯邦國防軍 BWI 組織資助 Element 的 MLS 開發（BwMessenger 專案）

2.5 Triple Ratchet（SPQR — 2025 後量子升級）

2025 年 10 月 Signal 引入 Sparse Post-Quantum Ratchet：

  • Double Ratchet 和 SPQR 並行執行
  • 雙方金鑰透過 KDF 混合，攻擊者必須同時破解古典和後量子演算法
  • 提供後量子前向保密（Forward Secrecy）和入侵後安全（Post-Compromise Security）

三、最佳實踐與常見陷阱

最佳實踐

  1. 金鑰刪除即時化：Message Key 解密後立即銷毀，Chain Key 步進後立即刪除舊值
  2. MAX_SKIP 限制：設定跳過訊息金鑰的上限（如 1000），防止 DoS 攻擊耗盡記憶體
  3. PreKey 補充機制：One-Time PreKey 用完後自動補充，避免降級到無 OPK 模式
  4. 多裝置同步：採用 Meta Labyrinth 模式 — 伺服器儲存加密密文，用戶端控制金鑰
  5. 備份加密獨立：雲端備份必須獨立加密，否則 E2EE 形同虛設
  6. Metadata 保護：E2EE 保護內容但不保護 metadata（誰和誰通訊、何時、頻率）

常見陷阱

  1. 金鑰未安全刪除：留存舊金鑰等於前向保密失效
  2. 忽略 Header 加密：明文 Header 洩露 Ratchet Public Key 和 Chain 索引
  3. 群組加密用個人金鑰逐一加密：效能災難，應用對稱群組金鑰
  4. 未驗證身份金鑰：中間人攻擊的主要入口（需 Safety Number 驗證機制）
  5. Web 平台限制：瀏覽器無法安全儲存長期私鑰（Meta 為 Web 開發專門方案）

四、與本專案的關聯

現有聊天室架構（Gun.js SEA）

當前 bot.js 使用 Gun.js SEA 進行加密：

  • SEA.encrypt(message, sharedSecret) 對稱加密
  • 多用戶 sharedSecrets Map 管理金鑰
  • Relay 伺服器中繼加密訊息

可改進方向

  1. 金鑰輪換：目前 sharedSecret 是靜態的，可參考 Double Ratchet 的 KDF Chain 實現持續金鑰更新
  2. 前向保密：引入 DH 棘輪，每次通訊交換新的 Ephemeral Key，舊訊息即使金鑰洩露也安全
  3. PreKey 機制：利用 Gun.js 的 P2P 儲存特性，將 PreKey Bundle 發布到去中心化網路
  4. 群組金鑰管理：目前所有用戶共用單一 sharedSecret，可引入 MLS 式群組金鑰樹
  5. 後量子準備：評估 ML-KEM 在瀏覽器 Web Crypto API 的可用性

實作建議


// 概念：在 Gun.js 上實現簡化版 Double Ratchet
const crypto = require('crypto');

// KDF Chain 步進
function kdfChain(chainKey) {
  const messageKey = crypto.createHmac('sha256', chainKey)
    .update(Buffer.from([0x01])).digest();
  const newChainKey = crypto.createHmac('sha256', chainKey)
    .update(Buffer.from([0x02])).digest();
  return { messageKey, newChainKey };
}

// 每條訊息使用獨立金鑰
function encryptMessage(plaintext, chainKey) {
  const { messageKey, newChainKey } = kdfChain(chainKey);
  const iv = crypto.randomBytes(16);
  const cipher = crypto.createCipheriv('aes-256-gcm', messageKey, iv);
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()]);
  const tag = cipher.getAuthTag();
  return { encrypted, iv, tag, newChainKey };
}

五、E2EE 協定比較

| 特性 | Signal Protocol | MLS (RFC 9420) | Gun.js SEA | Matrix Olm/Megolm |

|------|----------------|----------------|------------|-------------------|

| 金鑰協商 | X3DH | TreeKEM | ECDH (Web Crypto) | Curve25519 |

| 訊息加密 | Double Ratchet | Group Ratchet Tree | AES-GCM (靜態金鑰) | Megolm Ratchet |

| 前向保密 | 每條訊息 | 每次 epoch | 無 | 每次 session |

| 入侵恢復 | DH 棘輪步進 | Epoch 更新 | 需手動換金鑰 | 需重建 session |

| 群組擴展 | O(n) 每條訊息 | O(log n) 每次操作 | O(1) 共享金鑰 | O(n) 初始化 |

| 後量子支援 | SPQR (2025) | 研究中 | 無 | 無 |

| 去中心化 | 需中心伺服器 | 需擴展 | 原生 P2P | 聯邦式 |

| Metadata 保護 | 部分 (Sealed Sender) | 是 | 否 | 否 |

六、Meta Messenger E2EE 實戰經驗

Meta 的 Messenger E2EE 部署提供了大規模生產級的寶貴經驗：

八大安全概念：

  1. 傳輸機密性 2. 儲存機密性 3. 端點控制與驗證
  2. 裝置對裝置功能設計 5. 遙測日誌限制 6. 應用安全加固
  3. 受保護內容邊界明確 8. 第三方外部審查

Labyrinth 協定：伺服器端加密儲存協定，支援多裝置同步和金鑰輪換（裝置移除時）。

OHAI 匿名搜尋：透過 Oblivious HTTP 和匿名憑證，搜尋貼圖庫時不洩露用戶身份。
