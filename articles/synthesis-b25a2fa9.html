核心內容摘錄：

  • 本報告回答三個研究問題：第一，Model Context Protocol（MCP）在 HTTP 傳輸下，把「受保護的 MCP 伺服器」明確建模為 OAuth 2.1 資源伺服器，並透過 OAuth 2.0 Protected Resource Metadata（RFC 9728）讓客戶端取得授權伺服器位置、支援範圍與權杖呈遞方式；同時以 Resource Indicators（RFC 8707）把授權請求與權杖使用綁定到 MCP 伺服器的規範化 URI，降低權杖被轉用於其他資源的風險 [來源：Authorization（MCP Specification 2025-11-25）, 2025]。第二，RFC 9728 定義的 /.well-known/oauth-protected-resource 與 WWW-Authenticate 內 resource_metadata 參數，使「發現 → 驗證 metadata resource 與請求 URL 一致 → 取得授權伺服器 metadata」成為可自動化重播的測試序列；搭配 401/403 的 scope challenge 與 step-up 重新授權，可把「最小權限」從口號落實為狀態碼驅動的行為 [來源：RFC 9728 OAuth 2.0 Protected Resource Metadata, 2025]。第三，OAuth 解決的是「誰能以何種 scope 呼叫資源伺服器」，但無法消滅工具 metadata 與工具回覆中的間接注入；OWASP 社群與 MCP-SafetyBench 研究共同指出：連線時審查與執行時把伺服器輸出當可信內容之間存在結構性信任落差，需要 schema 化輸出、特權工具隔離、以及與 OAuth 正交的集中式政策／審計 [來源：MCP Tool Poisoning（OWASP）, 2025] [來源：MCP-SafetyBench, 2025]。綜合結論：企業若要銜接「政策引擎 schema」與「OAuth 整合測試」，應以「PRM+AS discovery+PKCE+resource」為一條可機械驗證的授權鏈，並以「工具輸出驗證＋taint／閘道」為第二條可觀測的資料鏈，兩者缺一不可。
  • MCP 讓代理能在標準介面上掛載多個外部工具，但遠端部署把問題從「本機 stdio 可控環境」推向「多租戶、跨網路、動態註冊」的開放系統。先前本專案連續研究已覆蓋 MCP 安全與 taint 傳播；context/continuity/auto-task-ai_deep_research.json 建議下一步深化「policy engine schema 與 OAuth 整合測試」。動機在於：若僅驗證 JWT 簽章與過期時間，仍可能忽略 audience 綁定、PRM 冒充、以及工具層面的資料流污染。本文因此聚焦「規範要求的授權發現與權杖綁定」如何具體化成 CI 可跑的測試矩陣，並對齊學術基準對整體風險的量化結論，避免把 OAuth 當成萬靈丹 [來源：How to build secure and scalable remote MCP servers, 2025]。
  • MCP 規範將客戶端、資源伺服器、授權伺服器的責任分界對齊 OAuth 2.1 家族：資源伺服器必須實作 RFC 9728，使客戶端能以標準化 HTTP 流程定位授權伺服器；客戶端則必須同時支援從 WWW-Authenticate 取得 resource_metadata URL，以及在缺省時依序探測 well-known 路徑 [來源：Authorization（MCP Specification 2025-11-25）, 2025]。RFC 9728 進一步要求：若 metadata 係由 WWW-Authenticate 提供 URL 取得，則回傳的 resource 必須與客戶端實際請求資源時使用的 URL 完全一致，否則客戶端必須拒絕使用該 metadata，藉此抑制中間人與 DNS 層級的冒充 [來源：RFC 9728, 2025]。對工程團隊而言，這三句話可直接轉成測試案例：偽造 resource、偽造 metadata 主機、與正確三元組（MCP endpoint、PRM、authorization_servers）對照。
  • 規範要求 MCP 客戶端在授權與換票請求皆帶上 resource，且其值必須是目標 MCP 伺服器的 canonical HTTPS URI；資源伺服器驗票時亦必須確認權杖係為該伺服器簽發之意圖 audience，失敗則 401 [來源：Authorization（MCP Specification 2025-11-25）, 2025]。Auth0 對 2025-06 修訂的解讀亦強調：此舉旨在避免惡意或遭入侵的伺服器拿著「為別的資源簽發的權杖」跨服務兌換 [來源：MCP Spec Updates from June 2025, 2025]。GitHub 工程部落格則把整體流程整理為八步驟（401 挑戰、取 PRM、DCR、PKCE+resource、同意、換票、Bearer 呼叫），明確指出「沒有 MCP 特規的怪異 OAuth」，利於直接套用現成函式庫與測試替身 [來源：How to build secure and scalable remote MCP servers, 2025]。
