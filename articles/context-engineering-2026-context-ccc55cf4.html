摘要（~200 字）

2026 年，Context Engineering（CE）已從 Prompt Engineering 的衍生工具演化為獨立的 AI 系統設計學科。Anthropic 定義 CE 為「在推理期間篩選、維護最優 token 集合的策略集合」，超越「如何問問題」，聚焦「此刻模型需要哪些資訊」。

CE 的核心洞見在於：Context Window 是有限的注意力預算。Transformer 的 n² 配對注意力機制使 token 增加必然導致精確度下降（Context Rot）——即使 Claude 有 200K、Gemini 3 Pro 有 2M tokens，「更多不等於更好」。業界最佳實踐收斂為：追求最小高訊號 token 集、RAG 與工具調用互補不競爭、三層記憶架構（短期/長期/參數）、Sub-Agent 分層壓縮。

本報告整合 Anthropic 工程實踐、Weaviate 記憶架構框架與業界研究，提煉 5 點可操作洞見，並連結至 daily-digest-prompt 系統的現有設計。

關鍵洞見

洞見 1：CE 學科定義與六大支柱

定義收斂：

  • Anthropic：「curating and maintaining the optimal set of tokens during LLM inference」
  • Weaviate：「designing the architecture that feeds an LLM the right information at the right time」
  • 業界共識：PE 問「怎麼問」，CE 問「模型此刻需要什麼」

六大支柱（Weaviate 框架）：

  1. Agent 編排：決策路由與多步驟執行協調
  2. 查詢增強：使用者輸入精煉（重寫、HyDE、多查詢）
  3. Retrieval：連接外部知識（RAG、向量搜尋）
  4. Prompting：引導推理的結構化指令
  5. Memory：短期/長期/工作記憶的分層管理
  6. Tools：賦予模型真實世界行動能力

與 PE 的根本差異：PE 是靜態的提示寫法；CE 是動態的資訊環境設計，隨 Agent 每一步決策而演化。

來源：Anthropic Engineering Blog、Weaviate Blog

洞見 2：Context Rot 效應與注意力預算原則

核心問題：Transformer 的 n² 配對注意力 → token 愈多，每個 token 「競爭」注意力愈激烈：

  • 精確召回能力下降（needle-in-a-haystack 效應）
  • 推理品質減弱，模型開始遺漏應捕捉的資訊
  • 幻覺率上升（Context Poisoning 累積）

生產量化：

  • 即使 Claude 200K tokens，長上下文精確度仍低於短上下文
  • Gemini 3 Pro 2M tokens、GPT-4.5 256K tokens — 視窗擴大不解決 Rot 問題
  • 團隊可在不降品質的前提下削減 60-80% 成本（Anthropic 2026 指引）

Anthropic 最佳實踐原則：

「找到最小可能的高訊號 token 集，最大化期望結果的概率。」

System Prompt 的 Goldilocks Zone：

  • 過度詳細 → Brittle（每個邊界情況都需更新）
  • 過度模糊 → 不可靠（模型自行詮釋，不一致）
  • 最佳：使用 XML tags 或 Markdown 標頭劃分清晰段落（、）

來源：Anthropic Engineering Blog、Web search（finout.io、moltbook-ai.com）

洞見 3：RAG vs 工具調用的 CE 統一決策框架

角色分工：

| 維度 | RAG | 工具調用 |

|------|-----|---------|

| 用途 | 餵入知識（靜態資料） | 賦予行動（動態執行） |

| CE 視角 | 知識 Retrieval 層 | 行動執行層 |

| 互補關係 | 互補，共用 context window 預算 | |

關鍵技術：

  1. Just-In-Time Retrieval：

- 保持輕量識別符（file paths、query strings、links）在 context

- 按需透過工具動態載入完整資料

- 避免預先載入所有可能需要的資料（預載 vs 按需的混合策略）

  1. 工具 RAG 化（Tool Retrieval）：

- 對工具描述應用 RAG，動態取最相關工具注入 context

- 研究數據：工具 RAG 化提升選擇準確率 3 倍（2025 論文）

- OpenAI 建議：每 Agent < 20 工具，準確率從 10 工具後開始下滑

  1. MCP 標準化整合：

- Model Context Protocol 解決 M×N 工具整合問題（M 模型 × N 工具 → M+N）

- 統一工具描述格式，降低 CE 複雜度

  1. RAG 分塊策略的 CE 影響：

- 小塊：精確 embedding，缺少上下文 → 注入時需補周邊段落

- 大塊：豐富上下文，「嘈雜」embedding → 消耗更多 context window

- 最佳：動態分塊 + 語義邊界對齊

來源：Weaviate Blog、SmartScope Blog（RAG vs MCP）、Elastic Search Labs、LangChain Blog

洞見 4：三層記憶架構與四種失效模式

三層記憶：

| 層次 | 位置 | 特徵 | 適用場景 |

|------|------|------|---------|

| 短期（In-Context） | 當前 context window | 即時存取，有限容量 | 當前推理步驟、工具輸出 |

| 長期（External） | 向量 DB（Qdrant/Weaviate 等） | 持久化，語義搜尋 | 歷史互動、領域知識 |

| 參數（Parametric） | 模型權重 | 不可實時更新 | 基礎推理能力、常識 |

| 工作記憶（Working） | 臨時外部存儲 | 跨步驟中間狀態 | 多步驟任務（訂票、代碼生成） |

四種失效模式（必須防範）：

  1. Context Poisoning：幻覺資訊在 Agent 迭代中累積 → 後續推理基於錯誤前提

- 防範：每步驗證工具輸出；隔離 web 來源內容與高權限操作

  1. Context Distraction：過量歷史資料覆蓋新的推理需求

- 防範：定期 Compaction；限制 message history 保留輪數

  1. Context Confusion：無關工具/文件佔據 window → 模型選錯工具或引用無關知識

- 防範：工具 RAG 化；動態 Retrieval 代替靜態預載

  1. Context Clash：矛盾資訊同時存在 → 模型決策癱瘓

- 防範：單一真相來源原則；Memory 定期去重與衝突解決

來源：Weaviate Blog

洞見 5：長期 Agent 執行的三大 CE 策略

1. Compaction（壓縮）：

  • 時機：context window 接近上限
  • 原則：保留架構決策、丟棄冗餘工具輸出
  • 實作：摘要 + 保留關鍵事實清單（類 BufferWindow 壓縮策略）
  • 本系統對應：skills/context-compressor/SKILL.md

2. Structured Note-Taking（結構化筆記外部記憶）：

  • Agent 在執行過程中主動維護外部持久記憶
  • 支援追蹤數千步驟而不造成 context window 膨脹
  • 本系統對應：context/continuity/*.json（各任務的跨輪記憶）

3. Sub-Agent 分層架構：

  • 專責子 Agent 處理局部任務，使用 10,000+ tokens 完成深度分析
  • 返回給協調 Agent 的結果：1,000-2,000 tokens 的精煉摘要
  • 壓縮比：5-10x
  • 本系統對應：run-todoist-agent-team.ps1 Phase 1→2→3 分層；子 Agent 透過 results/*.json 交接

Token 成本優化數據：

  • Prompt Caching：cache hit 時輸入成本降低 90%（Anthropic API）
  • Batch Processing：成本降低 50%（Anthropic Message Batches API）
  • Model Routing：Haiku / Sonnet / Opus 按任務複雜度路由 → 整體成本降低 60-80%

來源：Anthropic Engineering Blog、finout.io Anthropic Pricing Guide 2026
