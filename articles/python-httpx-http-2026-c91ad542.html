---

技術概述

httpx 是基於 httpcore 傳輸層的現代 Python HTTP 客戶端，同時支援同步（Client）與非同步（AsyncClient）操作介面，從根本上解決了 requests 只能同步、aiohttp 只能非同步的分裂問題。其主要技術特性：

  • 完整 HTTP/2 支援（multiplexing + header compression），需 pip install 'httpx[http2]'
  • 預設 5 秒超時（安全預設，相較於 requests 的無限等待）
  • 完整型別標註（Type hints，IDE 友好）
  • 相容三種非同步後端：asyncio、trio、anyio
  • requests-compatible API，遷移成本低

適用場景：FastAPI/ASGI 框架的 API 呼叫、AI Agent 工具函式（tool calls）、多 API 並行批次請求、爬蟲。

與主要替代方案的差異：

| 特性 | httpx | requests | aiohttp |

|------|-------|----------|---------|

| Native async | ✅ | ❌ | ✅ |

| HTTP/2 | ✅ | ❌ | ❌ |

| 同步介面 | ✅ | ✅ | ❌ |

| 超高並發（>300）效能 | ⚠️ | N/A | ✅ |

| 與 requests API 相容 | ✅ | — | ❌ |

[來源：fooish.com Python httpx 教學, 2026; Oxylabs HTTPX vs Requests vs AIOHTTP 比較]

核心概念與原理

1. 傳輸架構（Transport Layer）

httpx 架構分三層：

  1. httpx.AsyncClient / httpx.Client — 高層公共 API，處理重定向、Cookie、認證
  2. httpcore（底層傳輸）— 管理 HTTP/1.1 連接池和 HTTP/2 多路複用
  3. asyncio / trio socket — 實際網路 IO

AsyncHTTPTransport 是預設非同步傳輸實作，HTTPTransport 是同步版本。兩者均支援 retries 參數進行底層網路層級重試（連接失敗、逾時），但不處理 HTTP 狀態碼（如 429）的應用層重試。

[來源：Python httpx 官方文件 Transports, 2026]

2. 連接池管理（Connection Pooling）

httpx 透過 httpx.Limits 控制三個連接池參數：

  • max_keepalive_connections=20：允許的 keep-alive 連接數（default 20）
  • max_connections=100：最大總連接數，超過則進入等待佇列（default 100）
  • keepalive_expiry=5：空閒 keep-alive 連接的存活秒數（default 5s）

連接池的設計目標是重用 TCP 連接，避免每次請求的 TCP handshake 和 TLS negotiation 開銷。PoolTimeout（從 httpx.Timeout 的 pool 參數設定）在無法及時從池取得連接時拋出。

核心原則：不要在熱路徑（hot loop）內建立 Client。每次 async with httpx.AsyncClient() 都會建立全新連接池，無法重用舊連接，連接建立成本高昂。

[來源：Python httpx 官方文件 Clients — "Creating an AsyncClient is expensive", 2026]

3. 四維超時系統（Timeout System）

httpx 的超時分成四個獨立階段，每個階段都可獨立配置：

| 超時類型 | 說明 | 例外類型 |

|---------|------|--------|

| Connect | 建立 socket 連接的等待時間 | ConnectTimeout |

| Read | 等待回應資料塊的間隔時間 | ReadTimeout |

| Write | 發送請求資料塊的等待時間 | WriteTimeout |

| Pool | 從連接池取得連接的等待時間 | PoolTimeout |

預設行為：所有階段共用 5 秒總逾時（httpx.Timeout(5.0)）。

細粒度設定方式：


# 不同階段設定不同超時
timeout = httpx.Timeout(30.0, connect=5.0, pool=10.0)
# 30s read/write，5s connect，10s pool 等待
client = httpx.AsyncClient(timeout=timeout)

注意：httpx.TimeoutException 是所有超時例外的父類，可一次捕捉所有類型。

[來源：Python httpx 官方文件 Timeouts — "HTTPX is careful to enforce timeouts everywhere by default", 2026]

4. AsyncClient 生命週期管理

兩種管理方式：


# ✅ 推薦：context manager，自動 aclose()
async with httpx.AsyncClient() as client:
    response = await client.get(url)

# ⚠️ 需手動管理：開發者負責 aclose()
client = httpx.AsyncClient()
try:
    response = await client.get(url)
finally:
    await client.aclose()  # 不可省略！

在 Streaming 手動模式下，Response.aclose() 同樣必須呼叫，否則 TCP 連接留置不釋放。

[來源：Python httpx 官方文件 Async Support, 2026; GitHub Issues #1155 e2b-dev/E2B]

5. HTTP/2 多路複用

啟用 http2=True 後，httpx 可在單一 TCP 連接上同時發起多個請求（stream multiplexing + header compression）：

  • 需要伺服器支援 ALPN 協商，否則自動 fallback 到 HTTP/1.1
  • 基準測試：HTTP/2 模式比 HTTP/1.1 快 ~17%（1000 requests benchmark: 8.5s vs 10.22s）
  • 但 aiohttp（HTTP/1.1）在高並發場景仍快 httpx HTTP/2 兩倍以上（針對非 HTTP/2 服務器）

[來源：Oxylabs HTTPX vs Requests vs AIOHTTP 比較, 2025]

最佳實踐

1. 應用層級單一客戶端（Single Client Instance）

情境：長時間運行的 AI Agent 服務或 FastAPI 應用

建議：在 lifespan 事件或模組層級建立單一 AsyncClient


# FastAPI lifespan 最佳實踐
from contextlib import asynccontextmanager
import httpx
from fastapi import FastAPI

@asynccontextmanager
async def lifespan(app: FastAPI):
    # 啟動：建立共享客戶端
    app.state.http_client = httpx.AsyncClient(
        limits=httpx.Limits(
            max_keepalive_connections=20,
            max_connections=100,
            keepalive_expiry=30,
        ),
        timeout=httpx.Timeout(30.0, connect=10.0),
    )
    yield
    # 關閉：正確釋放連接
    await app.state.http_client.aclose()

app = FastAPI(lifespan=lifespan)

依據：官方文件明確指出「Creating an AsyncClient is expensive」，並推薦 lifespan 模式。

反例：在 request handler 內 async with httpx.AsyncClient()，每次請求重建連接池。

[來源：Python httpx 官方文件 Clients; BetterStack httpx guide, 2026]

2. 結構化超時配置策略

依 API 類型選用不同超時預算：

| API 類型 | connect | read | 備註 |

|---------|---------|------|------|

| 本地 KB API | 3s | 30s | localhost，TCP 連接快 |

| 外部 LLM API（如 Claude） | 5s | 120s | 考慮 cold start |

| ntfy 通知 | 5s | 10s | 快速失敗為佳 |

| 爬蟲/外部站點 | 5s | 30s | 一般建議 |

絕對禁止 timeout=None：在 AI Agent 工作流中，上游服務偶爾無回應，無限等待會讓整個 Phase 掛起，造成 Task Scheduler 0x41301 殭屍狀態。

[來源：dasroot.net 2026-02; Python httpx 官方文件 Timeouts]

3. httpx-retries 聲明式重試（網路層）

httpx-retries 透過 RetryTransport 插入重試邏輯，處理連接失敗等網路層錯誤：


from httpx_retries import Retry, RetryTransport

retry = Retry(total=3, backoff_factor=0.5)
# total=3：最多重試 3 次
# backoff_factor=0.5：{0.5s, 1.0s, 2.0s} 退避延遲

async with httpx.AsyncClient(
    transport=RetryTransport(retry=retry)
) as client:
    response = await client.get("https://api.example.com/data")

依據：網路層重試（ConnectError、ConnectTimeout）與應用層重試（HTTP 429、503）分開處理，職責明確。

[來源：httpx-retries 官方文件, will-ockmore.github.io]

4. tenacity + httpx 處理 HTTP 429 Rate Limit

httpx-retries 不處理 HTTP 狀態碼，需用 tenacity 在應用層處理：


import asyncio
import random
import httpx
from tenacity import (
    retry, stop_after_attempt,
    wait_exponential, retry_if_exception_type
)

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential(multiplier=0.5, min=1, max=30),
    retry=retry_if_exception_type(httpx.HTTPStatusError),
)
async def api_call_with_retry(
    client: httpx.AsyncClient, url: str
) -> dict:
    response = await client.get(url)
    if response.status_code == 429:
        # 優先讀取 Retry-After header
        retry_after = float(response.headers.get("Retry-After", 1.0))
        jitter = random.uniform(0, 0.5)
        await asyncio.sleep(retry_after + jitter)
        response.raise_for_status()  # 觸發 tenacity retry
    response.raise_for_status()
    return response.json()

重試策略原則（2026 生產建議）：

  • 重試條件：HTTP 429、500、502、503、504 + ConnectError + TimeoutException
  • 不重試：401、403、400（客戶端錯誤）
  • Base delay: 0.5s，指數係數: 2.0，最大: 30s
  • 加 jitter（全隨機）防止 retry storm
  • 優先讀取 Retry-After header，勿用固定延遲

[來源：Fastio AI Agent Retry Patterns 2026; dasroot.net 2026-02]

5. asyncio.gather 並行請求模式


async def fetch_all(urls: list[str]) -> list[dict]:
    async with httpx.AsyncClient(
        limits=httpx.Limits(max_connections=50, max_keepalive_connections=20)
    ) as client:
        tasks = [
            client.get(url, timeout=httpx.Timeout(30.0, connect=5.0))
            for url in urls
        ]
        responses = await asyncio.gather(*tasks, return_exceptions=True)
    # 過濾例外
    return [
        r.json() for r in responses
        if not isinstance(r, Exception)
    ]

效能收益：5 個各需 2 秒的請求，並行執行總耗時 ~2 秒 vs 循序 10 秒（5x 加速）。

[來源：dailypypy.org 正體中文 httpx 教學; oneuptime.com 2026-02]

陷阱與反例

陷阱一：熱路徑建立多個客戶端（P0 — 最常見）


# ❌ 反例：每次呼叫重建連接池
async def bad_approach(url: str) -> dict:
    async with httpx.AsyncClient() as client:  # 每次都是全新連接池
        return (await client.get(url)).json()

# ✅ 正解：Client 提升到模組層級或 lifespan
_shared_client: httpx.AsyncClient | None = None

def get_client() -> httpx.AsyncClient:
    global _shared_client
    if _shared_client is None:
        _shared_client = httpx.AsyncClient(
            timeout=httpx.Timeout(30.0, connect=5.0),
        )
    return _shared_client

成因：AsyncClient 建立成本高（初始化 httpcore ConnectionPool、SSL context 驗證、後端事件循環綁定）。在高頻路徑重複建立完全抵銷連接複用效能收益，且可能造成 fd 洩漏。

[來源：Python httpx 官方文件 Clients — "Creating an AsyncClient is expensive"; GitHub Discussion #2444]

陷阱二：忘記 aclose() 造成資源洩漏

失敗模式：手動建立 AsyncClient（不用 async with）但沒有在 exception path 呼叫 aclose()，導致 TCP 連接和 fd 洩漏。

真實案例：GitHub Issue #1155 (e2b-dev/E2B) — httpx.AsyncClient 在 sandbox 強制終止時未關閉，生產環境出現連接池洩漏，需要定期重啟服務修復。

規避方法：永遠使用 async with，或確保 finally 區塊中呼叫 aclose()。

[來源：GitHub Issues #1155 e2b-dev/E2B, 2024; httpx 官方文件 Async Support]

陷阱三：超高並發場景 ReadError（>300 concurrent）

在 300 concurrent / 1000 total requests 的壓力測試中，httpx 出現隨機 httpx.ReadError，且只在高負載下非確定性地觸發，難以重現和除錯。aiohttp 在相同場景下無此問題。

根因：httpx 的 AsyncHTTPTransport 使用 httpcore 的連接池，在極高並發下連接競爭（pool contention）超過 max_connections 上限，請求被迫等待並最終 ReadError。

規避策略（按優先級）：

  1. 降低並發數：asyncio.Semaphore(50) 限制最大並行請求數
  2. 調高 max_connections：httpx.Limits(max_connections=200, max_keepalive_connections=50)
  3. 超高並發（>300）場景改用 aiohttp + TCPConnector(limit=N)

[來源：miguel-mendez-ai.com 高並發測試報告, 2024; GitHub Issue #838 encode/httpx — "HTTPX AsyncClient slower than aiohttp?"]

陷阱四：timeout=None 導致 AI Agent 無限掛起


# ❌ 危險：無超時限制
response = await client.get(llm_api_url, timeout=None)

失敗場景：LLM API 上游服務偶發無回應，timeout=None 讓整個 Agent Phase 永久掛起，造成 Windows Task Scheduler 進入 0x41301「Running」殭屍狀態，後續排程被阻塞。

規範：所有外部 API 呼叫均設置明確超時值，絕不使用 timeout=None。

[來源：Python httpx 官方文件 Timeouts; 本專案 zombie-lock-subprocess-hang 事故記錄]

與本專案關聯

本專案 tools/*.py 目前大量使用 Bash curl 發送 HTTP 請求（KB API、ntfy、Todoist API），存在以下系統性問題：

  1. 每次 curl 都啟動新進程（~10-50ms overhead）、無連接複用
  2. 重試邏輯散落各處，無統一策略
  3. 超時設定不一致（部分用 --max-time，部分未設）
  4. 錯誤處理依賴 exit code 而非結構化例外

落地場景一：KB API 呼叫統一化

建立 tools/http_client.py 工廠函式，供 research_saturation_analyzer.py、ntfy_dedup.py 等工具共享：


# tools/http_client.py — 可執行片段（官方文件 + httpx-retries 整合驗證）
import httpx
from httpx_retries import Retry, RetryTransport


def make_kb_client() -> httpx.AsyncClient:
    """建立 KB API 用的 AsyncClient（含重試 + 連接池）。"""
    return httpx.AsyncClient(
        base_url="http://localhost:3000",
        timeout=httpx.Timeout(30.0, connect=5.0),
        limits=httpx.Limits(
            max_keepalive_connections=10,
            max_connections=50,
            keepalive_expiry=30,
        ),
        transport=RetryTransport(Retry(total=3, backoff_factor=0.5)),
        headers={"Content-Type": "application/json"},
        http2=False,  # 本地 API 不支援 HTTP/2
    )


def make_ntfy_client() -> httpx.AsyncClient:
    """建立 ntfy.sh 用的 AsyncClient（快速失敗策略）。"""
    return httpx.AsyncClient(
        base_url="https://ntfy.sh",
        timeout=httpx.Timeout(10.0, connect=5.0),  # ntfy 要快速失敗
        transport=RetryTransport(Retry(total=2, backoff_factor=1.0)),
    )

落地場景二：研究飽和度分析工具非同步化

tools/research_saturation_analyzer.py 每次執行需查詢 KB API，遷移到 httpx.AsyncClient 後可並行查多個搜尋關鍵字，預計降低 30-50% 執行時間。
