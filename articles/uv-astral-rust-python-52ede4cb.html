uv — Rust 驅動的 Python 套件管理器完整指南（2026）

技術概述

uv 是 Astral（Ruff 背後的公司）用 Rust 開發的全方位 Python 套件管理器，一個工具取代 pip、pip-tools、pipx、poetry、pyenv、virtualenv 的功能。2026 年初 PyPI 月下載量已突破 7,500 萬次，超越 Poetry 的 6,600 萬次，成為 Python 生態系的事實標準。其核心優勢在於 Rust 實作帶來的 10-100 倍效能提升，以及統一管理 Python 版本、虛擬環境、依賴解析、鎖檔生成的一站式體驗。

---

核心概念與原理

架構設計

uv 的效能優勢源於兩個層面：

  1. Rust 原生實作：避免 Python 直譯器的啟動開銷與 GIL 限制，所有依賴解析、下載、安裝都在 Rust 層完成
  2. 並行下載與快取策略：多執行緒並行下載套件，搭配全域快取（~/.cache/uv/）實現極速重複安裝
  3. 跨平台鎖檔格式：uv.lock 記錄所有平台的解析結果，一次解析、處處安裝

兩檔案系統

| 檔案 | 角色 | 類比 |

|------|------|------|

| pyproject.toml | 宣告依賴（最低版本約束） | 如 requirements.in |

| uv.lock | 完整解析結果（精確版本） | 如 requirements.txt |

uv.lock 是跨平台的——即使你在 Linux 上鎖定，Windows 特有的依賴（如 tzdata）也會被記錄，確保可重現性。

核心命令速查

專案管理


# 初始化專案
uv init myproject

# 新增依賴
uv add requests pandas
uv add --dev pytest ruff        # 開發依賴
uv add --group gmail google-auth # 可選依賴群組

# 移除依賴
uv remove pandas

# 鎖定依賴版本
uv lock

# 同步環境（安裝 lockfile 中的所有依賴）
uv sync
uv sync --no-group dev           # 僅安裝生產依賴
uv sync --frozen                 # CI 用：嚴格依照 lockfile，不更新

# 執行腳本（自動同步環境）
uv run python script.py
uv run pytest tests/

Python 版本管理


# 安裝特定 Python 版本
uv python install 3.12
uv python install 3.9 3.10 3.11  # 同時安裝多版本

# 列出可用版本
uv python list

# 釘選專案 Python 版本
uv python pin 3.12

工具管理（取代 pipx）


# 一次性執行（不安裝）
uvx ruff check .
uvx black --check .

# 全域安裝工具
uv tool install ruff
uv tool list

pip 相容介面（漸進遷移用）


# 直接取代 pip（10-100x 加速）
uv pip install -r requirements.txt
uv pip compile requirements.in -o requirements.txt
uv pip sync requirements.txt

# 建立虛擬環境
uv venv .venv --python 3.12

應用程式專案（如 daily-digest-prompt）

name = "daily-digest-prompt"

[dependency-groups]

最佳實踐與常見陷阱

最佳實踐

  1. 務必將 uv.lock 加入版本控制：這是可重現性的關鍵，等同於 npm 的 package-lock.json
  2. CI 中使用 --frozen 旗標：uv sync --frozen 確保完全依照 lockfile 安裝，不會意外更新版本
  3. 用 uv run 取代手動啟動 venv：uv run python script.py 自動確保環境同步，無需 source .venv/bin/activate
  4. 善用 dependency-groups 分離關注點：將 dev、test、docs、gmail 等依賴分群管理
  5. 定期執行 uv lock --upgrade：安全地升級所有依賴到最新相容版本
  6. 快取管理：uv cache prune 清理未使用的快取，避免磁碟膨脹

常見陷阱

  1. Windows 路徑問題：uv run --project D:/path 需要正斜線，反斜線可能導致路徑解析失敗
  2. --no-editable 容易遺忘：生產部署時應使用 uv sync --no-editable，避免依賴原始碼路徑
  3. 混用 pip 與 uv：在 uv 管理的環境中直接用 pip 安裝會繞過鎖檔，導致環境漂移
  4. optional-dependencies vs dependency-groups 混淆：[project.optional-dependencies] 是 PyPI 發布用的，[dependency-groups]（PEP 735）是本地開發用的。應用程式專案建議用後者
  5. 忘記 requires-python 約束：不設定會導致 uv 選擇最新 Python 版本，可能與專案不相容

與本專案的關聯（daily-digest-prompt）

現行使用模式

本專案在以下場景強制使用 uv：

  1. Hooks 執行：所有 Python hooks（pre_bash_guard、pre_write_guard、cjk_guard 等）都透過 uv run --project D:/Source/daily-digest-prompt python 執行，確保依賴隔離
  2. 測試執行：uv run pytest tests/ 執行 556 個測試
  3. 工具腳本：uv run python tools/md_to_docx.py 等工具一律透過 uv
  4. 依賴管理：pyproject.toml 定義 4 個核心依賴 + dev/gmail 可選群組

潛在優化方向

  1. 遷移至 [dependency-groups]：目前使用舊版 [project.optional-dependencies]，建議遷移到 PEP 735 的 [dependency-groups] 語法
  2. 新增 uv.lock：目前專案未提交 lockfile，建議執行 uv lock 並提交 uv.lock
  3. CI 整合：若未來加入 GitHub Actions，使用 uv sync --frozen 確保可重現性
  4. 升級 requires-python：目前設定 >=3.9，考慮升級至 >=3.11 以利用較新的語言特性（如 tomllib 標準庫）

與 Hooks 系統的整合要點


# .claude/settings.json 中的 hook 命令格式
"uv run --project D:/Source/daily-digest-prompt python D:/Source/daily-digest-prompt/hooks/pre_bash_guard.py"

--project 旗標確保 uv 使用專案的 pyproject.toml 解析依賴，即使 CWD 不在專案目錄內（解決 hook CWD 漂移問題）。
