一、專案概述

RAGFlow（infiniflow/ragflow）是由 InfiniFlow 團隊開發的開源 RAG 引擎，定位為「結合尖端 RAG 與 Agent 能力的上下文層（Context Layer）」。截至 2026 年 3 月已累積 74,100 Stars、8,200 Forks，採用 Apache 2.0 授權，被 GitHub Octoverse 2025 報告列為年度增長最快的專案之一。

二、核心技術架構

2.1 DeepDoc 深層文件理解引擎

RAGFlow 最核心的差異化技術是 DeepDoc——自建的文件理解引擎，包含 Vision（視覺分析）和 Parser（解析器）兩大模組：

  • Vision 模組：執行版面配置分析（Layout Analysis），判斷文字區塊是否連續、哪些區域需要表格結構辨識（TSR），以及圖片與圖說的配對
  • Parser 模組：支援 PDF、Word、Excel、Email 等 23 種格式的深度解析，PDF 輸出包含帶位置資訊的文字區塊、裁切圖片表格（含自然語言翻譯）、及帶圖說的圖片

可視化分塊驗證是 DeepDoc 的殺手級功能：使用者可直觀檢視文件被如何分塊解析，支援人工干預調整，這在企業級 RAG 部署中至關重要——因為分塊品質直接決定檢索準確率。

2.2 TreeRAG 階層檢索架構

RAGFlow 提出的 TreeRAG 解決了 RAG 領域最根本的矛盾：小分塊提高語義匹配精度但破碎上下文，大分塊保持連貫但引入雜訊。

離線階段：

  • LLM 分析文件，構建階層式摘要的多層樹狀結構
  • 同時生成語義增強元素：摘要、關鍵字、實體、問題

線上階段：

  • 細粒度相似度搜尋定位相關分塊
  • 利用樹狀結構動態聚合父節點/兄弟節點，組成語義完整的「大片段」

這種「先精確定位、再擴展閱讀」的策略有效緩解了「Lost in the Middle」效應，且不會線性增加上下文成本。

2.3 PTI 攝取管線（Parse-Transform-Index）

RAGFlow 將傳統的 RAG 工作流重構為類似 ETL 的三階段管線：

Parser 節點：

  • 處理文件讀取與版面辨識
  • 支援 8 大類 23 種檔案格式
  • PDF 可選 DeepDoc（內建）、MinerU（業界領先）、Naive（純文字）三種解析器

Chunker 節點：

  • Token-based：可自訂分段大小（預設 512 tokens），支援重疊功能維持語義連續性
  • Title-based：基於正則表達式的階層式分段，適合技術手冊等結構化文件

Transformer 節點：

  • 彌合使用者查詢與文件之間的「語義鴻溝」
  • 生成補充內容：摘要、關鍵字、問題、metadata
  • 三種模式：Improvise（創意）、Precise（忠實）、Balance（平衡）

Indexer 節點：

  • 寫入檢索引擎（Infinity、Elasticsearch、OpenSearch）
  • 支援混合搜尋：全文索引（關鍵字精準度）+ Embedding（語義搜尋）
  • 可選檔名加權與欄位特定索引策略

三、從 RAG 到 Context Engine 的典範轉移

3.1 Context Engine 框架

RAGFlow 的 2025 年終回顧提出了「RAG 正在演化為 Context Engine」的核心論點。完整的資料基礎涵蓋三個檢索領域：

  1. 領域知識（Domain Knowledge）：傳統 RAG 式的企業文件檢索
  2. 工具資料（Tool Data）：對數百/數千個工具描述進行語義搜尋（解決 MCP 等協定未涉及的「工具選擇」問題）
  3. 對話/狀態資料（Conversation/State Data）：動態互動歷史與 Agent 記憶

這種統一方法從 Prompt 中「硬編碼上下文拼接」轉向「宣告式、基於即時意圖的動態上下文組裝」。

3.2 工具檢索創新

當工具數量超過數十個時，盲目將所有描述塞入上下文會導致「選擇癱瘓」並浪費 Token。RAGFlow 的解決方案：

  • 建立工具描述的索引庫（向量、關鍵字或混合）
  • 根據當前任務上下文動態檢索相關工具
  • 附帶「操作手冊」（Playbooks）——正確順序使用工具的元知識

關鍵洞見：即使簡單的 BM25 關鍵字檢索也優於靜態硬編碼的工具描述。

3.3 記憶系統與 RAG 的同構關係

RAGFlow 指出記憶和 RAG 共享相同的功能核心（基於檢索），但服務不同的資料源：

  • RAG：靜態企業知識資產
  • Memory：動態生成的互動日誌、對話歷史、Agent 狀態

記憶的複雜性集中在生命週期管理——決定寫入時機、摘要觸發條件、片段關聯方式。

四、競品比較與市場定位

| 維度 | RAGFlow | LangChain | LlamaIndex |

|------|---------|-----------|------------|

| 核心強項 | 深層文件理解 + 可視化分塊 | 快速原型 + 多用途鏈式組合 | 索引策略 + 精準檢索 |

| 目標用戶 | 企業級 RAG 部署、非開發者 | 全端 AI 開發者 | 文件密集型 QA 系統 |

| 工具整合 | Agent + MCP 工作流 | 50K+ 整合 | 160+ 資料連接器 |

| 部署模式 | Docker 容器 + 可視化 UI | SDK / 程式碼優先 | SDK / 程式碼優先 |

| 文件處理 | DeepDoc 深度解析（含 OCR、TSR、圖片理解） | 基本解析 | 中等解析 |

| 分塊策略 | 12 種模板 + 可視化人工干預 | 基本分割器 | 多種索引策略 |

市場定位：RAGFlow 聚焦於「文件理解品質」這個 RAG 系統中最被低估但最關鍵的環節，而非追逐更多整合或更靈活的鏈式組合。

五、可借鏡的技術項目（對 Daily-Digest-Prompt 系統的啟示）

啟示 1：PTI 管線架構可套用於知識庫攝取

本系統的知識庫（localhost:3000）目前採用單一管道匯入。RAGFlow 的 PTI（Parse-Transform-Index）三階段管線提供了更精細的資料處理模型：

  • Parse：不同格式用不同解析器（目前系統已有 contentText → Tiptap JSON 轉換）
  • Transform：匯入時自動生成摘要、關鍵字、問題（可增強搜尋召回率）
  • Index：混合索引策略（本系統已有 BM25 + 向量混合搜尋）

具體建議：在 POST /api/import 流程中加入 Transform 階段——自動從 contentText 提取關鍵字和 2-3 句摘要，存入筆記 metadata，提升 hybrid search 的 BM25 分數。

啟示 2：TreeRAG 的「先定位後擴展」策略

本系統的 hybrid search 回傳固定 topK 結果，每筆結果是獨立的分塊。RAGFlow 的 TreeRAG 提示了一個改進方向：當搜尋命中某筆記的一個段落時，可以動態擴展讀取該筆記的前後段落，提供更完整的上下文。

具體建議：hybrid search API 可增加 expand_context: true 參數，命中時額外回傳同筆記的相鄰分塊。

啟示 3：工具檢索取代硬編碼 Skill 索引

RAGFlow 的「工具資料語義搜尋」概念直接對應本系統的 SKILL_INDEX.md。目前 Agent 啟動時載入完整 Skill 索引（約 300 行），當 Skill 數量持續增長，這將占用寶貴的 context window。

具體建議：將 SKILL_INDEX.md 的觸發關鍵字和能力描述向量化，存入知識庫。Agent 啟動時僅載入前 5 個最相關的 Skill 描述，而非全部索引。這正是 RAGFlow 提出的「即使簡單 BM25 都優於硬編碼」洞見。

啟示 4：可視化分塊驗證的品質保證理念

本系統的研究筆記匯入後缺乏「品質驗證」環節。RAGFlow 的可視化分塊檢視啟發了一個想法：新筆記匯入後，自動執行一次「自查詢」（用筆記標題搜尋自己），驗證該筆記是否能被有效檢索到。

具體建議：在 knowledge-query Skill 的匯入流程末尾加入驗證步驟——匯入後 30 秒用 hybrid search 查詢該筆記標題，若 score < 0.7 則記錄警告，表示該筆記的索引品質不佳。

啟示 5：Context Engine 三域統一的記憶架構

RAGFlow 將 Domain Knowledge、Tool Data、Conversation State 統一為 Context Engine 的框架，與本系統的三個資料層（digest-memory.json + SKILL_INDEX + research-registry.json）形成對映。目前三者各自獨立，未來可考慮統一的「上下文組裝」機制。

六、關鍵洞見摘要

  1. 「分塊品質 > 模型能力」：RAGFlow 團隊的核心信念——RAG 系統中文件理解和分塊品質的影響遠大於 LLM 選擇。可視化分塊驗證不是奢侈功能，而是生產級部署的必需品。
  1. 「簡單重疊分塊常勝複雜策略」：RAGFlow 的經驗反直覺地指出，離線處理允許放寬分塊要求，簡單的重疊分塊常優於複雜切割策略（後者可能破壞語義單元）。
  1. 「RAG 不死，它在演化」：企業對 RAG 的採用實際上在深化而非衰退。RAG 從特定的「檢索增強生成」模式演化為以「智能檢索」為核心能力的 Context Engine，這一趨勢不可逆轉。
  1. 「長上下文不取代 RAG」：直接灌入長上下文是「暴力策略」，資訊洪流和非線性計算開銷是其致命傷。正確做法是「檢索優先、長上下文收容」的協同策略。
  1. 「多模態 RAG 的瓶頸是工程而非理論」：張量索引儲存成本爆炸是核心挑戰——百萬頁圖片語料在 float32 精度下需約 TB 級儲存。解決方向包括張量量化壓縮和 Token 剪裁。
