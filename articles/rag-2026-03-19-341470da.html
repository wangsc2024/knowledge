二、品質缺口分析（7 項）

G1：Phase 2 結果檔案缺失無自動通知（Critical）

  • 問題：Phase 2 Agent 未產出結果檔時，PowerShell 自動補寫 done_cert 為 {status: FAILED, quality_score: 0}，但無主動告警
  • 影響：FSM 顯示多個 Phase 2 有 0 agents done，代表週期性結果缺失
  • 建議：Phase 3 Pre 新增自動告警步驟 — 若 verify-all 失敗 count > 3，發送 ntfy critical 告警

G2：自動任務公平性崩潰（Critical）

  • 問題：29 個自動任務中僅 6 個有執行記錄（20.7%），Fairness 指數 1.993
  • 根因假設：(a) 自動任務被 Todoist 任務搶佔 slot，(b) 無可處理項目判定過於寬鬆，(c) round-robin 狀態機未持久化
  • 建議：徹底檢查 run-todoist-agent-team.ps1 的 round-robin 邏輯，新增 fairness_audit 日誌

G3：品質追蹤無自動化管道（High）

  • 問題：quality-tracker SKILL 已定義完整流程，但未整合至 Phase 3
  • 缺口：需建立 done_cert → quality-trend.json 的 ETL 管道
  • 建議：Phase 3 後新增子步驟，自動萃取品質分數並計算移動平均

G4：Schema 漂移偵測未自動化（High）

  • 問題：agent-result-validator SKILL 已定義檢測邏輯，但未在 Phase 3 前執行
  • 風險：task_key 命名漂移導致靜默組裝失敗
  • 建議：在 done_cert verify-all 後新增 agent-result-validator 驗證步驟

G5：快取命中率低迷（Medium）

  • 問題：310 hits / 1498 total（20.7%），遠低於 40% 目標
  • 建議：使用 cache-optimizer SKILL 新增 --analyze-misses 模式

G6：I/O 單位成本過高（High）

  • 問題：平均每次工具呼叫產出 24.3KB，超標 4.86 倍
  • 建議：引入 context-budget-monitor SKILL，超過 3KB 自動委派子 Agent

G7：每日成功率未達標（High）

  • 問題：91 runs 中 15 failed（16.5%），失敗原因未自動分類
  • 建議：擴展 trace_analyzer.py 新增 --classify-failures 模式

三、外部最佳實踐參考

3.1 智慧品質閘門設計

根據 2026 年業界最佳實踐（Mabl、Anthropic、Google Cloud），品質閘門設計應從「統一規則」轉向「上下文感知」：

  1. 混合評估（Hybrid Evaluation）：自動化計分提供可重複性與規模，人工判斷捕捉自動化遺漏（Anthropic Engineering Blog）
  2. 多維度品質計分：將品質信號分解為可解讀的維度模組，組合為複合分數（arxiv 2603.04028 — Proof of Quality）
  3. 操作約束即品質指標：延遲、每任務成本、Token 效率、工具可靠性皆為一級評估目標（Mabl）
  4. 設計驗證為一級任務：將檢查變成獨立任務，讓每個子目標可量測（Anthropic Agent Evaluation）

3.2 Schema 驗證整合

  • 結構化輸出強制：JSON Schema 有效性 + 基於評分標準的合規性檢查（Codecademy LLM Eval Framework）
  • CI/CD 整合評估：每次相關變更都執行測試套件，用確定性檢查（JSON 有效性）+ 統計指標 + 模型評估器計分（Braintrust）
  • 部署後煙霧測試：驗證生產環境 Agent 輸出、日誌捕獲與回滾就緒度（Parasoft）

3.3 任務完成度評估

  • Task Completion：Agent 是否完成指定任務（Confident AI）
  • Tool Correctness：Agent 是否呼叫正確的工具（Google Cloud Agent Evaluation）
  • Error Recovery：Agent 是否正確處理錯誤與升級邏輯（Mabl CI/CD Agents）
