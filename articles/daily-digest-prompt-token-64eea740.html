Daily-Digest-Prompt Token 消耗效率與排程產能最佳化

七日資料驅動分析、空跑消除策略與多後端成本模型調優建議

研究日期：2026-03-10

類型：專案優化 / 深度思維 / 知識庫

適用對象：daily-digest-prompt 專案

---

一、現況診斷：七日 Token 消耗模式分析

1.1 消耗數據概覽（2026-03-03 至 2026-03-10）

| 日期 | 估算 Tokens | 工具呼叫次數 | 輸入字元 | 輸出字元 | 每次呼叫 Tokens |

|------|------------|------------|---------|---------|---------------|

| 03-03 | 224K | 1,547 | 786K | 0 | 145 |

| 03-04 | 3.75M | 4,530 | 2.32M | 10.8M | 828 |

| 03-05 | 85K | 152 | 88K | 211K | 562 |

| 03-06 | 2.27M | 1,507 | 669K | 7.26M | 1,504 |

| 03-07 | 16.3M | 4,879 | 2.77M | 54.4M | 3,347 |

| 03-08 | 8.46M | 3,673 | 1.88M | 27.7M | 2,303 |

| 03-09 | 30.5M | 2,438 | 1.04M | 105.6M | 12,507 |

| 03-10 | 1.41M | 890 | 328K | 4.61M | 1,587 |

1.2 關鍵發現

  1. 極端波動：最低 85K（03-05）至最高 30.5M（03-09），差距 358 倍。這表明系統缺乏穩定的每日 Token 預算控制。
  2. 輸出主導成本：03-09 的 output_chars 達 105.6M，而 input_chars 僅 1.04M（比值 101:1）。以業界輸出 token 計費約為輸入的 4-5 倍計算，輸出 token 佔總成本 >95%。
  3. 高密度日的效率反常：03-09 僅 2,438 次工具呼叫卻產出 30.5M tokens，平均每次呼叫 12,507 tokens — 這暗示某些自動任務產出了超長的研究報告或程式碼，但這些長文本不一定全部有效。
  4. 空跑浪費：run-fsm.json 顯示多個 todoist 執行的 Phase 2 為「0 agents done, 0 failed」（如 b2b09c51、f4823ed9），這些空跑仍消耗了完整的 Phase 1 查詢 token + Phase 3 組裝 token。

1.3 失敗模式統計

| 失敗類型 | 7 天總計 | 趨勢 |

|---------|---------|------|

| Timeout | 7 | 03-01~03-04 集中 |

| Phase Failure | 6 | 零散分佈 |

| API Error | 0 | 穩定 |

| Circuit Open | 0 | 穩定 |

每次 timeout 浪費的 token 約等於正常執行的 80-100%（等到超時才中斷），7 次 timeout 估計浪費 3-5M tokens。

二、五大優化策略

策略 1：空跑消除 — Phase 0 預判機制（預估節省 10-15%）

問題：當 Todoist 無待辦且自動任務已達上限時，系統仍啟動完整的 Phase 1 Agent 查詢（~2 分鐘、~30K tokens），然後才在 Phase 1 末尾判定為 idle。

建議：在 run-todoist-agent-team.ps1 的 Phase 0（PowerShell 層）新增快速預判：


# Phase 0.5: 快速預判（PowerShell 層，0 token 消耗）
$autoTasks = Get-Content 'context/auto-tasks-today.json' | ConvertFrom-Json
$allExhausted = Test-AllAutoTasksExhausted -Tasks $autoTasks -Limits $frequencyLimits
$lastPlanType = (Get-Content 'results/todoist-plan.json' -EA 0 | ConvertFrom-Json).plan_type

if ($allExhausted -and $lastPlanType -eq 'idle') {
    Write-Log "Phase 0.5: 預判 idle（自動任務全滿 + 上次 idle）→ 跳過本次執行"
    Set-FsmState -Phase 'skipped' -Detail 'pre-judge-idle'
    exit 0
}

預期效果：每日 44 次排程執行中，估計 10-15 次為空跑（深夜 02:00-07:00 + 任務清空後的尾時段），可節省 300K-450K tokens/日。

策略 2：輸出 Token 預算控制（預估節省 20-30%）

問題：model-selection.yaml 的 phase1_max_tokens 控制輸入，但未控制輸出。研究任務的 LLM 傾向產出 5,000-15,000 字的長文報告，但知識庫筆記有效閱讀長度通常在 1,500-3,000 字。

建議：

  1. 在 prompt 層加入輸出長度指引（零成本）：

- 研究報告：「核心分析 1,500-2,500 字，附錄 500 字以內」

- 系統分析：「摘要 300 字 + 行動項 5 條以內」

  1. 在 model-selection.yaml 新增 max_output_tokens：

```yaml

output_budgets:

research: 4000 # 研究任務輸出上限

system: 2000 # 系統維護輸出上限

creative: 6000 # 創意任務（遊戲/Podcast）可較高

default: 3000

```

  1. Claude CLI 參數：claude -p --max-tokens 4000（若 CLI 支援）

參考：業界最佳實踐指出，精簡 prompt 指引可節省 30-50% 輸出 token（來源：Elementor Engineers Medium 文章）。

策略 3：研究任務去重增強 — 成本感知（預估節省 15-20%）

問題：現有去重機制基於「主題完全相同才冷卻」，但 KB 中 AI 相關筆記已超過 46 篇，同主題不同角度的研究邊際效益遞減。

建議：

  1. 成本效益門檻：在 dedup-policy.yaml 新增：

```yaml

cost_aware:

daily_research_token_budget: 5000000 # 5M tokens/日

per_research_token_estimate: 800000 # 每次研究約 800K tokens

max_research_per_day: 6 # 基於預算計算

diminishing_return_threshold: 5 # 同 domain 超過 5 篇後減半頻率

```

  1. 動態研究頻率：當日 token 已超過 daily_research_token_budget 的 70% 時，剩餘研究任務改用 Haiku/OpenRouter free tier（成本降 5-10x）。

策略 4：排程密度智能化 — 從固定時刻到自適應（預估節省 10-15%）

問題：目前排程為固定 cron（每小時整點 single + 半點 team），即使 02:00-07:00 幾乎不會有新的 Todoist 任務。

建議：

  1. 分時段頻率：

```yaml

scheduling_zones:

peak: # 08:00-23:00：每 30 分鐘一次（維持現狀）

interval: 30

off_peak: # 00:00-07:00：每 2 小時一次（減少 75%）

interval: 120

idle: # 若連續 3 次 plan_type=idle → 暫停 4 小時

backoff: 240

```

  1. 自適應退避：連續 N 次 idle 後指數退避。參考 Jenkins 的 H 符號散列策略，避免所有任務同時啟動。

預期效果：凌晨 6 小時從 12 次降至 3 次，每次節省 ~30K tokens = 270K tokens/日。

策略 5：Prompt 快取最大化 — 共用前言靜態化（預估節省 5-10%）

問題：每次 claude -p 呼叫都重新載入 preamble.md、SKILL_INDEX.md、CLAUDE.md 等共用 context，這些內容在同一天內幾乎不變。

建議：

  1. Anthropic Prompt Caching：若 Claude API 支援 prompt caching（2026 年已普遍支援），將不變的系統 prompt 標記為 cacheable。官方數據顯示快取命中可減少 75-90% 的 input token 成本。
  1. 共用 context 預編譯：Phase 0 生成一份 cache/compiled-context.md，合併 preamble + SKILL_INDEX + 當日配置快照，減少每次 Agent 啟動時的重複讀取。
  1. Codex 端：利用 Codex 的 cached web search 模式（已部分實施），確保同日重複主題搜尋命中快取。

三、多後端成本模型現況與調優

3.1 現有後端配置評估

| 後端 | 適用任務 | 優勢 | 待優化 |

|------|---------|------|--------|

| claude_sonnet | Phase 1/3、預設 | 工具鏈完整 | Phase 1 idle 判定可提前 |

| claude_sonnet45 | 系統任務（6 個） | 推理強 | 部分任務（skill_audit）可降級 |

| claude_haiku | 緊急降級 | 便宜 | 可作為 Phase 1 查詢預設 |

| codex_exec (gpt-5.4) | 研究任務（11 個） | WebSearch 內建 | 輸出過長，需加 budget |

| codex_standard | 創意/QA/GitHub | 程式碼強 | 3 個任務全停用(daily_limit=0) |

| openrouter_free | git_push | 免費 | 可擴展至更多低複雜度任務 |

3.2 調優建議

  1. Phase 1 預設改用 Haiku：Phase 1 的本質是「Todoist 查詢 + 路由 + 計分」，邏輯相對固定，Haiku 足以勝任。估計節省 Phase 1 成本 60-70%。
  1. 停用任務清理：codex_standard 對應的 3 個任務（creative_game/qa_optimize/github_scout）全部 daily_limit=0，可從 task_rules 中移除，減少配置認知負擔。
  1. OpenRouter 擴展：log_audit（固定 Bash 序列）和 self_heal（讀狀態 + 簡單修復）的邏輯複雜度低，可從 claude_sonnet45 降級至 openrouter_standard，大幅降低成本。

五、監控與持續改進

5.1 建議新增度量

  1. Token 效率比：有效輸出字元 / 總輸出字元（有效 = 實際寫入 KB 或結果檔案的內容）
  2. 空跑率：idle 執行次數 / 總排程執行次數（目標 < 20%）
  3. 單次研究成本：研究任務 tokens / 研究任務次數（目標 < 800K）
  4. 後端使用比例：各後端佔比趨勢（目標：Claude Opus 佔比逐步降至 < 30%）

5.2 告警閾值建議


# config/token-alerts.yaml（建議新增）
alerts:
  daily_warn: 5000000     # 5M tokens/日
  daily_critical: 10000000 # 10M tokens/日
  per_run_warn: 500000     # 50 萬 tokens/次
  idle_rate_warn: 0.3      # 空跑率 > 30%
