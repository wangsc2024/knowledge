deep-research Skill 品質門檻機制與 web-research Skill 互補整合策略

研究日期：2026-03-21 | 類型：todoist_research（Skill 整合研究）

摘要

本研究分析 199-biotechnologies/claude-deep-research-skill（v2.3.1）的品質門檻機制，並探討其與既有 web-research Skill 的互補整合策略。deep-research 提供企業級 8 階段研究管線，包含自動化驗證迴圈、反幻覺協議與多角色紅隊審查；web-research 則提供輕量級 6 步驟框架，聚焦來源品質分級與 KB 匯入。兩者可形成分層互補架構：web-research 處理日常研究（2-5 分鐘），deep-research 處理重大決策研究（10-45 分鐘）。

---

一、品質門檻機制（Quality Gates）

1.1 雙腳本自動驗證

deep-research 內建兩個 Python 驗證腳本，形成完整品質閘門 [1]：

  • validate_report.py（9 項結構檢查）：Executive Summary 字數（200-400 字）、必要章節存在性、引用格式 [N] 一致性、參考文獻與引用匹配、無佔位符文字（TBD/TODO）、字數合理範圍（500-10000）、最少 10 個來源、無斷裂內部連結
  • verify_citations.py（引用驗證）：DOI 解析確認引用存在、標題/年份匹配偵測錯誤元資料、標記可疑條目（近期年份無 DOI、無 URL、驗證失敗）

1.2 驗證迴圈協議

每份報告生成後強制執行驗證迴圈 [1]：

  1. 執行 validate_report.py
  2. 執行 verify_citations.py
  3. 任一失敗 → 讀取錯誤、修正、重新驗證
  4. 最多 3 次重試，仍失敗則停止並報告

1.3 反疲勞協議（Anti-Fatigue Protocol）

每個章節完成前必須通過 5 項檢查 [1]：

  • 段落數 ≥ 3（主要章節）
  • 散文比例 ≥ 80%（限制條列式輸出）
  • 零佔位符
  • 豐富證據（具體數據、統計、引用）
  • 引用密度（重大主張在同一句中引用）

1.4 反幻覺協議（Anti-Hallucination Protocol）

六項核心規則 [1]：

  • 每個事實主張必須立即引用來源 [N]
  • 區分事實（來自來源）與綜合分析（自身推論）
  • 使用明確標記如「According to [1]...」
  • 推論須標記為「This suggests...」
  • 不確定時說「No sources found for X」而非捏造
  • 禁止模糊歸因（「Studies show...」「Experts believe...」）

二、8 階段管線的品質控制點

Phase 3: RETRIEVE — 並行搜尋與 FFS 品質閾值

依研究模式設定不同品質閾值 [2]：

| 模式 | 來源數量 | 平均可信度 | 時間上限 |

|------|----------|-----------|----------|

| Quick | 10+ | >60/100 | 2 分鐘 |

| Standard | 15+ | >60/100 | 5 分鐘 |

| Deep | 25+ | >70/100 | 10 分鐘 |

| UltraDeep | 30+ | >75/100 | 15 分鐘 |

Phase 4: TRIANGULATE — 交叉驗證

核心主張需 3+ 獨立來源；單一來源資訊須標記；記錄共識與爭議區域 [2]。

Phase 4.5: OUTLINE REFINEMENT — 動態大綱調整

根據實際證據調整研究方向（非基於猜測），避免「鎖定」問題。最多 50% 大綱重組 [2]。

Phase 6: CRITIQUE — 多角色紅隊審查

深度/超深度模式啟用 3 個批評角色 [2]：

  • Skeptical Practitioner：實務可信度
  • Adversarial Reviewer：同儕審查視角
  • Implementation Engineer：可執行性

若發現關鍵知識缺口，迴環回 Phase 3 補充搜尋（delta-queries，限時 3-5 分鐘）。

三、與 web-research Skill 的互補分析

3.1 功能對比

| 面向 | web-research | deep-research |

|------|-------------|---------------|

| 來源品質 | 4 級分級（A/B/C/D） | 0-100 可信度評分 |

| 驗證方式 | 手動交叉驗證 | 自動化腳本（9+N 檢查）|

| 引用管理 | 格式規範 | citation_manager.py 自動追蹤 |

| 去重 | registry + KB hybrid | 無（獨立運作）|

| KB 整合 | 原生匯入流程 | 無（產出至 ~/Documents/）|

| 耗時 | 2-5 分鐘 | 5-45 分鐘 |

| 報告格式 | KB 筆記 | Markdown + HTML + PDF |

3.2 互補整合策略

分層路由：依研究複雜度選擇 Skill [3][4]：

  • 日常研究（Todoist 自動任務）→ web-research（快速、KB 整合）
  • 重大決策研究（手動觸發）→ deep-research（深度、多格式輸出）

品質門檻借鏡：

  • web-research 可借鏡 deep-research 的反幻覺協議，在研究筆記中要求逐句引用
  • web-research 的 4 級分級可映射至 deep-research 的 0-100 分：A=90-100, B=70-89, C=50-69, D=30-49

工具複用：

  • source_evaluator.py（可信度評分）可獨立呼叫，強化 web-research 的來源評估
  • verify_citations.py（引用驗證）可用於 KB 筆記的品質審計

3.3 整合注意事項

  • deep-research 耗時較長（10-45 分鐘），不適合排程自動任務（可能超過 timeout） [5]
  • deep-research 產出至 ~/Documents/，需額外步驟匯入 KB
  • 兩個 Skill 的 description 關鍵字有重疊（"deep research"、"研究報告"），需注意路由優先級

---

四、Claude Code Skill 最佳實踐（官方建議）

根據 Claude Code 官方文件 [4]：

  • Skill 可透過 context: fork 在獨立子 Agent 中執行
  • allowed-tools 限制工具存取，確保安全
  • 支援 !command`` 語法注入動態上下文
  • 建議 SKILL.md 不超過 500 行，詳細內容移至 reference/ 目錄
  • deep-research 的架構（SKILL.md 薄層 + reference/ 詳細文件）符合此最佳實踐

---

五、結論與建議

  1. 品質門檻機制完善：9 項結構檢查 + 引用驗證 + 3 次重試迴圈 + 反幻覺協議，形成多層防護
  2. 互補而非替代：web-research 適合日常快速研究，deep-research 適合深度分析報告
  3. 可複用元件：source_evaluator.py 和 verify_citations.py 可獨立使用於其他研究流程
  4. 整合建議：在 routing.yaml 中新增 deep-research 觸發條件，限定手動觸發（disable-model-invocation: true）
