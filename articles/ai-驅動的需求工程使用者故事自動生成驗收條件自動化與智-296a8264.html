AI 驅動的需求工程：使用者故事自動生成、驗收條件自動化與智能需求分析

技術概述

AI 驅動的需求工程（AI-Driven Requirements Engineering, AI4RE）是將大型語言模型（LLM）、自然語言處理（NLP）和機器學習技術應用於軟體需求的擷取、分析、規格化與驗證的新興領域。根據 2026 年 Wiley 發表的系統性文獻回顧（涵蓋 238 篇研究），這個領域的發表量從 2022 年的 4 篇爆發式成長到 2024 年的 113 篇，顯示技術成熟度正在快速提升。核心價值在於將需求工程師從繁瑣的文件整理中解放，轉型為「AI 協調者」角色，專注於策略性決策與利害關係人溝通。

核心方法與工具

1. LLM 驅動的需求生成

模型類別代表應用場景
生成式模型GPT-4、Claude需求文件撰寫、使用者故事生成
程式碼導向Codex需求轉可執行程式碼
編碼器模型BERT需求文本結構化分析
混合系統Human-in-the-loop結合機器與人類專業判斷

關鍵統計：GPT 系列模型佔研究的 67.3%，研究焦點分布為分析（30.0%）、擷取（22.1%）、管理僅 6.8%（嚴重不足）。

2. NLP 需求品質分析

NLP 工具主要分為六大功能類別：

  1. 品質檢查：即時偵測模糊、不一致、複雜度過高
  2. 資訊擷取：從文件/郵件/會議記錄中提取需求
  3. 分類：自動將需求分為功能性/非功能性
  4. 建模：生成 UML 圖、資料流圖
  5. 追溯性：建立需求-設計-測試的雙向鏈結
  6. 搜尋與檢索：語義搜尋相關需求

實際工作流程（Step-by-Step）

階段一：需求擷取（Elicitation）

1. 收集原始輸入
   - 利害關係人訪談錄音/逐字稿
   - 會議記錄、郵件、即時通訊
   - 既有系統文件、競品分析

2. AI 處理
   - NLP 擷取關鍵需求陳述
   - LLM 摘要利害關係人意圖
   - 自動標記優先級與分類

3. 產出初稿
   - 使用者故事初稿（As a... I want... So that...）
   - 功能需求清單
   - 識別衝突與缺口

階段二：使用者故事生成

Prompt 範本（GenAI User Story Generation）：

## System Prompt
你是資深產品經理，擅長撰寫清晰的使用者故事。
根據以下需求描述，生成符合 INVEST 原則的使用者故事。

## 輸入
- 功能描述：[從擷取階段輸出]
- 目標用戶：[角色]
- 業務目標：[目的]

## 輸出格式
### 使用者故事
As a [角色], I want [功能] so that [價值]

### 驗收條件（Given-When-Then）
Given [前置條件]
When [使用者行為]
Then [預期結果]

### 非功能需求
- 效能：[回應時間限制]
- 安全：[認證/授權要求]

階段三：驗收條件自動化

RAGcceptance 方法（ASE 2025 最新研究）：

  1. 輸入多模態需求（文件 + UI 截圖）
  2. RAG 檢索相關歷史需求與驗收條件
  3. LLM 生成候選驗收條件
  4. 領域專家審核與修正

效果：整合多模態資訊顯著提升驗收條件的相關性、正確性與可理解性。實務評估確認可減少手動工作量，捕捉領域專家可能遺漏的細微需求。

階段四：品質驗證

驗證檢查清單：
□ 模糊性偵測（ambiguity detection）
□ 一致性驗證（cross-requirement consistency）
□ 完整性檢查（missing scenarios）
□ 可測試性評估（testability）
□ 合規性檢查（compliance with standards）
□ 衝突解析（conflict resolution）

階段五：追溯性管理

需求 → 設計規格 → 測試案例 → 原始碼
  ↕        ↕         ↕         ↕
AI 自動建立雙向追溯鏈結
自動影響分析（需求變更時識別受影響的下游產物）

程式碼/配置範例

Python + OpenAI 使用者故事生成

import openai

def generate_user_story(feature_description: str, persona: str) -> dict:
    """從功能描述自動生成使用者故事與驗收條件"""
    prompt = f"""
    根據以下功能描述，為 {persona} 生成使用者故事。
    
    功能描述：{feature_description}
    
    請輸出：
    1. 使用者故事（As a... I want... So that...）
    2. 3-5 個驗收條件（Given-When-Then 格式）
    3. 非功能需求（效能、安全、可用性）
    4. 邊界條件與例外情境
    """
    
    response = openai.chat.completions.create(
        model="gpt-4",
        messages=[
            {"role": "system", "content": "你是資深產品經理和需求分析師。"},
            {"role": "user", "content": prompt}
        ],
        temperature=0.3  # 低溫度確保一致性
    )
    return response.choices[0].message.content

Jira 整合自動化（GenAI → Jira User Story）

from jira import JIRA

def create_jira_story_from_ai(project_key: str, ai_output: dict):
    """將 AI 生成的使用者故事自動建立為 Jira Issue"""
    jira = JIRA(server='https://your-org.atlassian.net', token_auth='...')
    
    issue_dict = {
        'project': {'key': project_key},
        'summary': ai_output['story_title'],
        'description': ai_output['full_story'],
        'issuetype': {'name': 'Story'},
        'customfield_10001': ai_output['acceptance_criteria'],  # AC 欄位
        'priority': {'name': ai_output['priority']}
    }
    
    new_issue = jira.create_issue(fields=issue_dict)
    
    # 自動建立子任務（驗收條件 → 測試案例）
    for ac in ai_output['acceptance_criteria']:
        sub_task = {
            'project': {'key': project_key},
            'summary': f"Test: {ac['title']}",
            'issuetype': {'name': 'Sub-task'},
            'parent': {'key': new_issue.key}
        }
        jira.create_issue(fields=sub_task)
    
    return new_issue.key

關鍵數據：

  • OpenAI 模型在需求分類達 94% F1 Score
  • SpaCy 提供 83% 準確度 + 0.1 秒/查詢的即時分析
  • 大型需求文件（數千條）可在秒級完成品質掃描

與本專案的應用場景

Daily-Digest-Prompt 可受益面向

  1. Todoist 任務品質提升：

- 自動為 Todoist 任務生成結構化驗收條件 - 在 templates/shared/quality-gate.md 中加入 AI 品質檢查 - config/routing.yaml 可加入「需求品質分數」作為路由因子

  1. SKILL.md 規格化：

- 用 AI 驗證 SKILL.md 的 triggers/allowed-tools 一致性 - 自動生成 Skill 的使用場景（Given-When-Then 格式） - 提升 SKILL_INDEX.md 的路由精準度

  1. 自動任務模板改善：

- 為 templates/auto-tasks/*.md 自動生成驗收條件 - 確保每個自動任務模板都有明確的品質閘門

  1. 系統審查增強：

- 在 config/audit-scoring.yaml 中加入「需求品質」維度 - 檢查 specs/ 目錄下的規格文件是否符合 INVEST 原則

風險與注意事項

三大核心挑戰（2026 研究共識）

  1. 可再現性（66.8%）：同一 prompt 可能產生不同結果，需固定溫度參數與種子值
  2. 幻覺（63.4%）：LLM 可能生成看似合理但不正確的需求，必須人工審查
  3. 可解釋性（57.1%）：AI 為何生成特定驗收條件缺乏透明度

實務風險

  • 過度依賴：>90% 研究仍處於早期開發階段，僅 1.3% 達生產級整合
  • 領域知識不足：LLM 對特定業務領域的術語和規則理解有限
  • 合規風險：金融/醫療等受監管領域需額外驗證
  • 變更管理：AI 生成的需求可能與既有系統限制衝突

最佳實踐

  1. 永遠保持 Human-in-the-loop：AI 生成初稿，人類做最終決策
  2. 低溫度設定：生成需求時使用 temperature=0.2-0.3 確保一致性
  3. Few-shot Prompting：提供 2-3 個高品質使用者故事範例作為引導
  4. 迭代精修：第一次生成 → 品質檢查 → 修正 → 再驗證
  5. 版本控制：所有 AI 生成的需求都必須納入版本管理
