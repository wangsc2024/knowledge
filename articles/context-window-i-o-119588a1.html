Context Window 管理與 I/O 預算優化 — Agent 系統效能瓶頸診斷與壓縮策略實踐

產出日期：2026-04-01

研究系列：context-compression（foundation 階段）

來源品質：2個A級官方文件 + 1個B級技術文章

---

📋 摘要

Context Window 是 LLM 在單次請求中可處理的 token 總量，本質上是模型的「工作記憶」。隨著 Agent 對話延長，累積的歷史訊息將消耗 Context Window、提高成本並增加延遲。本報告整合 Claude API 官方文件、Microsoft Agent Framework 與 Maxim.AI 的最佳實踐，建立 Context Window 管理的基礎知識框架，涵蓋 token 計算方法、壓縮策略分類與效能診斷指標體系。

🎯 核心洞見

1. Context Window = 工作記憶預算

  • Token 計算：英文 ≈ 字數×1.3、程式碼 ≈ 行數×18、學術文本 ≈ 字數×1.4
  • 2026 年規模：128K-200K（主流）、1M（旗艦）、10M（前沿）

2. Context 膨脹三重成本

  • 成本：每次對話包含完整歷史，長對話指數增長
  • 效能：200K token 請求比 8K 慢 10-20 倍
  • 準確度：Context Rot（模型聲稱 200K 但在 130K 時效能下降）

3. 壓縮策略分類

| 策略 | 激進度 | 成本降幅 | 需要 LLM |

|------|--------|---------|----------|

| Tool Result Compaction | 低 | 20-40% | 否 |

| Summarization | 中 | 50-70% | 是 |

| Sliding Window | 高 | 70-85% | 否 |

| Truncation | 極高 | 85-94% | 否 |

4. Context Awareness（Claude 4.5+）

  • Token Budget 追蹤：注入總預算 → 每次工具呼叫後更新剩餘量
  • 模型可根據剩餘預算決定行為

5. 診斷指標體系

| 指標 | 健康閾值 | 告警閾值 |

|------|---------|----------|

| avg_io_per_call | < 8,000 | > 12,000 |

| context_usage_rate | < 60% | > 80% |

| token_efficiency | > 70% | < 50% |

🚀 建議行動

立即可行（1-3 天）

  1. 建立 token 監控（整合 Token Counting API）
  2. 實施 Tool Result Compaction

中期優化（1-2 週）

  1. 建立 Pipeline 壓縮策略（Tool → Summarization → Truncation）
  2. 配置 Context Budget Monitor

長期演進（1 個月）

  1. 系列研究推進（foundation → mechanism → application → optimization）
