Execution Trace Schema 實作指南 — Agent 工作流追蹤欄位、OpenTelemetry/OpenInference 映射與 daily-digest-prompt 落地模式

研究日期：2026-03-20

研究類型：技術研究

今日任務涉及技術：[品質閘門、RAG 知識庫、GitHub 研究工作流、Vite 同步、AI 模型蒸餾、資料契約、Execution Trace Schema、ntfy 通知審查、快取 TTL 治理]

本次研究主題：Execution Trace Schema 統一與 Agent 工作流結構化追蹤。

技術概述

Execution Trace Schema 的價值，不只是把 trace_id、span_id、status 寫進日誌，而是讓本地 JSONL、結果檔與觀測平台共用同一套追蹤語意。近年的實務方向很清楚：底層以 OpenTelemetry 做通用 tracing，AI 層再用 GenAI、MCP semantic conventions 與 OpenInference span kind 補足 agent、tool、retriever 等語意。對 daily-digest-prompt 這類多步驟、批次型、自動化專案而言，最重要的不是追求很多欄位，而是建立一個低基數、可關聯、可 flush、可回放的最小契約，讓 results/.json、logs/structured/.jsonl 與未來觀測後端都能沿著同一條 trace 主線工作。

核心概念與原理

三層追蹤模型

  1. 業務層：task_key、phase、tool_or_skill、artifacts。
  2. 追蹤層：trace_id、span_id、parent_span_id、status、duration_ms。
  3. AI 語意層：gen_ai.operation.name、gen_ai.agent.name、gen_ai.tool.name、openinference.span.kind、session.id。

OpenTelemetry 的角色

OpenTelemetry 已提供 GenAI spans、Agent spans、MCP spans 的語意規格。對本專案最實用的點：

  • Agent invocation 可用 gen_ai.operation.name=invoke_agent，並帶 gen_ai.agent.name。
  • 若跨程序或跨協定呼叫，使用 CLIENT span 比 INTERNAL 更清楚。
  • MCP tool call 可用 mcp.method.name=tools/call 搭配 gen_ai.tool.name，讓不同工具協定仍可放進同一追蹤樹。

OpenInference 的角色

OpenInference 要求每個 span 明示 openinference.span.kind，常見值有 AGENT、TOOL、CHAIN、RETRIEVER、RERANKER、LLM。這讓後續分析可以直接回答：哪個 phase 最慢、哪種工具最常失敗、哪類 KB 查詢最耗時。

建議欄位映射

| 本地欄位 | 建議標準欄位 |

|---|---|

| trace_id | OTel trace id |

| span_id | OTel span id |

| parent_span_id | parent span id |

| task_key | gen_ai.agent.name |

| phase | workflow.phase |

| tool_or_skill | gen_ai.tool.name 或 mcp.method.name |

| status | span status + error.type |

| duration_ms | span duration |

| artifacts | workflow.artifacts |

| run_id | session.id |

最小可行 span 切分

  • Task Root Span：單次任務執行。
  • Phase Span：去重、KB 查詢、Web 研究、KB 匯入、registry 更新。
  • Tool Span：每次 shell、KB API、WebSearch、WebFetch。
  • Optional AI Span：之後若接 LLM，再細分 LLM、RETRIEVER、RERANKER。

最佳實踐與常見陷阱

最佳實踐

  1. 先固定低基數 enum，例如 phase、tool_or_skill、status。
  2. 同時保留本地 JSONL 與標準欄位映射，不把觀測平台當唯一資料源。
  3. 在 span 建立時就放入採樣與路由需要的欄位。
  4. 程序結束前強制 flush；Phoenix 文件明確提醒 batch processor 若不 shutdown()，queued spans 可能丟失。
  5. 將 MCP tool call 與一般 execute_tool spans 對齊，避免兩套資料模型。

常見陷阱

  • 把完整 prompt 或隨機 URL 放入 span name，造成高基數污染。
  • 只有結果沒有 parent_span_id，無法還原 phase 與 tool 關係。
  • 錯誤只寫字串，不寫 status=ERROR 與 error.type。
  • PowerShell 平行工作流同時寫同一 trace 檔，造成交錯。
  • 用結果檔承載所有 span 細節，導致摘要與追蹤職責混在一起。

與本專案的關聯

daily-digest-prompt 已有 results/.json、logs/structured/.jsonl、state/* 等輸出，但它們的關聯鍵仍偏鬆散。若導入本研究建議，可先做三件事：

  1. 所有自動任務啟動時建立單一 trace_id 與 root span。
  2. 每個 Phase 寫入統一事件：phase_start、phase_end、tool_call、artifact_written。
  3. 將 KB 查詢 / 匯入與 Web 研究映射為 TOOL 或 RETRIEVER span。

直接收益是：

  • 更容易定位 running / interrupted / partial 的根因。
  • 品質閘門可以直接統計哪個 phase 最常失敗。
  • 未來要接 Phoenix 或 OTel Collector，不必重做欄位設計。

程式碼範例


{
  "trace_id": "7f5c5d3e4f6a4b8a9d0e1f2a3b4c5d6e",
  "span_id": "9ab12cd34ef56a78",
  "parent_span_id": "1aa22bb33cc44dd5",
  "session.id": "2026-03-20.todoist-auto-tech_research",
  "openinference.span.kind": "TOOL",
  "gen_ai.operation.name": "execute_tool",
  "gen_ai.tool.name": "knowledge-query",
  "workflow.phase": "kb_import",
  "status": "OK",
  "duration_ms": 842,
  "artifacts": ["results/todoist-auto-tech_research.json"]
}


$event = [ordered]@{
  trace_id = $traceId
  span_id = $spanId
  parent_span_id = $parentSpanId
  'session.id' = $runId
  'openinference.span.kind' = 'TOOL'
  'gen_ai.operation.name' = 'execute_tool'
  'gen_ai.tool.name' = 'knowledge-query'
  'workflow.phase' = 'kb_search'
  status = 'OK'
  duration_ms = 512
}
$event | ConvertTo-Json -Depth 5 -Compress | Add-Content 'logs/structured/execution-trace.jsonl'
