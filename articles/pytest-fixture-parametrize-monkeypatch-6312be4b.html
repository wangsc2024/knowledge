pytest 進階測試模式完整指南

技術概述

pytest 是 Python 生態系中最廣泛使用的測試框架，其 Fixture 系統、參數化機制和 Monkeypatch 工具構成了高效測試的三大支柱。本文系統性整理 pytest 的 10 大進階測試模式，從 Fixture 工廠到 CLI 工具測試，涵蓋生產級測試套件的完整最佳實踐。特別針對 Hook 系統（如 PreToolUse/PostToolUse stdin/stdout 管線測試）提供實用設計模式。

---

一、Fixture 工廠模式（Factory Fixture）

1.1 基本工廠：返回建構函數而非物件


import pytest

@pytest.fixture
def make_customer():
    """工廠 fixture：返回建構函數，允許測試自訂參數"""
    def make(
        first_name: str = "Cosmo",
        last_name: str = "Kramer",
        email: str = "test@example.com",
        **rest
    ):
        return Customer(
            first_name=first_name,
            last_name=last_name,
            email=email,
            **rest
        )
    return make

def test_customer_email(make_customer):
    customer = make_customer(email="custom@test.com")
    assert customer.email == "custom@test.com"

命名慣例：使用 make_ 前綴（如 make_customer）明確表示這是工廠而非直接物件。

1.2 進階工廠：使用 factory_boy 庫


from factory import Factory, Sequence, LazyAttribute

class UserFactory(Factory):
    class Meta:
        model = User
    
    username = Sequence(lambda n: f'user{n}')
    email = LazyAttribute(lambda obj: f'{obj.username}@example.com')
    active = True

適用場景：物件有複雜的預設值邏輯、需要序列化 ID、或有延遲計算的屬性。

二、Fixture 組合模式（Fixture Composition）

2.1 工廠依賴工廠


@pytest.fixture
def make_transaction(make_customer, make_sale):
    """組合多個工廠 fixture，提供合理預設值"""
    def make(transaction_id, customer=None, sale=None):
        if customer is None:
            customer = make_customer()
        if sale is None:
            sale = make_sale()
        return Transaction(
            transaction_id=transaction_id,
            customer=customer,
            sale=sale,
        )
    return make

核心價值：避免重複建構邏輯，允許測試覆寫任意參數，同時維持合理預設值。

2.2 conftest.py 組織策略


tests/
├── conftest.py          # 全域 fixture（DB 連線、API mock）
├── hooks/
│   ├── conftest.py      # Hook 專用 fixture（stdin/stdout mock）
│   ├── test_pre_bash_guard.py
│   └── test_post_tool_logger.py
└── skills/
    ├── conftest.py      # Skill 專用 fixture（API response mock）
    └── test_todoist.py

規則：

  • 每層 conftest.py 中的 fixture 對該目錄及子目錄自動可用，無需 import
  • 較具體的 conftest.py 可覆寫上層同名 fixture
  • 避免在 conftest.py 中放測試函數（只放 fixture 和 hook）

三、Parametrize 參數化策略

3.1 基本參數化


@pytest.mark.parametrize("test_input,expected", [
    ("3+5", 8),
    ("2+4", 6),
    pytest.param("6*9", 42, marks=pytest.mark.xfail),  # 預期失敗
])
def test_eval(test_input, expected):
    assert eval(test_input) == expected

3.2 堆疊參數化（笛卡爾積）


@pytest.mark.parametrize("x", [0, 1])
@pytest.mark.parametrize("y", [2, 3])
def test_foo(x, y):
    pass  # 產生 4 個測試：(0,2), (0,3), (1,2), (1,3)

典型應用：測試多資料庫引擎 × 多輸入值的矩陣組合。

3.3 Fixture 參數化


@pytest.fixture(params=["sqlite", "postgres", "mysql"])
def db_engine(request):
    engine = create_engine(request.param)
    yield engine
    engine.dispose()

def test_query(db_engine):  # 自動跑 3 次
    result = db_engine.execute("SELECT 1")
    assert result is not None

3.4 間接參數化（Indirect Parametrize）


@pytest.fixture
def numbers(request):
    return request.param

@pytest.mark.parametrize("numbers", [(2, 3), (5, 7)], indirect=True)
def test_addition(numbers):
    num1, num2 = numbers
    assert add(num1, num2) == num1 + num2

3.5 自訂 ID 提升可讀性


@pytest.mark.parametrize("cmd,expected", [
    pytest.param("rm -rf /", True, id="dangerous-delete"),
    pytest.param("ls -la", False, id="safe-list"),
    pytest.param("echo $SECRET", True, id="env-leak"),
], ids=str)
def test_bash_guard(cmd, expected):
    assert is_dangerous(cmd) == expected

3.6 動態參數化（pytest_generate_tests hook）


# conftest.py
def pytest_generate_tests(metafunc):
    if "rule_name" in metafunc.fixturenames:
        rules = load_yaml_rules("config/hook-rules.yaml")
        metafunc.parametrize("rule_name", [r["name"] for r in rules])

應用場景：從 YAML 配置動態生成測試案例，配置變更自動反映到測試。

四、Monkeypatch 最佳實踐

4.1 九大 Monkeypatch 方法

| 方法 | 用途 | 範例 |

|------|------|------|

| setattr(obj, name, value) | 替換屬性/方法 | mock API 呼叫 |

| delattr(obj, name) | 刪除屬性 | 測試缺失屬性行為 |

| setitem(mapping, name, value) | 修改字典值 | mock 配置字典 |

| delitem(obj, name) | 刪除字典項 | 測試缺失配置 |

| setenv(name, value) | 設定環境變數 | mock TOKEN/SECRET |

| delenv(name) | 刪除環境變數 | 測試缺失 env |

| syspath_prepend(path) | 修改 sys.path | 測試模組載入 |

| chdir(path) | 切換工作目錄 | 測試路徑邏輯 |

| context() | 限定 patch 範圍 | 保護 pytest 內部 |

4.2 雙向資料綁定模式（Two-Way Data Binding）


@pytest.fixture
def mock_send_email(monkeypatch):
    """透過閉包實現雙向資料流：mock 收集資料 + 測試可讀取"""
    sent_emails = []

    def _send_email(recipient, subject, body):
        sent_emails.append((recipient, subject, body))
        return True

    monkeypatch.setattr("myapp.email.send_email", _send_email)
    return sent_emails  # 測試可透過此列表驗證呼叫記錄

def test_order_confirmation(mock_send_email):
    process_order(order_id=123)
    assert len(mock_send_email) == 1
    assert mock_send_email[0][1] == "Order Confirmation"

4.3 狀態切換模式（State Toggle）


@pytest.fixture
def mock_api(monkeypatch):
    """可在測試中動態切換 API 行為"""
    state = {"should_fail": False, "call_count": 0}

    def _call_api(*args, **kwargs):
        state["call_count"] += 1
        if state["should_fail"]:
            raise ConnectionError("API unreachable")
        return {"status": "ok"}

    monkeypatch.setattr("myapp.api.call", _call_api)
    return state

def test_retry_on_failure(mock_api):
    mock_api["should_fail"] = True
    with pytest.raises(ConnectionError):
        call_with_retry(max_retries=3)
    assert mock_api["call_count"] == 3

4.4 禁止事項

  • 不要 patch 內建函數（open, compile 等）——可能破壞 pytest 內部
  • 用 monkeypatch.context() 包裹標準庫 patch——限定影響範圍
  • 不要在 monkeypatch 中修改全域可變狀態——改用工廠模式建立獨立實例

五、Hook 測試設計模式（stdin/stdout 管線）

本專案的 Claude Code Hooks（pre_bash_guard.py、post_tool_logger.py 等）採用 stdin→處理→stdout 的管線架構，需要特殊的測試模式。

5.1 stdin/stdout Mock 模式


import json
import sys
from io import StringIO

@pytest.fixture
def mock_stdin(monkeypatch):
    """工廠 fixture：模擬 stdin JSON 輸入"""
    def _set_input(data: dict):
        monkeypatch.setattr('sys.stdin', StringIO(json.dumps(data)))
    return _set_input

@pytest.fixture
def capture_stdout(monkeypatch):
    """捕獲 stdout JSON 輸出"""
    output = StringIO()
    monkeypatch.setattr('sys.stdout', output)
    return output

def test_pre_bash_guard_blocks_nul(mock_stdin, capture_stdout):
    mock_stdin({
        "tool_name": "Bash",
        "tool_input": {"command": "echo test > nul"}
    })
    from hooks.pre_bash_guard import main
    main()
    result = json.loads(capture_stdout.getvalue())
    assert result["decision"] == "block"

5.2 YAML 規則驅動測試


import yaml

@pytest.fixture
def hook_rules():
    """載入外部化規則配置"""
    with open("config/hook-rules.yaml") as f:
        return yaml.safe_load(f)

def test_all_bash_rules_have_tests(hook_rules):
    bash_rules = [r for r in hook_rules["bash"] if r.get("enabled", True)]
    for rule in bash_rules:
        assert rule["name"], f"Rule missing name: {rule}"
        assert rule["pattern"], f"Rule {rule['name']} missing pattern"

5.3 JSONL 日誌驗證模式


@pytest.fixture
def log_file(tmp_path):
    """建立臨時 JSONL 日誌檔"""
    log_path = tmp_path / "structured" / "tool-usage.jsonl"
    log_path.parent.mkdir(parents=True)
    return log_path

def test_post_tool_logger_writes_jsonl(log_file, monkeypatch):
    monkeypatch.setenv("STRUCTURED_LOG_DIR", str(log_file.parent))
    # ... 執行 logger ...
    lines = log_file.read_text().strip().split("\n")
    for line in lines:
        record = json.loads(line)
        assert "ts" in record
        assert "tool" in record
        assert "tags" in record

七、Fixture 作用域（Scope）策略


@pytest.fixture(scope="session")
def db_connection():
    """整個測試 session 只建立一次"""
    conn = create_connection()
    yield conn
    conn.close()

@pytest.fixture(scope="module")
def api_client(db_connection):
    """每個測試模組建立一次"""
    return APIClient(db_connection)

@pytest.fixture  # scope="function"（預設）
def clean_state(db_connection):
    """每個測試函數重置狀態"""
    db_connection.rollback()
    yield
    db_connection.rollback()

作用域層級：session > package > module > class > function

規則：高作用域 fixture 不能依賴低作用域 fixture（session 不能用 function 級 fixture）。

八、與本專案的關聯

本專案（daily-digest-prompt）有 496 個 pytest 測試，覆蓋 7 個 Hook 模組 + 2 個 Skill 模組。以下是直接可應用的模式：

8.1 已使用的模式

  • stdin/stdout mock：所有 Hook 測試使用 StringIO mock stdin 輸入
  • monkeypatch.setenv：環境變數（HOOK_SECURITY_PRESET）測試
  • tmp_path：JSONL 日誌輪替測試
  • parametrize：pre_bash_guard 的 144 個測試案例大量使用參數化

8.2 可引入的改進

  1. 工廠 fixture 統一 Hook 輸入建構：目前各測試檔獨立建構 stdin JSON，可抽取 make_hook_input 工廠到 tests/hooks/conftest.py
  2. 動態參數化從 hook-rules.yaml 生成：使用 pytest_generate_tests 從 YAML 自動產生規則測試，配置變更即刻反映
  3. 狀態切換模式用於 Circuit Breaker 測試：取代手動修改 state dict
  4. caplog 替代部分 stdout 捕獲：對 logging 模組的輸出驗證更精確

8.3 效能優化建議

  • 將 YAML 規則載入提升至 scope="session" — 496 個測試只載入一次
  • 使用 pytest-xdist 並行執行（pytest -n auto）— 預估可將 496 個測試從 ~30 秒降至 ~10 秒
  • 使用 @pytest.mark.slow 標記長時間測試，CI 可選擇性跳過

九、常見陷阱與解法

| 陷阱 | 症狀 | 解法 |

|------|------|------|

| Fixture 作用域不匹配 | ScopeMismatch 錯誤 | 確保高作用域不依賴低作用域 |

| Monkeypatch 全域狀態洩漏 | 測試順序依賴 | 使用工廠模式建立獨立實例 |

| Parametrize 過度組合 | 測試爆炸（千個案例） | 精選代表性案例 + boundary testing |

| conftest.py 過大 | 難以維護 | 按模組拆分到子目錄 conftest.py |

| Fixture 清理不完整 | 資源洩漏 | 用 yield + finally 或 addfinalizer |

| Mock 過度 | 測試脆弱、假綠燈 | 只 mock 外部邊界（API、檔案系統） |
