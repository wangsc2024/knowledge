一、核心問題診斷

本次優化聚焦三個既有研究尚未深入的系統性問題：

| 問題維度 | 嚴重度 | 現況指標 | 目標指標 |

|----------|--------|----------|----------|

| 狀態檔案無界增長 | 🔴 高 | research-registry 101 entries (~56KB) | ≤30 entries (<15KB) |

| 配置交叉引用無驗證 | 🟡 中 | 0 項跨檔完整性檢查 | 覆蓋 3 組關鍵引用 |

| 平均 I/O 過大 | 🔴 高 | 43KB/次 (目標 5KB) | ≤10KB |

二、狀態檔案生命週期分析

2.1 七類 JSON 狀態檔現況

| 檔案 | 用途 | 寫入者 | 清理機制 | 問題 |

|------|------|--------|----------|------|

| context/research-registry.json | 研究去重 | Agent | 7 天 retention（宣告但未強制） | 101 entries, 56KB, 持續膨脹 |

| state/run-fsm.json | FSM 執行狀態 | PS 腳本 | max_entries=20（已實作） | 正常 |

| state/token-usage.json | Token 消耗 | PS 腳本 | 7 天滾動 | 正常 |

| state/failure-stats.json | 失敗統計 | PS 腳本 | 無 TTL | 累積增長，無自動清理 |

| context/behavior-patterns.json | 行為模式 | 未實作 | N/A | 空檔案，功能從未啟用 |

| state/config-metrics.json | 配置度量 | analyze-config.ps1 | 30 天歷史 | 正常 |

| context/research-series.json | 長期系列追蹤 | Agent | 永久保留（設計如此） | 正常 |

2.2 根因分析

research-registry.json 的 retention_days: 7 定義在 dedup-policy.yaml，但清理邏輯散佈在各 prompt 模板的「寫入後」步驟中。問題在於：

  1. 清理依賴 Agent 自律：每次研究任務結束時，prompt 要求「移除超過 7 天的舊 entry」，但 LLM 可能跳過
  2. 無機器強制層：不像 Hook 有 Python 攔截，registry 清理完全靠 prompt 指令
  3. summary 欄位未同步：目前 summary 為空 {}，失去快速預檢功能，每次都要遍歷全部 entries

2.3 建議修正

P0-A：registry 清理機器化


新增 tools/cleanup-registry.py：
  - 讀取 dedup-policy.yaml 的 retention_days
  - 自動刪除超過 7 天的 entries
  - 重新計算 summary 欄位（total, by_type, recent_3d_topics）
  - 由 Phase 0（run-agent-team.ps1、run-todoist-agent-team.ps1）呼叫
  - 預期效果：registry 穩定在 30 entries 以內

P0-B：failure-stats.json 加入 TTL


state/failure-stats.json 加入 30 天滾動清理：
  - 在 check-health.ps1 或 Phase 0 執行
  - 保留最近 30 天記錄，超過的自動移除
  - 預期效果：檔案大小穩定

P1-A：behavior-patterns.json 決策


二擇一：
  選項 A（啟用）：在 post_tool_logger.py 新增模式偵測，每 session 結束時寫入
  選項 B（移除）：刪除空檔案，從 system-insight 移除相關指標
  建議：選項 B（YAGNI 原則，目前無具體使用場景）

三、配置交叉完整性驗證

3.1 三組關鍵跨檔引用（目前無驗證）

| 來源檔 | 引用目標 | 引用類型 | 風險 |

|--------|----------|----------|------|

| frequency-limits.yaml 的 template 欄位 | templates/auto-tasks/*.md | 檔案路徑 | 模板不存在→靜默失敗 |

| routing.yaml 的 skill 名稱 | skills/SKILL_INDEX.md 的 Skill 列表 | 名稱比對 | Skill 改名→路由斷裂 |

| prompts/team/todoist-auto-.md 的輸出檔名 | Phase 3 assemble 預期的 results/todoist-auto-.json | 命名規範 | 命名不一致→Phase 3 找不到結果 |

3.2 歷史教訓

本專案已三次因跨檔引用不一致導致生產故障：

  1. 2026-02-24：prompt 重命名但 $dedicatedPrompts hardcoded 未更新
  2. 2026-02-18：3 個 auto-task prompt 輸出檔名用舊規範（缺 auto- 前綴）
  3. routing.yaml 標籤覆寫：Claude Code/GitHub 被 code-task 優先級搶贏 research-task

3.3 建議修正

P0-C：擴展 validate_config.py 新增 3 項跨檔檢查


# 新增驗證項目：
1. template_path_check：
   - 遍歷 frequency-limits.yaml 每個 task 的 template 欄位
   - 檢查對應 templates/auto-tasks/{template}.md 是否存在
   - 失敗→報告缺失模板

2. skill_name_check：
   - 解析 routing.yaml 的所有 skill 引用
   - 比對 skills/SKILL_INDEX.md 的 Skill 列表
   - 失敗→報告未註冊 Skill

3. prompt_output_check：
   - 掃描 prompts/team/todoist-auto-*.md 的輸出檔名定義
   - 驗證命名規範一致性（auto- 前綴）
   - 失敗→報告不一致的檔案

預期效果：未來配置修改後，pre-commit hook 自動捕捉跨檔引用斷裂，避免生產故障。

四、快取命中率錯位修正

4.1 現況分析

| API | TTL | 排程頻率 | 理論命中率 | 實際命中率 |

|-----|-----|----------|-----------|------------|

| Todoist | 45 min | 每 30 min | ~50% | 偏低 |

| Pingtung News | 360 min | 每 8h (3次/日) | ~62% | 偏低 |

| HackerNews | 180 min | 每 8h (3次/日) | ~37% | 偏低 |

| Knowledge | 60 min | 每次研究 | ~15% | 極低 |

| Gmail | 360 min | 按需 | N/A | N/A |

聚合命中率 13.92% 的主因：

  1. Todoist 雙軌排程：整點（single）+ 半點（team），但兩軌各自維護快取，互不共享
  2. Knowledge API 頻繁呼叫：每個自動任務都做 hybrid search 去重 + 匯入，TTL 60 min 不夠覆蓋密集研究期
  3. Phase 0 快取預計算只寫 cache/status.json，但各 Agent 不一定讀取

4.2 建議修正

P1-B：快取共享機制


1. Todoist 快取統一：
   - single 模式和 team 模式共用同一份 cache/todoist.json
   - Phase 0 預取寫入，Phase 1 直接讀取
   - 預期命中率提升至 ~65%

2. Knowledge API TTL 延長：
   - 去重查詢結果快取至 cache/kb-dedup-{topic_hash}.json
   - TTL 延長至 120 min（研究任務間隔通常 > 30 min）
   - 預期命中率提升至 ~30%

3. 快取狀態標籤強制：
   - 在 post_tool_logger.py 加入快取命中/未命中標籤
   - 每日 check-health.ps1 報告分 API 命中率
   - 為 TTL 調優提供數據支撐

五、自動任務公平性修正

5.1 偏斜根因

21 個自動任務的 daily_limit 分配不均：

  • 佛學研究 4 個任務 × 3 次/日 = 12 次（佔 27%）
  • AI/技術研究 6 個任務 × 合計 17 次/日 = 17 次（佔 38%）
  • 系統維護 3 個任務 × 合計 5 次/日 = 5 次（佔 11%）
  • 其他 8 個任務共 11 次/日 = 11 次（佔 24%）

暫停任務（creative_game_optimize=0, qa_optimize=0, unsloth=0）拉大標準差。

5.2 建議修正

P2-A：正式移除或歸檔暫停任務


1. daily_limit=0 的任務從 round-robin 池移除（不參與計算）
2. 在 frequency-limits.yaml 加入 status: active/paused/archived 欄位
3. round-robin 只考慮 status=active 的任務
4. 預期公平性 σ/μ 從 3.16 降至 ~1.2

P2-B：I/O 優化


1. research-registry.json 讀取改用 summary-only 模式（已設計但 summary 為空）
   → P0-A 修復後 summary 恢復，I/O 從 56KB 降至 <1KB
2. 大型 prompt 檔案拆分：preamble.md (24 行) 已精簡，但 quality-gate 等可進一步壓縮
3. Glob/Read 加 limit 參數：research-registry 讀取時用 offset+limit

八、風險與限制

  1. Registry 清理可能影響去重準確度：7 天 window 縮短可能漏掉跨週重複，但 KB hybrid search 作為第二層防護可彌補
  2. 快取共享需要鎖定機制：Todoist 雙軌共用快取可能出現競態條件，建議用檔案鎖（PowerShell [System.IO.File]::Open 排他鎖）
  3. 跨檔驗證新增維護成本：validate_config.py 測試案例需從 37 增至 ~50，但一次性投入後長期受益
