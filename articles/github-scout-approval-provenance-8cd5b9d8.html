GitHub Scout：approval provenance 與雙層 tool boundary 契約 - 2026-04-03

專案觀測摘要

  • browser-use/browser-use：74.2k stars；以瀏覽器自動化為主，工具邊界同時碰到瀏覽器 DOM/API 與可選遠端執行環境。
  • OpenHands/OpenHands：65.6k stars；覆蓋 CLI、Local GUI、Cloud 與企業部署，具 RBAC/permissions 敘事，適合觀察 operator override 入口。
  • lastmile-ai/mcp-agent：7.8k stars；聚焦 MCP orchestration、human input、Temporal durable execution，適合借鏡 override 與 trace contract。
  • guardrails-ai/guardrails：6.3k stars；聚焦輸入/輸出 validator 與 structured data guardrails，屬內容層 guardrail surface。
  • e2b-dev/E2B：以隔離 sandbox 為核心，代表 execution environment boundary 的典型設計。

改進提案

  1. P0 / 低風險：把 github_scout 的 tool_boundary 契約收斂為結構化物件，單層用 {summary}，雙層用 {execution_environment_boundary, tool_api_boundary}。
  2. P1 / 高風險：為 tool_boundary 與 operator_override 建立 approval provenance event envelope，串接 trace_id、operation_name、error correlation 與後續 logger sidecar。

審查結論

  • P0 採納：可行性高、變更僅限 config/template/prompt/skill 契約層，可回滾。
  • P1 保留為 plan_ready：需人工審核，因為涉及 logger/error 契約與下游解析相容性。

已落實項目

  • 已更新 config/github-scout-review-policy.yaml，新增 tool_boundary_normalized_shape 與物件化建議。
  • 已更新 templates/auto-tasks/github-scout.md、prompts/team/todoist-auto-github_scout.md、skills/github-scout/SKILL.md，讓雙層 tool_boundary 有一致輸出範例。

待人工審核項目
