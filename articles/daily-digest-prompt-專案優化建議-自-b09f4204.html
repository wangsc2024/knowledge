Daily-Digest-Prompt 專案優化建議

自動任務投資回報率（ROI）與知識密度提升策略

研究日期：2026-02-23 任務 ID：6g2FfvWmxqhCWx85 分析焦點：從 200 筆執行記錄與 142 筆研究註冊表反推自動任務的實際產出效率，提出精準優化方案 差異化：前期報告覆蓋架構（2/15）、可觀測性（2/17）、排程效能（2/18）、KB 生命週期（2/19），本文首次從 ROI 經濟學角度 分析

1.3 失敗模式分析

5 次失敗全部集中在兩個 Agent：

  • todoist-team：4 次 Phase 1 失敗（查詢階段超時或 API 不可用）
  • daily-digest：1 次 all attempts failed

Phase 1 是系統最脆弱的環節 — 佔失敗總數的 80%。

二、問題識別：三大結構性瓶頸

2.1 問題一：自動任務 Token 消耗不成比例

現象：18 個自動任務佔用每日 45 次執行配額中的絕大部分（Todoist 真實待辦約 12 筆，~4 runs 完成），但知識庫產出的邊際效用遞減。

數據佐證：

  • 142 筆研究註冊表中，2/16 一天產出 30+ 筆，相當密集
  • KB 412 筆中，佛學（~25%）+ AI（~20%）佔 45%，主題集中度高
  • 研究去重機制雖然有效避免完全重複，但「換角度研究同主題」仍會產生大量近似內容
  • todoist-team 平均耗時 698s（~12 分鐘），每次消耗可觀的 API Token

根因：自動任務的「產量驅動」設計（daily_limit 累加 = 45 次/日）沒有「品質閘門」把關單篇研究的知識密度。系統追求的是「每天產出 N 篇」而非「每篇都有高密度新知識」。

2.2 問題二：Todoist-team Phase 1 脆弱性

現象：4 次 Phase 1 失敗佔總失敗數 80%，且全部為 todoist-team。

數據佐證：

  • Phase 1 timeout 已從 300s 提升至 420s，仍偶爾不足
  • 單一模式 todoist 反而 100% 成功（89/89），暗示 team 模式的 Start-Job 並行開銷是額外風險因素
  • Phase 1 需要：API 呼叫 + 路由 + 計分 + 寫出 N 個 task prompt 檔案，步驟多且串行

根因：Phase 1 是「單點故障」（Single Point of Failure），任何一步失敗都會導致整個 team 模式執行失敗。而 Phase 1 的 420s timeout 涵蓋了太多串行步驟。

2.3 問題三：記憶體系碎片化

現象：系統有 4 個獨立的狀態/記憶檔案，更新邏輯分散。

  • context/digest-memory.json：Agent 獨佔寫入，schema_version=2
  • context/auto-tasks-today.json：頻率追蹤，每日歸零
  • context/research-registry.json：研究去重，7 天滾動
  • state/scheduler-state.json：PowerShell 獨佔，200 筆上限

問題：

  1. digest-memory.json 的 skill_usage 數據停留在單次 session，不會累積跨 session 統計
  2. trends 陣列只有 4 天，缺乏長期趨勢視角
  3. 沒有機制將 scheduler-state.json 的 200 筆執行數據與 research-registry.json 的 142 筆研究數據關聯分析
  4. 知識庫 412 筆的主題分佈信息無處持久化

三、優化方案

方案一：引入知識密度評分機制（Knowledge Density Score）

目標：從「產量驅動」轉向「價值驅動」，每篇研究匯入 KB 前先評估知識密度。

設計：

在 config/scoring.yaml 或新建 config/knowledge-density.yaml 中定義：

knowledge_density:
  version: 1
  # 評估維度（匯入前自評）
  dimensions:
    novelty: 30        # 新穎度（KB 中是否已有類似內容）
    actionability: 25   # 可行性（是否有具體可執行的建議/步驟）
    depth: 25           # 深度（是否超越表面概述，有原理分析）
    cross_reference: 20 # 交叉引用（是否連結到已有知識）
  
  # 門檻
  minimum_score: 60     # 低於此分數建議合併到已有筆記而非新建
  high_quality: 80      # 高於此分數為高品質產出
  
  # 匯入策略
  below_minimum: "merge_or_skip"  # 合併到最相關的既有筆記，或跳過
  above_minimum: "create_new"     # 建立新筆記

實施步驟：

  1. 在 templates/sub-agent/research-task.md 的品質閘門中加入密度評分
  2. 在 templates/auto-tasks/ 的 8 個研究模板中加入同樣機制
  3. post_tool_logger.py 記錄每次匯入的密度分數，供趨勢分析

預期效果：KB 新增筆記的平均品質提升 30%+，減少低密度重複內容。

方案二：Phase 1 分階段容錯（Staged Fallback）

目標：將 Phase 1 的串行步驟拆分為可獨立容錯的子階段，降低整體失敗率。

設計：

Phase 1 現行流程（串行，420s 共享 timeout）：
  1. Todoist API 查詢
  2. 三層路由篩選
  3. TaskSense 計分排序
  4. 寫出 plan.json + N 個 task prompt 檔案

Phase 1 優化後（分階段 timeout + checkpoint）：
  1a. Todoist API 查詢（120s timeout）
      → 失敗 → 使用快取降級
      → 成功 → checkpoint: todoist-raw.json
  1b. 路由 + 計分（60s timeout）
      → 失敗 → 使用上次的路由結果
      → 成功 → checkpoint: routing-result.json
  1c. 規劃 + 寫出檔案（180s timeout）
      → 失敗 → 減少任務數（max_tasks 3 → 1）再試
      → 成功 → plan.json ready

實施步驟：

  1. 在 prompts/team/todoist-query.md 中加入 checkpoint 寫出邏輯
  2. 在 run-todoist-agent-team.ps1 中加入 Phase 1 的子階段偵測
  3. 新增 results/phase1-checkpoint.json 作為中間狀態

預期效果：Phase 1 失敗率從 4.3%（4/93）降至 <1%，因為每個子階段都有獨立的降級路徑。

方案四：自動任務能量配比動態調整

目標：根據已有 KB 的主題分佈，動態調整各群組的 daily_limit。

設計：

在 config/frequency-limits.yaml 中新增動態調整區塊：

dynamic_rebalancing:
  enabled: true
  trigger: "system-insight auto-task"
  rules:
    - condition: "KB 同主題筆記 > 50 篇"
      action: "該主題 daily_limit -= 1（最低 1）"
    - condition: "KB 同主題筆記 < 10 篇"
      action: "該主題 daily_limit += 1（最高原始值 × 1.5）"
    - condition: "某研究方向 7 天無新 entry"
      action: "優先排入下次輪轉"

預期效果：避免佛學研究（目前 25.5%）持續增長而其他方向（遊戲 5.2%、安全 2.5%）長期不足。

方案五：研究品質回饋迴路

目標：建立「研究 → 匯入 → 評估 → 下次研究調整」的閉環。

設計：

當前流程（開環）：
  研究 → 匯入 KB → 結束

優化後（閉環）：
  研究 → 密度評分 → 匯入 KB → 7 天後自動評估被引用次數
  → 若引用率 > 30% → 標記為高價值，同主題繼續深入
  → 若引用率 = 0% → 標記為低價值，減少同方向研究

在 context/research-registry.json 的每個 entry 中新增欄位：

{
  "date": "2026-02-23",
  "topic": "...",
  "density_score": 78,
  "cited_count_7d": 0,
  "value_rating": "pending"
}

實施步驟：

  1. system-insight 自動任務每日掃描 KB 搜尋日誌，統計各筆記被 hybrid search 匹配的次數
  2. 將引用次數回寫到 research-registry.json
  3. 下次同類研究時，優先選擇「高引用率方向」深入，減少「零引用方向」

預期效果：形成正向增強迴路，研究方向自動向「最有價值的知識」收斂。

五、預期整體效果

  1. Phase 1 可靠性：失敗率從 4.3% 降至 <1%（~3x 改善）
  2. KB 知識密度：新筆記平均品質分數提升 30%+
  3. 主題多樣性：Shannon 多樣性指數從目前 ~0.75 提升至 >0.85
  4. 資源效率：減少低價值研究的 Token 消耗，同等配額下產出更高品質知識
  5. 系統自覺性：統一儀表板讓系統具備「元認知」能力，能主動識別並調整自身行為
