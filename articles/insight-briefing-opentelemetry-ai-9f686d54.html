可觀測性工程進階：OpenTelemetry 分散式追蹤、AI 管線 Span 設計與成本歸因實踐

日期：2026-05-28

類別：軟體工程方法論 / 可觀測性

來源品質：A 級 3 篇（opentelemetry.io 官方、uptrace.dev、redhat.com），B 級 2 篇

摘要

OpenTelemetry 的 GenAI 語義約定（Semantic Conventions）在 2026 Q1 進入穩定版，為 LLM 呼叫的 token 計數、成本歸因、多 Agent 追蹤提供了跨廠商統一標準。關鍵數據：output tokens 成本為 input tokens 的 4–8 倍，cache_read tokens 已納入標準屬性集，auto-instrumentation 的 OTel overhead 不足 1ms/call。

核心洞見

洞見 1：GenAI 語義約定 2026 Q1 穩定版——標準屬性集已確立

必填：gen_ai.operation.name、gen_ai.provider.name

推薦（token 計數）：gen_ai.usage.input_tokens、gen_ai.usage.output_tokens、gen_ai.usage.cache_read.input_tokens、gen_ai.usage.cache_creation.input_tokens

Span 命名：{gen_ai.operation.name} {gen_ai.request.model}

洞見 2：成本歸因公式與 Output Token 主導性

Cost (USD) = (input_tokens × input_rate + output_tokens × output_rate) / 1,000,000

2026 年 4 月定價：Claude Sonnet 4.6 $3.00/$15.00（5×），GPT-5 $1.25/$10.00（8×）

Output tokens 成本為 input tokens 的 4–8 倍，completion 長度是主要成本驅動。

洞見 3：Span 層級設計——父子結構與 Context Propagation

父 span（agent.run）聚合總成本，子 span 記錄各步驟成本。W3C traceparent header 跨服務邊界傳遞 trace context。Auto-instrumentation overhead < 1ms/call。

洞見 4：隱私優先的 Prompt 捕捉設計

Prompt 預設不捕捉（privacy-first），改用 Span Events（可在 Collector 層過濾）而非 Span Attributes。內容上限 500–1000 字元。

洞見 5：四大成本優化信號

  1. 模型降級空間：output_tokens 少但成本高
  2. Prompt 膨脹：input_tokens 單調增長
  3. 重試循環：同 operation 多次 LLM 呼叫
  4. 對話歷史增長：per turn input_tokens 線性增加

建議行動（本系統）

  1. 近期：post_tool_logger.py 拆分 cache_read_tokens 與 billable_tokens
  2. 中期：autonomous_harness.py 加入 phase_cost_usd 至 spans 結構
  3. 長期：評估整合 opentelemetry-instrumentation-anthropic 套件
