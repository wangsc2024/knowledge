Harness Engineering：AI Agent 系統優化完整指南

研究日期

2026-04-19

研究來源

  • Martin Fowler - Harness Engineering for Coding Agents
  • Anthropic & OpenAI Best Practices
  • LangChain - Improving Deep Agents

---

核心定義

Agent = Model + Harness

"Harness" 指的是 AI Agent 除了模型本身之外的所有系統元件。Harness Engineering 是設計和實現以下系統的學科：

  • 約束 (Constrain)：限制 Agent 可以執行的操作
  • 指引 (Inform)：告訴 Agent 應該如何執行
  • 驗證 (Verify)：確認 Agent 是否正確執行
  • 修正 (Correct)：當 Agent 出錯時進行糾正

核心原則

"用機制強制品質，而非提示"
The combination of linters, Hooks, tests, and ADRs compounds.

底層模型的重要性低於其周圍的系統。LangChain 的實證證明了這點：他們的 coding agent 從 52.8% 提升到 66.5% (Terminal Bench 2.0)，從 Top 30 躍升至 Top 5，完全沒有改動模型本身。

---

兩大控制機制

1. Feedforward Controls（前饋控制 / 指引）

預防性機制：在輸出生成前防止不期望的結果。

實作方式：

  • 架構文件 (Architecture Documentation)
  • 編碼慣例 (Coding Conventions)
  • 引導指令 (Bootstrap Instructions)
  • AGENTS.md / CLAUDE.md 等指引文件

目標：在第一次嘗試時就提高正確率。

2. Feedback Controls（回饋控制 / 感測器）

觀察性機制：在生成後觸發自我修正。

兩種執行模式：

  • 計算型 (Computational)：確定性、毫秒至秒級執行

- 測試 (Tests)

- Linters

- 型別檢查器 (Type Checkers)

- 格式化工具 (Formatters)

  • 推論型 (Inferential)：語義分析、較慢但更豐富

- LLM as Judge

- Code Review Agents

- 語義驗證

策略原則：計算型控制適合持續使用；推論型適合高風險階段審查。

---

三大調控類別

1. 可維護性 Harness (Maintainability Harness)

  • 程式碼品質與結構
  • 現有工具最成熟的領域
  • 涵蓋：Linters、Formatters、Style Guides

2. 架構適配性 Harness (Architecture Fitness Harness)

  • 效能與設計特性
  • 涵蓋：Fitness Functions、Architecture Decision Records (ADRs)
  • 確保系統符合架構願景

3. 行為正確性 Harness (Behaviour Harness)

  • 功能正確性驗證
  • 仍在發展有效方法中
  • 涵蓋：Integration Tests、Mutation Testing、E2E Tests

---

多 Agent 專業化架構

角色分離原則

不要將所有工作壓在單一 Agent，而應分散到專門化 Agent：

| 角色 | 職責 | 關鍵能力 |

|------|------|----------|

| Planner | 將 prompts 展開為完整規格 | 需求分析、任務拆解 |

| Generator/Builder | 實作功能 | 程式碼生成、API 呼叫 |

| Evaluator/QA | 測試輸出 | 驗證、測試、品質評估 |

重要警告：

"The evaluator is critical because agents consistently rate their own work too generously."
Evaluator 至關重要，因為 Agent 持續性地過度評估自己的工作。

---

策略性實作原則

1. Keep Quality Left（品質左移）

定義：將檢查分散到開發生命週期的早期階段。

實作層級：

  1. 最快 (Pre-Commit Hooks)：格式化、基本 lint
  2. 快 (Local Tests)：單元測試、型別檢查
  3. 中 (CI Pipeline)：整合測試、mutation testing
  4. 慢 (Post-Integration)：E2E 測試、手動審查

原則：

"The quality of a feedback loop is proportional to its speed."
反饋循環的品質與其速度成正比。

將能在 CI 運行的 linters 移到 pre-commit hooks；將只在 pre-commit hooks 運行的 formatters 移到 PostToolUse Hooks。

2. Harnessability（可控性）

認識到某些程式碼庫比其他更容易控制：

  • 強型別語言優於動態型別
  • 清晰的模組邊界優於緊耦合
  • 既有框架優於自造輪子

3. 人類角色重新定義

有效的 harness 不會消除人類輸入，而是將人類輸入導向最重要的地方，透過自動化常規品質守門。

---

LangChain 實證案例：+13.7 分的 Harness 改進

成果

  • Terminal Bench 2.0 分數：52.8% → 66.5%
  • 排名躍升：Top 30 外 → Top 5
  • 唯一改變：Harness（系統 prompts、工具、middleware）
  • 模型：GPT-5.2-Codex（未改動）

關鍵技術

#### 1. Self-Verification Loop（自我驗證循環）

循環結構：


Planning & Discovery → Build → Verify → Fix → (repeat)

實作：

  • PreCompletionChecklistMiddleware：強制 Agent 在完成前執行驗證通道
  • 防止在寫完初始解決方案後過早退出

#### 2. Context Engineering（情境工程）

增強環境感知：

  • LocalContextMiddleware：映射目錄結構與可用工具
  • 明確提示可測試程式碼標準與邊緣案例
  • 時間預算警告：防止低效執行

#### 3. Pattern Detection（模式偵測）

LoopDetectionMiddleware：

  • 追蹤檔案編輯
  • 當 Agent 陷入重複相同方法時介入
  • 鼓勵重新考慮策略

#### 4. Reasoning Optimization（推理優化）

"Reasoning Sandwich" 方法：

  • 高推理預算：Planning 階段、Verification 階段
  • 標準推理預算：Implementation 階段

避免全程使用最大推理預算的浪費。

---

Anthropic & OpenAI 最佳實踐

情境窗口管理

Anti-Pattern：依賴 context 壓縮

Best Practice：為每個 session 建立新的 client/context，用結構化交接產物橋接 sessions

"The agent's context window is ephemeral. Everything that must survive between sessions needs to be written to disk."
Agent 的情境窗口是短暫的。所有必須在 sessions 之間存活的資訊都需要寫入磁碟。

狀態持久化優於情境

原則：將關鍵資訊結構化到檔案中，而非依賴情境。

實作：

  • ADRs (Architecture Decision Records)
  • Task Lists (tasks.md)
  • State Files (state/*.json)
  • Memory Layers (context/continuity/*.json)

一次一個任務

原則：將每個 session 聚焦在單一任務上。

理由：

  • 防止情境耗盡
  • 維持可恢復性
  • 避免跨 sessions 的複合錯誤（最常見的失敗模式之一）

建置前驗證

強制規範：在 session 開始時總是執行基準測試。

原因：

"Compounding bugs across sessions is one of the most common failure modes."
跨 sessions 的複合錯誤是最常見的失敗模式之一。

快速反饋循環

整合工具：

  • Type Checkers
  • Linters
  • UI Automation

提供對無效輸出的即時反壓。

Evaluator 評分標準

避免：主觀性評估（"這好嗎？"）

採用：具體的、可測量的成功標準與可測試的閾值

---

關鍵 Anti-Patterns

1. 過早複雜化

在證明單一 Agent 的限制前就開始多 Agent 複雜性。

2. 知識外部化

將必要知識儲存在 repository 外部。

3. 允許佔位符實作

允許 placeholder implementations（如 // TODO: implement this）。

4. 長情境無重置

長情境窗口沒有定期重置。

5. 過載指令檔案

指令檔案超過 ~100 行（保持簡潔）。

對 daily-digest-prompt 專案的優化建議

立即可行的改進

#### 1. 強化 Pre-Commit Hooks

將更多 linters 從 CI 移至 pre-commit：

  • YAML 語法驗證
  • JSON schema 驗證
  • Markdown linting
  • 敏感資訊掃描

#### 2. 實作 Self-Verification Loop

在自動任務完成前加入驗證步驟：


# config/verification-checklist.yaml
auto_task_verification:
  - result_file_exists
  - result_file_valid_json
  - required_fields_present
  - summary_length_adequate
  - no_placeholder_content

#### 3. 增強 LoopDetectionMiddleware

擴展 hooks/agent_guardian.py 的 loop detection：

  • 偵測相同檔案的重複編輯模式
  • 偵測相同 API 端點的重複呼叫
  • 偵測相同錯誤的重複觸發

#### 4. Context Engineering

為每個 auto-task 建立 LocalContextMiddleware 等效物：


// context/auto-task-{task_key}-env.json
{
  "available_tools": ["WebSearch", "WebFetch", "Bash"],
  "environment": {
    "kb_service": "http://localhost:3000",
    "project_root": "D:/Source/daily-digest-prompt"
  },
  "constraints": {
    "max_duration_seconds": 600,
    "max_api_calls": 10
  }
}

#### 5. 實作 Reasoning Budget

在 config/llm-router.yaml 加入 reasoning budget 配置：


reasoning_budgets:
  planning: high      # Phase 0, Phase 1
  implementation: standard  # Phase 2
  verification: high  # Phase 3, validation

中期改進

#### 6. Evaluator Agent 獨立化

將 Phase 3 assemble 步驟拆分：

  • Phase 3a: Collect results (current)
  • Phase 3b: Independent Evaluator Agent

- 驗證結果品質

- 評分（1-5）

- 決定是否需要重試

#### 7. 架構適配性 Fitness Functions

在 config/fitness-functions.yaml 定義可自動檢查的架構特性：


fitness_functions:
  - name: config_complexity
    metric: yaml_line_count
    threshold: 200
    action: warn
  - name: prompt_coherence
    metric: duplicate_instruction_ratio
    threshold: 0.3
    action: alert

#### 8. Mutation Testing Integration

在重要模組加入 mutation testing：


# tests/mutation-test-critical.sh
mutmut run --paths-to-mutate=hooks/,tools/validate_results.py

長期願景

#### 9. Harness Coverage Dashboard

建立 dashboard 追蹤 harness 覆蓋率：

  • Computational checks coverage
  • Inferential checks coverage
  • Feedback loop speed distribution

#### 10. Agent Specialization Refactoring

將 Todoist Agent 重構為專業化角色：

  • Planner Agent: todoist-task-planner.ps1
  • Generator Agent: todoist-task-generator.ps1 (多個並行)
  • Evaluator Agent: todoist-task-evaluator.ps1
