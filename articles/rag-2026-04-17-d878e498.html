一、快取與檢索優化（P0 最高優先）

1.1 語義快取擴展至 LLM API

現況問題：

  • 當前僅知識庫查詢啟用語義快取（tools/kb_semantic_cache.py）
  • Claude API 呼叫無快取層，相似查詢重複計費

業界實證：

  • AWS Best Practices (2026)：語義快取可降低 20-30% Token 成本
  • 相似度閾值建議：初始 0.92-0.95（高精度），逐步 A/B 測試調整至 0.85-0.90
  • TTL 分層：靜態文件 24h、助理回覆 1-4h、即時資料 5-15min

實施方案：


# 擴展 tools/kb_semantic_cache.py 支援多命名空間
# namespace: kb_query（已實施）
#           claude_api（新增）
#           groq_api（新增）
#           todoist_query（新增）

uv run python tools/kb_semantic_cache.py store \
  --namespace claude_api \
  --profile "task_optimize:model=opus" \
  --query "<task_description>" \
  --response-file claude_response.json

預期效益：

  • Token 成本降低 20-30%
  • API 延遲降低 80%+（快取命中時 <100ms vs API 2-5s）

1.2 自適應 TTL 動態調整

現況問題：

  • 所有快取統一 TTL（如 knowledge.json 60min）
  • 未依資料波動與存取模式調整

核心原理（RFC 5861 + Redis EXPIRE GT/LT）：

  • 熱門項目：延長 TTL（EXPIRE key 7200 GT）
  • 低命中項目：縮短 TTL 釋放記憶體
  • Stale-While-Revalidate：過期後先回傳舊值，背景異步更新

實施方案：


# config/cache-policy.yaml 新增 adaptive_ttl 規則
adaptive_ttl:
  knowledge:
    base_ttl: 3600  # 1h
    max_ttl: 86400  # 24h
    min_ttl: 300    # 5min
    hit_threshold: 5  # 5 次命中後延長至 max_ttl
    miss_threshold: 3  # 3 次未命中縮短至 min_ttl

  claude_api:
    base_ttl: 7200  # 2h
    max_ttl: 43200  # 12h
    strategy: "swr"  # Stale-While-Revalidate

預期效益：

  • 命中率從 27.9% → 45%+
  • 記憶體使用優化（淘汰冷數據）

1.3 混合搜尋去重與 Reranking

現況問題（筆記：「RAG 混合搜尋去重策略」）：

  • hybrid search 分數集中 0.333-0.500，區分度低
  • BM25（非 bounded）與向量（0-1）融合需正規化
  • 缺少 reranking 階段過濾誤命中

業界最佳實踐：

  1. 兩階段檢索：FTS5/BM25 抓關鍵詞 → 向量補足語義 → 文件級去重 → Reranker 重排
  2. RRF (Reciprocal Rank Fusion)：score = Σ 1/(k + rank_i)，k=60 是標準值
  3. 輕量 Reranker：Cohere Rerank API / jina-reranker-v2（本地）

實施方案：


# tools/hybrid_search_rerank.py（新增）
# 1. FTS5 召回 top 50
# 2. 向量召回 top 50
# 3. RRF 融合（k=60）
# 4. 去重（cosine > 0.95 視為重複）
# 5. Rerank top 20 → 最終 top 5

預期效益：

  • 精確度提升 15-25%
  • 誤命中率降低 40%+

  • Prompt Engineering (2022-24)：單次優化，無持久狀態
  • Context Engineering (2025)：單 Window Token 策展
  • Harness Engineering (2026)：跨 Window 持久狀態編排

Harness = Constraints + Feedback Loops + Quality Gates

三、LLM 路由動態學習機制（P1）

3.1 強化學習驅動的序列決策

現況問題：

  • 當前路由為靜態規則（tools/llm_router.py mapping 格式）
  • 未依任務複雜度動態選擇模型

業界突破（筆記：「LLM 路由決策動態學習機制」）：

  • Router-R1（arXiv 2506.09033）：Qwen2.5-3B + RL 訓練，Exact Match 0.416
  • xRouter：DAPO/GRPO 訓練，AIME/GPQA 達 GPT-5 近似精度，成本僅 1/8
  • NeuralUCB：線上學習，RouterBench 平均獎勵 0.60，成本僅最高品質 33%

複合獎勵函數：


# Router-R1 設計
r = R_format + (1-α) * R_outcome + α * R_cost

# xRouter 閘控設計（成本懲罰僅在任務成功時生效）
R_final = R_binary × (K − λC)  # λ=2 實驗最優

# NeuralUCB 對數正規化
r = q · exp(−λc̃)  # c̃ 為正規化成本

3.2 實施方案

Phase 1：任務複雜度評估器


# tools/task_complexity_evaluator.py（新增）
# 依 4 個維度評分（0-1）：
# - input_length：>500 tokens = 0.8+
# - output_required_length：>2000 tokens = 0.9+
# - domain_specificity：研究/深度思維 = 0.9+
# - dependency_count：需讀取 >5 檔案 = 0.8+

complexity_score = weighted_avg(4_dimensions)

# 路由規則：
# - score > 0.8 → Opus
# - 0.5 < score <= 0.8 → Sonnet
# - score <= 0.5 → Haiku

Phase 2：線上學習迴圈


# config/router-learning.yaml（新增）
learning:
  mode: "neural_ucb"  # 或 "static"
  exploration_rate: 0.1
  update_frequency: "daily"
  metrics:
    - success_rate
    - avg_cost
    - avg_latency

  fallback:
    min_success_rate: 0.9  # 低於此值回退靜態路由

預期效益：

  • 30% 簡單任務路由到 Haiku（成本 1/50）
  • 整體成本降低 25-40%
  • 品質保留 95%+（閘控設計）

五、知識活化 OODA 閉環（P2）

5.1 現況診斷

數據（筆記：「知識活化與 OODA 閉環實效性分析」）：

  • 107 筆研究積累
  • 轉化為 ADR：僅 18 筆（16.8%）
  • 實際實施：<10 筆（<9.3%）

根因：

  • 缺乏「研究→ADR→實施」追蹤機制
  • 高價值研究未被識別與優先化
  • 無自動回顧流程

5.2 實施方案

建立追蹤機制：


# context/knowledge-activation.yaml（新增）
research_to_adr:
  - research_note_id: "eec7ae69-d8a6-415b-8f29-9243c25f7cd1"  # 專案 RAG 優化日報
    title: "Semantic Caching 擴展"
    status: "proposed"  # proposed/accepted/implementing/done
    adr_ref: "ADR-042"
    priority: "P0"
    estimated_value: "20-30% Token 成本降低"
    created_at: "2026-04-19"

  - research_note_id: "0130d917-2316-4d40-af95-74f205707881"  # LLM 路由決策
    title: "RL-driven LLM Router"
    status: "proposed"
    adr_ref: null
    priority: "P1"
    estimated_value: "25-40% 整體成本降低"

自動回顧機制：


# config/frequency-limits.yaml 新增
knowledge_activation_review:
  allowed_days: [0]  # 週日
  daily_limit: 1
  execution_order: 99
  timeout_seconds: 1800

# 執行邏輯（templates/auto-tasks/knowledge-activation-review.md）：
# 1. 讀取 context/research-registry.json 最近 30 天
# 2. 計算引用次數（KB 筆記互引）、相關性分數
# 3. 識別 top 5 高價值研究
# 4. 自動生成 ADR proposal（寫入 improvement-backlog.json）

預期效益：

  • 知識活化率從 16.8% → 30%+
  • ADR 提案週期從 >7 天 → <3 天

  1. uv sync --extra semantic-cache（fastembed 依賴）
  2. Docker（OpenTelemetry Jaeger）
  3. 知識庫 localhost:3000 穩定運行

緩解措施

  • 語義快取：設定 confidence_threshold，低於閾值不快取
  • RL 路由：Phase 1 先用靜態複雜度評估器，Phase 2 再引入 RL
