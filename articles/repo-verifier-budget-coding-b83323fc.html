Repo 修補場景的 verifier budget 實測：Coding Agent 在 SWE-bench 類任務中的動態測試、候選搜尋與成本治理：深度技術研究報告

執行摘要

這份研究聚焦在 repo 修補型 coding agent 的 verifier budget 問題，也就是在固定推理與執行成本下，如何把預算分配給測試、候選搜尋、critic/PRM 排序，以及 repository 結構探索。現有證據顯示，repo 修補不是單次程式補全，而是 repository-level 的定位、修改、測試與回歸保證流程 [來源：SWE-BENCH: Can Language Models Resolve Real-World GitHub Issues?, 2024] [來源：SWE-bench Verified, 2026]。因此，最有價值的前段 verifier 仍是可執行測試與環境回饋，因為它們最直接對應「補丁是否真的讓 repo 回到可接受狀態」這個目標 [來源：R2E-Gym: Procedural Environment Generation and Hybrid Verifiers for Scaling Open-Weights SWE Agents, 2026]。

第二個結論是，SWE-bench Verified 雖仍具參考價值，但已不適合被當成單一主指標。OpenAI 在 2026 年 2 月 23 日公開指出，SWE-bench Verified 至少有 59.4% 的抽樣失敗案例含題目或測試缺陷，且污染會扭曲分數 [來源：Why SWE-bench Verified no longer measures frontier coding capabilities, 2026]。這與官方 leaderboard 仍主要呈現 resolved rate 的現況形成張力 [來源：SWE-bench Verified, 2026]。因此，budget 設計不能只追求排行榜分數，必須同步納入延遲、token、測試成本與 repo 健壯性風險。

第三個結論是，最佳 verifier budget 應採分層與動態配置。混合 execution-based 與 execution-free verifiers 的策略，比將資源全部押在單一路徑更穩健 [來源：R2E-Gym: Procedural Environment Generation and Hybrid Verifiers for Scaling Open-Weights SWE Agents, 2026]。而 RepoMirage 與 SWE-PolyBench 顯示，repo 結構擾動、語言差異與工具鏈差異都會大幅改變代理表現 [來源：RepoMirage: Do Code Agents Really Understand Repository Structures?, 2026] [來源：SWE-PolyBench: A multi-language benchmark for repository level evaluation of coding agents, 2026]。實務上，合理做法是先以測試與環境訊號做硬門檻，再把 critic、patch ranking 與額外搜尋保留給後段排序與疑難案件。

研究背景與動機

repo 修補型 coding agent 已從展示型 demo 走向真實維護流程。Anthropic 對真實工作負載的分析指出，coding agents 的高頻任務集中在修改、除錯、驗證與既有 codebase 維護，而不是從零開始寫新程式 [來源：Anthropic Economic Index: What types of work do coding agents actually do?, 2026]。這使 verifier budget 成為核心工程問題：如果前段花太多成本在昂貴測試，系統會太慢；如果太依賴廉價的 critic 或 ranking，則容易在真實 repo 上通過錯誤候選。

同時，評測環境本身正在失真。SWE-bench 將真實 GitHub issue 修補轉成 execution-based benchmark，開創了 repository-level 的代表性場景 [來源：SWE-BENCH: Can Language Models Resolve Real-World GitHub Issues?, 2024]。但到了 2026 年，前沿模型在 SWE-bench Verified 上的分數已受到污染與資料品質問題干擾 [來源：Why SWE-bench Verified no longer measures frontier coding capabilities, 2026]。如果研究者只依賴 resolved rate，往往會低估結構探索成本、上下文成本與測試治理成本，最後做出不適合 production 的 budget 設計。

核心技術解析

repo 修補任務的第一個技術特徵，是 verifier 必須服務於「互動式修補循環」，而不是單次輸出驗收。原始 SWE-bench 就已說明，代理需要跨檔案理解、環境互動與執行驗證 [來源：SWE-BENCH: Can Language Models Resolve Real-World GitHub Issues?, 2024]。R2E-Gym 進一步把這件事系統化，建立 8.1K 個可執行 repo 修補環境，並比較 execution-based 與 execution-free verifiers 的上限與互補性 [來源：R2E-Gym: Procedural Environment Generation and Hybrid Verifiers for Scaling Open-Weights SWE Agents, 2026]。這組證據支持一個明確設計原則：測試與環境回饋適合做前段 gating，critic/PRM 更適合在後段對剩餘候選做排序與資源再分配。

第二個技術特徵，是 repo 結構理解本身就需要 verifier 預算。RepoMirage 利用 function-preserving perturbation 測試 repository structure shift，結果顯示 GPT-4o 的 resolved rate 從 32.91% 掉到 7.59% [來源：RepoMirage: Do Code Agents Really Understand Repository Structures?, 2026]。這代表很多代理其實不是「修補能力不足」，而是被定位、導航與上下文選取擊倒。若把 budget 全花在最終測試，卻不保留預算給結構探索、失敗重試與檔案定位，就會讓代理在真實 repo 中提早失敗。

第三個技術特徵，是 budget profile 必須語言與工具鏈感知。SWE-PolyBench 把 repo-level benchmark 擴展到 Java、JavaScript、TypeScript 與 Python，並顯示不同語言下代理表現不均 [來源：SWE-PolyBench: A multi-language benchmark for repository level evaluation of coding agents, 2026]。這說明單一固定 budget ceiling 很容易失真。以 Python 為主的 repo，快速單元測試可能適合前段高頻執行；在重型 Java 或前端 monorepo 中，測試啟動成本更高，應提高前段靜態定位與候選壓縮比例，只將完整整合測試留給後段。

第四個技術特徵，是 repo-level instruction 並非越多越好。Evaluating AGENTS.md 的研究指出，repository context files 往往降低成功率，且增加超過 20% 的推論成本 [來源：Evaluating AGENTS.md: Are Repository-Level Context Files Helpful for Coding Agents?, 2026]。這與工程直覺相反，但很有啟發：context 本身就是預算消耗項。換句話說，verifier budget 不只是在「測試 vs critic」之間分配，也包含「給模型多少 repo 說明文字」這件事。真正有效的策略是最小必要 context，而不是無上限堆疊說明。

第五個技術特徵，是評測指標應從單一 resolved rate 轉向多維治理。SWE-bench 官方頁面仍以 resolved rate 為主要展示欄位 [來源：SWE-bench Verified, 2026]，但 OpenAI 已明確指出該指標無法穩定代表 frontier coding capability [來源：Why SWE-bench Verified no longer measures frontier coding capabilities, 2026]。業界解讀也逐漸將 SWE-bench-Live、OSWorld、PaperBench 與長期 CI 導向 benchmark 視為同一波轉向 [來源：AI Agent Evaluation Stack in 2026: Beyond Saturated SWE-bench Scores, 2026]。對 budget 治理而言，這意味每次 verifier 追加都要問四個問題：成功率多提升多少、延遲多增加多少、token 成本多多少、是否更接近真實維護流程。

關鍵洞見

  1. ✅ 已驗證：前段篩選最值得優先保留的 verifier 是可執行測試與環境回饋，因為它們最貼近 repo repair 的真實成功條件。支持來源 4：SWE-BENCH、SWE-bench Verified、R2E-Gym、Anthropic Economic Index。
  2. ✅ 已驗證：execution-free critic 與 patch ranking 不應取代測試，而應留在後段排序與預算再分配。支持來源 3：R2E-Gym、RepoMirage、SWE-agent Organization。
  3. ✅ 已驗證：repo 結構健壯性是 verifier budget 的一級風險，必須保留定位、導航與結構探索預算。支持來源 3：RepoMirage、R2E-Gym、SWE-BENCH。
  4. ✅ 已驗證：budget profile 應依語言、工具鏈與測試啟動成本分型，而非用單一固定上限。支持來源 4：SWE-PolyBench、RepoMirage、Evaluating AGENTS.md、Anthropic Economic Index。
  5. ✅ 已驗證：resolved rate 若不搭配延遲、token 與 verifier 成本一起看，會錯配真實 CI 維護目標。支持來源 5：SWE-bench Verified、OpenAI 對 SWE-bench Verified 的評估、Anthropic Economic Index、AI Agent Evaluation Stack、中文解讀文章。
  6. ⚠️ 待驗證：長期 CI 維護型 benchmark 可能更適合 budget 治理，但目前這一點在本輪資料中仍多來自二手報導，尚不足以作為主結論。支持來源 2：AI Agent Evaluation Stack、SWE-CI 二手報導。

實務應用建議

第一，將 verifier budget 明確拆成三層。L1 使用廉價且高信息密度的 repo 導航與快速檢查，包含檔案定位、關鍵測試子集與基本環境 sanity check。L2 使用可執行測試做硬 gating，只在候選壓縮後再投入較昂貴的測試成本。L3 才是 critic、patch ranking、補充搜尋與人工審查接口。這種配置最符合 R2E-Gym 所揭示的混合 verifier 優勢。

第二，對不同 repo 類型建立 profile。小型 Python library 可提高測試頻率；大型 monorepo、前端專案或 Java 專案則應提高上下文裁切與候選壓縮比例，避免在早期就被冷啟動測試成本吃掉全部 budget。這是對 SWE-PolyBench 與 RepoMirage 證據的直接落地。

第三，把 repo context 視為有成本的資源，而不是免費提示。AGENTS.md 類文件若要保留，應限制篇幅、只保留穩定工作流規則，並定期檢查是否真的提升成功率。若沒有增益，就應回退到更短的最小操作說明。

第四，報表層面不要只記 resolved rate。每輪實驗至少同步記錄 latency、token、測試執行時間、平均候選數與 fallback 次數。沒有這些欄位，就無法真正優化 verifier budget。

第五，將「疑難案件升級條件」寫進 agent policy。當代理連續兩輪無法定位正確檔案、快速測試反覆失敗，或 repo 結構訊號不穩定時，系統就應自動升高結構探索預算，甚至切換到更保守的候選搜尋模式。這能避免把昂貴測試浪費在明顯不成熟的候選上，也能把 RepoMirage 所揭示的結構風險納入 runtime 治理，而不是事後才在 leaderboard 上看到失敗。

與現有知識的連結

這份研究延續了前一輪「小型 coding agent 的 verifier budget 策略」的結論，但把焦點從抽象預算原則，推進到 repo repair 的具體場景。前一輪已指出可執行測試是最可靠的 verifier，PRM/critic 更適合候選排序；本輪則補上 repo-level benchmark 污染、結構健壯性、多語言差異與 context 成本四個關鍵變數，使 budget 設計從模型內部推理問題，轉成完整軟體工程治理問題。

未來展望

下一步最值得深化的方向，是用真實 repo 類型建立 budget profile 實測，例如 Python library、TypeScript frontend、Java service 各自採用不同 L1/L2/L3 比例，觀察成功率、延遲與成本曲線。此外，長期 CI 維護型 benchmark 若能取得一手來源，將可驗證「短期修補成功」與「長期可維護」之間是否存在明顯 trade-off。

另一個值得追蹤的方向，是把 verifier budget 與 agent observability 連結起來。若未來實驗能穩定收集檔案定位錯誤率、上下文命中率、測試選擇品質與 fallback 觸發原因，就能從「後驗分析」走向「即時調度」，讓 budget profile 依 repo 狀態動態切換，而不只是依靜態語言分類決定。
