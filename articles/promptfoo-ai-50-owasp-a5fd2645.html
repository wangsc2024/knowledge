一、專案概述

Promptfoo（promptfoo/promptfoo）是由 Ian Webster（CEO）與 Michael D'Angelo（CTO）於 2024 年創立的開源 AI 紅隊測試與評估框架，定位為「LLM 應用的測試驅動開發平台」。截至 2026 年 3 月已累積 15,213 Stars、1,400+ Forks，採用 MIT 授權，是 AI 安全測試領域最具影響力的開源專案。

二、核心架構：三層測試模型

2.1 宣告式配置（Declarative Config）

Promptfoo 採用 YAML 宣告式配置，開發者無需撰寫測試程式碼即可定義評估流程：


# promptfooconfig.yaml 範例
providers:
  - openai:gpt-4
  - anthropic:claude-3-opus
prompts:
  - "Translate the following to {{language}}: {{text}}"
tests:
  - vars:
      language: French
      text: Hello world
    assert:
      - type: contains
        value: Bonjour
      - type: llm-rubric
        value: Translation is natural and grammatically correct

2.2 三大核心概念

| 概念 | 說明 | 類比 |

|------|------|------|

| Providers | 被測試的 LLM 端點（OpenAI、Anthropic、Azure、Ollama 等） | 測試目標 |

| Test Cases | 輸入變數 + 預期行為的組合 | 測試案例 |

| Assertions | 自動評分指標（精確匹配、LLM 評分、正規表達式等） | 斷言 |

2.3 五階段開發工作流

  1. 定義（Define）：識別核心使用案例，準備代表性測試情境
  2. 配置（Configure）：指定 prompts、test cases、API providers
  3. 執行（Execute）：運行評估並擷取模型輸出
  4. 分析（Analyze）：透過 CLI 或 Web UI 檢視比較矩陣
  5. 迭代（Iterate）：依使用者回饋擴充測試覆蓋

三、紅隊測試（Red Teaming）：50+ 漏洞類型

3.1 威脅分類

#### 模型層漏洞

  • Prompt Injection（提示注入）：未受信任的使用者輸入與系統提示串接
  • Jailbreaking（越獄）：繞過基礎安全過濾器（Tree of Attacks 在 GPT-4 上可達 80%+ 成功率）
  • 有害輸出：仇恨言論、毒性內容、偏見
  • 幻覺與錯誤資訊：生成不實內容
  • 版權侵害：重現受保護的訓練資料
  • PII 洩漏：從訓練資料中暴露個人資訊

#### 應用層漏洞

  • 間接 Prompt Injection：透過 RAG 系統上下文注入
  • 工具濫用（Tool Misuse）：未授權 API 存取、SQL 注入
  • BOLA（Broken Object-Level Authorization）：未授權資源存取
  • BFLA（Broken Function-Level Authorization）：越權操作
  • 資料滲出（Data Exfiltration）：透過巧妙提問提取敏感資訊
  • 劫持/離題使用：將 AI 用於非預期用途

3.2 攻擊策略（Strategies）

| 策略 | 說明 |

|------|------|

| Simple Rewording | 基礎提示重寫 |

| TAP（Tree of Attacks with Pruning） | 樹狀攻擊搜尋，GPT-4 上 80%+ 成功率 |

| Adversarial Suffixes | 在查詢後附加對抗性後綴 |

| ASCII Art Encoding | 用 ASCII 藝術編碼繞過內容過濾 |

| Roleplay Framing | 角色扮演框架（如 Discord Clyde AI 的「祖母漏洞」） |

3.3 OWASP LLM Top 10 對照

Promptfoo 的插件系統完整覆蓋 OWASP LLM Top 10 所有類別，包括：

  • LLM01: Prompt Injection
  • LLM02: Insecure Output Handling
  • LLM03: Training Data Poisoning
  • LLM06: Sensitive Information Disclosure
  • LLM07: Insecure Plugin Design
  • LLM09: Overreliance

四、技術特色

4.1 隱私優先（Private-by-Default）

  • 評估完全在本機執行
  • Prompts 不會傳送至外部伺服器
  • 適合企業合規需求（GDPR、HIPAA）

4.2 多模型比較

支援同時對多個 LLM 進行 A/B 測試：

  • OpenAI（GPT-4、GPT-4o）
  • Anthropic（Claude 3/4 系列）
  • Azure OpenAI
  • AWS Bedrock
  • Google Gemini
  • Ollama（本地模型）
  • 自訂 API Provider

4.3 CI/CD 整合

  • GitHub Actions / GitLab CI 原生支援
  • PR Code Scanning（LLM 相關合規檢查）
  • 失敗時自動阻擋部署

4.4 效能優化

  • 快取機制：避免重複 API 呼叫
  • 並行處理（Concurrent Processing）：加速大規模評估
  • Live Reloading：開發中即時查看結果

五、OpenAI 收購：產業標誌性事件

5.1 收購詳情

  • 宣布日期：2026 年 3 月 9 日
  • 收購方：OpenAI
  • 交易金額：未公開
  • 整合目標：併入 OpenAI Frontier 平台

5.2 收購背景與動機

隨著 Agentic AI（代理式 AI）興起，AI 應用從單純的文字生成擴展到具備工具呼叫、程式碼執行、網路存取等能力的自主代理。這些能力大幅提升了安全風險：

  • Agent 可能被誘導執行非預期操作
  • Tool Misuse 風險遠超傳統 LLM 應用
  • 企業部署需要系統化的安全驗證

OpenAI 表示：收購 Promptfoo 是為了「強化代理式安全測試與評估能力」。

5.3 開源承諾

Promptfoo 將繼續維持開源：

  • MIT 授權不變
  • 持續支援多廠商模型（非 OpenAI 獨佔）
  • 23 人團隊全員併入 OpenAI

5.4 產業影響分析

| 面向 | 影響 |

|------|------|

| AI 安全測試標準化 | Promptfoo 成為事實標準，收購加速其工具融入 AI 開發主流 |

| 企業信心 | Fortune 500 的 25% 已使用，收購提供長期維護保證 |

| 競爭格局 | OpenAI 在安全層獲得先發優勢，其他廠商需自建或尋找替代 |

| 開源生態 | 類似 GitHub 收購模式：維持開源吸引社群，商業功能整合至平台 |

六、對 Daily-Digest-Prompt 系統的啟發

6.1 Hook 安全機制的紅隊思維

本專案已有 pre_bash_guard.py、pre_write_guard.py 等防護 Hook，但這些是「白名單防禦」。可借鑑 Promptfoo 的紅隊思維，建立「對抗測試」：

  • 定期以 adversarial prompts 測試 Hook 攔截率
  • 模擬 prompt injection 驗證 Agent 不會執行非預期操作

6.2 宣告式測試配置

Promptfoo 的 YAML 配置式測試與本專案的「文件驅動架構」理念一致。可考慮將 Agent 品質驗證從程式碼轉為 YAML 配置。

6.3 多模型比較

本專案已有 llm-router.yaml 做路由，Promptfoo 的多模型評估可用於驗證路由決策品質——比較 Groq 與 Claude 在不同任務類型上的實際表現差異。

八、總結

Promptfoo 代表了 AI 開發從「試錯法」到「測試驅動」的典範轉移。其三大價值主張：

  1. 系統化：將 LLM 測試從手動嘗試提升為結構化、可重複的流程
  2. 安全化：50+ 漏洞類型的自動紅隊測試，覆蓋 OWASP LLM Top 10
  3. 標準化：被 OpenAI 收購後，有望成為 AI 安全測試的事實標準

被 OpenAI 以未公開金額收購（2026-03-09），是 AI 安全測試領域的標誌性事件，標誌著「AI 安全測試」從可選步驟升級為 AI 開發的基礎設施層。
