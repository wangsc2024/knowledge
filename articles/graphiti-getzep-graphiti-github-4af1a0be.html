本次研究：Graphiti（25,605 stars，2,550 forks，383 open issues）— 為 AI Agent 建立即時、可增量更新、具時間有效性的脈絡圖（temporal context graph），並以 Episode 維持來源溯及（provenance）。

---

專案概述

Graphiti 由 Zep 團隊維護並以 Apache-2.0 開源，定位是「動態環境中的 Agent 記憶／脈絡工程」：把非結構化對話、結構化 JSON 與外部事件流，持續編入可查詢的圖，而不是一次性的靜態文件 RAG 批次摘要 [來源：getzep/graphiti README, 2026]。核心使用者是要在生產環境處理狀態會改變的助理與 Agent 開發者：例如 CRM 訂單狀態、使用者偏好被新事實取代、或合規上需要「某時間點何者為真」的稽核場景 [來源：Zep Graphiti Overview 文件, 2026]。與僅做向量相似度的記憶層相比，它把「事實如何隨時間演變」當成一級公民：邊（fact/relationship）帶有效性視窗，舊事實可被標記失效而非直接抹除，以保留歷史解釋鏈 [來源：README 與 Zep 文件, 2026]。官方宣稱檢索多為亞秒級，並在 Show HN 討論中補充延遲有很大比例落在第三方 embedding API（約 70%），自架 embedding 可把 P90 壓到約 10ms 量級 [來源：Hacker News item=41378691, 2024]。

技術架構

語言與執行：主程式庫為 Python 3.10+；圖資料庫可插拔，README 列舉 Neo4j 5.26、FalkorDB 1.1.2、Kuzu 0.11.2、Amazon Neptune（並搭配 OpenSearch Serverless 做全文）等驅動 [來源：README, 2026]。LLM／嵌入：預設 OpenAI；可選 Azure OpenAI、Anthropic、Groq、Google Gemini；本地可用 Ollama 的 OpenAI 相容端點，但 README 強調需 Structured Output 能力，否則小模型易造成 schema 不符合而 ingestion 失敗 [來源：README, 2026]。圖內資料模型：實體為節點、事實／關係為邊，邊上承載自然語言描述與時間欄位；一切衍生資訊溯及 Episodes（原始 ingest 資料流）以滿足稽核與除錯 [來源：README + Zep Overview, 2026]。檢索管線：語義向量 + BM25 全文 + 圖距離／中心節點 rerank 的混合策略；ingestion 以並行 LLM 呼叫為主，預設以 SEMAPHORE_LIMIT（預設 10）節流以避免供應商 429 [來源：README, 2026]。對外介面：內建 MCP Server（mcp_server/）讓 Claude／Cursor 等客戶端以協定操作 episode 與搜尋；另有 FastAPI server/ 提供 REST [來源：README, 2026]。學術背書：arXiv:2501.13956《Zep: A Temporal Knowledge Graph Architecture for Agent Memory》與專案連動，可用於理解設計動機與實驗主張 [來源：README badge, 2026]。

架構示意（邏輯分層）：


Episodes (raw) → LLM 抽取/對齊實體與邊 → 圖儲存 (Neo4j/FalkorDB/...)
                                    ↘ 嵌入與全文索引
檢索 ← 混合搜尋 (semantic + BM25 + graph rerank) ← 查詢

與 Microsoft GraphRAG 的文件批次社群摘要路線不同，Zep 文件明確對照：GraphRAG 偏靜態批次與秒級到十秒級延遲；Graphiti 針對連續更新與雙時間／失效邊語意 [來源：Zep Overview 對照表, 2026]。DEV Community 的 Graphiti vs Mem0 實測則從實作面補刀：Graphiti 把「語義與圖結構」放在同一儲存與 ID 空間，利於做「已注入上下文的 fact 去重」這類 metadata contract；Mem0 向量與可選圖層若各自更新，會出現作者所稱 split-brain（圖已改、向量仍舊敘事） [來源：DEV juandastic Graphiti vs Mem0 benchmark, 2026]。

功能特色與差異化

| 維度 | Graphiti | Mem0（對照，依 DEV 實測與向量優先設計） | 依據 |

|------|------------|----------------------------------------|------|

| 事實時間語意 | 邊上自然語言 fact + valid/invalid 時間戳 | 向量為主；圖邊偏薄、與向量無共用 ID | DEV benchmark；README |

| 矛盾／更新 | 失效舊邊並保留歷史 | 實測出現「兩條都看似當前」的並存 | DEV Phase 2 |

| Token／成本 | 較高（實測約 1.68x） | 較低 | DEV 成本表 |

| 敘事／感官細節 | 因果骨架較完整 | 向量原子句較利於「感官觸發」檢索 | DEV Phase 3 |

| 部署複雜度 | 必備圖庫 + LLM | 向量庫為主，較輕 | README vs 生態描述 |

中文技術圈整理亦強調：即時增量、雙時間、混合檢索與 MCP 整合 是對岸開發者最常在「一天一個開源項目」類文章中複述的賣點 [來源：掘金技術社群 Graphiti 解讀文, 2025–2026]。

使用方式（快速上手）

  1. 依 README 啟動 Neo4j 或 docker run FalkorDB。
  2. pip install graphiti-core 或加上 [falkordb]／[anthropic] 等 extra。
  3. 設定 OPENAI_API_KEY（或他廠相容 client）。
  4. 參考 examples/quickstart：建索引、寫入 episode、跑 hybrid search。

from graphiti_core import Graphiti

# 典型建構：URI / user / password；生產環境建議改為注入 Neo4jDriver / FalkorDriver
graphiti = Graphiti("bolt://localhost:7687", "neo4j", "password")
# 依 quickstart：add_episode、search、…

完整參數（Azure、Gemini、Ollama）見 README 範例區塊 [來源：README, 2026]。

潛在應用（對 daily-digest-prompt）

  1. 連續性 JSON 的圖式化：將 context/continuity/*.json 的 runs 改寫為 episode ingest，用時間失效表達「上一輪建議已被本輪覆蓋」，降低純文字重複與版本衝突；落地步驟：先對單一 task_key 做 dual-write 驗證再切流。
  2. Skill／Workflow 決策追蹤：把 ADR 與 workflow 選擇寫成邊上 fact，供 assemble-digest 類任務做「為何當時選 A」的 point-in-time 查詢；落地：每筆 digest 產出後寫入一則 structured JSON episode。
  3. 與現有 KB 的互補：本 repo 已有向量混合搜尋（skills/knowledge-query）；Graphiti 可承接「會過期的事實」（例如 API 配額、路由策略版本），向量庫保留長文敘事；落地：僅對高頻變動欄位走 Graphiti，避免全量遷移成本。

優缺點評估

優點

  • 時間語意與 provenance 一級設計，利於合規與除錯 [Zep 文件, 2026]。
  • 混合檢索 + rerank，對結構化追問較友善 [README, 2026]。
  • MCP／REST 雙出口，與 Cursor 生態契合 [README, 2026]。

缺點（有來源）

  • 成本：第三方實測 Phase 3 narrative ingestion，Graphiti token 約為 Mem0 的 2.25x [DEV, 2026]。
  • 小模型風險：官方警告非結構化輸出可靠度不足 [README, 2026]。
  • 營運重量：需圖資料庫與嵌入管線，對僅需「相似度記一筆」的場景過重 [README 需求列表；DEV 結論, 2026]。
  • Telemetry：預設匿名統計，需 GRAPHITI_TELEMETRY_ENABLED=false 關閉 [README, 2026]。

不適用情境：無圖 DBA 能力、無法承擔較高 LLM token、且記憶以「平面片段」即足夠的簡單 chatbot（DEV 文亦建議多數標準 Agent 可優先 Mem0） [DEV, 2026]。
