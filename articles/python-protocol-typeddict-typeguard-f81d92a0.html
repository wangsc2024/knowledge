Python 靜態型別系統進階實踐：Protocol、TypedDict、TypeGuard 與型別檢查器選型（2026）

技術概述

Python 的漸進式型別系統（Gradual Typing）自 PEP 484（Python 3.5，2015 年）引入以來，已演進為完整的靜態型別功能集。2026 年，核心機制包含：Protocol（PEP 544，結構式子型別）、TypedDict（PEP 589）、TypeGuard（PEP 647）、TypeIs（PEP 742，Python 3.13）、TypeVar 預設值（PEP 696）、ReadOnly（PEP 705）。

這套系統解決的核心問題：讓 Python 在享有動態語言靈活性的同時，也能透過靜態分析提前發現型別錯誤。與 Java/C# 的名義型別系統（Nominal Typing）不同，Python 選擇結構式子型別——一個物件是否符合某型別，取決於它是否擁有所需的屬性與方法，而非繼承關係。[來源：PEP 544，2020]

替代方案比較：Protocol（結構式靜態，無執行期負擔）vs ABC（名義式靜態+執行期）vs Pydantic（名義式執行期驗證，中等負擔）vs TypedDict（結構式靜態，無負擔）。

核心概念與原理

Protocol：結構式子型別的靜態 duck typing

Protocol 是 Python 型別系統中最強大的創新之一。根據官方規範，「Class Dog 是 Animal 的結構子型別，只要前者具有後者所有屬性與方法，且型別兼容」。與 ABC 的關鍵差異：繼承 ABC 必須明確聲明，但滿足 Protocol 只需「有正確的方法」——無需繼承關係。[來源：typing.python.org/protocols，2026]


from typing import Protocol, runtime_checkable

class Saveable(Protocol):
    def save(self) -> None: ...

class Database:
    def save(self) -> None:
        print("Saved to DB")

class FileStore:  
    def save(self) -> None:
        print("Saved to file")

def persist(storage: Saveable) -> None:
    storage.save()

persist(Database())   # 通過型別檢查，無需繼承
persist(FileStore())  # 通過型別檢查，無需繼承

Protocol 組合：多個 Protocol 可透過多重繼承組合，但必須明確帶上 Protocol 基底類別。

@runtime_checkable：讓 Protocol 支援 isinstance() 檢查，但只檢查方法名稱存在性，不驗證簽名型別。[來源：Real Python，2025]

Protocol 屬性宣告的規範：實例屬性必須在類別層級宣告，而非 __init__ 內部——這是常見錯誤來源：


class Logger(Protocol):
    filename: str          # 類別層級宣告（正確）
    def log(self, msg: str) -> None: ...

class BadLogger:
    def __init__(self):
        self.filename = "app.log"  # 型別檢查器無法識別（錯誤）

TypedDict：帶型別的字典（零執行期負擔）

TypedDict 允許為字典指定每個 key 的型別，完全不產生執行期負擔。Python 3.13 新增 ReadOnly（PEP 705）支援唯讀欄位：


from typing import TypedDict, ReadOnly, NotRequired

class TaskResult(TypedDict):
    agent: str
    status: str
    summary: str
    kb_imported: NotRequired[bool]  # 可選欄位（PEP 655）

class ImmutableConfig(TypedDict):
    api_key: ReadOnly[str]   # 唯讀欄位（Python 3.13，PEP 705）
    timeout: int

TypedDict vs Pydantic 選型：TypedDict 適合內部 API 與受信任資料源（零執行期負擔）；Pydantic 適合外部 API 與使用者輸入（執行期驗證 + 自動型別轉換）。[來源：packetcoders.io，2025]

TypeGuard 與 TypeIs：使用者定義的型別縮窄

TypeGuard（PEP 647）與 TypeIs（PEP 742）的核心差異是負向分支（else）的行為：TypeGuard 只縮窄正向分支；TypeIs 正向與負向分支都縮窄（更直觀）。


from typing import TypeIs  # Python 3.13+ 或 typing_extensions

def is_string(val: str | int) -> TypeIs[str]:
    return isinstance(val, str)

def process(val: str | int) -> None:
    if is_string(val):
        print(val.upper())   # val: str
    else:
        print(val + 1)       # val: int（TypeIs 能縮窄 else 分支）

官方文件明確建議：「使用者通常應優先選擇 TypeIs，其行為更加直觀。」標準庫的 inspect.isgenerator、iscoroutine、isawaitable 已改用 TypeIs。[來源：typing.python.org，2026；rednafi.com，2024；PEP 742，2022]

型別參數預設值（Python 3.13，PEP 696）


from typing import TypeVar
T = TypeVar("T", default=str)  # 預設型別為 str

class Container[T = str]:      # 新語法（PEP 695 + PEP 696）
    def __init__(self, value: T) -> None:
        self.value = value

型別檢查器選型（2026 最新格局）

2026 年生態系新增 ty（Astral）與 Pyrefly（Meta）。規範符合度（pydevtools benchmark，2026-04）：

  • pyright: 97.8%（Microsoft，VS Code 整合最佳）
  • Zuban: 96.4%（新創，mypy 語義 + 編譯速度）
  • Pyrefly: 87.8%（Meta，1.8M lines/sec）
  • mypy: 58.3%（PSF 參考實作，文件最豐富）
  • ty: 53.2%（Astral，alpha，uv 生態整合）

速度：ty 比 mypy 快約 7 倍；Pyrefly 宣稱比 mypy 快 14 倍（理想環境）。

選型建議（社群共識，HN 2024）：mypy 適合既有管線維護；pyright 適合 VS Code 整合；ty 在 uv 生態整合有優勢但 alpha 不建議 production；多工具並行（編輯器用 pyright/ty，CI 用 mypy）是常見策略。[來源：pydevtools.com，HN 2024]

最佳實踐

1. mypy strict 漸進式導入策略


# pyproject.toml
[tool.mypy]
strict = true
show_error_codes = true
warn_unused_ignores = true

[[tool.mypy.overrides]]
module = ["legacy_module.*"]
disallow_untyped_defs = false

[[tool.mypy.overrides]]
module = ["some_untyped_lib"]
ignore_missing_imports = true

漸進遷移路徑：新增模組一律嚴格 → 修改舊模組補 type hints 後移除 override → CI 失敗立即修正，不累積技術債。[來源：careers.wolt.com，2024]

2. Protocol 設計原則

  • 介面最小化：只宣告客戶端真正使用的方法
  • 讀寫分離：mutable 屬性易導致 invariance 問題，優先用 @property 標示唯讀
  • 避免過度依賴 @runtime_checkable：只做名稱檢查，不做簽名驗證
  • 重要實作類別加 explicit inheritance 讓 mypy 強制驗證

3. TypeIs 取代 TypeGuard

新代碼統一改用 TypeIs（Python 3.13+，或 typing_extensions 回填），避免 TypeGuard else 分支不縮窄的陷阱。標準庫已逐步採用 TypeIs。

4. CI 整合


uv run mypy --config-file pyproject.toml hooks/ skills/ tools/

5. TypedDict NotRequired

用 NotRequired（PEP 655）標示可選欄位，比 Optional 更語義清晰，且不會讓 type checker 誤認欄位存在但值可為 None。

陷阱與反例

陷阱 1：Protocol 被意外滿足（Accidental Compliance）

Protocol 只看結構，不看意圖。越通用的 Protocol 越容易被意外滿足（例如 str 的 encode() 方法可能意外符合某個 Encoder Protocol）。規避：縮窄 Protocol 方法簽名，或改用 ABC 名義式約束。[來源：Real Python，2025]

陷阱 2：TypeGuard else 分支不縮窄


from typing import TypeGuard

def is_int(val: str | int) -> TypeGuard[int]:
    return isinstance(val, int)

def process(val: str | int) -> None:
    if is_int(val):
        pass  # int
    else:
        val.upper()  # mypy 報錯：int 沒有 upper()（else 分支是 str | int，不是 str）

TypeGuard 設計上只保證正向縮窄，負向不做假設。規避：改用 TypeIs。[來源：rednafi.com，2024；PEP 742，2022]

陷阱 3：mypy ignore_errors 全局靜默

全局 ignore_errors = True 讓 mypy 形同虛設。正確做法：用 # type: ignore[error-code] 針對特定行抑制，搭配 show_error_codes = true 取得 error code。「never ever silence mypy by using ignore_errors = True」。[來源：careers.wolt.com，2024]

陷阱 4：過度解讀 mypy 58% 規範符合度

mypy 只符合 58.3% 的型別規範測試案例，但大多數失敗是複雜泛型邊緣情形。真實開發中選錯工具（如用 mypy 處理高度泛型 API 庫）才是主要痛點，而非 58% 本身的問題。[來源：sinon.github.io，2025；pydevtools.com，2026]

與本專案（daily-digest-prompt）關聯

落地場景 1：tools/validate_results.py 加入 Protocol

本專案的 validate_results.py 驗證 results/todoist-auto-*.json 格式。可用 Protocol 定義 validator 介面，讓未來新增 validator 無需繼承：


from typing import Protocol

class ResultValidator(Protocol):
    def validate(self, data: dict) -> list[str]: ...

class SchemaValidator:
    def validate(self, data: dict) -> list[str]:
        errors = []
        for field in ["agent", "task_key", "status", "summary"]:
            if field not in data:
                errors.append(f"missing required field: {field}")
        return errors

def run_all_validators(data: dict, validators: list[ResultValidator]) -> list[str]:
    return [e for v in validators for e in v.validate(data)]

落地場景 2：results/ JSON 的 TypedDict 化

本專案的 config/schemas/results-auto-task-schema.json 已有明確 schema，可對應為 TypedDict 在 Python 層早期捕捉缺失欄位：


from typing import TypedDict, NotRequired

class AutoTaskResult(TypedDict):
    agent: str
    task_key: str
    status: str
    summary: str
    done_cert: NotRequired[dict]
    cli_exit_note_zh: NotRequired[str]
    result_guard_zh: NotRequired[str]
    kb_imported: NotRequired[bool]
    error: NotRequired[str | None]

落地場景 3：hooks/ 加入 mypy 覆蓋

本專案 hooks/ 有 529 個測試但 mypy 覆蓋不明。建議在 CI 中加入：uv run mypy hooks/ tools/ --ignore-missing-imports --no-strict-optional，逐步升到 strict 模式，消滅 Any 型別的擴散。
