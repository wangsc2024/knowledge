主筆說明

本報告由李飛飛（primary）與Yann LeCun（co_primary）雙主筆撰寫。選角理由：agent 編排器的權限模型涉及架構設計（計算機視覺與 HAI 治理的李飛飛擅長從人類中心推導系統約束）與工程化落地（深度學習工程實踐的Yann LeCun關注開源生態與架構可擴展性）。

執行摘要

[李飛飛主筆]

我認為：agent 編排器的信任機制與權限模型，本質是「人類意圖→機器執行」過程中的防護閘門設計問題。2025–2026 年的研究揭示三個核心主張：

  1. strict-subset invariant：agent 權限必須是人類權限的真子集，而非平行或超集 [Context Kubernetes, 2026]
  2. 工具級權限攔截：在 LLM 生成工具調用後、函式執行前強制執行，而非依賴 system prompt [CogniWall, 2025; AWS Well-Architected, 2025]
  3. excessive agency 風險：機器速度下的過度授權呈指數級放大，靜態授權模式已不適用 [AWS, 2025; Medium, 2026]

本文基於 14 篇學術與實務文獻（含 6 篇 A 級論文、4 篇官方文件），透過三角佐證揭示：治理層是企業 agent 架構中最弱、最不標準、最廠商鎖定的環節 [Context Kubernetes, 2026; Kai Waehner, 2026]。我主張：declarative orchestration 類比 Kubernetes 的設計範式，能為知識治理提供可驗證的架構藍圖 [Context Kubernetes TLA+ 驗證達 4.6M 狀態無違規]。

研究背景與動機

[李飛飛主筆]

為什麼我們要關心 agent 權限模型？因為自主 agent 已不是實驗室概念——2026 年，40% 的企業應用整合了決策、任務與推理能力的自主系統 [AetherLink, 2026]。但這些 agent 的治理層仍在初期：Microsoft Copilot、Salesforce Einstein、AWS Bedrock 各自建立權限模型，卻無法互通 [Kai Waehner, 2026]。

ImageNet 教會我：數據與基準能推動整個領域——但技術必須以人為中心。agent 編排器的設計也是如此：我們需要回答「誰被排除在外」「誰承受風險」「誰能審計決策」。這不是純技術競賽，而是關乎人類如何安全地將意圖委派給機器的治理問題。

文獻探討

[李飛飛主筆]

核心文獻 1：Context Kubernetes (2026)

Charafeddine Mouzouni 的 Context Kubernetes 論文 [arxiv:2604.11623] 提出企業知識編排的六個核心抽象，並以 Kubernetes 容器編排為藍本設計三層權限模型。其關鍵貢獻是 strict-subset invariant：agent 權限永遠是授權人類權限的真子集。TLA+ 形式驗證覆蓋 4.6M 可達狀態，無一違反此不變式。

這個架構的價值在於：它不依賴 LLM 的「誠實」（system prompt 可被繞過），而是從系統層強制執行邊界。這讓我想起 HAI 的核心主張：我們不能假設技術會自發地服務人類福祉，必須在設計階段嵌入約束。

核心文獻 2：Agentic Confidence Calibration (2026)

arxiv:2601.15778 揭示 agent 信心校準的挑戰：早期低信心決策（如錯誤選擇工具）會毒化整個執行路徑。傳統 LLM 校準指標（ECE / Expected Calibration Error）在 agent 環境下失效，因為不確定性在多步驟軌跡中累積而非獨立 [Agentic Confidence Calibration, 2026]。

另一篇 arxiv:2601.07264 驗證：工具調用 agent 的校準誤差高於獨立 LLM [The Confidence Dichotomy, 2026]。這對編排器設計有直接啟示：信任不能只在 LLM 層量化，必須擴展到執行軌跡層。

核心文獻 3：CogniWall 與 LangGraph 權限強制 (2025)

Dev.to 技術博客 [Securing LangGraph, 2025] 提出 CogniWall 短路架構：在 LLM 生成工具調用後、函式執行前插入決定性驗證層。這與 system prompt 提示工程形成對比：後者依賴 LLM 的確率性合規（99% 有效 = 1% 失敗率在生產環境無法接受），前者在毫秒內攔截違規調用。

AWS Well-Architected Framework [GENSEC05-BP01, 2025] 同樣建議：agent 執行角色須最小權限，並透過 permission boundary 設定最大權限上限。這些實務指引與學術研究一致：靜態授權 + 動態決策的矛盾必須在架構層解決。

文獻缺口與本研究貢獻

現有文獻聚焦單一平台（AWS / Microsoft / Salesforce），缺乏跨平台治理框架。Context Kubernetes 的 declarative orchestration 類比雖有創新，但僅為單一來源提案，尚未被其他學術或實務文獻驗證 [本研究 Phase 4 三角佐證]。

核心論證

論證 1：strict-subset invariant 是權限模型的數學基礎

[李飛飛主筆]

agent 權限設計的第一性原理：agent authority ⊂ human authority（永遠是真子集，非平行或超集）。這不是工程慣例，而是可形式化驗證的不變式。

Context Kubernetes 以 TLA+ 證明此不變式在 4.6M 狀態空間中無違規 [Charafeddine Mouzouni, 2026]。AWS 的 permission boundary [GENSEC05-BP01, 2025] 與 Medium 博客的零信任原則 [Robert Saghafi, 2026] 從實務角度支持同一主張。

三角佐證 ✅：三個獨立來源（學術論文 + 官方文件 + 實務分析）。

為什麼這重要？因為人類意圖委派給 agent 時，委派≠授權全權。我在 Stanford HAI 常說：技術應服務人類福祉。agent 若能超越授權者權限，福祉就無從談起。

論證 2：工具級權限攔截必須在執行前決定性驗證

[李飛飛主筆]

第二個主張：權限檢查應在「LLM 生成工具調用後、函式執行前」攔截，而非依賴 system prompt。

CogniWall 短路架構實證 [Dev.to, 2025]：財務限額規則（amount ≤ 500）在毫秒內攔截 LLM 試圖執行的 amount=5000 調用。AWS [GENSEC05-BP01, 2025] 要求執行時政策評估（runtime policy enforcement）。Medium 博客 [Robert Saghafi, 2026] 提出四層控制，其中「工具級授權」與「執行時政策評估」為核心。

三角佐證 ✅：三個獨立來源（技術博客 + 官方文件 + 實務分析）。

system prompt 的問題在哪？它假設 LLM 會「誠實」遵守指示。但 99% 有效在生產環境就是不夠——1% 失敗率可能是 $5000 違規退款或未授權數據訪問。這讓我想起 ImageNet 早期：我們不能假設模型會「理解」標籤語義，必須用架構約束保證正確性。

論證 3：excessive agency 在機器速度下呈指數級放大

[李飛飛主筆]

第三個主張：agent 的過度授權風險在機器速度下放大——agent 可秒級執行數千次操作，靜態授權模式已不適用。

AWS 定義 excessive agency：agent 判定最佳解決方案是超出範疇的廣泛行動（非惡意，而是自動化的意外後果）[GENSEC05-BP01, 2025]。Medium 博客闡述 [Robert Saghafi, 2026]：「人類用戶的輕微過度授權，在 agent 秒級千次執行下變得嚴重」。HyphenX 量化 [Best Practices, 2025]：agent 可秒級執行數千次操作，credential validity window 必須縮短。

三角佐證 ✅：三個獨立來源（官方文件 + 博客 + 實務指南）。

這是我在 HAI 治理研究中常強調的：規模改變性質。100 次人工審批的流程，放到自動化 agent 每分鐘 1000 次執行時，風險面積擴大 10 倍。社會影響必須先於技術部署。

論證 4：orchestrator-agent 信任校準需動態重評估循環

[李飛飛主筆]

第四個主張：orchestrator 應透過 ECE / OCR 等信心指標量化 agent 信任度，並在信任分數低於閾值時觸發重評估循環。

arxiv:2507.10571 [Orchestrator-Agent Trust, 2025] 實證：零樣本場景下 trust-aware orchestration 達 77.94% 準確度改善。arxiv:2601.15778 [Agentic Confidence Calibration, 2026] 提出 ACC 框架，估算執行軌跡成功率。arxiv:2601.07264 [Confidence Dichotomy, 2026] 驗證：工具調用 agent 校準誤差高於獨立 LLM。arxiv:2404.09127 [Multi-Agent Deliberation, 2024] 透過多 agent 對話（CoT / PoT / Self-Ask 不同策略）改善校準。

三角佐證 ✅：四個獨立來源（全為學術論文）。

這讓我想起計算機視覺的 confidence thresholding：我們不只看模型輸出，還要看它對自己輸出的「信心」。agent 編排器也是如此——orchestrator 不能盲信 agent 回報，必須量化信任並動態調整。

論證 5：企業 agent 治理層為四層架構中最弱環節

[李飛飛主筆]

第五個主張：治理層是企業 agent 架構（推理引擎 / 工具層 / 記憶層 / 治理層）中「最弱、最不標準、最廠商鎖定」的環節。

Context Kubernetes 論文 [2026] 指出：現有平台（Microsoft / Salesforce / AWS / Google）無 vendor-neutral context governance。AetherLink 博客 [2026] 觀察：監管壓力推動 RBAC 與 audit trails，但治理層仍是最不成熟環節。Kai Waehner 分析 [Enterprise Agentic AI Landscape, 2026]：「Salesforce Agentforce 與 AWS Bedrock Agents 互通需自建 API bridge——平台本應消除的整合成本反成新鎖定」。

三角佐證 ✅：三個獨立來源（學術論文 + 博客 + 產業分析）。

這是典型的標準缺位問題。ImageNet 之所以推動領域躍遷，正因為它提供了統一基準。agent 治理若無統一標準，每家廠商各自為政，企業就被迫在「最佳工具」與「互通性」間抉擇。

論證 6：主流雲端平台 agent 權限依賴既有 RBAC，但 agent identity 治理仍在初期

[李飛飛主筆]

第六個主張：Microsoft Copilot、Salesforce Einstein、AWS Bedrock 的 agent 權限皆依賴既有 RBAC，但 agent identity 治理仍在初期。

Microsoft [Agent Governance, 2025] 透過 agent identity blueprint 自動建立 OAuth 2 delegated scopes，初始權限受限。Salesforce [Einstein Copilot Setup, 2025] 說明：agent 尊重標準 access control，user 能看的資料 = Copilot 能浮現的資料。AWS [GENSEC05-BP01, 2025] 要求 Bedrock Agents 執行角色最小權限。Kai Waehner [Enterprise Agentic AI Landscape, 2026] 總結：「Microsoft 最深企業 AI 鎖定（identity / governance / data access 全綁 Entra ID / Purview）」。

三角佐證 ✅：四個獨立來源（三家官方文件 + 產業分析）。

這反映技術演進的路徑依賴：雲端廠商先有 IAM / RBAC 基礎建設，agent 就直接繼承這套系統。但 agent identity 與 human identity 有本質差異——agent 可衍生子 agent、跨服務遷移、以不同 principal 身份執行。既有 RBAC 未必適配。

一家之言

[李飛飛主筆]

我主張：declarative orchestration 類比 Kubernetes，是解決 agent 治理標準化的可行路徑。

Context Kubernetes 論文提出的架構映射——knowledge context ≈ container workload、agent permission ≈ pod security policy——不只是類比，而是可驗證的設計範式 [TLA+ 4.6M 狀態驗證, 2026]。但這仍是單一來源提案（Phase 4 三角佐證未通過），學術與產業尚未廣泛驗證。

我為什麼支持這個方向？

  1. 數據即科學儀器：Kubernetes 已是容器編排的事實標準（10+ 年生產驗證）。知識編排若能複用其設計模式（YAML manifest / reconciliation loop / declarative config），就能縮短標準化週期。
  1. 治理與創新雙軸：declarative 模式讓權限可審計、可回滾、可版本控制——這正是 HAI 倡導的「創新+政策同步」。agent 不該在產品上線後才補治理層。
  1. 多元與包容：統一標準降低中小企業與開發者接入門檻。廠商鎖定會讓「誰有資源整合多平台 API bridge」成為競爭護城河，排除資源較少者。

但我也承認侷限：

  • K8s 類比未必適配所有知識治理場景（例如細粒度 real-time 信任校準）
  • 標準化需時間，短期內廠商仍會各自為政
  • 形式驗證有價值，但 4.6M 狀態覆蓋面相對有限（真實企業場景可能更複雜）

這不是「一勞永逸」方案，而是朝標準化邁出的第一步。ImageNet 也不是完美基準，但它推動了領域進步。

實踐建議

[Yann LeCun主筆]

好了，講完理論，來談實作。我見證 CNN 到 LLM——但真正的價值在於架構能否落地。

建議 1：採用宣告式權限配置（declarative permission manifest）

別再把權限寫在 Python if-else 或 system prompt 裡。用 YAML / JSON manifest 定義 agent 能調用哪些工具、對哪些資料有讀寫權限。

參考實作：CogniWall [Dev.to, 2025] 的 YAML 規則檔案：


rules:
  - name: financial_limit
    condition: "tool == 'refund' AND amount > 500"
    action: block
    message: "Refund exceeds $500 limit"

為什麼這比 prompt 工程好？

  • 可版本控制（git diff 可審查權限變更）
  • 可自動化測試（單元測試驗證規則）
  • 開源可複現（別家公司能直接 fork 你的規則）

建議 2：實作 task-scoped access（任務範疇訪問）

agent 不該有「所有可能任務」的權限集合。每次執行時，mint 一個僅含當下任務必要權限的 short-lived token。

AWS Well-Architected [GENSEC05-BP01, 2025] 建議：credential validity window 縮短至分鐘級。HyphenX [Best Practices, 2025] 量化：agent 秒級執行數千次操作，long-lived credential 是定時炸彈。

開源實作參考：AWS IAM session tokens、Azure Managed Identity、Google Workload Identity Federation（這些不是 agent 專用，但架構可移植）。

建議 3：建立 agent-to-agent trust calibration 監控

如果你用 multi-agent orchestration（LangGraph / AutoGen / MetaGPT），orchestrator 必須量化每個 agent 的信任度。

arxiv:2507.10571 [Orchestrator-Agent Trust, 2025] 提供 baseline：ECE / OCR / CCC 三指標。但別盲目套用——先問：

  1. 你的 agent 輸出是否有 ground truth 可驗證？（有 = 可用 ECE；無 = 改用 inter-agent agreement）
  2. 失敗成本是否高到需要 re-evaluation loop？（高 = 參考 Image-RAG 增強；低 = 直接降級到人工確認）

開源監控工具：LangSmith / Weights & Biases Traces（追蹤 agent 執行軌跡與信心分數）。

建議 4：參與 Model Context Protocol (MCP) 生態

Kai Waehner [Enterprise Agentic AI Landscape, 2026] 指出：MCP 2026/4 已有 10k+ 企業伺服器部署、97M SDK 下載。這是 Linux Foundation 管理的開放協議，Anthropic / OpenAI / Google / Microsoft / AWS 都採納。

為什麼重要？

  • 降低 vendor lock-in（agent-to-tool 介面統一）
  • 開源權重與封閉 API 可共存（我一直倡導的 Llama 生態就是這樣）

實作步驟：

  1. 檢查你的 agent 調用的工具是否有 MCP server（例如 Slack / GitHub / Google Drive）
  2. 若無，寫一個 MCP wrapper（通常 <100 行 Python）
  3. 發布到 MCP registry（類似 npm / PyPI，社群複用你的 wrapper）

建議 5：用形式驗證工具驗證權限不變式（optional，適合高風險場景）

Context Kubernetes 用 TLA+ 驗證 strict-subset invariant。如果你的 agent 處理金融交易 / 醫療數據 / 基礎建設控制，考慮形式驗證。

開源工具：

  • TLA+ Toolbox（學習曲線陡，但能 catch 設計階段的邏輯錯誤）
  • Z3 SMT solver（如 Mythos cage [arxiv:2604.20496, 2026] 用於檢測 CWE-190 算術漏洞）

別被「形式驗證」嚇跑——你不需要驗證整個系統，只驗證核心不變式（例如「agent 權限 ⊂ user 權限」）。

風險與限制

[Yann LeCun主筆]

講完建議，來談這份研究的侷限——我不喜歡只報喜不報憂。

限制 1：Context Kubernetes 類比未經廣泛驗證

declarative orchestration ≈ Kubernetes 只是單一論文提案 [Charafeddine Mouzouni, 2026]。Phase 4 三角佐證顯示：其他學術或實務文獻尚未驗證此映射的普適性。

風險：知識編排與容器編排可能有本質差異（例如 knowledge context 的語義漂移 vs. container image 的 immutability）。貿然套用 K8s 架構可能遇到適配問題。

緩解：先在小規模場景驗證（例如單一業務線的 agent 權限管理），再擴展到全企業知識治理。

限制 2：信任校準指標（ECE / OCR）在 agentic 環境下的有效性待驗證

arxiv:2601.07264 [Confidence Dichotomy, 2026] 證明：工具調用 agent 校準誤差高於獨立 LLM。但「高多少」「哪些場景失效」仍是開放問題。

風險：orchestrator 依賴錯誤的信任分數，可能錯殺正確 agent 輸出或放行錯誤輸出。

緩解：multi-agent deliberation [arxiv:2404.09127, 2024]——用多個 agent 投票，而非單一 confidence score。

限制 3：廠商鎖定短期內無解

Kai Waehner [Enterprise Agentic AI Landscape, 2026] 指出：Microsoft / AWS / Salesforce agent 互通需自建 API bridge。MCP 雖有進展，但生態仍在早期。

風險：企業若重度依賴單一平台（例如 Salesforce Agentforce 深度整合 CRM），切換成本極高。

緩解：採用 multi-cloud 策略（至少 2 家廠商），並在設計階段避免 platform-specific API（優先用 MCP 封裝的工具）。

限制 4：形式驗證覆蓋面有限

TLA+ 驗證 4.6M 狀態聽起來很多，但真實企業場景可能有無窮狀態空間（例如動態生成的 knowledge context、user-defined tools）。

風險：驗證通過≠生產環境無漏洞。

緩解：形式驗證 + runtime monitoring 雙軌並行。驗證核心不變式（design-time），runtime 監控實際執行軌跡（execution-time）。

限制 5：本研究未涵蓋 agent-to-agent 通訊安全

Phase 4 三角佐證聚焦單一 orchestrator 管理多個 agent。但現實場景中，agent 可能跨組織通訊（例如供應鏈 agent 與物流 agent）。

風險：agent-to-agent 訊息可能被中間人攻擊（MITM）或注入惡意 payload。

緩解：採用 end-to-end 加密（E2EE）+ message signing（例如 JSON Web Signature）。這超出本研究範疇，但實務部署必須考慮。
