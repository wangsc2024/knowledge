Dify — 開源 Agentic Workflow 平台深度洞察

2026-03 完成 $30M Pre-A 融資 | 全球 140 萬+ 部署

一、專案概述

Dify 是一個生產級 LLM 應用開發平台，將 AI 工作流、RAG 管線、Agent 能力、模型管理與應用可觀測性整合為單一產品。其核心設計哲學是「從原型到生產的橋樑」——透過視覺化畫布降低 AI 應用的開發門檻，同時保有企業級部署能力。

技術棧

  • 後端：Python（FastAPI）
  • 前端：React（TypeScript）
  • 資料庫：PostgreSQL + pgvector
  • 容器化：Docker Compose / Kubernetes
  • 向量儲存：內建 pgvector，可擴展至 Milvus、Chroma

二、核心架構三層模型

| 層級 | 功能 | 設計亮點 |

|------|------|----------|

| LLM Orchestration | 模型連接與切換 | 模型無關架構，支援數十家推理供應商 |

| Visual Studio | 拖拽式 AI 工作流設計 | 節點式畫布，含 LLM、RAG、條件分支、HTTP、程式碼執行節點 |

| Deployment Hub | 一鍵部署為 API/聊天機器人 | Backend-as-a-Service，完整 API 覆蓋 |

三、七大可借鏡技術項目

1. 節點式工作流引擎（Node-Based Workflow Canvas）

設計模式：DAG（有向無環圖）工作流，每個節點為獨立執行單元（LLM 呼叫、知識庫檢索、條件分支、程式碼區塊、HTTP 請求）。

可借鏡之處：

  • 我們的 config/pipeline.yaml 可參考 Dify 的節點定義格式，讓管線步驟具備條件分支與並行執行能力
  • Agent Node 同時支援 Function Calling 與 ReAct 兩種推理策略，可套用於 todoist agent 的任務決策
  • 45 分鐘完成客服工作流（vs LangChain 需一整天），證明視覺化編排的生產力倍增效果

技術洞察：畫布在 20+ 節點時出現效能下降，暗示前端渲染引擎可能採用即時計算而非虛擬化，這是大型工作流的已知瓶頸。

2. RAG 管線全生命週期管理

架構：

  • 文件攝入 → 自動分塊（可調 chunk size/overlap）→ 嵌入模型選擇 → pgvector 儲存 → 檢索
  • 支援 PDF、PPT 等常見格式的自動文字抽取
  • 50 文件知識庫測試中達到約 85% 檢索相關度

可借鏡之處：

  • 嵌入模型可在每個工作流節點獨立選擇，這種「每步驟模型彈性」直接對應我們 llm-router.yaml 的 per-task-type 路由
  • pgvector 作為向量儲存的「零額外依賴」方案（PostgreSQL 內建），可評估是否取代我們知識庫目前的向量方案

已知限制：metadata filtering 需 workaround，非原生支援——這是選型時的關鍵考量點。

3. 模型無關路由（Model-Agnostic Router）

設計：

  • 支援 OpenAI、Anthropic、Google、Meta、Azure、Hugging Face、Ollama 等數十家供應商
  • 每個工作流節點可獨立指定模型
  • 內建 Token 使用量與成本追蹤
  • 成本優化策略：分類任務用便宜模型，生成任務用高階模型

可借鏡之處：

  • 我們的 llm-router.yaml 已實作 task_type→model mapping，Dify 進一步在 workflow 層級做到 per-node 路由，粒度更細
  • Dify 的成本追蹤整合到 UI 面板，我們可在 tools/budget_guard.py 加入類似的 per-task 成本歸因

4. 可觀測性整合（Observability Stack）

內建能力：

  • 日誌顯示每個節點的輸入、輸出、Token 消耗量、執行時長
  • 原生整合 Opik、Langfuse、Arize Phoenix 三大可觀測性平台

可借鏡之處：

  • 我們的 post_tool_logger.py 已記錄結構化 JSONL，可參考 Dify 加入 per-node 執行時長（對應我們的 per-skill 耗時）
  • Langfuse 整合模式可作為未來接入外部可觀測平台的參考架構
  • 「企業級可觀測性仍在成熟中」——提醒我們內建日誌 + 外部平台是目前業界的最佳實踐組合

5. Agent 雙模式框架（Function Calling + ReAct）

設計：

  • Function Calling：直接工具呼叫，低延遲，適合確定性任務
  • ReAct：推理-行動循環，適合需要多步推理的複雜任務
  • 50+ 內建工具（Google Search、DALL·E、Stable Diffusion、WolframAlpha、程式碼直譯器）
  • 自訂工具透過 API 整合

可借鏡之處：

  • 我們的 todoist agent 目前主要是單步執行模式，可引入 ReAct 式推理做複雜任務的多步拆解
  • 工具註冊表設計（50+ 工具 → Plugin Marketplace）與我們的 skills/SKILL_INDEX.md 理念一致

已知限制：多 Agent 協作不如 CrewAI/AutoGen 成熟；工具呼叫失敗時的 Agent 除錯不透明。

6. 插件市集與擴展性（Plugin Marketplace v1.0）

設計：

  • 社群貢獻的插件，消除常見整合的客製程式碼
  • Slack、資料庫查詢等預建整合

可借鏡之處：

  • 我們的 27 個專案 Skill 本質上就是一個「內部插件系統」
  • Dify 的 Plugin Marketplace 提供了一個更正式的分發模型——版本管理、依賴宣告、安裝/解除安裝流程
  • 可考慮為 Skill 系統加入 install/uninstall 生命週期管理

7. Backend-as-a-Service 模式

設計：

  • 每個 AI 應用自動產生完整 REST API
  • 涵蓋業務邏輯整合所需的所有端點
  • 一鍵從 Workflow → 可部署 API

可借鏡之處：

  • 我們的 bot/ Express 伺服器可參考此模式，為每個自動任務產生標準化 API 端點
  • Dify 的「Prompt IDE」（模型效能比較環境）可啟發我們建立 prompt 版本比較機制

五、核心洞察與啟示

洞察 1：「視覺化 ≠ 玩具」範式確立

Dify 以 132K Stars 與 $30M 融資證明，視覺化工作流引擎已從「低程式碼玩具」進化為「生產級基礎設施」。關鍵在於它同時提供 API-first 的程式化存取，讓視覺化設計與程式碼控制並存。

洞察 2：RAG 已成 AI 平台的「標配」

所有主流 AI 平台（Dify、n8n、LangChain）都將 RAG 視為核心能力而非可選功能。pgvector 的零額外依賴方案降低了 RAG 的導入門檻，但 metadata filtering 的缺失暴露了「整合式 RAG」vs「專用 RAG 引擎」的取捨。

洞察 3：可觀測性是 AI 應用的「必經之路」

Dify 原生整合三大可觀測性平台（Opik、Langfuse、Arize Phoenix），反映業界共識：AI 應用的除錯不能靠 print，必須有結構化追蹤。這驗證了我們 post_tool_logger.py 結構化日誌的方向正確。

洞察 4：模型路由的粒度正從「應用級」走向「節點級」

Dify 支援每個工作流節點獨立選擇模型與參數，這比我們目前的 task_type 級路由更細緻。未來趨勢是「每個決策點都有最適合的模型」。

洞察 5：Plugin/Skill 生態是護城河

Dify 的 Plugin Marketplace v1.0、LangChain 的 Integration Hub——開源 AI 平台的競爭力最終取決於生態系統豐富度。我們的 27 Skill 系統具備類似潛力。

六、對 Daily-Digest-Prompt 專案的具體建議

  1. 管線節點化：將 config/pipeline.yaml 演進為節點式定義（含條件分支），實現「天氣不好時跳過戶外建議」等動態管線
  2. per-Skill 耗時追蹤：在 post_tool_logger.py 加入 skill_name + duration_ms 欄位，實現 Dify 式 per-node 可觀測性
  3. 模型路由細粒度化：llm-router.yaml 從 task_type 級擴展到 step 級，讓摘要用快速模型、分析用深度模型
  4. Skill 生命週期管理：為 SKILL_INDEX.md 加入版本、依賴、啟用/停用狀態，參考 Dify Plugin Marketplace
  5. RAG 向量方案評估：知識庫若遷移至 PostgreSQL + pgvector，可減少一個外部依賴
