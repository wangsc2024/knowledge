Claude Code Worktree Isolation 完整指南

Git 工作樹隔離、Agent 宣告式隔離與 WorktreeCreate/WorktreeRemove Hook 事件的多 Agent 安全並行開發模式

一、概述

Claude Code 於 v2.1.50（2026-02-23）正式引入 Worktree Isolation 機制，解決多 Agent 並行開發時的程式碼衝突問題。此功能利用 Git Worktree 為每個 Agent session 建立獨立工作目錄，各 Agent 在自己的分支和檔案空間中作業，共享相同的 Git 歷史和遠端連線。

核心價值：

  • 消除並行衝突：多個 Claude Code session 同時修改程式碼不再互相覆蓋
  • 宣告式隔離：Agent 定義檔加入 isolation: worktree 即自動啟用
  • 自動生命週期管理：session 結束時依據變更狀態自動清理或保留
  • 非 Git VCS 支援：透過 Hook 事件擴展到 SVN、Perforce、Mercurial

---

二、CLI 使用方式

2.1 基本啟動


# 指定 worktree 名稱
claude --worktree feature-auth

# 自動生成名稱（如 bold-oak-a3f2）
claude --worktree

# 搭配 tmux 在獨立 session 中啟動
claude --worktree feature-auth --tmux

2.2 目錄結構

每個 worktree 建立在專案根目錄下：


.claude/worktrees/{worktree-name}/   # 獨立工作目錄

分支自動建立為 worktree-{worktree-name}，從預設遠端分支衍生。

2.3 清理行為

Session 結束時根據變更狀態自動決策：

| 狀態 | 行為 |

|------|------|

| 無變更 | Worktree 和分支自動移除 |

| 有變更或提交 | Claude 詢問使用者是否保留或移除 |

建議在 .gitignore 加入：


.claude/worktrees/

清理殘留 worktree：


git worktree list    # 列出所有 worktree
git worktree prune   # 清理過期 worktree

2.4 Session 中途啟用

在已啟動的 session 中，可直接要求 Claude「在 worktree 中工作」，系統會自動建立 worktree 並切換。

三、Agent 宣告式隔離

3.1 Agent 定義檔

在 .claude/agents/ 目錄中定義 Agent 時，透過 isolation: worktree frontmatter 宣告自動隔離：


---
name: refactor-agent
description: Agent that performs isolated refactoring work
isolation: worktree
---
You are a refactoring specialist. Analyze the codebase and refactor...

3.2 行為特性

  • 每次 Agent 執行自動建立獨立 worktree
  • Agent 完成後無變更則自動清理
  • 有變更時保留 worktree 供人工審閱或合併
  • 搭配 Agent Teams 使用，每個 teammate 可在各自 worktree 中並行作業

3.3 Subagent 搭配

當 subagent 使用 isolation: "worktree" 時，其工作在獨立 worktree 中進行。subagent 結束時觸發 WorktreeRemove 事件進行清理。

四、WorktreeCreate Hook 事件

4.1 觸發時機

  • claude --worktree 啟動時，在 TUI 渲染前 觸發
  • subagent 使用 isolation: "worktree" 時觸發
  • 不支援 matcher：每次觸發都會執行

4.2 輸入 JSON


{
  "session_id": "abc123",
  "transcript_path": "/Users/.../.claude/projects/.../abc.jsonl",
  "cwd": "/Users/.../my-project",
  "hook_event_name": "WorktreeCreate",
  "name": "feature-auth"
}

| 欄位 | 說明 |

|------|------|

| name | worktree 的 slug 識別碼（使用者指定或自動生成） |

4.3 輸出行為

  • stdout 必須輸出：建立的 worktree 目錄的絕對路徑
  • Claude Code 使用此路徑作為隔離 session 的工作目錄
  • 非零 exit code：worktree 建立失敗
  • 只支援 type: "command" hook handler

4.4 設定範例（SVN）


{
  "hooks": {
    "WorktreeCreate": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'NAME=$(jq -r .name); DIR=\"$HOME/.claude/worktrees/$NAME\"; svn checkout https://svn.example.com/repo/trunk \"$DIR\" >&2 && echo \"$DIR\"'"
          }
        ]
      }
    ]
  }
}

4.5 非 Git VCS 整合模式

配置 WorktreeCreate hook 後，--worktree flag 會呼叫自訂 hook 而非執行 git 命令：

  1. Hook 從 stdin 讀取 JSON 取得 name
  2. 用該 VCS 的命令建立工作副本
  3. 將其他輸出導向 stderr（避免干擾路徑回傳）
  4. 最後一行 echo 工作副本的絕對路徑到 stdout

五、WorktreeRemove Hook 事件

5.1 觸發時機

  • --worktree session 結束並選擇移除時
  • subagent 使用 isolation: "worktree" 完成後
  • 不支援 matcher

5.2 輸入 JSON


{
  "session_id": "abc123",
  "transcript_path": "/Users/.../.claude/projects/.../abc.jsonl",
  "cwd": "/Users/.../my-project",
  "hook_event_name": "WorktreeRemove",
  "worktree_path": "/Users/.../my-project/.claude/worktrees/feature-auth"
}

| 欄位 | 說明 |

|------|------|

| worktree_path | 被移除的 worktree 目錄絕對路徑（即 WorktreeCreate stdout 輸出的路徑） |

5.3 行為特性

  • 無法阻擋移除：WorktreeRemove hook 不支援 decision control
  • 用於清理操作（移除版本控制狀態、封存變更等）
  • hook 失敗僅在 debug 模式記錄
  • 只支援 type: "command" hook handler
  • 未配置 WorktreeRemove 時，worktree 目錄留在磁碟上

5.4 設定範例


{
  "hooks": {
    "WorktreeRemove": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'jq -r .worktree_path | xargs rm -rf'"
          }
        ]
      }
    ]
  }
}

六、與 Agent Teams 的整合

6.1 並行安全

Agent Teams 中最常見的問題是檔案衝突：兩個 teammate 編輯同一檔案導致覆寫。Worktree Isolation 從根本解決此問題：


Teammate A → .claude/worktrees/auth-impl/     （獨立分支）
Teammate B → .claude/worktrees/api-refactor/   （獨立分支）
Teammate C → .claude/worktrees/test-coverage/  （獨立分支）

每個 teammate 在各自 worktree 中作業，完成後用 git merge 合併。

6.2 搭配品質閘門 Hook

  • TeammateIdle：teammate 閒置前檢查 worktree 中的建置產物
  • TaskCompleted：任務完成前驗證 worktree 中的測試結果

九、最佳實踐

9.1 何時使用 Worktree Isolation

| 適合 | 不適合 |

|------|--------|

| 多個 Agent 並行修改不同模組 | 單一 Agent 順序修改 |

| Agent Teams 的 teammate 各自負責 | 只做唯讀研究（不修改檔案） |

| 需要安全實驗性修改（可隨時丟棄） | 快速單檔修改 |

| 非 Git VCS 專案的隔離需求 | — |

9.2 與 Subagent 的選擇

| 維度 | Worktree Isolation | Subagent |

|------|-------------------|----------|

| 檔案隔離 | 完全隔離（獨立目錄） | 共享工作目錄 |

| 分支管理 | 自動建立獨立分支 | 無分支隔離 |

| 清理機制 | 自動（無變更則清理） | session 結束即清理 |

| 開銷 | 較高（建立 worktree + 分支） | 較低 |

| 適用場景 | 有寫入操作的並行任務 | 唯讀研究或快速查詢 |

9.3 Daily-Digest-Prompt 專案啟示

對於本專案的 Team 並行模式（run-*-team.ps1），Worktree Isolation 可在以下場景提升安全性：

  1. Todoist Phase 2 並行執行：N 個 Agent 同時執行任務時，若多個任務涉及同一檔案的修改，worktree 隔離可避免覆寫
  2. 系統審查 Phase 1：4 個 Agent 並行評估不同維度，若涉及自動修正，隔離可確保修正不衝突
  3. 自動任務中的程式碼修改：creative-game-optimize、qa-system-optimize 等涉及檔案修改的自動任務

---

十、完整 Hook 生命週期位置


SessionStart → UserPromptSubmit → [agentic loop: PreToolUse → PostToolUse/Failure → Stop] → SessionEnd
                                                     ↕
                              WorktreeCreate ←→ WorktreeRemove（獨立於主迴圈）

WorktreeCreate 和 WorktreeRemove 是獨立的設定/拆除事件，不屬於 agentic loop，在 session 建立/銷毀時觸發。
