一、專案概述

OpenClaw（前身為 Clawdbot → Moltbot → OpenClaw）是由 PSPDFKit 創辦人 Peter Steinberger 開發的開源個人 AI 助理。不同於傳統 AI 聊天機器人，OpenClaw 是一個自主型 AI Agent，能在使用者的本地機器上運行，透過訊息平台（WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等 20+ 平台）作為使用介面，執行真實操作：排程會議、分類信箱、填寫表單、執行腳本、預訂餐廳。

二、核心架構：Gateway 控制平面

2.1 Gateway 設計模式

OpenClaw 的核心是一個稱為 Gateway 的 Node.js 單進程，預設監聽 127.0.0.1:18789。Gateway 是中央控制平面，同時管理所有訊息平台連線。

訊息處理流程：

  1. 訊息從任一平台進入 Gateway
  2. Gateway 路由至對應的 Agent Session
  3. Agent 執行「大腦迴圈」（Brain Loop）
  4. 回應透過正確的通道返回

安全模型：預設綁定 localhost，遠端存取需透過 SSH Tunnel 或 Tailscale 明確配置。

2.2 Agent Brain Loop

Agent 的處理迴圈每次請求最多執行 20 次迭代：

  1. 組裝上下文（對話歷史 + 工作區檔案）
  2. 發送至已設定的 LLM
  3. 接收回應
  4. 執行請求的工具呼叫
  5. 串流回覆

關鍵設計：每個 Session 序列化處理，防止 Race Condition。

2.3 與 Daily-Digest-Prompt 架構對比

| 面向 | OpenClaw | Daily-Digest-Prompt |

|------|----------|--------------------|

| 控制平面 | Gateway（Node.js 單進程） | PowerShell 7 排程腳本 |

| 訊息介面 | 20+ 訊息平台 | ntfy.sh 推播 |

| Agent 引擎 | 內建 Brain Loop | Claude Code CLI (claude -p) |

| 排程機制 | Heartbeat Daemon（30 min） | Windows Task Scheduler |

| 記憶系統 | Markdown 檔案 | JSON 檔案（digest-memory.json） |

| 插件系統 | Skills（700+） | Skills（26 專案 + 56 全域） |

三、Heartbeat 自主排程系統

3.1 機制

每 30 分鐘，Heartbeat Daemon 自動喚醒 Agent：

  1. 讀取 HEARTBEAT.md 指令
  2. 判斷需要執行的操作
  3. 自主執行（無需人類提示）

3.2 成本優化策略

兩階段判斷：

  • 第一階段：確定性檢查（不消耗 LLM Token）
  • 第二階段：僅在偵測到顯著變化時，才升級至 LLM 處理
可借鑑點：這與 Daily-Digest-Prompt 的 Phase 0 預計算模式（PowerShell 預計算 cache/status.json，LLM 讀 boolean valid 欄位）異曲同工。OpenClaw 的兩階段設計可進一步啟發我們的快取降級策略。

四、Markdown 記憶系統

4.1 檔案結構

記憶儲存在 ~/.openclaw/workspace/ 下的純文字 Markdown 檔案：

| 檔案 | 用途 |

|------|------|

| AGENTS.md | 定義 Agent 行為 |

| SOUL.md | 人格設定（身份、行為、邊界） |

| TOOLS.md | 工具使用慣例 |

| MEMORY.md | 長期記憶 |

4.2 設計哲學

  • 人類可讀：任何時候都能直接檢視 Agent 的知識
  • 版本可控：支援 Git 版控，追蹤 Agent 知識演變
  • 語義搜尋：使用 SQLite + 向量嵌入實現跨對話語義搜尋
可借鑑點：OpenClaw 的 SOUL.md 概念可類比為 Daily-Digest-Prompt 的 CLAUDE.md。但 OpenClaw 將「人格」獨立為一個檔案，分離了行為定義與專案配置，這種關注點分離值得參考。

五、Skill 插件生態系統

5.1 架構

Skills 是模組化 Markdown 擴充，儲存在 ~/.openclaw/workspace/skills/：

  • ClawHub 登記冊：700+ 社群 Skills
  • 熱載入：安裝 Skill 無需重啟伺服器
  • 涵蓋領域：Gmail、GitHub、Notion、Obsidian、Spotify、Home Assistant、Twitter/X 等

5.2 MCP 整合

透過 mcporter 模組整合 MCP：

  • MCP 整合與核心運行時解耦
  • 新增或變更 MCP 伺服器無需重啟 Gateway
  • 保持靈活性和可擴展性
可借鑑點：OpenClaw 的 Skill 熱載入機制值得 Daily-Digest-Prompt 參考。目前我們的 Skill 是靜態載入的（Agent 啟動時讀取 SKILL_INDEX.md），若能實現動態 Skill 發現和載入，可提升系統靈活性。

六、爆發式增長的技術洞察

6.1 為何 60 天達 250K Stars？

  1. 產品-市場適配（PMF）：AI 能力成熟後，瓶頸從「Agent 能做什麼」轉向「在哪裡接觸 Agent」。OpenClaw 透過整合現有訊息平台解決了這個問題
  2. 零切換成本：使用者不需離開 WhatsApp/Telegram，直接在慣用平台上使用 AI Agent
  3. 本地優先：資料不離開使用者的機器，消除隱私疑慮
  4. MIT 授權：完全開源，吸引開發者有機貢獻
  5. 基礎設施定位：不與特定 AI 能力競爭，而是作為 Agent 中介軟體

6.2 開發強度與 AI 輔助開發

  • 68 天產出約 550K 行存活程式碼
  • ~102 commits/day, ~8K 淨新增行/day
  • 暗示大量使用 AI 輔助開發（雖未明確證實）

七、可借鑑的技術項目

7.1 對 Daily-Digest-Prompt 的啟發

| 借鑑項目 | OpenClaw 做法 | DDP 現況 | 改進建議 |

|---------|-------------|---------|--------|

| 確定性預檢查 | Heartbeat 兩階段：先確定性檢查，再 LLM | Phase 0 預計算 cache/status.json | 可擴展預檢查至更多場景 |

| 人格與配置分離 | SOUL.md 獨立定義人格 | CLAUDE.md 混合配置+行為 | 考慮抽出行為層 |

| Skill 熱載入 | 安裝無需重啟 | 靜態 SKILL_INDEX.md | 動態 Skill 發現機制 |

| 多通道統一路由 | Gateway 控制平面 | 單通道 ntfy.sh | 如有需求可擴展至多通道 |

| Markdown 記憶 | 純文字 + Git 版控 | JSON 檔案 | Markdown 更利人類審閱 |

| MCP 解耦整合 | mcporter 獨立模組 | 直接在 prompt 中整合 | 可參考解耦設計 |

7.2 通用技術借鑑

  1. 本地閘道模式：單進程控制平面統一管理多通道，適用於任何需要多來源整合的 Agent 系統
  2. Session 序列化：防止並行處理導致的 Race Condition，簡單但有效
  3. 成本控制分層：確定性檢查 → LLM 處理的兩階段模式，可大幅降低 API 成本
  4. 平台化思維：將 Agent 定位為基礎設施而非應用，透過插件生態擴展能力
