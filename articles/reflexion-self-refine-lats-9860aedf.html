Reflexion、Self-Refine 與 LATS — LLM Agent 自我反思架構的工程化設計模式

主題概述

自我反思（Self-Reflection）已成為 Agentic AI 最具影響力的設計模式之一。其核心理念是：不信任模型的第一次輸出，而是讓 Agent 暫停、批判自身成果、迭代改進直到達到品質門檻。從 2023 年 Reflexion 與 Self-Refine 論文發表，到 2025-2026 年 Process Reward Models（PRM）與多 Agent 辯論的成熟，反思模式已從簡單的「檢查你的答案」提示，演化為一套完整的工程化框架生態系。

本文系統整理三大核心反思架構（Reflexion、Self-Refine、LATS）的機制差異、工程實踐要點與已知陷阱，為 Agent 系統設計者提供可落地的選型與實作指引。

---

一、三大反思架構比較

1. Reflexion（Shinn et al., 2023）— 跨試次語言強化學習

核心機制：Agent 在多次嘗試間以自然語言記錄失敗原因，存入「情節記憶」（episodic memory），後續試次讀取記憶避免重蹈覆轍。

三元件架構：

  • Actor：基於當前任務與歷史反思生成動作（可用 CoT 或 ReAct）
  • Evaluator：評分模組（二元通過/失敗、啟發式規則、或 LLM 評判）
  • Self-Reflection Model：接收失敗軌跡與評分，生成自然語言反思

效能：

  • HumanEval pass@1 從 76.4% 提升至 91%（+15 pp）
  • HotPotQA 精確匹配提升 3-5 個百分點
  • 成本遠低於微調（fine-tuning），僅需額外推理花費

適用場景：程式碼生成、多步推理、需要從錯誤中學習的任務

工程要點：

  • 反思記憶需持久化（JSON/向量資料庫），跨 session 保留
  • 記憶視窗建議 3-5 條，過多反思反而增加噪訊
  • 評估器盡可能使用確定性工具（單元測試、型別檢查）而非 LLM 判斷

2. Self-Refine（Madaan et al., 2023）— 單次任務內迭代精煉

核心機制：Generate → Feedback → Refine 三步迴圈，同一 LLM 同時擔任生成者、批評者與修正者，無需額外訓練資料或強化學習。

迴圈結構：

  1. 初始生成（Initial Output）
  2. 回饋（Feedback）：模型接收 prompt + 初始輸出，指出問題的定位（localization）與改進指示
  3. 精煉（Refine）：根據回饋修正輸出
  4. 重複步驟 2-3 直到滿足停止條件

效能：

  • 跨 7 項任務平均提升 ~20%（人類評估與自動指標）
  • GPT-4 + Self-Refine 在程式碼優化、數學推理、對話生成均顯著優於單次生成

適用場景：內容生成打磨、程式碼審查、創意寫作、文件準備

工程要點：

  • 溫度不對稱：生成用較高溫度（0.7-0.8 鼓勵創意），批評用較低溫度（0.2-0.3 確保一致性）
  • 成本優化：批評角色可用較便宜的模型（如 GPT-4 生成 + GPT-3.5 反思）
  • 迭代上限：通常 1-3 輪最佳，超過反而引入新錯誤或過度對沖
  • 不適合需要跨試次學習的場景（此時用 Reflexion）

3. LATS（Language Agent Tree Search, Zhou et al., 2023）— 蒙地卡羅樹搜尋 + 反思

核心機制：將推理、行動與規劃統一在樹搜尋框架中，結合 Monte Carlo Tree Search（MCTS）與 LLM 的生成和評估能力，系統性地探索多條解題路徑。

六步操作循環：

  1. Selection：從根節點沿最有前景的路徑選擇
  2. Expansion：在選定節點展開新的候選動作
  3. Evaluation：LLM 評估候選節點的價值
  4. Simulation：模擬候選路徑的未來軌跡
  5. Backpropagation：將評估結果回傳至祖先節點
  6. Reflection：在回溯階段生成語言反思，指導後續搜尋

效能：

  • HumanEval pass@1 達 94.4%（SOTA）
  • HotPotQA EM 達 0.61
  • WebShop 平均分 75.9

適用場景：高風險任務、複雜多步問題解決、需要探索多條路徑的場景

工程要點：

  • 計算成本極高：每個任務需要數十至數百次 LLM 呼叫
  • 僅用於高價值任務（合規審查、關鍵程式碼、安全評估）
  • 可搭配快取和剪枝策略降低成本

二、反思模式選型矩陣

| 維度 | Simple Reflection | Self-Refine | Reflexion | LATS |

|------|-------------------|-------------|-----------|------|

| 複雜度 | 低 | 中 | 中高 | 高 |

| LLM 呼叫數 | 2x | 3x(1+cycles) | 每試次 3x | 數十至數百 |

| 記憶 | 無 | 單次任務內 | 跨試次持久 | 樹結構 |

| 最佳用途 | 低風險快速任務 | 品質精煉 | 適應性學習 | 高風險複雜推理 |

| 成本 | $ | $$ | $$ | $$$$ |

三、多 Agent 反思模式（進階）

Generator-Critic 配對

兩個獨立 Agent 分別負責生成與批評，迭代精煉。避免單模型的「自我一致性陷阱」（self-consistency trap）。

Multi-Agent Reflexion（MAR）

多個推理人格（diverse reasoning personas）+ 裁判模型（judge），將「行動、診斷、批評、聚合」分離為獨立角色，減少共享盲點。MAR 在 HumanEval 上從 76.4 提升至 82.6（+6.2 pp）。

Process Reward Models（PRM）

對推理的每一步打分，而非只評估最終輸出。ThinkPRM（2025）以更少的訓練標籤達到強效能，實現細粒度反饋與早期錯誤偵測。

四、已知陷阱與緩解策略

| 陷阱 | 描述 | 緩解策略 |

|------|------|----------|

| 自我一致性陷阱 | 模型自信地辯護錯誤答案 | 使用不同模型/溫度進行批評，或外部工具驗證 |

| 諂媚式反思 | 模型同意自己的初始輸出而非真正批評 | 明確批評指令：「找出至少 2 個問題」 |

| 品質退化 | 過度反思引入新錯誤或過度對沖 | 設硬性迭代上限（2-3 輪），監控品質趨勢 |

| 成本放大 | 每輪反思倍增計算成本 | 分級策略：簡單任務跳過反思，複雜任務才啟用 |

| 無限迴圈 | 反思與精煉永不收斂 | 設終止條件：固定次數、品質門檻、收斂偵測 |

五、工程化落地建議（可直接實施的設計決策）

1. 分級反思策略


if task_risk == "low":
    use direct_generation  # 無反思
elif task_risk == "medium":
    use self_refine(max_cycles=1)  # 1 輪 Self-Refine
elif task_risk == "high":
    use reflexion(max_trials=3)  # 3 次 Reflexion
elif task_risk == "critical":
    use lats(max_depth=5)  # LATS 樹搜尋

2. 溫度不對稱原則

  • 生成角色：temperature=0.7-0.8（鼓勵多樣性）
  • 批評角色：temperature=0.2-0.3（確保一致判斷）
  • 精煉角色：temperature=0.5（平衡創意與約束）

3. 外部接地（Grounding）

反思品質的關鍵在於是否有外部事實錨點：

  • 程式碼任務 → 單元測試、型別檢查
  • 事實性問答 → 搜尋引擎驗證
  • 數學推理 → 計算器確認
  • 內部 LLM 判斷僅作為最後手段

4. 反思記憶管理

  • 記憶格式：結構化 JSON（含 timestamp、task_context、failure_reason、improvement_plan）
  • 記憶視窗：最近 3-5 條，過舊的壓縮或淘汰
  • 持久化：向量資料庫或本地 JSON 檔案
  • 檢索：語義搜尋匹配相關歷史反思

5. 品質監控

  • 追蹤每輪反思的品質分數變化
  • 若連續 2 輪分數未提升 → 提前終止
  • 記錄反思命中率（reflection hit rate）：反思實際改善輸出的百分比
