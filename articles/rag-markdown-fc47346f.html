RAG 知識庫匯入工作流實務 — 結構化 Markdown、去重查詢與驗證回寫

研究日期：2026-03-18

研究類型：技術研究

今日任務涉及技術：[Vite 靜態網站同步、GitHub 推送、RAG 知識庫匯入、AI 技術研究工作流]

本次研究主題：RAG 知識庫匯入的結構化 Markdown 設計、去重查詢與驗證回寫流程（因知識庫已有結構化輸出與 Vite 同步筆記，但尚無聚焦匯入工作流本身的專門研究）

技術概述

RAG 知識庫的研究任務，真正容易出錯的地方通常不在「有沒有寫出筆記」，而在「能否穩定匯入、避免重複、確認成功、把結果回寫到後續流程」。2025-2026 年的實務趨勢很明確：上游用結構化輸出降低格式漂移，下游用 deterministic ID 或 idempotent upsert 保證重試安全，中間再以 Markdown 作為人類可審閱與機器可索引的共同格式。對 daily-digest-prompt 這類多 Agent、自動化、Windows 環境專案而言，最穩的做法不是追求單一步驟成功，而是建立「先查重、再匯入、再驗證、最後回寫」的閉環流程。

核心概念與原理

1. 內容格式：以 Markdown 為主體，metadata 最小但完整

研究筆記應區分兩層：

  • contentText：完整 Markdown，提供全文搜尋、RAG 檢索、人工審閱
  • metadata：至少包含 title、tags、source、日期、研究主題

這種設計有兩個好處。第一，Markdown 本身兼具可讀性與可分塊性；第二，後端可依標題、段落、清單、程式碼區塊做 chunking。Weaviate 在 2025-09 的 chunking 文章指出，RAG 成敗往往不是檢索器本身，而是資料如何被切塊；若來源是 PDF 等視覺格式，應先轉成結構化文字，例如 Markdown，再做 chunking，否則欄位、段落與順序容易混亂。

2. 上游結構化輸出：先保證 schema，再談匯入

若研究內容來自 LLM，自由文字直接進入匯入步驟會帶來兩種風險：

  • 欄位缺漏，例如漏掉 tags 或 summary
  • 型別漂移，例如陣列變字串、null 處理不一致

OpenAI 在 Structured Outputs 文件與文章都強調：json_schema + strict: true 能把模型輸出限制在指定 schema 內，而不是只保證「看起來像 JSON」。這對自動研究流程特別重要，因為你不是在做一次性對話，而是在做可重試的資料管線。實務上應把「研究摘要 JSON」與「匯入筆記 JSON」視為兩份契約：

  • 研究摘要 JSON：給 Agent 間交接與結果檔
  • 匯入筆記 JSON：給 KB API

兩者結構不必相同，但都應有明確 schema 與驗證。

3. 去重策略：標題相似度不足，必須結合內容與時間

單靠標題判斷重複容易失準。較穩健的去重流程應為：

  1. 先查 registry，確認 3 天內是否有完全相同 topic
  2. 再查 KB hybrid search，判斷是否已有高度相似筆記
  3. 最後才決定是換題、換角度，或直接補強既有系列

這種三層去重很適合本專案：

  • registry 解決「最近做過沒」
  • KB search 解決「知識庫裡是不是已經有了」
  • series 管理解決「是不是該從 foundation 進到 mechanism / application」

4. 成功驗證：匯入成功不等於流程完成

最常見的假成功是：HTTP 200 回來了，但沒有 note id、沒有寫入索引、沒有回寫 registry。正確驗證至少包含：

  • API 回應是否有 id 或 noteIds
  • 是否能再查到該筆筆記
  • registry / continuity 是否完成回寫
  • 最終結果 JSON 是否標示 kb_imported=true/false

如果缺少這一步，自動任務很容易在 UI 顯示成功，實際上 Phase 3 或下游卻找不到產物。

最佳實踐與常見陷阱

最佳實踐

  • 以 Markdown 作為知識正文，保留標題、列表、程式碼區塊，方便 chunking 與全文索引。
  • 匯入前先做 hybrid search 去重，再做 registry 冷卻判定。
  • 以 deterministic ID 或等價的 idempotent key 思維設計匯入，避免重試造成重複資料。
  • 讓匯入流程具有「可驗證完成」條件，例如必須拿到 noteIds[0] 才算成功。
  • 將 summary、topic、kb_imported 同步回寫結果檔，讓後續 phase 可直接消費。

常見陷阱

  • 只驗證 JSON 可解析，卻不驗證欄位語意與必填欄。
  • 只有標題去重，沒有看內容相似度與近期研究紀錄。
  • 重試匯入時使用隨機 ID，導致同一筆研究生成多份重複資料。
  • 成功寫入 KB 後未更新 registry，下一次任務又重做同一主題。
  • Windows 環境用 inline JSON POST，導致編碼或 quoting 問題；改用檔案式 payload 更穩定。

與本專案的關聯

這個主題直接對應 daily-digest-prompt 的每日研究任務。專案目前的風險不是「不會研究」，而是：

  • 多個 Agent 都會產生 JSON 與 Markdown
  • KB 可用時要匯入，不可用時要優雅降級
  • 後續 Phase 3 仰賴固定結果檔與 KB 狀態

因此最適合本專案的工作流是：

  1. 先建立 results/todoist-auto-*.json 佔坑檔
  2. 讀取 registry / continuity / 今日任務
  3. 萃取技術主題並做 KB 去重
  4. 產出結構化 Markdown 筆記
  5. 以 JSON 檔發送匯入請求
  6. 驗證回應中是否含 note id
  7. 更新 registry、series、continuity、最終結果檔

這樣做的價值在於，即使任務半途失敗，也不會讓系統誤以為「什麼都沒發生」。

程式碼範例


{
  "notes": [
    {
      "title": "RAG 知識庫匯入工作流實務 — 結構化 Markdown、去重查詢與驗證回寫",
      "contentText": "# 標題\\n\\n## 技術概述\\n- 使用 Markdown 作為全文索引內容\\n- 匯入前先做 KB 去重\\n",
      "tags": ["技術研究", "RAG 知識庫匯入", "daily-digest"],
      "source": "import"
    }
  ],
  "autoSync": true
}


$body = @{
  notes = @(
    @{
      title = "RAG 知識庫匯入工作流實務 — 結構化 Markdown、去重查詢與驗證回寫"
      contentText = (Get-Content -Raw temp-tech-note.md)
      tags = @("技術研究", "RAG 知識庫匯入", "daily-digest")
      source = "import"
    }
  )
  autoSync = $true
} | ConvertTo-Json -Depth 8

$result = Invoke-RestMethod -Uri "http://localhost:3000/api/import" `
  -Method Post `
  -ContentType "application/json; charset=utf-8" `
  -Body $body
