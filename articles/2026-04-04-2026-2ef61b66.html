二、核心洞見：三層回饋機制的結構同構

洞見 1：Agent↔System↔Governance 的三層閉環同構

| 層級 | 回饋機制 | 時間尺度 | 記憶載體 | 本系統對應 |

|------|---------|---------|---------|----------|

| L1 Agent | Reflexion verbal memory + Self-Refine critique | 秒～分鐘 | 短期語言記憶 | continuity/*.json |

| L2 System | Circuit Breaker 狀態轉換 + Harness Gap Loop | 分鐘～小時 | state/*.json | Harness B4-A gap_tracker |

| L3 Governance | 約束演化（proposed→embedded）+ ADR 審查 | 天～週 | context/adr-registry.json | arch-evolution Skill |

關鍵發現：三層之間存在 資訊瀑布效應（Information Cascade）——

  • L1 Agent 的反思結果（key_findings）流入 L2 的 continuity 記憶
  • L2 的 Gap Loop 累積的 pattern 觸發 L3 的約束提案
  • L3 的新約束（embedded constraint）反向注入 L1 的 preamble 規則

這與 Adeptiv AI 在 2026 年提出的「治理即運行基礎設施」（Governance as Operational Infrastructure）理念高度吻合：靜態策略文件創造的是「控制的危險幻覺」，唯有嵌入執行工作流的動態治理才有實效。

洞見 2：Generator-Discriminator Gap 在品質閘門中的映射

研究指出 LLM 的「生成能力」與「鑑別能力」之間存在顯著落差（Generator-Discriminator Gap），且此落差具有 任務依賴性——結構化任務（如 3-SAT）的自我鑑別準確率遠高於理解類任務。

映射到本系統：tools/quality_gate.py 的 dual_mode（確定性 + LLM 評估）正是對此 Gap 的工程解法：

  • 確定性層（B2-A）處理結構化檢查（欄位完整性、格式合規）→ 鑑別準確率高
  • LLM 層（B2-C）處理語義品質（內容深度、創新性）→ 鑑別準確率較低，需旗標控制

啟示：應為 B2-C 引入 Process Reward Model（PRM） 思維——不僅評估最終輸出，也對中間步驟（研究搜尋品質、去重判斷、引用密度）進行分步評分。

洞見 3：Semantic Caching 與 Reflexion Memory 的功能等價

Semantic Caching 的核心機制——將請求轉為向量嵌入，在向量資料庫中比對相似度，超過閾值則返回快取回應——與 Reflexion 框架的 verbal memory 在功能上等價：

| 維度 | Semantic Cache | Reflexion Memory |

|------|---------------|------------------|

| 輸入 | 用戶 prompt embedding | 任務 + 反思文本 |

| 比對 | 餘弦相似度 > threshold | 語義相關性判斷 |

| 命中 | 返回快取回應 | 載入過去反思指導行為 |

| 未命中 | 轉發至 LLM | 從零開始推理 |

啟示：本系統的 config/dedup-policy.yaml 已在研究去重中實作了「語義快取」邏輯（hybrid search score > 0.85 即為重複）。可擴展為 自適應閾值——根據主題類別的知識密度動態調整：

  • 高密度主題（如 AI Agent 架構，已有 18+ 篇）→ 閾值提高至 0.90（更嚴格去重）
  • 低密度主題（如認知科學，僅 4 篇）→ 閾值降低至 0.75（鼓勵探索）

洞見 4：Context Length Penalty 與 Continuity 記憶的權衡

研究顯示當前 SoTA 模型在 8-32K tokens 以上出現效能退化（「注意力分散」效應），長執行序列導致 Agent 提前停止或遺失焦點。

映射到本系統：

啟示：可依任務 token 預算動態調整 continuity 保留筆數（研究類保留 3 筆、簡報類保留 5 筆）。

三、六項系統優化建議

建議 1：Draft-Verify 擴展為 Mini Self-Refine Loop

  • 現狀：templates/shared/draft-verify.md 執行一輪 Draft→Verify
  • 優化：在 Phase D 加入 structured critique prompt（「列出本次輸出的三個最弱環節」），Phase V 針對性修正
  • 預期效果：品質分數 +0.3～0.5（參考 Self-Refine 論文的迭代增益曲線）
  • 實施複雜度：低（僅修改模板文件）

建議 2：dedup-policy.yaml 引入自適應閾值

  • 現狀：固定 saturation_threshold=3
  • 優化：新增 adaptive_threshold 段落，依 diversity_guard.categories 各類別的現有筆記數動態計算
  • 公式：effective_threshold = base_threshold + log2(category_count) * 0.02
  • 預期效果：高密度主題自動收緊去重，低密度主題鼓勵探索
  • 實施複雜度：中（需修改去重判斷邏輯）

建議 3：quality_gate.py 引入分步評分（PRM 思維）

  • 現狀：B2-A 確定性 + B2-C LLM 終態評估
  • 優化：在 B2-A 增加 3 個中間檢查點（搜尋品質、去重合規、引用密度），各佔 20% 權重
  • 預期效果：更早發現品質問題，減少終態重工
  • 實施複雜度：中（修改 quality_gate.py）

建議 4：continuity max_runs 動態化

  • 現狀：所有任務固定 max_runs=5
  • 優化：在 config/memory-policy.yaml 增加 runs_by_task_type 映射（research:3, briefing:5, optimization:4）
  • 依據：研究類任務 context 較重，減少保留筆數可降低 context length penalty
  • 實施複雜度：低（配置修改 + preamble 讀取邏輯微調）

建議 5：L1→L2 回饋橋接——自動提取 Gap Signal

  • 現狀：continuity 的 key_findings 僅供同任務下次參考
  • 優化：在 on_stop_alert.py 增加邏輯——若 continuity 中連續 2 次 status=partial/failed，自動寫入 improvement-backlog.json
  • 預期效果：Agent 級失敗模式自動升級為系統級改進項目
  • 實施複雜度：中（修改 hook + backlog_writer）

建議 6：治理可觀測性指標對齊

  • 現狀：check-health.ps1 有 4 個健康度區塊
  • 優化：參考 Adeptiv AI 的「30+ operational metrics」框架，增加 3 個治理指標：

- 約束新鮮度（constraint freshness）：ADR 平均年齡

- 回饋迴路延遲（feedback latency）：從 gap 發現到約束嵌入的平均天數

- 自適應比率（adaptation ratio）：已嵌入約束 / 已提案約束

  • 實施複雜度：中（修改 check-health.ps1 + 新增 adr-registry 統計邏輯）
