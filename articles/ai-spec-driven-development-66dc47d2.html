AI 需求工程的最後一哩：Spec-Driven Development 與 PR Acceptance Criteria Gate

技術概述

2025-2026 的 AI 需求工程實務，重點已從「讓模型幫你寫 user story」轉向「讓需求成為可持續驗證的主工件」。GitHub Spec Kit 把規格放在開發最前面，先生成 spec、再生成 plan 與 tasks；Cucumber Gherkin 讓 Rule 與 Scenario 直接承接業務規則與驗收條件；Atlassian Rovo Dev 已能在 pull request 階段比對 linked Jira work item 的 acceptance criteria，回報已滿足、缺漏或仍需人工判定。研究面也顯示，LLM 在 Requirements Engineering 的主流用例已集中在 elicitation、validation 與 test generation，但多數還停留在 controlled environment，因此真正可落地的關鍵不是「生成更多文字」，而是建立 spec-first、traceability、PR gate 與 deterministic validation 的閉環。

核心方法與工具

  • Spec-Driven Development（SDD）：用規格而不是程式碼當主工件。先明確寫出需求、驗收條件與待澄清事項，再生成技術計畫與任務。
  • [NEEDS CLARIFICATION] 守衛：在需求階段禁止模型自行補空白，凡是授權範圍、錯誤語義、成功條件、例外情境不明確，一律標記待澄清。
  • Acceptance Criteria Normalization：把鬆散需求轉成可驗證敘述，每條 AC 只描述一件事，必須可觀測、可衡量、可對應測試。
  • Gherkin / BDD：以 Feature、Rule、Scenario、Given/When/Then 把 acceptance criteria 轉成可執行規格，避免需求與測試分離。
  • Requirements-aware PR Gate：在 PR 階段把 work item、branch name、acceptance criteria、測試結果與 code review 串起來，不讓「測試有過、需求沒滿足」的變更直接進主線。
  • Trust Layer：高風險需求仍需 schema、contract、formal rule 或人工審查，不能把 LLM 產出的需求直接當真相。

實際工作流程（step-by-step）

  1. 收集原始需求：來源可以是訪談逐字稿、issue、support ticket、故障回報、產品想法或既有 prompt/workflow 缺陷。
  2. 生成需求草稿：用 LLM 先整理成 feature summary、user story、business rule、non-functional constraint；未指定資訊全部標為 [NEEDS CLARIFICATION]。
  3. 切成單一結果導向的 stories：每個 story 只保留一個主要 user outcome，不把多個流程或多個角色塞進同一條。
  4. 正規化 acceptance criteria：每條 AC 必須寫成「前置條件 / 觸發 / 可觀測結果」，避免抽象語句如「更好用」「更穩定」。
  5. 下沉成 Gherkin：用 Rule 表示業務規則，用 Scenario 表示具體情境；Then 只驗證可觀測輸出，不驗證資料庫內部狀態。
  6. 產生 implementation plan：將規格映射到資料模型、API contract、驗證情境與任務切分，並保留需求到技術決策的追溯關係。
  7. 先建測試骨架再補實作：優先產生 contract / integration / e2e / unit 測試骨架，讓模型補齊程式碼時被規格與測試雙重約束。
  8. PR 階段做 requirements-aware 檢查：branch name 帶 Jira key，PR 連到單一 work item；系統比對 acceptance criteria 與變更內容，標示已滿足、缺漏或需人工確認。
  9. 把營運回饋回寫規格：incident、使用者回報、品質告警不只修 code，也要回寫到 spec 與 acceptance criteria，讓下一輪生成與驗證有依據。

程式碼 / 配置範例

1. 規格草稿模板（spec-first）


# Feature: ai_sysdev result reliability

## User story
As a maintainer, I want every `ai_sysdev` run to leave a valid result file,
so that Phase 3 never encounters `result_file_missing`.

## Clarifications
- [NEEDS CLARIFICATION: KB 匯入失敗時要標成 `partial` 還是 `failed`？]
- [NEEDS CLARIFICATION: `kb_imported` 在失敗路徑應為 `false` 還是省略？]

## Acceptance criteria
- AC-1: 任務開始後、任何外部呼叫前，`results/todoist-auto-ai_sysdev.json` 必須存在。
- AC-2: 若 KB 匯入失敗，結果檔仍須可被 `tools/validate_results.py` 驗證通過。
- AC-3: 若研究完成但 KB 匯入失敗，`status` 應為 `partial`，並保留錯誤原因。

2. Gherkin 驗收規格


Feature: ai_sysdev result guard

  Rule: Fail-safe result file exists before research starts
    Scenario: Task bootstraps placeholder result file
      Given the ai_sysdev task is triggered
      When the agent finishes loading the preamble
      Then results/todoist-auto-ai_sysdev.json should exist
      And the file should contain task_key "ai_sysdev"

  Rule: KB failure does not break result validity
    Scenario: KB import API is unavailable
      Given the knowledge base API returns an error
      When the agent finalizes the task
      Then the result status should be "partial"
      And the error field should explain the KB failure
      And validate_results.py should pass

3. PR Gate 範例


name: requirements-gate
on:
  pull_request:

jobs:
  acceptance-trace:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate result schema and BDD scenarios
        run: |
          uv run pytest tests/bdd -q
          uv run python tools/validate_results.py results/todoist-auto-ai_sysdev.json

效果量化（若有）

  • GitHub Spec Kit 官方 workflow example：/speckit.specify -> /speckit.plan -> /speckit.tasks 的示例把傳統約 12 小時的文件工作壓到約 15 分鐘。這是 toolkit 官方示例，不是獨立實驗，但可反映 spec-first automation 的流程設計目標。
  • 2025 Requirements Engineering SLR：彙整 74 篇 2023-2024 的 primary studies，指出 LLM 在 RE 的重點已從 defect detection / classification 轉向 elicitation、validation 與 test generation，而且常用 zero-shot / few-shot prompting。
  • 2025 Procedia CIRP 研究：在 textual product requirements 的品質評估比較中，GPT-4o 優於其他模型，表現接近人工評估者。
  • 限制：公開研究仍以 controlled environment 為主，真正跨團隊、跨 sprint 的產線 ROI 與穩定性數據還不多，因此現階段更可信的是「流程可驗證性」而不是「單次生成速度」。

與本專案應用場景

  • 本專案大量依賴 prompt、workflow、results schema 與守衛腳本，天然適合用 spec-first 流程整理需求，而不是直接改 prompt 後再補救。
  • 對 results/todoist-auto-*.json、tools/validate_results.py、done_cert、check-health.ps1 這種可明確驗證的機制，可先寫 acceptance criteria，再生成測試或 schema 驗證步驟。
  • 對 context/improvement-backlog.json 與 state/failed-auto-tasks.json 中的重覆故障，可先讓 LLM 抽出 user-facing impact，再整理成 Rule 與 Scenario，避免修一處破另一處。
  • 若未來導入 requirements-aware PR gate，最實際的切入點不是全面導入 Jira Product Discovery，而是先要求 branch / task / result schema 三者的 traceability 一致，再逐步把 BDD 與 PR 檢查接進來。

風險與注意事項

  • 不要在 spec 階段偷塞實作細節：Spec Kit 強調先寫 what 與 why，不要太早決定 how，否則需求文件會被框架選型綁死。
  • Acceptance criteria 不清楚，PR gate 就會失真：Atlassian Rovo Dev 明確要求 AC 要短、清楚、可衡量；若 work item description 為空，系統甚至不會搜尋 AC。
  • Then 要驗證可觀測結果：Cucumber 明確建議不要把資料庫內部狀態當 Then 的主要驗證標的，否則需求與使用者感知脫節。
  • LLM 需求生成仍有可信度邊界：最新研究雖然看好 LLM 在 RE 的產能，但仍指出多數研究只在 controlled environment 評估；高風險規格仍需 formal method、schema、contract test 或人工 review。
  • PR gate 只能擋已形式化的需求：若需求本身沒有 story ID、AC ID、rule 名稱與 branch 關聯，後面的自動檢查會變成表面化 lint，而非真正的 requirements traceability。
