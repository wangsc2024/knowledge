AI 程式碼審查的 CI/CD 落地指南：SARIF、CodeQL／Semgrep 與 LLM 修補建議分級閘門

研究日期：2026-03-11

研究類型：AI 系統開發 / 程式碼審查 / CI/CD 實作

---

技術概述

2025-2026 年的 AI 程式碼審查，已從「在 PR 上產生幾條自然語言評論」進化為可嵌入 CI/CD 的多層審查管線。成熟做法不是讓 LLM 直接當唯一裁判，而是把 靜態分析（CodeQL、Semgrep）、標準化結果格式（SARIF）、AI 審查代理（如 GitHub Copilot code review 或團隊自建 LLM reviewer）與 分級閘門政策（哪些問題阻擋合併、哪些僅提示）串成閉環。GitHub 官方文件顯示，Code Scanning 原生支援 SARIF 匯入；GitHub Copilot 也可對 pull request 做自動或手動程式碼審查；Semgrep 則提供在 CI 與 PR 中執行規則掃描的標準流程。這代表團隊可以用「規則引擎保底、LLM 補語意、人工做最終裁決」的方式，把 AI Code Review 真正落到工程治理，而不是停留在 demo。

核心方法與工具

1. SARIF 作為統一中介格式

  • SARIF（Static Analysis Results Interchange Format）是 GitHub Code Scanning 可接受的標準結果格式。
  • 優點是讓不同掃描器輸出落到同一個檢視平面，便於在 PR 與 Security 頁面集中處理。
  • 實務上可把 CodeQL 當「深度語意安全分析」、Semgrep 當「規則快速覆蓋」，若工具支援也可把自建檢查器輸出成 SARIF。

2. CodeQL：高可信度安全與資料流分析

  • 適合找 injection、taint flow、權限繞過、危險 API 使用等較深層問題。
  • 官方 github/codeql-action 已整合初始化、分析與結果上傳流程。
  • 建議把 CodeQL 結果視為高權重訊號，可直接進入 blocking gate。

3. Semgrep：規則化快速掃描與自訂政策

  • 適合快速涵蓋 framework anti-pattern、敏感字串、危險函式、團隊自訂安全政策。
  • 可先用社群規則或產品內建規則，再逐步加入 repo 專屬規則。
  • 建議把 Semgrep 用於 fast feedback，特別適合 pre-merge 檢查與增量掃描。

4. LLM Reviewer：補足語意、設計與可維護性判斷

  • 適合審查命名、抽象邊界、可測試性、規格一致性、文件缺漏、需求偏差。
  • 不建議直接作為唯一 blocking 來源，因為幻覺與不穩定性仍存在。
  • 最佳角色是：總結 diff 風險、產出修補建議、把靜態分析結果轉成更易行動的工程語言。

5. 分級閘門（Severity Gate）

  • blocker：CodeQL 高嚴重度、Semgrep 高信心安全告警、秘密外洩、授權或法規違反。
  • warning：LLM 發現可維護性風險、複雜度升高、測試缺漏、文件未更新。
  • info：命名、註解、重構建議、風格一致性。
  • 原則是 只有高信心、可重現、可追責的結果才阻擋合併；LLM 多數情況先做 advisory。

實際工作流程（step-by-step）

  1. PR 建立或更新：開發者提交 pull request。
  2. CodeQL 掃描：在 GitHub Actions 中執行 init 與 analyze，結果進入 GitHub Code Scanning。
  3. Semgrep 掃描：針對 diff 或整個 repo 執行規則掃描，若支援則輸出 SARIF，上傳到同一視圖。
  4. LLM 審查：使用 GitHub Copilot code review 或自建 reviewer，閱讀 diff、測試變更與規格檔案，產生語意層建議。
  5. 結果分級：將 CodeQL / Semgrep / LLM 發現映射到 blocker、warning、info。
  6. PR Gate 決策：

- 有 blocker → 失敗，要求修復後再跑。

- 只有 warning/info → 可合併，但要求 reviewer 確認或建立 follow-up issue。

  1. 修補建議回寫：LLM 根據 blocker/warning 自動生成修補片段、測試補強建議或文件更新清單。
  2. 人工覆核：人類 reviewer 對高風險建議做最終判定，避免 AI 誤修。
  3. 合併後追蹤：統計誤報率、審查耗時、阻擋率、修補接受率，作為下一輪規則與 prompt 調優基礎。

程式碼／配置範例

範例 1：GitHub Actions 整合 CodeQL + Semgrep


name: ai-code-review

on:
  pull_request:
  push:
    branches: [main]

jobs:
  codeql:
    runs-on: ubuntu-latest
    permissions:
      actions: read
      contents: read
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: javascript-typescript, python
      - uses: github/codeql-action/autobuild@v3
      - uses: github/codeql-action/analyze@v3

  semgrep:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      security-events: write
    steps:
      - uses: actions/checkout@v4
      - uses: returntocorp/semgrep-action@v1
        with:
          config: >-
            p/default
            p/security-audit

範例 2：PR Gate 判定邏輯


  review-gate:
    runs-on: ubuntu-latest
    needs: [codeql, semgrep]
    steps:
      - name: Decide merge gate
        run: |
          echo "若 CodeQL / Semgrep 有高嚴重度問題則 exit 1"
          echo "LLM 建議僅產出評論與修補建議，不直接當唯一 gate"

範例 3：LLM Reviewer Prompt 骨架


你是 PR reviewer。
請根據 diff、測試檔、README 與規格檔案，輸出：
1. 需求偏差
2. 安全風險
3. 可維護性問題
4. 缺少的測試
5. 建議修補方案

規則：
- 不要重複靜態分析已明確指出的問題
- 若信心不足，標示為 warning 而非 blocker
- 若涉及安全或資料外洩，必須附上重現條件

效果量化（公開資料與可追蹤指標）

  • GitHub 於 2024 年公開的 Copilot 研究指出，使用 Copilot 的開發者在指定任務中完成速度可明顯快於未使用組；此數據雖不是專指 code review，但說明 AI 對開發工作流具生產力提升潛力。
  • 在 CI/CD 落地時，較實用的量化方式不是只看「評論數」，而是追蹤：

- PR 平均審查等待時間（review lead time）

- 高嚴重度問題攔截率

- 誤報率 / 人工駁回率

- 修補建議採納率

- 合併後 hotfix 率與 rollback 率

  • 若團隊採「靜態分析 blocking + LLM advisory」模式，通常可先降低誤阻擋風險，再逐步提高 AI 自動化比例。

與本專案的應用場景（daily-digest-prompt 如何受益）

  1. Skill 與模板變更審查：對 skills/、templates/、scripts/ 的 PR 加上 Semgrep 規則與 LLM reviewer，可提早發現危險 shell 指令、路徑處理錯誤、Windows 相容性問題。
  2. JSON/YAML 政策治理：針對 config/、context/、results/ 建立規則，偵測 schema 漂移、欄位缺漏、未更新 retention / dedup 設定。
  3. 知識庫匯入品質控管：LLM 可檢查筆記是否缺少來源、是否低於字數門檻、是否沒有工作流程或範例碼。
  4. PR 自動摘要：讓 reviewer 快速理解某次研究任務改了哪些流程、是否有 registry / series / result 三者不同步問題。
  5. 安全防呆：對 curl、檔案寫入、暫存檔清理與 Windows nul 禁令建立專屬規則，降低自動化腳本失誤。

風險與注意事項

  • 不要讓 LLM 單獨決定 merge gate：AI 評論適合作輔助，不適合無條件阻擋。
  • SARIF 是格式，不是品質保證：若規則本身噪音高，集中展示只會集中干擾。
  • 需控制評論噪音：同一 PR 若同時有 CodeQL、Semgrep、LLM 留言，容易讓開發者疲勞；應做去重與分級。
  • 保護敏感程式碼與祕密：若使用外部 LLM reviewer，須確認資料治理與儲存政策。
  • 先從 advisory 開始：成熟順序建議是 可見化 → advisory → 部分 blocking → 政策化治理。
