Daily-Digest-Prompt 系統熵增與複雜度治理優化建議

從 19 個自動任務的擴張軌跡反思配置膨脹、認知負荷與結構性簡化路徑

研究日期：2026-03-01
任務 ID：6g2FfvWmxqhCWx85
差異化：前期報告覆蓋安全(2/26)、Token 經濟學(2/27)、排程效能(2/28)、Q2 路線圖(3/1 AM)；本文首次從 系統熵增 角度分析專案複雜度的結構性問題
分析基礎：586 行 CLAUDE.md、2224 行配置 YAML、24 個 Skills、120 條研究註冊表、200 筆排程記錄、636 筆 KB

---

一、問題本質：系統熵增的三個維度

1.1 配置膨脹（Config Entropy）

| 指標 | 數值 | 風險等級 |

|------|------|---------|

| CLAUDE.md 行數 | 586 行 | 🔴 高（LLM Context Window 固定消耗） |

| config/*.yaml 總行數 | 2,224 行（16 個 YAML） | 🟡 中（按需載入但維護成本高） |

| SKILL_INDEX.md 行數 | 231 行 | 🟡 中（每次啟動全量載入） |

| frequency-limits.yaml | 326 行（19 個自動任務定義） | 🟡 中（新增任務需同步 3+ 處） |

| routing.yaml | 233 行（20 個標籤映射） | 🟡 中 |

根因：專案從 2 月 14 日至今 16 天，配置量以每天約 140 行的速度增長。每新增一個功能（Skill/自動任務/路由規則），至少需要同步修改 3-5 個文件（CLAUDE.md + frequency-limits.yaml + routing.yaml + SKILL_INDEX.md + 可能的 prompt 模板）。這是典型的 散彈式修改（Shotgun Surgery） 反模式。

1.2 認知負荷（Cognitive Load）

Agent 每次啟動需要理解的配置上下文：

| 載入項 | Token 估算 | 必要性 |

|--------|-----------|--------|

| CLAUDE.md（系統注入） | ~4,000 tokens | 必要但可精簡 |

| SKILL_INDEX.md（每次必讀） | ~1,500 tokens | 必要但可分層 |

| preamble.md（共用前言） | ~200 tokens | 必要 |

| frequency-limits.yaml | ~2,000 tokens | 僅自動任務需要 |

| routing.yaml | ~1,500 tokens | 僅 Todoist Agent 需要 |

| 合計 | ~9,200 tokens | — |

問題：每日摘要 Agent 不需要 routing.yaml，Todoist Agent 不需要 pipeline.yaml，但 CLAUDE.md 把所有資訊都注入了。這相當於每次讀一本 300 頁的手冊，只為了找其中 30 頁的資訊。

1.3 研究註冊表膨脹（Registry Bloat）

| 指標 | 數值 | 說明 |

|------|------|------|

| 總條目 | 120 條 | 7 天滾動視窗理論上限 ~70 條（10/天），實際超標 71% |

| 檔案大小 | 65.2 KB | 每次研究任務全量讀取 |

| 單日最高 | 37 條（2/27） | 遠超預估的 10 條/天 |

| task_type 種類 | 14 種 | 去重判定需遍歷所有 |

根因：dedup-policy.yaml 設定 retention_days=7，但每天自動任務產出量（~30 條/天）遠超預估。120 條 × 平均 500 chars = 60KB 全量讀取，在 LLM context 中約佔 4,000-5,000 tokens。

二、五項結構性優化建議

建議 1：CLAUDE.md 分層瘦身（P0 — 影響每次執行）

現狀：586 行 CLAUDE.md 包含架構說明、目錄樹、流程圖、排程表、ADR 索引等，所有內容每次 session 全量注入。

建議：將 CLAUDE.md 拆分為三層：

  1. L1 核心層（目標 < 150 行）：僅保留規則、禁令、快速參考表
  2. L2 參考層（按需讀取）：架構說明、流程圖移至 docs/architecture.md
  3. L3 深度層（Skill 觸發）：ADR 索引、排程表等移至對應 Skill 或配置

預期效益：

  • Context Window 每次節省 ~2,500 tokens（從 ~4,000 降至 ~1,500）
  • 每日 44 次執行 × 2,500 tokens = 每日節省 ~110K tokens
  • CLAUDE.md 維護負擔降低 60%

實施風險：低。拆分純屬重組織，不影響功能。Agent 需要詳細資訊時按需讀取 L2/L3 文件。

建議 2：研究註冊表壓縮視圖（P1 — 影響研究任務品質）

現狀：120 條完整 entry，每條含 date/task_type/topic/kb_note_title/kb_imported/tags[]，全量讀取。

建議：實作兩層視圖：

  1. 摘要視圖（取代全量讀取）：

`json

{

"summary": {

"total": 120,

"by_type": {"todoist_research": 35, "shurangama": 11, ...},

"recent_3d_topics": ["Implementation Intentions", "Pong v1.2", "Frogger v2.1", ...],

"saturated_types": ["todoist_research"]

}

}

`

  1. 詳細視圖（僅在去重判定時按 task_type 篩選讀取）

預期效益：

  • 從 65KB 全量讀取降至 ~2KB 摘要 + 按需 ~5KB 篩選
  • Token 消耗從 ~5,000 降至 ~500（常態）+ ~1,000（判定時）

實施方式：

  • 在 research-registry.json 中新增 summary 頂層欄位
  • 每次寫入 entry 時同步更新 summary
  • 研究模板改為「先讀 summary → 需要去重時才讀對應 task_type 的 entries」

建議 3：自動任務新增流程標準化（P1 — 降低散彈式修改）

現狀：新增一個自動任務需要修改至少 5 個文件：

  1. config/frequency-limits.yaml（新增任務定義）
  2. emplates/auto-tasks/.md（建立模板）
  3. prompts/team/todoist-auto-.md（建立團隊 prompt）
  4. CLAUDE.md（更新架構說明和統計數字）
  5. skills/SKILL_INDEX.md（若涉及新 Skill）

建議：建立 Convention over Configuration 機制：

  1. 自動發現：PowerShell 腳本已改為動態掃描（Get-ChildItem todoist-auto-.md），模板也應自動發現 emplates/auto-tasks/.md
  2. 自動推導： requency-limits.yaml 的 asks 段落從模板檔案的 frontmatter 自動生成
  3. 檢查工具： alidate_config.py 新增「自動任務一致性檢查」— 確認 template、prompt、frequency-limits 三處同步

預期效益：

  • 新增自動任務從「修改 5 個文件」降為「建立 1 個模板 + 1 個 prompt」
  • 消除 CLAUDE.md 中手動維護統計數字的負擔
  • 降低命名不一致導致的靜默失效風險（歷史上已發生 3 次）

建議 4：run-fsm.json 生命週期管理（P2 — 預防資料膨脹）

現狀：18 條記錄（10.3KB），2 條仍為 running（含 system-audit 和 todoist），無清理機制。

問題：

  1. unning 狀態的條目永遠不會被清理（若進程異常結束）
  2. 無 max_entries 上限，檔案隨時間線性膨脹
  3. Agent 每次讀取全量 JSON，浪費 Context Window

建議：

  1. 新增 max_entries: 20 上限，超過時移除最舊的已完成條目
  2. 新增 stale_timeout_hours: 2：running 超過 2 小時自動標記為 imeout
  3. 在 un-*-team.ps1 腳本結尾清理完成條目（僅保留最近 10 條）

預期效益：

  • 檔案大小穩定在 ~6KB 以下
  • 消除殭屍 running 狀態
  • 減少 Agent 讀取的無效資料

建議 5：快取命中率歸零的根本修復（P0 — 已識別但未解決）

現狀：連續 6 天 cache_hits = 0，每次都打真實 API。cache-policy.yaml 的 TTL 已調整（todoist 45min, HN 180min），但仍為 0%。

根因深入分析：

  1. 團隊模式的孤立進程：每個 claude -p 是獨立進程，啟動時讀取的 cache/*.json 可能已過期
  2. 寫入-讀取競爭：Phase 1 的 5 個 Agent 同時啟動，各自讀取同一個快取檔案，但各自的 API 回應不會寫回給其他 Agent
  3. Todoist 即時性矛盾：cache-policy.yaml 明確標註「Todoist Agent 不使用快取（需即時資料）」，但 daily-digest 的 fetch-todoist 也不用快取，因為它的 prompt 直接呼叫 API

建議：

  1. Phase 0 預取：在 un-agent-team.ps1 的 Phase 1 啟動前，用 PowerShell 腳本預取 API 資料並寫入 cache/*.json，Phase 1 Agent 讀取快取而非呼叫 API
  2. 跨 Agent 快取共享：Phase 1 Agent A 完成 API 呼叫後，除了寫入 results/.json，也更新 cache/.json，後續 Agent 可受益
  3. 每日摘要專用快取策略：區分「Todoist 任務執行」（需即時）和「每日摘要報告」（可容忍 30 分鐘延遲）

預期效益：

  • 快取命中率從 0% 提升至 30-50%
  • Phase 1 平均耗時從 114s 降至 60-80s
  • 減少 API quota 消耗

五、結論：從「加法思維」到「減法思維」

專案過去 16 天的成長軌跡是典型的加法思維：發現問題 → 新增 Skill → 新增自動任務 → 新增配置。這在初期是正確的策略，但當系統達到 24 個 Skill、19 個自動任務、2,224 行配置後，邊際效益開始遞減，維護成本則加速增長。

「減法思維」的核心原則：

  1. 能合併的就合併：buddhist-research.md 已示範參數化模板共用 3 個任務，tech/ai 研究系列也可考慮類似做法
  2. 能自動推導的就不手動寫：統計數字、一致性檢查、檔案發現都應自動化
  3. 能按需載入的就不預載：CLAUDE.md 的架構說明、ADR 索引不需要每次注入
  4. 能壓縮的就不全量載入：研究註冊表、run-fsm.json 都應有摘要視圖
最好的程式碼是不需要寫的程式碼。最好的配置是不需要維護的配置。
