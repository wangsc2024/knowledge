治理事件追蹤整合：OTel Span × DDP Governance Event Log 雙軌可觀測性架構

主題 slug：govlog-otel-trace-integration

---

摘要（Executive Summary）

2025-2026 年 LLM 可觀測性生態已以 OpenTelemetry 為強制基礎層（生產採用率年增一倍至 11%），但同時暴露出其結構極限：OTel 只記錄「發生了什麼」，無法評判「是否合乎政策」。這催生出雙層架構：OTel 遙測管線（OTLP 攝取 + Collector 扇出 + ClickHouse 列存）作為不可篡改事件記錄，加上目的性治理層於其上。

對 DDP 系統而言，state/governance-events.jsonl 已是一套成熟的治理事件流，harness_trace_id 欄位提供了 trace 關聯的雛型。本簡報萃取 5 項洞見，說明如何以最小成本升格這套機制，形成「治理事件 × OTel span 雙軌關聯」可觀測性層。

---

核心洞見（5 項）

洞見 1：OTel Collector 作為治理政策執行點（Enforcement Point）

核心論點：將 OTel Collector 定位為 inline PII 過濾 + OPA 政策評估點，而非只是數據路由器。

證據來源：

  • Red Hat 生產 agentic tracing 指南（2026-04）描述 Collector processor pipeline 作為 policy enforcement point，實現 inline PII 過濾、OPA 評估、graduated enforcement
  • 實測結果：L0–L4 漸進式執行機制（L0=允許 → L1=告警 → L2=標記 → L3=重導向 → L4=隔離），P50 端到端延遲 127ms，VPR（違規偵測率）99.7%
  • 升級計算公式：min(4, base + floor(violations_in_window / k))，加上 per-agent circuit breaker

DDP 對應：DDP 的 hooks/ layer 已具備類似攔截能力（PreToolUse, PostToolUse）。OTel Collector processor 是同一概念在可觀測性層的對等實作——治理決策可從 hooks 下沉到 Collector，在資料進入後端前完成。

---

洞見 2：JSONL 治理事件 × OTel Span 雙流關聯設計

核心論點：大 payload（prompts/completions）應掛在 span events 而非 span attributes，因此 JSONL 治理事件可視為與 OTel span 樹並行的事件流，透過 trace_id / span_id join 實現統一稽核重建。

證據來源：

  • OTel LLM Working Group 建議：大 payload 掛 span events 而非 attributes（後端常有 attribute size 限制）
  • Langfuse OTLP 端點（/api/public/otel）的雙層映射：trace-level attributes（userId、sessionId、metadata、tags）vs. observation-level attributes（個別步驟→spans/generations/events）
  • 有 model 屬性的 span 自動轉換為 generation type；langfuse.* namespace 優先於通用 GenAI 語義約定

實際管線設計：


JSONL 治理事件（append-only）
   + OTel span 樹（attribute-level 輕量 metadata）
   → 週期性 join on trace_id + span_id
   → 統一稽核重建視圖

DDP 對應：state/governance-events.jsonl 已含 trace_id（harness_trace_id），缺少的是 span_id 與 W3C traceparent 標頭傳播。最小改造路徑：在 govlog 寫入時附加 span_id，即可實現雙流關聯。

---

洞見 3：W3C Trace Context 傳播完整性是跨 Agent 治理的關鍵瓶頸

核心論點：自動注入只在 HTTP 路徑有效；Claude Code CLI → PS1 → MCP server 等非標準邊界需手動 inject，任何斷點都讓跨 Agent 血統追蹤（lineage）失效。

證據來源：

  • arxiv 2604.05119 的 Governance Telemetry Schema（GTS/ATS）研究：跨 Agent 血統可見性是點對點 guardrails 無法填補的關鍵缺口
  • 案例：EU PII 跨 OrderAgent→ShippingAgent→AnalyticsAgent 的跨管轄違規，只有在完整委派鏈血統維護下才可偵測
  • Llama Stack → MCP servers 邊界需手動呼叫 opentelemetry.propagate.inject()；自動插桩不覆蓋非標準 hop

DDP 對應：DDP 的 PS1 排程（run-todoist-agent-team.ps1）→ claude -p 呼叫 → 各 Agent phase 形成一個跨越三個邊界的執行鏈；目前 harness_trace_id 由 PS1 產生傳至 Agent，但 Agent 內部子呼叫（KB 匯入、ntfy 發送）並無 span 傳播。

---

洞見 4：Langfuse 自架 TCO 分析——DDP 規模不達盈虧平衡點

核心論點：Langfuse v3 自架 TCO 約 $3,000–4,000/月，雲端 Pro $199–300/月；自架只在 2–5M events/月以上或嚴格資料落地需求時才划算。

證據來源：

  • v3 四服務架構：Postgres（交易/元資料）+ ClickHouse（分析/trace 儲存）+ Redis/Valkey（佇列/快取）+ S3 相容儲存（大型 payload）+ web/worker containers
  • ClickHouse 是主要成本驅動；Langfuse Enterprise 授權費需加計 ClickHouse 商業授權
  • 2025-06 起核心功能改 MIT 授權（tracing、evals、prompt management、annotation queues、playground），只有薄企業合規功能保持商業授權
  • DevOps 維護穩定期每週數小時，ClickHouse 升級時更多

DDP 對應（Taleb 反模式警告）：DDP 當前的 govlog 事件量遠低於 2M/月，Langfuse 雲端 Pro tier 在此規模下更具成本效益。自架 Langfuse 可作為中長期規劃，但短期應優先以輕量本地方案（如 Grafana Tempo + 自建 ClickHouse 小型單機）驗證雙流關聯 POC。

---

洞見 5：治理遙測 Schema（GTS）pattern — 標準擴展而非替換

核心論點：GTS/ATS 在標準 OTel span 之上附加治理元資料（source_agent, operation_class, jurisdiction, sensitivity, lineage, verification_status），形成「OTel GenAI 語義約定 + GTS 擴展」雙層 schema，不替換標準協議。

證據來源：

  • arxiv 2604.05119：GTS 模式定義 8 個治理維度（source agent、receiving agent、operation、classification、jurisdiction、sensitivity、lineage、verification status）
  • OTel 2025 趨勢報告：89% 的生產用戶視廠商規範符合性為關鍵；從供應商分發（Datadog、Grafana、Elastic）的比率 44%→60% YoY
  • langfuse.* attribute namespace 為同類設計先例——在標準 GenAI 語義約定之上加 product-specific namespace

DDP 對應：DDP govlog 已有隱式 GTS 欄位（task_key=source_agent, action=operation, loop_mode=enforcement_level, created_at=timestamp）。顯式化映射至 OTel attribute namespace（如 ddp.gov.*）即可與 GTS 模式對齊，無需重建 schema。

---

多透鏡洞察（L2 Analysis）

透鏡選擇

本次套用三組思考透鏡（來源：temp/insight-briefing-lens.json）：

  • Sam Altman（primary）：規模突破、長期投資、持續迭代 lens
  • Jensen Huang（primary）：工程加速、軟體定義、算力最大化 lens
  • Taleb（adversarial）：反脆弱、尾部風險、反模式識別 lens

框架化建議

Sam Altman 透鏡：治理可觀測性不是成本中心，是解鎖 Full Autonomy 的前提條件。只有當每個 Agent 決策都可追蹤、可重播、可問責，才能安全地放大自動化規模——這是 100× 而非 2× 的槓桿。OTel Collector as policy enforcement point 是把「監控」升格為「可信基礎設施」的關鍵轉折，值得長期投資。

Jensen Huang 透鏡：JSONL + OTLP 雙流架構是「軟體定義可觀測性」——不依賴昂貴 APM 授權，用 ClickHouse 列存引擎在標準硬體上達 10:1 壓縮；Collector processor pipeline 讓政策邏輯「燒入」資料管線，治理從應用代碼走向基礎設施。DDP 現有的 PowerShell 管線（deterministc layer first）正是這種思路的自然延伸。

Taleb 反模式檢查：

  1. ❌ 「監控幻覺」：只在 HTTP 邊界做 W3C Trace Context 傳播，以為覆蓋了完整可觀測性——實際 MCP、async subprocess、PWsh 跨進程邊界全是斷點，高嚴重度治理違規恰好在這些盲區發生
  2. ❌ 「Langfuse 自架迷信」：規模未達 2-5M events/月 就上 ClickHouse 集群，運維複雜度超過可觀測性帶來的收益
  3. ❌ 「完美 schema 等待主義」：等待 GTS 完整標準化才開始，錯失早期用 ddp.gov.* namespace 漸進驗證的機會

透鏡盲區自陳

三個 primary/adversarial 透鏡均偏向技術實現最優化；可能低估組織採用阻力（「誰來維護 OTel Collector 規則？」）和治理事件的隱私處理複雜度（govlog 本身可能含 PII，用 OTel 廣播前需先處理）。

---

建議行動

  1. 立即可做（一行修改）：在 tools/lib/govlog.py 的 log_event() 函式，額外記錄 span_id（可用 uuid4() 作臨時產生，後期替換為 OTel context propagation）——為雙流關聯奠基，不依賴完整 OTel 部署
  1. 短期 POC（1-2 週）：建立 tools/gov_trace_correlate.py：讀取 state/governance-events.jsonl，join context/continuity/*.json 中的 harness_trace_id，產生 state/gov-trace-correlation-report.json——用確定性 join 先驗證關聯性，不需 Langfuse
  1. 中期評估（觀察期後）：依 DDP govlog 實際月均事件量決定 Langfuse 部署策略——低於 500k events/月 → Langfuse Cloud Pro；500k–2M → Cloud Pro + export；2M+ → 考慮自架 Lite（Postgres only，不上 ClickHouse 集群）
  1. 長期基礎設施（6+ 個月）：在 PS1 排程調度層（run-todoist-agent-team.ps1）實作 W3C Trace Context 跨 phase 傳播，使 Phase 0→1→2→3 形成連貫 span 樹
