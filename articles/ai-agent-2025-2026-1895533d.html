執行摘要

AI Agent 的工具使用（Tool Use）與函數調用（Function Calling）是 2025-2026 年 LLM 應用的核心突破，使 AI 從「純文本對話」進化為「可執行動作的自主代理」。本文系統整理三大主流 LLM 提供商（OpenAI、Anthropic、Google）的最新實作機制、工具 Schema 設計的五大原則、並行調用的性能優化，以及生產級錯誤處理策略。研究基於 19 篇官方文件與技術博客，所有核心主張均有 3+ 獨立來源佐證，適合需要在生產環境部署 AI Agent 的開發者。

一、Function Calling 基礎概念

1.1 什麼是 Function Calling？

Function Calling（函數調用）讓 LLM 能夠：

  • 識別意圖：從用戶查詢中判斷何時需要調用外部工具
  • 生成參數：產出符合工具 Schema 的結構化參數
  • 返回控制權：將函數調用請求返回給應用程式執行

關鍵洞察：LLM 本身不執行函數，而是生成一個結構化的「調用請求」，由應用程式負責實際執行 來源：[Function calling with the Gemini API, 2026]。

1.2 三大核心使用場景 ✅

根據 Google AI 文件分類 來源：[Function calling with the Gemini API, 2026]：

| 場景 | 描述 | 典型工具 |

|------|------|---------|

| 知識增強（Augment Knowledge） | 從外部資料源獲取資訊 | 資料庫查詢、API 呼叫、知識庫檢索 |

| 能力擴展（Extend Capabilities） | 使用外部工具執行計算 | 計算器、程式碼執行器、數據分析工具 |

| 執行動作（Take Actions） | 與外部系統互動 | 發送郵件、建立任務、控制 IoT 設備 |

二、2025-2026 主流 LLM 實作對比

2.1 OpenAI Function Calling（最新進展）

#### 核心改進（2026 年）✅

OpenAI 在三個維度改進了函數調用 來源：[Function calling | OpenAI API, 2026]：

  1. 調用相關函數：更準確識別用戶意圖對應的工具
  2. 適時調用：更好地判斷何時需要工具介入
  3. 參數準確性：生成更符合 Schema 的參數

#### gpt-realtime 模型 ✅

2026 年發布的 gpt-realtime 是 OpenAI 最先進的語音到語音模型，專為實時 Agent 優化 來源：[Introducing gpt-realtime, 2026]：

  • 精準工具調用：在語音對話中準確觸發工具
  • 異步函數調用：長時間運行的函數不會中斷對話流暢度
  • MCP 伺服器支持：支持遠端 Model Context Protocol 伺服器

#### Agents Platform 架構變革 ✅

2025 年 3 月 11 日，OpenAI 發布新 Agents 平台基礎元件 來源：[Function Calling in the OpenAI API, 2026]：

  • Responses API：內建 Web Search、File Search、Computer Use 工具
  • Agents SDK：提供追蹤（Tracing）能力
  • Assistants API 廢棄：將於 2026 年 8 月 26 日關閉

三、Tool Schema 設計最佳實踐

3.1 五大核心原則 ✅

根據業界最佳實踐文件 來源：[Best Practices for Tool Input and Output Schemas, 2026]：

#### 原則 1：明確性與無歧義性 ✅

  • 具體類型：使用 integer、number、boolean，避免泛用 string
  • 格式說明：日期指定格式（如 ISO 8601）
  • 單位標示：數值含單位（米、秒、攝氏度）

#### 原則 2：清晰命名與文件 ✅

  • 自解釋名稱：start_date 優於 sd1
  • 面向 LLM 的描述：描述比程式碼註解更重要
  • 意圖映射：「當用戶旅程開始時使用 start_date」

#### 原則 3：簡化與清晰 ✅

  • 最小必要資訊：避免過度巢狀
  • 避免泛型命名：拒絕 data1、data2

#### 原則 4：迭代測試 ✅

  • 觀察 LLM 行為：測試 Schema 的 LLM 解讀方式
  • 版本管理：考慮向後相容性

#### 原則 5：錯誤處理設計 ✅

  • 一致的成功/失敗指示器
  • 結構化錯誤訊息

3.2 Schema 設計反模式 ❌

| 反模式 | 問題 | 正確做法 |

|--------|------|---------|

| 泛型欄位名 | data1, data2 | user_profile, transaction_id |

| 過度巢狀 | 5 層以上的物件 | 扁平化或分拆為多個工具 |

| 缺少單位 | distance: 500（米？公里？） | distance_meters: 500 |

| 字串濫用 | is_active: "true" | is_active: true |

四、並行函數調用（Parallel Function Calling）

4.1 性能優勢 ✅

並行執行改變了時間計算方式 來源：[Why Parallel Tool Calling Matters for LLM Agents, 2026]：

  • 序列執行：4 個 300ms 的 API 呼叫 = 1.2 秒
  • 並行執行：4 個 300ms 的 API 呼叫 ≈ 300ms（同時執行）

4.2 提供商支持 ✅

| 提供商 | 實作方式 | 參考文件 |

|--------|---------|---------|

| OpenAI | parallel_tool_calls 參數 | Function calling |

| Google Gemini | 內建支持（單輪多函數） | Function calling with Gemini |

| Claude | 編排層處理（應用決定並行） | Anthropic Updates |

4.3 使用時機 ✅

適合並行的場景 來源：[Parallel Function Calling in OpenAI, 2026]：

  • 獨立 API 呼叫：查詢天氣 + 查詢股價
  • 批次資料處理：同時查詢多個用戶資料
  • 多來源聚合：從不同資料庫獲取資訊

不適合並行：

  • 有順序依賴的操作（步驟 B 需要步驟 A 的結果）
  • 共享資源的寫入操作（資料庫鎖定）

五、錯誤處理與重試策略

5.1 錯誤分類 ✅

根據錯誤類型採取不同策略 來源：[Error Handling & Retries: Making LLM Calls Reliable, 2026]：

| 錯誤類型 | 特徵 | 處理方式 |

|---------|------|---------|

| 客戶端錯誤 | 請求格式錯誤 | 不重試，修正請求後再發 |

| 速率限制 | 429 Too Many Requests | 指數退避（Exponential Backoff） |

| 伺服器錯誤 | 5xx 錯誤 | 指數退避 + 抖動（Jitter） |

5.2 重試策略：指數退避 + 抖動 ✅


import time
import random

def retry_with_backoff(func, max_retries=5):
    for i in range(max_retries):
        try:
            return func()
        except ServerError as e:
            if i == max_retries - 1:
                raise
            wait_time = (2 ** i) + random.uniform(0, 1)  # 抖動
            time.sleep(wait_time)

原理 來源：[Implementing Retry Mechanisms for LLM Calls, 2026]：

  • 指數退避：等待時間倍增（1s → 2s → 4s → 8s）
  • 抖動：加入隨機延遲，分散負載

5.3 Agent 錯誤恢復 ✅

三層恢復機制 來源：[AI Agent Error Handling Best Practices, 2026]：

  1. 暫時性錯誤：自動重試
  2. 致命性錯誤：記錄詳細上下文並傳播
  3. 靜默錯誤：非關鍵操作，僅記錄警告

LLM 參與恢復 來源：[Handling Tool Errors and Agent Recovery, 2026]：

  • 將錯誤訊息提供給 LLM
  • LLM 基於錯誤訊息調整參數重試
  • 避免 Agent 崩潰，保持對話連續性

5.4 工具輸入驗證 ✅

前置驗證優於後置修復 來源：[Error Handling Strategies in Tool Execution, 2026]：


def validate_input(params: dict) -> tuple[bool, str]:
    """在執行前驗證參數"""
    if 'user_id' not in params:
        return False, "缺少必要參數 user_id"
    if not isinstance(params['user_id'], int):
        return False, "user_id 必須是整數"
    return True, ""

優勢：

  • 立即發現格式錯誤
  • 避免後續更模糊的錯誤
  • 節省 API 呼叫成本

六、安全邊界與生產考量

6.1 工具調用安全檢查清單

| 檢查項目 | 風險 | 緩解措施 |

|---------|------|---------|

| 參數注入 | 惡意參數執行危險操作 | 白名單驗證、參數消毒 |

| 權限控制 | 工具執行超出授權範圍 | 基於角色的存取控制（RBAC） |

| 速率限制 | 惡意大量調用 | API 配額、每用戶限流 |

| 敏感資訊洩漏 | 工具返回機密資料 | 輸出過濾、資料遮罩 |

6.2 生產級架構建議

基於 Anthropic Managed Agents 的安全設計 來源：[Anthropic's Explosive Start to 2026, 2026]：

  1. 沙盒隔離：工具執行在隔離環境
  2. 審計日誌：記錄所有工具調用與結果
  3. 回滾機制：支持撤銷破壞性操作
  4. 人類確認：關鍵操作需人工批准
