PowerShell 7 的 JSONL 結構化日誌查詢、效能優化與常見陷阱

今日任務涉及技術：PowerShell 7、JSONL 結構化日誌、phase_failure 排查、query-logs.ps1、結構化欄位 schema、知識庫去重。

本次研究主題：PowerShell 7 的 JSONL 結構化日誌查詢、效能優化與常見陷阱（因知識庫雖已有 JSONL 與 PowerShell 基礎筆記，但尚無直接面向 query-logs.ps1/log_audit 場景的深入研究）。

執行摘要

如果 daily-digest-prompt 要把 logs/structured/*.jsonl 當成可靠的排障資料面，關鍵不是「把物件寫成 JSON」而已，而是同時守住三件事：第一，檔案格式必須維持 JSONL/NDJSON 的基本契約，也就是每行一筆完整事件、UTF-8、避免 BOM，才能安全追加與逐行處理 [來源：JSON Lines, 2026][來源：NDJSON Spec, 2014][來源：about_Character_Encoding, 2026]。第二，欄位 schema 要先固定可跨來源查詢的核心欄位，例如時間、嚴重度、事件分類、檔案路徑與 trace 關聯，再把任務特有細節放到擴充欄位，否則日後做 phase failure 聚合時會很難寫穩定查詢 [來源：OpenTelemetry Logs Data Model, 2026][來源：ECS Guidelines and Best Practices, 2026][來源：ECS Base fields, 2026]。第三，PowerShell 7 雖然已具備很好的 JSON 與編碼能力，但若直接用 Out-File/> 寫物件、忽略 ConvertFrom-Json -AsHashtable、或沒控制 DateKind 與 append 編碼，仍然很容易造成資料不可逆、時間誤判或跨平台讀取失敗 [來源：Out-File, 2026][來源：ConvertFrom-Json, 2026][來源：PowerShell 7.5 GA, 2025]。

對本專案最實用的結論是：把 JSONL 當成「穩定事件列車」，以逐行或小批次讀取為主，寫入時明確輸出字串化 JSON 並指定 utf8NoBOM，讀取時先做 schema/JSON 驗證，再用可預測的欄位名稱聚合。這樣 query-logs.ps1 才能從「臨時 grep 工具」升級成可維護的 observability 工具。

研究背景

今日已完成任務中，log_audit 明確涉及 phase_failure 趨勢分析與 query-logs.ps1 修正；同日的 system_insight 也在觀察成功率、I/O 超標與公平性等指標。這代表專案已經不只是「留紀錄」，而是開始依賴結構化日誌做根因分析、健康診斷與後續修復決策。現有知識庫已經有 JSONL 格式與 PowerShell 7 自動化的基礎研究，但仍缺一篇把兩者接起來、直接回應專案 log pipeline 需求的應用層筆記。

核心發現

主張 1：JSONL 適合專案日誌，前提是把「每行一筆事件、UTF-8、無 BOM」當成硬性契約。已驗證 ✅（3+ 來源）

JSON Lines 明確要求每行是一個合法 JSON 值，且應採用 UTF-8，不應包含 BOM；這正是 append-only 日誌與串流處理最需要的性質 [來源：JSON Lines, 2026][來源：NDJSON Spec, 2014]。PowerShell 7 的文字輸出預設已是 utf8NoBOM，而 Microsoft 也明確建議為相容性避免在 UTF-8 檔案中使用 BOM [來源：about_Character_Encoding, 2026]。

對 daily-digest-prompt 的意義是：results/.json 可以是單一 JSON 文件，但 logs/structured/.jsonl 必須是可逐行掃描的事件流。只要某一行混入格式化表格、BOM、非 JSON 前綴或半行資料，後續 ConvertFrom-Json 與外部分析工具就會同時失效。

主張 2：高價值日誌不是「欄位越多越好」，而是先固定核心欄位，再把專案細節放擴充欄位。已驗證 ✅（4+ 來源）

OpenTelemetry 的 logs data model 將時間、嚴重度、trace context 與 attributes 分開，強調每筆事件都應能穩定表達發生時間、來源與上下文 [來源：OpenTelemetry Logs Data Model, 2026]。Elastic ECS 也建議優先填充 core fields，因為跨 dashboard、搜尋、報表與告警的可重用性建立在這些欄位之上 [來源：ECS Guidelines and Best Practices, 2026]。ECS 同時把 @timestamp 視為所有事件的必備欄位，並用 event.kind、event.category、event.type、event.outcome 做分類；log.level、log.file.path 等則描述記錄機制本身 [來源：ECS Base fields, 2026][來源：ECS categorization fields, 2026][來源：ECS Log fields, 2026]。

對本專案，建議至少固定下列欄位：timestamp、level、event_kind、event_type、event_outcome、agent、task_key、session_id、trace_id、message、log_file、attributes。真正會變動的任務細節，例如 tool_name、kb_note_id、phase、duration_ms，則放在擴充欄位。這樣 phase_failure、success_rate、cache audit 才能共用查詢骨架。

主張 3：在 PowerShell 中，寫 JSONL 時不能把 `Out-File`/`>` 當作結構化輸出工具。已驗證 ✅（3+ 來源）

Out-File 會走 PowerShell formatting system，文件直接提醒這種輸出除非輸入已是字串，否則不適合程式化處理 [來源：Out-File, 2026]。同時，Out-File -Append 與 >> 不會自動匹配既有檔案編碼，若未明示 -Encoding，就可能在 append 時悄悄破壞原始檔案的編碼一致性 [來源：about_Character_Encoding, 2026]。JSONL 本身又要求每行是獨立 JSON 值與 UTF-8，這使得「先把物件序列化成單行 JSON 字串，再用明確編碼附加」成為比直接導出物件更安全的策略 [來源：JSON Lines, 2026]。

實務上應使用：


$line = ($event | ConvertTo-Json -Compress -Depth 8)
Add-Content -LiteralPath $logPath -Value $line -Encoding utf8

不要使用：


$event | Out-File $logPath -Append

因為後者很可能產出表格化文字、截斷欄位或造成編碼不一致。

主張 4：大型 JSONL 查詢應以逐行或小批次流式處理為主，不要預設整檔 `-Raw` 載入。已驗證 ✅（3+ 來源）

Get-Content 預設逐行回傳內容；-Raw 會把整個檔案當成單一字串回傳 [來源：Get-Content, 2026]。-ReadCount 可控制每次送入 pipeline 的行數，值越大通常能降低總處理時間，但也會延後第一批輸出回來的時間，在大型檔案上差異尤其明顯 [來源：Get-Content, 2026]。JSONL/NDJSON 的設計本來就是為了逐筆處理與串流，而不是每次重建整個文件 [來源：JSON Lines, 2026][來源：NDJSON Spec, 2014]。

對本專案來說，查詢 logs/structured/*.jsonl 時，建議預設走：


Get-Content $path -ReadCount 200 |
  ForEach-Object {
    foreach ($line in $_) {
      if (-not [string]::IsNullOrWhiteSpace($line) -and ($line | Test-Json)) {
        $obj = $line | ConvertFrom-Json -AsHashtable -DateKind String
        if ($obj.event_outcome -eq 'failure') { $obj }
      }
    }
  }

這比先 Get-Content -Raw 再切行更穩定，因為它保留了 JSONL 一行一事件的處理模型，也避免大檔一次灌入記憶體。

主張 5：PowerShell 7.5 的 JSON 行為已更適合日誌分析，但要顯式使用新參數才有價值。已驗證 ✅（3+ 來源）

ConvertFrom-Json -AsHashtable 可以避免大小寫近似鍵被吃掉、空字串鍵報錯，且對某些資料結構處理更快；在 PowerShell 7.3 起還能保留鍵順序 [來源：ConvertFrom-Json, 2026]。PowerShell 7.5 新增 -DateKind，可明確指定時間字串保留為字串、轉為 UTC 或保留 offset，避免日誌時間在跨時區分析時被默默改寫 [來源：ConvertFrom-Json, 2026][來源：PowerShell 7.5 GA, 2025]。而 Test-Json 在 PowerShell 7.4 起改以 System.Text.Json 做嚴格解析，並支援 schema 驗證，這使它很適合先驗證再轉換的日誌管線 [來源：Test-Json, 2026]。

因此，若本專案要提升 query-logs.ps1 的健壯性，推薦流程應改為：Test-Json 檢查單行合法性與 schema → ConvertFrom-Json -AsHashtable -DateKind String 解析 → 自行決定哪些欄位要再轉型。

核心概念與原理

JSONL 真正有價值的地方，不只是「每行一個 JSON」這麼簡單，而是它把日誌從「文本」變成「事件資料列」。每筆事件都可以獨立讀取、獨立驗證、獨立聚合，所以當某一行損壞時，不需要整檔回滾；當檔案很大時，也不需要先還原成一個巨大 JSON array [來源：JSON Lines, 2026][來源：NDJSON Spec, 2014]。

在 observability 層，OpenTelemetry 把 log record 拆成 Timestamp、Severity、TraceId/SpanId、Body 與 Attributes，其背後邏輯是把「跨系統必需欄位」與「單次事件的上下文」分層 [來源：OpenTelemetry Logs Data Model, 2026]。Elastic ECS 則把這種思路落到可搜尋的欄位標準：@timestamp 是必備；event.kind/category/type/outcome 用來做分類；log.level、log.file.path 描述日誌來源；如果沒有對應欄位，再用 custom fields 補充 [來源：ECS Base fields, 2026][來源：ECS categorization fields, 2026][來源：ECS Log fields, 2026]。

PowerShell 端的難點則在於它同時是 shell、物件管線與文字工具。這很強，但也意味著「畫面上看起來正確」不代表「寫到檔案後仍是可機器解析的 JSONL」。Out-File 針對的是顯示輸出；真正的 JSONL 應該先序列化成一行純字串，再明示編碼寫入 [來源：Out-File, 2026][來源：about_Character_Encoding, 2026]。讀取時則要先決定你要的是整檔字串、逐行字串，還是小批次行陣列；這會直接影響記憶體足跡與查詢速度 [來源：Get-Content, 2026]。

最佳實踐與常見陷阱

最佳實踐

  1. 固定最小 schema。至少包含 timestamp、level、event_kind、event_type、event_outcome、message、agent、task_key、session_id、trace_id、attributes。
  2. 寫入時先 ConvertTo-Json -Compress，再 Add-Content -Encoding utf8；不要把物件直接 pipe 到 Out-File。
  3. 讀取時先 Test-Json，再 ConvertFrom-Json -AsHashtable -DateKind String，避免時間欄位與鍵名被自動改寫。
  4. 查大檔時使用 Get-Content -ReadCount N 做小批次處理；只有小檔或一次性轉檔才考慮 -Raw。
  5. 把 trace_id、phase、tool_name、kb_note_id 放進 attributes 或穩定擴充欄位，讓 phase 之間能做關聯查詢。

常見陷阱

  1. 直接 >> 追加字串而未指定編碼，導致同一檔案混入不同編碼 [來源：about_Character_Encoding, 2026]。
  2. 用 Out-File 輸出物件，實際寫進去的是格式化表格，不是 JSON [來源：Out-File, 2026]。
  3. 對大型 .jsonl 直接 Get-Content -Raw | ConvertFrom-Json，因為 JSONL 整檔不是單一合法 JSON 文件 [來源：NDJSON Spec, 2014][來源：JSON Lines, 2026]。
  4. 忽略 ConvertFrom-Json -AsHashtable，讓大小寫近似鍵、空字串鍵或順序敏感欄位失真 [來源：ConvertFrom-Json, 2026]。
  5. 未顯式設定 -DateKind，跨時區聚合時把來源時間悄悄轉成本地時間 [來源：ConvertFrom-Json, 2026]。

與本專案的關聯

對 daily-digest-prompt，這份研究最直接的落地點有三個。

第一，query-logs.ps1 應該明確把「合法 JSONL 行」與「解析後事件」分成兩階段，先做 Test-Json，再轉成 hashtable。這可以避免單一壞行讓整段查詢中止，也讓 phase failure 的統計更穩。

第二，post_tool_logger.py、各 phase results、以及 system_insight 若要互相對照，最好逐步向共同欄位靠攏，例如 timestamp、event_outcome、task_key、agent、session_id、trace_id。這不一定要一次全面套 ECS，但至少要向 ECS/OTel 的欄位分層靠齊。

第三，若之後要把日誌匯入外部分析工具或知識庫，現在先把 UTF-8、無 BOM、單行 JSON、可分類欄位做對，未來遷移到 Elastic、OTel collector 或自建查詢索引才不需要重寫歷史資料。

程式碼範例


function Read-JsonlEvents {
    param(
        [Parameter(Mandatory)]
        [string]$Path,

        [int]$ReadCount = 200
    )

    Get-Content -LiteralPath $Path -ReadCount $ReadCount -Encoding utf8 |
        ForEach-Object {
            foreach ($line in $_) {
                if ([string]::IsNullOrWhiteSpace($line)) { continue }
                if (-not ($line | Test-Json)) { continue }

                $event = $line | ConvertFrom-Json -AsHashtable -DateKind String

                [pscustomobject]@{
                    Timestamp   = $event.timestamp
                    Level       = $event.level
                    Outcome     = $event.event_outcome
                    Agent       = $event.agent
                    TaskKey     = $event.task_key
                    SessionId   = $event.session_id
                    TraceId     = $event.trace_id
                    Message     = $event.message
                    Attributes  = $event.attributes
                }
            }
        }
}


function Write-JsonlEvent {
    param(
        [Parameter(Mandatory)]
        [string]$Path,

        [Parameter(Mandatory)]
        [hashtable]$Event
    )

    $line = $Event | ConvertTo-Json -Compress -Depth 8
    Add-Content -LiteralPath $Path -Value $line -Encoding utf8
}
