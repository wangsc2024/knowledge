AI 輔助突變測試（Mutation Testing）與 CI Quality Gate

技術概述（約 150 字）

突變測試透過對程式碼注入「突變」（例如把 > 改成 >=、把常數加一），再執行既有測試套件，檢查測試是否能偵測並殺死這些錯誤版本。相較於單純行覆蓋率，突變分數（Mutation Score, MSR）更能反映「測試是否真能抓錯」。現行工具鏈（Stryker、PIT、mutmut 等）已可與 CI 整合；大型語言模型則適合擔任測項設計助理（建議邊界案例、補強 assertion），但不應取代實際執行突變與測試——否則無法得到可稽核的 MSR。

核心方法與工具

| 生態 | 代表工具 | 備註 |

|------|-----------|------|

| JavaScript/TS | Stryker | 支援增量、並行、報表與 dashboard |

| Java/JVM | PIT (Pitest) | 與 Maven/Gradle 整合成熟 |

| Python | mutmut、Cosmic Ray | 適合腳本型專案與小型模組 |

| .NET | Stryker.NET | 與 dotnet test 管線銜接 |

指標：MSR = 被殺死的突變數 / 總突變數（已排除被標記為等價的突變）。另需區分「無法殺死」是因為測試不足，還是突變在語意上等價。

實際工作流程（step-by-step）

  1. 基線：確保單元測試在乾淨分支上全綠，並固定種子（若測試有隨機性）。
  2. 本地試跑：以預設或縮小範圍（單一模組）執行突變測試，產生 HTML/JSON 報表，檢視「存活突變」清單。
  3. 增量模式：在 CI 僅對變更檔或依賴圖子集跑突變（Stryker incremental、或僅對 PR diff），控制時間成本。
  4. 門檻策略：初期採 軟門檻（報告＋警告），待 MSR 穩定後再設 硬門檻（例如 MSR ≥ 某團隊共識值，且僅統計非等價突變）。
  5. LLM 輔助（可選）：將「存活突變片段＋對應測試」餵給模型，請其建議額外斷言或案例；人類審核後再合併。
  6. 治理：將 MSR、執行時間、突變數寫入儀表板或 results/*.json 類可稽核欄位，與 flaky 測試指標分開追蹤。

程式碼／配置範例

Stryker（`stryker.conf.json` 片段）


{
  "packageManager": "npm",
  "reporters": ["html", "clear-text", "progress"],
  "testRunner": "jest",
  "coverageAnalysis": "perTest",
  "mutate": ["src/**/*.js", "!src/**/*.spec.js"]
}

GitHub Actions（概念性 job）


mutation:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: "20"
    - run: npm ci
    - run: npx stryker run --incremental
實際快取目錄、並行數與 timeout 請依專案調整；Windows 排程／本機 runner 亦適用相同邏輯。

效果量化（實務觀察）

  • 產業與教學材料普遍認為：突變測試能揭露「高覆蓋但弱斷言」的測試；具體 MSR 目標應依專案風險與執行成本共識化，不宜抄固定百分比作為全專案 KPI。
  • 執行成本常為導入瓶頸：需搭配增量、並行、夜間全量與 PR 子集等分層策略。

與 daily-digest-prompt 的應用場景

  • 自動任務與 Hook：Python/PowerShell 測試可選定關鍵模組做突變門檻，與既有 pytest／品質閘門並列，驗證「守衛邏輯是否真的被測到」。
  • 多後端／沙箱備援：突變測試驗證的是本機測試套件，與 Cursor CLI／OpenRouter 配額無直接衝突；可放在非 LLM 關鍵路徑的 CI job，避免與 Phase2 搶佔時間。
  • 文件驅動架構：當 config/*.yaml 或 schema 變更時，可用突變測試確認 hooks／驗證器測試是否跟進。

風險與注意事項

  1. 等價突變：過多誤判會打擊信心，需使用工具兪示的 equivalent mutant 標記或審核流程。
  2. 時間與 CI 額度：全量突變可能極慢；必須增量與分層。
  3. LLM 幻覺：模型可能建議語意錯誤的測試；需 code review 與實際綠燈。
  4. 與 flaky 測試共存：flaky 會讓突變結果不穩定，宜先治理 flaky（見既有 CI/CD 智能化筆記）。
