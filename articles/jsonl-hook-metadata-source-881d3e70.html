JSONL Hook metadata.source 與 OpenTelemetry GenAI 資料來源語意對齊

執行摘要

本筆記承接 improvement-backlog backlog_kso_20260408_cache_source_logging：結構化日誌需具備可聚合的 API／快取來源維度，才能將快取命中率從「全域平均」下鑽到「依端點／依資料來源」的資料驅動 TTL 調優。OpenTelemetry GenAI 語意慣例以 gen_ai.data_source.id 標示 RAG／工具／外部儲存的資料來源識別項；本專案 Hook 層採 metadata.source（與既有 cause_chain.cache_source 並存）對齊同一治理意圖：在 api-call、cache-read、cache-write 類事件上，一律寫入可解析的來源欄位（無法判定時顯式為 unknown），避免下游分析將整批事件誤計為無來源。

研究背景

  • 既有 KB 與系統優化報告已指出：post_tool_logger 若未輸出 metadata.source，快取／API 事件在儀表上會呈現 unknown，阻斷分 API 的 miss 根因與 TTL 實驗設計。
  • OpenTelemetry 文件（Gen AI spans／attributes registry）建議在檢索與工具執行場景使用穩定的 data source 識別，與 db.* 等屬性並用時可進一步描述儲存後端 OpenTelemetry Gen AI spans、Gen AI attributes registry。

核心發現（三角佐證）

  1. 治理需求（專案內）：context/improvement-backlog.json 將「記錄 metadata.source」列為快取命中率提升的前置條件（✅ 內部 backlog）。
  2. 標準參照（社群／標準）：OTel GenAI 以 gen_ai.data_source.id 統一「代理／RAG 使用的資料來源」識別，與本專案 JSONL 的 metadata.source 可建立一對一語意對照（✅ 標準文件）。
  3. 實作現況（程式）：hooks/post_tool_logger.py 已透過 detect_api_sources／cause_chain.cache_source 推斷來源；本次補上 metadata.source 使與 cache_source_analyzer 等下游欄位期望一致（✅ 程式碼落地）。

實務應用建議

  1. 儀表與查詢：JSONL 聚合時以 metadata.source 為 group-by 主鍵之一；與 cause_chain.cache_status 交叉分析 hit／miss 比例。
  2. 與 OTel 匯出對齊：若未來 otel-log-standardizer 將 JSONL 轉 OTLP，可將 metadata.source 映射為 gen_ai.data_source.id 或 server.address 的補充 discriminator。
  3. TTL 實驗：先在單一高流量來源（如 todoist、knowledge）做 TTL A/B，再擴展至多來源；每次調整需保留 7 天對照窗口與 miss 分類（3Cs／工具既有分類）。
