AI 輔助變更影響分析：monorepo 架構決策與技術債排序實務（2026）

承接既有筆記：

  • AI 架構治理閉環實作：ADR 自動化、Fitness Function 與技術債優先級排序
  • AI-Driven Technical Debt Management 完整指南

技術概述

AI 輔助變更影響分析（Change Impact Analysis, CIA）正在從「找出哪幾個模組會受影響」升級為「把架構追蹤、ADR 決策、CI 節點與技術債優先級串成閉環」。2025-2026 年的最新趨勢是：先用依賴圖與 affected graph 建立可驗證的事實基，再讓 LLM 補上語義層推理，例如這次變更會打到哪些邊界、哪個 ADR 需要更新、哪些高摩擦模組值得先重構。系統文獻回顧指出，LLM 在軟體架構任務上已展現潛力，但 conformance checking 仍是明顯缺口；因此最佳實務不是讓 LLM 單獨判斷，而是把它放在 graph、traceability 與品質閘門之上。

核心方法與工具

  1. 依賴圖事實基
  2. nx affected 依 PR 只執行受影響專案，適合 monorepo 的第一層 impact 邊界。
  3. project graph / import graph / test ownership map 作為受影響面估算依據。
  1. 架構追蹤與 ADR 自動化
  2. LLM 可用於架構元件名稱抽取與 traceability 補齊，減少人工維護成本。
  3. DRAFT 方法把 few-shot、RAG 與 fine-tuning 合併，用於 ADR 草稿生成與決策知識重用。
  1. 技術債優先級排序
  2. 以 hotspot、變更頻率、依賴扇出、失敗率、模組複雜度建立 impact score。
  3. CodeScene 類工具把技術債量化到 PR review 與 IDE refactoring 建議，讓債務處理靠近日常開發。
  1. LLM 判讀層
  2. 摘要這次變更影響的服務、測試與 ADR。
  3. 為高風險模組產生「先驗證後重構」建議，而不是直接大改。

實際工作流程

  1. 收集變更集：讀取 git diff --name-only origin/main...HEAD。
  2. 生成受影響圖：以 Nx/Bazel/自製 graph 計算直接與間接受影響模組。
  3. 合併品質訊號：串接歷史 build failure、測試失敗率、ownership churn、Code Health 指標。
  4. 建立 impact score：例如 0.35dependency_fanout + 0.25change_frequency + 0.2test_failure_rate + 0.2code_health_risk。
  5. LLM 摘要與決策：輸入受影響節點、相關 ADR、近期 incident/CI 資料，輸出影響摘要、需要補的 ADR、建議驗證路徑。
  6. CI 執行最小必要任務：僅跑 affected build/test/lint，對高 impact 模組再加完整 regression。
  7. 回寫知識：把這次影響分析結果記到 ADR、變更說明或 knowledge base，形成可回放的決策歷史。

程式碼 / 配置範例


# .github/workflows/impact-analysis.yml
name: impact-analysis
on: [pull_request]
jobs:
  affected:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npx nx affected -t lint,test,build --base=origin/main --head=HEAD
      - run: python tools/impact_score.py > impact-report.json


# tools/impact_score.py
from math import fsum

weights = {
    'fanout': 0.35,
    'change_freq': 0.25,
    'test_fail_rate': 0.20,
    'code_health_risk': 0.20,
}

metrics = {
    'fanout': 0.82,
    'change_freq': 0.64,
    'test_fail_rate': 0.30,
    'code_health_risk': 0.71,
}

score = fsum(weights[k] * metrics[k] for k in weights)
print({'impact_score': round(score, 3), 'priority': 'high' if score >= 0.65 else 'medium'})

- related_adr: ADR-012, ADR-036
- hotspots: prompt-router churn high, kb-importer code_health 7/10

1. 這次 PR 的主要架構風險
2. 建議補充或更新的 ADR
3. 必跑驗證清單
4. 若要償還技術債，哪一個模組 ROI 最高

效果量化

  • Nx 官方文件指出，nx affected 能只對 PR 受影響專案執行任務，直接消除不相關專案的 CI 浪費；remote caching 進一步加速後續 CI 重跑。
  • CodeScene 官方案例頁宣稱可帶來 30% 更佳的程式碼品質與 50% 更少的非計畫性工作，並把 automated PR reviews、Code Health 與 AI-powered refactoring 串進開發流程。
  • DRAFT 論文在 4,911 筆 ADR 資料集上評估，顯示 ADR 生成若結合 few-shot、RAG 與 fine-tuning，效果優於單純 prompt。
  • 2025 年的軟體架構 LLM 系統回顧分析 18 篇研究，指出 checking conformance 仍不足，意味真正的生產落地仍需要 graph 與規則驗證護欄。

與本專案的應用場景

daily-digest-prompt 可以把這套方法用在自動任務與配置變更上：

  • 針對 config/*.yaml、templates/、skills/ 變更建立 affected graph，縮小驗證範圍。
  • 變更 memory-policy、dedup-policy、results schema 時，自動列出受影響 workflow 與 prompt。
  • 把 context/research-registry.json、workflows/index.yaml、results/ 驗證規則視為 ADR/契約層，讓 LLM 在 PR 階段提醒是否需要同步更新。
  • 針對高 churn 檔案（如 prompt 模板、registry、hook 規則）建立 impact score，優先做重構或測試補強。

風險與注意事項

  • 沒有 graph 或 ownership 事實基時，LLM 容易把語義相近但實際無依賴的模組誤判為受影響。
  • impact score 必須可解釋，否則工程團隊無法信任排序結果。
  • monorepo 的 graph 若維護不完整，affected 分析會漏跑驗證。
  • LLM 只能輔助 ADR 草稿與風險摘要，不能取代最終架構簽核。
  • 若將 CI/incident 資料送進模型，需先做敏感資訊過濾。
