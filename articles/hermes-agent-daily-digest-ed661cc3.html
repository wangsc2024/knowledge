Hermes Agent 架構研究與 daily-digest-prompt 優化借鑒（2026-04）

研究動機：daily-digest-prompt 系統已實施 Harness Engineering 與 OODA 閉環架構，本研究聚焦 Hermes Agent 的自我改進機制、記憶管理與多平台部署策略，提取可供系統優化借鑒的設計模式。

執行摘要

Hermes Agent 是 NousResearch 於 2026 年 2 月釋出的開源 AI Agent 框架，核心創新為「自我改進學習迴圈」與「三層記憶架構」。與 Claude Code / Cursor 等編輯器整合型 Agent 不同，Hermes Agent 以基礎設施常駐型定位，透過技能累積、會話持久化與跨平台部署，實現「與用戶共同成長」的 Agent 範式。

關鍵數據：

  • 15+ 平台整合（CLI、Telegram、Discord、Slack 等）透過統一 Gateway [Hermes Agent Documentation, 2026]
  • 6 種執行後端（local、Docker、SSH、Daytona、Singularity、Modal）支援從 $5 VPS 到無伺服器部署 [GitHub README, 2026]
  • 47 個內建工具分佈於 20 個工具集 [Architecture Documentation, 2026]
  • 三層記憶系統（MEMORY.md + SESSION DB + 外部記憶提供者）實現跨會話知識保留 [Memory Documentation, 2026]

核心架構特點

1. 三層式架構（Three-Tier Design）

Hermes Agent 採用分層設計，從入口層到引擎層到執行層：

層級 1：多入口層

  • CLI：終端機 UI（~8,500 行 Python）
  • Gateway API：訊息平台統一路由（14 個平台適配器）
  • ACP 適配器：編輯器整合（類似 Claude Code 的 MCP）
  • Cron 排程器：自動化任務執行

層級 2：編排引擎（AIAgent 核心）

  • run_agent.py 約 9,200 行，負責對話循環、重試邏輯與故障轉移
  • 提示系統從 SOUL.md、記憶、技能與上下文檔案組裝系統提示
  • 提供者解析將 (provider, model) 對應到 API 憑證
  • 工具系統支援動態調度與自註冊（import 時自動登記）

層級 3：執行後端

  • Local：本地終端執行
  • Docker / Singularity：容器隔離
  • SSH：遠端機器執行
  • Daytona / Modal：無伺服器持久化（閒置時休眠，喚醒時恢復環境）

借鑒價值：daily-digest-prompt 的 Phase 0–3 管線與 Hermes 的分層理念相似，但可強化：

  • ✅ 已實施：Phase 分離、Agent Team 並行、多後端路由（LLM Gate）
  • 🔄 可優化：引入類似 Gateway 的統一會話路由（目前 Todoist / Gmail / 系統審查各自獨立）

2. 自我改進學習迴圈（Built-in Learning Loop）

Hermes Agent 的核心差異化特性，包含四個子機制：

（1）技能創建（Skill Creation）

  • 當 Agent 解決複雜問題後，自動將過程寫入可重用的技能文件（Markdown 格式）
  • 技能包含：問題描述、解決步驟、驗證方法、注意事項
  • 相容 agentskills.io 開放標準，可跨系統分享

（2）技能改進（Skill Improvement）

  • 技能使用時若發現更佳方法，Agent 會「自我提示」更新技能內容
  • 內建版本控制與差異比對

（3）知識持久化（Knowledge Persistence Nudge）

  • Agent 會「提示自己」將重要資訊寫入 MEMORY.md
  • 包含環境事實、慣例、已完成工作與修正建議

（4）會話搜尋（Session Search）

  • 所有 CLI 與訊息會話儲存於 SQLite（~/.hermes/state.db）
  • FTS5 全文搜尋 + Gemini Flash 摘要，提供語義檢索

借鑒價值：

  • ✅ 已實施類似機制：

- skills/ 目錄管理 90+ 技能（frontmatter + 版本號）

- context/digest-memory.json 跨次記憶（連續天數、完成率）

- hooks/post_tool_logger.py 結構化日誌（50MB 輪轉）

  • 🔄 可借鑒改進：

- 自動技能提煉：當前技能手動建立，可參考 Hermes 的「任務完成後自動提示撰寫技能」機制

- 技能版本演化：目前僅追蹤版本號，可引入「使用中改進」的回饋迴圈

- 會話語義搜尋：當前日誌為 JSONL 格式，可引入 SQLite FTS5 提升檢索效能

3. 三層記憶系統（Three-Tier Memory Architecture）

Tier 1：內建持久記憶（Bounded Curated Memory）

  • MEMORY.md（2,200 字元上限，~800 tokens）：Agent 個人筆記

- 環境事實（作業系統、工具版本）

- 慣例（命名規則、專案結構）

- 已完成工作與修正建議

  • USER.md（1,375 字元上限，~500 tokens）：使用者偏好

- 溝通風格（簡潔 vs 詳細）

- 工作習慣（時區、工作時段）

- 技術偏好（語言、框架）

記憶管理：Agent 透過 memory 工具自主新增、替換或刪除條目；記憶滿時自動合併或淘汰舊項。

Tier 2：會話搜尋（Session Search）

  • SQLite 儲存所有 CLI 與訊息會話（~/.hermes/state.db）
  • FTS5 全文索引 + 譜系追蹤（parent / child 會話關聯）
  • 搜尋結果經 Gemini Flash 摘要後呈現

Tier 3：外部記憶提供者（External Memory Providers）

  • 內建 8 個外部記憶插件：Honcho、OpenViking、Mem0、Hindsight、Holographic、RetainDB、ByteRover、Supermemory
  • 提供知識圖譜、自動事實擷取等進階能力

安全機制：

  • 子字串比對更新（避免覆寫無關內容）
  • 字元上限強制與自動合併
  • 注入模式掃描（防範 prompt injection）
  • 自動去重

借鑒價值：

  • ✅ 已實施類似機制：

- context/digest-memory.json（schema_version=2，連續天數追蹤）

- context/workflow-state.json（工作流狀態持久化）

- state/ 目錄（執行狀態、度量、健康指標）

  • 🔄 可借鑒改進：

- 有界記憶設計：當前 digest-memory 無字元上限，可參考 Hermes 的自動合併機制

- 使用者偏好檔：建立類似 USER.md 的 context/user-profile.yaml，記錄溝通偏好、技術棧、工作習慣

- 會話譜系追蹤：當前日誌為平面 JSONL，可引入 parent/child 關聯（如 Phase 0→1→2 的依賴鏈）

4. 多平台部署與基礎設施彈性

15+ 平台整合（透過單一 Gateway 程序）：

  • 即時通訊：Telegram、Discord、Slack、WhatsApp、Signal、Matrix、Mattermost
  • 異步通訊：Email、SMS
  • 企業平台：DingTalk、Feishu、WeCom
  • 其他：BlueBubbles、Home Assistant

跨平台對話連續性：同一使用者在不同平台的會話共享記憶與技能庫。

6 種執行後端：

  • Local / SSH：傳統部署（適合完全控制與除錯）
  • Docker / Singularity：容器隔離（適合多租戶與安全要求）
  • Daytona / Modal：無伺服器持久化

- 閒置時環境休眠（幾乎零成本）

- 喚醒時恢復完整狀態（包含檔案系統與套件）

- 官方測試：$5/月 VPS 即可穩定執行

借鑒價值：

  • ✅ 已實施部分機制：

- config/llm-gate.yaml 多後端路由（claude_code、cursor_cli、codex、openrouter_runner、Gemma）

- run-todoist-agent-team.ps1 支援不同 Agent 類型

  • 🔄 可借鑒改進：

- 統一 Gateway 模式：將 Todoist / Gmail / 系統審查統一為單一 Gateway，支援更多觸發來源（如 Slack、Email）

- 無伺服器部署探索：評估 Modal / Railway 等平台，實現閒置零成本（當前為 Windows 排程器 + 本地執行）

5. 與其他框架的對比

vs. Claude Code / Cursor（根據 OpenAIToolsHub 評測）：

  • Claude Code / Cursor 優勢：更精緻、社群更大、程式碼品質更高（搭配 frontier 模型時）
  • Hermes Agent 優勢：成本更低、資料隱私（可純本地執行）、擴充性更強
  • 合規情境：對無法將程式碼傳送至第三方 API 的團隊，Hermes + 本地 Ollama 是少數全私有化選項

vs. OpenClaw：

  • Hermes Agent 被稱為「OpenClaw 的開源替代」
  • 提供相同等級的 Agentic 能力，但無「黑箱服務」問題

模型依賴性：

  • Frontier 模型（GPT-4o、Claude Sonnet）：多步驟任務表現紮實
  • 本地 70B 模型：工具選擇與多步驟規劃明顯較弱

A2. 有界記憶管理（參考 MEMORY.md 的 2,200 字元上限）

改進 context/digest-memory.json：


# tools/update_digest_memory.py
def update_memory(new_entry):
    memory = load_json('context/digest-memory.json')
    memory['entries'].append(new_entry)
    
    # 字元上限檢查（設為 5000 字元）
    total_chars = sum(len(json.dumps(e)) for e in memory['entries'])
    if total_chars > 5000:
        # 自動合併：保留最近 3 天 + 重要度 >=8 的舊項
        memory['entries'] = consolidate_entries(memory['entries'])
    
    save_json('context/digest-memory.json', memory)

B. 中期優化方向

B1. 統一 Gateway 模式

建立 gateway.ps1 統一入口：


# gateway.ps1 - 統一會話路由
param(
    [ValidateSet('todoist', 'gmail', 'slack', 'ntfy', 'system-audit')]
    [string]$Source,
    [string]$Payload
)

# 路由邏輯
switch ($Source) {
    'todoist' { & run-todoist-agent-team.ps1 }
    'gmail'   { & run-gmail-agent.ps1 }
    'slack'   { & gateway-slack-handler.ps1 -Payload $Payload }
    # ...
}

B2. 自動技能提煉

在 results/todoist-auto-*.json 的 done_cert.quality_score >= 4 時觸發：


# Invoke-SkillDistillation.ps1
if ($result.done_cert.quality_score -ge 4 -and $result.type -eq 'todoist_research') {
    $prompt = @"
根據以下任務結果，提煉為可重用的技能文件：

任務：$($result.content)
完成狀態：$($result.status)
品質分數：$($result.done_cert.quality_score)

輸出技能文件應包含：
1. 問題描述
2. 解決步驟
3. 驗證方法
4. 注意事項
"@
    Invoke-LlmGate -Profile skill_distillation -Prompt $prompt -OutputPath "skills/auto-generated/skill-$(Get-Date -Format 'yyyyMMdd-HHmmss').md"
}

B3. SQLite 日誌後端

取代當前的 JSONL 格式，引入 FTS5 搜尋：


# tools/migrate_logs_to_sqlite.py
import sqlite3
import json
from pathlib import Path

db = sqlite3.connect('state/tool-logs.db')
db.execute('''
    CREATE VIRTUAL TABLE IF NOT EXISTS logs USING fts5(
        timestamp, session_id, parent_session, tool, description, status, content
    )
''')

# 遷移現有 JSONL
for log_file in Path('logs/structured').glob('*.jsonl'):
    with open(log_file, 'r', encoding='utf-8') as f:
        for line in f:
            entry = json.loads(line)
            db.execute('INSERT INTO logs VALUES (?, ?, ?, ?, ?, ?, ?)',
                      (entry['timestamp'], entry.get('session_id'), entry.get('parent_session'),
                       entry['tool'], entry['description'], entry['status'], json.dumps(entry)))
db.commit()
