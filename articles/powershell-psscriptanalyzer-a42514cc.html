本次研究主題：PowerShell PSScriptAnalyzer 自訂規則與品質閘門（因知識庫尚無此主題的深入研究）

本文為 powershell-quality-gates 系列第 foundation 階段，承接既有專案優化與測試治理脈絡，建立 PowerShell 靜態分析、自訂規則與測試品質閘門的基礎實務。

技術概述

PSScriptAnalyzer 是 PowerShell 官方生態的靜態分析器，用來在執行前檢查 .ps1、.psm1、.psd1 的安全性、可維護性與相容性。到 2026-03-14 為止，PowerShell Gallery 的最新版本是 1.24.0，更新日期為 2025-03-18；Microsoft Learn 的新版說明顯示它已把最低需求提升到 Windows PowerShell 5.1，並加強 UseCorrectCasing、設定型別判斷、parse error 回報與 -EnableExit 行為。對本專案而言，PSScriptAnalyzer 最有價值的地方不是「一次性掃 code」，而是把安全與品質規則變成固定閘門，讓大型 PowerShell 編排腳本在提交前就先暴露問題。

核心概念與原理

1. 分三層看 PSScriptAnalyzer

  • 規則層：內建規則涵蓋命名、可讀性、安全性、相容性與格式。Invoke-ScriptAnalyzer 預設會跑所有內建規則，也可透過 IncludeRule、ExcludeRule、Severity 篩選。
  • 設定層：可用 PSScriptAnalyzerSettings.psd1 宣告規則組合。官方文件指出可採 built-in preset、explicit settings 與 implicit discovery 三種模式；如果把同名檔案放在專案根目錄，掃描專案根目錄時會自動發現。
  • 治理層：-EnableExit 會在分析完成後以 error 診斷數量作為 shell exit code，這對排程前 preflight、git hook 或 CI 都很重要，因為它能把靜態分析直接升級為阻擋條件。

2. 自訂規則的實作模型

Microsoft Learn 對 custom rule 的要求很明確：

  • 函式要有 comment-based help，至少包含 .DESCRIPTION
  • 輸出型別應為 DiagnosticRecord
  • 參數必須含 Ast 或 Token，而且名稱結尾也要是 Ast 或 Token
  • 自訂規則函式必須 Export-ModuleMember

這代表自訂規則不是「字串搜尋小工具」，而是直接建立在 PowerShell AST / token 模型上。適合本專案的規則例子包括：

  • 禁止在核心編排腳本中直接寫死外部 URL
  • 要求所有結果 JSON 都必須含 agent、type、status、summary
  • 禁止使用 Write-Host 作為正式流程狀態輸出
  • 禁止產生 nul 檔名或使用 > nul

3. 抑制與例外治理

官方文件允許用 SuppressMessageAttribute 局部抑制規則，也能透過設定檔排除規則。這很適合處理歷史包袱，但不能把 suppression 當成永久逃生門。較穩健的做法是：

  • 只在有明確理由時抑制
  • 盡量把 justification 寫清楚
  • 定期用 -SuppressedOnly 稽核哪些抑制仍存在

4. 與 Pester 的關係

PSScriptAnalyzer 解決的是「靜態問題」，Pester 解決的是「行為問題」。Pester v5 文件已改為以 New-PesterConfiguration 啟用 coverage，而不是舊的 -CodeCoverage 參數。這表示比較好的閘門順序是：

  1. 先跑 PSScriptAnalyzer，擋掉語法、風格、安全與相容性
  2. 再跑 Pester，確認行為與覆蓋率
  3. 最後才進行真正的排程或同步工作

最佳實踐與常見陷阱

最佳實踐

  • 先從 settings file 開始，不要一開始就自訂一堆規則：先把 Error/Warning、排除規則、專案預設值穩定下來，再新增 custom rule。
  • 把內建規則分群：例如 安全、格式、相容性 三層，避免每次掃描都把噪音混在一起。
  • 善用 -EnableExit 與 -ReportSummary：這能讓腳本在自動化中變成標準閘門，而不是人工閱讀報表。
  • 自訂規則只做專案契約，不做全部風格規範：專案契約像是結果 JSON 結構、禁止 nul、禁止未經快取的外部 curl，這些最值得寫成 custom rule。
  • Pester coverage 設實際門檻：先從 50-60% 起步，比宣稱 90% 但長期無法達成更有效。

常見陷阱

  • 把 -Fix 當萬能修復：官方文件明講有些修正可能改動編碼或需要其他人工調整，所以不能在未備份情況下直接批量執行。
  • 誤用 suppression 讓規則形同虛設：短期可接受，長期會讓品質閘門失去可信度。
  • 忽略 parser errors：新版文件指出 parser errors 會作為 diagnostic records 輸出。若只看 Warning 而忽略 ParseError，會把真正會中斷執行的問題漏掉。
  • 只有 Pester 沒有靜態分析：這會讓簡單的風格、安全與相容性問題一路跑到測試階段才曝光。

與本專案的關聯

我在本專案用全文搜尋檢查後，沒有找到現成的 PSScriptAnalyzer、Invoke-ScriptAnalyzer、Pester 設定或引用，代表目前 daily-digest-prompt 的 PowerShell 腳本量雖然不少，但尚未把靜態分析與單元測試正式納入固定閘門。這與今日完成的「專案優化（安全+品質4項+959測試）」高度相關，因為：

  • 專案內有多支 .ps1 編排腳本，容易累積風格與安全債
  • 任務流程對結果 JSON 契約敏感，適合加上 custom rule
  • 排程、自動化、知識庫同步都很需要先做 preflight 檢查

落地建議：

  1. 在 repo 根目錄加入 PSScriptAnalyzerSettings.psd1
  2. 新增 tools/Invoke-ProjectQualityGate.ps1
  3. 讓它依序執行 Invoke-ScriptAnalyzer、Invoke-Pester
  4. 把知識庫同步腳本與主要排程入口納入掃描範圍

程式碼範例

1. 基本設定檔


@{
    Severity = @('Error', 'Warning')
    IncludeRules = @(
        'PSAvoidUsingWriteHost',
        'PSUseApprovedVerbs',
        'PSAvoidUsingPlainTextForPassword',
        'PSUseCompatibleSyntax'
    )
    ExcludeRules = @(
        'PSAvoidUsingCmdletAliases'
    )
}

2. 專案品質閘門腳本


$analyzer = Invoke-ScriptAnalyzer `
    -Path . `
    -Recurse `
    -Settings .\PSScriptAnalyzerSettings.psd1 `
    -ReportSummary

if ($analyzer.Where({ $_.Severity -eq 'Error' }).Count -gt 0) {
    throw 'PSScriptAnalyzer 發現 Error 等級問題'
}

$config = New-PesterConfiguration
$config.Run.Path = '.\tests'
$config.CodeCoverage.Enabled = $true
$config.CodeCoverage.Path = '.\tools', '.\scripts', '.\*.ps1'
$config.CodeCoverage.CoveragePercentTarget = 60

$pester = Invoke-Pester -Configuration $config
if ($pester.FailedCount -gt 0) {
    throw 'Pester 測試失敗'
}

3. 自訂規則骨架


function Measure-ResultJsonContractAst {
    [OutputType([Microsoft.Windows.PowerShell.ScriptAnalyzer.Generic.DiagnosticRecord[]])]
    param(
        [Parameter(Mandatory)]
        [System.Management.Automation.Language.ScriptBlockAst]
        $ScriptBlockAst
    )

    if ($ScriptBlockAst.Extent.Text -match 'todoist-auto-tech_research\\.json' -and
        $ScriptBlockAst.Extent.Text -notmatch '"summary"') {
        [Microsoft.Windows.PowerShell.ScriptAnalyzer.Generic.DiagnosticRecord]@{
            Message  = '結果 JSON 必須包含 summary 欄位'
            Extent   = $ScriptBlockAst.Extent
            RuleName = $PSCmdlet.MyInvocation.InvocationName
            Severity = 'Warning'
        }
    }
}

Export-ModuleMember -Function Measure-ResultJsonContractAst
