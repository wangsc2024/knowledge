一、核心定義與範式轉移

1.1 Harness Engineering 是什麼

Harness Engineering 是設計、構建並迭代 AI Agent 周邊環境、工具、約束與反饋迴圈的學科。核心公式：

Agent = Model + Harness

  • Model：提供智慧（LLM 推理能力）
  • Harness：使智慧可用（約束、驗證、反饋）

來源：Martin Fowler、OpenAI、Epsilla Blog

1.2 為何需要 Harness Engineering

2026 年統計顯示：

  • 81% AI Agent 已進入生產運行
  • 88% 組織曾發生 AI Agent 安全事件
  • 改變 Harness（不改模型）可在同一 benchmark 產生 6 倍效能差距

來源：Meta-Harness 研究、SSRN 學術論文

二、四大支柱（The Four Pillars）

2.1 Constrain（約束）

定義：通過架構邊界與依賴規則約束 Agent 能做什麼

實作方式：

  • 檔案寫入權限邊界（資料所有權規範）
  • 工具白名單（allowed-tools in SKILL.md）
  • API 端點限制（只能呼叫特定服務）

關鍵原則：約束解空間 → 加速收斂

  • Agent 能生成任意內容時，浪費 tokens 探索死路
  • Harness 定義明確邊界時，Agent 更快收斂到正確解

來源：Epsilla、Augment Code

2.2 Inform（告知）

定義：透過 Context Engineering 與文檔告知 Agent 應做什麼

實作方式：

  • 結構化 prompt 模板（templates/auto-tasks/）
  • SKILL.md 操作指南
  • Repository Impact Map（符號依賴分析）

關鍵原則：Make AI look at real code before planning

  • 基於符號分析的依賴關係圖
  • 將下游任務錨定在實際代碼結構上

來源：NxCode Guide

2.3 Verify（驗證）

定義：通過測試與 Lint 驗證 Agent 是否正確執行

分層檢查策略（來源：Red Hat Developer）：


PostToolUse Hook（毫秒）> pre-commit（秒）> CI（分鐘）> 人工審查（小時）

推檢查至最快層：

  • Hook 層：即時攔截（如 pre_bash_guard 禁止讀取 .env）
  • pre-commit：快速驗證（如 validate_config.py）
  • CI：完整測試套件（1801 個測試）
  • 人工審查：最後防線

架構即代碼 — Fitness Functions：

  • 將架構規則編碼為 CI 測試
  • 範例：「服務層不可引入執行層」→ 寫測試強制
  • Agent 無需理解架構推理，只需通過測試即可合併 PR

來源：Martin Fowler

2.4 Correct（修正）

定義：透過反饋迴圈與自我修復糾正錯誤

專業化 Agent 分工（來源：Augment Code）：

  • Planner — 擴展簡短 prompt 為完整產品規格
  • Generator — 基於規格生成代碼
  • Evaluator/QA — 以用戶視角測試應用，評分並反饋

關鍵發現：Agent 對自我評價過於樂觀，必須獨立評估

  • 生成 Agent 自評常給 90+ 分
  • 獨立 QA Agent 評估更客觀
  • Evaluator 給予具體反饋（非籠統「很好」）

三、三大維度（The Three Dimensions）

來源：SSRN 學術論文

3.1 Context（上下文）

宣告性知識：

  • 是什麼（What）— 系統架構、數據結構
  • 範例：CLAUDE.md、SKILL.md、API 文檔

程序性知識：

  • 如何做（How）— 操作步驟、工作流
  • 範例：templates/auto-tasks/.md、workflows/.yaml

3.2 Constraint（約束）

產出前引導（Feedforward Controls）：

  • Prompt 規範（如 preamble.md）
  • Schema 定義（JSON schema validation）
  • 範例模板（few-shot examples）

產出後強制（Feedback Controls）：

  • Hook 攔截（pre_write_guard、pre_bash_guard）
  • CI 測試失敗 → 阻止合併
  • 靜態分析（lint、type check）

3.3 Convergence（收斂）

迭代評估約束：

  • 監控約束違規率（每週 Hook 攔截次數 / 總操作數）
  • 分析常見失敗模式（trace_analyzer.py）

識別缺口：

  • 哪些架構規則尚未自動化驗證？
  • 哪些反饋迴圈缺失？

精煉規則直至結構冪等：

  • 目標：違規率 < 5%
  • 達成「結構冪等」— Harness 不再需要頻繁修正

四、核心原則與最佳實踐

4.1 核心原則

「用機制強制品質，而非 prompts」（Enforce quality with mechanisms, not prompts）

順序原則（Sequential Principles）：


1. achieve detectability → grant autonomy
2. establish constraints → optimize throughput

先建立可偵測性（Hook、測試、日誌），再授予自主性（允許 Agent 自動執行）。

先建立約束（邊界、規則），再優化吞吐量（並行、加速）。

來源：Epsilla、Red Hat Developer

4.2 架構模式

Repository Impact Map：

  • 基於符號分析（AST、import graph）的依賴關係圖
  • Agent 在規劃前先查看真實代碼依賴
  • 避免盲目修改（如：不知道修改 X 會影響 Y、Z）

Fitness Functions 作為架構護欄：

  • 架構規則 → CI 測試
  • Agent 只需通過測試，無需理解架構哲學
  • 範例：

- test_data_ownership.py — 驗證檔案寫入權限邊界

- test_hook_precision.py — Hook 精準率 <5% false positive

- test_constraint_coverage.py — 約束嵌入率 ≥70%

分層反饋速度優化：

| 層級 | 反饋時間 | 適用檢查 |

|------|---------|----------|

| PostToolUse Hook | 毫秒 | 敏感操作攔截（讀 .env、寫系統檔案） |

| pre-commit | 秒 | 格式驗證、schema 檢查 |

| CI | 分鐘 | 完整測試套件、Fitness Functions |

| 人工審查 | 小時 | 架構決策、安全審查 |

五、對本專案的優化建議

本專案已導入 Harness Engineering（B0-B3 四批次，1801 測試）。基於研究發現，建議以下優化方向：

建議 A：強化 Fitness Functions 覆蓋率

現狀：B0 批次已建立 9 個 Fitness Functions（config/benchmark.yaml）

優化方向：

  • 將隱性架構規則編碼為 CI 測試
  • 範例：資料所有權規範 → tests/fitness/test_data_ownership.py
  • 範例：「config/*.yaml 必須有 version 欄位」→ tests/fitness/test_config_versioning.py

預期效果：

  • 減少人工審查負擔（機器可驗證的不需人工）
  • 違規即時攔截（CI 失敗 → 阻止合併）

建議 B：引入 Repository Impact Map

現狀：Agent 修改檔案前未查看依賴關係

優化方向：

  • 為 skills/、hooks/、config/ 建立符號依賴圖
  • 工具選擇：Python: pydeps、import-linter；通用: Graphviz
  • Agent 執行任務前先讀取相關模組的真實依賴關係

預期效果：

  • 減少盲目修改（如：不知道修改 llm-router.yaml 會影響 3 個 prompt）
  • 提升修改準確性（Agent 看到完整依賴鏈）

建議 C：專業化 Agent 角色強化

現狀：

  • ✅ 已有 Planner（assemble-digest.md）
  • ✅ 已有 Executor（todoist-auto-*.md）
  • ❌ 缺少獨立 Evaluator 角色

優化方向：

  • 新增獨立品質稽核任務（每日 1 次）
  • 功能：抽樣已完成任務、評分、將反饋寫入 context
  • 下次執行該任務時讀取反饋並改進

預期效果：

  • 客觀品質評估（避免 Agent 自評過於樂觀）
  • 持續品質改進迴圈

建議 D：加速反饋迴圈

現狀：部分檢查在 CI 層執行（分鐘級）

優化方向：

  • 將高頻檢查前移至 Hook 層（毫秒級）
  • 範例：YAML schema 驗證 → 從 validate_config.py（pre-commit）拆分到 pre_write_guard.py（Hook）
  • 範例：JSON 格式檢查 → 從 CI 前移到 Hook
  • 保留複雜檢查在 CI（如完整測試套件）

預期效果：

  • Agent 即時感知錯誤（寫檔時立即攔截 vs. 提交後才發現）
  • 減少無效 commit（格式錯誤在 Hook 層就被阻止）

建議 E：Convergence 量化追蹤

現狀：缺少約束系統健康度量化指標

優化方向：

  • 新增指標（寫入 state/convergence-metrics.json）：

- 約束違規率 = 每週 Hook 攔截次數 / 總操作數

- 自我修復成功率 = self-heal 成功次數 / 總觸發次數

- Fitness Functions 通過率 = CI 通過次數 / 總執行次數

  • 目標閾值：

- 違規率 < 5%（達成「結構冪等」）

- 自我修復成功率 > 80%

- Fitness Functions 通過率 > 95%

  • 每週生成趨勢報告（convergence-trend.md）

預期效果：

  • 可視化 Harness 健康度
  • 及時發現約束漂移（違規率突增）
  • 數據驅動優化決策

八、總結與行動清單

8.1 核心要點

  1. Harness 比模型更重要：改變 Harness 可產生 6 倍效能差距（不需換模型）
  2. 機制強制品質：用測試、Hook、CI 強制，而非依賴 prompt 自律
  3. 分層反饋速度：Hook（毫秒）> pre-commit（秒）> CI（分鐘）> 人工（小時）
  4. 專業化分工：Planner + Generator + 獨立 Evaluator（避免自評過於樂觀）
  5. 收斂量化追蹤：違規率 < 5% = 結構冪等
