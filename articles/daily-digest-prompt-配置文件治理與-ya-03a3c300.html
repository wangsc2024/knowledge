Daily-Digest-Prompt 配置文件治理與 YAML 工程化

研究日期：2026-02-24 研究動機：專案 config/ 目錄已累積 16 個 YAML + 2 個 Markdown 配置文件，跨 7 個 Python hooks、6 個 PowerShell 腳本、30+ 個 prompt 模板交叉引用。隨著系統演進，配置複雜度成為維護瓶頸。本文從 YAML 工程化角度提出系統性治理方案。

一、現況分析：配置文件全景地圖

1.1 配置文件清單（16 YAML + 2 MD = 18 個）

合計約 1,500+ 行 YAML 配置，由 13+ 個不同消費者讀取。

1.2 版本分布不一致

  • v3：frequency-limits.yaml（最活躍，反映最多次修改）
  • v2：routing.yaml、hook-rules.yaml、scoring.yaml（中等修改）
  • v1：其餘 10 個（從未升版或較新建立）
  • 無版本：3 個（retro-games、creative-game-mode、schemas/migration-rules）

問題：版本號沒有統一的遞增策略。routing.yaml v2 代表什麼程度的變更？是破壞性改動（SemVer MAJOR）還是新增欄位？消費者無法判斷。

1.3 Schema 驗證覆蓋率

validate_config.py 目前驗證 7 個 YAML 的基本結構：

  • pipeline.yaml、routing.yaml、cache-policy.yaml、frequency-limits.yaml
  • scoring.yaml、notification.yaml、hook-rules.yaml

未覆蓋的 9 個配置沒有 schema 驗證：timeouts.yaml、dedup-policy.yaml、topic-rotation.yaml、health-scoring.yaml、audit-scoring.yaml、benchmark.yaml、retro-games.yaml、creative-game-mode.yaml、migration-rules.yaml。

覆蓋率 = 7/16 = 43.75%，不到一半。

二、問題診斷：六大配置治理缺口

2.1 YAML Sprawl（配置蔓延）

根據業界定義，YAML sprawl 是「配置文件不受控制地增長」的現象。本專案 18 個配置文件分散在 config/ 目錄，缺乏：

  • 配置分類學（taxonomy）：哪些是「行為配置」、哪些是「數據配置」、哪些是「安全配置」？
  • 依賴關係圖：routing.yaml 引用 scoring.yaml 的概念（tiebreaker），但沒有明確的依賴宣告
  • 變更影響分析：修改 timeouts.yaml 的 phase2_timeout_by_type 會影響哪些腳本？需要人工追蹤

2.2 配置漂移（Configuration Drift）

配置漂移指「實際運行的配置與版本控制中的配置不一致」：

  • PowerShell 腳本可能硬編碼：若 run-todoist-agent-team.ps1 中有 timeout 常數，與 timeouts.yaml 的值可能不同步
  • prompt 模板引用過期配置鍵：若 frequency-limits.yaml 改欄位名，引用它的 20 個 prompt 模板不會自動更新
  • 無自動漂移偵測：缺少 CI 級別的配置一致性檢查

2.3 跨文件引用不透明

配置之間存在隱式依賴：

  • frequency-limits.yaml 的 tasks.*.template 引用 templates/auto-tasks/ 中的檔案路徑
  • routing.yaml 的 mappings.*.template 引用 templates/sub-agent/ 中的檔案路徑
  • pipeline.yaml 的 steps.*.skill_files 引用 skills/ 目錄
  • 但這些路徑引用沒有驗證：若模板被移動或重命名，配置文件不會報錯，只在運行時失敗

2.4 Schema 驗證不足

僅 43.75% 的覆蓋率意味著超過一半的配置修改是「盲飛」。特別是：

  • timeouts.yaml（控制所有 Agent timeout）無 schema，打錯欄位名會導致 Agent 使用預設值而非配置值
  • audit-scoring.yaml（150 行複雜結構）無 schema，權重配置加總不等於 100 也不會被偵測
  • dedup-policy.yaml（影響所有研究模板）無 schema

2.5 版本遷移策略缺失

雖然有 validate_config.py --migrate 功能和 migration-rules.yaml，但：

  • 遷移規則僅定義了單個文件的欄位重命名/移除
  • 缺少跨文件遷移：若 routing.yaml v2→v3 修改了 labelMapping 格式，所有引用該格式的 prompt 模板也需要同步更新
  • 缺少遷移測試：沒有自動化測試驗證遷移後的配置是否仍然正確

2.6 文件驅動的副作用

專案採用「文件驅動架構」是正確方向，但帶來一個副作用：LLM Agent 是配置的主要消費者，而非程式碼。這意味著：

  • 配置格式的「正確性」不僅取決於 schema，還取決於 LLM 能否正確解讀
  • 過度複雜的嵌套結構（如 routing.yaml 的三層路由）增加 LLM 解讀錯誤的機率
  • Agent 的 context window 限制下，大量配置文件會擠壓有用內容的空間

三、優化方案：五層配置治理框架

3.1 第一層：配置分類與命名規範

方案：建立配置分類學（Config Taxonomy），將 18 個文件分為 4 類：

類別前綴文件修改頻率
行為配置（Behavior）—pipeline, routing, frequency-limits, scoring低（架構定型後穩定）
參數配置（Parameters）—cache-policy, timeouts, dedup-policy, topic-rotation中（按需調參）
安全配置（Security）—hook-rules極低（審慎修改）
評分配置（Scoring）—health-scoring, audit-scoring, benchmark低
領域配置（Domain）—notification, retro-games, creative-game-mode中

預期效果：新增 config/README.md 中的分類表，讓人類和 Agent 都能快速定位配置用途。

3.2 第二層：全面 Schema 驗證

方案：將 validate_config.py 的覆蓋率從 43.75% 提升到 100%。

優先順序（依風險排序）：

  1. P0：timeouts.yaml — 直接影響 Agent 執行（數值型驗證 + 正數檢查）
  2. P0：audit-scoring.yaml — 權重加總驗證（=100）
  3. P1：dedup-policy.yaml — retention_days/cooldown_days 合理範圍
  4. P1：benchmark.yaml — target 格式驗證
  5. P2：其餘 4 個

實作方式：在 config/schemas/ 目錄建立 JSON Schema 檔案，validate_config.py 已支援此模式。

3.3 第三層：跨文件引用驗證

方案：新增 validate_config.py --check-refs 模式，驗證所有路徑引用是否指向存在的檔案。

檢查範圍：

# frequency-limits.yaml 中的模板路徑
tasks.*.template → 驗證檔案存在

# routing.yaml 中的模板路徑
label_routing.mappings.*.template → 驗證檔案存在

# pipeline.yaml 中的 Skill 路徑
steps.*.skill_files → 驗證檔案存在
finalize.*.config_ref → 驗證檔案存在

預期效果：模板重命名時，check-health.ps1 會立即報告引用失效。

3.4 第四層：版本策略統一

方案：所有配置文件統一採用語義化版本（Simplified SemVer）：

場景版本變更範例
新增可選欄位MINOR +1v1 → v2
修改現有欄位語義MAJOR +1v2 → v3
修正註釋/格式不變版本號—
移除欄位MAJOR +1—

配合：validate_config.py 在驗證時，若偵測到 version 欄位缺失，自動警告。

3.5 第五層：配置依賴圖可視化

方案：建立 config/dependency-graph.md，用 Mermaid 圖呈現配置間依賴關係：

graph TD
    routing --> scoring[scoring.yaml]
    routing --> templates[templates/sub-agent/]
    frequency --> templates_auto[templates/auto-tasks/]
    pipeline --> skills[skills/*/SKILL.md]
    pipeline --> cache[cache-policy.yaml]
    hook_rules --> hooks[hooks/*.py]
    notification --> ntfy[ntfy-notify SKILL]
    timeouts --> scripts[run-*.ps1]

預期效果：任何配置修改前，先查依賴圖確認影響範圍。

五、與業界最佳實踐的對照

5.1 YAML Sprawl 防護

根據 GraphApp.ai 的定義，YAML sprawl 在微服務和大型專案中極為常見。本專案 16 個 YAML 在「中等規模」範圍，但考慮到消費者是 LLM Agent（context window 有限），應更積極控制。

建議：考慮合併低修改頻率的配置（如 health-scoring + benchmark 合併為 metrics-config.yaml），減少文件數量。

5.2 Configuration Drift Detection

根據 Snyk 和 Puppet 的研究，配置漂移是生產環境事故的主要原因之一。IaC 工具（Terraform、Ansible）通過「聲明式配置 + 持續比對」解決此問題。

對應：本專案的 validate_config.py + check-health.ps1 已具備基礎漂移偵測能力，但建議：

  • 每次 git commit 前自動執行 validate_config.py（加入 pre-commit hook）
  • check-health.ps1 新增「配置一致性」維度

5.3 Schema-First Development

根據 23andMe 的 Yamale 和 JSON Schema 社群的實踐，「先定義 schema，再寫配置」是最佳做法。

對應：目前專案是「先寫配置，後補 schema」，建議新增配置文件時強制同步提供 schema。

六、具體行動計畫

Phase 1（立即可做，~2h）

  1. 補齊 timeouts.yaml 的 JSON Schema
  2. 補齊 audit-scoring.yaml 的 JSON Schema（含權重加總驗證）
  3. 為缺少 version 的 3 個 YAML 補上版本號
  4. 更新 config/README.md 加入分類表

Phase 2（本週完成，~4h）

  1. 實作 validate_config.py --check-refs 模式
  2. 補齊剩餘 7 個 YAML 的 Schema
  3. 建立 config/dependency-graph.md

Phase 3（長期優化）

  1. 考慮合併低頻配置文件（metrics-config.yaml）
  2. 評估 Yamale 等輕量 schema 驗證工具
  3. Git pre-commit hook 整合配置驗證
