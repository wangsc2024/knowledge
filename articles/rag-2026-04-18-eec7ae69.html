三、技術實作細節

核心設計模式

Observer-Reflector 模式（來自 Mastra 框架）

\\\

被動記錄層（Observer）：

├─ state/scheduler-state.json

├─ context/system-insight.json

└─ cache/status.json

主動反思層（Reflector）：

├─ tools/generate_insights.py

└─ context/digest-insights.json ← 新增

自動執行層（Action）：

└─ （待實作：自動應用建議）

\\\

關鍵技術決策

#### 決策 1：分離 Observations 與 Insights

動機：

  • Observations = 原始事實（\scheduler-state.json\、\system-insight.json\）
  • Insights = 主動反思（\digest-insights.json\）
  • 分離可避免資料污染，保持單一職責

實施：

  • 讀取多個 state 檔案
  • 生成獨立的 \digest-insights.json\
  • 保持原始資料不變

#### 決策 2：閾值驅動的洞察生成

動機：

  • 避免無意義的洞察洪流
  • 只在指標異常時生成

閾值設定：

\\\python

failure_rate > 0.25 → critical

failure_rate > 0.10 → warning

cache_hit_ratio < 0.25 → critical

cache_hit_ratio < 0.40 → warning

entropy_score > 0.60 → critical

entropy_score > 0.53 and trend=degrading → warning

success_rate < 0.75 → critical

success_rate < 0.90 → warning

\\\

#### 決策 3：可執行建議（Actionable Recommendations）

動機：

  • 洞察必須伴隨具體行動
  • 減少「發現問題但不知如何修復」

建議格式：

\\\json

{

"action": "adjust_timeout",

"target": "config/timeouts.yaml",

"params": { "建議": "延長 timeout +50%" },

"steps": ["步驟1", "步驟2"],

"expected_improvement": "失敗率 -30%"

}

\\\

六、關鍵學習與洞見

學習 1：Observational Memory 的威力

來自知識庫的啟發：

Mastra 框架的 Observer-Reflector-Memory 三層架構，將被動記錄升級為主動反思，實現 Agent 自我演化。

實踐驗證：

  • ✅ 自動發現成功率低於目標（Critical）
  • ✅ 自動發現熵增趨勢（Warning）
  • ✅ 提供可執行建議（4 個具體 targets）

深度洞見：

  • 從「事後諸葛」到「預測式維護」：傳統系統需人工查日誌才發現問題，Observational Memory 主動分析並預測風險
  • 閾值即智慧：關鍵不在記錄資料，而在設定正確的閾值與反思邏輯
  • 可執行性是核心：洞察若無對應行動，等同無用資訊

學習 2：知識庫驅動開發的價值

流程：

  1. 查詢 RAG 知識庫 → 獲得 5 大優化洞見
  2. 深度理解 Mastra/Claude Code/DeepEval 框架
  3. 映射到 daily-digest-prompt 專案
  4. 實作最高優先級優化（Observational Memory）

成效：

  • 0.5 小時完成 432 行核心程式碼
  • 8 個測試全數通過
  • 立即生成 2 個有效洞察

關鍵要素：

  • 知識庫品質：筆記需涵蓋架構模式、實作細節、同構性識別
  • 執行速度：從洞見到實作僅需 30 分鐘
  • 品質保證：TDD 驅動，測試先行

學習 3：系統熵增的根因

觀察：

  • Hook 規則分散在 13 個 Python 腳本
  • Skills 手動維護（97 個）
  • 配置膨脹（13 個 YAML）

根因分析：

  1. 缺乏統一抽象層：每個 Hook 獨立實作，無共用框架
  2. 手動維護成本高：Skill 觸發詞需手動更新 SKILL_INDEX.md
  3. 配置分散：相關配置散布多檔，難以統一管理

解決方向（Phase 2）：

  • Hook Registry：統一抽象層
  • Skill Discovery：自動化維護
  • 配置整合：合併相關 YAML
