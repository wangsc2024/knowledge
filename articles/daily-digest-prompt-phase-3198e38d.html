Daily-Digest-Prompt 自動任務排程效能與資源利用率優化建議

研究日期：2026-02-26
分析焦點：18 個自動任務的排程效能、並行瓶頸、Timeout 調適策略與資源利用率
差異化：聚焦排程層面的運行時效能優化（前次研究聚焦狀態管理/Token 經濟學）

一、專案排程現狀摘要

1.1 排程架構

| 排程 | 頻率 | 腳本 | 平均耗時 |

|------|------|------|----------|

| todoist-team | 每小時半點 02:30-23:30 | run-todoist-agent-team.ps1 | 749s (p50=720s, p90=992s) |

| todoist-single | 每小時整點 02:00-23:00 | run-todoist-agent.ps1 | 699s (p50=618s, p90=1215s) |

| daily-digest-team | 3 次/日 | run-agent-team.ps1 | 470s (p50=452s, p90=544s) |

| system-audit | 1 次/日 00:40 | run-system-audit-team.ps1 | ~20min |

1.2 執行品質（200 次歷史樣本）

  • 成功率：98.5%（197/200），3 次失敗均為 todoist-team Phase 1 超時
  • Section-level timeout：0（Phase 2 動態 timeout 機制運作良好）
  • 排程重疊：0 次（不同 Agent 30 分鐘錯開策略有效）
  • 超出 30 分鐘時段：0 次（無排程溢出）

1.3 耗時分布

| 區間 | 次數（近 50 次）| 百分比 |

|------|----------------|--------|

| 5-10 分鐘 | 8 | 16% |

| 10-15 分鐘 | 26 | 52% |

| 15-20 分鐘 | 13 | 26% |

| 20-30 分鐘 | 3 | 6% |

核心觀察：超過半數集中在 10-15 分鐘，但排程時段為 30 分鐘/60 分鐘，存在約 15-20 分鐘的閒置窗口。

二、識別的六大優化機會

2.1 排程密度優化 — 閒置窗口的利用

現狀問題：

  • todoist-team p50=720s（12 分鐘），但排程時段為 60 分鐘
  • 即使 p90=992s（16.5 分鐘），仍有 43.5 分鐘閒置
  • 18 個自動任務合計 45 次/日上限，但每次 team-mode 僅並行 2 個

優化建議：

  1. 提升 max_auto_per_run.team_mode 從 2 到 3：Phase 2 可同時啟動 3 個 auto-task Agent（而非 2 個）。因並行執行取 max 而非 sum，timeout 不會線性增加。估計可加速 45 次任務的完成速度約 33%。
  1. 自適應排程密度：根據前一次 duration_seconds 動態調整下次排程間隔。概念：若上次僅花 7 分鐘，下次可提前 15 分鐘觸發（前提是不與其他排程衝突）。
  1. idle 模式快速退出：plan_type 為 idle 時，Phase 2 timeout 僅為 buffer（120s），但 Phase 1 仍用完整 420s。可在 Phase 1 偵測到 idle 時提前返回，將整體 idle 耗時從 ~420s 降至 ~120s。

2.2 Timeout 調適策略 — 從靜態到動態

現狀問題：

  • Phase 2 已有按任務類型的動態 timeout（research=600s, code=900s 等），但缺乏歷史學習機制
  • creative_game_optimize 固定 900s，但多數情況 600-700s 即完成
  • ai_deep_research 固定 720s，但 4 階段研究偶爾需要更多時間

借鑑 Astraea 的自適應方案：


# 建議新增 config/adaptive-timeout.yaml
adaptive_timeout:
  enabled: true
  strategy: "ewma"  # 指數加權移動平均
  alpha: 0.3        # 學習率（新觀測權重 30%）
  safety_margin: 1.3  # 在 EWMA 上加 30% 安全邊際
  min_timeout: 180    # 最低 3 分鐘
  max_timeout: 1200   # 最高 20 分鐘
  history_window: 10  # 追蹤最近 10 次執行

核心演算法：


new_timeout = EWMA(recent_durations) × safety_margin
如果近 3 次均成功且 actual < timeout * 0.5 → 降低 safety_margin 至 1.2
如果近 3 次有 timeout → 提升 safety_margin 至 1.5

預期效果：

  • 經常快完成的任務（如 buddhist research）：timeout 從 600s 自動降至 ~480s
  • 偶爾超時的任務（如 ai_deep_research）：timeout 從 720s 自動升至 ~900s
  • 整體 Phase 2 等待時間減少約 15-20%

2.3 Phase 1 查詢效能優化

現狀問題：

  • 3 次失敗（3/200 = 1.5%）全部發生在 Phase 1，均為超時
  • Phase 1 timeout = 420s，含重試共 ~900s
  • Phase 1 的主要瓶頸：Todoist API 呼叫 + routing 計算 + plan.json 寫入

優化建議：

  1. 快取前次 routing 結果：若 Todoist 任務列表未變化（比對 hash），直接複用前次 plan.json 的路由結果，僅更新計分排序。
  1. Phase 1 分段 timeout：將 420s 拆分為 API 呼叫（120s）+ 路由計算（120s）+ 寫入（60s），每段獨立超時，精準定位瓶頸。
  1. Fallback 優化：現有的 Fallback（檢查 plan.json 檔案時間）已證明有效，但可增加 plan.json 內容完整性驗證（如檢查 plan_type 欄位存在）。

2.4 雙軌模式的資源效率分析

現狀數據：

  • todoist-team：118 次執行，avg=749s，失敗 3 次
  • todoist-single：67 次執行，avg=699s，失敗 0 次

關鍵洞見：

  • todoist-single 零失敗但平均耗時接近，且 p90=1215s 比 team 的 992s 高 22%
  • team-mode 的 Phase 1 overhead（~120-200s 的 Job 啟動和 JSON 解析）在任務量少時不划算
  • 但 team-mode 可並行處理多任務，在 auto-task 場景下優勢明顯

優化建議：

  1. 根據時段自適應選擇模式：

- 白天活躍時段（08:00-22:00）：team-mode（可能有 Todoist 任務需並行處理）

- 深夜自動任務時段（02:00-07:00）：考慮僅用 single-mode（減少 Phase 1 開銷）

  1. 合併為單一入口：將 todoist-single 和 todoist-team 合併為一個腳本，根據 Phase 1 查詢結果（任務數量和類型）動態決定是否啟用並行模式。任務 ≤1 時走單一路徑，≥2 時啟用 Start-Job 並行。

2.5 背壓（Backpressure）機制

現狀問題：

  • 當 API 回應變慢或知識庫服務負載高時，所有 Agent 仍以原速發送請求
  • 無法根據下游服務健康狀態調整請求頻率

借鑑 Agent.xpu 與 Astraea 的背壓設計：


# 建議新增 config/backpressure.yaml
backpressure:
  enabled: true
  monitors:
    - api: "todoist"
      latency_threshold_ms: 3000
      error_rate_threshold: 0.1
      action: "reduce_parallelism"
    - api: "knowledge"
      latency_threshold_ms: 5000
      error_rate_threshold: 0.2
      action: "skip_kb_import"
  actions:
    reduce_parallelism:
      max_auto_per_run: 1  # 從 2 降至 1
      cooldown_minutes: 15
    skip_kb_import:
      description: "暫停知識庫寫入，僅快取結果"
      cooldown_minutes: 30

預期效果：

  • API 延遲高時自動降低並行度，避免雪崩
  • 知識庫故障時不阻塞研究流程，結果暫存待後續匯入
  • 與現有 Circuit Breaker 互補：CB 是「全斷」，背壓是「降速」

2.6 排程資料可觀測性增強

現狀問題：

  • scheduler-state.json 僅記錄時間、狀態、耗時，缺少資源利用率資訊
  • 無法判斷 Phase 2 的並行效率（如 3 個 Agent 中 2 個 5 分鐘完成，1 個 12 分鐘完成）

優化建議：

  1. 新增 Phase-level 計時：
    2. 
{
  "phase_timings": {
    "phase1": { "start": "...", "end": "...", "duration_s": 180 },
    "phase2": {
      "start": "...", "end": "...", "duration_s": 480,
      "agents": [
        { "name": "auto-shurangama", "duration_s": 420, "status": "success" },
        { "name": "auto-tech_research", "duration_s": 480, "status": "success" }
      ],
      "parallelism_efficiency": 0.88
    },
    "phase3": { "start": "...", "end": "...", "duration_s": 120 }
  }
}
  1. 並行效率公式：
    2. 
parallelism_efficiency = sum(agent_durations) / (max(agent_durations) × agent_count)

若 3 個 Agent 各花 400s/450s/480s，效率 = 1330/(480×3) = 0.92（很高）。

若 3 個 Agent 各花 120s/120s/480s，效率 = 720/(480×3) = 0.50（兩個 Agent 閒置過長）。

  1. 趨勢儀表板：在 check-health.ps1 中新增排程效率趨勢（近 7 天的 p50/p90/並行效率）。

四、與學術研究的對照

4.1 Astraea（狀態感知排程引擎）

Astraea（arXiv: 2512.14142）提出的「狀態感知排程」核心思想：根據請求的歷史狀態（已完成步驟、API 等待時間）動態調整排程優先級。本專案可借鑑：

  • Service Time Predictor：類似本建議的 EWMA timeout，Astraea 用離線 profiling + 線上學習預測每個請求的完成時間
  • Adaptive Cache Manager：類似本專案的 api-cache，但 Astraea 加入了「高/低水位閾值」的動態切換策略
  • JCT 降低 25.5%：證明自適應排程相比靜態排程有顯著效益

4.2 Agent.xpu（異構排程）

Agent.xpu 的「inter-XPU backfill」概念在本專案中的映射：

  • 當一個 auto-task Agent 提前完成時，可「回填」另一個 auto-task（類似 backfill 填補氣泡）
  • 現有架構的限制：Phase 2 使用 Wait-Job 等待所有 Job，無法中途加入新 Job
  • 長期建議：改用事件驅動的 Job 管理（PowerShell 的 Register-ObjectEvent 或自訂佇列）

4.3 去中心化自適應任務分配

Nature 論文（s41598-025-21709-9）提出的兩層架構（自適應控制器 + 選擇性廣播）對本專案的啟發：

  • Phase 1 的「查詢→路由→計畫」本質上就是集中式調度器
  • 未來可考慮讓 auto-task Agent 自主拉取任務（pull model），而非由 Phase 1 推送（push model）
  • Pull model 的優勢：Agent 完成當前任務後立即拉取下一個，無需等待統一的 Phase 3

五、快速勝利（Quick Wins）

以下 3 項可在 30 分鐘內實施，立即見效：

5.1 提升並行度


# config/frequency-limits.yaml
max_auto_per_run:
  team_mode: 3  # 從 2 改為 3

5.2 idle 快速返回

在 Phase 1 Agent 偵測到 idle 時，plan.json 加入 "early_exit": true，Phase 2 讀取後跳過等待。

5.3 Phase 計時記錄

在 run-todoist-agent-team.ps1 的 Update-State 中，新增 phase_durations 欄位：


$sections["phase_durations"] = @{
    phase1 = $phase1Duration
    phase2 = $phase2Duration
    phase3 = $phase3Duration
}

六、總結

本專案的排程系統已達到 98.5% 成功率，架構穩健。主要優化方向不在「修復」而在「提效」：

  1. 短期（P0-P1）：提升並行度、增加可觀測性、Phase 1 分段超時 — 預計提升整體吞吐量 20-30%
  2. 中期（P2）：自適應 timeout（EWMA） — 預計減少 15-20% 的無效等待時間
  3. 長期（P3）：雙軌合併、背壓機制、事件驅動排程 — 為未來擴展（更多自動任務/更多 API）奠定基礎

核心理念：從靜態配置驅動轉向歷史數據驅動的自適應排程，讓系統根據自身執行經驗自動調優。
