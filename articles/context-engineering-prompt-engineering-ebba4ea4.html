Context Engineering 完整指南 — 從 Prompt Engineering 到系統級上下文架構的演進

技術概述

Context Engineering（上下文工程）是 2025-2026 年 AI Agent 開發領域最重要的範式轉移。Anthropic 將其定義為「策劃和維護 LLM 推論時最佳 token 集合的策略」。與 Prompt Engineering 聚焦「怎麼問」不同，Context Engineering 聚焦「模型在推論時能存取什麼資訊」，是一門設計動態系統的學科 — 在正確的時間、以正確的格式，提供正確的資訊與工具。

核心洞見：Agent 成功的關鍵主要取決於上下文品質，而非模型精細度。大多數失敗源於不充分的上下文，而非模型本身的限制。

---

一、Prompt Engineering vs Context Engineering

| 維度 | Prompt Engineering | Context Engineering |

|------|-------------------|--------------------|

| 核心問題 | 「我該怎麼措辭？」 | 「模型現在需要存取什麼資訊？」 |

| 作用範圍 | 單次輸入-輸出對 | 模型看到的所有內容（記憶、歷史、工具、系統提示） |

| 本質 | 撰寫靜態指令 | 設計動態資訊系統 |

| 關係 | Context Engineering 的子集 | 包含 Prompt Engineering + 更多 |

| 產出 | 一段最佳化的 prompt 文字 | 一套完整的上下文架構系統 |

關鍵區別：Prompt Engineering 是離散任務（寫好一段指令），Context Engineering 是迭代式的系統設計（什麼上下文配置最可能產生模型的期望行為）。

---

二、七大上下文組件

一個完整的 Context Engineering 系統包含七個組件：

  1. Instructions / System Prompt — 行為準則與範例
  2. User Prompt — 即時任務或問題
  3. State / History — 當前對話線程
  4. Long-Term Memory — 跨 session 持久化知識
  5. Retrieved Information (RAG) — 外部、即時更新的知識來源
  6. Available Tools — 工具定義與能力描述
  7. Structured Output — 回應格式規範

---

三、四大核心策略

3.1 Scratchpads（暫存筆記）

Agent 使用暫存筆記保存資訊供未來參考。類似人類的「工作備忘錄」，在多步驟任務中維持連貫性。

3.2 Context Trimming（上下文修剪）

使用硬編碼啟發式規則修剪上下文 — 移除舊訊息、按重要性過濾、或使用訓練過的修剪器。核心原則：每個 token 都消耗模型的「注意力預算」，Transformer 架構需要 n-squared 成對 token 關係計算。

3.3 Isolation（隔離策略）

不同任務需要不同資訊，隔離技術將上下文分散到專門化系統中，而非將所有上下文塞入單一模型視窗。子 Agent 架構是典型實現 — 專門化子 Agent 以乾淨的上下文視窗處理聚焦任務，返回精煉摘要（1,000-2,000 tokens）給協調主 Agent。

3.4 State Object Isolation（狀態物件隔離）

Agent 的執行時狀態設計為結構化 schema，多個欄位中只有一個（如 messages）在每步暴露給 LLM，其他欄位保持隔離供選擇性使用。

---

四、長期任務技術

4.1 Compaction（壓縮）

對話接近上下文限制時進行摘要，保留關鍵細節（架構決策、未解決的 bug），丟棄冗餘工具輸出。實施建議：先最大化召回率，再迭代改善精確度。

4.2 Structured Note-Taking（結構化筆記）

Agent 持久化外部記憶（如 NOTES.md 檔案、記憶工具），可在後續檢索。這使得跨多次上下文重置仍能維持連貫性，支援多小時任務序列。

4.3 Sub-Agent Architecture（子 Agent 架構）

專門化子 Agent 以乾淨的上下文視窗處理聚焦任務，返回精煉摘要給協調主 Agent。這將詳細搜尋上下文與高階綜合工作隔離開來。

---

五、三層記憶架構

5.1 Short-term Memory（短期記憶）

活躍上下文視窗，保存「最近對話/推理、工具輸出、和檢索到的文件」。

5.2 Long-term Memory（長期記憶）

外部儲存（通常是向量資料庫），保存：

  • 情節記憶：過去事件、用戶互動、偏好
  • 語義記憶：通用和領域知識
  • 程序記憶：例行程序、工作流、決策步驟

5.3 Working Memory（工作記憶）

任務專屬的暫時空間，如在旅行訂票過程中維持日期資訊，不永久儲存。

核心原則：「最糟的記憶系統是忠實儲存所有內容的系統。」系統應在儲存前過濾互動，通常透過模型反思指派重要性分數。

六、四大上下文失敗模式

| 失敗模式 | 說明 | 對策 |

|---------|------|------|

| Context Poisoning | 幻覺資訊透過 Agent 重複使用而複合 | 驗證工具輸出、交叉比對 |

| Context Distraction | 過多歷史淹沒推理能力 | 主動修剪、摘要壓縮 |

| Context Confusion | 不相關工具擁擠決策空間 | 動態工具註冊、最小工具集 |

| Context Clash | 矛盾資訊造成推理停滯 | 優先級規則、衝突解決機制 |

---

七、Context Rot（上下文腐爛）

2026 年最受關注的挑戰之一：隨著上下文視窗中 token 數量增加，模型準確回憶該上下文資訊的能力下降。即使現代模型支援 100K-1M token 視窗，並不代表填滿它是最佳策略。

應對原則：尋找「最小高訊號 token 集合，最大化期望結果的可能性」。

---

八、與 Daily-Digest-Prompt 專案的關聯

本專案（daily-digest-prompt）已在多個維度實踐了 Context Engineering 的核心理念：

| CE 概念 | 本專案實踐 |

|---------|----------|

| 七大組件 | System Prompt (SKILL.md) + User Input (Todoist API) + State (digest-memory.json) + Long-term Memory (context/) + RAG (KB API) + Tools (22 Skills) + Structured Output (config/digest-format.md) |

| Isolation | 子 Agent 架構（Phase 1 並行 - Phase 2 組裝），各 Agent 只看自己需要的上下文 |

| Compaction | Phase 1 Agent 返回精煉 JSON，而非原始 API 回應 |

| Structured Note-Taking | digest-memory.json、research-registry.json 跨 session 持久化 |

| Context Trimming | scheduler-state.json 保留最近 200 筆、JSONL 日誌 50MB 輪轉 |

| State Object Isolation | 配置層（YAML）與模板層（Markdown）分離，按需載入不預載 |

可改進方向

基於 Context Engineering 理論，本專案可進一步優化：

  1. Context Pressure 監控：在 post_tool_logger 追蹤 context token 累積量，在 agent_guardian 加入 context_pressure 警告（已列入 improvement-backlog P0）
  2. 動態工具註冊：目前 22 個 Skill 全列在 SKILL_INDEX.md，可改為按任務類型動態載入相關 Skill 子集
  3. 三層記憶正式化：將現有的 digest-memory.json（短期）+ research-registry.json（工作記憶）+ KB（長期）明確對齊三層記憶架構
  4. Context Rot 防護：為長時間執行的任務（如 tech-research）加入上下文品質監控，主動摘要而非累積

---

九、最佳實踐摘要

  1. 從最小開始：先用基本 prompt 測試，根據觀察到的失敗再增加指令
  2. 系統性組織：使用 XML 標籤或 Markdown 標題結構化 prompt
  3. 視上下文為有限資源：每個 token 消耗注意力預算
  4. 優先訊號雜訊比：寧可精煉 500 token 也不要 5000 token 的雜訊
  5. 工具設計自包含：工具描述清楚、參數明確、返回 token 高效
  6. Just-In-Time 載入：Agent 維持輕量識別符，動態載入資料而非預載所有內容
  7. 監控 Context Health：追蹤上下文使用率、失敗模式、記憶品質
