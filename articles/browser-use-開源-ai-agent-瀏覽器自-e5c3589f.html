Browser Use — 開源 AI Agent 瀏覽器自動化平台深度研究

研究日期：2026-02-24 研究範圍：Browser Use 平台架構、五層系統設計、DOM 處理引擎、事件驅動架構、LLM 整合模式，以及對本系統（Daily Digest）的可借鏡技術分析 來源品質：A 級 3 個（GitHub 官方 repo、官方網站、官方文件）+ B 級 4 個（技術評測、架構分析、對比文章）

一、專案概述

Browser Use（browser-use/browser-use）是 2024 年創立於舊金山的開源專案，截至 2026 年 2 月已獲得 78,900+ Stars、9,300+ Forks、293 位貢獻者，採用 MIT 授權。

核心定位：讓 AI Agent 能操控網頁瀏覽器，以自然語言描述任務，自動完成網頁導航、表單填寫、資料擷取等操作。定位為「AI 使用網路的方式」（The way AI uses the web）。

解決的問題：傳統的瀏覽器自動化（Selenium、Playwright）需要開發者撰寫精確的選擇器和操作步驟，維護成本高且容易因頁面變動而失效。Browser Use 透過 LLM 理解頁面語義，讓自動化變得更加彈性和智能。

核心價值：

  • 自然語言驅動的瀏覽器自動化
  • 多 LLM 提供者支持（ChatBrowserUse、Claude、Gemini、Ollama 本地模型）
  • 開源核心 + 商業雲端服務的雙軌模式
  • 被 Fortune 500 企業信任（Airbnb、Amazon、Google、Microsoft、Stripe、Uber）

二、技術架構（五層系統設計）

2.1 架構總覽

Browser Use 由五個主要層次組成：

┌─────────────────────────────────┐
│   Layer 5: LLM Integration      │  提供者抽象、結構化輸出
├─────────────────────────────────┤
│   Layer 4: Tools & Actions      │  Action 定義、執行管線
├─────────────────────────────────┤
│   Layer 3: DOM Processing       │  DOM 提取、增強、序列化
├─────────────────────────────────┤
│   Layer 2: Browser Sessions     │  CDP 連線、事件路由、守望者
├─────────────────────────────────┤
│   Layer 1: Agent System         │  Agent 迴圈、訊息管理、狀態追蹤
└─────────────────────────────────┘

2.2 Agent 系統（Layer 1）

Agent 遵循 Observe → Think → Act 的迴圈模式：

  1. 觀察（Observe）：提取當前瀏覽器狀態（DOM 結構 + 螢幕截圖）
  2. 思考（Think）：將觀察結果送交 LLM，決定下一步行動
  3. 行動（Act）：執行 LLM 決定的瀏覽器操作
  4. 重複：直到任務完成或達到最大步數

2.3 Browser Session 管理（Layer 2）

  • 透過 Chrome DevTools Protocol (CDP) 直接控制瀏覽器
  • 事件路由機制處理頁面載入、導航、錯誤等事件
  • 支持真實使用者 Profile（保留已登入的 session）
  • 看門狗（Watchdog）機制偵測無響應或卡住的狀態

2.4 DOM 處理引擎（Layer 3）

這是 Browser Use 的核心差異化技術：

  • 多源 DOM 提取：同時從 Accessibility Tree 和原始 DOM 取得頁面結構
  • LLM 優化序列化：將複雜的 DOM 樹轉為 LLM 容易理解的結構化 JSON，僅保留可互動元素（按鈕、連結、表單欄位）
  • DomService：專用服務負責 DOM 增強和序列化，大幅減少送入 LLM 的 token 量

2.5 事件驅動架構（Layer 2-3 交叉）

使用 EventBus 實現元件間的鬆耦合通訊：

  • Action 事件：ClickElementEvent、TypeTextEvent、NavigateToUrlEvent、ScrollEvent
  • 生命週期事件：BrowserStartEvent、BrowserStopEvent、TabCreatedEvent
  • 狀態事件：頁面載入完成、導航成功等

2.6 工具與行動系統（Layer 4）

支援可擴展的自定義工具：

@tool
def save_to_file(content: str, filepath: str):
    """Save extracted content to a file"""
    with open(filepath, 'w') as f:
        f.write(content)

開發者可透過裝飾器定義自定義工具，Agent 會在適當時機自動選擇使用。

2.7 LLM 整合（Layer 5）

  • 提供者抽象層：統一介面支援多種 LLM（OpenAI、Anthropic、Google、本地 Ollama）
  • ChatBrowserUse：專為瀏覽器任務優化的模型，完成任務速度比通用模型快 3-5 倍
  • 雙重理解：同時分析 DOM 結構（JSON）和螢幕截圖（Vision），提高準確性
  • 結構化輸出：LLM 回傳結構化的 Action 物件，而非自由文字

四、使用方式（快速上手）

4.1 安裝

uv init && uv add browser-use && uv sync
uvx browser-use install  # 安裝瀏覽器依賴

4.2 基本使用

from browser_use import Agent, Browser, ChatBrowserUse
import asyncio

async def main():
    browser = Browser()
    agent = Agent(
        task="在 GitHub 上搜尋 browser-use 的 star 數量",
        llm=ChatBrowserUse(),
        browser=browser,
    )
    result = await agent.run()
    print(result)

if __name__ == "__main__":
    asyncio.run(main())

4.3 CLI 使用

browser-use open https://example.com    # 導航
browser-use state                       # 列出可互動元素
browser-use click 5                     # 點擊元素
browser-use type "搜尋文字"              # 輸入文字
browser-use screenshot file.png         # 截圖
browser-use close                       # 關閉瀏覽器

六、潛在應用（與自身專案的連結）

6.1 Daily-Digest-Prompt 專案

  1. 新聞擷取增強：目前使用 curl + MCP 查詢屏東新聞。Browser Use 可用於擷取需要 JavaScript 渲染的動態新聞頁面，或登入後才能存取的資訊。
  1. Todoist 進階操作：某些 Todoist 功能可能沒有 API 支援（如批量操作、進階篩選器），可用 Browser Use 透過網頁介面操作。
  1. 知識庫研究自動化：讓 AI Agent 自動瀏覽、理解和摘要網頁內容，比 WebFetch + WebSearch 更深入。
  1. 架構借鏡 — 事件驅動：Browser Use 的 EventBus 模式值得本專案 Hook 系統參考，可將 Hook 從「攔截模式」擴展為「事件訂閱模式」。

6.2 遊戲專案 (D:\Source\game)

  • 自動化測試：用自然語言描述遊戲測試場景（「打開遊戲，等待載入，點擊開始按鈕，驗證分數顯示為 0」）
  • 跨瀏覽器測試：自動在 Chromium、Firefox、WebKit 上測試遊戲相容性

七、優缺點評估

優點

  1. 極低學習門檻：自然語言描述任務，非開發者也能使用
  2. 強大的 DOM 理解：多源提取 + LLM 優化序列化，準確率高
  3. 模型靈活性：支援雲端 LLM 和本地 Ollama，部署彈性大
  4. MIT 開源：無商業使用限制
  5. 活躍維護：每週更新，293 位貢獻者
  6. 生態完善：CLI、模板市場、雲端服務一應俱全
  7. 2026 年增長最快的開源專案之一

缺點

  1. 速度依賴 LLM：每步操作需等待 LLM 回應，比純 Playwright 慢很多
  2. 成本：ChatBrowserUse 輸出 $2.00/1M tokens，大量使用時成本可觀
  3. 確定性不足：LLM 的非確定性可能導致同一任務不同次執行路徑不同
  4. 框架整合有限：目前與 CrewAI、AutoGen、PhiData 等主流 Agent 框架的整合仍不完善
  5. 雲端依賴：Stealth 模式、CAPTCHA 解決等進階功能需要商業雲端版
  6. 資源消耗：跑多個瀏覽器實例時記憶體消耗大
