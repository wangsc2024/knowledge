Harness Engineering 治理三角優化：risk_tiers × 人在回路 × 執行契約

研究日期：2026-04-11

去重說明：與知識庫既有「Reliability Surface／Chaos」「B3 儀表三角」「Policy-as-Code 約束驗證」等筆記互補；本筆記聚焦 路徑風險分級、人工審核觸發與執行契約（冪等／重試／審核）的三方對齊，並對照業界 SRE 錯誤預算與 Agent 治理框架。

---

執行摘要

本專案 config/autonomous-harness.yaml 已將 Harness 工程拆為 B0–B5 元件，並明確 OODA（context/workflow-state.json）與 Harness（state/autonomous-runtime.json 等）的寫入邊界。在「優化方案」層級，建議把三條已存在但易分散理解的軸線收斂為 治理三角：

  1. risk_tiers（B0-E）：依路徑 glob 對應 critical／high／medium／low，每級綁定 required_checks（如 preflight、schema_validation、human_review、test_pass）。這與 Open Policy Agent 社群常見的 依路徑與情境評分的策略決策 同構：以宣告式規則決定「要跑哪些檢查」，而非在程式裡散落 if/else [來源：hoop.dev Risk-based Access Control with OPA, 部落格]。
  2. human_review_gates：定義「哪些變更路徑觸發人工審核」（如 hooks、run-*.ps1、排程行為），與 risk_tiers 中 critical 路徑要求 human_review 互補——前者是觸發器集合，後者是分級策略 [對照本專案 YAML 註解]。
  3. execution_contracts：集中 冪等鍵欄位、可安全重試 vs 需審核後重試、human_review_gates 與 LangGraph／Trigger.dev 啟發的 durable 執行語意，使「變更節奏」可被治理。

業界側，Google SRE 錯誤預算（error budget） 將 SLO 轉成可耗盡的配額：當服務在滾動視窗內超出預算，應暫停非必要變更直至恢復 SLO，並對高消耗事件要求事後檢討與行動項 [來源：Google SRE Workbook, Appendix B Example Error Budget Policy, Thurgood 等, 2018]。Cloud Monitoring 對 burn rate（預算燃燒率） 的告警則補上「短視窗內異常消耗」的早期預警 [來源：Google Cloud Documentation, Alerting on burn rate]。此兩者可 類比 到本專案：以 state/api-health.json 熔斷、state/failed-auto-tasks.json、state/scheduler-state.json 等訊號作為「可靠性預算」代理指標，在預算見底時 自動收緊 變更策略（例如提高審核層級、暫停非 P0 設定變更），與既有 component_flags 三層優先序（runtime_override > daily_limit > component_flags）對齊。

Agent 治理方面，新加坡 IMDA《Model AI Governance Framework for Agentic AI》（2026-01-22）強調 事前風險界定、工具／資料最小權限、可逆性與人在回路 [來源：Mayer Brown 對該框架之摘要, 2026-04]。AGENTSAFE 等框架則將風險分類轉成 設計／執行期／稽核 控制 [來源：Khan 等, AGENTSAFE 預印本, arXiv:2512.03180]。與本專案 risk_tiers + execution_contracts + 補償面（B5） 的組合方向一致。

---

一、本專案治理三角對照表

| 軸線 | 設定位置（摘要） | 優化重點 |

|------|------------------|----------|

| 路徑→檢查策略 | risk_tiers.*.paths + required_checks | 新增或調整路徑時，同步更新 tools／config 的 preflight 規則，避免「列在 critical 卻未跑 schema」的漂移 |

| 審核觸發 | human_review_gates.high_risk_paths + review_required_when | 將「提案敘述無法證明冪等」等條件與 execution_contracts.idempotency 連動，形成可測的 PR／self-heal 檢查清單 |

| 執行契約 | execution_contracts（冪等、重試、人工審核） | 對 knowledge_import、ntfy_notification、todoist_mutation 等 強制 trace_id + task_key + operation_name + attempt 的一致性，便於與 OTel／JSONL 對齊稽核 |

component_flags 優先序（runtime_override > daily_limit > enabled）應在營運手冊中與「錯誤預算見底時的降級」一併說明：緊急封鎖與頻率停用屬 硬停，元件旗標屬 行為調整，避免混淆 [來源：本專案 autonomous-harness.yaml B0-F 註解]。

---

二、可落地的優化方案（依影響排序）

2.1 「可靠性預算」代理指標與變更閘

  • 定義 1～3 個 SLI（例如：Phase2 結果檔完整率、排程連續失敗次數、API 熔斷開啟次數），以滾動視窗追蹤。
  • 當指標低於門檻時，啟動 變更凍結策略（對照 SRE：error budget exhausted 時暫停功能釋出 [來源：Google SRE Workbook, 2018]）：僅允許 P0／資安與 有 rollback 證明 的變更。
  • 短視窗 燃燒率 告警：對單一小時內異常激增的失敗／熔斷事件發出「快燃」告警 [概念對照：Google Cloud burn rate 說明]。

2.2 risk_tiers 與 PR／self-heal 的「同一真相來源」

  • 將 required_checks 與 CI／preflight_gate.py 的步驟 編號對齊，避免文件與程式不一致。
  • critical 路徑變更預設 兩人審核或 maintainer，呼應高風險 Agent 操作需可稽核軌跡的治理趨勢 [對照：WEF AI Agents 評估與治理基礎架構之風險評估與漸進式治理敘述, 2025 脈絡]。

2.3 執行契約與人在回路的「計畫先行」

  • 對標 Osprey 類「plan-first」：在實際改動 hooks／排程前產出可審查的變更計畫與依賴 [來源：Osprey 框架 arXiv:2508.15066v3, 2025-12]；與本專案 checkpoint、never_retry_without_review 清單一致化。

三、與既有 KB 筆記的關係

  • Reliability Surface／Chaos：補強「事後韌性」；本筆記補強「變更前治理與契約」。
  • B3 儀表三角：補強可觀測性指標；本筆記補強 誰在何種路徑上必須被攔下來審。
  • Policy-as-Code 約束驗證：補強外部工具映射；本筆記補強 本 repo 內生的三層 YAML 如何閉環。

五、實務應用建議（給未來自己的三句話）

  1. 改 YAML 路徑 glob 時，同步跑 uv run python hooks/validate_config.py 並更新 risk tier 對照表。
  2. 熔斷或失敗任務激增時，先視為 error budget 見底，縮小變更面而不是加新功能。
  3. 所有跨系統副作用 必須落在 execution_contracts.idempotency.required_for 所列操作類型上，並保留可追溯的 attempt／trace。
