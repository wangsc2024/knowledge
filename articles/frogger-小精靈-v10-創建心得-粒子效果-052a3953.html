Frogger (小精靈) v1.0 創建心得

專案背景

經典復古遊戲 Frogger 的 HTML5 Canvas 現代化重製。遊戲核心是引導青蛙穿越車流與河流到達對岸的 5 個歸巢點。

核心技術實作

1. 雙區域碰撞設計

馬路區域和河流區域的碰撞邏輯完全不同：

  • 馬路：碰到車輛 = 死亡（標準 AABB 碰撞）
  • 河流：必須站在原木/烏龜上才能存活，否則落水死亡
  • 河流平台移動：青蛙站在原木上時，需隨原木同步移動（每幀更新 frog.x += lane.dir speed dt）

2. 車道物件管理

每條車道獨立管理物件（車輛/原木/烏龜）：

  • 方向（dir: 1/-1）控制移動方向
  • 速度隨關卡遞增（speedMult = 1 + (level-1) * 0.12）
  • 物件超出畫面後循環到另一端（wrap around）

3. 歸巢系統

頂部 5 個歸巢點等距分布，青蛙到達後：

  • 標記為已佔據（occupied）
  • 獎勵分數 = 50 + 剩餘時間 * 10
  • 5 個全部到達 → 升級（+1000 分獎勵）

4. 粒子效果

三種死亡方式各有不同粒子效果：

  • 被車撞：紅色+橙色碎片 + 螢幕震動
  • 落水：藍色水花
  • 歸巢成功：綠色光效
  • 使用 swap-and-pop 回收，池上限 200

5. 8-bit 音效合成

使用 Web Audio API 即時合成 7 種音效：

  • hop（跳躍）：方波 300→500Hz
  • splash（落水）：鋸齒波 400→80Hz
  • squash（被撞）：鋸齒波 200→50Hz
  • home（歸巢）：C5→E5→G5 上行音
  • levelup（升級）：四音上行
  • gameover（結束）：下行悲傷音
  • tick（倒數）：高頻短促

設計要點

關卡難度設計

5 種車道類型各有不同速度：

  • 慢車（speed 1.2）→ 卡車（0.8）→ 快車（1.8）→ 賽車（2.5）→ 卡車（1.0）
  • 河流：短原木、烏龜群、長原木各有不同寬度和速度
  • 賽車（row 8）是最大威脅：速度 2.5x，僅 1 輛但極快

時間壓力

30 秒時間限制：

  • 計時條在右下方，隨時間變色（綠→黃→紅）
  • 最後 5 秒有 tick 音效提示
  • 超時直接死亡

經驗教訓

  1. 河流碰撞比馬路複雜得多：需要處理「站在平台上」的持續判定，而非單次碰撞
  2. 物件循環 (wrap) 邊界：需要在物件完全離開畫面後才重置，否則會有閃爍
  3. 移動冷卻 (cooldown)：120ms 冷卻防止快速重複移動，讓操控更精準
  4. smooth movement：青蛙移動用 lerp 而非瞬移，視覺更流暢
  5. dt cap：限制 dt 最大 50ms，防止切換分頁後回來的巨大 dt 造成物件穿越

適用場景

這些技術模式適用於：

  • 有多條車道/通道的遊戲（碰撞分區管理）
  • 需要「站在移動平台上」機制的遊戲
  • 時間壓力 + 分數獎勵設計
  • 物件循環出現的射擊/跑酷遊戲
