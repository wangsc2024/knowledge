一、專案概述

Aider（Aider-AI/aider）是由 Paul Gauthier 創建的開源 AI 配對程式設計工具，以「AI pair programming in your terminal」為定位。截至 2026 年 2 月底已累積 41,000+ Stars、168 位貢獻者、577 個依賴專案，PyPI 總安裝量超過 510 萬次，每週處理 150 億 tokens。採用 Apache 2.0 授權。

| 指標 | 數值 |

|------|------|

| GitHub Stars | 41,000+ |

| PyPI 安裝量 | 5.1M+ |

| 每週 Token 處理量 | 15B |

| 支援程式語言 | 100+ |

| 授權 | Apache 2.0 |

| 「奇點指標」| 88%（最新版本 88% 的程式碼由 Aider 自身撰寫）|

核心價值

Aider 解決了傳統 AI 程式碼輔助工具的核心痛點：單檔案視角的局限性。大多數 AI 編碼助手只分析游標附近的上下文來建議程式碼片段，而 Aider 透過 RepoMap 技術建立全碼庫感知，讓 LLM 理解整個專案的架構、依賴關係和設計模式，實現跨檔案的智能編輯。

二、技術架構

2.1 RepoMap — 核心競爭力

RepoMap 是 Aider 最具技術深度的元件，位於 aider/repomap.py，實現了一套基於圖論的程式碼上下文生成系統。

#### 資料結構

標籤（Tag）使用 namedtuple 定義：


Tag(rel_fname, fname, line, name, kind)
# kind: "def"（定義）或 "ref"（引用）

#### Tree-Sitter AST 解析

  • 使用 py-tree-sitter-languages 模組解析原始碼為抽象語法樹
  • 透過 .scm 查詢檔案（位於 aider/queries/）提取函數、類別、變數的定義與引用
  • 對於 Tree-Sitter 僅提供定義的語言，使用 Pygments 作為引用提取的 fallback

#### 圖構建與 PageRank 排序

系統建立 networkx.MultiDiGraph：

  • 節點：相對檔案路徑
  • 邊：透過共享識別符號連接的檔案間引用關係
  • 邊權重：多重乘法器

| 條件 | 權重乘數 |

|------|----------|

| 基礎 | ×1.0 |

| 在對話中提及的識別符 | ×10 |

| snake/kebab/camelCase ≥8 字元 | ×10 |

| 私有識別符（_ 前綴） | ×0.1 |

| 定義在 >5 個檔案中 | ×0.1 |

| 來自聊天檔案 | ×50 |

| 引用次數加成 | ×√(num_refs) |

#### 個人化向量

PageRank 的初始權重偏好特定檔案：

  • 聊天中的檔案：+100 / len(fnames)
  • 提及的檔案：+100 / len(fnames)
  • 路徑組件匹配提及識別符的檔案：+100 / len(fnames)

#### Token 預算管理

使用二分搜尋找到符合 max_map_tokens（預設 1024）的最大標籤數量：

  • 起始：middle = min(max_map_tokens // 25, num_tags)
  • 收斂：O(log N) 迭代
  • 目標：輸出在預算的 15% 範圍內
  • 大型文字使用 1% 行採樣估算 Token 數

2.2 三級快取架構

| 層級 | 類型 | 用途 |

|------|------|------|

| TAGS_CACHE | 磁碟持久化（diskcache） | 儲存檔案修改時間與提取的標籤 |

| map_cache | 記憶體字典 | 快取格式化後的 repo map |

| tree_cache | 記憶體快取 | 快取渲染的程式碼片段 |

SQLite 錯誤時自動回退到記憶體儲存。

2.3 多模式聊天系統

| 模式 | 用途 | 特點 |

|------|------|------|

| Code | 直接程式碼編輯 | LLM 直接修改檔案 |

| Architect | 架構設計 | 架構師模型提案 → 編輯器模型執行 |

| Ask | 諮詢討論 | 不修改檔案，純討論 |

| Help | 使用說明 | Aider 使用指引 |

Architect 模式特別值得注意：它將「思考」與「執行」分離，讓較強的推理模型負責設計決策，再由編輯專用模型翻譯為具體的檔案變更。

三、功能特色

3.1 與同類工具的差異化

| 特性 | Aider | Cursor | Claude Code |

|------|-------|--------|-------------|

| 介面 | 終端 CLI | VS Code IDE | 終端 CLI |

| 碼庫感知 | RepoMap PageRank | RAG 檔案系統索引 | Context Window |

| 模型自由度 | 任意 LLM | 內建模型為主 | Claude 專屬 |

| 成本 | 免費（付 API 費用） | 訂閱制 | API 費用 |

| Git 整合 | 自動 commit | 有 | 有 |

| 開源 | Apache 2.0 | 非開源 | 非開源 |

| SWE-bench | 79% | 87% | 80.9% |

3.2 核心功能清單

  1. 全碼庫感知：RepoMap + Tree-Sitter 建構專案結構圖
  2. 100+ 語言支援：Python、JavaScript、Rust、Go、C++ 等
  3. Git 自動提交：每次變更自動生成描述性 commit message
  4. Lint + Test 自動化：修改後自動執行 linting 和測試，發現問題自動修正
  5. 圖片/網頁輸入：接受視覺參考和文件上下文
  6. 語音編碼：語音輸入程式碼修改需求
  7. IDE 整合：可在 VS Code 等 IDE 中透過註解觸發
  8. Prompt Caching：減少重複 API 呼叫成本

3.3 「奇點指標」（Singularity Metric）

Aider 追蹤一個獨特的指標：最新版本中有多少比例的程式碼是由 Aider 自身撰寫的。目前這個數字達到 88%，意味著 Aider 幾乎完全自我迭代。這不僅是技術成就的展示，也是產品品質的最佳證明 — 開發者信任 Aider 到足以用它來開發 Aider 本身。

四、使用方式

快速上手


# 安裝
python -m pip install aider-install
aider-install

# 使用 Claude Sonnet
cd /your/project
aider --model sonnet --api-key anthropic=sk-xxx

# 使用 DeepSeek（成本極低）
aider --model deepseek --api-key deepseek=xxx

# 使用本地模型（Ollama）
aider --model ollama/codellama

常用操作


# 在對話中加入檔案
/add src/main.py tests/test_main.py

# 切換模式
/architect   # 切換到架構師模式
/code        # 切換到程式碼模式

# 執行測試
/test pytest

# 語音輸入
/voice

YAML 配置


# .aider.conf.yml
model: sonnet
auto-commits: true
map-tokens: 2048
lint-cmd: ruff check
test-cmd: pytest

六、潛在應用（與 Daily-Digest-Prompt 的連結）

6.1 RepoMap 啟發：Skill 依賴圖譜

Aider 的 RepoMap 使用 PageRank 排序檔案重要性 — 這個概念可以應用到 Daily-Digest-Prompt 的 Skill 系統：

  • 建立 Skill 依賴圖（哪些 Skill 經常被一起使用）
  • 使用 PageRank 識別「核心 Skill」vs「邊緣 Skill」
  • 動態調整 Skill 索引的載入順序，優先載入高關聯性 Skill

6.2 三級快取策略借鑑

Aider 的 TAGS_CACHE（磁碟）→ map_cache（記憶體）→ tree_cache（記憶體）三級快取策略，可啟發 Daily-Digest-Prompt 的 API 快取系統：

  • 目前：單層 JSON 檔案快取
  • 可改進：加入記憶體層快取（同一 session 內的重複查詢）+ 磁碟持久化（跨 session）

6.3 Architect 模式的雙 Agent 分工

Aider 的 Architect 模式將「設計」與「執行」分離，這與 Daily-Digest-Prompt 的 Todoist 團隊模式（Phase 1 規劃 → Phase 2 執行）高度相似，驗證了這種分層設計的有效性。

6.4 Token 預算管理

Aider 使用二分搜尋 + 採樣估算來控制 Token 使用，對 Daily-Digest-Prompt 的 Context Engineering 有借鑑價值 — 尤其是 SKILL_INDEX.md 的 Token 成本控制。

七、優缺點評估

優點

  1. 全碼庫感知：RepoMap + PageRank 是目前開源工具中最精密的上下文生成系統
  2. 模型無關：支援幾乎任何 LLM，包括本地模型，不鎖定供應商
  3. 零成本工具：工具本身免費，只需支付 LLM API 費用
  4. 自我驗證：88% 的程式碼由自身生成，是最佳的品質背書
  5. Git 原生：自動 commit 使得 AI 修改完全可追溯、可回退
  6. 多模式靈活：Architect / Code / Ask 模式適應不同場景
  7. Lint + Test 閉環：修改後自動驗證品質

缺點

  1. 學習曲線：純終端操作，對非 CLI 用戶不友善
  2. SWE-bench 分數略低：79% vs Cursor 87%，在複雜任務上可能不如 IDE 整合工具
  3. 無原生 GUI：缺乏視覺化程式碼差異展示
  4. Token 消耗：RepoMap 雖有 Token 預算控制，但大型專案仍可能消耗大量 Token
  5. 依賴外部 API：雖支援本地模型，但最佳效果仍需付費 API
