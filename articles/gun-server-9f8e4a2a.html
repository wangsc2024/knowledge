★ Insight ─────────────────────────────────────

Gun.js relay 的架構特殊性：

  1. Gun.js 的 RAD（Radisk）使用檔案鎖定機制，因此不能用 Node.js Cluster 多進程部署 — 這是 Gun 與一般 Express 應用最大的部署差異。正確做法是 fork 單進程 + PM2 自動重啟
  2. Gun.js 的 SEA（Security, Encryption, Authorization）提供的是端對端加密而非傳輸層安全 — SEA 保護的是訊息內容，但 WebSocket 連線本身若未走 TLS 則仍可被嗅探 metadata（誰連了誰、連線頻率等）
  3. express.static(__dirname) 是一個典型的資訊洩漏漏洞 — 整個伺服器目錄（含原始碼、package.json、.env）都變成可下載資源

結構化摘要

✅ 執行摘要：完成 Gun Relay Server 穩定性與安全性優化計畫 v1.0 的撰寫與知識庫存入。計畫基於 my-gun-relay/index.js（204 行）和 bot/bot.js（674 行）的實際程式碼分析，識別 9 項安全漏洞 + 8 項穩定性缺陷，提出 6 階段 4 週實施方案（~13 小時工時）。

📚 主要發現：

  1. 安全不對稱：bot.js 客戶端有完整防護（helmet、rate-limit、timing-safe），relay 伺服器端幾乎裸奔
  2. 三大 P0 問題：無 Rate Limiting、Token 字串直接比較（timing attack 風險）、無全域例外處理
  3. 靜態檔案洩漏：express.static(__dirname) 暴露整個伺服器目錄含原始碼
  4. API Key 可選：if (apiKey) 機制使得忘記設定環境變數 = 完全無認證
  5. Gun.js 部署限制：RAD 檔案鎖定不支援 Cluster，只能用 fork + PM2 單進程自動重啟
