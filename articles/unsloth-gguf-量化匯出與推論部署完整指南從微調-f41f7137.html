Unsloth GGUF 量化匯出與推論部署完整指南

概述

本指南涵蓋 Unsloth 微調完成後，將模型轉換為 GGUF 格式並部署到 Ollama、vLLM、LM Studio 等推論引擎的完整管線。重點包括：16 種以上量化方法的選擇策略、Dynamic 2.0 智慧量化技術、QAT（Quantization-Aware Training）訓練時量化、以及五條不同的匯出-部署路徑。GGUF（GPT-Generated Unified Format）是 llama.cpp 生態系的標準格式，也是本地部署的事實標準。

一、GGUF 量化方法全覽

1.1 支援的量化格式

Unsloth 支援 16 種以上量化方法，以下依推薦程度排列：

方法位元數說明適用場景
q4_k_m4-bit推薦預設。注意力與前饋層的一半使用 Q6_K，其餘 Q4_K生產部署平衡
q5_k_m5-bit同上但使用 Q5_K，更高精度品質優先
q8_08-bit快速轉換，資源需求中等開發測試、快速驗證
f1616-bit最快轉換，保留 100% 精度，但慢且佔記憶體基準比對
q4_0 / q4_14-bit原始 4-bit 方法，q4_1 精度略高於 q4_0極致壓縮
q5_0 / q5_15-bit漸進式品質提升中等壓縮
q4_k_s / q5_k_s4/5-bitK-Quant 小型變體記憶體受限環境
q2_k2-bit極端壓縮，attention.vw 用 Q4_K，其餘 Q2_K實驗性
q3_k_s / q3_k_m / q3_k_l / q3_k_xs3-bit3-bit 系列（小/中/大/超小）低資源設備
q6_k6-bit高品質壓縮品質與大小平衡
iq2_xxs / iq2_xs / iq3_xxs2-3 bitiQuant 系列，2.06-3.06 bpw極端壓縮研究

1.2 量化選擇決策樹

目標是什麼？
├─ 生產部署（平衡品質/速度/大小）→ q4_k_m（首選）
├─ 品質優先（可接受較大檔案）→ q8_0 或 q5_k_m
├─ 極致壓縮（邊緣設備/手機）→ q2_k 或 iq2_xxs
├─ 基準比對（無損參考）→ f16
└─ Apple Silicon 優化 → Q4_NL, Q5.1, Q5.0, Q4.1, Q4.0

二、五條匯出-部署管線

管線 A：直接匯出 GGUF（最簡單）

from unsloth import FastLanguageModel

# 載入微調後的模型
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="./finetuned_model",
    max_seq_length=2048,
)

# 直接存為 GGUF（一行搞定）
model.save_pretrained_gguf(
    "output_dir",
    tokenizer,
    quantization_method="q4_k_m"  # 推薦預設
)

# 或推送到 Hugging Face Hub
model.push_to_hub_gguf(
    "username/model-name",
    tokenizer,
    quantization_method="q4_k_m"
)

注意：記憶體不足時可調整：

model.save_pretrained_gguf(
    "output_dir", tokenizer,
    quantization_method="q4_k_m",
    maximum_memory_usage=0.5  # 預設 0.75，降至 0.5 避免 OOM
)

管線 B：先合併再手動轉換（最靈活）

# 步驟 1：合併 LoRA 並存為 16-bit（canonical checkpoint）
model.save_pretrained_merged(
    "merged_model",
    tokenizer,
    save_method="merged_16bit"  # 高保真檢查點
)
# 步驟 2：編譯 llama.cpp
apt-get install pciutils build-essential cmake curl libcurl4-openssl-dev -y
git clone https://github.com/ggml-org/llama.cpp
cmake llama.cpp -B llama.cpp/build \
    -DBUILD_SHARED_LIBS=OFF -DGGML_CUDA=ON -DLLAMA_CURL=ON
cmake --build llama.cpp/build --config Release -j --clean-first \
    --target llama-cli llama-mtmd-cli llama-server llama-gguf-split
cp llama.cpp/build/bin/llama-* llama.cpp

# 步驟 3：轉換為 GGUF
python llama.cpp/convert_hf_to_gguf.py merged_model \
    --outfile model-Q8_0.gguf --outtype q8_0 --split-max-size 50G

管線 C：匯出到 Ollama（本地對話）

# 步驟 1：存 LoRA adapter
model.save_pretrained("lora_adapter")

# 步驟 2：匯出為 GGUF（建議先用 Q8_0 快速驗證）
model.save_pretrained_gguf(
    "ollama_model", tokenizer,
    quantization_method="q8_0"
)

# 步驟 3：Unsloth 自動生成 Modelfile（含 chat template）
# 步驟 4：ollama create my-model -f Modelfile
# 步驟 5：ollama run my-model

關鍵警告：Chat Template 必須訓練與推論一致！這是 Ollama 部署後表現差的最常見原因。

管線 D：匯出到 vLLM（高吞吐量伺服器）

# 存為 16-bit 合併模型（vLLM 推薦格式）
model.save_pretrained_merged(
    "finetuned_model", tokenizer,
    save_method="merged_16bit"
)
# 安裝 vLLM
pip install --upgrade pip && pip install uv
uv pip install -U vllm --torch-backend=auto

# 啟動伺服器
vllm serve /path/to/finetuned_model

vLLM 支援 LoRA 熱插拔（hot-swapping），可在同一基礎模型上動態切換不同 LoRA adapter。

管線 E：QAT + TorchAO 匯出（最高品質量化）

from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="unsloth/Qwen3-4B-Instruct-2507",
    max_seq_length=2048,
    load_in_16bit=True,
)

# 啟用 QAT（訓練時量化）
model = FastLanguageModel.get_peft_model(
    model,
    r=16,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_alpha=32,
    qat_scheme="int4",  # 支援: fp8-int4, fp8-fp8, int8-int4, int4
)

# ... 正常訓練 ...

# 轉換並匯出
from torchao.quantization import quantize_
from torchao.quantization.qat import QATConfig

quantize_(model, QATConfig(step="convert"))
model.save_pretrained_torchao(
    model, "tokenizer",
    torchao_config=model._torchao_config.base_config,
)

QAT 可恢復樸素量化損失的 70% 精度，在 GPQA/MMLU Pro 等基準上提升 1-3%，且推論時無額外開銷。

三、Unsloth Dynamic 2.0 智慧量化

3.1 核心原理

傳統量化對所有層套用相同位元數，Dynamic 2.0 則逐層動態調整量化類型：

  • 敏感層（如 attention）保留較高精度（6-bit）
  • 穩健層（如 feed-forward）降至較低精度（4-bit）
  • 每個模型、每一層的組合都不同

3.2 校準資料集

使用超過 150 萬 token 的高品質、人工策展清洗資料，針對對話效能優化。為避免過擬合，基準測試時使用 Wikipedia 資料集而非校準資料集。

3.3 效能數據

MMLU 5-Shot（Gemma 3 27B）：

  • Unsloth Q4_K_XL：71.47%
  • Google QAT 版本：70.64%
  • Unsloth 在更小檔案（約少 2GB）下達到更高精度

KL 散度改進：Q2_K_XL 變體達到約 7.5% KLD 降低（相比標準 imatrix 量化）。

3.4 支援格式

  • Q4_NL, Q5.1, Q5.0, Q4.1, Q4.0（Apple Silicon / ARM 優化）
  • IQ1_S, IQ1_M, IQ2_XXS, IQ2_M, Q2_K_XL, Q3_K_XL

3.5 企業級應用

DeepSeek-V3.1 壓縮案例：671 GB → 192 GB（壓縮 75%），MoE 層量化到更低位元，注意力層保留 4-6 bit。

四、QAT vs PTQ 選擇指南

維度Post-Training Quantization (PTQ)Quantization-Aware Training (QAT)
訓練成本零（直接量化）需額外訓練時間
精度損失4-bit 以下顯著可恢復 70% 損失
推論開銷無無（轉換後等同 PTQ）
檔案大小與量化位元對應與 PTQ 相同
適用場景快速部署、精度可接受極致品質、生產環境
Unsloth 支援save_pretrained_gguf()qat_scheme 參數

五、常見問題與解法

5.1 Ollama/vLLM 推論品質差

根因排序：

  1. Chat Template 不匹配（最常見）：訓練與推論必須使用相同模板
  2. EOS Token 錯誤：導致長文本生成亂碼
  3. Start Token 多餘：推論引擎重複添加序列起始符號

解法：使用 Unsloth conversational notebooks 強制 chat template。

5.2 GGUF 轉換記憶體不足（OOM）

model.save_pretrained_gguf(
    "output", tokenizer,
    quantization_method="q4_k_m",
    maximum_memory_usage=0.5  # 從預設 0.75 降到 0.5
)

5.3 量化失敗（llama.cpp 編譯問題）

手動編譯 llama.cpp 後重試，確保 CUDA toolkit 版本匹配。

5.4 多 GGUF 匯出注意事項

匯出多種量化格式時，每次只將一行設為 True，避免同時匯出多種格式導致等待過久。

六、部署架構建議

本地開發/測試

微調 → save_pretrained_gguf(q8_0) → Ollama → 測試對話品質

生產環境（API 服務）

微調 → save_pretrained_merged(16bit) → vLLM serve → REST API

邊緣設備/行動端

微調 + QAT(int4) → save_pretrained_torchao → ExecuTorch

社群分享/模型市集

微調 → push_to_hub_gguf(q4_k_m) → Hugging Face Hub
