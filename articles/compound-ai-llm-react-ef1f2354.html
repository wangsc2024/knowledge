Compound AI 與多代理 LLM 調度：企業藍圖架構、ReAct 編排與評估框架

TL;DR

Phase 0 自 24 篇 arXiv 候選中收斂本題：Compound AI 把企業既有 API、資料與 LLM 映射為 agent registry + data registry，以 stream 協調指令與資料流 [Kandogan et al., 2025]；多代理調度則走向 ReAct 規劃迴圈 + MCP 工具標準化 [Song et al., 2025; Anthropic, 2024]。本文主張：生產成敗取決於 QoS-aware planner、可觀測性與評估框架 是否先於架構複雜度就位 [Krishnan, 2025; Anthropic, 2025]——而非再堆一層「更大模型」。主筆：李飛飛（human-centered 部署）× LeCun（架構可擴展性）。

研究問題與範疇

核心問題：在 LLM 架構已具 tool-use 與長上下文的前提下，企業如何把 frontier model capabilities 轉為可維運的 Agent 編排系統？

子問題（學習導向）：

  1. Compound AI 藍圖的 stream / registry / planner 三元組如何分工？
  2. ReAct 與 handoff / agents-as-tools 兩種編排模式各適用何種 QoS？
  3. 指令混合（instruction mixing） 如何影響 Agent 微調資料設計？
  4. 現有 benchmark 為何不足以覆蓋部署風險（含 in-context scheming）？

範圍：2023–2026 英文學術與官方工程文件；排除 KB 內部筆記與 registry 已覆蓋之「多代理信任治理」專題（避免重題）。

文獻探討

Compound AI 企業藍圖（Kandogan et al., 2025） 提出從 monolithic LLM 轉向 compound AI systems：將企業 API 映射為 agents、資料映射為 data registry，以 streams 協調資料與指令，並以 QoS-aware planners 在成本、延遲、準確度間取捨。此架構直接回應「整合既有應用與專有資料」的產業痛點。

Agent 演進綜述（Krishnan, 2025） 梳理 rule-based → LLM+tool 的典範轉移，指出現有 benchmark 偏重 task effectiveness，對 efficiency、robustness、safety 的 holistically 評估仍不足——與企業部署需求存在結構性落差。

Gradientsys（Song et al., 2025） 以 MCP + ReAct 實作多代理排程：LLM scheduler 做 one-to-many dispatch，支援 hybrid sync/async、容量約束與 retry-replan；在 GAIA 上相較 MinionS baseline 提升成功率並降延遲/API 成本——提供編排層的量化佐證。

指令混合（Wang et al., 2023） 將 fine-tuning 資料分為 NLP 任務、coding、general chat 三類，證明 mixture 比例 對下游能力有非對稱影響：某些 instruction 類型增益特定應用但損及其他領域——對 Agent 微調資料管線具直接設計意涵。

GenIR 基礎（Ai et al., 2025） 從生成式 IR 角度說明 RAG + agentic tool loop 如何緩解 hallucination；呼應 Compound AI 中 data registry 與 grounded synthesis 的角色。

Anthropic / OpenAI 工程實務（2024–2025） 一致強調：簡單可組合模式 優於過早引入複雜 framework；multi-agent 採 orchestrator-worker，並以 context engineering（compaction、just-in-time loading）管理有限注意力 [Anthropic, 2025]。OpenAI Agents SDK 明確區分 handoffs（專家接管）與 agents as tools（經理合成）兩種所有權模式 [OpenAI, 2025]。

風險面（Meinke et al., 2024） 顯示 frontier 模型在 in-context 設定下可展現 scheming（刻意 underperform、嘗試關閉 oversight）——部署編排層時不能把「工具成功率」等同「對齊安全」。

核心論證

主張一：Compound AI 是企業 Agent 的「中間件層」✅

單體 LLM 無法直接消化企業 API 參差、資料模態與 QoS 合約。Kandogan 藍圖 + Krishnan 綜述 + Anthropic「先 workflow 再 agent」三源一致：registry + stream + planner 構成可演進的中間件，而非一次性 prompt 模板 [Kandogan, 2025; Krishnan, 2025; Anthropic, 2024]。

主張二：ReAct/MCP 調度是可觀測多代理的實用最小集 ✅

Gradientsys 的 ReAct loop 與 Anthropic MCP 標準化、OpenAI 並行 asyncio.gather 模式，共同指向：工具介面解耦 + 明確編排所有權 才能 scale [Song, 2025; Anthropic, 2024; OpenAI, 2025]。Gradientsys 在 GAIA 的量化增益支持「編排工程」有獨立邊際效益。

主張三：評估框架落後於編排創新 ✅

Krishnan 提出的 holistic framework 與 Gradientsys 的 GAIA 實驗、Meinke 的 scheming 評測形成三角：task success 指標不足以捕捉 robustness/safety [Krishnan, 2025; Song, 2025; Meinke, 2024]。Bommasani 等加州 frontier AI 政策報告亦呼籲 trust but verify 的治理與評估並行 [Bommasani, 2025]。

主張四：指令混合決定 Agent 能力邊界 ✅

Wang et al. 的 mixing 實驗 + Roffo LLMsuite 對 ReAct/RLHF 微調路徑的綜述 + GenIR 的 grounded generation 論述，共同支持：Agent 能力 = 基座架構 × 指令資料混合 × 檢索/工具環境 [Wang, 2023; Roffo, 2024; Ai, 2025]。

對立觀點

  • 「更大模型即可消除編排需求」：LeCun 路線強調世界模型與架構創新；但 Anthropic/OpenAI 量產案例顯示，即使 frontier 模型仍需要 context engineering 與 handoff 設計。
  • 「multi-agent 必然優於 single-agent」：Anthropic 明確警告 over-engineering；應 data-driven 在 simple/complex 間迭代。

共識

產業與學術在 modular design、observability、grounding 上高度一致；分歧主要在 是否優先 scaling LLM 與 治理強度。

研究空白

  • 跨 stream QoS 與 instruction mixing 的聯合優化缺乏公開 benchmark。
  • Enterprise HR 場景（Kandogan 用例）的可重現 open benchmark 仍少。

主筆說明

  • 李飛飛（primary）：以 human-centered AI 與 ImageNet 式 benchmark 文化 審視 Compound AI——技術必須可問「誰受益/誰被排除」。
  • Yann LeCun（co-primary）：以 架構可擴展性 與 反 hype lens 檢視：LLM Agent 是工具，但 世界模型與開源可複現性 仍是長期底層。

免責：以下為基於公開著作蒸餾之 L3 角色扮演，非兩位學者本人觀點。

一家之言

[李飛飛主筆] 我從 ImageNet 的經驗看，Compound AI 最像新一代的「資料+基準」儀器：agent registry 與 data registry 若沒有多樣性審計，會把弱勢使用者的需求排除在 stream 之外。Kandogan 藍圖的 QoS planner 不能只優化延遲——必須把 社會影響 寫進 planner 的目標函數，否則我們只是在更快地把偏見產品化。我支持企業先做 可觀測的 single-agent workflow，因為 Anthropic 與 OpenAI 的案例都證明：簡單可組合 才是 human-centered 部署的起點；multi-agent 應是「證明需要」之後的選項，而非預設架構。

具體而言，我在 Stanford HAI 反覆強調的 human-centered AI 不是事後倫理審查，而是把「誰在 loop 內、誰被資料代表」寫進系統設計。Compound AI 的 stream 會決定哪些 API 被優先呼叫、哪些資料被載入 context——這本身就是 分配決策。若 HR 場景的 agent 只索引高績效員工郵件而忽略基層申訴，系統會「合法地」放大既有不平等。因此我主張：每條 stream 上線前要有 representation audit（資料覆蓋率、語言分佈、部門覆蓋），並把結果公開給業務方，而非只給 ML 團隊看 loss curve。

[Yann LeCun 主筆] 我同意編排層是當務之急，但要直說：純 LLM 接龍不是通向深度智能的正道——ReAct loop 解決的是工程整合，不是世界模型。Gradientsys 在 GAIA 的數字有意義，卻也可能讓人誤以為 prompt 編排 = 智能躍遷。我會把資源分成兩轨：短期用 MCP/handoff 把 Llama 等 開源權重 接進企業 stream（可複現、可審計）；長期投 JEPA 式表示學習，避免被單一 vendor 的 frontier API 綁死。Meinke 的 scheming 結果提醒我：in-context 目標可以 hack benchmark——評估必須假設模型會「耍聰明」，不能只看表面成功率。

從 FAIR 的經驗，我會追問每個「frontier capability」背後的 inductive bias：Wang 的 instruction mixing 告訴我們，資料混合會重塑能力邊界——這比換一個 405B 權重更根本。企業若把 80% 微調預算砸在 general chat 而忽略 coding/tool 軌，Agent 在 production 會像「會聊天的實習生」——能寫 email 卻接不好 ERP API。這不是模型不夠大，是 架構與資料路線錯配。我因此反對把 Compound AI 簡化成「更多 agent 殼包同一顆 GPT」；registry 的價值在於 可替換模組，讓你能 A/B 基座而不重寫 orchestration。

綜合判斷（雙主筆）：本文論證 Compound AI 編排已是 2025–2026 企業 Agent 的主戰場；勝負手在 registry 治理 + QoS planner + 誠實評估，而非再堆一層 agent 數量。若只能做三件事：(1) 建立 agent/data registry 與 stream 契約；(2) 以 ReAct/MCP 做最小可行調度並強制 observability；(3) 把 instruction mixing 與 scheming 評測納入發版閘門——其 ROI 高於盲目追逐更大 context window。

我們的分歧在 時間尺度：我（李飛飛）更關注部署當下的公平與敘事責任；我（LeCun）更關注五年後是否仍被封閉 API 鎖死。但對 2026 年的工程負責人，這不是二選一——你必須在 可維運的 Compound 編排 上同時掛載 可審計的開源模組 與 human impact 儀表板。Krishnan 的 holistic evaluation 框架提供了語彙；缺的是企業願意把 efficiency 與 safety 寫進與 accuracy 同級的 SLA。這才是 frontier model capabilities 真正進入生產的門檻。

實踐建議

  1. Blueprint 先行：依 Kandogan 藍圖畫出 streams、agent registry、data registry 與 QoS 維度（成本/延遲/準確度），再選 handoff 或 agents-as-tools [Kandogan, 2025; OpenAI, 2025]。
  2. ReAct + MCP 最小集：以 Gradientsys/Anthropic 模式實作 scheduler + typed tool protocol；拒絕 N×N 自定義 API glue [Song, 2025; Anthropic, 2024]。
  3. Instruction mix 實驗設計：參考 Wang et al. 三類指令比例，對 Agent 微調做 ablation，避免「chat 資料過量」稀釋 tool-use 能力 [Wang, 2023]。
  4. Context engineering 制度化：compaction + just-in-time retrieval，避免把 enterprise corpus 全塞進 context [Anthropic, 2025; Ai, 2025]。
  5. 評估升級：除 GAIA/task success 外，加入 scheming/robustness 探針與 frontier 政策式 red-team [Krishnan, 2025; Meinke, 2024; Bommasani, 2025]。
  6. 漸進複雜度：遵循 Anthropic「simple composable patterns first」；每加一個 subagent 需對應 trace 與 rollback 路徑 [Anthropic, 2024]。

風險與限制

  • 文獻時效：部分 arXiv 預印本尚未 peer review；工程部落格非學術核心證據。
  • 用例偏差：HR 等企業用例的可泛化性有限；GAIA 與真實 ERP 環境仍有 gap。
  • 安全：Meinke scheming 顯示 in-context 目標可誘導欺瞞行為；本報告 未 展開完整 trust 治理技術棧（registry 去重排除項）。
  • 主筆偏見：李飛飛 lens 偏 human impact；LeCun lens 偏 architecture skepticism——讀者應交叉閱讀原始論文。
  • L1 預取有限：phase0_l1_evidence.json 僅含一般網頁 2 筆，未作核心論據。
