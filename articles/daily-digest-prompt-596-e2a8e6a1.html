Daily-Digest-Prompt 知識庫知識熵增治理與快取系統實效性診斷

日期：2026-02-28 | 任務：運用 RAG 知識庫檢視專案並提出優化建議
研究角度：知識品質治理 + 快取實效性（區別於前三篇的狀態管理、排程效能、Token 經濟學）

核心問題：每日約 25 筆自動匯入，但缺乏「知識蒸餾」機制。以 research-registry.json 最近 7 天 90+ entries 為例，雖然去重機制（3 天冷卻 + KB 語義搜尋）防止了完全重複，但同類主題的「角度微調」仍產生高度相似的知識塊：

  • 「Unsloth 研究」：4 篇覆蓋框架比較、部署管線、核心架構、GRPO 訓練，有重疊的基礎知識

二、問題根因深層分析

2.1 知識熵增的根本驅動力

當前系統的研究去重機制（dedup-policy.yaml）設計目標是「避免完全重複」，但未解決「知識碎片化」：

  1. 去重粒度過粗：topic_cooldown_days=3 只防止 3 天內完全相同 topic，但「楞嚴經卷三」vs「楞嚴經卷四」被視為不同 topic
  2. 缺乏知識合併：8 篇楞嚴經研究各自獨立，沒有「系列總覽」或「進度追蹤」的彙整機制
  3. 匯入品質無閘門：只要 hybrid search score < 0.85 即可匯入，缺乏「增量價值」評估
  4. 標籤爆炸：1955 個標籤（平均 3.3 個/筆記），但缺乏標籤正規化（如「AI」vs「人工智慧」、「fine-tuning」vs「微調」）

2.2 快取零命中的連鎖效應

快取零命中 → 每次 API 呼叫都是新請求 → digest-memory.json 記錄 cache_hits=0 → 品質閘門驗證時無法判斷快取是否正常 → 靜默退化。

關鍵設計缺陷：快取命中追蹤僅在 digest-memory.json 的 skill_usage 區塊記錄計數器，但沒有告警機制。連續 4 天零命中無人察覺，直到本次深度分析才發現。

三、具體優化建議

建議 1：KB 知識蒸餾機制（優先級：P1）

目標：定期將碎片化知識合併為結構化系列，降低知識密度

方案：

  • 新增「知識蒸餾」自動任務（建議 weekly，可加入 system-insight 群組）
  • 觸發條件：同 tag 下筆記數 ≥ 5 篇且無「系列總覽」
  • 行為：讀取同 tag 所有筆記 → 生成「系列總覽」→ 在各筆記加上 series_id tag → 匯入總覽
  • 佛學系列優先：楞嚴經 10+ 篇 → 「楞嚴經全卷研究導覽」；法華經、教觀綱宗同理
  • 技術系列次之：Unsloth 4 篇 → 「Unsloth 從入門到部署完整學習路徑」

預期效果：

  • 知識可發現性提升（搜尋一篇總覽即可獲得全脈絡）
  • 新研究選題更精確（看到已有什麼、缺什麼）
  • RAG 查詢品質提升（總覽文件的語義覆蓋更廣）

建議 2：快取系統健康監測與自動修復（優先級：P1）

目標：將快取零命中從靜默退化升級為可察覺問題

方案：

  1. on_stop_alert.py 增強：分析 digest-memory.json 的 cache_hits，連續 2+ 天零命中時發送 warning 通知
  2. self-heal 增強：自愈任務掃描 cache/ 目錄，清理過期超過 degradation_max_age_hours 的快取檔案，以及非快取檔案（不在 cache-policy.yaml sources 中的檔案）
  3. cache 寫入驗證：每個 Phase 1 Agent 完成時，用 Read 驗證快取檔案已更新（時間戳 ≤ 5 分鐘前），失敗則在 Phase 2 日誌中標記
  4. 快取目錄即時清理：在 run-agent-team.ps1 Phase 2 完成後，掃描 cache/ 並清理不在 policy 中的檔案

預期效果：

  • 快取問題在 48 小時內被察覺（不再靜默 10+ 天）
  • cache/ 目錄保持乾淨（自動清理 2.86MB 垃圾）
  • API 呼叫次數降低（恢復快取命中後，每日減少 ~10 次 API 呼叫）

建議 3：標籤正規化與知識分類治理（優先級：P2）

目標：解決 1955 個標籤的碎片化問題，提升搜尋精確度

方案：

  1. 標籤映射表：config/tag-normalization.yaml

- 同義詞合併：AI = 人工智慧 = Artificial Intelligence

- 層級關係：Unsloth → parent: LLM fine-tuning → parent: AI

- 棄用標籤：識別使用次數 ≤ 1 的標籤，標記為待清理

  1. 匯入時正規化：research-task.md 和 auto-task 模板在匯入前查詢映射表
  2. 定期審計：skill-audit 自動任務新增「標籤健康度」檢查項

預期效果：

  • 有效標籤數從 1955 → ~500（預估 75% 可合併或清理）
  • 同義搜尋準確率提升
  • 匯入時標籤一致性強制保證

建議 4：匯入品質閘門強化（優先級：P2）

目標：從「防重複」升級到「評估增量價值」

方案：

  1. 增量價值評估：匯入前用 hybrid search 取得 top-3 相似筆記，若 score > 0.7 則要求新筆記必須有 ≥ 30% 的獨有內容（非摘要或改寫）
  2. 最小內容長度：contentText < 300 字的筆記標記為 draft，不計入正式知識庫統計
  3. 系列筆記自動連結：同 tag 同週匯入的筆記自動加上 series: -week 標籤
  4. 匯入限速：每日單一自動任務最多匯入 3 筆（已有 max_auto_per_run，但匯入數量無上限）

建議 5：Research Registry 生命週期優化（優先級：P3）

目標：registry 從「記錄」升級到「知識圖譜的索引」

方案：

  1. 結構增強：entry 新增 kb_note_id 欄位（匯入後回填 noteId），建立 registry ↔ KB 雙向連結
  2. 歸檔機制：超過 retention_days 的 entries 不直接刪除，移到 context/research-archive.json（月度歸檔）
  3. 趨勢分析：每週統計 task_type 分佈，識別過度集中或不足的研究方向
  4. 匯入狀態追蹤：kb_imported=false 的 entries 加入 retry_count，超過 3 次則標記放棄原因

七、結論

本專案在兩週內從 339 → 596 筆知識庫筆記，展現了強大的知識捕獲能力。然而，知識的「量」已超越「質」的管理能力 — 96% 為自動匯入、1955 個未正規化標籤、碎片化的系列研究缺乏彙整。同時，快取系統的靜默失效暴露了可觀測性的盲點。

建議優先實施「快取健康監測」（P1，2-3 天可完成）和「知識蒸餾機制」（P1，需要更深入的設計），前者解決即時的效能浪費，後者開啟知識從「量」到「質」的轉化之路。
