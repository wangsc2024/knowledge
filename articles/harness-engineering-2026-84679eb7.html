Harness Engineering 深化優化方案：從理論到實踐（2026）

核心公式與定義

Agent = Model + Harness（Mitchell Hashimoto, 2026）

Harness Engineering 是一門專注於設計約束、工具、反饋迴路、文件與驗證系統的學科，目標是引導不可預測的 AI Agent 產出可靠、可維護、可擴展的軟體輸出。

關鍵洞察：88% 的 AI Agent 專案無法進入生產環境，瓶頸不在模型能力，而在缺乏生產級 Harness。

四大核心組件

1. 架構約束與邊界（Architectural Constraints）

定義：機器可讀的規範（schemas、ADR、linter rules）強制執行模組分離與領域規則，同時在集中化框架內保持局部自主性。

實作要點：

  • 使用 AGENTS.md / CLAUDE.md 編碼架構決策
  • Linter 配置為 error（非 warning），零容忍違規
  • 類型系統縮小有效輸出空間
  • 約束應機器可執行，不依賴人工審查

2. 反饋迴路與可觀測性（Feedback Loops）

定義：閉環系統追蹤失敗、聚類錯誤模式，並將修正回饋到 Harness 中——無需修改模型本身。

效能證據：LangChain 實證透過系統化追蹤將成功率從 52.8% 提升至 66.5%。

設計原則：

  • 錯誤訊息必須是可執行提示，而非二元 pass/fail
  • 範例：不輸出「違規偵測」，而是「使用 logger.info({event: 'name', ...data}) 取代 console.log」
  • 啟用 Agent 自我修正，減少人工介入

3. 驗證與防護機制（Verification & Guardrails）

混合驗證策略：

  • 確定性工具（linter、type checker、schema validator）
  • AI 驅動驗證（語義合規性、架構一致性檢查）
  • 定期「垃圾回收」防止熵累積

三層控制架構（實施順序）：

  1. 預防控制（Preventive）：約束 Harness 在生成前縮小解空間
  2. 修正控制（Corrective）：反饋迴路傳回結構化錯誤信號
  3. 強制控制（Enforcement）：品質門檻在 CI 層攔截不合規程式碼

4. 生命週期管理與腳手架（Lifecycle Management）

標準化基礎設施：

  • 標準化 CI/CD 流程
  • 文件自動生成
  • 可觀測性儀表板
  • 從 CLI scaffold 演進至宣告式 workflow prompts

深化優化方案（2026-04 建議）

優化 A：引入 Plan-Execute-Verify (PEV) 模式

現狀缺口：部分自動任務直接執行，缺乏顯式計畫分解與執行前驗證。

建議實施：

  1. 在 templates/auto-tasks/*.md 插入 PEV 三階段：

- Plan：明確分解步驟 + 驗收條件（AC）

- Execute：執行前閘門驗證工具呼叫與參數

- Verify：後驗證對齊規格，捕捉測試遺漏的架構違規

  1. 在 config/benchmark.yaml 新增 PEV 合規度指標：

```yaml

pev_compliance:

threshold: 0.85 # 85% 任務需通過 PEV 檢查點

measure: "plan_explicit_count / total_tasks"

```

預期收益：降低 15-20% 的任務失敗率（特別是複雜研究類任務）。

優化 B：熵監控與定期重構 Agent

現狀缺口：長期運行專案未建立熵累積監控機制。

建議實施：

  1. 新建 tools/entropy_monitor.py：

- 追蹤配置檔案行數、ADR 數量、Hook 規則複雜度

- 計算熵分數：entropy_score = (config_lines / baseline) × (adr_count / baseline) × (hook_complexity / baseline)

- 閾值：entropy_score > 1.5 觸發告警

  1. 排程「重構 Agent」（每週執行）：

- 自動識別冗餘 ADR、過時配置

- 提出合併/刪除建議

- 生成重構 PR

  1. 整合至 check-health.ps1 新增 [系統熵分數] 區塊。

預期收益：維持長期可維護性，防止配置膨脹（目前 analyze-config.ps1 已有基礎，需擴展為主動重構）。

優化 C：適應性函數（Fitness Functions）擴展

現狀缺口：benchmark.yaml 的 fitness_functions 主要是靜態閾值，缺乏自適應能力。

建議實施：

  1. 引入 動態閾值調整：

```python

# tools/adaptive_fitness.py

def adjust_threshold(metric_name, historical_data, target_percentile=0.9):

"""基於歷史數據自動調整 fitness function 閾值"""

baseline = np.percentile(historical_data, target_percentile)

return baseline * 1.1 # 設定為 P90 + 10% buffer

```

  1. 擴展評估維度（參考 2026 研究）：

- 任務解決率（task_resolution_rate）

- 程式碼流失率（code_churn_rate）：修改後被回滾的比例

- 驗證稅（verification_tax）：人工審查時間抵銷

- 缺陷逃逸率（defect_escape_rate）：進入生產的 bug 數

- Pass@1 率（first_attempt_success）：首次嘗試成功率

  1. 在 state/fitness-history.jsonl 記錄歷史數據，每月自動調整閾值。

預期收益：Fitness Functions 從靜態規則演進為自適應系統，隨專案成熟度提高標準。

優化 D：結構化輸入強制（Structure In, Structure Out）

現狀缺口：部分 prompt 允許模糊輸入，導致輸出不穩定。

建議實施：

  1. 在 templates/shared/preamble.md 新增「輸入驗證」段落：

- 所有任務必須明確 Repository Scope（影響範圍）

- 必須明確 Specific File Paths（不可猜測）

- 必須引用 Actual Symbol Names（函數、類別名稱）

  1. 新建 hooks/pre_task_input_validator.py：

- 檢查輸入是否含糊（如「優化系統」而無具體範圍）

- 強制要求補充具體路徑與符號

- 不合規任務直接拒絕

  1. 在 Phase 1（todoist-query）加入「輸入具體化」步驟，將模糊任務轉換為結構化規範。

預期收益：減少幻覺生成，提高輸出一致性（Red Hat 實證：結構化輸入可降低 30-40% 的錯誤率）。

優化 E：反饋迴路可執行化

現狀缺口：Hook 錯誤訊息部分為描述性（"violation detected"），缺乏可執行指引。

建議實施：

  1. 改寫 hooks/*.py 的錯誤訊息模板：

```python

# 舊版（描述性）

raise PreventHookError("nul 檔案禁令違規")

# 新版（可執行）

raise PreventHookError(

"禁止使用 '> nul'。請改用 '> /dev/null 2>&1'。\n"

"範例：curl ... > /dev/null 2>&1"

)

```

  1. 在 config/hook-rules.yaml 為每條規則新增 fix_template 欄位：

```yaml

- name: nul_prohibition

pattern: "> nul"

fix_template: "將 '{match}' 替換為 '> /dev/null 2>&1'"

```

  1. Hook 失敗時自動提供修正建議，啟用 Agent 自我修正。

預期收益：減少 50% 的重複錯誤（Agent 從錯誤訊息直接學習修正方式）。

優化 F：Two-Phase Workflow 導入（Impact Map → Structured Task）

現狀缺口：複雜任務直接執行，未建立影響範圍地圖（Impact Map）。

建議實施：

  1. 在 Phase 1（todoist-query）新增「Impact Analysis」子步驟：

- 使用 LSP/MCP 分析受影響檔案與模組

- 識別依賴關係與副作用

- 生成 3-5 行 Impact Map

  1. 在 Phase 2 前插入「Impact Map Review」檢查點：

- 若影響範圍 > 5 個檔案 → 要求人工確認

- 若涉及核心模組（ADR、Hook、Skills）→ 自動標記高風險

  1. Impact Map 儲存於 state/impact-maps/{task_id}.json，供後續審計追蹤。

預期收益：在 3 行地圖中捕捉錯誤，成本遠低於在 PR 中發現（Red Hat 實證）。

總結

Harness Engineering 不是一次性專案，而是持續演化的紀律。本優化方案基於 2026 年最新研究與本專案已有基礎（B0-B3 批次、1801 測試），提出六大深化方向（PEV 模式、熵監控、適應性函數、結構化輸入、可執行反饋、Two-Phase Workflow），預期將任務成功率從 75% 提升至 90% 以上，實現從「能用」到「生產級可靠」的質變。

關鍵原則重申：

  • Agent = Model + Harness：改善環境優先於期待更強模型
  • Structure In, Structure Out：結構化輸入保證結構化輸出
  • 預防 > 修正 > 強制：三層控制按順序實施
  • 可執行反饋：錯誤訊息即修正指引
  • 持續監控：熵分數與適應性函數防止系統退化
