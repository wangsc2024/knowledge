技術概述

混合搜尋的核心不是把全文搜尋與向量搜尋「都打開」而已，而是要把兩種檢索訊號放在正確的順序裡：先用 FTS5/BM25 抓住關鍵詞精確性，再用向量搜尋補足語意召回，之後做文件級去重與候選集合裁剪，最後才交給 reranker 重排。2026 年的主流實務已從「線性加權分數」轉向更穩健的 RRF、relative score fusion 與兩階段/三階段檢索。對 daily-digest-prompt 這類筆記量持續成長、查詢類型混合精確詞與語意意圖的系統而言，關鍵不是追求單次最強模型，而是建立一條可調參、可觀測、可去重的檢索管線。

核心概念與原理

1. FTS5 / BM25：負責精確詞與結構化訊號

SQLite FTS5 以倒排索引做全文檢索，內建 bm25() 排序。官方文件指出 FTS5 的 BM25 分數越小代表越相關，常見用法是 ORDER BY bm25(ft)。這一層最擅長處理：

  • 專有名詞：如 OpenClaw、Todoist API v1、Vite
  • 檔名、函式名、錯誤字串
  • 查詢中具強約束的精確詞

若 query 明顯有精確詞，應保留一條 lexical 通道，不要只靠 embedding。

2. 向量搜尋：補足語意召回

向量搜尋擅長找語意相近但表面詞不同的內容，例如「知識庫同步」與「build-time import pipeline」。它能補 lexical recall 的盲點，但若單獨使用，常會把語意相近卻不含關鍵詞的舊筆記一起拉進來，增加重複與誤召回。

3. 融合不是線性加權，而是穩健合併排名

Azure AI Search 目前將 hybrid query 定義為文字查詢與 vector query 並行執行，再用 RRF 合併結果；官方也明確把 maxTextRecallSize 作為新的可調參數，讓 BM25 側能決定要餵多少候選到融合器。Qdrant 也直接指出，不應草率用 0.7 vector + 0.3 bm25 這種線性組合，因為兩種分數空間通常不可線性分離。

實務上可把融合策略理解成兩類：

  • RRF：看排名位置，不強依賴不同分數尺度可比性，穩健耐用。
  • Relative Score Fusion：先正規化再融合，保留更多原始分數資訊。Weaviate 已在 v1.24 把 relativeScoreFusion 設為預設。

4. 去重應放在 rerank 之前，且以文件層級處理

若同一篇筆記被切成多個 chunk，同一文件可能同時從 BM25 與向量搜尋多次命中。若不先做文件級去重，reranker 會把算力浪費在同文不同 chunk 上，結果也容易被重複片段洗版。

建議流程：

  1. 先分別取得 lexical topN 與 vector topN
  2. 以 noteId 為主鍵、title + chunkHash 為備援做去重
  3. 每篇文件只保留最強 chunk 與必要 metadata
  4. 再把 20–50 筆候選送進 reranker

5. Reranking：最後一哩，不是第一層檢索

Qdrant 與 Cohere 的文件都強調 reranker 應用在「小候選集」而非全庫。Cohere 目前的 Rerank API v2 建議單次文件數不要超過 1,000，並可用 max_tokens_per_doc 控制長文截斷；新一代可用模型包含 rerank-v4.0-pro 與 rerank-v4.0-fast。對本專案來說，更合適的做法是把 rerank 放在查詢最後，只重排 20–40 個候選。

2026 可觀察到的實務進展

  1. Azure AI Search：到 2026-01-20 更新的文件，hybrid query 已把 maxTextRecallSize 納入 2025-11-01-preview 參數，用來細調 BM25 召回量；另可搭配 vector weighting 與 semantic ranking。這代表業界已從「能不能 hybrid」進化到「如何控制兩側召回貢獻」。
  2. Weaviate：文件顯示 relativeScoreFusion 自 v1.24 成為預設，代表保留分數資訊的融合策略逐漸成熟，特別適合分數分布仍有辨識度的場景。
  3. Qdrant：官方教學已把 dense + sparse + late interaction（ColBERT）視為標準多階段管線，並以 prefetch 實作先召回再 rerank。這對需要兼顧精確詞與語意理解的 RAG 系統很有啟發。
  4. Cohere：目前文件顯示 rerank-v4.0-pro / fast 已成為最新 rerank 模型族，v2 API 也明確把 model、top_n、max_tokens_per_doc 規格化，代表 reranking 已從「實驗性加分項」變成標準後處理元件。

最佳實踐與常見陷阱

最佳實踐

  • 先粗召回，再精重排：FTS5 與向量搜尋各取前 20–50 筆，去重後再 rerank。
  • 用排名融合，少碰硬加權：預設先用 RRF；若觀測到分數分布穩定，再試 relative score fusion。
  • 把去重鍵設計清楚：優先 noteId，次選 source + canonical_title + normalized_chunk_text_hash。
  • 分開觀測三段指標：lexical 命中率、vector 命中率、rerank 後 MRR / nDCG。
  • 對中文資料做正規化：標題、標點、全半形、英數大小寫、空白要先清洗，否則 chunk 級重複難以收斂。

常見陷阱

  • 把 BM25 分數直接拿去和 cosine similarity 線性相加：尺度不同，容易造成排序漂移。
  • 未去重就直接 rerank：浪費延遲與成本，還會讓同一文件多次霸榜。
  • rerank 候選太大：結果更慢、成本更高，且收益遞減。
  • topK 只看最終返回數：真正重要的是各階段候選大小，例如 lexical 30、vector 30、dedup 後 25、rerank 回 8。
  • 忽略 query 類型差異：像 OpenClaw 這類品牌詞需要 lexical 偏重；像「知識保留管線」則更需要語意側。

與本專案的關聯

daily-digest-prompt 目前已明顯同時依賴：

  • Todoist 任務文字作為查詢輸入
  • 本地知識庫作為研究去重與補充依據
  • 匯入後的筆記再被後續任務二次查詢

因此最值得優化的不是單次搜尋命中，而是「研究前去重 + 研究後可再檢索」的閉環：

  1. 研究前：先用 FTS5 抓精確主題詞（如 Vite、OpenClaw），再用向量搜尋擴展相近主題。
  2. 融合後：以 noteId 與標題正規化做文件級去重，避免同筆筆記多 chunk 干擾。
  3. 匯入前：若最高相似結果 > 0.85，視為高度重複；若介於 0.4–0.85，改走「承接前文、補機制/應用/優化」策略。
  4. 匯入後：將 tags、title、canonical topic 一起寫好，提升下一輪 lexical recall。

這樣能直接支援本專案最常見的兩種任務：

  • 「今天要不要換研究主題？」
  • 「這篇新筆記之後還找不找得到？」

程式碼範例


from collections import defaultdict

def hybrid_retrieve(query, lexical_hits, vector_hits, reranker, top_n=8):
    bucket = defaultdict(list)

    for hit in lexical_hits:
        bucket[hit["noteId"]].append({**hit, "channel": "lexical"})
    for hit in vector_hits:
        bucket[hit["noteId"]].append({**hit, "channel": "vector"})

    candidates = []
    for note_id, hits in bucket.items():
        best = sorted(hits, key=lambda item: item["score"], reverse=True)[0]
        candidates.append({
            "noteId": note_id,
            "title": best["title"],
            "content": best["contentText"],
            "channels": sorted({h["channel"] for h in hits}),
            "max_score": best["score"],
        })

    rerank_input = [f"{c['title']}\n\n{c['content'][:1200]}" for c in candidates[:30]]
    reranked = reranker(query=query, documents=rerank_input, top_n=top_n)

    final_results = []
    for item in reranked:
        final_results.append(candidates[item["index"]])
    return final_results

實作建議參數（給 daily-digest-prompt）

  • lexical 召回：topK = 25
  • vector 召回：topK = 25
  • 去重後候選：20–30
  • rerank 輸出：6–10
  • 高重複門檻：score >= 0.85
  • 可承接深化區間：0.40 <= score < 0.85
