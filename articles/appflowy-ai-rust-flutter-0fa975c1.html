二、核心技術架構分析：Rust+Flutter 雙引擎

AppFlowy 的架構決策是整個專案最值得研究的部分 — 它選擇了「Flutter 負責 UI、Rust 負責核心邏輯」的雙引擎模式，這在開源界是極少見的組合。

2.1 三層架構


┌─────────────────────────────────────┐
│  Presentation Layer (Flutter/Dart)  │  BLoC 狀態管理 + Widget 渲染
├─────────────────────────────────────┤
│  FFI Bridge Layer (Protobuf)        │  Event-Based RPC + 通知訂閱
├─────────────────────────────────────┤
│  Business Logic Layer (Rust)        │  5 大 Manager + CRDT 引擎
└─────────────────────────────────────┘
  1. Presentation Layer（Flutter/Dart）：所有 UI 渲染使用 Flutter Widget，狀態管理採用 BLoC（Business Logic Component）模式，i18n 用 easy_localization
  2. FFI Bridge Layer：雙向通訊使用 Event-Based RPC + Protobuf 序列化，Dart FFI 綁定 Rust 實作
  3. Business Logic Layer（Rust）：5 個核心 Manager（User、Folder、Database、Document、AI）透過 AFPluginDispatcher 路由事件

2.2 為何選擇 Rust + Flutter?

這個架構決策揭示了深思熟慮的工程取捨：

| 決策 | 選擇 | 理由 |

|------|------|------|

| UI 框架 | Flutter | 單一程式碼庫支援 6 個平台（含 Web），原生渲染效能 |

| 核心引擎 | Rust | 記憶體安全、零成本抽象、適合 CRDT 高頻操作 |

| FFI 序列化 | Protobuf | 強型別、高效二進位編碼、語言中立 |

| 狀態管理 | BLoC | 可測試性高、事件驅動、與 Flutter 生態整合良好 |

關鍵洞察：Rust 不僅提供效能優勢，更重要的是它的 ownership 模型天然適合 CRDT 的 immutable update 語義。Flutter 的 Hot Reload 則大幅加速 UI 迭代。這種「各取所長」的混合架構，犧牲了一定的開發複雜度（FFI 邊界維護成本），換取了長期可維護性和效能保證。

2.3 Event Dispatch 系統

AppFlowy 的 FFI 通訊模式值得特別注意：

  • async_event() / sync_event()：前端發起的請求-回應模式
  • observe_notification()：後端主動推送通知，實現響應式 UI 更新
  • AFPluginDispatcher：中央事件路由器，每個 Manager 註冊自己的事件處理器
  • 所有事件均為強型別（UserEvent、FolderEvent、DatabaseEvent、AIEvent）

這套系統的精妙之處在於：前端完全不需要知道 Rust 的實作細節，只需發送型別安全的事件並監聽通知。這是一種語言無關的微服務通訊模式，只不過是跨語言而非跨進程。

三、CRDT 即時協作引擎

3.1 Yrs CRDT 核心

AppFlowy 的協作系統建立在 Yrs（Yjs 的 Rust 實作）之上，這是目前最成熟的 CRDT 函式庫之一：

  • 文件編輯：文字操作自動合併，無需伺服器仲裁
  • 資料庫儲存格：Last-Write-Wins（時間戳排序）
  • 資料夾結構：Tree CRDT 確保層級一致性
  • 列排序：Position-based CRDT 跨用戶端維持順序

3.2 三層插件架構


AppFlowyCollabBuilder（工廠）
  ├── Disk Plugin       → 即時本地持久化（RocksDB/IndexedDB）
  ├── Sync Plugin       → WebSocket 即時雲端同步
  └── Awareness Plugin  → 用戶存在感知（游標位置、線上狀態）

同步流程：

  1. 本地變更觸發 Yrs transaction
  2. SyncPlugin.after_transaction() 編碼更新
  3. 更新在本地佇列中排隊，已連線時即時傳送
  4. 伺服器合併更新並廣播給其他用戶端
  5. 遠端更新透過 MessageHandler 應用

3.3 離線優先（Offline-First）策略

| 策略 | 實作方式 |

|------|----------|

| 即時本地持久化 | 所有變更立即寫入 CollabKVDB |

| 完整離線功能 | 無雲端連線也能完整使用 |

| 佇列同步 | 離線變更在重連後自動同步 |

| CRDT 合併 | 重連時自動衝突解決 |

| 斷線重試 | 指數退避策略（exponential backoff） |

四、雙重儲存策略

AppFlowy 的儲存架構展現了對不同資料特性的深刻理解：

| 儲存層 | 技術 | 用途 | 特性 |

|--------|------|------|------|

| SQLite | Diesel ORM | 結構化中繼資料（用戶、工作空間、聊天記錄） | ACID 交易、高效查詢 |

| CollabKVDB | RocksDB(桌面) / IndexedDB(Web) | CRDT 編碼的協作文件 | 快速衝突解決、離線優先 |

| KVStore | 鍵值存儲 | 用戶偏好設定 | 輕量、快速讀寫 |

設計考量：SQLite 擅長關聯式查詢和 ACID 保證，但不適合高頻的 CRDT 更新操作。RocksDB 的 LSM Tree 架構天然適合寫入密集型場景，且支援前綴掃描（prefix scan）便於按文件 ID 檢索。

五、Local-First AI 整合

AppFlowy 的 AI 策略是當前「隱私優先 AI」趨勢的最佳實踐案例：

5.1 雙軌 AI 路由

  • 本地 AI：整合 Ollama，支援 Mistral 7B、Llama 3、DeepSeek R1、Gemma 3、Phi4 等模型
  • 雲端 AI：可選接入雲端 LLM 服務
  • AIManager 負責路由決策：依用戶設定自動切換本地/雲端模式

5.2 隱私保證

  • 本地 AI 模式下資料零外傳 — 所有推理在用戶裝置完成
  • 低延遲 — 無需網路往返
  • 離線可用 — 無需網路也能使用 AI 功能

六、可借鏡的技術項目

6.1 Event-Based RPC 跨語言通訊模式

借鏡價值：AppFlowy 的 Protobuf + Event Dispatch 模式可應用於任何需要跨語言互操作的系統。對 daily-digest-prompt 而言，若未來 Hooks 需要 Python↔PowerShell 更結構化的通訊，可參考此模式建立型別安全的事件系統。

實踐要點：

  • 定義事件 Schema（Protobuf / JSON Schema）
  • 建立中央路由器（Dispatcher）
  • 實作通知訂閱（觀察者模式）

6.2 CRDT Offline-First 資料同步

借鏡價值：AppFlowy 的三層插件架構（Disk + Sync + Awareness）提供了可複用的離線優先同步模式。對知識庫系統（localhost:3000）而言，若需支援多裝置同步，Yrs CRDT 是經過生產驗證的選擇。

實踐要點：

  • 本地變更立即持久化（不依賴網路）
  • 同步佇列 + 指數退避重試
  • CRDT 合併語義消除衝突仲裁需求

6.3 雙重儲存策略（結構化 + 非結構化分離）

借鏡價值：SQLite（中繼資料）+ KV Store（文件內容）的分離模式，啟發了如何設計混合儲存架構。daily-digest-prompt 目前使用 JSON 檔案存儲狀態，若規模成長可考慮 SQLite 存結構化資料 + 檔案系統存 CRDT 文件。

6.4 Plugin-Based Manager 初始化模式

借鏡價值：AppFlowy 的 5 個核心 Manager 透過 AFPluginDispatcher 註冊事件處理器，Manager 之間解耦。這與 daily-digest-prompt 的 Skill 系統設計理念一致 — 每個 Skill 是獨立的處理單元，透過 SKILL_INDEX 路由。可強化現有 Skill 系統的事件機制。

6.5 Local-First AI 整合模式

借鏡價值：AppFlowy 的 AIManager 雙軌路由（本地 Ollama / 雲端 LLM）為 daily-digest-prompt 提供了 AI 整合參考。未來若需離線 AI 能力（如本地摘要生成），可參考此架構實作 Ollama 整合層。

6.6 Domain-Driven Design 前端組織

借鏡價值：AppFlowy 的 Flutter 前端嚴格遵循 DDD 四層架構（Presentation → Application → Domain → Infrastructure），每層有明確的依賴方向。這種分層紀律適用於任何需要長期維護的前端專案。

八、架構啟示與反思

8.1 雙語言架構的取捨

AppFlowy 證明了「正確的語言用在正確的地方」比「一種語言走天下」更有價值。Rust 的記憶體安全保證消除了 CRDT 引擎中最容易出錯的部分，而 Flutter 的跨平台能力避免了為 6 個平台維護 6 套 UI。FFI 橋接的維護成本是真實的代價，但被長期收益抵消。

8.2 CRDT vs OT 的選擇

AppFlowy 選擇 CRDT 而非 OT（Operational Transformation），這意味著放棄了中央伺服器仲裁的簡潔性，換取了真正的離線優先能力和分散式架構的可能性。對於開源專案而言，這是更明智的選擇 — 用戶可以完全不依賴中央伺服器運作。

8.3 AGPL-3.0 授權的策略考量

AGPL-3.0 是最嚴格的開源授權之一 — 任何修改後的程式碼（即使僅在伺服器端運行）都必須開源。這保護了 AppFlowy 社群生態，同時透過商業版（Cloud 方案）實現永續營收。

---

九、總結

AppFlowy 不僅是一個生產力工具，更是現代開源軟體工程的典範。它展示了：

  1. 雙語言架構可以在工程複雜度和長期維護性之間取得最佳平衡
  2. CRDT + Offline-First 是資料主權時代的正確技術選擇
  3. Local-First AI 證明了隱私和智能並非互斥
  4. Plugin 架構 + Event Dispatch 提供了可擴展性的工業級實踐
  5. Domain-Driven Design 在 Flutter 前端中依然適用且有效

對於任何構建知識管理、協作工具或 AI 工作空間的團隊，AppFlowy 的架構決策都值得深入研究和借鏡。
