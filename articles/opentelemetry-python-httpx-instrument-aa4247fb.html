OpenTelemetry Python 官方 httpx instrument 與 HTTP Semantic Conventions 雙軌稽核（Agent 熱路徑）

技術概述

本主題銜接既有「httpx Event Hooks + AsyncClient」研究，改從標準化分散式追蹤視角說明：當 tools/ 與排程 Agent 以 httpx 呼叫 localhost:3000（KB）與 ntfy 時，除了自訂 event_hooks 寫 JSONL，尚可並行採用 opentelemetry-instrumentation-httpx 產出符合 HTTP Semantic Conventions 的 client span，使延遲、重試、錯誤類型與後端 Tempo／Azure Monitor 等可對齊。與純 httpx hooks 的差異在於：OTel 關心 span 生命週期、屬性命名空間、取樣與與 log 的 trace 關聯；hooks 則擅長應用層稽核（例如 leaf_hash、payload 大小）。兩者可分工：transport 層交給官方 instrument，業務欄位留在 JSONL 或 LoggingHandler 注入的 trace_id [來源:OpenTelemetry HTTPX Instrumentation,2026][來源:HTTP semantic conventions declared stable,2023][來源:Semantic conventions for HTTP spans,2026]。

核心概念與原理

1. Instrumentor 與 Transport 兩條路徑

官方文件提供 全域 HTTPXClientInstrumentor().instrument() 與 單例 instrument_client(client)，以及不透過 monkey-patch、改以 SyncOpenTelemetryTransport／AsyncOpenTelemetryTransport 包一層底層 HTTPTransport／AsyncHTTPTransport 的模式 [來源:OpenTelemetry HTTPX Instrumentation,2026]。原理是：在 httpx 送出請求前建立 SpanKind.CLIENT 的 span，於收到標頭或錯誤時結束；request_hook／response_hook（及 async 變體）在「span 已建立後」與「span 將結束前」回呼，參數是 transport 層的 RequestInfo／ResponseInfo，而非高階 httpx.Request 物件——與 httpx 自己的 event_hooks 觸發點不同，兩者並用時須清楚邊界以免重複讀 body [來源:OpenTelemetry HTTPX Instrumentation,2026]。

2. HTTP Semantic Conventions 穩定版與遷移開關

OpenTelemetry 於 2023 將 HTTP semconv 標為 stable（v1.23.0）：http.method→http.request.method、http.status_code→http.response.status_code、http.url→url.full、net.peer.→server.／client. 與 network. 分離；並以環境變數 OTEL_SEMCONV_STABILITY_OPT_IN 控制僅發新屬性（http）、新舊並發（http/dup）或維持舊版預設，以利儀表板與告警規則漸進遷移 [來源:HTTP semantic conventions declared stable,2023][來源:Semantic conventions for HTTP spans,2026]。對 client span，規格要求 http.request.method、server.address、server.port、url.full 等欄位層級，並定義 重試／redirect 時 http.request.resend_count 的語意：每次實際重送遞增，便於區分「單次 API 呼叫」與「多次 on-wire 嘗試」[來源:Semantic conventions for HTTP spans,2026]。

3. 重複 span 與 async/sync hook 混用根因

歷史 issue #554 指出：早期實作在 同一 httpx.Client 重複請求 時，對 _transport 重複包裝，導致第 n 次請求出現 n 層巢狀 span [來源:GitHub Issue HTTPX instrumentor wraps transport repeatedly,2021]。維護者後續以 wrapt／transport 層重構（PR #2909 等）改善與 httpx 版本的相容性 [來源:WebSearch opentelemetry httpx wrapt PR]。2024–2026 另有一條線：若 多個 Client 共用同一 httpx.HTTPTransport() 再分別 instrument_client，transport 可能被加掛兩次而產生雙 span；社群 PR #3106 討論在 transport 上加 _is_instrumented_by_opentelemetry 旗標以避免重複 instrument，惟該 PR 已因 stale 關閉，代表行為仍須以「一 transport 只對應一條 instrument 鏈」自管 [來源:GitHub PR httpx add is_instrumented flag,2024–2026]。另：AsyncClient 必須使用 async_request_hook／async_response_hook，否則同步 hook 在 async 路徑會出現未 await 的錯誤類型問題，PR #1920 即針對此修復 [來源:WebSearch opentelemetry httpx async hooks PR 1920]。

4. Logs 與 Traces 的 SDK 行為（Python 1.30）

opentelemetry-python v1.30.0（2025-02-04） 明確：一律初始化 logs SDK；OTEL_PYTHON_LOGGING_AUTO_INSTRUMENTATION_ENABLED 只控制是否掛上 stdlib logging 的 handler，而非整體 logs SDK 是否存在；並修正 logs body None 序列化以對齊 1.31+ 資料模型 [來源:GitHub Release open-telemetry/opentelemetry-python v1.30.0,2025]。這影響「想把 JSONL 管線改成 OTel Logs + OTLP」時的啟動順序與測試假設。

最佳實踐

  1. Client 單例化 + 單一路徑 instrument：在 Agent 長生命週期程式中，對 KB／ntfy 共用單一 AsyncClient，僅在工廠函式內呼叫一次 instrument_client 或啟用全域 instrument()，並避免與手動 AsyncOpenTelemetryTransport 疊床架屋 [來源:OpenTelemetry HTTPX Instrumentation,2026][來源:GitHub Issue #554,2021]。
  2. 先決定 semconv 模式：在過渡期對 staging 設 OTEL_SEMCONV_STABILITY_OPT_IN=http/dup，對 prod 設 http，並在 Grafana 查詢同時支援新舊欄位直到儀表板改版完成 [來源:HTTP semantic conventions declared stable,2023]。
  3. 敏感標頭顯式列出：以 OTEL_INSTRUMENTATION_HTTP_CAPTURE_HEADERS_CLIENT_REQUEST 等變數白名單擷取 header，搭配 …SANITIZE_FIELDS 遮罩 set-cookie、session 類欄位；文件強調「全抓 headers 有外洩風險」[來源:OpenTelemetry HTTPX Instrumentation,2026]。
  4. health／metrics 端點排除：使用 OTEL_PYTHON_HTTPX_EXCLUDED_URLS 逗號分隔 regex，避免對 /api/health 等高頻低價值請求產生成本與 cardinality [來源:OpenTelemetry HTTPX Instrumentation,2026]。
  5. 雙軌：OTel trace + 既有 JSONL：Microsoft Learn（zh-tw）說明可透過 自訂 SpanProcessor 在 OnEnd 注入維度；對照本專案可把 task_key、backend 寫入 span attribute，同時保留 hooks/post_tool_logger.py 的 JSONL 供離線稽核，兩者以 trace_id 關聯 [來源:Microsoft Learn 在 Application Insights 中新增和修改 OpenTelemetry,2026]。

陷阱與反例

  • 重複包裝 transport：同 client 連續請求或共用 raw transport 導致 span 倍增——根因與緩解見 issue #554 與 PR #3106 討論；反例是「每支工具函式內都 instrument_client」[來源:GitHub Issue #554,2021][來源:GitHub PR #3106,2026]。
  • 在 response_hook 內盲目讀滿 body：transport hook 拿到的是低階 stream，行為與 httpx 高階 response 讀取一致地受「是否已讀取」約束；大檔下重複讀或與 raise_for_status 順序錯置仍可能踩 httpx 已知討論（與前篇 KB 筆記互參）。此處 OTel 並未自動替你緩衝 body [來源:OpenTelemetry HTTPX Instrumentation,2026]。
  • 忽略 4xx 語意與 span status：HTTP semconv 說明 client 端 4xx 建議設 Error，但應用若把 404 當正常探測，需在 exporter 前以 processor 覆寫（Azure 文件展示 Http4xxSuccessProcessor 模式）；直接關閉 status 會讓 SLO 誤判 [來源:Semantic conventions for HTTP spans,2026][來源:Microsoft Learn 修改遙測,2026]。

與本專案（daily-digest-prompt）關聯

  • 場景 A：在 uv run python 呼叫 KB POST /api/import 的共用模組外，包一層 HTTPXClientInstrumentor.instrument_client，於 async_response_hook 寫入 http.response.status_code 與自訂 attribute ddp.task_key，與 results/todoist-auto-*.json 的驗證流程對齊，失敗 trace 可直接在 Tempo 用 url.full 篩 localhost:3000 [來源:OpenTelemetry HTTPX Instrumentation,2026]。
  • 場景 B：run-todoist-agent-team.ps1 觸發的子程序若未來改為長駐 Python runner，可啟用 OTel Logs SDK（1.30 行為） 將原本 logs/structured/*.jsonl 的欄位逐步雙寫 OTLP，仍保留 JSONL 7 天清理政策於 PS 層 [來源:GitHub Release v1.30.0,2025][來源:專案慣例 CLAUDE.md 日誌保留]。

程式碼範例（官方文件節錄 + 自訂註解）


# 來源：OpenTelemetry HTTPX Instrumentation 官方文件（Read the Docs）節錄
import httpx
import asyncio
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor

url = "https://example.com"
HTTPXClientInstrumentor().instrument()  # 全域；若只用單一 AsyncClient 可改 instrument_client

async def get(url: str):
    async with httpx.AsyncClient() as client:
        return await client.get(url)

asyncio.run(get(url))

自寫驗證：對 localhost:3000 請求請改為工廠單例 + OTEL_PYTHON_HTTPX_EXCLUDED_URLS=".*/api/health" 以降低噪音。
