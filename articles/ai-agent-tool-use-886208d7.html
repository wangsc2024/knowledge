AI Agent Tool Use 契約測試與 Schema-Driven Guardrails：工具呼叫可靠性實作指南（2026）

技術概述

當 AI Agent 進入多工具、多 handoff、可執行副作用的系統開發場景後，失敗點不再只在 prompt，而在工具契約是否夠明確、schema 是否被嚴格檢查、以及失敗是否能在呼叫前被阻斷。2025-2026 年官方文件已逐步把這件事產品化：Anthropic 強調詳細 tool description、input_examples 與 strict schema；OpenAI Agents SDK 則把 tool guardrails、tripwire 與 blocking/parallel 模式納入執行框架。這代表實務重點已從「讓模型會呼叫工具」轉成「讓工具呼叫可驗證、可阻斷、可回放」。

核心方法與工具

  • Schema-first 工具定義：用 JSON Schema 明確描述必填欄位、列舉值、格式限制與範例輸入。
  • Strict tool use：對高風險工具啟用嚴格 schema 驗證，避免模型產生半合法參數。
  • Tool guardrails：在工具前後加入 input/output guardrails，對敏感參數、空回應、越權結果設 tripwire。
  • Contract tests：以固定輸入、錯誤輸入、邊界輸入、注入輸入測試 tool schema 和 guardrail 行為。
  • Spec-driven 開發骨架：把工具契約、驗收條件、測試矩陣與實作任務維持在同一條規格鏈中。

實際工作流程（step-by-step）

  1. 定義工具契約：先寫工具名稱、使用時機、禁用時機、參數意義、回傳欄位與失敗模式。
  2. 建立 schema 與 examples：為每個工具補 required、enum、pattern、input_examples，避免模糊輸入。
  3. 把 guardrails 綁在工具而非只綁在 agent：特別是 manager/handoff 流程，工具層才能攔截每次呼叫。
  4. 撰寫 contract tests：至少覆蓋 happy path、schema mismatch、prompt injection、敏感資訊外洩、空輸出/過量輸出。
  5. 把 tripwire 連到 CI：若 guardrail 阻斷、schema 驗證失敗或工具輸出不符契約，CI 直接 fail。
  6. 回放與觀測：記錄工具名稱、引數、guardrail 決策、tripwire 次數與被阻斷成本，作為後續優化依據。

程式碼/配置範例

1. Anthropic 工具定義：schema + examples + strict


{
  "name": "github_manage_pr",
  "description": "管理 PR 生命週期。僅在使用者明確要求建立、審查或合併 PR 時使用。若缺少 repo、branch 或 action，不要猜測，應要求補充。回傳應只包含 PR 編號、狀態與下一步需要的人類確認資訊。",
  "strict": true,
  "input_schema": {
    "type": "object",
    "properties": {
      "action": {
        "type": "string",
        "enum": ["create", "review", "merge"]
      },
      "repo": { "type": "string" },
      "pr_number": { "type": "integer", "minimum": 1 },
      "review_comment": { "type": "string" }
    },
    "required": ["action", "repo"]
  },
  "input_examples": [
    { "action": "create", "repo": "org/app" },
    { "action": "review", "repo": "org/app", "pr_number": 42, "review_comment": "請補 migration 測試" }
  ]
}

2. OpenAI Agents SDK：tool guardrails


import json
from agents import (
    Agent,
    Runner,
    ToolGuardrailFunctionOutput,
    function_tool,
    tool_input_guardrail,
    tool_output_guardrail,
)

@tool_input_guardrail
def validate_pr_args(data):
    args = json.loads(data.context.tool_arguments or "{}")
    if args.get("action") == "merge" and not args.get("pr_number"):
        return ToolGuardrailFunctionOutput.reject_content(
            "merge 動作必須提供 pr_number"
        )
    if "token" in json.dumps(args).lower():
        return ToolGuardrailFunctionOutput.raise_exception(
            "偵測到敏感憑證，不允許送入工具"
        )
    return ToolGuardrailFunctionOutput.allow()

@tool_output_guardrail
def validate_pr_result(data):
    text = str(data.output or "")
    if "merged":
        pass
    if "ERROR" in text:
        return ToolGuardrailFunctionOutput.raise_exception(
            "工具輸出含錯誤訊息，需人工接管"
        )
    return ToolGuardrailFunctionOutput.allow()

@function_tool(
    tool_input_guardrails=[validate_pr_args],
    tool_output_guardrails=[validate_pr_result],
)
def github_manage_pr(action: str, repo: str, pr_number: int | None = None):
    return {"action": action, "repo": repo, "pr_number": pr_number, "status": "ok"}

agent = Agent(name="Repo Agent", tools=[github_manage_pr])
result = Runner.run_sync(agent, "幫我合併 org/app 的 PR 42")

3. CI 中的 contract test 矩陣


tool_contract_tests:
  cases:
    - name: merge_requires_pr_number
      input:
        action: merge
        repo: org/app
      expect: reject
    - name: block_secret_leak
      input:
        action: review
        repo: org/app
        review_comment: "token=sk-live-xxx"
      expect: tripwire
    - name: create_pr_happy_path
      input:
        action: create
        repo: org/app
      expect: allow

效果量化（生產力提升數據，若有）

  • IBM 2025 的 agile epics 案例研究指出，17 位產品經理的使用者研究顯示，LLM 參與需求品質評估具有可行性與高滿意度，但仍存在採用阻力，表示 AI 最適合先扮演 quality gate，而非完全自動決策。
  • ICSE 2025 關於 automated traceability 的研究顯示，requirements smells 對「是否存在 trace link」的預測有小但顯著影響；這代表工具或需求契約若描述含糊，LLM 的追溯與測試生成品質會實際下降。
  • 推論：在 agent tool use 場景中，先改善 tool description、schema 嚴格度與 guardrail 覆蓋率，通常比單純換大模型更划算。這是根據上述研究結果與官方文件綜合推論，不是單一來源直接量測。

與本專案的應用場景（daily-digest-prompt 如何受益）

  • 可把 knowledge-query、Todoist、通知與研究匯入視為高價值工具，為每個工具補上 schema、禁止條件與 input_examples。
  • 對 curl 類 KB 匯入流程加 preflight contract test：標題必填、tags 非空、contentText 長度達標、source=import。
  • 對研究類自動任務加入 tripwire：若 topic 重複、registry 未更新、KB 匯入回應不含 note id，直接標記 partial。
  • 針對多步驟任務可區分 blocking guardrails 與 parallel guardrails：有副作用的寫入走 blocking，純搜尋與觀測可接受 parallel。

風險與注意事項

  • 不要只做 agent-level guardrails：manager/handoff 流程可能繞過最外層檢查，工具層 guardrails 才能逐次攔截。
  • strict schema 不是萬靈丹：它能限制格式，但不能保證語義正確，仍需 contract tests 和人類審查。
  • 工具描述過短會直接降低可靠性：官方文件已明確把詳細描述列為工具表現最重要因素之一。
  • parallel guardrail 有成本/風險折衷：延遲較低，但 guardrail 失敗時可能已消耗 token 或觸發工具副作用。
  • 觀測必須落地：若不記錄被阻斷原因、tripwire 次數與工具錯誤型別，就無法做後續優化。
