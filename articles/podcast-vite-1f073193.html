對話腳本

主持人 A（解說）：今天我們來聊一個實用的技術主題：如何把知識庫變成高效能的靜態網站。假設你有一個 RAG 系統或 Obsidian 筆記，裡面累積了幾百篇文章，想要公開分享給更多人看，傳統的做法是透過 API 查詢，但這樣有個問題——搜尋引擎看不到、離線看不了、而且還需要後端服務持續運行。

主持人 B（提問）：聽起來確實麻煩。那如果改成靜態網站呢？是不是就像把筆記轉成 HTML 檔案，直接放到 CDN 上？

主持人 A（解說）：沒錯！這就是 SSG——靜態網站生成的核心概念。建構時把所有頁面預先渲染成純 HTML，部署後完全不需要後端。好處是 SEO 友善、可以離線存取、而且 Cloudflare Pages 或 Netlify 的免費方案就能承載，TTFB 可以壓到 50 毫秒以內。

主持人 B（提問）：那為什麼特別選 Vite 系列的方案？市面上不是還有 Next.js 或 Hugo 嗎？

主持人 A（解說）：Vite 的優勢在於建構速度和開發體驗。我們來看三個主流方案。第一個是 VitePress，它是 Vue 生態的原生選擇，初次造訪時是純靜態 HTML，後續導航自動升級為 SPA，內建 MiniSearch 全文搜尋，而且 HMR 延遲只要 100 毫秒。500 頁的文件站建構時間只需 8.4 秒。

主持人 A（解說）：第二個是 Astro Starlight，它採用 Islands Architecture，只載入互動所需的 JavaScript，而且支援 React、Vue、Svelte 混用。最大亮點是效能——同樣 500 頁的站點，產出大小只有 1.2MB，LCP 只要 820 毫秒，還內建零配置的 Pagefind 搜尋引擎。

主持人 B（提問）：這兩個聽起來都很強。那第三個方案是什麼？

主持人 A（解說）：第三個是 Vite-SSG 或 Vite-React-SSG，它們是純 Vite 加上自動 SSR 和 SSG 合併。適合需要完全自定義 UI 的場景，比如你已經有一個 React 或 Vue 應用，想要靜態化，或者需要自建搜尋和路由系統。彈性最高，但設定複雜度也相對高一些。

主持人 B（提問）：那我應該怎麼選？有沒有一個簡單的判斷標準？

主持人 A（解說）：可以這樣判斷：如果你是 Vue 團隊，或者筆記數量在 500 頁以內，選 VitePress 最快上手。如果追求極致效能、或者筆記量在 500 到 10000 頁之間，選 Astro Starlight。如果你需要完全自定義前端，或者已經有現成的 React/Vue 應用要靜態化，那就選 Vite-SSG。

主持人 B（提問）：框架選好了之後，那知識庫的內容怎麼變成網站？總不能手動複製貼上吧？

主持人 A（解說）：當然不行，這就需要一條自動化管線。流程是這樣的：首先從知識庫 API 查詢筆記，然後用 Python 腳本做差異偵測——可以用 SHA-256 雜湊或時間戳來判斷哪些筆記有更新。接著把 contentText 轉成 Markdown 檔案，注入 frontmatter（包含標題、標籤、日期），最後依標籤分類到不同目錄。

主持人 B（提問）：你剛才提到差異偵測，為什麼需要這一步？

主持人 A（解說）：因為增量同步可以大幅提升效能。如果每次都全量重建 300 多頁，可能需要好幾分鐘。但如果只處理有變動的 10 到 20 頁，可能幾秒就完成了。推薦的混合策略是：先用時間戳初篩出可能變更的筆記，再用內容雜湊確認是否真的改過，最後只重建變更的頁面，搜尋索引也採用差異更新而非全量重建。

主持人 B（提問）：這樣確實聰明。那建構完之後，效能優化還有哪些要注意的？

主持人 A（解說）：部署端的優化可以從幾個方向著手。首先是壓縮，啟用 Brotli 可以減少 15 到 25% 的傳輸量。圖片改用 WebP 或 AVIF 格式，可以再省 30 到 50%。然後 Cloudflare Pages 預設就支援 HTTP/3，可以降低連線延遲。如果要做成 PWA，可以加上 Service Worker 離線快取，讓使用者完全離線也能瀏覽。

主持人 B（提問）：這套方案實際應用在你們的專案上效果如何？

主持人 A（解說）：我們有一個案例是把本地 RAG 知識庫同步到靜態網站。具體流程是：Python 腳本從 localhost:3000 抓取筆記，做增量同步後推送到 GitHub，然後 Cloudflare Pages 自動部署。建議加上 GitHub Actions 排程，每 6 小時自動同步一次，這樣知識庫更新後網站也會自動跟著更新，完全不需要手動介入。

主持人 A（解說）：總結一下：Vite SSG 生態提供了三個強大的選擇——VitePress 適合快速上手、Astro Starlight 適合追求效能、Vite-SSG 適合自定義需求。配合增量同步管線和邊緣部署，可以把知識庫變成高效能、低成本的公開網站，而且整套流程都可以自動化。

主持人 B（提問）：這個方案真的很實用，尤其對於想要分享知識但又不想維護後端服務的人來說。我覺得最有價值的是增量同步的概念，既省時間又省資源，而且搭配 GitHub Actions 之後幾乎是零維護成本。感謝今天的分享！
