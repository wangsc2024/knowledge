推理時運算擴展的 API 控制面：OpenAI／Gemini／Claude 對照（2026-04-10）

概述

推理時運算擴展（test-time compute／inference-time scaling）指在推論階段投入更多計算（更長內部推理、重試、採樣與驗證等），以換取數學、程式、科學推理等難題上的正確率。相對於單純擴大預訓練規模，近年產品線普遍把「可調整的推理預算」暴露成 API 參數與計費單位，使開發者能在延遲、成本與正確率之間做路由與 SLO 設計。

與知識庫內既有「方法論／PRM／產業格局」類筆記互補：本文聚焦 三大供應商官方文件與部落格中可操作的 control surface。

重點摘錄

OpenAI（o 系列與「學習推理」敘事）

  • OpenAI 在介紹透過大規模強化學習讓模型學會使用 chain of thought 時明確指出：o1 的表現會隨訓練算力（train-time compute）與「思考時間」（test-time compute）增加而提升，且兩者 scaling 的瓶頸與預訓練不同，仍待研究。
  • 評測上，官方說明除非另有註記，o1 在許多推理基準上使用「最大 test-time compute」設定，以凸顯推理模型與 GPT-4o 的差距。
  • 編程／競賽情境：官方以 IOI 為例，展示在固定提交次數下，test-time selection（依公開測資、模型自產測資與學習式評分函數挑選提交）可顯著加分；放寬提交上限時，即使不做 selection，分數仍可大幅上升——說明 推理品質與「推論時搜尋／選擇策略」可分開優化。

Google Gemini（thinkingBudget、thinkingLevel、thought summaries）

  • Gemini 2.5／3 系列在 API 層以內部「thinking」流程強化推理與多步規劃；文件說明可透過 thinkingBudget（2.5） 或 thinkingLevel（Gemini 3 起建議） 控制推理深度與成本。
  • thinkingBudget（2.5）：以「思考 token 數量」引導模型；0 可關閉思考（視型號而定，Pro 文件載明無法完全關閉思考）、-1 表示動態依題目調整。
  • thinkingLevel（3.x）：離散等級（如 low／medium／high），文件說明未指定時 3 系列預設動態 high，且 3.1 Pro 無法關閉思考。
  • 可觀測性：可開啟 thought summaries（includeThoughts），取得濃縮推理摘要；計價上，思考 token 會反映在輸出計費（文件提供 thoughtsTokenCount 等欄位語意），與最終回答摘要並存。

Anthropic Claude（Extended Thinking 與 Adaptive Thinking）

  • Extended Thinking：在 Messages API 以 thinking 物件啟用；啟用後回傳內容包含 thinking 區塊與 text 區塊（串流與多輪時需依文件保留／回傳結構，避免竄改 thinking）。
  • Claude Opus 4.6／Sonnet 4.6：文件建議優先使用 thinking: { type: adaptive }，並可以 effort（如 high／max）作軟性引導；舊式 type: enabled + budget_tokens 仍可用但朝向淘汰。
  • 與工具呼叫：文件說明 interleaved thinking（在工具呼叫之間插入推理）在新型號上的預設與限制；實務上影響 agent 管線的 context 組裝與 token 帳務。

分析與實務含義

  1. 產品路由：同一題可拆成「低延遲非推理模型」與「高推理預算模型」兩層；後者適用於數學證明、長程式、關鍵決策支援。
  2. FinOps：三家皆將「隱藏推理／思考」納入 輸出 token 經濟；若未監控 thinking token，SLO 只看 latency 會低估帳單與尾延遲。
  3. 可觀測性：Gemini thought summaries、OpenAI 對 hidden CoT 的摘要策略、Anthropic 雙區塊結構，對 除錯與安全審計 有不同取捨，整合到自家 OTel／eval 管線時需對齊欄位語意。
  4. 與學術框架的銜接：Snell 等提出的 compute-optimal test-time scaling（依難度調整 proposer／verifier 資源）可對照為：API 參數是「單輪推理預算」，而 採樣＋選擇／外部 verifier 仍屬應用層設計。
