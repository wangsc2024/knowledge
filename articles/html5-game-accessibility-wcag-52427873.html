HTML5 遊戲無障礙設計（Game Accessibility）完整指南

技術概述

遊戲無障礙設計（Game Accessibility）是移除不必要的障礙，讓所有玩家（包括視覺、聽覺、運動、認知障礙者）都能享受遊戲的設計實踐。全球約有 10-20% 的人口具有某種形式的障礙，遊戲產業正快速將無障礙從「可選功能」提升為「核心設計原則」。

本研究涵蓋三大標準體系（GAG、XAG、WCAG）在 HTML5 遊戲中的應用，並提供四類障礙的具體適配策略與 Canvas 遊戲的技術實作模式。

---

一、三大無障礙標準體系

1.1 Game Accessibility Guidelines (GAG)

gameaccessibilityguidelines.com 提供的開放指南，按實施成本與影響範圍分為三級：

| 級別 | 定義 | 範例 |

|------|------|------|

| Basic | 低成本、廣泛影響 | 控制重映射、字體大小、色彩對比 |

| Intermediate | 需要規劃、良好設計 | 巨集系統、UI 可調整、練習模式 |

| Advanced | 複雜適配、深度障礙 | 語音 GPS、聲納地圖、手語翻譯 |

按障礙類型分為 6 大類別：

  • 運動障礙（Motor）：控制重映射、靈敏度調整、避免連打 QTE、按住替代切換
  • 認知障礙（Cognitive）：簡單語言、互動教學、進度提示、可調速度
  • 視覺障礙（Vision）：高對比、可調字體、螢幕閱讀器、音訊描述
  • 聽覺障礙（Hearing）：字幕/標題、視覺音效指示、單聲道切換
  • 語音障礙（Speech）：文字聊天替代語音、符號式溝通
  • 通用（General）：難度選擇、輔助模式、自動存檔、無障礙資訊公開

1.2 Xbox Accessibility Guidelines (XAG)

Microsoft 於 2020 年發布的業界標準（目前 v3.2），由 Gaming & Disability Community 共同開發。每條指南包含：

  • Goal：預期影響
  • Scoping Questions：適用性判斷
  • Implementation Guidelines：最低實作要求
  • Potential Player Impact：受影響的障礙類型

配套資源：Gaming and Disability Player Experience Guide (GADPEG) 提供各障礙類型玩家遇到的具體屏障描述。

1.3 WCAG 2.1 在遊戲中的應用

研究顯示，WCAG 2.1 AA 級在遊戲領域的合規率僅約 34.1%（「可感知」原則最高），主要差距在於：

  • 遊戲不使用標準 OS UI，而是自訂渲染
  • Canvas/WebGL 內容對輔助技術不透明
  • 遊戲本身的挑戰性與無障礙之間存在張力

二、四類障礙的具體適配策略

2.1 運動障礙適配

核心原則：不假設玩家的輸入能力

| 層級 | 措施 | 實作要點 |

|------|------|----------|

| Basic | 控制重映射 | 所有按鍵可自訂，含鍵盤/手把/觸控 |

| Basic | 靈敏度調整 | 滑桿控制移動/瞄準靈敏度 |

| Basic | 大按鈕/間距 | 觸控元素 ≥ 48x48px，間距 ≥ 8px |

| Intermediate | 避免連打/QTE | 提供自動執行或單鍵替代 |

| Intermediate | 按住→切換 | 所有長按操作可改為切換模式 |

| Intermediate | 巨集系統 | 複雜操作序列綁定為單一按鍵 |

| Advanced | 輸入冷卻 | 0.5 秒防誤觸延遲（可選） |

| Advanced | 輔助技術相容 | 支援 Switch Control、眼動追蹤等 |

2.2 視覺障礙適配

核心原則：資訊不僅依賴視覺傳達

| 層級 | 措施 | 實作要點 |

|------|------|----------|

| Basic | 色彩對比 ≥ 4.5:1 | WCAG AA 標準，UI 文字與背景 |

| Basic | 不以固定色彩傳達資訊 | 加形狀/圖案/文字輔助 |

| Basic | 可調字體大小 | 預設 ≥ 16px，可放大至 200% |

| Intermediate | 色盲模式 | 紅綠色盲（deuteranopia）為最常見 |

| Intermediate | 高對比模式 | 前景/背景可自訂色彩 |

| Advanced | 螢幕閱讀器支援 | ARIA 標記 + Fallback DOM |

| Advanced | 音訊描述 | 環境/事件的語音描述 |

| Advanced | 聲納地圖 | 可 ping 的空間音訊導航 |

色盲模式設計要點：

  • 三種主要色盲：紅色盲（Protanopia）、綠色盲（Deuteranopia）、藍色盲（Tritanopia）
  • 不要僅「換色」，而是加入形狀/紋理/圖示區分
  • 提供色盲模擬器供開發者自測

2.3 聽覺障礙適配

核心原則：聲音資訊必須有視覺替代

| 層級 | 措施 | 實作要點 |

|------|------|----------|

| Basic | 字幕 | 所有重要對話，含說話者標示 |

| Basic | 音效視覺化 | 方向箭頭、閃爍圖示 |

| Basic | 獨立音量控制 | 音效/語音/背景音樂分開調整 |

| Intermediate | 背景音描述 | 重要環境音效以文字/圖示呈現 |

| Intermediate | 單聲道切換 | 立體聲→單聲道選項 |

| Advanced | 字幕速度控制 | 適當的每分鐘字數（不超過 140 WPM） |

2.4 認知障礙適配

核心原則：降低認知負荷，容許不同處理速度

| 層級 | 措施 | 實作要點 |

|------|------|----------|

| Basic | 簡單語言 | 避免專業術語，用白話文 |

| Basic | 自訂遊玩速度 | 速度滑桿或暫停功能 |

| Basic | 互動教學 | 邊玩邊學，非文字牆 |

| Intermediate | 目標提示 | 隨時可查看當前任務 |

| Intermediate | 練習模式 | 無失敗的安全環境 |

| Intermediate | 進度摘要 | 長敘事提供劇情回顧 |

| Advanced | 全語音朗讀 | 選單/文字皆可語音播放 |

三、HTML5 Canvas 遊戲的無障礙技術實作

3.1 Canvas 的核心挑戰

Canvas 是即時模式（Immediate Mode）位圖繪製，沒有 DOM 結構，螢幕閱讀器完全無法存取內容。這是 HTML5 遊戲無障礙的最大技術障礙。

3.2 Fallback DOM 模式（Sub DOM）

在 標籤內放置 fallback 內容，為螢幕閱讀器提供替代存取：


<canvas id="game" width="800" height="600" role="img" aria-label="Space Invaders Game">
  <!-- Fallback DOM for screen readers -->
  <div role="status" aria-live="polite">
    <p>Score: <span id="a11y-score">0</span></p>
    <p>Lives: <span id="a11y-lives">3</span></p>
    <p>Level: <span id="a11y-level">1</span></p>
  </div>
  <div role="log" aria-live="assertive" id="a11y-events">
    <!-- Game events announced here -->
  </div>
</canvas>

同步更新 Fallback DOM：


function updateAccessibility(gameState) {
  document.getElementById('a11y-score').textContent = gameState.score;
  document.getElementById('a11y-lives').textContent = gameState.lives;
  
  // Announce important events
  if (gameState.event) {
    const eventLog = document.getElementById('a11y-events');
    const announcement = document.createElement('p');
    announcement.textContent = gameState.event;
    eventLog.appendChild(announcement);
    // Cleanup old announcements
    while (eventLog.children.length > 5) {
      eventLog.removeChild(eventLog.firstChild);
    }
  }
}

3.3 ARIA Live Region 動態公告


class AccessibilityAnnouncer {
  constructor() {
    this.polite = document.createElement('div');
    this.polite.setAttribute('aria-live', 'polite');
    this.polite.setAttribute('role', 'status');
    this.polite.style.cssText = 'position:absolute;left:-9999px;';
    
    this.assertive = document.createElement('div');
    this.assertive.setAttribute('aria-live', 'assertive');
    this.assertive.style.cssText = 'position:absolute;left:-9999px;';
    
    document.body.appendChild(this.polite);
    document.body.appendChild(this.assertive);
  }
  
  announce(message, priority = 'polite') {
    const target = priority === 'assertive' ? this.assertive : this.polite;
    target.textContent = '';
    requestAnimationFrame(() => { target.textContent = message; });
  }
}

// 使用範例
const a11y = new AccessibilityAnnouncer();
a11y.announce('Level 3 started', 'assertive');
a11y.announce('Score: 1500');

3.4 鍵盤完全可操作


class AccessibleKeyboardHandler {
  constructor(canvas) {
    canvas.setAttribute('tabindex', '0');
    canvas.setAttribute('role', 'application');
    canvas.setAttribute('aria-label', 'Game area, use arrow keys to move');
    
    this.bindings = new Map();
    canvas.addEventListener('keydown', (e) => this.handleKey(e));
  }
  
  remap(action, newKey) {
    this.bindings.set(newKey, action);
  }
  
  handleKey(e) {
    const action = this.bindings.get(e.code);
    if (action) {
      e.preventDefault();
      action();
    }
  }
}

function drawEntity(ctx, entity, palette) {
  ctx.fillStyle = palette[entity.type];
  switch(entity.type) {
    case 'player':  // 三角形
      drawTriangle(ctx, entity.x, entity.y, entity.size);
      break;
    case 'enemy':   // 方形
      ctx.fillRect(entity.x, entity.y, entity.size, entity.size);
      break;
    case 'powerup': // 圓形
      drawCircle(ctx, entity.x, entity.y, entity.size / 2);
      break;
  }
}

3.6 W3C 提案：Engine-to-ARIA Bridge

W3C 2018 年 Web Games Workshop 提出的未來方向：

  • 利用 Accessibility Object Model (AOM) 建立遊戲引擎到 ARIA 的橋接
  • 三層適配策略：OS 級（縮放/觸覺）→ 引擎級（重映射/字幕/TTS）→ 遊戲級（標題/控制）
  • 多模態渲染：視覺瞄準可轉為音訊或觸覺回饋
  • 讀取 OS 使用者偏好（對比度/字體大小/字幕樣式）自動適配

四、最常被抱怨的四大問題

根據 gameaccessibilityguidelines.com 統計：

  1. 控制重映射：玩家無法按自己的需求調整按鍵配置
  2. 文字大小：字體過小，尤其在電視/大螢幕上
  3. 色盲支援：約 8% 的男性有某種色覺異常
  4. 字幕呈現：字幕背景透明度、大小、位置不佳

這四項全部屬於 Basic 級別，實作成本低但影響巨大。

六、HTML5 遊戲無障礙檢查清單

Basic 必做（所有遊戲）

  • [ ] 鍵盤完全可操作（tabindex + role + 事件處理）
  • [ ] 色彩對比 ≥ 4.5:1（文字/UI）
  • [ ] 資訊不僅靠色彩傳達（加形狀/文字）
  • [ ] 字體可調整（最小 16px，可放大）
  • [ ] 控制可重映射
  • [ ] 暫停功能（Esc 或觸控按鈕）
  • [ ] 閃爍頻率 < 3Hz（避免光敏性癲癇）
  • [ ] 無障礙功能資訊公開（設定頁面）

Intermediate 建議（中型遊戲）

  • [ ] 色盲模式（至少 Deuteranopia）
  • [ ] 高對比模式
  • [ ] 遊戲速度可調
  • [ ] Canvas Fallback DOM（螢幕閱讀器）
  • [ ] ARIA Live Region 動態公告
  • [ ] 觸控元素 ≥ 48x48px
  • [ ] 音量獨立控制（音效/音樂/語音）
  • [ ] 字幕支援（含說話者標示）

Advanced 進階（大型遊戲）

  • [ ] 完整螢幕閱讀器支援
  • [ ] 全語音朗讀（TTS）
  • [ ] 音訊空間導航（聲納地圖）
  • [ ] 輔助技術設備相容性
  • [ ] 障礙玩家參與測試

七、對本專案（禪境遊廊）的應用建議

禪境遊廊已有的基礎：WCAG 2.1 合規、響應式設計、CSS Grid。

建議優先實作：

  1. 所有遊戲加入 Canvas Fallback DOM（分數/生命/狀態）
  2. 加入全域色盲模式切換（遊戲門戶設定）
  3. 確保所有遊戲鍵盤完全可操作（tabindex + role="application"）
  4. 觸控按鈕尺寸統一 ≥ 48x48px
  5. 加入 ARIA Live Region 公告重要遊戲事件
