專案RAG優化日報 2026-03-27

Hook 管線效能優化與磁碟空間自動治理策略

研究日期：2026-03-27

研究範疇：Hook 執行效能瓶頸、Session 級快取機制、磁碟空間生命週期自動化管理

專案：daily-digest-prompt（每日摘要 Agent 系統）

---

一、研究背景與動機

本專案採用 9 個 Python Hook（4 PreToolUse + 3 PostToolUse + 1 PostToolUseFailure + 1 Stop）作為機器強制層，每次工具呼叫同步觸發 2-4 個 Hook。Hook 總計 ~250KB Python 程式碼，同步執行的效能開銷成為瓶頸。同時 output/（984MB）、results/（591MB）、podcasts/（270MB）等目錄缺乏自動清理，磁碟持續膨脹。

與過往差異：3/20-21 聚焦 state/ JSON 管理；3/26 聚焦 Hook 安全性；本次聚焦 Hook 效能 + 全域磁碟生命週期。

---

二、20 項關鍵發現

A. Hook 管線效能（8 項）

  • A1 HIGH：9 個 Hook 全部同步執行，每次工具呼叫增加 200-500ms
  • A2 HIGH：Hook 檔案過大（validate_config 71KB、post_tool_logger 41KB、on_stop_alert 40KB），首次載入 ~1-2 秒
  • A3 MEDIUM：無 Session 級快取，每次重新 import + 讀取 hook-rules.yaml，重複 I/O 佔 30-40%
  • A4 MEDIUM：hook_utils.py (24KB) 被所有 Hook import，模組載入冗餘
  • A5 MEDIUM：PostToolUse 雙重觸發（[*] + [Write]/[Edit]），Write 操作 Hook 開銷翻倍
  • A6 LOW：CJK Guard 對非 CJK 檔案（.json/.yaml）仍執行字元掃描
  • A7 LOW：無 Hook 執行時間監控，效能退化不可見
  • A8 MEDIUM：PreToolUse 4 個 Hook 各啟動獨立 Python 進程（每個 ~100-200ms）

B. 磁碟空間管理（7 項）

  • B1 CRITICAL：output/ 984MB 無清理
  • B2 CRITICAL：results/ 591MB 無清理
  • B3 HIGH：podcasts/ 270MB 無歸檔
  • B4 HIGH：137 個 io-guard-state 孤立檔案無 TTL
  • B5 MEDIUM：logs/ 52MB 保留策略未強制執行
  • B6 LOW：tmp/ 20MB 手動清理
  • B7 LOW：無磁碟使用度量

C. 配置效能交叉問題（5 項）

  • C1 MEDIUM：hook-rules.yaml (15KB) 每次 Hook 觸發重新解析
  • C2 MEDIUM：context-usage.json 204KB 無上限
  • C3 MEDIUM：timeouts.yaml 每 Phase 重新載入（2-5 秒）
  • C4 LOW：research-registry.json 87 entries 可能含過期
  • C5 LOW：47 個 YAML 無跨檔語義驗證

---

三、12 項優化方案

P1 高投報率（本週）

P1-A Hook Session 快取：Phase 0 預計算 cache/hook-rules-compiled.json，Hook 讀 JSON（~1ms）取代 YAML（~50ms）。每日節省 ~100 秒。

P1-B 磁碟自動清理腳本：tools/cleanup-disk.ps1 三層清理 — output/ 7天刪除、results/ 14天壓縮、io-guard-state 3天刪除、logs 7天壓縮、tmp 1天刪除。Task Scheduler 每日 00:00。回收 ~1.2GB。

P1-C 磁碟健康指標：check-health.ps1 新增磁碟區塊（output/ >500MB、state/ >200 檔案警戒）。

P2 中等投報率（1-2 週）

P2-A Hook 進程合併：pre_tool_dispatcher.py 統一入口，Python 進程 4→1。業界對標：Lefthook 5x 加速（5.149s→1.016s）。節省 ~300ms/次。

P2-B CJK Guard 智慧跳過：副檔名白名單（.json/.yaml/.lock）直接 exit(0)。~60% 跳過。

P2-C PostToolUse 去重：合併 [*] 和特定 matcher 為單一入口。

P2-D context-usage.json 滾動窗口：保留最近 100 session，超出歸檔。

P3 長期改善

P3-A Hook 自監控：計時裝飾器 + hook-perf.jsonl + p95 告警。

P3-B 三層壓縮歸檔：熱（7天）→溫（7-30天 gzip）→冷（>30天 tar.gz）→可選 R2。對標 Harness Artifact Lifecycle。

P3-C YAML 預編譯：Phase 0 將 timeouts/hook-rules/routing 編譯為 cache/config-compiled.json。

P3-D 跨檔語義驗證：validate_config.py --cross-validate 模式。

---

四、量化效益

  • Hook 單次延遲：~500ms → ~200ms（-60%）
  • 每日 Hook 總開銷：~100秒 → ~40秒（-60%）
  • 磁碟佔用：~1.92GB → <500MB（-74%）
  • state/ 檔案數：371 → <50（-86%）
  • io-guard-state：137 → 0（-100%）
