執行摘要（約 200 字）

2025-2026 年間，AI Agent 工具呼叫品質工程從「結果正確性」進化為「多層精準度量化 + 閉環自動糾錯」的工程實踐。ToolCritic（2025）定義 8 類工具呼叫錯誤並透過診斷回饋實現 +13% 準確率提升；RAG-Repair 針對執行後語義錯誤，結合 RAG 上下文修復達 7-15% pass rate 改善。Datadog 2026 年生產數據揭示：每月 840 萬次 LLM 錯誤中 rate limit 佔 1/3，僅 28% 利用 prompt cache，工具呼叫品質監控尚未成業界標配。

本報告提煉 5 項核心洞見，涵蓋三層錯誤分類框架、多維精準度指標、雙軌自動糾錯設計、生產觀測盲點，以及評估閉環架構，可直接對映本系統 tool_calling_quality_audit 自動任務的改善方向。

---

核心洞見

洞見 1：工具呼叫錯誤的三層分類框架

ToolCritic（arXiv:2510.17052）定義 8 類工具呼叫錯誤，可按偵測時機分為三層：

| 層級 | 偵測時機 | 典型錯誤類型 | 修復難度 |

|------|---------|------------|---------|

| L1 靜態 | 執行前 | 工具名稱幻覺（hallucination）、語法錯誤、參數類型不符 | 低（規則可偵測）|

| L2 動態 | 執行中 | 參數值語義錯誤、提前呼叫（premature invocation）| 中（需上下文判斷）|

| L3 反思 | 執行後 | 輸出誤解（argument misalignment）、結果循環依賴 | 高（需 RAG 修復）|

來源：ToolCritic: Detecting and Correcting Tool-Use Errors in Dialogue Systems — A 級

本系統對映：tools/tool_calling_quality_audit 的三層管線（parse→validate→semantic）對應此框架，但 L3 反思層尚無自動修復機制。

---

洞見 2：工具選擇精準度的五維量化指標

Maxim AI 評估指南（getmaxim.ai）與 TRAJECT-Bench（arXiv:2510.04550）定義工具呼叫精準度需同時量化 5 個維度：

  1. 工具選擇準確率（Tool Selection Accuracy）：是否選對工具（0-1 分）
  2. 呼叫決策 F1（Invocation F1）：是否該呼叫工具 — 偽陽性（無需呼叫卻呼叫）vs 偽陰性（需要呼叫卻不呼叫）
  3. 參數生成精準率（Parameter Accuracy）：工具選對但參數錯誤（最常見缺口）
  4. 執行成功率（Execution Success Rate）：工具呼叫實際成功比例
  5. 工具序列精準率（Sequence Accuracy）：多步驟工具呼叫的順序與依賴正確性

來源：Observability and Evaluation Strategies for Tool-Calling AI Agents — B 級

本系統對映：現有 tool_calling_quality_audit 僅追蹤第 1、4 維度，缺乏呼叫決策 F1 與序列精準率。

---

洞見 3：雙軌自動糾錯架構 — 前置診斷 + 後置 RAG 修復

2025 年研究收斂出「雙軌」自動糾錯設計：

軌道 A：ToolCritic 前置診斷回饋


工具呼叫生成 → ToolCritic 偵測錯誤 → 生成 "thought" 回饋 → 主 LLM 自我修正
準確率提升：+13%（vs zero-shot baseline）

軌道 B：RAG-Repair 後置反思修復（arXiv:2510.17874）


工具執行失敗 → 分析失敗類型 → RAG 從官方文件檢索修復上下文 → 重生成工具呼叫
pass rate 改善：+7 至 +15%（因模型而異）
correctness 改善：+36%
最大受益：小模型（Llama3.1 8B: +24%）

關鍵發現：troubleshooting 文件作為 RAG 來源比官方文件高出 10% pass rate，因其含有具體的錯誤情境與修復路徑。

來源：Repairing Tool Calls Using Post-tool Execution Reflection and RAG — A 級

---

洞見 4：生產觀測盲點 — 2026 Datadog 數據揭示三個缺口

Datadog State of AI Engineering 2026 揭示生產環境現況：

| 指標 | 數值 | 意涵 |

|------|------|------|

| LLM 呼叫錯誤率（2026-03）| 2% | 840 萬次/月錯誤 |

| Rate limit 佔比 | ~1/3 錯誤 | 容量設計問題，非工具品質問題 |

| Single-service-call agents | 59% | 多數 Agent 為 monolithic，工具序列品質問題被低估 |

| Prompt cache 利用率 | 28% | 嚴重低估，與工具呼叫重複成本直接相關 |

| Agent 框架採用率 | 9% → 18%（2025-2026）| 可觀測性需求倍增 |

三個工具品質盲點：

  1. Rate limit 錯誤與工具呼叫品質問題混合統計，掩蓋真實工具錯誤率
  2. 59% 單次呼叫 agents 的品質監控完全缺失（沒有序列追蹤）
  3. 工具呼叫評估尚未成為業界標配（相較對話品質評估普遍滯後）

---

洞見 5：「推理陷阱」— 強化 LLM 推理反而加劇工具幻覺

The Reasoning Trap（arXiv:2510.22977，2025）提出反直覺發現：強化 LLM 推理能力（chain-of-thought、long reasoning）反而會放大工具幻覺。

機制：

  • 更長的推理鏈 → 更多中間步驟 → 更多「自圓其說」機會 → 工具呼叫決策更易偏離事實
  • 工具幻覺（fabricating tool name / parameters）在 reasoning-enhanced 模型中高於同等 instruct 模型

實務建議：

  1. 工具路由層應保持短路徑決策（routing logic 越簡單越好，勿在工具選擇前做多步推理）
  2. 高精準度場景：工具選擇用 classifier（deterministic）而非 LLM reasoning
  3. 評估時分開追蹤 reasoning-trace 步驟數 vs 工具呼叫準確率，尋找「甜蜜點」

來源：The Reasoning Trap: How Enhancing LLM Reasoning Amplifies Tool Hallucination — A 級

---

建議行動

  1. 補齊 5 維量化指標：tool_calling_quality_audit 補充呼叫決策 F1（偽陽性/偽陰性）與工具序列精準率追蹤
  2. 建立 troubleshooting 文件 RAG 庫：將 tools/TOOL_INDEX.md 與常見錯誤模式建立 RAG 快捷索引，支援 L3 自動修復
  3. 隔離 rate limit 錯誤：從工具品質指標中過濾 429/rate_limit 錯誤，確保真實工具選擇準確率可見
  4. 工具路由保持 deterministic：高頻工具路由（ntfy/KB/Todoist）改用規則路由或分類器，避免 LLM reasoning 引入幻覺
