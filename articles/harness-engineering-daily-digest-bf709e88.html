研究日期：2026-04-22

核心目標：可靠性 +50-70%、人工干預 -60%、維護成本 -40%

---

執行摘要

本指南整合 2026 年 Harness Engineering 最新實踐（Anthropic、OpenAI、LangChain、Google、Meta），提出 P0-P2 三階段優化方案，預期 3.5 個月內提升系統可靠性 50-70%、降低 token 使用 20-30%、減少人工干預 60%。

核心發現

  1. Harness > Model：LangChain 僅改 Harness（未換模型），accuracy 從 52.8% → 66.5%（+13.7%）
  2. 70/30 黃金比例：70% Computational + 30% Inferential 驗證最優（成本降 70%）
  3. 40% Reset Rule：Context 在 40% 時重置優於 80% 壓縮（消除焦慮）
  4. Separated Evaluator：測試案例與評估器解耦（2026 年標準）
  5. PEV Pattern：Plan-Execute-Verify 三階段分離提升可觀測性

二、P0-P2 三階段優化方案

P0：立即實施（1-2 週，高影響 / 低成本）

#### P0-A：Separated Evaluator Pattern（可靠性 +40%）

目標：建立獨立評估器目錄，將測試邏輯從 prompt 中解耦。

核心原理：

將測試案例（Cases）與評估器（Evaluators）分離，靈活應對 AI agent 非確定性輸出。

實施步驟：


# 1. 建立目錄結構
prompts/evaluator/
├── base_evaluator.py          # 基礎評估器類
├── research_task_evaluator.py # 研究任務評估器
├── todoist_task_evaluator.py  # Todoist 任務評估器
└── system_audit_evaluator.py  # 系統審查評估器

# 2. 定義評估器介面
class BaseEvaluator:
    def evaluate_task_completion(self, output: dict) -> float
    def evaluate_quality(self, output: dict) -> dict
    def evaluate_safety(self, output: dict) -> bool

# 3. 實施具體評估器（範例）
from deepeval.metrics import HallucinationMetric, AnswerRelevancyMetric

class ResearchTaskEvaluator(BaseEvaluator):
    def evaluate_task_completion(self, output: dict) -> float:
        score = 0.0
        if output.get("kb_note_id"): score += 0.4
        if output.get("sources_count", 0) >= 10: score += 0.3
        if len(output.get("key_findings", "")) > 100: score += 0.3
        return score

預期成果：

  • ✅ 測試覆蓋率 0% → 60%
  • ✅ False positive rate ↓ 50%
  • ✅ 可除錯性 ↑（評估分數分解為 sub-metrics）

工時估算：22h（約 3 個工作日）

#### P1-C：Agentic Testing 原型（測試案例數 +50%）

目標：為 todoist-auto-* 任務啟用自主測試生成。

  4. Report：生成人類可讀的測試報告

P2：中期演進（1-2 月，戰略投資）

#### P2-A：Plan-Execute-Verify（PEV）架構重構（完成度 +25%）

目標：將現有三階段（Query-Execute-Assemble）升級為 PEV 模式。

架構對比：

| 階段 | 現有（QEA） | 優化後（PEV） |

|------|------------|-------------|

| Phase 1 | Query（查詢 Todoist） | Plan（分解任務為可驗證步驟） |

| Phase 2 | Execute（執行任務） | Execute（按計畫執行，每步驟附檢查點） |

| Phase 3 | Assemble（組裝輸出） | Verify（獨立驗證器檢查計畫完成度） |

核心優勢：

  • ✅ 可檢視性：計畫是 first-class artifact，可記錄、檢查、審計
  • ✅ 可除錯性：失敗時可定位具體計畫步驟（而非黑盒 CoT）
  • ✅ 可靠性：驗證層攔截低品質輸出，避免錯誤級聯傳播

實施步驟：


# 計畫 Schema 範例
plan:
  goal: "處理用戶退款請求"
  steps:
    - id: 1
      action: retrieve_order_info
      params: {order_id: "ABC123"}
      expected_output: {status, purchase_date, amount}
    - id: 2
      action: check_refund_policy
      depends_on: [1]
      expected_output: {eligible: boolean, reason: string}
    - id: 3
      action: process_refund
      depends_on: [2]
      condition: "$step2.eligible == true"

預期成果：

  • ✅ 任務完成度 +25%
  • ✅ 計畫偏離率 -60%
  • ✅ 可除錯性大幅提升

工時估算：48h（約 6 個工作日）

#### P2-B：Meta-Harness Tuner（Token 效率 +15%）

目標：端到端 Harness 參數優化（timeout、token budget、retry policy）。

優化目標：

  • 最小化 token 使用
  • 最大化成功率
  • 平衡延遲與成本

實施步驟：


# tools/meta_harness_optimizer.py（新建）
class MetaHarnessOptimizer:
    def optimize(self) -> dict:
        # 定義參數空間
        params = {
            "timeout": (60, 600),
            "max_retries": (1, 5),
            "token_budget": (5000, 50000)
        }
        
        # 目標函數（最小化）
        def objective(x):
            timeout, max_retries, token_budget = x
            simulated_results = self._simulate(timeout, max_retries, token_budget)
            token_cost = np.mean([r["tokens"] for r in simulated_results])
            success_rate = np.mean([r["success"] for r in simulated_results])
            return (token_cost / 10000) * 0.3 + (1 - success_rate) * 0.7
        
        # 執行優化
        result = minimize(objective, x0=[120, 2, 20000], bounds=list(params.values()))
        return {"timeout": int(result.x[0]), "max_retries": int(result.x[1])}

預期成果：

  • ✅ Token 效率 +15%
  • ✅ 成功率 +10%

工時估算：24h（約 3 個工作日）
