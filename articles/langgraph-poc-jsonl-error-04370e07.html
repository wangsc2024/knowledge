LangGraph 中斷恢復與工具重放 PoC：評測設計與 JSONL／error-object 追溯整合（2026）

執行摘要

本研究承接前次「MCP Host 與 LangGraph Interrupt」主軸之後續角度，聚焦 中斷節點之狀態恢復與工具調用重放，目標是把人機協同核准、可恢復編排與本儲庫既有 JSONL／error-object 觀測平面對齊成可落地的 PoC。綜合 LangGraph 官方對 Human-in-the-loop、Persistence／checkpoint 與 interrupt API 的說明 [來源：LangGraph Human-in-the-loop, 2025]、[來源：LangGraph Persistence, 2025]、[來源：LangGraph interrupt API, 2025]，共識是可中斷圖必須與執行緒狀態持久化並設計，否則核准與恢復將無法在長任務中穩定重入。MCP 規範則將 工具與資源邊界 置於 Host，使核准政策有明確掛點 [來源：MCP Specification, 2025]。OWASP LLM Top 10 亦反覆要求限制 過度代理 並保留審計軌跡 [來源：OWASP LLM Top 10, 2024]。對照 docs/plans/github-scout-20260331-post-tool-logger-schema-linkage.md 所描述之本專案 JSONL 與 error-object 聯動构想 [來源：儲庫架構計畫文件, 2026]，已形成「編排層 interrupt／Host 核准／觀測層 JSONL」三位一體的實務路線。待驗證項為：僅快照圖狀態而不記錄工具參數雜湊與模型版本指紋時，重放仍可能因上游漂移而失敗 [來源：內部分析, 2026]。

背景與動機

context/continuity/auto-task-ai_deep_research.json 指出下一深化方向為 PoC 驗證 interrupt 恢復與工具重放；同時 context/research-series.json 中 langgraph 系列仍處 foundation／application 缺口。對 daily-digest-prompt 而言，Phase 1→2→3 管線與 Hooks 已提供 post_tool_logger 結構化事件；若 Agent 編排在 LangGraph／類圖框架下引入 interrupt，缺乏與 thread_id、tool_call_id、run_id 對齊的欄位，將使 self-heal 與 OODA 閉環 難以關聯到具體一次核准決策。

核心技術解析

可中斷與恢復：LangGraph 將 human-in-the-loop 視為可在節點邊界暫停、待人類輸入後再恢復的機制；其可靠度依賴 persistence 所提供的 checkpoint／thread 語意，使執行不必從頭重建完整上下文 [來源：LangGraph Human-in-the-loop, 2025]、[來源：LangGraph Persistence, 2025]。

interrupt 的原語位置：Python API 層以 interrupt 暴露「帶著狀態切出」的行為，利於與表單收集、工具二次確認與多輪核准銜接 [來源：LangGraph interrupt API, 2025]。

工具邊界與治理：MCP 將 capabilities 暴露與 Host 端治理連結；高風險工具路徑應在 Host 進行 policy 與同意流程，而非下放到模型單方面決策 [來源：MCP Specification, 2025]。OWASP LLM 風險清單同樣將「不受控的工具使用」列為核心威脅類型之一，需技術與流程並重 [來源：OWASP LLM Top 10, 2024]。

與本 repo 對齊：儲庫內計畫文件已點出 error-object-schema 與 post_tool_logger JSONL 應與執行 metadata（如 trace／proposal）機讀對照 [來源：儲庫架構計畫文件, 2026]，與本研究所述 PoC 的事件模型是一致方向。

關鍵洞見（三角佐證）

  • ✅ 已驗證：持久化 checkpoint 與 interrupt 是人類核准後能否「無損重入」的前置條件；缺一事則 PoC 無法穩定觀測（4+ 方向文件與架構文件共識）。
  • ✅ 已驗證：MCP Host 的政策層與 LangGraph 節點層可分工——前者管工具允許集合，後者管圖內控制流與恢復點（3+ 來源）。
  • ✅ 已驗證：JSONL＋error-object 可作為 唯一真相來源（SSOT） 的稽核投影面；PoC 事件應優先映射至此平面而非另造日誌格式（3+ 來源：規範＋儲庫文件）。
  • ⚠️ 待驗證：若未對工具輸入做 規範化雜湊 與 模型指紋 紀錄，「重放」可能僅能驗證控制路徑而無法驗證輸出品質一致性（目前主要為推理性延伸，待實驗）。
  • ⚠️ 待驗證：除錯向 time-travel／子圖 API 對生產追蹤欄位的欄位映射表是否一次到位（待實驗）。

實務應用建議（daily-digest-prompt／個人專案）

  1. 欄位契約：在 post_tool_logger 事件中加入 thread_id、interrupt_id（或節點序號）、mcp_tool_name 與 approval_decision，與現有 task_key／phase 並列；拒絕新增第三套追蹤鍵。
  2. PoC 範圍：先做 deterministic stub tool（固定輸出）＋固定 thread_id，對 interrupt → resume → 完成 與 中斷後取消 兩條路徑量測 JSONL 事件數與順序。
  3. 對齊 error-object：當工具被拒或逾時，產生符合 config/schemas/error-object-schema.json 之 instance，指向對應 JSONL 行號或檔案偏移欄位（若已實作）。
  4. 與排程協調：PoC 任務列入低頻手動或 run-todoist-agent-team 實驗槽位，避免與生產 digest 搶佔資源；結果寫入 results/ 與 state/scheduler-state.json 模式與其他自動任務一致。

與現有知識的連結

本主題延續 registry 與 continuity 已涵蓋之 LangGraph interrupt／MCP Host 脈絡，但聚焦 重放評測與追溯契約，避免與前次長標題主題逐字重複；並補上 langgraph research-series 自 foundation 往 application 推進所需的實作型知識。

未來展望

下一步可將 approval 理由碼 與 工具调用重放 seed 固定化，並以 property-based 測試檢查「任一 interrupt 點恢復後工具序列仍與golden trace 一致」；另可評估與 OpenTelemetry GenAI semantic conventions 的 span 對齊（本專案 未來計畫/待辦事項.md 已提及相關方向）。
