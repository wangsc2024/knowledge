AI Agent 約束演化機制深度研究：從 proposed 到 embedded 的四型分類、轉化路徑與當前系統優化建議

研究日期：2026-04-05
系列：constraint-evolution-governance（mechanism 階段）
研究類型：深度研究洞察簡報（insight-briefing）

摘要（Executive Summary）

本研究深入分析 Daily-Digest-Prompt 專案的約束演化機制，建立四型分類框架（技術/流程/配置/工具），設計 proposed→embedded 轉化路徑決策樹，並針對當前 constraint-registry 中 11 個 rejected 約束提出根因分析與優化方案。

關鍵發現：

  1. 當前系統已具備三層約束演化管線（constraint_evolve.py v2），但 pattern 提取過於粗糙導致 11/14 約束被 rejected
  2. 缺少約束分類框架，所有約束統一以 Hook 處理，未考慮不同類型的最佳落地方式
  3. 需要引入 context-aware 分類與 LLM 輔助 pattern 精煉

建議行動：

  1. 建立約束分類器（四型分類 + 決策樹）
  2. 強化 pattern 提取邏輯（summary NER + context 分析）
  3. 建立三層驗證機制（pre-embed pytest + post-embed 回歸 + runtime 監控）

核心洞見

1. 約束分類四型框架

  • 技術約束（Hook 實作）：檔案路徑、命令格式驗證
  • 流程約束（Prompt Template）：執行順序、依賴檢查
  • 配置約束（Config YAML）：閾值、權限邊界
  • 工具約束（Tool Script）：複雜驗證邏輯

2. 轉化路徑決策樹

  • pattern 可正規化 → Hook
  • 需人工判斷 → Prompt
  • 純數值閾值 → Config
  • 複雜邏輯 → Tool

3. 11 個 rejected 約束的根本問題

  • Pattern 提取過於粗糙（summary 未精確解析）
  • 缺少 context-aware 分類（誤將 false positive 視為約束）
  • 需要引入 LLM 輔助 pattern 精煉

4. 驗證機制三層

  • Pre-embed：pytest 測試覆蓋
  • Post-embed：回歸測試（確保不誤攔）
  • Runtime：監控告警（檢測新失敗模式）

5. 優先級排序原則

  • P0：安全約束（已有 11 次以上觸發）
  • P1：穩定性約束（影響成功率）
  • P2：效率約束（優化流程）

實作路線圖

Phase A：強化 Pattern 提取（1-2 週）

  • 實作 NER 提取（檔案路徑、命令參數、API 錯誤）
  • 實作 false positive 判定
  • 重新處理 rejected 約束

Phase B：分類器與決策樹（1 週）

  • 設計 post-embed 回歸測試
  • 設定 runtime 監控告警
