本地 LLM 生產部署可靠性工程

日期: 2026-06-28 | 主題: 本地 LLM 生產部署可靠性工程：VRAM 預算管理 × llama.cpp 穩定性 × 多模型共存策略

研究類型: insight_briefing

摘要

本地 LLM 生產部署的可靠性問題在 2025-2026 年急遽受到重視。本報告整合 llama.cpp 社群、AI gateway 廠商與生產事故回顧，聚焦三大核心問題：VRAM 預算閘門設計（admission control 防 OOM race condition）、llama.cpp server 靜默崩潰監控（40% 生產事故發生於啟動後 10 秒）、Tier Fallback Chain 的靜默降級陷阱（100% 完成率但 0% schema 完整率）。

關鍵洞見

  1. VRAM 預算必須在 allocation 前計帳（mutex-protected VRAM ledger），保留 10% headroom（24GB 卡 ≤22GB 放行）
  2. 靜默崩潰是頭號殺手：40% 事故在啟動後 10 秒內，systemd Restart=on-failure + /health 輪詢是最低標配
  3. KV Cache Swapping：非活躍 KV state 移至 host RAM，24GB 卡 4 並發/35B 模型可穩定在 22.05 GB
  4. Tier Fallback 靜默降級陷阱：100% 完成率但 0% schema 完整率；需 payload 正規化 + quality floor 驗證
  5. 生產 Fallback 共識：hop budget cap 2-4 跳、7 類故障形狀分類器、指數退避 100ms→500ms→2000ms

多透鏡洞察（L2）

  • Sam Altman 透鏡：本地 LLM 可靠性工程的目標是縮小本地基礎設施與雲端 API 的可靠性差距，使 local-first 策略在成本與 SLA 上同時可行
  • Jensen Huang 透鏡：VRAM 是 GPU 上最稀缺的資源，VRAM 預算管理是「GPU 工廠」的效能核心；KV Cache 量化是垂直整合思維
  • Taleb 反模式：「Retry 三次」迷信（Narrative Fallacy）；靜默降級脆弱性；VRAM 容量規劃過度最佳化忽略 tail case

建議行動（DDP）

  • 驗證 vram-arbiter headroom=2GB 在 12B server 啟動前生效
  • 補 per-port health 輪詢（port 8082）
  • 孤兒進程清理閘門：啟動前確認 port 乾淨
  • Tier fallback schema 驗證：Sonnet45 → local Gemma 路徑
