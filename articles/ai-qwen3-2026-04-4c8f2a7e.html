品質評分：4/5

---

執行摘要

2026 年 4 月，Alibaba 推出 Qwen3 系列（0.6B–235B 參數），提出混合推理（Hybrid Reasoning）範式：在同一模型中整合「慢思考」（Thinking Mode，長 CoT 推理鏈）與「快思考」（Non-Thinking Mode，直接回答）。技術報告（arXiv 2505.09388）揭示了支撐此能力的四階段訓練管線：長 CoT 冷啟動 → 推理強化學習（RL）→ 思考融合（Thinking Fusion）→ 通用行為對齊 RL。

核心結論（五點）：

  1. 四階段管線的關鍵不是規模而是工程順序：每個階段解決前序遺留的不同問題，缺一不可。
  2. 思考預算呈對數縮放關係：投入算力越多精度提升越顯著，但邊際效益遞減；小模型（<7B）獲益更大（+15-20%），大模型（+5-10%）。
  3. Thinking vs Non-thinking 存在 2 倍以上的性能差距（Qwen3-32B MATH-500: 0.95 vs 0.44；LiveCodeBench: 0.54 vs 0.29），在推理密集型任務中切換代價顯著。
  4. 生產環境非思考模式需要框架版本管控：vLLM/SGLang 部分版本仍輸出思考痕跡，需雙層保障。
  5. Qwen3.6 的 Always-on CoT + Gated DeltaNet 線性注意力代表另一設計哲學：以確定性推理深度換取開發者便利性，在 SWE-bench Verified 達 78.8%，以 27B dense 超越 397B MoE 模型。

---

研究背景與動機

傳統 LLM 在「快速回應」與「深度推理」之間存在根本取捨：一般指令追蹤模型反應迅速但推理能力有限；推理模型（如 o1 系列、DeepSeek-R1）擅長複雜問題但回應慢且成本高。開發者不得不維護兩套模型，根據任務類型手動路由。

Qwen3 的混合推理試圖消除這一缺口：讓同一模型視任務複雜度自適應選擇推理深度。這不只是模型能力問題，更是訓練方法論的創新——如何在單一訓練流程中同時優化兩種截然不同的推理行為，且不讓它們相互干擾？

此研究的時代意義在於：若成功，混合推理範式將取代「推理模型 vs 通用模型」的二元市場格局，重塑 AI 基礎設施的部署架構。Qwen3.6 的 Always-on CoT 路線進一步提出了第三條路：乾脆消除用戶選擇，讓深度推理成為預設行為。

---

核心技術解析

3.1 四階段訓練管線：逐步建構混合能力

第一階段：長 CoT 冷啟動（Long Chain-of-Thought Cold Start）

從 Qwen2.5 checkpoint 出發，使用高品質的長思考鏈 SFT 數據進行微調。目的是讓模型初步學習「展開推理過程」的行為模式，為後續 RL 提供合理的初始策略。此階段的數據品質至關重要——低品質的假 CoT 數據可能導致後續 RL 訓練崩潰。

第二階段：推理強化學習（Reasoning RL）

使用 GRPO（Group Relative Policy Optimization）算法，配合 3,995 個 query-verifier pairs 進行訓練 [來源：Qwen 官方 Blog, 2026]。獎勵信號主要基於規則式驗證器（數學答案正確性、代碼執行結果），避免 reward hacking。Qwen3 的關鍵工程貢獻是 Entropy 控制機制：強制保持 policy entropy 穩定或單調遞增，防止模式崩潰（mode collapse）。此機制使模型在長時間 RL 訓練中保持輸出多樣性，是大規模 RL 成功的隱藏因子。

第三階段：思考融合（Thinking Mode Fusion）

這是四階段管線的核心創新 [來源：arXiv 2505.09388, 2026]。通過 SFT 將「非思考」能力注入到已具備強推理能力的模型，使用混合 SFT 數據（thinking 樣本 + non-thinking 樣本）同時訓練。關鍵洞見：non-thinking 能力並非從零訓練，而是在強推理基礎上注入，避免「追兩兔兩不得」的問題。架構基礎採用 GQA + SwiGLU + RoPE + RMSNorm（移除 QKV-bias，改用 QK-Norm）[來源：DebuggerCafe, 2026]。

第四階段：通用行為對齊 RL（General RL）

針對指令追蹤、多輪對話、格式規範等通用行為進行最終對齊，確保思考融合後的模型在非推理場景仍表現穩定。此階段可類比傳統的 RLHF，但建立在混合推理能力的基礎上。訓練數據涵蓋 36 萬億詞量、119 種語言 [來源：搜狐科技, 2025]。

強到弱蒸餾（Strong-to-Weak Distillation）

為降低小模型完整走四個階段的成本，Qwen 使用旗艦模型（235B/32B）的輸出知識蒸餾至小模型（0.6B-7B），跳過完整管線的部分步驟 [來源：DebuggerCafe, 2026; 騰訊新聞, 2025]。

3.2 Thinking Budget Control：可量化的算力-精度取捨

Thinking budget 的本質是生成 token 計數器：設定最大 thinking tokens 上限，模型在此預算內完成推理鏈，超過則截斷 [來源：DebuggerCafe, 2026]。

官方確認：thinking budget 能提供 scalable performance improvements correlated with compute budget（性能提升與算力投入正相關）[來源：Qwen 官方 Blog, 2026]。

獨立研究（arXiv 2508.12140，醫療推理領域）量化了三個效率區間：

  • 0-256 tokens：高效率即時區間，適合低延遲應用
  • 256-512 tokens：精度-效率均衡區間
  • 512+ tokens：高精度模式，適合 STEM、診斷類任務

對數縮放關係（logarithmic scaling law）在多個獨立評測中確認，且不同領域的最優預算不同（神經學 > 心血管），說明最優預算是任務感知的，不存在通用最優值。

EvalScope 官方評測揭示：「問題難度越高 → 生成越長 → 資源效率反而上升」（31%→43%）[來源：EvalScope, 2026]，說明模型能自適應地在困難問題上投入更多思考，這本身就是 budget control 的隱式機制。

關鍵數據對比（Qwen3-32B）：

  • MATH-500：Thinking 0.95 vs Non-Thinking 0.44
  • AIME24：Thinking 0.70
  • LiveCodeBench：Thinking 0.54 vs Non-Thinking 0.29
  • GPQA Diamond：Thinking 0.60

3.3 Qwen3.6 的架構革新：Always-on CoT 與線性注意力

Qwen3.6-27B 代表另一設計哲學：Always-on CoT——所有查詢自動應用結構化推理，無需開發者配置 [來源：dasroot.net, 2026]。

關鍵架構創新：

  • Gated Delta Networks（線性注意力）：在 3:1 比例中替換標準 Transformer 注意力，解決長上下文的速度懸崖問題，使 262K（可擴展至 1M）context 實用化 [來源：QwenLM/Qwen3.6 GitHub, 2026]
  • MoE 布局：10 × (3×Gated DeltaNet→MoE + 1×Gated Attention→MoE)
  • Thinking Preservation：跨對話輪次保留思考上下文，避免 Agent 多步任務中重新推理的冗餘計算 [來源：QwenLM/Qwen3.6 GitHub, 2026]
  • 百萬 Agent 環境強化學習（Qwen3.5）：前一代 Qwen3.5 系列透過百萬 Agent 環境的 RL 訓練強化 agentic 能力，Qwen3.6 在此基礎上精煉

性能數據：SWE-bench Verified 78.8%，Terminal-Bench 2.0 61.6%，在 agentic coding benchmark 超越 397B MoE 模型，驗證了「最優參數效率點而非最大規模」的設計哲學 [來源：MarkTechPost, 2026]。

關鍵洞見

I1 ✅（5 來源）：四階段管線的核心是 Stage 3「模式融合」——non-thinking 能力是注入的而非訓練的

Non-thinking 能力並非獨立訓練的新行為，而是通過在強推理模型基礎上注入混合 SFT 數據實現的。這一設計避免了同時訓練兩種行為模式時的干擾問題，是 Qwen3 成功的核心工程洞見。支持來源：arXiv 2505.09388、Qwen 官方 Blog、DebuggerCafe、騰訊新聞、Sohu。無明顯反方觀點。

I2 ⚠️（2 來源）：Entropy 控制是 Qwen3 RL 穩定性的隱藏機制

保持訓練 policy entropy 穩定或單調遞增，防止模式崩潰，使大規模長時間 RL 訓練成為可能。支持來源：Qwen 官方 Blog、arXiv 技術報告。具體實作細節（閾值設定、entropy 計算方式）在公開資料中尚不詳盡，屬待驗證技術細節。

I3 ✅（3 來源）：思考預算的「規模反轉效益」——小模型獲益更大

小模型（<7B）從 thinking budget control 獲益 +15-20% 精度提升，大模型僅 +5-10%。這對邊緣部署有特殊工程價值。支持來源：arXiv 2508.12140（醫療推理）、Qwen 官方 Blog、EvalScope 文件。反方觀點：醫療推理結果是否能泛化到其他領域尚需更多驗證。

I4 ⚠️（2 來源）：Gated DeltaNet 線性注意力解決長上下文速度懸崖

Qwen3.6 的線性注意力設計在原理上消除了 Transformer 的 O(n²) 注意力瓶頸，使 1M context 實用化。支持來源：dasroot.net、QwenLM/Qwen3.6 GitHub。社群 benchmark 聲稱比 Claude Opus 4.6 快 3 倍，官方未確認，需謹慎解讀。

I5 ⚠️（2 來源）：Thinking Preservation 是 Agent 系統的核心創新

跨輪次保留思考上下文消除了多步 Agent 任務中的重複推理計算。支持來源：QwenLM/Qwen3.6 GitHub、dasroot.net。具體機制（hidden state vs context window 層面）尚不明確，是值得追蹤的開放問題。

I6 ✅（3 來源）：強到弱蒸餾使小模型能跳過完整四階段管線

旗艦模型知識蒸餾至小模型的工程策略，大幅降低訓練成本。支持來源：DebuggerCafe、arXiv 2505.09388、騰訊新聞。反方觀點：蒸餾後小模型的 thinking budget 可調性尚未系統評測。

I7 ✅（3 來源）：生產環境非思考模式需框架版本管控，非開箱即用

vLLM、SGLang 部分版本的 enable_thinking=False 實作存在不一致性，部分版本仍輸出思考痕跡。支持來源：Ray GitHub issue #52979、LM-Studio 社群 issue #1559、搜尋結果社群回報。反方觀點：官方文件聲稱 enable_thinking=False 可正確工作，可能是框架版本問題而非基礎設計缺陷。

實務應用建議

1. 任務路由策略：依複雜度動態分配 thinking budget

不要二元選擇 thinking/non-thinking，而是建立任務複雜度評分器（0-10分），線性映射到 budget 上限（0→0 tokens，10→512+ tokens）。此策略可在保持用戶體驗的同時控制成本，特別適合 API 驅動的 SaaS 產品。

落地步驟：(1) 建立任務複雜度分類器（可用小模型實現），(2) 配置 budget 閾值表（對應業務場景），(3) 監控 budget 使用率與任務完成質量的相關性，(4) 每月根據實測數據調整閾值。

2. 邊緣部署優先考慮小模型 + thinking budget

由於小模型從 budget control 獲益更大（+15-20%），邊緣設備（手機、邊緣服務器）應優先部署 3B-7B Qwen3 模型並啟用 thinking budget，而非部署更大的 dense 模型。風險：邊緣設備計算資源限制可能使 thinking tokens 成為瓶頸，需測試最大可接受 budget 上限。

3. Agent 系統選型：Qwen3.6 的 Thinking Preservation 優先於 budget 靈活性

多步 Agent 任務（代碼生成、研究助理、複雜工作流）應優先評估 Qwen3.6 的 Thinking Preservation 特性，而非 Qwen3 的動態切換能力。Always-on CoT 消除了 Agent 框架需要手動管理 thinking 模式的複雜性。部署需求：最少 64GB RAM + GPU 加速。

4. 生產框架版本管控（風險緩解）

在 non-thinking 模式生產部署前，建立框架版本白名單（已驗證 enable_thinking=False 正確工作的版本），並加入輸出後處理層（檢測並過濾 thinking token 殘留）。避免靜默失敗造成成本超支。

5. 評估流程：優先使用任務特定 benchmark 而非通用 benchmark

Thinking mode 在 MATH-500/AIME 等推理密集型 benchmark 的優勢（0.95 vs 0.44）在通用問答場景不會完全複現。部署前應用業務場景真實樣本測試，確認 thinking budget 的性價比。

與現有知識的連結

本研究與知識庫中的 MoE LLM 部署研究（MoE 生產部署全景 2026、DeepSeek-V3 稀疏激活架構）形成互補：MoE 部署研究聚焦推理服務層（all-to-all 通訊、負載均衡、並行策略），而本次研究聚焦訓練方法論層（四階段管線、mixed SFT、RL entropy 控制）。兩者的交集在於 Qwen3.6 的 MoE 架構：理解 Qwen3 的訓練方法論有助於解釋 Qwen3.6 MoE 在 agentic coding 場景的出色表現（路由穩定性 + Always-on CoT 相乘效果）。

Qwen3 的四階段管線也與 MoE Fine-tuning 研究（LoRA 在 expert routing 層的行為分析）高度相關：若用 LoRA 對 Qwen3-30B-A3B 進行 fine-tuning，Stage 3 的思考融合行為是否會被破壞？這是值得追蹤的交叉研究問題。此外，本專案的 LangChain Canary 評測任務（langchain_canary_eval）未來可考慮加入 Qwen3 thinking/non-thinking 模式的對比評測，補強現有評測維度。

未來展望

當前四階段管線仍有幾個未解問題值得深化：

  1. Entropy 控制機制的完整工程細節：閾值設定、entropy 計算方式、與 KL divergence 正則化的關係——這些是能否複製 Qwen3 訓練成功的關鍵工程參數，社群復現實驗尚在早期。
  2. 蒸餾後小模型的 thinking budget 可調性：旗艦模型知識蒸餾後，小模型是否完整保留了四階段管線賦予的 budget 可調性，還是蒸餾過程壓縮了此能力？
  3. Thinking Preservation 的確切實作機制：是在 KV cache 層面保留，還是在 context window 中顯式嵌入？這直接影響 Agent 框架的記憶體管理策略。
  4. Always-on CoT 的成本結構分析：Qwen3.6 Always-on CoT 在低複雜度任務上的不必要推理開銷是否可接受？與 Qwen3 動態切換的 TCO（總擁有成本）對比需要系統評測。

下次研究建議方向：Qwen3 四階段管線的社群複製實驗報告，或 Always-on CoT vs 動態切換的生產成本結構實測分析。
