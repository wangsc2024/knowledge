Daily-Digest-Prompt 專案 Skill 驅動架構優化策略

從 62 Skill 精簡到執行效能與成本治理的實踐指南

研究日期：2026-03-23

任務 ID：6g2FfvWmxqhCWx85

研究動機：以 Skill 系統為切入點，全面分析本專案的架構複雜度、Token 消耗模式與自動任務調度效率，制定可落地的分階段優化計畫。

---

執行摘要

本研究針對 Daily-Digest-Prompt 專案現有 62 個 Skill（27 專案 + 35 工具）、36 個自動任務（日均 ~40 次執行）、日均 ~12.5M Token 消耗的系統現況，結合 2026 年 AI Agent 成本優化與多 Agent 架構設計最佳實踐，提出五維度優化策略：Skill 精簡合併、模型路由分層、Token 預算治理、自動任務調度優化、快取命中率提升。預期可實現 Token 消耗降低 30-50%、自動任務執行效率提升 20%、Skill 維護成本降低 40%。

---

一、現況診斷

1.1 系統規模指標

| 維度 | 當前值 | 業界建議 | 差距 |

|------|--------|---------|------|

| Skill 總數 | 62 個 | 15-25 個核心 | 偏高，存在合併空間 |

| 自動任務數 | 36 個（33 活躍） | 依需求 | 需評估 ROI |

| 每日執行次數 | ~40 次 | 依需求 | 部分可合併 |

| 日均 Token 消耗 | ~12.5M | 依預算 | 有 30-50% 優化空間 |

| 快取命中率 | 22.7% | 40-60% | 待提升 |

| 配置文件數 | 38+ YAML | 精簡為主 | 存在整合空間 |

| FSM 成功率 | 97%+ | 99%+ | 接近目標 |

1.2 Token 消耗分佈

根據 Phase 分佈分析（2026-03-23）：

  • Phase 2（自動任務並行）：4.2M（33.6%，最耗）
  • Phase 3（組裝通知）：1.4M（11.2%）
  • Phase 1（查詢規劃）：1.2M（9.6%）
  • 其餘為手動任務與系統開銷

關鍵發現：Phase 2 的並行 Agent 執行佔總消耗三分之一，是優化重點 [Moltbook-AI, 2026]。

---

二、五維度優化策略

2.1 Skill 精簡合併（降低維護成本 40%）

原則：「每個 Skill 應聚焦單一領域」[Calmops, 2026]，但功能高度重疊的 Skill 應合併。

候選合併清單：

| 候選群組 | 現有 Skill | 合併方案 | 預期效果 |

|---------|-----------|---------|----------|

| 品質監控 | quality-tracker + agent-result-validator + task-quality-gate | → unified-quality-gate | 3→1，減少 Skill 間交接開銷 |

| 系統觀測 | system-insight + context-budget-monitor + execution-journal | → system-observability | 3→1，統一觀測層 |

| Skill 治理 | skill-registry-sync + skill-lifecycle-manager + skill-test-scaffolder | → skill-governance | 3→1，Skill 元管理統一 |

| 研究策略 | kb-research-strategist + research-saturation-analyzer | → research-planner | 2→1，研究規劃統一 |

| Hook 治理 | hook-registry + prompt-lint | → config-lint | 2→1，配置審計統一 |

精簡後預估：62 → ~50 個 Skill（減少 ~20%），核心維護對象從 27 降至 ~20。

動態載入策略：採用 DynamicSkillLoader 模式 [Calmops, 2026]，僅在觸發時載入 SKILL.md，設定最大併發數限制，記憶體超限時自動驅逐低優先度 Skill。

2.2 模型路由分層優化（Token 成本降低 40-70%）

現有 llm-router.yaml 已實作 O(1) mapping，但可進一步細化：

三層模型路由策略 [Moltbook-AI, 2026]：

| 層級 | 任務類型 | 推薦模型 | 成本比 |

|------|---------|---------|--------|

| Tier 1（輕量） | 新聞摘要、格式轉換、狀態查詢 | Groq/Haiku | 1x |

| Tier 2（標準） | 研究分析、任務規劃、內容生成 | Sonnet 4.6 | 20x |

| Tier 3（高階） | 架構決策、系統審查、深度研究 | Opus 4.6 | 100x |

現有可立即優化：

  • hackernews-ai-digest：已使用 Groq 前處理 ✅
  • pingtung-news：可增加 Groq 快速摘要 ⚠️
  • kb_system_optimize：屬系統維護，可降至 Sonnet ⚠️
  • 佛學研究 4 任務（8 次/日）：產出導向，維持 Sonnet/Opus ✅

2.3 Token 預算治理（消耗降低 30-50%）

三層預算機制 [Moltbook-AI, 2026]：


# 建議新增 config/token-budget.yaml
budgets:
  per_request:
    max_output_tokens: 4096  # 限制單次輸出
    thinking_budget: 8000    # 限制推理 Token
  per_task:
    simple_task: 50000       # 簡單任務上限
    research_task: 200000    # 研究任務上限
    system_task: 100000      # 系統維護上限
  per_day:
    warn: 10000000           # 10M 警告
    critical: 18000000       # 18M 暫停

Context 管理優化：

  • 「傳送 50K Token 卻只需 5K」是常見浪費 [Moltbook-AI, 2026]
  • 選擇性工具輸出：研究任務的 KB 查詢結果僅保留相關片段
  • 漸進式摘要：Phase 2 長 session 自動壓縮舊訊息
  • ADR-036 壓縮策略已就位，需確認觸發閾值合理性

2.4 自動任務調度優化（執行效率提升 20%）

現況：36 個任務、~40 次/日，round-robin 輪轉。

優化方向：

A. 任務合併（減少啟動開銷）

  • 佛學研究 4 類（楞嚴經/教觀綱宗/法華經/淨土宗）每日 8 次 → 可考慮合併為 2 個 slot（上午佛學 A + 下午佛學 B），內部輪轉子類
  • 系統維護 7 任務 12 次/日 → 合併為 3 個綜合維護 slot

B. 時段感知調度 [Nature Scientific Reports, 2025]

  • 根據 failure-hour-analyzer 的 time-slot-risk.json 數據，避開高失敗時段
  • 高成本任務（深度研究）安排在離峰時段
  • 低延遲任務（通知、狀態查詢）安排在整點

C. 動態任務分配 [Nature Scientific Reports, 2025]

  • Contract Net Protocol：任務發布後由最適合的 Agent 競標
  • 根據當前系統負載動態調整並行 Agent 數量（3-5 → 2-4）

2.5 快取命中率提升（22.7% → 40%+）

已識別改進項（improvement-backlog 高優先級）：

| 策略 | 預期提升 | 實施難度 |

|------|---------|----------|

| TTL 分層（高頻端點短 TTL、低頻長 TTL） | +8-12% | 低 |

| Stale-While-Revalidate（過期快取先用再更新） | +5-8% | 中 |

| 語義快取（相似查詢共用結果） | +5-10% | 高 |

| Per-endpoint 分析（識別熱點端點） | 輔助決策 | 低 |

Anthropic Prompt Caching：重複系統提示可減少 90% 輸入成本 [Moltbook-AI, 2026]。本專案的 preamble.md（139 行共用前言）在每次 Agent 啟動時重複傳送，是 Prompt Caching 的理想候選。

  • [ ] 合併 5 組候選 Skill（13 → 5）

五、風險與緩解

| 風險 | 影響 | 緩解措施 |

|------|------|----------|

| Skill 合併引入回歸 | 功能缺失 | 合併前完整測試套件覆蓋 |
