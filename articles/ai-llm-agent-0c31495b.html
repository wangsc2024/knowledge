複合式 AI 與 LLM 基座：企業 Agent 串流編排與能力整合路徑

TL;DR

本文主張：企業 AI 的下一階段競爭力，取決於能否把 frontier LLM 嵌入「可註冊、可規劃、可觀測」的複合式架構，而非單純升級模型參數。

Phase 0 從 24 篇 arXiv 候選中收斂本題，避開既有 registry 已覆蓋的「多代理信任治理」「負例 scheming 防範」等重題，改聚焦 LLM 基座能力 × Agent 編排整合 這條仍缺系統性藍圖的軸線。

核心結論（4 點）：

  1. 范式轉移已發生：CAIS 調查與 Kandogan 藍圖均指出，產業正從單體 LLM 轉向整合 retriever、tool、planner 的複合系統 [Chaudhry et al., 2025; Kandogan et al., 2025] ✅
  2. Stream 是企業編排的核心抽象：agent/data registry + task/data planner + stream 訊息流，構成可對接既有 API 與資料資產的整合層 [Kandogan et al., 2025; arXiv 2406.00584] ✅
  3. 編排複雜度需與任務匹配：Anthropic 實務與 Gradientsys 實驗共同顯示，簡單 composable pattern 往往優於過早引入多代理；並行 subagent 可顯著降延遲但 token 成本可達 10–15× [Anthropic, 2024; Song et al., 2025] ✅
  4. 基座能力仍受 instruction/data 混合制約：指令微調的資料混合比例會顯著影響下游 agent 表現，編排層不能假設「換大模型就萬能」 [Wang et al., 2023; Xu et al., 2023] ✅

研究問題與範疇

研究問題

在 keywords「large language model architecture」「AI agent orchestration」「frontier model capabilities」交會處，本報告回答：

  1. 2025 年學術與工業界如何定義 Compound AI / CAIS，與單體 LLM 的能力邊界為何？
  2. 企業場景中，Agent 編排的主流架構模式（stream、registry、planner）如何與 LLM 基座對接？
  3. Frontier 能力（長上下文、工具使用、多步推理）在生產部署中應如何被「能力工程」與「系統編排」分工？

文獻探討

1. 複合式 AI 系統（CAIS）綜述

Chaudhry 等人 2025 年的 CAIS 調查將複合系統定義為：LLM 與 retriever、agent、tool、orchestrator 等外部模組整合，以克服單體模型在記憶、推理、即時 grounding 與多模態上的限制 Chaudhry et al., 2025。其 taxonomy 沿 RAG、LLM Agent、MLLM、Orchestration 四軸展開，指出評測與互操作性仍是開放問題。

Krishnan 2025 從演化史梳理 agent 從 rule-based 到「LLM + 感知 + 規劃 + 工具」的模組化架構，並提出涵蓋有效性、效率、穩健性與安全性的 holistic evaluation framework Krishnan, 2025。

2. 企業藍圖：Stream + Registry + Planner

Kandogan 等人提出 enterprise compound AI 的 blueprint architecture：以 streams 協調 agent 間的資料與指令流；以 agent registry 與 data registry 對接企業既有模型、API 與多模態資料；以 task/data planners 在 cost、accuracy、latency 等 QoS 約束下分解與映射任務 Kandogan et al., 2025。同團隊 2024 版本進一步將 stream 建模為可動態生產、監控與消費的訊息序列 Kandogan et al., 2024。

3. 多代理調度與 ReAct 編排

Gradientsys 展示以 typed MCP 與 ReAct planning loop 驅動的一對多調度：LLM scheduler 將 heterogeneous agent 並行派工，並以 SSE observability 暴露中間推理 Song et al., 2025。在 GAIA benchmark 上，相較 MinionS baseline 取得更高成功率、更低延遲與 API 成本。

4. LLM 基座：指令混合與能力曲線

Wang 等人 2023 將 instruction tuning 資料分為 NLP 下游、coding、general chat 三類，證明 mixing 策略會顯著影響不同應用場景的增益與負面轉移 Wang et al., 2023。WizardLM 的 Evol-Instruct 則顯示，以 LLM 自動演化複雜指令可逼近 ChatGPT 級 follow-complex-instruction 能力 Xu et al., 2023。

5. 資訊生成范式（GenIR）與 RAG 定位

Ai 等人 2025 的 GenIR 章節區分 information generation 與 information synthesis，指出 RAG 是緩解 hallucination、對接外部 corpus 的關鍵范式 Ai et al., 2025。

6. 一線工程實務：簡單可組合優先

Anthropic 總結：最成功的 agent 實作使用簡單、可組合的 pattern Anthropic, 2024。其 multi-agent Research 產品採 orchestrator-workers，並強調 parallel tool calling 可將研究時間縮短最高 90% Anthropic, 2025。

核心論證

論點一：單體 frontier LLM 已觸及「系統整合」瓶頸 ✅

事實：CAIS 調查將 standalone LLM 的記憶、即時 grounding、工具使用局限列為複合系統興起的動因 Chaudhry et al., 2025。Kandogan 藍圖列出企業落地障礙：既有應用整合、專有資料/API、成本與延遲 Kandogan et al., 2025。Krishnan 綜述呼應同一趨勢 Krishnan, 2025。

推論：Frontier model capabilities 是必要非充分條件；架構層決定能力能否在 SLA 內穩定外化。Thinkata 與 Prabhakar 的產業評述將此描述為分層 stack 的演化 Thinkata, 2025; Prabhakar, 2025。

論點二：Stream 式編排比「框架選型」更具長期性 ✅

事實：Kandogan 2024/2025 將 stream 定義為 orchestration 的一等公民 Kandogan et al., 2024; Kandogan et al., 2025。Gradientsys 以 MCP typed stream + SSE 實作可觀測調度 Song et al., 2025。

推論：stream + registry 抽象更貼近企業 event/log 基礎設施。Prabhakar 2025 的 planner model 可視為 stream 上的 planner 實例化 Prabhakar, 2025。

論點三：多代理是「按需引入」的加速器，不是預設架構 ✅

事實：Anthropic 建議僅在簡單方案不足時增加 agentic 複雜度 Anthropic, 2024；multi-agent 可能消耗 10–15× token。Research 系統證明 orchestrator-workers + 並行工具可帶來數量級延遲改善 Anthropic, 2025。Gradientsys 在 GAIA 上驗證 ReAct 調度效益 Song et al., 2025。

推論：編排層應內建 effort scaling，否則 frontier 能力會以成本與故障模式外溢。

論點四：基座 instruction 能力與編排層必須 co-design ✅

事實：Instruction mixing 顯示不同指令類型組合對下游任務有非對稱影響 Wang et al., 2023；WizardLM 證明複雜指令資料可系統性提升 follow 能力 Xu et al., 2023。Roffo 將 RAG、ReAct、LangChain 與 fine-tuning 並列為整合工具箱 Roffo, 2024。

推論：Agent 編排假設的規劃與工具選擇能力，受 基座微調混合 制約；registry 應記錄 instruction profile 與 eval 成績。

一家之言

[李飛飛主筆] 以人為中心的複合式 AI，不是更複雜的 AI

我從 ImageNet 的經驗學到：基準與資料管線會重塑整個領域，但技術必須服務人類福祉。當我閱讀 Kandogan 的 registry 藍圖與 CAIS 調查時，我看到的不只是架構炫技，而是「誰的資料被註冊、誰被排除在 agent 可觸及的 knowledge 之外」的治理問題。Stream 式編排若缺少 human-in-the-loop 與影響評估，會把 frontier 能力快速放大到客戶、員工與弱勢群體——這與 Bommasani 等人加州 frontier AI 報告強調的 trust but verify 一致 Bommasani et al., 2025。

我認為企業應把 compound AI 視為 社會技術系統：agent registry 不只是 metadata，而是「能力與責任的公開清單」。Anthropic 強調 transparency 與可觀測 planning steps Anthropic, 2024，正是 human-centered AI 在編排層的最低要求。若你只做 planner 而不做 影響評估與弱勢資料覆蓋審計，你只是在更快的自動化——不是更負責任的 AI。

[Yann LeCun 主筆] LLM 是工具，世界模型才是下一跳——但編排仍不可少

我會直說：很多「agent 框架」只是 prompt 工程的包裝。LLM 擅長接龍，不等於理解世界狀態；JEPA/世界模型路線才是長期智能。但這不否定 Kandogan stream 與 Gradientsys MCP 的價值：在當前產品週期，我們仍需要 可組合、可觀測 的 compound 系統把現有 LLM 用到極致。

我特別同意 Anthropic「先簡單後複雜」與 Wang 等人 instruction mixing 的警示：不要以為換更大的 Llama/GPT 就能跳過資料與架構工作。開源權重 + registry 可讓企業在 planner 層做 A/B——這比封閉 monolith 更符合創新節奏。若你的編排不能 swap 基座、重跑 eval，那不是 architecture，是 vendor lock-in。

綜合判斷（合著）：2026 年的贏家不是「最多 agent 的 stack」，而是 stream 可移植 + registry 可問責 + planner 可證明 ROI 的 stack；frontier capabilities 是燃料，編排是引擎，human-centered governance 是方向盤。

實踐建議

  1. Phase 0 選題對齊：新研究系列先跑 paper harvest（≥20 候選），用 negative_topics 去重，避免與 KB 既有 trust/scheming 專題撞題。
  2. MVP 用單 agent + RAG：依 Anthropic 建議，先 prompt chaining / routing 證明業務指標，再引入 orchestrator-workers Anthropic, 2024。
  3. 引入 stream 抽象：對照 Kandogan blueprint，將 chat、tool result、planner decision 統一為可記錄 stream Kandogan et al., 2025。
  4. 建立 agent/data registry：記錄模型版本、instruction mix、eval 分數與 QoS；planner 路由應讀 registry 而非 hardcode endpoint。
  5. 並行要有 budget：參考 Anthropic Research 的 parallel subagent + parallel tools，但設定 subagent 上限與 token budget Anthropic, 2025。
  6. 觀測 Gradientsys 式 SSE：生產環境暴露 agent 活動與 retry/replan 事件 Song et al., 2025。
  7. 對齊 GenIR 雙范式：區分 generation vs synthesis 任務，選 RAG 或 fine-tune 路徑 Ai et al., 2025。
