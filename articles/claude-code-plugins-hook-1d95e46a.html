Claude Code Plugins 的企業分發與 hook observability：marketplace 治理、設定分層與 telemetry 契約

技術概述

Claude Code 的 plugin 系統已把 commands、agents、skills、hooks 與 MCP servers 收斂為可版本化、可分享的目錄單位，真正的工程難點不在「怎麼做出 plugin」，而在「怎麼安全分發、怎麼統一啟用、怎麼觀測 rollout 後的行為」。本研究聚焦這三件事：團隊如何以 marketplace 與 settings 分層分發 plugin、hooks 在執行期實際能攔截哪些事件，以及如何把這些事件映射到 OpenTelemetry，讓 plugin 擴充不只是可安裝，還能被稽核、比較與除錯。[來源：Plugins, 2026] [來源：Plugin marketplaces, 2026] [來源：Monitoring, 2026]

研究背景

今天完成的 Todoist 任務可辨識出兩個直接相關的技術脈絡：Superpowers / oh-my-claudecode 代表社群化 skill 與 plugin 生態，Claude Code CLI hooks、slash commands 與 skills 代表官方擴充面。延伸後可整理出今日任務涉及技術：Claude Code、plugins、skills、hooks、slash commands、subagents、marketplaces、OpenTelemetry observability。近 3 天內 registry 已有 Claude Code CLI hooks、slash commands 與 skills 的擴充架構，因此本次改選更偏實作治理的角度，避免與既有機制筆記重疊。

核心發現

第一，Claude Code 官方已把 plugin 當成團隊級分發單位，而不是單純的本機客製化資料夾。plugin.json 定義 plugin 身分與版本，marketplace.json 定義可被發現與安裝的目錄；團隊則可在 .claude/settings.json 用 extraKnownMarketplaces 與 enabledPlugins 把 marketplace 與 plugin 啟用狀態隨 repo 一起分發。[來源：Plugins, 2026] [來源：Plugin marketplaces, 2026] [來源：Claude Code settings, 2026] 這代表 plugin rollout 的最小治理單位應是「repo 設定 + marketplace 來源 + plugin 版本」，而不是文件口頭約定。已驗證✅

第二，hooks 應視為 plugin rollout 的治理平面，而不是只拿來跑格式化腳本。Claude Code hooks 可配置在使用者、專案、本機專案與 enterprise managed policy；plugin hooks 啟用後會與既有 hooks 合併，且同一事件下的多個 hooks 會並行執行。[來源：Hooks reference, 2026] [來源：Claude Code settings, 2026] [來源：Plugins reference, 2026] 由於 hooks 能附著在 PreToolUse、PostToolUse、SubagentStop、Notification 等事件，實務上最有價值的用途是做 preflight policy、結果檢查、審計記錄與 rollout guard，而非把業務流程硬塞進 hook script。已驗證✅

第三，若要讓 plugin 生態可維運，光靠 hook script 日誌不夠，至少要把 Claude Code 原生 telemetry 與 hook 自訂事件接成同一條 OTel 管線。Claude Code 已支援匯出 metrics 與 logs/events，內建事件至少包含 claude_code.tool_result、claude_code.api_request、claude_code.user_prompt，且可透過 managed settings 在組織層統一開啟與設定 OTLP 端點。[來源：Monitoring, 2026] [來源：Claude Code settings, 2026] OpenTelemetry 的 GenAI semantic conventions 則提供模型請求、token、錯誤與 span 屬性的通用欄位；agent spans 規格進一步把 workflow、task、tool execution 這類多步驟代理行為標準化。[來源：Semantic conventions for generative AI systems, 2026] [來源：Semantic conventions for generative client AI spans, 2026] [來源：Monitoring, 2026] 實作上最穩定的方式不是讓 hook 直接自創一套 schema，而是把 hook 產生的事件補充到既有 OTel logs/spans，保留 session.id、tool_name、success、duration_ms 等內建欄位，再額外加入 plugin 與 matcher 資訊。已驗證✅

第四，設定分層本身就是 rollout 風險控制機制。Claude Code settings 的優先序是 enterprise managed policy、CLI 參數、本機專案、共享專案、使用者設定；企業政策不可被覆寫。[來源：Claude Code settings, 2026] [來源：Hooks reference, 2026] [來源：Monitoring, 2026] 因此團隊不應把安全需求只寫在 README，而應把必要 marketplace、telemetry env 與禁止覆寫的安全設定放進 managed settings 或至少共享專案設定，再把個人實驗性 plugin 留在 user/local scope。已驗證✅

深度分析

從結構上看，Claude Code plugins 比早期只靠 slash commands 或本機 skills 更適合團隊協作，因為它把「可分享元件」與「安裝來源」分開建模。plugin 本體是版本化目錄，marketplace 是分發目錄；兩者分離後，團隊就可以在同一 repo 上固定 marketplace 來源，在不同專案啟用不同 plugin 組合。[來源：Plugins, 2026] [來源：Plugin marketplaces, 2026] 這種模型與套件管理很像，但更偏工作流治理，因為 plugin 不只是函式庫，還會把 hooks、agents、skills 與 MCP server 一起帶進執行環境。[來源：Plugins reference, 2026]

真正需要補強的是 observability。官方 monitoring 頁面已提供 Claude Code 本體的 metrics 與 event schema，例如工具執行結果、token 使用、成本與 session 指標。[來源：Monitoring, 2026] 但 plugin hook 自己做的行為，例如 PostToolUse 後再跑 linter、在 SubagentStop 後寫審計紀錄，若沒有額外欄位，就很難從後端區分「Claude 原生事件」與「plugin 附加行為」。因此建議每個 plugin hook 至少補四類欄位：plugin.name、plugin.version、hook.event_name、hook.matcher。若是 rollout 驗證，再補 rollout.scope、rollout.marketplace、rollout.decision。這些欄位不必脫離 OTel，而是作為 log attributes 或 span attributes 附加在既有 event 上。[來源：Monitoring, 2026] [來源：Semantic conventions for generative AI systems, 2026] [來源：Semantic conventions for generative client AI spans, 2026]

另一個常見誤區是把所有 plugin hook 都視為可串行推理步驟。官方 hooks 文件明確指出多個相符 hooks 會平行執行。[來源：Hooks reference, 2026] 這意味 hook 腳本應盡量短、冪等、具 timeout，且避免隱性共享狀態。若團隊把部署、格式化、發通知與風險掃描都綁在同一事件上，沒有清楚的 timeout 與失敗語義，就會導致 rollout 後的失敗來源難以定位。對 daily-digest-prompt 這種多階段代理系統來說，最實際的做法是讓 hook 只做 guard、emit event、觸發輕量檢查；真正的重工作仍回到獨立工具或子代理。

最佳實踐與常見陷阱

最佳實踐方面，第一是把 plugin 分發切成三層。共享 .claude/settings.json 只放團隊必要的 marketplace 與正式 plugin；.claude/settings.local.json 留給個人試驗；enterprise managed policy 只放不可繞過的安全、telemetry 與網路治理。[來源：Claude Code settings, 2026] [來源：Plugin marketplaces, 2026] 第二是把 hook 當 policy 與 audit 平面，對 PreToolUse、PostToolUse、SubagentStop 設計明確責任，而不是一個 script 包全部。[來源：Hooks reference, 2026] 第三是 rollout 前後都要驗證：安裝後確認 /plugin marketplace list、/plugin、/help，執行時再看 OTel 是否出現 tool_result 與自訂 hook 事件。[來源：Plugin marketplaces, 2026] [來源：Plugins, 2026] [來源：Monitoring, 2026]

常見陷阱則有四個。第一，直接在 plugin 內寫絕對路徑，官方 reference 已指出路徑應保持相對並善用 ${CLAUDE_PLUGIN_ROOT}。[來源：Plugins reference, 2026] 第二，把 user scope plugin 混進團隊共享 repo，造成他人無法重現安裝結果。[來源：Claude Code settings, 2026] 第三，只開啟 Statsig/Sentry 這類產品層 telemetry，卻沒有把 OpenTelemetry logs 與本地 hook 行為接上，最後只能看到使用統計，看不到 rollout 故障點。[來源：Data usage, 2026] [來源：Monitoring, 2026] 第四，忽略 OTel GenAI conventions 仍在演進中的穩定性標記；若團隊自行落地，應先固定使用的 convention 版本與欄位集，避免後續 dashboard 與查詢語句破裂。[來源：Semantic conventions for generative AI systems, 2026]

與本專案的關聯

對 daily-digest-prompt，這份研究最直接的落點有三個。第一，可把目前 repo 內散落的 hooks、skills 與 slash commands 打包為內部 plugin marketplace，讓新成員只需信任 repo 即可安裝必要工具。第二，可在 .claude/settings.json 顯式宣告核心 plugin 與 marketplace，避免不同機器有不同啟用集合。第三，可把現有 JSONL/OTLP 流程往前延伸：Claude Code 原生 event 進 OTel，plugin hooks 再補充自訂欄位，最後以 task_key、plugin.name、hook.event_name 做 dashboard 與告警。

程式碼範例


{
  "enabledPlugins": {
    "team-guards@daily-digest": true,
    "research-tools@daily-digest": true
  },
  "extraKnownMarketplaces": {
    "daily-digest": {
      "source": {
        "source": "github",
        "repo": "your-org/claude-plugins"
      }
    }
  },
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_LOGS_EXPORTER": "otlp",
    "OTEL_EXPORTER_OTLP_ENDPOINT": "http://otel-collector:4317"
  }
}


{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/emit_hook_event.ps1",
            "timeout": 15
          }
        ]
      }
    ]
  }
}

上述做法的重點不是 script 本身，而是讓 hook 事件能補上 plugin.name=team-guards、hook.event_name=PostToolUse、hook.matcher=Write|Edit，並與 claude_code.tool_result 共享同一條觀測管線。

知識缺口與未來方向

目前官方 monitoring 文件偏重 Claude Code 本體的 metrics 與 events，對 plugin hook 自訂欄位如何命名、是否有官方建議 schema，查無此資料。因此若團隊近期要落地，應先在內部定義最小欄位契約與版本，未來再視 OpenTelemetry GenAI/agent conventions 穩定化後收斂。下一步值得深化的方向是 optimization：plugin rollout 的版本釘選、逐步啟用、失敗回退與觀測 SLO。
