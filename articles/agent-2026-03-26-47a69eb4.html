摘要（200 字）

本簡報基於 system-insight 2026-03-25 數據與知識庫既有研究，提出 Agent 執行效能的四維監控與治理框架。核心發現：多維度指標呈現分化態勢 — Skill 覆蓋率超標（101.6%，實際 65 個 Skill vs 索引 64 個）反映能力擴張，但快取命中率持續低迷（23.8% vs 目標 40%）、I/O 預算超標 107.7%（24,925 chars vs 目標 12,000）達 emergency 等級，顯示資源效率與 Context 壓力嚴峻。任務公平性持續改善（0.78→0.664）但仍超標 32.8%。建議：優先執行 Context I/O 壓縮工作流，並行調優快取 TTL 與公平性權重，建立跨指標關聯分析的閉環治理機制。

關鍵洞見

洞見 1：Skill 覆蓋率 101.6% 顯示索引滯後，非真正超標

現象：

  • 實際使用 65 個 Skill，但 SKILL_INDEX.md 記載 64 個，覆蓋率計算為 65/64 = 101.6%
  • 這不是「過度使用」，而是索引未同步更新

證據：

  • system-insight.json："unique_skills_detected": 65, "skill_index_total": 64
  • JSONL 日誌統計顯示 65 個不同 Skill 被呼叫

行動建議：

  • 執行 uv run python tools/skill_registry_sync.py 掃描所有 SKILL.md frontmatter，更新 skills/SKILL_INDEX.md
  • 驗證：下次 system-insight 執行後覆蓋率應回歸 100% 左右

---

洞見 2：快取命中率 23.8% 持續低迷，TTL 設定可能過於保守

現象：

  • 近 7 天快取命中率僅 23.8%（752 hits / 3155 total），遠低於 40% 目標
  • 較上次（23.4%）微幅改善 +1.7%，但改善速度緩慢

潛在根因（需用 cache-optimizer Skill 驗證）：

  1. TTL 過短：部分 API 的快取時效設定過於保守（如 60min），導致頻繁過期
  2. 動態參數過多：查詢參數變化大（如 Todoist 每次篩選條件不同），難以命中
  3. 冷啟動頻繁：排程重啟清空記憶體快取（若有）

行動建議：

  • 執行 cache-optimizer Skill 分析各端點命中率，識別低命中端點
  • 針對低命中端點調整 config/cache-policy.yaml TTL（如 knowledge_query 60min → 180min）
  • 監控：下次 system-insight 驗證命中率是否提升至 30%+

---

洞見 3：I/O 預算超標 107.7% 達 emergency，Context 壓力極大

現況：

  • avg_io_per_call 達 24,925 chars，超標 107.7%（目標 12,000 chars）
  • 較上次（24,408）惡化 +2.1%，持續惡化趨勢
  • 風險：接近 Claude Code Context Window 上限，可能觸發壓縮或失敗

證據：

  • system-insight.json alerts: "level": "critical", "severity": "emergency"
  • recommendations: "立即執行 workflows/context-io-budget-control-workflow.yaml"

行動建議（最高優先級）：

  1. 立即執行：workflows/context-io-budget-control-workflow.yaml（Delegation + Summary 雙策略）
  2. Delegation：重量級操作（讀取 5+ 檔案、複雜分析）委派子 Agent
  3. Summary：主 Agent 只接收摘要 JSON（≤200 行），不直接累積大量檔案內容
  4. 驗證：下次 system-insight 監控 avg_io_per_call 是否降至 <18,000 chars

關聯影響：I/O 超標也可能拖累快取效率（大量 Read 操作繞過快取層）

---

洞見 4：任務公平性持續改善但仍超標，需微調權重

現況：

  • auto_task_fairness 0.664，較上次（0.78）改善 -14.9%，但仍超標 32.8%（目標 <0.5）
  • 顯示 round-robin 機制逐步恢復，但仍有不均

潛在根因：

  • 部分任務因依賴缺失或時段限制反覆跳過（如 gmail 服務未啟動）
  • 權重設定可能不當（如研究類任務權重過高）

行動建議：

  • 執行 task-fairness-analyzer Skill 診斷剩餘不均根因
  • 檢查 config/frequency-limits.yaml 是否有任務 allowed_days 限制過嚴
  • 微調 config/scoring.yaml 權重或調整 round-robin 優先級邏輯

---

洞見 5：跨指標關聯分析顯示連鎖效應

發現：

  1. I/O 超標 → 快取繞過：大量 Read 操作可能繞過 api-cache 層，降低快取命中率
  2. 公平性改善 → 執行次數增加 → I/O 壓力上升：更多任務被執行，累積 I/O 增加
  3. Skill 覆蓋擴張 → 工具呼叫多樣化 → 快取分散：新 Skill 引入新 API 端點，稀釋快取命中率

啟示：

  • 優化不能單點突破，需要系統性協調
  • 優先處理 I/O 預算（最緊急），再優化快取（次要），最後微調公平性（已改善中）
  • 建立跨指標監控儀表板（system-insight 已提供基礎，可視化待補強）
