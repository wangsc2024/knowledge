技術研究：Cursor CLI 非互動 agent（`agent -p`）沙箱、`sandbox.json` 與無頭寫入邊界（2026-05-03）

技術概述

Cursor CLI 讓開發者在終端機直接驅動與編輯器同源能力的 AI agent，支援互動式 agent 與非互動 agent -p（--print）兩類場景；後者典型用途是 PowerShell 排程、run-todoist-agent-team.ps1 Phase 2、或 CI 內嵌的自動化任務檔 [來源: Cursor CLI overview, 2026]。與「純聊天 API」不同，CLI agent 會實際呼叫讀寫檔案、Shell、以及（若已設定）MCP 工具；因此產品方以沙箱界定「技術上允許的副程序邊界」，並以互動核准或旗標（如 --force、--approve-mcps）處理政策上是否自動放行 [來源: Implementing a secure sandbox for local agents · Cursor, 2026]。相較之下，OpenAI Codex CLI 以另一套命名（workspace-write、danger-full-access）與 approval_policy（never／on-request 等）描述類似雙層控制，且官方文件明確區分「沙箱邊界」與「核准策略」不得混為一談 [來源: OpenAI Codex — Sandbox concepts, 2026]。本專案同時面對 Windows 本機排程、多後端路由與 AGENTS.md 對 CLI／MCP 的邊界說明，理解 Cursor 這條鏈路的預設與可調參數，是降低 Phase 2 假失敗與過度放寬權限的關鍵。

核心概念與原理

1. 非互動 -p 並不等於「自動寫入」

官方中文〈參數〉說明：--print「可存取所有工具，包括 write 和 shell」，但無頭文件另段指出：在腳本中要實際改檔需搭配 --force（--yolo 同義），否則「僅提議變更」[來源: 參數｜Cursor Docs（zh-Hant）, 2026；使用 Headless CLI（cn）, 2026]。這解釋了常見誤解：以為 -p 在 CI 內必然落盤，結果只得到文字建議而無檔案 diff。原理上，無頭模式缺少互動式「套用變更」確認，故產品以 --force 顯式承擔風險，與互動模式的人類按鈕對齊。

2. sandbox.json：路徑、網路與合併優先序

sandbox.json 可置於 ~/.cursor/sandbox.json（使用者全域）與 repo 根目錄 /.cursor/sandbox.json（工作區）；兩者並存時工作區優先合併。欄位層級上，type 預設 workspace_readwrite，可降到 workspace_readonly 或 insecure_none（完全關閉沙箱）；networkPolicy 以 default、allow[]、deny[] 控制出站，且 deny 永遠贏過 allow；私有網段與 metadata IP 預設封鎖以降低 SSRF 面 [來源: sandbox.json Reference, 2026]。此外，.cursor/*/.json、.git/config、.cursorignore 等路徑恒為寫入保護；.cursor/rules/、skills/ 等子目錄可寫，直接對應本 repo 的 Skill／規則治理目錄結構 [來源: sandbox.json Reference, 2026]。

3. 跨平台實作與 Windows 路徑

工程部落格說明：macOS 以動態生成的 Seatbelt profile 限制整棵子程序樹；Linux 以 Landlock + seccomp，並對 .cursorignore 做「不可讀寫的映射」；Windows 則在 WSL2 內執行 Linux 沙箱，原因是通用開發工具鏈的可支援原生 Windows 沙箱仍不足，官方與 Microsoft 合作推進中 [來源: Implementing a secure sandbox for local agents · Cursor, 2026]。Microsoft Learn（zh-tw）定義 WSL2 為輕量 VM 內的完整 Linux 核心，與 Windows 共用部分資源但隔離 PID／掛載等命名空間 [來源: 什麼是 WSL／WSL2, Microsoft Learn, 2025]。因此在本專案「純 pwsh + 本機 repo」場景，若 Cursor 終端沙箱依賴 WSL2，未安裝或未啟用 WSL2 可能導致與文件敘述不一致的邊界行為，需以實機 agent about／實際寫檔測試佐證而非臆測。

4. 與 Codex 威脅模型的對照（三角佐證中的一邊）

Codex 文件將沙箱描述為「讓 agent 在受限環境內自主行動」；當指令超出邊界才進入核准流程；並列出 read-only、workspace-write、danger-full-access 與 approval_policy 的組合矩陣 [來源: OpenAI Codex — Sandbox concepts, 2026]。社群曾回報 workspace-write + never 下 apply_patch 與 shell 寫檔皆 failed in sandbox 的案例（Linux 主機；issue 已關閉），顯示非互動旗標互動仍可能踩到實作／發版邊角 [來源: GitHub openai/codex#5824, 2025-2026]。與 Cursor 對照：兩者都把「技術邊界」與「人／旗標核准」拆開，但參數名與預設不完全 1:1，遷移腳本不可直接複製 Codex 參數字串到 Cursor。

5. MCP 與 --approve-mcps

中文參數表列 --approve-mcps：「自動核准所有 MCP 伺服器」，並有 agent mcp list 等子命令管理設定來源（.cursor/mcp.json）[來源: 參數｜Cursor Docs（zh-Hant）, 2026]。論壇存在「CLI 無法正確觸發 MCP 工具」之類報告（頁面主文未完整展開），與本專案 AGENTS.md「CLI 需在 cli.json 允許 Mcp(...)」敘述可並讀：代表 MCP 能力 = 設定檔允許 ∩ 執行面（IDE vs CLI）支援 [來源: Cursor Community Forum #129387, 2025-08 建立；AGENTS.md 專案內規範]。

6. 可量測的產品宣稱（實驗／營運）

Cursor 工程文宣稱：導入沙箱後，相較未沙箱化 agent「少約 40% 的核准停頓」，並輔以內部 offline eval 調整 Shell 工具錯誤訊息，使模型在權限不足時較少無效重試 [來源: Implementing a secure sandbox for local agents · Cursor, 2026]。此數字屬單一廠商營運樣本，不宜外推為普遍 SLA，但可作為「沙箱 + 更清楚錯誤回饋 → 降低人類中斷」方向的旁證。

最佳實踐

  1. 無頭寫檔：固定帶 --workspace + 明確 --force 策略

情境：Phase 2 寫 results/todoist-auto-*.json。依據：cn Headless 文件寫明無 --force 可能只產生建議 [來源: 使用 Headless CLI（cn）, 2026]。反例：在 production branch 全域 --yolo 與未限制 workspace，等於把「互動確認」整段移除，與沙箱精神相悖。

  1. 在 repo 根放精簡 sandbox.json：預設拒絕外連、白名單套件註冊表

依據：networkPolicy.default: deny 範本與 npm／PyPI 域名白名單示例 [來源: sandbox.json Reference, 2026]。落地：對齊本專案 curl ntfy、Todoist API、Ollama 等，把真實 hostname 列入 allow[]，避免 agent 在沙箱內對合法監控端點無限重試。

  1. 敏感路徑留在 deny／仰賴內建保護

官方已保護 .git/config、.cursorignore 等；若仍要讀 secrets，應以「工作副本不含密文」為原則，而非關閉沙箱 [來源: sandbox.json Reference, 2026]。

  1. MCP：排程任務顯式 --approve-mcps 或事先 agent mcp login

依據：參數表與 mcp 子命令 [來源: 參數｜Cursor Docs（zh-Hant）, 2026]。反例：假設 IDE 內已登入、CLI 排程自動繼承 OAuth state——論壇路線顯示此假設不穩。

  1. 多後端／配額降級時，把 cli_exit_note_zh 與實際旗標寫進結果 JSON

與今日 todoist-history 中 cursor_cli fallback 敘事對齊：後端切換不改變本機沙箱檔案邏輯，但改變的是「哪個二進位在跑 agent」；結果檔應記錄 task_execution 歸因（若 schema 有）以利 check-health [來源: Langfuse 與 daily-digest 串接筆記提及多後端 metadata；本 repo state 觀測]。

陷阱與反例

  1. 混淆「沙箱關閉」與「核准關閉」

Codex 文件與 #5824 顯示：approval never 仍可能在 workspace-write 下被沙箱拒絕寫入 [來源: OpenAI Codex — Sandbox concepts, 2026；openai/codex#5824]。若遷移心智模型到 Cursor，亦應分開看待 --sandbox disabled 與 --force。

  1. Windows 未準備 WSL2 卻預設「與 Linux 同級沙箱」

Cursor 工程文明指 Windows 走 WSL2 內 Linux 沙箱路徑 [來源: Implementing a secure sandbox for local agents · Cursor, 2026]。反例：在純 pwsh、無 WSL 的機器以文件示例推論「一定可沙箱化 npm」。

  1. MCP 在 CLI 失敗時盲目升級到 insecure_none

論壇路徑顯示 CLI MCP 問題可能與版本／程序有關 [來源: Cursor Community Forum #129387]。若直接 type: insecure_none 換取工具可用，等同放棄 networkPolicy 與檔案邊界，應先比對 agent mcp list 與 IDE 設定差異。

與本專案（daily-digest-prompt）關聯

  • Phase 2 任務檔與 agent -p：run-todoist-agent-team.ps1 產生之任務內容與 results/todoist-auto-*.json 守門，應假設無頭模式預設不寫檔，除非腳本層已傳 --force；並以 --workspace 鎖定本 repo，避免落到錯誤工作目錄 [來源: 使用 Headless CLI（cn）, 2026]。
  • 與 ntfy／httpx 研究銜接：先前 tech_research 已覆蓋 ntfy 與 httpx；本主題補上「沙箱 networkPolicy.default: deny 時，必須顯式放行 ntfy.sh 與 KB localhost:3000」否則 Phase 2 內嵌 curl 測 KB 會在沙箱視角與主機實際成功不一致——應以 sandbox.json allow 清單治理 [來源: sandbox.json Reference, 2026]。

程式碼範例

以下為官方文件拼接之可執行示意（實際環境需已安裝 Cursor CLI 並完成驗證；來源：cn Headless + CLI overview）。


# 非互動檢視（可能不套用變更）
agent -p --workspace "D:/Source/daily-digest-prompt" \
  "列出 tools 目錄下與 ntfy 相關的 Python 模組檔名"

# 非互動且允許寫入（謹慎用於自動化）
agent -p --force --workspace "D:/Source/daily-digest-prompt" \
  --output-format text "將 HEARTBEAT.md 第一行加上註解時間戳（若政策不允許請跳過）"
