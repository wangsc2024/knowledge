二、12 Session 技術實作全覽

Session 1：Agent Loop — 「Bash is all you need」

核心模式（約 20 行有效程式碼）：


User Prompt → messages[] → LLM → response
     ↑                                ↓
     |                    stop_reason == 'tool_use'?
     |                    /                        \
     |                  yes                         no
     |                   ↓                           ↓
     +--- execute tools ←                    return text
          append results

關鍵設計決策：

  • 終止條件極簡：if response.stop_reason != 'tool_use': return
  • 僅一個工具定義：bash（name + description + input schema）
  • 工具描述中顯式教導模型常用模式（cat/grep/find/echo/sed/python）
  • subprocess.run(command, shell=True, capture_output=True, text=True) 擷取 stdout + stderr
  • 結果格式化為 tool_result content block 注入對話歷史

核心洞察：「模型佔 80%，程式碼佔 20%」— 複雜的代理行為源於模型智慧，而非架構複雜性。

---

Session 2：Tools — 「The loop didn't change」

核心原則：擴展工具不需要重寫迴圈。

  • 新增工具 = 新增 handler
  • 迴圈結構保持不變（仍是 while + stop_reason 檢查）
  • 工具分發邏輯獨立於核心迴圈

---

Session 3：TodoWrite — 「Plan before you act」

核心原則：可見的規劃改善結果品質。

  • 引入任務規劃機制
  • 模型先產出計畫，再逐步執行
  • 規劃是「工具」而非「系統指令」

---

Session 4：Subagents — 「Process isolation = context isolation」

遞迴子代理模式（自然浮現，無需框架支持）：

  • 父代理透過 bash 工具執行：python v0_bash_agent.py 'subtask'
  • 子程式偵測命令列參數 → 建立 history = []（全新上下文）
  • 子程式輸出印到 stdout → 被父程式的 subprocess 擷取
  • 結果透過正常 tool_result 機制回流

核心洞察：子代理功能不需要特殊框架 — 它自然地從 bash 的程式產生能力浮現。

---

Session 5：Skills — 「Load on demand, not upfront」

三層漸進式揭露（Progressive Disclosure）：

  1. Tier 1（啟動時）：Frontmatter 元資料（name/description），約 100 tokens，用於 Skill 發現
  2. Tier 2（選中時）：完整 SKILL.md 注入，500-5,000 字的詳細指令
  3. Tier 3（執行時）：捆綁資源（腳本、參考文件）透過 Read 工具按需載入

技術實現：

  • Skill 選擇完全依賴 LLM 推理（無演算法、無 embedding、無分類器）
  • 所有可用 Skill 格式化為文字描述放入 Skill 工具的 prompt 中
  • Claude 原生地將用戶意圖與描述匹配
  • 15,000 字元 Token 預算限制可用 Skill 數量

Skill 注入雙訊息設計：

  1. 可見訊息（isMeta: false）：50-200 字元狀態標籤
  2. 隱藏訊息（isMeta: true）：完整 SKILL.md 指令（對 API 可見，對 UI 隱藏）

執行上下文修改：

  • 預核准 allowed-tools 中列出的工具（無需用戶確認）
  • 可選覆蓋模型（如指定 opus）
  • 權限範圍限定於 Skill 執行期間

---

Session 6：Compact — 「Strategic forgetting」

核心原則：上下文壓縮使長 session 成為可能。

  • 策略性遺忘非關鍵訊息
  • 保留結構化狀態（如 TodoWrite 的任務清單）
  • 壓縮演算法需平衡資訊保留與 token 節省

---

Session 7：Tasks — 「State survives /compact」

核心原則：基於檔案的持久化使狀態在上下文壓縮後存活。

  • 任務狀態寫入檔案系統
  • 壓縮後重新讀取檔案恢復狀態
  • 設計模式：檔案是唯一的真實來源（single source of truth）

---

Session 8：Background Tasks — 「Fire and forget」

核心原則：執行緒 + 通知佇列實現非阻塞操作。

  • 啟動背景任務不等待完成
  • 任務完成時透過佇列通知
  • 主迴圈繼續處理其他工作

---

Session 9：Agent Teams — 「Append to send, drain to read」

JSONL 郵箱協議：

  • 儲存路徑：team_inbox///.jsonl
  • 每行一個 JSON：{id, from, text, timestamp, read}
  • 寫入 O(1)（追加）vs Claude Code 的 JSON Array O(N)（全文讀寫）

訊息處理模式：

  1. 寫入：直接 append 到 JSONL 檔案
  2. 讀取：逐行解析，過濾未讀
  3. 標記已讀：僅在 prompt 完成時批次處理
  4. 審計追蹤：JSONL 天然保留完整歷史

---

Session 10：Team Protocols — 「Same request_id, two protocols」

雙層有限狀態機（FSM）：

  1. 成員狀態（粗粒度）：ready → busy → shutdown_requested → shutdown

- 錯誤狀態允許恢復路徑

- guard: true 防止清理期間的競態條件

- force: true 在崩潰恢復時繞過驗證

  1. 執行狀態（細粒度）：約 10 個狀態追蹤 prompt loop 精確位置

- 支援詳細 UI 呈現

- 恢復邏輯可管理

Shutdown 與 Plan Approval：

  • 使用相同 request_id 追蹤請求-回應對
  • 兩種協議（shutdown/plan_approval）共用相同 FSM 基礎設施

---

Session 11：Autonomous Agents — 「Poll, claim, work, repeat」

自組織模式（無需協調者）：

  1. Agent 輪詢任務清單
  2. 聲明鎖（Claim Lock）認領任務
  3. 執行任務
  4. 完成後回到步驟 1

身份持久化（Identity Re-injection）：

  • 每次 prompt 迴圈注入 Agent 身份
  • 避免上下文壓縮後丟失角色認知

---

Session 12：Worktree Isolation — 「Isolate by directory, coordinate by task ID」

核心原則：以目錄隔離執行，以任務 ID 協調。

  • 每個 Agent 在獨立的 Git worktree 工作
  • 避免並行修改同一檔案的衝突
  • 任務完成後合併結果

---

三、跨 Session 設計模式提煉

模式 1：不可變迴圈核心

迴圈結構從 s01 到 s12 從未重寫。所有新功能都是在迴圈外部添加的。

模式 2：檔案作為通訊媒介

從 s07 的任務持久化到 s09 的 JSONL 郵箱，檔案系統是 Agent 間唯一的共享狀態。

模式 3：漸進式揭露

s05 的 Skill 載入體現了「只在需要時載入」的原則。15,000 字元預算迫使精簡化。

模式 4：自然浮現的能力

s04 的子代理不需要框架 — bash 的 subprocess 能力足矣。複雜行為從簡單機制浮現。

模式 5：雙層狀態管理

s10 的粗/細粒度 FSM 模式適用於任何需要生命週期管理的系統。

| s12 Worktree 隔離 | Start-Job 程式隔離（PowerShell 7） |

---

五、可直接採用的改進方向

  1. JSONL 郵箱優化：將 results/*.json 從 JSON 改為 JSONL 追加模式，降低並行寫入衝突風險
  2. Identity Re-injection：在 auto-task prompt 中加入明確的 Agent 身份段落，防止長 session 身份漂移
  3. Claim Lock 機制：為 frequency-limits.yaml 的 round-robin 加入 claim lock，防止並行 session 重複執行同一任務
  4. 雙層 FSM 模式：將 scheduler-state.json 的單一狀態擴展為粗/細粒度雙層，改善除錯體驗
