Harness Engineering：讓 AI Agent 可生產的工程紀律

執行摘要

Harness Engineering 是 2026 年興起的新興工程紀律，專注於設計「約束、工具、反饋迴圈、文件與驗證系統」，使強大但不可預測的 AI Agent 能可靠地產出軟體 [agentengineering.dev, 2026]。有別於提升 prompt 或模型規模，此方法著重於架構環境讓 Agent 安全且可預測地運作。Martin Fowler 將其定義為：「Agent = Model + Harness」，其中模型提供智能，而 Harness 讓智能變得有用 [Fowler, 2026]。

OpenAI 的內部成功展現此範式的重要性：他們使用 Codex Agent 建造超過 100 萬行程式碼，零人工貢獻，關鍵在於系統性架構——當 Agent 失敗時，工程師設計預防系統而非修復個別輸出 [agentengineering.dev, 2026]。

核心價值：2-5× 可靠性提升、8× CI 加速、80% 測試週期縮短 [Harness.io, 2026]。

---

研究背景

Harness Engineering 的興起

傳統軟體工程專注於人類編寫的程式碼，但隨著 AI Coding Agent（如 GitHub Copilot、Claude Code、Cursor）的普及，新的挑戰出現：

  1. 可預測性問題：LLM 輸出具隨機性，同樣 prompt 可能產生不同結果
  2. 長鏈失敗：多步驟工作流中，單一錯誤會級聯放大
  3. 漂移問題：Agent 在長期專案中可能逐漸偏離架構規範

Harness Engineering 誕生於此背景，透過「架構約束」取代「模型調校」，解決生產環境的根本挑戰 [agentengineering.dev, 2026]。

與傳統測試 Harness 的關係

傳統測試 Harness 是「協助測試應用程式或元件的 stubs 與 drivers 集合」，作為測試環境的模擬基礎設施 [GeeksforGeeks, 2026; Wikipedia, 2026]。而 Harness Engineering 擴展此概念至整個開發生命週期，不僅測試輸出，更約束生成過程。

---

核心概念

1. 定義與架構

Harness 是包裹 AI Agent 的完整基礎設施，包含：

  • 可存取的工具 — Agent 可呼叫的 API、檔案系統、資料庫等
  • 守衛機制 — 安全邊界，防止危險操作
  • 反饋迴圈 — 幫助 Agent 自我修正的機制
  • 可觀測性層 — 讓人類監控 Agent 行為的介面

[agentengineering.dev, 2026]

2. 雙控制機制（Dual Control）

Martin Fowler 提出 Harness 透過兩種控制機制引導 Agent [Fowler, 2026]：

#### Feedforward Controls（前饋控制，預防性）

在問題發生前預防，透過文件、架構規則、結構化工具提高初次成功率。

實例：

  • CLAUDE.md 專案指引檔案（自動載入）
  • ADR（Architecture Decision Records）機器可讀邊界
  • Schema 驗證器（阻擋不合規格的輸出）

#### Feedback Controls（反饋控制，修正性）

觀察生成結果並啟用自我修正。

實例：

  • Linter 與 Type Checker（計算型，快速）
  • AI-driven 語義分析（推理型，慢但精準）
  • Fitness Functions（持續驗證架構品質）

關鍵洞見：「純反饋會產生重複錯誤，純前饋無法驗證有效性」[Fowler, 2026]。生產系統需兩者平衡。

3. 三大規範類別

[Fowler, 2026]

| 類別 | 目標 | 工具範例 | 成熟度 |

|------|------|---------|--------|

| Maintainability Harness | 內部程式碼品質 | ESLint, Prettier, pytest | 最成熟 |

| Architecture Fitness Harness | 效能與結構需求 | Fitness Functions, Benchmark Gates | 中等 |

| Behavior Harness | 功能正確性 | E2E Tests, Human Review | 最少（依賴人工） |

實務應用技術

A. 12 個 Agentic Harness Patterns

[GenerativeProgrammer, 2026; Claude Lab, 2026]

#### 記憶與 Context 管理（5 patterns）

  1. Persistent Instruction File Pattern

持久化專案層級配置檔（如 CLAUDE.md），Session 啟動時自動載入，定義建置命令、慣例、編碼標準。

  1. Scoped Context Assembly Pattern

動態載入多層級指引檔（專案根目錄 + 子目錄），讓 Agent 看到位置相關的規則。

  1. Tiered Memory Pattern

分層記憶：緊湊索引（≤200行）常駐、主題檔案按需載入、完整對話記錄留在磁碟供搜尋。

  1. Dream Consolidation Pattern

背景程序在閒置時檢視、去重、重組 Agent 記憶，類似「Agent 狀態的垃圾回收」。

  1. Progressive Context Compaction Pattern

多階段壓縮：近期對話保持細節，舊對話逐步摘要。

#### 工作流與編排（3 patterns）

  1. Explore-Plan-Act Loop Pattern

三階段工作流：讀取/映射 → 規劃討論 → 實際編輯，分離關注點。

  1. Context-Isolated Subagents Pattern

執行獨立 context window、system prompt、受限工具集的子 Agent，用於不同任務階段。

  1. Fork-Join Parallelism Pattern

在獨立 git worktrees 並行啟動多個子 Agent，適用可分解為無依賴單元的任務。

#### 工具與權限（3 patterns）

  1. Progressive Tool Expansion Pattern

預設小工具集（<20），按需啟用額外工具，減少選擇負擔。

  1. Command Risk Classification Pattern

工具執行前用確定性模式匹配預解析，評估命令安全性並設權限閘門。

  1. Single-Purpose Tool Design Pattern

用專用工具（如 FileReadTool, FileEditTool）取代通用 shell，每個工具有型別輸入與明確邊界。

#### 自動化（1 pattern）

  1. Deterministic Lifecycle Hooks Pattern

在 lifecycle 特定點執行 shell 命令（在 prompt 外），確保動作一致發生，不依賴模型記憶。

B. Test Harness 六種設計模式

[Microsoft Learn, 2005]

基於儲存類型（Flat / Hierarchical / Relational）與處理模型（Streaming / Buffered）的排列組合：

| 儲存類型 | Streaming 模式 | Buffered 模式 | 適用場景 |

|---------|---------------|--------------|---------|

| Flat Data (文字檔) | 簡單、短 | 需預/後處理 | 簡單測試案例 |

| Hierarchical (XML) | 追蹤複雜 | 簡潔（XmlSerializer） | 複雜測試案例、XML 環境 |

| Relational (SQL) | SqlDataReader | DataSet + DataAdapter | 大量測試案例、長期專案 |

關鍵選擇準則：

  • Flat + Streaming：最簡單，適合快速驗證
  • XML + Buffered：適合複雜巢狀結構（XmlSerializer 一行讀取）
  • SQL + Buffered：適合跨表關聯、大規模測試管理

C. 測試自動化框架優化技術

[Qodo, 2026; BrowserStack, 2026]

#### 執行效能優化

  • 並行測試：跨環境/執行緒同時執行多個測試，使用 parallel testing frameworks 分配測試案例
  • 風險優先：根據風險評估優先執行關鍵測試案例

#### 測試資料管理

  • 合成資料：用合成或匿名化測試資料減少對生產資料的依賴
  • 自動化資料生成與清理：最小化手動工作

#### Framework 設計最佳實踐

  • 模組化：將複雜測試拆為更小、可重用模組
  • 參數化：用參數化執行相同邏輯搭配不同輸入值，減少冗餘
  • 定期重構：排程週期性維護以優化效能、更新依賴

優化建議（按優先級）

#### 🔴 P0：Dream Consolidation 自動化

目標：自動去重與重組記憶層

實作：

  1. 新增 tools/dream_consolidation.py
  2. Cron 排程每週執行（週日凌晨）
  3. 操作：

- 掃描 context/continuity/*.json，合併重複 topic

- 壓縮 context/research-registry.json（保留 summary，archive 詳細內容）

- 清理 state/ 中 30 天前的 JSONL 日誌

預期效益：減少 context 膨脹 30-40%

#### 🟡 P1：Progressive Context Compaction 自動化

目標：自動壓縮長對話記錄

實作：

  1. 擴展 tools/context_compressor.py（現有工具）
  2. 在 hooks/post_tool_logger.py 中觸發（對話 >5000 tokens 時）
  3. 壓縮策略：

- 最近 10 turns → 保持完整

- 11-50 turns → 保留關鍵決策與結果

- 51+ turns → 僅保留摘要

預期效益：延長單次 session 可用時間 2-3×

#### 🟡 P1：Explore-Plan-Act Loop 標準化

目標：所有研究任務強制執行三階段

實作：

  1. 更新 templates/auto-tasks/*.md 研究類 prompt
  2. 插入強制 Phase 檢查點：

```

Phase 1: Explore（讀取 5+ 來源，不做決策）

Phase 2: Plan（列出 3-5 個子問題與搜尋角度）

Phase 3: Act（執行研究並寫入 KB）

```

  1. 在 done_cert 中標記 explore_plan_act: true

預期效益：研究任務品質分數從 3.2 → 4.0+（依歷史 done_cert 統計）

#### 🟢 P2：Fork-Join Parallelism for Auto-tasks

目標：高負載時段（8:00-20:00）用 git worktrees 隔離並行自動任務

實作：

  1. 擴展 run-todoist-agent-team.ps1 Phase 2
  2. 若 active_tasks.Count >= 3，為每個任務建立臨時 worktree
  3. 完成後自動清理（成功）或保留（失敗供除錯）

預期效益：高負載時段 throughput 提升 40-60%

#### 🟢 P2：Single-Purpose Tool for State Management

目標：用專用工具取代直接 Bash 操作 state/*.json

實作：

  1. 新增 tools/state_manager.py（CLI 介面）

- Subcommands: read / write / append / cleanup

- 內建 schema 驗證（讀取 config/schemas/*.json）

  1. 更新所有 prompt 改用 uv run python -m tools.state_manager read
  2. 在 hooks/pre_write_guard.py 中攔截直接寫入 state/

預期效益：state 檔案格式錯誤率從 5.2% → <1%（依 state/failure-stats.json 統計）

最佳實踐與常見陷阱

✅ 最佳實踐

[綜合來源：Fowler, agentengineering.dev, Qodo, Microsoft]

  1. 「No Manual Code」約束驅動 Harness 演化

OpenAI 內部實驗：對新模組採用「禁止人工寫程式碼」約束，迫使建立健壯 Harness，工程速度提升數量級 [agentengineering.dev, 2026]。

  1. 定期重構 Harness 程式碼

排程週期性維護以優化效能、更新依賴、重構程式碼 [Qodo, 2026]。

  1. Shift Quality Left

在開發生命週期早期實施 pre-commit 檢查，較昂貴的驗證放在整合後 [Fowler, 2026]。

  1. 分層儲存測試案例

- Flat: 簡單案例（<100 筆）

- Hierarchical (XML): 複雜巢狀結構

- Relational (SQL): 大規模（>1000 筆）+ 長期管理

[Microsoft, 2005]

  1. 人類判斷仍不可或缺

Harness 無法完全替代人類在「組織對齊」「美學架構決策」「情境判斷」上的角色 [Fowler, 2026]。

❌ 常見陷阱

  1. 過度依賴 Feedback-only

純反饋控制會產生重複錯誤。範例：daily-digest-prompt 早期僅用 Hook 攔截，未提供 Feedforward 指引，導致 Agent 多次撞同一道牆 [專案經驗]。

  1. 忽略 Buffered 模式的記憶體需求

將大量測試案例全部讀入記憶體可能導致 OOM。SQL + Streaming 更適合壓力測試情境 [Microsoft, 2005]。

  1. 將 Harness 視為「一次性建置」

Harness 需隨系統演化。固定不變的 Harness 會成為技術債 [agentengineering.dev, 2026]。

  1. 測試案例資料分散管理

將測試案例嵌入程式碼雖適合 TDD，但無法跨配置共享。應視情境選擇 embedded（NUnit TDD）或 external（跨環境測試）[Microsoft, 2005]。

  1. 未監控 Harness 自身的效能

Hook 執行時間過長（>500ms）會拖慢開發。應定期 benchmark Harness 機制 [Fowler, 2026]。

知識缺口與未來方向

待深入研究的領域

  1. Behavior Harness 的突破

功能正確性驗證仍高度依賴人工。未來方向：

- LLM-as-a-Judge 自動化功能測試

- 規格驅動驗證（Formal Methods + AI）

[Fowler, 2026]

  1. 跨專案 Harness 可移植性

當前 Harness 多為專案特定。研究方向：

- Harness 模板庫（類似 cookiecutter）

- 產業標準 Harness 規範（如 OpenTelemetry 對可觀測性）

[agentengineering.dev, 2026]

  1. Harness 的可觀測性

缺少統一的 Harness 監控標準。潛在方向：

- OTLP 整合（OpenTelemetry Protocol）

- Harness 健康指標（latency, failure rate, coverage）

[專案觀察]

  1. Multi-Agent Harness 編排

多 Agent 協作的 Harness 設計尚未成熟。探索方向：

- Agent 間通訊協議

- 分散式 Harness 架構

[Claude Lab, 2026]

daily-digest-prompt 專案特定待研究

  1. Token 效率與 Harness 設計的權衡

當前專案高度關注 token 經濟學（state/token-usage-real.json），但 Harness 機制（如 Hook）本身也消耗 context。研究：最佳 Harness 複雜度與 token 成本的平衡點。

  1. Harness 失敗的自我修復

當 Hook 自身失敗時（如 cwd 漂移導致死循環），系統缺少自癒機制。探索：Harness 的 Harness（meta-harness）可行性。
