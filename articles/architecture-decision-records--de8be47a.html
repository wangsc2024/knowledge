Architecture Decision Records (ADR) 方法論完整指南

技術概述

Architecture Decision Records（ADR，架構決策記錄）是一種輕量級文件化方法，用於記錄軟體專案中「架構層級的重要決策」。由 Michael Nygard 於 2011 年在 Cognitect 部落格首次提出，其核心理念是：決策記錄不只是事後補寫的文件，而是團隊思考與溝通的工具。2024-2026 年間，ADR 結合 LLM（大型語言模型）輔助生成、Agent 自動化追蹤等技術，從傳統的手動文件化演進為可機器輔助的「活文件」（Living Documentation）架構。

核心概念與原理

Michael Nygard 原始定義

架構顯著決策（Architecturally Significant Decision）指影響以下面向的決策：

  • 系統結構（Structure）
  • 非功能特性（Non-functional Characteristics）
  • 依賴關係（Dependencies）
  • 介面設計（Interfaces）
  • 建構技術（Construction Techniques）

Nygard 格式（最簡）

每筆 ADR 是一個短文件（1-2 頁），包含五個欄位：

欄位說明
Title簡短描述決策（如「ADR-001: 使用 PostgreSQL 作為主資料庫」）
Context決策的背景脈絡（技術約束、商業需求、團隊能力）
Decision做出的具體決策
Status決策狀態（Proposed → Accepted → Deprecated → Superseded）
Consequences決策的正面與負面後果

MADR 4.0（Markdown Architectural Decision Records）

MADR 是目前最主流的 ADR 模板標準，2024 年 9 月發佈 4.0 版。相比 Nygard 格式，增加了更多結構化欄位：

YAML Front Matter（可選）：

status: accepted
date: 2026-02-24
decision-makers: [Alice, Bob]
consulted: [Carol]
informed: [Dave]

核心段落：

  1. Context and Problem Statement — 描述需要決策的情境
  2. Decision Drivers — 影響決策的力量或考量（如效能、成本、團隊熟悉度）
  3. Considered Options — 評估過的替代方案（含優缺點分析）
  4. Decision Outcome — 選擇的方案與理由
  5. Consequences — 正面與負面結果
  6. Confirmation — 如何驗證決策是否正確實施

可選段落：

  • Pros and Cons of the Options（各方案優缺點詳列）
  • More Information（參考資料、相關 ADR 連結）

模板變體：

  • adr-template.md：完整模板（含所有段落 + 說明）
  • adr-template-minimal.md：僅必填段落 + 說明
  • adr-template-bare.md：完整段落，無說明（直接填寫）
  • adr-template-bare-minimal.md：僅必填段落，無說明

最佳實踐

八大核心實踐

  1. 與程式碼共存版控：ADR 存放在 docs/decisions/ 或 docs/adr/，隨程式碼一起提交 Git。好處：PR 審查時同時審查決策變更。
  1. 使用清晰無術語的語言：避免過度技術縮寫，確保新成員也能理解。
  1. 保持簡潔但有實質內容：每篇 ADR 約 1-2 頁，不要寫成完整設計文件。
  1. 記錄「為什麼」而非只記「什麼」：Context 和 Decision Drivers 比 Decision 本身更重要。
  1. 追蹤決策生命週期：

- Proposed → 提案中，待審議 - Accepted → 已接受，正在實施 - Deprecated → 已棄用（但保留歷史） - Superseded by ADR-XXX → 被新決策取代（保留並標註）

  1. 絕不刪除 ADR：即使決策被推翻，原始 ADR 也只標記為 Superseded，永不刪除。這保留了決策演進的完整歷史。
  1. 在 Code Review 中引用 ADR：PR 審查時引用相關 ADR 編號，建立程式碼與決策的雙向連結。
  1. 定期審查 ADR 有效性：季度或半年一次檢視所有 Accepted ADR，確認是否仍然適用。

大規模組織的 ADR 策略

MADR 4.0 建議用子目錄組織：

docs/decisions/
  backend/
    0001-use-postgresql.md
    0002-adopt-event-sourcing.md
  frontend/
    0001-choose-react.md
  infrastructure/
    0001-kubernetes-deployment.md

好處：各組件獨立編號，避免跨團隊的序號衝突。

LLM 輔助 ADR 生成（2025-2026 趨勢）

Metaprompting 方法論

Equal Experts 提出的 LLM 輔助 ADR 生成流程：

步驟 1：決策核心句提取 團隊將決策精煉為一句話：「我們在 2 月 24 日決定使用 YAML 作為配置格式，因為它支援注釋、層級清晰、且 LLM 理解力優於 JSON。」

步驟 2：Metaprompt 設計 設計一個「生成 prompt 的 prompt」，確保每個決策特定的 prompt 都遵循團隊標準：

  • 唯一識別碼格式（如 ADR-DDP-001）
  • 必填段落（Background、Alternatives、Decision）
  • 語氣與風格規範
  • 專案特定的術語表

步驟 3：多模型交叉審查 用不同 LLM 作為「審查者」（Judge），檢查生成的 ADR 是否有邏輯瑕疵、幻覺引用或不一致的理由。

品質保證守則

  • 參考來源必須驗證：LLM 經常幻覺參考資料，每個連結都必須人工檢查
  • 避免可信度陷阱：AI 生成的文字流暢度容易讓審查者放鬆警惕，必須有明確的「人工審查 checklist」
  • 回溯性文件化加速：最大的價值在於快速補建歷史 ADR — 收集決策陳述後，一個上午可生成數十篇

進階應用場景

  1. 隱式 ADR 推斷：從設計文件、PR 討論、Slack 對話中自動推斷未記錄的架構決策
  2. ADR 品質審查：LLM 檢查已有 ADR 的邏輯一致性與完整性
  3. 矛盾偵測：跨 ADR 集合偵測互相矛盾的決策
  4. 審查者推薦：根據 ADR 內容領域自動建議適合的審查人選

工具生態系統

adr-tools（Nat Pryce）

經典命令列工具，支援 Nygard 格式：

adr init doc/architecture/decisions
adr new "Use PostgreSQL as primary database"
adr list
adr link 5 "Amends" 3

adr-log

從 MADR 檔案自動生成架構決策日誌（索引頁）。

ADR Agent（macromania/adr-agent）

AI Agent 輔助 ADR 撰寫，提供：

  • Chat 互動式 ADR 生成
  • 多 Agent 協作（生成 → 驗證 → 修正）
  • 基於 LLM 的替代方案分析

GitHub + LLM 自動化管線

Iraj Hedayati 提出的 CI/CD 整合方案：

  1. PR 合併時觸發 GitHub Action
  2. 腳本收集 PR 資訊（標題、描述、變更檔案）
  3. 呼叫 LLM API 生成 ADR Markdown
  4. 存入 docs/decisions/ 並提交
  5. 開發者審查後確認或修改

潛在改進方向

  1. 採用 MADR 4.0 模板：將 adr-registry.json 的 entries 同時生成獨立的 Markdown ADR 檔案，便於人類閱讀
  2. ADR 編號標準化：目前用 ADR-001~008 序號，可加入分類前綴（如 ADR-SEC-001 安全類）

# ADR-009: 採用 Hook 機器強制層取代 Prompt 自律

## Context and Problem Statement

Agent 的自律行為（如禁止 `> nul` 重導向）依賴 Prompt 宣告，但 Prompt 自律可被上下文壓縮或指令覆蓋。
需要一種不可繞過的強制機制。

## Decision Drivers

* Agent 曾在上下文壓縮後違反 nul 禁令
* 安全規則不應依賴 LLM 的遵從性
* 需要可審計的攔截記錄

## Considered Options

1. 加強 Prompt 措辭（更多次重複禁令）
2. Hook 機器強制層（PreToolUse/PostToolUse）
3. 外部 Proxy 攔截所有工具呼叫

## Decision Outcome

選擇方案 2：Hook 機器強制層。

因為：
- Claude Code 原生支援 Hooks（無額外依賴）
- PreToolUse 在工具執行前攔截，零延遲
- Python 實作可測試（496 個測試覆蓋）
- 攔截事件自動記錄到 JSONL 結構化日誌

## Consequences

**正面：**
- nul 檔案問題徹底消除
- 所有攔截事件可審計
- 測試覆蓋確保規則正確性

**負面：**
- Hook 執行增加每次工具呼叫約 50ms 延遲
- 規則過嚴可能誤攔合法操作（需持續調整）

## Confirmation

透過 `python hooks/query_logs.py --blocked` 確認攔截記錄；
透過 `pytest tests/hooks/` 確認 496 個測試全通過。
