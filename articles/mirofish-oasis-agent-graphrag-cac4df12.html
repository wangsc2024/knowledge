一、專案概述

MiroFish（666ghj/MiroFish）是一款開源的群體智能預測引擎（Swarm Intelligence Prediction Engine），定位為「簡潔通用的群體智能引擎，預測萬物」。截至 2026 年 3 月已累積 10,200+ Stars、1,100+ Forks，2025 年 11 月創建，增長迅速。

核心理念

與傳統 AI Agent 框架（如 CrewAI、Mastra）不同，MiroFish 不是「讓 Agent 完成任務」，而是「讓數千 Agent 自由互動以預測未來」。其核心方法論是：

  1. 從真實世界擷取「種子資訊」（新聞、政策草案、小說、財務訊號）
  2. 自動建構高保真數位平行世界（Digital Parallel World）
  3. 數千個具有獨立人格、長期記憶與行為邏輯的智能 Agent 自由互動、社會演化
  4. 觀察群體湧現行為（Emergent Behavior）產生預測洞見

這套方法的獨特價值在於：多個獨立心智的分歧、說服、過度反應、結盟——產生真正的「驚喜」和群體模式，這是單一模型難以偽造的。

二、七層技術架構

MiroFish 採用清晰的分層架構：

| 層級 | 技術選型 | 職責 |

|------|---------|------|

| 展示層 | Vue.js 3 + Vite + Element Plus + D3.js 7 | SPA 前端，port 3000 |

| API 客戶端層 | Axios 封裝（simulation.js, report.js, graph.js） | HTTP 通訊抽象 |

| 後端 API 層 | Flask 3.0+ | Blueprints 路由（simulation/report/project/graph/IPC），port 5001 |

| 服務層 | SimulationManager, ReportManager, TaskManager | 業務邏輯核心 |

| 執行層 | Python subprocess + IPC + Zep Graph Memory Sync | OASIS 模擬子程序編排 |

| 外部服務 | OpenAI-compatible LLM + Zep Cloud + OASIS | LLM 推理 + 知識圖譜 + 模擬引擎 |

| 資料層 | File System + SQLite + JSON 狀態檔 | 模擬痕跡與報告儲存 |

技術棧

  • 前端：Vue.js 3, Vite, vue-router 4, D3.js 7, Element Plus
  • 後端：Python 3.11-3.12, Flask 3.0+, uv 套件管理
  • AI：OpenAI SDK 1.0+, zep-cloud 3.13, camel-ai 0.2.78
  • 文件處理：PyMuPDF 1.24+
  • 部署：Docker Compose（Nginx + Python/uv 容器）或 npm run dev 開發模式

三、五階段工作流管線

階段 1：知識圖譜建構（Graph Building）

上傳文件觸發實體抽取，使用 GraphRAG 技術在 Zep Cloud 上建構知識圖譜。產出 graph_id 供後續階段使用。從種子材料中提取實體、關係和時間事實。

階段 2：環境設定（Environment Setup）

OasisProfileGenerator 根據知識圖譜中提取的實體，使用 LLM 推理生成平台特定的 Agent 角色（Twitter CSV 格式、Reddit JSON 格式）。每個 Agent 具有獨立的人格背景、態度、記憶和決策邏輯。

階段 3：模擬執行（Simulation Execution）

run_parallel_simulation.py 啟動 OASIS 子程序，執行雙平台（Twitter + Reddit）並行模擬。自動解析預測需求，動態更新 Agent 時序記憶。產出 actions.jsonl 和 SQLite 模擬痕跡。

階段 4：報告生成（Report Generation）

ReportAgent 實作 ReACT 模式（Reasoning + Acting），配備三個專用工具：

  • insight_forge：深度洞見萃取
  • panorama_search：全景式模擬資料搜尋
  • interview_agents：訪談模擬中的 Agent

透過多輪推理合成結構化 Markdown 報告。

階段 5：深度互動（Deep Interaction）

聊天介面支援與 ReportAgent 對話或批量 Agent 訪談（透過 IPC 通訊）。使用者可以探查因果、測試干預假設。

狀態流轉


ProjectCreated → GraphBuilding → GraphComplete → SimulationCreated → Preparing → Ready → Running → Completed → ReportGenerating → ReportComplete

四、核心元件深度分析

4.1 OASIS 模擬引擎

MiroFish 的模擬引擎基於 CAMEL-AI 團隊的 OASIS（Open Agent Social Interaction Simulations），這是一個高效能社交媒體模擬框架，支援百萬級 Agent 互動模擬。OASIS 提供 Twitter/Reddit 環境實作、Agent 動作生成和 SQLite 痕跡記錄。

4.2 Zep 長期記憶系統

整合 Zep Cloud 作為 Agent 的長期認知記憶層。ZepGraphMemoryUpdater 在模擬執行期間同步活動回知識圖譜。ZepToolsService 提供語義查詢抽象（search_graph, get_entities, get_episodes），供 ReportAgent 存取。

4.3 Knowledge Capsule Layer（知識膠囊層）

2026 年 3 月最新 PR 提出的「Knowledge Capsule Layer」專注於 Agent 記憶生命週期管理，強化群體智能的知識累積與衰減機制。

4.4 SimulationManager

管理模擬生命週期、狀態持久化、檔案操作、透過 SimulationRunner 啟動子程序。

七、與 Daily-Digest-Prompt 系統的借鑑啟示

7.1 群體模擬 vs 多 Agent 任務執行

MiroFish 的多 Agent 是「模擬者」（simulator），而 DDP 的多 Agent 是「執行者」（executor）。但兩者共用的設計模式值得關注：

  • 記憶分層：MiroFish 的 Zep 長期記憶 ↔ DDP 的 digest-memory.json + research-registry.json
  • 狀態機流轉：MiroFish 10 狀態流轉 ↔ DDP 的 FSM（run-fsm.json）
  • 子程序編排：MiroFish 的 Python subprocess ↔ DDP 的 PowerShell Start-Job

7.2 ReACT ReportAgent 模式

MiroFish 的 ReportAgent 使用 ReACT（Reasoning + Acting）+ 三工具模式，與 DDP 的子 Agent 模板驅動架構異曲同工。可借鑑的改進：

  • 為研究類子 Agent 增加 insight_forge 式深度分析工具
  • 引入 interview 機制讓 Agent 之間交叉驗證結論

7.3 GraphRAG 知識建構

MiroFish 將文件 → 知識圖譜 → Agent 背景的流程，可啟發 DDP 的知識庫匯入優化：

  • 匯入筆記時自動抽取實體關係
  • 建立跨筆記的知識網路（目前筆記間是孤立的）

7.4 Knowledge Capsule Layer

記憶生命週期管理（累積、衰減、壓縮）是 DDP 可直接借鑑的方向，特別是 research-registry.json 的 7 天 TTL 可以進化為更智慧的「知識膠囊」衰減模型。

八、限制與風險

  • Windows 實驗性：macOS 為主要平台，Windows 支援尚不完善
  • Token 成本：大規模模擬（2000+ Agent × 40 輪）的 LLM 呼叫成本可觀
  • 驗證困難：群體湧現行為的預測準確度難以量化驗證
  • OASIS 依賴：核心模擬引擎依賴 CAMEL-AI 的 OASIS 框架
