LLM 代理系統的失敗感知架構：從負軌跡訓練到自癒編排的三層整合設計原則

我以Yann LeCun視角和你聊，基於公開言論推斷，非本人觀點。（Co-primary: distill-yann-lecun）

本報告由兩位主筆的視角共同形塑：李飛飛從 Human-centered AI 的視角強調，代理系統的失敗不是技術噪音，而是影響真實人類福祉的可量化風險——數據資源的浪費與部署可靠性的缺失都需要被嚴肅對待。Yann LeCun 則從架構批判的立場補充：LLM 代理的脆弱性根植於自回歸語言模型缺乏世界模型（world model）的本質缺陷，任何不觸及這一根因的修補都是「在沙上建樓」。兩種視角的張力恰好界定了本報告的論證空間：不否認 LLM 代理的當下價值，但必須以架構誠實性面對其失敗模式。

TL;DR

大型語言模型代理在多步規劃與工具呼叫任務中的失敗率長期被低估：推理錯誤佔所有失敗的 61.8%，工具使用錯誤佔 21.1%（Where LLM Agents Fail, 2025）。問題的根源是當前開發實踐中的「成功偏差」——系統設計者將失敗軌跡視為廢棄資料，而非訓練訊號。本報告主張三層整合框架：訓練層（負軌跡微調）→ 編排層（重試-重規劃機制）→ 自我修正層（迭代自癒）。實證顯示自癒架構可達 98.8% 任務成功率，遠超純重試架構的 94.5%（Self-Healing Agentic Orchestrators, 2026）。失敗感知設計不是補丁，而是 LLM 代理系統走向生產可靠性的架構前提。

研究問題與範疇

核心問題

大型語言模型代理（LLM Agents）在工具使用、多步規劃、指令遵循等任務中普遍存在系統性失敗，但現有架構設計和訓練範式如何系統性處理這些失敗？「失敗感知」（failure-aware）的訓練與編排架構是否能顯著提升代理的生產可靠性？

  • 納入：LLM 代理的失敗模式分類、負軌跡訓練方法、編排層重試/重規劃機制、多層次自我修正架構、多代理系統的韌性設計
  • 排除：代理安全性/欺騙行為（已有專題報告）、強化學習代理（非 LLM 核心）、純推理基準評測（無工具呼叫場景）
  • 時間範疇：2023-2026 年（LLM 代理範式成熟期）

文獻探討

失敗模式的系統性分類

2025 年的 AgentDebug 框架（"Where LLM Agents Fail and How They Can Learn From Failures", arXiv 2509.25370）建立了迄今最全面的 LLM 代理失敗分類法：推理錯誤（61.8%）、工具使用錯誤（21.1%）、指令遵循錯誤（11.5%）、其他（5.6%）。這一分類揭示了失敗的高度集中性——超過六成源自 LLM 核心推理能力，而非工具 API 設計問題，這直接指向訓練層干預的必要性。

LifelongAgentBench（2025）在持續學習場景中識別出四類失敗模式：格式違規、上下文溢出、承諾失敗、最終提交錯誤——後兩者均與代理缺乏「失敗感知」的恢復機制直接相關。

負軌跡訓練：失敗作為訓練訊號

Wang 等人（2024）的奠基性工作 "Learning From Failure: Integrating Negative Examples when Fine-tuning Large Language Models as Agents"（arXiv 2402.11651）首次系統論證：失敗軌跡包含成功軌跡無法覆蓋的錯誤空間信息。他們提出的 NAT（Negative-Aware Tuning）方法通過在微調時加入前綴/後綴標記，告知模型應生成成功或失敗軌跡，在數學推理、多跳問答和策略問答三類任務上均取得大幅提升。

2025 年的 "Co-Evolving Agents: Learning from Failures as Hard Negatives"（arXiv 2511.22254）進一步強化這一思路，引入一個專門處理失敗軌跡的輔助失敗代理（failure agent），將其精煉為「硬負例」（hard negatives），並通過兩個代理的協同演化（co-evolution）持續提升系統整體表現。

"Trial and Error: Exploration-Based Trajectory Optimization for LLM Agents"（arXiv 2403.02502, 2024）則從探索角度出發，系統地將試錯軌跡（包括失敗路徑）整合進代理優化，展示了失敗軌跡在探索空間中的導航價值。

編排層的失敗處理機制

Gradientsys（arXiv 2507.06520, Song et al., 2025）是目前設計最完備的多代理調度框架之一，其核心貢獻是將重試-重規劃（retry-and-replan）機制整合進 ReAct 規劃迴路：當子代理失敗時，調度器可選擇重試同一動作、切換工具、或觸發完整重規劃，並在 GAIA 基準上以更低延遲和 API 成本超越 MinionS 基線。

2026 年的 "Self-Healing Agentic Orchestrators for Reliable Tool-Augmented Large Language Model Systems"（arXiv 2606.01416）提供了不同恢復策略的直接比較：自癒架構（98.8%）> 純重試（94.5%）> 完整重規劃（93.8%）> 靜態工作流。這一結果反直覺——完整重規劃反而劣於純重試——說明重規劃引入的額外 LLM 調用本身是故障點。

"From Agent Loops to Structured Graphs: A Scheduler-Theoretic Framework for LLM Agent Execution"（arXiv 2604.11378, 2026）則提出形式化的調度理論框架，允許代理在失敗時自主選擇重試、跳過或重規劃，而無需預設嘗試上限，為動態失敗處理提供了理論基礎。

自我修正的能力邊界

"Can LLMs Correct Themselves? A Benchmark of Self-Correction in LLMs"（arXiv 2510.16062, 2025）的結論頗具衝擊性：未有外部反饋訊號時，LLM 的自我修正能力極為有限。這與 "Confidence v.s. Critique: A Decomposition of Self-Correction Capability for LLMs"（arXiv 2412.19513, 2024）的分析一致——LLM 的自我批評（critique）能力優於置信度校準能力，但仍無法可靠地獨立修正錯誤。

2024 年的 "Can LLMs Learn from Previous Mistakes? Investigating LLMs' Errors to Boost for Reasoning"（arXiv 2403.20046）在推理任務上的研究顯示，LLM 能夠在上下文中吸收前次錯誤的隱式教訓，但效果高度依賴錯誤的呈現方式與任務類型。

多代理韌性與拓撲設計

"On the Resilience of LLM-Based Multi-Agent Collaboration with Faulty Agents"（arXiv 2408.00989, 2024）提供了拓撲結構影響韌性的量化證據：層次化架構（hierarchical topology）在代理出現故障時性能下降僅 5.5%，遠優於線性（10.5-23.7% 下降）與全連接拓撲。這直接支持了 Gradientsys 和 Context Kubernetes 所採用的中心調度架構。

Context Kubernetes（arXiv 2604.11623, Mouzouni, 2026）雖以知識編排為主要訴求，但其三層代理許可權模型（agent authority ⊂ human authority）和 TLA+ 驗證（4.6M 狀態零違規）提供了可靠性保障的形式化基礎。

核心論證

主張：失敗感知三層架構是 LLM 代理從實驗室走向生產的必要條件

本文認為，當前 LLM 代理系統的主要可靠性瓶頸不是模型能力本身，而是系統設計中對失敗的系統性忽視。論證分三層：

層 1 — 訓練層的成功偏差問題

傳統代理微調幾乎清一色使用成功軌跡。Wang 等人（2024）指出這造成了訓練空間的嚴重扭曲——失敗軌跡所覆蓋的錯誤模式是成功軌跡無法觸及的，因此微調模型在面對類似失敗情境時無法正確判斷「這是一條應當規避的路徑」。NAT 方法的實驗結果顯示，在數學推理任務上加入負軌跡的模型顯著優於純正軌跡微調基線，而所用資料量相同甚至更少。

本文闡明：負軌跡不是資料污染，而是一種特殊形態的監督訊號，其語義內容是「錯誤路徑識別」，類似強化學習中負向獎勵的概念，但操作成本遠低於真實環境的 RL 探索。Co-Evolving Agents（2025）的硬負例演化進一步證實，失敗軌跡的價值可通過專門設計的訓練機制被系統性提取。

層 2 — 編排層的失敗處理設計

當前大多數代理框架將重試實現為簡單的固定次數循環，這種設計在以下場景會系統性失敗：（a）同一失敗原因重複觸發（工具 API 格式錯誤）；（b）失敗的真正原因是上游計劃錯誤（重試子任務無法修復）。

本文論證，Gradientsys 和 Self-Healing Agentic Orchestrators 所展示的「感知-診斷-恢復」三步編排才是正確的架構設計原則：（1）感知：從工具返回值、執行日誌中提取失敗訊號；（2）診斷：分類失敗類型（是否可重試？是否需要重規劃？）；（3）恢復：選擇最低成本的恢復路徑。自癒架構之所以優於完整重規劃（98.8% vs 93.8%），正是因為其診斷步驟避免了在不必要場景觸發昂貴的完整 LLM 重規劃。

層 3 — 自我修正層的外部反饋依賴

自我修正不能是閉環的——這是 2025 年自我修正基準研究（arXiv 2510.16062）的核心發現。本文進一步主張：自我修正層的設計應以外部反饋作為觸發器，而非依賴模型的語言自省。工具返回值、形式化驗證結果、其他代理的審查輸出，都是有效的外部反饋來源。

三層架構的整合設計原則：訓練層減少失敗發生率（防錯）→ 編排層在失敗發生時最低成本恢復（容錯）→ 自我修正層在有外部訊號時進行深度糾偏（糾錯）。三層缺一，系統可靠性存在結構性缺口。

一家之言

（李飛飛主筆視角）

當我們談論 LLM 代理的「失敗」，我們不應只談技術錯誤率。失敗是有代價的，而這個代價由真實的人承受——一個醫療資訊代理給出錯誤建議，一個法律助理誤解合約條款，一個商業代理執行了錯誤的交易。數據是科學儀器，而被丟棄的失敗軌跡是我們尚未校準的儀器。

我要強調的是：負軌跡訓練不僅僅是提升準確率的技術手段，它是一種以人為中心的資料倫理——承認失敗的存在，系統性地從中學習，而非用「正確率 X%」的聚合指標掩蓋對弱勢案例的忽視。Wang 等人（2024）的工作顯示失敗軌跡作為訓練訊號的價值，但更深層的啟示是：系統設計者應當把「代理在哪裡失敗」作為第一類可觀測指標，就像 ImageNet 把分類錯誤作為研究啟動器一樣。

在治理維度，Context Kubernetes（2026）的三層許可權模型——「代理許可權永遠是人類許可權的真子集」——是我所認為的正確方向。這不是保守主義，而是 Human-centered AI 的系統實踐：在代理自主性不斷擴張的當下，失敗的後果必須可追溯、可管控、可糾偏。

（Yann LeCun 共同主筆視角）

我必須說一句讓人不舒服的話：我們在失敗感知架構上花費的巨大工程精力，從某種意義上是在為一個根本性缺陷做補丁。LLM 的自回歸預測機制缺乏對世界的因果模型——它不知道為什麼工具呼叫失敗，只能從訓練資料中統計哪些呼叫模式更常見。這就是為什麼 "Can LLMs Correct Themselves?"（2025）告訴我們，沒有外部反饋時自我修正幾乎無效。

但這並不意味著失敗感知架構是無用的。恰恰相反——在具備世界模型的下一代架構成熟之前，我們需要務實地把失敗訊號設計進系統。Gradientsys 的重試-重規劃機制，以及 Self-Healing Agentic Orchestrators 的自癒比較實驗，都是工程層面的正確方向。開源社群正在快速迭代這些機制，這正是開放生態的價值。

我的觀點是：現有的三層框架（訓練・編排・自我修正）是良好的中期架構，但研究社群應當並行投資於具備世界模型的代理架構，因為那才是失敗感知問題的根治路徑，而非持續的工程補丁。架構創新週期是真實的，每十年出現一個新範式——我們距離下一個範式時刻不遠了。

實踐建議

對代理系統研究者

  1. 建立失敗資料集：在代理基準評測中，保留並分類所有失敗軌跡（依據 AgentDebug 的四類分法），作為開源數據集發布以支持負軌跡訓練研究的重現性。
  2. 採用 NAT 微調策略：在代理微調中明確加入負軌跡，搭配前綴/後綴標記（參見 arXiv 2402.11651），而非僅使用成功軌跡。

對代理框架開發者

  1. 感知-診斷-恢復三步編排：重試邏輯應配備失敗類型分類器（工具 API 錯誤、規劃錯誤、上下文錯誤等），根據分類選擇恢復策略，而非統一重試。
  2. 層次化拓撲優先：多代理系統設計參考 Resilience of LLM-Based Multi-Agent Collaboration 的實驗結論，採用層次化中心調度架構（故障下降 5.5% vs 全連接的 23.7%），避免全連接拓撲的脆弱性。
  3. 自我修正依賴外部觸發器：不要期望 LLM 純語言自省能修正錯誤；設計工具返回值解析器和形式化驗證鉤子，作為自我修正的驅動訊號。

對企業部署者

  1. 可觀測性是前提：在部署 LLM 代理前，建立失敗率監控（按 AgentDebug 四類分法），可觀測性是失敗感知架構的外部反饋基礎。
  2. 許可權邊界設計：遵循 Context Kubernetes 的三層許可權原則，確保代理許可權是人類許可權的真子集，為失敗後的人工介入留有空間。

風險與限制

研究局限性

  1. 負軌跡資料取得成本：NAT 方法依賴已標記的失敗軌跡，在新場景中的初始失敗軌跡收集需要預先部署，構成冷啟動問題。
  2. 自癒架構的 LLM 成本：Self-Healing Agentic Orchestrators 的自癒機制在低資源部署場景中可能不可行——每次失敗都觸發 LLM 診斷查詢會顯著增加 API 成本。
  3. 階段混淆風險：三層框架中，編排層的重規劃與自我修正層的迭代糾偏在實踐中容易混淆，設計者需要明確界定觸發條件。

更廣泛的風險

  1. 失敗模式分佈漂移：以歷史失敗軌跡訓練的模型，在面對新型失敗模式時仍可能無效——失敗感知架構提升的是已知失敗類型的魯棒性，而非未知失敗類型的泛化性。
  2. 「自癒幻覺」風險：高自癒率（98.8%）可能誤導部署者降低人工監控，而實際上剩餘的 1.2% 失敗案例可能集中在高風險低頻場景，其危害遠超隨機失敗。
