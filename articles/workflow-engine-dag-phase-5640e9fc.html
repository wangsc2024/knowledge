一、技術概述

Workflow Engine（工作流引擎）是負責定義、排程、執行和監控任務流程的軟體系統。核心資料結構為 DAG（Directed Acyclic Graph，有向無環圖），其中節點代表任務，邊代表依賴關係。2025-2026 年的工作流編排生態正經歷三大轉變：從批處理轉向事件驅動即時觸發、從任務中心轉向數據資產中心、以及 AI/LLM 工作流的原生整合。主要工具包括 Apache Airflow（3.2 億次下載）、Dagster、Prefect、Kestra、Temporal 和 n8n 等。

二、核心概念與原理

2.1 DAG 基礎

DAG 由三個要素組成：

  • 節點（Node/Task）：工作流中的最小執行單位，封裝輸入參數、執行邏輯和輸出結果
  • 有向邊（Edge）：表示任務間的依賴關係，A → B 表示 B 依賴 A 的輸出
  • 無環約束：DAG 不允許循環依賴，確保存在合法的執行順序

2.2 拓撲排序（Topological Sort）

拓撲排序是 DAG 排程的數學基礎，將圖中節點排列成線性序列，使得對每一條有向邊 (u, v)，u 都排在 v 之前。

Kahn's Algorithm（BFS 方法）：


def topological_sort_kahn(graph):
    in_degree = {node: 0 for node in graph}
    for node in graph:
        for neighbor in graph[node]:
            in_degree[neighbor] += 1
    
    queue = [n for n in in_degree if in_degree[n] == 0]
    result = []
    
    while queue:
        node = queue.pop(0)
        result.append(node)
        for neighbor in graph[node]:
            in_degree[neighbor] -= 1
            if in_degree[neighbor] == 0:
                queue.append(neighbor)
    
    if len(result) != len(graph):
        raise ValueError("循環依賴偵測！")
    return result

DFS 方法：適合需要偵測循環或產生穩定排序的場景，使用遞迴搭配已訪問集合。

時間複雜度：O(V + E)，V 為節點數，E 為邊數。

2.3 兩大架構分類

| 類型 | 代表工具 | 核心理念 | 適用場景 |

|------|---------|---------|----------|

| 任務中心型 | Airflow, Luigi, Kestra, n8n | 以 DAG 組織任務控制流，對任務內容保持中立 | 通用工作流自動化 |

| 數據中心型 | Dagster, Temporal, Flyte | 以數據資產（表格、模型）為首要關注 | 數據管線、ETL |

三、工作流引擎六大核心元件

3.1 任務定義（Task Definition）

每個任務封裝三要素：

  • 輸入：從上游任務接收的數據或配置
  • 執行邏輯：實際業務邏輯（API 呼叫、數據轉換、計算）
  • 輸出：傳遞給下游任務的結果

3.2 圖表示（Graph Representation）

常用鄰接表表示 DAG：


dag = {
    'fetch_todoist': [],
    'fetch_news': [],
    'fetch_hackernews': [],
    'assemble_digest': ['fetch_todoist', 'fetch_news', 'fetch_hackernews']
}

3.3 排程器（Scheduler）

排程器負責：

  1. 掃描 DAG 識別「就緒」任務（所有前置依賴已完成）
  2. 將就緒任務提交到執行佇列
  3. 支援時間觸發（cron）和事件觸發（webhook/data event）

排程策略：

  • 向前掃描：依拓撲排序順序逐層執行
  • 向後查看：檢查前一週期是否執行過，避免重複（Eppo 的 Paddle 系統採用此策略，搭配 15 分鐘寬限期）

3.4 執行引擎（Execution Engine）

執行引擎的核心設計決策：

| 模式 | 優點 | 缺點 | 適用場景 |

|------|------|------|----------|

| 單線程 | 簡單可預測 | 無法利用並行 | 任務間強順序依賴 |

| 多線程 | 利用 CPU 多核 | 共享狀態管理複雜 | CPU 密集型任務 |

| 多進程 | 隔離性強 | 進程間通訊開銷 | I/O 密集型、需隔離 |

| 分散式 | 水平擴展 | 架構複雜度高 | 大規模生產環境 |

3.5 持久化層（Persistence Layer）

儲存：

  • 任務狀態（pending/running/success/failed/retrying）
  • 執行歷史（時間戳、耗時、輸出）
  • DAG 執行元資料

Eppo 的三表模型：

  • dag_runs：單一 DAG 執行實例
  • dag_task_runs：計算圖中的節點
  • dag_task_run_dependencies：節點間的依賴關係

3.6 監控與告警

即時追蹤：任務延遲、失敗率、重試次數、佇列深度。

四、並行執行策略

4.1 任務並行（Task Parallelism）

識別 DAG 中相互獨立的任務（無依賴邊），同時執行：


       ┌─ fetch_todoist ──┐
start ─┼─ fetch_news ──────┼─ assemble_digest
       └─ fetch_hackernews ┘

此模式中，三個 fetch 任務無依賴關係，可並行執行，assemble_digest 等待全部完成後執行。

4.2 數據並行（Data Parallelism）

相同邏輯處理不同數據分片：


query → [task_1, task_2, task_3] → assemble

Todoist 團隊模式的 Phase 2 即為此模式：Phase 1 查詢規劃後，Phase 2 的 N 個並行執行 Agent 各處理一個任務。

4.3 LLM Compiler Pattern

AI Agent 領域的創新排程模式：

  1. 規劃器（Planner）：LLM 分析任務並生成 DAG
  2. 拓撲排序與驗證：系統檢查循環依賴、建立執行順序
  3. 並行執行與結果整合：獨立任務並行，依賴任務等待

相比 ReAct（逐步推理-執行）模式，LLM Compiler 在任務具「寬度大於深度」特徵時表現更優。

五、容錯與錯誤處理

5.1 重試策略


retry_config = {
    'max_retries': 3,
    'backoff': 'exponential',  # 1s → 2s → 4s
    'retry_on': ['TimeoutError', 'ConnectionError'],
    'dead_letter_queue': True
}

5.2 故障隔離

  • 任務級隔離：單一任務失敗不影響其他獨立分支
  • Phase 級隔離：Phase 1 的某個 Agent 失敗，不影響其他 Phase 1 Agent
  • Circuit Breaker：連續失敗後自動斷路，避免級聯故障

5.3 僵屍任務偵測

Eppo 的守護程序模式：定期掃描長時間處於 running 狀態的任務，自動標記為 failed 並觸發重試。

七、與本專案的關聯

7.1 現有 Phase 管線即為 DAG

Daily-Digest-Prompt 的團隊模式本質上是手動實作的 DAG 排程引擎：

每日摘要（run-agent-team.ps1）：


Phase 1 (並行): [Todoist, News, HN, Gmail, Security]
                        ↓ (全部完成)
Phase 2 (串行): Assemble Digest

Todoist 團隊（run-todoist-agent-team.ps1）：


Phase 1: Query → Planning
              ↓
Phase 2 (並行): [Task_1, Task_2, ..., Task_N]
              ↓ (全部完成)
Phase 3: Assemble + Notify

7.2 可借鑑的設計模式

  1. Eppo 三表模型：目前 scheduler-state.json 是扁平記錄，可考慮分離 dag_runs（Phase 執行實例）和 task_runs（個別任務狀態），更精確追蹤失敗點
  1. 向後查看排程：目前的 config/frequency-limits.yaml 已實作類似概念（檢查今日已執行次數），可進一步加入寬限期防止邊界情況
  1. Worker 自動擴展：Eppo 根據 Redis 佇列深度動態調整 Worker 數量。本專案的 Phase 2 可根據任務數量動態調整 Start-Job 並行度
  1. LLM Compiler Pattern：Phase 1 的 Query Agent 已扮演「Planner」角色，但目前 Phase 2 的任務是靜態分配的。若引入 LLM Compiler 模式，Phase 1 可動態生成任務 DAG，Phase 2 按拓撲排序自動並行
  1. 僵屍任務偵測：目前的 timeout 機制是硬性的（Wait-Job -Timeout），可加入守護程序式的定期心跳檢查

八、程式碼範例：簡易 DAG 排程器（Python）


import asyncio
from enum import Enum
from dataclasses import dataclass, field
from typing import Callable, Any

class TaskStatus(Enum):
    PENDING = 'pending'
    RUNNING = 'running'
    SUCCESS = 'success'
    FAILED = 'failed'

@dataclass
class Task:
    name: str
    fn: Callable
    deps: list[str] = field(default_factory=list)
    status: TaskStatus = TaskStatus.PENDING
    result: Any = None

class DAGScheduler:
    def __init__(self):
        self.tasks: dict[str, Task] = {}
    
    def add_task(self, task: Task):
        self.tasks[task.name] = task
    
    def _get_ready_tasks(self) -> list[Task]:
        ready = []
        for task in self.tasks.values():
            if task.status != TaskStatus.PENDING:
                continue
            deps_met = all(
                self.tasks[dep].status == TaskStatus.SUCCESS
                for dep in task.deps
            )
            if deps_met:
                ready.append(task)
        return ready
    
    async def execute(self):
        while any(t.status == TaskStatus.PENDING for t in self.tasks.values()):
            ready = self._get_ready_tasks()
            if not ready:
                raise RuntimeError('Deadlock: 無就緒任務但仍有待處理任務')
            
            # 並行執行所有就緒任務
            results = await asyncio.gather(*[
                self._run_task(task) for task in ready
            ], return_exceptions=True)
            
            for task, result in zip(ready, results):
                if isinstance(result, Exception):
                    task.status = TaskStatus.FAILED
                else:
                    task.status = TaskStatus.SUCCESS
                    task.result = result

    async def _run_task(self, task: Task):
        task.status = TaskStatus.RUNNING
        dep_results = {d: self.tasks[d].result for d in task.deps}
        return await task.fn(dep_results)

九、最佳實踐摘要

  1. 模組化設計：任務定義、排程器、執行引擎、持久化層各自獨立，可替換
  2. 冪等性（Idempotency）：確保任務重試不會產生副作用
  3. 可觀測性：為每個 DAG 執行和任務記錄結構化日誌
  4. 漸進式複雜度：從簡單的線性 Phase 開始，按需引入 DAG 依賴解析
  5. 故障域隔離：獨立分支的失敗不應級聯影響
  6. 避免過度並行：並行度受限於 API Rate Limit 和系統資源
  7. Dead Letter Queue：多次重試仍失敗的任務放入 DLQ，供人工審查
