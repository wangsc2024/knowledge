LLM 驅動之 Lean 4 可驗證程式生成：階層式證明搜尋與 2026 前沿 深度研究報告

AI深度研究Lean4 可驗證程式生成2026研究報告

LLM 驅動之 Lean 4 可驗證程式生成：階層式證明搜尋與 2026 前沿（Goedel-Code-Prover／Verina／WybeCoder）：深度技術研究報告

執行摘要（200-400 字）

本研究聚焦 2025–2026 年 LLM 驅動之 Lean 4 可驗證程式生成前沿，以 Verina 基準、Goedel-Code-Prover 之階層式證明搜尋，以及 WybeCoder 之命令式混合驗證為三條主線。綜合多來源後可見：可驗證生成必須同時產出實作、形式規格與機器檢查證明，其中「證明合成」遠比程式與規格步驟困難，構成結構性瓶頸 [來源：Verina: Benchmarking Verifiable Code Generation（官方專頁）, 2025]。階層分解、編譯器／SMT 回饋與搜尋式推論，是當前提升端到端驗證成功率的主流路徑 [來源：Goedel-Code-Prover: Hierarchical Proof Search…, 2026；WybeCoder: Verified Imperative Code Generation, 2026]。不同文獻宣稱的 Verina 成功率數字，會因模型是否專用化、搜尋與代理預算、pass@k 與評測切片而呈現巨大落差，應分「通用 LLM 單次」「專用神經證明器＋搜尋」「高預算多代理」三類維度解讀，不宜直接橫向比排名。對日常軟體工程，啟發在於把「規格＋機器檢查」思維遷移到 Schema、Hook 與結果檔守衛等管線，並以可分解的驗證任務清單取代單次 LLM 一次到位。

研究背景與動機

形式化方法與大型語言模型的交會點，正從「輔助寫證明」走向「端到端可驗證程式生成」：模型不只要寫出能編譯的程式，還要滿足 Hoare 風格或測試導向的規格，並通過 Lean 等證明助理的機器檢查。Verina 以 189 道 Lean 任務模組化評估程式生成、規格生成與證明生成，揭示通用模型在證明維度上的斷層 [來源：Verina: Benchmarking Verifiable Code Generation（官方專頁）, 2025]。相對地，Goedel-Code-Prover 與 WybeCoder 分別代表「Lean 內階層式神經證明搜尋」與「命令式程式＋SMT＋Lean 的代理式混合路線」，為 2026 年工程落地提供互補參考 [來源：arXiv:2603.19329, 2026；arXiv:2603.29088, 2026]。

核心技術解析（每個主張內嵌引用）

結構性瓶頸在「證明」而非「程式」：Verina 官方量測顯示，通用 LLM 在 pass@1 下，程式與規格維度尚可，但證明成功率可落至個位數百分比級別，與前兩者形成強烈對比 [來源：Verina: Benchmarking Verifiable Code Generation（官方專頁）, 2025]。這與「可驗證生成＝實作＋規格＋證明」的三元結構一致，說明瓶頸主要發生在機器檢查證明的合成 [來源：WybeCoder: Verified Imperative Code Generation, 2026]。

Goedel-Code-Prover 的階層式證明搜尋：該路線將驗證目標分解為較簡子目標，並在戰術層完成補全；分解分數同時服務訓練獎勵與推論排序，並以混合強化學習在連續分解獎勵與監督回放之間取得穩定 [來源：Goedel-Code-Prover: Hierarchical Proof Search…, 2026]。專案網站亦描述推論管線：並行分解候選、QuickCheck 過濾、依分數遞迴展開與葉節點編譯修復，使加算力能在分解與補全兩階段帶來可觀的單調或穩定改善 [來源：Goedel-Code-Prover 專案網站與技術說明, 2026]。

WybeCoder 的命令式與混合驗證：相對於 Lean 函數式題材，WybeCoder 強調代理式「邊生成邊證明」，讓程式、不變式與證明共演化，並結合自動 VC、SMT（如 cvc5）與 Lean 內互動式證明；並提供由 Verina／Clever 轉向命令式規格的腳本，使評測可比較不同程式風格 [來源：WybeCoder: Verified Imperative Code Generation, 2026；facebookresearch/wybecoder, 2026]。

評測錨點與可重現性：sunblaze-ucb/verina 公開資料格式與任務旗標，支援 code_gen／spec_gen／proof_gen 等管線對齊論文與 HuggingFace 資料集，使後續工作能在共同錨點上累積 [來源：sunblaze-ucb/verina（評測程式與資料格式）, 2025]。

關鍵洞見（含三角佐證狀態 ✅/⚠️）

  • 證明合成是主短板，驅動專用神經證明器與搜尋管線 — 已驗證✅（Verina、Goedel-Code-Prover、WybeCoder 三線敘述一致）。
  • 分解分數若對齊訓練與部署启发式，可降低目標不一致 — 已驗證✅（Goedel-Code-Prover 論文與網站）。
  • 「同一 Verina 成功率」數字的可比性受限 — 已驗證✅：通用模型單次、專用 8B 階層搜尋、與多代理混合路線分屬不同實驗協定，應對照附錄與預算 [分歧點見三角化摘要]。
  • 函數式 Lean 管線與命令式 Velvet／Loom 為互補而非零和 — 已驗證✅（Goedel-Code-Prover vs WybeCoder 覆蓋面不同）。
  • 將 Schema／Hook 守衛類比為「可搜尋的分解式驗證」 — 待驗證⚠️（工程類比，非學界共識定理）。

實務應用建議（3-5 點）

  1. 以 Verina 類切片自評：若團隊導入可驗證生成，宜分別量測「能跑」「規格對」「能證」三階，避免只看單一成功率。
  2. 導入階層分解思維：將大規格拆成子引理或子守衛（對應本 repo 的結果檔 schema、hook 規則），降低單次模型輸出負擔。
  3. 預算與報告維度對齊：對外報告時明確標註 pass@k、迭代上限與是否專用模型，避免跨文獻誤比。
  4. 命令式路線評估：若主力程式碼為命令式與不變式，可平行評估 WybeCoder 類混合管線與純 Lean 函數式路線的維護成本。
  5. 媒體與單一來源敘事：對產業新聞中宣稱的大規模形式化成果，技術細節應回歸正式論文核實 [來源：新浪科技報導, 2026 — 待驗證⚠️]。

與現有知識的連結

本專案已具結果檔 schema、Phase2 守衛與 Hook 強制層，與「規格＋機器檢查」文化相通；差異在於多數欄位仍以約束與測試表達，而非 Lean 定理。可驗證生成前沿提醒我們：高價值下一步往往是可組合、可搜尋的驗證步驟，與 continuity 中「verifier／coding agent」脈絡可銜接為「把關鍵不變式從慣例提升為可執行檢查」。

未來展望

短期內預期會看到更多「分解策略學習＋工具回饋」的論文與開源權重；中長期則關注命令式與函數式評測是否收斂到共同的中間表示，以及通用推理模型能否在固定預算下縮小與專用證明器的差距。對本 repo，可選題包含：結果檔欄位的漸進形式化、與 research-registry／ADR 的追溯連結強化。
