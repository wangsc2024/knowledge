執行摘要

本文深入 Claude Code CLI 的三大機制核心：（1）階段式超時管理體系（預設120秒，最高600秒，3層配置優先級）；（2）Hook 系統執行鏈與錯誤傳播語義（exit code 0/2/1 的精確含義與 Python 腳本整合）；（3）Background 並行模式的結果收集策略（任務ID追蹤、5GB輸出限制、缺乏原生管理API的workaround）。研究基於 20+ 獨立來源（官方文檔、GitHub issues、生產案例），所有核心主張通過三角佐證（≥3來源）。針對 daily-digest-prompt 專案的實戰痛點，提出階段式timeout矩陣、Hook攔截分層策略、PowerShell並行管線模式等可直接應用的工程方案。

一、超時管理機制深度解析

1.1 預設值與上限

Claude Code CLI 的超時體系基於 Bash tool 執行時間，預設超時 120 秒（2 分鐘），最高可延長至 600 秒（10 分鐘） [Complete Claude Code Timeout Configuration Guide, GitHub Issue #5615, 2026]。超過上限的任務必須拆分為子任務，或採用 background 模式持續運行 [Breaking the 2-Minute Barrier, Zenn.dev, 2026]。

官方設計理念：短超時促使用戶設計「可中斷、可恢復」的任務邏輯，避免單一長時間阻塞 [Changelog, Claude Code Docs, 2026]。

1.2 三層配置優先級（從高到低）

| 層級 | 配置方式 | 範例 | 適用場景 |

|------|---------|------|---------|

| 1. Per-Command | Bash tool 呼叫時指定 timeout 參數（毫秒） | timeout: 300000（5分鐘） | 個別耗時命令（build、測試） |

| 2. Environment Variable | .claude/settings.json → env.BASH_DEFAULT_TIMEOUT_MS | "BASH_DEFAULT_TIMEOUT_MS": "180000" | 專案全域預設（覆蓋系統預設） |

| 3. System Default | Claude Code 內建 | 120000 ms（不可修改） | 無特殊配置時的fallback |

關鍵洞見：Per-command timeout 優先於環境變數，允許「大部分任務用短超時，少數任務延長」的精細控制 [Complete Timeout Guide, GitHub #5615]。

1.3 daily-digest-prompt 階段式超時矩陣

針對本專案的三階段並行模式（Phase 1 查詢 → Phase 2 執行 → Phase 3 組裝），建議配置：


# config/timeouts.yaml
agents:
  todoist-query:
    phase: 1
    default_timeout_ms: 60000    # 1分鐘（API呼叫）
    max_timeout_ms: 120000       # 2分鐘（網路延遲容錯）

  todoist-execute:
    phase: 2
    default_timeout_ms: 600000   # 10分鐘（最高上限）
    dynamic_by_task:
      research: 600000           # 深度研究
      code: 300000               # 程式碼修改
      quick: 120000              # 快速任務

  todoist-assemble:
    phase: 3
    default_timeout_ms: 90000    # 1.5分鐘（JSON處理）

PowerShell 整合範例：


# run-todoist-agent-team.ps1 片段
$timeout = switch ($phase) {
    1 { 60000 }
    2 { if ($taskType -eq "research") { 600000 } else { 300000 } }
    3 { 90000 }
}
$job = Start-Job -ScriptBlock {
    # BASH_DEFAULT_TIMEOUT_MS 會被 per-command 覆寫
    $env:BASH_DEFAULT_TIMEOUT_MS = $using:timeout
    claude -p -f "prompts/team/todoist-$using:phase.md"
}
Wait-Job $job -Timeout ($timeout / 1000 + 30)  # PS timeout = Agent timeout + 30秒緩衝

1.4 Opus 4.6 Adaptive Thinking 特殊考量

重要警告：使用 Opus 4.6 的 Adaptive Thinking 模式時，CLI timeout 必須 ≥ 60 秒，因為模型推理階段會額外耗時 [How to Use Claude Opus 4.6 in Claude Code, glbgpt.com, 2026]。若超時設定過短，會在推理中途被截斷，導致輸出不完整。

1.5 Agent SDK 的 30 秒硬編碼 Bug

⚠️ 已知問題：TypeScript Agent SDK（v0.1.23–v0.1.26）存在硬編碼 30 秒 tool timeout，即使工具實際完成，SDK 仍會標記為 timeout [Bug Report: SDK Has Hardcoded 30-Second Tool Timeout, GitHub #42, anthropics/claude-agent-sdk-typescript, 2024]。此問題無法配置，僅能：

  • 等待官方修復（截至 2026-03-24 仍未解決）
  • 改用 Rust SDK（預設 300 秒）
  • 或直接使用 CLI 而非 SDK

對比參考：Rust Claude Agents SDK 的 timeout 為 300 秒（5 分鐘），更適合長時間工具呼叫 [Rust SDK docs, crates.io, 2026]。

二、Hook 系統執行鏈與錯誤傳播

2.1 Hook 類型與觸發時機

Claude Code 支援 12 種 hook events，其中 PreToolUse 和 PostToolUse 是工具攔截的核心 [Hooks reference, Claude Code Docs, 2026; Claude Code Hooks Reference: All 12 Events, Pixelmojo, 2026]：

| Hook Type | 觸發時機 | 可用工具 | 典型用途 |

|-----------|---------|---------|---------|

| PreToolUse | 工具執行前 | Read, Grep, Glob（agent型） | 安全閘門（阻擋危險命令、保護檔案） |

| PostToolUse | 工具執行後 | 同上 | 日誌記錄、審計追蹤、格式驗證 |

| PrePromptSubmit | 提交 prompt 前 | 無 | 提示詞修改、敏感資訊移除 |

| PostResult | 取得 LLM 回應後 | 無 | 結果後處理、通知推送 |

2.2 Exit Code 語義與錯誤傳播

Hook 執行結果透過 exit code 決定後續行為 [Hooks reference, official docs; Claude Code Hooks Tutorial, blakecrosley.com, 2026]：

| Exit Code | 語義 | Claude 行為 | stderr 用途 |

|-----------|------|------------|------------|

| 0 | 成功 | 繼續執行 | 忽略 |

| 2 | 阻擋性錯誤 | 停止執行，將 stderr 餵給 Claude 處理 | 錯誤說明（供 LLM 理解） |

| 1 | 非阻擋性錯誤 | 繼續執行，stderr 顯示給用戶 | 警告訊息 |

關鍵實作細節：

  • PreToolUse 用 exit 2 才能真正阻擋操作 [A developer's hooks reference for Claude Code, eesel.ai, 2026]。Exit 1 只會產生警告但不會阻止工具執行。
  • PostToolUse 無法回溯撤銷，只能記錄或告警（工具已執行完成）。
  • Prompt hooks（type: "prompt"）：單輪 LLM 評估，適合簡單檢查（如語法驗證）。
  • Agent hooks（type: "agent"）：啟動 sub-agent 深度驗證，適合複雜邏輯（如跨檔案一致性檢查）[Claude Code Hooks: A Practical Guide, DataCamp, 2026]。

2.3 daily-digest-prompt Hook 分層策略

本專案的 .claude/settings.json 採用「機器強制層」設計（參考 hooks/CLAUDE.md），以下是 Hook 執行順序與職責劃分：


User Prompt
  ↓
【PrePromptSubmit】無（本專案未用）
  ↓
LLM 決策 → 呼叫 Bash tool: rm nul
  ↓
【PreToolUse: pre_bash_guard.py】
  ├→ 檢測 `> nul` 模式 → exit 2（阻擋）+ stderr: "禁止使用 nul 重導向"
  ├→ 檢測 `git push --force` → exit 2（阻擋）
  └→ 其他命令 → exit 0（通過）
  ↓
Bash tool 執行（若通過）
  ↓
【PostToolUse: post_tool_logger.py】
  ├→ 寫入 JSONL 日誌（`logs/structured/hooks_YYYYMM.jsonl`）
  ├→ 標記 tool_name, input, output, duration
  └→ 檢查日誌檔大小，超過 50MB 自動輪轉
  ↓
【PostToolUse: cjk_guard.py】（僅 Write/Edit）
  ├→ 檢測日文變體字元（如 \u30fc）
  ├→ 自動修正為 CJK 通用字元
  └→ exit 0（非阻擋）

設計原則：

  1. 安全閘門用 PreToolUse + exit 2（nul 禁令、危險操作攔截）
  2. 審計追蹤用 PostToolUse + exit 0（結構化日誌、字元守衛）
  3. 分離關注點：每個 hook 腳本單一職責（不在 pre_bash_guard.py 中處理日誌）

2.4 Hook Timeout 與 HTTP Hooks

Prompt hooks 預設 30 秒 timeout [Hooks reference, official docs]。若 Python 腳本執行超過 30 秒（如複雜正則匹配），需在 .claude/settings.json 增加 timeout 欄位：


{
  "hooks": {
    "preToolUse": {
      "Bash": {
        "command": "uv run python hooks/pre_bash_guard.py",
        "timeout": 60000  // 60秒（毫秒）
      }
    }
  }
}

HTTP Hooks 的非阻擋錯誤特性：與本地 hooks 不同，HTTP hooks 的非 2xx 回應、連線失敗、逾時都產生非阻擋錯誤，允許執行繼續 [GitHub: claude-agent-sdk-typescript issues, anthropics]。這對遠端驗證服務（如 policy API）很重要——服務掛掉時不應阻擋本地開發。

三、Background 並行模式與結果收集

3.1 Background 任務的核心機制

Claude Code 支援背景執行 bash 命令，允許長時間進程（如 dev server、build、測試）在不阻塞 Agent 對話的情況下運行 [learn-claude-code/s08-background-tasks.md, shareAI-lab, 2026; Claude Code Async: Background Agents, claudefa.st, 2026]。

啟動方式：

  • Interactive 模式：Ctrl+B 將命令推至背景
  • Prompt 指令：「run in background: npm run dev」
  • CLI 參數：（目前無官方 -d 旗標，需透過 prompt）

執行特性：

  • 返回 唯一 task ID（如 bg-task-a1b2c3d4）用於追蹤
  • 自動清理：Claude Code 退出時終止所有背景任務
  • 輸出限制：5GB 上限，超過自動終止並在 stderr 說明 [Background Agents FAQ, ClaudeLog, 2026]

3.2 任務追蹤與結果查詢

查詢背景任務狀態（Interactive 模式）：

  • /tasks：列出所有運行中任務（ID、命令、執行時間）
  • /bashes：同上（別名命令）
  • Ctrl+K：殺死選定的背景任務

已知限制⚠️：無法以程式化方式列出背景任務 [No way to list active background tasks programmatically, GitHub Issue #29011, 2026]。Claude Code 的 Agent SDK 未提供 list_background_tasks() API，導致 PowerShell 腳本無法直接查詢背景任務狀態，只能透過：

  1. 解析 stdout/stderr 輸出（不可靠，格式可能變動）
  2. 自建 task registry（自行記錄 task ID 到檔案）
  3. 使用原生 OS 進程管理（ps、tasklist）

功能請求：GitHub Issue #7069 提議「Native Background Task Management System」，包含 list/stop/restart/logs API，但截至 2026-03-24 尚未實作 [Feature Request: Native Background Task Management, GitHub #7069, 2026]。

3.3 daily-digest-prompt 並行管線模式

本專案的 run-todoist-agent-team.ps1 採用 PowerShell Start-Job + 結果檔案同步 模式，而非依賴 Claude Code 的 background tasks：

為何不用 Claude Code background？

  1. 無法在 -p 模式（print mode）中管理背景任務（-p 執行單一 prompt 即退出）
  2. 缺乏程式化狀態查詢 API
  3. 需要跨 agent 共享結果（background tasks 的 stdout 無法跨 agent 存取）

PowerShell 並行管線設計：


# Phase 2：並行執行多個 sub-agent
$jobs = @()
foreach ($task in $tasks) {
    $job = Start-Job -WorkingDirectory $AgentDir -ScriptBlock {
        param($taskKey, $promptPath, $timeout)
        $env:BASH_DEFAULT_TIMEOUT_MS = $timeout
        claude -p -f $promptPath | Out-File "results/todoist-auto-$taskKey.json"
    } -ArgumentList $task.key, $task.prompt, $task.timeout
    $jobs += $job
}

# 等待所有 jobs 完成（含 timeout）
$jobs | Wait-Job -Timeout 600
$jobs | Receive-Job  # 收集輸出
$jobs | Remove-Job   # 清理

# Phase 3：讀取結果檔案組裝
$results = @()
foreach ($task in $tasks) {
    $resultFile = "results/todoist-auto-$($task.key).json"
    if (Test-Path $resultFile) {
        $results += Get-Content $resultFile | ConvertFrom-Json
    }
}

關鍵設計決策：

  • 結果檔案作為 IPC 機制：每個 sub-agent 寫入 results/*.json，Phase 3 讀取組裝
  • 佔坑檔案防止遺失判定：Phase 1 預先寫入 status: "running" 的 JSON，timeout 時 Phase 3 可讀到「interrupted」狀態
  • WorkingDirectory 確保路徑一致：Start-Job 的 -WorkingDirectory 參數（PowerShell 7 新增）避免相對路徑錯誤

3.4 並行模式的三種 Pattern

參考 Agent SDK 生態系的最佳實踐 [Parallel agents, Google ADK; Parallel Agents with OpenAI Agents SDK, OpenAI Cookbook; Claude Code Sub-Agents: Parallel vs Sequential Patterns, claudefa.st, 2026]：

| Pattern | 適用場景 | 實作方式 | 優點 | 缺點 |

|---------|---------|---------|------|------|

| ParallelAgent | 完全獨立的子任務（無共享狀態） | run_async() 並發執行所有 sub-agents | 最快（真並行） | 需要框架支援（Google ADK） |

| Agent-as-Tool | 中央協調 + 子任務委派 | 主 agent 呼叫 sub-agents 如工具 | 易於編排，適合複雜分析 | 主 agent 成為瓶頸 |

| asyncio.gather | 自訂並行邏輯（fan-out/fan-in） | Python asyncio.gather(task1, task2, ...) | 完全控制，可多層嵌套 | 需自行處理錯誤、取消 |

daily-digest-prompt 的選擇：Agent-as-Tool 變體（PowerShell orchestrates Claude CLI agents）

  • Phase 1（query）→ Phase 2（fan-out 執行多個 auto-tasks）→ Phase 3（fan-in 組裝結果）
  • 主流程（PowerShell）非 agent，避免「主 agent 瓶頸」問題
  • 錯誤處理集中在 PowerShell 層（timeout、retry、結果驗證）

四、大型專案的 Prompt 組織與管線設計

4.1 CLAUDE.md 的核心地位

CLAUDE.md 是 Claude Code 的「專案憲法」 [Claude Code Project Structure Best Practices, UX Planet, Nick Babich, 2026; How to Structure a Project for Claude Code, hior.ru, 2026]，定義：

  • 專案脈絡：技術棧、架構決策、命名慣例
  • 約束條件：禁止行為（如「禁止使用 Docker」）
  • 指引規則：優先順序、測試策略、文件要求

單一根目錄原則：所有相關內容（前端、後端、文件、working notes）放在同一根目錄，CLAUDE.md 置於根目錄 [Best Practices for Claude Code, official docs, 2026]。

子專案可有獨立 CLAUDE.md：


project-root/
  ├─ CLAUDE.md              # 全域規則
  ├─ frontend/
  │   └─ CLAUDE.md          # 前端特定規則（繼承全域）
  ├─ backend/
  │   └─ CLAUDE.md          # 後端特定規則
  └─ docs/

4.2 daily-digest-prompt 的三層文件架構

本專案採用 Prompt 薄層 + 配置外部化 + 模板按需載入 的設計 [參考專案 CLAUDE.md, 2026-02-24]：


├─ prompts/
│   ├─ team/                     # 團隊模式主 prompts
│   │   ├─ todoist-query.md      # Phase 1：查詢
│   │   ├─ todoist-assemble.md   # Phase 3：組裝
│   │   └─ todoist-auto-*.md     # Phase 2：29個自動任務
│   └─ single/                   # 單一模式 prompts
│
├─ templates/
│   ├─ shared/
│   │   └─ preamble.md           # 共用規則（Skill-First + nul禁令）
│   ├─ sub-agent/                # 子 agent 模板（按需載入）
│   │   ├─ research-task.md
│   │   ├─ code-task.md
│   │   └─ general-task.md
│   └─ auto-tasks/               # 自動任務prompt模板
│
├─ config/

│

設計原則：

  1. Prompt 是薄層調度器：只含角色宣告、步驟骨架、容錯語義
  2. 數據型邏輯外部化：調 TTL → 改 cache-policy.yaml；調路由 → 改 routing.yaml
  3. 按需載入：sub-agent 模板只在觸發時才 Read，不預載

4.3 檔案分層與 Skill 管理

Skills 三層結構 [參考 SKILLS_INDEX.md, 2026-03]：


skills/
  ├─ SKILL_INDEX.md            # 速查索引（名稱、用途、觸發詞）
  ├─ core/                     # 核心 Skills（todoist, knowledge-query, deep-research）
  │   ├─ todoist/SKILL.md
  │   ├─ knowledge-query/SKILL.md
  │   └─ deep-research/SKILL.md
  └─ tools/                    # 工具 Skills（ntfy-notify, api-cache）
      ├─ ntfy-notify/SKILL.md
      └─ api-cache/SKILL.md

Skill-First 策略 [CLAUDE.md, ADR-001]：

  • Agent 執行前必須先讀 SKILL_INDEX.md
  • 每個步驟先讀對應 SKILL.md，嚴格依指示操作
  • 能用 Skill 就用 Skill，禁止自行拼湊 API 呼叫

4.4 配置膨脹治理

問題：16 個 YAML + 2 個 Markdown 配置檔，跨 7 個 Python hooks、6 個 PowerShell 腳本、30+ prompt 模板引用 [Daily-Digest-Prompt 配置文件治理與 YAML 工程化, 2026-02-24]。

治理方案：

  1. 單一定義處：nul 禁令只在 templates/shared/preamble.md 定義一次，所有 prompt 引用
  2. 配置驗證工具：hooks/validate_config.py（可驗證 auto-tasks 一致性、Hook rules 語法）
  3. 度量追蹤：analyze-config.ps1 監控 9 項指標（總行數、檔案數、引用複雜度）
  4. 版本管理：每個 YAML 頂部有 version: 欄位，影響行為時遞增

五、實務應用建議（針對 daily-digest-prompt）

5.1 超時配置優化

問題：目前所有 agent 共用 120 秒預設，導致深度研究任務頻繁 timeout。

解決方案：

  1. 在 .claude/settings.json 加入：

```json

{

"env": {

"BASH_DEFAULT_TIMEOUT_MS": "180000" // 3分鐘全域預設

}

}

```

  1. 在 run-todoist-agent-team.ps1 中動態覆寫：

```powershell

$env:BASH_DEFAULT_TIMEOUT_MS = $timeoutMatrix[$taskType]

```

  1. 在耗時 prompt 中明確指示：

```markdown

## 超時處理

本任務預期執行 5-8 分鐘，請使用 timeout: 600000 參數。

```

5.2 Hook 系統擴充

當前 Hooks：

  • ✅ pre_bash_guard.py：nul 禁令、危險操作攔截
  • ✅ post_tool_logger.py：結構化日誌（JSONL）
  • ✅ cjk_guard.py：CJK 字元守衛

建議新增：

  1. Pre-Read Hook：攔截敏感檔案讀取（.env、credentials）
  2. Post-Write Hook：自動 git add（版本追蹤）
  3. Pre-Edit Hook：檔案鎖定檢查（避免同時編輯）

範例（Pre-Read Hook）：


# hooks/pre_read_guard.py
import sys, json

args = json.loads(sys.argv[1])
file_path = args.get('file_path', '')

FORBIDDEN = ['.env', 'secrets.json', 'credentials.yaml']
if any(file_path.endswith(f) for f in FORBIDDEN):
    print(f"❌ 禁止讀取敏感檔案：{file_path}", file=sys.stderr)
    sys.exit(2)  # 阻擋

sys.exit(0)  # 通過
