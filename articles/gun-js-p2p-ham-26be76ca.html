一、技術概述

Gun.js（GunDB）是一個開源的即時去中心化離線優先圖形資料庫引擎，以 JavaScript 撰寫，核心壓縮後僅約 9KB。它能在瀏覽器內直接執行，實現裝置間的 P2P 資料同步，支援離線編輯後自動合併。其核心定位為「自由鬥士的資料庫」，強調數據主權與去中心化。

關鍵數據：

  • GitHub Stars：18K+（amark/gun）
  • 效能：20M+ API ops/sec（記憶體內操作）
  • 壓縮大小：~9KB gzipped
  • 生產用戶：Internet Archive、HackerNoon
  • 網路流量：去中心化應用日推 TB 級 P2P 流量

二、核心概念與原理

2.1 圖形資料模型（Graph Data Model）

Gun.js 使用圖形結構表示資料與關係。每個節點有唯一的 soul（識別符），資料以標準 JSON 格式儲存。邏輯圖形被正規化為 JSON 子集。

重要限制：節點內容僅支援 null、boolean、number、string 和連結（reference）。不支援陣列和巢狀物件。需使用 set() API 模擬無序陣列。


// 基本資料操作
const gun = Gun(['https://your-relay.example.com/gun']);

// 寫入
gun.get('users').get('alice').put({
  name: 'Alice',
  age: 30,
  online: true
});

// 即時監聽
gun.get('users').get('alice').on((data, key) => {
  console.log(`${key} updated:`, data);
});

// 集合操作（模擬陣列）
gun.get('chatroom').get('messages').set({
  text: 'Hello World',
  sender: 'alice',
  ts: Date.now()
});

2.2 HAM 衝突解決演算法（Hypothetical Amnesia Machine）

HAM 是 Gun.js 的核心衝突解決機制，結合時間戳與向量時鐘來解決分散式系統中的並發衝突。

設計哲學：

  • 保證強最終一致性（Strong Eventual Consistency, SEC），而非線性一致性
  • 偏好高可用性（Availability）而非強一致性（CAP 定理中的 AP）
  • 離線編輯在重新連線後自動合併

運作原理：

  1. 每次寫入都帶有狀態向量（state vector）和時間戳
  2. 系統中每個變更都經過 HAM 處理
  3. 較新的時間戳覆蓋較舊的值
  4. 相同時間戳時使用確定性規則（值的字典序比較）
  5. 所有對等節點最終收斂至相同狀態

與 CRDT 的關係：Gun.js 是圖形 CRDT 的實作，但使用自己的 HAM 演算法而非傳統 CRDT 原語。HAM 本質上是一個 Last-Writer-Wins Register（LWW-Register）的變體。

2.3 DAM 網路協議（Daisy-chain Ad-hoc Mesh-network）

Gun.js 實作了自己的傳輸層協議 DAM，用於 P2P 網路中的對等節點協調。

特點：

  • 菊花鏈式的臨時網狀網路
  • 透過訊息傳遞協調而不產生過多重複廣播
  • 支援 WebSocket 和 WebRTC 傳輸
  • 自動處理節點發現與連線管理

2.4 SEA 加密體系（Security, Encryption, Authorization）

SEA 是 Gun.js 的加密層，使用最新的原生 Web Crypto API。

四大密碼學原語：

| 功能 | 演算法 | 用途 |

|------|--------|------|

| 簽章 | ECDSA（P-256） | 資料完整性驗證 |

| 加密 | AES-GCM | 端對端訊息加密 |

| 金鑰交換 | ECDH | 共享密鑰協商 |

| 密碼派生 | PBKDF2 | 帳號密碼保護 |


// SEA 使用者認證
const user = gun.user();

// 註冊
await user.create('username', 'password');

// 登入（密鑰對由 PBKDF2 從密碼派生）
await user.auth('username', 'password');

// 端對端加密訊息
const sharedKey = await SEA.secret(recipientEpub, senderPair);
const encrypted = await SEA.encrypt('secret message', sharedKey);
const decrypted = await SEA.decrypt(encrypted, sharedKey);

// 資料簽章
const signed = await SEA.sign('important data', senderPair);
const verified = await SEA.verify(signed, senderPub);

三、RAD 儲存引擎（Radisk Storage Engine）

RAD 是 Gun.js 在 Node.js 環境中的預設儲存引擎，基於基數樹（Radix Tree）實作。

3.1 架構設計

  • 記憶體內基數樹 + 磁碟持久化
  • 將 Gun 的圖形資料分塊為傳統檔案
  • 正確合併更新到每個批次
  • 管理高負載下的記憶體分配

3.2 關鍵配置參數


const gun = Gun({
  peers: ['wss://relay.example.com/gun'],
  radisk: true,  // 啟用 RAD 持久化（Node.js 預設）
  axe: false,    // 停用內建 relay（若不需要）
  file: 'data',  // 資料儲存目錄
  // RAD 效能調校
  opt: {
    until: 250,    // 批次寫入間隔（毫秒），預設 250
    batch: 10000,  // 強制刷盤前最大項目數，預設 10K
    chunk: 10 * 1024 * 1024  // 分塊大小（Node.js 預設 10MB）
  }
});

3.3 儲存後端

RAD 對實際儲存引擎沒有意見，支援：

  • Node.js：fs（預設）、Amazon S3、LevelDB
  • 瀏覽器：localStorage（預設，有 5MB 限制）、IndexedDB、OPFS
  • 其他：MongoDB、Cassandra、BitTorrent

四、最佳實踐與常見陷阱

4.1 生產級部署最佳實踐

Relay Server 部署：


// relay-server.js
const Gun = require('gun');
const express = require('express');
const app = express();
const server = app.listen(8765);

const gun = Gun({
  web: server,
  radisk: true,      // 必須啟用持久化！
  axe: false,        // 關閉內建路由（自行管理）
  file: './gun-data', // 指定資料目錄
  multicast: false   // 生產環境關閉組播
});

進程管理：使用 pm2 或 systemd 確保 relay 在機器重啟後自動恢復。

4.2 常見陷阱

| 陷阱 | 說明 | 解法 |

|------|------|------|

| 忘記啟用 radisk | 未設 radisk: true，重啟資料全消 | 明確配置 radisk: true |

| 陣列不支援 | 直接存陣列會靜默失敗 | 使用 .set() 建立無序集合 |

| 刪除困難 | 無法真正刪除，只能設 null（tombstone）| 設計資料模型時預先考慮 |

| ACK 判斷錯誤 | !ack.err 將 null/undefined 誤判 | 改用 ack && ack.ok === true |

| .on() 記憶體洩漏 | 重連後未 .off() 清理監聽 | 重連前清理所有 .on() 訂閱 |

| localStorage 5MB 限制 | 瀏覽器預設用 localStorage | 生產環境改用 IndexedDB |

| 公鑰未驗證 | MITM 可替換 epub 公鑰 | 實作公鑰固定或信任鏈 |

| 網路分區 | 斷線期間的腦裂問題 | 接受最終一致性，設計冪等操作 |

4.3 資料模型設計原則

  1. 扁平化：避免巢狀結構，用引用連結節點
  2. 冪等寫入：操作可安全重試
  3. 時間戳即版本：HAM 依賴時間戳，確保系統時鐘同步
  4. 分區容忍：設計時假設網路可能隨時斷開
  5. tombstone 管理：定期清理 null 標記，避免資料膨脹

五、與 daily-digest-prompt 專案的關聯

5.1 直接應用

本專案今日新增了 Gun.js P2P 聊天室系統，涉及三個元件：

  1. wsc-bot01（Bot 端）：Node.js 執行，使用 Gun.js + SEA 加密與前端通訊，整合 LLM API 處理訊息
  2. my-gun-relay（中繼伺服器）：提供 WebSocket relay + 前端聊天介面
  3. chatroom-scheduler.py（排程器）：與現有 daily-digest-prompt 系統整合

關鍵修復（今日審查報告中識別的 P0 問題）：

  • radisk: true 未啟用 → 重啟資料消失
  • ACK 邏輯 !ack.err 語義錯誤 → 訊息發送狀態判斷錯誤
  • .on() 監聽未清理 → 重連後訊息重複
  • bot epub 公鑰未驗證 → MITM 攻擊面

5.2 架構啟發

  1. 離線優先模式：Gun.js 的離線優先策略與 daily-digest-prompt 的 API 快取降級機制（api-cache Skill）理念一致
  2. 衝突解決：HAM 的 LWW-Register 策略可啟發 scheduler-state.json 的並發寫入問題解決
  3. P2P 通知：未來可用 Gun.js 替代 ntfy.sh 作為自有通知基礎設施
  4. CRDT 知識串聯：本 KB 已有 CRDT 研究（2026-02-26），Gun.js 的 HAM 是 CRDT 的實際應用案例
