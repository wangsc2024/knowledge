系列：gunjs-decentralized-db（optimization 階段）
承接 application 階段知識，本文聚焦生產環境的穩定性、安全性與效能最佳化。

現況架構

my-gun-relay/index.js：204 行 Express + Gun.js relay server

  • Port 8765，SEA ECDH 加密通訊
  • Webhook 端點 POST /api/task（Bearer token 認證）
  • RAD 持久化至 radata/
  • Cloudflare Workers keepalive 每 10 分鐘探測

安全漏洞評估（9 項）

高嚴重度

  1. 無 Rate Limiting：relay 端完全沒有速率限制，DDoS 可直接癱瘓服務
  2. 無安全標頭：缺少 helmet，存在 XSS/clickjacking/MIME sniffing 風險
  3. Token 比較不安全：使用字串直接比較（!==），易受 timing attack

中嚴重度

  1. CORS 過於寬鬆：origin: '*' 對 Gun peer 合理，但 /api/task 不應如此
  2. 未強制 WSS：本地用 WS，需確保外部一律走 Cloudflare TLS
  3. API key 可選：未設定環境變數時直接跳過驗證
  4. Gun.js 版本差異：relay ^0.2020.1241 vs bot ^0.2020.1235

低嚴重度

  1. 無 IP 黑名單
  2. 靜態檔案全暴露：express.static(__dirname) 可能洩漏原始碼

穩定性缺陷評估（8 項）

高嚴重度

  1. 無全域例外處理：未捕獲錯誤直接 crash
  2. 單進程部署：一個 crash = 全服務中斷

中嚴重度

  1. 無 Graceful Shutdown：強制關閉可能損壞 radata
  2. 無記憶體洩漏偵測：長期運行無監控
  3. Heartbeat 僅外部：最長 10 分鐘才發現離線
  4. 無自動重連 peer：外部 peer 斷線後靜默中斷

低嚴重度

  1. radata 無備份策略
  2. 日誌不結構化

穩定性優化方案

1. PM2 進程管理

  • 使用 fork 模式（Gun radata 不支援 cluster 多進程共享寫入）
  • max_memory_restart: 300M
  • 指數退避重啟延遲
  • 配置檔：ecosystem.config.js

2. 全域例外處理 + Graceful Shutdown

  • process.on('unhandledRejection') — 記錄但不退出
  • process.on('uncaughtException') — 給 Gun 1s flush 時間後退出
  • SIGTERM/SIGINT — 先關 HTTP server，5s 超時強制退出

3. 增強 Health Check

  • 回傳 memory（rss/heapUsed/heapTotal）、requestCount、uptimeHuman
  • 每 60 秒自我 heartbeat 檢查（記憶體警告 > 200MB、sharedSecret 重建）

4. 應用層 Rate Limiting

  • express-rate-limit：/api/ 路由 30 req/min
  • Gun WebSocket 由 Gun 內部處理，不受 Express 限制

5. 負載均衡（NGINX）

  • WebSocket proxy（proxy_read_timeout 86400s）
  • API rate limiting zone（30r/m）
  • 拒絕 .js/.json/.md 直接存取

6. radata 備份策略

  • PowerShell 腳本每日 00:10 備份（在 relay 重啟前）
  • 保留最近 7 天，自動清理舊備份

安全性加固方案

1. Helmet 安全標頭

  • contentSecurityPolicy: false（Gun SEA 需要 eval）
  • crossOriginEmbedderPolicy: false（允許跨域嵌入）

2. Timing-Safe Token 驗證

  • 使用 crypto.timingSafeEqual() 替代字串比較
  • 固定長度 Buffer 比較防止長度洩漏

3. 強制 API Key

  • 啟動時檢查 API_SECRET_KEY 環境變數
  • 未設定時拒絕啟動並輸出錯誤訊息

4. 靜態檔案安全

  • express.static 改為只暴露 public/ 子目錄
  • index.html 移至 public/

5. CORS 分級策略

  • /gun 路由：寬鬆 CORS（origin: '*'）
  • /api/ 路由：白名單 origin + 允許無 origin（server-to-server）

6. WSS 加密

  • 方案 A（推薦）：Cloudflare Proxy（現行，確認 Full strict 模式）
  • 方案 B：自建 TLS（Let's Encrypt + https.createServer）

7. 依賴安全掃描

  • npm audit 每週排程
  • high/critical 漏洞透過 ntfy 告警

8. 防火牆規則

  • Windows：僅允許 LocalSubnet 存取 8765
  • Linux：ufw 允許 127.0.0.1 + Cloudflare IP 範圍

9. 請求體大小限制

  • express.json({ limit: '100kb' })

效能最佳化

1. 記憶體管理

  • 每 5 分鐘記錄 heap/rss 結構化指標
  • PM2 max_memory_restart 300M 作為安全網

2. Gun.js 調優

  • axe: false（小規模不需 mesh optimizer）
  • multicast: false（關閉 UDP）
  • chunk: 2MB（RAD 分片大小）
  • until: 2000（防洪等待時間）

3. 結構化日誌

  • JSON 格式日誌（level, msg, data, ts）
  • 替代 console.log 純文字

實施時間表（6 Phase / 4 週）

| Phase | 時間 | 內容 | 優先級 | 工時 |

|-------|------|------|--------|------|

| 1 | Week 1 | Helmet + Timing-Safe + 強制 API Key + 例外處理 + Graceful Shutdown + 靜態檔案 | P0 | 2h |

| 2 | Week 1-2 | Rate Limiting + 請求體限制 + PM2 | P0 | 3h |

| 3 | Week 2 | Health Check 增強 + 備份 + 結構化日誌 | P1 | 2h |

| 4 | Week 3 | CORS 分級 + WSS 驗證 + npm audit 排程 | P1 | 2h |

總預估工時：~13 小時

安全不對稱問題

bot.js（客戶端）已有完整防護（helmet、rate-limit、timing-safe、graceful shutdown、全域例外處理），但 relay（伺服器端）幾乎裸奔。本計畫的核心目標就是消除這個「堡壘客戶端 + 裸奔伺服器」的反常格局。
