一、核心哲學：約束即賦能

"Every time you discover an agent has made a mistake, you take the time to engineer a solution so that it can never make that mistake again." — Mitchell Hashimoto

Harness Engineering 的核心悖論：限縮 Agent 的解空間，反而提升產出效率。當 Agent 可以產生任意輸出時，它會浪費 Token 探索死路；當 Harness 定義清楚邊界，Agent 更快收斂到正確解。

二、三層約束架構（可實作分層）

2.1 依賴層級（Dependency Hierarchy）


Types → Config → Repo → Service → Runtime → UI

每層只能從左側層匯入，透過以下機制機械式強制（非提示詞自律）：

| 強制手段 | 實作方式 | 適用時機 |

|---------|---------|----------|

| 自訂 Linter | 靜態分析掃描 import 路徑 | CI/CD 每次提交 |

| LLM-based Auditor | Agent 程式碼的合規審查 | PR Review 階段 |

| Pre-commit Hook | 提交前攔截違規 | 開發者本地 |

| 結構測試（ArchUnit 式）| 驗證產出結構正確性 | 自動化測試套件 |

2.2 三級 Harness 複雜度

| 等級 | 投入時間 | 實作項目 |

|------|---------|----------|

| L1 | 1-2 小時 | CLAUDE.md/.cursorrules、pre-commit hooks、測試套件、一致目錄結構 |

| L2 | 1-2 天 | AGENTS.md、CI 強制架構約束、共用 prompt 模板、文件 linting |

| L3 | 1-2 週 | 自訂中介軟體、熵值清理 Agent、Harness 版本化、效能儀表板 |

三、反饋迴路實作模式

3.1 中介軟體鏈（Middleware-First Design）

LangChain 的可組合方式，每層中介軟體鏈式串接：


# 概念性 Middleware Pipeline
pipeline = [
    LocalContextMiddleware(),      # 程式庫映射，注入本地上下文
    LoopDetectionMiddleware(),     # 追蹤重複檔案編輯，防止 doom loop
    ReasoningSandwichMiddleware(), # 規劃/驗證高推理力，實作中推理力
    PreCompletionChecklistMiddleware()  # 提交前驗證清單
]

Doom Loop 偵測：中介軟體追蹤檔案編輯次數，當同一檔案被反覆修改超過閾值，自動中斷並請求人類介入。

Reasoning Sandwich：規劃與驗證階段使用高推理力模型（如 o3），實作階段使用中推理力（如 claude-sonnet），最佳化運算成本。

3.2 Generator-Evaluator 模式（Anthropic GAN 式）


Generator Agent → 產出程式碼/內容
       ↓
Evaluator Agent → 使用 Playwright/pytest 等工具做 E2E 驗證
       ↓
   通過 → 提交
   失敗 → 回饋修正指令給 Generator

關鍵發現：模型無法可靠地自我評估。分離 Generator 與 Evaluator 比教會 Agent 自我批評更實際。

3.3 Stripe「兩擊升級」規則

Stripe 的 Minions 系統（每週 1,300+ PR）使用 Blueprint 編排：

  • 確定性節點：Linter 執行、Git commit（不用 Agent）
  • Agent 節點：功能實作、失敗修復
  • 兩擊規則：Agent 連續失敗兩次 → 升級至人類，防止無限重試迴圈

四、自癒機制四層框架

4.1 架構概覽


┌─────────────────────────────────────────┐
│ Layer 4: Learning Loop                  │
│   失敗日誌 → 分析 → 系統改善            │
├─────────────────────────────────────────┤
│ Layer 3: Automatic Recovery             │
│   Context Refresh → Task Simplify →     │
│   Human Escalation                      │
├─────────────────────────────────────────┤
│ Layer 2: Failure Detection              │
│   Pre-action validation:                │
│   quality < 85 || driftScore > 0.3      │
├─────────────────────────────────────────┤
│ Layer 1: Health Monitoring              │
│   品質分數 · 錯誤率 · P95 延遲 ·       │
│   Context Drift 偵測                    │
└─────────────────────────────────────────┘

4.2 自動恢復序列

  1. Context Refresh：清除工作記憶，從持久儲存重新載入
  2. Task Simplification：將複雜動作分解為更小步驟
  3. Human Escalation：自動恢復失敗時標記人類審查

4.3 熵值清理 Agent（Garbage Collection）

定期排程（每日/每週）執行的專責 Agent：

  • 驗證文件與程式碼是否一致
  • 掃描先前遺漏的約束違規
  • 識別並修正模式偏差
  • 追蹤解決循環或不必要的依賴

4.4 量化成效

| 指標 | 導入前 | 導入後 |

|------|--------|--------|

| 失敗率 | 23% | 3% |

| 自動恢復率 | — | 85% |

| 人類介入頻率 | 每日 | 每週 |

五、語義圖作為持久化 Harness（Epsilla 方法）

將約束編碼為圖結構：

"The nodes, edges, and schema of the graph are the constraints."

三大優勢：

  1. 結構性屏障：防止無效 Agent 動作
  2. 持久化反饋迴路：修正永久編碼進世界模型
  3. 可擴展 Agent 協調：透過共享圖狀態取代脆弱的 Agent 間協定

錯誤被轉化為結構性知識，防止再次發生 — 真正的自癒是環境重新設計。

七、對 Daily-Digest-Prompt 專案的應用映射

本專案已實踐多項 Harness Engineering 模式：

| Harness 模式 | 專案對應實作 |

|-------------|-------------|

| Pre-commit Hook 強制 | hooks/pre_bash_guard.py、pre_write_guard.py |

| Loop Detection | agent_guardian.py LoopDetector（doom loop 偵測）|

| Generator-Evaluator | Phase 2 執行 → Phase 3 組裝驗證 |

| 熵值清理 Agent | run-system-audit-team.ps1（每日 00:40 排程）|

| 約束外部化 | config/*.yaml（14+ 配置檔）|

| L2 AGENTS.md | CLAUDE.md + templates/shared/preamble.md |

| Health Monitoring | check-health.ps1（6 維度健康度）|

| 兩擊升級 | config/timeouts.yaml + Guardian CircuitBreaker |

可強化方向

  1. Reasoning Sandwich：Phase 1 規劃用 o3，Phase 2 實作用 sonnet — 目前 llm-router.yaml 已有基礎
  2. 語義圖約束：將 adr-registry.json + research-series.json 結構化為圖，提供 Agent 導航
  3. Pre-Completion Checklist Middleware：在 hook_pipeline.py 新增提交前驗證清單

---

八、設計原則：可拆卸 Harness

Build "rippable harnesses" — 過度工程會在模型進步時失效。
  • 約束應隨模型能力提升而逐步移除
  • 從基礎 Linting 開始，隨模式浮現再新增約束
  • 保持 Provider-Agnostic：Harness 應跨 Claude/GPT/Gemini 運作
