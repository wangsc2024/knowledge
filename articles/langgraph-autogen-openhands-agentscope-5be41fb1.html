技術概述

今日管線中 github_scout 已觀測四個代表性專案：LangGraph、AutoGen、OpenHands、AgentScope。本文從狀態持久化／人機協作（HITL）／安全執行邊界三個軸線，補齊 scout 摘要之外的機制差異，便於在 daily-digest-prompt 這類「多階段排程＋多後端 Agent」系統中選型與對照自家 Harness。[來源：LangGraph Persistence 官方文件, 2025；Microsoft Research AutoGen v0.4 部落格, 2025；OpenHands Runtime Architecture 文件；AgentScope GitHub／arXiv, 2024–2026]

核心概念與原理

1. LangGraph：以 checkpoint 為中心的圖狀態機

LangGraph 在編譯圖時掛載 checkpointer，於每個 super-step 邊界保存圖狀態快照（checkpoint），並以 thread_id 作為執行緒主鍵；官方明列持久化支援 人機協作（檢視／中斷／核准）、跨互動記憶、time travel 重播除錯，以及節點失敗後從最後成功步驟恢復的容錯。來源：LangGraph Persistence, LangChain Docs, 2025

與「僅記對話」不同，文件另引入 Store 介面，用於跨 thread_id 保留使用者級長期記憶；checkpointer 管單一對話緒、store 管跨對話，兩者常並用。[來源：同上]

2. AutoGen 0.4：事件驅動分層與可觀測性

AutoGen v0.4 以 Core／AgentChat／Extensions 三層重構：Core 提供事件驅動的代理基元；AgentChat 提供接近 v0.2 抽象層級的高階 API（如群組對話、程式碼執行）；Extensions 承載模型客戶端與第三方整合。框架強調非同步訊息、模組化元件、內建度量與訊息追蹤，並標榜 OpenTelemetry 支援以利業界標準可觀測性。來源：AutoGen v0.4 部落格, Microsoft Research, 2025

AgentChat 層延續 RoundRobinGroupChat、SelectorGroupChat 等團隊型態，並在部落格與遷移說明中提及串流、狀態儲存與暫停後恢復等能力，適合對話式多代理快速原型。來源：同上；AutoGen Migration Guide, 2025

3. OpenHands：Docker 沙箱與動作執行邊界

OpenHands Docker Runtime 以容器隔離執行任意程式碼；後端透過 REST 與容器內 Action Executor 通訊，於沙箱內初始化 Bash、瀏覽器、Jupyter、外掛等，並回傳觀測結果。文件明確將動機歸因於安全、環境一致性、資源控管、隔離與可重現性。來源：OpenHands Runtime Architecture, 2025

SDK 另將 LocalWorkspace（本機行程）與 Remote／Docker／API 遠端 Workspace 分層，使同一套 Agent API 可在開發與生產隔離環境間切換。來源：OpenHands SDK Architecture Overview

4. AgentScope：訊息導向編排與工具生態

AgentScope（阿里通義實驗室）以 production-ready 為定位，README 與 PyPI 說明強調內建 ReAct、工具／記憶、MCP 與 A2A 支援，以及 MsgHub／pipeline 等多代理編排；學術論文將平台描述為以訊息交換為核心的開發者導向多代理平台，並論及分散式與管線化互動模式。來源：agentscope-ai/agentscope GitHub, 2026；AgentScope: A Flexible yet Robust Multi-Agent Platform, arXiv:2402.14034, 2024

最佳實踐與常見陷阱

  • 先決定「狀態單位」：若工作流需精確從中斷點續跑與稽核每一步，優先評估 LangGraph 的 thread／checkpoint 模型是否可直接映射到你的 task_key／Phase。來源：LangGraph Persistence, 2025
  • 對話型多代理＋工具：AutoGen AgentChat 適合快速組隊與 UI／Studio 驗證；注意 v0.4 與 v0.2 執行模型差異，依官方遷移指南調整。來源：Microsoft Research Blog, 2025
  • 執行不可信程式碼：OpenHands 類 容器化 Action Executor 是強邊界；本機 subprocess 僅適合受信任場景。來源：OpenHands Runtime Architecture, 2025
  • 與 MCP／互通協定：若管線已採 MCP 工具，AgentScope／n8n／多框架的 MCP 整合度可作為選型加權；避免僅看 Stars 數忽略執行隔離與可觀測性。來源：AgentScope GitHub, 2026

與 daily-digest-prompt 的關聯

  • Phase 1→2→3 與 thread_id 類比：每日自動任務可對照「有 checkpoint 的工作流」— 結果 JSON、scheduler-state、continuity 檔扮演不同層級的狀態儲存；LangGraph 的 thread／checkpoint／store 分層有助區分「單次執行緒狀態」與「跨任務使用者／專案記憶」。
  • 多後端與沙箱：任務內文提到的 Cursor CLI／Codex／OpenRouter 備援，對應 OpenHands 文件中的「本機 vs 隔離執行」取捨：高風險動作應落在具 Action Executor 邊界的環境，而非僅依 LLM 自律。
  • 可觀測性：AutoGen 對 OpenTelemetry 的強調與本專案 JSONL Hook、Phoenix／OTel 研究方向一致，可作為多代理實驗時的統一追蹤層選項。來源：Microsoft Research Blog, 2025

程式碼範例（概念對照）

LangGraph 掛載 checkpointer 與指定 thread_id（節錄自官方文件模式）：


from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)
config = {"configurable": {"thread_id": "daily-digest-phase2"}}
graph.invoke(initial_state, config)

來源：LangGraph Persistence, 2025
