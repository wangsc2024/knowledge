研究類型：技術研究

今日任務涉及技術：[RAG 知識庫、GitHub 熱門專案研究、Vite 靜態同步、專案優化、PowerShell 自動化、知識庫洞見整理]

本次研究主題：GitHub 熱門專案研究工作流：GraphQL Search、排序訊號與去重策略。

選題原因：今日已完成任務中，至少兩項 recurring task 直接要求「找出熱門 GitHub 專案並深入研究」或「研究 GitHub 最熱門方案」，但現有知識庫多是單一專案個案研究，缺少一套可重複執行的搜尋、排序、去重與人工審核流程。

技術概述

GitHub 熱門專案研究的真正難點，不是能不能列出 star 數高的 repository，而是能否穩定找出「對目前任務有用、不是舊題重複、且值得深入」的候選清單。到 2026 年，較穩的做法是把 GitHub GraphQL Search 當成候選蒐集層，用 search query 一次拿到 repository 名稱、star、fork、更新時間、topic 與 owner 等必要欄位；再配合搜尋語法與排序限定詞先做粗篩；最後在本地或 RAG 知識庫端做去重與品質分數。對 daily-digest-prompt 而言，這比單靠網頁 Trending 或手動搜尋更穩，也更容易寫成自動任務。

核心概念與原理

1. GraphQL Search 是候選蒐集層，不是最終決策器

GitHub GraphQL search 查詢可跨資源搜尋，並支援 GitHub 的搜尋語法；官方文件也明確說明 search connection 最多回傳 1,000 筆結果。這代表它適合做「先縮小宇宙，再分頁抓候選」，不適合拿來掃描整個 GitHub。實務上應把 query 寫成高訊號條件，例如：

  • topic:rag language:python stars:>500 pushed:>=2025-01-01 sort:stars-desc
  • topic:mcp language:typescript archived:false mirror:false sort:updated-desc
  • topic:agent-framework good-first-issues:>5 stars:>300 sort:stars-desc

先限制主題、語言、活躍度與最小 star，才能讓後續去重與人工審核成本可控。

2. 排序要分成探索排序與研究排序

GitHub 文件指出，一般搜尋結果可以用 Sort 選單或 sort 限定詞依 relevance、stars、forks、updated 等訊號排序。對研究任務而言，至少要拆成兩種排序模式：

  • 探索排序：sort:updated-desc 或 sort:stars-desc，目的是先抓近期仍活躍的熱門專案。
  • 研究排序：在本地再加權 stars、updatedAt、forkCount、topic 命中數、是否 archived、是否已有 KB 筆記。

這樣能避免「高 star 但兩年沒動」與「剛爆紅但品質未確認」混在一起。

3. GraphQL 的優勢是欄位精準，代價是要管成本與限制

GitHub GraphQL 文件強調它相對 REST 更精準、更彈性，但也有明確限制：連線必須帶 first/last，單次通常 1 到 100，總 node 不可超過 500,000；對 GitHub.com 的一般使用者，常見 primary rate limit 是每小時 5,000 points。另有 secondary rate limits，例如最多 100 個 concurrent requests，以及 GraphQL endpoint 每分鐘 2,000 points。這表示自動研究工作流不能無腦併發，也不能把 repository、issues、releases、README、topics 全塞進一支查詢；應先用輕量 search 撈候選，再對前 5 到 10 個候選做第二階段細查。

4. 2025-11-07 後，官方已不鼓勵在文件內直接試 GraphQL Explorer

GitHub changelog 於 2025-11-07 明確宣布退休 GraphQL Explorer，並建議改用本地 GraphQL client。對 daily-digest-prompt 的意義很直接：若未來要把 GitHub 熱門專案研究任務自動化，應優先採用 gh api graphql、Insomnia、Altair 或腳本化 client，而不是依賴文件頁內的互動測試工具。這也更適合排程與 CI。

最佳實踐與常見陷阱

最佳實踐

  1. 先窄查詢，再補細節。第一階段只抓 nameWithOwner、url、stargazerCount、forkCount、updatedAt、repositoryTopics、isArchived。
  2. 查詢條件固定模板化。將 topic、language、stars、pushed、sort 做成參數，避免每次人工改寫造成偏差。
  3. 搜尋結果進 KB 前先做去重。以 repository URL 或 owner/name 當自然鍵，並比對近 7 天 registry 與知識庫標題。
  4. 用二階段評分。第一階段用 GitHub 訊號篩掉低價值 repo；第二階段再用專案需求相關性、文件品質、近 90 天活躍度與議題成熟度評估。
  5. 若使用 GitHub CLI，優先 gh api graphql。官方 CLI 手冊已支援 graphql endpoint 與 pagination，適合 terminal / script workflow。

常見陷阱

  1. 只看 stars。star 是人氣訊號，不是可研究性或可落地性的保證。
  2. 忽略 1,000 結果上限。若 query 太寬，後半段熱門專案根本抓不到。
  3. 過度深查單一 query。容易撞 GraphQL node limit、timeout 或 secondary rate limit。
  4. 不做 archived / mirror 過濾。會把已停止維護或鏡像倉庫誤列為候選。
  5. 不做跨類型去重。GitHub 研究、todoist_research、tech_research 都可能重複分析同一 repo。

與本專案的關聯

對 daily-digest-prompt，這套工作流可直接支援兩類任務：

  • 研究1項 GitHub 最熱門方案並將研究結果寫入 RAG 知識庫
  • 參考 RAG 知識庫找出一個熱門的 GitHub 專案運用 skill 進行深度瞭解

建議落地方式：

  1. 用 GraphQL Search 取得候選 repo 清單。
  2. 用本地規則計算研究分數，例如：score = stars_weight + recency_weight + topic_match - duplicate_penalty。
  3. 對前 3 名候選做 README / release / docs 深查。
  4. 匯入 KB 前，先比對 context/research-registry.json 與 KB hybrid search，避免跨任務重複。
  5. 把最終候選與淘汰理由一起寫進結果 JSON，方便之後 audit。

程式碼範例


gh api graphql -f query='\
query($q:String!, $endCursor:String) {\
  search(query:$q, type:REPOSITORY, first:20, after:$endCursor) {\
    pageInfo { hasNextPage endCursor }\
    nodes {\
      ... on Repository {\
        nameWithOwner\
        url\
        description\
        stargazerCount\
        forkCount\
        updatedAt\
        isArchived\
        repositoryTopics(first:10) {\
          nodes { topic { name } }\
        }\
      }\
    }\
  }\
}' -F q='topic:mcp stars:>300 archived:false sort:stars-desc'

後處理規則示意：


保留條件：
- isArchived = false
- updatedAt 在近 180 天內
- KB / registry 無完全相同 topic
- repositoryTopics 命中本次研究主題

排序建議：
- 40% stars
- 25% recency
- 20% topic match
- 15% docs/readme quality
