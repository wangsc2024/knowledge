---

一、技術概述

AI-Driven Infrastructure as Code（AI-IaC）是將大型語言模型（LLM）整合到基礎設施即程式碼工作流程中的新興範式。傳統 IaC 要求工程師手動撰寫 Terraform HCL、Pulumi TypeScript/Python 或 CloudFormation YAML，而 AI-IaC 透過自然語言提示自動生成、審查與修復基礎設施配置，將佈建中等複雜度多雲環境的時間從 2-3 天壓縮至 2 小時以內。2026 年全球雲端運算支出預計超過 9,400 億美元，AI 驅動的漂移偵測、自動修復建議與策略即程式碼生成已從實驗階段進入生產環境，主要 IaC 供應商已將 LLM 驅動助手直接整合到 CLI 工具中。

三、TerraFormer 神經符號框架（學術前沿，2026）

架構設計

TerraFormer 是結合監督微調（SFT）與強化學習（RL）的神經符號框架，透過形式化驗證工具的回饋來訓練 LLM 生成可部署的 IaC 配置。

三層形式化驗證器：

  1. FV-i（語法驗證）：terraform validate 確保 HCL 語法正確
  2. FV-ii（可部署性驗證）：terraform plan 確認配置可實際部署
  3. FV-iii（策略合規驗證）：Open Policy Agent (OPA) 檢查安全與合規策略

訓練數據：

  • TF-Gen：152,475 個生成實例（自然語言 → Terraform 配置）
  • TF-Mutn：52,516 個突變實例（首個 IaC 突變數據集，模擬真實世界的配置修改）

強化學習獎勵設計（GRPO）：

  • 0 分：編譯失敗
  • 0.5 分：編譯成功
  • 1.0+ 分：部署成功 + 策略合規度比例分（最高 2.0 分）

基準測試結果

| 模型 | 參數量 | TF-Gen 正確率 | TF-Mutn 正確率 |

|------|--------|---------------|----------------|

| GPT-4.1 | 未知 | 10.37% | 41.62% |

| DeepSeek-R1 | 671B | 11.65% | 42.03% |

| Claude Sonnet 3.7 | ~100B+ | 16.90% | 52.03% |

| TerraFormer† (14B) | 14B | 18.04% | 55.14% |

關鍵發現：僅 14B 參數的 TerraFormer 在正確率上超越了 50 倍以上參數量的通用模型，達到 100% Linter 合規和 69.34% 安全合規。

創新點

  1. 首個 IaC 突變數據集：TF-Mutn 更貼近真實開發（修改現有配置而非從零生成）
  2. 可擴展自動化數據策劃：LLM 引導的修復迴圈成本約 $15,000，對比人工標註 ~$500 萬
  3. 模組化管線：可擴展到 Pulumi、Ansible、CloudFormation
  4. 形式化驗證作為訓練信號：將編譯錯誤和策略違規轉為結構化獎勵，超越傳統稀疏二元回饋

四、實際工作流程（Step-by-Step）

4.1 AI 輔助 Terraform 生成工作流


步驟 1：自然語言描述需求
  「為 AWS 3 層應用建立 Terraform：
   VPC（3 AZ 公有/私有子網），ALB 在公有子網，
   ASG 在私有子網，RDS 在私有子網，最小端口 SG」

步驟 2：AI 生成初始配置
  → versions.tf / variables.tf / main.tf / outputs.tf

步驟 3：迭代精修
  → 「加入 KMS 加密 RDS 和 EBS」
  → 「加入 CloudWatch 警報 ASG CPU > 70%」
  → 「啟用 ALB 存取日誌到 S3」

步驟 4：安全掃描
  → checkov -d . / tfsec .

步驟 5：策略合規檢查
  → OPA/Sentinel 驗證

步驟 6：部署驗證
  → terraform plan → 人工審查 → terraform apply

4.2 Prompt Engineering for IaC 最佳實踐

  1. 明確指定檔案結構：要求分離 versions.tf、variables.tf、main.tf、outputs.tf
  2. 提供應用程式碼上下文：讓 LLM 理解環境變數需求和資料庫 schema
  3. 明確資源依賴順序：指出 ECR registry 必須先於 Docker image push
  4. 指定命名規範和標籤標準：確保生成的資源符合組織慣例
  5. 請求特定輸出：指明下游操作需要的 output 值

五、程式碼/配置範例

5.1 AI 生成的 Terraform 模組結構


# versions.tf
terraform {
  required_version = ">= 1.9"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# main.tf（AI 生成的 VPC 模組）
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.16.0"

  name = var.project_name
  cidr = var.vpc_cidr

  azs             = ["ap-northeast-1a", "ap-northeast-1c", "ap-northeast-1d"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]

  enable_nat_gateway = true
  single_nat_gateway = true

  tags = var.common_tags
}

5.2 OPA 策略合規範例（Rego）


package terraform.security

# 禁止 0.0.0.0/0 入站規則
deny[msg] {
  resource := input.resource.aws_security_group_rule[name]
  resource.type == "ingress"
  resource.cidr_blocks[_] == "0.0.0.0/0"
  msg := sprintf("SG rule '%s' allows 0.0.0.0/0 ingress — use specific CIDRs", [name])
}

# 強制 S3 加密
deny[msg] {
  bucket := input.resource.aws_s3_bucket[name]
  not bucket.server_side_encryption_configuration
  msg := sprintf("S3 bucket '%s' missing encryption", [name])
}

七、與本專案的應用場景

daily-digest-prompt 專案可從 AI-IaC 受益：

  1. 排程基礎設施自動化：若未來遷移到雲端，AI 可自動生成 Terraform 配置（定時觸發器、Lambda 函數、通知服務）
  2. Policy-as-Code 理念借鑑：OPA/Rego 的策略合規模式可應用到本專案的 Hook 系統（hook-rules.yaml 已實現類似概念）
  3. TerraFormer 的 GRPO 獎勵設計：「漸進式結構化獎勵」（語法→部署→合規）可啟發本專案的 quality-gate 品質閘門分層驗證
  4. 形式化驗證作為訓練信號：概念可應用到 Hooks 測試自動化（validate_config.py 的 YAML schema 驗證已是簡化版本）
  5. 漂移偵測與自動修復：Pulumi Neo 的 AI 自動修復模式對應本專案的 self-heal 自愈迴圈（ADR-008 OODA 閉環）

八、四大風險與注意事項

| 風險類型 | 說明 | 緩解策略 |

|---------|------|----------|

| 不安全預設值 | AI 可能生成 0.0.0.0/0 入站規則、未加密資源 | 強制 Checkov/tfsec CI 掃描 + OPA 策略阻擋 |

| 憑證洩漏 | 在公共 LLM 提示中暴露 AWS 金鑰/密碼 | 使用 Vault 管理密碼、永不在提示中包含憑證、優先自建私有模型 |

| 幻覺配置 | 生成不存在的參數或無效模組版本 | 必須 terraform plan 驗證 + 人工審查 |

| 供應鏈攻擊 | 引用未經審核的第三方模組可能含後門 | 模組白名單制度、僅允許經審核的 registry 來源 |

黃金法則：將 AI 生成的 IaC 視為「資淺工程師的程式碼」— 必須經過同行審查、安全掃描和策略驗證後才可合併。
