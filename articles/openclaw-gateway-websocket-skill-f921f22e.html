執行摘要

OpenClaw 作為 GitHub 史上增長最快的開源專案（210K+ stars），其核心競爭力不僅在於多頻道整合（30+ 渠道），更在於其精心設計的內部機制：WebSocket 雙工協議路由、Markdown-to-Prompt 技能熱載入與Lane-based FIFO 並發控制。本文深入 mechanism 階段，透過官方文件、源碼分析與社群討論的三角佐證，揭示 OpenClaw Gateway 的核心實作細節。

關鍵發現：

  1. Gateway 採用單 port 多協議設計（HTTP + WebSocket on 18789），透過 attachGatewayUpgradeHandler() 動態升級連線 dev.to, 2026
  2. Skill 系統本質是「Prompt 工程即程式碼」— Markdown 指令直接注入 Agent context，MCP Server 提供工具執行能力 OpenClawMCP Blog, 2026
  3. Lane-based Queue 確保 Session 序列化（同一用戶訊息永不並行）同時支援多 Lane 並行（main/cron/subagent 各自獨立並發預算）LumaDock, 2026

一、Gateway WebSocket 路由機制

1.1 架構概覽：Hub-and-Spoke 模型

OpenClaw 採用中央化的 Hub-and-Spoke 架構 OpenClaw Architecture，Gateway 作為唯一的訊息路由中樞，連接：

  • Spokes（輻射端）：30+ 頻道整合（WhatsApp、Telegram、Slack、Discord、iMessage 等）
  • Hub（中樞）：Gateway 背景程序（預設 port 18789）
  • Agent Runtime：實際處理 LLM 請求與工具執行的運行時

關鍵設計決策：單一 Gateway 程序管理所有頻道連接（"single process, multi-channel" philosophy Gateway Protocol），避免傳統多程序架構的 Session 同步問題。

1.2 WebSocket vs HTTP：為何選擇全雙工協議？

傳統 AI Agent 框架多採用 HTTP Request-Response 模型，但 OpenClaw Gateway 選擇 WebSocket 作為核心協議，原因如下 [Meta Intelligence, 2026]：

| 需求 | HTTP | WebSocket | OpenClaw 選擇 |

|------|------|-----------|-------------|

| 即時任務派發 | ❌ Client 需輪詢 | ✅ Server 主動推送 | WebSocket |

| 長時間執行進度回報 | ❌ 需 Server-Sent Events | ✅ 雙向即時通訊 | WebSocket |

| 連線維持成本 | ❌ 每次請求建立新連線 | ✅ 持久連線 | WebSocket |

| 多 Agent 並行通訊 | ❌ 需額外協調層 | ✅ 單一連線多路復用 | WebSocket |

實作細節：Gateway 監聽 WebSocket upgrade 事件於 attachGatewayUpgradeHandler() (src/gateway/server-http.ts:650)，將帶有 Upgrade: websocket header 的請求轉發給 WebSocketServer (ws library)，一般 HTTP 請求則由 handleRequest() 處理 How OpenClaw Serves HTTP, WebSocket on Single Port。

1.3 Gateway Protocol v3：統一訊息格式

OpenClaw 目前使用 Protocol Version 3 (PROTOCOL_VERSION = 3)，所有通訊透過 JSON 編碼的 GatewayFrame discriminated union 進行 DeepWiki。

訊息路由流程（已驗證✅）：


inbound message → access control checks → session resolver → agent dispatcher

多 Agent 路由（binding 配置）OpenClaw Multi-Agent：


# 範例：不同 WhatsApp DM 路由到不同 Agent
bindings:
  - match:
      peer.kind: "direct"
      peer.e164: "+15551234567"
    agent: "personal-assistant"
  - match:
      peer.kind: "direct"
      peer.e164: "+15559999999"
    agent: "work-automation"

二、Skill 熱載入與 Markdown-to-Prompt 機制

2.1 Skill 解剖：YAML Frontmatter + Markdown 指令

OpenClaw Skill 本質是「可執行的 Prompt 文件」OpenClaw Skills Guide，檔案結構：


---
name: web-search
version: 1.0.0
description: Search the web using SerpAPI
dependencies:
  - serpapi-mcp
---

# Web Search Skill

When user asks to search the web:
1. Use serpapi_search tool with query parameter
2. Parse top 3 results
3. Summarize findings

關鍵洞察：Markdown 本體直接注入 Agent 的 context window，成為 LLM 的 system prompt 一部分 Build Custom OpenClaw Skill with MCP。

2.2 MCP Server：Tools 的執行層

Skill (指令) 與 MCP Server (工具) 的分工 VoltAgent/awesome-openclaw-skills：

| 層級 | 職責 | 技術 |

|------|------|------|

| Skill (SKILL.md) | 定義「何時」「如何」使用工具 | Markdown Prompt Engineering |

| MCP Server | 提供實際可呼叫的工具 | TypeScript/Python + MCP SDK |

實例：serpapi-mcp 提供 serpapi_search 工具，web-search.md Skill 告訴 Agent「用戶問『今天天氣』時呼叫此工具」。

2.3 Skill Registry 與熱載入（⚠️ 待深入驗證）

已知機制（來自官方文件與社群討論）：

  • Skill 位於 ~/.openclaw/workspace/skills/ 目錄
  • OpenClaw 可自動生成 Skill 框架 (openclaw skill generate)
  • ClawHub 為官方 Skill 註冊中心（5,700+ 技能）GitHub openclaw/openclaw

未驗證細節（需源碼分析補充）：

  • Skill 載入時機（啟動時 vs. 運行時）
  • YAML metadata 如何解析為依賴關係
  • Skill 更新後是否需重啟 Gateway

三、Session 並發控制與 Task Queue 實作

3.1 核心可靠性保證：序列化寫入 Per Session

OpenClaw Queue 系統的核心承諾 OpenClaw Command Queue：

"serialize writes per session, throttle global work, and define exactly what happens when new input arrives mid-run"

Session Key 鎖定機制 LumaDock Concurrency Guide：

  • 每個 Session 進入 Queue 時分配唯一 lane：session:
  • 同一 lane 的訊息嚴格 FIFO 序列化（保證對話連貫性）
  • 不同 Session 的 lane 可並行處理（互不阻塞）

3.2 Lane-Based 並發控制

OpenClaw 定義三個預設 Lane OpenClaw Session Concurrency, Medium：

| Lane | 用途 | 預設並發上限 | 隔離價值 |

|------|------|------------|----------|

| main | 用戶即時訊息 | 4 | 防止 cron job 阻塞對話 |

| cron | 定時任務 | 未明確（可配置） | 長時間任務不影響主對話 |

| subagent | sessions_spawn 子任務 | 8 | 並行子任務處理 |

關鍵設計：不同 Lane 資源獨立，cron job 長時間執行不會搶占 main lane 的並發預算 GitHub Issue #1159。

3.3 Mid-Run Message Handling

當新訊息到達時，系統有三種策略 OpenClaw Session Management：

  1. Steer current run：將新輸入注入當前執行中的 turn
  2. Wait for followup turn：排入 Queue 等待當前 turn 完成
  3. Both：同時執行上述兩者

Session Isolation 配置（per-channel-peer）Session Tools：


session:
  dmScope: per-channel-peer

確保每個用戶獨立 Session lane，A 用戶的長任務不阻塞 B 用戶。

四、對 Daily-Digest-Prompt 的借鑑啟示

4.1 Gateway 模式 vs. 多 Agent 並行

現狀：daily-digest-prompt 的 run-todoist-agent-team.ps1 採用 PowerShell 啟動多個獨立 claude -p 程序，透過檔案系統 (results/*.json) 協調。

OpenClaw 啟示：

  • 單一 Gateway 程序：統一管理所有 Agent 連線，避免檔案鎖定競爭
  • WebSocket 通訊：Agent 之間透過 Gateway 即時通訊，取代輪詢檔案系統
  • Session Registry：集中管理進行中的任務，減少 scheduler-state.json 寫入衝突

4.2 Skill 熱載入 vs. 靜態模板

現狀：daily-digest-prompt 有 27 個 SKILL.md，但需手動更新 SKILL_INDEX.md，Agent 啟動時一次性載入所有 Skill。

OpenClaw 啟示：

  • 按需載入：只在任務需要時載入對應 Skill（減少 context window 佔用）
  • 自動索引：掃描 skills/ 目錄自動建立 Registry（類似 ClawHub）
  • Skill 依賴解析：YAML frontmatter 的 dependencies 欄位自動載入所需 MCP Server

4.3 Lane-Based Queue vs. Round-Robin 輪轉

現狀：daily-digest-prompt 的自動任務用 execution_order 輪轉，無法處理「長時間研究任務阻塞快速通知任務」的問題。

OpenClaw 啟示：

  • 優先級 Lane：high-priority（即時通知）、normal（todoist 任務）、background（研究）
  • Per-Task Session Key：相同任務（如 ai_deep_research）序列化，不同任務並行
  • 並發預算分配：high-priority lane 配置更多 slots

4.4 Session Transcript vs. continuity JSON

現狀：context/continuity/auto-task-*.json 儲存最近 5 次執行記錄。

OpenClaw 啟示：

  • JSONL 格式：每次執行 append 一行，不需讀取-修改-寫入（避免並發問題）
  • 基於 Session 的記憶：不同頻道（todoist/ntfy/slack）各自獨立 transcript
  • 自動壓縮：舊記錄定期歸檔或摘要（OpenClaw 源碼有相關實作 GitHub security）

4.5 WebSocket 心跳與重連（⚠️ 待驗證）

潛在需求：daily-digest-prompt 的 Gun.js relay 需要斷線重連機制。

OpenClaw 啟示（需源碼驗證）：

  • Gateway 應實作 WebSocket ping/pong 心跳
  • Client 斷線後自動重連，Session 狀態保留
  • 重連後繼續執行中的任務（需 Task 狀態持久化）

4.6 安全強化（2026 年更新）

OpenClaw 最新安全措施 GitHub Releases 2026.3.2：

  1. 拒絕 symlink/hardlink 種子檔（防 workspace 外檔案注入）
  2. Device pairing 改用短期 bootstrap token（取代共享 Gateway 憑證）
  3. 停用 workspace plugin 自動載入（需明確信任決策）

daily-digest-prompt 應用：

  • Hook 系統應檢測 symlink（防繞過 allowed_write_paths 白名單）
  • ntfy token 應有 TTL（目前長期有效）
