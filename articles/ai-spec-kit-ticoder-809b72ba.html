AI 需求工程中的意圖形式化：Spec Kit、TiCoder、nl2postcond 與 repo-native 評估閉環

研究日期：2026-04-24

研究類型：ai_sysdev

技術概述

2025-2026 的 AI 需求工程，關鍵不再只是讓模型代寫 PRD、user story 或 acceptance criteria，而是把自然語言需求逐步轉成「可檢查、可追溯、可回歸驗證」的規格資產。GitHub 的 Spec Kit 把 spec、plan、tasks 放到開發主線，讓需求先於實作；Microsoft Research 則把這個問題往更嚴格的方向推進，提出 intent formalization，把模糊需求轉成 tests、postconditions 甚至 DSL 等可檢查規格。再往後一層，GitHub Models 讓 prompt、測試資料與 evaluators 直接存進 repo，以 .prompt.yml 形式做比較與評分。三者組合後，需求工程可以形成一條閉環：需求輸入 → 規格化 → 模糊點澄清 → 驗收條件形式化 → repo-native 評估與 CI 閘門。

核心方法與工具

1. Spec-Driven Development（GitHub Spec Kit）

  • 核心思想：規格不再只是開會文件，而是可驅動後續 plan、tasks 與 implementation 的主工件。
  • 標準流程：/speckit.constitution → /speckit.specify → /speckit.plan → /speckit.tasks → /speckit.implement。
  • 實務價值：先固定「要做什麼」與「品質原則」，再讓 AI 實作，能降低 prompt 漂移與需求漏失。

2. Intent Formalization（Microsoft Research, 2026）

  • 核心命題：AI 寫出的程式碼是否真的符合使用者意圖。
  • 方法：把自然語言需求翻成可檢查的 formal specifications，層級可從 tests、assertions、functional specs 到 DSL。
  • 關鍵瓶頸：規格正確性沒有天然 oracle，仍需人類透過輕量互動與 proxy artifacts 驗證。

3. TiCoder / TDUIF（Test-Driven User-Intent Formalization）

  • 做法：先讓模型生成候選程式與候選測試，再讓人類只回答少量「這個測試是否符合意圖」。
  • 作用：把最模糊、最容易分叉的需求點用測試明確化，避免直接接受第一版生成程式。
  • 適合情境：需求不完整、邊界條件多、函式行為容易誤解的任務。

4. nl2postcond

  • 做法：把 docstring、註解或自然語言規格轉成 formal postconditions/assertions。
  • 用途：補足 acceptance criteria 太口語、太難直接機器檢查的問題。
  • 實務定位：介於 user story/BDD 與完整形式驗證之間，成本較低，但已可形成有效 guardrail。

5. GitHub Models repo-native evaluation

  • 將 prompt、testData、expected output、evaluators 直接存成 .prompt.yml。
  • 可以在 UI 或 CLI 中對多模型、多 prompt 版本做一致性比較。
  • evaluator 可用 similarity、relevance、groundedness，也可用 custom prompt 做 pass/fail gate。

實際工作流程（step-by-step）

工作流 A：需求澄清到可執行規格

  1. 收集來源材料：PRD、會議紀錄、issue、使用者回饋、既有限制。
  2. 先寫 constitution：品質原則、回應時限、資料保護、可觀測性、驗收責任邊界。
  3. 用 Spec Kit 產生 feature spec，限定先描述 what/why，不要太早進入 framework 細節。
  4. 把 spec 中高歧義條目抽出，要求模型生成 candidate examples / candidate tests。
  5. 由 PM、工程師或 domain owner 快速標記 Yes/No/Undef，保留真正符合意圖的例子。
  6. 將通過的例子轉成 Given-When-Then、單元測試或 postconditions。
  7. 把這些規格連到 tasks 與實作工作單，要求每個 task 對應至少一個可機器檢查的 acceptance artifact。

工作流 B：驗收條件 repo 化與評估閘門

  1. 把 prompt 與 acceptance cases 寫進 .prompt.yml。
  2. 對每個需求至少準備 3 類案例：正常路徑、邊界條件、反例/禁止行為。
  3. 設定 evaluator：similarity 檢查是否貼近預期，relevance 檢查是否有答到需求，groundedness 檢查是否超出輸入事實。
  4. 需要更嚴格時，用 custom prompt evaluator 檢查「是否滿足 AC-001 到 AC-005」。
  5. 在 PR 或 nightly CI 執行評估；若分數低於門檻或 fail，阻擋合併。
  6. 新需求變更時先改 spec / prompt / testData，再更新實作，避免 code 先跑。

程式碼／配置範例

範例 1：需求規格片段（EARS + AC）


# Feature: 自動摘要結果檔驗證

## User Story
作為維運者，我要在自動任務結束後立刻驗證 `results/todoist-auto-*.json`，
以便在 Phase 3 組裝前攔住格式錯誤。

## EARS
When an auto task writes a result JSON,
the validation service shall check required fields within 5 seconds.

## Acceptance Criteria
- AC-001: 缺少 `task_key` 時必須回傳 `failed` 或 `format_failed`
- AC-002: `summary` 長度不足時不得標記 `success`
- AC-003: 驗證失敗時必須留下可診斷訊息

範例 2：將驗收條件寫進 `.prompt.yml`


name: Result Validation Requirement Check
model: openai/gpt-4.1
modelParameters:
  temperature: 0.2
messages:
  - role: system
    content: |
      你是需求驗收評估器。只根據輸入 JSON 與 acceptance criteria 判斷。
  - role: user
    content: |
      驗收條件：
      {{criteria}}

      待評估輸出：
      {{candidate_output}}

testData:
  - criteria: |
      AC-001: 缺少 task_key 時必須失敗
      AC-002: summary 長度不足時不得 success
    candidate_output: |
      {"status":"success","summary":"太短"}
    expected: fail

evaluators:
  - name: AC gate
    prompt:
      content: |
        若 candidate_output 違反任一 acceptance criteria，輸出 fail；否則輸出 pass。
        只允許 pass 或 fail。

範例 3：用 postcondition 形式化模糊需求


def remove_duplicates(numbers):
    result = ...
    assert all(numbers.count(i) == 1 for i in result)
    assert all(i in numbers for i in result)
    return result

這類 assertion 比「移除重複元素」更可檢查，也更適合交給 AI 與測試框架驗證。

效果量化

  • Microsoft Research 的 TiCoder 論文指出，在 MBPP 上，最佳方法把 pass@1 從 48.39% 提升到 70.49%；若最多允許 5 次互動，可到 85.48%。
  • 同一研究中，系統能在平均 1.69 次使用者查詢內，為 90.40% 的樣本產出符合意圖的非平凡功能測試。
  • nl2postcond 專案報告指出，LLM 生成的 postconditions 能抓到 Defects4J 中 64 個真實歷史 bug，說明自然語言規格轉 formal assertions 不只是學術展示。
  • GitHub Models 文件顯示，prompt 與 testData 可以直接 repo 化，並以 similarity、relevance、groundedness 等 evaluator 做可重複比較；這代表需求驗證可進入 CI，而不是留在會議或人工 demo。

與本專案應用場景

  • 本 repo 已大量依賴 prompt、workflow、schema、result JSON；最適合導入的是「需求先 repo 化，再讓 agent 實作」。
  • 可把 results/todoist-auto-*.json、done_cert、cli_exit_note_zh、result_guard_zh 轉成正式 acceptance criteria，放進 spec 或 .prompt.yml。
  • 對 workflows/.yaml、templates/shared/.md 的新增需求，可先生成 EARS/BDD 與 counterexample cases，再交給 agent 改檔。
  • 對高風險自動任務（如 schema 驗證、KB 匯入、Phase handoff），可增加 custom evaluator，檢查輸出是否符合既定契約。
  • 對 prompt 變更，可建立 regression dataset，避免需求描述小改卻讓 agent 行為飄移。

風險與注意事項

  • 不能把「生成更多需求文字」誤當成需求工程完成；沒有可檢查 artifact，最後仍是 vibe coding。
  • Intent formalization 不是零人工。真正該由人判斷的，是歧義點、風險邊界與規格優先順序。
  • evaluator 本身也可能有偏差，因此高風險 gate 應搭配 deterministic checks，例如 JSON Schema、單元測試、字串規則與 policy-as-code。
  • 需求形式化的成本要分層投入：低風險功能用 examples/tests 即可，高風險模組才上升到 contracts 或 DSL。
  • 若 repo 沒有維護好 traceability，spec、tests、prompt、code 很快又會分離。
