二、五大瓶頸深度分析

瓶頸 1：快取命中率歸零（P0 — 最高優先）

數據：digest-memory.json 的 trends 記錄連續 4 天（2/25-2/28）cache_hits = 0，同時 api_calls = 5-7/次。

根因推斷：

  1. 團隊模式中每個 Sub-Agent 是獨立 claude -p 進程，無法共享記憶體快取
  2. cache/*.json 的 TTL 檢查可能在每次 Phase 1 啟動時判定為過期（Phase 1 間隔 > TTL）
  3. 每日摘要 3 次執行（08:00/11:15/21:15）間隔 3-10 小時，遠超多數 API 的 TTL
  4. Todoist 每小時整點/半點觸發，API 快取 TTL 若 < 30 分鐘則永遠失效

影響：每次都打真實 API，增加延遲（Phase 1 平均 114s > 60s 目標）和 API quota 壓力。

建議修正：

  • 調整 config/cache-policy.yaml 中 Todoist API 的 TTL 為 25 分鐘（同小時整點和半點可共享快取）
  • 新聞/HN 的 TTL 調至 3 小時（同日 3 次每日摘要可命中 2 次）
  • 在 run-todoist-agent-team.ps1 的 Phase 1 結束後，將 API 回應寫入共享快取路徑
  • 在每日摘要團隊模式中，Phase 1 的 5 路 Agent 啟動前先由腳本預取快取檔案

瓶頸 2：Phase 1 超標（P1）

數據：Phase 1 平均 114s，目標 <= 60s（benchmark.yaml）。

根因：

  1. Phase 1 每個 Sub-Agent 都獨立啟動 claude CLI → 冷啟動開銷 ~10-15s
  2. 每個 Agent 都讀取 SKILL_INDEX.md（14KB）+ preamble.md → Context Window 膨脹
  3. API 無快取命中（瓶頸 1），每次都等待真實回應

建議修正：

  • 優先解決瓶頸 1（快取命中），可將 Phase 1 降至 60-80s
  • 精簡 Phase 1 Agent 的 prompt：fetch-*.md 不需要載入完整 SKILL_INDEX，只需對應的單一 Skill
  • 考慮合併低延遲的 fetch Agent（如 fetch-todoist + fetch-gmail 共用一個 Agent）

瓶頸 3：Skill 使用率偏低（P1）

數據：24 個 Skill 中只有 11 個被使用（46%），遠低於 70% 目標。

根因分析：

  • 核心 Skill（todoist/news/hackernews/habits/learning/memory/cache/scheduler-state/ntfy）每次必用 = 9-10 個
  • 工具 Skill（task-manager/skill-scanner/system-audit/todoist-task-creator/arch-evolution/chatroom-query）僅在特定自動任務中觸發
  • game-design、github-scout、web-research、kb-curator 等使用率低

建議：

  • 重新定義「使用率」指標：區分「每日摘要使用率」與「全系統使用率」
  • 每日摘要 Skill 使用率目標改為 >=80%（基於必用 Skill）
  • 全系統使用率以「週覆蓋率」計算（7 天內至少用過 1 次），目標 >= 85%
  • 低頻 Skill 檢視是否應降級為「按需 Skill」而非「核心 Skill」

瓶頸 4：run-fsm.json 無上限 + 殘留 running 狀態（P2）

數據：17 條記錄，2 條仍為 running（含正在執行的），無清理機制。

風險：隨時間累積膨脹，Agent 每次讀取全量 JSON 浪費 Context Window。

建議：

  • 加入 max_entries 上限（建議 20），自動移除最舊的已完成條目
  • self-heal 自動任務增加偵測項：running 狀態超過 2 小時的條目自動標記為 timeout
  • 在 PowerShell 腳本中加入清理邏輯（每次寫入前先 prune）

瓶頸 5：研究註冊表膨脹（P2）

數據：119 條 / 66KB，4 天日均 ~30 條。按此速率 7 天窗口可達 200+ 條。

根因：每次自動任務都寫入 entry（含 tags 陣列），檔案只增不減。

建議：

  • 嚴格執行 7 天滾動清理：每次寫入前先移除 > 7 天的條目
  • 壓縮 entry 格式：移除冗餘欄位（如 kb_imported=true 的佔 97%，可改為只記錄 false）
  • 建立 research-registry-archive.json 存歷史條目，主檔案保持精簡

五、結論

Daily-Digest-Prompt 在 2 月完成了從「手動腳本」到「自主 Agent 系統」的跨越。系統審查 S 級（90.47 分）、穩定 98.5% 成功率、634 筆知識庫、496 個測試案例，這些數字證明了架構決策的正確性。

但營運數據揭示了三個結構性問題：快取失效（0% 命中率）、Phase 1 超標（114s > 60s）、Skill 使用率失真（46% < 70%）。這些並非架構缺陷，而是配置調優未跟上系統演化速度。Phase A 的 4 項修正（~5 小時工時）可立即改善 3 個不達標指標。

核心洞見：系統已從「建設期」進入「營運優化期」。下一步不是加更多功能，而是讓現有功能跑得更高效。快取修復一項就可能將 Phase 1 從 114s 降至 70s、API 呼叫減少 40%、成本節省 20%+。
