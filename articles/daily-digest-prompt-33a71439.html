二、關鍵發現

🔴 P0 - 立即處置（0-24 小時）

#### 1. API 金鑰暴露（SEC-001 - CRITICAL）

問題: .env 檔案暴露 7 個 production API 金鑰

  • Todoist API Token
  • OpenRouter API Key
  • Obsidian API Key
  • Gemini API Key
  • Mistral API Key
  • Nvidia API Key
  • Groq API Key

影響: 完整服務存取權限、計費 API 濫用風險、配額耗盡

修復: 詳見 docs/SECURITY-REMEDIATION.md

  • 立即 revoke 所有金鑰
  • 清理 git 歷史（BFG / filter-branch）
  • 實施 secret vaulting（1Password / Azure Key Vault）
  • 建立金鑰輪換政策

---

#### 2. Assembly Side Effects Missing（39 筆失敗）

問題: Phase 3 LLM 宣稱完成但 todoist-hourly.json 未實際更新

影響: Phase 3 重試、累積失敗計數、組裝延遲

根因: 檔案寫入驗證邏輯缺失、LLM prompt 不夠明確

修復:

  1. Phase 3 完成後加入檔案存在性驗證
  2. 檢查 LastWriteTime 是否晚於 phase3Start
  3. 實施 3 秒 polling retry（最多 3 次）
  4. 強化 LLM prompt 的檔案寫入要求

預期成果: Phase 3 失敗率從 39 筆/week 降至 <7 筆/week（-82%）

---

#### 3. Agent 崩潰頻率偏高（51 筆/30 天）

問題: agent_crash 事件高發（2026-04-17: 14 筆，2026-04-16: 11 筆）

根因:

  • 11 個 critical sessions 超過 300M tokens（最大 918K tokens）
  • Phase 2 並行度過高導致記憶體壓力
  • 無 session size limits

修復:

  1. 實施 session size limits（最大 500 messages / 10M tokens）
  2. Phase 2 並行度動態調整（根據記憶體使用率）
  3. 加入 exponential backoff（首次立即重試 → 30 秒 → 2 分鐘）
  4. Session health check 機制

預期成果: agent_crash 從 14 筆/day 降至 <3 筆/day（-79%）

---

🟡 P1 - 高優先級（1-7 天）

#### 4. 系統熵增惡化（entropy=0.537）

問題: Entropy score 從 0.44 (2026-03-31) 升至 0.537 (2026-04-18)，趨勢惡化

影響: 配置膨脹（58 個 YAML、480+ keys）、維護成本上升

修復:

  1. 整合跨 Adapter 關注點（frequency-limits + llm-gate + llm-router → adapter-registry.yaml）
  2. 清理重複配置（使用 YAML anchors）
  3. 執行 analyze-config.ps1 -Trend 監控改善

預期成果: Entropy score 降至 <0.50，配置檔案 <50 個

---

#### 5. Token 預算治理強化

問題: 雖然日均使用率僅 9.7% (3.4M/35M)，但單一 session 達 918K tokens

影響: Context 窗口竄改風險、Phase 0 快取預計算失效時 fallback 讀整檔（2.6MB）

修復:

  1. 驗證 Phase 0 cache/status.json 預計算成功率（目標 >99.5%）
  2. 實施 context 用量檢查（超過 80% 觸發壓縮）
  3. ADR-036 Context 壓縮感知機制驗證

預期成果: 所有 session <5M tokens，Phase 0 預計算 >99.5% 成功率

---

#### 6. 快取效率恢復

問題: 快取命中率從 27% 降至 5.6%，Knowledge base 快取缺失

影響: 額外 API 呼叫（+17 次相比 7 天均值）、token 浪費

修復:

  1. Knowledge base 快取預熱（Phase 0 加入 KB 預查詢）
  2. 調整 TTL 配置（Todoist: 120→240 分鐘）
  3. 實施 cache warming cron job（每 6 小時）

預期成果: 快取命中率恢復至 >40%，API 呼叫降低 30%

三、架構優勢（保持）

1. 文件驅動架構（8.5/10）

  • Prompt 薄層：80-140 行，僅含角色宣告與步驟骨架
  • 配置外部化：YAML 支援注釋、層級清晰
  • 模板按需載入：子 Agent 模板只在觸發時才 Read

2. Skill-First 策略（9.0/10）

  • 102 個 Skills 涵蓋所有維度（核心 21 + 工具 18 + 實驗 37 + 其他 26）
  • Frontmatter 標準化：name / version / description / allowed-tools / triggers≥3
  • 版本管理：v1.0.0 至 v0.5.0 並存，實驗 Skill 清晰標示

3. Hook 機器強制層（9.0/10）

  • 15 個 Hook 模組：638 個測試函數
  • 規則完全外部化：config/hook-rules.yaml（唯一真相來源）
  • DeerFlow 短路機制：快速拒絕、fail-open policy

4. 測試覆蓋完善（8.0/10）

  • 2403 個測試（2026-04-17）：hooks (804) + skills (129) + tools (1229) + config (241）
  • 反模式規範：時間敏感性（DI + _FIXED_NOW）、版本脆性（範圍 >= 1）
