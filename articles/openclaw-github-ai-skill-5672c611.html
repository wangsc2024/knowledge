一、專案概述

OpenClaw（openclaw/openclaw）是 2026 年 GitHub 史上增長最快的開源專案，由開發者 Steinberger 主導，於 2026 年 1 月底因 Moltbook 專案的病毒式傳播而爆紅，短短數週內從 9,000 Stars 飆升至超過 247,000 Stars。2026 年 2 月 14 日，Steinberger 宣布加入 OpenAI，專案將移交至開源基金會管理。

核心數據

| 指標 | 數值 |

|------|------|

| Stars | 247,000+（2026-03-02） |

| Forks | 47,700+ |

| 整合數量 | 50+ 第三方服務 |

| 預設 Skill | 100+ 預配置 AgentSkills |

| 支援頻道 | 22+ 通訊平台 |

| 授權 | MIT |

| 安全漏洞 | 512 個（含 8 個 Critical） |

| 惡意 Skill | 341/2,857（約 12%） |

核心定位

OpenClaw 是運行在本地裝置上的個人 AI 助手，作為 AI 模型與 50+ 整合服務之間的本地閘道器（Local Gateway），所有數據永不離開用戶機器。

二、四層技術架構深度解析

OpenClaw 的核心技術優勢來自其清晰的四層架構設計，每層職責分明：

2.1 Channel Adapter（頻道適配層）

  • 職責：統一 22+ 通訊平台的訊息收發介面
  • 支援頻道：WhatsApp、Telegram、Slack、Discord、LINE、Signal、iMessage、Google Chat、Microsoft Teams、Matrix、IRC 等
  • 設計模式：Adapter Pattern — 將各平台的 API 差異封裝為統一的訊息介面
  • 可借鏡：任何需要多通道整合的系統都可採用此模式，避免為每個平台寫獨立邏輯

2.2 Gateway（閘道核心層）

  • 職責：作為 Session、路由、頻道連線的 Single Source of Truth
  • 功能：請求路由、權限管控、Session 管理、AI 模型調度
  • 設計亮點：所有控制流都經過 Gateway，確保可審計性與單一控制點
  • 可借鏡：API Gateway 模式在微服務架構中已是標配，OpenClaw 將此概念應用於 AI Agent 調度

2.3 Agent（代理執行層）

  • 職責：連接外部 LLM（Claude、DeepSeek、GPT 系列、本地模型）執行任務
  • 特點：多模型抽象層（Multi-Model Abstraction），不綁定特定 AI 供應商
  • 自擴展能力：Agent 能自主撰寫新 Skill 以擴展自身能力，無需人工介入
  • 設計革新：從「靜態工具集」進化為「自我進化能力系統」— 這是 Agentic AI 架構的重大範式轉移

2.4 Skill Plugin（技能插件層）

  • 職責：100+ 預配置 Skill，涵蓋瀏覽器控制、Shell 命令、檔案操作、智慧家居、音樂串流等
  • 市集機制：ClawHub 公共市集，社群貢獻 Skill
  • 安全隱患：12% 惡意 Skill 率暴露了社群 Skill 市集的嚴重治理問題

2.5 Memory System（記憶持久層）

  • 儲存格式：本地 Markdown 文件
  • 功能：持久記憶、用戶偏好、跨 Session 適應性行為
  • 設計選擇：Markdown 而非資料庫 — 人類可讀、易於手動編輯、Git 可追蹤
  • 可借鏡：本專案的 context/*.json 記憶機制與此理念一致，但 OpenClaw 選用 Markdown 而非 JSON，更適合非結構化的個人偏好數據

三、安全危機：2026 年首個重大 AI Agent 安全事件

OpenClaw 的爆紅伴隨著前所未有的安全危機，成為 AI Agent 時代的重要警示：

3.1 曝露規模

  • 135,000+ 個 OpenClaw 實例暴露於公共網際網路，遍布 82 國
  • 15,000+ 個實例直接存在遠端程式碼執行（RCE）漏洞
  • 42,000+ 個控制面板無認證保護
  • CVE-2026-25253：CVSS 8.8 的關鍵漏洞

3.2 五大安全弱點分析

| 弱點類型 | 描述 | 根因 |

|---------|------|------|

| 無強制 HITL | 不強制人類審批高風險操作 | 便利性優先於安全性的設計哲學 |

| Prompt Injection | 網頁/文件中的惡意提示可操控 Agent | 缺乏輸入消毒層 |

| 供應鏈攻擊 | 12% ClawHub Skill 為惡意軟體 | 無 Skill 審核機制 |

| OAuth 權限過寬 | 請求超出必要的 OAuth 權限範圍 | 缺乏最小權限原則實施 |

| 影子 IT 風險 | 20% 企業員工未經 IT 審批即部署 | 個人工具與企業環境的邊界模糊 |

3.3 安全啟示（直接可操作）

  1. 零信任原則：任何 Agent 組件（包括 Skill）都不應獲得隱式信任
  2. 強制 Human-in-the-Loop：高風險操作（檔案刪除、API 呼叫、資金操作）必須人類確認
  3. Skill 審核管線：社群 Skill 必須經過自動化安全掃描 + 人工審核才可上架
  4. 最小權限原則：OAuth scope 應精確到單一操作，非全域授權
  5. 持續監控：即時追蹤 Agent 的權限使用與異常行為

五、2026 三大 Agent 架構趨勢（從 OpenClaw 生態觀察）

5.1 本地優先運動（Local-First Movement）

Ollama（162K Stars）、Open WebUI（124K Stars）、OpenClaw（247K Stars）共同驗證：開發者要求 AI 在自己的硬體上運行。驅動力包括隱私顧慮、API 成本消除、客製化需求。

可操作洞察：本專案已是 Local-First（Claude Code CLI + 本地排程），符合趨勢方向。

5.2 從「回應型 AI」到「行動型 AI」

幾乎所有平台都納入自主 Agent 行為。演化路徑：

  • Level 0：AI 回答問題（ChatGPT 式）
  • Level 1：AI 使用工具（Function Calling）
  • Level 2：AI 自主執行（OpenClaw 式，檔案操作 + 程式執行 + 工作流編排）
  • Level 3：AI 自我進化（自主撰寫 Skill 擴展能力）

可操作洞察：本專案位於 Level 2，skill-forge 具 Level 3 雛型。

5.3 視覺化工作流組合（No-Code Agent Pipeline）

Langflow、Dify、n8n 證明拖拉式介面是設計 AI Agent 管線的首選方式，讓非 ML 專家也能建立生產系統。

可操作洞察：本專案的 YAML 配置驅動架構是文字版的 No-Code，已在同一精神下運作。

七、結論與行動建議

OpenClaw 的現象級成功與安全危機為 AI Agent 領域提供了極其珍貴的正反面教材：

值得學習

  1. 四層架構清晰分離：Channel → Gateway → Agent → Skill 的職責劃分值得作為 Agent 系統設計的參考模板
  2. Markdown 記憶：人類可讀的持久化格式，比 JSON 更適合非結構化偏好數據
  3. 自擴展 Skill 機制：Agent 能自主產生新能力，是 Agentic AI 的下一個前沿

引以為鑑

  1. 安全不可後置：OpenClaw 的「便利優先、安全可選」導致 135K 實例暴露，本專案的 Hook 強制層設計更為穩健
  2. 社群 Skill 需審核：12% 惡意率是致命警告，任何 Skill 市集都需要自動化 + 人工雙層審核
  3. 權限必須最小化：OAuth 全域授權是安全反模式，每個 Skill 都應有明確的 allowed-tools 白名單
