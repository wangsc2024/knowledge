LLM 輔助技術債檢測、ACE 自動化重構與架構侵蝕預防的工業級實踐

---

一、技術概述

AI 驅動的技術債管理（AI-Driven Technical Debt Management）是將大型語言模型（LLM）與靜態分析工具結合，自動化偵測、分類、優先排序並修復軟體系統中累積的技術債務。2025-2026 年，此領域從「工具輔助偵測」演進至「Agentic 自動重構」——AI 不僅識別問題，更能自主執行跨檔案重構並驗證行為等價性。核心轉變在於：傳統技術債管理是週期性清掃（periodic cleanup），AI 原生 SDLC 將其轉為持續衛生（continuous hygiene），品質、測試、文件、依賴升級、重構成為 always-on 能力。

---

二、六類技術債分類

| 類別 | 說明 | AI 可介入程度 |

|------|------|-------------|

| 程式碼債 | 不可維護、不可讀的程式碼，缺乏模組化 | 高（自動重構） |

| 文件債 | 缺失或過時的文件 | 高（自動生成） |

| 測試債 | 自動化測試覆蓋率不足 | 高（自動測試生成） |

| 架構債 | 次優系統設計決策 | 中（偵測為主，修復需人工） |

| 基礎設施債 | 維護不良的部署與擴展系統 | 中（基於 IaC 分析） |

| 安全債 | 被忽視的安全最佳實踐與漏洞 | 高（SAST/DAST 自動掃描） |

三、核心方法與工具

3.1 ACE（Augmented Code Engineering）— CodeScene 的工業級自動重構框架

架構設計：ACE 是目前最成熟的 LLM 輔助重構系統之一，其核心在於「驗證優先」而非「生成優先」。

四步驟工作流程：


步驟 1：Code Smell 偵測
├── 使用 CodeHealth 指標分析 5 類 Code Smell
│   ├── Complex Conditional（複雜條件判斷）
│   ├── High Cyclomatic Complexity（高循環複雜度）
│   ├── Deep Nested Logic（深層巢狀邏輯）
│   ├── Bumpy Road（序列巢狀模式）
│   └── Large Method（過長方法）
└── 目前支援 25+ 種 Code Smell，跨 30+ 程式語言
     ↓
步驟 2：上下文感知 LLM 選擇
├── ML 模型根據待重構程式碼的屬性選擇最適 LLM
├── 整合 OpenAI、Anthropic、Google、LLaMA 系列
└── 動態 Prompt 建構：程式語言 + 原始碼 + 函數上下文 + Smell 類別 + 精確位置
     ↓
步驟 3：三層驗證
├── 語法驗證（Syntactic）：靜態分析工具確認無語法錯誤或新警告
├── 品質驗證（Quality）：確認目標 Code Smell 已解決且 CodeHealth 提升
└── 語意驗證（Semantic）：比對原始與重構後程式碼，驗證行為保留
     ↓
步驟 4：信心等級輸出
├── High：僅預期變更，完全驗證通過 → 直接建議採用
├── Mid：含非預期次要變更 → 標註呈現給開發者
└── Low：捨棄，不顯示給使用者

關鍵績效數據：

  • 基線 LLM 效能：37% 重構在功能上正確
  • ACE 精確度：98% 驗證通過的建議能提升 CodeHealth 同時保留行為
  • ACE 召回率：52% 偵測到的 Code Smell 獲得重構建議
  • 函數大小限制：目前支援最多 130 行的函數
  • 精確度從 37% → 98% 的關鍵：驗證層「捨棄不正確的解」

3.3 架構侵蝕偵測（Architecture Erosion Detection）

新興指標 — 幻覺耦合（Hallucinated Coupling）：

LLM 正確實作類別但錯誤導入違反控制反轉（IoC）原則的依賴。最新研究發現：

  • 專有模型（GPT-5.1）：0% 架構違規率
  • 開源模型：呈現顯著偏差（critical divergence）
  • 這是首個量化「AI 合成微服務中架構侵蝕」的實證框架

14 個 AI 架構應用主題（arXiv 2504.04334）：

| 類別 | 應用 |

|------|------|

| 架構設計與決策 | LLM 候選架構生成、設計模式辨識、ADR 知識圖譜分析、ML 前瞻式決策 |

| 架構演化與適應 | ML 元件適應、架構回復/逆向工程、資源管理、效能估算、重構、韌性增強 |

六大 AI 特定挑戰：

  1. 一次性建議缺乏持續適應能力
  2. 文件與演化系統不同步
  3. 模式匹配缺乏上下文推理
  4. AI 建議缺乏領域專業整合
  5. 缺乏通用的量化評估指標
  6. 模組級診斷無法解決系統級問題

四、實際工作流程（Step-by-Step）

4.1 持續技術債管理管線


階段 1：即時偵測（Detect）
├── PR/Commit 觸發 CodeScene CodeHealth 分析
├── SonarQube 靜態掃描（Code Smell + 漏洞）
├── AI 架構侵蝕檢查（依賴方向 + 耦合度）
└── 結果寫入品質閘門（Quality Gate）
     ↓
階段 2：智能分類（Classify）
├── 自動分類六類技術債（程式碼/文件/測試/架構/基礎設施/安全）
├── 嚴重度評估（ML 模型根據歷史資料預測影響）
└── 優先排序（業務影響 × 修復成本 × 風險評分）
     ↓
階段 3：自動修復（Remediate）
├── 高信心 Code Smell → ACE 自動重構（98% 精確度）
├── 中信心 → 生成建議 + 人工審查
├── 測試債 → AI 自動生成補充測試案例
├── 文件債 → LLM 自動生成/更新文件
└── 架構債 → 生成 ADR + 重構計畫（需人工決策）
     ↓
階段 4：驗證與回饋（Verify）
├── 自動化測試驗證行為等價性
├── CodeHealth 分數變化追蹤
├── 技術債趨勢儀表板更新
└── 回饋至 ML 模型改善未來偵測

4.2 CI/CD 整合實作


# .github/workflows/tech-debt-check.yml
name: Tech Debt Quality Gate
on: [pull_request]

jobs:
  code-health:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: CodeScene Analysis
        uses: codescene-oss/codescene-ci-cd@v2
        with:
          codescene-delta-analysis: true
          fail-on-declining-code-health: true
          coupling-threshold-percent: 30
      
      - name: SonarQube Scan
        uses: sonarqube-quality-gate-action@v1
        with:
          quality-gate: 'wait'
      
      - name: AI Debt Classification
        run: |
          python scripts/classify_debt.py \
            --input sonar-report.json \
            --model gpt-4o \
            --output debt-report.md

五、程式碼範例

5.1 ACE 風格的自動重構前後對比

重構前（Deep Nested Logic + Complex Conditional）：


def process_order(order, user, inventory):
    if order is not None:
        if user is not None and user.is_active:
            if order.total > 0:
                for item in order.items:
                    if item.id in inventory:
                        if inventory[item.id] >= item.quantity:
                            inventory[item.id] -= item.quantity
                        else:
                            return {"error": "insufficient_stock", "item": item.id}
                    else:
                        return {"error": "item_not_found", "item": item.id}
                return {"status": "success", "order_id": order.id}
            else:
                return {"error": "invalid_total"}
        else:
            return {"error": "inactive_user"}
    else:
        return {"error": "no_order"}

ACE 自動重構後（Guard Clause + Early Return）：


def process_order(order, user, inventory):
    if order is None:
        return {"error": "no_order"}
    if user is None or not user.is_active:
        return {"error": "inactive_user"}
    if order.total <= 0:
        return {"error": "invalid_total"}
    
    for item in order.items:
        if item.id not in inventory:
            return {"error": "item_not_found", "item": item.id}
        if inventory[item.id] < item.quantity:
            return {"error": "insufficient_stock", "item": item.id}
        inventory[item.id] -= item.quantity
    
    return {"status": "success", "order_id": order.id}

5.2 SATD（Self-Admitted Technical Debt）自動偵測


import re
from pathlib import Path

SATD_PATTERNS = [
    (r'#\s*(TODO|FIXME|HACK|XXX|WORKAROUND)\b', 'high'),
    (r'#\s*(TEMP|TEMPORARY|QUICK FIX)\b', 'medium'),
    (r'#\s*(REFACTOR|OPTIMIZE|CLEANUP)\b', 'low'),
]

def scan_satd(repo_path: str) -> list[dict]:
    """掃描程式碼庫中的自承認技術債。"""
    findings = []
    for py_file in Path(repo_path).rglob('*.py'):
        for i, line in enumerate(py_file.read_text().splitlines(), 1):
            for pattern, severity in SATD_PATTERNS:
                if re.search(pattern, line, re.IGNORECASE):
                    findings.append({
                        'file': str(py_file),
                        'line': i,
                        'content': line.strip(),
                        'severity': severity,
                        'type': 'SATD'
                    })
    return findings

七、與 Daily-Digest-Prompt 專案的應用場景

7.1 直接可借鑑的模式

  1. CodeHealth 閘門整合：在 pre_write_guard.py 或 post_tool_logger.py 中加入 Code Smell 偵測邏輯，即時標記新增技術債
  2. SATD 掃描自動化：system-audit Skill 可整合 SATD 掃描（FIXME/TODO/HACK），追蹤 context/tech-debt-backlog.json 中的項目年齡
  3. 架構侵蝕預防：利用 arch-evolution Skill 的 ADR 機制，每次架構變更都記錄決策理由，防止無意識偵蝕
  4. 品質閘門量化：templates/shared/quality-gate.md 可參考 CodeHealth 的量化基準（10 分制），從定性轉為定量評估

7.2 已有的相關機制

本專案的以下機制與 AI 技術債管理理念一致：

  • hooks/agent_guardian.py：三層守護（ErrorClassifier + CircuitBreaker + LoopDetector）— 類似 ACE 的三層驗證
  • config/audit-scoring.yaml：7 維度 38 子項評分 — 可擴充技術債維度
  • OODA 閉環（system-insight → audit → arch-evolution → self-heal）— 持續改善循環

八、風險與注意事項

  1. 語意等價性仍是未解問題：ACE 論文承認「程式語意等價性在自動程式修復中仍是大致未解的研究問題」，大規模重構需人工審查
  2. AI 生成程式碼的「合成技術債」：AI 生成的程式碼「功能性高但系統性缺乏架構判斷」，可能引入新形式的技術債（模型漂移、資料依賴、可解釋性問題）
  3. 過度依賴風險：AI 不是魔法棒，團隊仍需維持批判性思維和策略性品質決策能力
  4. 函數規模限制：ACE 目前僅支援函數級重構（≤130 行），API 級別變更仍需人工處理
  5. 隱私考量：程式碼送至雲端 LLM 的安全風險，CodeScene ACE 承諾「永不將程式碼用作訓練資料」
  6. 假陽性/假陰性：AI 偵測仍有誤判，必須維持人工審查流程
