快取命中率優化基礎知識

專案當前快取命中率僅 23.9%，遠低於業界標準 90%+，存在 66.1% 改善空間。

核心概念

快取命中率公式：


CHR = hits ÷ (hits + misses) × 100%

業界標準：

  • 一般 Web 應用：90%+
  • 靜態內容：80%+

TTL 策略分層

| 內容類型 | 建議 TTL |

|---------|----------|

| 靜態資產（含版本號） | 天/週/年 |

| API 回應（高頻更新） | 5-10 分鐘 |

| 新聞/內容 | 15-60 分鐘 |

| 知識庫筆記 | 60 分鐘 |

| HTML 頁面 | 5 分鐘 |

來源：AWS CloudFront, Fastly 官方文件

自適應 TTL 演算法

f-TTL 演算法：

  • 雙層快取架構
  • 效能提升：達同等命中率所需快取空間節省 49%
  • 來源：arXiv 學術論文

快取金鑰優化原則

原則：只包含真正影響內容的參數

反模式：

  • 包含所有 query params → 碎片化嚴重
  • 包含 Accept-Language → 語言變體爆炸
  • 包含追蹤 cookies → 每個用戶獨立快取

來源：AWS CloudFront, Fastly

命中率提升技術

1. TTL Jitter（防雷擊效應）

為 TTL 加入隨機偏移量：


ttl_actual = ttl_base + random.randint(-jitter, +jitter)

來源：Redis

2. Stale-While-Revalidate

快取過期時仍回傳舊資料，同時背景更新快取。

3. 混合淘汰策略（LFU + TTL）

效果（Ekata 案例）：

  • DRAM 使用降低 70%
  • 無延遲犧牲
  • 年省數十萬美元

來源：Redis

診斷流程

當命中率 < 目標時，依序檢查：

  1. Cache-Control 標頭 — 檢查 no-cache/no-store
  2. TTL 設定 — 比較實際更新頻率與 TTL
  3. 快取金鑰碎片化 — 分析 unique cache keys
  4. 流量模式 — 爬蟲/冷門路徑存取
  5. 後端效能 — 回源請求延遲

來源：Fastly, AWS

專案應用建議

短期（1-2 週）：

  • 分析當前 TTL 設定合理性
  • 實施 TTL jitter（± 10%）
  • 檢查快取金鑰碎片化

中期（1-2 月）：

  • 實施 stale-while-revalidate
  • 建立命中率儀表板
  • A/B 測試 TTL 策略

長期（3-6 月）：

  • 考慮自適應 TTL 演算法（d-TTL）
  • 實施混合淘汰策略（LFU + TTL）
  • 快取預熱自動化
