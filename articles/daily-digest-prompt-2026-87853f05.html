Daily-Digest-Prompt 執行管線效率與容錯架構優化建議

基於 RAG 知識庫跨領域知識（Context Engineering、AIOps 日誌分析、Workflow Engine 設計模式、FSM 狀態機最佳實踐）與專案實際運行數據的深度分析。

二、問題識別與根因分析

問題 1：run-fsm.json 無 TTL 清理，殘留 stale runs 無限累積

根因：FSM 狀態機只有寫入邏輯（新增 run + 更新 phase state），沒有：

  1. 完成後清理（completed run 不移除）
  2. 超時偵測（running 狀態無 heartbeat，永遠不會自動標記失敗）
  3. 容量限制（runs 物件無上限，會無限增長）

影響：

  • 檔案持續增長，讀寫效能緩慢下降
  • 無法從 FSM 狀態準確判斷「當前真正在執行的 run」
  • stale run 汙染監控數據

問題 2：並行實例的 stderr 日誌檔案名稱衝突

根因：兩個 daily-digest 使用相同 timestamp 生成 stderr 檔案名 assemble-stderr-20260303_174747.log，導致 Windows 檔案鎖競爭。

影響：Phase 2 組裝 Agent 完全失敗，整個 digest 丟失。

問題 3：loop-state 檔案無生命週期管理

根因：post_tool_logger.py 每個 session 建立 state/loop-state-{sid[:8]}.json，但 session 結束後不清理。

影響：

  • 檔案持續累積（目前 67 個，每日新增 ~20 個）
  • 對功能無直接影響，但增加目錄掃描時間

問題 4：research-registry.json 去重清理未觸發

根因：去重清理邏輯寫在模板中由 LLM 執行，依賴 LLM「記得」先清理舊 entries。在 context window 壓力下，LLM 可能跳過這一步。

影響：

  • registry 持續膨脹（125 entries），讀寫效能下降
  • 去重判定基於過多歷史數據，可能錯誤地阻止有價值的重複研究

問題 5：scheduler-state.json 遷移殘留

根因：已從 entries 陣列模式遷移到 run-fsm.json，但 scheduler-state.json 仍保留（82KB），且 entries 陣列為空。

影響：

  • 82KB 空殼檔案每次都被讀取
  • 腳本中仍有讀寫此檔案的邏輯，增加維護負擔

三、優化建議

建議 1：run-fsm.json 衛生機制（高優先）

方案：在 PowerShell 團隊腳本的啟動階段加入 FSM 衛生檢查。


衛生規則：
1. 移除所有 updated 超過 2 小時的 runs（超時自動標記 failed + 移除）
2. 移除所有 phase 全部 completed 的 runs（正常完成清理）
3. 保留最近 5 個 runs 供除錯（FIFO）
4. 偵測同 agent_type 的並行 run → 記錄 warning 到日誌

實作位置：run-todoist-agent-team.ps1、run-agent-team.ps1、run-system-audit-team.ps1 的 Phase 0 區塊。

預期效益：

  • run-fsm.json 維持在 1-5 個 runs（從目前 9 個降至穩態 1-2 個）
  • 消除 stale run 汙染，監控準確度提升
  • 估計減少 JSON 讀寫量 60-80%

建議 2：並行安全的日誌檔案命名（高優先）

方案：stderr 檔案名加入 run_id 或 PID 防撞。


# 現行：$StderrFile = "$LogDir\assemble-stderr-$Timestamp.log"
# 改為：$StderrFile = "$LogDir\assemble-stderr-${Timestamp}_${PID}.log"

影響範圍：3 個團隊腳本中的 Start-Job stderr 重導向。

預期效益：

  • 完全消除並行實例的檔案鎖競爭
  • Phase 2 組裝失敗率降至 0%

建議 3：loop-state 自動清理（中優先）

方案 A（推薦）：在 on_stop_alert.py（Stop hook）中加入清理邏輯。Session 結束時，刪除當前 session 的 loop-state 檔案。

方案 B：在 self-heal 自動任務中加入定期清理（每日一次掃描 state/ 移除 >24h 的 loop-state 檔案）。

預期效益：

  • state/ 目錄從 67+ 檔案降至 <5 個
  • 目錄掃描效能微幅改善

建議 4：research-registry 機器化清理（中優先）

方案：將去重清理從 LLM prompt 移到 PowerShell 腳本層。


# 在 run-todoist-agent-team.ps1 Phase 0 加入：
python -X utf8 -c "
import json
from datetime import datetime, timedelta
...
entries = [e for e in data['entries'] 
           if datetime.strptime(e['date'], '%Y-%m-%d') >= cutoff]
data['entries'] = entries
...
"

預期效益：

  • registry 穩定在 20-40 entries（7 天窗口內的實際量）
  • 去重判定更精確，不被過期數據汙染
  • LLM 不再需要記住清理步驟

建議 5：並行排程衝突偵測與預防（中優先）

方案：在團隊腳本啟動時檢查 run-fsm.json 中是否已有同 agent_type 的 running 實例。


決策邏輯：
1. 讀取 run-fsm.json
2. 若有同 agent_type 的 running phase → 檢查 updated 時間
   a. <30 分鐘 → 等待或跳過本次（避免重複執行）
   b. >=30 分鐘 → 標記舊 run 為 stale，繼續本次執行
3. 記錄衝突事件到日誌

預期效益：

  • 消除同類型 Agent 並行衝突
  • 根絕 Phase 2 檔案鎖競爭的根本原因

建議 6：config 檔案健康度指標（低優先）

現狀：19 個配置檔案共 2,446 行，部分配置間存在數值冗餘（如 timeout 值在 timeouts.yaml 和 run-todoist-agent-team.ps1 的 $TimeoutBudget 中重複定義）。

方案：

  1. 統一 timeout 來源：腳本從 config/timeouts.yaml 動態讀取，移除 hardcoded $TimeoutBudget
  2. 在 validate_config.py 中加入交叉引用檢查（偵測 YAML 與 PS1 的數值不一致）

預期效益：

  • 消除配置漂移風險（YAML 改了但 PS1 忘改）
  • 減少維護者心智負擔

四、優先級排序與實施建議

| 優先級 | 建議 | 預估工作量 | 風險等級 | ROI |

|--------|------|-----------|---------|-----|

| P0 | 建議 2：並行安全日誌命名 | 30 分鐘 | 低 | 極高 |

| P0 | 建議 1：FSM 衛生機制 | 2 小時 | 低 | 高 |

| P1 | 建議 5：排程衝突偵測 | 1 小時 | 中 | 高 |

| P1 | 建議 4：registry 機器化清理 | 1 小時 | 低 | 中 |

| P2 | 建議 3：loop-state 清理 | 30 分鐘 | 低 | 低 |

| P2 | 建議 6：config 健康度 | 3 小時 | 中 | 中 |

實施順序建議：

  1. 先做 P0（建議 2 + 1），立即消除已知失敗模式
  2. 再做 P1（建議 5 + 4），預防未來問題
  3. P2 可納入下一輪 skill-audit 或 system-insight 自動任務中漸進實施

五、與知識庫已有知識的關聯

本報告的分析方法借鑑了知識庫中以下研究：

  1. FSM 設計模式（2026-02-28）：狀態機需要 TTL、容量限制、超時偵測 → 直接對應建議 1
  2. Workflow Engine 設計模式（2026-02-28）：DAG 排程引擎的冪等性與衝突偵測 → 對應建議 5
  3. AIOps 日誌分析：結構化日誌系統已具備，但缺少自動化修復閉環 → 對應建議 3、4
  4. Context Engineering：Prompt 薄層調度器的設計已成熟，但「機器化強制」仍有盲區（registry 清理依賴 LLM）→ 對應建議 4 的設計哲學
  5. 知識熵增治理（2026-02-28）：已識別品質稀釋問題，本報告聚焦的是「運行時效率」而非「知識品質」，互為補充

---

六、結論

專案的文件驅動架構和 Skill-First 策略已相當成熟（26 Skills、19 YAML 配置、544 個測試）。但在高頻執行（每日 44 次）的壓力下，運行時資源管理成為新的瓶頸：

  1. 狀態檔案的生命週期管理缺失（FSM、loop-state、registry）
  2. 並行執行的隔離性不足（stderr 檔案名衝突）
  3. 從「LLM 自律」到「機器強制」的遷移尚未覆蓋所有環節（registry 清理仍依賴 LLM）

六項建議按 P0→P1→P2 排序，P0 預計 2.5 小時可完成，即可消除已知的 Phase 2 組裝失敗問題。這與專案 ADR-005（Hook 機器強制層取代 Prompt 自律）的設計哲學一脈相承——凡是可以機器化的，就不要依賴 Agent 自律。
