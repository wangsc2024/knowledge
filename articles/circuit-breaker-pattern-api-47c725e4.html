Circuit Breaker Pattern 完整指南 — API 彈性架構、三態狀態機與故障降級最佳實踐

技術概述

Circuit Breaker（熔斷器）模式源自 Michael T. Nygard 的《Release It!》一書，是分散式系統中防止級聯故障（cascading failures）的核心彈性模式。正如實際電路中的斷路器在偵測到過載時自動斷電保護電路一樣，軟體中的 Circuit Breaker 在偵測到下游服務異常時，自動停止發送請求，避免資源耗盡與雪崩效應。Martin Fowler 將其描述為「將受保護的函數呼叫包裝在熔斷器物件中，由該物件監控失敗狀況」。此模式在微服務架構中尤為關鍵 — 單一服務故障可透過同步呼叫鏈擴散至整個系統。

---

一、核心概念：三態狀態機

Circuit Breaker 本質上是一個有限狀態機（FSM），具有三個明確的狀態，每個狀態有清晰的轉換規則：

1.1 Closed（關閉 = 正常運作）

  • 請求正常傳遞至下游服務
  • 內部維護失敗計數器，成功呼叫將計數器重置為零
  • 計數器為時間視窗制（非累計制），在固定間隔自動重置，防止偶發故障誤觸發
  • 當指定時間內的失敗次數超過閾值時 → 轉為 Open

1.2 Open（開啟 = 阻斷請求）

  • 所有請求立即失敗，不實際呼叫下游服務（fail fast）
  • 可返回預設回應（fallback）、快取資料、或錯誤訊息
  • 同時啟動一個逾時計時器（reset timeout）
  • 計時器到期 → 轉為 Half-Open
  • 核心價值：保護已故障的服務不被繼續轟炸，給予恢復空間

1.3 Half-Open（半開 = 試探恢復）

  • 允許有限數量的試探性請求通過
  • 維護成功計數器，若連續成功達到閾值 → 轉為 Closed（恢復正常）
  • 任何失敗立即 → 轉回 Open，重新啟動逾時計時器
  • 核心價值：防止剛恢復的服務被瞬間洪水請求再度壓垮

狀態轉換圖


     ┌─成功──→ 重置計數器 ┐
     │                    │
  [Closed] ←─────────────┘
     │

二、關鍵配置參數

| 參數 | 典型值 | 說明 |

|------|--------|------|

| failure_threshold | 5 | 觸發 Open 狀態的失敗次數 |

| failure_rate_threshold | 50% | 滑動視窗內的失敗率閾值 |

| sliding_window_size | 100 calls | 統計視窗大小 |

| reset_timeout | 30-60 秒 | Open 狀態持續時間 |

| half_open_max_calls | 3-5 | Half-Open 允許的試探請求數 |

| success_threshold | 3 | Half-Open 轉 Closed 所需連續成功次數 |

| timeout_duration | 5000ms | 單次呼叫逾時 |

進階配置：

  • 漸增逾時（Increasing Timeout）：Open 狀態每次觸發延長等待時間（如 30s → 60s → 120s）
  • 錯誤篩選（Error Selectivity）：並非所有錯誤都應觸發熔斷 — 僅服務不可用（HTTP 503）、逾時等應計入失敗，業務邏輯錯誤（HTTP 400）不應觸發
  • 健康端點探測：在 Open 狀態主動 ping 下游服務的 /health 端點，而非單純等待計時器

三、與其他彈性模式的搭配

3.1 Circuit Breaker + Retry Pattern

兩者互補但目的不同：

  • Retry：處理暫態故障（transient faults），預期操作最終會成功
  • Circuit Breaker：阻止對持續性故障服務的無謂呼叫

最佳實踐：在 Retry 中嵌入 Circuit Breaker 感知 — 當 Circuit Breaker 回報故障非暫態時，停止 Retry。

3.2 搭配 Exponential Backoff + Jitter

重試公式：sleep = min(base × 2^attempt, MAX_BACKOFF) + random_jitter

加入隨機抖動（jitter）防止所有客戶端同步重試（thundering herd 問題）。

3.3 搭配 Bulkhead Pattern

將資源（執行緒、連線池）隔離，確保一個下游服務的故障不會耗盡所有資源。Circuit Breaker 配合 Bulkhead 可更精細地控制故障影響範圍。

3.4 搭配 Fallback Pattern

在 Open 狀態提供降級策略：

  • 返回快取資料（最常用）
  • 切換到備用服務
  • 返回預設值
  • 將請求排入佇列待後處理

Python 實作範例（PyBreaker）


import pybreaker
import requests

# 定義熔斷器
api_breaker = pybreaker.CircuitBreaker(
    fail_max=5,           # 5 次失敗觸發 Open
    reset_timeout=30,     # 30 秒後嘗試 Half-Open
    exclude=[ValueError], # 業務錯誤不計入
)

# 監聽狀態變化
class BreakerListener(pybreaker.CircuitBreakerListener):
    def state_change(self, cb, old_state, new_state):
        print(f"[Circuit Breaker] {old_state.name} → {new_state.name}")

api_breaker.add_listener(BreakerListener())

@api_breaker
def call_api(url):
    response = requests.get(url, timeout=5)
    response.raise_for_status()
    return response.json()

# 使用（含 fallback）
def get_data_safe(url, fallback_data=None):
    try:
        return call_api(url)
    except pybreaker.CircuitBreakerError:
        print("[Fallback] 服務暫時不可用，使用快取資料")
        return fallback_data

5.2 可優化方向

  1. 時間視窗計數：將 consecutive_failures 改為滑動視窗（如「5 分鐘內 3 次失敗」）
  2. 錯誤分類：HTTP 429（限流）vs 503（不可用）vs timeout 分別處理，429 自帶 Retry-After
  3. 主動健康探測：在 failed 狀態定期 ping API health 端點，而非等待下次排程
  4. 漸增冷卻時間：首次故障等 1 分鐘、第二次 5 分鐘、第三次 15 分鐘
  5. AI 自適應閾值：根據歷史流量模式動態調整閾值（Azure 文件提及的 2026 趨勢）

六、常見陷阱與避免方法

6.1 閾值設定不當

  • 過低：正常流量中的偶發錯誤就觸發熔斷，導致可用性下降
  • 過高：故障已擴散才觸發，失去保護意義
  • 最佳實踐：初始值設為 50% 失敗率 + 最少 10 次呼叫，再根據監控數據微調

6.2 單一熔斷器覆蓋多個獨立資源

  • 問題：多個 shard/region 共用一個 Circuit Breaker，導致健康的 shard 也被阻斷
  • 解法：每個獨立資源一個 Circuit Breaker 實例

6.3 忽略 Half-Open 洪水問題

  • 問題：Half-Open 允許太多試探請求，壓垮剛恢復的服務
  • 解法：嚴格限制 half_open_max_calls（3-5 次），且用健康端點替代真實請求

6.4 不當的逾時配置

  • 問題：reset_timeout 太短導致服務還沒恢復就嘗試重連
  • 解法：根據下游服務的恢復特性設定，並使用漸增逾時

6.5 缺乏監控

  • Martin Fowler 特別強調：「熔斷器是監控的最佳切入點」
  • 必須記錄所有狀態變化、暴露詳細狀態資訊、支援手動觸發/重置

七、2026 年趨勢：AI 自適應 Circuit Breaker

Microsoft Azure 文件指出，傳統 Circuit Breaker 依賴預先配置的固定閾值，行為確定但可能非最佳。2026 年的新趨勢是引入 AI/ML 動態調整：

  • 即時流量分析：根據當前流量模式自動調整失敗率閾值
  • 異常偵測：ML 模型識別非典型故障模式（如漸進式退化 vs 突發崩潰）
  • 歷史學習：從過去的故障事件學習最佳恢復時間
  • Service Mesh 整合：Istio/Envoy 層級的 Circuit Breaker 可跨服務統一管理，無需修改應用程式碼
