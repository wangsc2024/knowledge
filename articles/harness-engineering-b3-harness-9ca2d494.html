Harness Engineering B3 儀表三角與「Harness 覆蓋率」開放問題：配置熵、約束覆蓋率、機制飽和的閾值治理（2026-04-09）

Harness Engineering知識庫研究B3配置熵約束覆蓋率Fitness Function2026-04-09

概述

本筆記承接 config/autonomous-harness.yaml 與專案 Harness Engineering 分層（B0–B5），將優化焦點收斂到 B3 觀測層：以「配置熵（entropy_score）」「約束覆蓋率（constraint_coverage）」「機制飽和（mechanism saturation）」形成可操作的儀表三角，並對照 Martin Fowler 與演化式架構文獻中的 Architecture fitness / Fitness Function，回答「如何系統化地提升 Harness 對行為空間的覆蓋與警戒品質」。

---

重點

  1. Fowler 的開放問題：在〈Harness engineering for coding agent users〉中，作者明確指出需要「類似程式碼覆蓋率與突變測試」的手段來評估 Harness 覆蓋率與品質；並區分 Maintainability、Architecture fitness、Behaviour 三類調節對象，其中 Architecture fitness 直接對應 Fitness Functions（性能、可觀測性約束等）。
  2. 演化式架構的適能函式：Continuous Architecture 將適能函式定義為對「架構特性是否被保留」的客觀、可自動化量測；區分原子/整體、觸發/持續、靜態/動態等維度，並建議每週檢視「失敗的適能函式」並制定行動計畫（建議起步追蹤約 3 條，成熟團隊 5–6 條）。
  3. 本專案落地對齊：

- 配置熵：對應 tools/entropy_score.py 與 state/entropy-trend.json 所代表的「設定空間複雜度」監控，可作為 continual / dynamic 類適能函式的輸入，用於偵測無意義膨脹與治理漂移。

- 約束覆蓋率：對應 tools/constraint_coverage.py 與 state/constraint-metrics.json，將「規則是否涵蓋實際失敗模式」量化，呼應 Fowler 所說的 Harness 覆蓋率問題。

- 機制飽和：對應 state/mechanism-saturation-last-check.json 與 config/dedup-policy.yaml 的飽和閾值語意，用於避免同一類介入（如攔截）重複堆疊而無新知識增益。

  1. component_flags 與補償面：autonomous-harness.yaml 中 intervention_sources 將 circuit_breaker、constraint_evolution、quality_gate_llm 等與 state 檔案／日誌指標綁定；優化方向是讓 B3 儀表與這些補償面指標共用同一套閾值語言（SLO/燃燒率/週檢），避免「有旗標、無行動」。

---

分析

| 維度 | 外部理論 | 本專案元件 | 可落地優化 |

|------|-----------|------------|------------|

| 架構適能 | Fitness function 每週檢視、持續監控 | check-health.ps1 約束健康、quality_trend | 將「失敗適能函式」清單與 state/quality-trend.json 對齊 owner 與修正 SLA |

| Harness 覆蓋 | Fowler：需類似覆蓋率概念 | constraint_coverage.py | 對高風險路徑（risk_tiers.critical）強制最低覆蓋率門檻 |

| 持續驗證 | Triggered vs Continual | Hook JSONL、post_tool_logger | 將 blocked 事件率與 mechanism 飽和聯動，觸發 arch-evolution 審查而非僅計數 |

風險：若只增加指標而不綁定 行動閉環（backlog → gap → embedded），會落入 Fowler 所說的「verification theater」。因此 B3 儀表必須與 B4 Gap Loop、B5 補償面審查共用同一套「失敗→提案→嵌入」契約。
