AI-Driven Requirements Quality Assurance 完整指南

Multi-Agent 需求品質自動驗證、QUS 框架 13 準則實踐與 Given-When-Then 驗收條件自動生成的工業級工作流

---

一、技術概述

AI 驅動的需求品質保證（Requirements QA）是需求工程（RE）自動化的核心環節。與傳統人工審查相比，透過 LLM Multi-Agent 架構可實現需求文件的自動驗證、品質評分與修正建議。2024-2026 年間，該領域論文數量從 23 篇（2023）暴增至 113 篇（2024），年成長率達 136%，反映出研究社群對 LLM4RE 的高度關注。

核心價值在於：將原本依賴資深分析師經驗的品質判斷，轉化為可量化、可自動化的流程——從需求擷取（Elicitation）、規格撰寫（Specification）、品質驗證（Validation）到追溯性管理（Traceability），每個環節都可由 AI 輔助或自動化。

---

二、核心框架：QUS（Quality User Story）13 準則

2.1 QUS 框架概述

QUS 由 Lucassen 等人（Utrecht University, 2016）提出，定義了使用者故事品質的 13 個準則，分為三個層次：

| 層次 | 準則 | 說明 |

|------|------|------|

| 語法層（Syntactic） | Well-formed | 遵循 "As a [role], I want [feature], so that [benefit]" 格式 |

| | Atomic | 每個 story 只描述一個功能 |

| | Minimal | 不含多餘的語法元素 |

| 語意層（Semantic） | Conceptually sound | 角色（who）不等於功能（what） |

| | Problem-oriented | 描述問題而非解決方案 |

| | Unambiguous | 避免歧義用語 |

| | Conflict-free | 不與其他 story 矛盾 |

| 語用層（Pragmatic） | Full sentence | 完整句子 |

| | Estimatable | 可估算工時 |

| | Unique | 不與其他 story 重複 |

| | Uniform | 格式一致 |

| | Independent | 可獨立交付 |

| | Complete set | 覆蓋完整功能範圍 |

2.2 AQUSA 自動化工具

AQUSA（Automatic Quality User Story Artisan）是基於 QUS 框架的 NLP 自動驗證工具：

  • 輸入：一組使用者故事文本
  • 處理：NLP 語法解析 + 規則比對
  • 輸出：品質缺陷報告 + 修正建議
  • 實證：分析 18 家公司、1023 個使用者故事
  • 局限：Recall 尚未達 100%，但已展現實用價值

三、核心方法：Multi-Agent 需求品質驗證

3.1 ALAS（Autonomous LLM-based Agent System）架構

研究提出的 Multi-Agent 系統包含四個核心元件：


┌─────────────────────────────────────┐
│         ALAS Reference Model         │
├─────────┬──────────┬────────────────┤
│  Task   │  Agents  │ Shared KB      │
│ 定義目標 │ 專責角色  │ 共享上下文     │
│ 與範圍   │ LLM 實例 │ 對話歷史       │
└────┬────┴────┬─────┴───────┬────────┘
     │         │             │
     v         v             v
   Phase 1: Task Preparation
   - Persona 定義（角色專責化）
   - K-shot prompting（品質範例）
   - Chain-of-Thought（推理鏈）
   - Fact-checking list（驗證清單）

   Phase 2: Task Conduction
   - Agent 順序互動
   - 每次回應更新 Shared KB
   - 動態 Prompt 調整

3.2 Multi-Agent 工作流程（Step-by-Step）

步驟 1：需求擷取 Agent


輸入：會議紀錄、簡報、訪談摘要
處理：LLM 摘要 + 關鍵需求提取
輸出：結構化需求清單（JSON/Markdown）

步驟 2：使用者故事生成 Agent


輸入：結構化需求清單
處理：依 QUS 格式生成 user stories（30 次迭代取共識模式）
輸出：user stories + acceptance criteria
Prompt 模式：
  "Generate user stories following the template:
   As a [role], I want [feature], so that [benefit].
   Acceptance Criteria:
   Given [context], When [action], Then [outcome]."

步驟 3：品質驗證 Agent


輸入：生成的 user stories
處理：逐一檢查 QUS 13 準則 + INVEST 標準
輸出：品質評分 + 缺陷報告
評分維度：
  - Syntactic score (0-5)
  - Semantic score (0-5)
  - Pragmatic score (0-5)
  - Overall quality (加權平均)

步驟 4：修正建議 Agent


輸入：品質缺陷報告 + 原始 stories
處理：針對每個缺陷提供具體修正建議
輸出：改善版 user stories

步驟 5：人工審查閘門


由 Product Owner 審查 AI 修正版
確認：業務脈絡正確性、技術可行性、優先級排序

3.3 實測結果

| 指標 | 原始 stories | GPT-3.5 改善版 | GPT-4 改善版 |

|------|-------------|---------------|-------------|

| 整體滿意度 | 3.0 | 4.0 | 3.71-4.0 |

| 清晰度 | 中等 | 顯著提升 | 顯著提升 |

| 商業價值表達 | 不足 | 改善 | 大幅改善 |

| 簡潔性 | 良好 | 下降（S2: 3-3.17） | 更複雜 |

關鍵發現：AI 改善版在清晰度和商業價值表達上明顯優於原始版本，但 GPT-4 輸出傾向增加複雜度——5 位參與者認為 story 粒度過大。人工審查仍為不可或缺的環節。

四、Given-When-Then 驗收條件自動生成

4.1 BDD（Behavior-Driven Development）驗收條件格式

Gherkin 語法是 BDD 工具（如 Cucumber）的標準語言：


Feature: 使用者登入
  Scenario: 成功登入
    Given 使用者已註冊帳號 "user@example.com"
    And 使用者在登入頁面
    When 使用者輸入正確的帳號和密碼
    And 點擊「登入」按鈕
    Then 系統導向使用者的個人儀表板
    And 顯示歡迎訊息 "歡迎回來"

  Scenario: 密碼錯誤
    Given 使用者已註冊帳號 "user@example.com"
    And 使用者在登入頁面
    When 使用者輸入錯誤的密碼
    And 點擊「登入」按鈕
    Then 系統顯示錯誤訊息 "帳號或密碼不正確"
    And 使用者仍在登入頁面

4.2 AI 自動生成工作流程


步驟 1：使用者故事輸入
  "As a registered user, I want to login to my account"

步驟 2：LLM 分析行為邊界
  → 正常路徑（happy path）
  → 異常路徑（error cases）
  → 邊界條件（edge cases）

步驟 3：生成 Given-When-Then 場景
  → 每個路徑 1-3 個場景
  → 自動補充 And/But 條件

步驟 4：品質驗證
  → 每個場景聚焦單一行為
  → 語言清晰無歧義
  → 可自動化測試

步驟 5：Gherkin 格式化輸出
  → 直接可供 Cucumber/SpecFlow 執行

4.4 Prompt Engineering 最佳實踐


## 驗收條件生成 Prompt 模板

你是一位資深 QA 工程師。根據以下使用者故事，
生成完整的 BDD 驗收條件（Given-When-Then 格式）。

### 要求：
1. 至少涵蓋：正常路徑、錯誤處理、邊界條件
2. 每個場景聚焦單一行為
3. Given 描述前置狀態，When 描述動作，Then 描述預期結果
4. 使用具體的測試數據（非泛用描述）
5. 確保可自動化執行

### 使用者故事：
{user_story}

### 輸出格式：
Feature: [功能名稱]
  Scenario: [場景名稱]
    Given [前置條件]
    When [使用者動作]
    Then [預期結果]

5.2 品質驗證閘門


# 品質驗證偽代碼
def validate_user_story(story: str) -> QualityReport:
    # 語法層驗證
    syntactic = check_qus_syntactic(story)  # Well-formed, Atomic, Minimal
    
    # 語意層驗證
    semantic = check_qus_semantic(story)    # Unambiguous, Problem-oriented
    
    # 語用層驗證
    pragmatic = check_qus_pragmatic(story)  # Estimatable, Independent
    
    # INVEST 標準
    invest = check_invest(story)            # I-N-V-E-S-T 六維度
    
    # 驗收條件品質
    ac_quality = validate_acceptance_criteria(story.acceptance_criteria)
    
    return QualityReport(
        syntactic_score=syntactic.score,
        semantic_score=semantic.score,
        pragmatic_score=pragmatic.score,
        invest_score=invest.score,
        ac_completeness=ac_quality.completeness,
        overall=weighted_average([syntactic, semantic, pragmatic, invest]),
        defects=collect_defects([syntactic, semantic, pragmatic]),
        suggestions=generate_fixes(defects)
    )

5.3 常見品質缺陷類型

| 缺陷類型 | 說明 | AI 偵測率 |

|---------|------|----------|

| Conjunction overuse | 一個 story 含多個 and/or | 高（語法分析） |

| Format inconsistency | 不遵循 As-a/I-want/So-that | 高（模式匹配） |

| Ambiguity | 使用 "某些"、"可能" 等歧義詞 | 中高（語意分析） |

| Missing rationale | 缺少 "so that" 子句 | 高（格式檢查） |

| Redundancy | 與其他 story 重複 | 中（向量相似度） |

| Implementation leak | 描述解決方案而非問題 | 中（LLM 判斷） |

七、與本專案的應用場景

7.1 daily-digest-prompt 的需求品質提升

本專案（daily-digest-prompt）採用規格驅動開發（SDD），其 specs//spec.md 可直接受益：

  1. User Story 自動生成：從 spec.md 的功能描述自動生成結構化 user stories
  2. AC 自動補全：為 tasks.md 中每個任務自動生成 Given-When-Then 驗收條件
  3. QUS 品質閘門：在 spec.md 定稿前，自動驗證所有 user stories 的 13 準則合規性
  4. AQUSA 整合：將 AQUSA 工具整合到 check-health.ps1，定期掃描需求品質

八、風險與注意事項

  1. 輸入品質依賴性：LLM 輸出品質高度依賴輸入資訊的完整性——會議紀錄殘缺則生成結果必然不足
  2. 隱性知識缺口：資深分析師的隱性知識（tacit knowledge）無法被 LLM 捕捉，需人工補充
  3. 複雜度膨脹：GPT-4 傾向生成過度詳細的 stories（簡潔性評分下降），需設定粒度控制
  4. 領域特異性：通用 LLM 在垂直領域（醫療、金融、法規）的需求理解仍有局限，需 fine-tuning 或 RAG 補強
  5. 成本考量：GPT-4 成本約為 GPT-3.5 的 7 倍（€24.80 vs €3.60），需權衡品質與成本
  6. 幻覺風險：LLM 可能生成看似合理但實際不存在的需求，驗證閘門不可省略
  7. 合規性：在受管制產業（醫療器材、航空），AI 生成的需求文件可能需要額外的認證流程
