Agent 團隊協調效能與配置治理優化

優化摘要

本次對 daily-digest-prompt 專案進行 Agent 團隊協調效能與配置治理的深度分析，識別 5 項核心痛點（P0–P4），涵蓋 Phase 預算失控、配置膨脹、狀態生命週期缺失、測試覆蓋率盲區、loop-state 洪泛等問題。結合 2025–2026 年多 Agent 編排、可觀測性、配置熵控制的最新業界實踐，提出 12 項具體優化建議。

---

一、痛點分析與量化影響

1.1 Phase 2 超時預算失控（P0-嚴重）

現況：run-todoist-agent-team.ps1 的 Phase 2（並行任務執行階段）缺乏全局 timeout 預算。38 個自動任務中，tech_research 超時設為 2,600 秒（43 分鐘）、podcast_create 為 2,400 秒（40 分鐘）。若兩者同時執行，Phase 2 耗時可達 83 分鐘，遠超 30 分鐘排程間隔，導致排程衝突。

量化影響：

  • 最壞情況 Phase 2 耗時：83 分鐘（vs 預期 15–30 分鐘）
  • 排程衝突風險：每 30 分鐘一次的 todoist-team 排程可能重疊
  • 短任務（skill_forge=900s）被長任務阻塞，浪費等待時間

優化建議：

  1. 新建 config/phase-timeout-policy.yaml，定義全局 Phase 2 硬上限（建議 1,800 秒）
  2. 單任務預算上限不超過全局預算的 60%（建議 1,080 秒）
  3. 超時行為策略：kill 超時任務 → 進入降級模式 → 記錄降級原因至 state/phase-degradation.json
  4. 引入任務優先級佇列：P1 任務優先分配時間資源，P3 任務在預算不足時延後 [來源：arXiv, 2025]

1.2 配置膨脹（P0-嚴重）

現況：config/ 目錄包含 50 個配置檔、總計 7,236 行。frequency-limits.yaml 達 1,171 行，混合了頻率限制、超時覆寫、模型選擇三種職責，違反單一責任原則。

量化影響：

  • 新開發者理解系統需閱讀 50 個配置檔
  • frequency-limits.yaml 1,171 行中，約 40%（~470 行）屬於超時/模型選擇，可拆分
  • 配置修改後的驗證路徑不明確（缺乏 INDEX.md 導航）

優化建議：

  1. 將 frequency-limits.yaml 拆分為三個單一職責檔案：純頻率限制（~200 行）、AutoTaskTimeoutOverride 合併至 timeouts.yaml、模型選擇移至 model-selection.yaml
  2. 新建 config/INDEX.md（檔案導航 + 修改檢查清單）
  3. 長期目標：配置熵分數從目前值降低 30%（透過 entropy_score.py 追蹤）[來源：IEEE Configuration Complexity Model, 2007]

1.3 狀態生命週期管理缺失（P1-重要）

現況：context/ 102 個 JSON 檔案（最大 688 KB）、state/ 232 個 JSON 檔案（最大 894 KB），無統一 TTL 或清理策略。adr-registry.json（688 KB）持續膨脹、kb-note-scores.json（894 KB, 30,161 行）無輪轉機制。手動備份檔案（podcast-history.backup-*）無清理排程。

量化影響：

  • 狀態檔案總大小 > 2 MB，持續增長
  • 無 schema 版本管理（scheduler-state.json 缺 schema_version）
  • context/ 和 state/ 的邊界定義模糊，導致檔案歸屬混亂

優化建議：

  1. 新建 config/state-lifecycle.yaml，為每個狀態檔定義 TTL、最大容量、清理策略
  2. 大型 JSON 檔（>500 KB）引入輪轉機制：保留最新 N 筆 + 壓縮歸檔
  3. 統一 schema_version 欄位要求
  4. 獨立 Cleanup Script 納入 health-check 定期執行

1.4 Tools 測試覆蓋率盲區（P2-中等）

現況：pyproject.toml 的 [tool.coverage.run] source = ["hooks", "skills"] 排除了 tools/ 目錄。tools/ 有 300+ 測試但不計入覆蓋率報告，Harness 核心元件（llm_router.py、budget_guard.py）的覆蓋率不透明。

量化影響：

  • 公開覆蓋率指標無法反映 tools/ 的真實狀態
  • Harness 元件的 bug 可能漏網

優化建議：

  1. 將 "tools" 加入 [tool.coverage.run] source
  2. 分層覆蓋率閾值：hooks+skills ≥44%、tools ≥30%、全套 ≥50%
  3. 針對 LLM 輸出的非確定性測試，引入語義匹配斷言（而非精確比對）[來源：Tricentis QA Trends 2026]

1.5 Loop-State 檔案洪泛（P3-低）

現況：state/ 目錄有 165 個 loop-state-*.json 檔案，清理機制僅在 run-agent-team.ps1 啟動時掃描 3 天前的檔案。

優化建議：

  1. 改為保留最新 N 個（建議 30 個），而非時間閾值
  2. 考慮拆分至 state/loop-state/ 子目錄

二、業界最佳實踐整合

2.1 多 Agent 編排優化

  • 專責化原則：每個 Agent 專注單一任務，減少 45% 交接次數、決策速度提升 3 倍 [IBM Research, 2025]
  • 模型分層（Model Tiering）：快速便宜模型做分流，複雜推理交給高階模型，節省 40–60% 成本 [Microsoft Azure AI, 2025]
  • Fan-out/Fan-in + Early Stopping：並行排程多 Agent 獨立解題，先完成者終止其餘，可達 1.8x 加速 [arXiv:2507.08944, 2025]

2.2 可觀測性強化

  • OpenTelemetry 事實標準：v1.20.0 支援 eBPF 零程式碼插樁 [OpenTelemetry, 2026]
  • 三大核心指標：Latency、Tokens/sec、Cost 構成推論可觀測性三角 [dasroot.net, 2026]
  • 品質回饋閉環：觀測輸出品質、偵測 prompt drift、觸發品質降級告警 [LangWatch, 2026]

2.3 配置熵控制

  • 自癒配置趨勢：自癒自動化從 2025 年 40% → 2026 年 55% [Virtue Market Research, 2026]
  • 配置管理市場：2025 年 58.3 億美元 → 2030 年 122.4 億美元（CAGR 16%）
  • 本專案已有基礎：entropy_score.py、analyze-config.ps1、config/benchmark.yaml，走在前沿

五、結論

本專案已具備成熟的 Agent 團隊架構（3 Phase Fan-out/Fan-in）、配置熵量測工具（entropy_score.py）和結構化日誌系統（post_tool_logger.py），但在 Phase 預算控制、配置單一職責、狀態生命週期 三個維度仍有顯著改善空間。建議優先實施 P0 的 Phase timeout 全局預算和 frequency-limits.yaml 拆分，可在一週內完成並帶來可量化的穩定性與維護性提升。
