一、技術概述

AI 驅動的文件生成（AI-Driven Documentation Generation）是利用大型語言模型（LLM）自動分析程式碼結構、語意與上下文，產出高品質技術文件的技術實踐。2026 年的文件自動化已從單純的 docstring 生成，演進為涵蓋 API 參考、架構文件、變更日誌、onboarding 指南的全棧文件管線。

根據業界統計，手動文件撰寫佔開發時間的 20-30%，而 AI 工具可將此降至約 5%，節省 15-25% 的開發時間。Gartner 預測到 2026 年，超過 30% 的 API 需求將來自 AI Agent 而非人類開發者，推動文件從「人類可讀」向「LLM 友善」的範式轉移。

3.3 變更日誌自動化工作流


# Changeish 風格的變更日誌自動化
import subprocess
import openai
from datetime import datetime

def generate_changelog():
    # 步驟 1：收集 Git 歷史
    git_log = subprocess.run(
        ['git', 'log', '--oneline', '--since=1.week'],
        capture_output=True, text=True
    ).stdout
    
    # 步驟 2：組裝 LLM Prompt
    prompt = f"""Based on these git commits, generate a structured changelog:
    
    {git_log}
    
    Format:
    ## [{datetime.now().strftime('%Y-%m-%d')}]
    ### Added
    ### Changed  
    ### Fixed
    """
    
    # 步驟 3：呼叫 LLM
    response = openai.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": prompt}]
    )
    
    # 步驟 4：附加到 CHANGELOG.md
    with open('CHANGELOG.md', 'a') as f:
        f.write(response.choices[0].message.content)

四、llms.txt — LLM 友善文件新標準

4.1 概念與設計

受 robots.txt 啟發，llms.txt 是放置於網站根目錄的輕量級 Markdown 檔案，專門引導 LLM 理解網站內容。


# My API Documentation

> Overview: RESTful API for task management
> Auth: OAuth 2.0 Bearer Token
> Base URL: https://api.example.com/v1

## Endpoints
- [Tasks](/docs/tasks.md): CRUD operations for tasks
- [Projects](/docs/projects.md): Project management
- [Users](/docs/users.md): User authentication and profiles

## Quick Start
- [Getting Started](/docs/quickstart.md)
- [Authentication Guide](/docs/auth.md)

4.2 Mintlify 的自動生成

Mintlify 在每個託管專案部署時自動生成 llms.txt 和 llms-full.txt，零配置即可讓 AI Agent 存取機器可讀文件。

4.3 設計原則

  1. 高層概覽：提供 API、驗證方式、聯絡點的結構化摘要
  2. 機器可解析：純 Markdown，無需 HTML 渲染
  3. 安全考量：速率限制、驗證機制、邊界設定
  4. 可操作性：不只告知（inform），還要賦能（enable）

六、與本專案的應用場景（daily-digest-prompt）

6.1 直接可用

  1. SKILL.md 自動生成：新增 Skill 時，LLM 分析 SKILL.md 模板 + 程式碼，自動產出標準化文件
  2. 變更日誌自動化：整合 Conventional Commits + AI 摘要，每次 git-push 自動任務時附帶 CHANGELOG 更新
  3. Hook 文件自動更新：hooks/*.py 修改後，自動更新 CLAUDE.md 的 Hook 清單區塊

6.2 中期可導入

  1. llms.txt 生成：為知識庫 API (localhost:3000) 生成 llms.txt，讓 AI Agent 更好地理解 API 結構
  2. system-audit 報告模板化：審查報告結構可由 LLM 根據維度分數動態生成，取代靜態模板
  3. ADR 自動文件化：arch-evolution Skill 產出 ADR 後，自動生成人類可讀摘要

6.3 長期願景

  1. Swimm 風格活文件：CLAUDE.md 與 config/*.yaml 的文件區塊隨程式碼變更自動同步
  2. 多 Agent 文件生成：類似 AI-Doc-Gen 的多 Agent 架構，分析專案 23 個 Skill 並產出統一文件

---

七、風險與注意事項

  1. 幻覺風險：LLM 可能生成看似合理但不正確的 API 描述，必須有人工審查閘門
  2. 過時文件比缺少文件更危險：自動生成的文件若未隨程式碼更新，會誤導開發者
  3. 上下文窗口限制：大型程式碼庫需分段處理，單次提交過多程式碼會降低文件品質
  4. 風格一致性：不同 LLM 呼叫可能產出風格不一的文件，需統一 prompt 模板
  5. 安全洩漏：程式碼送入雲端 LLM 可能洩漏敏感邏輯，考慮使用本地模型（Doc-Comments-AI + Ollama）
  6. 成本控制：大型專案的全庫文件生成可能消耗大量 token，需設定預算上限
  7. llms.txt 安全：確保 llms.txt 不暴露內部 API 結構或敏感端點

8.3 CI/CD 整合（GitHub Actions）


name: Auto Documentation
on:
  push:
    branches: [main]
    paths:
      - 'hooks/**'
      - 'skills/**'
      - 'config/**'

jobs:
  generate-docs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Generate Changelog
        run: |
          pip install changeish
          changeish --since last-tag --output CHANGELOG.md
      
      - name: Update API Docs
        run: |
          pip install doc-comments-ai
          doc-comments-ai --path ./hooks --model gpt-4o
      
      - name: Commit & Push
        run: |
          git add -A
          git diff --cached --quiet || \
            git commit -m "docs: auto-update documentation"
          git push
