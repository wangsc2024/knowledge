一、專案概述

Browser Use（browser-use/browser-use）是一款開源 Python 框架，讓 LLM 能夠自主控制網頁瀏覽器執行複雜任務。專案定位為「讓網站對 AI Agent 可存取」，透過自然語言指令即可完成表單填寫、資料擷取、電商操作、研究任務等多步驟瀏覽器自動化工作。

二、七層架構模型

Browser Use 的架構從自然語言任務到 Chrome DevTools Protocol（CDP）命令分為七層：

第 1-2 層：任務與推理層

自然語言任務流入 Agent.step()，實作 Sense-Think-Act 迴圈。Agent 是中央調度器，協調訊息管理、LLM 呼叫與動作執行三大子系統。

第 3-4 層：動作層

LLM 輸出透過動態生成的 ActionModel（Pydantic schema）進行驗證。內建動作包括：

| 動作 | 用途 |

|------|------|

| navigate(url) | CDP Page.navigate 帶等待條件 |

| click(index) | 依 DOM 索引點擊元素，處理自動完成偵測 |

| input_text(index, text) | 輸入文字，支援敏感資料處理 |

| extract(text_format) | 從頁面擷取結構化資料 |

| scroll(direction, amount) | 捲動頁面或特定元素 |

| go_back() | 瀏覽器返回 |

| search(query) | 搜尋頁面內容 |

| done(result) | 任務完成訊號 |

自訂動作透過 @tools.action() 裝飾器擴展，無需修改核心程式碼。

第 5-7 層：執行層

事件透過 EventBus（bubus 函式庫）分派給專門的 Watchdog 處理器執行 CDP 命令。此解耦設計確保可擴展性與韌性。

三、核心技術創新

3.1 DOM 壓縮管線（DomService + DOMTreeSerializer）

這是 Browser Use 最關鍵的技術創新。DomService.build_enhanced_tree() 建立 EnhancedDOMTreeNode：

  1. 三源合併：同時擷取 CDP DOM 樹、Accessibility 樹與 Layout 快照
  2. 語義-結構合併：將無障礙（AX）語義資料與 DOM 結構資料融合
  3. 非互動元素過濾：移除純裝飾性/不可操作的節點
  4. 包裝層簡化：消除冗餘的巢狀容器
  5. 索引分配：為可點擊/可輸入的元素分配數字索引

效能指標：

  • 壓縮比：10,000+ 節點 → ~5KB 最佳化文字（50-200 個互動元素）
  • Token 消耗：每步 5-10K tokens（未壓縮需 100K+）
  • 優勢：使用數字索引而非 XPath，避免 DOM 變化導致的定位脆弱性

3.2 視覺感知 + DOM 結構化雙模式

Browser Use 結合兩種網頁理解方式：

  • DOM 模式：解析 HTML 結構，擷取所有可點擊連結、按鈕、表單欄位與文字
  • 視覺模式：擷取頁面截圖（base64），讓視覺 LLM 理解佈局、圖片、設計元素
  • 融合策略：視覺模型同時接收截圖 + DOM 索引，實現精確的元素定位

3.3 Event-Driven Watchdog 架構

BrowserSession 透過 EventBus 分派事件：

事件類型：

  • 用戶動作事件：NavigateToUrlEvent、ClickElementEvent、TypeTextEvent、ScrollEvent
  • 生命週期事件：BrowserStartEvent、BrowserStopEvent、BrowserConnectedEvent
  • 狀態監控事件：NavigationCompleteEvent、FileDownloadedEvent、TabCreatedEvent

Watchdog 處理器：

  • DOMWatchdog：導航後重建 DOM
  • DownloadsWatchdog：監控檔案下載完成
  • CrashWatchdog：偵測並恢復崩潰（指數退避重試）
  • DefaultActionWatchdog：執行 CDP 命令
  • PopupsWatchdog：處理對話框/警示

3.4 ActionLoopDetector（迴圈偵測器）

防止 Agent 陷入重複行為：

  • 監控動作雜湊值與頁面指紋
  • 偵測到重複時發出 nudge（如「試著向下捲動」）
  • 追蹤 consecutive_failures 計數

四、Agent 執行流程


1. BrowserSession 擷取狀態 → DomService 建立合併 DOM 樹
2. DOMTreeSerializer 壓縮 10,000+ 節點為 ~200 個互動元素
3. MessageManager 組裝：系統提示 + 任務 + DOM 索引 + 截圖 + 歷史
4. Agent 呼叫 LLM（結構化輸出 Schema = AgentOutput）
5. LLM 回傳：actions[]、memory、evaluation、next_goal
6. Tools.execute_action() 分派事件 → Watchdog 執行 CDP → ActionResult
7. 更新 History；迴圈重複直到任務完成或達 max_steps

訊息壓縮系統

MessageManager.maybe_compact_messages() 在接近 token 上限時壓縮歷史：

  • 透過專用 LLM 呼叫壓縮舊對話輪次
  • 保留 AgentHistoryList 供結構化輸出呼叫使用
  • 可透過 compact_every_n_steps 和 compaction_llm 配置

七、架構設計模式分析

7.1 與 Daily-Digest-Prompt 的設計同構性

| Browser Use 模式 | Daily-Digest-Prompt 對應 |

|------------------|------------------------|

| EventBus + Watchdog | Hook 系統（PreToolUse/PostToolUse） |

| ActionLoopDetector | agent_guardian.py 的 LoopDetector |

| MessageManager 訊息壓縮 | Agent Team 的 Phase 分段 Context 管理 |

| 自訂 @tools.action() | Skill 系統的 SKILL.md 宣告式擴展 |

| BrowserProfile 配置 | config/ 目錄的 YAML 外部化配置 |

7.2 可借鑑的設計決策

  1. 數字索引取代 XPath：Browser Use 為每個互動元素分配整數索引，比路徑定位更穩健。啟示：Agent 系統中的外部資源參照宜用穩定 ID 而非路徑。
  2. Nudge 機制：偵測到迴圈時不直接中斷，而是向 LLM 提供提示引導跳出。比硬性熔斷更優雅。
  3. Pydantic Schema 驗證：所有 LLM 輸出都透過 Pydantic model 驗證，確保結構化與型別安全。

八、應用場景

  1. 求職自動化：解析履歷 → 填寫線上申請表
  2. 電商代理：自然語言購物清單 → Instacart 下單
  3. 資料擷取：將非結構化網頁轉為結構化 JSON
  4. QA 測試：自然語言測試案例 → 自動化端到端測試
  5. 監控告警：持續監控網頁變化、價格追蹤
  6. MCP 整合：作為 AI 助手的瀏覽器能力擴展

---

九、限制與挑戰

  1. 記憶體消耗：每個 Chrome 實例佔用 200-500MB，並行執行需注意資源管理
  2. 反偵測對抗：部分網站的 CAPTCHA 和指紋偵測可能阻擋自動化
  3. LLM 成本：每步 5-10K tokens，複雜任務可能需要 20-50 步
  4. 確定性不足：LLM 決策的非確定性可能導致相同任務不同結果
  5. 安全風險：AI Agent 控制瀏覽器存在 prompt injection 和憑證外洩風險

---

十、技術趨勢判讀

Browser Use 代表 2026 年 AI Agent 發展的三大趨勢：

  1. 從 API 到瀏覽器：越來越多服務沒有 API，瀏覽器自動化成為 AI Agent 的「最後一哩」
  2. DOM 理解即能力：高效的 DOM 壓縮與理解成為瀏覽器 Agent 的核心技術壁壘
  3. MCP 標準化整合：Browser Use 的 MCP Server 模式正成為 AI 助手擴展瀏覽器能力的標準範式
