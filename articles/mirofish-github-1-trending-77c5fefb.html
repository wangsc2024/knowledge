二、核心架構

2.1 設計理念：湧現式預測（Emergent Prediction）

MiroFish 的根本創新在於不使用傳統統計模型或機器學習做預測，而是建構「數位培養皿」（Digital Petri Dish）——讓數千至百萬個 LLM 驅動的自主 Agent 在模擬環境中互動，透過群體行為湧現產生預測結果。

與傳統 ML 預測的關鍵差異：

  • 傳統方式：歷史數據 → 統計模式 → 概率預測
  • MiroFish：情境注入 → Agent 群體互動 → 行為湧現 → 敘事性預測報告

2.2 底層框架：OASIS

基於 CAMEL-AI 的 OASIS（Open Agent Social Interaction Simulations） 框架，支援：

  • 最高 100 萬個 Agent 同時運行
  • 23 種社交行為：關注、評論、轉發、按讚、靜音、搜尋等
  • Agent 間自主形成社交網路拓撲

2.3 五階段模擬管線（Simulation Pipeline）


階段 1：知識圖譜建構（GraphRAG）
  ├── 種子資料注入（新聞/政策/財經信號/論文）
  ├── 實體與關係抽取
  └── 個體/群體記憶結構注入

階段 2：環境生成
  ├── Agent 人設生成（性格、背景、行為規則）
  ├── 社交網路拓撲建構
  └── 模擬參數配置

階段 3：並行模擬執行
  ├── 數千 Agent 同時運行
  ├── 自動解讀預測請求
  ├── 時間記憶動態更新
  └── 環境持續演化（觀點、敘事、行為隨時間變化）

階段 4：報告生成
  ├── ReportAgent 分析結果
  ├── 結構化預測報告
  └── 趨勢、風險、行為洞察摘要

階段 5：深度互動
  ├── 與個別 Agent 對話，探索其決策邏輯
  ├── 與 ReportAgent 追問分析
  └── 探索模擬中的社會動態

2.4 記憶系統

使用 Zep Cloud 管理 Agent 長期記憶：

  • 每個 Agent 維持獨立的人格描述檔
  • 跨模擬輪次累積經驗
  • 支援個體記憶與群體記憶注入

2.5 知識處理：GraphRAG

使用 GraphRAG 架構建構知識圖譜：

  • 從種子材料（新聞、政策文件、財經報告）抽取實體與關係
  • 知識圖譜作為 Agent 模擬的共享知識基礎
  • 支援跨輪次知識累積

三、技術棧與部署

| 組件 | 技術 |

|------|------|

| 後端 | Python 3.11-3.12 |

| 前端 | Vue.js |

| Agent 框架 | OASIS（CAMEL-AI）|

| 記憶管理 | Zep Cloud |

| 知識圖譜 | GraphRAG |

| LLM 支援 | 任何 OpenAI SDK 相容 API（推薦阿里通義千問 Qwen-Plus）|

| 部署方式 | Docker（前端 :3000 / 後端 :5001）或原始碼安裝 |

部署注意事項

  • 模擬輪次建議 < 40 輪（計算資源考量）
  • Zep Cloud 免費層可用於小規模實驗
  • 每個 Agent 需要 LLM API 呼叫，計算成本高

四、應用場景

  1. 輿情預測：模擬公眾對爭議性事件的反應軌跡與情緒演化
  2. 金融預測：模擬交易者心理、羊群效應、市場情緒傳播
  3. 政策影響分析：跨利害關係人群體模擬政策實施後果
  4. 情境規劃：企業「What-if」沙盤推演
  5. 社群媒體建模：模擬爭議事件的擴散動態
  6. 虛構情境推演：用虛構設定測試社會動態假說

六、限制與風險

  1. 計算成本高：每 Agent 每輪需 LLM API 呼叫，大規模模擬費用可觀
  2. 結果隨機性：相同場景可能產出不同結果（非確定性）
  3. 早期階段：v0.1.2，僅 4 個月開發歷程
  4. 驗證困難：預測準確度需真實世界結果回饋確認
  5. 文化偏差：Agent 行為可能無法完美捕捉真實人類文化差異

---

七、對本專案的啟發

  1. Agent 規模思維：MiroFish 展示了百萬級 Agent 模擬的可行性，對日常 Agent 系統設計提供了新的規模視角
  2. GraphRAG 整合模式：知識圖譜 + Agent 記憶的結合，可借鏡於知識庫系統的深度搜尋
  3. 湧現式設計：不預設結果，讓系統行為自然湧現——這與 Skill-First 的「按需載入」理念相通
  4. Vibe Coding 實踐：20 歲開發者 10 天完成專案，AI 輔助開發的生產力範例
