Harness Engineering 系統優化執行方案

執行摘要

基於 2026 年 4 月最新研究（14 篇權威來源三角佐證），針對 Daily-Digest-Prompt 系統提出三階段可執行優化方案，預期可提升系統可靠性 40-60%、降低 token 使用 20-30%、減少人工干預 50%。

核心數據支撐：

  • Harvey Legal Agents：2% → 98% 完成率（Autoresearch + Rubric）[Artificial Lawyer, 2026]
  • Meta-Harness：TerminalBench-2 達 76.4%（Opus 4.6），Context tokens 降 77% [Stanford arxiv 2603.28052]
  • Context 排名優化：成本降 91%（$48k → $4.5k/月），品質提升 20% [Shaped.ai, 2026]

---

一、三大核心洞見（2026 突破）

洞見 1：Autoresearch + Rubric = 自動化品質提升引擎

實證來源：Harvey Legal Agents 實驗（12 個複雜法律任務）

機制：


Agent 執行任務 → LLM Judge 依 Rubric 評分 → Coding Agent 分析失敗叢聚
→ 形成改進假設 → 修改系統組件 → 重新執行 → 循環

關鍵數據：

  • Complaint 起草：2% → 98% 完成率（提升 4800%）
  • 平均成功率：40.8% → 87.7%（+115% 相對提升）
  • 7/12 任務達 >90% 成功率

核心引言：

"When the rubric is high quality, the agent can hill-climb surprisingly far." [Artificial Lawyer, 2026]

對系統的意義：

  • 當前自修復成功率約 40%，引入 Rubric 可提升至 65-75%
  • 失敗模式可被自動識別並形成新約束

---

洞見 2：Meta-Harness — 自動優化優於手工調優

實證來源：Stanford IRIS Lab Meta-Harness 論文 [arxiv:2603.28052, 2026]

技術架構：

  • Coding Agent（Claude Code）訪問完整執行歷史
  • 用 grep/cat 選擇性檢查（非一次性載入全部）
  • 偵測長期失敗 pattern（40 次迭代達收斂）
  • 自動生成新 Harness 版本並評估

性能數據：

| 任務 | 指標 | Meta-Harness | 手工最佳 | 改進 |

|------|------|-------------|---------|------|

| Text Classification | Accuracy | ACE +7.7 pp | Baseline | +7.7% |

| | Context Tokens | 11.4K | 50.8K | -77% |

| TerminalBench-2 (Opus 4.6) | Pass Rate | 76.4% | 74.7%（KIRA）| 排名 #2 |

與現有方法對比：

  • TextGrad/OPRO：0.002-0.026M tokens/iteration
  • Meta-Harness：~10M tokens/iteration（1000 倍診斷資訊，但總成本更低）

對系統的意義：

  • 當前手工維護 35 個 auto-task prompt，可實現自動化優化
  • 執行歷史（state/*.jsonl）是改進的關鍵資產

洞見 3：Context 排名比填充重要 10 倍

實證來源：Shaped.ai Context Window 優化 [Shaped.ai Blog, 2026]

核心物理約束：

"Attention cost scales O(n²) with context length" — 2x tokens = 4x compute cost

量化對比（RAG 場景）：

| 策略 | Tokens/Query | 成本/月（10K 查詢/日） | 品質 |

|------|-------------|---------------------|------|

| Naive RAG（200 chunks） | 50,000 | $48,000 | 中 |

| Shaped 排名（10 結果） | 2,500 | $4,500 | 高 |

| 改進 | -95% | -91% | +20% |

為什麼 Vector Similarity 失敗？

"Cosine similarity 衡量 embedding 空間距離，而非有用性（usefulness）。" [Shaped.ai, 2026]

正確做法：多檢索器管線（6 層）：

  1. Vector search（語義相似度）
  2. Lexical search（BM25 關鍵字）
  3. Collaborative filtering（用戶行為模式）
  4. Trending signals（實時熱度權重）
  5. ML scoring ensemble（LightGBM/XGBoost 排名模型）
  6. Diversity reordering（MMR 多樣性）

對系統的意義：

  • 當前 KB 查詢僅用 vector + lexical（2/6 層）
  • 加入行為追蹤可顯著提升相關性

二、三階段優化方案

Phase A：快速勝利（2 週，P0）

#### A1：建立 Rubric 評分系統

目標：為 3 個高價值任務建立量化評分標準

核心配置（config/task-rubrics.yaml）：


tech_research:
  threshold: 7.0  # 最低可接受分數（滿分 10）
  dimensions:
    - name: source_quality
      weight: 0.25
      criteria:
        - "≥3 個官方文件或學術來源"
        - "來源發布日期 ≤6 個月"

    - name: triangulation
      weight: 0.30
      criteria:
        - "每個核心主張有 3+ 獨立來源佐證"
        - "引用格式正確（[來源, 年份]）"

    - name: actionability
      weight: 0.20
      criteria:
        - "包含實務應用建議章節"
        - "至少 3 個可執行行動項"

    - name: completeness
      weight: 0.15
      criteria:
        - "字數 ≥500 且 ≤10000"
        - "必要章節齊全（摘要、分析、建議、參考來源）"

    - name: structure
      weight: 0.10
      criteria:
        - "每個主要章節 ≥3 段落"
        - "標題層級正確（#/##/###）"

預期效果（3 週內）：

  • 自修復成功率：40% → 65%
  • 輸出品質可量化追蹤（每日 35 次評分記錄）
  • 低分任務自動觸發重做或降級

投資回報：開發工時 8h，月省 3-5 小時人工修正

---

#### A2：Embedded Rubrics（分數嵌入錯誤訊息）

目標：讓 Agent 看到違反約束的即時影響

實施範例（validate_config.py 輸出）：


❌ Error: frequency-limits.yaml missing 'daily_limit' for task 'tech_research'

📊 Rubric Impact（預估）：
- Completeness 維度：-2.0 分（當前 6.5/10）
- 當前總分：6.2/10（警戒線：7.0/10）
- ⚠️ 低於閾值，任務將被標記為 quality_gate_blocked

💡 Suggested Fix（預期改進 +2.5 分 → 總分 8.7/10）：
1. 開啟 config/frequency-limits.yaml
2. 在 tech_research 段落下新增：
   daily_limit: 6
3. 驗證修正：
   uv run python tools/validate_config.py

預期效果：Agent 自我修正準確度提升（知道優先修什麼）

---

#### A3：用戶行為追蹤（KB 查詢）

目標：為 Context 排名優化累積訓練資料

KB API 擴展：


{
  "noteId": "abc123",
  "feedback": {
    "session_id": "todoist-1234",
    "used_in_final_output": true,
    "relevance_score": 4,
    "usage_type": "direct_quote|paraphrase|context_only|rejected",
    "timestamp": "2026-04-22T10:30:00+08:00"
  }
}

預期效果：2 週內累積 100+ 行為樣本，為 ML 排名模型準備資料

Phase B：基礎建設（4 週，P0-P1）

#### B1：Fitness Functions 自動驗證

目標：ADR 定義的架構特性可自動量化驗證

實施步驟：

  1. 擴展 tools/validate_adr.py 加入 fitness 驗證模組
  2. 建立 tools/fitness_checkers/（模組化檢查器）
  3. 整合到 check-health.ps1（每次執行驗證）

預期效果：架構侵蝕即時偵測，ADR 成為「可執行規範」

投資回報：開發工時 16h，減少架構漂移修復成本（每次 ~2-4h）

---

#### B2：自動約束提取（失敗模式 → 新約束）

目標：從執行日誌自動識別失敗模式，提議新約束

核心工具（tools/constraint_miner.py）：從 trace.jsonl 挖掘失敗模式，用 LLM 叢聚相似失敗，提取共同 pattern，生成 constraint proposal。

預期效果：每週自動提出 2-4 個約束提案，人工識別工作量降低 70%

投資回報：開發工時 24h，每週省 1-2h 人工日誌分析

---

#### B3：熵趨勢追蹤與減熵處方

目標：建立熵分數歷史基線，觸發閾值自動化

實施步驟：

  1. 擴展 tools/entropy_score.py（定期計算並追蹤趨勢）
  2. 建立減熵處方機制（tools/entropy_remediation.py）
  3. 整合到排程（每日 06:00）

預期效果：熵分數穩定在 2.0-2.5，減熵處方自動化率 80%

投資回報：開發工時 12h，每月省 2-3h 人工清理

---

Phase C：智能演化（8 週，P1-P2）

#### C1：Meta-Harness 實施（Prompt 自動優化）

目標：讓系統自動優化 prompt 模板

核心機制：分析失敗模式 → Claude Opus 4.6 提出改進假設 → 生成新版本 → A/B Testing → 顯著改善則採用

預期效果：

  • 3 個高價值任務平均 Rubric 分數：7.2 → 8.5
  • Prompt 優化週期：手動 2-4 週 → 自動 3-5 天

投資回報：開發工時 32h，每月省 4-8h prompt 手動調優

---

#### C2：ML 排名模型（KB 查詢）

目標：用機器學習提升 KB 查詢相關性

技術架構：

  1. 準備訓練資料（Phase A3 累積）
  2. 訓練 LightGBM 模型（Learning to Rank）
  3. 整合到 KB API（/api/search/ranked）

特徵工程：Vector similarity、BM25 score、Note freshness、Tag overlap、Historical usage、Session context

預期效果：

  • KB 查詢相關性（NDCG@5）：0.65 → 0.82
  • 平均 query tokens：2500 → 1500

投資回報：開發工時 24h，月省 ~1M tokens（約 $10-15）

---

#### C3：端到端自動優化迴圈

目標：整合 A-C2，形成完整閉環

架構圖：


執行任務 → Rubric 評分 → 失敗模式叢聚
    ↓                         ↓
Meta-Harness        約束挖掘器
自動優化 Prompt      提議新約束
    ↓                         ↓
A/B Testing         人工審議
    ↓                         ↓
替換最佳版本    → embedded 到 config/
    ↓
Fitness Functions 驗證
    ↓
熵趨勢監控 → 觸發減熵處方

預期效果（12 週後）：

  • 系統自修復率：40% → 75%
  • 人工干預頻率：每週 5-8 次 → 每週 1-2 次
  • 平均任務品質（Rubric）：7.2 → 8.3

投資回報：開發工時 16h，每週省 3-6h 人工維護
