二、RAG 檢索品質度量化

2.1 工具設計

新建 tools/rag_quality_check.py，採用 8 組覆蓋不同知識領域的基準查詢：

| 類別 | 查詢 | 預期結果數 |

|------|------|----------|

| RAG | RAG 管線效率優化 | ≥2 |

| Agent | Agent 架構治理 | ≥3 |

| 系統 | Todoist API 遷移 | ≥1 |

| 研究 | 知識庫跨領域遷移 | ≥1 |

| 排程 | 自動任務排程效率 | ≥1 |

| 安全 | Hook 攔截規則 | ≥1 |

| 認知 | 貝葉斯推理決策 | ≥1 |

| AI | 深度學習 Transformer | ≥1 |

度量指標：

  • Precision@K：前 K 個結果中 score ≥ 0.4 的比例
  • MRR（Mean Reciprocal Rank）：第一個相關結果的倒數排名
  • 分數分布：high(≥0.7) / medium(0.4-0.7) / low(0.2-0.4) / irrelevant(<0.2)

2.2 基線結果（2026-04-03）

| 指標 | 值 | 評估 |

|------|----|----- |

| Precision@3 | 66.7% | 良好（目標 ≥70%） |

| Precision@5 | 40.0% | 需改善（目標 ≥50%） |

| MRR | 1.000 | 優秀（首筆必中） |

| 平均分數 | 0.2725 | 偏低（多數結果落在 low 區間） |

| 分數標準差 | 0.0193 | 穩定（跨查詢一致性高） |

| 達標率 | 8/8 (100%) | 所有查詢找到足夠結果 |

2.3 關鍵洞見

  1. MRR 完美但 P@5 不足：搜尋引擎「找得到」但「排得不好」— 首筆必中，但第 4-5 名結果品質下降
  2. high 區間結果為 0：所有查詢的 max_score 均 ≤ 0.5，說明向量相似度或 RRF 融合權重需調整
  3. 分數一致性高（stddev=0.019）：不同領域查詢表現均勻，無特定領域弱點

2.4 優化建議

  • 短期：調整 RRF 融合的 k 參數（目前可能過大，壓低了高分文件的排名）
  • 中期：引入輕量 Reranking（如 cross-encoder 或 LLM-based rerank）提升 P@5
  • 長期：建立 Ground Truth 標註集，從 benchmark 升級為 evaluation pipeline

3.2 ADR Fitness 覆蓋率

| 指標 | 值 |

|------|----|

| 已接受 ADR 總數 | 388 |

| 含 fitness function | 1 |

| 覆蓋率 | 0.3% |

3.3 根因分析

  1. 審議流程缺失：constraint-registry.json 中 7 筆 pattern 均為 review_required，但無自動或手動審議機制觸發接受/拒絕
  2. ADR 產出速度遠超 fitness 補齊速度：每日新增 5-10 ADR，但 fitness function 需人工定義
  3. 建議：建立自動審議閾值（trigger_count ≥ 10 自動接受）+ ADR fitness 批量生成腳本
