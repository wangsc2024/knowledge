AI 驅動 CI/CD 優化進階實作：flaky test 治理、測試選擇與 build 預測工作流（2026）

技術概述

AI 驅動的 CI/CD 已從單純「重跑失敗測試」走向資料驅動的測試選擇、flaky quarantine 與 pipeline 排程協同。根據 Launchable 官方文件，Predictive Test Selection 會用歷史測試結果、變更檔案、測試特徵與環境資訊訓練模型，目標是在最短時間內優先跑出最可能失敗的測試。Buildkite Test Engine 則把 flaky 偵測、標記、隔離與 test splitting 產品化；Atlassian 進一步展示了大型 monorepo 中如何用 Flakinator 把偵測、隔離、通知、Jira 指派與恢復流程串成平台。對 daily-digest-prompt 這類自動化專案，這代表可以把 CI 從「全部都跑」升級為「先蒐集訊號，再針對風險變更跑最值錢的測試」。

核心方法與工具

  • Predictive Test Selection（Launchable）：根據測試歷史、變更特徵、檔案相似度與環境 flavor，對完整測試集排序，再依 optimization target 切出子集。
  • Flaky Detection + Quarantine（Buildkite）：以 monitor 偵測 flaky，打上 flaky label，並將測試轉為 muted 或 skipped；bktec 可在執行時自動排除 quarantined tests。
  • 平台化治理（Atlassian Flakinator）：收集 test run metadata、用多種偵測機制與 Bayesian scoring 建立 flakiness score，再自動建立 Jira/Slack 任務與恢復流程。
  • 必要資料欄位：test_id、suite、status、duration_ms、retry_count、commit_sha、branch、changed_files、runner、environment、owner。
  • 治理指標：build success rate、median feedback time、quarantined test count、false positive rate、recovered builds、test minutes saved。

實際工作流程

  1. 蒐集測試遙測

在 CI 中對每次測試上報測試名稱、結果、耗時、重試次數、commit SHA 與變更檔案。

  1. 建立 flaky 判定層

先用規則法抓同一 commit 下 pass/fail flip，之後再加入歷史視窗、重試頻率、duration variance 等訊號。

  1. 自動標記與隔離

對疑似 flaky 測試自動加 flaky label，先 muted 而非直接刪除，以保留資料並避免阻塞主線。

  1. 導入測試選擇

對 pull request 先跑預測子集；main branch 或 nightly 再跑 full suite 以回補標籤資料與模型訓練資料。

  1. 回寫責任歸屬

依 code owner 或 test owner 自動建立 issue / Slack 通知，要求在 SLA 內修復或解除 quarantine。

  1. 監控成效

追蹤 CI 時間下降比例、retest 次數、誤隔離率與恢復後再犯率；若資料漂移，回退到較保守的規則模式。

程式碼 / 配置範例

1. GitHub Actions 蒐集測試遙測並產生最小資料集


name: ci
on:
  pull_request:
  push:
    branches: [main]

jobs:
  tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Run unit tests with JUnit output
        run: npm test -- --reporter=junit --reporter-options output=reports/junit.xml
      - name: Export CI telemetry
        run: python scripts/export_test_telemetry.py reports/junit.xml "$GITHUB_SHA" > reports/test-telemetry.jsonl
      - uses: actions/upload-artifact@v4
        with:
          name: test-telemetry
          path: reports/

2. 最小 flaky 偵測腳本


from collections import defaultdict

def detect_flips(records):
    by_test_commit = defaultdict(set)
    for row in records:
        key = (row["test_id"], row["commit_sha"])
        by_test_commit[key].add(row["status"])
    return [
        {"test_id": test_id, "commit_sha": sha, "is_flaky": True}
        for (test_id, sha), statuses in by_test_commit.items()
        if {"passed", "failed"}.issubset(statuses)
    ]

效果量化

  • Atlassian（2025-12-08）：Jira Frontend 曾有 21% master build failures 來自 flaky tests；Jira backend 約 15% repo failures 與 flaky tests 有關；每年浪費 150,000+ 小時；Flakinator 已恢復 22,000+ builds，識別 7,000 個獨特 flaky tests，系統處理量超過 350 million test executions/day。
  • Launchable 官方文件：其模型明確以「在最短測試時間內找出最可能失敗的測試」為目標；這不是單純檔案對映，而是結合變更特徵、歷史失敗關聯與測試行為的排序問題。
  • Buildkite 官方文件：透過 bktec 可同時做 intelligent test splitting 與 quarantined test 排除。依文件描述，這會直接帶來更快且更穩定的 pipeline。

以上數值為來源直接提供；若推估 daily-digest-prompt 的效益，我的推論是先做 flaky 隔離與測試分層，通常比直接導入完整 ML 模型更快見效。

與本專案的應用場景

  • tools/validate_results.py、tools/research_series_advisor.py、各種 prompt schema 驗證可先被標為「高價值快速測試」，PR 階段優先執行。
  • 若 Playwright 或整合測試偶發失敗，可先建立 flaky-tests.json 或 state/quarantined-tests.json，在主線上 soft-fail，但 nightly 持續收集資料。
  • logs/structured/*.jsonl 已有可觀測性基礎，可擴充成測試遙測來源，先做規則式 flip 偵測，再考慮 ML 排序。
  • round-robin / result validation 相關腳本變更頻繁，適合先導入 change-based test selection，縮短每次自動任務修改後的回饋時間。

風險與注意事項

  • 資料品質不足：缺少穩定的 test ID、重試資訊或 changed files，模型與規則都會失真。
  • 過度隔離：把真正的產品缺陷誤判成 flaky，會讓主線品質假性變好。
  • 樣本偏差：只在 PR 跑子集會造成訓練資料偏掉，因此必須保留 main / nightly full suite。
  • 治理成本：quarantine 若沒有 owner、SLA 與解除條件，只會把測試債務藏起來。
  • 工具綁定風險：Buildkite、Launchable 屬平台能力；若本專案短期不採購，應先做自建規則版最小閉環。
