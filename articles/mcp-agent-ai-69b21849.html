mcp-agent 深度研究報告

概述

本文為 ai-workflow-automation 系列第 mechanism 階段，承接既有的 Dify 平台層與 Temporal durable workflow 研究，深入補齊 MCP 協議層的工作流自動化機制。mcp-agent 是 LastMile AI 推出的 Python 框架，核心主張是把 MCP 當成 AI 工作流的共通控制面，並以可組合的工作流模式取代大型黑箱式 Agent 平台。它要解決的不是單一聊天代理，而是「如何讓多個工具、資料來源、MCP Server 與長流程任務，用同一套協議與執行模型協同運作」。

目標使用者主要是要打造生產級 Agent 工作流的工程團隊：一類是已經擁有多個 MCP Server、希望快速串成可部署代理系統的開發者；另一類是需要 pause/resume、人工核准、可觀測性與長任務持久化的團隊。相較於 Dify、n8n 這種偏視覺化的平台，mcp-agent 更接近「MCP-native 的程式碼優先工作流 runtime」。

技術架構

mcp-agent 的核心結構可拆成四層。第一層是 MCPApp runtime，負責載入設定、管理 secrets、初始化 server registry、logger、OTEL tracing 與 executor。第二層是 Agent / AgentSpec，定義模型可呼叫哪些 MCP servers、有哪些 instruction 與人類輸入 callback。第三層是 AugmentedLLM 與 workflow factory，將 Agent 包裝成可執行的推理單元，並透過 create_parallel_llm(...)、create_router_llm(...)、create_orchestrator(...)、create_evaluator_optimizer_llm(...) 等 helper 拼成更高階的工作流。第四層是 execution engine，開發期可用 asyncio 在單程序記憶體內執行，進入生產後只要把 execution_engine 切到 temporal，相同 workflow 就能獲得 durable execution、retry、signals、query、workflow replay 與 child workflow 能力。

這個架構有兩個特點。第一，它不是 graph-first，而是 code-first。官方直接強調開發者可以用 if、while 這種一般程式控制流取代節點圖編排，這讓複雜條件邏輯、循環與自訂狀態管理更自然。第二，它把 MCP 放在架構中心而不是外掛層，工具、資源、提示模板、Roots、Elicitation、Sampling 都是同一個協議面的一部分，因此 agent 可以同時消費工具與資料，而不只是 function calling。

工作流核心能力

mcp-agent 的工作流能力可分成三類。第一類是模式庫。官方文件列出完整的 Parallel、Router、Intent Classifier、Planner/Orchestrator、Deep Research、Evaluator-Optimizer 與 Swarm 模式，幾乎對應到 Anthropic《Building Effective Agents》中的主要設計模式，並且可以彼此巢狀組合。這意味著一個工作流可以先做 intent classification，再路由到某個 orchestrator，內部再 fan-out 到多個 specialist agents，最後用 evaluator 做品質把關。

第二類是 MCP 原生能力。它不只支援 Tools，還支援 Resources、Prompts、Roots、Elicitation 與 Sampling。這點對工作流自動化很關鍵，因為企業任務經常不是單純叫工具，而是先讀資源、提取 prompt 模板、等待人類核准、再繼續執行。官方也支援 stdio、SSE、WebSocket、streamable HTTP 四種 transport，代表同一套 workflow 可以接本地 subprocess server，也能接遠端雲端 MCP 端點。

第三類是營運控制能力。它提供 TokenCounter 監看 token 用量、OpenTelemetry tracing、詳細 execution summary，以及 human input / signals 機制。搭配 Temporal 時，可以把長任務停在等待核准、等待外部事件或等待資料補件的狀態，再由 operator 恢復。這使它比一般「一次回答型」Agent 框架更接近真正的業務流程引擎。

AI 整合機制

mcp-agent 的 AI 整合是「LLM provider 抽象層 + MCP 工具面 + workflow pattern」三者的組合。模型層面支援 OpenAI、Anthropic、Google、Azure、Bedrock、Ollama 等 provider；Agent 本身只定義能力邊界與 server_names，真正執行時再透過 attach_llm(...) 綁定某個 AugmentedLLM。這讓同一個 workflow 可以維持結構不變，僅替換模型供應商或請求參數。

更深一層來看，它不是把 LLM 看成唯一主角，而是把 LLM 當成 workflow 中的推理節點。MCP servers 負責把外界能力與上下文以標準介面暴露出來，workflow patterns 負責規劃、路由、分工與評估，Temporal 或 asyncio 則決定執行生命週期。這種分層讓 AI 整合從「單一模型調工具」升級為「可持久化的 Agent runtime」。限制則在於：若團隊沒有 MCP server 資產、也不熟 Python async/Temporal，初期導入門檻會高於 Dify 或 LangFlow 這類視覺化產品。

部署與維運

部署面向分成四種路徑。最輕量的是本機 asyncio 開發模式，適合原型、除錯與單節點運行；第二種是自架 Temporal，適合 on-prem 或 air-gapped 環境；第三種是官方 mcp-c managed cloud，CLI 會處理 bundle、secrets、auth、client install 與 observability；第四種是把應用直接暴露成標準 MCP server，讓 Claude、Cursor、ChatGPT Apps 或自家 host 直接連線。官方文件明確指出，cloud 路徑下每個 stdio MCP server 會被容器化，workflow 則交給 Temporal 執行，這使生產環境的可隔離性與恢復能力都比單純本機腳本高。

維運上，mcp-agent 的優勢是同一份 workflow 程式可在 asyncio 與 Temporal 間切換，能先用低成本方式開發，再把可靠性升級到生產級。不過代價也很明確：若你需要 durability，就必須接受 Temporal 叢集、worker、task queue 與 observability 管線的額外複雜度；若只想要快速拖拉出 AI flow，這條路會比 no-code 平台重。

與 daily-digest-prompt 的連結

對 daily-digest-prompt 而言，mcp-agent 最值得借鑑的不是某一個 API，而是三個控制面概念。第一，將工具暴露、資料讀取、人工核准與工作流恢復統一放在 MCP 之下，能讓現有 Skill / Hook / 本機腳本逐步轉成可被 Agent 標準化呼叫的能力面。第二，採用「程式碼優先 + 模式工廠」的做法，與本專案既有的 prompt/template 驅動設計很相容，未必需要改成圖形工作流。第三，若未來要做長任務研究、自動補件、人工審批後續執行，asyncio -> Temporal 的雙引擎切換概念可作為本專案從腳本型自動化升級到 durable orchestration 的路線。

更具體地說，可以借鑑它的 TokenCounter 與 OTEL tracing 模型，將多步研究任務的 token、分支與失敗點納入統一觀測；也可以借鑑 human input / signals，把目前需要人工確認的研究、匯入、發送步驟轉成可暫停再恢復的流程。若把 daily-digest-prompt 視為「以檔案與 Skill 為中心的 Claude Agent 系統」，那 mcp-agent 提供的是一個把這些能力整理成 MCP-native runtime 的參考實作。

優缺點評估

優點：

  • MCP 原生，不把工具整合作為附屬插件，而是把 Tools、Resources、Prompts、Roots、Elicitation、Sampling 都納入同一工作流面。
  • 兼具簡單與擴展性：開發期可用 asyncio 快速迭代，生產期可切到 Temporal 而不需重寫 workflow。
  • 模式庫完整，直接提供 parallel、router、orchestrator、deep research、evaluator、swarm 等可組合流程。
  • 程式碼優先，對需要條件分支、循環、狀態與自訂控制流的工程團隊很友善。
  • 具備生產級觀測與控制能力，包含 token accounting、OTEL tracing、retry、pause/resume、human-in-the-loop。

缺點 / 限制：

  • 偏 Python 工程框架，不適合非技術使用者；與 Dify、n8n 相比缺少成熟的視覺化建模體驗。
  • 若要發揮完整價值，通常需要 MCP server 資產與 Temporal 維運能力，導入門檻高於單純 SDK。
  • 官方雲端 mcp-c 仍在 Beta，對穩定性、定價與企業治理需求較高的團隊仍需自行驗證。
  • 因為主張「用程式控制流取代 graph」，大型團隊若強依賴視覺化審查與非工程角色協作，未必比 LangGraph Studio 或 Dify workflow 更易溝通。
  • 社群規模雖快速成長，但與 Dify、n8n、LangGraph 這類更成熟生態相比，整體整合與第三方教學仍較少。
