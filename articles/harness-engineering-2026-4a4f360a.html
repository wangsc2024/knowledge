Harness Engineering 深度研究與系統優化方案 (2026)

版本: 3.0
日期: 2026-04-22
狀態: 研究報告 + 實施建議
基於: Daily Digest Prompt 系統現有實施 (v1.0/v2.0) + 2026 業界最新實踐

---

執行摘要

本研究整合 Martin Fowler、Anthropic 與 2026 年產業界最新 Harness Engineering 實踐，結合 Daily Digest Prompt 系統既有基礎（B0-B3 批次實施），提出下一代優化方案。

核心發現：

  • 2026 共識：基礎設施優化勝過模型升級（實驗證明 10x 提升來自編輯格式變更，而非模型換代）
  • 關鍵轉變：從「Agent 寫程式」轉向「設計讓 AI 可靠寫程式的系統」
  • 實證證據：Context reset 在 40% 使用率時優於等到 80% 才壓縮
  • 成本控制：Computational 驗證優先（毫秒級），Inferential 驗證策略性部署（秒級）

1. Harness Engineering 核心框架

1.1 定義與邊界

Agent = Model + Harness

Harness 包含除模型外的所有機制，分為兩層：

  1. 內建層：Agent 構建者提供（如 Claude Agent SDK）
  2. 使用者層：特定場景客製化（本專案的 Hooks、Skills、OODA 流程）

1.2 四大組件（agent-engineering.dev 標準）

| 組件 | 定義 | 本專案對應 | 成熟度 (v2.0) |

|------|------|-----------|--------------|

| 架構約束 | 透過規則限制 Agent 行為空間 | Hooks、hook-rules.yaml、約束演化管線 | 8.5/10 → 目標 9.5 |

| 反饋迴路 | 執行結果回饋至決策層 | quality_gate.py、constraint_evolve.py | 9.0/10 → 目標 9.5 |

| 驗證護欄 | 多層品質門檻 | Draft-Verify、結果驗證、Schema 檢查 | 8.8/10 → 目標 9.5 |

| 生命週期管理 | 狀態追蹤與時間閾值 | timing-policy.yaml、OODA workflow | 9.2/10 → 目標 9.5 |

2. 雙控制機制：Feedforward vs Feedback

2.1 Martin Fowler 框架

Guides（前饋控制） — 預防性引導，在行動前提高首次成功率

  • 系統提示（System Prompts）
  • AGENTS.md / CLAUDE.md
  • Skill 指引文件
  • Good Examples Repository（本專案 v2.0 新增）

Sensors（反饋控制） — 事後校正，觀察輸出並促進自我修正

  • 靜態分析工具（Linter、Type Checker）
  • 測試執行（Unit / Integration / E2E）
  • Quality Gate LLM 評分
  • Hook 攔截與修復建議

關鍵洞察：

單獨使用任一方式都不足：僅有反饋導致重複犯同樣錯誤；僅有前饋則永遠無法驗證有效性。

2.2 本專案實施現狀與改進

| 機制 | 現有實作 | v2.0 改進方向 | 預期效果 |

|------|---------|--------------|---------|

| Feedforward | preamble.md、CLAUDE.md、Skill 模板 | + Good Examples Registry（優秀案例庫） | 首次成功率 72% → 85% |

| Feedback | Hooks、quality_gate、constraint_evolve | + Success Pattern Extractor（成功模式自動升級） | iteration_count 1.3 → 1.1 |

| Feedback → Feedforward | ❌ 斷裂 | ✅ 自動升級管線 | 減少重複錯誤 15% |

3. 驗證金字塔：Computational vs Inferential

3.1 核心原則

| 類型 | 特性 | 成本 | 適用場景 |

|------|------|------|---------|

| Computational（確定性） | 規則式、快速、可重複 | 毫秒級 | JSON Schema、必填欄位、regex pattern |

| Inferential（推理式） | 語義分析、非確定性 | LLM 調用（秒級） | 文意完整性、邏輯連貫性、創意品質 |

最佳比例：70% Computational + 30% Inferential（Martin Fowler 建議）

3.2 本專案優化方案（v2.0 三層金字塔）


           ┌─────────────┐
           │  Tier 3     │  LLM 語義評分（僅 Critical 任務）
           │  5% 流量    │  成本：Haiku ~5s
           └─────────────┘
          ┌───────────────┐
          │  Tier 2       │  啟發式檢查（keyword 密度、引用完整性）
          │  20% 流量     │  成本：regex + NLP ~200ms
          └───────────────┘
         ┌─────────────────┐
         │  Tier 1         │  確定性檢查（schema、必填欄位）
         │  100% 流量      │  成本：<10ms
         └─────────────────┘

預期效果：

  • LLM 呼叫從 30 次/日 → 9 次/日（減少 70%）
  • 驗證延遲從 150s → 50s
  • 誤判率 < 5%（Tier 2 閾值設為 8.0/10）

4. Context Management：Reset vs Compaction

4.1 Anthropic 實證研究

關鍵發現：

Context compaction 會讓模型焦慮接近上限（context anxiety），導致提早結束任務。

兩種策略對比：

| 策略 | 觸發點 | 優點 | 缺點 | 適用場景 |

|------|--------|------|------|---------|

| Compaction | > 80% | 保留 context 連貫性 | 模型焦慮、資訊損失 | Opus 4.5 內建自動壓縮 |

| Clean Reset | 40-60% | 無焦慮、清晰狀態 | 需要結構化 Handoff | 多會話長時間任務 |

4.2 本專案實施方案（v2.0 Context Reset Coordinator）

三級重置策略：

| Context 使用率 | 動作 | 實作 |

|---------------|------|------|

| < 60% | 正常繼續 | 無動作 |

| 60-75% | Artifact Handoff | 寫入 state/agent-handoff/{trace_id}.json，清空 context，新 Agent 讀取繼續 |

| > 75% | Emergency Reset | 強制終止，標記 status=partial，checkpoint 寫入 handoff |

Handoff Schema 設計：


{
  "trace_id": "abc123",
  "original_task": "深度研究 Harness Engineering",
  "completed_steps": ["WebSearch", "WebFetch", "初步分析"],
  "artifacts": {
    "web_content": "file://state/agent-handoff/abc123-web.md",
    "kb_drafts": ["file://state/agent-handoff/abc123-kb-01.md"]
  },
  "next_steps": ["撰寫優化方案", "寫入知識庫", "驗證"],
  "context_snapshot": null
}

與現有 ADR-036 的關係：

  • ADR-036（Summary/BufferWindow）保留作為 > 90% 最後手段
  • 60-75% 區間改用 Handoff，避免資訊損失

5. 多 Agent 架構：專業化分工

5.1 Anthropic 三 Agent 模式

架構：Planner → Generator → Evaluator

| 角色 | 職責 | 輸入 | 輸出 |

|------|------|------|------|

| Planner | 任務分解與資源配置 | 功能列表 + 環境狀態 | 執行計畫（JSON） |

| Generator | 實際編碼實作 | 執行計畫 + codebase | 程式碼變更 + git commit |

| Evaluator | 品質驗證與測試 | 變更集 + 測試腳本 | 通過/失敗 + 改進建議 |

核心優勢：

  • 專業化：每個 Agent 只做一件事，提示詞可獨立調優
  • 可觀測性：Planner 輸出計畫 JSON 可單獨審查
  • 避免自評偏差：Generator 與 Evaluator 分離，消除「AI 驗證 AI」盲點

5.2 本專案四角色設計（v2.0 提案）

| 角色 | 職責 | 現有對應 | 改進方向 |

|------|------|---------|---------|

| Planner | 任務選擇與分解 | Phase 1 內嵌邏輯 | 獨立為 todoist-planner.md |

| Generator | 任務執行 | Phase 2 子 Agent | 無需變更 |

| Evaluator | 品質驗證 | quality_gate.py | 增強 Tier 2 Heuristic |

| Assembler | 輸出組裝 | Phase 3 todoist-assemble | 無需變更 |

實施效益：

  • Planner 可獨立 A/B 測試（不影響執行層）
  • 計畫 JSON 提供完整 audit trail
  • 消除 OODA vs Harness 職責重疊

6. 2026 產業界最新實踐

6.1 十大關鍵洞察

  1. 基礎設施 > 模型升級

- 實驗證明：改變編輯格式帶來 10x 提升，模型升級僅 1.2x

- LangChain 在 TerminalBench 2.0 從低排名躍升至第 5 名，僅靠 Harness 優化

  1. 文檔是最強 Harness

- "If your AGENTS.md is vague, your agent output will be vague"

- 投資精確、機器可讀的文檔作為 Agent 的 Ground Truth

  1. Repository-Local Knowledge

- Agent 視角：不在 context 中的內容等同不存在

- Google Docs、Slack 討論不可訪問，僅 repo 內的版本化文件可見

  1. 40% Context Reset Rule

- 在 40% 使用率時重置 context，搭配結構化 handoff 文檔

- 工具過載問題：僅載入需要的工具（Tool Search 節省 85% token）

  1. Network Isolation for Evaluation

- 評估環境網路隔離現已成為必要要求，非可選項

  1. Max Iterations Hard Limit

- 嚴格的 max_iterations 計數器，超過閾值強制終止

- 防止 Agent 無限迴圈

  1. Repository Impact Map

- 讓 AI 在規劃前看實際程式碼

- 從符號分析建立影響地圖，讓下游任務基於真實程式碼

  1. Meta-Harness Optimization

- 將整個 Harness 視為聯合優化目標

- Proposer Agent 訪問所有先前 Harness 候選、分數、執行 trace

  1. Guides + Sensors 系統化

- 當前 Feedforward/Feedback 控制散落各處

- 需要工具化集中配置、同步、系統性推理

  1. Keep Quality Left

- 將檢查分佈於整個開發週期

- 早期發現成本最低（LSP、Linter）

- 後集成檢查（變異測試、性能測試）

- 持續監測（死碼、依賴掃描、SLO 漂移）

6.2 行業共識（2026）

"If 2025 was the year AI agents proved they could write code, 2026 is the year the industry learned that the agent isn't the hard part—the harness is."

工程師不再直接寫程式，而是設計讓 AI 可靠寫程式的系統。

7. 本專案優化路徑（v2.0 → v3.0）

7.1 v2.0 五大策略回顧

| 策略 | 核心內容 | 實施難度 | ROI |

|------|---------|---------|-----|

| 1. Feedforward 增強 | Good Examples Registry | 中 | 高（首次成功率 +13%） |

| 2. Context Management | Clean Reset + Handoff | 高 | 中（減少焦慮，提升長任務穩定性） |

| 3. Multi-Agent 精細化 | 獨立 Planner Agent | 中 | 中（可觀測性提升） |

| 4. Computational-First | 三層驗證金字塔 | 低 | 極高（LLM 成本 -70%） |

| 5. Feedback → Feedforward | Success Pattern Extractor | 中 | 高（減少重複錯誤） |

7.2 v3.0 新增優化（基於 2026 實踐）

#### 優化 A：Repository Impact Map（程式碼影響地圖）

問題：Agent 規劃時缺乏對現有程式碼的結構化理解。

方案：

  • 從符號分析建立 context/code-impact-map.json
  • 包含：函式依賴圖、模組耦合度、熱點檔案
  • Planner Agent 讀取此地圖後再規劃

工具：tools/code_impact_analyzer.py（基於 AST 分析）

#### 優化 B：Meta-Harness Tuner（自動調優引擎）

問題：Harness 參數（timeout、閾值、權重）當前手動調整。

方案：

  • Proposer Agent 訪問歷史執行 trace
  • 基於成功率/失敗模式自動建議參數調整
  • 人工審核後應用

工具：tools/harness_tuner.py --propose --apply

#### 優化 C：Unified Guides & Sensors Dashboard

問題：Feedforward/Feedback 控制散落 15+ 檔案。

方案：

  • 建立 config/guides-sensors-registry.yaml 統一註冊
  • 視覺化儀表板顯示每個控制的觸發率、有效性
  • 自動偵測「未觸發規則」（Dead Guards）

工具：tools/harness_dashboard.py（輸出 HTML 報告）

#### 優化 D：Network-Isolated Sandbox（隔離沙箱）

問題：當前評估環境無網路隔離。

方案：

  • 使用 Windows Sandbox 或 Docker 執行測試
  • 僅允許白名單 API（localhost:3000 知識庫、ntfy.sh）
  • 所有外部 HTTP 請求必須透過 api-cache 代理

實作：更新 config/autonomous-harness.yaml 增加 network_policy

12. 結論與下一步

關鍵要點

  1. Harness > Model：實驗證明基礎設施優化帶來的提升遠超模型升級
  2. 雙控制必要性：Feedforward 與 Feedback 必須並存且形成閉環
  3. 70/30 黃金比例：70% Computational + 30% Inferential 驗證最優
  4. 40% Reset Rule：Context 在 40% 時重置優於 80% 壓縮
  5. 專業化分工：Planner-Generator-Evaluator 三角色分離提升可觀測性

建議實施順序

  1. 立即啟動（本週）：Tier 2 Heuristic 層（快速勝利，成本 -70%）
  2. 第一季（3 個月）：Good Examples + Context Reset + Planner 獨立
  3. 第二季（3-6 個月）：Meta-Tuner + Harness Dashboard + Network Isolation
  4. 持續優化：每月審查 SRS 指標，季度更新 Good Examples

2026 核心轉變

工程師的角色不再是寫程式，而是設計讓 AI 可靠寫程式的系統。

Harness Engineering 是這個轉變的核心實踐。
