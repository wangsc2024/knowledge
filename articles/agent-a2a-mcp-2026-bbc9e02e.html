多 Agent 協作協定（A2A + MCP）：跨框架互通標準的落地實踐與安全邊界

研究日期：2026-04-17 | 來源：8 個（5 高可信 + 3 中可信）

核心結論

  1. 雙軌標準已確立：MCP（垂直工具整合）+ A2A（水平 Agent 協作）已成為業界共識，Google/Anthropic/OpenAI/Microsoft 共同背書
  2. A2A 生產就緒：一年內從 50 → 150+ 組織，Azure AI Foundry 與 Amazon Bedrock AgentCore 原生整合
  3. 安全是最大瓶頸：2/3 企業因安全疑慮不敢擴展 Agentic AI
  4. MCP 工具投毒是最高威脅：OWASP MCP Top 1，工具 metadata 層注入可在使用者授權前觸發
  5. 協定選擇有業務成本：任務完成時間差異最高 36%

架構對比

| 維度 | MCP | A2A |

|------|-----|-----|

| 整合方向 | 垂直（Agent ↔ 工具/資料）| 水平（Agent ↔ Agent）|

| 傳輸底層 | JSON-RPC 2.0（HTTP/Stdio/SSE）| HTTP + SSE + JSON-RPC 2.0 |

| 核心原語 | Tools、Resources、Prompts | Agent Card、Task、Message、Artifact |

| 比喻 | USB-C 接頭（即插即用）| 水平匯流排（Agent 協調）|

| 主要威脅 | 工具投毒、提示注入 | Card 偽造、任務注入 |

| 治理機構 | Linux Foundation AAIF | Google 主導，開放標準 |

安全架構：零信任 + RFC 8693 委派 Token


使用者 → Orchestrator (token: user@scope=all)
              ↓ Token Exchange (RFC 8693)
         HR Agent (token: subject=user, actor=hr_agent, scope=hr_only)
              ↓ Token Exchange
         HR API (token: subject=user, actor=hr_agent, scope=read_employee)

每一跳的 Token scope 只縮不擴，攻陷中間 Agent 無法橫向移動。

新興治理：IATP 動態信任評分（Microsoft 2026-04）

  • 0–1000 分動態信任評分，根據行為歷史即時計算
  • 五個行為層級對應不同授權範圍
  • Gated Agent 架構：跨層傳遞前輸出必須轉換為驗證過的安全資料型別

落地路線圖

Stage 1 MCP（工具呼叫）→ Stage 2 ACP（非同步多模態）→ Stage 3 A2A（組織內多 Agent）→ Stage 4 ANP（跨平台去中心化）

企業策略：MCP First（先整合工具），A2A Gradually（待多 Agent 需求出現後引入）

未解問題

跨組織 Agent 身份聯邦（corp-a.com ↔ corp-b.com 的委派信任）仍無統一標準，預計 NIST AI Agent Standards 2026-2027 落地後有解。
