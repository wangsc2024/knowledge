以數位憑證衍生遮罩機制實現身分導向資料隔離之深度研究報告

執行摘要

本報告的核心結論是：若僅以公開憑證或公鑰衍生 salt，雖能降低跨租戶與跨版本的關聯性，但不足以抵抗低熵個資的離線字典攻擊；實務上應以憑證指紋做身分綁定，再由 HSM 保護的租戶主密鑰透過 HKDF 派生遮罩金鑰，對需可逆欄位採 AES-GCM，對需不可逆分析鍵採 HMAC，並以 Zero Trust、mTLS、版本化 metadata、雙軌輪替與審計機制維持多租戶隔離與可追溯性。對自由文本，本地小模型可作為選配，但對結構化欄位並非必要。

摘要與導論

Abstract. 本研究探討一種「以數位憑證衍生遮罩機制」的設計框架，用於在 Zero Trust 架構下實現 Identity-aware Data Masking。研究問題不在於「是否加密全表」，而在於如何僅對個資與敏感資訊做選擇性遮罩，同時保留必要的分析、查詢與稽核能力。依 GDPR 與 EDPB 對 pseudonymisation 的定義，若資料仍可透過額外資訊重新歸屬到自然人，它仍屬個人資料；因此，設計重點應是把「可重新識別的額外資訊」與業務資料分離管理，並對該額外資訊施以更高等級的密碼與治理保護。NIST Zero Trust 亦明確要求存取決策以身分、裝置狀態與每次請求為核心，而非信任網段位置。

本報告採取的立場是：憑證在此設計中主要負責「身分綁定、租戶界定、版本治理與請求授權」；真正提供離線抗猜測能力者，必須是受保護的秘密金鑰材料，而非公開可得的憑證本身。 這一判斷來自 HKDF、HMAC 與 AEAD 的設計前提：HKDF 要從輸入 keying material 萃取並擴展出強金鑰；HMAC 的安全性建立在共享祕密；AES-GCM 的機密性與完整性則要求金鑰保密且 nonce/IV 不重複。若把公開憑證或公鑰直接當成唯一祕密來源，則最多只能提供 domain separation 或 anti-precomputation 的部分效果，無法對低熵欄位提供充分保護。

本研究範圍限定於 selective masking：只對個資與敏感資訊欄位施作遮罩，而不是對整個資料列或整個資料庫一體加密。台灣個資法明列姓名、出生年月日、身分證字號、護照號碼、指紋、聯絡方式、財務情況等為個人資料，並將病歷、醫療、基因、性生活、健康檢查及犯罪前科視為更高敏感等級之資料；GDPR 亦將可直接或間接辨識自然人的資訊納入個人資料範疇，並在序言中特別指出 online identifiers 可能導致再識別。因此，企業實作時應至少將「直接識別碼」「高風險準識別碼」「特殊敏感資料」區分為不同遮罩策略。

文獻探討

近十年相關研究大致可分成四條脈絡。第一條是 Zero Trust 與身分中心化存取控制。NIST SP 800-207 將 Policy Engine、Policy Administrator、Policy Enforcement Point、CDM 與 identity governance 定義為 ZTA 的核心組件，並強調每次請求都須依當前身分與裝置狀態做 least-privilege 決策。這一脈絡直接支持本研究的基本主張：遮罩不應只是資料庫靜態處理，而應是與身分、裝置、租戶、請求上下文共同運作的政策執行結果。

第二條是 pseudonymisation 與 re-identification 治理。GDPR Article 4(5) 將 pseudonymisation 定義為：在沒有額外資訊的情況下，資料不得再歸屬於特定資料主體，且該額外資訊必須分別保存並受到技術與組織措施保護。EDPB 2025 指南進一步指出，pseudonymised data 仍屬 personal data；pseudonymisation 可以降低風險、可允許一定程度分析與紀錄連結，但通常不足以單獨滿足所有合規要求，需要搭配存取控制、分離保存與授權管理。這些文獻意味著：本報告中的「可逆遮罩」與「不可逆遮罩」都不應被誤認為自動達成匿名化。

第三條是 密碼學衍生與加密資料庫的洩漏問題。HKDF 的 extract-then-expand 模式被設計用來從初始 keying material 派生強金鑰，且 RFC 5869 明確說明 salt 可提升不同用途之間的獨立性；NIST SP 800-56C 則將 shared secret 加上其他資訊衍生 keying material 系統化。另一方面，HMAC 被定義為以共享祕密為基礎的 keyed-hash；AES-GCM 則提供同時具備機密性與完整性的 AEAD。與此同時，近年的 encrypted database 與 searchable encryption 文獻反覆指出：一旦系統暴露 deterministic equality、frequency、order、volume 或 interoperability 類型的洩漏，攻擊者就可能以頻率分析、注入或 leakage-abuse 方式逐步重建明文分布。因此，任何需要「可比對性」的遮罩都必須被視為一種受控洩漏設計，而非零洩漏設計。

第四條是 敏感資訊自動辨識與去識別化。臨床文本去識別研究顯示，transformer 類模型在 i2b2/2014 PHI 任務上可達相當高的 precision/recall；混合式方法把深度學習、規則、上下文與 surrogate replacement 結合，也能在實務資料上取得穩定效果。新近研究顯示 GPT-4 在臨床去識別可有很高 precision，但 recall 未必同等突出；另有研究以 GPT-4 做 privacy-safe augmentation 以改善模型跨資料集泛化。另一方面，Microsoft Presidio 的官方文件也明確承認，自動化 PII 偵測不能保證找出所有敏感資訊，因此必須搭配額外防護。這些文獻支持一個務實結論：對結構化欄位，規則與欄位治理往往比 LLM 更可靠；對自由文本，最佳成本效益通常是規則 + 輕量模型/專用 NER 的混合架構，而不是預設把所有敏感內容送去大型雲端 LLM。

系統設計

本報告建議的系統遵循三個設計原則。第一，租戶必須是所有衍生流程的一級維度：所有 key/salt/AAD 均必須顯式含有 tenant_id，否則多租戶資料隔離將退化為應用邏輯假設。第二，證書只作為身分綁定與上下文域分割，不作為唯一秘密來源。第三，比較需求與顯示需求分離：同值比對、索引、群組等可比較需求應使用單獨的受控 blind index；真正對外展示的敏感值則應使用 randomized AEAD。這種分層方式同時回應 NIST Zero Trust 的 per-request access、GDPR/EDPB 的 additional information separation，以及 deterministic encryption 洩漏研究的警示。

憑證版本管理不宜直接把 leaf certificate 視為持久化資料的唯一根。RFC 8705 明確指出 certificate-bound token 在憑證更新後會失效；若把 leaf cert 直接當作持久化遮罩金鑰來源，則每次 renewal 都會破壞歷史資料可比對性，造成 join 斷裂與大規模重遮罩。因此，建議採 identity_root_id 穩定、cert_version 浮動 的雙層模型：leaf cert 用於授權與當前請求綁定，而持久化遮罩鍵由租戶根祕密與穩定主體識別推導。只有當身分根、租戶根或 key_version 改變時，才需要 remask/reindex。

建議的 metadata 至少應包含：tenant_id、identity_root_id、cert_thumbprint_sha256、cert_serial、cert_version、key_version、field_name、algorithm_id、mask_mode、record_id、random_nonce、blind_index_scope、schema_ver、aad_hash、created_at、rotated_from、status。其中 cert_thumbprint_sha256 可依 RFC 8705 的 x5t#S256 表示法建立一致的證書指紋欄位；key_version 與 cert_version 必須分離，才能在「證書輪替」與「密鑰輪替」之間做正交治理。

安全分析

本系統的威脅模型可依 STRIDE 映射到六類風險：Spoofing（偽造證書或冒用主體）、Tampering（竄改 metadata、AAD 或 blind index）、Repudiation（否認重識別與存取行為）、Information Disclosure（資料庫外洩、低熵欄位離線猜測、橫向租戶關聯）、Denial of Service（KMS/HSM 或 Policy Engine 成為瓶頸）、以及 Elevation of Privilege（藉由錯誤租戶政策、憑證映射或 wildcard 證書共享擴大存取）。NIST ZTA、RFC 5280、RFC 8705 與 NIST key-management guidance 的共同意涵是：PE/PA/PEP、信任錨、私鑰保護、版本治理與審計紀錄必須一體設計，否則遮罩只是在失控環境裡做局部加密。

在彩虹表與字典攻擊方面，若設計採用 per-record random_nonce、record_id、field_name 與 tenant_id 進入 final_salt，則攻擊者無法有效重用預先計算表；每一筆記錄都需要獨立重算。但這個結論只有在「token 仍包含祕密鍵材料」時才成立。若只有公開 cert-based salt，對低熵欄位的攻擊複雜度仍是線性的：T ≈ |D| × |R| / rate，其中 |D| 是候選明文域大小、|R| 是目標筆數。對數十、數百、數萬級候選域的欄位，這個成本在離線情境並不高，因此公開憑證鹽值不能宣稱具備強抗彩虹表能力，它只能降低大規模預先重用。

在關聯攻擊與頻率分析方面，任何 deterministic token 或 blind index 只要在同一 scope 下重複使用，便會暴露 equality pattern；一旦資料分布高度偏斜，攻擊者即可利用外部統計或背景知識做推測。這也是商用 encrypted database 與 deterministic encryption 文件反覆提醒的核心風險。因此，建議把 scope 限縮到「tenant + identity_root_id + field_name + key_version」，並將低熵欄位改採 bucketization、generalization 或只保留顯示遮罩，不輸出穩定 token。

在形式化安全性方面，可把不可逆模式視為：在 HMAC 為 PRF、K_mask 保密、final_salt 具唯一性且攻擊者不得操控 HKDF salt 的前提下，token 對未持有密鑰的攻擊者呈現計算上難以反推的 pseudorandom 值。HKDF 本身明確要求應把 salt 視為與 IKM 獨立，且不應由攻擊者任意操控；AES-GCM 則要求 nonce 唯一，且 96-bit IV 有利於互通與實作簡化。故本設計的安全邊界不只是演算法選型，更取決於：salt 是否被攻擊者輸入、nonce 是否重複、以及 KMS/HSM 是否隔離。

若以 96-bit random nonce 建立 final_salt 或 AEAD nonce，則隨機碰撞機率近似 n(n-1)/(2·2^96)。以本報告的工具計算，n = 10^9 時約為 6.31×10^-12，n = 10^12 時約為 6.31×10^-6；若 token 截斷為 128 位元，10^9 個 token 的隨機碰撞近似約 1.47×10^-21，若保留 256 位元則更低至 4.32×10^-60 等級。這些數值代表：只要 nonce 管理正確、scope 分割合理，偶然碰撞不會是主要風險；主要風險仍是 deterministic linkage 與密鑰治理失敗。

在自簽憑證可行性上，本報告的結論是「可行，但僅適合封閉生態系、內部 mTLS、服務對服務 API、裝置到平台或同組織零信任環境；不適合作為公網瀏覽器相容信任的唯一方案。」RFC 8705 明確允許在 mutual-TLS client authentication 中採用 self-signed certificates，也允許 PKI 模式；RFC 5280 則指出 self-signed certificate 可作為 certification path 的起點與 trust anchor。可是，一旦場景轉為公網 user-facing 服務，OWASP 建議應使用受作業系統與瀏覽器信任的 CA，否則信任分發、撤銷、告警與使用者體驗都會變得脆弱。

自簽憑證與 CA-signed 的主要差異，不是「密碼學强度」而是「信任分發與治理邊界」。兩者都可使用強金鑰、SHA-256 指紋與 mTLS；真正不同的是：CA-signed 依賴公眾 trust store 與 CA/B Forum 生態，較適合 public-facing；self-signed 則把信任責任回收給組織自己。這會降低外部簽發成本，但提高內部指紋白名單、trust anchor 部署、撤銷、短生命週期、自動化更新與稽核的要求。若採自簽方案，建議至少實施 fingerprint allowlist、私有根/中繼 CA 置於 HSM、mTLS、短期 leaf cert、自動化輪替、內部 CT-like append-only log、以及憑證與資料存取事件的聯合審計。Certificate Transparency（RFC 9162）雖主要服務公開憑證生態，但其 append-only log 精神很適合在內網實作「私有透明度日誌」。

實驗設計與結果討論

若要驗證本設計，資料集應同時涵蓋結構化欄位與自由文本。結構化部分可用合成資料產生高、低熵欄位分布；自由文本部分可選用 i2b2/n2c2 2014 de-identification、MIMIC-IV-Note、Enron Email，以及完全合成的 PII 資料集。需注意，i2b2/n2c2 與 MIMIC 類資料通常需要 DUA/credentialed access，而 PhysioNet 也明確提醒，MIMIC 等 credentialed data 的使用條款不允許將資料送往第三方 API 或線上平台處理；這一點直接影響是否需要本地部署 NLP/LLM。

表中樣本量是工程驗證起點；正式研究可再依預期 effect size 做 power analysis。

對NER 與動態遮罩，本報告的判斷是：本地 LLM 不是預設必要條件，而是「自由文本占比高、語言多樣、欄位治理差、且不能把資料送到外部 API」時的選配能力。 文獻顯示 transformer 與混合式系統已能在去識別上取得很高的實務表現；GPT-4 類模型可帶來高 precision，但 recall、成本、治理與資料外送風險未必最優。Presidio 官方也警示自動偵測不保證完整，因此制度面仍需多層防護。綜合之下，對大多數政府或企業內部系統，建議採 「結構化欄位：schema/rule-first；半結構化文本：regex + dictionary + small NER；高變異自由文本：再評估 local small LLM」 的階梯式架構。

從資源成本看，是否部署本地 LLM 可先以模型尺寸估算。官方模型卡顯示 Llama 3.1 提供 8B、70B、405B 等級，Qwen2.5 亦有 0.5B 至 72B 多種尺寸；若以 4-bit 量化粗估，8B 權重本體約為 4 GB，70B 約為 35 GB，實際部署尚需額外記憶體與 KV cache，因此 7B–8B 級模型通常較適合單機/單 GPU 場景，70B 級則顯著提高硬體門檻。若需要提升本地 serving 效率，可考慮使用 vLLM 等高吞吐引擎，其研究顯示在同等延遲下可把吞吐提高約 2–4 倍。換言之：若目的僅是 PII/NER，先用小模型或專用模型，比直接上大模型更符合成本與治理原則。（本段中的記憶體數字屬依參數量與量化位寬計算之工程估算。）

就預期研究結果而言，文獻與標準共同支持以下假設。其一，公開憑證 only 的 salt 方案在多租戶 anti-correlation 上會有效，但在低熵欄位抗離線猜測上會顯著劣於 HSM-secret-enhanced 的 HKDF+HMAC/AES-GCM 方案。其二，若系統必須保留回復原值能力，AES-GCM + blind index 大機率會成為最佳折衷。其三，若系統直接把 leaf certificate 做為持久化遮罩根，則憑證輪替必然造成 join 破裂；若採本報告的 identity_root_id + tenant_root_secret_v + cert_version 分層，則可於不重遮罩歷史資料的前提下支援 leaf renewal 與 dual-read。其四，對文本遮罩，local LLM 只在「高變異、低結構、受 DUA/主權限制」情境下具明顯邊際效益。

法規與治理

從 GDPR 視角來看，本設計最關鍵的法遵點有三個。第一，可逆遮罩與可重新識別的不可逆 token，都通常仍屬 personal data；因為控制者持有 additional information 或可藉由其他資料完成 attribution。第二，Article 25 要求 data protection by design and by default，Article 32 要求依風險採取適當技術與組織措施，包括 pseudonymisation 與 encryption。第三，EDPB 2025 明言 pseudonymisation 可降低風險並協助資料最小化，但通常不會單獨構成充分保護。因此，企業在法律文件上不宜把此類機制表述為「匿名化完成」，而應更精確地表述為「有條件可重新識別之 pseudonymisation / reversible protection」。

在台灣法制下，個資法已將一般個資與特殊敏感資料作出明確區分，並要求機關指定專責或專人辦理個資安全維護，且在資料遭竊取、竄改、毀損、滅失或洩漏時通知當事人並採取即時有效應變措施。這表示本設計若導入可逆遮罩，還必須同時導入：重識別流程申請、最小必要揭露、事件留痕、批次導出控管、密鑰/證書異常通報、與依角色區分的 access justification。換句話說，合規的焦點不只在演算法，還在誰可以解、何時可解、為何而解、如何留痕。

若組織涉及境外雲端 NLP 或 LLM，則應做比一般 masking 更嚴格的治理判斷。對像 MIMIC 這類 credentialed dataset，資料使用條款已明確限制不可把資料送交第三方線上平台；即便在一般企業資料情境，若模型供應商可接觸原始文本，也涉及處理者委外、跨境傳輸、存取審計與二次利用界線。故本報告建議：凡是含敏感個資的自由文本，若需要模型參與偵測，優先考慮本地或專屬 VPC/on-prem；若只是結構化欄位，則完全沒有使用大型 LLM 的必要。

治理面最務實的控制組合是：欄位分級、證書/密鑰分層、append-only 審計、兩人覆核重識別、短生命週期 leaf cert、輪替 runbook、定期 red-team linkage test，以及將 masking policy 納入 SIEM/CDM 的持續監測。 這樣的控制組合與 NIST ZTA 對 continuous diagnostics、least privilege、control plane isolation 的要求相符，也與個資法和 GDPR 對風險導向措施的精神一致。

結論

綜合標準、法規與近年文獻，本報告提出的答案可以濃縮為一句話：「憑證導向」值得做，但不可把公開憑證誤當成祕密；真正可靠的 Identity-aware Data Masking，應是『憑證綁定身分與租戶邊界』＋『HSM 保護的祕密派生資料遮罩金鑰』＋『可比較性與顯示遮罩分離』＋『Zero Trust 與版本治理』。 因此，若您的目標同時涵蓋 selective masking、Zero Trust、多租戶與輪替，本報告建議以 HKDF + HMAC/AES-GCM + blind index + x5t#S256 metadata + identity_root_id 雙層版本化 為主體設計；自簽憑證在封閉內網與 mTLS 場景下是可行的，但需以指紋白名單、HSM、短生命週期與私有透明度日誌補強；本地 LLM 則應被定位為自由文本去識別的選配能力，而不是結構化個資遮罩的前提條件。
