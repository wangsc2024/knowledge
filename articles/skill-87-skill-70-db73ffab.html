Skill 觸發詞覆蓋率全面盤點與優化方案 — 87 個 Skill 觸發詞缺口分析、四維評估模型與 70% 覆蓋率達成路徑

Skill 觸發詞覆蓋率全面盤點與優化方案

概述

本研究針對 daily-digest-prompt 專案的 87 個 Skill，系統性分析其觸發詞（triggers）的覆蓋率現況，識別缺口並提出優化方案。目標是將觸發詞覆蓋率從目前約 52% 提升至 70% 以上。

現況統計

| 指標 | 數值 |

|------|------|

| Skill 總數 | 87 |

| 觸發詞總數 | 803 |

| 平均觸發詞數 | 9.2 個/Skill |

| 低於最低標準（<7） | 3 個（3.4%） |

| 恰好達標（=7） | 27 個（31%） |

| 良好覆蓋（≥10） | 35 個（40.2%） |

| 中等覆蓋（8-9） | 22 個（25.3%） |

四維覆蓋率評估模型

維度定義

  1. 數量充足度（權重 25%）：觸發詞數量是否 ≥10 個
  2. 中英雙語覆蓋（權重 25%）：是否同時包含中文與英文觸發詞
  3. 同義詞多樣性（權重 25%）：同一概念是否有多種表達方式（口語、書面、縮寫）
  4. 動作導向短語（權重 25%）：是否包含動詞開頭的使用情境（如「幫我查」「我想知道」）

現況覆蓋率計算

| 類別 | Skills 數 | 佔比 | 加權覆蓋率 |

|------|----------|------|----------|

| A 級（4 維全達） | 12 | 13.8% | 100% |

| B 級（3 維達標） | 23 | 26.4% | 75% |

| C 級（2 維達標） | 25 | 28.7% | 50% |

| D 級（1 維達標） | 24 | 27.6% | 25% |

| F 級（0 維達標） | 3 | 3.4% | 0% |

加權平均覆蓋率 = (12×100 + 23×75 + 25×50 + 24×25 + 3×0) / 87 = 52.0%

重點缺口分析

缺口一：低於最低標準的 3 個 Skill

| Skill | 觸發詞數 | 缺失類型 |

|-------|---------|----------|

| add-podcast-task | 6 | 缺英文同義詞、缺動作短語 |

| cache-miss-feedback-loop | 6 | 缺口語化表達、缺動作短語 |

| cache-miss-reason-observer | 6 | 與 cache-miss-feedback-loop 高度重疊 |

缺口二：恰好 7 個觸發詞的 27 個 Skill（最大改善空間）

這 27 個 Skill 佔總量 31%，是提升覆蓋率的關鍵槓桿點。常見問題：

  1. constraint 系列（4 個）：constraint-evolve、constraint-embedding-verifier、constraint-pipeline-automator — 缺少「規則」「政策」「約束管理」等口語化表達
  2. execution 系列（2 個）：execution-failure-root-cause、execution-metrics-reconstructor — 缺少「為什麼失敗」「成功率多少」等問句形式
  3. Skill 治理系列（4 個）：skill-forge、skill-registry-sync、skill-test-scaffolder、skill-usage-coverage-diagnostic — 缺少動詞形式「建立 Skill」「同步索引」
  4. 品質/監控系列（5 個）：draft-verify-reflector、pre-mortem、pre-flight-check、prompt-lint、todo-scanner — 缺少「檢查品質」「程式碼掃描」等泛用表達
  5. 快取系列（2 個）：cache-optimizer、cache-swr-manager — 與其他快取 Skill 缺區分度
  6. 系統運維系列（6 個）：entropy-health-monitor、otel-log-standardizer、phase-snapshot-recovery、scheduler-heartbeat-monitor、self-heal、state-lifecycle-manager — 缺少「系統不穩」「排程掛了」等口語觸發

缺口三：五大系統性缺失模式

| # | 缺失模式 | 影響 Skills 數 | 範例 |

|---|---------|--------------|------|

| 1 | 缺少口語化表達 | ~45 個 | 「幫我看」「怎麼了」「出問題了」|

| 2 | 缺少問句形式 | ~50 個 | 「為什麼失敗」「怎麼修」「哪裡壞了」|

| 3 | 缺少中文縮寫/別稱 | ~20 個 | 「KB」→「知識庫」已有，但「TTL」→缺「過期時間」|

| 4 | 缺少英文全稱 | ~15 個 | 只有縮寫沒有全稱（如 SWR 缺 stale-while-revalidate 已有，但 DRR 缺 Deficit Round Robin）|

| 5 | 跨 Skill 區分度不足 | ~8 對 | cache-optimizer vs cache-swr-manager vs adaptive-ttl-optimizer 三者觸發詞高度重疊 |

優化方案：達成 70% 覆蓋率的路徑

策略一：補齊低於標準的 3 個 Skill（+3% 覆蓋率）

add-podcast-task 新增：

  • podcast 排程、podcast 自動、create podcast、建立 podcast、錄製任務

cache-miss-feedback-loop 新增：

  • 快取為什麼 miss、miss 回饋、快取自動調整、cache feedback、快取學習

cache-miss-reason-observer 新增：

  • 快取為什麼沒命中、cache miss 觀察、快取失效原因、miss 分析報告

策略二：將 27 個 =7 Skill 提升至 ≥10（+15% 覆蓋率）

每個 Skill 新增 3-5 個觸發詞，優先補充：

| 補充類別 | 範例新增觸發詞 |

|---------|-------------|

| 口語化 | 「幫我檢查」「看一下」「怎麼修」 |

| 問句式 | 「為什麼」「哪裡出錯」「怎麼辦」 |

| 動詞式 | 「執行」「分析」「修復」「建立」 |

| 場景式 | 「系統慢」「排程掛了」「品質下降」 |

#### 具體建議（優先 15 個高影響力 Skill）

  1. self-heal +4：自動修復、故障排除、系統壞了、自動恢復
  2. pre-mortem +4：風險預警、會不會失敗、任務風險、潛在問題
  3. pre-flight-check +4：啟動前檢查、依賴檢查、環境就緒、準備好了嗎
  4. skill-forge +4：建立新 Skill、skill 自動產生、能力補充、forge
  5. constraint-evolve +4：約束升級、規則更新、約束管理、政策演化
  6. execution-failure-root-cause +4：為什麼失敗、錯在哪裡、失敗診斷、出錯原因
  7. entropy-health-monitor +4：系統複雜度、配置膨脹、系統亂了、熵
  8. todo-scanner +4：程式碼 TODO、待修清單、技術債掃描、HACK 標記
  9. git-smart-commit +4：分批提交、自動 commit 訊息、git 分群、提交分類
  10. prompt-lint +4：prompt 品質、模板健康、prompt 太長、模板膨脹檢查
  11. draft-verify-reflector +4：草稿驗證、品質檢查流程、兩階段品質、品質審核
  12. phase-snapshot-recovery +4：中斷恢復、斷點續跑、管線恢復、Phase 斷了
  13. scheduler-heartbeat-monitor +4：排程還活著嗎、心跳檢查、排程斷線、排程監控
  14. state-lifecycle-manager +4：state 太多了、清理狀態檔、狀態膨脹、state 過期
  15. otel-log-standardizer +4：日誌轉格式、OTel、OTLP 匯出、日誌正規化

策略三：系統性補充五大缺失模式（+5% 覆蓋率）

  1. 口語化觸發詞規範：每個 Skill 至少 1 個口語化觸發詞（「幫我...」「看一下...」）
  2. 問句式觸發詞規範：診斷類 Skill 必須包含 1 個問句（「為什麼...」「哪裡...」）
  3. 去重區分標記：高重疊 Skill 對新增區分性觸發詞，如 cache-optimizer 專注「分析」、cache-swr-manager 專注「背景更新」
  4. 中文全稱補齊：所有英文縮寫必須有中文全稱對應
  5. 場景觸發詞：運維類 Skill 新增故障場景描述（「系統慢」「排程掛了」）

預期效果

| 策略 | 投入 | 預期覆蓋率提升 |

|------|------|---------------|

| 策略一：補齊 3 個不達標 | 15 個新觸發詞 | +3% → 55% |

| 策略二：提升 27 個邊界 Skill | ~100 個新觸發詞 | +15% → 70% |

| 策略三：系統性補充 | ~50 個新觸發詞 | +5% → 75% |

| 合計 | ~165 個新觸發詞 | 75%（超越目標） |

跨 Skill 觸發詞衝突熱點

| 衝突組 | 共享觸發詞 | 建議區分策略 |

|--------|----------|------------|

| cache-optimizer / cache-swr-manager / adaptive-ttl-optimizer | 快取、TTL、命中率 | optimizer→「分析效率」、swr→「背景刷新」、adaptive→「自動調整 TTL」 |

| execution-journal / execution-metrics-reconstructor | 執行追蹤、日誌 | journal→「Skill chain」、reconstructor→「指標計算」 |

| task-manager / auto-task-creator / todoist-task-creator | 新增任務 | manager→「管理」、auto-creator→「自動任務」、todoist-creator→「Todoist 排程」 |

| research-saturation-analyzer / kb-research-strategist | 研究、飽和 | saturation→「收穫遞減」、strategist→「深化路線」 |

| cache-miss-reason-observer / cache-miss-feedback-loop | cache miss、miss 原因 | observer→「觀察診斷」、loop→「自動修正」 |

實施優先順序

  1. P0 — 立即修復：3 個不達標 Skill（add-podcast-task、cache-miss-feedback-loop、cache-miss-reason-observer）
  2. P1 — 高影響力：15 個高頻使用 Skill 的觸發詞擴充（上表）
  3. P2 — 系統性改善：其餘 12 個 =7 Skill 的觸發詞補充
  4. P3 — 衝突消歧：5 組跨 Skill 觸發詞衝突的區分策略

來源與方法

  • 資料來源：直接讀取 87 個 skills/*/SKILL.md 的 frontmatter triggers 陣列
  • 評估方法：四維覆蓋率模型（數量、中英雙語、同義詞多樣性、動作導向）
