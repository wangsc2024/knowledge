二、核心架構：六大 AI 基元（Primitives）

Mastra 的設計哲學是提供一組可組合的 AI 基元，開發者按需組合：

2.1 Agents（自主代理）

  • 使用 LLM + Tools 解決開放式任務
  • 內建推理、工具選擇、迭代執行循環
  • 支援可選停止條件（stopping condition）
  • 多 Agent 工作流支援

2.2 Workflows（工作流引擎）

  • Graph-Based 執行引擎：非線性 DAG 編排
  • 直覺語法：.then()、.branch()、.parallel()
  • Human-in-the-Loop：支援 suspend() 暫停 + 等待人類審批後恢復
  • 支援條件分支、並行執行、錯誤處理

2.3 Tools（工具整合）

  • 函式即工具（Function-as-Tool）
  • MCP 伺服器支援：可將 Agent、Tools 等暴露為 MCP 資源
  • 自訂工具定義與驗證

2.4 Memory（記憶系統）— Observational Memory

（核心創新，見第三節詳述）

2.5 RAG（檢索增強生成）

  • 內建 RAG 管線
  • 可與 Observational Memory 並用（OM 管理 Agent 經驗，RAG 管理外部知識）

2.6 Evals（品質評估）

  • 內建評估框架
  • 支援自動化品質和準確度測量
  • 可觀測性（Observability）與追蹤（Tracing）整合

附加：Mastra Studio

  • 互動式開發與測試 UI
  • 即時預覽 Agent 行為
  • 模板系統（Chatbot、PDF 互動、資料庫查詢、程式碼審查）

三、Observational Memory（OM）— 核心技術創新深度剖析

3.1 三層記憶架構

OM 採用漸進式壓縮的三層結構：


┌───────────────────────────────────────┐
│  Layer 1: Raw Messages（原始訊息）    │
│  → 最近的完整對話紀錄                 │
│  → 超過 30,000 tokens 觸發壓縮       │
├───────────────────────────────────────┤
│  Layer 2: Observations（觀察日誌）     │
│  → Observer Agent 壓縮產出            │
│  → 帶日期的濃縮觀察筆記               │
│  → 超過 40,000 tokens 觸發反思       │
├───────────────────────────────────────┤
│  Layer 3: Reflections（反思摘要）      │
│  → Reflector Agent 合併相關觀察       │
│  → 最高層級的模式識別與整合           │
└───────────────────────────────────────┘

3.2 雙 Agent 協作機制

Observer（觀察者）：

  • 觸發條件：未觀察訊息超過 30,000 tokens
  • 行為：壓縮對話為濃縮觀察筆記
  • 特殊功能：追蹤「當前任務」和「建議回應」，確保 Agent 能續接中斷處
  • 壓縮後丟棄原始訊息

Reflector（反思者）：

  • 觸發條件：觀察日誌超過 40,000 tokens
  • 行為：合併相關觀察、移除已過時資訊
  • 保留事件驅動的結構（event-based structure）

3.3 非同步緩衝系統（Async Buffering）

| 參數 | 預設值 | 說明 |

|------|--------|------|

| bufferTokens | 0.2 | 每 20% 閾值觸發背景預計算（~6,000 tokens）|

| bufferActivation | 0.8 | 啟動時保留 20% 閾值的訊息 |

| blockAfter | 1.2 | 安全閥：1.2 倍閾值時強制同步觀察 |

| reflection.bufferActivation | 0.5 | 50% 觀察閾值時開始背景反思 |

關鍵設計：背景預計算確保壓縮觸發時零延遲，不會造成對話中斷。

3.4 壓縮效能

| 工作負載類型 | 壓縮比 |

|-------------|--------|

| 純文字對話 | 3–6x |

| 工具密集型 Agent（如 Playwright）| 5–40x |

實例：50,000+ token 的 Playwright 互動 → 壓縮至數百 token 的觀察筆記。

3.5 Scope 選項

  • Thread Scope（預設）：每個對話線程獨立觀察，最適合長期 Agentic 場景
  • Resource Scope（實驗性）：跨對話線程共享觀察，實現跨會話記憶

四、基準測試：OM vs RAG

4.1 LongMemEval 基準

LongMemEval 是專門測試 AI 系統長期記憶能力的基準，包含 500 個問題，每個附帶約 50 段對話。

| 模型 | OM 準確率 | RAG 準確率 | 差距 |

|------|-----------|-----------|------|

| GPT-5-mini | 94.87% | — | — |

| GPT-4o | 84.23% | 80.05% | +4.18% |

4.2 Token 成本降低

  • 透過 provider prompt caching，成本降低最高 10 倍
  • 平均有效上下文窗口維持在 ~30,000 tokens
  • 穩定的觀察區塊創造可快取的 prompt 前綴

4.3 OM vs RAG 定位差異

| 維度 | Observational Memory | RAG |

|------|---------------------|-----|

| 資料來源 | Agent 自身歷史經驗 | 外部語料庫 |

| 檢索方式 | 直接讀取壓縮日誌（無檢索階段）| 每輪動態向量檢索 |

| Prompt 快取 | 穩定前綴，高快取命中率 | 每輪變動，難以快取 |

| 適用場景 | 長期工具密集型 Agent | 開放式知識探索 |

| 關係 | 互補非互斥 | 互補非互斥 |

五、企業早期應用模式

Mastra 早期客戶展示的三類典型場景：

  1. CMS/SaaS 內嵌 Agent：記住用戶偏好、報表格式、內容類型決策
  2. AI SRE/運維：追蹤已調查的告警、採取的行動、決策依據
  3. 文件處理工作流：保留歷史處理決策和邊界案例，減少重複用戶互動

六、對 Daily-Digest-Prompt 系統的五大借鑑啟示

啟示 1：跨次記憶的漸進壓縮

現狀：digest-memory.json 僅記錄連續天數、完成率等統計指標。

借鑑：引入 OM 的三層壓縮思路 — 近期詳細紀錄 → 中期觀察摘要 → 長期模式反思。可在 digest-memory.json 加入 observations 欄位，定期壓縮歷史行為模式。

啟示 2：非同步緩衝的預計算模式

現狀：Phase 0 已有 PS 預計算機制（cache/status.json）。

借鑑：Mastra 的 bufferTokens=0.2 漸進預計算思路可延伸 — 不僅在 Phase 0 一次性計算，而是在每次 Todoist 執行後增量更新快取狀態，降低 Phase 0 冷啟動時間。

啟示 3：Human-in-the-Loop 的 Suspend/Resume

現狀：Todoist 團隊模式的 Phase 3 組裝是單向（Phase 2 完成 → Phase 3 組裝）。

借鑑：Mastra 的 suspend() + 人類審批模式可用於高風險自動任務 — 當 Agent 判斷需要人類確認時（如大量修改配置），暫停執行並透過 ntfy 發送確認請求。

啟示 4：Graph-Based Workflow 編排

現狀：團隊模式用 PowerShell 腳本線性編排 Phase 0→1→2→3。

借鑑：Mastra 的 .then()/.branch()/.parallel() 語法啟示 — 未來若團隊模式複雜度增加，可考慮引入 DAG 編排引擎，支援條件分支（如根據快取命中情況跳過某些 Phase）。

啟示 5：Agent 經驗 vs 外部知識的雙軌記憶

現狀：知識庫（KB）用於外部知識，digest-memory 用於運行統計。

借鑑：明確區分 Agent 自身經驗記憶（哪些任務做得好、哪些失敗模式反覆出現）與外部知識記憶（研究成果、技術文件），建立獨立的 Agent 行為觀察日誌，作為系統自愈（self-heal）的輸入源。
