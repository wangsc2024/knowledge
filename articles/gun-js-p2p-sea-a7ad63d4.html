Gun.js 最佳實踐完整指南

核心概念

Gun.js 是開源去中心化、即時同步、離線優先的圖形資料庫（~9KB gzipped, 20M+ ops/sec）。五層架構：HAM（衝突解決）→ Graph（記憶體圖）→ SEA（加密）→ RAD（儲存）→ DAM（網路）。

安裝與初始化

  • npm install gun，Node.js 需 @peculiar/webcrypto
  • 生產環境配置多個 relay peer，禁用不需要的儲存後端
  • React 中建議 localStorage: false 改用 IndexedDB

資料模型設計最佳實踐

  1. 絕不使用原生陣列 → 用 gun.set() 建立集合
  2. 使用確定性 ID 作為 soul → 避免深層巢狀產生隨機 UUID
  3. 扁平化資料結構 → 關聯資料分散為獨立節點透過引用連結
  4. 建立索引節點 → 為常用查詢建立專用索引避免全圖遍歷
  5. 使用時間前綴作為鍵 → 支援 RAD 字典序範圍查詢

同步機制（DAM）

  • P2P WebSocket 架構，至少部署 2 個 relay peer
  • 必須 HTTPS（WebCrypto 需要安全上下文）
  • Gun 自動重連不可靠，需自行實作心跳 + 手動重連
  • DAM 透過鄰居 peer ID 和 fast hash 去重，減少冗餘訊息

持久化（RAD）

  • Node.js 預設 RAD（基數樹 fs），瀏覽器用 localStorage/IndexedDB
  • RAD 參數：chunk（分塊大小）、until（批次間隔 250ms）、batch（最大批次 10K）
  • 支援自訂 adapter（只需 put/get 兩個方法）
  • 生產環境建議搭配 S3 等外部儲存

安全性（SEA）

  • SEA.pair()：ECDSA + ECDH 金鑰對
  • SEA.sign/verify：ECDSA-SHA256 簽章/驗簽
  • SEA.encrypt/decrypt：AES-GCM 加解密
  • SEA.secret()：ECDH 共享秘密（端對端加密）
  • SEA.certify()：路徑級別授權憑證
  • SEA.work()：PBKDF2 工作量證明
  • 生產環境禁止 SEA.throw = true
  • 錯誤檢查用 === undefined 而非 try-catch

效能優化

  1. 用 .once() 做一次性讀取，避免 .on() 的持續訂閱開銷
  2. 及時 .off() 取消訂閱防止洩漏
  3. RAD 字典序分頁避免載入整個集合
  4. 批次寫入用 partial update 合併
  5. Web Worker 隔離 Gun 到背景線程

HAM 衝突解決

  • 基於 state vector 的 CRDT：state 較新者勝出
  • 同 state 衝突：JSON.stringify 字典序較大者勝出
  • state > 系統時鐘：延遲處理（防止惡意未來時間戳攻擊）
  • 保證強最終一致性（SEC），不保證線性化
  • 不適合 Double Spending 場景

常見陷阱

  1. 不支援原生陣列
  2. 無法真正刪除資料（只能 tombstone: put(null)）
  3. 根層級不能存放基元值
  4. 深層巢狀可能產生隨機 UUID
  5. .on() 可能回傳 partial 資料
  6. localStorage 5MB 限制
  7. 自動重連不可靠

適用場景

  • 即時聊天（5星）、協作白板（4星）、IoT（4星）
  • 社群媒體（3星，需自行設計 ACL）
  • 電商（2星）、銀行交易（1星，缺乏強一致性）
