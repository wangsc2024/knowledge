版本: 1.0.0

工作流結構

前置條件（3 項）

  1. cache-policy.yaml 檔案存在
  2. cache-policy.yaml 已修改（git diff 非空）
  3. context/cache-baseline.json 存在（記錄調整前的基準線）

驗證步驟（6 步驟）

  1. baseline_capture: 捕捉基準線（記錄 TTL 調整前的命中率、API 呼叫次數、stale 次數）
  2. ttl_adjustment: TTL 調整（依據 cache-policy.yaml 的 adaptive_ttl 規則調整 TTL）
  3. observation_period: 觀測期（執行 3-7 天，收集新的命中率數據）
  4. hit_rate_comparison: 命中率比對（對比調整前後的 hit_rate、stale_rate、miss_rate）
  5. per_source_validation: 各來源驗證（檢查 todoist/news/hackernews/knowledge/gmail 各自的命中率）
  6. report_generation: 產出驗證報告（彙整驗證結果並產出 Markdown 報告）

成功條件

  • 絕對達標: hit_rate >= 40%（目標值，來自 cache-policy.yaml hit_rate_targets.overall）
  • 相對提升: hit_rate 相對提升 >= 20%（容許基準線低於 40% 但有顯著改善）

回退條件

  • hit_rate 下降 > 10%（命中率惡化超過 10%）
  • stale_rate > 90%（過期快取比例過高）
  • miss_rate > 30%（未命中比例過高）

回退程序

  1. git checkout HEAD~1 -- config/cache-policy.yaml（回退至上一版本）
  2. 記錄回退原因至 context/cache-ttl-rollback-history.json
  3. 發送 ntfy 通知（priority=4）

整合點

對齊檔案

  • config/cache-policy.yaml（TTL 來源與 adaptive_ttl 規則）
  • config/benchmark.yaml（hit_rate_targets 定義）
  • logs/structured/post_tool_logger.jsonl（命中率數據來源）

新建檔案

  • context/cache-baseline.json（基準線記錄）
  • context/cache-comparison-report.json（驗證結果）
  • context/cache-ttl-rollback-history.json（回退歷史）
  • docs/cache-ttl-validation-report-YYYYMMDD.md（人類可讀報告）

預期成果
