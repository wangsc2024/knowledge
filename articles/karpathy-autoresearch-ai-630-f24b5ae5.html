一、專案概述

autoresearch（karpathy/autoresearch）由前 Tesla AI 總監、OpenAI 共同創辦人 Andrej Karpathy 於 2026 年 3 月開源，是一個讓 AI Agent 在單 GPU 上自主進行 ML 實驗的框架。專案在發布後 3 天內獲得 22,983 Stars，登上 HN 最高討論分（198 分），Karpathy 的推文更在 2 天內獲得 860 萬次觀看 [GitHub Trending Weekly, 2026-03-11]。

二、三原語架構設計（Three Primitives）✅ 已驗證

autoresearch 的核心創新不在 AI 模型本身，而在於將自主實驗迴圈提煉為三個可泛化的原語（primitives）：

2.1 可編輯資產（Editable Asset）

Agent 僅被允許修改 一個檔案（train.py），其餘皆為唯讀。這個約束帶來三大好處 [GitHub README; Ken Huang Substack, 2026-03]：

  • 搜尋空間可控：Agent 無法修改資料前處理或評估邏輯，確保實驗可比較
  • 每次變更都是可審閱的 diff：人類可在事後逐一檢視 Agent 的「假說」
  • 防止不可逆的環境破壞：prepare.py 的分詞器和資料載入器始終穩定

2.2 標量指標（Scalar Metric）

使用 val_bpb（validation bits per byte） 作為唯一評判標準，越低越好。關鍵設計 [GitHub README; program.md]：

  • 與詞表大小無關：不同 vocab_size 的架構變更可公平比較
  • 單一數字決策：簡化 Agent 判斷為「改善或退回」二元選擇
  • 自動記錄：每次實驗結果寫入 results.tsv（commit hash / val_bpb / peak memory / keep or discard）

2.3 時間限制循環（Time-Boxed Cycle）

每次訓練固定 5 分鐘，超過 10 分鐘自動終止。此設計 [GitHub README; VentureBeat, 2026-03]：

  • 平台無關可比性：無論 H100 或 RTX 4090，每次實驗的「時間預算」相同
  • 自動排程：~12 次/小時，一晚可跑 ~100 次
  • 資源可預測：VRAM 和電力消耗可精確估算

三、三檔案極簡架構

| 檔案 | 角色 | Agent 可修改？ |

|------|------|---------------|

| prepare.py | 資料集下載、BPE 分詞器訓練（預設 8,192 tokens）、DataLoader、評估工具 | 否（唯讀） |

| train.py | GPT 模型定義、Muon + AdamW 優化器、訓練迴圈（~630 行） | 是（唯一可編輯） |

| program.md | Agent 的行為指令（人類編寫，定義目標、規則、記錄格式） | 否（人類編輯） |

Agent 可調參數（train.py 內）

  • DEPTH：模型層數（預設 8，可降至 4）
  • vocab_size：詞表大小（預設 8,192，可降至 256 做 byte-level）
  • MAX_SEQ_LEN：序列長度
  • WINDOW_PATTERN：注意力機制模式（預設 "SSSL"，可簡化為 "L"）
  • 學習率：實驗中發現從 0.5 提升至 4.7 有顯著改善
  • DEVICE_BATCH_SIZE / TOTAL_BATCH_SIZE：必須為 2 的冪次

四、實驗迴圈機制（The Karpathy Loop）✅ 已驗證


while True:  # 「永不停止」指令
    1. Agent 閱讀 train.py 原始碼
    2. 形成改善假說（如修改學習率、重構注意力機制）
    3. 修改 train.py
    4. git commit（記錄假說）
    5. uv run train.py > run.log 2>&1（固定 5 分鐘）
    6. grep "^val_bpb:" run.log → 提取指標
    7. 記錄至 results.tsv
    8. val_bpb 改善 → keep（保留修改）
       val_bpb 退步 → discard（回滾至上一版本）
    9. 回到步驟 1

program.md 的關鍵指令：「NEVER STOP」 — Agent 須無限自主循環直到人類手動中斷 [program.md]。

六、設計哲學與可泛化性分析

6.1 「非 ML 專屬」的通用模式

Karpathy 明確指出三原語的組合「與 GPU 和神經網路無關——它適用於任何你能評分的事物」[VentureBeat, 2026-03]。關鍵洞見：

| 領域 | 可編輯資產 | 標量指標 | 時間限制 |

|------|-----------|---------|----------|

| ML 訓練 | train.py | val_bpb | 5 分鐘 |

| 行銷實驗 | 廣告文案/版面 | 轉換率 | 24 小時 A/B 測試 |

| 程式碼優化 | 效能關鍵函式 | 延遲 ms | CI/CD 管線 |

| Prompt 工程 | Prompt 模板 | 評分/成功率 | 固定 Token 預算 |

6.2 與既有系統的比較

| 面向 | autoresearch | Optuna/Ray Tune | OpenClaw |

|------|-------------|-----------------|----------|

| 搜尋方式 | LLM Agent 閱讀原始碼、語義理解 | 數學最佳化（TPE/BO） | Plugin 呼叫 |

| 修改範圍 | 任意程式碼變更（架構、優化器重構） | 僅數值超參數 | 預定義 API |

| 可審閱性 | 每步是 git diff | 數值日誌 | 執行日誌 |

| 極簡度 | 3 個檔案、630 行 | 數千行 + 配置 | 25 萬+ Stars 的大型專案 |

七、對 Daily-Digest-Prompt 系統的五大借鑑啟示

7.1 「三原語」可直接對映自動任務框架

| autoresearch 原語 | Daily-Digest-Prompt 對應 |

|-------------------|-------------------------|

| Editable Asset（train.py） | Prompt 模板（templates/auto-tasks/*.md） |

| Scalar Metric（val_bpb） | done_cert.quality_score（1-5） |

| Time-Boxed Cycle（5 分鐘） | config/timeouts.yaml phase2_timeout |

行動建議：現有自動任務已具備三原語雛形，可進一步強化——讓 Agent 基於 quality_score 自動微調 Prompt 模板（類似 autoresearch 修改 train.py）。

7.2 「NEVER STOP」迴圈 vs 有限次數迴圈

autoresearch 的無限迴圈適合探索式研究，但 daily-digest 的排程制（每 30 分鐘）更適合「有限預算、最佳結果」模式。借鑑點：在 Phase 2 自動任務中加入「品質不達標時自動重試」機制（最多 N 次，類似 autoresearch 的 keep/discard）。

7.3 單一檔案約束原則

autoresearch 限制 Agent 只修改一個檔案，大幅降低出錯風險。借鑑點：強化 Hook 機制，限制自動任務 Agent 只能寫入 results/*.json（已部分實現於 pre_write_guard.py），進一步收窄 blast radius。

7.4 Git 作為實驗日誌

autoresearch 每步都 git commit，形成完整的「假說→實驗→結果」追蹤鏈。借鑑點：git-smart-commit Skill 可增強為自動為每次自動任務結果建立 commit，實現類似的審計軌跡。

7.5 program.md 模式 vs SKILL.md 模式

兩者的設計理念驚人相似：人類撰寫「指令文件」引導 AI Agent 行為。autoresearch 用 program.md（單一任務指令），daily-digest 用 SKILL.md（多技能模組化指令）。借鑑點：可考慮為特定深度研究任務建立 program.md 風格的「單次任務聚焦指令」，減少 Skill 載入開銷。

---

八、限制與風險

  • 單 GPU 限制：僅適合小模型實驗（nanochat 等級），無法直接擴展到大規模訓練 [GitHub README]
  • 指標過度簡化：val_bpb 無法捕捉模型品質的所有面向（如毒性、偏見、多任務能力）
  • Agent 失控風險：「NEVER STOP」指令缺乏安全邊界（如 VRAM OOM、磁碟空間耗盡）
  • 可重現性：不同 GPU 型號的 5 分鐘訓練量不同，影響跨硬體比較

---

九、結論

karpathy/autoresearch 的核心貢獻不是「又一個 AutoML 工具」，而是提煉出了自主實驗迴圈的最小可行模式（三原語）。這個模式的威力在於其泛化性——任何「可量化、可修改、可時間限制」的優化問題都能套用。對 Daily-Digest-Prompt 而言，最直接的借鑑是將現有的 quality_score + timeout + prompt 模板三者整合為一個閉環自動優化迴圈，讓系統不只是「執行任務」，而是「持續改進執行方式」。
