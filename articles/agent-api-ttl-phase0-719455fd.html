深度研究洞察簡報：Agent API 快取命中率治理（2026-06-02）

摘要

本專案 system-insight 警示 cache_hit_ratio=0.211（低於 warning 門檻 0.4）；cache/status.json（Phase 0 預計算，2026-06-02T05:30+08:00）顯示 5/5 API 來源 TTL 過期（hit_ratio=0.0）。業界趨勢指向 分層靜態／動態語義快取（Apple Krites，最高約 3.9× 靜態答案重用）與 工具呼叫 memoization（canonical hash + TTL）。對本 repo 的建議：強化 Phase 0 預填排程、對齊 config/cache-policy.yaml 的 adaptive_ttl（命中率 <20% 且低波動 → TTL×2），並將 KB 語義快取 miss 納入命中率儀表板。

關鍵洞見

1. 命中率低落的主因是「快取過期」而非「查詢語義相近但未命中」

  • 證據：cache/status.json 五個來源（todoist、pingtung-news、hackernews、gmail、knowledge）皆 valid: false、miss_reason: ttl_exceeded；todoist 已超 TTL 150 分鐘（TTL 120m）。
  • 含義：Phase 2 Agent 讀到 valid=false 後必然走 live API，統計上拉低 cache_hit_ratio。
  • 行動：確保 run-agent-team.ps1 Phase 0／Phase 1 fetch 在 insight_briefing 等高 IO 任務前執行；監控 exceeded_by_min 欄位作為排程 SLA。

2. 分層語義快取（Krites）可擴大安全重用，且不增加關鍵路徑延遲

  • 來源：Apple ML Research — Semantic Caching（arXiv:2602.13165）；靜態閾值未命中但落在 grey zone 時，非同步 LLM judge 驗證後將靜態答案 promote 至動態層。
  • 證據：對話／搜尋 workload 靜態答案重用最高約 3.9×；關鍵路徑行為與標準 GPTCache 閾值策略相同。
  • 行動：tools/kb_semantic_cache.py 可評估「near-miss + 背景 judge」模式，與現有 lookup/store 分離，避免阻塞 Agent 回合。

3. 工具層快取應區分「可快取讀取」與「副作用寫入」

  • 來源：DEV — Caching Vercel AI SDK tools；tool-call-cache。
  • 證據：建議 TTL：一般搜尋 1–24h、新聞 30–120s、使用者狀態 30–120s；禁止快取 sendEmail／write_file 等有副作用工具。
  • 行動：對 tools/*.py CLI 高頻子命令（如 kb_semantic_cache query）以 SHA-256(canonical JSON) 做檔案快取，並在錯誤回應時 不寫入 cache。

4. Phase 0 預計算 `cache/status.json` 是 Agent 的單一讀取點（設計正確，需排程對齊）

  • 證據：CLAUDE.md 載明 LLM 直接讀 valid 欄位；config/cache-policy.yaml v7 已定義各來源 TTL 與 adaptive_ttl（hit_ratio < 0.20 AND volatility == low → ttl_minutes *= 2）。
  • 行動：將 cache_hit_ratio 分解為「TTL 過期占比」vs「語義 near-miss」；在 write_system_insight.py 追加 cache_ttl_exceeded_ratio 指標。

5. KB 語義快取與 API 快取應分帳監控

  • 證據：config/cache-policy.yaml 已將 kb_hybrid TTL 由 2880→3600m（2026-06-01，因 system-insight 低命中率）；kb_semantic_cache stats --days 7 可彙總 degraded 比例。
  • 行動：insight_briefing 執行前強制 uv run python -m tools.kb_semantic_cache stats --days 7，寫入簡報附錄。
