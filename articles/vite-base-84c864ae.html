Vite 靜態站部署的快取失效、Base 路徑與原子發佈策略

執行摘要

本文主張：對 Vite 靜態站而言，真正決定上線穩定度的不是「能不能 build 出 dist/」，而是三個部署邊界是否被正確設計：base 是否對齊實際掛載路徑、靜態資產是否維持內容雜湊與快取分層、發佈流程是否具備可回溯的原子切換點。這三者若任一失真，就會在 GitHub Pages 上表現為白屏、404 或資產路徑錯誤，在 Cloudflare Pages 上表現為自訂網域快取過久、預覽與正式環境不一致，或 SPA 路由與靜態檔交錯時的行為偏差。

本研究的第一個核心結論是：GitHub Pages 對 Vite 最敏感的設定是 base 與 artifact 結構。Vite 官方部署指南明確要求在 https://.github.io// 場景下把 base 設成 //，而 GitHub Pages 的 Actions 流程則要求部署 artifact 的入口檔位於 artifact 根層，且 deploy-pages 需要明確的 pages: write、id-token: write 與 needs 依賴 [來源：Deploying a Static Site, 2026][來源：Using custom workflows with GitHub Pages, 2026][來源：Creating a GitHub Pages site, 2026]。第二個核心結論是：內容雜湊檔名才是長快取的前提，不能把「所有靜態檔都強快取」當作最佳實踐。Vite 的 asset graph 會替被引用的資產產生 hash 檔名，而 web.dev 也指出 hashed URL 才適合搭配長 Cache-Control；Cloudflare 則進一步提醒，若在 custom domain 上對非 immutable 資源加自訂快取，反而會讓部署後的舊內容持續被送出 [來源：Static Asset Handling, 2026][來源：Love your cache, 2020][來源：Serving Pages, 2026]。

第三個核心結論是：Cloudflare Pages 的預設行為比 GitHub Pages 更接近「前端靜態站友善預設」。它會在沒有頂層 404.html 時自動視為 SPA，並以 Etag、Tiered Cache 與預覽部署做出較完整的發佈隔離；但這個優勢只在你不亂加自訂快取規則時成立 [來源：Serving Pages, 2026][來源：Preview deployments, 2026][來源：部署静态站点, 2026]。第四個核心結論是：若同一份 Vite 產物要同時支援子路徑與不同靜態主機，工程上更穩的做法不是到處硬編碼根路徑，而是讓程式使用 import.meta.env.BASE_URL 取資料，並盡量把可變資料與 immutable bundle 分離。這點也和本專案現況一致：game_web/vite.config.js 使用 base: './'，game_web/src/main.js 以 fetch(\${import.meta.env.BASE_URL}knowledge-sync.json\) 讀取同步資料，正是為了減少子路徑部署風險。

研究背景與動機

今天完成的 Todoist 任務是「知識庫同步更新 Vite 網站並推送 GitHub」，它表面上是一次內容同步與發佈操作，實際上碰到的是靜態站部署最典型的三個工程問題：內容放上去之後能不能正確被找到、改版之後能不能立刻看到新資產、以及多分支或多平台部署時能不能把 preview 與 production 隔開。這類問題之所以常反覆出現，是因為它們跨越了 bundler、CDN、Git hosting 與前端程式碼本身四個層次，單看其中一層很容易誤判。

知識缺口在於：多數教學只教「怎麼把 Vite 丟上 GitHub Pages 或 Cloudflare Pages」，卻不把部署穩定性的條件拆開講。base 錯、public 資產放錯、快取規則加錯、預覽流程缺失，這些都不會在 vite build 當下報錯，而是要到正式部署後才暴露。對需要長期維護靜態知識站的專案來說，這種延遲型錯誤比 build fail 更麻煩，因為它通常要靠使用者或人工巡檢才會發現。

因此，本研究不再討論「Vite 能不能部署」，而是聚焦「怎樣部署才不容易出現路徑錯、舊檔殘留、預覽污染正式站、或 SPA 路由異常」。研究目標是建立一套可直接落地的判斷框架，讓後續不論是部署到 GitHub Pages、Cloudflare Pages，或兩者並行，都能用同一套邏輯檢查風險。

文獻探討

Vite 官方文件提供了最接近事實基線的部署規範。英文版與中文版部署指南都指出，Vite 預設 build 輸出是 dist/，而 GitHub Pages 若部署於 repo 子路徑，base 必須設定為 //；若是 user site 或 custom domain，則維持 / 即可 [來源：Deploying a Static Site, 2026][來源：部署静态站点, 2026]。同一份指南還給出 GitHub Actions 範本，包含 concurrency、Node setup、artifact upload 與 deploy-pages，這代表 Vite 官方已把「Actions artifact 發佈」視為 GitHub Pages 的標準做法，而不是把 dist/ 直接 commit 到特殊分支。

GitHub 官方文件補足了 Pages 端的限制條件。Using custom workflows with GitHub Pages 指出 deploy-pages 需要 pages: write 與 id-token: write 權限，且 needs 必須指向產出 artifact 的 build job，否則部署 job 可能會找不到要發布的內容 [來源：Using custom workflows with GitHub Pages, 2026]。Creating a GitHub Pages site 進一步說明，若用 Actions workflow 發佈，artifact 必須在根層包含 entry file；若走 branch publishing，非 Jekyll 產物要加 .nojekyll，而且 workflow 用 GITHUB_TOKEN 推出的 commit 不會再觸發新的 Pages build [來源：Creating a GitHub Pages site, 2026]。這些規則說明 GitHub Pages 的穩定性高度依賴流程正確，而不是靜態檔本身正確就足夠。

在快取層，Vite 的 Static Asset Handling 提供的是資產命名與 URL 解析機制：被 source code 引用的資產會進入 build graph、取得 hash 檔名，public/ 下的檔案則保留原始名稱直接複製到輸出根層 [來源：Static Asset Handling, 2026]。web.dev 的 Love your cache 則從瀏覽器快取角度指出，只有帶內容雜湊的資源，才適合給很長的 Cache-Control: max-age=31536000, immutable；不能改名的使用者入口頁（例如 index.html）不適合套同樣策略 [來源：Love your cache, 2020]。這和 Vite 的 public/import 分流剛好形成互補：可雜湊的檔案進 asset graph，不可雜湊但必須固定檔名的檔案留在 public/，兩者快取策略不能混用。

Cloudflare Pages 官方文件則補足「平台預設」層面的差異。Serving Pages 指出，若沒有頂層 404.html，Pages 會將站點視為 SPA，所有路由回落到根；同時 Pages 預設已做足快取最佳化、送出 Etag、使用 Tiered Cache，且明白警告不要輕率在 custom domain 上再加自訂 caching，否則可能在部署後繼續送出 stale assets [來源：Serving Pages, 2026]。Preview deployments 則說明每個分支或 PR 都有獨立 preview URL，hash-based preview URL 可長期直接存取，而 branch alias 只會指向該分支最新版本 [來源：Preview deployments, 2026]。這使 Cloudflare 在「預覽隔離」與「部署原子感」上，天然比 GitHub Pages 更完整。

核心論證

子題一：`base` 不是小設定，而是部署正確性的第一道邊界

事實陳述：Vite 在 GitHub Pages repo 子路徑部署時，base 必須與最終 URL 子路徑一致；若使用 Actions artifact 發佈，entry file 也必須位於 artifact 根層，否則站點雖已部署，仍可能在執行時找不到靜態資產或首頁 [來源：Deploying a Static Site, 2026][來源：部署静态站点, 2026][來源：Creating a GitHub Pages site, 2026]。

來源佐證：Vite 官方英中兩版都把 // 視為 GitHub Pages repo-site 的必要 base；GitHub Docs 則把 artifact root entry file 列為 Actions 發佈前提，並補充 deploy-pages 需配對 build job 與權限 [來源：Deploying a Static Site, 2026][來源：Using custom workflows with GitHub Pages, 2026][來源：Creating a GitHub Pages site, 2026]。

推論：因此，GitHub Pages 上最常見的白屏不是「Vite build 壞掉」，而是建置產物與掛載路徑模型不一致。若專案要同時部署到 repo 子路徑、user site 與其他靜態主機，單靠硬編碼 /repo/ 很脆弱。更穩的策略是像本 repo game_web/vite.config.js 一樣，採相對 base: './'，再在應用端用 import.meta.env.BASE_URL 組出資料路徑；這樣可把「路徑正確性」從人工記憶轉為建置輸出的一部分。

子題二：快取更新的關鍵不在「清快取」，而在「哪些檔應該可永久快取」

事實陳述：Vite 對 source-import 的資產會產生 hash 檔名；web.dev 認為 hashed filenames 是長生命週期快取的前提；Cloudflare Pages 也只建議對 immutable hashed assets 做客製快取，而非對整站或 mutable 資料一體套用 [來源：Static Asset Handling, 2026][來源：Love your cache, 2020][來源：Serving Pages, 2026]。

來源佐證：Vite 文件直接說明 referenced assets 會進入 build graph 並取得 hashed file names；web.dev 將內容 hash 與長快取綁在一起，說明不能重新命名的入口頁不應使用同一快取模型；Cloudflare 則點明 custom domain 加快取可能讓新部署仍送出 stale assets，但對帶內容 hash 的 CSS、JS 做快取是合理例外 [來源：Static Asset Handling, 2026][來源：Love your cache, 2020][來源：Serving Pages, 2026]。

推論：真正要避免 stale deploy，最優先的不是每次部署後人工 purge，而是把檔案切成兩類。第一類是可雜湊的 bundle、CSS、圖片，應該留在 Vite asset graph，吃長快取與 CDN 分發；第二類是固定名稱且內容可能更新的檔案，例如 knowledge-sync.json，應該視為 mutable payload，不要套與 bundle 同樣 aggressive 的 caching。這也解釋了為什麼本 repo 會把同步資料放在 game_web/public/knowledge-sync.json，再由 game_web/src/main.js 透過 BASE_URL 明確抓取：它本質上就是一個「固定 URL、變動內容」的資源。

子題三：Cloudflare Pages 比 GitHub Pages 更接近前端站的安全預設，但自訂快取仍可能反噬

事實陳述：Cloudflare Pages 在沒有頂層 404.html 時，會自動把站點視為 SPA；預設發送 Etag，資產由 Tiered Cache 服務；預覽部署會為每個分支建立獨立 URL 與 alias，而且 preview 不會污染正式 custom domain [來源：Serving Pages, 2026][來源：Preview deployments, 2026][來源：部署静态站点, 2026]。

來源佐證：Serving Pages 明確寫出 SPA fallback、Etag、Tiered Cache 與 Purge Everything 路徑；Preview deployments 則描述 hash-based preview URL、branch alias 與 preview 對 production/custom domain 的隔離；Vite 中文部署文也把 Cloudflare Pages 視為直接以 Git 導入與自動部署的標準平台之一 [來源：Serving Pages, 2026][來源：Preview deployments, 2026][來源：部署静态站点, 2026]。

推論：若目標是快速、安全地驗證前端變更，Cloudflare Pages 的分支預覽與內建 SPA 行為讓它更像「為現代前端預設最佳化的平台」。但這個優勢並不代表能無條件在 custom domain 上疊快取規則。Cloudflare 自己已警告，額外 caching 可能讓 redirect、Functions 或新資產切換失真。因此平台預設已經足夠時，應先維持預設，再只針對 hash assets 做最小化調整。

子題四：真正的原子發佈，不是「推上去就算完成」，而是能明確指出哪個版本被切到正式流量

事實陳述：Vite 官方給的 GitHub Pages workflow 使用 artifact upload 與 deploy action，並加入 concurrency；GitHub custom workflows 文件要求 build job 與 deploy job 之間有清楚的 needs 關係；Cloudflare preview deployments 則保留每次 preview 的 hash URL，branch alias 只指向最新版本 [來源：Deploying a Static Site, 2026][來源：Using custom workflows with GitHub Pages, 2026][來源：Preview deployments, 2026]。

來源佐證：Vite 範本 workflow 中 configure-pages、upload-pages-artifact、deploy-pages 是明確分段的；GitHub Docs 把 artifact 與 deploy job 相依列成必要條件；Cloudflare Docs 則說 ..pages.dev 會永久存在、branch alias 只轉向最新部署 [來源：Deploying a Static Site, 2026][來源：Using custom workflows with GitHub Pages, 2026][來源：Preview deployments, 2026]。

推論：對前端團隊而言，原子發佈的實務意義是「你知道哪一包被上線、能不能對照 preview、出問題能不能定位到前一包」。GitHub Pages 的 artifact 模式已比直接 commit gh-pages 分支穩定，因為它把 build 產物與 deploy 步驟分離；Cloudflare 則更進一步，讓每次 preview 版本本身就是可回看的實體 URL。若專案重視內容同步、人工驗收與多分支試跑，Cloudflare 的預覽治理能力更成熟。

一家之言

我認為 Vite 靜態站部署最常見的誤區，是把「平台上線」當成終點，而不是把「版本切換行為是否可預期」當成終點。GitHub Pages 與 Cloudflare Pages 其實都能部署 Vite，但兩者適合的治理模型不同。GitHub Pages 比較像是「你自己把 build 邊界與發佈流程設計好，它就能穩定服務 artifact」；Cloudflare Pages 比較像「平台先幫你把 preview、SPA fallback、CDN 分發預設做好，再由你決定是否額外微調」。

所以，如果專案的核心需求是低成本公開站點、repo 綁定、文件頁面或簡單靜態展示，GitHub Pages 已足夠；但如果專案常做內容同步、分支驗證、多輪預覽、或需要避免人為把 preview 與 production 搞混，Cloudflare Pages 的 operational model 明顯更好。真正該被優先優化的，也不是「哪家比較快」，而是專案是否有明確區分 immutable asset、mutable data、preview alias 與正式流量入口。沒有這個分層，換任何平台都只是把偶發故障改成延遲故障。

實踐建議

  1. 對 repo 子路徑部署明確建模。

步驟：GitHub Pages repo-site 用 base: '//'；若同一產物需在多個子路徑或檔案系統靜態主機上執行，改採相對 base，並統一從 import.meta.env.BASE_URL 推導 fetch/asset URL。風險：第三方腳本若仍假設根路徑 /，會在子路徑下失敗。

  1. 將 mutable JSON 與 hashed bundles 拆開管理。

步驟：JS/CSS/圖片盡量用 import 進 Vite asset graph；像 knowledge-sync.json 這種固定檔名資料留在 public/ 或等價靜態目錄。風險：若把可變 JSON 也加上超長 CDN/browser cache，部署後會看見舊內容。

  1. GitHub Pages 一律走 artifact workflow，不走手推 build branch。

步驟：使用 configure-pages、upload-pages-artifact、deploy-pages 三段式，保留 needs、權限與 concurrency。風險：若讓 workflow 再去推 Pages branch，可能踩到 GITHUB_TOKEN 不觸發 Pages rebuild 的限制。

  1. Cloudflare Pages 預設先維持原樣，再局部加速。

步驟：先吃平台預設 SPA fallback、Etag、Tiered Cache 與 preview deployments；只針對帶內容 hash 的資產加 cache rule。風險：若直接對 custom domain 全站加 cache，可能讓 redirect、Functions 或新版本資產切換不一致。

  1. 建立部署後驗證清單，而不是只看 CI 綠燈。

步驟：至少驗證首頁、子路由、JSON fetch、preview 與 production 是否一致；Cloudflare 若發現 stale 資產，再做 purge。風險：只看 workflow success，容易漏掉 runtime path/caching 問題。

風險與限制

本研究聚焦於 Vite 靜態站，不涵蓋 SSR、混合渲染或需要後端 session 的部署模型，因此結論不適用於需要伺服器端路由、動態權限控制或多區域寫入一致性的系統。其次，web.dev 的快取文章偏通用 Web 平台觀點，並非專為 Vite 或 GitHub Pages 撰寫，所以本文只把它用在「hashed asset 與 immutable caching」原則層，不直接外推到所有 Pages 平台細節。最後，本研究沒有實測本 repo 在 GitHub Pages 與 Cloudflare Pages 的 TTFB、cache-hit 或 rollback 時間，只能提供架構與流程層面的建議，不能把本文解讀為正式 benchmark。
