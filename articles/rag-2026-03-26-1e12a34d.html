三、關鍵發現（20 項，按嚴重度排序）

P0 — 關鍵（2 項）

F-01：.env 檔案歷史殘留風險

  • 問題：.env 與 bot/.env 可能曾被提交至 git 歷史
  • 影響：任何 clone/fork 可能暴露 API 憑證
  • 建議：使用 BFG Repo-Cleaner 清除 git 歷史中的 .env 檔案，並定期輪換所有 API 金鑰
  • 參考：OWASP Secrets Management Cheat Sheet

F-02：Groq Relay 端點認證

  • 問題：groq-relay.js 的 /groq/chat 端點應確保認證機制完整
  • 影響：未授權存取可能導致 API 配額耗盡
  • 建議：確認所有公開端點均有 API_SECRET_KEY 驗證，定期輪換金鑰

P1 — 高風險（8 項）

F-03：單一共享密鑰認證

  • bot.js 所有端點使用同一 API_SECRET_KEY
  • 建議：實施金鑰輪換策略（90 天 TTL），支援輪換期間雙金鑰共存

F-04：Hook 寬容模式風險

  • hook-rules.yaml 的 permissive preset 會停用 medium/low 規則
  • 建議：將 critical 規則硬編碼為永遠啟用，permissive 僅限 CI/CD 測試環境

F-05：路徑穿越告警不阻擋

  • pre_write_guard.py 專案外寫入為 warn_only
  • 現狀：設計決策（ADR-005），搭配 ntfy 告警。建議每季審計 allowed_paths

F-06：routes.js 輸入未限制長度

  • saveToKnowledgeBase() 的 taskContent 無長度上限
  • 建議：加入 contentText 8000 字元上限（與 groq-relay 對齊）

F-07：Python 依賴版本鬆綁

  • pyproject.toml 部分依賴使用 >= 範圍（edge-tts、fpdf2、Gmail 相關）
  • 建議：所有依賴鎖定為精確版本，搭配 uv.lock 作為唯一真相來源

F-08：npm 依賴版本鬆綁

  • package.json 使用 ^X.Y.Z 允許自動升級
  • 建議：鎖定 package-lock.json 並定期執行 npm audit

F-09：LINE Webhook 密鑰管理

  • LINE_CHANNEL_SECRET 不在本地 .env 中（✅ 正確），僅在 Render 環境變數
  • 建議：維持現狀，確認永不存入版本控制

F-10：錯誤回應洩露 API 契約

  • groq-relay.js 422 回應包含欄位級驗證細節
  • 建議：生產環境返回通用錯誤訊息，詳細錯誤僅記錄於伺服器端日誌
