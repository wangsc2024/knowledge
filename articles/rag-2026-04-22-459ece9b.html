執行摘要

本次分析涵蓋 5 篇 2026 年 RAG 與系統優化相關知識庫筆記（總計 53,128 字元），聚焦於 GraphRAG 知識圖譜、Harness Engineering 測試治理、Compound AI 路由優化、Vite 增量部署與 AI 市民服務平台等五大主題。

這些筆記反映了 RAG 系統從基礎檢索向知識圖譜增強、從單一模型向 Compound AI 路由、從手動部署向自動化工作流程的演進趨勢，並提供了可量化的測試覆蓋率基線（80% 單元測試、60% 變異測試）與實踐路徑。

---

關鍵洞見（Top 5）

  1. GraphRAG 將 RAG 從向量檢索提升至知識圖譜層次

- 透過實體關係建模提供更精確的上下文理解

- 相比傳統 RAG 提升全域問題回答準確度 20-40%

- LazyGraphRAG 延遲建圖策略降低索引成本 60-70%

  1. Harness Engineering 引入四大支柱作為系統穩定性基線

- 約束演化（constraint evolution）自動化率 ≥70%

- 品質門檻（quality gates）覆蓋率 ≥80%

- 熵分數（entropy score）< 0.5

- 覆蓋率量化（coverage metrics）變異測試 ≥60%

  1. Compound AI 透過路由層與 RAG 配置的聯合優化

- 實現模型選擇與檢索策略的協同決策

- 減少配置試錯時間 50-70%

- 六維決策：模型能力、成本預算、延遲要求、準確度目標、安全合規、負載均衡

  1. Vite 增量部署結合 Git 自動化工作流程

- Git diff 偵測 → 內容雜湊去重 → 選擇性失效快取

- 減少 CI/CD 時間 60-80%

- 全程 5-10 分鐘（commit hook → GitHub Actions → Cloudflare Pages）

  1. AI 市民服務平台需兼顧多模態輸入與政策知識庫的即時檢索能力

- 語音、影像、文字三通道統一管線

- 平均響應時間 < 3 秒

- ASR → 意圖識別 → 混合檢索 → 生成回應 → 多通道輸出

---

技術趨勢（Top 5）

1. 知識圖譜增強（GraphRAG）

從扁平向量檢索轉向結構化實體關係建模。

三層索引架構：

  • 實體層（Entity Extraction）
  • 社群層（Community Detection）
  • 摘要層（Hierarchical Summarization）

LazyGraphRAG 延遲建圖策略：僅為查詢相關文件建構子圖，適合動態知識庫場景。

2. 測試治理機械化

用約束演化管線與品質門檻取代人工審查，降低 LLM 依賴。

量化基線：

  • 單元測試覆蓋率 ≥80%
  • 變異測試分數 ≥60%
  • 熵分數 < 0.5
  • 約束演化自動化率 ≥70%

3. 路由與檢索聯合優化

Compound AI 系統中，模型路由決策與 RAG 配置不再獨立。

METIS 框架：將 top-k、chunk_size、reranker 等 RAG 超參數與模型路由決策放入同一優化空間，透過多目標優化（qLogNEHVI）求解帕累托前沿。

4. 增量部署自動化

靜態網站透過 Git diff 偵測與選擇性構建，減少 CI/CD 時間 60-80%。

三階段管線：

  1. Git diff 偵測變更（僅構建變動檔案）
  2. 內容雜湊去重（避免重複上傳）
  3. 選擇性失效快取（只清除受影響路徑）

5. 多模態 RAG 平台

政府服務場景需整合語音轉文字、OCR、文本檢索的統一管線。

五階段流程：

  1. ASR（自動語音辨識，Whisper）
  2. 意圖識別（NLU）
  3. 混合檢索（政策文件 + FAQ）
  4. 生成回應（LLM + RAG）
  5. 多通道輸出（文字 + 語音 + 視覺）

優化機會（針對 daily-digest-prompt 系統）

P0 優先級（立即執行）

#### 1. 套用 Harness Engineering 測試基線

實施路徑：

  • 在 check-health.ps1 中新增 [測試治理] 區塊
  • 追蹤 hooks/ 與 tools/ 的單元測試覆蓋率（目標 ≥80%）
  • 追蹤變異測試分數（目標 ≥60%）
  • 低於閾值時 ntfy 告警

預期收益：

  • 建立可量化的品質基線
  • 降低系統熵增風險
  • 減少手動審查成本

工作量: 2-3 天

#### 2. 優化 kb-sync-incremental.ps1 增量同步

實施路徑：

  • 階段 1: Git diff 偵測層（只同步變更筆記）
  • 階段 2: 內容雜湊去重（避免重複寫入）
  • 階段 3: 失效快取選擇性清除（只更新受影響索引）

預期收益：

  • Phase 0 KB 同步時間從 30-60 秒降至 10-20 秒
  • 減少 50-70% 執行時間

工作量: 1-2 天

P1 優先級（短期規劃）

#### 3. 實施 Compound AI 動態路由

實施路徑：

  • 新增 tools/compound_ai_router.py
  • 根據任務複雜度（token 估算）、預算剩餘（budget_guard.py）、模型可用性（API health）聯合決策模型選擇
  • 取代 llm-router.yaml 靜態 mapping

預期收益：

  • 提升模型選擇的適應性
  • 減少因靜態配置導致的 token 浪費或品質下降

工作量: 3-5 天

#### 4. 擴展 ntfy 多模態告警（TTS 語音播報）

實施路徑：

  • critical 級別同時推送：

- 文字（ntfy）

- 語音播報（TTS edge-tts）

- 截圖（Playwright 錯誤畫面）

預期收益：

  • 提升緊急告警的可達性
  • 改善緊急告警的響應速度

工作量: 2-3 天（非核心功能）

P2 優先級（中長期探索）

#### 5. 引入 GraphRAG 知識圖譜層

實施路徑：

  • 採用 LazyGraphRAG 延遲建圖策略
  • 將 research-registry.json 與 improvement-backlog.json 轉為知識圖譜
  • 建立 ADR↔Task↔Skill 的實體關係
  • 僅在 kb-research-strategist 查詢時動態建構子圖

預期收益：

  • 提升研究接續的上下文理解能力
  • 減少索引維護成本

工作量: 5-7 天（需大幅改造現有 JSON 儲存結構）
