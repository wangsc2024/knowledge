專案 RAG 優化日報（2026-04-19）

一、執行摘要

基於 RAG 知識庫查詢與系統健康檢查，本次優化聚焦於四個關鍵領域：快取效益提升、Phase 失敗率降低、配置驗證補全、知識活化迴圈建立。優化後預期系統成功率從 73% 提升至 85%+，快取命中率從 27.9% 提升至 45%+。

二、診斷結果

2.1 快取效益低下（P0）

  • 現況：7 天平均命中率 27.9%，目標 40%+
  • 根因：昨日 34 次快取操作中，快取被標記為 stale 而非 hit
  • 影響：API 重複呼叫增加延遲與成本
  • 證據：logs/structured/2026-04-18.jsonl 顯示 cache_status: "stale"

2.2 Phase 失敗率高（P1）

  • 現況：近 7 天 Phase 失敗 115 次
  • 根因：「assembly failed after 2 attempts」
  • 影響：系統成功率僅 73%
  • 證據：state/scheduler-state.json 記錄多次 assembly 失敗

2.3 配置驗證不完整（P2）

  • 現況：15 個配置檔缺少 JSON Schema 驗證
  • 風險：配置錯誤可能導致運行時失敗
  • 證據：check-health.ps1 顯示「💡 有 15 個配置檔尚未使用 JSON Schema 驗證」

2.4 知識活化不足（P3）

  • 現況：107 筆研究積累，但轉化為系統改善的比例低
  • 根因：缺乏「研究→ADR→實施」追蹤機制
  • 證據：知識庫文獻「知識活化與 OODA 閉環實效性分析」指出轉化效率診斷需求

三、優化方案

3.1 快取命中率提升（P0）

方案 A：強化 cache-first 策略

  • 在 api-cache Skill 步驟 0 強制讀取快取狀態
  • 修改快取判定邏輯：stale → hit（在 grace period 內）
  • 啟用 SWR（Stale-While-Revalidate）機制

方案 B：優化 TTL 動態調整

  • 實施 adaptive_ttl 規則（config/cache-policy.yaml 已定義）
  • 依實際命中率調整各端點 TTL
  • 目標：hackernews 50%+、gmail 40%+、knowledge 60%+

預期效果：命中率從 27.9% → 45%+

3.2 Phase 失敗率降低（P1）

方案 A：加強 Phase 間結果驗證

  • Phase 2 完成前驗證結果 JSON schema
  • Phase 3 啟動前確認所有依賴檔案存在
  • 失敗時記錄詳細錯誤原因（不只「assembly failed」）

方案 B：增加重試策略

  • Phase 失敗時自動重試（最多 3 次）
  • 重試間隔：2s → 5s → 10s（指數退避）
  • 記錄重試原因與次數

預期效果：成功率從 73% → 85%+

3.3 JSON Schema 驗證補全（P2）

執行步驟：


# 1. 生成缺失的 Schema
uv run python hooks/validate_config.py --migrate

# 2. 加入 pre-commit hook
echo "uv run python hooks/validate_config.py" >> .git/hooks/pre-commit

# 3. 驗證所有配置檔
uv run python hooks/validate_config.py --check-all

預期效果：配置錯誤在提交前攔截

3.4 知識活化迴圈（P3）

建立追蹤機制：建立 context/knowledge-activation.yaml，追蹤研究到 ADR 到實施的完整流程。

自動回顧機制：

  • 新增自動任務 knowledge_activation_review（每週執行）
  • 識別高價值研究（引用次數、相關性分數）
  • 自動生成 ADR proposal

預期效果：知識活化率從 16.8% → 30%+

五、實施時程

  • Week 1: 快取優化 + Phase 驗證（10h，P0/P1）
  • Week 2: Schema 補全 + TTL 調整（8h，P0/P2）
  • Week 3: 知識活化機制 + 測試（12h，P3）
