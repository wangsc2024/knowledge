PowerShell 7 並行程式設計完整指南（2026）

技術概述

PowerShell 7 提供四種主要並行執行機制：Start-Job（程序隔離）、Start-ThreadJob（執行緒輕量級）、ForEach-Object -Parallel（管線並行）、以及直接操作 .NET Runspace Pool（最低層級控制）。選擇正確的並行策略對效能影響極大——Microsoft 官方基準測試顯示，處理 20 個任務時 Start-Job 耗時 7.58 秒，Start-ThreadJob 僅 2.37 秒，ForEach-Object -Parallel 更只需 0.79 秒。本文深入比較四種方法的內部機制、效能特性、適用場景，並提供與 daily-digest-prompt 專案直接相關的生產級最佳實踐。

---

核心概念與架構

四種並行方法的本質差異

| 特性 | Start-Job | Start-ThreadJob | ForEach-Object -Parallel | Runspace Pool |

|------|-----------|-----------------|--------------------------|---------------|

| 執行單位 | 獨立程序 | 執行緒（Runspace） | 執行緒（Runspace Pool） | 執行緒（手動管理） |

| 隔離程度 | 最高（程序隔離） | 中等（執行緒隔離） | 中等（Runspace 隔離） | 最低（需自行管理） |

| 序列化 | 跨程序序列化/反序列化 | 無需（同程序記憶體） | 無需（同程序記憶體） | 無需 |

| 物件型別保真 | 低（經序列化可能失真） | 高（原生物件） | 高（原生物件） | 高 |

| 記憶體開銷 | 最高（每個 Job 一個 pwsh 實例） | 低 | 最低（Runspace Pool 重用） | 低 |

| ThrottleLimit | 無（需手動管理） | 有 | 有（預設 5） | 需手動設定 Pool Size |

| 適用版本 | PS 5.1+ | PS 7+（或從 Gallery 安裝） | PS 7.0+ | PS 5.1+（需 .NET 知識） |

| 程式碼複雜度 | 低 | 低 | 最低 | 最高 |

效能基準（Microsoft 官方 PS 7.5.1 測試）

處理 20 個任務（BatchSize=5）的實際測量結果：


Start-Job:                7.58 秒  ← 最慢（程序建立開銷）
Start-ThreadJob:          2.37 秒  ← 中等
ForEach-Object -Parallel: 0.79 秒  ← 最快（Runspace Pool 重用）
Start-Process:            0.26 秒  ← 特殊場景（僅適用外部程式）

關鍵發現：Start-Job 比 ForEach-Object -Parallel 慢 9.6 倍，主要瓶頸在於每個 Job 建立獨立 PowerShell 程序的開銷。

各方法深入解析

1. Start-Job — 程序隔離的重量級方案

內部機制：

  • 每個 Job 啟動一個全新的 pwsh.exe 程序
  • 輸入/輸出透過跨程序序列化（Remoting Serialization System）傳遞
  • 提供最強隔離性——一個 Job 崩潰不影響其他 Job 或主程序

PowerShell 7 新增功能：

  • -WorkingDirectory 參數：指定背景 Job 的工作目錄（PS 5.1 無此功能，是 daily-digest-prompt 遷移到 PS 7 的關鍵原因之一）
  • 若未指定，預設使用呼叫者的當前目錄

適用場景：

  • 需要最高程序隔離（防止一個任務崩潰影響全部）
  • 執行不受信任的腳本或外部工具
  • 任務輸出不需要型別保真（字串/簡單物件足夠）
  • 本專案場景：run-agent-team.ps1 的 Phase 1 五路並行擷取（每路啟動 claude -p，本質上是外部程式）

程式碼範例：


# daily-digest-prompt 的實際模式
$job = Start-Job -WorkingDirectory $AgentDir -ScriptBlock {
    $OutputEncoding = [System.Text.Encoding]::UTF8
    & claude -p --allowedTools "Read,Bash,Write" (Get-Content $using:promptFile -Raw)
} -Name "fetch-todoist"

# 等待完成（含 timeout）
$completed = $job | Wait-Job -Timeout 300
if (-not $completed) {
    $job | Stop-Job
    Write-Warning "Job timed out"
}
$result = $job | Receive-Job
$job | Remove-Job

2. Start-ThreadJob — 輕量級執行緒 Job

內部機制：

  • 在同一程序內建立獨立 Runspace 執行緒
  • 不經過序列化系統，輸出物件保持「活」的狀態（型別保真）
  • 自帶 -ThrottleLimit 參數控制並行數
  • PowerShell 7 內建 ThreadJob 模組

關鍵優勢：

  • 比 Start-Job 快 3-10 倍（無程序建立開銷）
  • 保持物件型別保真（不經序列化/反序列化）
  • 支援 -InitializationScript 預備 Runspace 環境

風險注意：

  • 隔離性較低——一個 ThreadJob 崩潰可能影響同程序的其他 ThreadJob
  • 共享資源時需注意執行緒安全

程式碼範例：


# 並行處理多個 API 快取檢查
$cacheFiles = @('todoist', 'pingtung-news', 'hackernews', 'gmail')
$jobs = $cacheFiles | ForEach-Object {
    Start-ThreadJob -ThrottleLimit 4 -ArgumentList $_ {
        param($api)
        $cachePath = "cache/$api.json"
        if (Test-Path $cachePath) {
            $cache = Get-Content $cachePath | ConvertFrom-Json
            $age = (Get-Date) - [datetime]$cache.timestamp
            [PSCustomObject]@{
                API = $api
                Cached = $true
                AgeMinutes = $age.TotalMinutes
                Expired = $age.TotalMinutes -gt $cache.ttl_minutes
            }
        } else {
            [PSCustomObject]@{ API = $api; Cached = $false }
        }
    }
}
$results = $jobs | Receive-Job -Wait -AutoRemoveJob

3. ForEach-Object -Parallel — 管線並行的現代首選

內部機制：

  • PowerShell 7.0 引入，底層使用 PSTaskPool 類別管理執行中的任務
  • PS 7.1 起預設重用 Runspace Pool（而非每次建立新 Runspace），大幅降低開銷
  • -ThrottleLimit 設定 Runspace Pool 大小（預設 5）
  • 可用 -UseNewRunspace 開關強制每次建立新 Runspace（較慢但更安全）

變數作用域：

  • 每個平行迭代在獨立 Runspace 中執行，具備隔離的作用域
  • 存取外部變數必須使用 $using: 修飾符
  • Runspace 啟動時為「乾淨」狀態，不繼承已載入的模組或函式

程式碼範例：


# I/O 密集型：並行查詢多個知識庫主題
$topics = @('PowerShell 7', 'CRDT', 'Circuit Breaker', 'ORM Migration')
$apiBase = 'http://localhost:3000'

$results = $topics | ForEach-Object -ThrottleLimit 10 -Parallel {
    $topic = $_
    $base = $using:apiBase
    try {
        $response = Invoke-RestMethod -Uri "$base/api/search/hybrid" `
            -Method POST `
            -ContentType 'application/json' `
            -Body (@{ query = $topic; topK = 5 } | ConvertTo-Json)
        [PSCustomObject]@{
            Topic = $topic
            ResultCount = $response.results.Count
            TopTitle = $response.results[0].title
            Status = 'Success'
        }
    } catch {
        [PSCustomObject]@{
            Topic = $topic
            ResultCount = 0
            TopTitle = $null
            Status = "Error: $($_.Exception.Message)"
        }
    }
}

4. Runspace Pool — .NET 層級的精細控制

內部機制：

  • 直接使用 System.Management.Automation.Runspaces 命名空間
  • RunspaceFactory.CreateRunspacePool(minRunspaces, maxRunspaces) 建立 Pool
  • 每個 Runspace 可獨立設定初始會話狀態（InitialSessionState）
  • 完全控制執行緒生命週期

適用場景：

  • 需要精確控制 Runspace 初始化（載入特定模組/變數）
  • 高效能批次處理（數百至數千個平行任務）
  • 需要同步原語（Mutex、Semaphore、ConcurrentDictionary）

程式碼範例：


# 建立 Runspace Pool
$pool = [RunspaceFactory]::CreateRunspacePool(1, 10)
$pool.Open()

$handles = @()
try {
    foreach ($item in $workItems) {
        $ps = [PowerShell]::Create()
        $ps.RunspacePool = $pool
        $ps.AddScript({
            param($data)
            # 處理邏輯
            Start-Sleep -Milliseconds 100
            "Processed: $data"
        }).AddArgument($item) | Out-Null
        
        $handles += [PSCustomObject]@{
            PowerShell = $ps
            Handle = $ps.BeginInvoke()
        }
    }
    
    # 收集結果
    $results = $handles | ForEach-Object {
        $_.PowerShell.EndInvoke($_.Handle)
        $_.PowerShell.Dispose()
    }
} finally {
    $pool.Close()
    $pool.Dispose()
}

最佳實踐與常見陷阱

1. ThrottleLimit 調校策略

| 工作負載類型 | 建議 ThrottleLimit | 原因 |

|-------------|-------------------|------|

| CPU 密集 | = 邏輯核心數 | 超過無益，反增切換開銷 |

| I/O 密集（API 呼叫） | 核心數 × 2-3 | 執行緒等待 I/O 時可讓位 |

| 網路密集 | 10-50（實測調整） | 受限於網路頻寬和 API 速率限制 |

| 混合型 | 核心數 + 2-5 | 平衡 CPU 和 I/O 等待 |

本專案適用：daily-digest-prompt 的 Phase 1 五路並行（Todoist + 新聞 + HN + Gmail + 安全審查）屬於 I/O 密集型（等待 API 回應 + claude CLI 執行），ThrottleLimit=5 合理。

2. 執行緒安全與資料收集

危險模式（避免）：


# 錯誤：標準陣列非執行緒安全
$results = @()
1..100 | ForEach-Object -Parallel {
    ($using:results) += "Item $_"  # 競態條件！
}

安全模式：


# 方法 1：使用 ConcurrentBag（推薦）
$bag = [System.Collections.Concurrent.ConcurrentBag[PSObject]]::new()
1..100 | ForEach-Object -Parallel {
    ($using:bag).Add([PSCustomObject]@{ Id = $_; Value = $_ * 2 })
}
$results = $bag.ToArray()

# 方法 2：直接輸出（ForEach-Object -Parallel 自動收集管線輸出）
$results = 1..100 | ForEach-Object -Parallel {
    [PSCustomObject]@{ Id = $_; Value = $_ * 2 }  # 自動透過管線安全收集
}

3. 錯誤處理模式

平行任務中的錯誤處理原則：

  • 在平行 ScriptBlock 內部處理例外（非外部）
  • 回傳狀態物件而非讓例外傳播
  • 使用 -ErrorAction Stop 確保例外被 try-catch 捕獲

$results = $tasks | ForEach-Object -Parallel {
    $task = $_
    try {
        # 業務邏輯
        $output = Invoke-RestMethod -Uri $task.url -TimeoutSec 30 -ErrorAction Stop
        [PSCustomObject]@{
            TaskId = $task.id
            Status = 'Success'
            Data = $output
            Error = $null
        }
    } catch {
        [PSCustomObject]@{
            TaskId = $task.id
            Status = 'Failed'
            Data = $null
            Error = $_.Exception.Message
        }
    }
}
# 事後統一處理
$failed = $results | Where-Object Status -eq 'Failed'
if ($failed) { Write-Warning "$($failed.Count) tasks failed" }

4. Timeout 管理模式


# Start-Job 搭配 Wait-Job -Timeout
$jobs = 1..5 | ForEach-Object {
    Start-Job -ScriptBlock { Start-Sleep 600; "Done $_" }
}
$completed = $jobs | Wait-Job -Timeout 300
$timedOut = $jobs | Where-Object State -eq 'Running'
$timedOut | Stop-Job
$timedOut | Remove-Job -Force

# ForEach-Object -Parallel 搭配 -TimeoutSeconds（PS 7.3+）
$results = $urls | ForEach-Object -Parallel {
    Invoke-RestMethod -Uri $_ -TimeoutSec 30
} -ThrottleLimit 10 -TimeoutSeconds 120

5. UTF-8 編碼注意事項（Windows 特有）

問題：Start-Job 的背景程序預設使用系統編碼（非 UTF-8），導致中文等非 ASCII 字元亂碼。

解決方案（daily-digest-prompt 已採用）：


$job = Start-Job -WorkingDirectory $AgentDir -ScriptBlock {
    # 強制 UTF-8 輸出編碼
    $OutputEncoding = [System.Text.Encoding]::UTF8
    [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
    
    # 執行含中文輸出的命令
    & claude -p --allowedTools "Read,Bash,Write" $using:prompt
}

ForEach-Object -Parallel 和 Start-ThreadJob 不受此問題影響：因為共享主程序的記憶體空間和編碼設定。

6. 記憶體管理與清理


# 黃金法則：完成後必須移除 Job
try {
    $jobs = 1..10 | ForEach-Object { Start-Job { ... } }
    $results = $jobs | Receive-Job -Wait -AutoRemoveJob  # 推薦：自動移除
} catch {
    # 確保清理
    Get-Job | Where-Object State -ne 'Running' | Remove-Job -Force
}

# ForEach-Object -Parallel 自動管理，無需手動清理

與本專案的關聯

daily-digest-prompt 的並行架構分析

本專案（daily-digest-prompt）大量使用 Start-Job 實現三種團隊並行模式：

| 模式 | 並行數 | 方法 | 原因 |

|------|--------|------|------|

| 每日摘要 Phase 1 | 5 路 | Start-Job | 啟動外部 claude -p CLI，本質上是外部程式 |

| Todoist Phase 2 | N 路（動態） | Start-Job | 每個子 Agent 獨立執行，需程序隔離 |

| 系統審查 Phase 1 | 4 路 | Start-Job | 獨立評估維度，需隔離避免交叉污染 |

為何選擇 Start-Job 而非更快的替代方案？

  1. 外部程式驅動：claude -p 是外部 CLI 程式，Start-Job 的程序隔離最適合此場景
  2. UTF-8 控制：每個 Job 可獨立設定 $OutputEncoding，確保中文輸出正確
  3. 故障隔離：一個 Agent 超時或崩潰不影響其他 Agent
  4. WorkingDirectory：PS 7 的 -WorkingDirectory 參數確保每個 Job 在正確的專案目錄下執行

可優化方向

  1. Phase 0 預檢查：Circuit Breaker 狀態檢查可改用 ForEach-Object -Parallel（I/O 密集，無需程序隔離）
  2. 快取批次讀取：多個快取檔案檢查可用 Start-ThreadJob 平行化
  3. 結果收集：Phase 完成後的 Receive-Job 可搭配 -AutoRemoveJob 簡化清理
  4. 動態 ThrottleLimit：Todoist Phase 2 的 N 路並行可根據系統負載動態調整

進階模式

同步雜湊表（Synchronized Hashtable）

用於跨執行緒安全共享狀態：


$syncHash = [hashtable]::Synchronized(@{
    Completed = 0
    Failed = 0
    Results = [System.Collections.Concurrent.ConcurrentBag[object]]::new()
})

$tasks | ForEach-Object -Parallel {
    $hash = $using:syncHash
    try {
        # 處理任務...
        $hash.Results.Add("OK: $_")
        [System.Threading.Interlocked]::Increment([ref]$hash.Completed)
    } catch {
        [System.Threading.Interlocked]::Increment([ref]$hash.Failed)
    }
}

Write-Host "Completed: $($syncHash.Completed), Failed: $($syncHash.Failed)"

Monitor 鎖（檔案寫入安全）


$lockObj = [object]::new()

$items | ForEach-Object -Parallel {
    $lock = $using:lockObj
    $result = "Processed: $_"
    
    [System.Threading.Monitor]::Enter($lock)
    try {
        Add-Content -Path 'output.log' -Value $result
    } finally {
        [System.Threading.Monitor]::Exit($lock)
    }
}
