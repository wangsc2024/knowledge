AI 驅動的架構優化與技術債檢測

技術概述

AI 驅動的架構優化結合了三大技術面向：架構適應度函數（Architecture Fitness Functions） 自動驗證架構特性、行為程式碼分析（Behavioral Code Analysis） 結合開發者活動模式辨識高影響技術債、以及 LLM 輔助重構 自動修正程式碼品質問題。2026 年，Forrester 預測超過 75% 技術決策者將面臨中度到重度技術債，而 AI 工具從被動掃描轉向主動預測與自動修正，形成「偵測→優先排序→自動修復→持續驗證」的閉環。Linux Foundation 2026 報告指出，高技術債組織維護成本增加最高達 65%，凸顯自動化管理的迫切需求。

核心方法與工具

1. 架構適應度函數（Architecture Fitness Functions）

源自 Neal Ford 與 Rebecca Parsons 的《Building Evolutionary Architectures》（第二版），適應度函數是任何驗證架構特性的自動化檢查，如同單元測試在 CI 中運行，失敗即表示架構偏移。

五大類型：

類型用途範例
依賴方向強制確保依賴從高層流向低層model 不可 import service
服務通訊邊界微服務間僅允許指定通訊api-gateway 只能呼叫 auth/user
資料庫存取模式限制 DB 操作僅在指定層僅 repositories 層可用 prisma/SQL
關鍵路徑延遲預算Runtime 驗證效能閾值POST /api/orders p95 < 500ms
元件大小限制防止 God Service每模組最多 50 檔案、400 行/檔

關鍵數據：純靠 Code Review 偵測架構偏移的成功率僅 17%（2,400 個 PR 調查），導入適應度函數後違規從年均 34 次降至 3 次。

2. 行為程式碼分析（CodeScene 方法論）

傳統靜態分析只標記壞程式碼，無法判斷哪些真正影響生產力。CodeScene 的行為分析結合：

  • Hotspot 偵測：高變更頻率 × 低 Code Health = 優先修復目標
  • CodeHealth 指標：聚合 25+ 情境因子成單一商業相關品質分數，經同行評審論文驗證具商業影響力
  • 開發者活動模式：分析版本控制歷史，揭露哪些程式碼消耗不成比例的工程時數

量化成效：導入 CodeScene 的團隊報告程式碼品質提升 30%、非計畫工作減少 50%。

3. LLM 輔助重構

LLM 在系統性、局部重構上表現優異（Magic Number 消除、Long Statement 拆分、Extract Method），但在跨模組、架構層級重構上仍需人工監督。

擅長領域（2026 現況）：

  • 跨多檔案的自動任務，保留語義完整性
  • 命名慣例統一、程式碼風格規範化
  • 已知模式的安全性修復

限制領域：

  • 需跨類別推理的架構重構
  • 領域邏輯相關的設計決策
  • 需業務脈絡理解的遷移

實際工作流程（Step-by-Step）

流程 A：架構適應度函數導入（4 個 Sprint）

Sprint 1：架構文件化 + 依賴檢查

  1. 與團隊記錄目前預期的架構規則（依賴方向、服務邊界）
  2. 撰寫依賴方向適應度函數
  3. 加入 CI Pipeline 的 PR 檢查

Sprint 2：邊界與存取檢查

  1. 新增服務通訊邊界函數
  2. 新增資料庫存取模式函數
  3. 修復基線違規

Sprint 3：效能與元件大小

  1. 整合延遲預算（需 Runtime 數據收集）
  2. 設定元件大小限制
  3. 建立儀表板追蹤趨勢

Sprint 4：文化嵌入

  1. 將適應度函數納入 RFC 審查流程
  2. 加入 Definition of Done
  3. 定期檢討閾值與規則

流程 B：CodeScene 技術債優先排序

  1. 連接版本控制：整合 Git 倉庫
  2. 自動分析：系統掃描程式碼 + 分析變更歷史
  3. Hotspot 視覺化：檢視系統地圖，辨識高頻變更 × 低品質的交集
  4. ROI 計算：CodeHealth 將技術債轉為財務數字
  5. 設定品質閘門：在 PR Review 中自動檢查 CodeHealth
  6. ACE 即時修復：IDE 中接收 AI 重構建議

流程 C：LLM 輔助重構管線

  1. 靜態分析掃描：識別 Code Smells 與複雜度熱點
  2. LLM Prompt 構建：提供程式碼上下文 + 重構目標
  3. 自動重構：LLM 生成重構建議
  4. 驗證閘門：自動測試 + 適應度函數 + 人工審查
  5. 合併與監控：持續追蹤重構後的品質趨勢

適應度函數範例（TypeScript）

// 依賴方向適應度函數
const TIERS = {
  'controllers': 3,
  'services': 2,
  'repositories': 1,
  'models': 0
};

describe('Architecture Fitness: Dependency Direction', () => {
  it('should not import from higher tiers', () => {
    const files = glob.sync('src/**/*.ts');
    files.forEach(file => {
      const tier = getTier(file);
      const imports = getImports(file);
      imports.forEach(imp => {
        const impTier = getTier(imp);
        expect(impTier).toBeLessThanOrEqual(tier,
          `${file} (tier ${tier}) imports ${imp} (tier ${impTier})`);
      });
    });
  });
});

服務邊界適應度函數

const ALLOWED_COMMUNICATIONS = {
  'api-gateway': ['auth-service', 'user-service'],
  'auth-service': ['user-service'],
  'user-service': [],  // 葉節點
};

describe('Architecture Fitness: Service Boundaries', () => {
  it('should only communicate with allowed services', () => {
    Object.entries(ALLOWED_COMMUNICATIONS).forEach(
      ([service, allowed]) => {
        const calls = getServiceCalls(service);
        calls.forEach(target => {
          expect(allowed).toContain(target,
            `${service} calls ${target} which is not allowed`);
        });
      });
  });
});

CI/CD Pipeline 整合（GitHub Actions）

name: Architecture Fitness
on:
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * *'

jobs:
  static-fitness:
    name: Static Fitness Functions
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:fitness:static

  runtime-fitness:
    name: Runtime Fitness Functions
    if: github.event_name == 'schedule'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run test:fitness:runtime

與本專案的應用場景（daily-digest-prompt）

1. Hook 系統的適應度函數

本專案的 hooks/ 目錄已有明確的架構規則。可建立適應度函數驗證：

  • 依賴方向：hooks 不可 import 非 hook_utils 的內部模組
  • 規則一致性：hook-rules.yaml 與實際 hook 程式碼的規則數量一致
  • 配置 Schema：validate_config.py 覆蓋所有 YAML 檔案

2. CodeHealth 啟發的品質追蹤

結合現有的 post_tool_logger.py JSONL 日誌，可計算：

  • Hotspot 分析：哪些 Skill/Hook 檔案最常被修改且品質最低
  • 變更頻率 × 複雜度：辨識需要優先重構的區域

3. 技術債自動偵測

  • validate_config.py 已涵蓋 13 個 YAML Schema 驗證，可擴展為技術債偵測器
  • check-health.ps1 可新增「架構健康度」維度
  • system-audit Skill 的 7 維度評分可整合適應度函數結果

4. LLM 重構應用

  • 現有 496 個測試提供重構安全網
  • cjk_guard.py 的 post-fix 模式已是 LLM 輸出自動修正的實例
  • 可擴展為更通用的 LLM 輸出品質閘門

風險與注意事項

  1. AI 生成程式碼的技術債悖論：GitClear 發現 AI 工具導致重複程式碼區塊增加 8 倍，重構程式碼比例顯著下降
  2. 適應度函數維護成本：規則本身也需要維護，過於嚴格會阻礙創新
  3. LLM 幻覺風險：LLM 可能生成看似合理但不正確的架構建議，必須有驗證機制
  4. 行為分析的隱私考量：分析開發者活動模式可能引發隱私疑慮
  5. 工具導入成本：企業級工具需授權費用，開源替代方案功能較有限
  6. 組織文化：TechDebt 2026 研討會強調工具無法取代組織對技術債管理的認同與協作
