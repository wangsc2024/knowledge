AI-Assisted Code Review 完整指南

技術概述

AI 程式碼審查（AI Code Review）是利用大型語言模型（LLM）與靜態分析技術，自動化程式碼品質檢查、安全漏洞偵測與 Pull Request 審查的系統化實踐。2025-2026 年間，這個領域經歷了從「輔助建議」到「自主修復」的根本轉變：Google DeepMind 的 CodeMender 能自主修補開源漏洞；Datadog 用 LLM 過濾 SAST 誤報；CodeRabbit、Qodo 等工具將 AI 審查深度嵌入 PR 工作流。核心價值在於將安全審查從「事後追趕」提升為「開發內建」，同時大幅縮短審查週期。

二、三大技術路線

#### 路線 1：LLM 原生審查（CodeRabbit / Qodo）

  • 直接用 LLM 分析 PR diff，生成結構化審查意見
  • 優勢：理解業務邏輯、偵測架構問題、自然語言互動
  • 局限：可能產生幻覺、對大型 diff 上下文受限

#### 路線 2：Neuro-Symbolic 混合（IRIS / LSAST）

  • LLM + 傳統靜態分析結合
  • IRIS：GPT-4 + 靜態分析偵測 120 漏洞中的 69 個（傳統工具僅 27 個）
  • LSAST：本地 LLM + 知識檢索系統，兼顧隱私與即時性
  • SAST 交叉驗證：96.9% 偵測率，回收 47% 基線遺漏

#### 路線 3：自主安全 Agent（CodeMender）

  • 多 Agent 系統 + Gemini Deep Think 推理
  • 工具鏈：靜態分析 + 動態分析 + 差異測試 + Fuzzing + SMT 求解器
  • 兩種模式：反應式（修補已知漏洞）+ 主動式（重寫不安全程式碼）
  • 成果：6 個月內對開源專案提交 72 個安全修復

工作流 A：PR 階段自動審查（推薦入門）


# .github/workflows/ai-code-review.yml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: CodeRabbit Review
        uses: coderabbitai/ai-pr-reviewer@v1
        with:
          openai_api_key: ${{ secrets.OPENAI_API_KEY }}
          review_comment_lgtm: false
          path_filters: |
            !*.md
            !*.json

流程說明：

  1. 開發者推送 PR → GitHub Actions 觸發
  2. AI 工具分析 diff → 生成 walkthrough 摘要
  3. 逐檔案留下審查意見（bug、安全、風格）
  4. 開發者在 PR 留言中與 AI 對話釐清
  5. 修正後重新觸發驗證 → 合併

def build_code_context(file_path: str, line_number: int, window: int = 50) -> str:
    """擷取漏洞周圍的程式碼上下文。"""
    lines = Path(file_path).read_text().splitlines()
    start = max(0, line_number - window)
    end = min(len(lines), line_number + window)
    return "\n".join(f"{i+1}: {l}" for i, l in enumerate(lines[start:end]))

風險與注意事項

技術風險

  1. LLM 幻覺：AI 可能誤報安全漏洞或提供錯誤修復建議，必須保留人工審查環節
  2. 上下文窗口限制：大型 PR（> 1000 行 diff）可能超出 LLM 上下文，需分批處理
  3. 程式碼隱私：雲端 AI 審查工具需將程式碼傳送至外部伺服器，機敏專案需選擇自託管方案（Qodo、LSAST 支援）

流程風險

  1. 過度依賴自動化：開發者可能因 AI 審查而降低人工審查品質，需明確「AI 初審 + 人工終審」的分工
  2. 規則漂移：AI 學習的編碼風格可能與團隊規範偏離，需定期校準 Living Rules
  3. 誤報疲勞：初期未調校的 AI 審查可能產生大量低價值意見，需逐步調整敏感度

導入建議

  • Phase 1：先從 PR 摘要生成開始（低風險、高可見性）
  • Phase 2：加入安全掃描（SAST + LLM 過濾）
  • Phase 3：啟用自動修復建議（需人工確認）
  • Phase 4：建立組織級 Living Rules（跨專案一致性）
