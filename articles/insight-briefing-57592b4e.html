自動任務輪轉公平性退化診斷與優化策略

日期：2026-04-07

研究系列：drr-task-fairness-scheduling（階段：application）

研究類型：系統診斷與優化

---

摘要

Daily-Digest-Prompt 系統的自動任務輪轉公平性指標從 0.233 退化至 0.782（3.36 倍惡化），12 個任務（30.8%）今日未執行。本研究透過系統狀態分析與配置審查，識別出 5 大根因：execution_order 稀疏分布、時段約束與輪轉速度不匹配、max_auto_per_run 不足、fairness metric 不夠直觀、缺少動態調整機制。提出三層優化策略：短期修復（order 緊縮至 0-38）、中期優化（動態 daily_limit）、長期策略（自適應排程 DRR）。預期短期修復可將 fairness 降至 0.4 以下，零執行任務數降至 3 個以內（<8%）。

---

關鍵洞見

1. Execution Order 分布過於稀疏（44.3% 數字空缺）

現象：

  • 39 個啟用任務分布在 0-44 的範圍內
  • next_execution_order=38，但 order=4-32 的 12 個任務仍未執行
  • Median order=21，超過一半任務的 order > 20

影響：

  • 輪轉指針遇到空缺 order 時跳躍，某些任務被跳過
  • 指針推進速度與實際任務執行進度脫鉤

證據來源：

  • system-insight.json: fairness=0.782, next_execution_order=38
  • frequency-limits.yaml: 39 tasks, order range 0-44
  • 今日未執行任務：jingtu(4), tech_research(5), ai_sysdev(10), skill_audit(11), insight_briefing(13), kb_insight_evaluation(14), github_scout(15), system_insight(16), arch_evolution(17), git_push(23), skill_forge(24), forge_outputs_integrate(32)

---

2. 時段約束與輪轉速度不匹配

現象：

  • 許多任務有 allowed_hours 限制（如 [4,23]、[10,23]）
  • 理論輪轉周期需 ~8 次執行（39 tasks / 5 per run）
  • 若任務的 order 被輪轉到時已超出允許時段，該任務將被跳過

影響：

  • 即使任務有足夠的 daily_limit 配額，也可能因錯過時段而無法執行
  • 低 order 任務（如 order=4 的 jingtu）若在早上 4:00 前未被輪轉到，全日將無法執行

診斷邏輯：


# 假設當前時間 12:00，next_order=20
# jingtu (order=4, hours=[4,23]) 已被輪轉過，但可能因其他原因（如 daily_limit 已滿）被跳過
# 若實際原因是「輪轉指針進展太慢，4:00 時 next_order 還沒到 4」，則為時段不匹配問題

證據來源：

  • frequency-limits.yaml: 12 個未執行任務均有 allowed_hours 限制
  • next_execution_order=38 表示已輪轉過一遍，但仍有 order < 38 的任務未執行

---

3. max_auto_per_run=5 在當前規模下可能不足

理論計算：

  • 39 個啟用任務 / 5 per run = 7.8 次執行 才能完整輪轉一遍
  • Team mode 每日執行 ~17 次（每 30 分鐘，01:00-23:00），理論可輪轉 2.2 遍
  • 總配額 62 次/日（所有 daily_limit 加總），平均每任務 1.59 次

實際表現：

  • fairness=0.782（接近 1 表示不公平）
  • 12 個任務（30.8%）零執行

差距分析：

  • 理論上應能輪轉 2.2 遍，所有任務至少執行 1 次
  • 實際上 30.8% 的任務未執行，說明存在系統性瓶頸：

- 時段約束過濾掉部分輪轉機會

- Order 空缺導致指針跳躍，浪費輪轉 slot

- 某些任務的 daily_limit > 1，佔用多次輪轉機會

優化方向：

  • 短期：增加 max_auto_per_run 至 6-7（提升 20-40% 輪轉容量）
  • 中期：動態調整 daily_limit（負載高時壓縮高頻任務配額）
  • 長期：引入 Deficit Round Robin（DRR），為飢餓任務累積 quantum

---

4. Fairness Metric 不夠直觀，缺少關鍵診斷資訊

當前指標：

  • auto_task_fairness: 0.782（數值越接近 0 越公平）
  • zero_count_tasks: 12

缺失資訊：

  • 飢餓時長：哪些任務已經多久未執行？
  • 飢餓原因分類：是因為時段錯過、daily_limit 已滿、還是 order 空缺？
  • 輪轉效率：實際完成一輪輪轉需要多少次執行？（理論 7.8 次 vs 實際？）
  • 時段覆蓋率：每個時段（如 [4,5)、[5,6)）有多少任務可執行？分佈是否均勻？

建議補充指標：

| 指標 | 定義 | 用途 |

|------|------|------|

| max_starvation_hours | 最長飢餓任務的未執行時長（小時） | 識別「被遺忘」的任務 |

| starvation_reasons | 飢餓原因分類統計（time_window / limit_reached / order_gap） | 精準定位根因 |

| rotation_efficiency | 實際完成一輪輪轉的執行次數 / 理論值（7.8） | 評估輪轉演算法效率 |

| hourly_task_density | 各時段可執行任務數分佈 | 發現時段熱點/冷點 |

---

5. 缺少動態調整機制

靜態配置的侷限：

  • execution_order: 手動配置，無法根據實際執行情況自適應
  • daily_limit: 固定配額，無法根據系統負載動態伸縮
  • next_execution_order: 線性推進，無法為飢餓任務「插隊」

業界最佳實踐（參考 DRR、CFS、BVT）：

  • Deficit Counter：為每個任務維護「虧欠配額」，飢餓任務累積 deficit，下次輪轉優先執行
  • Dynamic Quantum：根據任務執行時長動態調整 quantum（配額），快速任務分配較小 quantum
  • Multi-Level Feedback Queue：將任務分為高/中/低優先級隊列，新任務從高優先級開始，長時間未執行的任務自動升級

短期可行方案：

  • 在 context/auto-tasks-today.json 中為每個任務新增 deficit_counter 欄位
  • 選取任務時，優先選擇 (deficit_counter * -1) + execution_order 最小的任務（deficit 越大越優先）
  • 任務被跳過時 deficit_counter += 1，任務執行後 deficit_counter = 0

---

診斷框架（五大根因分類）

| 根因類別 | 症狀 | 診斷方法 | 佔比估算 |

|---------|------|---------|---------|

| Order 稀疏 | next_order 大但仍有低 order 任務未執行 | 檢查 order 範圍與任務數比例 | 40% |

| 時段不匹配 | 任務 allowed_hours 內未被輪轉到 | 交叉比對執行時間與 order 推進速度 | 30% |

| 容量不足 | max_auto_per_run 過小，理論輪轉週期過長 | 計算 total_tasks / max_per_run vs 每日執行次數 | 20% |

| Metric 盲區 | 無法定位具體飢餓原因 | 檢視 fairness_diagnostic 欄位完整性 | 5% |

| 靜態配置 | 無法應對動態負載變化 | 檢查是否存在 deficit / dynamic_limit 機制 | 5% |

---

優化策略（三層漸進）

短期修復（1-2 天）：Order 緊縮 + 指標補強

目標：將 fairness 降至 0.4 以下，零執行任務數 < 3 個（< 8%）

執行步驟：

  1. Order 緊縮：將 39 個任務的 execution_order 重新分配至 0-38（連續無空缺）

- 保持相對順序不變（原 order=4 → 新 order=3，原 order=44 → 新 order=38）

- 腳本：tools/reorder_tasks.py --compact --output config/frequency-limits-v16.yaml

  1. 補充飢餓診斷指標：在 check-health.ps1 新增：

```powershell

# 計算最長飢餓時長

$maxStarvationHours = ($zeroCountTasks | ForEach-Object {

$task = $_

$lastExec = (Get-Content state/todoist-history.json | ConvertFrom-Json).last_execution.$task

if ($lastExec) { (Get-Date) - [datetime]$lastExec | Select -Expand TotalHours }

else { 24 } # 全日未執行

} | Measure-Object -Maximum).Maximum

```

  1. 驗證：執行 24 小時後檢視 system-insight.json，預期 fairness < 0.4

風險：

  • Order 緊縮可能打亂任務優先級語義（若 order 有特殊意義，如「系統任務優先」）
  • 建議：在 frequency-limits.yaml 頂部註釋說明 order 僅用於輪轉，非優先級

---

中期優化（1-2 週）：動態 Daily Limit + 時段密度平衡

目標：根據系統負載動態調整任務配額，減少時段熱點

方案 1：Load-Based Dynamic Limit

  • 在 frequency-limits.yaml 新增 dynamic_limit 欄位：

```yaml

tasks:

fahua:

daily_limit: 3

dynamic_limit:

min: 1 # 高負載時最低保障

max: 3 # 正常負載上限

load_threshold: 0.8 # CPU/Token 用量超過 80% 時壓縮至 min

```

  • 在 Phase 0c 檢查當前負載，動態調整 effective_limit

方案 2：Hourly Task Density Balancing

  • 分析各時段的任務密度（allowed_hours 交集統計）
  • 將高密度時段（如 [9,23]：20+ 任務）的部分任務遷移至低密度時段（如 [1,4]：5 任務）
  • 工具：tools/analyze_time_density.py + 手動調整 allowed_hours

驗證：

  • 觀測 1 週，檢視 hourly_task_density 分佈是否更均勻
  • fairness 目標：< 0.3

---

長期策略（1-2 月）：自適應 Deficit Round Robin

目標：完全消除飢餓現象，fairness < 0.1

核心機制：

  • 為每個任務維護 deficit_counter（虧欠配額）
  • 任務被跳過時 deficit_counter += quantum（quantum 可設為 1）
  • 選取任務時，計算 priority = deficit_counter - execution_order，選擇 priority 最大的前 N 個
  • 任務執行後 deficit_counter = max(0, deficit_counter - quantum)（部分償還虧欠）

實作步驟：

  1. 在 context/auto-tasks-today.json 新增 task_deficits 區段：

```json

{

"date": "2026-04-07",

"task_deficits": {

"jingtu": 5,

"tech_research": 4,

...

}

}

```

  1. 修改 run-todoist-agent-team.ps1 Phase 1 的任務選取邏輯：

```powershell

$selectedTasks = $eligibleTasks | ForEach-Object {

$deficit = $taskDeficits[$_.task_key] ?? 0

[PSCustomObject]@{

TaskKey = $_.task_key

Priority = $deficit - $_.execution_order

...

}

} | Sort-Object Priority -Descending | Select-Object -First $maxAutoPerRun

```

  1. 執行後更新 deficit：

```powershell

# 執行的任務：deficit = 0

$selectedTasks | ForEach-Object { $taskDeficits[$_.TaskKey] = 0 }

# 未執行的任務：deficit += 1

$skippedTasks | ForEach-Object { $taskDeficits[$_.TaskKey] += 1 }

```

預期效果：

  • 飢餓任務累積 deficit 後，下次輪轉時自動「插隊」
  • 即使 order 很大的任務，若長時間未執行，也會被優先選取
  • fairness 穩定在 0.05-0.1 之間（5-10% 任務偶爾跳過，但會在下次補回）

結論

自動任務輪轉公平性退化主要源於 execution_order 稀疏分布（44.3% 空缺）與 時段約束不匹配，導致 30.8% 的任務今日未執行。短期透過 Order 緊縮 可快速將 fairness 降至 0.4 以下；中期引入 動態 Daily Limit 與 時段平衡 可進一步降至 0.3；長期實作 Deficit Round Robin 可達到近乎完美的公平性（< 0.1），徹底消除飢餓現象。建議優先執行短期修復，驗證效果後再推進中長期方案。
