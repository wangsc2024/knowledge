專案概述

OpenHands（前身 OpenDevin）是 All-Hands AI 開發的開源 AI 軟體工程 Agent 平台，2024 年初以社群驅動方式響應 Cognition Devin 發布，現已成長為 GitHub 上最受歡迎的自架 AI 代碼 Agent（72.1k stars，490+ contributors）。

其核心定位不是「AI 補全工具」，而是可自主完成完整工程任務的 AI 開發者：克隆倉庫、閱讀文件、編寫程式碼、執行測試、提交 PR，並能在過程中瀏覽網頁取得外部知識，全部在隔離沙箱中完成。與 GitHub Copilot 等補全工具最本質的差異是「執行完整任務」而非「建議程式碼片段」[來源：GitHub README, 2026]。

2026 年 3 月完成 $18.8M Series A 融資，MLSys 2026 論文以 Oral + Poster 雙軌呈現 V1 架構重設計，代表學術與工程的雙重認可[來源：MLSys 2026 Oral 3749]。

技術架構

V1 四層 SDK 模組

（來源：arxiv 2511.03690 Wang et al.、docs.openhands.dev）

V1 的最核心創新是將「Agent 邏輯」與「執行環境」完全解耦，形成四層 SDK：

| 套件 | 職責 |

|------|------|

| openhands.sdk | 核心抽象（Agent、Conversation、LLM、Tool、MCP） |

| openhands.tools | 具體工具實作（bash、file_editor、browser 等） |

| openhands.workspace | 執行環境（Local 用 subprocess、Remote 用 HTTP API） |

| openhands.agent_server | REST/WebSocket API 伺服器，即時事件串流 |

這種分層讓「本地開發」與「千台雲端 Agent 部署」共用同一套代碼庫，僅切換 Workspace 實作即可[來源：arxiv 2511.03690]。

事件溯源狀態模型（Event-Sourced State）

V1 的狀態管理採用函數式事件溯源模式。所有互動被視為「不可變事件」追加到 EventLog，ConversationState 是唯一可變實體，支援確定性重播（deterministic replay）和可靠的 session 恢復。

事件層次結構分三層：

  • Event（基底，所有事件的父類）
  • LLMConvertibleEvent（LLM 可見，含 ActionEvent 和 ObservationBaseEvent）
  • 內部書記事件（不進入 LLM context）

這個設計解決 V0 的核心問題：本地/遠端實作重複導致行為分歧。V1 之後本地與遠端行為完全對稱[來源：arxiv 2511.03690, MLSys 2026]。

工具系統（Action-Observation Pattern）

（來源：docs.openhands.dev/sdk/arch/tool-system）

工具系統四大職責：

  1. Type Safety：Pydantic 模型強制輸入/輸出 schema
  2. Schema Generation：自動生成 LLM 相容的 OpenAI 工具描述
  3. Execution Lifecycle：validate inputs → execute → wrap outputs
  4. Tool Registry：動態發現工具（Instance / Subclass / Factory 三種注冊模式）

工具定義兩種模式：

  • 直接實例化（無狀態工具，如 finish、think）
  • 子類工廠模式（有狀態工具，如 execute_bash、file_editor，透過 @classmethod create() 工廠方法配置 session 狀態）

MCP 整合（一等公民）

MCP 工具不是外掛，與原生工具等同對待。MCPToolDefinition 繼承標準 ToolDefinition，MCPToolExecutor 委派給 FastMCP 的 MCPClient，JSON Schemas 自動翻譯為 Action 模型。透過 Background Event Loop 模式橋接同步呼叫與非同步 MCP 協議[來源：docs.openhands.dev, arxiv 2511.03690]。

V0 vs V1 核心差異

| 面向 | V0（舊版） | V1（現版） |

|------|-----------|----------|

| 架構 | 單體、高耦合 | 四層模組分離 |

| 配置 | 140+ 欄位、15 個類 | ConversationState 單一可變來源 |

| 沙箱 | 硬編碼、強制容器化 | 可選（預設本地，需要才容器化） |

| 本地/遠端 | 重複實作，行為分歧 | Workspace 抽象統一 |

生產數據：V1 substantially reduces system-attributable failures，event-sourcing overhead 可忽略[來源：arxiv 2511.03690]。

功能特色

OpenHands vs 主要競品對比[來源：toolhalla.ai, vibecoding.app]：

| 功能 | OpenHands | Devin | SWE-agent |

|------|-----------|-------|-----------|

| 授權 | MIT（免費） | 專有 | MIT（免費） |

| 定價 | 自架免費 | $20-$500/月 | 自架免費 |

| SWE-bench Verified | 72%（Claude 4） | ~50% | 74%+ |

| 模型靈活性 | 任意 LLM | 鎖定私有模型 | 任意 LLM |

| 多 Agent | 支援 | 不支援 | 不支援 |

| 介面 | Web UI + VS Code | Web App + Slack | CLI 限定 |

| 自架 | 預設即可 | 僅 Enterprise | 預設即可 |

OpenHands 三大差異化：

  1. 模型靈活性：可自由切換 Claude 4 Opus、GPT-5、Gemini、Llama，或透過 Ollama 跑本地模型達到零邊際成本。Devin 鎖定私有模型，在 Claude 4 超越時無法受益[來源：toolhalla.ai]。
  2. 多 Agent 架構：唯一支援多 Agent 委派的開源 Coding Agent，可將複雜任務拆解給子 Agent 並行執行[來源：toolhalla.ai]。
  3. 完整開源 + 企業就緒：MIT 授權免費自架，同時提供 Kubernetes、RBAC、Jira/Linear/Slack 整合[來源：GitHub README]。

使用方式

快速啟動（Docker，約 10 分鐘）：


docker run -it --rm \
  -e SANDBOX_RUNTIME_CONTAINER_IMAGE=docker.all-hands.dev/all-hands-ai/runtime:0.45-nikolaik \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v ~/.openhands-state:/.openhands-state \
  -p 3000:3000 \
  --add-host host.docker.internal:host-gateway \
  docker.all-hands.dev/all-hands-ai/openhands:0.45

啟動後訪問 localhost:3000，輸入 LLM API Key，對話框輸入任務描述即可。

SDK 程式化使用：


from openhands.sdk import Agent, Conversation, LocalWorkspace

workspace = LocalWorkspace(path="./my_project")
conversation = Conversation(workspace=workspace)
agent = Agent(model="claude-opus-4-6", tools=["bash", "file_editor"])
result = await agent.run("Fix the failing test in tests/test_api.py", conversation)

v1.6.0（2026-03-30）新增 Kubernetes 支援和 Planning Mode beta，讓 Agent 先規劃再執行，有效減少迴圈問題。

潛在應用

場景 1：自動化 GitHub Issue 修復閉環

daily-digest-prompt 有大量 state/*.json 維護和 hook 規則修復任務，目前是人工介入。可將 OpenHands 接入 GitHub Issues，讓 Agent 自動判斷 P2/P3 等級 bug 是否有明確複現步驟，若有則自動修復並提 PR。

落地步驟：

  1. 用 docker run 部署 OpenHands（本地沙箱，代碼不外送）
  2. 配置 GitHub App 權限，webhook 觸發 Issue 事件
  3. 設定 Agent 門檻：僅處理有完整錯誤訊息 + 具體文件路徑的 issue
  4. 生成 PR 後人工審核，迭代提升 Agent 的 PR 品質

場景 2：prompt 批量重構自動化

當 prompts/team/todoist-auto-*.md 需要大規模升版（例如批量加入 preamble 守衛規則），可讓 OpenHands SDK 批量讀取、修改、驗證 prompt 檔，配合 hooks/validate_config.py 自動測試，整個流程在沙箱內完成。

場景 3：研究 → KB → PR 自動閉環

現有研究流程：LLM 生成報告 → 手動寫入 KB。OpenHands 的 GitHub Integration（Issue-to-PR workflow）可作為執行層，Agent 負責 git 操作、格式標準化（frontmatter、tags）、PR 描述撰寫，直接整合 tools/archive_writer.py 現有能力。

優缺點評估

優點

  1. 開源免費，企業就緒：MIT 授權無授權費，同時提供 Kubernetes / RBAC / 多用戶管理[來源：toolhalla.ai, GitHub README]
  2. 開源最高 SWE-bench 得分：72-72.8%（Claude 4 / Sonnet 4.5），比 Devin（~50%）領先 22 個百分點；V1 substantially reduces system-attributable failures[來源：arxiv 2511.03690, toolhalla.ai]
  3. 模型自由切換：任意 LLM 提供商（含本地 Ollama），零邊際成本可能[來源：vibecoding.app]
  4. 多 Agent 委派：唯一支援多 Agent 任務委派的開源 Coding Agent[來源：toolhalla.ai]

缺點

  1. Docker 依賴高：需 Docker-in-Docker 存取 Docker socket，企業受控環境常遇阻[來源：vibecoding.app]
  2. 資源消耗大：每個 session 4-8GB RAM，每個任務 LLM 成本 $0.15-$0.60[來源：vibecoding.app]
  3. 迴圈行為問題：Agent 有時重複同一失敗方法，需手動干預[來源：vibecoding.app, community reviews]
  4. 前端/視覺能力弱：React UI 和 CSS 佈局生成不可靠[來源：vibecoding.app]

不適用情境

  • 純前端/UI 設計驅動的專案
  • 無 DevOps 能力的初學者團隊（學習曲線陡）
  • 企業不允許容器化環境的安全合規場景
  • 對即插即用有強需求（Devin 更合適）
