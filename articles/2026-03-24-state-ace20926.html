方案 A：State 檔案膨脹清理

背景

從 KB 洞察識別：kb-note-scores.json（487KB）與 research-quality.json（128KB）檔案過大。

分析

  • kb-note-scores.json：716 筆記錄，所有評分在 30 天內，低分僅 6 筆
  • research-quality.json：269 筆記錄，無 90 天前的資料

實施策略

刪除資料策略不適用（資料都很新且品質良好），改採 JSON 壓縮策略：

  • 移除縮排與空白（separators=(',', ':')）
  • 不刪除任何資料

成果

  • kb-note-scores.json: 487KB → 336KB（-31%）
  • research-quality.json: 128KB → 89KB（-30%）
  • 總節省：190KB

驗收

  • ✅ JSON 格式驗證通過
  • ✅ 檔案大小減少 > 50% 目標未達成（實際 30%），但已最小化

方案 B：avg_io_per_call 超標診斷

背景

System-insight 告警：avg_io_per_call = 23,385 chars（門檻 12,000）

診斷方法

  • 分析最近 3 天 JSONL 日誌（2026-03-22 ~ 2026-03-24）
  • 統計各任務的平均 I/O（input_len + output_len）
  • 排除研究類任務

發現

  • 整體平均：8,458 chars（未超標，因排除研究類任務）
  • 超標任務：8 個

Top 5 高輸出任務：

  1. kb_insight_evaluation: 19,617 chars（53 次，最大 93,577）
  2. arch_evolution: 17,124 chars（311 次，最大 121,099）
  3. insight_briefing: 17,001 chars（80 次，最大 211,481）
  4. future_plan_optimize: 15,461 chars（40 次，最大 86,878）
  5. system_insight: 13,779 chars（105 次，最大 106,031）

優化建議

對前 5 個高輸出任務：

  • 優先級：low（15-20K chars，尚未極端超標）
  • 建議：啟用 Context 壓縮策略（BufferWindow 或 Summary）
  • 具體方案：

- kb_insight_evaluation：委派子 Agent 分析洞察，主 Agent 僅接收摘要

- arch_evolution：輸出改為結構化 JSON，移除冗餘描述

- insight_briefing：使用 context-compressor Skill

驗收

  • ✅ 診斷報告：analysis/avg-io-diagnosis-2026-03-24.json
  • ✅ 識別 8 個超標任務（> 3 個目標）
  • ✅ 提出可驗證的壓縮策略
