Circuit Breaker、Bulkhead 與 Retry Budget 在 AI Agent 系統的工程實踐

一、執行摘要（300 字）

傳統微服務的三大韌性模式——Circuit Breaker、Bulkhead、Retry Budget——在 AI Agent 系統中需要根本性重新設計。核心挑戰在於：LLM 的「失敗」不僅是 HTTP 錯誤碼，更包括語義降級（幻覺、品質下滑）等「看似成功的失敗」。2025-2026 年的產業實踐顯示，純粹的 Harness 約束防線需搭配漸進式降級鏈，才能在失敗時維持部分功能而非全面停擺。本報告整合 Portkey（LLM Gateway）、Athenic（Agent 重試策略）、agent-engineering.dev（Harness 生產實測）、GoCodeo（Agent 錯誤恢復）、harness-engineering.ai（Agent Harness 六組件）及 Rentier Digital（8 個月生產經驗）等 6 個獨立來源，提出「三層韌性模型」：Layer 1 重試預算控制（防止 Retry Storm）、Layer 2 Circuit Breaker 語義擴展（偵測品質降級）、Layer 3 Bulkhead 工具隔離（限制爆炸半徑）。同時針對 daily-digest-prompt 系統提出 5 項具體優化建議，預期可將端到端任務完成率從現有基線提升 15-20%。

---

二、研究背景

2.1 傳統韌性模式的 Agent 挑戰

微服務韌性三支柱在 Agent 系統面臨獨特挑戰：

| 模式 | 傳統用途 | Agent 系統新挑戰 |

|------|---------|----------------|

| Circuit Breaker | HTTP 錯誤率監控 | LLM 幻覺是 200 OK + valid JSON，傳統斷路器無法偵測 [Resilience Circuit Breakers for Agentic AI, 2026] |

| Bulkhead | 執行緒池/連接池隔離 | Agent 工具呼叫的資源消耗不可預測（Token 用量波動 10x） |

| Retry Budget | 固定重試次數 | 重試 LLM 呼叫可能產生完全不同結果，非等冪操作 |

2.2 Agent 特有的失敗模式

根據多來源交叉驗證，Agent 系統有五類主要失敗 [GoCodeo, 2026; Galileo, 2026]：

  1. 執行層錯誤：工具呼叫失敗（API timeout、資料庫連線中斷）
  2. 語義錯誤：語法正確但邏輯錯誤的 LLM 輸出（幻覺 API 呼叫、畸形查詢）
  3. 狀態錯誤：Agent 認知與實際環境狀態偏離
  4. 超時與延遲：規劃迴圈卡住、處理耗時過長
  5. 依賴錯誤：外部服務不可用、速率限制

三、核心發現：三層韌性模型

3.1 Layer 1：Retry Budget（重試預算控制）✅已驗證

核心主張：無預算控制的重試是 Agent 系統最大的隱形成本殺手。

佐證來源：

  • Portkey：「規模化時，持續失敗觸發重試風暴，堆積請求、推高 Token 用量、拖慢整個系統」[Portkey Blog, 2026]
  • Athenic：提供具體參數基線——初始延遲 1000ms、最大延遲 60s、乘數 2x、最大重試 5 次、Jitter 因子 0.3 [Athenic Blog, 2026]
  • harness-engineering.ai：「20 步管線中每步 95% 成功率 = 整體僅 36% 完成率」[harness-engineering.ai, 2026]

工程實作要點：


重試預算公式：baseDelay = min(initialDelay × multiplier^attempt, maxDelay)
實際延遲 = baseDelay ± (jitterFactor × baseDelay)

錯誤分類與重試策略：

| 錯誤類型 | HTTP 狀態碼 | 處理方式 |

|---------|-----------|----------|

| 速率限制 | 429 | 遵守 Retry-After header |

| 伺服器錯誤 | 500/502/503/504 | 指數退避 |

| 模型過載 | 529 (Anthropic) | 延長退避（120s+） |

| 認證錯誤 | 401/403 | 不重試，修復憑證 |

| Context 溢出 | 400 | 不重試，切換模型 |

Jitter 的關鍵作用：「100 個客戶端同時遇到速率限制，沒有 Jitter 就會同時重試，造成另一波尖峰。Jitter 將重試分散到不同時間點」[Athenic, 2026]。

3.2 Layer 2：Circuit Breaker 語義擴展 ✅已驗證

核心主張：Agent 系統的 Circuit Breaker 必須超越 HTTP 錯誤碼，納入語義品質監控。

佐證來源：

  • dasroot.net：傳統三態模型（CLOSED→OPEN→HALF-OPEN）的 Agent 適配 [dasroot.net, 2026]
  • Athenic：具體配置——5 次失敗/60 秒窗口觸發 OPEN，30 秒後進入 HALF-OPEN，需 2 次連續成功恢復 [Athenic, 2026]
  • Portkey：「監控失敗請求數、時間內失敗率、特定失敗狀態碼（429/502/503）」[Portkey, 2026]

Agent 特有擴展——語義斷路器：

傳統 Circuit Breaker 只看 HTTP 狀態碼，但 Agent 最大的失敗是「幻覺回應」（200 OK + 自信的錯誤內容）。語義斷路器需額外監控：

  • Schema 驗證失敗率（Pydantic 驗證）
  • 輸出品質分數（confidence markers）
  • 重複動作偵測（Agent 重複相同操作無進展）

三態轉換配置建議：


CLOSED → 正常請求流
  ↓ 5 次失敗 / 60 秒窗口（含語義失敗）
OPEN → 立即拒絕（返回快取/降級回應）
  ↓ 30 秒冷卻
HALF-OPEN → 允許探測請求
  ↓ 2 次連續成功
CLOSED（回復）
  ↓ 1 次失敗
OPEN（重新開路）

3.3 Layer 3：Bulkhead 工具隔離 ✅已驗證

核心主張：Agent 工具應依任務階段隔離，防止單一工具失敗影響整體管線。

佐證來源：

  • harness-engineering.ai：「規劃步驟不需要檔案系統寫入權限。程式碼執行步驟不需要網路搜尋」[harness-engineering.ai, 2026]
  • GoCodeo：「模組化 Agent 路由：chain-of-responsibility 遞減複雜度——主推理 Agent → 帶模板的恢復 Agent → 規則式 fallback 或人工升級」[GoCodeo, 2026]
  • Rentier Digital：「12 個工具造成的決策延遲大於能力增益」；「工具 30 天未呼叫就刪除」[Rentier Digital, 2026]

工具隔離策略：

| 隔離層 | 機制 | 效果 |

|-------|------|------|

| Phase 隔離 | 每個執行階段只暴露必要工具 | 減少決策點，降低失敗面 |

| Token 隔離 | 每個工具池有獨立 Token 預算 | 防止單一工具耗盡全局預算 |

| 資源池隔離 | 即時互動與批次處理分開資源池 | 批次任務不影響即時回應 |

4.2 Fallback Chain 降級鏈架構

實務中最有效的降級模式是四級模型降級鏈 [Athenic, 2026]：


1. Primary（GPT-4o / Claude Opus，最高品質）
   ↓ 失敗或超時
2. Secondary（Claude Sonnet / GPT-4o-mini，平衡品質與成本）
   ↓ 失敗
3. Budget Model（Gemini Flash，超低成本）
   ↓ 失敗
4. Cache Fallback（1 小時內快取回應）
   ↓ 全部失敗
5. Graceful Error Message（上下文相關的友善提示）

五、實務應用建議：daily-digest-prompt 系統落地

基於上述分析，針對本專案（daily-digest-prompt）提出 5 項具體優化建議：

建議 1：統一重試預算配置（P0）

目前各 API 呼叫的重試邏輯分散在多個腳本中。建議在 config/retry-budget.yaml 集中管理：

  • 外部 API（Todoist/ntfy）：max_retries=3, backoff=exponential, jitter=0.3
  • 知識庫（localhost:3000）：max_retries=2, backoff=fixed(500ms)
  • LLM（Groq/Claude）：max_retries=2, backoff=exponential, respect Retry-After

建議 2：Circuit Breaker 整合至 Phase 執行（P1）

在 run-todoist-agent-team.ps1 的 Phase 1→2→3 之間加入斷路器邏輯：

  • Phase 1 失敗率 > 50% → 跳過 Phase 2 的外部 API 呼叫，使用快取
  • 連續 3 個 Agent 超時 → 暫停 5 分鐘再繼續
  • 語義驗證失敗（JSON schema 不合規）計入失敗計數

建議 3：工具 Phase 隔離（P1）

依 Harness B0-E 風險分級，對自動任務的工具權限做階段隔離：

  • 研究階段：僅 WebSearch + WebFetch + Read（禁止 Write/Edit）
  • 寫入階段：僅 Write + curl（禁止 WebSearch）
  • 驗證階段：僅 Read + Bash（唯讀驗證）

建議 4：Fallback Chain 降級策略（P2）

在 config/llm-router.yaml 加入降級鏈配置：

  • Groq Fast → Claude Haiku → 快取回應 → 友善錯誤訊息
  • 降級時自動調整 done_cert.quality_score（降一級 -1 分）

建議 5：季度工具清理機制（P2）

依 Rentier Digital 的 8 個月生產經驗，建立季度審查流程：

  • 30 天未呼叫的工具標記為候選刪除
  • 每季審查 Skill 使用率，精簡不常用 Skill
  • 配置熵分數（B3-B）作為清理觸發指標

六、知識缺口與未來方向

  1. 語義 Circuit Breaker 的具體實作：目前業界對 LLM 品質降級的自動偵測仍處早期，需要更多生產數據驗證閾值
  2. Retry Budget 與 Token Budget 的交互：重試消耗的 Token 如何與整體預算治理（B4-A budget_guard）整合
  3. 跨 Agent 的 Bulkhead 協調：Team 模式下多個 Agent 共享資源池時的隔離策略
