Pydantic v2 深度實踐：AI Agent 系統中的結構化驗證、設定管理與 LLM 輸出解析（2026）

技術概述

Pydantic v2 是 Python 生態中最廣泛使用的資料驗證與序列化函式庫。2023 年 6 月發布的 v2 版本以 Rust 重寫核心驗證引擎（pydantic-core），帶來 4～50 倍效能提升。[來源：Pydantic 官方文件，2026；GitHub Discussion #6748] 它解決三個關鍵問題：

問題 1：型別安全邊界。Python 的動態型別系統在系統邊界（HTTP API 輸入、LLM 輸出、檔案讀取）極易引入 AttributeError 與 KeyError。Pydantic 在邊界強制執行結構化驗證，通過即成為已知結構的 Python 物件。

問題 2：LLM 輸出解析。LLM 回應是非結構化文字或 JSON，Pydantic v2 是 Instructor、PydanticAI、LangChain、LlamaIndex 等 2026 年主流 Agent 框架的共同解析骨幹。[來源：jangwook.net，2026]

問題 3：設定管理。pydantic-settings 提供統一的多來源設定載入（環境變數 / .env 檔 / Kubernetes secrets / CLI），含型別驗證與優先順序管理。[來源：Pydantic Settings 官方文件，2026]

與替代方案的差異：marshmallow 效能差距 5-20 倍且語法繁瑣；attrs 輕量但無 JSON Schema 生成；Python dataclasses 無驗證語義。2026 年 AI 生態中 Pydantic v2 是結構化輸出的事實標準，PyPI 週下載量超過 2,000 萬次。[來源：pypi.org/project/pydantic，2026]

---

核心概念與原理

2.1 驗證引擎架構：Rust + Python API 雙層設計

Pydantic v2 的核心是 pydantic-core，用 Rust 實作驗證邏輯並透過 PyO3 暴露 Python API。驗證流程：Python 呼叫 BaseModel(**data) → Rust 端執行型別強制與驗證 → 回傳已驗證 Python 物件。[來源：Pydantic 官方文件 models，2026]

效能數據（官方 benchmark）：

  • 簡單模型驗證：v2 比 v1 快 5-17 倍
  • 巢狀模型驗證：v2 比 v1 快 8 倍
  • JSON 解析（model_validate_json()）：比 Python 層 JSON + 驗證更快，因為 Rust 直接解析 JSON bytes

關鍵設計原則：宣告式約束（Field(ge=0, le=100)）由 Rust 直接處理，效能最佳；Python 層驗證器（@field_validator）需要 Python↔Rust 的上下文切換，相對較慢。[來源：Pydantic performance docs，2026]

2.2 四種驗證模式（field_validator）

Pydantic v2 的 @field_validator 支援四種執行模式，對應不同的處理時機 [來源：Pydantic validators 官方文件，2026]：

| 模式 | 執行時機 | 輸入型別 | 效能 | 適用場景 |

|------|---------|---------|------|----------|

| after（預設） | Rust 型別強制後 | 已確認型別（如 int） | 最快 | 業務邏輯驗證、範圍檢查 |

| before | Rust 處理前 | Any（原始輸入） | 中 | 資料清洗、格式轉換 |

| plain | 替換 Pydantic 內部驗證 | Any | 快 | 完全自定義型別處理 |

| wrap | 包裝整個驗證流程 | Any + handler 函式 | 最慢 | 補救失敗驗證（如截斷超長字串） |


from pydantic import BaseModel, field_validator, ValidationInfo
from typing import Any

class TaskResult(BaseModel):
    status: str
    summary: str
    quality_score: int

    # after 模式：型別已確認，直接驗證語義
    @field_validator('status')
    @classmethod
    def valid_status(cls, v: str) -> str:
        allowed = {'success', 'failed', 'partial'}
        if v not in allowed:
            raise ValueError(f'status 需是 {allowed}')
        return v

    # before 模式：原始輸入可能是任意型別
    @field_validator('quality_score', mode='before')
    @classmethod
    def parse_score(cls, v: Any) -> Any:
        if isinstance(v, str) and v.isdigit():
            return int(v)  # '4' → 4
        return v

@field_validator 必須以 @classmethod 裝飾（v1 不需要，是高頻遷移錯誤）[來源：Pydantic validators docs，2026]。也可同時驗證多個欄位：@field_validator('f1', 'f2') 或用萬用字元 '*'。

2.3 `@model_validator` — 跨欄位一致性驗證

@model_validator 作用於整個模型，最常用的是 mode='after'（所有欄位驗證完成後執行，self 已是已型別確認的實例）[來源：Pydantic validators docs，2026]：


from pydantic import BaseModel, model_validator
from typing_extensions import Self

class AgentOutput(BaseModel):
    status: str
    error: str | None = None
    kb_imported: bool = False
    summary: str

    @model_validator(mode='after')
    def check_consistency(self) -> Self:
        # 規則 1：failed 時必須有 error
        if self.status == 'failed' and self.error is None:
            raise ValueError('status=failed 時 error 欄位不可為 None')
        # 規則 2：failed 時 kb_imported 不應為 True
        if self.status == 'failed' and self.kb_imported:
            raise ValueError('status=failed 但 kb_imported=True，邏輯矛盾')
        return self

mode='before' 則接受 cls + data: Any（原始 dict），可在任何欄位解析前攔截整個輸入：


    @model_validator(mode='before')
    @classmethod
    def strip_extra_prefix(cls, data: Any) -> Any:
        if isinstance(data, dict) and 'agent' in data:
            # 自動去掉 'todoist-auto-' 前綴（相容舊格式）
            data['agent'] = data['agent'].replace('todoist-auto-', '')
        return data

2.4 `@computed_field` — 計算屬性自動納入序列化

v2.0 新增的 @computed_field 讓 @property 自動出現在 model_dump() / model_dump_json() 輸出中，無需手動 override 序列化邏輯 [來源：Pydantic models docs，2026]：


from pydantic import BaseModel, computed_field

class ResearchResult(BaseModel):
    start_ts: float
    end_ts: float
    topic: str

    @computed_field
    @property
    def duration_seconds(self) -> float:
        return round(self.end_ts - self.start_ts, 2)

    @computed_field
    @property
    def topic_slug(self) -> str:
        return self.topic.lower().replace(' ', '-')[:30]

r = ResearchResult(start_ts=1000.0, end_ts=1060.0, topic='Pydantic v2')
print(r.model_dump())  # 包含 duration_seconds=60.0, topic_slug='pydantic-v2'

v2.13 新增 exclude_if 參數，可條件性排除計算欄位輸出（如 exclude_if=lambda v: v is None）。

2.5 `TypeAdapter` — 非 BaseModel 型別驗證與效能陷阱

TypeAdapter 允許對任意 Python 型別（list[int]、dict[str, Any]、Union[...]、str）執行驗證與序列化，是 v1 parse_obj_as() 的替代品 [來源：Pydantic 官方文件，2026]：


from pydantic import TypeAdapter

# ✅ 模組層級單例，只建立一次
_LIST_INT_ADAPTER = TypeAdapter(list[int])
_TASK_IDS_ADAPTER = TypeAdapter(list[str])

def parse_task_ids(raw: list) -> list[str]:
    return _TASK_IDS_ADAPTER.validate_python(raw)  # 複用 adapter

# ❌ 每次呼叫都建立新 TypeAdapter → 重建 core schema，熱路徑效能崩壞
def bad_parse(raw: list) -> list[int]:
    adapter = TypeAdapter(list[int])  # 每次呼叫都初始化！
    return adapter.validate_python(raw)

原理：TypeAdapter(T) 建立時呼叫 Rust 端的 schema builder，是非 trivial 的計算。Pydantic 官方文件明確警告：「建立 TypeAdapter 有相當的初始化成本，建議對特定型別只建立一次並複用。」[來源：Pydantic TypeAdapter docs；GitHub Discussion #6748]

2.6 `model_config` + ConfigDict — 取代 v1 的 `Config` 類

v2 完全廢棄嵌套的 class Config，改用 model_config = ConfigDict(...) [來源：Pydantic migration guide，2026]：


from pydantic import BaseModel, ConfigDict

class StrictAPIOutput(BaseModel):
    model_config = ConfigDict(
        extra='forbid',            # 禁止未知欄位（防 LLM 幻覺欄位進入系統）
        frozen=True,               # 不可變物件（hashable，適合 set/dict key）
        str_strip_whitespace=True, # 自動 strip 字串空白
        validate_default=True,     # 驗證預設值
    )
    status: str
    summary: str

extra='forbid' 對 AI Agent 系統特別重要：LLM 可能回傳計畫外的欄位（如幻覺的 score、confidence 欄位），用 extra='forbid' 可立即在入口點捕獲這類問題，而不是讓它靜默進入業務邏輯。[來源：oneuptime.com，2026]

2.7 `pydantic-settings`：多來源設定管理

pydantic-settings 從 pydantic v2 主包獨立出來，需另外安裝 (pip install pydantic-settings) [來源：Pydantic Settings 官方文件，2026]：


from pydantic_settings import BaseSettings, SettingsConfigDict

class AppSettings(BaseSettings):
    model_config = SettingsConfigDict(
        env_prefix='DDP_',           # DDP_KB_URL, DDP_NTFY_TOPIC ...
        env_file='.env',             # 本地開發 .env 檔
        env_nested_delimiter='__',   # DDP_REDIS__HOST='localhost'
        env_ignore_empty=True,       # 空字串環境變數使用 default
    )
    kb_url: str = 'http://localhost:3000'
    ntfy_topic: str = 'wangsc2025'
    claude_timeout: int = 900
    max_parallel_tasks: int = 3

設定值優先順序（高→低）：CLI 引數 > 環境變數 > .env 檔 > secrets 檔 > 欄位預設值。[來源：Pydantic Settings 官方文件，2026]

最佳實踐

1. 在系統邊界集中驗證，內部程式碼信任 Pydantic 物件

在 HTTP 輸入、LLM JSON 輸出、JSON 檔案讀取等邊界點使用 model_validate() 驗證，一旦通過則在業務邏輯中直接使用已驗證物件，無需在每個函式重複驗證。[來源：oneuptime.com，2026] 反例：在每個處理函式都呼叫 model_validate()。正例：入口點驗證 → 傳遞已驗證物件 → 不再需要 if/else 防禦式檢查。

2. TypeAdapter 模組層級單例，絕不在熱路徑重建

如§2.5 所述，TypeAdapter(T) 初始化成本顯著。每個 Agent 每次執行可能驗證數百次 JSON 物件，若在迴圈或函式內部建立 TypeAdapter，累積開銷會逆轉 v2 的效能優勢。[來源：Pydantic performance docs，2026；GitHub Discussion #6748] 正確做法：模組頂層定義單例。

3. 盡量用宣告式約束，減少 Python 層驗證器

Rust 核心對宣告式約束（Field(ge=0, le=100, max_length=200)、constr(pattern=r'^[a-z_]+$')）的處理速度遠超 @field_validator 的 Python 函式（無 Python↔Rust 上下文切換）。建議：先嘗試宣告式表達，Python 驗證器只用於宣告式無法表達的邏輯（如「欄位 A 的值必須大於欄位 B」）。[來源：Pydantic performance docs，2026]


# ✅ 宣告式（Rust 處理）
from pydantic import Field
class Result(BaseModel):
    score: int = Field(ge=1, le=5)
    summary: str = Field(min_length=20, max_length=500)

# ❌ 不必要的 Python 驗證器
    @field_validator('score')
    @classmethod
    def check_score(cls, v):
        if not 1 <= v <= 5: raise ValueError('...')
        return v

4. 敏感資料用 SecretStr，防止意外日誌洩露


from pydantic import BaseModel, SecretStr

class AgentConfig(BaseModel):
    api_key: SecretStr  # str(api_key) → '**********'
    ntfy_token: SecretStr

# 取得真實值
config.api_key.get_secret_value()

Agent 系統常在 debug 時印 config，SecretStr 確保 API key 不會出現在日誌或錯誤追蹤中。[來源：oneuptime.com，2026]

5. pydantic-settings + @lru_cache 作設定管理，避免重複讀取環境變數


from functools import lru_cache
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix='DDP_', env_file='.env')
    kb_url: str = 'http://localhost:3000'
    ntfy_topic: str = 'wangsc2025'
    claude_timeout: int = 900

@lru_cache
def get_settings() -> Settings:
    return Settings()  # 整個 Python 行程只建立一次

@lru_cache 防止每次呼叫都重新讀環境變數（多次 os.environ 查詢 + Pydantic schema 重建）。[來源：Pydantic Settings docs；oneuptime.com，2026]

陷阱與反例

陷阱 1：Optional[T] 語義在 v2 逆轉（最高頻遷移錯誤）

v1 行為：Optional[str] 自動設置 default=None，欄位可缺失。

v2 行為：Optional[str] 等同 str | None，欄位仍是必填的，只是允許值為 None。[來源：Pydantic migration guide，2026；Medium migration article]


# ❌ v2 中這個欄位是必填！
class Model(BaseModel):
    name: Optional[str]  # 不傳會報 ValidationError

# ✅ v2 正確的可選語義
class Model(BaseModel):
    name: Optional[str] = None  # 明確 default=None
    # 或用 Union 語法：
    # name: str | None = None

成因：PEP 484 原本的設計意圖是允許 None 值，而非隱含 default。v2 修正了 v1 的「誤解性便利」，對齊語言規範。此錯誤在遷移大型專案時幾乎必定出現，bump-pydantic 工具能自動偵測但不能完全自動修正。

規避：所有 Optional 欄位遷移時手動加 = None；使用 mypy/pyright 靜態分析輔助掃描。

陷阱 2：TypeAdapter 在熱路徑每次重建，效能退化至 v1 水準


# ❌ 每次函式呼叫都建立 TypeAdapter → 重建 Rust schema
def parse_results(items: list) -> list[dict]:
    adapter = TypeAdapter(list[dict])  # 高頻呼叫時極慢
    return adapter.validate_python(items)

# ✅ 模組層級單例
_RESULTS_ADAPTER = TypeAdapter(list[dict])

def parse_results(items: list) -> list[dict]:
    return _RESULTS_ADAPTER.validate_python(items)  # 快 10-100 倍

成因：每個 TypeAdapter(T) 建立時觸發 Rust 端的 schema compilation（構建 core schema）。Pydantic 官方文件明確警告此成本「non-trivial」。[來源：Pydantic TypeAdapter docs，2026；GitHub Discussion #6748]

規避：grep 專案中所有 TypeAdapter( 出現位置，確認無一在函式內部（應全在模組頂層或類別屬性）。

陷阱 3：from_attributes=True 不分場合開啟，驗證純 dict 時效能損失

from_attributes=True 讓 Pydantic 嘗試用 getattr() 路徑讀取物件屬性（ORM 物件用）。若你的模型只驗證 dict（如 JSON 解析結果、results/*.json），開啟此設定會讓 Pydantic 在 dict 和 attribute 兩條路徑都嘗試，增加不必要的工作量。[來源：Pydantic performance docs，2026；GitHub Discussion #6748]


# ❌ 錯誤：驗證純 dict 時不需要 from_attributes
class ResultModel(BaseModel):
    model_config = ConfigDict(from_attributes=True)  # 不需要！

# ✅ 只在真正需要 ORM 支援時開啟
class UserORM(BaseModel):
    model_config = ConfigDict(from_attributes=True)  # 真正有 SQLAlchemy 物件

規避：明確分開「ORM 模型」和「純資料模型」，前者才開 from_attributes=True。

陷阱 4：wrap validator 濫用（隱藏效能地雷）

wrap 模式需要在 Python 層「具象化」（materialize）資料，跳出 Rust 快速路徑。Pydantic 官方文件明確警告：「若你在意效能，應避免 wrap validators。」[來源：Pydantic performance docs；validators docs，2026]


# ❌ 用 wrap 只因為「看起來更彈性」，實際上完全不需要
@field_validator('value', mode='wrap')
@classmethod
def unnecessary_wrap(cls, v, handler):
    return handler(v)  # 沒用到任何 wrap 特性，浪費

# ✅ after 就夠了，而且快得多
@field_validator('value', mode='after')
@classmethod
def simple_validation(cls, v: int) -> int:
    if v < 0: raise ValueError('不可為負數')
    return v

規避：wrap 只用於一個合理場景：需要在 ValidationError 發生後執行補救（如截斷超長字串後繼續驗證）。其他情況一律用 before/after。

與本專案（daily-digest-prompt）關聯

落地場景 1：tools/validate_results.py 升級至 Pydantic v2 模型

本專案的 validate_results.py 目前使用手寫 if/else 邏輯驗證 results/todoist-auto-*.json。引入 Pydantic v2 後可定義 AutoTaskResult 模型：


from pydantic import BaseModel, ConfigDict, field_validator, model_validator
from typing import Literal
from typing_extensions import Self

# 模組層級單例（規避陷阱 2）
_RESULT_ADAPTER = TypeAdapter(dict)  # 用於原始 JSON 預驗證

class AutoTaskResult(BaseModel):
    model_config = ConfigDict(
        extra='ignore',         # 允許額外欄位（相容舊結果）
        str_strip_whitespace=True
    )
    agent: str
    task_key: str
    status: Literal['success', 'failed', 'partial', 'format_failed']
    summary: str
    cli_exit_note_zh: str | None = None
    result_guard_zh: str | None = None

    @field_validator('summary')
    @classmethod
    def summary_not_placeholder(cls, v: str) -> str:
        placeholders = {'見上', '略', '執行中...', 'task_did_not_complete_or_timeout'}
        if v.strip() in placeholders or len(v.strip()) < 20:
            raise ValueError(f'summary 不合規：長度 {len(v)}，內容 {v!r}')
        return v

    @model_validator(mode='after')
    def check_failed_has_error_note(self) -> Self:
        if self.status == 'failed' and not self.cli_exit_note_zh:
            # 警告級別，不 raise（向下相容）
            pass
        return self

好處：ValidationError 自動彙總所有違規欄位（而不是遇到第一個就停止）；model_json_schema() 可直接生成 JSON Schema 文件供人工審查。

落地場景 2：pydantic-settings 統一多個 Python 工具的 config 讀取

本專案 tools/*.py 工具目前各自讀取 YAML/JSON 配置，缺乏統一型別驗證。可建立 tools/config.py：


from functools import lru_cache
from pydantic_settings import BaseSettings, SettingsConfigDict

class DailyDigestConfig(BaseSettings):
    model_config = SettingsConfigDict(
        env_prefix='DDP_',
        env_file='.env',
        env_nested_delimiter='__',
        env_ignore_empty=True,
    )
    ntfy_topic: str = 'wangsc2025'
    kb_base_url: str = 'http://localhost:3000'
    kb_alive_ttl_minutes: int = 30
    claude_timeout_seconds: int = 900
    max_parallel_tasks: int = 3

@lru_cache

任何工具調用 get_config().kb_base_url 即可；環境變數 DDP_KB_BASE_URL=http://custom:3000 自動覆蓋，無需修改程式碼。

程式碼範例（可執行片段）


# 完整示例：LLM 輸出驗證 + pydantic-settings 整合
# 來源：自寫，基於 Pydantic 官方文件範例改編

from functools import lru_cache
from typing import Any, Literal
from typing_extensions import Self
from pydantic import BaseModel, ConfigDict, Field, TypeAdapter, field_validator, model_validator
from pydantic_settings import BaseSettings, SettingsConfigDict

# --- 設定管理（lru_cache 防止重複初始化）---
class AppSettings(BaseSettings):
    model_config = SettingsConfigDict(env_prefix='APP_', env_file='.env')
    kb_url: str = 'http://localhost:3000'
    max_retries: int = 3

@lru_cache
def settings() -> AppSettings:
    return AppSettings()

# --- 模組層級 TypeAdapter 單例（規避熱路徑重建）---
_SUMMARY_ADAPTER = TypeAdapter(str)

# --- LLM 輸出驗證模型 ---
class ResearchOutput(BaseModel):
    model_config = ConfigDict(extra='forbid', str_strip_whitespace=True)

    topic: str = Field(min_length=5, max_length=100)
    kb_imported: bool
    status: Literal['success', 'failed']
    summary: str = Field(min_length=20)  # 宣告式約束，Rust 處理
    quality_score: int = Field(ge=1, le=5)  # 宣告式，無需 @field_validator

    @model_validator(mode='after')
    def check_failed_consistency(self) -> Self:
        if self.status == 'failed' and self.kb_imported:
            raise ValueError('status=failed 但 kb_imported=True，邏輯矛盾')
        return self

# --- 使用範例 ---
try:
    result = ResearchOutput.model_validate({
