一、專案概述

OWL（Optimized Workforce Learning）是由 CAMEL-AI 社群開發的開源多 Agent 協作框架，建構於 CAMEL-AI Framework 之上，專注於透過動態 Agent 協調實現真實世界任務自動化。

核心數據：

  • GitHub Stars：19.1K+
  • Forks：2.2K+
  • License：Apache 2.0
  • NeurIPS 2025 Accepted
  • GAIA Benchmark 開源第一：69.09%（超越所有開源替代方案與 Deep Research）
  • 首次開源：2025 年 3 月 7 日
  • 支援 Python 3.10/3.11/3.12

解決的核心問題：現有多 Agent 系統在真實世界任務中面臨三大挑戰——任務分解粒度不當、Agent 間協調低效、工具呼叫策略缺乏學習能力。OWL 透過結構化的三層架構加上強化學習訓練，系統性地解決這些問題。

---

二、技術架構：Planner-Coordinator-Worker 三層模型

2.1 架構概覽

OWL 採用 Planner-Coordinator-Worker 三層式架構（Tripartite Framework），設計理念模仿人類團隊協作模式——將複雜任務拆解為可執行的子步驟，由不同角色的 Agent 分工完成。


┌─────────────┐
│   Planner   │  戰略層：任務分解、子任務規劃、全局策略
├─────────────┤
│ Coordinator │  管理層：執行流程管理、Agent 間通訊、子任務分配
├─────────────┤
│   Worker    │  執行層：具備工具呼叫能力的專責 Agent
└─────────────┘

2.2 各角色職責

| 角色 | 職責 | 關鍵能力 |

|------|------|----------|

| Planner | 接收用戶任務，分解為結構化子任務 | 任務理解、策略制定、依賴關係分析 |

| Coordinator | 管理執行流程，協調 Agent 間通訊 | 子任務分配、狀態追蹤、錯誤恢復 |

| Worker | 執行具體操作，呼叫工具完成子任務 | 30+ 工具呼叫、多模態處理、程式碼執行 |

2.3 雙角色簡化模式

實際使用中，OWL 也支援「Planning Agent + Web Agent」的簡化雙角色模式：

  • Planning Agent：負責任務分解與策略制定
  • Web Agent（Execution Agent）：透過工具呼叫完成具體操作

此模式降低了系統複雜度，適合中等複雜度任務。

三、WORKFORCE 強化學習訓練

3.1 訓練方法：GRPO

OWL 採用 GRPO（Group Relative Policy Optimization） 強化學習方法訓練 Agent：

  • 群組相對獎勵比較：不使用絕對獎勵分數，而是在同組軌跡中比較相對表現
  • 軌跡學習：從實際任務執行軌跡中學習最佳策略
  • 協調效率優化：優化 Agent 間的協調策略而非單一 Agent 能力

3.2 訓練成果

  • 基於 Qwen2.5-32B-Instruct 模型，WORKFORCE 強化學習訓練提升了 16.37% 的 GAIA 效能
  • 2025 年 7 月釋出訓練資料集與模型 checkpoints
  • 消融實驗證明：

- Coordinator 角色對任務成功率有顯著影響

- Worker 專化（specialization）提升特定領域效能

- GRPO 訓練優於監督微調（SFT）方法

MCP 整合方式：


# mcp_servers_config.json 定義各伺服器
config_path = Path(__file__).parent / "mcp_servers_config.json"
mcp_toolkit = MCPToolkit(config_path=str(config_path))
await mcp_toolkit.connect()
tools = [*mcp_toolkit.get_tools()]

支援的 MCP 伺服器類型包括：Desktop Commander（檔案系統操作）、Playwright（網頁自動化）、Fetch（內容擷取）等。

五、GAIA Benchmark 表現

GAIA（General AI Assistants）是評估 AI 系統在真實世界任務的標準基準測試，分為三個難度等級。

5.1 成績對比

| 框架 | GAIA 分數 | 備註 |

|------|-----------|------|

| OWL (WORKFORCE) | 69.09-69.70% | 開源第一 |

| Deep Research | 低於 OWL | 商業方案 |

| 標準單 Agent LLM | 顯著低於 OWL | 缺乏多 Agent 協調 |

| 其他開源框架 | 58.18% 以下 | OWL 早期版本即達此分數 |

5.2 為何 OWL 能勝出

  1. 階層式任務分解：Planner 將複雜任務拆解為粒度適當的子任務
  2. 跨 Agent 協調：Coordinator 管理執行流程，避免重複工作
  3. 工具專化：Worker 具備 30+ 工具的專業呼叫能力
  4. 強化學習優化：GRPO 訓練讓 Agent 從經驗中學習最佳策略

六、快速上手

6.1 安裝（推薦 uv）


git clone https://github.com/camel-ai/owl.git
cd owl
pip install uv
uv venv .venv --python=3.10
source .venv/bin/activate  # macOS/Linux
uv pip install -e .

6.2 基本使用


from owl import construct_society, run_society

# 定義任務
task = "Find latest Apple stock price and create summary report"

# 建構多 Agent 協作環境
society = construct_society(task)

# 執行任務
answer, chat_history, token_count = run_society(society)
print(answer)

6.3 自訂工具鏈


from camel.toolkits import BrowserToolkit, SearchToolkit

# 選擇所需工具
tools = [
    *BrowserToolkit(headless=True).get_tools(),
    SearchToolkit().search_google,
]

# 帶工具建構 society
society = construct_society(task, tools=tools)

九、潛在應用：Daily-Digest-Prompt 系統借鑑

9.1 高度相關的設計模式

本專案（daily-digest-prompt）已採用多 Agent 並行模式（Phase 1 五路並行 → Phase 2 組裝），OWL 的架構提供以下借鑑：

  1. Coordinator 角色啟示：本專案的 Phase 2 組裝 Agent 可視為簡化的 Coordinator。OWL 證明了獨立 Coordinator 層對任務成功率的顯著影響——可考慮在 Todoist 團隊模式中強化 Phase 1 的任務協調機制。
  1. 工具專化 vs 通用 Agent：OWL 的 Worker 專化模式與本專案的「每個子 Agent 只做一件事」原則一致。差異在於 OWL 的 Worker 可動態組合工具，而本專案是在 prompt 層靜態指定。
  1. MCP 標準化整合：OWL 的 MCPToolkit 展示了 MCP 如何成為 Agent 工具的「USB-C」。本專案目前透過 curl 直接呼叫各 API，未來可考慮將 Todoist API、ntfy、知識庫 API 等封裝為 MCP 伺服器，統一工具介面。
  1. 強化學習啟示：GRPO 訓練展示了 Agent 策略可從經驗中學習。本專案的 context/research-registry.json 和 context/auto-tasks-today.json 累積的執行軌跡，理論上可用於類似的策略優化。

9.2 不適用的部分

  • 重量級依賴：OWL 需要完整 Python 環境 + 多模型 API Key，本專案以 Claude Code CLI 為單一執行引擎
  • 即時協調開銷：三層架構增加了延遲，本專案的並行模式（Start-Job + 獨立 claude -p）更適合批次處理場景

十、優缺點評估

優點

  • GAIA Benchmark 開源第一，有實證效能支撐
  • NeurIPS 2025 學術認可，框架設計經同行評審
  • 30+ 工具包覆蓋六大能力，開箱即用
  • GRPO 強化學習訓練是開源框架中唯一提供的
  • MCP 原生整合，工具擴展標準化
  • Apache 2.0 授權，商用友善
  • 多模型支援（OpenAI、Claude、Gemini、Qwen、Ollama）

缺點

  • 最佳效能需 GPT-4 級別模型，Token 成本高
  • 瀏覽器自動化等工具操作資源密集
  • 需配置多個 API Key，初始設定複雜
  • 相比 AutoGen（41.7K stars）社群較小
  • 三層架構增加系統複雜度與除錯難度
  • 強化學習訓練需大量計算資源，一般用戶難以自行訓練
