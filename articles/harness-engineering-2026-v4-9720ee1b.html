執行摘要

本文件整合 2026 年 4 月 20-21 日發布的最新 Harness Engineering 研究，提出七大增量優化方向，補充現有 v3.0 方案。核心發現：10x 性能提升來自基礎設施優化而非模型升級，且已有可複製的實施路徑。

關鍵數據：

  • Edit Format 優化：6.7% → 68.3%（10 倍改進，零模型變動）
  • LangChain 案例：排名 30 → 5（52.8% → 66.5%，僅改 harness）
  • Tool Organization：節省 85% context tokens（Anthropic 實測）
  • Context 閾值：40% 重置優於 80% 填滿（性能拐點）
  • 6x 性能差距：相同模型，純 harness 優化達成

一、實驗證據：Harness > Model（2026 年確認）

1.1 Can Bölük 實驗：Edit Format 的 10x 魔法

實驗設計：

  • 相同模型（Grok Code Fast 1）
  • 相同任務集
  • 唯一變數：edit format（輸出格式）

結果：


Hashline format:  68.3% 成功率
str_replace format: 6.7% 成功率
────────────────────────────────
改進倍數:         10.2x

啟示：

"The format in which you ask the model to output edits matters 10x more than which model you use."

對 Daily Digest Prompt 的意義：

  • 當前使用 Claude Code CLI 的 Edit tool，需驗證其 format 是否為最優
  • 建議實驗：Hashline vs. str_replace vs. unified diff（三種格式 A/B/C 測試）
  • 預期收益：若當前為次優格式，可能有 2-10x 改進空間

1.2 LangChain 案例：零模型改動，排名飛躍

背景：

  • Terminal Bench 2.0 全球排行榜
  • 原排名：30（52.8% 分數）
  • 目標：進入前 10

實施改動（100% harness 層）：

  1. 改進文檔載入機制（documentation loading）
  2. 加入驗證迴圈（validation loops）
  3. 增強追蹤（tracing）

結果：

  • 新排名：5（66.5% 分數）
  • 改進：+13.7 百分點
  • 模型變動：零

對 Daily Digest Prompt 的意義：

  • 當前 prompts/team/*.md 的文檔載入機制是否最優？
  • 驗證迴圈是否完整？（現有 Draft-Verify 是第一層，是否需要更多層？）
  • 追蹤機制是否足夠？（現有 state/llama-workflow-trace.jsonl，是否需要增強？）

1.3 Dex Horthy 發現：40% 是黃金閾值

實驗結果：


100K window @ 30% usage > 200K window @ 80% usage

性能拐點：

  • 40% 以下：性能穩定
  • 40-60%：開始下降
  • 60% 以上：急劇惡化

當前實施狀態：

  • ✅ CLAUDE.md 已提及「40% Reset Rule」
  • ❓ 但缺少自動監控與觸發機制

建議優化：


# config/context-management.yaml（新增）
thresholds:
  warning: 0.40  # 發出警告，建議重置
  critical: 0.60  # 強制重置，避免性能崩潰

reset_protocol:
  method: "structured_handoff"  # 生成交接文檔
  handoff_template: "templates/shared/context-handoff.md"
  preserve_fields:
    - task_key
    - current_phase
    - critical_state
    - pending_actions

實施步驟：

  1. Hook 層監控 context 使用率（通過 API 或啟發式估算）
  2. 達到 40% 時寫入 state/context-reset-suggest.flag
  3. Agent 讀取 flag 後觸發結構化交接流程
  4. 生成精簡交接文檔（< 500 tokens）
  5. 重啟新 context，載入交接文檔繼續

二、七大增量優化方向（基於最新研究）

優化 A：Edit Format 標準化與實驗

現狀評估：

  • Claude Code CLI 預設使用何種 format？（需確認）
  • 當前 Edit tool 調用是否有格式指定？

優化方案：

  1. Phase 1（驗證）：

- 建立 3 個測試任務（簡單/中等/複雜）

- 分別用 Hashline、str_replace、unified diff 執行

- 測量成功率與 token 使用

  1. Phase 2（標準化）：

- 選擇最優格式寫入 config/edit-format.yaml

- 在 templates/shared/preamble.md 加入格式規範

- Hook 層驗證 Edit 輸出符合標準格式

  1. Phase 3（持續優化）：

- 每月 A/B 測試新興格式（如 2026 年可能出現的 semantic-diff）

- 維護格式效能基準（state/edit-format-benchmark.json）

預期收益：

  • 保守估計：+20% Edit 成功率
  • 樂觀估計：+100%（若當前為次優格式）

優化 B：Tool Organization 極致精簡

數據支持：

Anthropic Tool Search saves ~85% context tokens by loading only needed tools.

當前狀態：

  • ✅ Hook 系統已限制 Write/Edit/Bash 等工具
  • ❓ 但每個 Agent 是否仍載入過多工具？

優化方案：

  1. 工具使用審計：

```bash

# 分析過去 30 天的 tool 使用情況

uv run python tools/tool_usage_analyzer.py --days 30 --output state/tool-usage-stats.json

```

  1. 角色專屬工具集：

```yaml

# config/role-toolsets.yaml（新增）

fetcher:

allowed: [Read, Bash, WebFetch, WebSearch]

forbidden: [Write, Edit] # 唯讀角色

analyst:

allowed: [Read, Grep, Bash]

forbidden: [Write, Edit, WebFetch] # 分析不應修改

assembler:

allowed: [Read, Write, Bash]

forbidden: [Edit, WebFetch] # 組裝不應增量編輯

```

  1. 動態工具載入：

- Phase 0 確定任務類型 → 載入對應工具集

- 禁止「萬能工具集」（防止 context 膨脹）

預期收益：

  • Token 節省：30-50%（保守估計，基於 Anthropic 85% 數據按比例縮小）
  • 副作用減少：工具越少，誤用越少

優化 C：自然語言 Harness 實驗

理論基礎：

Tsinghua University's breakthrough: natural-language-based harnesses significantly outperform traditional Python scripts.

當前狀態：

  • ✅ 已使用 Markdown prompt（自然語言）
  • ❓ 但部分邏輯仍用 PowerShell（非自然語言）

實驗設計：

  1. 對照組：當前 run-todoist-agent-team.ps1（PowerShell 調度）
  2. 實驗組：prompts/orchestrator/todoist-team-nl.md（自然語言調度）

```markdown

## 任務分解（自然語言）

1. 首先，讀取 Todoist API 獲取今日任務

2. 然後，對每個任務判斷是否符合執行條件：

- 條件 A：截止時間在 2 小時內

- 條件 B：優先級 ≥ P2

- 條件 C：尚未開始（無 in_progress 標記）

3. 接著，選出最多 3 個任務並行執行

4. 最後，彙整結果並推送通知

```

  1. 對比指標：

- 成功率

- Token 使用

- 人類可讀性（1-5 分，團隊評分）

預期收益：

  • 可維護性：+50%（自然語言更易理解與修改）
  • 錯誤率：-20%（邏輯更清晰，減少歧義）

優化 E：Embedded Linter Fix Instructions

數據支持：

Adding fix instructions directly in linter error messages improves agent compliance with constraints.

當前狀態：

  • ✅ Hook 系統已有 linter（hooks/validate_config.py）
  • ❌ 但錯誤訊息僅指出問題，未提供修復指引

優化範例：

Before（當前）：


❌ Error: frequency-limits.yaml missing 'daily_limit' for task 'ai_research'

After（優化）：


❌ Error: frequency-limits.yaml missing 'daily_limit' for task 'ai_research'

💡 Fix:
1. Open config/frequency-limits.yaml
2. Locate the 'ai_research' entry
3. Add: daily_limit: 6  # Recommended default
4. Run: uv run python hooks/validate_config.py --check-frequency-limits

實施步驟：

  1. 修改 hooks/validate_config.py：

- 每個錯誤類型加入 fix_instructions 欄位

- 輸出格式：錯誤 + 修復步驟

  1. 建立 config/linter-fix-templates.yaml：

```yaml

missing_daily_limit:

message: "Missing 'daily_limit' for task '{task_key}'"

fix_steps:

- "Open config/frequency-limits.yaml"

- "Locate the '{task_key}' entry"

- "Add: daily_limit: {suggested_value} # {reason}"

- "Run: uv run python hooks/validate_config.py --check-frequency-limits"

```

  1. Agent 讀取錯誤時直接獲得 actionable 指引

預期收益：

  • 自修復成功率：+60%
  • 人工干預：-40%

優化 G：Repository-First 知識管理

核心原則：

"Everything the agent needs must be in the repository" - knowledge in external tools becomes invisible to the system.

當前狀態：

  • ⚠️ 知識庫 API（localhost:3000）為外部依賴
  • ⚠️ Agent 需主動呼叫 API（可能忘記或失敗）

優化方案：

  1. 知識快取層：

```

config/kb-cache/

├── harness-engineering/ # 主題目錄

│ ├── index.json # 筆記索引

│ └── notes/

│ ├── note-1.md

│ └── note-2.md

└── rag-optimization/

└── ...

```

  1. 同步機制：

```bash

# scripts/sync-kb-to-repo.ps1（新增）

# 每日 02:00 執行，從 KB API 拉取重要筆記到 repo

$topics = @("Harness Engineering", "RAG 優化", "系統架構")

foreach ($topic in $topics) {

Invoke-RestMethod -Uri "http://localhost:3000/api/search/hybrid" `

-Method POST -Body (@{query=$topic; limit=10} | ConvertTo-Json) `

| Export-KbCache -Topic $topic

}

```

  1. Fallback 策略：

- Agent 優先讀 config/kb-cache/（repo 內）

- 若需最新資料，才呼叫 API

- API 失敗時，使用快取（stale but available）

預期收益：

  • 可用性：+30%（API 故障時仍可運作）
  • 速度：+50%（本地讀取 vs. HTTP 呼叫）
  • 可複製性：repo clone 即含知識

產出：

  • 1,000,000+ 行代碼
  • 人工編寫：0 行（100% AI 生成）

  • 快速反饋迴圈（CI/CD < 5 分鐘）
