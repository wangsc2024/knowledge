Open-SWE — LangChain 開源非同步 Coding Agent 框架深度洞察

研究日期：2026-03-25

GitHub：langchain-ai/open-swe（8.3K Stars, 983 Forks, MIT License）

發布日期：2026-03-17

核心定位：企業級內部 Coding Agent 框架，從 Stripe（Minions）、Ramp（Inspect）、Coinbase（Cloudbot）三家頂尖工程團隊的實戰模式中提煉通用架構

---

一、專案概述

Open-SWE 是 LangChain 基於 Deep Agents + LangGraph 打造的開源非同步 Coding Agent 框架。它不是一個 IDE 插件或 CLI 工具，而是一個雲端原生的長時間任務執行平台，可以從 Slack、Linear、GitHub 觸發，在隔離沙箱中自主完成程式碼變更並開 PR。

核心理念：像真正的隊友一樣工作 — 早上指派一批任務，下午收到一組 PR。

---

二、技術架構：七大支柱

2.1 Agent Harness（組合式架構）

  • 設計哲學：Compose over Fork — 基於 Deep Agents 組合而非分叉現有方案
  • 優勢：保留上游升級路徑，同時允許組織客製化編排、工具與中介軟體
  • 對比：Stripe 選擇 Fork（Goose）、Coinbase 從零建構、Ramp 同樣採用 Compose（OpenCode）

2.2 三層 Agent 管線（核心編排）


Manager → Planner → Programmer（含 Reviewer 子 Agent）

| Agent | 職責 | 關鍵行為 |

|-------|------|----------|

| Manager | 任務分派與狀態管理 | 接收 Slack/Linear/GitHub 觸發，路由到正確 repo |

| Planner | 程式碼庫分析與計畫制定 | 搜尋檔案、理解架構、生成逐步執行計畫 |

| Programmer | 程式碼撰寫與自我審查 | 編碼、測試、lint、開 PR；內含 Reviewer 子 Agent |

Human-in-the-Loop：Planner 生成計畫後會中斷等待，讓開發者 accept/edit/delete/revise。

2.3 沙箱隔離執行

  • 每個任務在獨立雲端 Linux 環境中執行
  • 可插拔提供者：Modal、Daytona、Runloop、LangSmith
  • 持久化：同一 thread 的後續訊息重用同一沙箱
  • 自動重建：沙箱不可達時自動重新建立
  • 並行執行：多任務各有獨立沙箱，無需排隊

2.4 精選工具集（~15 個）

設計原則：「較小、精選的工具集更易測試、維護和推理」

| 工具 | 功能 |

|------|------|

| execute | 沙箱內 Shell 命令 |

| fetch_url | 網頁轉 Markdown |

| http_request | API 呼叫（GET/POST 等）|

| commit_and_open_pr | Git commit + 開 Draft PR |

| linear_comment | Linear 工單更新 |

| slack_thread_reply | Slack 執行緒回覆 |

| Deep Agents 內建 | read_file、write_file、edit_file、ls、glob、grep、write_todos、task（子 Agent 生成）|

2.5 Context Engineering（上下文工程）

雙層資訊注入策略：

  1. AGENTS.md：倉庫根目錄檔案，編碼規範、測試要求、架構決策注入系統 prompt
  2. 任務上下文：完整 Linear issue 或 Slack thread 歷史預先組裝，減少工具呼叫開銷
這與本專案 CLAUDE.md 的角色完全一致 — 用靜態文件引導 Agent 行為。

2.6 確定性中介軟體（Middleware）

三個 Hook 在 Agent 迴圈的特定點注入確定性邏輯：

| Middleware | 功能 |

|-----------|------|

| check_message_queue_before_model | 在下次模型呼叫前注入後續訊息 |

| open_pr_if_needed | 安全網：Agent 遺漏開 PR 時自動補上 |

| ToolErrorMiddleware | 優雅錯誤處理 |

2.7 多觸發介面

  • Slack：@openswe + repo:owner/name 語法，thread 內狀態更新
  • Linear：在 issue 評論 @openswe，👀 反應確認
  • GitHub：在 Agent PR 上 tag @openswe 處理 review feedback
  • 確定性 Thread ID 路由後續訊息到對應的執行中 Agent

三、企業級架構模式比較

| 維度 | Open-SWE | Stripe Minions | Ramp Inspect | Coinbase Cloudbot |

|------|----------|----------------|--------------|-------------------|

| 架構 | Compose（Deep Agents）| Fork（Goose）| Compose（OpenCode）| 從零建構 |

| 沙箱 | 可插拔多選 | AWS EC2 | Modal | 自建 |

| 工具數 | ~15 精選 | ~500 精選 | OpenCode SDK | MCPs + Skills |

| 上下文 | AGENTS.md + threads | Rule files | 內建 | Linear-first |

| 編排 | 子 Agent + Middleware | Blueprints | Sessions | 三模式 |

四、七大可借鏡技術項目

4.1 Compose over Fork 架構策略

洞察：Open-SWE 不分叉 Deep Agents，而是在其上組合。這保留了上游升級能力，同時允許高度客製化。

借鏡：daily-digest-prompt 的 Skill-First 策略本質上就是 Compose 模式 — 用 SKILL.md 組合行為而非硬編碼。可進一步強化 Skill 的可插拔性，讓每個 Skill 像 Open-SWE 的工具一樣獨立替換。

4.2 確定性 Middleware 安全網

洞察：open_pr_if_needed 是關鍵設計 — 不信任 LLM 一定會執行關鍵步驟，用確定性邏輯作為兜底。

借鏡：本專案的 Hook 系統（pre_bash_guard、pre_write_guard）已實現類似機制。可參考 Open-SWE 的 check_message_queue_before_model 模式，在 Agent 迴圈中注入即時指令（如排程優先級變更）。

4.3 精選工具集哲學

洞察：Stripe 有 ~500 工具，Open-SWE 刻意精簡到 ~15 個。理由：「較小的工具集更易測試、維護和推理」。

借鏡：本專案有 28 個 Skills，可評估是否有合併或精簡空間。Open-SWE 的經驗證明，精選工具 + 良好的組合能力 > 大量特化工具。

4.4 AGENTS.md 上下文工程

洞察：用倉庫根目錄的靜態文件注入編碼規範、架構決策到系統 prompt，讓每次執行都遵循團隊約定。

借鏡：本專案的 CLAUDE.md 已實現此模式。Open-SWE 的做法驗證了「文件驅動 Agent 行為」的正確性（ADR-002）。可參考其分層設計：全域 AGENTS.md + 目錄層級覆寫。

4.5 持久化沙箱與 Thread 路由

洞察：同一 thread 的後續訊息重用同一沙箱，且用確定性 Thread ID 路由。這讓多輪對話保持狀態連貫。

借鏡：本專案的 Todoist Team 模式可參考此設計 — 讓同一任務的多階段執行共享狀態（目前透過 results/*.json 傳遞，可考慮更緊密的狀態綁定）。

4.6 子 Agent 隔離與 Task 工具

洞察：Deep Agents 的 task 工具原生支援生成子 Agent，每個子 Agent 有獨立的 middleware stack、todo list 和檔案操作。

借鏡：本專案的 Agent Team 策略已有類似概念（subagent_type 參數）。Open-SWE 的「每個子 Agent 有獨立 middleware」設計值得參考 — 目前本專案的 Hook 是全域的，可考慮 per-agent Hook 配置。

4.7 Human-in-the-Loop 中斷模式

洞察：Planner 生成計畫後主動中斷，等待人類審核後才交給 Programmer 執行。這在自動化與控制之間取得平衡。

借鏡：本專案的自動任務目前是全自動執行。對於高風險操作（如系統配置變更），可參考 Open-SWE 的中斷模式，加入人工確認關卡。

---

五、技術洞察總結

核心設計原則

  1. Compose > Fork > Build：在成熟框架上組合，保留升級路徑
  2. 精選 > 堆疊：15 個精選工具 vs 500 個工具海
  3. 確定性兜底：Middleware 確保關鍵步驟不被 LLM 遺漏
  4. 文件即配置：AGENTS.md 讓 Agent 行為可版本控制
  5. 沙箱即信任邊界：每任務隔離，錯誤爆炸半徑可控

與 daily-digest-prompt 的架構同構性

| Open-SWE 概念 | daily-digest-prompt 對應 |

|---------------|-------------------------|

| AGENTS.md | CLAUDE.md + SKILL.md |

| Middleware Hooks | pre_bash_guard / pre_write_guard |

| task 子 Agent | Agent Team（subagent_type）|

| 精選 ~15 工具 | 28 Skills（可評估精簡）|

| 沙箱隔離 | Phase 0/1/2 階段隔離 |

| Thread 路由 | results/*.json 檔案傳遞 |

| Human-in-the-Loop | 目前無（可新增高風險確認）|
