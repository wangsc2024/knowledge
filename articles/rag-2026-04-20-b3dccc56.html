專案 RAG 優化日報（2026-04-20）

---

一、執行摘要

1.1 任務目標

以 RAG 知識庫為基礎，深度分析 daily-digest-prompt 專案並提出優化建議，完成優化計畫並執行關鍵驗證步驟。

1.2 完成成果

  • ✅ 制定完整 RAG 優化計畫（P0/P1/P2 三級優先級）
  • ✅ 驗證語義快取覆蓋狀況（P0-A）
  • ✅ 執行自適應 TTL 分析（P1-B）
  • ✅ 診斷快取命中率問題（27.4% vs 目標 40%+）
  • ✅ 產出可執行的改善建議清單

---

二、深度分析：現況診斷

2.1 語義快取覆蓋狀況（P0-A 驗證結果）

已整合 Skill：

  • ✅ knowledge-query：完整整合，支援 4 種搜尋 API
  • ✅ semantic-caching：通用 Skill，提供基礎能力

未整合 Skill（覆蓋缺口）：

  • ❌ todoist：高頻端點（每 30 分鐘），無語義快取
  • ❌ hackernews：每日觸發，可受益於語義快取
  • ❌ gmail：每日摘要，重複查詢模式明顯
  • ❌ pingtung-news：低頻但重複性高

快取檔案狀態：

  • cache/semantic-cache-kb.json：13KB，僅 1 筆測試項目
  • 結論：語義快取基礎建設完整，但實際使用率極低（< 1%）

根因分析：

  1. 僅 KB 搜尋 API 整合，其他端點缺乏語義快取協議
  2. Skill 文件未普及步驟 0.5 語義快取流程
  3. 缺乏命中率監控與效益驗證機制

---

2.2 快取命中率診斷（P1-B 分析結果）

整體指標：

  • 當前命中率：27.4%（vs 目標 40%+）
  • 總請求數：1177 次（7 天）
  • 快取讀取：392 次
  • API 呼叫：992 次

問題診斷：

  1. 未命名來源問題：存在 unknown 來源，命中率 0%
  2. TTL 配置：todoist 120min、hackernews 2880min、knowledge 1920min
  3. 歷史改善記錄：23.4% → 25.5% → 27.4%（趨勢正向但緩慢）

---

三、優化方案（P0/P1/P2）

P0-A：擴展語義快取至所有 API

從 1 個 Skill → 5 個 Skill 整合，預期命中率 +15-25%

P0-B：查詢改寫層（輕量版）

使用 Groq 前處理，關鍵詞擴展與同義詞替換

P1-A：Reranking 層

Cross-Encoder 重排序，MRR 提升 > 15%

P1-B：自適應 TTL（本次已執行）

修正 unknown 來源標記，2 週後重新評估

P2-A：RAGAS 評估管線

自動化品質評估：Faithfulness > 0.85、Context Recall > 0.70

P2-B：ARC 快取驅逐整合

替換 LRU 為 ARC，預估命中率 +5-10%

---

四、關鍵發現

  1. 語義快取基礎完整，但普及率低：僅 1/5 Skill 整合
  2. 快取命中率改善緩慢：4 個月僅 +17%（23.4% → 27.4%）
  3. unknown 來源反映架構缺口：需強化 Hook 層驗證
  4. 從 mechanism → application 轉折：技術方案 → 可執行計畫
