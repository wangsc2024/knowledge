Skill Hybrid Routing 機制設計

執行摘要

本研究針對 Daily Digest Prompt 專案現有的 Todoist 三層路由機制（Tier 1 標籤路由 → Tier 2 關鍵字比對 → Tier 3 LLM 語義判斷），設計 Hybrid Routing 改進方案，結合向量語義匹配、加權融合與信心度校準，以提高 Skill 使用機率與路由準確度。

現有架構分析

三層路由現況（config/routing.yaml）

| 層級 | 方法 | 信心度 | 缺點 |

|------|------|--------|------|

| Tier 1 | 標籤精確匹配（^prefix） | 100% | 依賴用戶手動標記，未標記則漏接 |

| Tier 2 | 關鍵字硬匹配（keywords 陣列） | 80% | 只比對表面詞彙，語義近義詞無法命中 |

| Tier 3 | LLM 語義判斷（processable 列表） | 60% | 延遲高、成本高、輸出不穩定 |

核心問題

  1. Tier 2 語義盲區：「自動化測試」無法命中 keywords=["程式碼"]，但語義上應路由到程式開發 Skill
  2. Tier 3 成本過高：每次路由都需 LLM 呼叫，且結果不可快取（任務描述千變萬化）
  3. 層級斷裂：Tier 1→2→3 是瀑布式 fallback，無法融合多層信號提升準確度

Hybrid Routing 設計方案

架構：三信號加權融合（Weighted Signal Fusion）


任務輸入
  ├── Signal A：標籤精確匹配（Label Match）→ 權重 0.5
  ├── Signal B：語義向量相似度（Semantic Similarity）→ 權重 0.3
  └── Signal C：關鍵字 BM25 分數（Keyword BM25）→ 權重 0.2
  │
  ▼ 加權融合
  Skill 候選排名 → Top-K 選擇 → 信心度校準 → 路由決策

Signal A：標籤精確匹配（保留現有 Tier 1）

  • 來源：routing.yaml 的 label_routing.mappings
  • 分數計算：命中 = 1.0，未命中 = 0.0
  • 權重：0.5（最高，因為標籤是用戶明確意圖）
  • 優化：支援模糊標籤匹配（如「遊戲」→ 命中「遊戲開發」「遊戲優化」「遊戲研究」）

Signal B：語義向量相似度（新增）

  • 方法：將每個 Skill 的 description + triggers 預編碼為向量（fastembed），查詢時計算 cosine similarity
  • 模型：BAAI/bge-small-zh-v1.5（中文語義，384 維，~50MB）
  • 快取策略：Skill 向量離線預計算，存入 cache/skill-embeddings.json；任務向量即時計算
  • 分數計算：cosine_similarity(task_embedding, skill_embedding)
  • 權重：0.3

Signal C：關鍵字 BM25 分數（改進現有 Tier 2）

  • 改進點：從硬匹配改為 BM25 評分，支援部分匹配與 TF-IDF 加權
  • 語料庫：所有 Skill 的 triggers + description 關鍵字集合
  • 分數計算：BM25(task_description, skill_keywords_corpus)
  • 權重：0.2

融合公式


final_score(skill) = w_label × label_score + w_semantic × semantic_score + w_bm25 × bm25_score

其中：w_label=0.5, w_semantic=0.3, w_bm25=0.2

信心度校準

| final_score 範圍 | 路由決策 | 信心度 |

|-----------------|---------|--------|

| ≥ 0.7 | 直接路由，跳過 LLM | 高（90%+） |

| 0.4 ~ 0.7 | 路由 + LLM 確認（輕量 Groq） | 中（70-90%） |

| < 0.4 | fallback 到 LLM 完整判斷 | 低（<70%） |

LLM Fallback 優化（取代現有 Tier 3）

  • 模型選擇：Groq（llama-4-scout-17b）取代 Claude，延遲 <500ms
  • Structured Output：Pydantic schema 強制輸出 {skill: str, confidence: float}
  • 快取：相似任務描述的路由結果可語義快取（tools/kb_semantic_cache.py 機制復用）

實作路線圖

Phase 1：Metadata 工程（低成本，高回報）

  1. 為每個 SKILL.md 新增 semantic_description 欄位（2-3 句完整語義描述）
  2. 擴充 triggers 陣列，加入同義詞與近義詞
  3. 新建 config/skill-semantic-profiles.yaml：每個 Skill 的語義向量、關鍵字集

Phase 2：語義匹配引擎

  1. 新建 tools/skill_router.py：

- compute_label_score()：標籤匹配

- compute_semantic_score()：fastembed 向量相似度

- compute_bm25_score()：BM25 關鍵字評分

- fuse_scores()：三信號加權融合

  1. 預計算 Skill 向量：cache/skill-embeddings.json
  2. 整合到 run-todoist-agent-team.ps1 Phase 1

Phase 3：信心度校準與 LLM Fallback

  1. 整合 Groq 輕量 LLM 做低信心度確認
  2. 語義快取：相似查詢復用路由結果
  3. A/B 測試：新舊路由準確度對比

預期效益

| 指標 | 現況 | 預期改善 |

|------|------|----------|

| Skill 命中率 | ~65%（依賴標籤覆蓋） | ~85%（語義補足） |

| 路由延遲（P50） | ~2s（Tier 3 LLM） | ~200ms（向量計算） |

| LLM 路由呼叫比例 | ~35%（所有 Tier 3） | ~10%（僅低信心度） |

| 未匹配標籤處理 | 告警但不處理 | 語義自動路由 |

與現有架構的相容性

  • 向後相容：Signal A 完全保留 Tier 1 標籤路由邏輯
  • 漸進式啟用：可用 config/routing.yaml 的 feature flag 控制各 Signal 開關
  • 快取復用：語義快取使用現有 tools/kb_semantic_cache.py 基礎設施
  • 依賴最小化：fastembed 已在 pyproject.toml 的 semantic-cache extra 中
