  • 本主題處理的是 Python 非同步管線如何把「可重試的網路失敗」與「可預期的連線資源上限」拆開建模。httpx 同時提供高階 AsyncClient API 與底層 HTTPTransport／AsyncHTTPTransport；官方明確區分「連線層重試」與「讀到 HTTP 回應後的語意重試」責任邊界 [來源: Transports - HTTPX, 2026 文件抓取日]。在 daily-digest-prompt 這類 PowerShell 7 先 orchestrate、uv run python 再呼叫本機 KB（localhost:3000）與 ntfy 的架構中，常見反模式是每支工具各自 subprocess/curl：每次冷啟動 TLS、難以集中設定逾時與重試、也不利於把 429/503 與 ConnectTimeout 分類處理。改為單例或長生命週期 AsyncClient + 明確 httpx.Limits，可把「連線池內排隊」變成可觀測的 PoolTimeout，而不是隨機的 shell 失敗碼。若還需要對 HTTP 狀態碼 做退避，官方建議補上泛用重試工具（如 tenacity）或第三方 httpx-retries 的 RetryTransport [來源: Transports - HTTPX, 2026；API - HTTPX Retries, 2026]。
  • HTTPTransport(retries=N)（非同步則 AsyncHTTPTransport(retries=N)）只在 httpx.ConnectError 或 httpx.ConnectTimeout 時重送請求；一旦已經拿到回應（含 503），就不屬於這條路徑 [來源: Transports - HTTPX, 2026]。Stack Overflow 上常見的踩雷是誤用同步 HTTPTransport 搭配 AsyncClient（缺少 async context），以及以為 retries=3 會自動重試 POST 的 5xx——實際上必須改用 AsyncHTTPTransport 並理解其僅涵蓋連線層 [來源: Stack Overflow #77694820, 2023–2025]。若要預設重試 429/502/503/504 並尊重 Retry-After，httpx-retries 的 Retry 預設 status_forcelist 與 respect_retry_after_header=True 即對齊此缺口 [來源: API - HTTPX Retries, 2026]。
  • httpx.Limits 三個核心欄位：max_keepalive_connections（預設 20）、max_connections（預設 100）、keepalive_expiry（預設 5 秒）——來自官方 Resource Limits 頁 [來源: Resource Limits - HTTPX, 2026]。StatusCodeFYI 的連線池指南補充工程直覺：冷啟動 TLS 可能佔 約 80–220ms（DNS+TCP+TLS 1.3），因此 pool 的價值在「重用」；並建議 idle timeout 設得比上游 keep-alive 更短（範例：ALB 60s → client 55s），避免「伺服器已関閉閒置連線、客戶端池仍握著半死 socket」[來源: Connection Pooling Guide, StatusCodeFYI, 2026]。中文工程筆記亦歸納：PoolTimeout 多半來自 max_connections 上限或連線未釋放（未消耗 response body），解法通常是調整 Limits 或修正 async with 生命週期 [來源: 突破Python HTTP瓶颈：HTTPX连接池与资源管理实战指南, 2026]。

  4. 量化指標設計：為本主題設計 1–2 個 fitness function，寫入 config/benchmark.yaml。
  5. 演化路線圖：將本主題放上 context/improvement-backlog.json，標明 owner 與 ETA。
