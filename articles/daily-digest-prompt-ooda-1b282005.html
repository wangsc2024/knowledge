Daily-Digest-Prompt 知識活化與 OODA 閉環實效性分析

從 107 筆研究積累到系統改善的轉化效率診斷與知識 ROI 最佳化策略

研究日期：2026-03-12

類型：專案優化 / 深度思維 / 知識庫

適用對象：daily-digest-prompt 專案

---

一、問題本質：知識積累 ≠ 系統改善

1.1 現況數據快照

| 維度 | 數值 | 觀察 |

|------|------|------|

| 知識庫筆記總數 | 1,047 篇（932 import / 101 manual / 14 web） | 龐大知識資產 |

| 研究註冊表 entries | 107 筆 | 持續累積中 |

| ADR 總數 | 12 個（7 accepted / 2 deferred / 1 wontfix / 2 proposed） | 僅 11.2% 研究轉化為 ADR |

| 改善積壓清單 | 5 項（2 P1 + 3 P2） | 僅 4.7% 研究轉化為行動項 |

| 系統審查分數 | 87.18（A 級） | 分數穩定但改善速率趨緩 |

| 快取命中率 | 14.57% | 多次研究指出，仍未改善 |

1.2 知識轉化漏斗


107 筆研究 → 12 個 ADR → 5 項改善清單 → ? 項已實施
  100%         11.2%        4.7%           ~2.8%（3 項 done）

核心問題：系統投入大量 Token（每日 8M+）產出研究，但研究到行動的轉化率僅 ~5%。這不是知識「不夠」的問題，而是知識「不活」的問題。

二、OODA 閉環實效性診斷

2.1 閉環運行數據（2026-03-08 至 03-12）

| 日期 | Observe | Orient | Decide | Act | 備註 |

|------|---------|--------|--------|-----|------|

| 03-08 | - | completed | skipped | - | improvement-backlog 為空 |

| 03-09 | - | completed | skipped | - | 同上 |

| 03-10 | - | completed | skipped | - | 同上 |

| 03-11 | - | completed | completed | completed | 手動觸發，產出 11 ADR |

| 03-12 | - | completed | completed | completed | 正常推進 |

關鍵發現：5 天中有 3 天（60%）Decide 步驟被 skip。根據 OODA 研究，健康的閉環應追蹤「loops per week」和「percent of loops that produce a decision」作為健康指標（Umbrex OODA Framework）。本系統的 Decide 產出率僅 40%，遠低於健康閾值。

2.2 Skip 根因分析

ooda-workflow.yaml 的條件觸發邏輯：


# Orient 完成後檢查 improvement-backlog.json 是否為空
# 空則跳過 Decide/Act

問題在於 improvement-backlog.json 的更新來源是 system-audit，而 system-audit 每日 00:40 執行。如果 audit 未發現新問題（因為分數已達 87 A 級），backlog 就維持空狀態，OODA 的 Decide/Act 永遠不會觸發。

這形成了一個悖論：系統越穩定 → backlog 越空 → OODA 越不行動 → 深層優化無法推進 → 系統停滯在「夠好但不卓越」的水平。

2.3 OODA 改善建議

建議 O1：雙軌觸發機制

除了 audit-driven（被動）觸發外，增加 research-driven（主動）觸發：


# ooda-workflow.yaml 建議新增
decide_triggers:
  - source: improvement-backlog  # 現有（被動）
    condition: items.length > 0
  - source: research-registry    # 新增（主動）
    condition: |
      recent_7d_entries >= 5 AND
      entries_without_action_item >= 3
    action: "從近 7 日研究中提取可行動建議，轉化為 backlog items"

建議 O2：研究產出必須附帶 Action Item

每篇研究筆記匯入 KB 時，強制產出至少一個 actionable_insight：

  • 格式：{insight, effort, impact, target_file}
  • 儲存於 research-registry.json 的 entry 中
  • OODA Decide 階段讀取未處理的 actionable_insights

建議 O3：OODA 健康度量

在 check-health.ps1 新增 OODA 健康區塊：

  • decide_production_rate：近 7 日 Decide 實際執行次數 / Orient 完成次數
  • action_completion_rate：Act 完成項 / Decide 產出項
  • research_to_adr_rate：新 ADR / 新研究 entries
  • 警告閾值：decide_production_rate < 50% 觸發告警

三、知識活化效率診斷

3.1 知識庫利用率分析

問題 1：研究筆記「寫了就忘」

1,047 篇筆記中，被後續研究引用（score > 0.4 的 hybrid search hit）的比例估計不到 30%。大量筆記在匯入後從未被再次檢索。

問題 2：重複研究角度仍在發生

by_type 統計顯示 todoist_research 佔 48 筆（44.9%），遠超其他類型。飽和閾值 saturation_threshold=3 僅限制同一 task_type，但 todoist_research 內部的子主題多樣性未受控。

問題 3：快取命中率持續低迷

快取命中率 14.57% 已在以下研究中被指出但仍未改善：

  • 2026-03-01「系統熵增與複雜度治理」
  • 2026-03-10「Token 消耗效率與排程產能最佳化」
  • 2026-03-11「狀態檔案生命週期治理」

這正是「知識不活」的典型症狀 — 問題被研究、被記錄、但未被修復。

3.2 知識 ROI 計算框架

借鑑 Stravito（2026）的 Knowledge Management ROI 框架和 Larridin 的 Utilization × Proficiency × Value 模型，建議建立以下度量：


知識 ROI = (改善帶來的系統分數提升 + 節省的 Token 成本) ÷ 研究投入的 Token 成本

當前估算：

  • 研究投入：~107 筆 × ~50K tokens/筆 = ~5.35M tokens
  • 系統分數提升：從 ~75（初始）到 87.18 = +12.18 分
  • Token 節省：快取命中率若從 14.57% 提升至 40%，日省 ~2M tokens

目標：建立可追蹤的知識 ROI 儀表板，讓每次研究的投資回報可見化。

五、Groq Router 未啟用問題深入分析

5.1 現象

state/token-usage.json 顯示 2026-03-11 起新增 groq_calls / claude_calls 計數欄位，但值均為 0。tools/llm_router.py 和 config/llm-router.yaml 已在 AI 架構治理方案 Batch 1（P1-B）中實施完成，但實際 Agent 執行未調用路由器。

5.2 根因推測

  1. invoke-llm.ps1 未被任何 prompt/template 引用 — 實作了但未接線
  2. 現有 prompt 仍直接使用 claude -p，繞過了 LLM Router
  3. 缺少「路由器接線清單」：哪些任務應走 Groq、哪些走 Claude

5.3 建議

  • 盤點所有 curl.*groq 和 claude -p 呼叫點，建立接線矩陣
  • 優先將 news_summary 和 task_classify 類型路由至 Groq（低成本、低延遲）
  • 在 check-health.ps1 新增 Groq 路由健康度量

| Rank | 項目 | Priority | 建立日期 | 停滯天數 |

|------|------|----------|---------|--------|

| 1 | PS1 hardcoded timeout 消除 | P1 | ~03-11 | 1 天 |

| 2 | 依賴漏洞自動掃描 | P1 | ~03-11 | 1 天 |

| 3 | YAML 注釋密度提升 | P2 | ~03-11 | 1 天 |

| 4 | 行覆蓋率提升 | P2 | ~03-11 | 1 天 |

| 5 | GitHub Actions CI | P2 | ~03-11 | 1 天 |

所有項目都是 03-11 system-audit 新產出的，目前停滯天數短。但歷史模式顯示，若無人工觸發 OODA Act，這些項目可能長期停滯。

6.2 建議：自動化 Act 觸發

對於 effort=low 的項目（如 #3 YAML 注釋），可設計自動修復腳本：


# ooda-workflow.yaml 建議
act_auto_fix:
  enabled: true
  conditions:
    - effort: low
    - priority: [P2, P3]
    - auto_fixable: true  # 需在 backlog item 標記
  max_auto_fixes_per_day: 2

七、系統停滯風險：「A 級陷阱」

7.1 分析

系統審查分數 87.18（A 級）看似健康，但隱含停滯風險：

  1. 邊際改善遞減：從 C→B 只需修復明顯問題；從 A→A+ 需要結構性突破
  2. 研究飽和：107 筆研究中大量是「借鑑其他開源專案」，但「內部度量驅動的改善」比例低
  3. 快取命中率 14.57% 被 3 篇研究指出仍未修復，說明「研究→行動」管線斷裂

7.2 突破策略：從「外部學習」轉向「內部度量驅動」

| 現行模式 | 建議模式 |

|---------|--------|

| 研究開源專案 → 寫借鑑分析 | 分析內部度量 → 找到瓶頸 → 直接修復 |

| 每日 2-3 篇外部研究 | 每週 1 篇內部度量分析 + 1 個具體修復 |

| 知識「廣度」優先 | 知識「深度轉化」優先 |

九、深度思維反思

9.1 知識管理的「最後一哩」

本專案的知識積累能力卓越（1,047 篇），但知識「活化」能力薄弱。這與組織學習理論中的經典問題一致：「組織不知道自己知道什麼」（Organizational Learning Processes, Management Science 2020）。解方不是產出更多知識，而是建立知識到行動的自動管線。

9.2 OODA 的精髓是「節奏」而非「循環」

John Boyd 的 OODA 原始理論強調「tempo advantage」— 比對手更快地循環。本系統的 OODA 問題不是循環不正確，而是節奏太慢（60% skip rate）。解方是降低 Decide 的觸發門檻，讓系統持續產出小改善，而非等待大問題。

9.3 「A 級陷阱」的組織學啟示

當系統達到「夠好」水平後，持續改善的動力會下降。這與 Satisficing（滿意化）理論一致：決策者傾向接受「足夠好」的解決方案，而非追求最優。突破方法是設定更高的基準線（如將 A 級門檻從 85 提高到 90），強制系統保持改善壓力。
