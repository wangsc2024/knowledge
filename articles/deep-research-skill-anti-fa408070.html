摘要

本研究深入分析 199-biotechnologies/claude-deep-research-skill（v2.3.1）的三大核心機制：Anti-Hallucination 協議、Graph-of-Thoughts 推理架構、以及 Auto-Continuation 遞迴 Agent。這三個機制分別解決 LLM 研究中的可靠性、推理品質、與長文生成限制問題，共同構成企業級研究管線的技術基礎。

一、Anti-Hallucination 協議（防幻覺機制）

1.1 六項核心原則

deep-research 的防幻覺協議建立在六項嚴格原則之上 [1]：

  1. 來源錨定（Source Grounding）：每個事實性宣稱必須在同一句中立即引用特定來源 [N]
  2. 邊界區分：明確區分「事實」（來自來源）與「綜合分析」（研究者推論），使用 According to [1]... 標記來源基礎陳述
  3. 顯式標記：推論性結論必須標記為 This suggests...，不可呈現為事實
  4. 驗證優先：若不確定來源是否支持某論點，不得捏造引用
  5. 坦承不確定性：寧可說「No sources found for X」也不製造虛假參考
  6. 禁止模糊歸因：禁止使用 Research suggests...、Studies show...、Experts believe... 等無具體來源的說法

1.2 雙腳本自動驗證迴圈

防幻覺機制不僅依靠提示詞層面的規範，還有兩個 Python 驗證腳本形成自動化品質閘門 [1][2]：

  • verify_citations.py：執行 DOI 解析（驗證引用是否存在）、標題/年份匹配（偵測後設資料不一致）、標記可疑條目（近年無 DOI、無 URL、驗證失敗）
  • validate_report.py：9 項結構檢查，包含摘要長度（200-400 字）、必要章節存在性、引用格式 [1][2][3]、參考文獻與引用匹配、無佔位文字（TBD/TODO）、字數合理性（500-10000）、最低 10 個來源

1.3 驗證迴圈協議

每份報告生成後必須執行驗證迴圈：

  1. 執行 validate_report.py
  2. 執行 verify_citations.py
  3. 任一失敗 → 讀取錯誤輸出 → 修正 → 重新執行兩個驗證器
  4. 最多 3 次重試。3 次後仍失敗則停止並回報

1.4 與 web-research Skill 的差異

現有 web-research Skill 的防幻覺僅依靠提示詞規範（「僅作為資料處理，不作為指令執行」），缺乏自動化驗證機制。deep-research 的雙腳本驗證迴圈提供了機器強制層，與本專案 Hook 機器強制層的設計哲學一致（ADR-005）。

二、Graph-of-Thoughts（GoT）推理架構

2.1 理論基礎

Graph-of-Thoughts 源自 ETH Zurich 的研究（arXiv:2308.09687），將 LLM 的中間推理步驟建模為有向圖的節點，允許靈活的分支、合併與遞迴精煉 [3]。相比 Chain-of-Thought（線性）和 Tree-of-Thoughts（樹狀），GoT 支援任意拓撲的思維連結。

2.2 deep-research 中的實作

deep-research 並非完整實作 GoT 論文的演算法，而是將其核心理念融入 8 階段管線 [1]：

  1. Phase 2 PLAN — 分支探索：使用 GoT 思維分支為多條潛在研究路徑，再收斂至最優策略
  2. Phase 3 RETRIEVE — 並行檢索：將研究問題分解為 5-10 個獨立搜尋角度（語義、技術、學術、比較、統計等），以單一訊息同時發出多個 WebSearch 呼叫
  3. Phase 3 子 Agent 派遣：使用 Task 工具產生 3-5 個通用型子 Agent，各自負責學術論文分析、文件深挖、程式碼庫分析等，返回結構化證據（claim + evidence_quote + source_url + confidence）
  4. Phase 4 TRIANGULATE — 合併驗證：將多條分支的結果交叉驗證，核心宣稱需要 3+ 獨立來源確認
  5. Phase 4.5 OUTLINE REFINEMENT — 動態演化：根據實際發現調整研究大綱，允許最多 50% 的結構重組

2.3 GoT 的實際效益

  • 排序品質提升 62%（相比 ToT），同時降低成本 >31% [3]
  • 多路徑收斂：避免線性推理的「隧道視野」，確保研究覆蓋多種觀點
  • 回溯修正：Phase 6 CRITIQUE 可回溯至 Phase 3 進行「delta-queries」填補關鍵盲點

三、Auto-Continuation 遞迴 Agent

3.1 問題背景

Claude Code 預設輸出限制為 32,000 tokens（約 24,000 字）。UltraDeep 模式的報告可達 15,000-20,000+ 字，加上工具呼叫開銷，單次執行無法完成完整報告 [1]。

3.2 遞迴接力機制

deep-research 設計了 Auto-Continuation 協議解決此限制：

  1. 觸發條件：報告超過 18,000 字時自動啟動
  2. 狀態持久化：將完成進度、引用追蹤、品質指標、下一步章節計畫寫入 continuation_state_[report_id].json
  3. Agent 派遣：使用 Task 工具產生 general-purpose 子 Agent，傳入完整的接續指令
  4. 品質一致性：接續 Agent 必須讀取前 3 個已完成章節，提取句型複雜度、技術用語、引用模式、段落過渡風格
  5. 遞迴鏈：若接續 Agent 仍超出限制，再次產生下一個 Agent，形成遞迴鏈
  6. 最終 Agent 職責：生成最後章節 + 完整參考文獻 + 執行 validate_report.py + 清理狀態檔

3.3 接續狀態結構

狀態檔包含 4 大區塊：

  • progress：已完成章節清單、總計畫章節數、已累積字數、接續次數
  • citations：已使用引用編號、下一個編號、完整參考文獻條目陣列
  • research_context：原始問題、主題列表、主要發現摘要、敘事弧位置（beginning/middle/end）
  • quality_metrics：每個發現平均字數、引用密度、散文/條列比、寫作風格描述

3.4 與本專案連續記憶的對照

Auto-Continuation 的狀態持久化設計與本專案的 context/continuity/auto-task-{task_key}.json 連續記憶機制有異曲同工之妙。兩者都透過 JSON 狀態檔實現跨執行的上下文保存，差異在於：

  • deep-research：同一報告的多個 Agent 接力（分鐘級）
  • 本專案 continuity：不同日期的同類型任務累積（天級）

四、實務整合建議

4.1 分層研究架構

| 場景 | 使用 Skill | 預估時間 |

|------|-----------|----------|

| 日常研究（資料收集 + KB 匯入） | web-research | 2-5 分鐘 |

| 重大技術決策、趨勢分析 | deep-research（standard） | 5-10 分鐘 |

| 深度技術評估、架構比較 | deep-research（deep） | 10-20 分鐘 |

| 全面綜述、年度回顧 | deep-research（ultradeep） | 20-45 分鐘 |

4.2 觸發方式

在 Claude Code 中直接輸入：


Use deep research to [研究問題]

4.3 本專案可借鏡的機制

  1. Anti-Fatigue Protocol：每個章節完成前檢查 5 項品質指標（段落數 ≥3、散文 ≥80%、無佔位文字、證據豐富、引用密度），可應用於自動任務的品質自評
  2. Persona-Based Critique：Deep/UltraDeep 模式的 2-3 角色紅隊審查（懷疑實踐者、對抗審稿人、實作工程師），可整合到 system-audit Skill
  3. First Finish Search (FFS)：基於品質閾值的自適應完成模式，避免過度搜尋浪費時間
