Dify 深度研究報告

概述

Dify（langgenius/dify）是一個開源的 AI-native 工作流開發平台，由 LangGenius 團隊於 2023 年創立。截至 2026 年 3 月，已累積 132K GitHub Stars 與 20.5K Forks，成為全球最受歡迎的開源 AI 應用開發平台之一。Dify 的核心定位是「讓所有人都能用 AI 創造」，提供從視覺化工作流設計、RAG 知識庫、AI Agent 框架到一鍵部署的完整 LLMOps 平台。與同類工具相比，Dify 不只是自動化工具（如 n8n），而是 AI-native 的應用開發環境。

本文為 AI 工作流自動化系列 foundation 階段，與已有的 n8n（通用工作流引擎）和 Langfuse（觀測性平台）研究互補。

技術架構

蜂巢架構（Beehive Architecture）

Dify 在 v1.0 版本全面導入「蜂巢架構」（Beehive Architecture），借鑒六邊形架構（Hexagonal Architecture）思想。核心理念：每個模組可獨立開發、測試、部署，如同蜂巢的六邊形格子，既自給自足又緊密協作。這使得開發者可以調整單一模組而不影響整體系統。

微服務元件

| 服務 | 技術 | 職責 |

|------|------|------|

| api | Python Flask + Gunicorn (port 5001) | REST API、工作流引擎編排 |

| web | Next.js (React) | Studio 前端 UI |

| worker | Celery | 非同步任務（資料集索引、排程作業） |

| beat | Celery Beat | 定期任務排程 |

| plugin-daemon | 獨立程序 (port 5003) | Plugin 隔離執行環境 |

| sandbox | 隔離沙盒 | Code 節點安全執行 |

| ssrf_proxy | Squid Proxy | SSRF 防護 |

| nginx | Nginx (port 80/443) | 反向代理 |

三層資料流模式

  1. 同步請求-回應：Client -> API -> 工作流引擎 -> LLM/Knowledge 服務 -> 回應（支援 Blocking 與 SSE Streaming）
  2. 非同步處理：API 將任務入列 Redis -> Celery Worker 消費 -> 更新資料庫（用於資料集索引等耗時操作）
  3. Plugin 執行：工作流引擎 -> Plugin Daemon -> 隔離 Plugin 程式碼 -> 外部 API（安全隔離執行）

儲存層

  • 主資料庫：PostgreSQL（應用、工作流、使用者、對話記錄）
  • 快取/訊息佇列：Redis（Celery Broker + 快取）
  • 向量資料庫：可配置（預設 pgvector，支援 Weaviate、Qdrant、Milvus、Chroma 等 11+ 種）
  • 檔案儲存：Local FS / S3 / Azure Blob / 各雲端供應商

工作流核心能力

視覺化工作流建構器

Dify 的拖拉式畫布支援以下節點類型：

  • LLM 節點：呼叫任意 LLM（GPT-4、Claude、Llama、Gemini 等），可為每個節點獨立選擇模型
  • 知識庫檢索節點：RAG 管線整合，文件上傳即自動分塊
  • 條件分支節點：依信心度、內容分類等條件路由
  • 程式碼執行節點：在沙盒中安全執行 Python/JavaScript
  • HTTP 請求節點：呼叫外部 API
  • 工具節點：50+ 內建工具（Google Search、DALL-E、Stable Diffusion、WolframAlpha）
  • Agent 節點：支援 Function Calling 與 ReAct 推理策略的自主決策節點

RAG Pipeline

  • 拖拉上傳 PDF、PPT、DOC 等文件，自動分塊
  • 混合檢索（向量 + 關鍵字搜尋）
  • 多模態知識庫：統一文字與圖片到同一語義空間
  • 50 文件知識庫標準 Q&A 場景約 85% 檢索準確率

Plugin 生態系統

v1.0 導入的 Plugin 系統包含五類核心元件：

  1. Models：AI 模型管理 Plugin（擴充模型供應商）
  2. Tools：領域專用工具 Plugin（增強 Agent 與工作流能力）
  3. Agent Strategies：推理策略 Plugin（支援多步驟推理的自主工具選擇）
  4. Extensions：自訂擴充
  5. Endpoints：自訂 API 端點

每個 Plugin 為獨立套件，可單獨開發、部署與維護，並有版本控制與安全機制。Dify 規劃進一步模組化 RAG Engine（拆分為 ETL、Embedding、Index Building、Data Recall）。

AI 整合機制

模型支援

Dify 展現真正的「模型無關」設計：

  • OpenAI：GPT-4、GPT-4o、o1 系列
  • Anthropic：Claude 3.5 / Claude 4 系列
  • Google：Gemini 系列
  • Meta：Llama 3 系列
  • 本地模型：Ollama 整合
  • Azure OpenAI、Hugging Face 模型
  • 支援 OpenAI 相容 API 的任何模型

LLM 整合方式

  • 原生抽象層：統一的 Model Provider 介面，切換模型只需改設定
  • 每節點獨立模型選擇：同一工作流中不同步驟可用不同模型（成本最佳化）
  • Prompt IDE：視覺化 Prompt 設計與模型效能比較
  • LLMOps：生產環境的監控、分析與持續改進

整合限制

  • 結構化輸入/輸出僅支援 1 層深度成員定義（巢狀結構只能取得 Object）
  • 多 Agent 編排能力基本但不如專用框架（如 CrewAI）
  • 畫布效能在 20+ 節點的複雜工作流中會降級

部署與維運

部署選項（6 種）

| 模式 | 適用場景 |

|------|----------|

| Dify Cloud（託管服務） | 快速上手，免維運 |

| Docker Compose（自建） | 中小型部署，最常見 |

| Kubernetes + Helm | 大規模生產環境 |

| AWS CDK / Marketplace AMI | AWS 原生部署 |

| Azure Terraform | Azure 原生部署 |

| Google Cloud Terraform | GCP 原生部署 |

系統需求

  • 自建 VPS：$20-40/月即可承載中小型工作負載
  • 雲端定價：免費 Sandbox → $59/月 Professional → $159/月 Team
  • 主要依賴：PostgreSQL、Redis、向量資料庫（可選）

API 認證

  • Bearer Token 認證（每個應用有獨立 API Key）
  • 支援 Blocking 與 SSE Streaming 回應模式
  • 端點：/v1/workflows/run、/v1/chat-messages 等

與 daily-digest-prompt 的連結

直接啟發

  1. 蜂巢架構 vs Skill-First：Dify 的模組化 Plugin 理念與 daily-digest-prompt 的 Skill 模組化異曲同工。兩者都追求「改配置不改核心」的設計哲學。Dify 的 Plugin 獨立生命週期管理可借鑑到 Skill 版本管理。
  1. 三層資料流：Dify 的同步/非同步/隔離三層模式，對應 daily-digest-prompt 的 Phase 0（預計算）→ Phase 1-2（並行 Agent）→ Phase 3（組裝）架構。Dify 用 Celery + Redis 做非同步佇列，daily-digest-prompt 用 PowerShell Start-Job + 檔案交接。
  1. 可配置向量資料庫：Dify 支援 11+ 種向量 DB，daily-digest-prompt 目前固定使用 localhost:3000 知識庫。未來若需擴展，可參考 Dify 的抽象層設計。
  1. Code Sandbox 隔離：Dify 的 Code 節點在沙盒中安全執行，類似 daily-digest-prompt 的 Hook 機器強制層（pre_bash_guard + sandbox）概念。

可借鑑機制

  • 模型路由：每節點獨立選擇模型的設計，可擴展到 daily-digest-prompt 的 config/llm-router.yaml（目前僅 Groq vs Claude 二選一）
  • Plugin Marketplace：Skill 若未來開放社群共享，可參考 Dify 的 Plugin 打包與分發機制
  • Prompt IDE：Prompt 模板的視覺化編輯與 A/B 測試，比目前手動編輯 .md 模板更高效

優缺點評估

優點：

  • 完整一站式平台：工作流 + RAG + Agent + 部署 + 監控
  • 真正的模型無關：100+ LLM 供應商整合
  • 蜂巢架構實現高度模組化，Plugin 獨立生命週期
  • 自建無授權費用（Apache 2.0 開源）
  • 部署選項豐富（6 種模式，從 Docker 到 K8s）
  • 活躍社群（132K Stars、9,239 commits）
  • 多模態知識庫統一語義空間

缺點 / 限制：

  • 結構化 I/O 僅支援 1 層深度（複雜巢狀需變通）
  • 企業級治理功能仍在成熟中（SSO/RBAC 文件分散）
  • 無內建面向終端使用者的前端（需自行開發）
  • 文件更新偶爾落後功能釋出
  • 多 Agent 編排較基本，不如 CrewAI、AutoGen 等專用框架
  • 畫布在 20+ 節點複雜工作流中效能降級
  • 元資料過濾等進階 RAG 功能需變通方案

與同類平台比較

| 面向 | Dify | n8n | Flowise |

|------|------|-----|----------|

| 定位 | AI-native 開發平台 | 通用工作流自動化 | LLM Pipeline 工作台 |

| Stars | 132K | 177K | ~30K |

| AI 深度 | 原生（RAG + Agent + Prompt IDE） | 外掛式（LangChain 節點） | 原生（RAG + Agent Chain） |

| 整合數量 | 100+ LLM + 50+ 工具 | 400+ 服務整合 | LangChain 生態 |

| 最佳搭配 | AI「大腦」—推理與 RAG | 「神經系統」—連接一切 | 快速 RAG 原型 |

| 部署 | Docker / K8s / Cloud | Docker / K8s / Cloud | Docker |

最佳實踐：Dify 負責 AI 推理邏輯（Brain），n8n 負責企業系統連接與工作流編排（Nervous System），兩者互補而非取代。
