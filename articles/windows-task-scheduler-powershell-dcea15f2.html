Windows Task Scheduler 在 PowerShell 7 自動任務中的應用實踐：工作目錄、退出碼與排錯範本

技術概述

Windows Task Scheduler 本身只是觸發器與宿主，不會替你的 PowerShell 腳本補齊工作目錄、相對路徑、權限模型或可觀測性。如果直接把 pwsh.exe 和腳本路徑塞進排程，最常見的結果不是「完全不能跑」，而是任務被觸發、History 也出現 Action started，但腳本在 %windir%\\system32、錯誤的安全內容或不透明的退出碼下靜默失敗。[來源：New-ScheduledTaskAction, 2026][來源：Troubleshoot issues with scheduled tasks not running, 2026][來源：about_Pwsh, 2026] 對 daily-digest-prompt 這類以 PowerShell 協調多個自動任務的專案，真正要設計的是一個穩定的「執行契約」：明確工作目錄、使用絕對路徑、保留可診斷退出碼、選對 LogonType/RunLevel，並把 Operational log 與 stdout/stderr 收集納入標準範本。

核心發現（Phase 4 三角佐證）

  1. 已驗證：WorkingDirectory 不是可有可無的附加欄位；若省略，Task Scheduler 會在 %windir%\\system32 執行，這會直接影響相對路徑腳本、模組載入與輸出位置。[3 來源：New-ScheduledTaskAction, 2026；IExecAction::get_WorkingDirectory, 2022；about_Pwsh, 2026]
  2. 已驗證：pwsh -File 比 -Command 更適合排程腳本，因為它保留腳本參數語意且能透過 exit N 傳遞數值退出碼；若用 -Command 包裝外部程式或腳本，常會把非 0/1 的退出碼壓扁成 1。[3 來源：about_Pwsh, 2026；schtasks query, 2024；Troubleshoot issues with scheduled tasks not running, 2026]
  3. 已驗證：自動任務的 principal 設計要先選 LogonType，再談 RunLevel。S4U 不存密碼，但沒有網路與加密檔案存取；ServiceAccount 適合 Local System/Local Service/Network Service；需要最高權限時再加 RunLevel Highest，否則維持 Limited 較安全。[3 來源：New-ScheduledTaskPrincipal, 2026；TaskFolder.RegisterTask method, 2026；Troubleshoot issues with scheduled tasks not running, 2026]
  4. 已驗證：Task Scheduler 的除錯不應只看 GUI 的「Last Run Result」。標準手順應同時檢查 History/Status、TaskScheduler Operational log、腳本 transcript 或 stdout/stderr 日誌，再用 HRESULT 對照表解碼 0x00041303、0x8004131C 等結果。[3 來源：Troubleshoot issues with scheduled tasks not running, 2026；Task Scheduler error and success constants, 2026；schtasks query, 2024]

核心概念與原理

第一個關鍵原理是「工作目錄與腳本路徑是兩個不同層次的問題」。New-ScheduledTaskAction 明確提供 -WorkingDirectory 參數，若未指定，Task Scheduler 會在 %windir%\\system32 執行任務。[來源：New-ScheduledTaskAction, 2026] Win32 的 IExecAction::get_WorkingDirectory 也指出，這個值會寫入 Task Scheduler XML 的 WorkingDirectory 元素，而且是在註冊任務時檢查路徑是否合法，而不是等任務執行時才發現錯誤。[來源：IExecAction::get_WorkingDirectory, 2022] 同時，pwsh 自己也支援 -WorkingDirectory，而且官方文件特別提醒從 cmd.exe 或批次環境呼叫時，-File 後的腳本路徑若寫成 .\test.ps1，呼叫端可能找不到它，應優先使用明確路徑。[來源：about_Pwsh, 2026] 這三個來源可以互相印證一件事：排程任務不要依賴「目前目錄剛好正確」，而要同時把 Task Scheduler action 的工作目錄與 pwsh 啟動參數設成可預期值。

第二個原理是「pwsh -File 是排程的預設安全路徑」。about_Pwsh 說明 -File 會把後續內容視為腳本路徑與腳本參數，而且 File 必須是命令列最後一個 pwsh 參數；更重要的是，腳本若正常結束則退出碼為 0，若以 exit 明確結束則回傳指定數值，遇到 script-terminating error 時則為 1。[來源：about_Pwsh, 2026] 相比之下，-Command 在某些情況會把外部退出碼折成 1，若想保留原始退出碼，官方建議手動 exit $LASTEXITCODE。[來源：about_Pwsh, 2026] 再配合 schtasks /query /v /hresult 可把結果轉成 HRESULT 形式查核，[來源：schtasks query, 2024] 以及 Microsoft 的疑難排解文件建議先在 Task Scheduler 外部直接測試腳本，再放進任務中執行，[來源：Troubleshoot issues with scheduled tasks not running, 2026] 可得到一個實務結論：排程腳本應盡量用 pwsh.exe -NoProfile -NonInteractive -File <絕對路徑>，需要傳遞外部程式退出碼時在腳本末尾明確 exit $LASTEXITCODE 或 exit 1。

第三個原理是「帳號模型會直接改變任務能力」。New-ScheduledTaskPrincipal 列出可用的 LogonType，包括 Password、S4U、Interactive、ServiceAccount 與 InteractiveOrPassword 等；同一份文件也說明 RunLevel 只有 Highest 與 Limited 兩種主要取值，範例包含使用 LOCALSERVICE 與 ServiceAccount，以及以 BUILTIN\\Administrators + RunLevel Highest 執行。[來源：New-ScheduledTaskPrincipal, 2026] 更底層的 TaskFolder.RegisterTask 文件則補齊安全語義：TASK_LOGON_S4U 不會儲存密碼，但沒有網路或加密檔案存取能力；TASK_LOGON_SERVICE_ACCOUNT 用於 Local System、Local Service、Network Service；TASK_LOGON_INTERACTIVE_TOKEN_OR_PASSWORD 雖可回退，但官方不建議新任務使用，因為可靠性較差。[來源：TaskFolder.RegisterTask method, 2026] 另外，Microsoft 的疑難排解指南直接建議在除錯時切換成「Run only when user is logged on」，藉此判斷是不是安全內容造成的失敗。[來源：Troubleshoot issues with scheduled tasks not running, 2026] 這表示排程設計不只是「能不能跑」，而是先決定任務需不需要網路、需不需要登入桌面、需不需要高權限，再反推 principal。

第四個原理是「可觀測性要靠三層訊號交叉驗證」。第一層是 Task Scheduler 內建 History 與 Status；Microsoft 文件明確要求先看 task 是否真的被觸發、是否長時間停留在 Running，以及是否能從 History tab 找到具體錯誤。[來源：Troubleshoot issues with scheduled tasks not running, 2026] 第二層是 TaskScheduler Operational log，官方在同一篇文章中把它列為支援案例前應蒐集的必要資料。[來源：Troubleshoot issues with scheduled tasks not running, 2026] 第三層是腳本本身的 transcript 或 stdout/stderr。官方建議 PowerShell 任務可啟用 transcript，Microsoft Q&A 的實務回應也建議先包一層 .bat 或啟動器，把開始/結束時間與標準輸出、標準錯誤導到每日日誌檔，以判斷卡住的是 Task Scheduler、cmd.exe 還是實際腳本。[來源：Task Scheduler operational issues, 2023] 最後再用 HRESULT 常數對照，例如 0x00041303 代表任務尚未執行過，0x0004131C 代表已註冊但 principal 可能缺少 batch logon privilege。[來源：Task Scheduler error and success constants, 2026]

最佳實踐與常見陷阱

最佳實踐的第一條，是把 action 寫成明確、可重現的啟動契約，而不是只填一個腳本檔名。建議至少固定四個元素：pwsh.exe 絕對路徑、-NoProfile、-NonInteractive、腳本絕對路徑；若專案依賴相對路徑，再同時設定 -WorkingDirectory 與 Task Scheduler action 的 WorkingDirectory。[來源：New-ScheduledTaskAction, 2026][來源：IExecAction::get_WorkingDirectory, 2022][來源：about_Pwsh, 2026]

第二條，是讓退出碼成為正式契約，而不是把所有錯誤吞成成功。pwsh -File 能保留 exit N，schtasks /query /v /hresult 能把結果直接轉成 HRESULT 觀察。[來源：about_Pwsh, 2026][來源：schtasks query, 2024] 若任務最後一步呼叫外部程式，腳本應顯式傳遞 $LASTEXITCODE，避免 Task Scheduler 只看到模糊的成功或單一錯誤碼。

第三條，是最小權限優先。若任務只需本機檔案和服務控制，ServiceAccount + LocalService 或 NetworkService 往往比一般使用者帳密更乾淨；若任務必須碰網路磁碟或加密內容，就不該選 S4U。[來源：New-ScheduledTaskPrincipal, 2026][來源：TaskFolder.RegisterTask method, 2026] RunLevel Highest 只在明確需要時使用，否則 Limited 更符合自動任務的最小權限原則。[來源：New-ScheduledTaskPrincipal, 2026]

第四條，是標準化排錯流程。先在互動式 shell 直接執行腳本，接著人工 Start-ScheduledTask 或 GUI 手動觸發，再看 History、Operational log、transcript 與 stdout/stderr；最後才對照 HRESULT 常數。[來源：Troubleshoot issues with scheduled tasks not running, 2026][來源：Task Scheduler error and success constants, 2026][來源：schtasks query, 2024]

常見陷阱則很固定。其一是省略 WorkingDirectory，導致腳本在 system32 執行而找不到相對檔案。[來源：New-ScheduledTaskAction, 2026] 其二是把 -Command 當成通用入口，最後失去原始退出碼或踩到引號解析差異。[來源：about_Pwsh, 2026] 其三是用 S4U 跑需要網路分享或憑證的工作。[來源：TaskFolder.RegisterTask method, 2026] 其四是只看 GUI 的 Last Run Result，而不收腳本輸出與 Operational log。[來源：Troubleshoot issues with scheduled tasks not running, 2026][來源：Task Scheduler operational issues, 2023]

與本專案的關聯

daily-digest-prompt 的 PowerShell 入口腳本、排程任務與結果 JSON 寫入流程，很適合直接套用這套模式。具體做法是：讓排程只負責啟動固定的 pwsh 啟動器；啟動器負責設定工作目錄、 transcript、結構化結果檔與明確退出碼；真正的研究或組裝腳本則只專注在任務本身。這樣可以把「排程層失敗」與「任務層失敗」清楚分開，避免今天這類 closed_task_ids 與歷史檔脫鉤時連排程是否正常都難以判斷。

對本專案最有價值的落地點有三個。第一，將所有 Windows 排程統一改成絕對路徑 + 明確 WorkingDirectory。第二，把 schtasks /query /v /hresult 與 TaskScheduler Operational log 收集整合進健康檢查。第三，對每個排程任務建立一份標準 principal 決策表，明確標記哪些任務能用 service account、哪些任務需要互動式登入或網路憑證。

程式碼範例


$taskName = "DailyDigest-AgentTeam"
$root = "D:\\Source\\daily-digest-prompt"
$pwsh = "C:\\Program Files\\PowerShell\\7\\pwsh.exe"
$script = Join-Path $root "run-todoist-agent-team.ps1"
$logDir = Join-Path $root "logs\\scheduled"

New-Item -ItemType Directory -Force -Path $logDir | Out-Null

$argument = @(
  "-NoLogo",
  "-NoProfile",
  "-NonInteractive",
  "-WorkingDirectory `"$root`"",
  "-File `"$script`""
) -join " "

$action = New-ScheduledTaskAction `
  -Execute $pwsh `
  -Argument $argument `
  -WorkingDirectory $root

$trigger = New-ScheduledTaskTrigger -Daily -At 6:30AM

$principal = New-ScheduledTaskPrincipal `
  -UserId "LOCALSERVICE" `
  -LogonType ServiceAccount `
  -RunLevel Limited

$settings = New-ScheduledTaskSettingsSet `
  -StartWhenAvailable `
  -MultipleInstances IgnoreNew

Register-ScheduledTask `
  -TaskName $taskName `
  -Action $action `
  -Trigger $trigger `
  -Principal $principal `
  -Settings $settings `
  -Description "Run daily-digest auto task pipeline" `
  -Force

若要快速檢查結果，可再執行：


Get-ScheduledTaskInfo -TaskName "DailyDigest-AgentTeam"
schtasks /query /tn "DailyDigest-AgentTeam" /fo LIST /v /hresult
