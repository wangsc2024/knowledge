文件驅動 Agent 管線之輕量 SWE 風格縮樣：Workflow YAML、Hook 守衛與結果檔 Schema 邊界：深度技術研究報告

執行摘要（200-400 字）

本研究以「文件驅動」每日排程 Agent 為場景，將 JSON Schema、OpenAPI、平台工作流 YAML、雲端 Policy as Code，以及本倉 docs/ARCHITECTURE.md 等七則獨立來源交叉對照，回答三個問題：可驗證工件有哪些、失敗語意如何從模型自律轉成可觀測檢查點、以及「最小可重現／縮樣」觀念如何映射到結果檔與守衛訊息。結論是：以 config/schemas/results-auto-task-schema.json 固定 Phase 2 結果 JSON 的合法終態；以 Hook 規則與 config/cursor-cli-architecture-guardrails.yaml 承接 [AG-CURSOR-*]／FIX: 修復閉環；再以 workflows/index.yaml 把「task_types→workflow」外置成索引，與 Skill-First preamble 形成雙軌路由。三角佐證下，契約外顯、YAML 程序索引與機器守衛三類敘述彼此強化；分歧主要在「驗證標的是資料實例或 HTTP 介面」以及「YAML 由誰執行」。整體可視為輕量 SWE 縮樣：用最少案例偵測契約漂移，而非追求單次執行窮盡所有邊界。

研究背景與動機

排程觸發的 CLI／多後端 Agent 在 Phase 0–3 之間會產生大量「設定檔＋結果 JSON＋日誌」。若規範僅存在長篇 prompt，漂移難以在 CI 或合併前被機器擋下。研究動機是把公開規格中已成熟的「契約＋管線」語言，對照本專案既有 SSOT，形成可教學、可審計、可逐步自動化的邊界模型 [來源：Daily Digest Prompt 架構文件（本倉 docs/ARCHITECTURE.md）, 2026]。

核心技術解析

JSON Schema 作為結果檔與資料契約的詞彙層。 JSON Schema 以機器可讀方式描述实例是否有效，支援型別、格式與組合約束，利於在管線中做結構化驗證；與 results-auto-task-schema.json 的角色一致 [來源：JSON Schema: A Media Type for Describing JSON Documents (Validation vocabulary, Draft 2020-12), 2020-12]。

OpenAPI 對齊 HTTP 往返模型。 OpenAPI 標準化描述 REST 介面，契約測試常見模式是以規格為單一事實來源並在 CI 比對實作；與知識庫 /api/import 這類邊界自然對齊 [來源：OpenAPI Specification（OAS）, 2026]。

YAML 工作流承載「有序步驟」。 GitHub Actions 以 on、jobs、steps 等鍵約束行為，官方文件提供結構化欄位說明，利於靜態檢查或 checklist；本倉 workflows/index.yaml 則把多份人類／Agent 可讀程序檔掛上 task_types 路由，語意重心在索引與觸發，而非單一執行器 [來源：Workflow syntax for GitHub Actions, 2026]、[來源：Daily Digest Prompt 架構文件（本倉 docs/ARCHITECTURE.md）, 2026]。

Policy as Code 與 Hook／guardrails 的治理方向一致。 Azure Policy 將合規規則外顯為可稽核、可自動評估的定義，強調持續評估與拒絕不符合狀態的變更；對照本專案是把高風險操作從「模型記得不要」改為 Hook 攔截與架構守衛訊息 [來源：Azure Policy 概觀（Policy as Code）, 2026]、[來源：Azure Policy 概觀（正體中文）, 2026]。

最小可重現與縮樣。 最小可重現錯誤報告透過縮小範圍，讓他人以最少步驟重現；映射到 Agent 管線即：單一最小 results/todoist-auto-*.json 案例＋對應 cursor_cli_result_guard 期望 stderr，用以偵測 schema 與 FIX: 指引是否漂移 [來源：最小可重現錯誤報告（中文維基）, 2026]。

關鍵洞見（含三角佐證狀態 ✅/⚠️）

  • 將 Phase 2 結果檔視為釋出工件：以 schema 與 AG-CURSOR-* 守衛定義允許終態，與 JSON Schema 驗證同一治理平面。已驗證✅（4+ 獨立敘述來源）。
  • workflows/index.yaml 為程序路由表：與 Skill-First preamble 的「先讀索引」形成 Skill／Workflow 雙索引。已驗證✅（3+ 來源）。
  • 縮樣＝最小結果 JSON＋守衛 stderr 期望路徑。已驗證✅（維基最小重現＋本倉架構敘述＋ schema 語意）。
  • Policy 持續評估 vs 本倉 Phase 0–3：雲政策偏資源態，倉庫偏文件與 JSON 工件，但「左移檢查」一致。已驗證✅。
  • 三角佐證可具體化為：公開規格＋平台工作流語法＋本倉 ARCHITECTURE／guardrails。已驗證✅。
  • 若僅依單一類來源（如部落格）聲稱 AG-CURSOR 根因，應標待驗證並補本倉日誌樣本。待驗證⚠️（計畫內已標記）。

實務應用建議（3-5 點）

  1. 新增或修改自動任務結果欄位時，同步更新 results-auto-task-schema.json 與相關守衛測試，避免 Phase3 才爆發格式失敗。
  2. 為高頻 task_key 維護一份最小合法結果 JSON fixture，在 CI 或本地跑 schema validate＋（若可）result guard 乾跑。
  3. 在 workflows/index.yaml 掛上新任務對應的 workflow 條目，讓 todoist-auto-* 與「結果檔診斷」等流程共用同一索引慣例。
  4. 對 KB 匯入與查詢分別思考：資料實例走 schema；HTTP 契約可走 OpenAPI 或契約測試矩陣，與 knowledge-query Skill 步驟對齊。
  5. 研究或審查報告中，對無法從本倉 artifacts 或日誌驗證的因果聲稱，一律降級為「待驗證⚠️」並列出缺口。

與現有知識的連結

本專案已在 CLAUDE.md／AGENTS.md 層級固定 Skill-First、state 唯讀邊界與 nul 禁令；在工具層有 tools/result_file_diagnostic.py、workflows/result-file-diagnostic-workflow.yaml 等閉環。本報告把這些元件放回「契約＋管線＋縮樣」的一般軟體工程語彙，便於對外溝通與對照業界文獻 [來源：Daily Digest Prompt 架構文件（本倉 docs/ARCHITECTURE.md）, 2026]。

未來展望

後續可挑選 1–2 個實際 partial／format_failed 結果檔案例，做成固定 fixture 並接入 pytest 或守衛迴歸；另可評估對 KB POST /api/import 維護最小 OpenAPI fragment（僅內部使用）以降低欄位漂移。若多後端配額與沙箱備援持續演進，宜將 task_execution／fallback_kind 欄位一併納入縮樣集合。
