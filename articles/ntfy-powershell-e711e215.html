ntfy 自架通知可靠性：快取、反向代理、PowerShell 重試與限流

技術概述

ntfy 很適合拿來做自動化任務的完成通知，但它的可靠性不是只看「POST 成功」而已。真正影響送達率的，是訊息是否可回補、反向代理是否正確保留 WebSocket 升級、伺服器是否把不同使用者錯算成同一個 visitor，以及發送端是否尊重 429 節流與逾時設定。官方文件顯示，ntfy 預設會把訊息快取 12 小時，並提供 /v1/health 檢查；若在 proxy 後方部署卻沒開 behind-proxy，限流會失真；若 Nginx 沒有轉送 Upgrade/Connection，行動端常見的就是 Reconnecting... 與延遲通知。[來源：ntfy Configuration, 2026] [來源：ntfy Troubleshooting, 2026] [來源：NGINX WebSocket proxying, 2024]

核心概念與原理

主張 1：ntfy 的「可靠送達」基礎是伺服器快取，不是單純即時推送

ntfy 預設將訊息保留 12 小時，且可用 since= 與 poll=1 取回先前訊息；若設定 cache-file 或 database-url，快取可跨服務重啟保留。[來源：ntfy Configuration, 2026] publish 文件也明確指出，若發送時加 X-Cache: no，訊息雖會送給當下已連線客戶端，但之後無法再透過 since= 或 poll=1 補抓。[來源：ntfy Publish, 2026] 這代表對 daily-digest 這種「通知可晚到，但不能無痕遺失」的流程，預設應保留快取，而不是追求零殘留。PowerShell 端應把「topic 已發布」與「用戶端一定已讀」拆開，並把 ntfy 視為具短期耐久層的通知匯流排，而不是純 fire-and-forget webhook。[來源：ntfy Configuration, 2026] [來源：ntfy Publish, 2026] [來源：Microsoft Learn Invoke-RestMethod, 2025]

主張 2：自架在反向代理後時，`behind-proxy` 與 WebSocket 升級設定是同一條可靠性鏈上的兩個必要條件

ntfy 官方明說：若部署在 proxy 後方卻沒開 behind-proxy，所有訪客會被當成同一個 visitor，限流判斷會集中到 proxy IP，導致實際流量很小也可能被錯誤打到 429。[來源：ntfy Configuration, 2026] 另一邊，troubleshooting 文件指出 self-hosted 若出現 Reconnecting...、通知晚 30 分鐘以上才到，需同時檢查 behind-proxy 與 reverse proxy 的 WebSocket 設定。[來源：ntfy Troubleshooting, 2026] NGINX 官方再補足根因：Upgrade 與 Connection 是 hop-by-hop header，不會自動轉送，必須顯式設定；而且預設 60 秒無資料就可能關閉連線，需視情況調高 proxy_read_timeout 或由後端送 ping frame 維持連線。[來源：NGINX WebSocket proxying, 2024] 三者合起來可得出一個實務結論：在 ntfy 自架環境裡，很多「通知不穩」其實不是 ntfy 發送 API 壞掉，而是 proxy 層把長連線可靠性切斷了。[來源：ntfy Troubleshooting, 2026] [來源：NGINX WebSocket proxying, 2024] [來源：ntfy Configuration, 2026]

主張 3：發送端應讓 PowerShell 的 HTTP 重試邏輯與 ntfy 的 rate limit 對齊，而不是自訂硬編碼 sleep

PowerShell 7.6 Invoke-RestMethod 已內建 MaximumRetryCount、RetryIntervalSec、ConnectionTimeoutSeconds、OperationTimeoutSeconds 等參數。[來源：Microsoft Learn Invoke-RestMethod, 2025] 更重要的是，官方寫明當伺服器回 429 且帶 Retry-After header 時，cmdlet 會優先採用該值，而非呼叫端自己指定的 RetryIntervalSec。[來源：Microsoft Learn Invoke-RestMethod, 2025] ntfy 官方則說明 visitor request limit 預設 burst 60、每 5 秒回補 1 個 token，超出會回 429 Too Many Requests。[來源：ntfy Configuration, 2026] 這表示 Windows 自動化腳本最穩定的做法，不是自己寫固定 Start-Sleep 5 重試，而是直接用原生 web cmdlet 的 retry 機制，讓 client 與 server 的節流語意一致。這能避免本專案在排程高峰時因平行任務同時推送通知而形成自我放大重試風暴。[來源：Microsoft Learn Invoke-RestMethod, 2025] [來源：ntfy Configuration, 2026] [來源：ntfy Configuration, 2026]

主張 4：高負載或長時間運行時，應把快取層視為需要調校的資料庫，而不是單純開關

ntfy 官方文件指出，快取預設使用 SQLite 同步寫入；對個人主機通常足夠，但高負載場景建議開啟 WAL，並可使用 cache-batch-size、cache-batch-timeout 做批次非同步寫入。若看到 database locked，官方建議啟用這些設定。[來源：ntfy Configuration, 2026] 官方也直接公開了 ntfy.sh 的調校範例：cache-batch-size: 25、cache-batch-timeout: "1s"，搭配 pragma journal_mode = WAL、pragma busy_timeout = 15000 等 SQLite 啟動查詢。[來源：ntfy Configuration, 2026] 同一份文件還提醒，若走 proxy，proxy 的 open files limit 可能要設為目標連線數的兩倍。[來源：ntfy Configuration, 2026] 這些細節代表真正的耐久度瓶頸常在快取 I/O 與連線上限，而不是 publish API 本身。[來源：ntfy Configuration, 2026] [來源：NGINX WebSocket proxying, 2024] [來源：ntfy Troubleshooting, 2026]

主張 5：排程型提醒應優先用 ntfy 的 delayed delivery，而不是額外加一層本機 scheduler

ntfy 支援 At / Delay / In header 做延遲投遞，最短 10 秒、最長 3 天；而且 scheduled message 會連同快取一起保留到投遞後一段時間。[來源：ntfy Publish, 2026] 這代表對「30 分鐘後提醒重跑」「明早提醒人工檢查」這類短期補償通知，直接交給 ntfy 排程通常比額外建立一條 Windows Task Scheduler 任務更簡潔，也更容易與原始通知串起同一條 topic 記錄。[來源：ntfy Publish, 2026] 但官方同時指出，這功能不能與關閉快取並用，所以如果你需要 delayed delivery，就不能用 X-Cache: no。[來源：ntfy Publish, 2026] 這是設計取捨，不是 bug。[來源：ntfy Publish, 2026] [來源：ntfy Configuration, 2026] [來源：Microsoft Learn Invoke-RestMethod, 2025]

最佳實踐與陷阱

最佳實踐：

  1. 自架預設開啟 cache-file 與合理 cache-duration，讓短暫斷線可補抓。[來源：ntfy Configuration, 2026]
  2. Proxy 後方務必同時設定 behind-proxy: true 與 WebSocket upgrade header；Nginx 額外考慮 proxy_read_timeout。[來源：ntfy Configuration, 2026] [來源：NGINX WebSocket proxying, 2024]
  3. 發送端使用 Invoke-RestMethod 內建重試與 timeout，不要自訂粗糙 retry loop。[來源：Microsoft Learn Invoke-RestMethod, 2025]
  4. 用 /v1/health 做 preflight，失敗時把通知視為降級而非整個任務失敗。[來源：ntfy Configuration, 2026]
  5. 若流量明顯升高，調整 SQLite WAL、批次寫入與系統 nofile 限制。[來源：ntfy Configuration, 2026]

常見陷阱：

  1. 發送時用了 X-Cache: no，之後卻期待客戶端重連能補到訊息。[來源：ntfy Publish, 2026]
  2. Proxy 沒轉送 Upgrade/Connection，造成 WebSocket 退化成不穩的輪詢體驗。[來源：NGINX WebSocket proxying, 2024] [來源：ntfy Troubleshooting, 2026]
  3. 多個排程共用同一 proxy IP，卻忘記 behind-proxy，導致所有任務共同吃同一個 rate limit bucket。[來源：ntfy Configuration, 2026]
  4. 高頻發送仍手寫固定 sleep 重試，忽略 Retry-After，讓壅塞更嚴重。[來源：Microsoft Learn Invoke-RestMethod, 2025] [來源：ntfy Configuration, 2026]

與本專案關聯

daily-digest-prompt 目前大量依賴 ntfy 做 Phase 告警與任務完成通知。這份研究的直接落點有四個：

  1. ntfy-notify 類腳本不應預設送 X-Cache: no，否則手機短暫離線時會漏訊息。
  2. 若未來把 ntfy 放到 Nginx 後方，部署腳本應同時驗證 behind-proxy、WebSocket header 與 proxy_read_timeout。
  3. PowerShell 發送層可改用 Invoke-RestMethod -MaximumRetryCount ... -RetryIntervalSec ... -ConnectionTimeoutSeconds ...，把 429/5xx 行為標準化。
  4. 對「延後提醒」場景，可直接利用 ntfy delayed delivery，少維護一組 Windows 排程任務。

程式碼範例


function Send-NtfyReliable {
    param(
        [Parameter(Mandatory)] [string] $BaseUrl,
        [Parameter(Mandatory)] [string] $Topic,
        [Parameter(Mandatory)] [string] $Message,
        [hashtable] $Headers = @{},
        [int] $MaximumRetryCount = 3,
        [int] $RetryIntervalSec = 5,
        [int] $ConnectionTimeoutSeconds = 10,
        [int] $OperationTimeoutSeconds = 30
    )

    $health = Invoke-RestMethod -Method Get -Uri "$BaseUrl/v1/health" \
        -ConnectionTimeoutSeconds 5 -OperationTimeoutSeconds 5
    if (-not $health.healthy) {
        throw "ntfy unhealthy"
    }

    $mergedHeaders = @{
        "Title" = "daily-digest"
        "Priority" = "default"
        "Tags" = "robot_face,information_source"
    }
    foreach ($key in $Headers.Keys) {
        $mergedHeaders[$key] = $Headers[$key]
    }

    Invoke-RestMethod -Method Post -Uri "$BaseUrl/$Topic" \
        -Headers $mergedHeaders \
        -Body $Message \
        -ContentType "text/plain; charset=utf-8" \
        -MaximumRetryCount $MaximumRetryCount \
        -RetryIntervalSec $RetryIntervalSec \
        -ConnectionTimeoutSeconds $ConnectionTimeoutSeconds \
        -OperationTimeoutSeconds $OperationTimeoutSeconds
}

# 一般完成通知
Send-NtfyReliable -BaseUrl "https://ntfy.example.com" \
    -Topic "digest-alerts" \
    -Message "tech_research finished"

# 30 分鐘後補償提醒
Send-NtfyReliable -BaseUrl "https://ntfy.example.com" \
    -Topic "digest-alerts" \
    -Message "Phase 3 still pending" \
    -Headers @{ "At" = "30m" }


map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

location / {
    proxy_pass http://127.0.0.1:2586;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection $connection_upgrade;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_read_timeout 3600s;
}
