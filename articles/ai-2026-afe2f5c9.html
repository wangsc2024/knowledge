技術概述

2025-2026 年的 AI 程式碼審查重點，已從「讓 LLM 幫你留言」轉向「如何把靜態分析、AI triage、AI 修補與 CI 驗證串成低噪音閉環」。核心問題不是能不能找出更多問題，而是如何避免重複 alert、壓低 false positive、讓真正高風險 issue 先被看見，並確保 AI 產出的修補建議在 merge 前真的通過測試與重新掃描。最新實務顯示，成功團隊會同時管理三類訊號：掃描結果 metadata、AI triage 狀態、修補後驗證結果，而不是只看單次 PR 評論。

核心方法與工具

1. 以 SARIF metadata 做風險排序與去重

  • GitHub Code Scanning 會用 partialFingerprints 追蹤「邏輯上相同」的結果，避免同一問題在多次掃描中重複出現。
  • properties.precision 可標記規則的真陽性可靠度，properties.security-severity 可標記安全風險分數；GitHub 會依這些欄位排序，讓高風險、高精度結果優先浮現。
  • GitHub 對每次 run 最多只納入前 5,000 筆結果，因此 noisy rules 不只干擾人，也會擠掉真正重要的警報。

2. 將 AI triage 與人工裁決分離

  • Semgrep Assistant 會對部分 finding 做 auto-triage，並把可疑 false positive 標為 provisionally_ignored，先從 PR 評論流中隱藏，但不等於永久關閉。
  • 實務上應保留 open、reviewing、fixing、ignored、fixed 等狀態，讓 AI 只處理預篩與排序，人類保留最後裁決權。
  • 對 recurring false positive，應保存「triage memories」或等價規則例外，讓相同上下文在未來掃描時自動降噪。

3. 將 AI autofix 視為「候選修補」，不是直接 merge 的 patch

  • GitHub Copilot Autofix 以 CodeQL alert、程式碼片段與 query help 生成建議；官方明示它是 best-effort，不能保證每次都成功。
  • 修補建議必須經過三重驗證：CI 測試是否通過、alert 是否消失、語意行為是否維持。
  • 最穩定的流程不是「AI 直接改」，而是「AI 提 patch + 靜態掃描回歸 + 測試回歸 + 人工覆核」。

實際工作流程（step-by-step）

  1. 先跑 CodeQL、Semgrep 或其他 SAST，統一輸出 SARIF。
  2. 在產出階段補齊 partialFingerprints、precision、security-severity，避免重複 alert 與排序失真。
  3. 匯入 GitHub Code Scanning，讓 PR 只顯示 diff 內、且排序後的重要結果。
  4. 對高風險 finding 啟用 AI explanation / suggested fix；對低精度規則只保留 advisory，不直接 block。
  5. 對 AI 判定可能為 false positive 的 finding，先標記 provisionally_ignored 或等價暫存狀態，不直接刪除。
  6. 安全或維護團隊批次審核被忽略的 finding，將確認安全的模式寫入 triage memory、rule suppression 或 allowlist。
  7. 對接受的 AI 修補建議，重新執行單元測試、整合測試、lint、SAST re-scan。
  8. 只有在「測試通過 + alert resolved + diff 無新高風險 finding」三條件同時成立時，才允許 merge。
  9. 將 triage 自動化率、誤報率、修補接受率、重開率寫入 dashboard，持續調整 blocking policy。

程式碼／配置範例

GitHub Actions：分級執行 CodeQL + Semgrep + 驗證閘門


name: ai-code-review-optimized

on:
  pull_request:
  push:
    branches: [main]

jobs:
  codeql:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: github/codeql-action/init@v3
        with:
          languages: javascript,python
      - uses: github/codeql-action/autobuild@v3
      - uses: github/codeql-action/analyze@v3

  semgrep:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: returntocorp/semgrep-action@v1
        with:
          publishToken: ${{ secrets.SEMGREP_APP_TOKEN }}
          publishDeployment: ${{ secrets.SEMGREP_DEPLOYMENT_ID }}

  verify-fix:
    needs: [codeql, semgrep]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test
      - run: npm run lint

SARIF 規格重點：降低 noise、提升排序品質

  "ruleId": "js/sql-injection",
  "level": "error",
  "partialFingerprints": {
    "primaryLocationLineHash": "8f1a9f0c:14"
  },
  "properties": {
    "precision": "high",
    "security-severity": "8.7",
    "tags": ["security", "external/cwe/cwe-89"]

效果量化

  • GitHub 表示 Copilot Autofix 已覆蓋 JavaScript、TypeScript、Java、Python 中超過 90% 的支援 alert 類型，且對支援漏洞有超過三分之二可用「少量或無需修改」的建議；GitHub 另稱 GitHub Advanced Security 可讓團隊修補速度達傳統工具的 7 倍。
  • Semgrep 2025 年資料指出，Noise Filtering 部署後可立即減少約 20% 待分析 finding；其安全研究團隊對 Assistant 判定的同意率約 96%，使用者對 auto-triage 決策同意率約 95%。
  • Semgrep 2025 年 9 月進一步表示，Assistant 已能處理約 60% 新進 SAST triage 工作，且使用者稽核同意率仍超過 96%。
  • 單一 Fortune 500 beta 使用者在加入 5 個 triage memories 後，額外獲得 2.8 倍的降噪改善。

與本專案的應用場景

daily-digest-prompt 不是大型產品程式庫，但有大量 prompt、PowerShell、JSON、YAML 與 workflow 規則，最容易出現的是「規則誤報」「重複告警」「AI 建議修補後造成流程退化」。可直接套用的做法：

  • 把 hooks、PowerShell 腳本、prompt 規範檢查結果統一轉成 SARIF，集中進 GitHub Code Scanning 或同級 dashboard。
  • 對已知 benign pattern 建立 triage memory，例如 PowerShell 合法 here-string、Windows 路徑、prompt 模板中的固定關鍵詞，避免每次都被重報。
  • 對會改動 workflow 的 AI 修補建議，強制要求通過 npm test、必要 smoke test、以及重新掃描 hooks / prompt 規則後才可採納。
  • 針對 results/、context/、templates/ 等不同區域設定不同 blocking policy：執行路徑與安全規則採 blocking，內容檔與報告檔採 advisory。

風險與注意事項

  • AI triage 不能直接等同於真實 exploitability；任何自動忽略策略都需要可回溯與可重開。
  • precision 與 security-severity 如果填錯，排序會失真，反而讓高風險 issue 被淹沒。
  • 若 SARIF 缺少穩定 fingerprint，重跑掃描會產生重複 alert，讓指標失真。
  • AI autofix 可能修掉警報卻引入語意回歸，因此一定要配回歸測試與 re-scan。
  • 對小型專案而言，最常見失敗不是模型不夠強，而是 policy 太激進，導致評論噪音高於修補價值。
