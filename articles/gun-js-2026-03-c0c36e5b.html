Gun.js 加密聊天室任務指派系統：完整除錯研究報告

日期：2026-03-01

專案：daily-digest-prompt / wsc-bot

狀態：✅ 全工作流程驗證成功

一、系統架構概述

本系統實現一個基於 Gun.js 的去中心化加密聊天室，使用者可透過 UI 指派任務，Bot 接收並儲存，Worker 認領執行（claude -p），結果加密廣播回聊天室。

核心流程：index.html → SEA.encrypt → Gun Relay → bot.js 接收儲存 → process_messages.ps1 認領 → claude -p 執行 → /api/records/processed → bot.js sendSystemReply → Gun Relay → index.html 解密顯示

二、SEA 加密協定詳解

金鑰類型區分（關鍵易混淆點）

  • ECDH 金鑰對（epub/epriv）：Curve25519，用於計算共享密鑰 SEA.secret()
  • ECDSA 金鑰對（pub/priv）：P-256，用於簽章驗證 SEA.verify()

正確驗證方式：

  • ❌ 錯誤：SEA.verify(hw.sig, hw.epub) — 用 ECDH epub 驗 ECDSA 簽章，永遠失敗
  • ✅ 正確：SEA.verify(hw.sig, hw.pub) — 用 ECDSA pub 驗，正常運作

訊息格式（與 index.html 完全一致）


// 加密內容
const payload = JSON.stringify({ text: userMessage, ts: Date.now() });
const encryptedData = await SEA.encrypt(payload, sharedSecret);
// Gun 路徑
const msgId = 'msg_' + randomHex(12);
gun.get(chatRoomName).get(msgId).put(encryptedData); // 直接放，不包裝

三、握手協定

Bot 發布：gun.get('wsc-bot/handshake').put({ epub, sig, pub }) 含 ECDSA pub 供驗證

Bot 兼容路徑：gun.get('wsc-bot/handshake').get('bot-epub').put(myPair.epub)

Bot relay 重連補發：gun.on('hi', () => 補發 epub)（解決 Render.com 重啟問題）

客端 epub 發布（握手）：

  • gun.get('wsc-bot/handshake').get('client-epub').put(myPair.epub) // 向下相容
  • gun.get('wsc-bot/handshake').get('clients').get(myPair.pub).put(myPair.epub) // 多用戶
  • 發布後需等待 8 秒，讓 Bot 建立 sharedSecret

四、發現的 6 個 Bug 與修復

Bug 1：SEA.verify 使用錯誤金鑰

根因：SEA.verify(hw.sig, hw.epub) 永遠失敗（epub 是 ECDH 非 ECDSA）

修復：改為 SEA.verify(hw.sig, hw.pub)

檔案：D:\Source\my-gun-relay\index.html

Bug 2：Relay 重啟後 Bot epub 消失

根因：bot.js 只在 init() 發布一次 epub，relay 重啟後資料消失未補發

修復：gun.on('hi', ...) 加入補發邏輯，epubSig 升級為模組層級變數

Bug 3：Bot 自身回覆被誤存為新任務

根因：SEA.decrypt({text,ts}) 回傳物件，String(obj)='[object Object]' 繞過過濾

修復：先檢查 typeof raw === 'string'，否則取 raw.text

Bug 4：settings.json Hook 路徑損壞

根因：bash 將 D:\\Source\\... 的反斜線解析為逃逸字元，路徑損壞

修復：所有 hook command 改用正斜線 D:/Source/daily-digest-prompt/hooks/xxx.py

Bug 5：Worker CLAUDECODE 巢狀 Session 阻擋（最關鍵）

根因：Claude Code 設定 CLAUDECODE 用戶環境變數，Task Scheduler 繼承後 claude -p 拒絕啟動

症狀：Worker log 顯示 claude -p 在 <1 秒內完成，結果是錯誤訊息

修復：process_messages.ps1 在 claude -p 前 Remove-Item Env:\CLAUDECODE，finally 還原

wsc-bot01 不受影響：因使用 OpenAI codex exec（無 CLAUDECODE 限制）

Bug 6：測試腳本訊息格式錯誤

根因：put({ d: encrypted, ts }) 包裝格式，bot 嘗試 SEA.decrypt({d,ts}) 失敗

修復：直接 put(encryptedData)，且加密 JSON.stringify({text,ts}) 非純文字

六、Gun.js 重要行為特性

  • .once() null 行為：無快取時立即以 null 回呼，需多次 retry + fallback
  • .map().on() 歷史資料：附加後對現有所有資料觸發回呼，需用 processedMessages 去重
  • Render.com Free Tier：每 ~15 分鐘重啟，radisk 資料消失，需 relay reconnect republish

七、架構建議

  1. 使用本地 relay 開發（localhost:8765）比 Render.com 穩定
  2. epub 握手等待 8 秒確保 bot 建立 sharedSecret
  3. Task Scheduler 執行 claude -p 前必須清除 CLAUDECODE
  4. 嚴格遵守訊息格式：JSON.stringify({text,ts}) + 直接 put
  5. 多路徑握手：同時發布 client-epub 和 clients/
