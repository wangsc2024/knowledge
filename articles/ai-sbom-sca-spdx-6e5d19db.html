AI 驅動的 SBOM／供應鏈風險摘要與升級建議管線（SCA × SPDX × LLM）

技術概述

軟體物料清單（SBOM）把「專案實際帶進來的第三方元件與版本」結構化；軟體組成分析（SCA）再把已知的授權、CVE 與惡意樣態對照到 SBOM。當依賴數量變大，真正的瓶頸往往不是「有沒有掃到」，而是「人能不能在時間內做完 triage 與升級決策」。LLM 的定位不是取代 SCA，而是把 OSV／SARIF／diff 等證據整理成可審核的摘要與建議，並強制附上 Advisory ID、套件座標（PURL／pypi npm）、影響版本區間，讓人類與 CI 規則能做最後裁決。

核心方法與工具

| 層次 | 目的 | 常見選項 |

|------|------|----------|

| SBOM 產生 | 產出可交換清單 | SPDX（spdx-json）、CycloneDX（json/xml） |

| 產生器 | 從檔案系統／映像掃描 | Syft、官方語言套件匯出 |

| 漏洞比對 | 以 SBOM 或 lockfile 對 CVE | Grype、Trivy、osv-scanner |

| 權威資料 | 漏洞座標與影響版本 | OSV、各發行端 GHSA／NVD 等 |

| 供應鏈完整性（可選） | 產線是否被竄改 | SLSA provenance／build audit |

| 政策合規脈絡 | 為何要做 | 美國 NTIA SBOM 要素討論、歐盟《網路韌性法》（CRA）趨勢下的產品安全義務 |

實際工作流程（step-by-step）

  1. 鎖定邊界：定義本次 SBOM 覆蓋範圍（repo 根目錄、dist、容器映像、或單一服務目錄）。
  2. CI 產 SBOM 工件：於 pull_request 或 release 流程輸出 sbom.spdx.json／bom.json，上傳為 build artifact（保留版本與 commit SHA）。
  3. SCA 掃描：以 SBOM 或磁碟掃描產生 grype.json／trivy.json／osv.json；只保留可對應到 明確 advisory id 的項目進入「高信號」佇列。
  4. 風險分群：依嚴重度、是否 reachability（若有 call graph／靜態分析）、授權衝突、是否在執行路徑上，將 findings 分桶（例如 blocker / next_sprint / false_positive_candidate）。
  5. LLM 摘要閘門（可選）：提示詞必要求輸出表格欄位：advisory_id、package、ecosystem、introduced_fixed、evidence_links、recommended_action。禁止沒有 id 的「籠統描述」進 PR 評論。
  6. 人機裁決：維運／資安在 PR 標籤或 security tab 簽核；合併後把決策寫回 results 或 issue 以便下次差分。
  7. 升級與迴歸：對 fix 版本開 branch；跑單元測試／最小 E2E；必要時用 uv lock／npm audit fix 類指令產可重現鎖檔。

程式碼／配置範例

以 Syft 產 SPDX JSON（本機或 CI）


# 範例：掃描目前目錄（請在 repo 根目錄執行）
syft packages dir:. -o spdx-json=sbom.spdx.json

以 Grype 對 SBOM 比對漏洞


grype sbom:sbom.spdx.json -o json > grype-report.json

GitHub Actions（概念片段：於既有 workflow 加一步）


# 片段示例：實際 action 名稱／版本請依官方文件更新
- name: Generate SBOM (SPDX JSON)
  uses: anchore/sbom-action@v0
  with:
    path: .
    format: spdx-json
    artifact-name: sbom.spdx.json
若專案使用 uv／npm／pnpm，請優先以鎖檔（uv.lock／package-lock.json）作為真實來源，SBOM 產生器配置需與實際安裝路徑一致。

效果量化（保守陳述）

  • 合規與採購：CRA、美國行政命令 EO 14028 脈絡下，SBOM 已成多邊市場的「預設對話語言」；導入成本主要在治理與工具鏈，而非單次掃描。
  • 工程生產力：將 SCA 原始 JSON 變成「10 行可決策摘要」通常能把 PR 資安 review 從「逐条讀 scanner 輸出」改成「對準 blockers」；（實際節省比例依團隊與工件品質差異極大，應以自家 DORA／變更前置時間度量）。
  • 偵測覆蓋：SBOM≠安全；它是 事實基底。真正的覆蓋率仍取決於 lockfile 是否完整、scanner DB 是否更新、以及是否做 reachability。

與本專案（daily-digest-prompt）的應用場景

  • Python／Node 依賴治理：pyproject.toml＋uv.lock、可能的 MCP／CLI 外掛，可用 SBOM 追蹤「實際進 runtime 的套件集合」，對照 config/dependencies.yaml 與排程腳本 imports。
  • Agent 工具鏈：Skills、hooks、外部二進位／CLI（見 ADR 與 GitHub Scout 盤點）可比照「小型供應鏈」；LLM 摘要可把多份 scanner 輸出統一成 improvement-backlog 可吞食的格式。
  • 可追溯性對齊：與既有的 post_tool_logger JSONL／error-object 思路一致：每個高風險結論要能回指 artifact（SBOM hash、SARIF run id），而非只剩自然語言。

風險與注意事項

  • 幻覺與錯配：LLM 可能誤連 CVE 與套件名；務必要求輸出帶 OSV／GHSA id，並以 scanner 結果做雙重驗證。
  • 秘鑰與資料外洩：勿把內部-only SBOM／漏洞細節貼到公開模型；自建网关或 redact。
  • 維運負擔：SBOM 每 commit 產出可能很大；可改為「日構建／release-only」並儲存於 artifact storage。
  • 授權誤判：SCA 對授權衝突的 heuristic 仍可能誤報，需法務／OSS policy 覆核。
