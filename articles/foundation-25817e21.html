  • 天台智者一系以五時縱貫釋尊一代說法：華嚴、鹿苑（阿含）、方等、般若、法華涅槃。首時「華嚴」在化儀上屬頓（直對大根，不歷小乘施設），在化法上則常判為「兼」——同部廣演中，有圓機所領之圓實，亦有權機所見之別教行布；故不可把「頓教」誤等同於「僅圓教」，亦不可把華嚴部與後三時的漸門混為一談。蕅益《教觀綱宗》全書宗旨，即在教觀雙運下讀懂此種「部／教」雙軌。
  • > 原文：「夫教也者，聖人被下之言；觀也者，衆生進道之行。無教，則觀無所從起；無觀，則教徒設於虛。故有教無觀則罔，有觀無教則殆。」
  • 白話：教法是聖人應機施設的言教；觀行是眾生依教起修的進路。沒有教，觀失去依據；沒有觀，教成空談。偏一邊會迷罔或危殆。
  • 依《天台四教儀》總綱，五時為：一華嚴時、二鹿苑時（四阿含）、三方等時、四般若時、五法華涅槃時；並以五味相生（乳→酪→生酥→熟酥→醍醐）喻化儀之由淺入深（約鈍根歷五味；利根亦可味味入實）。
  • 《四教儀》釋頓教章明言：從部、時、味得名為頓——如來初成正覺，於寂滅道場為四十一位法身大士等圍繞，「如雲籠月」，現盧舍那身，說圓滿修多羅。然「若約機約教，未免兼權」：經中「初發心時便成正覺」等，對圓機說圓教；「處處說行布次第」則對權機說別教——故約部判屬頓教華嚴部，約教則稱「兼」（兼別圓等），非單一化法可盡。

核心內容摘錄：

  • 2026 年文件自動化的重點，已從「讓 LLM 幫你寫幾段說明」轉成「把文件變成可被人與 AI 同時消費的持續交付資產」。實務上，核心不是單點生成，而是建立一條以 OpenAPI 為單一真相來源（single source of truth）的文件管線：PR 階段先生成摘要，release 階段自動彙整變更，API 描述則保留機器可讀結構，再透過 MCP 與 llms.txt 對外暴露給代理與 IDE。這條路線的價值在於減少手寫重複、降低規格漂移，並讓 AI 助手直接讀規格、查文件、執行工具，而不是反覆吃長篇 Markdown。研究上，DocAgent 與 RepoAgent 顯示多代理與 repo-level context 對文件完整性與真實性有明顯幫助；LAPIS 更進一步指出，把 API 規格壓縮成 LLM 友善格式，可顯著降低 token 成本。
  • 1. 契約先行：OpenAPI Initiative 明確建議 design-first，並把 OAD 放入版本控制與 CI，避免程式實作與文件分岔。 2. 文件雙用途：OpenAPI summary / description / examples 既供人讀，也供 Swagger UI、ReadMe、Mintlify 等自動產生參考文件。 3. PR 級自動摘要：GitHub Copilot 可直接生成 PR 摘要；GitHub 文件明示應從空白描述開始，避免既有文字污染摘要。 4. Release 級自動變更日誌：GitHub 自動 release notes 會彙整 merged PR、contributors 與 full changelog；若要更細分，可配 Release Drafter 依 label 分類與解版號。 5. AI 原生文件平台：ReadMe 提供 Agent、Linter、Docs Audit、MCP Server；Mintlify 則提供 Agent Job、搜尋、頁面內容擷取與 llms.txt，可把 docs 變成可程式化維護的系統。 6. LLM 友善規格層：LAPIS 實測對 5 組 production API 規格，相較 OpenAPI YAML 平均減少 85.5% token、相較 JSON 減少 88.6%，適合作為 agent runtime 的壓縮表示。 7. 生成品質守門：DocAgent 採 Reader / Searcher / Writer / Verifier / Orchestrator 五角分工；ReadMe Linter 則把風格規範、錯字、broken links 檢查搬進文件工作流。
  • 1. 在 repo 維護 openapi.yaml，規定 API 變更先改規格，再改程式與 SDK。 2. PR 建立時，讓 Copilot 先產生變更摘要；作者只補「為什麼改」與風險說明。 3. CI 依 OpenAPI 重新生成或同步 API docs，並執行 style/lint 檢查。 4. 若使用 ReadMe，將 OpenAPI 與 Markdown 同步進 docs hub，再跑 Linter 或 Agent 修正文案與連結。 5. 若使用 Mintlify，可在 CI 或排程中呼叫 Agent Job/trigger update，批次修訂頁面與重新部署。 6. Release 前由 GitHub 自動 release notes 或 Release Drafter 聚合 PR，輸出 changelog 初稿。 7. 對外提供 llms.txt 與 MCP server，讓 IDE/agent 可直接 search/read/spec-execute，而不是只看靜態頁面。 8. 若要給 agent runtime 更小上下文，將 OpenAPI 轉成 LAPIS 或其他壓縮視圖，只保留操作、參數、錯誤、rate limit 與 flow 宣告。 9. 對 repo-level 文件生成需求，採 DocAgent/RepoAgent 類型架構，讓 verifier 負責 truthfulness，避免單輪摘要捏造。
