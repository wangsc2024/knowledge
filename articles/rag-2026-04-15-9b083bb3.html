一、執行摘要

  • Harness 改進比模型升級更有效（LangChain 實證：+26% 成功率）

二、五大維度深度洞見

2.1 架構創新：三層治理架構收斂

核心概念：

Claude Code Hooks、Google ADK、Stripe 不約而同採用 Guard-Orchestration-Execution 三層防護架構，證明此為業界最佳實踐。

關鍵技術：

  • Layer 1（Guard）：即時攔截不當操作（Claude Code PreToolUse Hook）
  • Layer 2（Orchestration）：動態注入與路由（Google ADK Agent 動態路由）
  • Layer 3（Execution）：子 Agent 分支執行（Stripe Agent Toolkit）
  • Mastra Observational Memory：實現 3-40× 上下文壓縮比（LongMemEval 94.87% 分數優於 RAG 3+ 分）
  • MCP + A2A 雙軌標準：垂直工具整合（MCP）+ 水平 Agent 協作（A2A），已獲四大廠商背書

實施建議：

  1. P0-緊急：升級 Claude Code 至 v2.0.10+，支援 PreToolUse Hook
  2. P1-重要：實施 Observational Memory 機制（digest-insights.json 已完成 Phase 1）
  3. P2-改進：評估 MCP 整合於知識庫 API 查詢（參考 Mastra 實作）

---

2.2 成本優化：複合效益最大化

核心概念：

模型路由、快取策略、Token 預算管控三管齊下，可實現 60-80% 複合成本節省。當前專案 Cache Read Tokens 達 77.14M（$144 累計成本），但缺乏監控體系。

關鍵技術：

  • 模型路由：節省 40-97% 成本（強化學習路由：91.0% 精度，成本僅 $0.3 vs $4.8）
  • 快取策略：節省 50-90% 成本（Anthropic Prompt Caching：90% 讀取成本節省）
  • KV cache-aware routing：疊加路由+快取雙重節省（llm-d, Red Hat 2025）
  • 信心度串聯式路由：達 GPT-4 97% 精度，僅需 24% 成本

實施建議：

  1. P0-緊急：建立 Cache Read Tokens 監控儀表板（當前 $144 成本無監控）
  2. P1-重要：實施模型路由策略（小任務 Haiku、複雜任務 Opus），預期節省 40-60% 成本
  3. P2-改進：優化快取命中率從 28.7% → 40%（參考 Anthropic Prompt Caching）

---

2.3 可觀測性：主動告警 + 根因分析 + 趨勢預測

核心概念：

OpenTelemetry GenAI semantic conventions 成為跨工具追蹤標準。當前專案 check-health.ps1 有 30 個靜態區塊，但缺乏實時告警觸發。

關鍵技術：

  • OpenTelemetry GenAI：gen_ai.* 屬性命名空間（工業標準）
  • 三層日誌架構：基礎設施健康 + 語意相關性 + 安全違規監控
  • Langfuse：整合 OTel + prompt 管理 + evaluation（25K stars）
  • 分散式追蹤：跨工具 trace correlation、Token 計量、自動化評估

實施建議：

  1. P0-緊急：建立實時告警觸發機制（整合 ntfy，建立 config/alert-rules.yaml）
  2. P1-重要：補充 post_tool_logger.py 的 OTel GenAI span 屬性
  3. P3-長期：評估 Langfuse 整合（self-host 部署滿足資料隱私）

---

2.4 品質閘門：Harness 改進優於模型升級

核心概念：

LangChain 實證：僅改 harness（不換模型）可提升 26% 成功率（Terminal Bench 2.0 從 52.8% → 66.5%）。當前專案每日成功率僅 70.5%（目標 90%），差距 19.5 個百分點。

關鍵技術：

  • Harness Engineering 四大組件：架構約束、回饋迴路、驗證護欄、生命週期管理
  • Hook 驅動品質閘門：PreToolUse 攔截 + 同步驗證
  • 約束驗證自動化：pre-commit hooks + CI/CD 整合
  • Fitness Functions：於 CI/CD 建立失敗叢聚分析 workflow

實施建議：

  1. P0-緊急：修復兩個高頻失敗點（assembly_side_effects_missing 34 次、result_file_missing 11 次），預期提升成功率 15.5%（從 70.5% → 86%）
  2. P1-重要：實施 Fitness Functions 於 CI/CD，建立失敗叢聚分析 workflow
  3. P2-改進：建立 config/quality-gates.yaml 品質閘門配置

  • Runaway Sessions 偵測：msgs>500 或 cache_read>50M

實施建議：

  1. P0-緊急：修復高頻失敗點（預期提升成功率 15.5%）
  2. P1-重要：建立失敗叢聚分析 workflow（參考 workflow-forge）
  3. P2-改進：優化熵分數至 <0.5（分析熵分數上升原因，識別複雜度熱點）
