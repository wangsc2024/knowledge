MCP（Model Context Protocol）生態系 2026 全景分析

一、概述

模型上下文協議（Model Context Protocol, MCP）是 Anthropic 於 2024 年 11 月開源的標準化協議，旨在統一 AI 模型與外部工具、資料源的整合方式。被譽為「AI 界的 USB-C」，MCP 在短短一年內從內部實驗成長為全行業採用的開放標準。截至 2026 年 3 月，MCP SDK 月下載量超過 9,700 萬次，活躍 MCP Server 超過 10,000 個，已被 Claude、ChatGPT、Gemini、Microsoft Copilot、Cursor、VS Code 等主流平台原生支援。

然而，2026 年也出現重大反面聲音：Perplexity 宣布棄用 MCP，轉回 REST API + CLI 方案，引發「MCP 是否過度設計」的產業辯論。本文多面向分析 MCP 的技術架構、生態發展、企業挑戰與爭議，為 AI Agent 系統設計提供決策參考。

---

二、技術架構

2.1 三層角色模型

| 角色 | 職責 | 類比 |

|------|------|------|

| MCP Host | AI 應用程式（Claude Desktop、Cursor IDE） | 瀏覽器 |

| MCP Client | Host 內嵌的通訊中介，管理與單一 Server 的連線 | HTTP Client |

| MCP Server | 外部工具/資料源的封裝，暴露 Tools/Resources/Prompts | Web Server |

2.2 通訊協議

  • JSON-RPC 2.0：支援 Request/Response、單向 Notification、有狀態 Session
  • 傳輸類型：stdio（本地）、HTTP + SSE（遠端，2025 年已上線）
  • Session 管理：維持對話上下文，支援斷線重連

2.3 三大原語（Primitives）

  1. Tools：AI Agent 可呼叫的外部函式（API 呼叫、工作流觸發、系統操作）
  2. Resources：可讀取/修改的外部資料（檔案、資料庫、即時系統資訊）
  3. Prompts：根據可用上下文動態調整的提示模板

2.4 安全模型

  • 使用者同意機制：所有操作需經使用者核准
  • 認證：OAuth 2.1（進行中）、Personal Access Tokens (PATs)、API Keys
  • 最小權限原則：Server 只暴露必要的存取範圍
  • 沙箱隔離：Server 運行在隔離環境中
  • 審計日誌：完整操作追蹤

三、2026 年路線圖四大焦點

3.1 傳輸可擴展性（Transport Scalability）

  • MCP Server Cards：透過 .well-known URL 暴露結構化 Server 元資料，讓瀏覽器/爬蟲/Registry 無需連線即可探索 Server 能力
  • Server Cards 工作組（WG）：協調標準制定
  • 邊緣/設備端輕量實作：降低運行門檻

3.2 Agent 通訊（Agent Communication）

  • Tasks 原語：提供「立即呼叫/稍後取回」的可靠模式
  • Agent Graphs：階層式組織 + 命名空間隔離
  • 非同步長時間任務：支援跨斷線的持久化操作
  • 監督 Agent：管理專責 Agent 團隊

3.3 治理成熟度（Governance Maturation）

  • AAIF（Agentic AI Foundation）：2025 年 12 月，Anthropic 將 MCP 捐贈給 Linux Foundation 旗下的 AAIF，由 Anthropic、Block、OpenAI 共同創立
  • SEP-1302：正式化工作組與興趣組
  • SEP-2085：繼任與修正程序
  • 結構化版本控制：MCP 1.1 → 2.0 明確升級路徑

3.4 企業就緒度（Enterprise Readiness）

  • 審計追蹤與合規：GDPR/HIPAA 支援
  • SSO 整合認證：與企業 IdP 整合
  • Gateway 行為標準化：統一代理行為
  • 配置可攜性：跨環境部署

五、反面觀點：Perplexity 棄用 MCP 的爭議

5.1 核心批評：Token 開銷問題

2026 年 3 月，Perplexity CTO Denis Yarats 於 Ask 2026 開發者大會宣布棄用 MCP，核心論點：

  • 72% 上下文浪費：典型 MCP 重度部署中，工具 Schema + 協議開銷在 Agent 處理任何用戶意圖前就消耗了 72% 的可用上下文窗口
  • 244 倍 Token 差距：2,500 個 API 端點作為 MCP Tools 需約 244,000 Token 描述；改用 CLI/API Client 模式僅需約 1,000 Token
  • 多 Server 認證摩擦：跨多個 MCP Server 的認證引入的可靠性衰減
  • 功能過剩：多數 MCP 功能在生產環境中從未被使用

5.2 CLI 替代方案的論點

  • AI 模型直接撰寫程式碼呼叫已授權的 API Client，而非透過 MCP 協議轉發
  • CLI 是「AI 最好的朋友」——存在數十年、穩定、輕量
  • Y Combinator CEO Garry Tan 獨立得出相同結論：CLI Agent 整合比 MCP 更可靠、更快

5.3 多方呼應

  • Cloudflare：指出 AI Tool Calling 機制存在根本問題
  • Apideck：同樣轉向 CLI/API 方案
  • Y Combinator：建構 CLI-based Agent 整合

5.4 平衡觀點

「MCP 沒有死，但它不再是預設答案。」— Micheal Lanham
  • Perplexity 的場景特殊：搜尋引擎需要極致 Token 效率，MCP 的「萬能接頭」設計不適合
  • Google 和 AWS 同時加倍投入 MCP，因其適合多工具協作的 Agent 場景
  • MCP 適合「工具多樣性高、需要標準化互操作」的場景
  • CLI/API 適合「端點數量大、需要極致 Token 效率」的場景

六、安全警告

研究發現 43% 的已測試 MCP 實作存在命令注入漏洞，突顯在廣泛企業部署前仍需大量安全強化工作。主要風險：

  1. Prompt Injection：透過 MCP Tool 回傳的資料注入惡意指令
  2. Session Hijacking：竊取 MCP 會話
  3. Over-Privileged Access：Server 授權範圍過大
  4. Token Theft：認證 Token 洩露

防護建議：參數驗證（Zod 等）、全面錯誤處理、後端資源 API 獨立認證、最小權限 Server 設計。

---

七、對 Daily Digest 專案的啟示

本專案已使用 MCP 工具（Slack、Cloudflare、GoDaddy），以下是基於研究的建議：

  1. 監控 Token 開銷：隨著 MCP Server 數量增加，注意上下文窗口消耗
  2. Server Cards 關注：當 .well-known 標準成熟後，可用於自動化 Server 發現
  3. 安全強化：確保所有 MCP Tool 回傳的資料經過消毒處理（本專案已有此機制）
  4. 混合策略：高頻/低延遲操作考慮 CLI/API 直連，多工具協作場景繼續用 MCP
  5. Agent Graphs 關注：階層式 Agent 管理與本專案的 Team 模式理念一致

---

八、結論

MCP 在 2026 年處於「成長痛」階段——已成為事實標準（97M 月下載、10K Server、四大 AI 巨頭支持），但也面臨 Token 效率、安全漏洞、企業就緒度等真實挑戰。Perplexity 的棄用不代表 MCP 失敗，而是凸顯了「一個協議不可能適合所有場景」的工程現實。正確的策略是根據使用場景選擇：多工具互操作用 MCP，Token 敏感的高頻操作用 CLI/API。
