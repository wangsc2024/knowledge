1. 效能基準測試

官方基準

  • 記憶體快取讀取：~20M ops/sec（MacBook Pro Chrome Canary）
  • 行動裝置：~5M ops/sec（Android Chrome）
  • 壓縮大小：~9KB gzipped

重要：以上為純記憶體測試，生產環境含磁碟 I/O 時延遲為 50-100 倍。

實測延遲

| 場景 | 延遲 |

|------|------|

| 本機 | 1-5ms |

| 區域網路 P2P | 10-50ms |

| 廣域網路 Relay | 100-500ms |

| 含磁碟同步 | +50-200ms |

並發能力

  • 單 Relay：50+ 並發 WebSocket
  • 每連線記憶體：~1-2MB
  • 訊息吞吐量：~100-500 msg/sec

決策矩陣（1-5 分）

|------|--------|-----|-----------|-----------|----------|

| 大小 | 5 | 4 | 1 | 5 | 2 |

場景推薦

  • 即時協作編輯 -> Yjs（首選）
  • 去中心化 P2P + 加密 -> Gun.js
  • CRDT 研究 -> Automerge
  • IoT/嵌入式 -> Gun.js（最小體積）
  • 圖形查詢 -> Gun.js（原生圖形模型）
  • 交易一致性 -> 以上都不適用
OrbitDB 已棄用（2019），不建議使用。

4. 容錯策略

HAM 衝突解決

  • 每個鍵值對攜帶 state（時間戳）
  • 較新的更新勝出
  • state 相同時以值的字典序決定

長時間離線處理

  • 重連可能觸發同步爆炸（大量更新）
  • 解決：增量批次同步（每批 50 筆）+ 恢復窗口（30 秒）
  • 監控 radata 成長（日增量 >50MB 告警）

資料損壞恢復

  • 啟動時驗證 RAD 檔案完整性（JSON.parse 檢查）
  • 每日增量備份（7 天保留）
  • 損壞時快速失敗（fail fast）而非帶傷運行

Split-Brain 防護

  • 多 Relay 環境下比對各 Peer 回應一致性
  • 等待 3 秒收集回應再判斷

5. 關鍵邊緣案例

| 案例 | 嚴重度 | 解決方案 |

|------|--------|----------|

| 未啟用 radisk | P0 | radisk: true |

| ACK !ack.err 邏輯錯誤 | P0 | ack && ack.ok === true |

| .on() 未 .off() | P0 | 重連前清理舊監聽 |

| ACK 超時 5 秒 | P1 | 調為 12-15 秒 |

| 解密失敗靜默 | P1 | 加入錯誤日誌 |

| 無重播攻擊防護 | P1 | 應用層時間戳驗證（60 秒窗口） |

6. 監控最佳實踐

健康檢查端點

  • 記憶體使用（heap used/total）
  • 連線 Peer 數量與 URL
  • radata 啟用狀態
  • 告警：heap >75%、peers = 0

同步延遲偵測

  • 每 5 分鐘發送 sync check marker
  • 量測寫入到 ACK 的往返時間

radata 成長趨勢

  • 每小時記錄大小
  • 7 天歷史保留
  • 日增量 >50MB 觸發告警

7. 安全進階

  • 金鑰輪換：建議 30 天週期，備份舊金鑰
  • 公鑰釘選：防止 MITM 替換 epub
  • 私鑰加密儲存：使用 AES-256-GCM + 環境變數密碼

8. 成熟度模型

| 等級 | 條件 | 適合 |

|------|------|------|

| L1 原型 | 單 relay、無持久化 | 開發測試 |

| L2 基礎 | radisk + PM2 + 健康檢查 | 1-10 用戶 |

| L3 穩定 | HTTPS + 金鑰管理 + 備份 | 10-50 用戶 |

| L4 企業 | 多 relay + 負載均衡 + 完整監控 | 50+ 用戶 |
