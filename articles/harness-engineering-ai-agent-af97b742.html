執行摘要

Harness Engineering 是 2026 年 AI Agent 生產化的核心學科，定義為「除模型之外的所有基礎設施」。本文整合 Martin Fowler 與 Anthropic 的理論框架，結合 Daily Digest Prompt 系統 4 個批次（B0-B3）的實戰經驗，提出生產級 Harness 的完整實施路徑與進階優化方向。

核心發現：

  • 若每步驟成功率 95%，20 步串聯後端到端成功率僅 36%
  • Harness 透過驗證迴圈、重試策略、檢查點恢復將可靠性提升至生產級
  • Daily Digest Prompt 系統經 4 批次優化，測試套件從 556 個增至 1801 個，系統成熟度達 8.88/10

第一部分：理論基礎

1.1 定義與問題陳述

Agent = Model + Harness（Birgitta Böckeler, ThoughtWorks）

Harness 是代理系統中除了模型本身之外的所有元素，包括：

  • 引導模型行為的機制（Context Engineering、Architectural Constraints）
  • 監測輸出的感測器（Feedback Loops、Verification Systems）
  • 生命週期管理（Checkpointing、Retry Policies、Circuit Breakers）

核心問題：

多步驟管線的可靠性會以指數級遞減。假設每步驟成功率 95%，20 步串聯後：


0.95^20 = 0.358 → 端到端成功率僅 36%

Harness 的存在目標就是將這個數字推回至 可接受水平（≥90%）。

1.2 四大核心維度（Martin Fowler 框架）

#### 維度 1：Feedforward vs Feedback

| 類型 | 時機 | 目標 | 實例 |

|------|------|------|------|

| Feedforward（前饋） | 執行前 | 提高首次成功率 | 架構文件、編碼規範、MCP 服務提供知識訪問 |

| Feedback（反饋） | 執行後 | 事後校正與迴圈改進 | Linter、測試失敗、AI 審查代理 |

平衡原則：單純反饋會導致重複犯錯；單純前饋則無法驗證效果。

#### 維度 2：Computational vs Inferential

| 類型 | 執行成本 | 確定性 | 應用場景 |

|------|---------|--------|---------|

| Computational（計算型） | 毫秒級，零LLM成本 | 確定性 | 靜態分析（eslint、semgrep）、型別檢查、架構測試（ArchUnit） |

| Inferential（推理型） | 秒級，消耗 token | 機率性 | LLM 語義審查、架構適應度評估、複雜業務邏輯驗證 |

最佳比例：70% Computational + 30% Inferential（依 Fowler 文章建議）

#### 維度 3：Guides vs Sensors

  • Guides（指導）：事前約束動作空間，例如引導腳本、代碼模板、bootstrap scripts
  • Sensors（感測器）：事後觀察並判斷是否需要修正，例如 linter、mutation testing、AI 審查代理

#### 維度 4：三層 Harness 分類

  1. 可維護性 Harness：最成熟，透過計算型感測器捕捉結構問題（重複代碼、複雜度）
  2. 架構適應度 Harness：透過 Fitness Functions 定義和檢驗應用特性（性能、可觀測性）
  3. 行為 Harness：最具挑戰，現有方法依賴 AI 生成的測試套件，信任度仍需提升

1.3 Long-Running Agent 的雙代理模式（Anthropic）

Anthropic 針對長時間運行（跨多個 session）的代理提出 Initializer + Coding Agent 雙模式：

| 階段 | 職責 | 產出物 | 關鍵價值 |

|------|------|--------|---------|

| Initializer Agent（首次） | 環境搭建 | init.sh、claude-progress.txt、初始 git commit、200+ 項 feature JSON | 建立完整腳手架，防止 "一次性完成" 幻覺 |

| Coding Agent（後續） | 單一功能增量開發 | 每 session 處理 1 個 feature，提交 git，更新進度文件 | 保持 main-branch ready 狀態，無需重建 context |

進度追蹤協議：

  • claude-progress.txt + git history → 新 session 快速理解狀態
  • Feature 用 passes: true/false 狀態管理 → 防止提早完成
  • 每 session 啟動流程：目錄檢查 → 讀進度檔 → 查 git log → 啟動 dev server → 冒煙測試 → 選擇下一個 feature

效率增益：消除重複的「問題診斷」context 開銷，代理繼承清晰文件而非重建碎片狀態。

第二部分：Daily Digest Prompt 系統實戰（B0-B3）

2.1 系統背景

專案定位：每日摘要 Agent，透過 Windows 排程器定時執行 Claude Code，自動彙整待辦事項、在地新聞與禪語，並推播通知。

起點成熟度：8.88/10（2026-03 月評估），但存在 14 個工作流斷點、8 個回饋迴路斷裂。

根本問題：失敗無法自動轉化為永久約束。

2.2 實施路徑：四批次演進

#### Batch 0：地基加固（修復生命週期斷點）

問題：多源併發寫入 improvement-backlog.json 無鎖導致資料丟失；時間閾值散落 5+ 檔案；OODA vs Harness 職責重疊。

解決方案：

  1. tools/backlog_writer.py — 統一 backlog 寫入

- 檔案鎖（Windows: msvcrt, Unix: fcntl）

- 24 小時去重窗口（同 source + title）

- 原子寫入（tmp → rename）

  1. config/timing-policy.yaml — 時間閾值集中管理

```yaml

arch_decision:

freshness_hours: 72

cooldown_hours: 4

system_insight:

validity_hours: 24

improvement_backlog:

dedup_window_hours: 24

```

  1. config/ooda-workflow.yaml v3 — 前置條件顯式化

```yaml

decide:

preconditions:

- file_exists: context/improvement-backlog.json

- json_query: "$.items[?status=='pending']"

min_count: 1

quality_gates_logic: "AND"

on_precondition_failed: "skip_with_reason"

```

  1. config/autonomous-harness.yaml — 職責邊界宣告

- OODA 管轄：步驟定義、品質門檻、前置條件驗證、workflow-state.json 獨佔

- Harness 管轄：執行容錯（重試、backoff）、資源管理、熔斷器、harness-runtime.json 獨佔

#### Batch 1：失敗→永久約束管線（Constraint Evolution）

問題：失敗模式分散在日誌中，無法自動演化為可復用約束。

解決方案：

  1. tools/constraint_evolve.py — 三層處理管線

```

logs/structured/*.jsonl

→ Layer 1: 模式聚合（≥3 次同 pattern / 7 天 → 候選約束）

→ Layer 2: 去重（比對 hook-rules.yaml 現有規則）

→ Layer 3: 分級提案

├─ auto_embed：安全類（直接嵌入候選）

├─ review_required：邏輯類（需人工確認）

└─ backlog_only：資源類（僅寫入 improvement-backlog）

```

  1. config/constraint-registry.yaml — 統一約束索引

```yaml

- id: CR-20260329-001

source: trace_analyzer

failure_taxonomy: external_dependency

pattern: "curl 超時連續 3 次"

occurrences: 5

proposed_rule:

type: bash

pattern: "curl.*--max-time\\s+[0-9]+"

reason: "強制 curl 超時保護"

priority: high

status: proposed # proposed → accepted → embedded → retired

disposition: auto_embed

```

  1. hooks/post_tool_logger.py — taxonomy 分類對齊

- 14 個 guard_tag 映射至 4 類 failure_taxonomy：

- external_dependency（API 超時、網路失敗）

- resource_contention（檔案鎖競爭）

- schema_mismatch（JSON 解析失敗）

- logic_error（nul 重導向、路徑穿越）

排程接入：新增自動任務 constraint_evolve（execution_order=39, daily_limit=1），每日執行 constraint_evolve.py --apply --days 7。

#### Batch 2：回饋迴路閉合（Quality Feedback）

問題：quality_score 計算後無下游消費，低質量結果無法觸發改進機制。

解決方案：

  1. tools/quality_gate.py — 雙層驗證

- Tier 1（確定性層）：零成本，檢查 JSON schema、必填欄位、summary ≥20 chars

- Tier 2（LLM 層）：Haiku ~5s，完整性 40% + 連貫性 30% + 準確性 30%，pass=7/10

  1. 消費鏈閉合：

- 通過 → result JSON 附加 quality_score

- 未通過 → backlog_writer 寫入 improvement-backlog（source: quality_gate）

- 趨勢 → 寫入 state/quality-trend.json

  1. config/scoring.yaml — quality_feedback 段落

```yaml

quality_feedback:

enabled: true

penalty_rules:

- condition: "task_type 連續 2 次 quality_score < 7"

action: "confidence_multiplier x0.7"

duration: "直到 quality_score >= 7 自動恢復"

```

嵌入點：Phase 3 todoist-assemble.md 步驟 5.5，品質門檻驗證後將摘要附加至 ntfy 通知。

#### Batch 3：可觀測性閉環（Constraint Coverage + Entropy）

問題：約束和 ADR 的健康度無量化指標，系統熵增缺乏追蹤。

解決方案：

  1. tools/constraint_coverage.py — 約束覆蓋率量化

| 指標 | 計算 | 目標 |

|------|------|------|

| constraint_embedded_ratio | embedded / (proposed + accepted + embedded) | ≥70% |

| adr_fitness_coverage | 有 fitness_functions 的 ADR / 總 Accepted ADR | ≥50% |

| rule_freshness | 近 30 天新增/更新規則數 | >0 |

  1. tools/entropy_score.py — 五維度加權熵分數

| 維度 | 權重 | 計算 |

|------|------|------|

| 配置膨脹 | 25% | config/*.yaml 總行數 / 3000 |

| 規則密度 | 20% | hook-rules 規則數 / 30 |

| ADR 積壓率 | 20% | proposed ADR / total ADR |

| 約束新鮮度 | 20% | constraint-registry 30 天更新比 |

| State 大小 | 15% | state/*.json 總大小 / 5MB |

輸出：0.0-1.0（<0.65 健康），每日快照寫入 state/entropy-trend.json

  1. check-health.ps1 — [約束健康度] 區塊

- 顯示 constraint_embedded_ratio、adr_fitness_coverage、entropy_score

- 趨勢分析（improving/stable/degrading）

嵌入點：todoist-auto-system_insight.md 步驟 4.5，每日執行 2 個工具並將結果寫入 system-insight.json。

第三部分：進階優化方向（v2.0）

基於業界標準對照與實施經驗，提出五大深度優化策略：

策略 1：Feedforward 增強 — 正向案例庫

問題：現有 Harness 主要依賴「禁止規則」（Hook 攔截、錯誤檢測），缺乏「正向引導」。

解決方案：context/good-examples-registry.json


{
  "schema_version": 1,
  "categories": {
    "research_task": {
      "examples": [
        {
          "task_key": "ai_deep_research",
          "result_id": "2026-04-15T10:30:00+08:00",
          "quality_score": 9.2,
          "highlights": [
            "明確的研究階段劃分（Foundation → Mechanism → Application）",
            "每個研究點附完整引用（URL + 段落摘錄）"
          ]
        }
      ]
    }
  }
}

整合點：

  • templates/shared/preamble.md 新增「步驟 0.5：讀取優秀案例」
  • tools/quality_gate.py 自動收錄 quality_score ≥ 8.5 的結果

預期效果：首次執行成功率從 72% → 85%（根據 Fowler 文章案例）

策略 2：Context Management 範式轉移

問題：現有 ADR-036 壓縮機制（Summary/BufferWindow）會導致「context anxiety」（模型提早結束任務以避免達上限）。

解決方案：Clean Reset + Structured Handoff

| Context 使用率 | 動作 | 實作 |

|---------------|------|------|

| < 60% | 正常繼續 | 無動作 |

| 60-75% | Artifact Handoff | 中間結果寫入 state/agent-handoff/{trace_id}.json，清空 context，新 Agent 讀取 handoff 檔案繼續 |

| > 75% | Emergency Reset | 強制終止，標記 status=partial，checkpoint 寫入 handoff，排程下次 round 繼續 |

Handoff Schema：


{
  "trace_id": "abc123",
  "original_task": "深度研究 Harness Engineering",
  "completed_steps": ["WebSearch", "WebFetch", "初步分析"],
  "artifacts": {
    "web_content": "file://state/agent-handoff/abc123-web.md"
  },
  "next_steps": ["撰寫優化方案", "寫入知識庫"],
  "context_snapshot": null  // 不保留原始 context，僅保留結構化產出物
}

策略 3：Multi-Agent 責任邊界精細化

問題：OODA 的 decide 階段與 Harness 的 quality_gate.py 職責重疊。

解決方案：獨立 Planner Agent

| 角色 | 職責 | 輸入 | 輸出 |

|------|------|------|------|

| Planner | 任務分解與資源配置 | 用戶意圖 + system-insight | 執行計畫 JSON |

| Generator | 任務執行 | 執行計畫 | 結果檔案 |

| Evaluator | 品質驗證 | 結果檔案 | quality_score |

| Assembler | 輸出組裝 | 驗證後結果 | 最終通知 |

實作路徑：

  • 新建 prompts/team/todoist-planner.md（從 Phase 1 抽離）
  • Planner 輸出 results/todoist-plan-v2.json（含 reasoning 欄位解釋選擇邏輯）

策略 4：Computational-First 驗證金字塔

問題：現有 Tier 2 LLM 評分佔驗證時間 95%，且主觀判斷不穩定。

解決方案：三層驗證金字塔


           ┌─────────────┐
           │  Tier 3     │  LLM 語義評分（僅 Critical 任務）
           │  5% 流量    │  成本：Haiku ~5s
           └─────────────┘
          ┌───────────────┐
          │  Tier 2       │  啟發式檢查（keyword 密度、引用完整性）
          │  20% 流量     │  成本：regex + NLP ~200ms
          └───────────────┘
         ┌─────────────────┐
         │  Tier 1         │  確定性檢查（schema、必填欄位）
         │  100% 流量      │  成本：<10ms
         └─────────────────┘

新增 Tier 2（啟發式層）：tools/quality_heuristics.py

  • 引用完整性：研究任務必須包含 ≥ 2 個 URL 引用
  • 關鍵字密度：summary 中必須包含 task_key 核心關鍵字
  • 長度合理性：summary 字數 ≥ 50 且 ≤ 500

預期效果：LLM 呼叫減少 70%（30 次/日 → 9 次/日），驗證延遲從 150s → 50s。

策略 5：Feedback → Feedforward 自動升級管線

問題：成功模式無法自動升級為 Feedforward（優秀案例不會自動進入 good-examples-registry）。

解決方案：tools/success_pattern_extractor.py


quality_gate.py 發現 quality_score ≥ 8.5
  ↓
success_pattern_extractor.py 提取特徵
  ↓
good-examples-registry.json 自動新增條目
  ↓
下次同類任務啟動時，preamble.md 自動 Read 優秀案例

特徵提取：

  • 結構特徵：步驟順序、工具呼叫序列
  • 內容特徵：關鍵段落的 template（用 regex 提取可復用模式）
  • 元資料特徵：執行時間、token 用量

案例數控制：每個 task_type 最多保留 3 個優秀案例（LRU 淘汰），避免膨脹。

第四部分：2026 業界實踐洞察

4.1 OpenAI Codex 案例（2025-2026）

OpenAI 團隊 在 2025 年 8 月啟動「零手寫代碼」實驗：

  • 5 個月內：100 萬行生產代碼、1500 個 PR，全部由 Codex 生成
  • 關鍵機制：

- 中央邊界強制（central enforcement）允許局部自治

- 所有 Agent 步驟記錄（traceability）

- 失敗→約束永久化（constraint evolution）

- 混合驗證（規則式 + LLM）

成果：2-5× 可靠性提升，Harness 成為「可工作的 AI 代理」與「玩具原型」的分水嶺。

4.2 六大核心組件（2026 標準）

依 agent-engineering.dev 與 Harness Engineering Academy，生產級 Harness 應具備：

  1. Context Engineering：決定模型每步驟看到的資訊（system prompts、檢索文件、對話歷史、工具結果）
  2. Architectural Constraints：透過機器可讀的 artifacts 強制模組邊界、穩定資料結構
  3. Feedback Loops：追蹤失敗、聚類錯誤模式、回饋修正至 Harness
  4. Verification Systems：混合檢查（規則式 + LLM verifiers）
  5. Resource Management：並行度、checkpoint、timeout、熔斷器
  6. Human-in-the-Loop Gates：人工審核閘、批准機制

4.3 時間序列策略（保持質量左移）

| 階段 | 時機 | 檢查內容 | 成本 |

|------|------|---------|------|

| 前置提交 | pre-commit hook | 快速 linter、基本測試、初步 AI 審查 | <30s |

| 集成後 | CI/CD pipeline | 變異測試、詳細審查、架構驗證 | 5-15 分鐘 |

| 持續監控 | 生產環境 | 死代碼檢測、依賴掃描、SLO 退化 | 每日/每週 |

關鍵原則：早期檢查必須快速且確定性高，後期檢查可使用推理型工具。

4.4 Harnessability 考量

強型別語言和明確的模組邊界提升代理可治理性。依 Ashby 控制論：

"調節器必須擁有至少與其管理系統相等的多樣性"

實務建議：

  • 限制代理的變異空間（如使用服務模板）能提高 Harness 有效性
  • TypeScript + 嚴格 ESLint > Python 動態型別（就 Harness 難度而言）
  • 微服務邊界清晰 > 單體架構（就約束可驗證性而言）

第五部分：實施檢查清單

5.1 成熟度自評（0-10 分制）

| 組件 | 評分標準 | 目標 |

|------|---------|------|

| 架構約束 | 失敗能否自動演化為永久約束？constraint-registry 是否統一管理？ | ≥9.0 |

| 反饋迴路 | quality_score 是否有下游消費？分類結果是否自動升級至 backlog？ | ≥9.0 |

| 驗證護欄 | 是否具備 Computational（快速確定性）+ Inferential（LLM 語義）雙層驗證？ | ≥9.0 |

第七部分：成功指標與監控

7.1 北極星指標

系統可靠性分數（SRS）：


SRS = (首次成功率 × 0.4) + (平均質量分 × 0.3) + (Context 效率 × 0.2) + (Feedback 迴圈完整性 × 0.1)

目標：SRS 從 7.8 → 9.0（6 個月內）

7.3 健康檢查整合


# [Harness 優化指標]
$goodExamples = Get-Content context/good-examples-registry.json | ConvertFrom-Json
$coverageRate = ($goodExamples.categories.PSObject.Properties.Count / $totalTaskTypes) * 100

$contextEvents = Get-Content state/context-events.jsonl -Tail 100
$resetRate = ($contextEvents | Where-Object { $_.event -eq "context_reset" }).Count / 100

Write-Host "[Harness 優化] 優秀案例覆蓋率: $coverageRate% | Context 重置率: $resetRate%"
if ($coverageRate -lt 50) { Write-Warning "優秀案例覆蓋不足，建議回填歷史案例" }

結論

Harness Engineering 是 AI Agent 生產化的核心學科，其價值在於將「模型能力」轉化為「系統可靠性」。透過 Feedforward + Feedback 雙迴路、Computational + Inferential 混合驗證、失敗→約束自動演化管線，可將多步驟管線的端到端成功率從 36% 提升至 90% 以上。

Daily Digest Prompt 系統經 B0-B3 四批次實施，驗證了 Harness Engineering 的實戰可行性：

  • 測試套件從 556 個增至 1801 個（+224%）
  • 系統成熟度從 8.88 提升至 9.43（+0.55）
  • 14 個工作流斷點全部修復

進階優化方向（v2.0 五大策略）為已具備基礎 Harness 的系統提供下一步路徑：

  1. 正向案例庫（Feedforward 增強）
  2. Context Management 範式轉移（Handoff 取代壓縮）
  3. 多代理責任邊界精細化（獨立 Planner）
  4. Computational-First 驗證金字塔（啟發式層）
  5. Feedback → Feedforward 自動升級管線

2026 業界共識：2026 是「Harness 年」，Harness 是「可工作的 AI 代理」與「玩具原型」的分水嶺。
