對話腳本

主持人 A（解說）：今天我們來聊一個實戰技術主題：PowerShell 7 的並行執行。如果你曾經為了加速批次處理而苦惱，這集內容會很有幫助。

主持人 B（提問）：我聽說 PowerShell 7 有好幾種並行方法，它們有什麼不同嗎？

主持人 A（解說）：沒錯，主要有四種：Start-Job 是進程級隔離，每個任務啟動獨立的 pwsh 實例，最安全但開銷最大。Start-ThreadJob 是執行緒級共享進程，速度快三倍。而 ForEach-Object -Parallel 使用 Runspace 池，是最快的原生方案。

主持人 B（提問）：聽起來 ForEach-Parallel 最快，那為什麼還需要其他方法？

主持人 A（解說）：因為場景不同。比如你要啟動多個完全獨立的外部程式，像是呼叫 Claude CLI 跑不同的 prompt，這時候 Start-Job 的完全隔離反而是優勢——一個崩潰不會影響其他。ForEach-Parallel 適合管線式的批量處理，像是同時處理一百個檔案。

主持人 B（提問）：那效能差異有多大？

主持人 A（解說）：以 20 個任務為例，Start-Job 需要 7.58 秒，ThreadJob 只要 2.37 秒，而 ForEach-Parallel 僅需 0.79 秒。更誇張的是 I/O 密集型任務：原本循序執行要 485 秒，用 ForEach-Parallel 加到 100 個並行度後，只需 8.9 秒，加速 54 倍！

主持人 B（提問）：哇，這麼快！那我是不是應該把並行度開到最大？

主持人 A（解說）：要看工作類型。CPU 密集型任務的 ThrottleLimit 應該等於邏輯核心數，超過反而增加上下文切換開銷。但 I/O 密集型任務大部分時間在等待，可以開到 25 到 100 甚至更高。網路操作要小心 API 速率限制，通常 10 到 50 之間測試比較保險。

主持人 B（提問）：還有什麼需要注意的陷阱嗎？

主持人 A（解說）：並行執行有三個常見陷阱。第一是變數隔離：並行區塊無法直接存取父作用域變數，必須用 using 修飾符傳遞。第二是執行緒安全：如果要共享狀態，得用 ConcurrentBag 這類執行緒安全集合，不能直接操作陣列。第三是記憶體管理：Start-Job 執行完務必 Remove-Job，否則會記憶體洩漏。

主持人 B（提問）：那 Windows Task Scheduler 怎麼配合使用？

主持人 A（解說）：關鍵是用 pwsh.exe 而非 powershell.exe，因為後者是舊版 5.1。你可以設定複雜排程，比如每小時整點執行、失敗自動重試、或是基於前一任務成功才觸發下一階段。還要記得設定超時保護和錯誤日誌，並用最小權限原則執行任務。

主持人 A（解說）：總結一下：選對方法比一味追求速度重要。小資料集、強依賴順序、或除錯階段都不適合並行。但只要場景對了，PowerShell 7 的並行能力可以讓自動化工作提速數十倍，特別是搭配 Task Scheduler 後，能實現企業級的多階段管線架構。
