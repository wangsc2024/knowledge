Claude Code Agent Teams 深度實踐指南

概述

Claude Code Agent Teams（又稱 Swarm 模式）是 2026 年 2 月推出的實驗性功能，讓多個 Claude Code 實例作為協調團隊同時工作。與傳統的 Subagent 模式不同，Agent Teams 支援對等通訊（peer-to-peer messaging）、共享任務清單和自主工作認領，實現真正的多代理協作。

本文聚焦於 Agent Teams 的深度實踐模式，包括團隊拓撲選擇、協調策略、任務分解模式、成本優化與反模式規避。

---

一、核心架構與七大原語

1.1 架構組成

Agent Teams 由四個核心元件構成：

| 元件 | 角色 |

|------|------|

| Team Lead | 主 Claude Code session，建立團隊、分配任務、綜合結果 |

| Teammates | 獨立 Claude Code 實例，各自擁有完整 context window |

| Task List | 共享工作清單，支援依賴追蹤與自動解鎖 |

| Mailbox | 代理間通訊系統，支援直接訊息與廣播 |

1.2 七個工具原語

團隊透過七個工具運作：

  1. TeamCreate — 初始化團隊目錄與配置檔
  2. TaskCreate — 以 JSON 檔案定義工作單元
  3. TaskUpdate — 認領任務、回報狀態
  4. TaskList — 查看共享工作佇列
  5. Task（帶 team_name 參數） — 產生新隊友
  6. SendMessage — 對等直接通訊
  7. TeamDelete — 團隊清理

1.3 持久化儲存


~/.claude/teams/{team-name}/config.json    # 團隊配置
~/.claude/teams/{team-name}/messages/      # 收件匣
~/.claude/tasks/{team-name}/               # 任務清單

環境變數自動注入：CLAUDE_CODE_TEAM_NAME、CLAUDE_CODE_AGENT_ID、CLAUDE_CODE_AGENT_TYPE。

二、Agent Teams vs Subagents 決策矩陣

| 面向 | Subagents | Agent Teams |

|------|-----------|-------------|

| Context | 獨立 window，結果回傳呼叫者 | 獨立 window，完全獨立 |

| 通訊 | 只能回報給主 Agent | 隊友間直接通訊 |

| 協調 | 主 Agent 管理所有工作 | 共享任務清單 + 自主協調 |

| 最適場景 | 聚焦型任務，只需結果 | 需要討論與協作的複雜工作 |

| Token 成本 | 較低（~200K/3 agents） | 較高（~800K/3 agents + lead） |

| 生命週期 | 短暫，完成即結束 | 持久，可跨任務存活 |

決策規則：當工作者需要共享發現、互相挑戰、自主協調時用 Agent Teams；只需快速聚焦執行並回報結果時用 Subagents。

三、四大團隊拓撲模式

3.1 Parallel Specialists（並行專家）

多個專門化代理同時審查同一主題的不同面向，結果透過收件匣非同步聚合。

適用場景：程式碼審查、多維度分析、研究探索

範例：


Create an agent team to review PR #142. Spawn three reviewers:
- One focused on security implications
- One checking performance impact  
- One validating test coverage
Have them each review and report findings.

每個審查者套用不同的「濾鏡」，Lead 在所有人完成後綜合跨領域發現。

3.2 Sequential Pipeline（順序管線）

任務透過 blockedBy 陣列建立顯式依賴，前一任務完成時自動解鎖下一任務。

適用場景：需要階段閘門的工作流（設計 → 實作 → 測試）

執行波次：

  • Wave 1：獨立任務並行執行
  • Wave 2：被 Wave 1 阻擋的任務執行
  • Wave 3：最終依賴任務完成

3.3 Self-Organizing Swarm（自組織蜂群）

無依賴的獨立任務，工作者透過 TaskList 競爭認領，自然負載均衡。

適用場景：QA 蜂群、大量獨立測試、批次處理

實例：5 個 Agent 同時對部落格執行不同類型的品質檢查（HTTP 狀態碼、內容驗證、連結檢查、SEO、無障礙），執行時間約 3 分鐘。

3.4 Competing Hypotheses（競爭假設）

多個隊友各自調查不同理論，透過「科學辯論」模式互相反駁，最終收斂到最可能的根因。

適用場景：除錯、根因分析

關鍵機制：辯論結構避免「錨定效應」。循序調查容易被第一個合理解釋錨定，而並行調查者主動嘗試推翻彼此理論，存活的理論更可能是真正的根因。

四、控制機制與進階配置

4.1 顯示模式

| 模式 | 說明 | 適用環境 |

|------|------|----------|

| In-process（預設） | 所有隊友在主終端機；Shift+Down 切換 | 任何終端 |

| Split Panes | 各隊友獨立 pane，同時可見 | tmux 或 iTerm2 |

4.2 計畫審批機制

可要求隊友在實作前先提交計畫（唯讀 plan mode），Lead 審核後才允許執行：


Spawn an architect teammate to refactor the authentication module.
Require plan approval before they make any changes.

Lead 可自主做出審批決策，透過 prompt 影響判斷標準（例如：「只批准包含測試覆蓋的計畫」）。

4.3 Delegate 模式

按 Shift+Tab 啟動委派模式，限制 Lead 只能使用協調工具（產生隊友、發送訊息、管理任務），防止 Lead 自己動手實作而忽略委派。

4.4 Hooks 品質閘門

| Hook | 觸發時機 | 用途 |

|------|---------|------|

| TeammateIdle | 隊友即將閒置 | 返回 exit code 2 可發送回饋，讓隊友繼續工作 |

| TaskCompleted | 任務即將標記完成 | 返回 exit code 2 可阻止完成並發送回饋 |

4.5 模型配置策略

成本優化公式：Lead 使用 Opus（高推理能力），Teammates 使用 Sonnet（成本效益）。


Create a team with 4 teammates to refactor these modules in parallel.
Use Sonnet for each teammate.

五、最佳實踐

5.1 團隊規模

  • 建議：3-5 個隊友（平衡並行效益與協調開銷）
  • 任務分配：每位隊友 5-6 個任務（保持生產力，避免過度切換）
  • 15 個獨立任務 → 3 個隊友是好的起點

5.2 任務粒度

| 粒度 | 問題 |

|------|------|

| 太小 | 協調開銷超過收益 |

| 太大 | 隊友長時間無檢查點，浪費風險增加 |

| 剛好 | 自包含單元，產出明確交付物（函式、測試檔、審查報告） |

5.3 充分的上下文

隊友會載入 CLAUDE.md、MCP 伺服器、Skills，但不繼承 Lead 的對話歷史。在 spawn prompt 中包含足夠的任務特定細節：


Spawn a security reviewer teammate with the prompt: "Review the 
authentication module at src/auth/ for security vulnerabilities. 
Focus on token handling, session management, and input validation. 
The app uses JWT tokens stored in httpOnly cookies."

5.4 避免檔案衝突

兩個隊友編輯同一檔案會導致覆寫。每位隊友應擁有不同的檔案集。進階做法是使用 Git Worktrees 讓每個 Agent 工作在獨立的工作樹中。

5.5 Plan-First 工作流

最佳模式分兩階段：

  1. 規劃階段（低成本）：Plan mode 探索程式碼庫、建立任務分解（~10K tokens）
  2. 團隊階段（高成本）：將審核過的計畫交給團隊並行執行

這個檢查點防止高成本團隊朝錯誤方向前進。

5.6 Compound Engineering 模式

80% 規劃與審查，20% 執行的哲學：

  • /workflows:plan 生成詳細實作規格
  • /workflows:review 多代理程式碼審查
  • /workflows:compound 記錄學習供後續 Agent 使用

六、訊息傳遞協定

6.1 訊息類型

| 類型 | 用途 |

|------|------|

| message | 對單一隊友的直接訊息 |

| broadcast | 對所有隊友廣播（成本隨隊友數線性增長，慎用） |

| shutdown_request / shutdown_response | 優雅關閉 |

| plan_approval_request / plan_approval_response | 計畫審批 |

| idle_notification | 隊友閒置通知 |

6.2 通訊注意事項

  • 優先使用 message（一對一），避免 broadcast
  • 閒置通知是正常的 — 不需要對每次閒置做出反應
  • Lead 不需要輪詢，訊息會自動送達

---

七、反模式與常見陷阱

| 反模式 | 後果 | 正確做法 |

|--------|------|----------|

| 過度廣播 | N 個隊友 = N 倍訊息成本 | 優先用直接訊息 |

| Lead 自己動手實作 | 隊友閒置浪費 | 使用 Delegate 模式或提醒等待 |

| 模糊的 spawn prompt | 隊友產出品質差 | 提供具體的檔案路徑、技術細節 |

| 同一檔案多人編輯 | 互相覆寫 | 明確劃分檔案擁有權 |

| 忘記清理團隊 | 殘留 tmux session / 檔案 | 依序 shutdown → cleanup |

| 序列化可並行的工作 | 浪費時間 | 無依賴的任務同時啟動 |

| 一次性任務用 Team | 協調開銷大於收益 | 用 Subagent 替代 |

---

八、已知限制

  1. In-process 隊友不可恢復：/resume 和 /rewind 不會還原隊友
  2. 任務狀態可能滯後：隊友有時忘記標記完成
  3. 關閉可能緩慢：隊友需完成當前工具呼叫
  4. 每 session 一個團隊：不支援巢狀團隊
  5. Lead 固定：無法轉移領導權
  6. 權限統一繼承：所有隊友初始使用 Lead 的權限設定
  7. Split Panes 限制：不支援 VS Code 內建終端或 Windows Terminal

九、成本分析與使用時機

| 配置 | Token 消耗 |

|------|------------|

| 單一 session | ~200K |

| 3 Agent 團隊 | ~800K（每位隊友是獨立的完整 context window） |

| 5 Agent 團隊 | ~1.5M（估算） |

何時值得使用 Agent Teams：

  • 工作需要跨代理真正的協調（辯論、共享發現、多層特性）
  • 並行探索能顯著加速（天級串行壓縮為小時級並行）

何時不值得：

  • 序列性強的任務
  • 單一檔案修改
  • 簡單聚焦的研究查詢

十、實戰工作流範例

10.1 QA 蜂群（5 Agent）


├── qa-pages: 測試 16 個 URL 的 HTTP 狀態碼
├── qa-posts: 驗證 83 篇文章的 metadata 和圖片
├── qa-links: 檢查 146 個內部 URL 的斷連
├── qa-seo: 驗證 RSS、robots.txt、SEO metadata
└── qa-a11y: 無障礙與 HTML 結構審計

執行時間：~3 分鐘

10.2 跨層特性開發（3 Agent）


├── frontend-dev: React 元件開發
├── backend-dev: API 端點實作
└── test-writer: 端到端測試撰寫

10.3 大型重構（Plan-First）


Phase 1 (Plan): 單一 Agent 探索程式碼庫、建立任務分解
Phase 2 (Execute): 4 個 Agent 並行重構不同模組
Phase 3 (Review): 3 個 Agent 分別審查安全、效能、架構
