---

一、主題概述

Structured Generation（結構化生成）是 2026 年 LLM 工程化最關鍵的技術之一，解決了「如何讓語言模型可靠地產出符合 schema 的結構化資料」這個核心問題。從早期的 prompt engineering 到 2026 年的 constrained decoding，結構化輸出經歷了四個演進階段，最終實現了 100% schema 合規的生產級解決方案。

本研究涵蓋三大面向：

  1. 底層機制：Constrained Decoding 的 token masking、FSM 編譯、grammar-guided generation
  2. 工具生態：Outlines、Instructor、Pydantic AI、XGrammar、llguidance 五大框架
  3. 工程實踐：各 LLM 供應商（OpenAI、Anthropic、Google）的實作對比與生產最佳實踐

---

二、演進四階段

Phase 1：Prompt Engineering（2022-2023）

  • 方法：在 prompt 中要求模型「回傳 JSON 格式」
  • 成功率：80-95%
  • 問題：無格式保證、靜默失敗、邊界條件易出錯

Phase 2：Function Calling / Tool Use（2023-2024）

  • 方法：透過 API 的 function/tool 定義提供 JSON Schema
  • 成功率：95-99%
  • 代表：OpenAI Function Calling、Anthropic Tool Use
  • 問題：Schema 僅作為提示（hint），非硬約束；可能在有效型別內產生無效值

Phase 3：Native Structured Output（2024-2025）

  • 方法：在 token 生成層面強制 schema 合規
  • 成功率：100%（受限解碼保證）
  • 代表：OpenAI response_format、Gemini response_schema
  • 技術：Finite State Machine (FSM) + Token Masking

Phase 4：高效能受限解碼引擎（2025-2026）

  • 方法：開源引擎實現近零額外開銷的受限解碼
  • 代表：XGrammar、llguidance、Outlines v2
  • 突破：O(1) 有效 token 查詢、預編譯 grammar、批次解碼支援

三、核心技術：Constrained Decoding（受限解碼）

3.1 基本原理

Constrained Decoding 在模型的 logit 輸出與 sampling 步驟之間插入一個 logit processor，追蹤目標 grammar（JSON Schema、regex、CFG）中的當前位置，將無效 token 的 logit 設為負無窮大（遮罩），確保只有合法 token 被採樣。

3.2 數學公式


P_constrained(x_t) = P(x_t) / sum(P(x))  if x_t in V_valid
                   = 0                     otherwise

關鍵性質：有效 token 間的相對機率比保持不變。例如逗號和右大括號的 3:1 機率比在遮罩後仍為 3:1，不改變模型的「偏好」。

3.3 Grammar 編譯流程

  1. JSON Schema -> Context-Free Grammar (CFG)：將 schema 的 type/properties/required 等轉換為 production rules
  2. CFG -> Finite State Machine (FSM)：編譯為狀態轉移圖，狀態轉移預計算
  3. FSM -> Token Mask Index：每個狀態對應一組有效 token ID 的 bitmask

3.4 效能特性

| 實作方式 | 每步查詢複雜度 | 額外延遲 |

|---------|--------------|----------|

| 暴力 grammar 檢查 | O(n) | 2-5x |

| FSM 預編譯 | O(1) | 近零（<5%） |

| Outlines index | O(1) | 低（schema 編譯一次性開銷） |

四、工具生態系統

4.1 Outlines（Python，開源）

定位：最成熟的開源結構化生成框架

核心功能：

  • 將 JSON Schema 編譯為 index 結構，實現 O(1) 有效 token 查詢
  • 支援 regex、JSON Schema、CFG 三種約束格式
  • 與 vLLM、Transformers、llama.cpp 整合

程式碼範例：


import outlines

model = outlines.models.transformers("mistralai/Mistral-7B")

# Regex 約束
generator = outlines.generate.regex(model, r"([Yy]es|[Nn]o)")
result = generator("Is the sky blue? ")

# JSON Schema 約束
from pydantic import BaseModel
class Sentiment(BaseModel):
    label: str
    score: float

generator = outlines.generate.json(model, Sentiment)
result = generator("Analyze: 'Great product!'")

4.2 Instructor（Python/TypeScript，開源）

定位：高階 SDK 補丁，將 Pydantic 模型直接作為 response_model

支援供應商：OpenAI、Anthropic、Google、Mistral、Ollama

核心機制：

  1. 序列化 Pydantic model 為 JSON Schema
  2. 傳遞給 LLM API（structured output 或 tool use）
  3. 反序列化回應並驗證
  4. 驗證失敗時自動 re-prompting（含錯誤訊息）

4.3 Pydantic AI（Python，Pydantic 官方）

三種輸出模式：

| 模式 | 機制 | Schema 合規 |

|------|------|----------|

| Tool Output | 透過 tool call 回傳 | 99%+ |

| Native Output | 模型原生 JSON Schema 約束 | 100% |

| Prompted Output | Schema 注入 prompt + 嘗試解析 | 80-95% |

2026 基準：Pydantic 驗證速度比 JSON Schema 快 3.5 倍（高吞吐場景）

4.4 XGrammar 與 llguidance（引擎級）

XGrammar（2025-2026）：

  • 高效能受限解碼引擎
  • 實現近零額外開銷
  • 批次解碼支援
  • 整合於 vLLM、TensorRT-LLM

llguidance（Microsoft）：

  • 結合 grammar 約束與語義引導
  • 支援更複雜的約束類型（跨欄位依賴、條件 schema）

4.5 vLLM 整合

vLLM 原生支援 structured output，透過 guided_decoding_backend 參數選擇引擎：


from vllm import LLM, SamplingParams

llm = LLM(model="mistralai/Mistral-7B", guided_decoding_backend="outlines")
params = SamplingParams(
    guided_json={"type": "object", "properties": {"name": {"type": "string"}}}
)
outputs = llm.generate(["Extract name:"], params)

六、生產最佳實踐

6.1 Validation Sandwich 模式（必備）

永遠不要直接信任 LLM 輸出，即使使用了 structured output：


[Schema 約束（結構正確性）]
    |
[Pydantic/Zod 驗證（型別安全性）]
    |
[業務邏輯驗證（語義正確性）]

6.2 Schema 設計原則

  1. 使用 Enum 取代自由字串：分類欄位必須用 enum（避免幻覺值）
  2. 設定 min/max 約束：字串長度、數字範圍、陣列大小
  3. 避免過深巢狀：OpenAI 限制 5 層，過深影響生成品質
  4. 明確 required 欄位：減少 null 值出現
  5. 添加 description：幫助模型理解欄位語義

6.3 錯誤處理策略


def structured_extract(prompt, schema, max_retries=3):
    for attempt in range(max_retries):
        try:
            result = llm.parse(prompt, response_format=schema)
            schema.model_validate(result)  # 二次驗證
            validate_business_rules(result)  # 業務規則
            return result
        except ValidationError as e:
            prompt = f"{prompt}\n\n前次回應有誤：{e}\n請修正。"
    raise StructuredOutputError("Max retries exceeded")

6.4 反模式（Anti-Patterns）

| 反模式 | 問題 | 正確做法 |

|--------|------|----------|

| 用 regex 解析 JSON | 脆弱、無法處理巢狀 | 使用 structured output API |

| 信任 LLM 輸出不驗證 | 邊界條件失敗 | Validation Sandwich |

| Schema 過度複雜 | 生成品質下降 | 拆分為多次呼叫 |

| 忽略 enum 約束 | 幻覺分類值 | categorical -> enum |

| 用 string 替代具體型別 | 型別不安全 | 使用 int/float/bool/enum |

七、約束格式選擇指南

| 約束格式 | 適用場景 | 複雜度 | 保證程度 |

|---------|---------|--------|----------|

| Regular Expression | 簡單格式（日期、email、ID） | 低 | 100% |

| JSON Schema | 結構化資料萃取 | 中 | 100% |

| Context-Free Grammar | 程式碼生成、SQL | 高 | 100% |

| Choice List | 分類、選擇 | 極低 | 100% |
