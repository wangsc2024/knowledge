執行摘要

Harness Engineering 是 2026 年 AI Agent 工程的核心紀律，定義為「模型之外的所有東西」——包括上下文傳遞、工具介面、規劃產物、驗證迴圈、記憶系統與沙盒環境。研究顯示，遵循 Harness Engineering 實踐的團隊可獲得 2-5× 可靠性提升，而 OpenAI Codex 實驗更證明了完全由 AI 生成的 100 萬行程式碼系統的可行性。

本報告針對 Daily Digest Prompt 專案提出 P0-P2 三階段優化方案，涵蓋測試治理、約束演化、品質門檻與熵分數量化，預期可提升系統可靠性 40-60%、降低 token 使用 20-30%、減少人工干預 50%。

一、Harness Engineering 核心理念

1.1 定義與範疇

Martin Fowler 定義：Harness 是圍繞 AI 模型的同心圓結構，分為三層：

  • 模型核心（Model Core）：LLM 本身
  • Agent Builder Harness：工具提供者構建的框架（如 Claude SDK、LangChain）
  • 用戶外層 Harness：應用開發者設計的約束、驗證與反饋系統

Anthropic 洞察：「每個 Harness 組件都編碼了一個假設，關於模型無法獨立完成的事項。」核心原則是找到最簡單的解決方案，只在必要時增加複雜性。

1.2 兩大控制機制

1. Guides（前饋控制）：在行為發生前預防錯誤

  • 計算性控制（Computational）：確定性、快速（linters、type checkers、結構化測試）
  • 推論性控制（Inferential）：語義分析（LLM-as-judge）、慢速且機率性

2. Sensors（回饋控制）：行為發生後觀察與自我修正

  • 日誌、指標、追蹤系統
  • UI 自動化與視覺驗證
  • 評估器代理（Evaluator Agent）

1.3 三大規範類別

| 類別 | 說明 | 工具範例 |

|------|------|---------|

| Maintainability | 程式碼品質與內部結構 | ESLint, Black, pre-commit hooks |

| Architecture Fitness | 效能與設計特性 | Fitness functions, ArchUnit |

| Behaviour | 功能正確性（最不成熟領域） | Playwright, LLM-as-judge, contract testing |

二、Anthropic 三層架構模式

2.1 長時間執行應用的專業化代理

Planner（規劃者）：

  • 將簡短提示（1-4 句）擴展為完整產品規範
  • 刻意保持高層次，避免詳細技術規範級聯失敗
  • 主動整合 AI 功能到規範中

Generator（生成者）：

  • 按衝刺（Sprint）循環運作
  • 使用 React、Vite、FastAPI、SQLite 堆疊
  • 交接給 QA 前進行自我評估
  • 包含 git 版本控制

Evaluator（評估者）：

  • 使用 Playwright MCP 實時測試應用
  • 在衝刺前與生成器協商「衝刺合約」（Sprint Contract）
  • 明確定義測試標準與成功條件

2.2 關鍵發現與解決方案

上下文焦慮問題：

  • 現象：Claude Sonnet 4.5 在接近認知上限時提前結束工作
  • 解決方案：完全上下文重置（清空並創建新會話）比資訊壓縮更有效
  • 數據：Opus 4.6 可連貫運作 2 小時無需重置

自我評估的侷限：

  • 現象：模型傾向於「自信地稱讚工作，即使品質明顯平庸」
  • 解決方案：分離評估者（Separate Evaluator），使用少樣本示例校準
  • 效果：RetroForge 案例從 20 分鐘 $9（低品質）提升至 6 小時 $200（可用品質）

2.3 約束演化管線設計

將主觀判斷轉化為可量化標準（以前端設計為例）：

| 標準 | 量化指標 |

|------|---------|

| 設計品質 | 顏色、排版、佈局協調性評分（1-5） |

| 原創性 | 避免模板預設和 AI 典型特徵（布林） |

| 工藝 | 排版層級、間距、色彩和諧度（1-5） |

| 功能性 | 可用性和任務完成度（0-100%） |

迭代反饋循環：

  • 生成 5-15 次迭代
  • 每次基於評估者的具體批評進行改進
  • 整個循環需時 4 小時（RetroForge 案例）

四、2026 年 Harness Engineering 趨勢

4.1 工具與框架演進

核心工具棧（2025-2026 State-of-Practice）：

| 類別 | 主要工具 |

|------|---------|

| Agent Loop | LangGraph, Claude SDK, OpenAI Realtime API |

| Planning | TaskWeaver, Plan-and-Execute pattern |

| Context | Compaction API, Prompt Caching, LLMLingua |

| Tools | MCP Servers, Composio, instructor |

| Permissions | Claude Agent SDK, MCP Annotations, OWASP guidelines |

| Evaluation | SkillTester, AutoHarness, LLM Readiness Harness |

| Protocols | MCP, A2A, AG-UI |

| Observability | Event hooks, Hermes streaming, WebRTC transport |

4.2 Model Context Protocol（MCP）崛起

關鍵進展：

  • Streamable HTTP Transport（2025-11-25 規範）：支援遠端 MCP 服務 vs 本地進程
  • MCP Inspector：互動式除錯 UI
  • Google 2026 路線圖：橫向擴展、.well-known 發現、Tasks 原語（帶重試語義）

官方實作：GitHub、Slack、Postgres、Puppeteer 參考實作

第三方生態：Composio（250+ SaaS API 包裝器，託管 OAuth）

4.3 AI-Driven Pipeline Automation

Agentic Testing：

  • AI 系統自主決定測試內容
  • 生成相關測試案例
  • 執行並分析結果
  • 提出根因假設（minimal human direction）

Self-Healing Tests：

  • 使用 AI 定位器與模式識別
  • 當元素移動、標籤變更或佈局更新時自動適應

4.4 安全與權限治理

OWASP LLM06:2025 — Excessive Agency：

  • 最小權限原則（Principle of Least Privilege）
  • Deny-by-default 設計
  • 關鍵發現：批准疲勞（Approval Fatigue）— 使用者批准 93% 的提示，使 HITL 保障失效

Claude Agent SDK 五層評估：

  1. Hooks（前置檢查）
  2. Deny Rules（拒絕規則）
  3. Permission Mode（權限模式）
  4. Allow Rules（允許規則）
  5. canUseTool（工具權限）

Claude Auto Mode：

  • 兩階段分類器（快速閘門 + 推理層）
  • 僅對標記行為進行 chain-of-thought 推理

五、測試與品質門檻

5.1 熵分數量化（Entropy Metrics）

熵退化表現：

  • 變數命名不一致
  • 文件與實作不同步
  • 死碼累積
  • 測試覆蓋率下降

量化指標：

  • PercentageCoverage：總體覆蓋率百分比
  • NbLinesOfCodeCovered：已覆蓋行數
  • NbLinesOfCodeNotCovered：未覆蓋行數
  • BranchCoverage：分支覆蓋率

5.2 Test Harness 評估框架

Six-Layer Testing Guide（2026）：

| 層級 | 類型 | 說明 |

|------|------|------|

| Layer 0 | Data Metrics | 資料層指標（必須先驗證） |

| Layer 1+ | Agent Metrics | 代理層指標（依賴 Layer 0 穩定性） |

評估指標：

  • pass@k：k 次嘗試中至少一次通過的比率
  • LLM-as-judge：語義品質評分
  • Fault Injection：故障注入測試

Soft Failure Range：

  • 0.5-0.8 評分範圍：機率性代理的預期範圍
  • 33% 以上落在此範圍：需調查後才能上線

5.3 測試框架（2026）

DeepEval：

  • pytest-native assertions
  • LLM-as-judge 指標

Braintrust：

  • Trace-level 可觀測性
  • Golden dataset 管理

LangFuse：

  • 生產環境追蹤監控

六、對 Daily Digest Prompt 專案的優化建議

6.1 現況分析

已實施的 Harness 組件（來自專案記憶）：

  • ✅ 約束演化管線（Hook pipeline, Guardian system）
  • ✅ 品質門檻（benchmark.yaml, fitness_functions）
  • ✅ 覆蓋率量化（1620 測試，hooks 529 + skills 27 + tools 300）
  • ✅ 熵分數追蹤（mechanism-saturation, edit-frequency）
  • ✅ ADR 自動化（validate_adr.py）
  • ✅ Token 預算治理（budget_guard.py, 80% 警告 / 100% 暫停）
  • ✅ Agent Pool（coordinator.py, done_cert.py）

已達到的成熟度：B0-B3 四批次，1801 測試（根據記憶）

6.2 待優化領域（Gap Analysis）

對比業界最佳實踐（OpenAI、Anthropic、Microsoft），發現以下優化空間：

| 領域 | 現況 | 業界標準 | Gap |

|------|------|---------|-----|

| Context Compaction | 手動壓縮（context_compressor.py） | Claude Compaction API（84% 壓縮率） | 未使用官方 API |

| Evaluator Agent | 單一 Agent 自我評估 | 分離 Evaluator（Anthropic 模式） | 自我評估侷限 |

| MCP Integration | 部分使用（Slack、Pingtung News） | 完整 MCP 生態（GitHub、Postgres、Playwright） | 工具覆蓋不足 |

| Self-Healing Tests | 靜態測試（pytest） | AI-based locators（2026 趨勢） | 測試脆弱性 |

| Agentic Testing | 手動測試規劃 | 自主測試生成與執行 | 測試自動化不足 |

| Soft Failure Tracking | 僅追蹤 pass/fail | 0.5-0.8 範圍追蹤（33% 閾值） | 指標粒度不足 |

| Meta-Harness | N/A | 端到端 Harness 優化（10M tokens） | 缺乏整體優化 |

七、P0-P2 優化方案

P0：立即實施（1-2 週，高影響 / 低成本）

#### P0-A：引入 Separated Evaluator Pattern

目標：解決自我評估侷限，提升輸出品質

實作：

  1. 新建 prompts/evaluator/ 目錄
  2. 為關鍵任務（research、todoist、system-audit）建立獨立評估者 prompt
  3. 評估者使用 Playwright MCP 進行實際測試（非僅讀日誌）
  4. 在 run-todoist-agent-team.ps1 Phase 2.5 插入評估階段

預期效果：

  • 品質評分準確度 +30%
  • 重做率（rework rate）-40%
  • 參考：Anthropic RetroForge 案例（$9 低品質 → $200 可用品質）

檢查清單：

  • [ ] 建立 prompts/evaluator/research-evaluator.md（含少樣本範例校準）
  • [ ] 建立 prompts/evaluator/todoist-evaluator.md（含衝刺合約模板）
  • [ ] 修改 run-todoist-agent-team.ps1 新增 Phase 2.5（Evaluation）
  • [ ] 更新 config/timeouts.yaml 加入 evaluator timeout（120-180s）
  • [ ] 驗證：選取 3 個 research 任務，比對評估者評分 vs 人工評分，一致性 >80%

#### P0-B：啟用 Claude Compaction API

目標：將上下文壓縮率從手動 ~50% 提升至官方 API 的 84%

實作：

  1. 修改 tools/context_compressor.py，新增 use_official_api: bool = True 參數
  2. 呼叫 Claude API 的 /v1/messages 端點，使用 context_compaction 參數
  3. Fallback 機制：API 失敗時回退到現有手動壓縮

預期效果：

  • Token 使用 -20%
  • 上下文窗口利用率 +40%
  • Cache hit ratio +15%（更小的上下文更易命中快取）

檢查清單：

  • [ ] 閱讀 Claude API 文件（Compaction API 使用方式）
  • [ ] 修改 tools/context_compressor.py，新增 compress_with_api() 方法
  • [ ] 更新 config/dependencies.yaml 加入 claude_api.compaction 配置
  • [ ] 在 run-agent-team.ps1 Phase 2 插入壓縮檢查點
  • [ ] 驗證：比對 100 輪對話前後的 token 使用（目標：-20%）

#### P0-C：Soft Failure Tracking

目標：追蹤 0.5-0.8 評分範圍的「軟失敗」，提前發現品質退化

實作：

  1. 修改 state/quality-trend.json schema，新增 soft_failure_rate 欄位
  2. 在 tools/audit_verify.py 加入軟失敗偵測邏輯
  3. 33% 閾值觸發 ntfy 告警（priority=4）

預期效果：

  • 提前 2-3 輪發現品質退化
  • 減少生產失敗 50%

檢查清單：

  • [ ] 更新 state/quality-trend.json schema（加 soft_failure_rate, soft_failure_count）
  • [ ] 修改 tools/audit_verify.py，加入 is_soft_failure(score: float) -> bool 函式
  • [ ] 在 check-health.ps1 新增 [軟失敗率] 區塊
  • [ ] 配置 ntfy 告警規則（config/notification.yaml）
  • [ ] 驗證：手動插入 0.6 評分結果，確認告警觸發

結論

Harness Engineering 不是一次性實施，而是持續演進的工程紀律。隨著模型改進（如 Opus 4.6），有趣的 Harness 組合空間不會縮小，而是移動——工程師的工作是持續驗證舊假設、移除已失效組件、添加新的能力放大層。

本報告提出的 P0-P2 三階段方案預期可在 3.5 個月內將 Daily Digest Prompt 專案的可靠性提升 40-60%、降低 token 使用 20-30%、減少人工干預 50%，並建立與業界標準（OpenAI、Anthropic、Microsoft）接軌的工程實踐。

立即行動建議：從 P0-A（Separated Evaluator） 開始，2-3 天內可見初步效果，為後續優化建立信心與數據基礎。
