Harness Engineering 約束 SLI/SLO 工程實作

研究日期：2026-04-04
研究系列：harness-engineering（optimization 階段延續）
前置研究：「生產實測指標與自動化約束校準」（2026-04-03）
聚焦：將 Harness 從定性治理推進到可量化 SLI/SLO 體系

---

一、研究背景

前次研究建立了「量化約束校準四層模型」（漸進式品質門檻、Eval-to-Guardrail 生命週期、OTEL 原生可觀測性、多維 Fitness Function），但尚未定義具體的 SLI（Service Level Indicator）與 SLO（Service Level Objective）。本次研究補齊此缺口，從業界最新實踐中提煉可操作的指標體系。

---

二、核心發現：Agent SLI/SLO 六維度模型

2.1 從 SRE 到 Agent RE：指標遷移

傳統 SRE 的 SLI（延遲、錯誤率、吞吐量、可用性）無法直接套用到 AI Agent 系統。Conf42 SRE 2026 的案例研究指出，需新增 LLM 特有維度：

| SLI 維度 | 定義 | 建議 SLO | 量測方式 |

|----------|------|---------|----------|

| 任務成功率 | 端到端任務完成比例 | ≥95%（生產）/ ≥85%（預備）/ ≥70%（開發） | trace 級 pass/fail 判定 |

| 語義準確率 | 回應通過自動語義評估的比例 | ≥90% | embedding 相似度 + judge model |

| 幻覺率預算 | AI 生成內容中不實資訊的比例上限 | ≤5% | embedding-based similarity check |

| 工具選擇正確率 | Agent 選擇正確工具的比例 | ≥92% | trajectory 回放驗證 |

| 約束遵從率 | Agent 遵守 guardrail 的比例 | ≥99% | guardrail trigger rate 監控 |

| 端到端延遲 p95 | 95 百分位的請求處理時間 | ≤30s（對話）/ ≤120s（任務） | trace duration 統計 |

2.2 Guardrail 層級與延遲預算

每層 Guardrail 增加 10-50ms 延遲，業界建議總延遲罰款（penalty）控制：

  • 對話型 Agent：≤100ms 總 guardrail 延遲
  • 任務型 Agent（本專案類型）：≤500ms 可接受
  • 批次型 Agent：延遲不敏感，重點在正確性

2.3 錯誤預算（Error Budget）模型

Epsilla 與 OpenAI 共識：先建立約束再優化吞吐量（establish constraints before optimizing throughput）。


月錯誤預算 = 1 - SLO 目標
範例：任務成功率 SLO = 95% → 月錯誤預算 = 5%
若 30 天 1000 次執行 → 允許 50 次失敗
若提前用盡 → 凍結新功能部署，聚焦可靠性

三、Fitness Function 自動化校準模式

3.1 Trace-Based Evaluation（業界共識）

2026 年業界已收斂到以 trace 為基礎的評估方式，每個步驟記錄最小 payload：

| 欄位 | 用途 |

|------|------|

| traceId / stepId | 唯一識別 |

| tool_name | 工具選擇審計 |

| argument_hash | 參數驗證（不含 secrets）|

| duration_ms | 延遲追蹤 |

| token_usage | 成本監控 |

| result_summary | 離線回放 |

3.2 Eval-to-Guardrail 生命週期（三觸發機制）

  1. 開發期 Eval：20-50 個真實案例的 eval dataset（task success + tool correctness + trajectory quality + safety）
  2. 部署前 Gate：eval 結果作為 CI/CD 閘門，未通過 safety threshold 的版本阻擋部署
  3. 生產期 Guard：100% 流量監控，guardrail trigger rate >10% 觸發告警

3.3 六類 Guardrail 實作模式（Policy as Code）

現代實踐強調「政策即程式碼」——機器可讀的政策執行，獨立於 LLM 之外：

  1. 能力閘控（Capability Gating）：按環境（dev/staging/prod）白名單工具
  2. 人機協作閘門（HITL）：不可逆操作（發送訊息、刪除、付款）需人工批准
  3. 資料邊界（Secrets Boundary）：短期 token、輸出脫敏、禁止 raw secrets 進入模型上下文
  4. 注入防護（Injection Resilience）：檢索內容視為不受信任，指令與資料分離
  5. 速率與預算限制（Rate & Budget）：Token/工具呼叫/成本天花板，防止失控執行
  6. 沙箱隔離（Sandboxing）：高風險工具（shell、瀏覽器自動化）在受限上下文中執行

四、LangChain Deep Agents 案例：量化可靠性提升

LangChain 團隊在 Terminal Bench 2.0 上的成績：

  • 優化前：52.8%
  • 優化後：66.5%（+26% 相對提升）
  • 方法：純粹透過 trace failure 分析與 harness 調整，未更換模型

此案例證明：可靠性提升的主要槓桿在 harness 而非模型。OpenAI 案例進一步報告 2-5x 可靠性提升。

五、對 Daily-Digest-Prompt 系統的具體優化建議

5.1 P0：benchmark.yaml 新增 constraint_slo 區塊

建議在 config/benchmark.yaml 新增：


constraint_slo:
  task_success_rate:
    target: 0.90  # 本專案基於自動化場景，90% 為合理起點
    window: 7d
    source: "state/scheduler-state.json success_count / total_count"
  guardrail_compliance:
    target: 0.99
    window: 7d
    source: "logs/structured/*.jsonl blocked events / total events"
  avg_execution_time_p95:
    target: 180  # 秒，任務型 Agent
    window: 7d
    source: "state/scheduler-state.json duration 欄位"
  error_budget_monthly:
    formula: "1 - task_success_rate.target"
    alert_at: 0.80  # 預算用盡 80% 時告警

5.2 P1：check-health.ps1 整合 SLO 監控

在健康度腳本新增 [約束 SLO 監控] 區塊：

  • 從 scheduler-state.json 計算近 7 天任務成功率
  • 從 structured JSONL 日誌計算 guardrail 遵從率
  • 超出 SLO 時輸出告警 + ntfy 通知

5.3 P2：Eval Dataset 建立

從 results/ 歷史執行檔建立 eval baseline：

  • 收集最近 30 天成功/失敗案例
  • 標注失敗根因分類（timeout / API error / quality failure）
  • 作為後續自動回歸測試的基礎

5.4 P3：Error Budget Policy

實施錯誤預算政策：月錯誤預算用盡 80% 時凍結新自動任務部署，專注可靠性改善。

---

六、關鍵原則總結

  1. 先可偵測，再授權自治（Epsilla 2026）：先建立可觀測性，再擴大 Agent 自主範圍
  2. 先約束，再優化吞吐：約束是賦能而非限制
  3. Policy as Code：政策執行獨立於 LLM，機器可讀可測試
  4. Trace-Based Evaluation：以 trace 為單位評估，非單一輸出
  5. 漸進式門檻：dev → staging → prod 環境分級 SLO
