對話腳本

主持人 A（解說）：今天我們要聊一個很實用的技術主題：如何把你的個人知識庫，轉化成一個高效能的靜態網站。這篇研究深入探討了 Vite SSG 生態系統，特別聚焦在將 RAG 知識庫作為內容來源，透過自動化管線生成靜態網站。

主持人 B（提問）：聽起來很有趣！但我想問，為什麼要把知識庫變成靜態網站呢？直接用 API 查詢不是更方便嗎？

主持人 A（解說）：這是個好問題。研究指出了四大核心優勢：首先是 SEO 友善，搜尋引擎可以索引所有內容；第二是離線存取，靜態 HTML 完全不依賴後端服務；第三是高效能，透過 CDN 邊緣快取，TTFB 可以低於 50 毫秒；最後是低成本，像 Cloudflare Pages 或 Netlify 的免費方案就足夠承載。

主持人 B（提問）：原來有這麼多好處！文章提到了 VitePress、Astro Starlight 和 Vite-SSG 三種方案，這要怎麼選擇呢？

主持人 A（解說）：VitePress 是 Vue 生態系的首選，它的架構很特別：初次造訪時是純靜態 HTML，確保 SEO 和快速載入；後續導航會自動升級為 SPA，提供流暢的使用體驗。它內建 MiniSearch 全文搜尋，完全不需要額外配置，而且 Markdown 就是 Vue 單檔元件，可以直接嵌入 Vue 元件。

主持人 B（提問）：那 Astro Starlight 的特色是什麼？聽起來也很強大。

主持人 A（解說）：Astro 採用 Islands Architecture，也就是元件級的部分水合技術，只載入互動所需的 JavaScript。而且它框架無關，React、Vue、Svelte、Solid 的元件都可以混用。最大的優勢是產出體積小，在 500 頁的文件站測試中，VitePress 產出 2.7MB，而 Astro 只有 1.2MB，LCP 也更快，只需要 820 毫秒。

主持人 B（提問）：效能差異還真明顯！那 Vite-SSG 適合什麼場景呢？

主持人 A（解說）：Vite-SSG 是給需要完全自定義的專案使用的。它提供 getStaticPaths 函式來定義預渲染路徑，建構時透過 loader 取得資料並生成 manifest，瀏覽器端就不需要重新 fetch 了。如果你已經有 React 或 Vue 應用，想要靜態化但保持完全的佈局和路由控制權，這就是最佳選擇。

主持人 B（提問）：了解了框架選擇，那實際的內容管線是怎麼運作的呢？

主持人 A（解說）：整個管線分四個階段：第一步，從 RAG 知識庫透過 API 查詢筆記，比如呼叫 /api/notes；第二步，Python 同步腳本執行差異偵測、Markdown 轉換、Frontmatter 注入，並根據標籤分類到不同目錄；第三步，靜態網站建構器解析 Markdown、生成搜尋索引、預渲染 HTML；最後部署到 Cloudflare Pages 或 Netlify。

主持人 B（提問）：聽起來每次都要全量重建？這樣不會很慢嗎？

主持人 A（解說）：這正是增量同步的價值所在！研究提出了混合策略：先用時間戳初步篩選出 modified_at 晚於上次同步的筆記，再對這些筆記的 contentText 計算 SHA-256 雜湊值，與上次記錄比對，只有雜湊不同的才重新生成頁面。這樣可以把 300 多頁的全量建構，縮減到每次只處理 10 到 20 頁。

主持人 B（提問）：搜尋功能呢？靜態網站要怎麼實現全文搜尋？

主持人 A（解說）：研究提出了雙軌架構。靜態端在建構時生成客戶端搜尋索引，像 Pagefind、MiniSearch、Lunr.js 都可以，這樣完全不需要後端服務，而且支援離線搜尋。文章比較了四種引擎：Pagefind 約 150KB 查詢延遲小於 50 毫秒，FlexSearch 最輕量只有 100KB 延遲更低至 10 毫秒。動態端則可以選擇性保留 RAG API，提供進階的向量搜尋和 BM25 混合檢索，適合需要 AI 問答的場景。

主持人 B（提問）：部署到 Cloudflare Pages 的流程會很複雜嗎？

主持人 A（解說）：完全不會！可以透過 GitHub Actions 自動化整個流程。設定一個排程任務，比如每 6 小時觸發一次，執行 Python 同步腳本的增量模式，接著跑 npm run build 建構網站，最後用 cloudflare/pages-action 自動部署。Cloudflare Pages 會自動提供 HTTPS、全球 CDN 分發，甚至每個 Pull Request 都會自動生成預覽 URL。

主持人 B（提問）：聽起來很完整！還有什麼效能優化的技巧嗎？

主持人 A（解說）：研究列出了五大策略：Brotli 壓縮可以減少 15 到 25% 的傳輸量；圖片改用 WebP 或 AVIF 格式能再減少 30 到 50%；HTTP/3 在 Cloudflare 是預設啟用的，能降低連線延遲；如果做成 PWA，Service Worker 可以實現完全離線存取；最後是預載入關鍵資源，讓 LCP 控制在 1 秒以內。

主持人 A（解說）：總結一下技術選型建議：小型知識庫少於 500 頁，選 VitePress 最簡單快速；中型 500 到 5000 頁，Astro Starlight 效能最佳產出最小；如果已有 React 或 Vue 應用需要靜態化，用 Vite-SSG 保持完全彈性；任何方案都可以搭配 RAG API 做進階語義搜尋。

主持人 B（提問）：這真是一套完整的知識庫靜態化方案！對於想要分享個人筆記、建立技術文件站，或是把 Obsidian、Notion 筆記公開的人來說，這篇研究提供了非常實用的技術路徑。感謝今天的分享！
