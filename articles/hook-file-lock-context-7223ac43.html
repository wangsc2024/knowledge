Hook 跨平台檔案鎖重構：抽取 file_lock context manager 至 hook_utils.py

專案優化daily-digest-prompt程式碼品質DRY原則重構hooks

優化目標

消除 hooks 模組中重複的跨平台檔案鎖定邏輯（DRY 原則）。原先 post_tool_logger.py 的 _update_token_usage() 和 agent_guardian.py 的 CircuitBreaker._atomic_update() 各自實作了相同的 msvcrt/fcntl 鎖定模式（約 15 行重複程式碼），現統一抽取為 hook_utils.py 中的 file_lock context manager。

變更摘要

1. hook_utils.py — 新增 `file_lock` class

  • 實作為 context manager（__enter__ / __exit__）
  • 跨平台鎖定：Windows 優先用 msvcrt.locking(LK_NBLCK)，POSIX 用 fcntl.flock(LOCK_EX | LOCK_NB)
  • 兩者皆不可用時退化為無鎖模式（不中斷流程）
  • 鎖檔案（{filepath}.lock）在 __exit__ 時自動清理
  • atomic_write_json 的 docstring 更新：引導使用者搭配 file_lock 保護 read-modify-write

2. post_tool_logger.py — 簡化 `_update_token_usage()`

  • 移除 15 行手動鎖定/釋放邏輯
  • 改為 with file_lock(token_file): 一行搞定
  • 移除 atomic_write_json 的 try/except ImportError fallback（因 file_lock 已從同一模組匯入）
  • 程式碼行數從 74 行減至 40 行（-46%）

3. agent_guardian.py — 簡化 `CircuitBreaker._atomic_update()`

  • 優先路徑：_FILE_LOCK_AVAILABLE 為 True 時用 file_lock context manager（3 行）
  • Fallback 路徑：hook_utils 不可用時保留原始手動鎖定邏輯（向後相容）
  • Import 行更新：新增 file_lock 到 from hook_utils import ...

驗證結果

  • 語法檢查：3 個 Python 檔案全部通過
  • 測試套件：556 個測試全部通過（4.58 秒）
  • Git diff 統計：+103 行 / -74 行（淨增 29 行，含 47 行新 class 定義 + docstring）

設計決策

  • file_lock 使用小寫 class 名稱，因為它主要作為 context manager 使用（類似 contextlib.contextmanager），符合 Python 社群對工具型 context manager 的命名慣例
  • agent_guardian.py 保留 fallback 路徑，確保模組可獨立運作（不強制依賴 hook_utils）
  • 不吞掉例外（__exit__ 回傳 False），確保業務邏輯錯誤正確傳播
