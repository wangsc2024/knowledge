---

執行摘要

2026 年，提示快取（Prompt Caching）已從開發便利功能演化為生產 AI 系統的核心基礎設施。本研究聚焦五個核心問題：Anthropic 2026 年 TTL 縮短的影響、Gemini 與 Anthropic 快取機制差異、三層快取架構設計、快取斷點最佳實踐，以及對本專案（daily-digest-prompt）的直接應用。

核心發現：Anthropic 在 2026 年初將默認快取 TTL 從 60 分鐘縮短至 5 分鐘，導致高頻工作負載的有效 API 成本增加 30-60%，原本 80%+ 的節省效果降至 40-55%。要維持高節省率，必須主動選擇 1 小時 TTL 方案，並將快取設計納入 prompt 架構的第一優先級。跨三大提供商（Anthropic、Gemini、OpenAI）的對比分析顯示：明確快取控制與自動快取偵測各有場景優劣，但「靜態前置、動態後綴」是三方一致的核心原則。三層快取架構（語義快取 → 前綴快取 → 完整推論）是 2026 年生產環境的標準實踐，組合策略可達 80%+ 成本節省。

---

背景與動機

LLM 推論成本是生產 AI 應用最大的可控成本項目。以 Claude Sonnet 4.6 為例，未快取的輸入 Token 成本為 $3/MTok，而快取命中成本僅為 $0.30/MTok——90% 的差異。在每日執行數十次自動任務的系統中（如 daily-digest-prompt），此差異意味著數量級的月度成本差距。

2026 年初，Anthropic 悄然調整了快取 TTL 政策（從 60 分鐘 → 5 分鐘），引發了廣泛的生產成本衝擊，也迫使開發者重新審視快取架構設計。此背景下，深入理解三大提供商的快取機制差異、設計最佳實踐，以及多層快取策略，已成為任何 AI 系統架構師的必修課題。

Anthropic Prompt Caching 核心技術解析

機制原理

提示快取的本質是 KV Cache 重用：當 LLM 處理 prompt 時，每個 Token 會生成對應的 Key-Value 矩陣（注意力機制的內部表示）。若相同前綴在下次請求中出現，系統可直接跳過 prefill 計算，重用已儲存的 KV 矩陣，顯著降低計算成本與延遲。[來源：Redis LangCache Blog, 2026]

Anthropic 的快取控制體系

明確快取（Explicit Caching）：在 content blocks 上添加 cache_control 標記，最多支援 4 個獨立斷點。適合需要精確控制不同快取邊界的場景。[來源：Anthropic 官方文件, 2026] ✅

自動快取（Automatic Caching）：在請求頂層添加 cache_control，系統自動將斷點移到最後可快取的 block。適合多輪對話中持續增長的訊息歷史。[來源：Anthropic 官方文件, 2026] ✅

快取層級順序：tools → system → messages（後者依賴前者；修改工具定義會使系統 prompt 快取也失效）。[來源：Anthropic 官方文件, 2026] ✅

2026 年 TTL 政策變化與衝擊

2026 年初的 TTL 縮短（60 分鐘 → 5 分鐘）是本研究最具生產衝擊的發現：

| 情境 | TTL 變更前（60分鐘） | TTL 變更後（5分鐘） | 因應策略 |

|------|---------------------|---------------------|---------|

| 高頻自動化工作流（間隔 <5min） | 80%+ 節省 | 80%+ 節省（不受影響） | 維持現狀 |

| 標準批次處理（間隔 5-60min） | 60-80% 節省 | 40-55% 節省（嚴重衝擊） | 選用 1 小時 TTL |

| 低頻長週期任務（間隔 >1小時） | 首次呼叫後冷快取 | 首次呼叫後冷快取 | 重設計快取邊界 |

1 小時 TTL 方案：快取寫入成本從 1.25x 提升至 2x 基礎輸入成本，但對穩定前綴的長週期工作流仍划算。[來源：Anthropic 官方文件, DEV Community 2026] ✅

最小 Token 門檻

| 模型 | 最小可快取 Token 數 |

|------|-------------------|

| Sonnet 4.5 / 4.6 | 2,048 |

| Opus 4.5 / Haiku 4.5 | 4,096 |

| 舊版 Sonnet / Haiku 3 | 1,024 |

短於門檻的 prompt 靜默不快取，無錯誤提示。[來源：Anthropic 官方文件, 2026] ✅

三大提供商比較分析

| 特性 | Anthropic | Gemini | OpenAI |

|------|-----------|--------|--------|

| 快取控制方式 | 明確 cache_control 標記 ✅ | 1024 tokens 自動隱式 ✅ | 自動前綴偵測 ✅ |

| 顯式快取最小 tokens | 1,024-4,096 | 32,768 | 不適用（全自動） |

| 默認 TTL | 5 分鐘 | 1 小時 | 未公開（自動管理） |

| 讀取成本折扣 | 90% off（0.1x 基礎） | 75-90% off（依型號） | 50% off |

| 存儲費用 | 無 | 有（按 token 時間計） | 無 |

| 最佳場景 | 小到中型靜態前綴、工具定義 | 大文件（32K+）、長影片 | 對話歷史自動優化 |

[來源：Medium 三方對比, GeminiPricing.com, PromptHub, 2026] ✅

Gemini 的 32K 最小門檻決定其最佳用例是代碼倉庫分析、長影片字幕等超大文件場景；而 Anthropic 更適合典型 LLM 應用的系統 prompt + 工具定義快取。[來源：Medium, 2026] ✅

關鍵洞見與三角佐證

洞見 1：快取應被視為生產基礎設施，而非開發便利 ✅（4 來源佐證）

TTL 縮短事件揭示了一個核心教訓：若未將提示快取設計成生產基礎設施，任何提供商政策變化都會導致意外的成本衝擊。應與批次處理、重試限制、並行控制一樣，在系統設計時明確建模快取策略。[來源：Introl, SurePrompts, DEV Community, Anthropic 官方, 2026]

洞見 2：靜默失效是快取設計的最大風險 ✅（3 來源佐證）

快取失效時不會拋出錯誤，只是靜默回退到全量推論。最常見的根因：

  • 動態時間戳或 Request ID 嵌入靜態前綴
  • Python 以外語言（Swift、Go）的 dict/map 鍵順序不穩定，導致工具定義 JSON 序列化結果每次不同
  • 部署時 prompt 措辭細微變化使現有快取失效

監控要點：持續追蹤 cache_read_input_tokens 佔總輸入的比例（目標 >60%）。[來源：SurePrompts, Introl, Anthropic 官方, 2026]

洞見 3：三層快取架構是 2026 年生產標準 ✅（3 來源佐證）


請求 → [語義快取] 100% 節省，30-60% 命中率
      → [前綴快取] 50-90% 節省，穩定 prompt 可達 80%+ 命中率
      → [完整推論] 基準成本

31% 的 LLM 查詢與先前請求具語義相似性，是語義快取層存在的最強論據。[來源：Introl, Redis, 2026]

洞見 4：回溯視窗 20 block 限制在長對話中造成快取遺失 ⚠️（1 來源）

Anthropic 快取命中機制從斷點位置最多向前回溯 20 個 content block。若多輪對話後斷點距最後快取寫入超過 20 個 block，快取命中失效。設計長對話時應從起始就規劃多個快取斷點。

洞見 5：Thinking Blocks 可被快取，是長鏈推論的成本槓桿 ⚠️（1 來源）

Claude 4+ 系列的 thinking blocks（以前稱 chain-of-thought）可在 assistant 輪次中被快取，讀出時計為輸入 Token。在重複使用同一推論前綴的 multi-step agentic 工作流中，可顯著降低成本。

實務應用建議：daily-digest-prompt 的快取優化路徑

基於研究發現，本專案有以下具體優化機會：

1. 識別快取候選區段

  • 高優先級：各 todoist-auto-*.md prompt 的靜態系統指令段（通常佔 prompt 總長 50-70%），每次調用幾乎不變
  • 中優先級：工具定義（若使用 tools API）；preamble.md 引用的靜態規則
  • 低優先級：動態注入的 auto-tasks-today.json 內容、當日時間戳

2. 評估 TTL 選擇

本專案的 Todoist 自動任務調用間隔通常為 30 分鐘 - 1 小時，超過 5 分鐘 TTL 閾值。建議選用 1 小時 TTL 方案，快取寫入成本 2x 但對穩定前綴划算。

3. Prompt 架構重組

遵循「靜態置前」原則重組 prompt 模板：


[靜態：系統角色宣告 + 通用規則] ← cache_control 斷點
[靜態：工具定義（若有）] ← cache_control 斷點
[動態：當日任務狀態、計數器] ← 不快取
[動態：本次任務特定內容] ← 不快取

4. 監控指標建立

在 run-todoist-agent-team.ps1 Phase 2 結果收集中加入快取命中率追蹤：目標 cache_read_input_tokens / total_input_tokens > 0.60，低於此閾值觸發快取設計審查。

5. 注意 PowerShell 工具定義序列化

若未來整合工具定義 API，確認 ConvertTo-Json 輸出的鍵順序穩定（PowerShell 5.1+ 的 Hashtable 不保序，應改用 [ordered]@{} 或從固定 JSON 文件讀取）。

與現有知識的連結

  • 本專案已有 cache/status.json 機制追蹤 API 快取命中率，可延伸到追蹤 Claude Token 快取命中率
  • ADR-001「Skill-First 策略」的快取層架構（skills/api-cache/）可作為前綴快取邏輯的封裝層
  • config/cache-policy.yaml 的 TTL 配置設計哲學可類比應用到 Anthropic prompt cache TTL 選擇

---

未來展望

  1. Anthropic 快取 TTL 彈性化：預期 2026 Q3 提供更細粒度的 TTL 選項（如 15分鐘、30分鐘），降低從 5 分鐘跳到 1 小時的成本躍升
  2. KV Cache 感知路由（llm-d）：Red Hat 的 llm-d 項目已實驗性地在路由層感知 KV Cache 狀態，把請求路由到最可能命中快取的服務器節點
  3. 跨提供商快取標準化：OpenRouter 已支援跨提供商的 prompt caching 透通，未來可能出現統一的快取感知 LLM 代理層
