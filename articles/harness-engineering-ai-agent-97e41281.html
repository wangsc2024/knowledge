Harness Engineering 深度研究：AI Agent 生產可靠性的新典範（2026）

研究背景

2026年3月，軟體工程領域發生典範轉移：Agent 不是難點，Harness 才是。Harness Engineering 成為設計約束、工具、回饋迴圈、文件和驗證系統的新學科，使 AI Agent 在生產環境中可靠運行。

演化三階段

| 階段 | 時期 | 核心關注 | 限制 |

|------|------|---------|------|

| Prompt Engineering | 2022-2024 | 優化提示詞 | 無法應對生產複雜度 |

| Context Engineering | 2025 | 管理上下文 | 長期任務漂移 |

| Harness Engineering | 2026+ | 系統約束與回饋 | 生產可持續性 |

核心定義

Agent = Model + Harness

  • Model：基礎大型語言模型（如 GPT-4、Claude）
  • Harness：圍繞模型的所有約束、工具、回饋迴圈、文件和驗證系統

工程師的職責從「直接寫程式碼」轉變為「設計讓 AI 可靠寫程式碼的環境」。

架構組件

1. 雙重控制機制

#### Guides（前饋控制）

  • 定義：預先引導 Agent 行為
  • 實現：文件、規則、結構強制
  • 案例：Red Hat 的 Repository Impact Map 使用 LSP/MCP 掃描程式碼庫作為單一真相來源

#### Sensors（反饋控制）

  • 定義：觀察輸出並啟用自我修正
  • 實現：自訂 linter 訊息包含修正指令
  • 效果：LangChain 實驗中 Terminal Bench 分數從 52.8% 提升至 66.5%（無需更換模型）

2. 執行類型

| 類型 | 特性 | 工具範例 |

|------|------|----------|

| Computational | 確定性、毫秒級、可靠 | Tests、Linters、Type checkers |

| Inferential | 語義分析、慢、昂貴但準確 | Agent-reviewing-agent、LLM 驗證 |

3. 三個調控維度

#### Maintainability Harness

  • 複雜度指標
  • 測試覆蓋率缺口
  • 風格違規
  • 程式碼重複

#### Architecture Fitness Harness

  • Fitness Functions：定義並檢查架構特性的導引與感測器
  • 約束實施：架構測試與效能要求
  • 邊界驗證：模組依賴方向嚴格驗證

#### Behaviour Harness

  • 挑戰：最難實現的維度
  • 現況：依賴 AI 生成測試套件 + 人工驗證
  • 缺口：對 AI 生成測試過於信任，尚不足夠

約束演化管線

驅動因素

  1. Agent 漂移：無結構化護欄，長期任務累積熵
  2. 規模需求：OpenAI 2026 內部專案生成 100萬行+ Agent 寫的程式碼，零人工行
  3. 系統化修正：將一次性補丁轉為永久修復（linting 規則、約束）

六步驟實施順序

  1. 強制函數：採用「零人工程式碼」約束
  2. 宣告式意圖：高層級規格，非低層級指令
  3. 全面追蹤：每個 Agent 動作可追溯與失敗分析
  4. 編碼修正：將修正轉為可重用系統約束
  5. 混合驗證：交替使用規則與 LLM 驗證以平衡速度/準確度
  6. 定期重構：排程重構 Agent 防止長期漂移累積

品質門檻架構

四大基礎組件

  1. 架構約束

- 機械可讀 ADR（架構決策記錄）

- Schema 驗證器

- 模組邊界強制

- 集中監督 + 局部自治

  1. 回饋迴圈

- 失敗閉環追蹤

- 錯誤模式叢集化

- 修正回饋至 Harness

  1. 驗證混合方法

- 確定性工具（linters、type checkers）

- AI 驅動驗證（agent-reviewing-agent）

- 定期「垃圾回收」重構

  1. 生命週期腳手架

- 標準化 CI/CD

- 文件自動生成

- 可觀測性（從 repo 設定到部署）

熵監控模式

核心挑戰

在 Agent 驅動開發中，熵加速是因為 Agent 產生程式碼的速度快於人類審查。

解決方案

  • 定期稽核：排程 Agent 任務掃描不一致、死程式碼、文件漂移
  • 自動清理 Agent：專責重構、測試維護、文件更新
  • 漂移偵測器：捕捉 Agent 行為意外變化
  • 實時檢查：違反約束前即時標記輸出

時間分佈策略

| 階段 | 檢查類型 | 工具範例 |

|------|---------|----------|

| Pre-commit | 快速、確定性 | Linters、基礎審查 |

| Post-integration | 昂貴感測器 | Mutation testing、詳細審查 |

| Continuous | 漂移偵測 | 執行期監控 |

核心原則總結

約束解決空間原則

約束越多，輸出越可預測。

Cursor 團隊發現：

  • 未約束模型：浪費 token 探索死胡同路徑
  • 良好設計 Harness：開闢窄而明確的成功路徑

人類責任轉移

編碼 Agent 沒有社交問責、沒有對300行函數的審美厭惡、沒有『我們不這樣做』的直覺。

有效 Harness 將隱性開發者經驗外部化為明確、可測量的控制。

人類責任從「防止所有錯誤」轉為「將輸入導向最重要之處」。

對 Daily-Digest-Prompt 系統的優化建議

1. 約束演化管線強化

現狀：

  • ✅ 已有 constraint_evolve.py（proposed→accepted→embedded）
  • ✅ 已有 hook-rules.yaml 約束定義
  • ⚠️ 缺口：缺少自動從失敗日誌提取約束的管線

建議行動：

  • 實施每日自動任務 constraint_extraction
  • 掃描 llama-workflow-trace.jsonl 最近7天失敗記錄
  • 使用 LLM 識別重複失敗模式
  • 自動生成 improvement-backlog.json constraint_proposal 條目
  • 閾值：同類失敗 ≥3 次觸發提案

2. Fitness Functions 量化驗證

現狀：

  • ✅ 已有 benchmark.yaml fitness_functions 段落
  • ⚠️ 缺口：fitness 驗證未自動化，依賴人工審查

建議行動：

  • 擴展 validate_adr.py 加入 fitness 自動驗證
  • 加入 Pre-commit hook 階段驗證（快速、確定性）
  • 加入每日 system-audit 段落（連續趨勢）

3. 熵監控自動化

現狀：

  • ✅ 已有 entropy_score.py 計算熵分數
  • ✅ 已有 entropy-health-monitor SKILL
  • ⚠️ 缺口：熵趨勢無歷史追蹤，無自動觸發減熵處方

建議行動：

  • 建立 state/entropy-trend.json 記錄每日熵分數
  • 每日 entropy_health 任務檢查趨勢
  • 連續3天上升 → 標記 warning
  • 超過 critical → 觸發 self-heal 減熵處方

5. 漂移偵測與自動清理
