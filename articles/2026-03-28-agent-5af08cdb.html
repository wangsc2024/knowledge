認知科學⇔Agent 工程互滲模式與系統自治閉環成熟度分析

執行摘要（300 字）

近三天知識庫呈現一個前所未有的融合趨勢：認知科學與 Agent 工程不再是單向借鏡，而是雙向互滲。決策疲勞理論（Baumeister/Schwartz）直接驅動了 Agent 任務簡化設計；反過來，Agent 的 Draft-Verify 三階段防護模式（Pre-Mortem→執行→Post-Mortem）與認知心理學的雙重歷程理論（System 1/System 2）高度同構。MoE 稀疏專家路由機制已從 ML 基礎設施滲透至 Agent Skill 調度層，gstack/superpowers/Dify 三大框架（合計 270K+ Stars）不約而同收斂於「認知模式切換」設計哲學。系統自治方面，success-rate-guardian、workflow-forge、trace_analyzer 構成了完整的觀測→診斷→自癒閉環，成熟度已從「被動監控」躍升至「主動治理」。本報告提出六項可落地的系統優化方案，涵蓋認知負荷減載、MoE 啟發路由增強、Draft-Verify 品質閘門及 HITL 分級審批。

---

一、五大跨域融合模式

模式 1：認知科學→Agent 工程的雙向互滲

近三天新增了三篇認知科學筆記，形成完整的「決策品質提升鏈」：

  • 論證謬誤系統分類（形式/非形式 230+ 種）→ 可應用於 LLM 輸出自動驗證 [來源：KB 9010901a]
  • 結構化分析技術 SATs（ACH 競爭假設分析、紅隊思維）→ 直接對標 Agent 的 Draft-Verify 模式 [來源：KB 1519408b]
  • 決策疲勞與選擇超載（Baumeister 自我損耗模型）→ 驅動任務選擇演算法從最優化轉向滿足化 [來源：KB 5ce2b9fc]

雙向互滲的證據：Agent 執行品質保證機制（KB b85a950d）的三階段防護——Draft-Verify 雙 Agent 協作、Pre-Mortem 失敗預測、Blameless Post-Mortem——與 SATs 的否證優先、ACH 假設對抗幾乎完全同構。這不是巧合，而是兩個領域在「對抗認知偏誤」這一共同目標上的自然收斂 [來源：CIO, 2026]。

關鍵洞見：Agent 系統的品質問題，本質上是認知偏誤的工程化體現。確認偏誤→過度自信的自動化決策；錨定效應→對初始 prompt 的過度依賴。認知科學不只是類比靈感，而是 Agent 設計的理論基礎。

模式 2：Agent 框架的「認知模式切換」收斂

近三天研究了 4 個重量級 Agent 框架，它們不約而同收斂於同一設計原則：

| 框架 | Stars | 核心洞見 | 認知模式切換表現 |

|------|-------|---------|----------------|

| gstack | 23K | 不同開發階段需要不同認知模式 | 9 個角色指令（CEO/QA/工程經理） |

| superpowers | 118K | 先規格化再寫碼 | Spec-First + TDD 強制循環 |

| Dify | 129.8K | 視覺化 Workflow 編排 | 拖放式狀態機設計 |

| Crawl4AI | 62.7K | 記憶自適應調度 | LLM 多策略萃取引擎 |

收斂方向：四個框架都在解決同一個問題——讓 AI 在不同任務階段採用不同的推理策略，而非作為「萬能工具」一路到底。這與 Kahneman 的 System 1/System 2 理論高度對應：快速直覺（System 1）用於模式匹配和分類，深度推理（System 2）用於規劃和驗證 [來源：gstack KB 7a0145c5; superpowers KB 237acebd; Dify KB 4c76df93; Crawl4AI KB 312f5352]。

模式 3：MoE 路由機制→Skill 調度的遷移

Expert Routing 與 Sparse MoE 研究（KB 6a77382a）揭示了一個深刻的架構同構性：

| MoE 概念 | Agent Skill-First 對應 | 當前實作狀態 |

|---------|----------------------|------------|

| Router（門控網路） | routing.yaml（三層路由） | ✅ 已實作 |

| Top-K 專家選擇 | template_resolution 優先級 | ✅ 已實作 |

| 負載平衡損失函數 | fairness-rebalancer | ⚠️ 初步實作 |

| Capacity Factor | success-rate-guardian | ⚠️ 設計中 |

| 專家特化 | Skill 專責分工 | ✅ 已實作 |

| Token Drop | 任務 timeout/skip | ✅ 已實作 |

| Auxiliary Loss | 配置膨脹度量 | ✅ 已實作 |

外部研究佐證：2026 年 MoE 已成為前沿 AI 的無可爭議標準（DeepSeek-V4 的 1.5T 參數中僅喚醒 30B），「Top-K routing + load balancing」組合被證明是最穩健的路由策略 [來源：TokenRing, 2026; Botpress AI Agent Routing Guide]。

關鍵洞見：daily-digest-prompt 的 Skill 路由已無意中實作了 MoE 的大部分設計模式。缺失的是動態負載感知——當某個 Skill 連續失敗時，應自動降低其路由權重（類似 MoE 的 capacity factor）。

模式 4：系統自治閉環的三階成熟

近三天的 12 篇系統優化筆記揭示了系統正在經歷「三階成熟」：

| 成熟度階段 | 代表組件 | 近三天進展 |

|-----------|---------|----------|

| L1 被動監控 | check-health.ps1、query-logs.ps1 | 穩定運行 |

| L2 主動診斷 | trace_analyzer.py、system-insight | Hook 管線效能優化（KB fa99c033） |

| L3 自主治理 | success-rate-guardian、workflow-forge | 新增 Skill 設計（KB bd248327）+ 結果 Schema 驗證（KB 0b596c8a） |

MCP 生態標準化（KB 2b3f9c42）為 L3 提供了外部基礎設施：月 SDK 下載 9700 萬次、10000+ 伺服器、OpenAI/Google/Microsoft 均採用的事實標準。而 HITL Durable Orchestration（KB 89ed7a6f）則解決了 L3 的核心難題——中斷後帶狀態恢復，Agent 進入真實業務流程的基本能力。

外部佐證：2026 年 QA 領域報告指出自癒測試腳本可減少 95% 的手動維護工作量，多 Agent QA 系統已實現從天級到 2 小時的測試週期壓縮 [來源：ThinkSys QA Trends 2026; Tricentis QA Trends 2026]。

模式 5：「元工具鍛造」— 知識庫驅動的自我進化

最具前瞻性的模式出現在系統的自我進化能力中：

  • skill-forge（KB 12c93ace）生成了 fairness-rebalancer Skill
  • workflow-forge（KB 2fb449fb、0b596c8a）自動產出配置驗證清單和結果 Schema 驗證 workflow
  • 系統優化落實報告（KB cdc57ff5、40fc4695、a70197e0）形成 OODA 閉環的 Act 階段

這些元工具不只是自動化——它們是知識庫驅動的自舉式進化：系統觀察自身問題（Observe）→ 分析根因（Orient）→ 生成新 Skill/Workflow 解決方案（Decide）→ 自動落實（Act）。

二、對本系統的六項優化方案

基於以上五大模式，提出可直接落地的優化方案：

方案 1：認知負荷減載 — 任務選擇「滿足化」

問題：目前 scoring.yaml 的 6 因子計分追求最優化（找出「最佳」任務），但認知科學研究表明，過度最優化反而降低決策品質（Schwartz 選擇悖論）。

方案：在 scoring.yaml 新增 satisficing_threshold 參數，任務分數超過閾值即可執行，無需遍歷所有候選。當候選任務 ≥ 5 個時自動啟用，減少 Agent 的「選擇疲勞」。

預期效果：Phase 1 任務篩選耗時減少 30-40%，同時避免低優先級任務被永遠壓制。

方案 2：MoE 啟發的動態路由權重

問題：routing.yaml 的路由權重是靜態的，不會根據 Skill 的歷史成功率調整。

方案：在 routing.yaml 新增 dynamic_weight 段落，讀取 success-rate-guardian 的統計數據，對連續失敗 ≥ 3 次的 Skill 自動降低路由優先級（capacity factor），對連續成功的 Skill 提升權重（直到上限）。

預期效果：模擬 MoE 的 auxiliary load balancing loss，防止失敗 Skill 持續消耗執行配額。

方案 3：Draft-Verify 品質閘門

問題：自動任務的 KB 寫入缺乏自我批判步驟，偶爾產出重複或淺層內容。

方案：在研究類自動任務模板中，KB 寫入前加入輕量 Critique 步驟——用 3 個批判角色（懷疑論者/領域專家/實務工程師）各提一個問題，若任一問題無法回答則補充搜尋。

預期效果：品質評分 ≥ 4 的比例從目前約 65% 提升至 80%+。

方案 4：超日節律感知排程

問題：目前排程是固定時間觸發，未考慮人類用戶的精力週期。

方案：在 notification.yaml 中新增 ultradian_aware 配置，將重要通知（深度研究結果、系統警告）集中在精力高峰時段（08:00-10:00、14:00-16:00）推送，低優先通知在精力低谷時段靜默或合併。

預期效果：減少用戶被通知打斷的認知切換成本。

方案 5：論證謬誤偵測整合

問題：LLM 生成的研究報告可能包含邏輯謬誤（特別是滑坡謬誤和訴諸權威），目前無自動檢測機制。

方案：在 deep-research SKILL.md 的 Phase 7 Critique 中，新增第 10 項結構檢查——「論證謬誤掃描」，針對報告中的因果推論、類比推理、權威引用進行 SEPC 四步檢測。

預期效果：減少研究報告中的推理跳躍和過度推斷。

方案 6：HITL 分級審批框架

問題：目前所有自動任務的執行權限一致，高風險操作（配置修改、Skill 生成）與低風險操作（KB 寫入）無差別。

方案：基於政策感知 Agent（KB ca0c220f）和 HITL Durable Orchestration（KB 89ed7a6f）的洞見，在 config/hook-rules.yaml 中定義三級審批：L0 自動（KB 讀寫）、L1 記錄（配置修改+ntfy 通知）、L2 暫停（Skill 生成/系統架構變更需人工確認）。

預期效果：在保持自動化效率的同時，為高風險操作建立安全閘門。
