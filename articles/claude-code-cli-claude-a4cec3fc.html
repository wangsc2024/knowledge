Claude Code CLI 非互動批次模式（claude -p）：生產部署、Context 管理與 Token 效率最佳實踐（2026）

技術概述

Claude Code CLI 的 -p（--print）旗標啟用非互動單次執行模式，讓 Agent 無需 TUI 介面即可完成任務並退出。這是將 Claude Code 整合進 CI 管線、Windows Task Scheduler、cron job 或任何自動化腳本的核心機制。

相較於互動模式的 1-2GB RAM 需求，非互動模式僅需 512MB RAM，啟動即執行即退出，無介面開銷。在底層架構上，所有執行介面（互動 TUI、headless CLI、Agent SDK）共享同一個 queryLoop() 函式 [來源: arXiv 2604.14228v1, 2026]，差異僅在外層 QueryEngine 包裝器如何管理會話狀態。

非互動模式的核心價值在三個層面：成本效率（Prompt Caching 可降低 90% 快取區段的 token 成本）、可組合性（--output-format stream-json 讓多個 claude -p 進程形成管線）、以及可重現性（確定性輸出配合 --max-turns 防止無限迴圈）。

核心概念與原理

單一 queryLoop() 架構

根據 Claude Code 設計空間論文 [來源: arXiv 2604.14228v1, 2026]，系統採用「Unix utility」設計哲學，最小化框架強制性，讓模型推理自由運作。非互動模式透過 QueryEngine.submitMessage() 協調單次對話，最終仍委派給同一個 loop() 函式。這意味著 -p 模式享有完整的工具能力，包括 Bash 執行、檔案讀寫、MCP 呼叫等，而非降級版本。

五層 Context 壓縮管線

Context Window 是非互動模式的首要資源約束。系統在每次 model 呼叫前依序執行五個壓縮器 [來源: arXiv 2604.14228v1, 2026]：

  1. Budget reduction：對每個工具結果施加大小上限，防止大型輸出淹沒 context
  2. Snip：輕量裁剪舊歷史片段（最低成本）
  3. Microcompact：細粒度壓縮，具備 cache-awareness（避免破壞 prompt cache breakpoint）
  4. Context collapse：讀取時投影（read-time projection），重建較小的對話快照
  5. Auto-compact：由 model 產生語意摘要（最高成本，最後手段）

較便宜的層次先執行，僅在不夠時才觸發昂貴層次。此設計直接解釋為何長批次任務中 context 管理策略至關重要。

七層 Permission 模式

授權系統採用 deny-first 評估，提供七種模式 [來源: arXiv 2604.14228v1, 2026; code.claude.com/docs, 2026]：

| 模式 | 行為 | 適用場景 |

|------|------|----------|

| plan | 執行前需使用者批准 | 高風險任務探索 |

| default | 每個動作需批准 | 互動開發 |

| acceptEdits | 自動批准檔案編輯 | 程式碼修改任務 |

| auto | ML 分類器評估 | 半自動化 CI |

| dontAsk | 無提示，deny 規則仍執行 | 無人值守腳本 |

| bypassPermissions | 最小提示，保留安全關鍵檢查 | 隔離容器/VM 中 |

| bubble | 子代理升級內部模式 | 系統內部使用 |

重要：使用者對 permission prompt 的批准率約 93% [來源: arXiv 2604.14228v1, 2026]，顯示互動確認已出現「審批疲勞」，這是系統設計多層 permission 模式的根本動機。

Prompt Caching 機制

非互動批次執行的最大成本節省來源是 Prompt Caching [來源: platform.claude.com/docs, 2026]：

  • 最低 token 門檻：Sonnet 4.6 ≥ 2048、Opus 4.7/Haiku 4.5 ≥ 4096（低於門檻不觸發快取，也不報錯）
  • TTL 選項：預設 5 分鐘免費；1 小時 TTL 快取寫入 2× 費率，讀取僅 0.1× 費率（約 90% 成本節省）
  • 工作區隔離（2026-02-05 起）：快取以工作區為單位隔離，多工作區之間不共享
  • 快取失效觸發器：Tool 定義變更、Web Search 開關切換、Images 增減，均會使對應層及以下的快取失效

關鍵洞察：Claude Code 的系統 prompt 預設包含動態區段（工作目錄、環境資訊、git 狀態），這些區段會因機器/使用者不同而導致快取未命中。使用 --exclude-dynamic-system-prompt-sections 可將這些動態內容移至第一則 user message，讓系統 prompt 在多機/多使用者批次呼叫中保持一致，顯著提升快取命中率。

stream-json 輸出協議

--output-format stream-json 生成換行分隔的 JSON 事件流 [來源: backgroundclaude.com/blog/stream-json, 2026]：

  • stream_event：包含 event.delta.type == "text_delta" 時，event.delta.text 為 token 內容
  • system + api_retry subtype：包含 attempt、max_retries、retry_delay_ms、error_status、error（rate_limit/server_error/billing_error 等）欄位

此格式讓下游程序可即時解析進度、觸發取消、更新 CI 進度，並透過 session_id 欄位進行審計追蹤。

關鍵 CLI 旗標速查

| 旗標 | 說明 | 批次推薦 |

|------|------|----------|

| --bare | 跳過 hooks/skills/MCP/CLAUDE.md 探索，啟動快 10× | ✅ 必加 |

| --max-turns N | 限制代理轉數，超過以 error exit code 退出 | ✅ 必加 |

| --max-budget-usd N | 預算上限（API 用戶）| 依需求 |

| --exclude-dynamic-system-prompt-sections | 移除動態 prompt 區段，提升 cache 命中率 | ✅ 推薦 |

| --output-format stream-json | 即時事件流輸出 | 管線場景 |

| --fallback-model sonnet | 主模型過載時自動降級（print 模式）| 高可用需求 |

| --no-session-persistence | 不寫入磁碟（純批次）| 高頻批次 |

| --tools "Read,Grep" | 限制可用工具集 | 安全敏感 |

| --effort low/high/xhigh | 推理深度控制 | 成本調節 |

最佳實踐

1. 批次任務務必加 `--bare` 降低啟動延遲

--bare 旗標跳過 hooks、skills、plugins、MCP servers、auto memory 和 CLAUDE.md 的自動探索，啟動速度提升最多 10 倍 [來源: pasqualepillitteri.it, 2026; code.claude.com CLI 參考, 2026]。適用情境：單一功能腳本不需要 CLAUDE.md 配置的批次執行。


claude --bare -p "Explain what this file does" --output-format json

反例：忘記 --bare 導致每次批次呼叫都掃描整個 .claude/ 目錄，5000 次/日排程呼叫中累積數十分鐘浪費啟動時間。

2. 限制轉數與預算防止失控執行


claude -p --max-turns 5 --max-budget-usd 2.00 "Fix the failing test"

--max-turns 在達到限制時以 error exit code 退出（非靜默退出），讓 PowerShell 的 $LASTEXITCODE 能正確捕捉。--max-budget-usd 適用 API 使用者，在預算耗盡時自動停止。兩者組合是無人值守排程任務的基本防護。

3. 選擇適合任務的 Permission 模式

  • CI 唯讀分析：--permission-mode dontAsk --tools "Read,Grep,Glob" — 無提示且限制工具範圍
  • 自動修復：--permission-mode auto — ML 分類器攔截異常操作，但在 -p 模式下分類器反覆阻擋會直接 abort
  • 隔離容器：--dangerously-skip-permissions — 僅限容器/VM 環境使用

4. 提升 Prompt Cache 命中率


claude -p --exclude-dynamic-system-prompt-sections --model sonnet "Analyze this file"

確保系統 prompt 前段穩定、動態內容靠後、系統 prompt 長度超過模型最低快取門檻（Sonnet 4.6: 2048 tokens；Opus 4.7: 4096 tokens）。

5. stream-json 管線鏈接多 Claude 實例


claude -p --output-format stream-json "First task" \
  | claude -p --input-format stream-json --output-format stream-json "Refine results" \
  | claude -p --input-format stream-json "Final summary"

適用複雜多階段管線，讓中間階段結果無需暫存檔案即可串聯。

陷阱與反例

陷阱 1：Exit Code 1 是萬用錯誤碼，根因在上方輸出中

claude 進程以 exit code 1 退出是常見模式 [來源: github.com/anthropics/claude-code/issues, 2026]，常見原因：Anthropic 帳號服務條款未接受、認證狀態損壞、工作目錄不存在或無讀取權限、缺少 Git、會話 JSONL 損壞。根因診斷：加 --verbose 並捕捉 stderr，不要僅憑 exit code 1 判斷為 LLM 失敗。

陷阱 2：auto 模式在 `-p` 下分類器連續阻擋即 abort

--permission-mode auto 在互動模式下遭阻擋時會問使用者；但在 -p 非互動模式下，沒有 fallback 到人，分類器連續阻擋達閾值即直接終止 [來源: code.claude.com/docs/en/best-practices, 2026]。

反例：自動化 CI 任務使用 auto 模式但 prompt 包含寫入 production 路徑的指令，分類器阻擋後任務 abort，CI 報 failed 但無明確錯誤訊息。

陷阱 3：動態系統 prompt 導致快取持續失效


# 錯誤：每次呼叫 git status 不同 → 快取全未命中
claude -p "Analyze repository"

# 正確
claude -p --exclude-dynamic-system-prompt-sections "Analyze repository"

在同一主機跨多個呼叫中，git 狀態、當前目錄等動態資訊不同，即使 model 和 prompt 相同也會破壞 prefix 一致性。

陷阱 4：stream-json chunk 邊界問題

換行分隔 JSON 在 subprocess 的 stdout 緩衝中可能在邊界處被截斷 [來源: backgroundclaude.com/blog/stream-json, 2026]。不攜帶跨 chunk 部分行的解析器會在大型回應中隨機 crash。必須實作：累積緩衝 → 按換行分割 → 逐行 parse → 跨 chunk 攜帶部分行殘留。

陷阱 5：Session 恢復不還原 Permission 設定

--resume 重新開啟舊會話時，session-scoped permissions 不會被還原 [來源: arXiv 2604.14228v1, 2026]，這是設計上的安全原則。在批次自動化中重用舊會話時，若期待前次設定的 --allowedTools 仍生效，會遇到意外的 permission prompt。

與本專案關聯

場景 1：`run-todoist-agent-team.ps1` Phase 2 批次優化

本專案在 Phase 2 以 PowerShell 啟動多個 claude -p 子進程執行 auto-tasks（Start-CursorCliJob）。可落地改進：

  • 在純 LLM 輸出任務中加入 --bare 旗標，測量啟動時間差異（基準：每日 ~65 次呼叫）
  • 加入 --exclude-dynamic-system-prompt-sections，讓同類任務共享 prompt cache 前綴，估計可節省 40-60% 的 input tokens
  • 加入 --fallback-model haiku 應對主模型過載（本專案已發生 2026-04-15 Anthropic outage）

場景 2：token-efficiency 排程批次呼叫快取策略

run-token-efficiency.ps1 串接 5 個 Python 工具。若其中使用 claude -p 的環節統一設定：

  • ANTHROPIC_DEFAULT_SONNET_MODEL=claude-sonnet-4-6（確保 2048 門檻對齊）
  • 統一 system prompt 不含動態時間戳或 git 狀態
  • 同批次任務 5 分鐘內完成 → 共享同一個 5 分鐘 TTL 快取

預期每 2 小時批次可節省 50-70% input tokens 成本。

程式碼範例


# 批次優化：最快啟動 + cache 友善
claude --bare -p "Summarize this file" \
  --exclude-dynamic-system-prompt-sections \
  --model sonnet \
  --max-turns 3 \
  --output-format json \
  > results/output.json

# 限制工具 + 預算守衛
claude -p "Analyze security vulnerabilities" \
  --tools "Read,Grep,Glob" \
  --permission-mode dontAsk \
  --max-budget-usd 1.00 \
  --output-format json

# stream-json jq 解析（官方推薦模式）
claude -p --output-format stream-json "Explain this code" \
  | jq -rj 'select(.type=="stream_event" and .event.delta.type?=="text_delta") | .event.delta.text'

# PowerShell 批次迴圈模式（本專案適用）
$files = Get-ChildItem "src/*.py" | Select-Object -ExpandProperty FullName
foreach ($file in $files) {
  $result = & claude --bare -p "Add type hints to $file" `
    --tools "Read,Edit" `
    --permission-mode dontAsk `
    --max-turns 5 `
    --output-format json | ConvertFrom-Json
  Write-Host "Processed $file: $($result.status)"
}
