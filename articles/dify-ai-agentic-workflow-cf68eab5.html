一、專案概述

Dify（Do It For You）是由 LangGenius, Inc. 開發的開源 AI 工作流建構平台，定位為「production-ready platform for agentic workflow development」。專案於 2023 年啟動，2025 年 3 月發布 v1.0 里程碑版本，截至 2026 年 3 月已累積 131,000 GitHub Stars、20,400 Forks、9,116 Commits，是開源 AI 應用平台中 Stars 數最高的專案之一。

核心價值主張

Dify 解決的核心問題是：從 LLM 原型到生產部署之間的鴻溝。傳統上，開發者需要自行整合 LLM API、建構 RAG 管線、實作 Agent 邏輯、監控生產環境，每個環節都需要不同的工具鏈。Dify 將這些需求統一在一個視覺化平台中，讓非技術人員也能在 45 分鐘內建構完整的 AI 應用。

二、技術架構

2.1 三層模組化架構

Dify 的核心架構由三個獨立模組組成：

  1. LLM Orchestration（模型編排層）

- 統一介面連接數百個 LLM（GPT-4、Claude、Gemini、Llama、Qwen 等）

- 支援 OpenAI API 相容格式，可無縫切換模型

- 本地部署支援（Ollama、vLLM 等）

- Azure OpenAI、Hugging Face 模型整合

  1. Visual Studio（視覺化工作室）

- 拖放式畫布設計 AI 工作流

- 支援節點類型：LLM 呼叫、知識庫檢索、條件分支、程式碼執行、HTTP 請求、Agent 節點

- Prompt IDE：提示詞工程介面，支援變數注入與模型比較

- 預建應用模板，快速啟動

  1. Deployment Hub（部署中心）

- 一鍵部署為 API、聊天機器人、嵌入式小工具

- 自動生成 RESTful API

- 內建日誌追蹤：輸入/輸出、Token 消耗、節點耗時

2.2 Queue-Based Graph Execution Engine（佇列式圖執行引擎）

Dify 1.9.0 版本重建了工作流引擎，從舊有的線性執行模型改為佇列式圖執行模型：

  • 統一佇列排程：所有任務進入統一佇列，排程器管理依賴關係與執行順序
  • 移除並行限制：舊引擎在複雜並行場景有瓶頸，新引擎完全解除
  • 彈性節點連線：管線可從任意節點啟動、中途暫停與恢復
  • 奠定進階功能基礎：斷點除錯、Human-in-the-Loop、觸發式執行

這個設計與 Daily-Digest-Prompt 的 Phase 並行架構有顯著相似之處 — 都是將獨立任務放入佇列並行執行，再匯總結果。

2.3 Plugin-First 架構（v1.0）

Dify v1.0 引入了解耦式插件架構，五大核心組件：

  1. Models：AI 模型管理即插件，跨 chatbot/agent/workflow 使用
  2. Agent Strategies：推理策略插件（Chain-of-Thoughts、Tree-of-Thoughts、Function Call、ReAct）
  3. Tools：50+ 內建工具（網頁搜尋、計算機、程式碼解釋器）
  4. Extensions：社群擴展（Slack 通知、資料庫查詢等）
  5. Knowledge Pipeline Plugins：RAG 管線各階段可插拔

每個插件獨立打包、獨立部署、獨立維護，支援版本控制與安全審計。

三、Knowledge Pipeline RAG 系統

3.1 設計哲學

Dify 將 RAG 定義為「Context Engineering 的資料基礎設施」，Knowledge Pipeline 提供從原始資料到可信上下文的完整視覺化管線。

3.2 RAG 管線架構


原始文件 → 解析（Parsing）→ 清洗（Cleaning）→ 分塊（Chunking）→ 嵌入（Embedding）→ 向量儲存（Vector Store）→ 檢索（Retrieval）→ 重排序（Reranking）→ 生成（Generation）

3.3 關鍵技術特性

  • 階層式父子分塊（Parent-Child Chunking）：保留局部精確度與全局上下文，適合長技術文件
  • 多模態知識庫：文字與圖片統一到單一語義空間，支援多模態 RAG 與視覺推理
  • 多模態嵌入：將內容映射到向量空間進行快速相似性匹配
  • 多模態重排序：評估查詢與文字/圖片的具體相關性
  • 可調配置：區塊大小、重疊率、嵌入模型均可自訂
  • ~85% 檢索準確率（50 文件測試場景）

3.4 向量資料庫整合

| 向量資料庫 | 特點 |

|------------|------|

| pgvector（預設） | PostgreSQL 原生，零額外依賴 |

| Milvus | 分散式，高效能 |

| Qdrant | Rust 實作，混合搜尋 |

| Weaviate | GraphQL 介面 |

| Chroma | 輕量級 |

| TiDB Vector | 分散式語義搜尋 |

| Oracle | 企業級 |

3.5 Knowledge Pipeline 插件生態

管線各階段支援社群插件：

  • 解析：LlamaParse、Unstructured、Mistral OCR、EasyDoc、Docling
  • 分塊：可自訂分塊策略
  • 嵌入：支援多種嵌入模型
  • 外部工具：可透過 MCP 鉤入外部工具

四、Agent 框架

4.1 雙推理策略

  • Function Calling：結構化工具呼叫（OpenAI 風格），適合確定性任務
  • ReAct：推理-行動鏈式思考，適合開放式探索任務

4.2 Agent Node（自主推理節點）

在工作流中嵌入 Agent Node，賦予工作流「自主推理」能力：

  • 自動選擇工具
  • 多步驟推理
  • 上下文檢索
  • 與工作流其他節點無縫銜接

4.3 局限性

  • 多 Agent 編排相比 CrewAI、AutoGen 等專用框架仍較基礎
  • 缺乏 Agent 間通訊協議（對比 OWL/CAMEL-AI 的雙 Agent 編排）

七、優缺點評估

優點

  1. 開源 + 自託管：完全資料控制，無供應商鎖定
  2. 視覺化工作流：非技術人員可在 45 分鐘內建構 AI 應用
  3. Plugin-First 架構：v1.0 解耦設計，社群生態快速成長
  4. Knowledge Pipeline：RAG 管線視覺化，各階段可插拔
  5. 多模態 RAG：文字+圖片統一語義空間，業界領先
  6. 佇列式圖引擎：並行執行無瓶頸，支援斷點與 Human-in-the-Loop
  7. 內建 LLMOps：日誌、Token 追蹤、效能監控
  8. 模型無關：支援數百個 LLM，一鍵切換

缺點

  1. 畫布效能：20+ 節點時效能下降，複雜分支邏輯視覺混亂
  2. 多 Agent 編排基礎：相比 CrewAI/AutoGen 仍有差距
  3. 無內建前端：僅提供後端+API，前端需自行開發
  4. 企業功能待完善：SSO/RBAC 文件不集中，SOC 2/ISO 認證開發中
  5. 文件更新滯後：功能更新快，文件跟不上
  6. 高流量瓶頸：中小型工作負載表現佳，高流量場景可能有瓶頸
  7. Metadata 過濾：RAG 中的 metadata 過濾需要變通方案

八、潛在應用：Daily-Digest-Prompt 系統的借鑑

8.1 架構啟發

| Dify 機制 | Daily-Digest 對應 | 借鑑價值 |

|-----------|-------------------|----------|

| Queue-Based Graph Engine | Phase 並行架構（5 路 Phase 1） | Dify 的佇列排程更通用，可考慮引入依賴感知的任務佇列 |

| Knowledge Pipeline | knowledge-query Skill + KB 匯入 | Dify 的分塊策略（Parent-Child Chunking）可改善 KB 檢索品質 |

| Plugin-First Architecture | Skill-First 策略 | 極為相似！Dify 的插件解耦 = Daily-Digest 的 Skill 自包含 |

| Human Input Node | 目前無 | 可在 Todoist 任務流程中加入人工審核節點 |

| Built-in Observability | post_tool_logger + query-logs | Dify 的 Token 追蹤可借鑑，量化每次執行的 Token 成本 |

8.2 具體應用場景

  1. RAG 分塊策略升級：目前 KB 使用簡單分塊，可借鑑 Dify 的階層式父子分塊，改善長文件（如佛學經文研究）的檢索品質
  2. 多模態知識庫：未來 game-design 任務可能需要圖片+文字的統一檢索
  3. 工作流觸發器：Dify 的 Trigger 概念可延伸到 Daily-Digest — 例如 Todoist 新增高優先任務時自動觸發執行
  4. Token 經濟學監控：借鑑 Dify 的 Token 消耗追蹤，加入 post_tool_logger 的 Token 成本統計

九、快速上手

9.1 Docker Compose 部署


git clone https://github.com/langgenius/dify.git
cd dify/docker
cp .env.example .env
docker compose up -d

部署完成後訪問 http://localhost/install 初始化管理員帳號。

9.2 建立第一個工作流

  1. 登入 → Studio → 建立應用 → 選擇「Workflow」
  2. 拖入 LLM 節點 → 選擇模型（GPT-4/Claude/Llama）
  3. 拖入 Knowledge 節點 → 連接知識庫
  4. 加入條件分支 → 設定路由邏輯
  5. 發布 → 自動生成 API endpoint

9.3 API 呼叫範例


curl -X POST 'https://api.dify.ai/v1/workflows/run' \
  -H 'Authorization: Bearer YOUR_API_KEY' \
  -H 'Content-Type: application/json' \
  -d '{"inputs": {"query": "你的問題"}, "response_mode": "blocking"}'
