主題：從 23K chars/call 到 12K 基準線的系統性壓縮路徑

嚴重程度：Critical（超標 94.9%）

---

一、摘要（Executive Summary）

本簡報針對當前系統 Context I/O 預算嚴重超標問題（avg_io_per_call = 23385 chars，超標 94.9%）進行診斷分析，並提出系統性優化路徑。

核心發現：

  • 問題嚴重性被低估：23385 chars 已排除研究類任務，表示「一般任務」本身超標
  • 診斷鏈路不完整：關鍵依賴 context-budget-monitor Skill 缺失
  • 壓縮策略已定義但未執行：context-compression-history.json 不存在
  • 知識庫缺口明顯：僅 2 筆低相關筆記（score 0.5）
  • 閉環監控機制缺失：缺乏長期趨勢追蹤與自動觸發

建議行動：

  1. 【P0】建立 context-budget-monitor Skill：執行 per-session I/O 分析
  2. 【P0】執行 Summary 壓縮策略：compression_ratio=0.3，產出壓縮指示檔與歷史記錄
  3. 【P1】7 天後驗證效果：avg_io_per_call 是否降至 12000 以下
  4. 【P2】整合閉環監控：將驗證步驟整合至每週 system-insight 自動任務

二、關鍵洞察（Key Insights）

洞察 1：問題嚴重性被低估

發現：當前 avg_io_per_call = 23385 chars（超標 94.9%），但 config/benchmark.yaml 已排除 14 個研究類任務（ai_deep_research、github_scout 等）的 I/O 統計。

證據：

  • context/system-insight.json → metrics.avg_io_per_call: 23385
  • config/benchmark.yaml → exclude_task_keys: [14 個研究類任務]
  • raw_data_snapshot.excluded_research_calls: 668（已排除）

意義：23385 chars 是「一般任務」（todoist-query、email、podcast 等）的真實值，問題比表面數據更嚴重。

---

洞察 2：診斷鏈路不完整

發現：雖有完整 8 步驟 workflows/context-io-budget-control-workflow.yaml，但步驟 3 依賴的 context-budget-monitor Skill 狀態為「需建立」。

證據：

  • workflow 步驟 3：「執行 I/O 診斷分析」→ 需 Read skills/context-budget-monitor/SKILL.md
  • dependencies.skills.context-budget-monitor.status: "需建立"

影響：無法識別高 I/O Agent（top 5）與工具呼叫（top 10），診斷鏈路中斷。

解法：建立 context-budget-monitor Skill，實作 per-session I/O 分析（讀取 logs/structured/*.jsonl，統計 io_chars 欄位）。

---

洞察 3：壓縮策略已定義但未執行

發現：templates/shared/preamble.md 定義三級壓縮策略（BufferWindow/Summary/Delegation），但 context/context-compression-history.json 不存在，表示從未實際執行過壓縮。

證據：

  • preamble.md → Context 壓縮感知（ADR-036）機制：檢查 state/context-compression-hint-*.txt
  • 實際檢查：context/context-compression-history.json 不存在
  • workflow 步驟 6：「記錄壓縮動作至歷史」→ 建立 history.json

意義：缺乏壓縮執行歷史與效果驗證數據（before/after I/O 比較、effectiveness 改善百分比等），無法評估策略有效性。

建議：首次執行時詳細記錄 avg_io_before、avg_io_after、effectiveness（改善百分比）、params（compression_ratio 等），供未來參考。

---

洞察 4：預防與治理措施脫節

發現：preamble.md 的 Context 保護規則（委派子 Agent）是預防性措施，但當前已超標 94.9%，需要診斷性措施 + 治理性措施雙管齊下。

預防性措施（已有）：

  • 委派子 Agent 處理讀取 5+ 檔案或耗時搜尋
  • 主 Agent 只接收 ≤200 行 JSON 摘要

診斷性措施（缺失）：

  • 識別現有高 I/O 來源（top 5 Agent + top 10 工具呼叫）
  • 分析 I/O 分佈（Read/Write/Grep 佔比）

治理性措施（已定義但未執行）：

  • BufferWindow（保留最近 5 次關鍵結果）
  • Summary（強制壓縮，compression_ratio=0.3）
  • Delegation + Summary（雙策略，compression_ratio=0.2）

建議：預防措施持續執行，同時啟動診斷（建立 context-budget-monitor）+ 治理（執行 Summary 壓縮）雙軌。

洞察 6：閉環監控機制缺失

發現：workflow 步驟 7 建議「7 天後驗證效果」，但未整合至定期自動任務，缺乏長期趨勢追蹤與自動觸發機制。

當前狀態：

  • integration.automated_trigger.enabled: false（手動觸發）
  • 無自動提醒機制（7 天後執行驗證）
  • 無長期趨勢追蹤（歷次壓縮效果比較）

建議閉環設計：

  1. 7 天後驗證：整合至 system-insight 自動任務（每週執行），檢查 context-compression-history.json 最近一筆記錄的 timestamp，若 > 7 天則觸發驗證步驟
  2. 自動觸發：穩定後啟用 automated_trigger，當 system-insight 偵測 avg_io_per_call > 12000 時自動執行 workflow
  3. 長期追蹤：context-compression-history.json 累積歷次壓縮數據，供趨勢分析與最佳實踐識別

【立即】P0 行動（1-3 天內）

#### 1. 建立 context-budget-monitor Skill

五、附錄：三級壓縮策略詳解

A. BufferWindow 策略（warning 級別）

適用：avg_io_per_call 12000-15000 chars（超標 ≤ 25%）

機制：保留最近 N 次關鍵結果，捨棄中間過程

  • buffer_size: 5（保留最近 5 次）
  • apply_to: "研究類、分析類 Agent"

實作：在每個主要步驟開始前，檢查工作記憶累積量，若 > buffer_size 則移除最舊的 N-5 筆

---

B. Summary 策略（critical 級別）★ 當前建議

適用：avg_io_per_call 15000-18000 chars（超標 25%-50%）

機制：強制壓縮工作記憶為摘要，減少累積內容

  • compression_ratio: 0.3（壓縮至原始 30%）
  • summary_length: 200（摘要最多 200 字）

實作：

  1. 識別累積的大型內容（如多次 Read/Grep 結果）
  2. 提取關鍵資訊（任務摘要、已完成步驟、待完成步驟、關鍵數據）
  3. 捨棄原始詳細內容，僅保留摘要

---

C. Delegation + Summary 策略（emergency 級別）

適用：avg_io_per_call ≥ 18000 chars（超標 ≥ 50%）

機制：委派子 Agent 處理重量操作 + 強制壓縮

  • delegation_threshold: "讀取 5+ 檔案或執行耗時搜尋"
  • summary_compression_ratio: 0.2（壓縮至原始 20%）

實作：

  1. 委派：所有讀取 5+ 檔案或耗時搜尋的操作委派子 Agent
  2. 子 Agent 只回傳 ≤200 行 JSON 摘要
  3. 主 Agent 對累積內容執行更激進的 Summary 壓縮（compression_ratio=0.2）

六、結語

當前 Context I/O 預算超標 94.9% 屬 critical 級別，但已有完整診斷與壓縮工具鏈（workflow + preamble 機制），缺乏的是執行與驗證。建議立即啟動 P0 行動（建立 context-budget-monitor Skill + 執行 Summary 壓縮），7 天後驗證效果，並整合至閉環監控形成長期治理機制。

預期成果：

  • 短期（7 天內）：avg_io_per_call 降至 15000 chars 以下（改善 35%+）
  • 中期（1 個月內）：avg_io_per_call 降至 12000 chars 以下（達成基準線）
  • 長期（3 個月內）：建立自動觸發與閉環監控，avg_io_per_call 穩定在 12000 chars 以下
