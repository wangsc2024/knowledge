LangGraph Postgres 持久化與語意索引成本：PostgresSaver／PostgresStore／向量層的營運權衡（2026）

執行摘要（約 320 字）

在 LangGraph 架構中，可中斷續跑與跨對話記憶分屬兩條持久化軸：前者由 checkpointer（BaseCheckpointSaver，常見實作含 PostgresSaver）依 thread_id 序列化圖狀態快照；後者由 BaseStore（如 PostgresStore）以命名空間＋鍵儲存可跨 thread 的 JSON 文件，並可選擇性接上向量／語意檢索 [來源：LangGraph Persistence, LangChain Docs, 2026]。與僅使用記憶體相比，Postgres 後端帶來可預測的耐久性與多人／多程序並行能力，但成本轉為：資料表與索引膨脹、WAL／VACUUM 壓力、以及當 Store 啟用語意層時的 embedding 批次寫入與向量索引（HNSW／IVFFlat）記憶體 [來源：同前；並參考本知識庫 Qdrant 指南對向量層的討論，2026]。對 daily-digest-prompt 而言，本機知識庫已採 Qdrant + 混合檢索（見專案 KB 既有筆記），若未來將 Agent 狀態機遷移至 LangGraph，可將「對話／圖 checkpoint」留在 Postgres、「全域 RAG」留在既有 Qdrant，以避免在單一資料庫內重複建兩套向量索引。

技術概述（約 150 字）

PostgresSaver 負責在每個 super-step 邊界寫入 checkpoint，使圖可重播、可從失敗節點恢復，並支援人機協作暫停／恢復語意 [來源：LangGraph Persistence, LangChain Docs, 2026]。PostgresStore 則實作長期記憶槽位，典型用法是儲存使用者偏好、摘要或規則，並可與語意搜尋整合；官方文件強調 checkpointer 無法在 thread 間共享狀態，因此跨會話資訊必須走 Store [來源：同前；KB 筆記《LangGraph Store 與跨 thread 長期記憶》, 2026]。

核心概念與原理（含內嵌來源）

  1. Checkpoint 與 thread 邊界 — LangGraph 以 thread_id 作為對話／工作單元主鍵；checkpoint 序列使執行具備 durable execution 特性，與 Human-in-the-loop、interrupt 恢復同一套持久化基礎 [來源：LangGraph Persistence, LangChain Docs, 2026]；多代理比較筆記亦將「checkpoint 為中心」列為 LangGraph 與他框架的結構性差異 [來源：多代理編排框架深度比較, KB, 2026]。
  2. Store 與 checkpointer 正交 — 前者解「跨 thread 長期記憶」，後者解「單一 thread 內狀態機」；兩者並用時應避免把大物件重複塞進 checkpoint，而應讓可共享知識落在 Store，降低每步快照體積 [來源：LangGraph Store 與跨 thread 長期記憶, KB, 2026]。
  3. 語意索引成本維度 — 若 Store 層啟用向量檢索，需評估：embedding 呼叫費用、向量維度 × 文件數 的儲存、以及 HNSW 等索引的記憶體占用；獨立向量庫（如本專案 KB 使用的 Qdrant）可把 OLTP（Postgres）與 ANN 搜尋解耦，但增加網路 hop 與維運面 [來源：Qdrant 向量數據庫完整指南, KB, 2026]。
  4. 營運策略 — 對 checkpoint 表建議：保留天數／每 thread 最大版本數、定期 VACUUM／監控表膨脹、連線池上限與逾時 與圖節點 idempotency 設計並列為 SLO；對 Store 語意層則建議：批次 embedding、冷熱分層（舊摘要壓縮）、以及與 RAG 管線共用同一向量索引以避免重複 [來源：LangGraph Persistence, LangChain Docs, 2026；Qdrant 指南, KB, 2026]。

最佳實踐與常見陷阱

  • 實踐：生產環境優先選官方維護的 Postgres checkpointer 套件並在部署文檔中啟用 setup()／遷移流程；為 thread_id 與使用者／租戶建立可觀測的對照表，便於稽核與清理 [來源：LangGraph Persistence, LangChain Docs, 2026]。
  • 實踐：將「可重建的快取」放在圖外（或 Store 的可淘汰槽位），避免 checkpoint 快照包含巨型 payload 導致 I/O 爆炸。
  • 陷阱：誤把「需要跨使用者共享的知識」只存在某一 thread_id 的 checkpoint — 應上收至 Store 或外部 RAG（與本專案 knowledge-query 技能一致）。
  • 陷阱：在 Postgres 內同時承載高寫入 checkpoint 與高維度向量索引卻未做資源隔離，易與 OLTP 業務競爭 IOPS [來源：工程共識 + Qdrant 指南, KB, 2026]。

與本專案（daily-digest-prompt）的關聯

  • Phase 1→2→3 已有 JSON 結果檔與 Hook JSONL；若引入 LangGraph，可把 一次自動任務映射為 thread_id=task_run_id，checkpoint 對齊 Phase 重試／守衛失敗 的恢復點 [來源：LangGraph Persistence, LangChain Docs, 2026]。
  • 全域知識仍應走 localhost:3000 KB（Qdrant 後端健康可由 cache/kb_live_status.json 與 /api/health 監控），避免與 Agent 圖狀態混在同一資料模型。
  • 與 Cursor CLI／多後端配額 場景類比：Postgres 連線與向量索引亦需 斷路器與降級路徑（與 config/autonomous-harness.yaml 風險層級一致）。
