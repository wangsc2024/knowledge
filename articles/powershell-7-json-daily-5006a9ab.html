研究類型：技術研究

今日任務涉及技術：PowerShell 自動化、RAG 知識庫同步、結果 JSON 寫入、Vite 靜態同步、GitHub 推送、結構化日誌。

本次研究主題：PowerShell 7 結構化錯誤處理、JSON 契約與並行工作流。

說明：state/todoist-history.json 的 2026-03-19 daily_summary 只有完成數量、未保存同日任務文字，因此本文依 closed_task_ids 對應最近一次同 recurring task id 的歷史內容推定今日技術需求；這是基於專案資料的合理推論。

技術概述

截至 2025-02-12，PowerShell 7.5 GA 已正式發布，官方把重點放在品質、安全與穩定性；Microsoft Lifecycle 也顯示 PowerShell 7.5 與 7.4 LTS 都支援到 2026-11-10。對 daily-digest-prompt 這種以 Windows 排程、自動寫入 results/*.json、呼叫本機 KB API、並需處理多步驟腳本的專案而言，7.5 的實際價值不只在新功能，而在「把非零退出碼、JSON 序列化與 runspace 併發」整成一套可預期的失敗模型。若錯誤只看 $LASTEXITCODE、JSON 只靠預設深度、並行只無限制地開 -Parallel，流程很容易出現假成功、格式漂移或資源互搶。

核心概念與原理

1. 原生命令錯誤語意

$PSNativeCommandUseErrorActionPreference 讓原生命令的非零退出碼能依 $ErrorActionPreference 轉成 PowerShell error。這點很重要，因為 curl.exe、git.exe、robocopy.exe 這類外部程式原本只會留下 exit code；若腳本沒主動檢查，就可能在 API 呼叫失敗時仍繼續寫出 success 結果檔。

實務上應把規則拆成兩層：

  • 一般外部命令：開啟 $PSNativeCommandUseErrorActionPreference = $true
  • 例外命令：像 robocopy.exe 這種非零碼不一定代表錯誤，局部關閉後自行判讀 $LASTEXITCODE

這樣可以把「失敗是否中止流程」從散落各處的 if ($LASTEXITCODE ...)，收斂為統一的 PowerShell 錯誤語意。

2. JSON 契約不是 `ConvertTo-Json` 預設值

ConvertTo-Json 在 PowerShell 7.1 起會對深度截斷發出警告；到 7.5-preview.3 又新增 BigInteger 直接序列化為 JSON number。這代表 JSON 輸出行為會隨版本演進。如果專案結果檔被 Phase 3 當成資料契約，腳本就不能只依賴預設行為，必須明確指定：

  • -Depth：避免巢狀物件被截斷
  • -AsArray：當下游要求陣列格式時保持穩定
  • UTF-8 寫檔：避免 Windows 編碼干擾後續讀取

在本專案脈絡下，results/todoist-auto-*.json 是 handoff contract，不是單純輸出文字；因此 JSON 行為必須被視為 API 契約的一部分。

3. `ForEach-Object -Parallel` 的 runspace pool 模型

官方文件指出，ForEach-Object -Parallel 在 PowerShell 7 使用平行參數集，ThrottleLimit 預設值是 5。Microsoft Learn 也說明，自 PowerShell 7.1 起，ForEach-Object -Parallel 會預設重用 runspace pool，而不是每輪重建 runspace；-UseNewRunspace 才會強制每次都開新 runspace。

這背後代表三個實作判準：

  • 工作很輕時，不要為了「感覺比較快」濫用平行
  • 以 I/O 等待型任務為主時，平行最有價值
  • 若腳本需共享可變狀態，必須假設跨執行緒不安全

對自動化專案而言，最合理的方式通常不是全面平行，而是把「獨立 API 呼叫、獨立檔案處理、互不依賴的查詢」放進平行區，而把最終結果彙總與檔案落盤保留在主執行緒。

最佳實踐與常見陷阱

最佳實踐

  1. 先把外部命令錯誤提升為 PowerShell error，再用 try/catch/finally 統一收斂。
  2. 所有結果檔輸出都顯式指定 JSON 深度與 UTF-8 編碼。
  3. 把 ForEach-Object -Parallel 的 ThrottleLimit 當成容量參數，而不是效能開關。
  4. 平行區只做可重試、無共享狀態的工作；集中寫檔與狀態更新放在主線程。
  5. 以狀態檔先寫 running 佔坑，再在結尾覆寫 success / partial / failed，避免中途中止造成缺檔。

常見陷阱

  1. 只看 Invoke-RestMethod 成功、不驗證回應是否真的含預期欄位。
  2. 讓平行區直接共寫同一個 JSON 檔或 registry，導致 race condition。
  3. 依賴 ConvertTo-Json 預設深度，讓深層欄位靜默遺失。
  4. 把所有外部命令都當成「非零即錯」，忽略 robocopy 類特例。
  5. 對 trivial task 濫用 -Parallel，反而因 runspace 開銷變慢。

與本專案的關聯

這個主題直接對應 daily-digest-prompt 的幾個真實需求：

  • 自動任務必須先建立 results/todoist-auto-tech_research.json 佔坑，否則超時時 Phase 3 只能看到缺檔。
  • 知識庫健康檢查與匯入需要呼叫本機 API；若外部命令或 HTTP 呼叫失敗卻未被提升為 error，流程會出現「假成功」。
  • research-registry.json、continuity/.json、results/.json 都是結構化 handoff；一旦 JSON 深度或編碼不穩定，下游統計就會偏移。
  • 後續若把多個研究查詢、KB 搜尋或 log 分析做平行化，必須先定義 runspace pool 上限與主線程回寫策略。

程式碼範例


$ErrorActionPreference = 'Stop'
$PSNativeCommandUseErrorActionPreference = $true

$result = [ordered]@{
    agent = 'todoist-auto-tech_research'
    type = 'tech_research'
    status = 'running'
    summary = '執行中...'
    error = $null
}

$tasks = @('health', 'hybrid-search', 'import')

try {
    $checks = $tasks | ForEach-Object -Parallel {
        switch ($_ ) {
            'health' {
                [pscustomobject]@{ step = $_; ok = $true; detail = 'localhost:3000 healthy' }
            }
            'hybrid-search' {
                [pscustomobject]@{ step = $_; ok = $true; detail = 'topic dedup checked' }
            }
            'import' {
                [pscustomobject]@{ step = $_; ok = $true; detail = 'note import prepared' }
            }
        }
    } -ThrottleLimit 3

    $result.status = 'success'
    $result.summary = 'PowerShell 7 工作流完成，錯誤語意與 JSON 契約已驗證'
    $result.checks = $checks
}
catch {
    $result.status = 'failed'
    $result.summary = 'PowerShell 7 工作流失敗'
    $result.error = $_.Exception.Message
}
finally {
    $result | ConvertTo-Json -Depth 6 | Set-Content -Path 'results/todoist-auto-tech_research.json' -Encoding utf8
}

這段範例的重點不在平行本身，而在三個順序：

  1. 先統一錯誤語意
  2. 再做可隔離的平行處理
  3. 最後在主線程一次性寫入結果 JSON
