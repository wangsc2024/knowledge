Gemini CLI — Google 開源終端 AI Agent 架構深度洞察

概述

Gemini CLI 是 Google 於 2025 年開源的終端 AI Agent，以 Apache 2.0 授權發布於 GitHub（google-gemini/gemini-cli），截至 2026 年 3 月累積 99.6K Stars。它將 Gemini 模型直接帶入終端環境，定位為開發者 CLI-first AI 助手，支援程式碼理解、檔案操作、Shell 命令執行與 MCP 擴展，免費帳戶每日可用 1,000 次請求、每分鐘 60 次，搭載 Gemini 3 的 1M token 上下文視窗。

技術棧為 TypeScript + Node.js，透過 npm 發布（@google/gemini-cli），使用 esbuild 打包。

核心架構亮點

1. GEMINI.md 階層上下文發現機制

Gemini CLI 使用 GEMINI.md 作為專案層級的指令上下文檔案（類似本專案的 CLAUDE.md），但其設計有一個關鍵創新：自動向上遍歷目錄發現。系統從當前工作目錄開始，沿父目錄向上搜尋 GEMINI.md，直到遇到邊界標記（如 .git），所有找到的檔案按層級合併為最終上下文。

  • 支援自訂檔名（context.fileName 可設為字串或陣列）
  • 支援 includeDirectories 跨多個工作區
  • 這意味著組織層級可以有全域 GEMINI.md，專案層級有另一份，二者自動疊加

借鏡啟示：本專案目前 CLAUDE.md 僅有專案根目錄和全域兩層，可考慮支援子目錄級別的覆寫（如 prompts/CLAUDE.md、hooks/CLAUDE.md），已部分實現但缺乏自動合併語義。

2. 六層配置級聯（Configuration Cascade）

配置優先級從低到高：

  1. 系統預設值
  2. 使用者設定（~/.gemini/settings.json）
  3. 專案設定
  4. 系統覆寫（/etc/gemini-cli/settings.json）
  5. 環境變數
  6. CLI 參數

settings.json 按功能分區：general（vim 模式、檢查點）、ui（主題）、model（模型選擇、壓縮閾值）、tools（沙箱、Shell）、context（檔案發現）、security（環境變數遮蔽）、experimental（預覽功能）。

借鏡啟示：本專案用 YAML 分散配置（cache-policy.yaml、routing.yaml 等），但缺乏統一的級聯優先級模型。Gemini CLI 的六層模型提供了明確的覆寫語義。

3. MCP 工具整合與安全沙箱

#### MCP 配置

在 settings.json 的 mcpServers 區塊配置，支援三種傳輸：

  • command（stdio）
  • url（SSE）
  • httpUrl（HTTP 串流）

工具發現流程：

  1. 遍歷已配置的伺服器建立連線
  2. 呼叫每個伺服器的工具列表端點
  3. 用 includeTools/excludeTools 過濾
  4. 消毒工具名稱以確保 API 相容性
  5. 以命名空間前綴註冊（mcp_serverName_toolName）

#### 環境消毒（Environment Sanitization）

這是 Gemini CLI 最值得借鏡的安全設計：

  • 產生 MCP 伺服器程序時，自動遮蔽敏感環境變數
  • 匹配模式：TOKEN、SECRET、PASSWORD、KEY、AUTH、CREDENTIAL
  • 核心 key 如 GEMINI_API_KEY、GOOGLE_API_KEY 一律遮蔽
  • 使用者若在伺服器 env 區塊明確聲明變數，視為「知情同意」

借鏡啟示：本專案的 pre_read_guard.py 和 pre_bash_guard.py 已有敏感路徑攔截，但缺乏子程序級別的環境變數消毒機制。Gemini CLI 的 pattern-based redaction 可強化 Hook 系統。

4. 工具存取控制（Restrictive-by-Default）

合併策略遵循「限制優先」原則：

  • excludeTools 永遠優先於 includeTools
  • Extension 提供的工具清單與使用者清單做交集（intersect）
  • 排除清單做聯集（union）
  • 使用者對任何 Extension 工具保有否決權

另有 tools.allowed 命名模式可跳過確認（如 run_shell_command(git)），以及 tools.core 限制內建工具的白名單。

借鏡啟示：本專案的 Hook 系統（pre_bash_guard）是黑名單模式（攔截危險指令），可考慮增加白名單交集語義，讓新增工具預設被封鎖直到明確允許。

5. 沙箱執行環境

tools.sandbox 支援多種後端：

  • docker、podman、lxc
  • windows-native（原生 Windows 沙箱）
  • macOS Seatbelt profiles（permissive-open、restrictive-open、strict-open、strict-proxied、自訂）
  • tools.sandboxAllowedPaths 路徑白名單
  • tools.sandboxNetworkAccess 網路權限控制

借鏡啟示：本專案在 Windows 環境運行，目前無沙箱。Gemini CLI 的 windows-native 沙箱值得評估，可保護 Agent 執行 Shell 命令時的檔案系統安全。

6. 對話檢查點（Checkpointing）

Gemini CLI 支援儲存和恢復複雜對話 session，讓使用者可以：

  • 暫停長時間互動
  • 保留完整上下文後恢復
  • 在 CI/CD 管線中實現跨步驟的 Agent 狀態傳遞

借鏡啟示：本專案的跨次記憶（context/continuity/）是手動摘要式的，缺乏原生的對話狀態快照。Checkpointing 可為 Team 模式的多階段任務提供更精確的狀態傳遞。

7. 三階段發布節奏（Nightly → Preview → Stable）

  • Nightly：每日構建，最新功能
  • Preview：每週二 UTC 23:59 發布
  • Stable：每週 UTC 20:00 發布

搭配 GitHub Actions 整合，可作為 PR 審查和 Issue 分類的自動化代理（免費）。

借鏡啟示：本專案的 auto-task 系統可參考此三階段模型，將實驗性自動任務與穩定任務分開管理。

與 Claude Code CLI 的關鍵差異

| 維度 | Gemini CLI | Claude Code CLI |

|------|-----------|----------------|

| 上下文檔 | GEMINI.md（階層合併） | CLAUDE.md（專案+全域兩層） |

| 沙箱 | 多後端（Docker/Podman/Native） | 無原生沙箱 |

| MCP 安全 | 環境消毒 + 工具白/黑名單交集 | Hook 黑名單攔截 |

| 檢查點 | 原生 session 快照 | 無（靠 continuity JSON） |

| 配置 | 六層級聯 JSON | YAML 分散 + CLAUDE.md |

| 免費額度 | 1000 次/日 | 依 API 計費 |

| 授權 | Apache 2.0 | 商業 |

對 Daily-Digest-Prompt 的七大借鏡啟示

  1. 階層上下文合併：子目錄 CLAUDE.md 自動疊加至專案 CLAUDE.md，減少重複指令
  2. 環境變數消毒：在 Hook 層加入 pattern-based 敏感變數遮蔽，防止子 Agent 洩露 API Key
  3. 工具白名單交集：新增工具預設封鎖，需明確允許才可使用
  4. Windows 原生沙箱：評估 windows-native 沙箱為 Bash 工具添加隔離層
  5. 對話檢查點：為 Team 模式的跨階段 Agent 引入狀態快照機制
  6. 六層配置級聯：統一配置優先級模型，解決目前 YAML 分散管理的認知負擔
  7. 三階段發布模型：將實驗性自動任務（如 creative-game-optimize）與核心任務分離管理
