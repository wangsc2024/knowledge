✅ 執行摘要：建立了完整的 Markdown 指令、編輯與總結 Skill 指南（docs/research/Markdown_指令_編輯與總結_skill.md），涵蓋 33 種語法的詳細說明、CommonMark AST 解析管線機制、5 大渲染器兼容性矩陣、編輯工具快捷鍵手冊、以及手動/半自動兩種內容摘要方法（含 Python 腳本）。

📚 主要發現：

  1. AST 解析管線三階段：區塊解析（容器 vs 葉子）→ 行內解析（定界符堆疊配對）→ AST 輸出。這解釋了 ** 比 __ 更可靠的底層原因（intraword 側位規則）
  2. CommonMark 僅定義 20 種語法，GFM 擴展 7 種（表格、刪除線、任務清單等），第三方擴展 6 種（腳註、Mermaid、數學公式等）
  3. 渲染器兼容性差異集中在 2021-2023 年新增功能：GitHub 逐步加入腳註(2021)、數學公式/Mermaid(2022)、告警區塊(2023)；Typora 最寬鬆但不支援告警
  4. 編輯效率關鍵：一行一句（Semantic Line Breaks）減少 Git 衝突、表格不對齊 | 最小化 diff、批次操作用正則表達式
  5. TOC 生成的錨點規則：GitHub 用 小寫→移除標點→空格變連字號→重複加後綴，中文字元保留

★ Insight ─────────────────────────────────────

這份文件有一個值得注意的設計選擇：為何 CommonMark 不包含表格？因為表格語法與「段落中偶然出現的 | 字元」存在歧義，CommonMark 團隊決定讓核心規範保持最小化，將表格留給 GFM 等擴展。這是「規範最小化 vs 實用需求」的典型取捨 — 類似本專案 Skill-First 策略中「核心 Skill 精簡、工具 Skill 按需載入」的設計思路。
