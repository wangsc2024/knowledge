Unsloth 資料集準備與對齊訓練格式完整指南

本文涵蓋 Unsloth 支援的所有資料集格式（Raw Text、Alpaca、ShareGPT、ChatML）、Chat Template 系統、以及偏好對齊訓練（DPO/ORPO/KTO/GRPO）的資料準備與實戰配置。

一、Unsloth 支援的四大資料格式

1. Raw Text（連續預訓練 CPT）

用於在未結構化文本上繼續預訓練模型，保持自然語言流暢度。

{
  "text": "Pasta carbonara is a traditional Roman pasta dish..."
}

適用場景：領域適應（domain adaptation）、知識注入、語言風格遷移。

2. Alpaca 格式（單輪指令微調）

Alpaca 格式是最簡單的指令格式，設計用於單輪互動。

{
  "Instruction": "將以下英文翻譯成繁體中文",
  "Input": "Machine learning is a subset of artificial intelligence.",
  "Output": "機器學習是人工智慧的一個子集。"
}

欄位說明：

  • Instruction：任務描述（必填）
  • Input：使用者輸入（選填，可為空字串）
  • Output：期望的模型回應（必填）

Unsloth 會將每一列合併成一個完整的 prompt 進行訓練。

3. ShareGPT 格式（多輪對話）

ShareGPT 使用 conversations 欄位儲存 JSONL 格式的多輪對話。

{
  "conversations": [
    {"from": "human", "value": "請幫我解釋什麼是 LoRA？"},
    {"from": "gpt", "value": "LoRA（Low-Rank Adaptation）是一種參數高效微調方法..."},
    {"from": "human", "value": "它和全量微調有什麼差別？"},
    {"from": "gpt", "value": "主要差別在於訓練的參數量..."}
  ]
}

關鍵特點：

  • 使用 from/value 作為 key（非 role/content）
  • 角色名稱為 human 和 gpt
  • 最適合儲存多輪對話，貼近真實使用者互動模式

4. ChatML 格式（OpenAI 標準）

Hugging Face 預設使用的格式，也是目前最廣泛使用的格式。

{
  "messages": [
    {"role": "user", "content": "什麼是 1+1？"},
    {"role": "assistant", "content": "答案是 2！"}
  ]
}

特殊 Token：<|im_start|> 和 <|im_end|> 標記說話者的開始與結束。

二、Chat Template 系統

支援的模板

Unsloth 支援多種 Chat Template：

  • chatml、llama-3、gemma-3、phi-4、qwen-2.5
  • mistral、zephyr、alpaca、vicuna、unsloth

查詢可用模板

from unsloth.chat_templates import CHAT_TEMPLATES
print(list(CHAT_TEMPLATES.keys()))

套用模板到 Tokenizer

from unsloth.chat_templates import get_chat_template

tokenizer = get_chat_template(
    tokenizer,
    chat_template = "gemma-3",
)

格式化函數

def formatting_prompts_func(examples):
    convos = examples["conversations"]
    texts = [
        tokenizer.apply_chat_template(
            convo, tokenize=False, add_generation_prompt=False
        ) for convo in convos
    ]
    return {"text": texts}

from datasets import load_dataset
dataset = load_dataset("repo/dataset", split="train")
dataset = dataset.map(formatting_prompts_func, batched=True)

ShareGPT → ChatML 轉換

from unsloth.chat_templates import standardize_sharegpt

# 將 from/value 格式轉為 role/content 格式
dataset = standardize_sharegpt(dataset)
dataset = dataset.map(formatting_prompts_func, batched=True)

自訂欄位映射

mapping = {
    "role": "from",
    "content": "value",
    "user": "human",
    "assistant": "gpt"
}

tokenizer = get_chat_template(
    tokenizer,
    chat_template = "chatml",
    mapping = mapping,
)

三、進階資料轉換：to_sharegpt

將多欄位資料集合併為 ShareGPT 對話格式。

from unsloth.chat_templates import to_sharegpt

# 使用 {} 包裹欄位名，[[]] 標記選填區段
template = "乘客從 {Embarked} 上船。[[年齡為 {Age}。]][[票價為 {Fare}。]]"

語法規則：

  • {column_name}：引用資料集欄位
  • [[optional text {Col}]]：當欄位值缺失時，整段自動省略
  • output_column_name：指定輸出欄位

五、偏好對齊訓練資料格式

DPO / ORPO / KTO 共通格式

偏好訓練需要三個欄位：

{
  "prompt": "什麼是生命的意義？",
  "chosen": "生命的意義在於不斷成長、學習，並找到自己的目標與價值...",
  "rejected": "42 哈哈"
}

欄位說明：

  • prompt：問題或指令
  • chosen：偏好的回應（人類標註為較好的）
  • rejected：不偏好的回應（人類標註為較差的）

推薦資料集：argilla/ultrafeedback_binarized

ORPO 實戰配置

ORPO（Monolithic Preference Optimization）將 SFT 和偏好對齊合併為單一步驟，不需要參考模型。

from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/Meta-Llama-3-8B-bnb-4bit",
    max_seq_length = 2048,
    load_in_4bit = True,
)

model = FastLanguageModel.get_peft_model(
    model,
    r = 16,
    lora_alpha = 16,
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj"],
    use_rslora = True,
)

from trl import ORPOTrainer, ORPOConfig

orpo_config = ORPOConfig(
    learning_rate = 8e-6,
    beta = 0.1,                          # 偏好最佳化強度
    lr_scheduler_type = "cosine_with_restarts",
    max_length = 2048,
    per_device_train_batch_size = 4,
    num_train_epochs = 3,
    evaluation_strategy = "steps",
    optim = "adamw_8bit",
)

trainer = ORPOTrainer(
    model = model,
    args = orpo_config,
    train_dataset = dataset["train"],
    eval_dataset = dataset["test"],
    tokenizer = tokenizer,
)

trainer.train()

ORPO 監控指標：

  • rewards/chosen、rewards/rejected：對數機率
  • rewards/accuracies：chosen 超過 rejected 的頻率
  • rewards/margins：平均差值
  • nll_loss：SFT 元件的負對數似然

DPO 配置差異

from trl import DPOTrainer, DPOConfig

dpo_config = DPOConfig(
    beta = 0.1,
    per_device_train_batch_size = 4,
    gradient_accumulation_steps = 8,
    max_length = 1024,
    max_prompt_length = 512,
    optim = "adamw_8bit",
)

DPO 直接最佳化模型參數，不需要顯式獎勵模型，比 GRPO 更簡單。

六、GRPO 推理模型訓練

GRPO（Generative Reward Policy Optimization）用於訓練推理模型（如 DeepSeek-R1 風格）。

資料格式

GRPO 資料集至少需要 2 欄（問題和答案）：

{
  "question": "計算 15 × 23 的結果",
  "answer": "345"
}

關鍵：答案不應揭示推理過程（讓模型自己學習推理）。

推薦資料集：openai/gsm8k（數學推理）

獎勵函數設計

def format_reward(completions):
    """檢查輸出是否符合指定格式（如 XML tag）"""
    pattern = r"<think>[\s\S]*?</think>\s*<answer>[\s\S]*?</answer>"
    return [1.0 if re.match(pattern, c) else 0.0 for c in completions]

def answer_reward(completions, references):
    """驗證生成的答案是否正確"""
    rewards = []
    for comp, ref in zip(completions, references):
        extracted = extract_answer(comp)
        rewards.append(1.0 if extracted == ref else 0.0)
    return rewards

多目標最佳化：多個獎勵函數加總，同時最佳化格式正確性和答案準確性。

GRPOTrainer 配置

from trl import GRPOConfig, GRPOTrainer

config = GRPOConfig(
    num_generations = 8,            # 每個 prompt 生成幾個候選
    max_prompt_length = 512,
    max_completion_length = 1024,
    learning_rate = 5e-6,
    per_device_train_batch_size = 1,
    gradient_accumulation_steps = 4,
    num_train_epochs = 1,
    optim = "adamw_8bit",
)

trainer = GRPOTrainer(
    model = model,
    args = config,
    train_dataset = dataset,
    reward_funcs = [format_reward, answer_reward],
)

trainer.train()

GRPO 特有設定：

  • UNSLOTH_VLLM_STANDBY=1：延長上下文 30%
  • fast_inference = True：啟用 vLLM 加速推理
  • gpu_memory_utilization = 0.7：GPU 記憶體使用率

監控指標

  • reward：平均獎勵分數
  • reward_std：獎勵標準差
  • completion_length：平均生成長度
  • kl：與基礎模型的 KL 散度（應保持合理範圍）

七、各方法比較與選擇指南

方法資料需求複雜度適用場景
SFT（Alpaca/ShareGPT）instruction-output 對低通用指令跟隨
DPOprompt + chosen + rejected中有偏好標註資料時
ORPOprompt + chosen + rejected中一步到位（SFT+對齊）
KTOprompt + completion + label中只有二元好/壞標籤時
GRPOquestion + answer高訓練推理能力
SimPOprompt + chosen + rejected中簡化版 DPO

訓練流程選擇建議

  1. 快速起步：SFT（Alpaca 格式）→ 效果不足時加 DPO
  2. 一步到位：ORPO（同時完成 SFT + 偏好對齊）
  3. 推理增強：SFT → GRPO（獎勵函數引導推理格式）
  4. 資源受限：KTO（只需二元標籤，資料準備最簡單）

八、資料集準備最佳實踐

資料量建議

  • 最低門檻：100 列以上
  • 建議規模：1,000 列以上
  • 品質 > 數量：100 筆高品質資料優於 10,000 筆雜訊資料

格式一致性

  1. 確保所有列的格式完全一致（欄位名、角色名稱）
  2. 使用 standardize_sharegpt() 統一 ShareGPT 變體
  3. 檢查特殊字元（中文標點、換行符）是否正確處理

繁體中文微調注意事項

  • 優先選擇 Qwen 或 Breeze 系列模型（繁中語料比例較高）
  • 資料集中繁簡混用會降低品質，建議統一為繁體
  • 使用 max_seq_length 需考慮中文 tokenization 的膨脹率（約 1.5-2x）

常見問題

Q：ShareGPT 和 ChatML 如何選擇？ A：如果資料集已有 from/value 格式就用 ShareGPT + standardize_sharegpt()；如果是 role/content 格式就直接用 ChatML。兩者最終都會被模板系統統一處理。

Q：ORPO 和 DPO 差在哪？ A：ORPO 將 SFT 和偏好對齊合併為一個步驟，不需要參考模型，訓練更快但可能犧牲一些對齊精度。DPO 需要先做 SFT 再做偏好對齊，兩步但效果通常更穩定。

Q：GRPO 的獎勵函數如何設計？ A：獎勵函數是可組合的。建議至少包含「格式獎勵」（確保輸出格式正確）和「正確性獎勵」（驗證答案準確性）兩個函數，加總後作為總獎勵。

九、模型匯出

訓練完成後支援多種匯出方式：

# 儲存 LoRA adapter
model.save_pretrained("lora_model")

# 合併後儲存（16-bit）
model.save_pretrained_merged("fused_model", tokenizer, save_method="merged_16bit")

# GGUF 量化匯出（供 llama.cpp / Ollama 使用）
model.save_pretrained_gguf("ggufs", tokenizer, quantization_method="q4_k_m")
