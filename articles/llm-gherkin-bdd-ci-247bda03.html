LLM 驅動驗收測試與 Gherkin／BDD：兩階段管線與 CI 品質閘門（2026-05-03）

LLM 驅動驗收測試與 Gherkin／BDD（ai_sysdev 2026-05-03）

技術概述（約 180 字）

行為驅動開發（BDD）以 Gherkin 作為技術與非技術利害關係人共享的受限自然語言。2025–2026 年產業界與學界焦點在於：以 LLM 將「使用者故事」自動展開為可討論、可版控的驗收場景，並在第二階段把場景縫合 DOM／API 證據轉成可執行測試。工業個案（汽車供應鏈）以兩工具 AutoUAT→Test Flow 串接 GPT-4 Turbo、Gherkin 與 Cypress，搭配 PR 觸發的 GitHub Action，把探索式速度與人類審查閘門結合。

核心方法與工具

  • 兩階段分離：(i) US→Gherkin（澄清需求／ATDD）；(ii) Gherkin+HTML(淨化)→Cypress（降低 UI 漂移與幻覺同步爆量）。
  • HTML 預處理：移除 style/script 以降低 prompt（論文量測：輸入約 9500 tokens／腳本輸出約 750 tokens／每則 US 約 0.12€，2024/06 Azure 定價）。
  • Living Documentation：Cucumber JSON → Jenkins cucumber-living-documentation（Cukedoctor）把執行結果回寫為 HTML/PDF（https://www.jenkins.io/doc/pipeline/steps/cucumber-living-documentation/）。

工作流程（step-by-step）

  1. PO 填寫故事標題與描述；觸發 Power Platform→Azure OpenAI（論文 AutoUAT）。
  2. 產出 Gherkin 於 Teams；PO／QA 在 Three Amigos 會議定稿步驟詞彙表。
  3. Test Flow：由 JIRA issue key 拉 US+Gherkin；爬取待測 URL HTML（去樣式腳本）；呼叫 GPT-4 Turbo 產出 TypeScript Cypress。
  4. 人為強制 Gate：開發者審 PR 內生成腳本；語意不符分類（論文）：缺脈絡 12 件、一行內微錯 4 件、複合錯誤 4 件／共 50 案—補脈絡或一小行修正可把語意一致率自 60% 拉到 92%。
  5. CI：JUnit／Cucumber JSON 上傳 TestRail CLI；Living doc 於 main merge 後更新。

程式碼／設定範例

Gherkin 草案（LLM 輸出後應對齊 step catalog）


Feature: 購物車
  Scenario: 登入後加入有庫存商品
    Given 使用者已使用 email 登入
    When 使用者從商品頁加入購物車 sku "WIDGET-1"
    Then 購物車應顯示數量 1

GitHub Actions（論文 Fig.3：PR 觸發生成 Cypress 骨架）


# .github/workflows/llm-acceptance-from-jira.yml
on:
  pull_request:
jobs:
  gen-at:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Call Test-Flow API
        run: curl -fsS -X POST "$TESTFLOW_URL/gen" -d @payload.json
      - name: Commit generated Cypress if diff
        run: git add cypress/e2e && git commit -m "chore(at): llm cypress draft" || true

與本專案（daily-digest-prompt）場景

  • prompts/team/.md 與 results/todoist-auto-.json 可類比「可執行規格」：以 ADR／config/schemas/results-auto-task-schema.json 為單一真相，Given=前置檔案、When=Phase2 LLM、Then=validate_results.py 對齊 BDD Gate。
  • tools/*.py 機械步驟等同 step definitions，與多後端路由（Codex／OpenRouter／Claude）並存時，對齊論文的「租戶內模型＋人審」風險控管。

風險與注意事項

  • 幻覺場景：與 JIRA／Issue 單一來源交叉驗證（TestQuality 2026-03：https://testquality.com/how-to-generate-ai-gherkin-test-cases/）。
  • 步驟片語漂移：維護團隊 step catalog；AI 僅組合既有步驟。
  • 過量生成：風險分級挑自動化子集合。
