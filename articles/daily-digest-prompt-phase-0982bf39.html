一、系統成熟度量化評估

1.1 Agentic AI 成熟度對標（6 級模型）

依據 Salesforce Agentic Maturity Model 和 nexocode Crawl-Walk-Run-Scale 框架，本系統定位：

| 等級 | 描述 | 本系統達成度 |

|------|------|------------|

| Level 0 | 規則自動化（無推理） | 100% — Hook 規則層 |

| Level 1 | 檢索增強（RAG） | 100% — KB hybrid search |

| Level 2 | 推理與行動（ReAct） | 100% — Skill-First + TDD |

| Level 3 | 多 Agent 協作 | 90% — Team 並行模式 |

| Level 4 | 自適應優化 | 25% — 部分回饋迴圈 |

| Level 5 | 組織級 AGI | 0% — 超出當前範疇 |

評估結論：系統處於 Level 3.25，多 Agent 協作成熟但自適應能力不足。主要差距在於系統缺乏基於執行結果的自動策略調整機制。

二、六大優化建議（P0-P2 優先級）

P0-A：State 目錄資源洩漏治理

問題：state/ 目錄累積 264 個 loop-state-*.json 檔案，每個 Session 產生一個但從未清理。按每日 44 次執行計算，每月新增 ~1,320 個檔案。

建議方案：

  1. 在 run-agent-team.ps1 和 run-todoist-agent-team.ps1 的 Phase 0 加入清理邏輯：

```powershell

# 清理超過 3 天的 loop-state 檔案

Get-ChildItem state/loop-state-*.json |

Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-3) } |

Remove-Item -Force

```

  1. 在 check-health.ps1 新增 loop-state 檔案數量監控（警戒閾值：200）
  2. 考慮將 LoopDetector 改為單一檔案 + session key 索引，避免檔案碎片化

預期效益：state/ 檔案數從 282 降至 ~20，減少磁碟 I/O 和目錄掃描時間。

P0-B：Context 檔案膨脹治理

問題：context/behavior-patterns.json（236KB）持續成長，無滾動清理機制。context/improvement-backlog.json（25KB）和 context/research-series.json（26KB）同樣缺少大小控制。

建議方案：

  1. behavior-patterns.json：實施 90 天滾動窗口，超齡 entry 歸檔至 context/archive/behavior-YYYY-MM.json
  2. improvement-backlog.json：已完成的 backlog item 移至 done 陣列，定期清理已完成超過 30 天的項目
  3. 在 config/benchmark.yaml 新增 context 膨脹閾值：

```yaml

context_size_limits:

behavior_patterns_kb: 300

improvement_backlog_kb: 50

research_series_kb: 50

```

預期效益：context/ 總大小控制在 500KB 以下，避免 LLM 讀取時的 token 浪費。

P1-A：Token 經濟性優化（Tiered Model Strategy）

問題：2026-03-04 出現 3.75M token 峰值（10.8M 字元輸出），超出 1.5M 警戒線 2.5 倍。部分自動任務（如研究類）消耗大量 token 但產出可用小型模型處理。

建議方案：基於 config/llm-router.yaml 擴展分層模型策略

  1. 分級任務 Token 預算：

- 研究類任務（research-task, tech-research）：上限 200K tokens/次

- 系統維護類（self-heal, system-insight）：上限 100K tokens/次

- 簡單操作類（git-push, log-audit）：上限 50K tokens/次

  1. Prompt 壓縮：共用前言 templates/shared/preamble.md 考慮條件載入，非研究任務省略 KB 策略分析段落
  2. Token 追蹤增強：state/token-usage.json 新增 by_task_type 細分統計，支援成本歸因

預期效益：日均 token 消耗降低 30-40%，峰值控制在 2M 以內。

P1-B：自適應回饋迴圈（向 Level 4 邁進）

問題：系統已有 OODA 閉環架構（ADR-008），但缺乏基於執行結果的自動策略調整。例如：

  • 研究任務反覆產出低品質筆記（quality_score ≤ 2）但未自動調整搜尋策略
  • 自動任務某群組連續失敗但輪轉策略不變
  • Token 消耗超標但未觸發降級機制

建議方案：實施三個自適應回饋機制

  1. 品質驅動的任務調整：

- 在 results/ 寫入後，統計近 7 天同 task_type 的平均 quality_score

- 若平均 < 3.0 → 自動在下次執行時加載替代搜尋策略（如更換搜尋引擎、調整 topK）

  1. 失敗感知的輪轉調整：

- frequency-limits.yaml 新增 adaptive_cooldown 欄位

- 連續失敗 3 次的任務自動延長冷卻時間（1h → 2h → 4h）

  1. Token 預算自動降級：

- 當日 token 消耗達警戒線 80% 時，後續任務自動切換至 Groq（快速推理模型）

- 需擴展 config/llm-router.yaml 支援動態路由

預期效益：系統從「按規則執行」進化為「按結果學習」，Agentic 成熟度提升至 Level 3.75。

P2-A：腳本生態整合與 DX 改善

問題：根目錄有 34 個 .ps1 腳本，認知負擔日增。5 個 kb- 腳本功能重疊，3 個 create--tasks.ps1 可合併。

建議方案：

  1. KB 工具整合：5 個 kb-*.ps1 合併為 kb-manager.ps1 -Action backup|export|verify|git-backup|backup-all
  2. 任務建立工具整合：3 個 create-*-tasks.ps1 合併為 create-tasks.ps1 -Type audit|github|research
  3. CLI 入口統一：建立 ddp.ps1 作為統一入口（類似 git 子命令模式）：

```powershell

./ddp.ps1 run digest # = run-agent-team.ps1

./ddp.ps1 run todoist # = run-todoist-agent-team.ps1

./ddp.ps1 check health # = check-health.ps1

./ddp.ps1 kb backup # = kb-manager.ps1 -Action backup

```

預期效益：根目錄腳本從 34 個減至 ~20 個，新使用者上手時間降低 50%。

P2-B：可觀測性儀表板化

問題：系統已有完善的 JSONL 日誌、健康檢查、Token 追蹤，但全部透過 CLI 查詢，缺乏視覺化儀表板。結合 KB 中已有的 Langfuse（LLM 觀測平台）和 n8n（工作流自動化）研究，可建構更強的觀測能力。

建議方案：

  1. 輕量級 HTML 儀表板：利用 check-health.ps1 輸出產生 output/dashboard.html

- 排程健康度熱力圖（近 7 天 × 24 小時）

- Token 消耗趨勢圖（日均 / 峰值 / 預算線）

- Skill 使用頻率分布圖

- 研究去重覆蓋度視覺化

  1. Langfuse 整合評估：參考 KB 中 Langfuse 研究筆記，評估是否值得引入 OpenTelemetry 追蹤
  2. ntfy 告警分級：基於 config/notification.yaml 細化告警等級

- Critical：API 斷路器觸發、系統崩潰

- Warning：Token 超標、連續失敗

- Info：任務完成、daily digest 發送

預期效益：營運可見性從 CLI-only 提升至視覺化，問題發現時間從分鐘級降至秒級。

四、Phase 3 演進路線圖

階段 3.1：穩固基礎（1-2 週）

  • P0-A：清理 264 個 loop-state 檔案 + 自動清理機制
  • P0-B：context 檔案滾動清理 + 大小監控
  • 修復 Token 追蹤精度（by_task_type 細分）

階段 3.2：智慧化升級（2-4 週）

  • P1-A：分層 Token 預算 + 條件式 Prompt 載入
  • P1-B：三個自適應回饋機制實施
  • 評估 Groq 動態路由的可行性

階段 3.3：體驗優化（4-6 週）

  • P2-A：腳本整合 + CLI 統一入口
  • P2-B：HTML 儀表板 + 告警分級
  • 評估 Langfuse 整合 ROI

成熟度目標

  • 階段 3.1 完成後：Level 3.5（資源治理成熟）
  • 階段 3.2 完成後：Level 3.75（自適應初步實現）
  • 階段 3.3 完成後：Level 4.0（全面自適應 + 視覺化觀測）

五、風險與限制

| 風險 | 等級 | 緩解措施 |

|------|------|----------|

| Token 預算機制可能過度限制深度研究品質 | 中 | 設立研究類任務的例外閾值 |

| 自適應策略可能導致系統行為不可預測 | 中 | 所有自動調整均記錄至 JSONL 日誌 |

| 腳本整合可能影響現有排程 | 低 | 保持原腳本路徑不變，新入口僅作轉發 |

| Loop-state 清理可能刪除進行中的 session 狀態 | 低 | 僅清理超過 3 天的檔案 |
