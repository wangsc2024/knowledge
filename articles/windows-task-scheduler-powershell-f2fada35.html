Windows Task Scheduler 進階自動化完整指南

從 PowerShell ScheduledTasks 模組、CIM 事件觸發、XML 任務定義到生產級監控診斷的最佳實踐

技術概述

Windows Task Scheduler 是 Windows 作業系統內建的任務排程引擎，支援時間型觸發（daily/weekly/monthly）、事件型觸發（Event Log、系統啟動、使用者登入）及條件型執行（閒置、電源、網路）。PowerShell 7 的 ScheduledTasks 模組提供 19 個 Cmdlet，涵蓋任務的完整生命週期（建立、註冊、監控、匯出、刪除）。對於進階場景（如 Event Trigger），需搭配 CIM 類別或 XML 任務定義檔突破 Cmdlet 限制。在生產環境中，搭配診斷腳本與事件日誌分析可建立完整的排程健康監控體系。

一、ScheduledTasks 模組核心 Cmdlet

1.1 完整 Cmdlet 清單（19 個）

| Cmdlet | 功能 |

|--------|------|

| Register-ScheduledTask | 註冊新任務 |

| Unregister-ScheduledTask | 移除任務 |

| Set-ScheduledTask | 修改既有任務 |

| Get-ScheduledTask | 查詢任務定義 |

| Get-ScheduledTaskInfo | 取得執行時資訊（上次結果、下次執行時間） |

| Enable-ScheduledTask / Disable-ScheduledTask | 啟用/停用任務 |

| Start-ScheduledTask / Stop-ScheduledTask | 手動啟動/停止任務 |

| Export-ScheduledTask | 匯出 XML 定義字串 |

| New-ScheduledTask | 建立任務實例（未註冊） |

| New-ScheduledTaskAction | 定義執行動作 |

| New-ScheduledTaskTrigger | 定義觸發器 |

| New-ScheduledTaskSettingsSet | 定義任務設定 |

| New-ScheduledTaskPrincipal | 定義安全主體 |

| Register-ClusteredScheduledTask | 叢集排程註冊 |

| Set-ClusteredScheduledTask | 叢集排程修改 |

| Get-ClusteredScheduledTask | 叢集排程查詢 |

| Unregister-ClusteredScheduledTask | 叢集排程移除 |

1.2 標準任務建立模式


# 四步建立法：Action → Trigger → Settings → Register
$action = New-ScheduledTaskAction -Execute 'pwsh.exe' `
  -Argument '-ExecutionPolicy Bypass -File C:\Scripts\daily-task.ps1'

$trigger = New-ScheduledTaskTrigger -Daily -At '08:00AM'

$principal = New-ScheduledTaskPrincipal -UserId 'NT AUTHORITY\SYSTEM' `
  -LogonType ServiceAccount

$settings = New-ScheduledTaskSettingsSet `
  -AllowStartIfOnBatteries `
  -DontStopIfGoingOnBatteries `
  -StartWhenAvailable `
  -RestartCount 3 `
  -RestartInterval (New-TimeSpan -Minutes 1)

Register-ScheduledTask -TaskName 'DailyDigest' `
  -TaskPath '\ClaudeAgent\' `
  -Action $action -Trigger $trigger `
  -Principal $principal -Settings $settings

1.3 觸發器類型速查

| 觸發類型 | Cmdlet 參數 | 備註 |

|----------|------------|------|

| 每日 | -Daily -At '08:00' | 可加 -DaysInterval 2 隔日 |

| 每週 | -Weekly -DaysOfWeek Monday,Wednesday -At '09:00' | 支援多天 |

| 系統啟動 | -AtStartup | 開機後執行 |

| 使用者登入 | -AtLogon | 可指定 -User |

| 一次性 | -Once -At '2026-03-01 10:00' | 搭配 -RepetitionInterval 可重複 |

| 事件觸發 | 不支援 — 需用 CIM 類別 | 見下節 |

二、CIM 類別突破 — 事件觸發器

2.1 問題

New-ScheduledTaskTrigger 不支援 -OnEvent 參數，無法直接建立事件觸發任務。

2.2 CIM 解法


# 取得 CIM 觸發器類別
$CIMTriggerClass = Get-CimClass `
  -ClassName MSFT_TaskEventTrigger `
  -Namespace Root/Microsoft/Windows/TaskScheduler

# 建立觸發器實例（ClientOnly = 不提交到 WMI）
$trigger = $CIMTriggerClass | New-CimInstance -ClientOnly
$trigger.Enabled = $true

# 設定事件訂閱（XPath 篩選）
$trigger.Subscription = @'
<QueryList>
  <Query Id="0" Path="Application">
    <Select Path="Application">
      *[System[Provider[@Name='MyApp'] and EventID=1001]]
    </Select>
  </Query>
</QueryList>
'@

# 組合並註冊任務
$action = New-ScheduledTaskAction -Execute 'pwsh.exe' `
  -Argument '-File C:\Scripts\event-handler.ps1'
$principal = New-ScheduledTaskPrincipal -UserId 'NT AUTHORITY\SYSTEM' `
  -LogonType ServiceAccount
$settings = New-ScheduledTaskSettingsSet

Register-ScheduledTask -TaskName 'EventResponder' `
  -TaskPath '\EventTasks\' `
  -Action $action -Principal $principal `
  -Settings $settings -Trigger $trigger

2.3 事件訂閱 XPath 語法

| 場景 | XPath 範例 |

|------|----------|

| 特定 EventID | *[System[EventID=1001]] |

| 特定來源 | *[System[Provider[@Name='MyProvider']]] |

| 組合條件 | *[System[Provider[@Name='MyApp'] and EventID=1001 and Level=2]] |

| TaskScheduler 自身 | *[System[Provider[@Name='Microsoft-Windows-TaskScheduler'] and EventID=101]] |

三、XML 任務定義

3.1 XML Schema 結構


<?xml version="1.0" encoding="UTF-16"?>
<Task xmlns="http://schemas.microsoft.com/windows/2004/02/mit/task">
  <RegistrationInfo>
    <Date>2026-02-27T08:00:00+08:00</Date>
    <Author>DailyDigest</Author>
    <Description>每日摘要排程</Description>
  </RegistrationInfo>
  <Triggers>
    <CalendarTrigger>
      <StartBoundary>2026-02-27T08:00:00+08:00</StartBoundary>
      <ScheduleByDay><DaysInterval>1</DaysInterval></ScheduleByDay>
    </CalendarTrigger>
  </Triggers>
  <Principals>
    <Principal id="Author">
      <UserId>S-1-5-18</UserId>
      <RunLevel>HighestAvailable</RunLevel>
    </Principal>
  </Principals>
  <Settings>
    <MultipleInstancesPolicy>IgnoreNew</MultipleInstancesPolicy>
    <DisallowStartIfOnBatteries>false</DisallowStartIfOnBatteries>
    <StopIfGoingOnBatteries>false</StopIfGoingOnBatteries>
    <StartWhenAvailable>true</StartWhenAvailable>
    <RunOnlyIfNetworkAvailable>false</RunOnlyIfNetworkAvailable>
    <ExecutionTimeLimit>PT1H</ExecutionTimeLimit>
    <RestartOnFailure>
      <Interval>PT1M</Interval>
      <Count>3</Count>
    </RestartOnFailure>
  </Settings>
  <Actions Context="Author">
    <Exec>
      <Command>pwsh.exe</Command>
      <Arguments>-ExecutionPolicy Bypass -File C:\Scripts\task.ps1</Arguments>
      <WorkingDirectory>C:\Scripts</WorkingDirectory>
    </Exec>
  </Actions>
</Task>

3.2 匯出/匯入流程


# 匯出既有任務為 XML
Export-ScheduledTask -TaskName 'DailyDigest' | Out-File task-def.xml

# 從 XML 匯入（PowerShell）
Register-ScheduledTask -TaskName 'DailyDigest-Clone' `
  -Xml (Get-Content task-def.xml -Raw)

# 從 XML 匯入（schtasks 命令列）
schtasks /create /tn "DailyDigest-Clone" /xml "task-def.xml" /ru SYSTEM

3.3 XML vs PowerShell Cmdlet 比較

| 維度 | XML 定義 | PowerShell Cmdlet |

|------|---------|------------------|

| 事件觸發 | 完全支援 | 不支援（需 CIM） |

| 跨機器部署 | 匯出匯入即可 | 需重寫腳本 |

| 版本控制 | Git 友善 | 需另存腳本 |

| 可讀性 | 冗長但完整 | 簡潔但隱藏設定 |

| 進階設定 | 全部暴露 | 部分不支援 |

四、生產級設定最佳實踐

4.1 失敗重試配置


$settings = New-ScheduledTaskSettingsSet `
  -RestartCount 3 `                    # 最多重試 3 次
  -RestartInterval (New-TimeSpan -Minutes 2)  # 每次間隔 2 分鐘
  -ExecutionTimeLimit (New-TimeSpan -Hours 1) # 最長執行 1 小時
  -StartWhenAvailable                  # 錯過排程時補執行
  -MultipleInstances IgnoreNew         # 防止並行執行

4.2 安全性最佳實踐

| 實踐 | 說明 |

|------|------|

| 最小權限原則 | 使用 Managed Service Account (gMSA) 而非 SYSTEM |

| 腳本簽章 | 生產環境用 AllSigned 執行策略 |

| 路徑保護 | 腳本目錄限制 ACL，僅允許 SYSTEM 和管理員 |

| 避免密碼暴露 | 用 -LogonType ServiceAccount 或 gMSA 免密碼 |

| 審計追蹤 | 啟用 TaskScheduler Operational Log |

4.3 常見陷阱

| 陷阱 | 症狀 | 解法 |

|------|------|------|

| -ExecutionPolicy Bypass 濫用 | 安全風險 | 改用 RemoteSigned + 腳本簽章 |

| 忘記 -WorkingDirectory | 相對路徑失效 | Action 加 WorkingDirectory |

| 密碼過期 | 任務靜默失敗 | 改用 gMSA 或 SYSTEM |

| StopIfGoingOnBatteries 預設開啟 | 筆電斷電停止任務 | 明確設為 $false |

| ExecutionTimeLimit 預設 72 小時 | 長任務被殺 | 設為 PT0S 取消限制或合理值 |

| 互動式登入依賴 | 無人登入時不執行 | 改用 ServiceAccount LogonType |

5.1 事件日誌監控

TaskScheduler 事件記錄在：

Applications and Services Logs > Microsoft > Windows > TaskScheduler > Operational

| EventID | 意義 |

|---------|------|

| 100 | 任務啟動 |

| 101 | 任務啟動失敗 |

| 102 | 任務完成 |

| 103 | 動作啟動失敗 |

| 106 | 任務註冊（使用者或更新） |

| 110 | 任務觸發 |

| 111 | 任務終止 |

| 129 | 建立程序失敗 |

| 201 | 動作完成 |

| 202 | 動作完成（失敗） |

| 413 | 服務啟動時載入任務失敗 |

5.2 PowerShell 診斷腳本


# 取得所有排程任務健康摘要
function Get-TaskSchedulerHealth {
    $tasks = Get-ScheduledTask | Where-Object { $_.State -ne 'Disabled' }
    $results = foreach ($task in $tasks) {
        $info = Get-ScheduledTaskInfo -TaskName $task.TaskName -TaskPath $task.TaskPath
        [PSCustomObject]@{
            Name         = $task.TaskName
            Path         = $task.TaskPath
            State        = $task.State
            LastResult   = '0x{0:X}' -f $info.LastTaskResult
            LastRun      = $info.LastRunTime
            NextRun      = $info.NextRunTime
            MissedRuns   = $info.NumberOfMissedRuns
            IsHealthy    = ($info.LastTaskResult -eq 0)
        }
    }
    $results | Sort-Object IsHealthy, LastRun
}

# 偵測失敗任務
$failed = Get-TaskSchedulerHealth | Where-Object { -not $_.IsHealthy }
if ($failed) {
    Write-Warning "以下 $($failed.Count) 個任務執行失敗："
    $failed | Format-Table Name, LastResult, LastRun -AutoSize
}

5.3 JAMS 診斷工具功能

生產環境的進階診斷工具（如 JAMS）提供：

  • Exit Code 分析：將 30+ 種失敗碼對應到根因（憑證損壞、並行衝突、存取拒絕、逾時）
  • 可靠性指標：成功率、連續失敗次數、錯過執行追蹤
  • 配置風險偵測：停用重試、無限逾時、互動登入依賴、Queue New Instance 策略
  • 效能追蹤：平均/最大執行時間、高頻排程偵測、超時任務識別
  • 風險等級：Critical/High/Medium/Low/Minimal 五級
  • 報告格式：CSV + 互動式 HTML Dashboard

5.4 自建監控整合


# 整合 ntfy 告警的排程監控
function Watch-ScheduledTasks {
    param([string]$NtfyTopic = 'my-alerts')
    
    $failed = Get-ScheduledTask | ForEach-Object {
        $info = Get-ScheduledTaskInfo -TaskName $_.TaskName -TaskPath $_.TaskPath -ErrorAction SilentlyContinue
        if ($info -and $info.LastTaskResult -ne 0) {
            [PSCustomObject]@{
                Name = $_.TaskName
                Result = '0x{0:X}' -f $info.LastTaskResult
                LastRun = $info.LastRunTime
            }
        }
    }
    
    if ($failed) {
        $body = ($failed | ForEach-Object { "$($_.Name): $($_.Result)" }) -join "`n"
        Invoke-RestMethod -Uri "https://ntfy.sh/$NtfyTopic" `
          -Method Post -Body $body `
          -Headers @{ Title = 'Task Scheduler Alert'; Priority = '4' }
    }
}

六、與 daily-digest-prompt 專案的關聯

6.1 目前架構

本專案使用 setup-scheduler.ps1 從 HEARTBEAT.md 批次建立 6 個排程任務，每日合計 44 次執行：

  • 系統審查（00:40）、每日摘要（08:00/11:15/21:15）
  • Todoist 單一模式（每小時整點 02-23）
  • Todoist 團隊模式（每小時半點 02-23）

6.2 可改進方向

| 面向 | 目前做法 | 進階做法 |

|------|---------|----------|

| 任務建立 | schtasks 命令列 | 改用 Register-ScheduledTask + XML 定義檔（版本控制友善） |

| 失敗重試 | PS 腳本內 retry | 加上 -RestartCount 3 -RestartInterval PT2M（OS 層級重試） |

| 健康監控 | check-health.ps1 讀取日誌 | 加入 Get-ScheduledTaskInfo 的 NumberOfMissedRuns 檢查 |

| 事件觸發 | 無 | 可建立事件觸發任務（如 Agent 失敗時自動啟動 self-heal） |

| 部署 | setup-scheduler.ps1 命令式 | 匯出 XML 後用 Register-ScheduledTask -Xml 宣告式部署 |

| 並行控制 | PS 腳本內鎖定 | 加上 -MultipleInstances IgnoreNew 防止重複執行 |
