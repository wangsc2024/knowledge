一、技術概述

AI Agent 開發已從單一模型的 prompt-response 範式演進為多 Agent 協作的分散式系統架構。2026 年，Gartner 預測 40% 企業應用將具備任務專精 AI Agent（2025 年僅 5%）。核心技術支柱包含：(1) Multi-Agent 系統（MAS）設計模式，將複雜任務拆解為專責 Agent 協作；(2) Model Context Protocol（MCP）作為 Agent-工具整合的開放標準，類似 AI 領域的「容器」基礎設施；(3) 生產級 Agent 架構的七大元件（感知、推理、記憶、工具、編排、RAG、部署）。業界從「能不能做」轉向「如何可靠地做」——在 5% 失敗率下，20 步驟的 Agent 工作流將不可接受地失敗，生產環境要求端對端失敗率低於 1%。

二、核心方法與工具

2.1 八大 Multi-Agent 設計模式

| # | 模式 | 適用場景 | 核心機制 |

|---|------|---------|----------|

| 1 | Sequential Pipeline | 文件處理、資料管線 | 線性串接，每個 Agent 的輸出是下一個的輸入 |

| 2 | Coordinator/Dispatcher | 客服路由、意圖分派 | 中央 Agent 依意圖分析分派至專責 Agent |

| 3 | Parallel Fan-Out/Gather | 程式碼審查、多維分析 | 多 Agent 並行處理 → 合成器整合結果，減少 60-80% 時間 |

| 4 | Hierarchical Decomposition | 報告撰寫、大型任務 | 父 Agent 將複雜任務拆解為子任務，層層委派 |

| 5 | Generator/Critic | SQL 生成、程式碼品質 | 生成 Agent + 驗證 Agent 形成品質迴圈 |

| 6 | Iterative Refinement | 效能最佳化、文案打磨 | 多次修訂循環，支援硬性迭代上限或 Agent 主動退出 |

| 7 | Human-in-the-Loop | 金融交易、生產部署 | Agent 處理基礎工作，高風險動作暫停等待人工授權 |

| 8 | Composite Pattern | 實際生產系統 | 組合以上模式：Coordinator → 並行搜尋 → Generator/Critic 迴圈 |

2.2 推理引擎模式比較

| 模式 | 優勢 | 劣勢 | 最佳場景 |

|------|------|------|----------|

| ReAct | 動態適應、即時調整 | Token 消耗高、延遲不穩、成本難預測 | 探索性任務、動態環境 |

| Plan-and-Execute | 執行快、成本可預測、重規劃少 | 適應性差、初始計畫錯誤難補救 | 穩定環境、分析型工作流 |

| Multi-Agent | 吞吐量高、故障隔離、專業化 | 需分散式系統設計能力、通訊開銷 | 跨領域複雜任務 |

2.3 MCP（Model Context Protocol）工具整合標準

MCP 由 Anthropic 於 2024 年 11 月推出的開放標準，定義 LLM 與外部工具/資料源的統一通訊協定：

  • 三大原語：Resource（資料源）、Tool（可執行功能）、Prompt（提示模板）
  • 廠商中立：OpenAI、Google DeepMind、Microsoft 均已採用
  • 2026 進化：MCP Apps 成為互動介面標準；MCP Server 可作為自主 Agent（如「旅遊 Agent」自主與「訂房 Agent」協商）
  • 效率提升：按需載入工具、資料預過濾後才進入模型、單步驟執行複雜邏輯
  • 與 FastMCP 關係：FastMCP 提供 70% MCP 伺服器的 Pythonic 開發體驗

三、實際工作流程（Step-by-Step）

3.1 Multi-Agent 系統設計工作流


Step 1: 定義問題邊界與分工
  → 識別可拆分的子任務域（如：搜尋、分析、生成、驗證）
  → 確定每個 Agent 的角色、目標與可用工具

Step 2: 選擇架構模式
  → 單步驟明確任務 → 直接 LLM 呼叫
  → 多步驟有明確結構 → Plan-and-Execute
  → 動態探索性任務 → ReAct
  → 跨域複雜任務 → Multi-Agent

Step 3: 實作 Agent 通訊與狀態管理
  → 使用 session.state 作為共享白板
  → 每個 Agent 寫入唯一 key（防止競態條件）
  → 實作 output_key 傳遞鏈

Step 4: 建立品質閘門與安全控制
  → Generator/Critic 迴圈（品質閘門通過才放行）
  → Human-in-the-Loop 用於不可逆操作
  → 實作重試邏輯與錯誤降級

Step 5: 可觀測性與監控
  → 追蹤每個 Agent 的決策路徑與原因
  → 完整審計軌跡（法規合規 + 除錯用）
  → 行為可觀測性（不僅是指標，而是理解 Agent 為何如此決策）

Step 6: 漸進式複雜化
  → 先用 Sequential 串接 → 驗證 → 逐步加入並行與 Critic

3.2 MCP Tool 整合工作流


Step 1: 設計 Tool Schema
  → 精確的語意描述（直接影響路由準確度）
  → 跨組織標準化（Amazon 經驗：統一 schema 治理框架）
  → 包含輸入驗證、預期輸出格式

Step 2: 實作 MCP Server
  → 使用 FastMCP 框架（Python）或官方 TypeScript SDK
  → 定義 Resource、Tool、Prompt 三大原語
  → 實作錯誤處理與重試邏輯

Step 3: 連接 Agent 與 MCP Server
  → 配置 .mcp.json 指向 Server 端點
  → 按需載入工具（節省 context window）
  → 實作認證與權限控制

Step 4: 測試與驗證
  → Tool 單元測試（schema 驗證 + 功能驗證）
  → 整合測試（Agent 能正確選擇 + 使用 Tool）
  → 壓力測試（並發呼叫 + 故障注入）

四、程式碼/配置範例

4.1 Google ADK — Parallel Fan-Out/Gather 模式


from google.adk.agents import ParallelAgent, LlmAgent

# 定義專責 Agent
security_reviewer = LlmAgent(
    name="security_reviewer",
    instruction="Review code for security vulnerabilities",
    output_key="security_findings"
)

style_reviewer = LlmAgent(
    name="style_reviewer",
    instruction="Review code for style and readability",
    output_key="style_findings"
)

performance_reviewer = LlmAgent(
    name="performance_reviewer",
    instruction="Review code for performance issues",
    output_key="performance_findings"
)

# 並行執行
code_review_parallel = ParallelAgent(
    name="code_review",
    sub_agents=[security_reviewer, style_reviewer, performance_reviewer]
)

# 合成器：讀取所有 findings
synthesizer = LlmAgent(
    name="synthesizer",
    instruction="""Read {security_findings}, {style_findings}, 
    {performance_findings} from state and create unified report."""
)

4.2 Generator/Critic 品質迴圈（Google ADK）


from google.adk.agents import LoopAgent, LlmAgent

generator = LlmAgent(
    name="sql_generator",
    instruction="Generate SQL query based on user request",
    output_key="generated_sql"
)

critic = LlmAgent(
    name="sql_validator",
    instruction="""Validate {generated_sql}. 
    If valid: set escalate=True to exit loop.
    If invalid: provide feedback for regeneration."""
)

sql_pipeline = LoopAgent(
    name="sql_quality_loop",
    sub_agents=[generator, critic],
    max_iterations=5  # 硬性上限防止無限迴圈
)

六、與 Daily-Digest-Prompt 專案的應用場景

6.1 現有架構已採用的模式

| 專案模式 | 對應設計模式 | 實例 |

|---------|-------------|------|

| 每日摘要 Phase 1→2 | Parallel Fan-Out/Gather | 5 路並行擷取 → 組裝 Agent 合成 |

| Todoist 3 階段 | Sequential + Parallel + Coordinator | 查詢→N路並行→組裝通知 |

| 系統審查 | Parallel + Hierarchical | 4 路並行審查→組裝修正報告 |

| 子 Agent 模板路由 | Coordinator/Dispatcher | routing.yaml 三層路由分派 |

| 品質閘門 | Generator/Critic | quality-gate.md + refinement.md 迴圈 |

6.2 可改進的方向

  1. 語意快取整合：現有 file-based cache 可引入向量相似度比對（Redis LangCache 模式），對相似查詢直接回傳快取結果，減少 API Token 消耗
  2. MCP Server 化：將知識庫 API（localhost:3000）封裝為 MCP Server，讓所有 Agent 透過標準協定存取，取代 curl 硬編碼
  3. 行為可觀測性增強：post_tool_logger.py 已追蹤工具呼叫，可增加「決策原因」欄位（為何選擇此工具/此 Agent），提升除錯能力
  4. Circuit Breaker 進階化：現有 api-health.json 5 個 API 的斷路器，可借鏡 Generator/Critic 模式加入「自我修復驗證迴圈」

七、風險與注意事項

7.1 技術風險

  • Token 成本爆增：Multi-Agent 系統消耗 15× 以上 Token，必須實作用量監控與成本閾值告警
  • 競態條件：並行 Agent 共享狀態時，務必確保每個 Agent 寫入唯一 key（Google ADK 明確警示）
  • 級聯失敗：工具呼叫失敗可能級聯導致整個 Agent 工作流失敗，需實作錯誤隔離與降級策略
  • 無限迴圈：Generator/Critic 模式與 ReAct 模式需設定硬性迭代上限（max_iterations）

7.2 治理風險

  • Tool Schema 治理：Amazon 經驗指出，模糊的 Tool 語意描述會導致 Agent 在運行時選錯工具，需跨組織標準化
  • 認證管理：Agent 存取第三方服務的憑據管理常被低估（Redis Blog 特別強調應從第一天開始設計）
  • 審計合規：生產級系統需完整決策審計軌跡，不能僅依賴日誌

7.3 設計原則

  • 從簡單開始：先用 Sequential 串接驗證，再逐步加入並行與 Critic 迴圈
  • 3-7 Agent 規則：少於 3 個代表可用單一 Agent，超過 7 個協調開銷超過收益
  • Human-on-the-Loop：不可逆操作必須有人工審核閘門
