AI 驅動的需求工程（AI-Powered Requirements Engineering）

技術概述

2025-2026 年，AI 在需求工程（Requirements Engineering, RE）領域正經歷從「輔助工具」到「自動化管線」的轉型。根據 Wiley 期刊 2026 年的系統性文獻綜述，238 篇論文（2019-2025）的分析顯示：生成式預訓練模型佔應用的 67.3%，其中需求分析（30.0%）和需求引出（22.1%）是研究最集中的領域。然而，90% 以上的研究仍處於早期開發階段，僅 1.3% 達到生產級整合。2026 年的核心突破在於規格驅動開發（Spec-Driven Development, SDD）的興起 — 將規格文件從「附帶產物」提升為「唯一真相來源（Single Source of Truth）」，搭配 AI Agent 自動執行需求→設計→任務→實作的完整流程。

---

核心方法與工具

1. 規格驅動開發（Spec-Driven Development, SDD）

由 AWS Kiro（2025 年中發布）首創並由 Thoughtworks 推廣的三文件規格結構，2026 年已成為 AI 輔助開發的主流工作流：

三文件結構：

  • requirements.md：使用 EARS 表記法（Easy Approach to Requirements Syntax）撰寫需求
  • design.md：系統設計文件，含序列圖與架構決策
  • tasks.md：可執行的任務清單，含依賴關係與驗收條件

EARS 表記法範例：


# 格式：When [trigger], the [system] shall [action]

When the user submits a login form with valid credentials,
the authentication service shall return a JWT token within 2 seconds.

When the API rate limit is exceeded,
the gateway shall return HTTP 429 with a Retry-After header.

SDD 核心原則：

  • 規格是真相來源，不是程式碼
  • 團隊從「vibe coding」（即興提示）升級為結構化、可測試的規格
  • 重要決策在實作前完成，實作階段中斷次數大幅降低

2. Spec-Driven LLM Development（SDLD）

David Lapsley 於 2026 年 1 月提出的 SDLD 框架，將 SDD 與 Agentic 自動化結合：

四階段流程：

  1. Specify：撰寫範圍、意圖、約束的結構化規格
  2. Plan：AI 工具從規格推導實作步驟
  3. Execute：使用 Agent 編排框架安全執行任務
  4. Review & Test：對照規格驗證輸出，提交產物，迭代精修

3. GeneUS — 自動使用者故事生成工具

2024 年 IEEE 發表的 GeneUS 工具，使用 GPT-4.0 從需求文件自動生成使用者故事：

核心創新：採用 RaT（Refine and Thought）提示策略，透過精煉輸入上下文來降低幻覺

輸出格式（JSON）：


{
  "user_stories": [
    {
      "id": "US-001",
      "role": "系統管理員",
      "goal": "管理用戶角色與權限",
      "benefit": "確保資料安全和適當的存取控制",
      "acceptance_criteria": [
        "Given 管理員已登入, When 修改用戶角色, Then 權限立即生效",
        "Given 超過 3 次錯誤嘗試, When 用戶再次嘗試, Then 帳號鎖定 30 分鐘"
      ],
      "test_cases": [
        "驗證管理員可以成功指派角色",
        "驗證帳號鎖定機制在閾值超過後觸發"
      ]
    }
  ]
}

評估結果：50 位開發者給出中位數 4/5 分，使用者故事清晰度高，僅 5% 缺少技術細節，重複率僅 0.5%。

4. AutoUAT + Test Flow — 工業級驗收測試生成

2025 年 IEEE 工業案例研究，展示 LLM 驅動的驗收測試自動化兩步管線：

架構：

  • AutoUAT：從使用者故事生成 Gherkin 場景（整合 Microsoft Power Automate + Teams）
  • Test Flow：從 Gherkin 場景生成可執行的 Cypress 測試腳本（整合 JIRA + GitHub Actions）

技術細節：

  • 使用 GPT-4 Turbo（gpt-4-1106-preview），128K context window
  • 輸入提示平均 9,500 tokens，輸出平均 750 tokens
  • 每個測試腳本成本約 €0.12

評估結果：

  • AutoUAT：95% 正面評價（62/65 回應者認為有幫助）
  • Test Flow：100% TypeScript 語法正確，語意相關性從 60% 提升至 92%
  • 32% 需要小幅修正，8% 需要完全重新生成

實際工作流程（Step-by-Step）

流程 A：從零開始的 SDD 工作流（以 Claude Code 為例）


步驟 1：平行研究（Parallel Research）
  → 啟動多個子 Agent 同時調查技術領域
  → 合併發現到統一報告

步驟 2：規格創建（Specification Creation）
  → AI 生成完整技術規格文件
  → 涵蓋：架構分析、遷移計畫、實作檢查清單
  → 規格成為「唯一真相來源」

步驟 3：透過訪談精煉（Refinement via Interview）
  → 使用 AskUserQuestion 工具釐清歧義
  → 在實作前解決所有不確定性

步驟 4：任務委派與原子提交（Task Delegation + Atomic Commits）
  → 每個任務委派給獨立子 Agent
  → 每個任務 = 一個 Git commit（容易回滾）
  → 依賴感知的執行順序

實際案例（Alex Opalic, 2026）：

  • SQLite → IndexedDB 遷移
  • 14 個已完成任務 / 14 個 Git commits / 15+ 修改檔案
  • 耗時 ~45 分鐘 vs. 手工估計 2-3 天
  • Context 利用率 71%（143K/200K tokens）

流程 B：使用者故事 + 驗收條件自動化


步驟 1：需求文件收集
  → 會議紀錄、PRD、利害關係人訪談

步驟 2：AI 解析需求（NLP 分析）
  → 偵測缺陷：歧義性、不一致、複雜度、重複、遺漏、可測試性
  → 工具：ScopeMaster（NLP 引擎評估並交叉參照所有使用者故事）

步驟 3：自動生成使用者故事
  → 輸入：原始需求文件
  → 輸出：結構化 JSON（角色/目標/價值/AC/測試案例）
  → 策略：RaT 提示（精煉上下文→減少幻覺）

步驟 4：驗收條件生成
  → Given/When/Then 格式的 Gherkin 場景
  → AI 識別邊界案例和被忽略的情境

步驟 5：測試腳本生成
  → 從 Gherkin → Cypress/Playwright 可執行測試
  → CI/CD 整合（GitHub Actions）

步驟 6：人工審查 + 精修
  → AI 生成 ≠ 最終交付，人工審查不可省略

程式碼/配置範例

需求分析 Prompt 模板


# 角色
你是資深需求分析師，精通 EARS 表記法和 BDD。

# 任務
分析以下需求文件，產出：
1. 使用 EARS 表記法的結構化需求（每條需求含 ID、觸發條件、系統行為）
2. 使用者故事（As a [role], I want [goal], so that [benefit]）
3. 驗收條件（Given/When/Then 格式）
4. 識別的缺陷（歧義/不一致/遺漏/不可測試）

# 輸入
[貼上需求文件]

# 輸出格式
JSON 結構，含 requirements[]、user_stories[]、defects[]

Claude Code SDD 配置（specs/ 目錄結構）


# specs/<feature>/spec.md 結構
Feature: 用戶認證系統
Status: 草稿 | 審查中 | 凍結
Author: [姓名]
Date: 2026-02-23

## 功能需求
REQ-001: When the user submits valid credentials,
         the auth service shall issue a JWT with 24h expiry.

## 非功能需求
NFR-001: Authentication latency shall not exceed 200ms at P95.

## 驗收條件
AC-001: Given valid credentials, When login, Then JWT issued
AC-002: Given expired JWT, When API call, Then 401 returned

## 開放問題
- [ ] OAuth 2.0 or custom JWT?
- [ ] Rate limiting strategy?

自動化管線配置（GitHub Actions）


# .github/workflows/spec-review.yml
name: AI Spec Review
on:
  pull_request:
    paths: ['specs/**/*.md']

jobs:
  review-specs:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: AI Requirements Analysis
        run: |
          claude -p "分析 specs/ 中修改的規格文件，
          檢查：EARS 格式正確性、AC 完整性、
          需求間一致性、可測試性。
          輸出 JSON 報告。" \
          --allowedTools Read,Write \
          --output-format json > review.json
      - name: Post Review
        uses: actions/github-script@v7
        with:
          script: |
            const review = require('./review.json')
            // Post findings as PR comments

與本專案的應用場景（daily-digest-prompt 如何受益）

1. 強化現有「規格驅動 + TDD」開發規則

本專案已有 specs// 目錄結構和凍結機制。可進一步引入：

  • EARS 表記法：將 spec.md 中的需求統一為 When [trigger], the [system] shall [action] 格式
  • AI 自動生成 AC：從 spec.md 自動推導 tasks.md 中每個任務的驗收條件
  • 一致性驗證：用 AI 自動交叉檢查 spec ↔ plan ↔ tasks 的一致性

2. 自動任務品質提升

18 個自動任務目前由 prompt 模板驅動。可用 AI 需求分析技術：

  • 對研究模板輸出做「缺陷偵測」（檢查歧義性、可驗證性）
  • 自動從品質閘門（quality-gate.md）生成具體的 Gherkin 驗收場景

3. Todoist 路由的 SDD 化

將 config/routing.yaml 的三層路由規則視為需求規格：

  • AI 自動偵測路由規則間的衝突或遺漏
  • 從標籤映射自動生成測試案例，驗證路由行為

4. Hook 規則規格化

config/hook-rules.yaml 的 20 條規則可轉換為 EARS 格式需求，便於：

  • AI 自動偵測規則重疊或衝突
  • 自動生成覆蓋測試案例（目前 496 個測試的維護效率提升）

風險與注意事項

三大核心挑戰（來自 238 篇論文的系統性分析）

  1. 可重現性（66.8%）：AI 生成的需求因隨機性而每次可能不同，需版本化管理
  2. 幻覺（63.4%）：AI 可能捏造不存在的需求或約束，人工審查不可省略
  3. 可解釋性（57.1%）：AI 為何產出特定需求，難以向利害關係人解釋

實務注意

  • AI 生成 ≠ 最終交付：所有 AI 產出必須經人工審查
  • 上下文品質決定輸出品質：使用者故事清晰度直接影響測試生成品質
  • 成本考量：GPT-4 Turbo 每個測試腳本約 €0.12，大規模使用需預算規劃
  • 資料安全：使用公司託管模型（非公共 API）處理敏感需求
  • 避免過度自動化：需求引出（Elicitation）的人際互動面仍不可替代
