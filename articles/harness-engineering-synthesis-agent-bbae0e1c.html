一、synthesis 階段目標

將前三階段（foundation/application/optimization）的離散知識整合為一套可操作的統一治理框架，使 Harness Engineering 從「概念理解」走向「系統級實踐」。核心整合目標：

  1. 約束演化管線（application 階段成果）→ 作為治理框架的「規則生成引擎」
  2. 自癒機制（application 階段成果）→ 作為治理框架的「韌性保障層」
  3. 效能量測與 Fitness Function（optimization 階段成果）→ 作為治理框架的「品質閘門」
  4. 漂移偵測（optimization 階段成果）→ 作為治理框架的「早期預警系統」

---

二、統一治理框架：三維度模型（Context × Constraint × Convergence）

2.1 OpenAI 與 Epsilla 的三維度共識

Harness Engineering 在 2026 年已形成學科共識，核心框架為三個 C：

| 維度 | 定義 | 實作層 |

|------|------|--------|

| Context | Agent 的感知環境——宣告式與程序式知識 | CLAUDE.md、SKILL.md、config YAML、模板 |

| Constraint | 機器可執行的架構約束——產出前的指導與產出後的驗證 | Hook 守衛、Linter、結構測試、LLM-as-Judge |

| Convergence | 約束被評估→缺口被辨識→規則被精煉的迭代迴路 | OODA 閉環、constraint-evolve、system-insight |

三、統一治理框架的四層架構

Layer 1：規則生成引擎（約束演化管線）

輸入：結構化日誌（JSONL）、system-insight 告警、失敗軌跡

處理：

  • Phase A：日誌模式萃取（頻率×影響度×自動化難度計分）
  • Phase B：constraint-registry 分級提案（proposed → accepted）
  • Phase C：嵌入式規則落地（Hook 規則 / config YAML / 結構測試）

輸出：機器可執行約束（embedded 狀態）

關鍵原則：「不是告訴 Agent 寫好程式碼，而是機械式地強制定義好程式碼的樣子」（NxCode, 2026）。

Layer 2：韌性保障層（自癒機制）

四層自癒框架（由輕到重）：

  1. Tier 0 — 自動重試：冪等操作自動重試（指數退避）
  2. Tier 1 — 參數修正：自動修正已知參數錯誤（如 TTL 值超限→校正至閾值）
  3. Tier 2 — 流程降級：非關鍵路徑失敗時降級（如 KB 不可用→跳過知識庫寫入）
  4. Tier 3 — 人工介入：觸發告警（ntfy）通知人工審查

迴路偵測中介軟體：防止 Agent 在同一錯誤上重複嘗試（Doom Loop 偵測），超過 3 次相同錯誤即斷路。

Layer 3：品質閘門（Fitness Function + LLM-as-Judge）

三層評估指標：

| 層級 | 指標 | 閾值 |

|------|------|------|

| 檢索層 | precision@k、recall@k、NDCG | precision@5 ≥ 0.6 |

| 生成層 | faithfulness、schema validity、延遲 | faithfulness ≥ 0.8 |

| Agent 層 | 任務完成率、工具選擇準確率、拒絕率 | 成功率 ≥ 85% |

Fitness Function 自動校準：

  • 採用 30 天滑動窗口計算基準線
  • 偏差超過 ±2σ 觸發自動校準
  • 校準頻率：每週一次常規、異常事件觸發即時校準

Layer 4：早期預警系統（漂移偵測）

三類漂移偵測（Map–Measure–Manage 框架）：

| 漂移類型 | 偵測方法 | 回應策略 |

|---------|---------|----------|

| 行為漂移 | 任務完成率下降、推理一致性比對 | prompt 版本回滾 |

| 資料漂移 | 檢索品質指標（precision@k）衰退 | Embedding 重建 / chunk 策略調整 |

| 上游漂移 | API schema 驗證、工具回應格式檢查 | 適配器層自動調整 |

關鍵數據：研究顯示長時間運行的 Agent 中約 42% 會出現行為漂移，造成任務成功率下降（arxiv 2601.04170）。「1% 的排行榜差異無法偵測模型在五十步後偏離軌道的可靠性」（Phil Schmid, 2026）。

四、對 daily-digest-prompt 系統的具體實施路線圖

4.1 現況盤點（已實現的 Harness 組件）

本系統已在不自覺中實踐了多項 Harness Engineering 原則：

| 組件 | 現有實作 | 對應 Layer |

|------|---------|------------|

| Hook 守衛（6 個） | pre_bash_guard、pre_write_guard 等 | Layer 1 (Constraint) |

| constraint-evolve Skill | proposed→accepted→embedded 管線 | Layer 1 |

| self-heal Skill（4 Tier） | 漸進式自我修復 | Layer 2 |

| check-health.ps1 | 6 維度健康評分 | Layer 3 |

| system-insight | OODA 閉環分析 | Layer 4 |

| benchmark.yaml | 靜態效能基準 | Layer 3 |

4.2 缺口分析

| 缺口 | 當前狀態 | 目標狀態 |

|------|---------|----------|

| constraint_coverage | 0%（11 個 proposed 未落地） | ≥ 70% |

| 漂移偵測 | 無主動偵測 | 行為漂移自動告警 |

| Fitness Function 自動校準 | 手動基準 | 30 天滑動窗口自動校準 |

| LLM-as-Judge | 僅 DONE_CERT 自評 | 獨立品質門檻 |

| 日誌→約束 自動轉化 | 手動分析 | JSONL 自動萃取提案 |

4.3 三階段實施路線（30-60-90 天）

第一階段（Day 0-30）：偵測能力建立

  • 為 post_tool_logger.py 增加行為漂移指標（連續失敗計數、推理一致性分數）
  • 將 benchmark.yaml 的靜態閾值改為 30 天滑動窗口基準
  • 在 check-health.ps1 增加「漂移風險」維度（7→8 維度）

第二階段（Day 31-60）：自動化約束落地

  • 實作 JSONL→constraint 自動萃取工具（日誌模式分析→提案生成）
  • 將 11 個 proposed 約束逐步轉化為 embedded（Hook 規則 / 結構測試）
  • 引入 LLM-as-Judge 二次品質評估（取代純自評）

第三階段（Day 61-90）：收斂迴圈閉環

  • 建立 Convergence Dashboard（約束覆蓋率、漂移趨勢、自癒成功率）
  • 實作自動校準排程（每週日 00:00 重算 fitness function 基準）
  • 達成「結構性冪等」：新增約束不再導致系統行為不可預測

五、統一治理框架的核心原則

  1. 偵測先於自治（Detectability before Autonomy）：先建立完整可觀測性，再擴展 Agent 自主權限
  2. 約束即賦能（Constraints as Enablers）：限縮解空間使 Agent 更快收斂到正確解
  3. 模組化設計（Design for Deletion）：預期每段「聰明」邏輯終將被替換
  4. 失敗軌跡即訓練資料（Failure as Data）：捕捉失敗路徑作為約束演化的輸入
  5. 上下文持久性（Context Durability）：瓶頸已從模型能力轉向上下文在長時間執行中的保持能力

---

六、量化預期效果

根據 2026 年業界案例研究：

  • Harness 本身帶來 2-5× 可靠性提升（OpenAI & 獨立基準）
  • LangChain 透過中介軟體堆疊將成功率從 52.8% 提升至 66.5%（+26%）
  • 系統性 Harness 可將 Agent 漂移相關錯誤降低 40-60%

本系統若完成三階段路線，預期可將 daily_success_rate 從 82.8% 提升至 ≥ 90%、constraint_coverage 從 0% 提升至 ≥ 70%。
