OpenClaw — 2026 年 GitHub 最速爆紅 AI 助手框架深度研究

Hub-and-Spoke 三層架構、Workspace-First 設計哲學與 250K Stars 的技術解構

研究日期：2026-03-20

研究類型：GitHub 熱門專案深度分析

---

一、專案概述

OpenClaw 是由 PSPDFKit 創辦人 Peter Steinberger 開發的開源自主 AI 助手框架，於 2026 年 1 月底爆紅，60 天內從 9,000 Stars 飆升至 250,000+ Stars，超越 React 成為 GitHub 史上最速成長的軟體專案。2026 年 2 月 15 日，Steinberger 宣布加入 OpenAI，OpenClaw 將移交開源基金會管理。

核心數據

| 指標 | 數值 |

|------|------|

| Stars | 250,000+（2026-03） |

| 支援平台 | 20+ 訊息平台 |

| Skills 生態 | 5,705+ |

| 授權 | 開源（基金會管理） |

| 創建者 | Peter Steinberger（PSPDFKit 創辦人） |

| 定位 | 本地優先的個人 AI 助手作業系統 |

核心理念

OpenClaw 不是聊天機器人外殼 — 它是 AI Agent 的作業系統。Gateway 是唯一持有訊息會話的常駐程序，運行在你控制的機器上（家中 Mac mini 或隔離 VPS），所有資料本地儲存。

---

二、三層架構設計（Hub-and-Spoke）

OpenClaw 的核心是 Hub-and-Spoke 架構，將介面層（訊息來源）與助手運行時（智能執行）完全分離：

2.1 Gateway Layer（閘道層 — 會話管理）

  • 管理使用者會話生命週期與對話歷史
  • 訊息佇列實作並發控制（預設最多 10 個並發）
  • Session 隔離：per-channel-peer 模式，每位使用者在每個 Channel 擁有獨立 Session
  • 自動重試邏輯：指數退避（1s → 2s → 4s）
  • WebSocket 伺服器作為單一控制平面

2.2 Channel Layer（頻道層 — 平台適配）

  • Adapter Pattern：統一 StandardMessage 介面，將 WhatsApp/Telegram/Discord/Slack/Signal/iMessage 等平台格式正規化
  • 路由規則：dmPolicy（pairing/allowlist/open/disabled）+ mentionGating（群組聊天過濾）
  • 每個 Channel 是獨立插件，降低擴展成本

2.3 LLM Layer（模型層 — Provider 插件系統）

  • Provider Registry Pattern：每個 Provider 實作標準介面（chat()、supportTools()、initialize()）
  • 啟動時自動發現與註冊，無需修改核心程式碼
  • 支援串流回應（AsyncIterator Pattern）
  • 支援 Anthropic、OpenAI、Ollama、DeepSeek 及自訂模型

---

三、四大設計模式

3.1 Workspace-First 設計哲學

配置檔案作為 Agent 身份與行為的唯一真相來源（Source of Truth）：

| 檔案 | 用途 |

|------|------|

| SOUL.md | 定義 Agent 的目的與行為 |

| TOOLS.md | 指定可用能力 |

| IDENTITY.md | 個人化設定 |

| HEARTBEAT.md | 執行配置（事件驅動 + 輪詢） |

可借鏡：這與本專案的 CLAUDE.md + Skill-First 策略高度一致。差異在於 OpenClaw 將身份、工具、心跳分檔管理，而非集中在單一 CLAUDE.md。

3.2 四層概念模型

| 層級 | 職責 |

|------|------|

| Model | LLM 呼叫與回應生成 |

| Memory | 分層記憶管理 |

| Tools | 能力擴展與執行 |

| Orchestrator | 多步驟任務協調 |

3.3 Skill 模組化

  • Skills 是可過濾的模組化能力單元
  • Skill Registry 管理發現與載入
  • 依 Agent 角色和信任等級限定可用 Skills
可借鏡：本專案的 skills/SKILL_INDEX.md + config/routing.yaml 路由機制已實現類似概念，但 OpenClaw 的信任等級分層（Autonomy Level 0-5）可借鏡。

3.4 Heartbeat 驅動執行

支援事件驅動與輪詢雙模式的事件循環系統。

---

四、記憶系統（四層架構）

| 層級 | 名稱 | 說明 |

|------|------|------|

| Layer 1 | Session Context | 當前對話的模型上下文窗口 |

| Layer 2 | Daily Logs | 原始每日筆記 |

| Layer 3 | Long-term Memory | 手動維護的長期洞察 |

| Layer 4 | Semantic Vector Search | SQLite + embeddings 向量搜尋 |

可借鏡：本專案的 context/digest-memory.json（跨次記憶）+ context/research-series.json（系列追蹤）是 Layer 2-3 的實現，但缺乏 Layer 4 的向量搜尋。OpenClaw 的 12 層社群擴展記憶架構（Knowledge Graph + 語義搜尋 7ms GPU）展示了更完整的記憶金字塔。

---

五、安全模型

  • 白名單 Tool 執行：僅預定義工具可被呼叫
  • 參數驗證：嚴格輸入消毒防注入攻擊
  • 權限分層：管理員 vs 一般使用者不同存取等級
  • 審計日誌：所有 Tool 呼叫記錄可追溯
  • 自主等級光譜（Autonomy Level 0-5）：依實際需求對齊設計複雜度
可借鏡：本專案的 Hook 機器強制層（pre_bash_guard + pre_write_guard）已實現類似的輸入消毒與審計日誌，但缺乏「自主等級光譜」的明確分級。

---

六、可借鏡的技術項目

立即可用

  1. Adapter Pattern 多平台統一介面：OpenClaw 的 StandardMessage 介面模式可應用於本專案的 LINE Bot + ntfy 通知統一化
  2. Provider Registry Pattern：啟動時自動發現與註冊，消除條件判斷邏輯 — 本專案的 LLM Router 已部分實現（O(1) lookup），但可進一步支援動態註冊
  3. 指數退避重試：Gateway 的 1s→2s→4s 退避策略可應用於 API 呼叫失敗場景

中期借鏡

  1. Workspace-First 分檔管理：將 CLAUDE.md 拆分為 SOUL.md（角色）+ TOOLS.md（工具）+ HEARTBEAT.md（排程），降低單檔膨脹
  2. Autonomy Level 分級：為 Todoist 自動任務定義 0-5 級自主等級，高風險操作需人工確認
  3. 四層記憶架構：在現有 digest-memory.json 基礎上加入語義向量層，提升跨次記憶的召回品質

長期參考

  1. Session 隔離與並發控制：per-channel-peer 模式在多 Agent Team 同時執行時確保 Context 不互相汙染
  2. Skill 信任等級：依 Skill 風險等級設定可用範圍（唯讀 / 讀寫 / 系統操作），強化安全邊界

---

七、與本專案架構對照

| 面向 | OpenClaw | Daily Digest Prompt |

|------|----------|--------------------|

| 架構風格 | Hub-and-Spoke（Gateway 中心） | 文件驅動（Prompt 薄層 + YAML 配置） |

| 配置管理 | Workspace-First（4 檔分管） | CLAUDE.md 集中 + 外部 YAML |

| 記憶系統 | 4 層（Session→Daily→LT→Vector） | 2 層（digest-memory + research-series） |

| 安全模型 | 白名單 + 權限分層 + 審計 | Hook 機器強制 + JSONL 日誌 |

| 插件機制 | Provider Registry（自動發現） | Skill-First（手動索引） |

| 排程 | Heartbeat（事件+輪詢） | Windows Task Scheduler（cron） |

| 多平台 | 20+ Channel Adapter | LINE Bot + ntfy |

---

八、結論

OpenClaw 的成功不僅來自功能豐富，更來自其清晰的架構分層與極低的擴展成本。Hub-and-Spoke 模式讓新 Channel 只需實作一個 Adapter，Provider Registry 讓新模型只需註冊一個介面，Workspace-First 讓 Agent 行為完全由版本控制的文字檔定義 — 這三個設計決策共同降低了貢獻門檻，是 250K Stars 爆發的技術基石。
