摘要

本研究整合 2026 年最新的測試驅動開發（TDD）原則與 LLM Agent 測試策略，解決 AI Agent 非確定性行為的測試挑戰。研究涵蓋：(1) TDD 紅綠重構循環的核心原則，(2) AI Agent 測試金字塔的三層架構，(3) 測試層級分佈的 70/20/10 規則，以及 (4) 非確定性行為的評估技術。研究發現顯示，將傳統 TDD 與 AI Agent 特性結合，需要在單元測試（Mock LLM）、整合測試（Real API + 約束）與端到端測試（場景驗證）三層級間建立平衡，並透過多次執行平均來吸收變異性。

關鍵洞見

1. TDD 紅綠重構循環是設計規範，非測試策略

核心發現：TDD 本質上是一種設計規範（design discipline），而不僅僅是測試策略。紅綠重構循環（Red-Green-Refactor）強制開發者在撰寫實作程式碼之前先思考設計與需求。

三階段流程：

  • Red（紅燈）：先撰寫一個失敗的測試，定義你需要建構的功能
  • Green（綠燈）：撰寫最小程式碼使測試通過，不追求完美
  • Refactor（重構）：在綠燈保護下改善程式碼品質

證據來源：

  • Codecademy: Red, Green, Refactor — B 級
  • Monday.com: TDD Complete Guide 2026 — B 級
  • James Shore: Red-Green-Refactor — A 級（TDD 權威）

獨特洞見：alexop.dev 展示了如何強制 Claude Code 遵循 TDD 循環，證明 Agentic 工作流也能應用紅綠重構模式。

2. LLM Agent 測試金字塔：Mock → 約束 → 場景

核心挑戰：AI Agent 的非確定性（temperature 與 top-p 採樣）使得相同輸入產生不同輸出，打破傳統自動化測試的假設。

適應性測試金字塔（由 Derek C. Ashmore 2026 年提出）：

| 層級 | 測試類型 | LLM 狀態 | 特性 | 佔比 |

|------|---------|---------|------|------|

| Layer 1 | Unit Tests | Mock LLM | 毫秒級、零成本、確定性 | 70% |

| Layer 2 | Integration Tests | Real API + 約束參數 | 真實 API、降低變異性 | 20% |

| Layer 3 | E2E Tests | Real API 完整場景 | 多輪對話、工具鏈驗證 | 10% |

關鍵技術：

  • Layer 1：完全 Mock LLM，測試周邊邏輯（prompt 構建、回應解析、錯誤處理）
  • Layer 2：真實 API 呼叫但使用 temperature=0 或固定 seed，驗證 prompt 實際有效
  • Layer 3：場景式評估套件，模擬多輪對話或複雜工具使用序列

證據來源：

  • Medium: AI Agent Testing Pyramid (Derek C. Ashmore, 2026) — A 級
  • SitePoint: Testing AI Agents — B 級
  • arXiv: Automated structural testing of LLM-based agents — A 級（學術研究）

3. 非確定性評估：3+ 次執行 + 判定器分級

核心技術：因為 LLM Agent 本質上是隨機的，測量一致性需要執行相同任務多次並觀察變異性。

推薦做法：

  1. 平均 3+ 次執行：吸收非確定性變異，取平均評分
  2. 判定器優先順序：

- 確定性判定器（如正則表達式、JSON schema 驗證）：優先使用

- LLM 判定器（用另一個 LLM 評估輸出）：彈性但有成本

- 人工判定器：精準但昂貴，僅用於額外驗證

評估維度：

  • 正確性：輸出是否符合事實
  • 完整性：是否涵蓋所有必要資訊
  • 相關性：是否回答實際問題
  • 一致性：多次執行的穩定度

證據來源：

  • Anthropic: Demystifying evals for AI agents — A 級（官方指引）
  • AWS: AgentCore Evaluations — A 級
  • Comet: AI Agent Evaluation — B 級

4. 測試金字塔 70/20/10 規則與反模式警告

推薦分佈：

  • 70-80% 單元測試：快速回饋、低維護成本、大量覆蓋
  • 15-20% 整合測試：驗證模組協作、API 契約、資料庫互動
  • 5-10% 端到端測試：僅覆蓋關鍵使用者流程與高風險區域

常見反模式：許多團隊不小心顛倒金字塔 — 大量依賴 E2E 測試而跳過整合測試，導致回饋緩慢、不穩定的建置與延遲的錯誤發現。

實際案例：對於典型 Web 應用程式，建議：

  • 單元測試：700-800 個（快速、可並行執行）
  • 整合測試：150-200 個（驗證服務間互動）
  • E2E 測試：30-80 個（僅關鍵使用者旅程）

5. TDD 與 LLM Agent 的整合策略

核心挑戰：傳統 TDD 假設確定性行為，而 LLM Agent 本質上是非確定性的。

整合策略：

  1. 分層測試：

- 確定性層（業務邏輯、工具呼叫、狀態管理）：使用傳統 TDD 紅綠重構

- 非確定性層（LLM 回應生成）：使用 Mock（Unit）+ 約束 API（Integration）+ 多次評估（E2E）

  1. 可測試性設計：

- 依賴注入：LLM 介面抽象化，方便 Mock

- 分離關注點：業務邏輯與 LLM 呼叫分離

- 固定點驗證：測試輸入/輸出的固定點（如 JSON schema），而非精確內容

  1. CI/CD 整合：

- Unit 測試在每次 commit 執行（< 5 分鐘）

- Integration 測試在 PR 階段執行（< 15 分鐘）

- E2E 測試僅在 merge 到主分支或排程執行（夜間）

實務建議（來自本專案經驗）：

  • 本系統已有 1620 個測試（hooks 529 + skills 27 + tools 300），但缺乏 TDD 方法論指導
  • 建議建立測試策略文件，明確各層級的目標覆蓋率與品質門檻
  • 引入測試覆蓋率監控與 Mutation Testing（目前僅部分啟用）

- 明確 TDD 紅綠重構流程的實施規範

- 建立 LLM Agent 測試的特殊規範（Mock 策略、評估方法、多次執行標準）

  1. 優化測試金字塔比例：

- 審查現有 1620 個測試的分佈，確認是否符合 70/20/10 規則

- 若 E2E 測試過多，考慮降級為整合測試以提升執行速度

  1. 引入確定性判定器：

- 為 LLM Agent 輸出建立 JSON schema 驗證器

- 為關鍵流程建立正則表達式斷言（如結果 JSON 格式）

中期行動（1-2 個月）

  1. 實施 TDD 工作流：

- 新功能開發強制使用紅綠重構循環

- 建立 pre-commit hook 確保測試覆蓋率不下降

  1. 建立 LLM Agent 評估套件：

- 實作多次執行評估框架（平均 3+ 次）

- 建立一致性指標追蹤（變異係數、標準差）

  1. 整合 Mutation Testing：

- 擴展現有的 Mutation Score 計算至所有關鍵模組

- 設定 P0/P1 模組的 Mutation Score 門檻（≥60%）
