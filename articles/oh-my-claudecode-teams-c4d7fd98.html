一、專案概述

oh-my-claudecode（OMC）（Yeachan-Heo/oh-my-claudecode）是 2026 年 3 月 GitHub 上快速崛起的 Claude Code 多 Agent 編排框架，累計 14.7K Stars、2,192 commits。其核心哲學是「Teams-first，零學習曲線」——使用者無需記憶指令語法，以自然語言驅動 32 個專業 Agent 協作完成軟體工程全生命週期任務。

二、架構分析

2.1 四層編排策略

OMC 最獨特的設計是提供多種執行策略供不同場景選擇：

| 策略 | 特性 | 適用場景 |

|------|------|----------|

| Team Mode（正典） | 五階段管線：plan → prd → exec → verify → fix | 複雜多檔案功能開發 |

| Autopilot | 意圖偵測 + 19 Agent 自動調度 | 一般開發任務 |

| Ralph | 持久化驗證/修復迴圈，拒絕半途放棄 | 需要高完成度的任務 |

| Ultrawork (ulw) | 最大化並行，非依賴任務同時執行（上限 6 Agent） | 獨立子任務批量處理 |

| Pipeline | 循序分階段處理 | 有強依賴的工作流 |

對比本專案：我們的 run-todoist-agent-team.ps1 採用三階段（查詢→N並行執行→組裝），類似 OMC 的 Team Mode 但缺少 verify → fix 自動修復迴圈。

2.2 Agent 分工矩陣（32 Agent，4 Lane）

Build/Analysis Lane（7 Agent）：

  • Explore（程式碼搜尋）→ Analyst（需求釐清）→ Planner（任務排序）→ Architect（系統設計）→ Executor（實作）→ Verifier（驗證）→ Debugger（根因分析）

Review Lane（5 Agent）：

  • Style-Reviewer / Code-Reviewer / API-Reviewer / Security-Reviewer / Performance-Reviewer
  • 五種審查面向各由專責 Agent 負責，非一個 Agent 全包

Domain Specialists（8 Agent）：

  • Dependency-Expert / Test-Engineer / Quality-Strategist / Designer / Writer / QA-Tester / Git-Master / Researcher

Product Lane（4 Agent）：

  • Product-Manager / UX-Researcher / Information-Architect / Product-Analyst

設計洞察：OMC 將 Review 拆為五個獨立角色（Style / Code / API / Security / Performance），每個角色有專屬 system prompt，而非一個 code-reviewer 一次審所有面向。這種「微服務式」分工讓每個 Agent 的 context window 更聚焦。

2.3 Smart Model Routing

OMC 內建模型路由：

  • Haiku：簡單任務（格式檢查、命名建議、狀態查詢）
  • Sonnet：中等任務（Code Review、測試生成）
  • Opus：複雜推理（架構設計、多檔案重構、除錯）

宣稱可節省 30-50% API 成本。

對比本專案：我們的 tools/llm_router.py + config/llm-router.yaml 實現了類似機制（Groq vs Claude 路由），但 OMC 的路由更細緻到 Agent 粒度。

三、技術亮點

3.1 Skill 自學習機制（/learner）

OMC 最具創新性的功能之一：

  1. Agent 完成任務後，/learner 自動萃取可複用模式
  2. 經過品質門檻篩選後，寫入 .omc/skills/ 成為專案級 Skill
  3. 下次遇到相似上下文時自動注入
  4. 分兩層：專案級（.omc/skills/，高優先）和使用者級（~/.omc/skills/，跨專案）

可借鏡價值：本專案有 29 個手動維護的 Skill，但缺少自動 Skill 萃取機制。可參考 OMC 的 /learner 設計，在 todoist-auto 任務完成後自動提取成功模式。

3.2 Socratic Deep Interview（/deep-interview）

在編碼前透過蘇格拉底式提問揭露隱藏假設：

  • 不直接執行模糊指令，先反問「你真正想解決什麼問題？」
  • 挖掘邊界條件和隱含約束
  • 生成明確的 PRD 後才進入實作

對比本專案：我們的「規格」階段（spec.md）有類似功能，但未自動化觸發。

3.3 Ralph 持久化模式

啟用 Ralph 後，Agent 拒絕在任務未驗證完成前停止：

  • verify → 發現問題 → fix → 再 verify → 直到通過
  • 類似我們的 done_cert 機制，但 OMC 的迴圈是自動的

3.4 Session Replay & Analytics

  • 所有 Agent 互動記錄在 .omc/state/agent-replay-*.jsonl
  • 內建 Token/成本追蹤
  • HUD 狀態列顯示即時編排指標

對比本專案：我們有 hooks/post_tool_logger.py 做 JSONL 日誌，但缺少 Session Replay 視覺化和成本追蹤面板。

3.5 Multi-Provider 支援

OMC 不限於 Claude，還支援：

  • Codex CLI workers
  • Gemini CLI workers
  • /ccg Skill 同時向三家 Provider 請求，綜合最佳回答

四、可借鏡技術項目（按優先級排序）

P0：verify → fix 自動修復迴圈

現況：本專案的 Phase 3 組裝只彙整結果，不驗證不修復。

借鏡：在 run-todoist-agent-team.ps1 Phase 3 後加入 Phase 4（verify），驗證失敗時自動重跑對應子 Agent。

預期效益：減少人工介入修復失敗任務。

P1：Review Lane 五維度拆分

現況：code-reviewer Skill 一次審所有面向。

借鏡：將 Code Review 拆為獨立的 Style / Logic / Security / Performance / API 五個子 Agent 並行審查。

預期效益：每個審查 Agent 的 context window 更聚焦，審查深度提升。

P1：Skill 自學習與品質門檻

現況：29 個 Skill 全部手動撰寫和維護。

借鏡：任務完成後自動萃取成功模式，經品質門檻（覆蓋率、可複用性評分）後寫入 Skill。

預期效益：Skill 生態系自動成長，減少維護負擔。

P2：成本追蹤與 Token 經濟儀表板

現況：tools/budget_guard.py 做預算控管，但缺少歷史成本視覺化。

借鏡：OMC 的 Session Replay + Analytics，在 check-health.ps1 加入每日/每週 Token 消耗趨勢。

預期效益：成本透明化，便於優化高消耗任務。

P2：Autopilot 意圖偵測

現況：Todoist 任務靠 routing.yaml 三層路由，需手動維護規則。

借鏡：OMC 的 Autopilot 以 LLM 做意圖分類自動派發，類似我們的 tools/llm_classifier.py 但更深度整合。

預期效益：新增任務類型時無需手動修改路由規則。

P3：Session Replay JSONL 視覺化

現況：JSONL 日誌只能用 query_logs.py 查詢。

借鏡：OMC 的 .omc/state/agent-replay-*.jsonl 提供完整的 Session 重放能力。

預期效益：除錯效率提升，可回溯任何失敗任務的完整決策鏈。

五、批判性評價

優勢

  1. 零配置理念：安裝即可用，降低入門門檻
  2. Agent 粒度精細：32 Agent 的分工比大多數框架更細，每個 Agent 專注一件事
  3. 多策略靈活切換：Team/Autopilot/Ralph/Ultrawork 各有適用場景
  4. 成本感知：Smart Routing 自動降級到便宜模型

風險與局限

  1. Agent 爆炸：32 個 Agent 的互動複雜度是 O(n²)，協調開銷可能抵消效能增益
  2. Plugin 依賴：完全依附 Claude Code Plugin 系統，若 API 變更則全盤受影響
  3. 黑盒問題：Autopilot 的意圖偵測 + 自動派發缺乏透明度，出錯時難以診斷
  4. 單一維護者：核心程式碼主要由 Yeachan Heo 一人推動，Bus Factor = 1

七、結論

oh-my-claudecode 代表了 Claude Code 生態中「重量級編排」的方向——用 32 個專業 Agent + 5 種執行策略覆蓋軟體工程全流程。其最具啟發性的設計是：

  1. Skill 自學習打破了手動維護的瓶頸
  2. verify → fix 迴圈讓任務自動達到品質標準
  3. 五維度 Review Lane展示了如何讓審查更深入

本專案可優先借鏡 verify-fix 迴圈和 Skill 自學習機制，這兩項投入產出比最高。
