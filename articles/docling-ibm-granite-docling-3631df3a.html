一、專案概述

Docling（docling-project/docling）是由 IBM Research Zurich 的 AI for Knowledge 團隊開發的開源文件解析與轉換工具包，定位為「Get your documents ready for gen AI」。截至 2026 年 3 月已累積 54,700+ Stars、3,700+ Forks、155 個版本（最新 v2.76.0），被 2,800+ 個專案使用，由 LF AI & Data Foundation 支持，採用 MIT 授權。

二、核心架構：三層文件理解管線

Docling 的架構設計體現了「統一表徵、模組管線、靈活輸出」的理念，分為三個核心層次：

2.1 輸入層：多格式解析器

支援的文件格式涵蓋企業級文件處理的完整光譜：

  • 結構化文件：PDF（程式化 + 掃描）、DOCX、PPTX、XLSX
  • 標記文件：HTML、LaTeX、Markdown
  • 媒體：圖片（OCR）、音訊（語音轉文字）
  • 專業格式：USPTO 專利 XML、JATS 學術論文 XML、XBRL 財務報告

2.2 處理層：AI 模型 Ensemble 管線

Docling 的核心創新在於將多個專業 AI 模型組合成可定制的 Ensemble 管線：

版面分析模型（Heron Layout Model）

  • 訓練資料：約 81,000 頁手工標註頁面（專利、手冊、10-K 財報）
  • 技術：物件偵測（object detection）辨識頁面元素（文字區塊、圖片、表格、標題）
  • 精準度：接近人工標註的五個百分點以內

表格識別模型（TableFormer）

  • 源自 IBM DeepSearch 專案
  • 將圖片形式的表格轉換為結構化的行列資料
  • 在內部測試中超越同類表格識別工具

OCR 替代策略

  • 優先使用電腦視覺模型而非傳統 OCR
  • 處理速度提升 30 倍，同時降低錯誤率
  • 這是一個重要的架構決策：以視覺理解取代字元識別

2.3 輸出層：DoclingDocument 統一表徵

所有格式的文件經處理後統一轉換為 DoclingDocument 中間表徵格式，再輸出為：

  • Markdown：最適合 LLM 消費
  • HTML：保留版面結構
  • JSON：程式化處理
  • DocTags：Granite-Docling 專用的緊湊標記格式

三、Granite-Docling：258M 參數的單次推理 VLM

Granite-Docling 是 Docling 生態系統的重大進化——一個 258M 參數的視覺語言模型（VLM），能以單次推理完成整頁文件解析。

3.1 設計哲學：單模型 vs Ensemble

| 面向 | Docling（Ensemble） | Granite-Docling（單 VLM） |

|------|-------------------|-------------------------|

| 架構 | 多模型組合管線 | 單一 258M VLM |

| 靈活性 | 高（可替換各模型） | 中（一體化） |

| 部署複雜度 | 高（多模型依賴） | 低（單模型） |

| 推理次數 | 多次（各模型各推一次） | 一次（single-pass） |

| 適用場景 | 需要精細控制的企業管線 | 快速部署、邊緣推理 |

3.2 DocTags 創新標記語言

Granite-Docling 引入了 DocTags——一種專為文件結構設計的通用標記格式：

  • 明確語義標籤：為每種頁面元素提供專用標籤（表格、公式、程式碼、清單等）
  • 位置感知：記錄每個元素在頁面上的精確位置
  • 關係保留：維護元素間的層級關係（如標題對應圖片）
  • Token 效率：比 HTML/Markdown 更緊湊，減少 Token 使用量
  • 無損轉換：避免 HTML/Markdown 轉換中常見的資訊遺失

3.3 處理能力

以 258M 參數處理：圖表、表格、表單、程式碼、數學公式、註腳、標題——能力對標遠大於其規模的模型。IBM 稱其「capabilities rival those of systems several times its size」。

四、MCP Server：Agentic 文件處理管線

Docling-MCP（docling-project/docling-mcp，488 Stars）將 Docling 的文件處理能力暴露為 MCP（Model Context Protocol）工具，使 AI Agent 能動態呼叫文件處理功能。

4.1 暴露的 MCP 工具

| 工具 | 功能 |

|------|------|

| convert_pdf_to_docling_document | PDF 轉結構化 JSON |

| create_docling_document | 程式化建立文件 |

| add_title / add_section / add_list | 文件元素操作 |

| export_to_markdown | Markdown 輸出 |

| upload_to_milvus / retrieve_from_milvus | RAG 向量資料庫整合 |

4.2 Agentic RAG 管線

Docling-MCP 解決了 RAG 系統中的核心痛點——資料品質。典型流程：


PDF 上傳 → Agent 呼叫 Docling MCP → 結構化 DoclingDocument
→ 語義分塊 → 向量化存入 Milvus → Agent 檢索問答

Agent 可在此過程中動態決策：

  • 文件格式不明？→ 自動偵測並選擇解析策略
  • 表格品質不佳？→ 切換到 Granite-Docling VLM 重新解析
  • 需要特定章節？→ 利用 DoclingDocument 的結構化索引定位

六、可借鑑技術項目（對 Daily-Digest-Prompt 系統的啟示）

啟示 1：統一文件表徵格式（DoclingDocument 模式）

現狀：我們的系統處理多種來源（Todoist JSON、新聞 HTML、HN API、Gmail），各自有不同的中間格式。

借鑑：Docling 將所有文件格式統一為 DoclingDocument 再輸出——我們可以設計一個 DigestDocument 統一中間表徵，標準化所有資料來源的處理管線。這會大幅簡化 Phase 2 組裝 Agent 的複雜度。

啟示 2：Ensemble 管線 vs 單模型的架構抉擇

現狀：我們的 Phase 1 用 5 個並行 Agent（Ensemble 模式），各自負責一個資料來源。

借鑑：Docling 同時提供 Ensemble 管線和 Granite-Docling 單模型兩種選擇。這暗示一個演進方向——當 Groq Relay 能力足夠時，某些簡單的 Phase 1 任務（如新聞摘要、HN 翻譯）可以合併為單次 Groq 呼叫，降低 Agent 數量和啟動開銷。

啟示 3：MCP 作為 Agent 工具的標準化接口

現狀：我們的知識庫透過直接 curl 呼叫 REST API。

借鑑：Docling-MCP 將文件處理暴露為 MCP 工具，Agent 可直接呼叫而無需了解底層 API 細節。我們的知識庫（localhost:3000）可以同樣封裝為 MCP Server，讓 Agent 透過標準 MCP 協定操作知識庫，取代目前散落在各模板中的 curl 呼叫。

啟示 4：DocTags 的 Token 經濟學

現狀：我們的研究筆記以 Markdown 匯入知識庫。

借鑑：DocTags 以更緊湊的標記減少 Token 使用量。對於我們每日 44 次 Agent 執行，如果能在知識庫查詢結果中使用更緊湊的格式（而非完整 Markdown），可以降低 Context Window 消耗，讓單次執行能處理更多資訊。

啟示 5：視覺理解取代字元識別的範式轉移

通用啟示：Docling 選擇電腦視覺而非 OCR，獲得 30 倍速度提升。這體現了 AI 領域的一個重要範式：當底層技術足夠成熟時，端到端解決方案往往優於管線式解決方案（end-to-end > pipeline）。對 Agent 系統設計亦然——當 LLM 能力提升時，某些需要多步驟管線處理的任務可能被單次推理取代。

七、技術深度分析

7.1 版面分析的物件偵測方法

Docling 採用物件偵測而非規則引擎來分析文件版面，這帶來三個優勢：

  1. 泛化能力：81,000 頁標註資料涵蓋多種文件類型，不需要為每種格式寫規則
  2. 魯棒性：對掃描品質不佳的文件仍能辨識結構
  3. 可擴展性：新增文件類型只需標註新的訓練資料

7.2 TableFormer 的技術價值

表格是文件中最難處理的結構——跨行跨列合併、嵌套表格、不規則邊界。TableFormer 源自 IBM DeepSearch 的多年積累，在這個細分領域建立了技術護城河。

7.3 258M 的小模型大能力

Granite-Docling 以 258M 參數實現競爭力——這得益於：

  • 任務專一化：只做文件理解，不做通用語言生成
  • DocTags 設計：專用標記語言降低輸出空間複雜度
  • 高品質訓練資料：IBM 內部積累的大量標註文件

這呼應了 AI 領域的趨勢：專用小模型在特定任務上可以匹敵甚至超越通用大模型。
