一、專案概述

Ladybird（LadybirdBrowser/ladybird）是由 Andreas Kling 主導的完全獨立開源瀏覽器專案，從零打造瀏覽器引擎，不使用任何來自 Blink、WebKit 或 Gecko 的程式碼。專案以 501(c)(3) 非營利組織形式運營，目標是打破當前瀏覽器引擎被少數巨頭壟斷的局面。

核心數據

| 穩定版目標 | 2028 年 |

二、為什麼 Ladybird 值得關注

2.1 瀏覽器引擎壟斷問題

目前全球瀏覽器引擎僅剩三個：Google 的 Blink（Chrome/Edge/Opera/Brave/Vivaldi）、Apple 的 WebKit（Safari）、Mozilla 的 Gecko（Firefox）。Blink 市佔率超過 75%，形成事實上的壟斷。Ladybird 是近 20 年來第一個從零打造的獨立瀏覽器引擎，試圖打破這個壟斷。

2.2 獨立性與資金來源

  • 完全捐贈制：不接受搜尋引擎交易、不出售使用者資料、不做廣告
  • 不出售影響力：贊助商無法購買董事會席位或開發方向的決策權
  • 18 個月財務跑道：以永續步調招聘，維持至少 18 個月的營運資金
  • GitHub 共同創辦人 Chris Wanstrath 捐贈 100 萬美元協助成立非營利組織

2.3 重要贊助商

| 等級 | 贊助商 | 年費 |

|------|--------|------|

| 白金 | Cloudflare、FUTO、Shopify | $100K |

| 金 | — | $50K |

| 銀 | — | $10K |

| 銅 | 37signals | $5K |

Cloudflare 在 2025 年 9 月宣布贊助，強調「支持開放網路的未來」，其贊助為無附加條件的捐贈。

三、技術架構

3.1 從零打造的引擎組件

Ladybird 的每個主要元件都是自行開發：

  • LibWeb：HTML/CSS 渲染引擎
  • LibJS：JavaScript 引擎（ECMAScript 相容）
  • LibURL：URL 解析
  • LibHTTP：HTTP 網路層
  • LibGfx：2D 圖形渲染

3.2 第三方函式庫策略

最初繼承自 SerenityOS 的「自行實作一切」哲學，現已轉向務實路線：影像處理、音訊/影片、加密、圖形等功能改用成熟的第三方函式庫，將工程資源集中在瀏覽器引擎核心。

---

四、Rust 遷移 — AI 輔助的語言轉換

4.1 為什麼選 Rust（而非 Swift）

2024 年曾嘗試引入 Swift 作為 C++ 的後繼語言，但因 Swift 的跨平台支援不足和 C++ 互操作性問題，導致約一年的延遲。2026 年 2 月正式宣布改用 Rust，理由：

  1. 記憶體安全保證：瀏覽器是高風險攻擊面，Rust 的 borrow checker 可消除整類漏洞
  2. 成熟的生態系：Firefox（Servo/Stylo）和 Chromium 都已開始引入 Rust
  3. 優秀的 C++ 互操作性：透過 cxx 等工具可漸進式遷移

4.2 AI 輔助移植實踐

首個遷移目標是 LibJS 的詞法分析器、語法分析器、AST 和位元組碼產生器，選擇原因：

  • 模組相對獨立（self-contained）
  • 輸入輸出定義明確
  • 有完善的 ECMAScript 測試套件覆蓋

移植數據：

| 指標 | 數值 |

|------|------|

| 產出 Rust 程式碼 | ~25,000 行 |

| 所需時間 | 約 2 週 |

| 手動估計時間 | 數個月 |

| 使用的 AI 工具 | Claude Code + Codex |

| 移植方式 | 人工指導 + 數百個小型 prompt |

| 測試通過 | 52,898 個 ECMAScript 測試 + 12,461 個 Ladybird 回歸測試 |

| 回歸數量 | 零（byte-for-byte 輸出一致） |

| 效能影響 | 無（所有 JS 工作負載效能一致） |

Andreas Kling 強調這是「人工指導的翻譯，非自主程式碼生成」（human-directed, not autonomous code generation），並經過多個模型的對抗式審查。

4.3 漸進式遷移策略

C++ 仍是主要開發語言，Rust 移植是「長期平行進行的支線」。兩種語言透過明確的整合邊界共存，確保安全互操作。

---

五、對開源生態的啟示

5.1 AI 輔助大型程式碼遷移的範例

Ladybird 的 Rust 遷移是目前最具說服力的 AI 輔助語言轉換案例之一：

  • 規模：25,000 行非平凡的編譯器程式碼
  • 品質：零回歸、byte-for-byte 一致
  • 效率：2 週 vs 數月
  • 方法論：人機協作（非全自動），數百個 prompt + 多模型交叉審查

5.2 瀏覽器多樣性的重要性

Web 標準的健康演進需要多個獨立引擎的競爭。當單一引擎市佔過高時，該引擎的實作缺陷可能被當作「事實標準」，影響整個網路生態。Ladybird 的存在為 Web 標準提供了額外的獨立驗證。

5.3 非營利開源的永續模式

Ladybird 的資金模式——純捐贈制、不出售影響力、維持 18 個月跑道——為大型基礎設施開源專案提供了一個值得參考的永續經營範本。
