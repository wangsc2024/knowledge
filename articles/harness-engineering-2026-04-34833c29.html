一、Separated Evaluator Pattern（分離式評估器模式）

1.1 核心理念

問題：傳統測試框架期待確定性輸出（exact match），但 AI agent 對同一任務可能產生多種正確回應（"Yes, you are eligible" vs "You qualify for this program"），導致測試脆弱且維護成本高。

解決方案：將測試案例（Cases）與評估器（Evaluators）分離：

  • Cases 定義測試場景（inputs、expected behaviors、success criteria）
  • Evaluators 提供判斷邏輯（使用 LLM-as-judge、rule-based checks、semantic similarity 等）
  • Experiments 處理編排與報告（orchestration + result aggregation）

優勢：

  1. 靈活性：更換評估邏輯不需重寫測試案例
  2. 可組合性：同一組 Cases 可使用不同 Evaluators（語義相似度、任務完成度、安全性檢查）
  3. 可擴展性：新增評估維度（latency、cost、ethical alignment）不影響既有測試

1.2 實施架構


Test Suite
├── Cases (JSON/YAML)
│   ├── scenario_001.yaml    # 輸入、預期行為
│   ├── scenario_002.yaml
│   └── ...
├── Evaluators (Python/TS)
│   ├── task_completion.py   # 任務是否完成
│   ├── semantic_match.py    # 語義相似度 >0.85
│   ├── safety_check.py      # 無毒性/偏見輸出
│   └── cost_budget.py       # token 使用 <限額
└── Experiments (Orchestrator)
    └── run_eval.py          # 執行 Cases × Evaluators

範例（基於 DeepEval + LangChain）：


# Case 定義
case = {
    "input": "用戶詢問退款資格",
    "context": {"order_date": "2026-03-15", "status": "delivered"},
    "expected": "判斷符合 30 天退款政策"
}

# Evaluator 1: Task Completion
from deepeval.metrics import TaskCompletionMetric
metric1 = TaskCompletionMetric(threshold=0.8, model="claude-3-7-sonnet-latest")

# Evaluator 2: Semantic Similarity
from deepeval.metrics import AnswerRelevancyMetric
metric2 = AnswerRelevancyMetric(threshold=0.85)

# Experiment
from deepeval import evaluate
evaluate(test_cases=[case], metrics=[metric1, metric2])

1.3 關鍵評估維度

根據 2026 年最佳實踐，AI agent 評估應涵蓋：

| 評估層 | 指標 | 工具推薦 |

|--------|------|----------|

| Reasoning Layer | G-Eval、CoT Faithfulness、Hallucination | DeepEval、LangSmith |

| Action Layer | Tool Selection Accuracy、API Call Success Rate | Braintrust、Maxim AI |

| Trajectory Metrics | Plan Quality、Decision Path Coherence | Custom Evaluators |

| Outcome Metrics | Task Completion、User Satisfaction | End-to-end Tests |

重要原則：

  • Trajectory metrics 解釋「為何有效」（why it works）
  • Outcome metrics 驗證「是否有效」（if it works）
  • 兩者需並行追蹤，單獨使用任一者都不足

二、Plan-Execute-Verify Pattern（計畫-執行-驗證模式）

2.1 架構設計

傳統 ReAct Pattern 的問題：

  • 每步同時進行 reasoning + action（混合決策與執行）
  • 簡單查詢高效，但複雜多步驟任務易失敗或緩慢
  • 無法事後驗證計畫品質（計畫隱含在 chain-of-thought 中）

PEV Pattern 的改進：

  1. Planning Phase：由 reasoning-focused LLM 分析任務、分解子任務、排序優先級，輸出結構化計畫（JSON/YAML）
  2. Execution Phase：executor 逐步執行計畫，呼叫工具、更新系統、回報進度
  3. Verification Phase：驗證每步輸出（schema validation、quality checks、error detection）

核心優勢：

  • 可檢視性：計畫是 first-class artifact，可記錄、檢查、審計
  • 可除錯性：失敗時可定位具體計畫步驟（而非黑盒 CoT）
  • 可靠性：驗證層攔截低品質輸出，避免錯誤級聯傳播

2.2 實施範例


# Planning Output (structured plan)
plan:
  goal: "處理用戶退款請求"
  steps:
    - id: 1
      action: retrieve_order_info
      params: {order_id: "ABC123"}
      expected_output: {status, purchase_date, amount}
    - id: 2
      action: check_refund_policy
      depends_on: [1]
      params: {purchase_date: "$step1.purchase_date"}
      expected_output: {eligible: boolean, reason: string}
    - id: 3
      action: process_refund
      depends_on: [2]
      condition: "$step2.eligible == true"
      params: {order_id: "ABC123", amount: "$step1.amount"}

# Execution (step-by-step)
executor.run(plan)
  → step 1: API call → {status: "delivered", purchase_date: "2026-03-15", amount: 299}
  → step 2: rule check → {eligible: true, reason: "within 30-day window"}
  → step 3: refund API → {success: true, refund_id: "REF456"}

# Verification (after each step)
verify_step_1:
  - schema_valid: ✓ (contains status, purchase_date, amount)
  - data_quality: ✓ (amount > 0, date is valid ISO8601)

verify_step_2:
  - schema_valid: ✓ (contains eligible, reason)
  - logic_check: ✓ (eligible=true only if within policy window)

verify_step_3:
  - api_success: ✓ (HTTP 200)
  - idempotency: ✓ (refund_id recorded in state)

2.3 適用場景

適合 PEV Pattern 的任務：

  • 多步驟工作流程（>3 步驟）
  • 有明確階段/依賴關係（需先 A 後 B）
  • 需審計追蹤（金融、醫療、法律合規）
  • 成本敏感場景（計畫可重用，減少重複推理）

不適合的場景：

  • 簡單單步驟任務（overhead 過高）
  • 實時互動（latency 增加 2-3×）
  • 高度動態環境（計畫快速過時）

Trade-offs：

  • ✅ 可靠性 ↑ 40-60%（驗證層攔截錯誤）
  • ✅ 可除錯性 ↑（顯式計畫 + 中間狀態記錄）
  • ❌ Latency ↑ 2-3×（多次 LLM 呼叫）
  • ❌ Cost ↑ 30-50%（planning + execution + verification 三階段）

三、Constraint Harnesses 實施（約束馬具）

3.1 四層控制架構

根據 2026 年最佳實踐，Constraint Harnesses 應採用分層防禦（Defense in Depth）：

| 層級 | 機制 | 範例 | 失敗成本 |

|------|------|------|----------|

| Layer 1: Prevention | Permission boundaries、architectural constraints | 禁止修改 state/*.json、強制 DI 注入 | 最低（阻止於源頭） |

| Layer 2: Validation | Schema checks、linters、type checkers | JSON schema validation、YAML lint | 低（開發階段攔截） |

| Layer 3: Verification | Test suites、CI gates | 單元測試 >80% 覆蓋率、E2E tests | 中（部署前發現） |

| Layer 4: Monitoring | Runtime checks、circuit breakers、rollback | Error rate >5% → auto-rollback | 高（生產環境補救） |

關鍵設計原則：

  1. Prevention First：優先實施 Layer 1-2（成本最低、效果最佳）
  2. Layered Redundancy：不依賴單層防護（假設每層可能失敗）
  3. Feedback Loop：Layer 4 監控數據回饋至 Layer 1 設計（持續演化）

3.2 Permission Boundaries 實施

def validate_command(cmd: str, permissions: dict) -> tuple[bool, str]:
    # 檢查禁止命令模式
    for pattern in permissions["command_restrictions"]["forbidden"]:
        if re.search(pattern, cmd):
            return False, f"Forbidden command pattern: {pattern}"

    # 檢查檔案訪問
    if "rm" in cmd or ">" in cmd:
        for path in extract_paths(cmd):
            if not is_path_allowed(path, permissions["file_access"]):
                return False, f"Path not writable: {path}"

    return True, "OK"

3.3 Architectural Constraints

範例 2：依賴規則（基於 Google Cloud Architecture Patterns）


# .harness/architecture.py
DEPENDENCY_RULES = {
    "prompts/": ["config/", "templates/"],       # prompts 只能依賴 config + templates
    "hooks/": ["tools/", "config/"],             # hooks 可用 tools + config
    "tools/": ["config/"],                       # tools 只能依賴 config（無橫向依賴）
    "config/": [],                               # config 無依賴（基礎層）
}

def check_import(file_path: str, import_path: str) -> bool:
    file_layer = get_layer(file_path)
    import_layer = get_layer(import_path)

    allowed_deps = DEPENDENCY_RULES.get(file_layer, [])
    if import_layer not in allowed_deps:
        raise ArchitectureViolation(
            f"{file_path} cannot import from {import_path} "
            f"(layer {file_layer} → {import_layer} not allowed)"
        )

四、Quality Gates 與測試框架

4.1 Regression Gates（迴歸閘門）

核心概念：部署前自動執行評估套件，低於閾值則阻擋發布。

實施層級：

| Gate 類型 | 觸發時機 | 閾值範例 | 工具 |

|-----------|----------|----------|------|

| Unit Regression | Git pre-commit | Coverage ≥80%、Mutation Score ≥60% | pytest + mutmut |

| Integration Regression | PR merge | E2E success rate ≥95% | DeepEval + LangSmith |

| Performance Regression | Staging deployment | P95 latency ≤ baseline × 1.1 | Braintrust |

| Quality Regression | Production canary | User satisfaction ≥4.0/5 | Maxim AI |

範例配置（.harness/quality-gates.yaml）：


gates:

      coverage: 0.80
      mutation_score: 0.60

4.2 Agentic Testing 系統

2025-2026 最大轉變：從 AI-assisted testing（人類設計測試，AI 輔助執行）到 AI-agentic testing（AI 自主觀察、推理、生成、執行、分析測試）。

Agentic Testing 工作流程：


1. Observe Application
   ↓ (agent explores UI/API, builds behavior model)
2. Reason About Coverage
   ↓ (identifies untested paths, edge cases, failure modes)
3. Generate Test Cases
   ↓ (creates test scripts with assertions)
4. Execute Tests
   ↓ (runs in sandbox, captures results)
5. Analyze Results
   ↓ (classifies failures, suggests fixes)
6. Report Findings
   ↓ (generates human-readable summary)
7. Iterate
   └─→ back to step 1 (minimal human direction per cycle)

商用工具：

  • Mabl：錄製使用者流程 → 自動生成 E2E tests → 自適應維護（UI 變更時自動更新選擇器）
  • Testim：視覺 AI 識別元素（不依賴脆弱的 CSS selectors）
  • BlinqIO：自然語言測試規範 → 自動生成 Playwright 腳本
  • Parasoft：API contract testing + mutation testing

開源框架：

  • Browser Use（Python）：基於 LangChain 的瀏覽器自動化 agent
  • Stagehand（TypeScript）：支援 Playwright 的 agentic test generator

4.3 多層評估架構

根據 Braintrust、Galileo、Maxim AI 的 2026 年實踐，agent 評估應分層進行：

Layer 1: Individual Agent Evals

  • 輸入：well-formed inputs（正常格式、完整欄位）
  • 評估：單一 agent 的推理品質、工具選擇準確度
  • 範例：research-agent 收到合法查詢 → 評估檢索品質、摘要準確度

Layer 2: Interface Contract Tests

  • 驗證：agent A 的輸出 schema 符合 agent B 的輸入 schema
  • 範例：fetcher 輸出 {results: [...], metadata: {...}} → analyst 必須能解析此格式

Layer 3: End-to-End Pipeline Evals

  • 衡量：最終輸出品質（task completion、user satisfaction）
  • 捕捉：中間輸出（用於除錯瓶頸）
  • 範例：fetch → analyze → assemble 完整流程 → 評估最終摘要品質

Layer 4: Adversarial Injection Tests

  • 注入：故意損壞的中間輸出（缺失欄位、格式錯誤、惡意內容）
  • 評估：下游 agent 的 graceful degradation（是否崩潰 vs 返回錯誤訊息）
  • 範例：analyst 收到空 results → 應返回 {"status": "no_data", "error": "..."} 而非拋出異常

六、Daily Digest Prompt 專案優化方案（P0-P2）

6.1 P0：Separated Evaluator Pattern 實施（可靠性 +40%）

目標：建立獨立評估器目錄，將測試邏輯從 prompt 中解耦。

實施步驟：


1. 建立目錄結構：
   prompts/evaluator/
   ├── research_task_evaluator.py
   ├── todoist_task_evaluator.py
   ├── system_audit_evaluator.py
   └── base_evaluator.py

2. 定義評估器介面：
   class BaseEvaluator:
       def evaluate_task_completion(self, output: dict) -> float
       def evaluate_quality(self, output: dict) -> dict
       def evaluate_safety(self, output: dict) -> bool

3. 實施具體評估器：
   # prompts/evaluator/research_task_evaluator.py
   class ResearchTaskEvaluator(BaseEvaluator):
       def evaluate_task_completion(self, output: dict) -> float:
           # 檢查：是否有 kb_note_id？source ≥10？key_findings 非空？
           score = 0.0
           if output.get("kb_note_id"): score += 0.4
           if output.get("sources_count", 0) >= 10: score += 0.3
           if len(output.get("key_findings", "")) > 100: score += 0.3
           return score

       def evaluate_quality(self, output: dict) -> dict:
           # 語義相似度、hallucination 檢查、citation 準確度
           from deepeval.metrics import HallucinationMetric, AnswerRelevancyMetric
           # ... (詳細實施)
           return {"hallucination_score": 0.05, "relevancy": 0.92}

4. 整合至現有工作流程：
   # run-todoist-agent-team.ps1 Phase 3 插入評估步驟
   $evalResult = uv run python prompts/evaluator/research_task_evaluator.py `
       --result-file results/todoist-auto-research.json

   if ($evalResult.task_completion -lt 0.7) {
       Write-Warning "Task completion low: $($evalResult.task_completion)"
       # 觸發 ntfy 告警 + 記錄至 state/quality-issues.jsonl
   }

預期成果：

  • ✅ 測試覆蓋率從 0% → 60%（三大任務類型）
  • ✅ False positive rate ↓ 50%（語義相似度取代 exact match）
  • ✅ 可除錯性 ↑（評估分數分解為 sub-metrics）

6.2 P1：Quality Gates + Regression Testing（維護成本 -40%）

目標：建立自動化品質閘門，阻止低品質變更進入 main 分支。

實施步驟：


# .harness/quality-gates.yaml
gates:
  prompt_change:
    trigger: "prompts/**/*.md modified"
    tests:
      - name: "Prompt Consistency Check"
        command: "uv run python tools/validate_prompt_consistency.py"
        threshold: {consistency_score: 0.85}

      - name: "Token Efficiency Regression"
        command: "uv run python tools/token_efficiency_checker.py --baseline state/token-budget-state.json"
        threshold: {tokens_per_task_delta: "+10%"}  # 不得增加 >10%

    block_on_failure: true

  config_change:
    trigger: "config/**/*.yaml modified"

def relay_to_backend_with_trace(task_type: str, prompt: str, **kwargs):
    trace = langfuse.trace(name=f"{task_type}_request")
    generation = trace.generation(
        name="llm_call",
        model=kwargs.get("model"),
        input=prompt,
        metadata={"task_type": task_type, "backend": kwargs.get("backend")}
    )

    # 原有 relay 邏輯
    response = _relay_impl(task_type, prompt, **kwargs)

    generation.end(
        output=response.get("content"),
        usage={
            "prompt_tokens": response.get("usage", {}).get("prompt_tokens"),
            "completion_tokens": response.get("usage", {}).get("completion_tokens")
        }
    )

    return response

十、風險評估與緩解策略

風險 1：Langfuse 部署複雜度

風險：PostgreSQL + Docker 配置可能失敗（Windows 環境）

機率：Medium

影響：High（阻擋 P2 實施）

緩解策略：

  • ✅ Plan B：使用 Langfuse Cloud（SaaS 版本，30 天試用）
  • ✅ Documentation：準備詳細 troubleshooting guide（常見 Docker 問題）
  • ✅ Rollback：保留現有 log-based observability（不完全依賴 Langfuse）

風險 2：Agentic Testing 穩定性

風險：Browser Use agent 可能產生不穩定測試（flaky tests）

機率：High

影響：Medium（降低測試信心度）

緩解策略：

  • ✅ 人類審核：自動生成的測試需經人類 approve 才合併
  • ✅ Flaky Test Detection：追蹤測試通過率，<95% 者標記為 flaky
  • ✅ 漸進式採用：先從 10 個測試開始，驗證穩定後再擴展

風險 3：Quality Gates 過嚴格

風險：阻擋合理變更（false negatives）

機率：Medium

影響：Medium（降低開發速度）

緩解策略：

  • ✅ Bypass 機制：提供 --skip-gates flag（需留下 audit log）
  • ✅ 閾值調整：前兩週監控模式（warn only, not block）
  • ✅ Feedback Loop：收集 false negative 案例 → 調整閾值

  • 排名：Top 30 → Top 5
  • 模型：未變（僅 Harness 改進）

背景：驗證 AI 能否生成並維護 100 萬行生產級代碼

Harness 設計：

  1. Constraint System：dependency rules + architectural boundaries
  2. Verification Loops：type checker + linter + test suite（自動執行）
  3. Feedback Mechanisms：CI failure → auto-retry with error context
  4. Observability：追蹤每個 LLM 呼叫的 cost + latency + quality

成果：

  • 成功生成：1,000,000+ 行代碼（Python + TypeScript）
  • Bug Rate：與人類開發者相當（經 code review 驗證）
  • 結論：Harness Engineering 是大規模 AI coding 的關鍵

案例 3：Microsoft Azure SRE（800× 加速）

  1. Agentic Troubleshooting：agent 自動收集 logs、metrics、traces
  2. Root Cause Analysis：LLM 分析異常模式 + 提出假設
  3. Verification：自動執行診斷腳本（ping、telnet、log grep）
  4. Human-in-the-Loop：agent 提出修復建議 → 人類批准執行

十二、結論與下一步

核心要點回顧

  1. Separated Evaluator Pattern 是 2026 年 AI agent 測試的標準架構（解耦測試案例與評估邏輯）
  2. Plan-Execute-Verify Pattern 證明 Harness 比模型更重要（LangChain +13.7% accuracy without model change）
  3. Constraint Harnesses 採用四層防禦（prevention → validation → verification → monitoring）
  4. Agentic Testing 取代 AI-assisted testing（自主觀察、生成、執行、分析測試）
  5. Observability 整合成熟（Braintrust、Maxim AI、Langfuse、LangSmith、Arize Phoenix 五大平台）
  6. 81% agents in production, but only 14.4% secure → 系統化 Harness 實施是 2026 年核心挑戰
