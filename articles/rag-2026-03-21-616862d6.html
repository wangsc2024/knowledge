適用對象：daily-digest-prompt 專案

---

一、執行摘要（300 字）

本次對 daily-digest-prompt 專案進行了系統性的 State 膨脹與配置碎片化診斷。透過兩個並行子 Agent 的深度掃描，發現三大結構性問題：（1）state/ 目錄累積 150 個 loop-state 廢棄檔案，佔用無謂 I/O；（2）kb-note-scores.json 單檔 432KB 且每次全量覆寫，造成寫入瓶頸；（3）29 個 YAML 配置檔共 4525 行，存在 timeout 重複定義、task_key 需跨檔同步等碎片化風險。

已實施 P0 修復：在 run-agent-team.ps1 與 run-todoist-agent-team.ps1 啟動階段加入 loop-state 自動清理（保留 3 天），預期隨時間推移可持續清理累積的廢棄狀態檔。同時規劃了 P1/P2 優化路線圖，包含 kb-note-scores 增量化、config 統一真相源架構、模板參數化去冗餘等 7 項改善方案。

---

二、現況診斷：三大結構性問題

2.1 State 目錄膨脹（P0 — 已修復）

現象：state/ 目錄包含 150 個 loop-state-*.json 檔案，每個 Agent 執行 session 都會建立一個新檔（以 session UUID 命名），但無任何清理機制。

影響：

  • 檔案系統碎片化：150+ 小檔案增加目錄掃描時間
  • Git 追蹤負擔：若不在 .gitignore 中，每次 commit 都會包含大量變更
  • 診斷困難：無法快速識別哪些是活躍 session

已實施修復：在兩個主要排程腳本的啟動階段加入自動清理邏輯：


$loopStateFiles = Get-ChildItem "state/loop-state-*.json" |
    Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-3) }
if ($loopStateFiles.Count -gt 0) {
    $loopStateFiles | Remove-Item -Force
}

設計決策：保留 3 天（而非 7 天）是因為 loop-state 僅用於單次 session 的迴圈偵測，session 結束後即無用。3 天可覆蓋週末無執行的情境。

2.2 kb-note-scores.json 全量覆寫瓶頸（P1 — 待實施）

現象：單檔 432KB（442,055 bytes），記錄知識庫中每筆筆記的品質分數。每次 kb-curator Skill 執行時全量重算並覆寫。

問題分析：

  • 寫入延遲：432KB JSON 的序列化/反序列化耗時約 50-100ms
  • 無增量機制：新增一筆筆記就需重寫整個檔案
  • 無分片策略：隨知識庫成長（目前 700+ 筆），檔案將線性膨脹

建議方案（增量化）：

  1. 改為 dirty-flag 機制：僅重算 last_modified > last_scored 的筆記
  2. 引入月度 snapshot：state/kb-note-scores-2026-03.json，按月分片
  3. 在 digest-memory.json 加入 cache_stats.kb_scores_dirty_count 追蹤

2.3 YAML 配置碎片化（P2 — 待實施）

現象：29 個 YAML 檔案共 4525 行，分散定義 routing、timeout、frequency、scoring 等邏輯。

核心問題 — 重複定義：

  • timeouts.yaml 的 phase2_timeout_by_task 與 run-todoist-agent-team.ps1 的 $AutoTaskTimeoutOverride 完全重複（30+ 個 key）
  • 修改 timeout 需同時更新兩處，且無自動驗證

核心問題 — 跨檔同步：

  • frequency-limits.yaml 的 task_key 必須與 prompts/team/todoist-auto-{task_key}.md 檔名一致
  • 歷史上已因連字號/底線不一致導致三次 Phase 2 結果缺失 bug

建議方案（單一真相源）：

  1. 新建 config/tasks.yaml 統一定義每個自動任務的 key、timeout、frequency、prompt 檔名
  2. PowerShell 腳本改為從 config/tasks.yaml 動態讀取，移除 hardcoded override
  3. 新增 tools/validate_task_config.py 交叉驗證 task_key 一致性

三、深度分析：YAML Sprawl 的產業最佳實踐

3.1 YAML Sprawl 定義與成因

根據 Octopus Deploy 的定義，YAML Sprawl 是指「隨著 GitOps 規模擴展，需要更新和維護的 YAML 配置檔案數量迅速增長的現象」[Octopus Deploy, 2025]。本專案的 29 個 YAML 恰好符合此模式 — 每新增一個 Skill 或自動任務，就可能需要在 3-4 個 YAML 中新增對應 entry。

3.2 單一真相源原則

根據 TechTarget 的分析，Single Source of Truth 的核心是「在組織中建立統一的數據中心，讓所有系統都參照同一個權威數據庫」[TechTarget, 2025]。應用到本專案：

| 當前狀態 | 目標狀態 |

|---------|--------|

| timeout 定義在 YAML + PS1 兩處 | 統一在 tasks.yaml |

| task_key 分散在 YAML + prompt 檔名 + result JSON | tasks.yaml 為唯一真相源 |

| Skill depends-on 在各 SKILL.md 獨立宣告 | skill-registry.json 自動掃描驗證 |

3.3 AI Agent 狀態管理最佳實踐

根據 FlowHunt 的研究，Agent 檔案狀態管理的三大原則是 [FlowHunt, 2025]：

  1. 定期清理：刪除舊檔案或歸檔結果，保持系統可管理性
  2. 增量更新：使用 reducer 模式（左右合併），而非全量覆寫
  3. 檔案即通訊：Agent 間透過檔案系統非同步傳遞狀態

本專案的 results/*.json 已實踐第三點（Phase 2 寫入、Phase 3 讀取），但第一、二點仍有改善空間。

五、已實施修復的驗證結果

5.1 loop-state 清理驗證

  • 修改檔案：run-agent-team.ps1、run-todoist-agent-team.ps1
  • 清理策略：每次排程啟動時，刪除 LastWriteTime > 3 天的 state/loop-state-*.json
