KB 現況與本次缺口

知識庫先前已涵蓋 AI Code Review 的 foundation、mechanism、application 與 optimization：包含 diff-aware review、CodeQL/Semgrep/SARIF 管線、AI auto-triage、誤報治理與 Autofix 驗證閉環。本次未重複這些內容，而是補上 synthesis 階段：如何把 LLM review、SAST/SARIF、安全修補建議、需求規格、ADR 與 branch protection 串成同一條治理流程。

技術概述

2026 年的 AI 程式碼審查實務，已不再只是「讓模型在 PR 留評論」。真正可落地的做法是三層架構：第一層由 PR review agent 做摘要、風險提示與風格檢查；第二層由 CodeQL、Semgrep 等工具輸出 SARIF，提供可機器排序的安全訊號；第三層才讓 Autofix 或建議型 AI 產生修補方案，再交給 CI、重新掃描與人工核准做最後把關。GitHub 的 Copilot Autofix 已直接建立在 code scanning 結果之上；Google Gemini Code Assist 則把 PR 摘要、嚴重度分級與 repo 級設定做成 GitHub review agent；Qodo Merge 進一步展示了靜態分析、元件級互動與 suggestion validation 的產品形態。綜合來看，最佳實踐是把 AI review 視為「治理編排器」，而非單獨的審查員。

核心方法與工具

1. Review Agent 層

  • Gemini Code Assist on GitHub

- 新 PR 開啟時可自動產生摘要與 review。

- 支援 LOW 到 CRITICAL 的 review comment 嚴重度門檻。

- 可用 .gemini/config.yaml、.gemini/styleguide.md 做 repo 級規則管理。

  • Qodo Merge / PR-Agent

- 對 PR 變更做 component-level 靜態分析。

- 可互動觸發 /analyze、/test、/improve_component。

- 對可直接套用的建議做定位、語法有效性與 self-reflection 驗證。

2. Security Signal 層

  • GitHub Code Scanning + SARIF

- 用 upload-sarif 匯入第三方掃描結果。

- 透過 category 分開 CodeQL、Semgrep、secret 掃描或 monorepo 子系統結果。

- 利用同一 commit 下多個 category 形成單一 PR 風險面板。

  • CodeQL / Semgrep / Secret scanning

- CodeQL 提供高可信度資料流與語意規則。

- Semgrep 補充快速規則與 framework-specific policy。

- secrets 與 supply-chain 掃描作為阻斷式 gate，避免 review agent 掩蓋高風險項目。

3. Fix & Governance 層

  • Copilot Autofix

- 依 code scanning alert 與 code context 產出修補建議。

- 適合做「候選修補」，不適合作為無條件自動提交。

  • 需求與 ADR 對齊

- 把使用者故事、驗收條件、ADR 摘要放進 PR template 或 review prompt。

- 讓 AI review 從「語法/風格」提升到「是否違反設計決策與產品意圖」。

實際工作流程（step-by-step）

工作流 A：PR 全鏈路治理

  1. 開發者建立 PR，PR template 強制填入需求連結、驗收條件、相關 ADR。
  2. Gemini 或同類 review agent 先產生 PR summary，做 correctness / maintainability / security 的第一輪檢查。
  3. GitHub Actions 執行 CodeQL、Semgrep、secret scan，將結果以不同 category 上傳 SARIF。
  4. Branch protection 規則只對「阻斷式檢查」生效，例如高嚴重度安全告警、測試失敗、schema 驗證失敗。
  5. AI review comment 僅作 advisory；若對應到 SARIF 高風險告警，才升級成人工必看項目。
  6. 對可修補的 CodeQL 告警，由 Copilot Autofix 產出修補候選。
  7. 套用候選修補後，重新跑單元測試、lint、掃描與必要的語意回歸測試。
  8. Reviewer 依需求/ADR/測試結果決定 merge；若 AI 建議與 ADR 衝突，ADR 優先。
  9. 合併後把被接受與被駁回的 AI 建議回寫 triage memory，作為下一輪 prompt / style guide 調整依據。

工作流 B：風險分級規則

  1. CRITICAL/HIGH 安全告警：阻斷 merge，必須人工核准。
  2. MEDIUM 安全告警：允許修補建議，但必須重新掃描通過。
  3. LOW 或純風格建議：不阻斷，只做 reviewer queue 排序。
  4. 與 ADR 衝突的建議：即使低風險，也要求人工覆核。
  5. .github/workflows/**、部署腳本、權限設定等高風險檔案：禁用自動修補直送主分支。

程式碼 / 配置範例

1. GitHub Actions：多來源 SARIF 匯入


name: pr-ai-governance

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  security-signals:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      security-events: write
      actions: read
    steps:
      - uses: actions/checkout@v5

      - name: Run Semgrep
        run: semgrep scan --config auto --sarif --output semgrep.sarif

      - name: Upload Semgrep SARIF
        uses: github/codeql-action/upload-sarif@v4
        with:
          sarif_file: semgrep.sarif
          category: semgrep-app

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v4
        with:
          languages: python,javascript

      - name: Autobuild
        uses: github/codeql-action/autobuild@v4

      - name: Analyze
        uses: github/codeql-action/analyze@v4
        with:
          category: codeql-core

效果量化

  • GitHub 官方說明目前用 超過 2,300 個有測試覆蓋的 alert 持續評估 Copilot Autofix 品質；對許多測試案例，建議可在不修改下直接提交且 CI 仍通過。這代表 Autofix 已具備「可驗證候選修補」水準，但仍不是 100% 保證。
  • Gemini Code Assist on GitHub 的官方配額在 2026-03-23 文件中列為：consumer 版 33 PR/日、enterprise 版 100+ PR/日。這比較接近營運吞吐量指標，而非傳統生產力提升百分比。
  • Qodo Merge 官方揭露其靜態分析能力支援 8 種語言，且對可套用建議增加定位與語法有效性檢查。這代表產業趨勢正從「多給建議」轉向「只交付可驗證建議」。
  • 對於「開發速度提升 X%」這種泛用數字，官方文件目前多未提供可直接對比的嚴謹基準；因此在導入時應自行追蹤 review turnaround time、false-positive rate、accepted suggestion rate、reopened security bug rate。

與本專案的應用場景

daily-digest-prompt 最適合的落地方向不是導入單一 AI reviewer，而是建立「輕量但硬邊界清楚」的治理：

  1. 對 hooks/、tools/、config/.yaml、templates/* 建立不同 review policy。
  2. 把 results/*.json schema、registry/continuity 一致性、PowerShell shell safety 視為阻斷式規則。
  3. 把 AI comment 限定在 PR 摘要、規格對齊、漏測提醒與風險排序，不直接修改高風險腳本。
  4. 若未來上 GitHub Actions，可把 Semgrep/secret scanning 匯成 SARIF，再讓 Code Scanning 做單一入口。
  5. 對自動任務 prompt、workflow、hook 規則新增 ADR 摘要，讓 AI reviewer 能判斷「是否違反既有治理原則」。

風險與注意事項

  • 不要把 AI review 當單一真相來源：GitHub 也明示 Autofix 是 best-effort，存在非決定性與上下文截斷限制。
  • 高風險檔案應有白名單保護：Google 官方直接排除 .github/workflows 的 review suggestion，這是合理的安全邊界，可類推到本專案的 hooks 與部署腳本。
  • 多工具結果必須分 category：否則同 commit 的後上傳 SARIF 可能覆蓋先前結果，破壞 triage。
  • 建議先 advisory、後 blocking：先累積 accepted rate 與誤報率，再決定哪些規則能升級為 merge gate。
  • 記得管理忽略清單與 context：不該讓 vendor、build output、產物檔把 review 注意力稀釋。
