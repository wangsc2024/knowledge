RAG 文件匯入品質治理：閱讀順序、表格保真、OCR 與 chunking-ready Markdown 的工程選型

技術概述

在 RAG 管線裡，文件「能轉成文字」不等於「適合進知識庫」。真正影響檢索品質的是四件事：閱讀順序是否被破壞、表格是否失真、OCR 是否把噪音放大、輸出是否能直接支援 chunking 與 metadata。今天常見路線可粗分為兩類：Markdown-first（MarkItDown）與 structure-native（Docling、Unstructured；Tika 則偏 text+metadata 抽取層）。前者部署輕、適合一般 Office/HTML；後者在複雜 PDF、表格與版面語義更強，但計算與整合成本也更高 [來源：Microsoft MarkItDown README, 2026；Docling Documentation, 2026；Unstructured Partitioning, 2026；Apache Tika, 2026]。

三角佐證核心主張

主張 1：文件匯入器分成「Markdown-first」與「structure-native」兩大類；選錯類型會直接影響下游 chunking 與 grounding

MarkItDown 的官方定位很清楚：把多種文件快速轉成 Markdown，供 LLM 與文字分析管線使用，並且明說它不是面向人類閱讀的高保真重排工具 [來源：Microsoft MarkItDown README, 2026]。Docling 則用 DoclingDocument 保存正文、頁首頁腳、版面框、表格、圖片與 provenance，再依需求輸出 Markdown、DocTags 或 JSON [來源：Docling Documentation, 2026；Docling Document, 2026]。Unstructured 的 partition 會先做檔案型別 routing，再輸出帶元素與 metadata 的結構化結果；若要更細控制，官方建議直接使用型別專用 partition 函式 [來源：Unstructured Partitioning, 2026]。Apache Tika 則用單一介面抽取文字與 metadata，覆蓋格式最廣，但並不直接提供 RAG-ready 的 chunk abstraction [來源：Apache Tika, 2026]。

已驗證✅（4 個獨立來源）

主張 2：閱讀順序、表格保真與 OCR 邊界，是 parser 選型的主戰場；複雜 PDF 不能只看「能不能轉」

Docling 官方文件把「page layout、reading order、table structure」列為核心能力，且 DoclingDocument 會把 reading order 保存在 body tree 中 [來源：Docling Documentation, 2026；Docling Document, 2026]。MarkItDown OCR plugin 會把 PDF 內嵌圖像依位置插回垂直閱讀順序，掃描 PDF 則以 300 DPI 整頁 OCR 補救；但 Excel 圖像會附在資料表後，而非真正回插到列中 [來源：MarkItDown OCR Plugin README, 2026]。Unstructured 明確指出：PDF 表格抽取目前要靠 hi_res，而 hi_res 在多欄文件上可能出現排序問題；這種情況反而要改走 ocr_only [來源：Unstructured Partitioning, 2026；Unstructured Table Extraction from PDF, 2026]。Tika 官方則把 PDF OCR 分成「抽 inline images」與「整頁 render 後 OCR」兩路，並直接說明哪一種較好要視文件而定 [來源：TikaOCR, 2025；Apache Tika PDFParser, 2025]。

已驗證✅（6 個獨立來源）

主張 3：chunking-ready 不只是輸出 Markdown；真正關鍵是能否保留結構、表格上下文與 provenance

Docling 的 chunking 文件把做法分成兩條：先匯出 Markdown 再做後處理，或直接對 DoclingDocument 做 native chunking。官方進一步提供 repeat_table_header=True 來讓跨 chunk 的表格保留欄位上下文，並在 LangChain 範例中把 ExportType.DOC_CHUNKS 設成預設 [來源：Docling Chunking, 2026；Docling RAG with LangChain, 2026]。MarkItDown 的優勢是 Markdown 天生貼近純文字、對 LLM token 友善；但 chunking 規則仍要在下游自己補上 [來源：Microsoft MarkItDown README, 2026]。Unstructured 會輸出元素與 page breaks，適合在 chunk 前先做 element-level 過濾與重組 [來源：Unstructured Partitioning, 2026]。Tika 抽的是 text+metadata，若要保留結構邊界，仍需額外設計 chunker 與欄位映射 [來源：Apache Tika, 2026]。

已驗證✅（5 個獨立來源）

主張 4：對本專案最實際的策略不是單一工具，而是「routing + validation + fallback」

以下為根據多家官方文件推導出的工程結論，不是任何單一產品官方的原句建議。若檔案是一般 Office、HTML、CSV、JSON、email，而且主要目標是快速做 Markdown-based RAG 匯入，MarkItDown 的成本最低 [來源：Microsoft MarkItDown README, 2026]。若是複雜 PDF、掃描件、表格密集、需要 reading order 與 provenance，Docling 更適合做主路徑；若偏好 element API 與策略切換，Unstructured 也可勝任，但必須顯式指定 fast / hi_res / ocr_only，不能完全依賴預設 auto [來源：Docling Documentation, 2026；Docling Chunking, 2026；Unstructured Partitioning, 2026；Unstructured Partitioning Strategies, 2026]。若需求是超廣格式覆蓋、metadata 抽取、JVM sidecar 或 server 端統一入口，Tika 才是更合理的底層抽取層 [來源：Apache Tika, 2026；TikaOCR, 2025]。

推論✅（6 個獨立來源）

工具選型矩陣

| 工具 | 最佳場景 | 強項 | 主要限制 |

| --- | --- | --- | --- |

| MarkItDown | DOCX / PPTX / XLSX / HTML / extractable PDF 的快速 Markdown 匯入 | Python 輕量、Markdown token 友善、可加 OCR plugin、可接 Azure Document Intelligence | 官方明說不追求人類高保真；OCR plugin 依賴 LLM client；複雜版面能力有限 |

| Docling | 版面複雜、表格多、要保留 reading order / provenance 的 PDF 與混合文件 | layout-aware、表格結構、DoclingDocument、native chunking、DOC_CHUNKS | 成本較高；需要理解其中介格式與 chunking 模型 |

| Unstructured | 要元素化輸出、需要 fast/hi_res/ocr_only 策略切換、或已有其 API/SDK 生態 | partition routing、element metadata、table extraction、可本地或遠端 inference | hi_res 多欄排序有已知限制；表格要顯式開 hi_res |

| Apache Tika | 超廣格式覆蓋、metadata-first、JVM / server sidecar | 1000+ 格式、單一介面、可用 header 做 per-request OCR 配置 | 不是 Markdown-first；chunking 與結構治理需自行補 |

最佳實踐與陷阱

  1. 先做 routing，再做 conversion。至少要先判斷：是否 extractable text、是否掃描件、是否多欄、是否表格密集、是否需要 metadata。
  2. 對表格與多欄 PDF，不要只看轉換成功率。真正該驗的是欄位是否保留、閱讀順序是否錯位、chunk 後是否還看得懂表頭。
  3. 若用 MarkItDown OCR plugin，必須加硬性檢查 llm_client / llm_model。官方明說沒有 client 時 plugin 會載入，但 OCR 會被靜默跳過 [來源：Microsoft MarkItDown README, 2026]。
  4. 若用 Unstructured 做 PDF 表格抽取，應顯式設定 strategy="hi_res"，且把 skip_infer_table_types=False；否則表格能力不會如預期生效 [來源：Unstructured Table Extraction from PDF, 2026]。
  5. 若文件是多欄掃描 PDF，Unstructured 官方反而建議改用 ocr_only，因為 hi_res 可能排序錯亂 [來源：Unstructured Partitioning, 2026]。
  6. 若用 Tika，應把 OCR 當成可配置策略而不是固定開關。官方文件提供 per-request header 與 tika-config.xml 兩種方式；這很適合做 sidecar 服務，但也代表你要自己治理效能與設定漂移 [來源：TikaOCR, 2025；Apache Tika PDFParser, 2025]。
  7. 若用 Docling，建議直接用 native chunker 或 DOC_CHUNKS，不要只匯出 Markdown 再粗暴切段；尤其表格跨 chunk 時，repeat_table_header 非常有價值 [來源：Docling Chunking, 2026；Docling RAG with LangChain, 2026]。

與本專案關聯

本專案的 KB 匯入 API 以 Markdown contentText 為主要輸入，因此最務實的做法不是全面改成單一 parser，而是建立三段式管線：

  1. 預設路徑：一般 Office / HTML / CSV / JSON 先走 MarkItDown，快速產出 Markdown。
  2. 複雜 PDF 路徑：若偵測到掃描件、多欄、表格密集或抽樣驗證失敗，就改派 Docling；若團隊偏好 element API，則改派 Unstructured 並強制指定策略。
  3. 企業 / sidecar 路徑：若後續出現大量異質格式、metadata 要求或 JVM sidecar 需求，再加 Tika，不必一開始就把 Tika 放進主路徑。

我建議在匯入前新增一個輕量品質閘門，至少記錄這些欄位：parser_used、ocr_used、ocr_strategy、table_detected、reading_order_risk、chunking_mode、source_pages。這樣檢索品質出問題時，才能回溯是 conversion 還是 embedding / chunking 的問題。

程式碼範例


from dataclasses import dataclass

@dataclass
class DocProfile:
    suffix: str
    extractable_text: bool
    scanned: bool
    multi_column: bool
    table_heavy: bool
    needs_metadata: bool


def choose_ingestion_route(p: DocProfile) -> tuple[str, dict]:
    if p.needs_metadata:
        return "tika", {"ocrStrategy": "auto" if p.extractable_text else "ocr_only"}

    if p.suffix == ".pdf" and (p.scanned or p.table_heavy or p.multi_column):
        if p.multi_column and not p.extractable_text:
            return "unstructured", {"strategy": "ocr_only"}
        return "docling", {"export_type": "DOC_CHUNKS"}

    return "markitdown", {"use_plugins": p.scanned}


def validate_conversion(markdown_text: str, parser_used: str) -> dict:
    return {
        "parser_used": parser_used,
        "has_markdown_headings": "#" in markdown_text,
        "has_markdown_tables": "|" in markdown_text,
        "ocr_blocks_present": "[Image OCR]" in markdown_text,
        "needs_manual_review": len(markdown_text.strip()) == 0,
    }

結論

這次研究最重要的結論不是「哪個工具最好」，而是「哪種文件該用哪種抽取假設」。MarkItDown 適合作為本專案的預設匯入器；Docling 適合作為複雜 PDF / 表格 / 高保真結構保留路徑；Unstructured 適合作為 element-first 與策略可切換路徑；Tika 則更像廣格式 metadata 抽取 sidecar。若把選型邏輯前移，並在匯入前加一層品質驗證，本專案的 KB 檢索品質會比單純換 parser 更穩定。
