GitHub Scout：自癒系統 (Self-Healing Systems) 2026-07-02

專案觀測摘要

  1. keephq/keep（11997★，pushed 2026-07-01T06:37:53Z）— 開源 AIOps 與告警管理平台。核心模式：宣告式 YAML workflow catalog（alert 條件 → action），跨監控來源的告警關聯與 fingerprint 去重，50+ 整合。
  2. k8sgpt-ai/k8sgpt（7938★，pushed 2026-07-01T23:53:26Z）— Kubernetes 診斷與可選自動修復。核心模式：pluggable analyzer 產出結構化 finding（含信心指標），再依 finding 選擇 remediation workflow，而非一步到位 LLM 決策。
  3. AgentWrapper/agent-orchestrator（7857★，pushed 2026-07-01T16:26:00Z，經 ComposioHQ/agent-orchestrator 301 重導向確認 canonical repo）— 平行 coding agent 編排器。核心模式：每個任務獨立 git worktree + branch + PR，CI 失敗時 agent 在該隔離 worktree 內自動修復再送審，而非直接改動共用分支。

改進提案

  • P1（keep 借鑑）：backlog 告警關聯合併——為 improvement-backlog 補上弱關聯分組，避免單一根因（如過往 cursor 配額耗盡）拆成多筆孤立 backlog 項目。已產出計畫 docs/plans/github-scout-20260702-backlog-alert-correlation.md，因觸及 Pattern B SSOT tools/backlog_writer.py，未直接落實，待人工審核。
  • P1（k8sgpt 借鑑，暫緩）：diagnosis confidence 門檻———k8sgpt 的 analyzer confidence → remediation catalog 模式具參考價值，但 grep 確認本系統 tools/failure_recovery_router.py 與 config/failure-recovery-matrix.yaml 目前無 confidence 欄位，也無任何 RCA producer 提供信心分數。在無上游 producer 前引入 consumer 端門檻屬過早抽象，違反 YAGNI，本輪不落實，需先由人工評估是否在 RCA why-chain 補上 confidence 產出。
  • P1（三案綜合，已落實）：發現 config/github-scout-review-policy.yaml 的 topic_overrides 涵蓋 9 個研究主題，但 SKILL.md 步驟1 列出的 12 大搜尋主題中，「自癒系統」是唯一缺乏 topic_override 的主題，導致既有與未來 self-healing 相關研究缺乏結構化必填欄位。已直接新增 self_healing_systems topic_override（diagnosis_model／remediation_selection_model／human_approval_gate 三個必填欄位），並與本系統既有 tools/failure_recovery_router.py、prompts/team/todoist-auto-self_heal.md 對照。
