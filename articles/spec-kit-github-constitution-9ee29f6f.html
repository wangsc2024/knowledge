一、專案概述

Spec Kit（github/spec-kit）是由 GitHub 官方團隊開發維護的開源規格驅動開發（Spec-Driven Development, SDD）工具集，旨在將傳統軟體工程的「先寫規格再寫程式碼」的理想，透過 AI Agent 的能力真正落地為可執行的開發流程。截至 2026 年 3 月已累積 72,843 Stars、6,256 Forks，採用 MIT 授權，是目前 GitHub 上最具影響力的 AI 輔助開發方法論專案之一。

核心定位

Spec Kit 的核心哲學是「權力倒轉（Power Inversion）」——傳統開發中，規格服務於程式碼；SDD 中，程式碼服務於規格。規格不再是一次性的溝通文件，而是可執行的開發源頭，透過 AI 直接生成實作。這與本專案 daily-digest-prompt 的「文件驅動架構（Document-Driven Architecture）」理念高度契合，但 Spec Kit 將其推到了更高層次——不只是配置驅動，而是整個開發生命週期都由規格文件驅動。

二、核心技術架構分析

2.1 Constitution 治理模型（最高原則層）

Spec Kit 引入了一個極具創新性的概念：Constitution（憲章）。這是一份定義專案核心原則的文件（.specify/memory/constitution.md），包含：

  • 不可協商的原則（如 TDD 強制、Library-First 設計）
  • 技術標準（效能門檻、安全合規要求）
  • 治理規則（修改憲章需文件化、審批、遷移計畫）
  • 版本控制（Version + Ratified + Last Amended 三重時間戳）

關鍵設計：/speckit.analyze 命令在做一致性分析時，Constitution 具有不可協商的權威——如果規格、計畫或任務與 Constitution 衝突，分析結果自動標記為 CRITICAL。這比一般的 linting 規則強大得多，因為它建立了一個「原則優先」的決策鏈。

2.2 七階段 SDD 管線

Spec Kit 定義了一條嚴謹的七階段開發管線，每個階段對應一個 slash command：


Constitution → Specify → Clarify → Plan → Tasks → Analyze → Implement
（治理原則）  （需求定義）（釐清）  （技術方案）（任務分解）（一致性分析）（實作執行）

每個階段的設計要點：

  1. Constitution（/speckit.constitution）：建立不可變的治理基準線
  2. Specify（/speckit.specify）：從自然語言提取用戶故事，自動建立 Git 分支（數字前綴 + 短名稱），產出 spec.md
  3. Clarify（/speckit.clarify）：結構化的覆蓋率驅動質詢，在規劃之前消除歧義
  4. Plan（/speckit.plan）：生成技術方案（plan.md、data-model.md、api-spec.json、research.md），支援平行研究任務
  5. Tasks（/speckit.tasks）：從計畫生成有序任務清單，含依賴管理、並行標記 [P]、用戶故事分組
  6. Analyze（/speckit.analyze）：唯讀跨文件一致性分析，檢查規格 / 計畫 / 任務三者的覆蓋完整性
  7. Implement（/speckit.implement）：按任務清單執行，自動驗證前置條件（含 checklist 檢查）

關鍵洞察：這條管線的精妙之處在於漸進式揭露（Progressive Disclosure）——每個階段只載入當前需要的上下文，不預載未來階段的資訊，有效控制 LLM 的 context window 消耗。

2.3 Extension 插件架構（模組化生態）

Spec Kit 於 2026 年 1 月發布了 Extension System RFC，這是一個完整的插件生態設計：

架構核心：

  • 目錄結構：.specify/extensions/{name}/，每個 extension 含 extension.yml（宣告式 manifest）+ commands/ + scripts/
  • 命名慣例：speckit.{extension}.{command}（如 speckit.jira.sync）
  • 生命週期：Discovery → Installation → Validation → Registration → Execution
  • 安全設計：Manifest 驗證防惡意程式碼、擴充功能失敗隔離不影響核心、Fail-Safe 預設降級
  • Catalog：中央目錄（catalog.json + catalog.community.json）支援社群貢獻

設計原則：

  1. Convention Over Configuration（慣例優於配置）
  2. Fail-Safe Defaults（安全降級）
  3. Backward Compatibility（核心命令不受影響）
  4. Developer Experience（specify extension add jira 一行安裝）
  5. Security First（信任邊界明確）

2.4 跨 Agent 相容設計

Spec Kit 支援 20+ 個 AI coding agent，這是透過一個精心設計的抽象層實現的：

  • CLI 初始化：specify init --ai claude/gemini/copilot/cursor/... 生成對應的 agent 配置
  • Slash Command 模板化：每個 command 是一個 Markdown 檔案（含 YAML frontmatter），定義描述、腳本路徑、Handoff（Agent 間交接）
  • 腳本雙版本：每個自動化腳本同時提供 Bash（scripts/bash/）和 PowerShell（scripts/powershell/）版本
  • 通用模板系統：templates/ 目錄下的 spec/plan/tasks/constitution 模板對所有 Agent 通用
  • Generic Agent 支援：--ai generic --ai-commands-dir 允許整合任何未官方支援的 Agent

三、應用場景與生態系統

3.1 三種開發模式

| 模式 | 描述 | 場景 |

|------|------|------|

| 0-to-1 Greenfield | 從零開始生成完整應用 | 新專案啟動 |

| Creative Exploration | 平行探索多種技術棧實作 | 技術選型、原型驗證 |

| Iterative Brownfield | 漸進式增強既有系統 | 功能迭代、遺留系統現代化 |

3.2 生態系統整合

  • 專案管理：Jira / Linear / Azure DevOps Extension
  • 版本控制：自動 Git 分支命名（NNN-short-name）、PR 整合
  • 品質門檻：自動 Checklist 驗證（UX / 測試 / 安全）
  • 研究任務：/speckit.plan 階段可平行啟動研究 Agent 調查技術棧細節

四、可借鑑項目：對照 Daily-Digest-Prompt 系統的六大啟示

啟示 1：Constitution 治理模型 → 強化 Preamble 為「專案憲章」

Spec Kit 做法：constitution.md 是不可協商的最高原則，分析命令自動檢查合規。

本專案現狀：templates/shared/preamble.md 定義共用前言（nul 禁令 + Skill-First），但缺乏版本化與治理機制。

建議：

  • 將 preamble.md 升級為 constitution.md，加入版本號、批准日期、修訂歷程
  • Hook 系統（pre_bash_guard.py 等）與 Constitution 原則建立明確對應關係
  • 新增 /constitution.verify 自動化驗證命令，檢查所有 config/ YAML 是否與 Constitution 一致
  • 變更 Constitution 需經 ADR 流程（已有 arch-evolution Skill 可整合）

啟示 2：Analyze 唯讀一致性分析 → 強化 system-audit 的跨文件驗證

Spec Kit 做法：/speckit.analyze 是嚴格唯讀的跨文件一致性分析，建構語義模型後比對覆蓋率。

本專案現狀：system-audit Skill 做 7 維度 38 子項審查，但主要是逐檔案評分，缺少跨文件的「語義覆蓋率」概念。

建議：

  • 在 system-audit 中新增「配置一致性」子維度，自動比對 config/*.yaml 與 templates/ 的交叉引用完整性
  • 引入 Spec Kit 的「Requirements Inventory + Task Coverage Mapping」概念，為 frequency-limits.yaml 的 19 個任務建立需求追溯矩陣
  • 分析結果明確標記 CRITICAL / MAJOR / MINOR（而非單純分數），對應修正優先級

啟示 3：Extension 插件架構 → Skill 生態的模組化演進

Spec Kit 做法：Extension 系統有完整的 Manifest（extension.yml）、Catalog、安裝/移除命令、版本管理。

本專案現狀：24 個 Skill 以目錄形式管理，各含 SKILL.md，但缺乏標準化的 manifest schema、中央 catalog 與版本依賴管理。

建議：

  • 為每個 Skill 新增 skill.yml manifest（name / version / depends / triggers / allowed-tools / min-agent-version）
  • SKILL_INDEX.md 演進為 skill-catalog.json（結構化索引，LLM 可直接解析）
  • 引入 Skill 版本相容性檢查：升級 Skill 時自動驗證 config/ 依賴是否需要同步更新
  • 參考 Spec Kit 的 Fail-Safe 設計：Skill 載入失敗時 graceful degrade，不阻斷主流程

啟示 4：漸進式上下文揭露 → 優化 Prompt 的 Token 經濟學

Spec Kit 做法：每個階段只載入該階段需要的文件，/speckit.implement 才讀取 tasks.md，不在 /speckit.specify 階段預載。

本專案現狀：已有「按需載入」設計（子 Agent 模板、自動任務 prompt 只在觸發時 Read），但主 prompt 仍一次性載入較多配置。

建議：

  • 量化各 prompt 的 token 消耗，建立 Token Budget 基線（參考 antigravity-skills 的研究：80% 方差來自 Token 預算）
  • 對 hour-todoist-prompt.md（~140 行）實施分層載入：Phase 1 只載入路由規則，Phase 2 按路由結果載入對應模板
  • 在 config/pipeline.yaml 中為每個步驟標注 context_budget（最大 token 數），超出時自動摘要前序步驟

啟示 5：Handoff 機制 → 強化 Agent 間任務交接的結構化

Spec Kit 做法：每個 slash command 的 YAML frontmatter 包含 handoffs 欄位，明確定義下一步可交接的 Agent、初始 prompt 與是否自動發送。

本專案現狀：團隊模式透過 results/*.json 中間檔案交接，但交接格式未強制驗證，Phase 2/3 偶爾因命名不一致導致失敗（見 MEMORY.md 「結果檔案缺失根因」）。

建議：

  • 為 results/*.json 定義 JSON Schema（必含 plan_key、status、timestamp、output），Phase 2 啟動前自動驗證
  • 在每個 team prompt 的 YAML frontmatter 加入 handoff 欄位，明確聲明輸入/輸出依賴
  • 引入 Spec Kit 的「Script 前置檢查」模式：每個 Phase 啟動時先跑 check-prerequisites 確認所有前置檔案存在

啟示 6：Clarify 結構化釐清 → 強化研究任務的前置探索

Spec Kit 做法：/speckit.clarify 在規格定稿前進行結構化的覆蓋率驅動質詢，識別模糊區域。

本專案現狀：研究任務（templates/auto-tasks/ 下 10 個研究模板）直接進入研究階段，缺少前置的「問題空間界定」步驟。

建議：

  • 在研究模板中新增「Phase 0: 問題界定」步驟——先列出已知 / 未知 / 假設，再開始搜尋
  • 與 research-registry.json 整合：Phase 0 自動查詢近期同主題研究，避免重複探索已知領域
  • 研究品質閘門新增「覆蓋率評分」——不僅去重，還評估研究是否涵蓋所有關鍵面向

五、技術對比：Spec Kit vs Daily-Digest-Prompt

| 面向 | Spec Kit | Daily-Digest-Prompt | 差距與機會 |

|------|---------|---------------------|------------|

| 治理模型 | Constitution（版本化 + 不可協商） | Preamble（共用前言） | 可升級為 Constitution |

| 管線定義 | 7 階段 SDD + slash commands | pipeline.yaml + 手動步驟 | 可引入 command 化 |

| 插件生態 | Extension System（Manifest + Catalog） | Skill 目錄 + SKILL_INDEX.md | 可結構化為 catalog.json |

| 跨 Agent | 20+ Agent 支援 | Claude Code 專用 | 低優先（單一 Agent 即可） |

| 品質門檻 | Checklist 自動檢查 + Analyze | quality-gate.md + Hook 強制 | Hook 已更強（機器強制） |

| 上下文管理 | Progressive Disclosure | 按需載入 + 薄層 Prompt | 可量化 Token Budget |

| 文件交接 | Handoff YAML frontmatter | results/*.json | 可加 Schema 驗證 |

| 研究流程 | Clarify 結構化釐清 | 直接研究 + KB 去重 | 可加 Phase 0 問題界定 |

六、實作建議優先級排序

基於影響力 x 實作成本的評估，建議以下優先順序：

| 優先級 | 建議 | 影響 | 成本 | 對應模組 |

|--------|------|------|------|----------|

| P0 | results/*.json Schema 驗證 | 解決反覆出現的檔案缺失問題 | 低 | run-todoist-agent-team.ps1 |

| P1 | Skill catalog.json 結構化 | 提升 Skill 檢索效率與版本管理 | 中 | skills/SKILL_INDEX.md |

| P1 | Constitution 升級 | 治理原則版本化與自動合規檢查 | 中 | templates/shared/preamble.md |

| P2 | Token Budget 基線 | 量化 prompt 效率，指導優化方向 | 中 | config/pipeline.yaml |

| P2 | 研究 Phase 0 問題界定 | 提升研究品質與去重效果 | 低 | templates/auto-tasks/*.md |

| P3 | system-audit 跨文件語義覆蓋率 | 從分數升級到覆蓋率分析 | 高 | skills/system-audit/ |
