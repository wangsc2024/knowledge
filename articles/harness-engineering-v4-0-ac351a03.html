执行摘要

本研究整合 Martin Fowler、Anthropic、Red Hat、Google 与 2026 年产业界最新 Harness Engineering 实践，结合 Daily Digest Prompt 系统既有基础（B0-B3 批次实施），提出下一代优化方案。

核心发现：

  • 2026 共识：基础设施优化胜过模型升级（实验证明 10x 提升来自编辑格式变更，而非模型换代）
  • 关键转变：从「Agent 写程式」转向「设计让 AI 可靠写程式的系统」
  • 实证证据：Context reset 在 40% 使用率时优于等到 80% 才压缩
  • 成本控制：Computational 验证优先（毫秒级），Inferential 验证策略性部署（秒级）
  • Meta 实证：预计算引擎减少 40% tool calls，结构化 context 优于 context dumping

1. Harness Engineering 核心框架

1.1 定义与边界

Agent = Model + Harness

Harness 包含除模型外的所有机制，分为两层：

  1. 内建层：Agent 构建者提供（如 Claude Agent SDK）
  2. 使用者层：特定场景客制化（本专案的 Hooks、Skills、OODA 流程）

2026 产业共识：

"If 2025 was the year AI agents proved they could write code, 2026 is the year the industry learned that the agent isn't the hard part—the harness is."

工程师不再直接写程式，而是设计让 AI 可靠写程式的系统。OpenAI 已建立超过 100 万行代码的生产应用，其中工程师设计的是让 AI 可靠写代码的系统，而非直接写代码。

1.2 四大组件（agent-engineering.dev 标准）

| 组件 | 定义 | 本专案对应 | 成熟度 (v2.0) |

|------|------|-----------|------------|

| 架构约束 | 透过规则限制 Agent 行为空间 | Hooks、hook-rules.yaml、约束演化管线 | 8.5/10 → 目标 9.5 |

| 反馈回路 | 执行结果回馈至决策层 | quality_gate.py、constraint_evolve.py | 9.0/10 → 目标 9.5 |

| 验证护栏 | 多层品质门槛 | Draft-Verify、结果验证、Schema 检查 | 8.8/10 → 目标 9.5 |

| 生命周期管理 | 状态追踪与时间阈值 | timing-policy.yaml、OODA workflow | 9.2/10 → 目标 9.5 |

2. 双控制机制：Feedforward vs Feedback

2.1 Martin Fowler 框架

Guides（前馈控制） — 预防性引导，在行动前提高首次成功率

  • 系统提示（System Prompts）
  • AGENTS.md / CLAUDE.md
  • Skill 指引文件
  • Good Examples Repository（本专案 v2.0 新增）

Sensors（反馈控制） — 事后校正，观察输出并促进自我修正

  • 静态分析工具（Linter、Type Checker）
  • 测试执行（Unit / Integration / E2E）
  • Quality Gate LLM 评分
  • Hook 拦截与修复建议

关键洞察：

单独使用任一方式都不足：仅有反馈导致重复犯同样错误；仅有前馈则永远无法验证有效性。

2.2 本专案实施现状与改进

| 机制 | 现有实作 | v2.0 改进方向 | 预期效果 |

|------|---------|--------------|------|

| Feedforward | preamble.md、CLAUDE.md、Skill 模板 | + Good Examples Registry（优秀案例库） | 首次成功率 72% → 85% |

| Feedback | Hooks、quality_gate、constraint_evolve | + Success Pattern Extractor（成功模式自动升级） | iteration_count 1.3 → 1.1 |

| Feedback → Feedforward | ❌ 断裂 | ✅ 自动升级管线 | 减少重复错误 15% |

2.3 2026 最佳实践：Close the Feedback Loop

产业共识：

  • 将错误追溯回输入并修复 harness，而非仅修复输出
  • 使用 linter 阻止 PR 的确定性约束，优于 prompt 中的概率性遵守
  • Repository-First 设计：让 Agent 能直接从 repository 推理完整业务领域

本专案实施：

  • constraint_evolve.py 自动从失败模式提取约束规则
  • Hook 系统提供确定性拦截（非仅 prompt 建议）
  • hooks/hook-rules.yaml 作为单一真相来源

3. 验证金字塔：Computational vs Inferential

3.1 核心原则

| 类型 | 特性 | 成本 | 适用场景 |

|------|------|------|--------|

| Computational（确定性） | 规则式、快速、可重复 | 毫秒级 | JSON Schema、必填栏位、regex pattern |

| Inferential（推理式） | 语义分析、非确定性 | LLM 调用（秒级） | 文意完整性、逻辑连贯性、创意品质 |

最佳比例：70% Computational + 30% Inferential（Martin Fowler 建议）

2026 产业趋势：

  • Computational sensors 便宜且快速，可在每次变更时与 agent 并行运行
  • Inferential controls 成本较高且非确定性，但允许提供丰富的引导和额外的语义判断
  • Formal verification 在 2026 年成为主流趋势，提供完全自动化的反馈信号

3.2 本专案优化方案（v2.0 三层金字塔）


           ┌─────────────┐
           │  Tier 3     │  LLM 语义评分（仅 Critical 任务）
           │  5% 流量    │  成本：Haiku ~5s
           └─────────────┘
          ┌───────────────┐
          │  Tier 2       │  启发式检查（keyword 密度、引用完整性）
          │  20% 流量     │  成本：regex + NLP ~200ms
          └───────────────┘
         ┌─────────────────┐
         │  Tier 1         │  确定性检查（schema、必填栏位）
         │  100% 流量      │  成本：<10ms
         └─────────────────┘

预期效果：

  • LLM 呼叫从 30 次/日 → 9 次/日（减少 70%）
  • 验证延迟从 150s → 50s
  • 误判率 < 5%（Tier 2 阈值设为 8.0/10）

4. Context Management：Reset vs Compaction

4.1 Anthropic 实证研究

关键发现：

Context compaction 會让模型焦虑接近上限（context anxiety），导致提早结束任务。

两种策略对比：

| 策略 | 触发点 | 优点 | 缺点 | 适用场景 |

|------|--------|------|------|--------|

| Compaction | > 80% | 保留 context 连贯性 | 模型焦虑、资讯损失 | Opus 4.5 内建自动压缩 |

| Clean Reset | 40-60% | 无焦虑、清晰状态 | 需要结构化 Handoff | 多會话长时间任务 |

4.2 2026 最新实践：Context Engineering

Google 架构方法：

  • 避免 Context Dumping：不将大量负载直接放入 chat history
  • Artifacts 模式：将大数据作为命名、版本化的对象，由 ArtifactService 管理
  • Tiered Storage：Context 管理必须作为架构考量，与存储和计算同等重要

Meta 预计算引擎：

  • 使用专门的 AI agents 系统性读取文件，产生简洁的 context 文件编码部落知识
  • 将未文档化知识转为结构化、AI 可读的 context
  • 结果：每任务减少 40% AI agent tool calls

成本优化：

  • Anthropic prompt caching 对缓存的 input tokens 提供 90% 折扣
  • 单靠此功能可削减 20-30% agent 账单
  • 优化策略聚焦 context 管理，多数团队可达成 60-80% 成本节省

4.3 本专案实施方案（v2.0 Context Reset Coordinator）

三级重置策略：

| Context 使用率 | 动作 | 实作 |

|---------------|------|------|

| < 60% | 正常继续 | 无动作 |

| 60-75% | Artifact Handoff | 写入 state/agent-handoff/{trace_id}.json，清空 context，新 Agent 读取继续 |

| > 75% | Emergency Reset | 强制终止，标记 status=partial，checkpoint 写入 handoff |

Handoff Schema 设计：


{
  "trace_id": "abc123",
  "original_task": "深度研究 Harness Engineering",
  "completed_steps": ["WebSearch", "WebFetch", "初步分析"],
  "artifacts": {
    "web_content": "file://state/agent-handoff/abc123-web.md",
    "kb_drafts": ["file://state/agent-handoff/abc123-kb-01.md"]
  },
  "next_steps": ["撰写优化方案", "写入知识库", "验证"],
  "context_snapshot": null
}

与现有 ADR-036 的关系：

  • ADR-036（Summary/BufferWindow）保留作为 > 90% 最后手段
  • 60-75% 区间改用 Handoff，避免资讯损失

多 Agent 系统的 Context 共享原则（Google 2026）：

  • 若每个 agent 共享相同 context，會用无关细节混淆模型，并支付巨大的 KV-cache 惩罚
  • Agents 应透过最小、结构化的输出以受控方式沟通

5. 多 Agent 架构：专业化分工

5.1 Anthropic 三 Agent 模式

架构：Planner → Generator → Evaluator

| 角色 | 职责 | 输入 | 输出 |

|------|------|------|------|

| Planner | 任务分解与资源配置 | 功能列表 + 环境状态 | 执行计画（JSON） |

| Generator | 实际编码实作 | 执行计画 + codebase | 程式码变更 + git commit |

| Evaluator | 品质验证与测试 | 变更集 + 测试脚本 | 通过/失败 + 改进建议 |

核心优势：

  • 专业化：每个 Agent 只做一件事，提示词可独立调优
  • 可观测性：Planner 输出计画 JSON 可单独审查
  • 避免自评偏差：Generator 与 Evaluator 分离，消除「AI 验证 AI」盲点

5.2 本专案四角色设计（v2.0 提案）

| 角色 | 职责 | 现有对应 | 改进方向 |

|------|------|---------|--------|

| Planner | 任务选择与分解 | Phase 1 内嵌逻辑 | 独立为 todoist-planner.md |

| Generator | 任务执行 | Phase 2 子 Agent | 无需变更 |

| Evaluator | 品质验证 | quality_gate.py | 增强 Tier 2 Heuristic |

| Assembler | 输出组装 | Phase 3 todoist-assemble | 无需变更 |

实施效益：

  • Planner 可独立 A/B 测试（不影响执行层）
  • 计画 JSON 提供完整 audit trail
  • 消除 OODA vs Harness 职责重叠

6. 2026 产业界最新实践

6.1 十大关键洞察

  1. 基础设施 > 模型升级

- 实验证明：改变编辑格式带来 10x 提升，模型升级仅 1.2x

- LangChain 在 TerminalBench 2.0 从低排名跃升至第 5 名，仅靠 Harness 优化

  1. 文档是最强 Harness

- "If your AGENTS.md is vague, your agent output will be vague"

- 投资精确、机器可读的文档作为 Agent 的 Ground Truth

  1. Repository-Local Knowledge

- Agent 视角：不在 context 中的内容等同不存在

- Google Docs、Slack 讨论不可访问，仅 repo 内的版本化文件可见

  1. 40% Context Reset Rule

- 在 40% 使用率时重置 context，搭配结构化 handoff 文档

- 工具过载问题：仅载入需要的工具（Tool Search 节省 85% token）

  1. Network Isolation for Evaluation

- 评估环境网路隔离现已成为必要要求，非可选项

  1. Max Iterations Hard Limit

- 严格的 max_iterations 计数器，超过阈值强制终止

- 防止 Agent 无限回圈

  1. Repository Impact Map

- 让 AI 在规划前看实际程式码

- 从符号分析建立影响地图，让下游任务基于真实程式码

  1. Meta-Harness Optimization

- 将整个 Harness 视为联合优化目标

- Proposer Agent 访问所有先前 Harness 候选、分数、执行 trace

  1. Guides + Sensors 系统化

- 当前 Feedforward/Feedback 控制散落各处

- 需要工具化集中配置、同步、系统性推理

  1. Keep Quality Left

- 将检查分布于整个开发周期

- 早期发现成本最低（LSP、Linter）

- 后集成检查（变异测试、性能测试）

- 持续监测（死码、依赖扫描、SLO 漂移）

6.2 结构化 Context 优于自由形式

Google Agent Bake-Off 发现：

  • AI 在结构化 context 环境中写出更好的代码
  • 给 AI 结构化约束实施，如文件路径、符号名称、现有模式
  • 核心原则：输入越具體，输出越一致

实施建议：

  • 不要给 AI 自由形式的 tickets，而是设计它工作的环境
  • 用 custom linters 和 structural tests 强制规则
  • 在错误讯息中注入修复指令到 agent context

6.3 确定性约束优于概率遵守

对比：

  • ❌ 概率遵守：在 prompt 中告诉 agent 遵循标准（依赖 LLM 推理）
  • ✅ 确定性约束：接上 linter，违反标准时阻止 PR（强制执行）

本专案实施：

  • Hook 系统提供确定性拦截
  • validate_config.py、validate_results.py 提供硬性验证
  • Schema 检查在工具层而非 prompt 层

7.2 v3.0 新增优化（基于 2026 实践）

#### 优化 A：Repository Impact Map（程式码影响地图）

问题：Agent 规划时缺乏对现有程式码的结构化理解。

方案：

  • 从符号分析建立 context/code-impact-map.json
  • 包含：函式依赖图、模组耦合度、热点档案
  • Planner Agent 读取此地图后再规划

工具：tools/code_impact_analyzer.py（基于 AST 分析）

#### 优化 B：Meta-Harness Tuner（自动调优引擎）

问题：Harness 参数（timeout、阈值、权重）当前手动调整。

方案：

  • Proposer Agent 访问历史执行 trace
  • 基于成功率/失败模式自动建议参数调整
  • 人工审核后应用

工具：tools/harness_tuner.py --propose --apply

#### 优化 C：Unified Guides & Sensors Dashboard

问题：Feedforward/Feedback 控制散落 15+ 档案。

方案：

  • 建立 config/guides-sensors-registry.yaml 统一注册
  • 视觉化仪表板显示每个控制的触发率、有效性
  • 自动侦测「未触发规则」（Dead Guards）

工具：tools/harness_dashboard.py（输出 HTML 报告）

#### 优化 D：Network-Isolated Sandbox（隔离沙箱）

问题：当前评估环境无网路隔离。

方案：

  • 使用 Windows Sandbox 或 Docker 执行测试
  • 仅允许白名单 API（localhost:3000 知识库、ntfy.sh）
  • 所有外部 HTTP 请求必须透过 api-cache 代理

实作：更新 config/autonomous-harness.yaml 增加 network_policy

7.3 v4.0 进阶优化（整合产业最佳实践）

#### 优化 E：Pre-Compute Context Engine（Meta 模式）

问题：每次执行都重新计算 context，浪费 token 与时间。

方案（参考 Meta 实践）：

  • 建立专门的 context 预计算引擎
  • 系统性读取档案，产生简洁的 context 编码档
  • 将部落知识转为结构化、AI 可读的 context

预期效果：

  • 减少 40% tool calls（Meta 实证数据）
  • 提升 context 精确度，减少无关资讯

实作：


# tools/context_precompute_engine.py
class ContextPrecomputeEngine:
    def scan_repository(self):
        """扫描 repo，提取关键知识"""
        pass

    def generate_context_artifacts(self):
        """生成结构化 context 档案"""
        pass

    def update_incremental(self):
        """增量更新（仅变更档案）"""
        pass

#### 优化 F：Structured Context Artifacts（Google 模式）

问题：大数据直接放入 chat history 导致 context 污染。

方案（参考 Google 架构）：

  • 实作 ArtifactService 管理命名、版本化的对象
  • 档案、日志、配置作为 artifacts 而非直接内容
  • Agents 透过 artifact references 沟通

Schema 设计：


{
  "artifact_id": "log_audit_20260422",
  "type": "analysis_result",
  "version": "1.0",
  "storage": "state/artifacts/log_audit_20260422.json",
  "metadata": {
    "created_at": "2026-04-22T10:30:00+08:00",
    "created_by": "todoist-auto-log_audit",
    "size_bytes": 15360,
    "checksum": "sha256:abc123..."
  }
}

#### 优化 G：Custom Linters with Remediation（Repository-First）

问题：Hook 拦截后 agent 不知如何修正。

方案：

  • 强化 Hook error messages 包含修复指令
  • 错误讯息注入到 agent context
  • 提供 auto-fix suggestions

实作：


# hooks/enhanced_hook_with_remediation.py
def validate_with_remediation(file_path: str, content: str):
    errors = validate(content)
    if errors:
        remediation = generate_fix_instructions(errors)
        return {
            "valid": False,
            "errors": errors,
            "remediation": remediation,  # 新增：修复指令
            "auto_fix_available": True
        }

11. 关键设计决策（ADR 对齐）

ADR-041：40% Context Reset Rule

决策：在 context 使用率达 40-60% 时触发 Clean Reset，而非等到 80% 才压缩。

理由：

  • Anthropic 实证：context compaction 导致模型焦虑
  • 40% reset 提供足够的「心理空间」
  • 结构化 Handoff 避免资讯损失

实作：

  • 修改 tools/context_compressor.py 降低阈值
  • 新增 tools/context_reset_coordinator.py
  • Handoff schema 强制包含 artifacts 与 next_steps

ADR-042：Computational-First Verification

决策：70% 验证流量走 Computational 路径，30% 走 Inferential。

理由：

  • Computational 成本 < 10ms，Inferential 成本 > 5s
  • Martin Fowler 建议的黄金比例
  • 本专案实测 LLM 成本可减少 70%

实作：

  • Tier 1（100%）：Schema 验证（validate_results.py）
  • Tier 2（20%）：Heuristic（quality_heuristics.py）
  • Tier 3（5%）：LLM 评分（quality_gate.py）

ADR-043：Pre-Compute Context Engine

决策：建立预计算引擎，系统性提取 repo 知识为结构化 context。

理由：

  • Meta 实证减少 40% tool calls
  • 部落知识转为 AI 可读格式
  • 避免每次执行重新计算

实作：

  • tools/context_precompute_engine.py
  • Git hook 触发增量更新
  • 输出至 context/precomputed/ 目录

ADR-044：Artifact-Based Context Management

决策：大数据作为 Artifacts 管理，而非直接放入 chat history。

理由：

  • Google 架构建议避免 context dumping
  • 命名、版本化对象提升可追溯性
  • 减少 KV-cache 惩罚

实作：

  • tools/artifact_service.py（create/get/list/cleanup）
  • Artifacts 存储于 state/artifacts/
  • 7 天 TTL 自动清理

ADR-045：Enhanced Hook Remediation

决策：Hook 错误讯息必须包含修复指令，注入到 agent context。

理由：

  • 产业最佳实践（Repository-First 设计）
  • 确定性约束 + 修复指引 = 高效回馈
  • 减少 iteration 次数

实作：

  • 所有 Hook 函式回传 remediation 栏位
  • Agent 读取错误时自动取得修复建议
  • 支援 auto-fix（可选）

13. 结论与下一步

关键要点

  1. Harness > Model：实验证明基础设施优化带来的提升远超模型升级
  2. 双控制必要性：Feedforward 与 Feedback 必须并存且形成闭环
  3. 70/30 黄金比例：70% Computational + 30% Inferential 验证最优
  4. 40% Reset Rule：Context 在 40% 时重置优于 80% 压缩
  5. 专业化分工：Planner-Generator-Evaluator 三角色分离提升可观测性
  6. Artifacts 优于 Dumping：大数据作为命名对象而非直接内容
  7. Pre-Compute 引擎：Meta 实证减少 40% tool calls
  8. 确定性 + 修复指引：Hook 错误讯息注入修复指令到 context

建议实施顺序

  1. 立即启动（本周）：

- Tier 2 Heuristic 层（快速胜利，成本 -70%）

- Artifact Service（Google 模式）

  1. 第二季（3-6 个月）：

- Meta-Tuner + Harness Dashboard + Network Isolation

- Code Impact Map

  1. 持续优化：

- 每月审查 SRS 指标

- 每周监控 Dead Guards
