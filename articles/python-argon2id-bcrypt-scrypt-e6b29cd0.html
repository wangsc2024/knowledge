一、技術概述

密碼雜湊（Password Hashing）是將使用者明文密碼轉換為不可逆固定長度字串的核心安全機制，其目的在於即使資料庫洩漏，攻擊者也無法直接取得原始密碼。2025-2026 年的密碼安全領域經歷了重要轉變：Argon2id 被 OWASP 確立為首選演算法；Python 3.13 移除了 crypt 標準庫模組，導致長期霸主 passlib 出現相容性問題；FastAPI 生態系全面遷移至 pwdlib + argon2-cffi。本文系統性整理四大雜湊演算法的比較、Python 生態系的遷移路徑，以及與本專案（daily-digest-prompt 的 QA System）的直接應用。

二、OWASP 密碼儲存規範（2025 更新）

2.1 演算法選擇優先順序

OWASP Password Storage Cheat Sheet 明確建立了優先級階層：

  1. Argon2id（首選）— Password Hashing Competition 冠軍，記憶體硬性 + 時間硬性 + 抗並行攻擊
  2. scrypt（次選）— Argon2id 不可用時的替代方案
  3. bcrypt（僅限遺留系統）— 成熟穩定但不具記憶體硬性
  4. PBKDF2（僅限 FIPS-140 合規需求）— CPU 硬性但不抗 GPU 攻擊

2.2 各演算法推薦參數

| 演算法 | 最低配置 | 推薦配置 | 高安全配置 |

|--------|---------|---------|----------|

| Argon2id | m=19MiB, t=2, p=1 | m=64MiB, t=3, p=1 | m=128MiB, t=3, p=4 |

| scrypt | N=2^15, r=8, p=1 | N=2^17, r=8, p=1 | N=2^18, r=8, p=1 |

| bcrypt | cost=10 | cost=12~13 | cost=14 |

| PBKDF2 | SHA-256, 600K 迭代 | SHA-256, 600K+ 迭代 | SHA-512, 210K+ 迭代 |

2.3 鹽值與胡椒值

  • Salt（鹽值）：由現代函式庫自動管理，開發者無需手動處理
  • Pepper（胡椒值）：額外的防禦深度層，必須儲存在密鑰保管庫（Secrets Vault），不可與密碼雜湊放在同一資料庫

2.4 密碼長度處理

  • bcrypt 有 72 位元組上限，超過會被截斷——必須在應用層強制限制或使用 pre-hashing
  • 推薦 pre-hashing 模式：bcrypt(base64(hmac-sha384(data:$password, key:$pepper)))
  • Argon2id 無此限制

三、四大演算法效能與安全性比較

3.1 效能基準（Intel Xeon E5-2680 @ 2.7GHz）

| 演算法 | 配置 | 雜湊時間 | 記憶體用量 |

|--------|------|---------|----------|

| Argon2id | m=64MB, t=3, p=1 | 200ms | 64 MB |

| bcrypt | cost=12 | 250ms | 4 KB |

| scrypt | N=2^17, r=8, p=1 | 350ms | 128 MB |

| PBKDF2 | 600K 迭代 | 200ms | <1 MB |

3.2 GPU 攻擊抵抗力

| 演算法 | GPU 加速倍數（vs CPU） | 8字元密碼破解成本 | 評級 |

|--------|----------------------|------------------|------|

| Argon2id | ~2-5x | $500K+ | ★★★★★ |

| bcrypt | ~400x | $40K | ★★★★☆ |

| scrypt | ~10-20x | $300K | ★★★★☆ |

| PBKDF2 | ~5,000x | $5K | ★★☆☆☆ |

核心洞察：記憶體硬性演算法（Argon2id、scrypt）透過強制攻擊者為每次嘗試分配大量 RAM，大幅削弱 GPU 並行攻擊的優勢。

3.3 併發驗證的記憶體估算

100 位使用者同時驗證：

  • Argon2id (m=64MB)：~6.4 GB
  • scrypt (N=2^17)：~12.8 GB
  • bcrypt：~400 KB
  • PBKDF2：可忽略

設計決策：選擇 Argon2id 時需確保伺服器有足夠 RAM，或調低 memory_cost 配合較高 time_cost。

四、Python 生態系遷移指南（2025-2026 關鍵變化）

4.1 Passlib 棄用危機

  • passlib 長期未維護，最後穩定版 v1.7.4
  • Python 3.13 移除 crypt 標準庫 → passlib 的部分 hash scheme 直接失效
  • bcrypt 5.0.0 的 API 變更導致 CryptContext 在新版 bcrypt 上報錯
  • FastAPI 官方已將文件更新為使用 pwdlib 取代 passlib

4.2 遷移路徑

| 來源 | 目標 | 遷移難度 | 說明 |

|------|------|---------|------|

| passlib + bcrypt | pwdlib + argon2-cffi | 低 | FastAPI 官方推薦路徑 |

| passlib + bcrypt | 原生 bcrypt 庫 | 低 | 最簡方案，直接用 bcrypt.hashpw/checkpw |

| passlib + bcrypt | argon2-cffi 直接使用 | 中 | 最安全方案，需處理向下相容 |

| passlib CryptContext | 自建 CryptContext 替代 | 中 | 保留 scheme rotation 能力 |

4.3 pwdlib（FastAPI 新推薦）


# fastapi-users v13+ 預設使用 pwdlib + Argon2
# 向下相容：bcrypt 雜湊仍可驗證，登入時自動升級為 Argon2
from pwdlib import PasswordHash
from pwdlib.hashers.argon2 import Argon2Hasher

password_hash = PasswordHash((Argon2Hasher(),))
hashed = password_hash.hash("my_password")
assert password_hash.verify("my_password", hashed)

4.4 argon2-cffi 直接使用


from argon2 import PasswordHasher
from argon2.exceptions import VerifyMismatchError

ph = PasswordHasher(
    time_cost=3,        # 迭代次數
    memory_cost=65536,  # 64 MB
    parallelism=1,
    hash_len=32,
    salt_len=16
)

# 雜湊
hash_result = ph.hash(password)
# 輸出格式：$argon2id$v=19$m=65536,t=3,p=1$salt$hash

# 驗證
try:
    ph.verify(hash_result, password)
    # 檢查是否需要重新雜湊（參數升級時）
    if ph.check_needs_rehash(hash_result):
        new_hash = ph.hash(password)
        # 更新資料庫中的雜湊值
except VerifyMismatchError:
    return False

4.5 Lazy Upgrade 模式（無需強制重設密碼）


# 在使用者成功登入時自動升級雜湊演算法
def authenticate_and_upgrade(password: str, stored_hash: str) -> tuple[bool, str | None]:
    """驗證密碼並在必要時升級雜湊。
    返回 (驗證結果, 新雜湊或None)
    """
    # 偵測舊格式（bcrypt 以 $2b$ 開頭）
    if stored_hash.startswith('$2b$'):
        import bcrypt
        if bcrypt.checkpw(password.encode(), stored_hash.encode()):
            # 成功驗證，升級為 Argon2id
            new_hash = ph.hash(password)
            return True, new_hash
        return False, None
    
    # Argon2id 驗證
    try:
        ph.verify(stored_hash, password)
        if ph.check_needs_rehash(stored_hash):
            return True, ph.hash(password)
        return True, None
    except VerifyMismatchError:
        return False, None

五、最佳實踐與常見陷阱

5.1 安全最佳實踐

  1. 優先使用 Argon2id：新專案一律採用 Argon2id，不再選擇 bcrypt
  2. 定期調整參數：每年依硬體進步提高 cost/memory/iteration
  3. 常數時間比較：使用 hmac.compare_digest() 或演算法內建的 verify 函式，防止計時攻擊
  4. 限制密碼長度上限：設定合理上限（如 128 字元）防止 HashDoS
  5. PHC 字串格式：使用標準化格式儲存雜湊（$argon2id$v=19$m=65536,t=3,p=1$salt$hash），便於日後升級

5.2 常見陷阱

  1. bcrypt 72 位元組截斷：中文字元（UTF-8 3 位元組/字）實際可用字元數更少
  2. passlib + Python 3.13：crypt 模組移除導致直接崩潰
  3. bcrypt 5.0 API 變更：bcrypt.hashpw() 不再接受 unicode 字串，必須先 encode
  4. 記憶體不足：Argon2id m=64MB 在容器化環境（如 128MB limit）可能 OOM
  5. pepper 洩漏：pepper 一旦洩漏且硬編碼在程式碼中，比不用更糟

5.3 CryptContext DRY 原則

本專案（QA System）教訓：在 #22 優化中發現 pwd_context = CryptContext(schemes=['bcrypt']) 被重複定義 3 次（auth.py、users.py、main.py），導致：

  • 若某處修改 scheme 或 deprecated 設定，其他處不一致
  • 單元測試無法統一 mock
  • 違反 DRY 原則

修正方式：在單一檔案（如 security.py 或 user.py）定義，其他模組 import。

六、與本專案的關聯

6.1 QA System 密碼安全現況

  • 使用 passlib 的 CryptContext(schemes=['bcrypt'], deprecated='auto')
  • #22 優化已完成 pwd_context DRY 整合（3 處 → 1 處）
  • 目前僅使用 bcrypt，尚未遷移至 Argon2id

6.2 建議升級路徑

  1. 短期：維持 passlib bcrypt，確保 Python 版本 ≤3.12
  2. 中期：遷移至 pwdlib + argon2-cffi，搭配 lazy upgrade 模式
  3. 長期：移除 passlib 依賴，直接使用 argon2-cffi

6.3 daily-digest-prompt Hooks 的密碼保護

  • pre_bash_guard.py 已攔截 echo/printenv 讀取 TOKEN/SECRET/KEY/PASSWORD 環境變數
  • pre_read_guard.py 已攔截 .env/credentials 等敏感檔案讀取
  • 這些 Hook 規則確保密碼和金鑰不會意外洩漏到日誌或輸出
