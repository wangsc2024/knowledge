GitHub Agent 架構熱門專案分析

---

📊 專案概覽

1. OpenClaw (302k⭐)

專案: openclaw/openclaw

描述: 本地優先的 AI 助理，採用 Gateway 控制平面架構，支援 20+ 社交平台整合

核心模式:

  • ✨ 三層 Skill 系統: bundled（捆綁技能）→ managed（託管技能）→ workspace（工作區技能）
  • 🌐 WebSocket Gateway 控制平面: ws://127.0.0.1:18789 作為單一協調點
  • 💬 會話間通訊工具: sessions_list/history/send 實現多 Agent 協作
  • 🎣 媒體管道 Hook: 轉錄、大小限制、臨時檔案生命週期控制
  • ⏰ Cron + Wakeup 混合排程: 組合模式支援定期任務和事件驚醒

架構洞察:

「控制平面即產品」——通過統一的本地 Gateway 承載複雜性，讓多層技能系統和設備生態有機整合。

---

2. CrewAI (45.9k⭐)

專案: crewAIInc/crewAI

描述: 多 Agent 協作框架，支援角色扮演與任務編排

核心模式:

  • 🎭 雙模式協作: Crews（自主協作）+ Flows（精準編排）
  • 📋 YAML 配置驅動: 任務定義結構化，易於版本控制
  • 👥 角色特化: role（角色）+ goal（目標）+ backstory（背景故事）
  • 👔 Hierarchical 執行模式: 自動指派經理 Agent 進行計劃、委派、驗證
  • 🏗️ Pydantic BaseModel: 結構化狀態管理確保資訊一致性

設計價值:

明確的專業角色設定增強團隊協作的有效性和可追蹤性。

---

3. LangGraph (44.6k⭐)

專案: langchain-ai/langgraph

描述: 圖狀態機 Agent 編排框架，強調持久化執行與人機協作

核心模式:

  • 📊 圖形編排: 受 Pregel 與 Apache Beam 啟發的節點邊結構
  • 💾 持久化執行: 故障自動恢復，從中斷處繼續，支援長時間運行
  • 🤝 Human-in-the-loop: 任意時點檢視與修改 Agent 狀態
  • 🧠 雙層記憶架構: 短期工作記憶（ongoing reasoning）+ 長期跨會話記憶
  • 🎯 低抽象設計: 控制與持久性優先，功能作為可選建構塊

設計原則:

「功能作為建構塊，開發者自由選擇何時使用」——不強加抽象，保持靈活性。

---

💡 改進建議（6 項）

🔴 P0 - 持久化執行（故障自動恢復）

來源: LangGraph

目標檔案: state/run-fsm.json, run-todoist-agent-team.ps1

工作量: medium

建議: 強化 run-fsm.json 的殭屍偵測機制，新增「中斷點快照」功能。當 Phase 2 的某個 Agent 超時或失敗時，將已完成的 Agent 結果快照到 state/phase2-snapshot-{traceId}.json，下次執行時可選擇「從快照恢復」（跳過已完成的 Agent，僅重試失敗的）。

價值: 目前 timeout 後整個 Phase 2 都需重新執行，浪費資源。此改進可顯著提升容錯能力與資源利用率。

---

🟡 P1 - 三層 Skill 系統

來源: OpenClaw

目標檔案: skills/SKILL_INDEX.md, docs/ARCHITECTURE.md

工作量: medium

建議: 引入 Skill 分層機制（core → shared → workspace），將現有 26 個專案 Skills 按穩定度分層：

  • core（15 個）：核心不變的 Skills
  • shared（8 個）：跨專案共用
  • workspace（3 個）：實驗性 Skills

價值: 降低依賴地獄、提升 Skill 生命週期管理、清晰的穩定性預期。

---

🟡 P1 - Hierarchical 執行模式（經理 Agent）

來源: CrewAI

目標檔案: run-todoist-agent-team.ps1, prompts/team/todoist-query.md

工作量: high

建議: 在 Team 模式引入「經理 Agent」角色：Phase 1 查詢後，經理 Agent 分析任務複雜度與依賴關係，動態調整 Phase 2 的並行策略。

範例:

  • 2 個研究任務有依賴 → 改為順序執行
  • 4 個獨立任務 → 全並行

價值: 目前 run-todoist-agent-team.ps1 的並行邏輯是固定的，經理 Agent 可提升編排智慧與適應性。

---

🟢 P2 - 會話間通訊工具

來源: OpenClaw

目標檔案: prompts/team/todoist-query.md, skills/task-manager/SKILL.md

工作量: low

建議: 參考 sessions_list/history/send 模式，在 Team 模式的 Phase 2 並行 Agent 間建立通訊機制。

實作方式: Phase 2 結果 JSON 新增 inter_agent_messages 欄位，Phase 3 assemble 時讀取並轉發。

範例: 研究 Agent 完成後通知 todoist Agent「已發現相關筆記」，todoist Agent 可選擇性調整優先級。

---

🟢 P2 - 角色特化（role + goal + backstory）

來源: CrewAI

目標檔案: prompts/team/todoist-.md, templates/sub-agent/.md

工作量: low

建議: 為各 Agent prompt 新增 frontmatter 的 role/goal/backstory 欄位。

範例: todoist-auto-ai_deep_research.md

  • role: 「AI 技術深度研究員」
  • goal: 「每日規劃一個 AI 技術深度研究」
  • backstory: 「擁有 5 年 AI 論文閱讀經驗」

價值: 強化角色認同，提升輸出一致性。

---

🟢 P2 - Human-in-the-loop（任意時點干預）

來源: LangGraph

目標檔案: prompts/team/todoist-query.md, config/approval-rules.yaml

工作量: high

建議: 引入「干預點」配置：在 config/approval-rules.yaml 定義哪些任務類型需要人工批准。

流程:

  1. git_push 任務在 Phase 2 完成後，寫入 state/approval-pending-{task_key}.json
  2. 發送 ntfy 通知，等待用戶回覆「approve」或「reject」
  3. Phase 3 檢查批准狀態後才執行

適用場景: 高風險操作（git push、KB 大量匯入、系統配置變更）。

---

🎯 執行優先級建議

  1. 立即實施: P0 持久化執行（解決資源浪費痛點）
  2. 本週實施: P1 三層 Skill 系統（架構改善，影響範圍大）
  3. 兩週實施: P1 經理 Agent（複雜度高但價值顯著）
  4. 持續優化: P2 項目按需求與資源排程
