cache/knowledge.json 修復與 Knowledge API 快取策略優化

日期：2026-04-22

優先級：P0-CRITICAL

影響範圍：系統成功率（當前 74%，目標 ≥ 90%）

摘要

當前系統 cache/knowledge.json 檔案缺失，導致 760 次 Knowledge API 呼叫全部 miss，cache_hit_ratio=0%，嚴重影響系統成功率（74% vs. 90% 門檻）。根因分析顯示：(1) 快取檔案需手動初始化，缺乏自動生成機制；(2) 無健康監控與修復流程；(3) TTL 設定為 32h，但未配合檔案產生機制。修復策略包含三層：立即修復（初始化腳本）、中期強化（健康監控）、長期治理（自動恢復）。預期可將 cache_hit_ratio 從 0% 提升至 60%+，系統成功率提升至 85%+。

關鍵洞見

1. 根因三角分析

根因 A：快取檔案初始化缺口

  • 發現：cache 目錄下無 knowledge.json，但 cache-policy.yaml 已定義 TTL=1920min
  • 證據：ls cache/ 輸出無 knowledge.json；system-insight 顯示 760 次全 miss
  • 影響：每次 knowledge-query API 呼叫都直接打知識庫，無快取保護

根因 B：缺乏快取健康監控

  • 發現：system-insight 監控其他快取，但未監控 knowledge 快取健康
  • 證據：cache_sources_broken: ["knowledge"]，但未觸發自動修復
  • 影響：檔案缺失後長時間未被偵測，累積 760 次 miss

根因 C：初始化與 TTL 設定脫節

  • 發現：cache-policy.yaml 設定 TTL 32h，假設檔案已存在，但無對應初始化機制
  • 證據：api-cache SKILL.md 描述「若檔案不存在 -> 直接呼叫 API」，未說明如何首次建檔
  • 影響：首次執行或檔案誤刪後無法自動恢復

建議行動

立即行動（P0）

  1. 建立快取初始化腳本 tools/initialize_knowledge_cache.py
  2. 執行初始化
  3. 驗證檔案建立

中期強化（P1）

  1. 加入健康監控（check-health.ps1）
  2. 觀測 7 天命中率

長期治理（P2）

  1. 修改 knowledge-query 流程，確保 API 成功後寫入快取
  2. 建立自動恢復機制（整合至 self-heal）

預期效益

  • cache_hit_ratio (knowledge): 0% → 60%+
  • daily_success_rate: 74% → 85%+
  • knowledge API 延遲: 200-500ms → <50ms
