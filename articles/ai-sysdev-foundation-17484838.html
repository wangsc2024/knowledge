核心內容摘錄：

  • ## 技術概述 2025-2026 的 AI 需求工程實務，重點已從「讓模型幫你寫 user story」轉向「讓需求成為可持續驗證的主工件」。GitHub Spec Kit 把規格放在開發最前面，先生成 spec、再生成 plan 與 tasks；Cucumber Gherkin 讓 Rule 與 Scenario 直接承接業務規則與驗收條件；Atlassian Rovo Dev 已能在 pull request 階段比對 linked Jira work item 的 acceptance criteria，回報已滿足、缺漏或仍需人工判定。研究面也顯示，LLM 在 Requirements Engineering 的主流用例已集中在 elicitation、validation 與 test generation，但多數還停留在 controlled environment，因此真正可落地的関鍵不是「生成更多文字」，而是建立 spec-first、traceability、PR gate 與 deterministic validation 的閉環。
  • ## 核心方法與工具 - Spec-Driven Development（SDD）：用規格而不是程式碼當主工件。先明確寫出需求、驗收條件與待澄清事項，再生成技術計畫與任務。 - [NEEDS CLARIFICATION] 守衛：在需求階段禁止模型自行補空白，凡是授權範圍、錯誤語義、成功條件、例外情境不明確，一律標記待澄清。 - Acceptance Criteria Normalization：把鬆散需求轉成可驗證敘述，每條 AC 只描述一件事，必須可觀測、可衡量、可對應測試。 - Gherkin / BDD：以 Feature、Rule、Scenario、Given/When/Then 把 acceptance criteria 轉成可執行規格，避免需求與測試分離。 - Requirements-aware PR Gate：在 PR 階段把 work item、branch name、acceptance criteria、測試結果與 code review 串起來，不讓「測試有過、需求沒滿足」的變更直接進主線。 - Trust Layer：高風險需求仍需 schema、contract、formal rule 或人工審查，不能把 LLM 產出的需求直接當真相。
  • ## 實際工作流程（step-by-step） 1. 收集原始需求：來源可以是訪談逐字稿、issue、support ticket、故障回報、產品想法或既有 prompt/workflow 缺陷。 2. 生成需求草稿：用 LLM 先整理成 feature summary、user story、business rule、non-functional constraint；未指定資訊全部標為 [NEEDS CLARIFICATION]。 3. 切成單一結果導向的 stories：每個 story 只保留一個主要 user outcome，不把多個流程或多個角色塞進同一條。 4. 正規化 acceptance criteria：每條 AC 必須寫成「前置條件 / 觸發 / 可觀測結果」，避免抽象語句如「更好用」「更穩定」。 5. 下沉成 Gherkin：用 Rule 表示業務規則，用 Scenario 表示具體情境；Then 只驗證可觀測輸出，不驗證資料庫內部狀態。 6. 產生 implementation plan：將規格映射到資料模型、API contract、驗證情境與任務切分，並保留需求到技術決策的追溯関係。 7. 先建測試骨架再補實作：優先產生 contract / integration / e2e / unit 測試骨架，讓模型補齊程式碼時被規格與測試雙重約束。 8. PR 階段做 requirements-aware 檢查：branch name 帶 Jira key，PR 連到單一 work item；系統比對 acceptance criteria 與變更內容，標示已滿足、缺漏或需人工確認。 9. 把營運回饋回寫規格：incident、使用者回報、品質告警不只修 code，也要回寫到 spec 與 acceptance criteria，讓下一輪生成與驗證有依據。

  • 白話：求寶者須經極長、多怖畏的曠野；雖有明達導師引路，大眾於中路疲憊恐懼，寧可退回，象徵眾生於生死煩惱長途與佛道久修前心生退轉。
