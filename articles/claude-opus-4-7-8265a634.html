Claude Opus 4.7 最新技術特性與遷移指南

研究摘要

Claude Opus 4.7 是 Anthropic 於 2026 年 4 月 16 日發布的最新旗艦模型，是目前最強大的 Claude 模型。本研究整理其核心特性、重大變更、能力提升與遷移指南。

核心規格

  • API Model ID: claude-opus-4-7
  • Context Window: 1M tokens（標準定價，無長文本加價）
  • Max Output: 128k tokens
  • 支援功能: Adaptive Thinking、Tool Use、Vision（高解析度）、Task Budgets（Beta）
  • 定價: $5 / $25 per MTok（與 Opus 4.6 相同）

重大新功能

1. 高解析度圖片支援（High-Resolution Image Support）

  • 最大解析度提升: 從 1568px / 1.15MP → 2576px / 3.75MP
  • 1:1 像素映射: 座標系統直接對應實際像素，無需縮放計算
  • 能力提升:

- 低層級感知（指向、測量、計數）

- 圖片定位（自然圖片邊界框定位與偵測）

  • 應用場景: Computer Use、截圖分析、文件理解工作流
  • 注意事項: 高解析度圖片消耗更多 tokens，不需要時應降採樣

2. 新增 `xhigh` Effort Level

  • Effort Parameter: 調整智慧度 vs. token 成本的平衡
  • 建議設定:

- xhigh: 程式碼與 agentic 任務

- high: 大多數智慧敏感任務

  • 僅適用於 Messages API（Claude Managed Agents 自動處理）

3. Task Budgets（任務預算，Beta）

  • 功能: 給 Claude 建議的 token 預算，用於完整 agentic 循環
  • 機制: 模型看到倒數計時器，用於排定工作優先級與優雅完成
  • 最小值: 20k tokens
  • 設定方式:

- Beta header: task-budgets-2026-03-13

- Output config: {"task_budget": {"type": "tokens", "total": 128000}}

  • 與 max_tokens 的差異:

- max_tokens: 硬性單次請求上限（模型不可見）

- task_budget: 建議性全循環上限（模型可見並自我調節）

  • 適用場景: 需要模型在 token 額度內自我調節的任務
  • 不適用: 開放式 agentic 任務（品質優先於速度時）

破壞性變更（Breaking Changes）

1. 移除 Extended Thinking Budgets

  • 不再支援: thinking: {"type": "enabled", "budget_tokens": N}
  • 替代方案: 使用 Adaptive Thinking + Effort
  • 預設行為: Adaptive Thinking 在 Opus 4.7 預設關閉，需明確設定 thinking: {type: "adaptive"} 啟用

2. 移除 Sampling Parameters

  • 不再支援: temperature, top_p, top_k 的非預設值
  • 替代方案: 使用 prompting 引導模型行為
  • 注意: temperature = 0 從未保證完全確定性輸出

3. Thinking Content 預設省略

  • 變更: 思考內容預設不在回應中顯示（thinking 欄位為空）
  • 影響: 延遲略有改善，但串流時會出現長暫停
  • 恢復方式: 設定 "display": "summarized" 以顯示思考進度

thinking = {
    "type": "adaptive",
    "display": "summarized",  # 或 "omitted"（預設）
}

4. Tokenizer 更新

  • 變更: 新的 tokenizer，處理文字時可能使用 1x～1.35x tokens（最多增加 35%）
  • 建議: 更新 max_tokens 參數，預留額外空間
  • API: /v1/messages/count_tokens 的計數結果與 Opus 4.6 不同

能力提升

知識工作（Knowledge Work）

  • .docx 修訂與 .pptx 編輯: 改進追蹤變更與投影片版面的生成與自我檢查
  • 圖表與圖形分析: 改進使用圖片處理工具（如 PIL）的程式化 tool calling，包括像素級數據轉錄
  • 建議: 移除現有 prompt 中的多餘緩解措施（如「雙重檢查投影片版面」），重新建立基準

記憶（Memory）

  • 檔案系統記憶: 更擅長跨回合寫入與使用檔案系統記憶（筆記本、結構化記憶存儲）
  • Agent 改進: 維護 scratchpad 的 agent 將更擅長記錄與利用筆記
  • 工具支援: 使用 Memory Tool 獲得託管式 scratchpad（無需自建）

視覺（Vision）

  • 參見上方「高解析度圖片支援」

行為變更（Behavior Changes）

這些變更不會破壞 API，但可能需要更新 prompt：

  1. 更字面化的指令遵循（特別是低 effort level）：模型不會自動推廣指令或推測未明確要求的事項
  2. 回應長度校準至感知任務複雜度：不再預設固定冗長度
  3. 預設較少 tool calls：更多使用推理，提高 effort 會增加工具使用
  4. 更直接、有主見的語氣：減少驗證式措辭與表情符號（相比 Opus 4.6 的溫暖風格）
  5. 更定期的進度更新：長 agentic traces 中更多中期狀態訊息（若已有強制狀態訊息的 scaffolding，可嘗試移除）
  6. 預設較少 subagents：可透過 prompting 調控
  7. 即時網路安全防護：涉及禁止或高風險主題的請求可能被拒絕；合法安全工作請申請 Cyber Verification Program

遷移指南

基本遷移檢查清單

  1. Thinking 設定:

- 移除 budget_tokens

- 改用 {"type": "adaptive"} + output_config: {"effort": "high"}

- 若需顯示思考過程：加 "display": "summarized"

  1. Sampling 參數:

- 移除所有非預設的 temperature, top_p, top_k

- 改用 prompt engineering 引導行為

  1. Token 計數:

- 更新 max_tokens 參數（預留 1.35x 空間）

- 重新測試 compaction triggers

  1. Effort Level:

- 程式碼與 agentic 任務考慮使用 xhigh

- 一般智慧任務至少使用 high

  1. Task Budget（可選）:

- 僅在需要模型自我調節 token 使用時設定

- 開放式任務不建議使用
