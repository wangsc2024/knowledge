1.2 資料模型

每筆筆記（Note）的完整 schema：

| 欄位 | 類型 | 說明 | 索引 |

|------|------|------|------|

| id | UUID | 主鍵 | PK |

| title | string | 筆記標題（支援標題搜尋） | FTS5 |

| contentText | string(MD) | Markdown 純文字（全文搜尋+向量化） | FTS5 + Vector |

| content | JSON | Tiptap JSON（前端編輯器用，由 contentText 自動生成） | — |

| tags | string[] | 標籤陣列 | 標籤索引 |

| source | enum | manual / import / web | — |

| createdAt | ISO datetime | 建立時間 | — |

| updatedAt | ISO datetime | 最後更新時間 | — |

| isDeleted | boolean | 軟刪除標記 | — |

1.3 搜尋引擎架構

| 端點 | 演算法 | 適用場景 | 回傳格式 |

|------|--------|---------|---------|

| GET /api/notes?limit=N | 直接列舉 | 標題 regex 搜尋 | {notes: [...]} |

| POST /api/search/keyword | BM25 (FTS5) | 精確詞彙匹配 | {results: [{title, score, ...}]} |

| POST /api/search/hybrid | BM25 + Vector | 語義+關鍵字混合 | {results: [{title, score, contentText, ...}]} |

| POST /api/search/retrieve | Hybrid + Format | 直接取得整合上下文 | {formattedContext: "..."} |

  │

三、資訊利用可行性分析

3.1 三種檢索方法比較

| 方法 | 精確度 | 召回率 | 適用場景 | 優缺點 |

|------|--------|--------|---------|--------|

| 標題搜尋 (regex) | ★★★★★ | ★★☆☆☆ | 知道確切筆記名 | 最快最準，但只搜標題 |

| BM25 關鍵字 | ★★★☆☆ | ★★★★☆ | 精確術語匹配 | 中文斷詞影響大，可能混入不相關結果 |

| Hybrid 混合搜尋 | ★★★★☆ | ★★★★★ | 概念性搜尋 | 語義理解佳，但分數區分度低(0.33-0.50) |

| Retrieve 整合 | ★★★★☆ | ★★★★☆ | LLM 直接消費 | 回傳格式化上下文，省去逐筆 fetch |

3.2 實測結果摘要

測試 1：「Agentic RAG 多代理檢索」（技術概念）

  • Hybrid 找到 Dify、FTS5 指南等相關筆記，語義理解正確
  • BM25 結果相似但排序不同
  • 建議：技術概念用 hybrid

測試 2：「楞嚴經 五十陰魔」（佛學專有名詞）

  • Hybrid 第 2 名命中目標，但第 1 名是 Crawl4AI（不相關）
  • BM25 同樣混入不相關結果
  • 建議：專有名詞搜尋應先用標題 regex，再 hybrid 補充

測試 3：「Canvas 碰撞偵測」（遊戲技術）

  • Hybrid 第 1 名精確命中目標
  • BM25 結果較差，混入 QA 報告和佛學筆記
  • 建議：遊戲技術用 hybrid 優於 keyword

3.3 組合搜尋策略（推薦）


研究型任務最佳搜尋流程：

Step 1: 標題精確搜尋
  GET /api/notes?limit=200 → regex filter
  ↓ 命中 → 直接讀取 contentText
  ↓ 未命中 ↓

Step 2: Hybrid 語義搜尋
  POST /api/search/hybrid {query, topK:10}
  ↓ score ≥ 0.4 的結果 → 讀取全文
  ↓ 結果不足 ↓

Step 3: BM25 關鍵字搜尋（備援）
  POST /api/search/keyword {query, topK:10}
  ↓ 交叉比對 Step 2 結果，補充遺漏

Step 4: Retrieve 整合上下文（LLM 消費用）
  POST /api/search/retrieve {query, topK:5}
  → 直接取得 formattedContext 供分析

五、示範深度研究報告

主題：「知識庫驅動的 Agentic RAG 架構——從本系統實踐看多層檢索策略的設計取捨」

系列：kb-architecture | 階段：foundation | 日期：2026-03-08
本文基於對 daily-digest-prompt 知識庫的實際架構分析

#### 一、主題概述

本知識庫採用 SQLite FTS5 + Qdrant 向量資料庫的雙引擎架構，提供三種搜尋模式。這種「多層檢索」設計是 Agentic RAG 的核心模式——Agent 根據任務性質動態選擇最適合的檢索策略。

根據 KB 筆記「SQLite FTS5 全文搜尋引擎完整指南」和「Dify — 開源 AI 工作流平台深度洞察」，我們已有 FTS5 和 RAG 管線的基礎知識。本文從實測數據出發，分析各檢索策略的實際表現。

#### 二、關鍵發現

發現 1：Hybrid Search 的分數正規化瓶頸

實測中 hybrid search 的分數集中在 0.333-0.500，區分度低。這源於 BM25 和向量分數的正規化難題——BM25 是非 bounded 分數，向量 cosine similarity 是 0-1，兩者融合需要 min-max 正規化，導致壓縮。

| 查詢類型 | Hybrid 分數範圍 | 有效區分度 |

|----------|----------------|-----------|

| 技術概念 | 0.333-0.500 | 低 |

| 佛學專有名詞 | 0.392-0.500 | 低 |

| 遊戲技術 | 0.333-0.500 | 低 |

發現 2：BM25 中文斷詞的盲點

「楞嚴經 五十陰魔」查詢中，BM25 將 Crawl4AI 爬蟲文章排在第一名。推測原因是 FTS5 的中文斷詞可能將「陰」「魔」等單字分別匹配到不相關文件。這是 CJK 語言 BM25 的已知限制。

發現 3：每筆記平均 10.6 chunks 的分塊策略

9,345 vectors / 883 notes ≈ 10.6 chunks/note。以平均 4,418 字/筆記計算，每 chunk 約 417 字。這在 RAG 最佳實踐中屬於適中的分塊大小（建議 200-500 字/chunk）。

發現 4：標籤體系的長尾效應

2,863 唯一標籤中，Top 30 標籤佔總使用次數的 ~40%，剩餘 2,833 個標籤佔 ~60%。這種長尾分佈表示標籤系統需要治理——建議建立核心詞彙表，將同義標籤合併。

#### 三、與既有知識的連結

  • 承接 KB 筆記「Dify — 開源 AI 工作流平台深度洞察」中 RAG 管線設計的啟示
  • 承接 KB 筆記「SQLite FTS5 全文搜尋引擎完整指南」中 BM25 與向量混合檢索架構
  • 補充 KB 中缺少的：實測數據驅動的搜尋策略選擇指南

#### 四、實踐應用

推薦的動態搜尋策略選擇器：


def select_search_strategy(query: str, task_type: str) -> str:
    """根據查詢特徵和任務類型選擇最佳搜尋策略"""

    # 1. 專有名詞（含中文經典名、技術術語）→ 標題優先
    if has_proper_nouns(query):
        return "title_first_then_hybrid"

    # 2. 概念性查詢 → hybrid 優先
    if task_type in ("research", "explore", "synthesis"):
        return "hybrid_with_retrieve"

    # 3. 精確程式碼/函數名 → BM25 關鍵字
    if has_code_tokens(query):
        return "keyword_first"

    # 4. 預設 → hybrid
    return "hybrid"

#### 五、結論與展望

本知識庫的「雙引擎+三種搜尋」架構已具備深度研究的基礎能力。主要優化方向：

  1. 搜尋策略自動選擇：基於查詢分析動態切換搜尋端點
