一、概述與定位

n8n（念作 "nodemation"）是一個開源的工作流自動化平台，從傳統的「無程式碼工作流」演進為「AI Agent 原生平台」，代表工作流自動化領域的典範轉移。它融合 500+ 整合、AI Agent、人工審核與自訂程式碼，打造可在生產環境可靠運行的彈性工作流。

核心差異化：n8n 不強迫使用者在「無程式碼」與「全程式碼」之間二選一，而是提供「視覺化 + 程式碼」的混合模式，讓非技術人員也能建構複雜的 AI 工作流，同時保留工程師的完整控制力。

---

二、六大 AI Agent 類型

n8n 基於 LangChain JavaScript 框架實作六種專責 Agent 節點：

| Agent 類型 | 用途 | 適用場景 |

|-----------|------|----------|

| Conversational Agent | 對話式互動 | 客服、FAQ 機器人 |

| ReAct Agent | 推理-行動循環 | 複雜多步驟任務、工具呼叫 |

| Plan and Execute Agent | 先規劃再執行 | 多步驟研究、報告生成 |

| OpenAI Functions Agent | OpenAI 函數呼叫 | 結構化輸出、API 整合 |

| SQL Agent | 資料庫查詢 | 自然語言轉 SQL |

| Tools Agent | 工具編排 | 多工具協調使用 |

Agent 運作模式：觀察環境 → 評估可用工具 → 決定行動 → 執行 → 重複直到目標達成。這與 ReAct 論文的 Reasoning + Acting 範式一致。

三、三層記憶系統

n8n 的記憶管理分為三個層次，分別對應不同場景：

1. Simple Memory（揮發性）

  • 訊息在工作流執行期間持續，重啟後消失
  • 用途：測試、單次互動、Demo

2. Window Buffer Memory（滑動視窗）

  • 保留最近 N 條訊息，舊訊息自動丟棄
  • 用途：成本控制、聚焦對話、避免 context overflow
  • 設計亮點：與 LLM token 限制解耦，開發者自訂視窗大小

3. Persistent Memory（持久化）

  • 支援 MongoDB、Redis、PostgreSQL、Zep 後端
  • Session ID 決定載入/儲存哪個對話
  • 用途：生產級聊天機器人、客服、需要對話歷史的應用

架構洞察：三層記憶系統與 Daily-Digest-Prompt 的 digest-memory.json（持久化跨次記憶）+cache/status.json（快取揮發性）設計有同構性。

四、Queue Mode 水平擴展架構

n8n 的 Queue Mode 是其生產級部署的核心，採用經典的生產者-消費者模式：

架構五元件


Traefik (反向代理 + SSL)
  ↓
n8n Main (UI + API + 排程，不執行工作流)
  ↓
Redis (訊息佇列，Job Broker)
  ↓
n8n Workers (執行引擎，可水平擴展)
  ↓
PostgreSQL (持久化儲存)

關鍵設計決策

  1. Main 不執行工作流：確保 UI/API 始終回應，不被重負載阻塞
  2. Worker 故障隔離：單個 Worker 崩潰不影響 Main 或其他 Worker
  3. Redis 緩衝：流量突增時 Job 在 Redis 緩衝，等 Worker 空閒時處理
  4. 動態擴展：docker compose up -d --scale n8n-worker=3 即可增加 Worker

效能調校

| 工作流類型 | N8N_WORKER_CONCURRENCY 建議 | 原因 |

|-----------|-------------------------------|------|

| I/O 密集（API 呼叫） | 10-20 | 大部分時間在等待回應 |

| CPU 密集（資料處理） | 1-5 | 避免 context switching 開銷 |

關鍵監控指標：Queue Depth — 若持續上升代表 Worker 無法跟上流量。

六、多 Agent 系統與人機協作

Multi-Agent 模式

n8n 支援多 Agent 協作，透過 Workflow Tool（Agent 觸發其他工作流）實現 Agent 間通訊。這是一種「工作流即 Agent 通訊協議」的設計，與 A2A（Agent-to-Agent）協議的精神一致但更輕量。

Human-in-the-Loop

  • 人工審核節點：在 Agent 決策前插入人工確認
  • 效能監控：內建審計軌跡追蹤 Agent 行為
  • 業務規則閘門：用條件節點限制 Agent 行動範圍

七、對 Daily-Digest-Prompt 系統的五大借鑑啟示

啟示 1：Main/Worker 分離模式

n8n 的「Main 不執行工作流」設計，對應我們的 PowerShell Phase 0（預計算，不呼叫 LLM）。這驗證了將調度層與執行層分離的價值。

啟示 2：Queue Depth 監控

我們的 state/run-fsm.json 目前缺少類似的佇列深度指標。可考慮追蹤待執行自動任務的累積數量，作為系統健康度的先行指標。

啟示 3：Worker Concurrency 按任務類型調整

n8n 按 I/O 密集 vs CPU 密集調整並發數。我們的 config/timeouts.yaml 已按任務類型設定 timeout，但未按任務特性調整並行度。

啟示 4：Memory 三層設計的正式化

n8n 的 Simple/Window/Persistent 記憶分類，可作為我們整理 digest-memory.json（持久化）、cache/*.json（視窗式）、Agent 上下文（揮發性）的參考框架。

啟示 5：Workflow-as-Tool 模式

n8n 的「Agent 可觸發其他工作流」設計，與我們的「自動任務 Prompt 可觸發子 Agent」模式類似。差異在於 n8n 有視覺化編輯器，而我們用 Markdown 模板。

八、與 Dify 的對比分析（跨主題連結）

| 維度 | n8n | Dify |

|------|-----|------|

| 核心定位 | 工作流自動化 + AI 擴充 | AI 應用開發平台 |

| Agent 實作 | LangChain JS（6 種 Agent） | 自研 Agent 引擎 |

| 視覺化 | 工作流視覺化 | 應用+工作流雙視覺化 |

| 擴展模式 | Queue Mode + Redis Worker | 微服務 + Celery |

| 整合數量 | 500+（偏自動化） | 50+（偏 AI 模型） |

| 適合場景 | IT 自動化 + AI 增強 | AI 原生應用開發 |

| Stars | 85K+ | 131K+ |

關鍵差異：n8n 是「自動化平台加 AI」，Dify 是「AI 平台加工作流」。兩者從不同方向走向同一個交集：AI 驅動的工作流自動化。
