一、專案概述

Dify（langgenius/dify）是目前 GitHub 上增長最快的開源 LLM 應用開發平台之一，截至 2026 年 2 月已累積 130,000+ Stars、20,300+ Forks、9,053 次 commit。它定位為 Production-ready Agentic Workflow Platform，將 RAG 管道、Agent 框架、模型管理、可觀測性與工作流引擎整合於一體，讓開發者從原型到生產部署僅需一個平台。

二、核心架構分析

2.1 Queue-Based Graph Engine（工作流引擎）

Dify 的工作流引擎經歷了從單執行緒單體架構到佇列驅動圖執行引擎的重大重構（Issue #24067），這是其最具技術深度的設計決策。

#### 舊架構的痛點

  1. 緊耦合：元件間高度依賴，測試與維護困難
  2. 單執行緒瓶頸：限制了可擴展性
  3. 狀態散落：狀態管理分散在各模組中
  4. 擴展困難：新增功能需深入修改核心

#### 新架構核心元件


┌────────────────────────────────────┐
│       Command Orchestrator         │  命令路由與調度
├────────────────────────────────────┤
│       Worker Pool Manager          │  動態伸縮工作執行緒
├─────────┬──────────┬───────────────┤
│ Node    │ Edge     │ Execution     │  狀態管理（分離式）
│ State   │ State    │ State         │
├─────────┴──────────┴───────────────┤
│       Event Router                 │  圖事件 vs 節點事件分離
├────────────────────────────────────┤
│       Response Coordinator         │  串流回應集中處理
└────────────────────────────────────┘

#### 執行流程

  1. 命令進入統一佇列
  2. Worker Pool 並行處理命令（動態伸縮）
  3. Event Router 路由到適當的處理器
  4. State Manager 追蹤節點/邊/執行狀態
  5. Response Coordinator 處理輸出串流

#### 動態伸縮參數

| 參數 | 用途 |

|------|------|

| GRAPH_ENGINE_MIN_WORKERS | 基線 Worker 數量 |

| GRAPH_ENGINE_MAX_WORKERS | 上限 Worker 數量 |

| GRAPH_ENGINE_SCALE_UP_THRESHOLD | 佇列深度觸發擴容 |

| GRAPH_ENGINE_SCALE_DOWN_IDLE_TIME | 閒置超時縮容 |

#### 並行執行模式（四種）

  1. Simple Parallel：Start → N 個並行分支 → End（最多 10 條）
  2. Nested Parallel：多層巢狀並行（最多 3 層）
  3. Iterative Parallel：迴圈內並行處理
  4. Conditional Parallel：條件分支並行

#### 錯誤處理策略

  • Retry：自動重試失敗節點
  • Default Value：失敗時使用預設值繼續
  • Fail Branch：觸發備用分支
  • Abort：終止整個工作流

2.2 Knowledge Pipeline（知識管道）

Dify 2.0 引入的 Knowledge Pipeline 是其 RAG 能力的革命性升級，將文件攝入與處理流程視覺化、模組化、可擴展化。

#### 設計理念

  • 將 RAG 資料處理管道延伸至工作流畫布體驗
  • 以 Plugin 架構支援多模態資料（文字、圖片、表格）
  • 每個步驟如拼圖般可自由組合

#### 核心能力

  1. 多資料來源整合：本地檔案、線上文件、雲端硬碟、網頁爬蟲
  2. 視覺化節點編排：拖拉式設計文件處理流程
  3. Plugin 驅動的攝入框架：透過插件無縫整合各類資料源
  4. DSL 匯出/匯入：Pipeline 可分享與重用
  5. Hybrid Retrieval：結合全文檢索與向量搜尋

2.3 Plugin 生態系統

Dify v1.0.0（2025 年 2 月）將所有模型與工具遷移至 Plugin 架構，實現了核心平台與擴展能力的徹底解耦。

#### Plugin 五大類型

| 類型 | 用途 |

|------|------|

| Models | AI 模型管理（OpenAI、Gemini、DeepSeek 等） |

| Tools | 領域專用能力（Perplexity、Discord、Slack 等） |

| Agent Strategies | Agent 推理策略（多步驟推理、工具選擇） |

| Extensions | 自訂端點與 API 整合 |

| Bundles | 多個 Plugin 的組合包 |

#### 安全與隔離

  • Plugin 運行於獨立 Go daemon 進程，與主 Python API 服務隔離
  • 每個 Marketplace 插件經過嚴格程式碼審查
  • 明確的資料處理宣告，使用者完全掌控權限
  • 支援遠端除錯（連接 SaaS 服務，操作轉發本地環境）

三、核心創新點分析

3.1 Graph Engine 的佇列驅動設計

傳統的 LLM 工作流引擎多採用 DAG（有向無環圖）直接執行模式，Dify 創新地引入佇列調度，實現了：

  • 非同步非阻塞 DB 寫入：複雜工作流的總執行時間趨近最長分支時間，而非所有分支的加總
  • 動態 Worker 伸縮：根據佇列深度自動調整並行度
  • 分層處理：可在執行流程中插入 Debug 日誌、限制層等擴展

3.2 Knowledge Pipeline 的「內容即工作流」理念

將 RAG 的文件處理從黑盒子變為視覺化、可編排的工作流，讓使用者能精確控制文件從「原始內容」到「結構化可檢索知識」的每一步轉換。

3.3 Plugin 的進程級隔離

不同於多數平台的函式級或容器級隔離，Dify 選擇用 Go daemon 提供 Plugin 運行時，兼顧效能與安全性。

四、可借鑑的技術項目

對照 Daily-Digest-Prompt 專案

| Dify 技術 | 借鑑方向 | 本專案對應 |

|-----------|---------|----------|

| Queue-Based Graph Engine | 工作流引擎佇列化 | run-*-team.ps1 的 Phase 並行模式可引入佇列排程 |

| 動態 Worker 伸縮 | Agent 並行度自適應 | 目前 Start-Job 數量固定，可依 API 狀態動態調整 |

| 四種錯誤處理策略 | 分級容錯機制 | 現有 retry 機制可擴展為 default-value / fail-branch 模式 |

| Knowledge Pipeline 視覺化 | RAG 管道可觀測性 | kb-curator Skill 可增加處理步驟追蹤 |

| Plugin 隔離執行 | Hook 獨立進程化 | hooks/*.py 目前同進程執行，可考慮隔離 |

| DSL 匯出/匯入 | 配置可攜性 | config/*.yaml 已外部化，可加 DSL 格式支援跨實例遷移 |

| Marketplace 生態 | Skill 分享機制 | skills/ 已模組化，可設計 Skill 匯出/匯入標準 |

具體可實施項目

  1. Phase 執行佇列化：將 run-agent-team.ps1 的 Phase 1 從固定 5 個 Start-Job 改為佇列模式，依 API 健康狀態（api-health.json）動態決定並行數
  2. Fail Branch 錯誤策略：在 Phase 2 組裝失敗時，不僅重試，還可觸發「降級分支」（如：只推播已成功的 Phase 1 結果）
  3. Skill DSL 標準：設計 Skill 的 DSL 匯出格式，讓 skills/ 下的 Skill 可在不同 daily-digest-prompt 實例間遷移
