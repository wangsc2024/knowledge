一、專案概述

OpenClaw（原名 Clawdbot）是 GitHub 史上增長最快的開源專案，60 天內從 9,000 飆升至 210,000+ Stars，以 MIT 授權的「本地優先個人 AI 助手」定位席捲開發者社群。其核心理念是：在你自己的硬體上運行 AI 助手，透過你已經在用的通訊軟體（WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等 23+ 頻道）與之互動。

核心數據

| 指標 | 數值 |

|------|------|

| GitHub Stars | 210,000+（2026-03） |

| 增長速度 | 60 天 9K → 210K（史上最快） |

| 授權 | MIT（允許商用） |

| 技術棧 | TypeScript / Node.js 22+ |

| 支援頻道 | 23+（WhatsApp、Telegram、Slack、Discord、Signal 等） |

| 社群技能 | ClawHub 5,700+ 個 |

| 記憶儲存 | 純 Markdown + YAML 檔案 |

| 排程機制 | Heartbeat 守護程序（預設 30 分鐘） |

二、核心架構：三層設計模型

OpenClaw 的架構可拆解為三個核心層：

2.1 Gateway（閘道器 — 神經中樞）

設計本質：單一 Node.js 程序，監聽 127.0.0.1:18789（WebSocket），扮演所有訊息的路由與認證中樞。

關鍵設計決策：

  • Loopback 綁定：預設只綁定 localhost，外部存取需透過 Tailscale/VPN 隧道 — 安全性與便利性的平衡
  • WebSocket 雙向通訊：保持長連接，支援即時串流回應
  • Session 隔離：每個 Agent/Workspace/Sender 各自獨立 Session，防止上下文交叉污染
  • 裝置配對認證：Challenge/Nonce 簽章機制，新裝置需明確核准

訊息處理迴圈：

  1. 平台送訊息至 Gateway
  2. Gateway 執行認證 + 路由檢查
  3. Agent Session 從檔案系統載入對話歷史
  4. LLM 接收訊息 + 可用工具/技能
  5. 模型處理請求、執行工具呼叫
  6. 回應串流回原始平台

2.2 Skill System（技能系統 — 可擴展能力層）

設計本質：「寫一個 Markdown 檔案，丟進目錄，Agent 即刻掌握」— 零編譯、零重啟的能力擴展。

技能定義格式：


# SKILL.md frontmatter
name: my-skill
version: 1.0.0
description: 技能描述
triggers:
  - 觸發關鍵字
allowed-tools: Bash, Read, Write

三層優先級：

  1. Workspace Skills（最高）：專案目錄下的技能
  2. Managed Skills：ClawHub 安裝的技能
  3. Bundled Skills（最低）：內建技能

ClawHub 生態系統：5,700+ 社群技能，涵蓋日曆管理、程式碼審查、研究自動化。具備 install/update/publish 功能，類似 npm/pip 的生態動態。

自我擴展能力：Agent 可以在 Session 中自行撰寫新 Skill 並立即調用 — 能力在執行時複合增長。

2.3 Heartbeat（心跳守護 — 主動排程層）

設計本質：將 AI 助手從「被動回應」轉變為「主動巡檢」的關鍵機制。

雙層檢查策略：

  • Tier 1（廉價檢查）：快速確定性腳本，檢查新郵件、行事曆變更、系統警報
  • Tier 2（模型升級）：僅在偵測到重大變化時才呼叫 LLM 進行解讀和決策

設計巧思：

  • 預設每 30 分鐘檢查一次，但只在有事值得報告時才發訊息
  • 讀取 HEARTBEAT.md 定義檢查邏輯，可配置頻率、目標聯絡人、活躍時段
  • 成本優化：避免無意義的 LLM API 呼叫

---

三、記憶系統：Markdown-First 設計哲學

核心決策：完全捨棄資料庫，所有狀態以 Markdown/YAML 檔案儲存於 ~/clawd/。

檔案結構

| 檔案 | 用途 |

|------|------|

| AGENTS.md | Agent 行為定義 |

| SOUL.md | 人格與核心指令 |

| TOOLS.md | 可用能力列表 |

| HEARTBEAT.md | 主動檢查配置 |

| memory/YYYY-MM-DD.md | 每日對話日誌 |

| MEMORY.md | 長期記憶事實 |

| skills//SKILL.md | 個別技能定義 |

為何選擇 Markdown？

  1. 可檢查性：任何文字編輯器都能直接查看 Agent 的知識與記憶
  2. 版本控制：Git 追蹤每次變更，完美對接開發者工作流
  3. 備份簡單：目錄複製即可完成備份
  4. 可搜尋：grep 即可搜尋所有歷史
  5. 透明度：消除 AI 黑箱，用戶完全掌控 Agent 知道什麼

---

四、頻道路由：23+ 平台統一抽象

Channel Layer 設計：標準化頻道轉接器，將各平台的協議特異性轉換為一致的內部格式。

支援平台：WhatsApp、Telegram、Slack、Discord、Signal、iMessage、Google Chat、LINE、Matrix、IRC、Microsoft Teams、BlueBubbles、Mattermost、Nextcloud Talk、Nostr、Synology Chat、Twitch、Zalo、WebChat 等。

優雅降級：功能不對應時自動降級（如純文字平台不渲染富文本），而非拋出錯誤。

統一路由規則：允許列表/封鎖列表在所有平台統一套用，集中配置一處管理。

五、安全架構：分層防禦模型

OpenClaw 的安全設計承認一個現實：「Prompt Injection 仍未解決」。

多層防護

| 層級 | 機制 | 說明 |

|------|------|------|

| L1 身份認證 | 裝置配對 + @mention | 群組需 @mention 才觸發 |

| L2 存取控制 | 允許列表/封鎖列表 | 未授權用戶訊息直接丟棄 |

| L3 權限粒度 | 讀/寫分離 | 讀郵件 vs 發郵件 分開授權 |

| L4 工具審批 | 危險操作確認 | 檔案刪除、Shell 命令、付款需明確同意 |

| L5 沙箱模式 | Docker 隔離 | 封鎖檔案寫入和 Shell 存取 |

| L6 模型魯棒性 | Reader Agent 模式 | 外部內容先由唯讀 Agent 摘要，再交給工具 Agent |

Cisco 安全揭露

Cisco 曾揭露 OpenClaw 存在「嚴重安全漏洞，包括供應鏈攻擊和 prompt injection」，強調技能整合前需進行安全審計。

六、爆發增長的五大因素分析

  1. Chat-Native 介面：不需學新 App，直接在日常通訊軟體中使用 AI
  2. 本地優先信任：資料不離開用戶設備，解決企業最大顧慮
  3. MIT 授權：允許商用，降低採用門檻
  4. 零摩擦擴展：Markdown Skill 即寫即用，ClawHub 生態滾雪球
  5. Ambient AI 定位：從「工具」變成「環境」，Heartbeat 讓 AI 主動服務

七、對 Daily-Digest-Prompt 系統的六大借鑑啟示

啟示 1：Heartbeat 雙層檢查策略

OpenClaw 做法：Tier 1 廉價腳本檢查 → Tier 2 模型升級（僅必要時）。

借鑑方向：目前系統的 Heartbeat 每次都觸發完整 Claude 呼叫。可引入 Tier 1 預檢（PowerShell 快速判斷是否有新任務/新事件），僅在有實質變化時才啟動 LLM Agent，預估可降低 40-60% 的 API 成本。

啟示 2：Self-Extending Skill 機制

OpenClaw 做法：Agent 可在 Session 中自行撰寫新 SKILL.md 並立即使用。

借鑑方向：目前系統的 Skill 只能由人工建立。可設計「Skill 自動生成」功能 — 當 Agent 發現某個重複操作模式時，自動提案新 Skill。搭配現有 skill-creator Skill 和 pre_write_guard.py 的保護，風險可控。

啟示 3：Reader Agent 安全模式

OpenClaw 做法：外部內容先由唯讀 Reader Agent 摘要消毒，再交給有工具權限的 Agent 處理。

借鑑方向：目前系統的 WebFetch 內容直接進入主 Agent 上下文。可在研究類任務中引入 Reader Agent 中間層，降低 Prompt Injection 風險。與現有的安全檢查（HTML/XML 移除 + injection 模式偵測）形成雙層防護。

啟示 4：多頻道統一路由

OpenClaw 做法：23+ 頻道透過 Channel Layer 統一抽象。

借鑑方向：目前系統僅有 ntfy 單一通知管道。可設計 Channel Layer 抽象（ntfy / LINE / Telegram / Discord），統一訊息格式，依優先級選擇管道。

啟示 5：裝置配對認證

OpenClaw 做法：Challenge/Nonce 簽章 + Token 範圍限定。

借鑑方向：目前系統的 Gun.js 聊天室使用共用密鑰。可借鑑 OpenClaw 的裝置配對機制，每個連線設備獲得範圍限定的 Token。

啟示 6：檔案即狀態哲學的驗證

OpenClaw 驗證了：daily-digest-prompt 採用的 Markdown + YAML + JSON 狀態管理模式是業界趨勢。OpenClaw 的 210K Stars 證明「捨棄資料庫、擁抱檔案」的架構決策是正確的。兩個系統的架構同構性（HEARTBEAT.md、SKILL.md、MEMORY.md、YAML 配置）並非巧合，而是 AI Agent 架構的趨同演化。

九、架構同構性深度分析

兩個獨立演化的系統呈現驚人的架構同構：

  1. HEARTBEAT.md：兩者都用 Markdown 定義排程邏輯
  2. SKILL.md：兩者都用 YAML frontmatter + Markdown body 定義能力
  3. MEMORY.md：兩者都用 Markdown 持久化長期記憶
  4. 本地優先：兩者都在用戶設備上執行，不依賴雲端
  5. 配置外部化：兩者都將行為配置從程式碼分離到 YAML/Markdown

這種趨同演化揭示了 AI Agent 架構的自然選擇壓力：當 Agent 需要可檢查性、可版本控制性、和人類可讀性時，Markdown + YAML 是最優解。

---

十、風險與挑戰

  1. 安全風險：授予 AI 檔案系統和 Shell 存取權是雙面刃，Cisco 已揭露重大漏洞
  2. 技能品質不均：ClawHub 5,700+ 技能品質參差不齊，缺乏嚴格審核機制
  3. 供應鏈攻擊：第三方技能可能包含惡意程式碼，類似 npm 的安全挑戰
  4. Prompt Injection 未解：官方文件坦承這是未解問題，Reader Agent 僅是緩解而非根治
  5. 營運負擔：自託管意味著用戶承擔基礎設施管理責任
