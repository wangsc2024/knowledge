everything-claude-code 深度研究：四層架構、Instinct 學習引擎與 Token 經濟學的 50K Stars 方法論解析

專案概述

everything-claude-code 是 Anthropic Hackathon 冠軍 Affaan Mustafa 開源的 Claude Code 配置增強層，GitHub ~50K Stars、6K+ Forks。歷經 10+ 個月密集使用打磨，將 Claude Code 從對話工具轉變為可編排的 AI Agent 平台，包含 13 個專用 Agent、50+ Skills、32 個命令、多語言規則體系與持續學習機制。

---

一、四層架構設計（核心方法論）

1.1 架構分層

everything-claude-code 採用明確的四層架構，從使用者互動到知識演化層層遞進：

| 層級 | 組件 | 職責 |

|------|------|------|

| 使用者互動層 | Commands + Rules | 32 個斜線命令 + 分層規則系統 |

| 智能層 | 13 Agents + 50+ Skills | 階層式委派 + 按需載入 |

| 自動化層 | Hooks（Pre/Post/Stop） | 事件觸發自動化 + 驗證閘門 |

| 學習層 | Continuous Learning v2 | 雙系統持續知識累積 |

1.2 設計哲學

  • 分層解耦：每層有明確的職責邊界，Rules 提供靜態約束、Agents 提供動態推理、Hooks 提供自動化執行、Learning 提供演化能力
  • 按需載入：Skills 僅在觸發時載入，避免常駐占用 Context Window
  • 權限最小化：每個 Agent 僅擁有完成其任務所需的最小工具集

---

二、13 Agent 階層委派模型

2.1 Agent 分類與職責

| 分類 | Agent | 工具權限 |

|------|-------|----------|

| 編排層 | Planner, Architect | 廣泛存取 |

| 開發層 | TDD Guide, Build Error Resolver, Refactor Cleaner | 程式碼+測試 |

| 品質層 | Code Reviewer, Security Reviewer, Go/Python/DB Reviewer | 唯讀 |

| 交付層 | E2E Runner, Doc Updater | 特定範圍 |

2.2 委派策略

  • Orchestrator Pattern：Planner/Architect 進行任務分解，再委派給專用 Agent
  • Sequential Phase：RESEARCH → PLAN → IMPLEMENT → REVIEW → VERIFY，每階段一個 Agent，產出作為下一階段輸入
  • Iterative Retrieval：子 Agent 回傳結果後，Orchestrator 評估是否需要追問，最多 3 輪循環

2.3 與 daily-digest-prompt 對比洞察

| 維度 | everything-claude-code | daily-digest-prompt |

|------|----------------------|---------------------|

| Agent 數量 | 13 個專用 Agent | Phase-based（查詢/執行/組裝） |

| 委派模式 | 階層委派（Orchestrator→Specialist） | 管線委派（Phase 1→2→3） |

| 權限控制 | Agent 層級工具限制 | Hook 層級規則攔截 |

| 持久化 | 子 Agent 無持久狀態 | results/*.json 中間檔案 |

三、Continuous Learning v2 — Instinct 學習引擎（最大創新）

3.1 核心概念

這是 everything-claude-code 最獨特的創新。v2 引入「本能（Instinct）」概念，取代 v1 的完整 Skill 直接生成：

  • Instinct：原子級學習單元，包含觸發條件 + 單一行為 + 信心分數 + 領域標籤 + 證據追蹤
  • 信心分數：0.3（暫定建議）到 0.9（高度確定），≥0.7 自動生效

3.2 觀察機制

為何用 Hooks 而非 Skills：Hooks 100% 確定性觸發，v1 的 Skill 觸發率僅 50-80%，遺漏大量模式。

兩個 Hook 運作：

  • PreToolUse：捕捉執行前的上下文
  • PostToolUse：記錄工具呼叫結果

資料串流至 observations.jsonl（本地私密）。

3.3 信心分數演化

| 提升因素 | 降低因素 |

|---------|----------|

| 重複觀察到相同模式 | 使用者明確更正 |

| 使用者未修改建議 | 長期未觀察到 |

| 外部匯入的類似 instinct 驗證 | 出現矛盾證據 |

3.4 Skill 演化流程


觀察(Hooks) → observations.jsonl → 背景分析(Haiku) → Instinct 生成
→ 3+ 相關 Instinct 聚類 → /evolve 命令 → Skill/Command/Agent

3.5 目錄結構


~/.claude/homunculus/
├── observations.jsonl     # 原始觀察日誌
├── instincts/personal/    # 自動學習的 instinct
├── instincts/inherited/   # 匯入的 instinct
└── evolved/               # 演化產物
    ├── agents/
    ├── skills/
    └── commands/

3.6 關鍵命令

  • /instinct-status：顯示已學習的 instinct 及分數
  • /evolve：聚類 instinct 為 skill/command
  • /instinct-export：匯出 instinct 庫
  • /instinct-import ：匯入他人 instinct

3.7 深度洞察

Instinct 系統本質上是 LLM 的條件反射建模。它將「觀察→模式識別→行為形成→信心校準」的認知循環外部化，使 Agent 能在無記憶的 LLM 基礎上建立類似「肌肉記憶」的持久行為模式。這與 daily-digest-prompt 的 digest-memory.json 記錄型記憶形成互補：前者捕捉行為模式（HOW），後者記錄事件狀態（WHAT）。

四、Token 經濟學與 Context 管理

4.1 模型選擇策略

| 任務類型 | 推薦模型 | 理由 |

|---------|---------|------|

| 探索/簡單編輯 | Haiku | 快速、低成本 |

| 多檔案實作（90%） | Sonnet | 最佳平衡 |

| 架構/安全 | Opus | 深度推理 |

核心原則：預設 Sonnet，僅在首次嘗試失敗或跨 5+ 檔案時升級 Opus。

4.2 Context Window 保護

  • 200K Window 在啟用過多 MCP 後可能僅剩 70K
  • 建議：配置 20-30 個 MCP，但同時啟用 ≤10 個，活躍工具 <80
  • 替代策略：將 Context 密集的 MCP（GitHub/Supabase）替換為 CLI + Skill 組合
  • 專用工具（如 mgrep）比通用 grep 節省約 50% Token

4.3 Session 管理

  • 階段間使用 /clear 清空 Context
  • Stop Hook（非 UserPromptSubmit）觸發狀態保存，避免延遲
  • 中間產出存為 .claude/*.tmp，每日新建避免 Context 汙染

五、安全模型 — AgentShield

5.1 五類掃描

  1. Secrets Detection：14 種模式偵測器（API Key、Token、密碼等）
  2. Permission Audit：Agent 工具權限過度開放檢查
  3. Hook Injection Analysis：Hook 注入風險評估
  4. MCP Server Profiling：MCP 伺服器風險分析
  5. Agent Config Review：Agent 配置弱點掃描

5.2 紅隊測試

三個 Opus Agent 對抗推理：攻擊者(Attacker)、防禦者(Defender)、審計者(Auditor)，超越模式匹配的深度安全評估。

5.3 與 daily-digest-prompt 對比

| 維度 | AgentShield | daily-digest-prompt Hooks |

|------|-------------|---------------------------|

| 方法 | 靜態掃描 + 紅隊推理 | Runtime 攔截（pre_bash/write/read_guard） |

| 觸發 | 按需掃描 | 每次工具呼叫 |

| 範圍 | 配置安全性 | 執行安全性 |

| 深度 | 3 Opus 對抗推理 | 規則匹配 + 正則 |

互補洞察：兩種方法覆蓋安全光譜的不同端——靜態（發現潛在漏洞）vs 動態（攔截即時威脅）。

七、自動化方案 — 命令與 Hooks 整合

7.1 命令系統（32 個）

| 類別 | 命令 | 用途 |

|------|------|------|

| 核心 | /plan, /tdd, /e2e | 工作流啟動 |

| 品質 | /code-review, /build-fix, /refactor-clean | 品質閘門 |

| 協作 | /multi-plan, /multi-execute | 多 Agent 編排 |

| 學習 | /learn-eval, /evolve | 持續學習 |

| 基礎建設 | /pm2, /sessions, /checkpoint | 生命週期管理 |

7.2 Hooks 自動化

  • PostToolUse：自動格式化（prettier）、類型檢查（tsc）、Lint
  • PreToolUse：阻擋非 README/CLAUDE 路徑的 .md 建立、提醒使用 tmux
  • Stop：掃描修改檔案的 console.log、保存 Session 狀態

7.3 並行化策略

  • Git Worktrees 實現並發任務
  • 最多 3-4 個同時 Claude 實例
  • Cascade Pattern：從左至右、從舊至新掃描
  • PM2 生命週期管理（v1.4.0 新增）

八、深度洞察與對 daily-digest-prompt 的啟示

8.1 最有價值的可借鑑項目

  1. Instinct 學習引擎：daily-digest-prompt 目前的「記憶」是事件記錄型（digest-memory.json），缺少行為模式的自動提取與演化機制。Instinct 系統的「觀察→模式→信心→演化」閉環值得借鑑。
  1. Token 經濟學實踐：用 CLI + Skill 替代 Context 密集的 MCP、專用搜尋工具節省 50% Token、模型選擇策略（預設 Sonnet + 按需 Opus）——這些都是可直接應用的優化。
  1. AgentShield 紅隊方法：三 Agent 對抗推理（攻擊者/防禦者/審計者）比 daily-digest-prompt 的規則匹配更深入，可作為 system-audit 的安全維度增強。

8.2 方法論差異比較

| 維度 | everything-claude-code | daily-digest-prompt |

|------|----------------------|---------------------|

| 定位 | 通用開發增強插件 | 專案特定的排程自動化系統 |

| 學習機制 | Instinct 信心演化 | 事件記錄 + 去重 Registry |

| 安全模型 | 靜態掃描 + 紅隊 | Runtime Hook 攔截 |

| 並行化 | Git Worktrees + PM2 | Phase-based Start-Job |

| 配置方法 | Plugin Marketplace 安裝 | 文件驅動架構 + YAML 配置 |

| 記憶持久化 | Session Hooks + .tmp 檔案 | digest-memory.json + research-registry.json |

| Skill 管理 | /skill-create + /evolve 自動生成 | 人工建立 SKILL.md + SKILL_INDEX.md |

8.3 融合建議

  1. 引入 Instinct 採集：在 post_tool_logger.py 增加行為模式識別，累積足夠次數自動建議新 Skill
  2. Token 經濟學審計：建立 Token 使用追蹤（每個 Sub-Agent 的 input/output tokens），識別優化點
  3. 紅隊安全層：在 system-audit 的安全維度加入 Agent 對抗推理，超越純規則匹配
  4. Skill 自動演化：借鑑 /evolve 命令，讓高頻使用的模板自動升級為正式 Skill
