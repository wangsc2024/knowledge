AI-Driven Requirements Engineering

LLM 輔助需求分析、使用者故事自動生成、驗收條件形式化與需求追溯性矩陣的工業級實踐

研究日期：2026-03-03

任務類型：ai_sysdev（AI 應用於系統開發）

---

一、技術概述

需求工程（Requirements Engineering, RE）是軟體開發生命週期中最關鍵卻最耗時的前期活動，傳統上高度依賴人類分析師的經驗與手動作業。2025-2026 年，LLM 正在根本性地改變 RE 的工作方式：從需求擷取（Elicitation）、分析（Analysis）、規格化（Specification）到驗證（Validation）的全流程，都出現了 AI 輔助甚至半自動化的工具與方法論。

根據實證研究，57.1% 的 RE 專業人員使用使用者故事（User Story）作為主要需求格式，其中 66.7% 已採用 LLM（主要是 ChatGPT）輔助作業，75% 的 LLM 使用者將其應用於使用者故事生成。LLM 輔助的需求撰寫速度比人類專家快 720 倍，成本僅為人工產出的 0.06%，且在利害關係人對齊度上高出 +1.12 分。然而，LLM 生成的使用者故事在多樣性與創造力上低於人類，驗收條件的滿足頻率也較低，因此「人機協作」（Human-AI Synergy）而非全自動化仍是當前最佳實踐。

二、核心方法與工具

2.1 四大 AI 技術在 RE 中的角色

| 技術 | 應用場景 | 具體能力 |

|------|---------|----------|

| LLM（GPT-4/Claude/Llama） | 需求生成、規格化、品質評估 | 使用者故事撰寫、SRS 文件產生、ISO 29148 合規檢查 |

| NLP | 需求擷取、模糊性偵測 | 從文件/郵件/對話中萃取需求、標記模糊語句 |

| ML 分類器 | 需求分類、一致性檢查 | 功能/非功能需求分類、矛盾偵測、缺口分析 |

| 預測分析 | 風險預測、瓶頸預警 | 基於歷史資料預測範疇蔓延、資源瓶頸 |

2.2 模型選擇策略

| 任務類型 | 推薦模型架構 | 代表模型 |

|---------|------------|----------|

| 需求理解/分類 | Encoder-only | BERT, RoBERTa |

| 需求生成/撰寫 | Decoder-only | GPT-4, Claude, Llama 3 |

| 摘要/轉換 | Encoder-Decoder | T5, BART |

| 追溯性分析 | RAG + LLM | GPT-4 + 向量搜尋 |

2.3 主要工具生態

  • GeneUS：學術研究工具，採用 RaT (Refine and Thought) Prompt 技術，從需求文件自動生成結構化使用者故事 + 測試案例，JSON 格式輸出可整合 Jira/Azure DevOps
  • RAAM（Requirements Analysis using AI for Mapping）：美國政府組織採用，需求生成加速 9,000%，追溯性分析時間減少 84%
  • Visure Requirements ALM：企業級平台，NLP 擷取 + ML 分類 + 自動追溯 + 即時驗證，支援 DO-178C/IEC 62304 等法規合規
  • TVR（Traceability Validation & Recovery）：RAG 方法，13 個 LLM 用於追溯連結的驗證與缺失連結恢復
  • Neuro-symbolic Pipeline：結合 LLM + SMT Solver，將自然語言需求形式化為可驗證的邏輯規格

三、實際工作流程（Step-by-Step）

3.1 使用者故事自動生成管線


步驟 1：文件擷取
  輸入：PRD/MRD/郵件/會議紀錄
  處理：NLP 解析 → 純文字萃取
  輸出：結構化需求片段

步驟 2：需求精煉（RaT Prompting）
  a) Refine Phase：移除無意義 Token、冗餘資訊
  b) Thought Phase：Chain-of-Thought 推理生成
  輸出：精煉後的需求陳述

步驟 3：使用者故事生成
  Prompt 模板：
  -------
  你是一位需求工程師。根據以下精煉需求，生成結構化使用者故事。
  每個使用者故事必須包含：
  - Who（角色）：系統的使用者或利害關係人
  - What（功能）：需要完成的具體動作
  - Why（目的）：業務價值或使用者目標
  - Acceptance Criteria（驗收條件）：Given-When-Then 格式
  - Definition of Done（完成定義）
  - Constraints（限制條件）
  -------
  輸出：JSON 格式使用者故事集合

步驟 4：測試案例規格化
  從驗收條件自動衍生測試案例
  包含：正常路徑、邊界條件、異常處理
  輸出：JSON 測試規格（可匯入 Jira/Azure DevOps）

步驟 5：品質評估
  使用 RUST 評估框架：
  - Readability（可讀性）
  - Understandability（可理解性）
  - Specifiability（可規格化程度）
  - Technical Aspects（技術面向）
  中位數評分目標：≥ 4/5

步驟 6：人工審查與修正
  專家驗證技術完整性、上下文特定性
  迭代精修直到品質通過

3.2 需求追溯性矩陣（RTM）自動化管線


步驟 1：需求文件向量化
  將所有需求文件轉為 Embedding 向量
  儲存於向量資料庫

步驟 2：連結建立
  使用 Embedding 相似度 + LLM 分類
  自動建立 需求 ↔ 設計 ↔ 測試 的追溯連結

步驟 3：驗證與恢復（TVR 方法）
  a) 驗證現有連結的正確性（RAG 查詢上下文）
  b) 偵測缺失連結並自動恢復
  c) LLM 生成連結理由（Rationale）

步驟 4：影響分析
  需求變更時自動計算影響範圍
  通知受影響的設計/測試/實作

4.1 Prompt Engineering for RE（九大主題框架）

LLM 需求工程的 Prompt 設計遵循九大主題：


# RE Prompt 工程框架
re_prompt_themes = {
    "context": "提供專案背景、領域、技術棧（e.g., '這是一個醫療設備管理系統'）",
    "persona": "指定角色（e.g., '你是 CMMI Level 5 認證的需求工程師'）",
    "templates": "提供輸出格式模板（e.g., 'As a [role], I want [feature] so that [benefit]'）",
    "disambiguation": "要求明確標記模糊語句（e.g., '標記所有含 should/may/some 的需求'）",
    "reasoning": "Chain-of-Thought（e.g., '先分析利害關係人，再衍生功能需求'）",
    "analysis": "多角度分析（e.g., '從安全性、效能、可用性三個角度評估'）",
    "keywords": "領域特定關鍵字（e.g., 'FDA 21 CFR Part 11 合規需求'）",
    "wording": "語言品質要求（e.g., '每個需求必須是單一、可測試的陳述'）",
    "few_shot": "提供高品質範例（最有效的 Prompting 策略）"
}

4.2 使用者故事生成 Prompt 範例


You are a senior requirements engineer specializing in {domain}.
Generate user stories from the following requirements document.

Format each story as JSON:
{
  "id": "US-001",
  "who": "<role>",
  "what": "<feature description>",
  "why": "<business value>",
  "acceptance_criteria": [
    {"given": "...", "when": "...", "then": "..."}
  ],
  "priority": "MoSCoW: Must/Should/Could/Won't",
  "nfr": ["ISO 25010 quality attributes if applicable"],
  "test_cases": [
    {"scenario": "...", "expected": "...", "type": "happy/edge/error"}
  ]
}

Requirements Document:
---
{refined_requirements}
---

IMPORTANT:
- Each story must be independently testable
- Flag ambiguous requirements with [AMBIGUOUS] tag
- Generate at least 3 acceptance criteria per story
- Include NFR considerations (performance, security, accessibility)

4.3 非功能需求自動推斷（ISO 25010 對齊）


def infer_nfrs(functional_requirements: list[str], domain: str) -> dict:
    """
    從功能需求自動推斷非功能需求。
    基於 ISO/IEC 25010:2023 品質模型的八大特性。
    研究顯示 Fine-tuned LLM 的分類準確率達 80.4%。
    """
    iso_25010_attributes = [
        "functional_suitability",  # 功能適用性
        "performance_efficiency",  # 效能效率
        "compatibility",           # 相容性
        "usability",               # 可用性
        "reliability",             # 可靠性
        "security",                # 安全性
        "maintainability",         # 可維護性
        "portability"              # 可移植性
    ]
    
    prompt = f"""
    Given these functional requirements for a {domain} system:
    {functional_requirements}
    
    For each ISO 25010 quality attribute, infer:
    1. Whether it's applicable (True/False)
    2. Specific NFR statement
    3. Measurable acceptance threshold
    4. Rationale for inclusion
    """
    # LLM 推斷 → 專家驗證 → 最終 NFR 清單
    return llm_generate(prompt)

六、與本專案的應用場景

6.1 Daily-Digest-Prompt 系統的直接受益點

  1. 規格驅動開發（Spec-Driven Development）增強

- 本專案已採用 specs//spec.md 規格驅動模式

- 可導入 LLM 輔助從口頭需求自動生成 spec.md 初稿

- 驗收條件（AC）自動從 spec 衍生，對應到 tasks.md

  1. 自動任務生成管線

- 目前 hour-todoist-prompt.md 的路由規則是手動維護的

- 可用 LLM 分析新增需求自動推斷任務類型、優先級、模板匹配

  1. 品質閘門自動化

- 現有 templates/shared/quality-gate.md 是靜態檢查清單

- 可用 LLM 動態評估每次 Skill 執行結果的品質分數

  1. 需求追溯性矩陣

- 建立 spec → plan → tasks → tests → evidence 的自動追溯

- 變更 spec 時自動通知受影響的下游文件

  1. Prompt 品質分析

- 用 RE 的 Prompt 九大主題框架評估系統中各 prompt 的品質

- 識別模糊指令、缺失上下文、不明確的驗收條件

七、風險與注意事項

  1. 幻覺風險：LLM 可能生成看似合理但技術上不正確的需求，尤其在輸入文件過長時（GeneUS 研究發現長文件增加幻覺率）
  2. 多樣性不足：LLM 傾向生成模式化的使用者故事，缺乏人類的直覺洞見與創新需求發現
  3. 驗收條件品質：LLM 生成的 AC 滿足頻率低於人類撰寫，需要專家審查
  4. 領域適應性：通用 LLM 在高度專業領域（醫療、航太、金融法規）需要額外微調或 RAG 增強
  5. 資料隱私：將內部需求文件送交雲端 LLM 存在資料外洩風險，敏感專案應考慮本地部署模型
  6. 信任建立：研究顯示利害關係人傾向信任人類撰寫的需求（即使 LLM 品質相當），透明化 AI 輔助流程是關鍵
  7. Prompt 依賴性：RE 的 Prompt 品質直接決定輸出品質，few-shot prompting 一致性地優於 zero-shot 和 chain-of-thought

八、Prompt Engineering 最佳實踐（九大主題）

根據系統性文獻回顧，RE 領域的 Prompt Engineering 歸納為九大主題：

  1. Context（上下文）：提供專案背景、領域、技術棧，避免去語境化的需求
  2. Persona（角色設定）：指定 RE 專家角色，提升專業用語準確度
  3. Templates（輸出模板）：提供結構化格式（JSON/Gherkin/表格），確保一致性
  4. Disambiguation（歧義消除）：要求標記模糊詞彙（shall/should/may），強制明確化
  5. Reasoning（推理鏈）：Chain-of-Thought 先分析再生成，RaT 先精煉再思考
  6. Analysis（多角度分析）：從安全/效能/可用性等面向交叉檢視
  7. Keywords（領域關鍵字）：注入法規/標準/術語，對齊領域知識
  8. Wording（措辭規範）：強制單一、可測試、無歧義的需求陳述
  9. Few-shot（範例引導）：提供 2-5 個高品質範例，是最有效的策略

---

九、Multi-Agent Debate（MAD）策略

研究發現 Multi-Agent Debate 策略能顯著提升 RE 品質：

  • F1-score：從 0.726 提升至 0.841（僅一輪互動）
  • Cohen's κ：與人類分析師達到實質一致（>0.7）
  • 機制：多個 LLM Agent 各持不同觀點，辯論後收斂到最佳需求表述
  • 適用場景：複雜需求的歧義消除、多利害關係人衝突解決
