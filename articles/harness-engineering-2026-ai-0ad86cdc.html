Harness Engineering 2026：AI Agent 系統優化完整指南

核心定義

Agent = Model + Harness

  • Model：基礎大型語言模型（GPT-4、Claude）
  • Harness：圍繞模型的所有約束、工具、回饋迴圈、文件和驗證系統
工程師職責從「直接寫程式碼」轉變為「設計讓 AI 可靠寫程式碼的環境」。

---

三大核心洞見（2026-04-22）

洞見 1：Autoresearch + Harness = Hill-Climbing Engine

來源：Harvey Legal Agents 實驗

機制：


Agent 嘗試任務 → LLM 依 rubric 評分 → Coding agent 分析失敗叢聚
→ 形成改進假設 → 修改系統組件 → 重新執行 → 循環

實證結果：

  • 12 個複雜法律任務
  • 平均成功率：40.8% → 87.7%（+115% 相對提升）
  • 5 個任務初始 2-7% → 最終 7 個任務 >90%（1 個達 100%）

關鍵引言：

"When the rubric is high quality, the agent can hill-climb surprisingly far."

---

洞見 2：Meta-Harness — 自動化優於手工

來源：Meta-Harness 論文（Stanford）

技術架構：

  • Coding Agent (Claude Code) 訪問完整歷史
  • 用 grep/cat 選擇性檢查（非一次性載入全部）
  • 偵測長期失敗 pattern
  • 生成新 Harness → 評估 → Iteration（40 次達收斂）

性能數據：

| 任務 | 指標 | Meta-Harness | 手工最佳 | 改進 |

|------|------|-------------|---------|------|

| Text Classification | Accuracy | ACE +7.7 pp | - | +7.7% |

| | Context Tokens | 11.4K | 50.8K | -77% |

| TerminalBench-2 (Opus 4.6) | Pass Rate | 76.4% | 74.7% | 排名 #2 |

| TerminalBench-2 (Haiku 4.5) | Pass Rate | 37.6% | - | 排名 #1 |

與現有方法對比：

  • TextGrad/OPRO：0.002-0.026M tokens/iteration
  • Meta-Harness：~10M tokens/iteration（多 1000 倍診斷資訊）

---

洞見 3：Context 排名比填充重要 10 倍

來源：Shaped.ai Context Window 優化

核心物理約束：

"Attention cost scales O(n²) with context length"

量化對比：

| 策略 | Tokens/Query | 成本/月（10K 查詢/日） | 品質 |

|------|-------------|---------------------|------|

| Naive RAG（200 chunks） | 50,000 | $48,000 | 中 |

| Shaped 排名（10 結果） | 2,500 | $4,500 | 高 |

| 改進 | -95% | -91% | +20% |

為什麼 Vector Similarity 失敗？：

"Cosine similarity measures distance in embedding space. It does not measure usefulness."

正確做法：多檢索器管線（6 層）：

  1. Vector search（語義）
  2. Lexical search（關鍵字）
  3. Collaborative filtering（用戶行為）
  4. Trending signals（實時熱度）
  5. ML scoring ensemble（排名模型）
  6. Diversity reordering（多樣性）

結果：從 10,000+ 候選中選出 10 個相關結果，<50ms

雙重控制機制

Guides（前饋控制）

  • 定義：預先引導 Agent 行為
  • 實現：文件、規則、結構強制
  • 案例：Red Hat Repository Impact Map 使用 LSP/MCP 掃描程式碼庫作為單一真相來源

Sensors（反饋控制）

  • 定義：觀察輸出並啟用自我修正
  • 實現：自訂 linter 訊息包含修正指令
  • 效果：LangChain 實驗中 Terminal Bench 分數從 52.8% 提升至 66.5%（無需更換模型）

---

約束演化管線（六步驟）

  1. 強制函數：採用「零人工程式碼」約束
  2. 宣告式意圖：高層級規格，非低層級指令
  3. 全面追蹤：每個 Agent 動作可追溯與失敗分析
  4. 編碼修正：將修正轉為可重用系統約束
  5. 混合驗證：交替使用規則與 LLM 驗證以平衡速度/準確度
  6. 定期重構：排程重構 Agent 防止長期漂移累積

八項深化優化

優化 H：Autoresearch 自動化改進迴圈

實施路徑：

  • Phase 1（2 週）：建立高品質 rubrics（35 個任務各 5-10 維度）
  • Phase 2（3 週）：Autoresearch Engine（失敗叢聚分析 + 假設形成）
  • Phase 3（持續）：每週六自動執行，發現 P0 假設自動通知

預期收益：

  • Task 成功率：75% → 90%（3 個月內）
  • 人工改進工作量：-60%

---

優化 I：Context Window 智能排名

實施步驟：

  1. 用戶行為追蹤（kb_feedback 欄位）
  2. ML 排名模型訓練（LightGBM）
  3. 整合進 KB API（top-5 模式）

預期收益：

  • Context tokens：-60%
  • 查詢品質：+15-25%

---

優化 J：Embedded Rubrics（分數嵌入錯誤訊息）

範例：


❌ Error: frequency-limits.yaml missing 'daily_limit'

📊 Rubric Impact:
- 完整性：-2 分
- 當前總分：6/10（警戒線：7/10）

💡 Fix（預期改進 +3 分）:
1. Open config/frequency-limits.yaml
2. Add: daily_limit: 6
3. Run: validate_config.py

優化 K：Harvey-Style Legal Rubrics

Tech Research Rubric 範例（總分 10.0）：

| 維度 | 權重 | 說明 |

|------|------|------|

| 來源品質 | 25% | ≥3 個一手來源（論文/官方文檔/GitHub）= 5 分 |

| 三角佐證 | 30% | ≥3 個獨立來源佐證核心論點 = 5 分 |

| 實用性 | 20% | 程式碼範例 + 部署指南 + 常見問題 = 5 分 |

| 完整性 | 15% | 涵蓋：定義/原理/實作/優缺點/適用場景/替代方案 = 5 分 |

| 結構品質 | 10% | 標題清晰/表格輔助/程式碼格式化/引用完整 = 5 分 |

品質門檻：

  • ≥8.0：優秀，直接發布
  • ≥7.0：良好，Minor 修訂後發布
  • ≥6.0：及格，需要改進特定維度
  • <6.0：不及格，重新研究或放棄

核心原則

約束解決空間原則

「約束越多，輸出越可預測。」
  • 未約束模型：浪費 token 探索死胡同路徑
  • 良好設計 Harness：開闢窄而明確的成功路徑

人類責任轉移

「編碼 Agent 沒有社交問責、沒有對300行函數的審美厭惡、沒有『我們不這樣做』的直覺。」

有效 Harness 將隱性開發者經驗外部化為明確、可測量的控制。

人類責任從「防止所有錯誤」轉為「將輸入導向最重要之處」。
