Daily Digest Prompt 專案優化報告

優化摘要

本次優化針對 deep scan 識別的 3 個問題進行修正，涵蓋配置一致性、效能瓶頸與文件同步。

修正項目

1. timeouts.yaml 幽靈 key 移除

  • 問題：phase2_timeout_by_task 含 creative_game key，但 frequency-limits.yaml 只有 creative_game_optimize
  • 影響：幽靈 key 不會被任何 auto-task 匹配，佔用配置空間且可能造成維護混淆
  • 修正：移除 creative_game，僅保留 creative_game_optimize: 900，註釋合併

2. post_tool_logger.py ErrorClassifier 單例化

  • 問題：每次 Bash 工具呼叫都在 main() 內新建 ErrorClassifier() 實例（第 248 行）
  • 影響：每日約 44 次 Agent 執行 x 20-100 次 Bash 呼叫 = 880-4400 次不必要的物件建立
  • 修正：改為模組層級單例 _error_classifier = ErrorClassifier()，在 import 時初始化一次
  • 向下相容：ImportError 時 _error_classifier = None，AGENT_GUARDIAN_AVAILABLE flag 不變

3. SKILL 計數同步（CLAUDE.md vs SKILL_INDEX.md）

  • 問題：CLAUDE.md 多處寫「22 個 Skill」，但實際有 23 個（17 核心 + 6 工具），SKILL_INDEX.md 正確
  • 根因：新增 git-smart-commit Skill 後 CLAUDE.md 未同步更新
  • 修正：CLAUDE.md 3 處 22→23、5 工具→6 工具

驗證結果

  • 語法檢查：post_tool_logger.py 通過 py_compile
  • 配置驗證：timeouts.yaml 結構正確（4 個 task key）
  • 測試套件：532 個測試全部通過（11.42s，無回歸）

技術洞見

  • Hook 模組層級單例模式適用於無狀態分類器（ErrorClassifier），但不適用於有狀態元件（LoopDetector 需依 session 初始化）
  • 配置 YAML 的 key 應與 frequency-limits.yaml 保持一對一對應，建議未來加入 validate_config.py 交叉驗證
  • SKILL 計數散布在 CLAUDE.md 多處，建議未來改為引用 SKILL_INDEX.md 的單一定義處（符合 DRY 原則）
