oh-my-codex（OMX）— Multi-Agent Orchestration Layer 深度洞察

專案概述

oh-my-codex（Yeachan-Heo/oh-my-codex）是韓國開發者 Yeachan Heo 建立的 Multi-Agent Orchestration Layer，定位為 OpenAI Codex CLI 的工作流編排層。2026 年 4 月 Trendshift 排名第二，7,700+ Stars。同作者另有姊妹專案 oh-my-claudecode（針對 Claude Code），兩者共享核心架構理念但分別針對不同 LLM 平台優化。

核心哲學：「You are the conductor, not the performer」 — 使用者是指揮家，AI Agent 是演奏者。OMX 不取代底層 LLM CLI，而是在其上提供結構化的編排、記憶與品質閘門。

---

一、五大架構支柱

1. Staged Team Pipeline（分階段團隊管線）

OMX 的團隊執行遵循嚴格的五階段管線：


team-plan → team-prd → team-exec → team-verify → team-fix (loop)
  • team-plan：架構設計與權衡分析，需人工審批才進入下一階段
  • team-prd：產品需求文件生成，明確驗收標準
  • team-exec：N 個 Worker 並行執行，每個 Worker 自動獲得獨立 git worktree
  • team-verify：驗證階段，確認產出符合 PRD
  • team-fix：修復迴圈，直到驗證通過

可借鏡：本專案（daily-digest-prompt）的 Phase 0→1→2 管線可強化為 plan→exec→verify→fix 四階段閉環，在 Phase 2 組裝後加入 verify 階段自動驗證結果完整性。

2. Scope-Based Skills 階層（範圍式技能繼承）


Project scope (.omx/skills/) → 優先級高、版本控制、團隊共享
User scope (~/.omc/skills/) → 跨專案 fallback、個人偏好
  • Skills 使用 YAML frontmatter + 觸發關鍵字自動注入
  • /learner 指令從對話中自動萃取可複用模式，存為 Skill
  • 嚴格品質門檻過濾低質量 Skill

可借鏡：本專案的 skills/ 目前是單層結構。可考慮引入「專案 Skill」與「全域 Skill」雙層優先級，讓跨專案的通用 Skill（如 ntfy-notify）自動 fallback，專案特有 Skill 覆寫全域版本。

3. 五類 MCP Server（持久狀態與程式碼智慧）

| MCP Server | 功能 | 持久化位置 |

|-----------|------|----------|

| State Server | 執行模式讀寫 (state_read/write/clear) | .omx/state/ |

| Memory Server | 跨 context window 記憶 | .omx/project-memory.json |

| Notepad Server | 優先級筆記（抗 context 壓縮） | .omx/notepad.md |

| Code Intelligence | LSP hover/goto/references/diagnostics | 即時 |

| Trace Server | 執行追蹤可視化 | .omx/logs/ |

可借鏡：本專案的 context/continuity/ 和 context/digest-memory.json 類似 Memory Server，但缺少 Notepad 概念。可引入「優先級筆記」機制 — 在 context 壓縮時保留關鍵決策備忘，防止長時間執行的 Agent 遺忘核心目標。

4. Hook 可擴展系統


omx hooks install --presets=workspace-context,memory,safety,review
omx hooks status

六種 Hook 事件（oh-my-claudecode 版本）：

  • session-start / stop：生命週期管理
  • keyword-detector：每次 prompt 攔截
  • ask-user-question：輸入請求回呼
  • pre-tool-use / post-tool-use：工具呼叫前後包裝

支援 OpenClaw 整合，將事件轉發至外部閘道（使用 {{sessionId}}、{{toolName}} 等模板變數）。

可借鏡：本專案的 Hook 系統已相當成熟（6 個 Hook），但缺少「Hook 預設套件」概念。可將常用 Hook 組合打包為 preset（如 safety = pre_bash_guard + pre_write_guard + pre_read_guard），簡化新專案初始化。

5. 智慧模型路由（Cost Optimization）

| 任務複雜度 | 模型層級 | 用途 |

|-----------|---------|------|

| Simple | Spark（Haiku 級） | 格式化、查表 |

| Standard | Default（Sonnet 級） | 實作、測試 |

| Complex | xhigh（Opus 級） | 架構設計、深度除錯 |

可借鏡：本專案的 config/llm-router.yaml 已有類似機制（mapping 格式，O(1) lookup），但 OMX 的三層分級更簡潔。可考慮在 router 中加入「任務複雜度自動偵測」，而非完全依賴手動 task_type 對應。

二、獨特執行模式

| 模式 | 特色 | 適用場景 |

|------|------|--------|

| Autopilot | 自主端到端，含自我修正迴圈 | 完整功能開發 |

| Ralph | 持續驗證/修復直到完成 | 不允許 partial 的嚴格任務 |

| Ultrawork | 最大並行（5+ Agent） | 批次修復 |

| Pipeline | 嚴格順序處理 | 多步驟轉換 |

| Team | 分階段管線 + 驗證迴圈 | 協調式團隊開發 |

Ralph 模式值得特別注意：它不允許「silent partial」— 任務要嘛完全完成，要嘛明確報告卡住的原因。這與本專案的 DONE_CERT 認證機制理念一致，但 Ralph 更進一步：它會自動重試直到 verify 通過或達到上限。

---

三、Git Worktree 隔離策略

OMX v0.10.0 的一大突破是 自動 worktree 隔離：

  1. 每個 Team Worker 自動獲得獨立 worktree：.omx/team//worktrees/worker-N
  2. 三種整合策略：

- Merge（--no-ff -X theirs）：Worker 清潔推進

- Cherry-pick：歷史分岔時

- Cross-worker rebase：順序依賴時

  1. 跨 worktree 狀態共享：OMX_TEAM_STATE_ROOT 確保所有 Worker 讀同一份狀態

可借鏡：本專案的 run-todoist-agent-team.ps1 使用子 Agent（.claude/worktrees/），但缺少自動整合策略。可參考 OMX 的三策略模型，在 Phase 3 組裝時根據子 Agent 結果衝突程度選擇合適的整合方式。

---

四、Autoresearch 自主研究模式

omx autoresearch 提供迭代式自主研究：

  • 自動探索主題 → 精煉問題 → 綜合發現
  • 多輪迭代，每輪深化前一輪的發現
  • 自動終止：當研究目標被滿足時自動停止

可借鏡：本專案的 kb-research-strategist Skill 已有五階段模型（foundation→synthesis），但缺少「自動終止判定」。可參考 OMX 的 autoresearch，加入研究飽和度自動偵測 — 當新搜尋結果與已知發現的重疊率超過閾值時，自動結束研究迴圈。

五、與 daily-digest-prompt 對比分析

| 面向 | oh-my-codex (OMX) | daily-digest-prompt |

|------|------------------|--------------------|

| 定位 | Coding Agent 編排 | 每日任務自動化 Agent |

| 管線模型 | plan→prd→exec→verify→fix | Phase 0→1→2 |

| 記憶機制 | MCP Server + notepad | continuity JSON + digest-memory |

| Skill 系統 | 雙層 scope + 自動萃取 | 單層 SKILL_INDEX + 手動維護 |

| Hook 系統 | 6 事件 + preset 套件 | 6 Hook + 逐一配置 |

| 模型路由 | 三層自動分級 | llm-router.yaml mapping |

| 狀態共享 | OMX_TEAM_STATE_ROOT 環境變數 | results/*.json 檔案交接 |

| 品質閘門 | verify→fix 自動迴圈 | DONE_CERT + Draft-Verify |

六、七大可借鏡技術項目（優先級排序）

P0：高價值、低成本

  1. Notepad 抗壓縮記憶：在 context/ 新增 notepad.md，context 壓縮時優先保留此檔內容。成本：修改 preamble.md 加入讀取指令
  2. Hook Preset 套件化：將現有 6 個 Hook 打包為 safety/logging/full preset，簡化配置。成本：新增 hooks/presets.yaml

P1：高價值、中等成本

  1. Verify→Fix 自動迴圈：Phase 2 後加入驗證階段，自動檢查結果完整性並觸發修復。成本：修改 run-agent-team.ps1 加入 Phase 2.5
  2. 研究飽和度自動終止：為研究任務加入重疊率偵測，超過 80% 重疊自動停止。成本：修改 dedup-policy.yaml + 研究模板

P2：中等價值

  1. 雙層 Skill Scope：引入全域 Skill fallback 機制（~/.claude/skills/ → 專案 skills/）。成本：修改 SKILL_INDEX.md 索引邏輯
  2. 跨 Worktree 狀態共享：用環境變數或共用目錄確保所有子 Agent 讀同一份狀態。成本：修改 PS1 腳本注入環境變數
  3. 任務複雜度自動偵測：在 llm-router 中加入啟發式複雜度評分，自動選擇模型層級。成本：修改 tools/llm_router.py
