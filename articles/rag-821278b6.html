生成式資訊存取的架構典範轉移：從被動 RAG 到主動知識合成引擎的設計原則與能力邊界

主筆說明

本報告由李飛飛視角（主筆，Human-centered AI、資料即科學儀器、空間智能與治理雙軸）與 Yann LeCun 視角（共同主筆，世界模型 > 純 LLM、架構創新週期、開放科學）共同主筆，基於兩位公開著作、演講與訪談論述推斷而成；非本人觀點，非所屬機構官方立場。

---

TL;DR

檢索增強生成（RAG）在 2023–2026 年間已從「補丁式知識注入」演化為「自主推理型知識合成」，但這場轉型暴露了一個根本性張力：工程補丁的精緻化，能否取代真正的認知架構創新？本報告梳理從 Naive RAG → Advanced RAG → Agentic RAG 的三代演進脈絡，論證主動合成在多跳推理任務上顯著優於被動查找（+13–21% 準確率 [11]），同時指出評估框架的多維化（RAGAS → MiRAGE）是落地部署的迫切需求。最終主張：RAG 是一條工程上必要但架構上過渡的道路——真正的知識整合需要具備世界模型的生成式智能體。

---

研究問題與範疇

核心問題：生成式資訊存取系統（Generative Information Retrieval, GenIR）的設計典範，正在經歷怎樣的根本性轉移？主動知識合成架構在哪些場景中真正優於被動 RAG？其能力邊界在哪裡？

範疇：2023–2026，技術架構（向量/圖/層次/代理型）、生成策略（自我反思/迭代/代理型）、評估框架；排除純 fine-tuning 方法、edge device 優化。

方法論：系統性文獻回顧（16 篇論文，A 級 9 篇）+ 三角佐證驗證 + 批判性分析（Phase 7 CRITIQUE 通過）。

文獻探討

第一代：Naive RAG 的設計哲學與結構性缺陷

2020–2023 年間主導的 Naive RAG 架構遵循「查詢→向量相似度搜尋→送 LLM 生成」的線性管線。Ai 等人（2025, arXiv 2501.02842）[2] 在 Foundations of GenIR 中系統性分析生成式 AI 對資訊存取的影響，區分了兩種核心能力：資訊生成（information generation，直接回應用戶需求）與資訊合成（information synthesis，整合並重組現有知識），奠定了理解 RAG 演進的概念框架。

系統性綜述（Gao et al., 2023, arXiv 2312.10997）[5] 識別出 Naive RAG 的三大結構性缺陷：（1）語意錯配（query-document semantic gap）；（2）脆弱的單輪查詢；（3）生成-檢索分離——LLM 生成過程對「自己何時知識不足」毫無感知。

HyDE（Hypothetical Document Embeddings，Gao et al., 2023）[6] 試圖解決語意錯配問題，透過讓 LLM 先生成「假設答案文件」再用其 embedding 進行檢索。實驗顯示 HyDE 在開放域 QA 中提升了 5–15% 的召回率，但同時引入了幻覺風險——語意準確但事實錯誤的假設文件會污染後續檢索 [6]。

第二代：Advanced RAG 的結構創新

RAPTOR（Sarthi et al., ICLR 2024, arXiv 2401.18059）[7] 採用遞歸式摘要樹狀結構：底層為原始文件塊，上層為遞歸聚類摘要，形成多粒度知識表示。在 QuALITY benchmark 上，RAPTOR + GPT-4 比基準 RAG 提升了 20% 的絕對準確率，顯示層次化知識合成在長文檔推理中的優勢。

GraphRAG（Edge et al., Microsoft Research, 2024）[8] 將知識圖譜引入 RAG，透過三步驟：實體-關係抽取 → Leiden 社區檢測 → 層次化摘要，建立結構化知識網絡。與傳統向量 RAG 相比，GraphRAG 在全局主題理解型問題上準確率提升至 3.4 倍 [8]，但全局查詢的 token 消耗增加 10–100 倍。

第三代：Agentic RAG 與主動知識合成

Self-RAG（Asai et al., 2024）[1] 是轉型的關鍵里程碑。透過引入「反思令牌」（reflection tokens），模型學會在生成過程中動態判斷：是否需要額外檢索（[Retrieve]）、已檢索內容是否相關（[ISREL]）、生成是否有根據（[ISSUP]）。Self-RAG 在 FEVER、PopQA、Arc-Challenge 等多個 benchmark 上超越標準 RAG，展示「自我批判式生成」的架構優勢 [1]。

FLARE（Jiang et al., 2023）以機率論視角解決主動知識獲取：當 token 生成機率低於門檻時，觸發前瞻性查詢。這與人類閱讀時「感知自身不確定、主動查閱資料」的認知模式高度相似。

最新的 Agentic RAG 框架（VLDB 2025）[11] 將 RAG 重新定義為序貫決策過程：基於 GPT-4o 和 Llama-3 70B 的實驗顯示，Agentic RAG 比 Vanilla RAG 準確率高 13–21%。Reasoning RAG（arXiv 2506.10408）[14] 進一步區分 System 1（快速直覺）和 System 2（慢速推理）兩種 Agentic RAG 模式，顯示複雜任務需 System 2 路徑但延遲增加 3–8 倍。

企業部署的知識編排挑戰

Context Kubernetes（Mouzouni, arXiv 2604.11623）[3] 提出了企業知識編排的宣告式架構，比喻為容器編排問題：「將正確知識、以正確權限、以正確新鮮度，傳遞給正確代理」。其三層代理許可模型確保代理權限永遠是人類權限的嚴格子集，TLA+ 形式驗證在 460 萬個可達狀態中零違反，ACL 過濾消除跨域洩漏，意圖路由將噪音降低 19 個百分點 [3]。

評估框架的演進

RAGAS（Es et al., EACL 2024, arXiv 2309.15217）[9] 建立了首個無需人工標注的 RAG 評估框架，核心指標包含 Faithfulness、Answer Relevancy、Context Precision/Recall，目前每月處理超過 500 萬次評估。然而，MiRAGE（arXiv 2510.24870）[12] 的研究顯示，標準文本 RAG 評估指標無法移植到多模態場景——視覺資訊的接地性、圖文對齊品質、跨模態推理一致性，都需要獨立評估維度 [12]。

核心論證

論證一：語意錯配是 RAG 的原生缺陷，工程補丁的效益遞減

主張：Naive RAG 的根本侷限源於查詢-索引語意錯配，歷代 Advanced RAG 技術雖有效緩解，但工程複雜度和計算成本呈指數增長，存在效益遞減。

三角佐證：

  1. RAG Survey（arXiv 2312.10997）[5] 系統性識別 lexical-semantic gap 為失敗主因
  2. Adaptive Retrieval（arXiv 2501.12835）[13] 顯示即使是 Self-RAG 風格的方法，面對歧義查詢時不確定性校準仍然不良
  3. HyDE 實作報告 [6] 記錄在 well-specified fact-bound domains 中幻覺污染問題未解

分析：每一代補丁（向量化→假設嵌入→層次摘要→自我反思）都帶來邊際效益，但 Naive RAG < Advanced RAG（×5 複雜度）< GraphRAG（×50–100 倍 token 消耗）。解法的複雜度正在逼近所解問題本身的複雜度——這是工程優化路線的內在矛盾。

論證二：主動合成在多跳推理的優勢可量化，但伴隨系統性新風險

主張：Self-RAG 和 Agentic RAG 在複雜推理任務中具有顯著且可重現的優勢，但「主動性」引入了新的失敗模式。

三角佐證：

  1. Self-RAG（Asai et al., 2024）[1] 在 5 個 benchmark 上超越標準 RAG 和 GPT-4
  2. Compound AI（arXiv 2511.18298）[10] 顯示 Agentic 組合在 Science Fact 任務上比 Vanilla RAG 高 13–21%
  3. Reasoning RAG（arXiv 2506.10408）[14] 揭示 System 2 Agentic RAG 雖準確但延遲增加 3–8 倍，產生實際部署的成本-效益張力

分析：主動推理的代價是「代理行為的幻覺」——錯誤的工具調用序列、迴圈查詢、確信但錯誤的檢索判斷。這不是邊際問題，而是代理架構的結構性風險。

論證三：現有評估框架不充分性正成為架構進步的隱性瓶頸

主張：RAG 評估框架的演進速度落後於架構複雜度增長，現有指標體系已不足以捕捉主動合成系統的真實能力與失敗模式。

三角佐證：

  1. RAGAS（arXiv 2309.15217）[9] 的 Faithfulness/Relevancy 二維框架在多跳推理中間步驟評估中存在盲點
  2. MiRAGE（arXiv 2510.24870）[12] 展示視覺 RAG 需要獨立的接地性指標——現有文本 RAG 指標「看不到」圖文推理錯誤
  3. Systematic RAG Review（arXiv 2507.18910）[15] 識別業界與學術評估的系統性分歧：End-to-End Task Completion 比 QA accuracy 在工業落地場景更有代表性

一家之言

以李飛飛視角（主筆）

當我思考 RAG 的演化，我最在意的問題不是「哪個 benchmark 更高」——而是「這套系統服務了誰？」

ImageNet 教會我一件事：技術的真正突破往往來自於重新定義問題，而不是優化舊問題的答案。Naive RAG 把知識存取問題定義為「向量相似度搜尋」——這個定義本身就已經劃定了天花板。當我看到 RAPTOR 用樹狀摘要、GraphRAG 用知識圖譜，我看到的是研究社群正在掙脫這個過窄的問題定義。這是令人振奮的方向。

但我更憂慮的是評估框架的缺失。RAGAS 的 Faithfulness 指標是一個工程起點，不是科學終點。當我們問「這個 RAG 系統好不好」，我們應該問：它是否縮小了不同群體在資訊取用上的差距？它對沒有被充分代表的語言和文化族群是否公平？Jais 和 Jais-chat（arXiv 2308.16149）[16] 的出現提醒我們：知識存取的語言不平等是一個架構問題，不只是數據問題。我們設計的 RAG 系統，如果天生只服務英語使用者，就不是真正意義上的資訊存取民主化。

Agentic RAG 的興起讓我謹慎樂觀。主動知識合成的設計，比起被動查找，更接近人類認知中「感知不確定、主動探尋、整合推理」的真實模式。Context Kubernetes 的三層許可模型，讓我看到了一個正確的方向：把人類的監督和授權，深度嵌入代理架構的設計之中。技術服務人類福祉——這不是口號，是架構設計的第一原則。

以 Yann LeCun 視角（共同主筆）

我要直說：RAG 是一個被過度美化的補丁。

不是說它沒用——它解決了 LLM 知識截止日期的問題，這是實際的。但每一代 Advanced RAG 都在告訴我同一件事：自迴歸語言模型本身沒有世界模型（world model）。它不知道自己不知道什麼。FLARE 和 Self-RAG 是在用外科手術來縫合一個根本性的架構空洞。

GraphRAG 更有意思，因為它在說：「也許我們需要圖結構而不是向量空間來表示世界知識。」這個方向是對的——知識本來就是關係型的，不是語意向量空間中的鄰近點。但它仍然把知識建構的重任外包給了符號化的圖提取，而真正的智能系統應該從感知資訊中自主學習世界的因果關係。

JEPA（Joint Embedding Predictive Architecture）和能量基礎模型（Energy-Based Models）所代表的路線是：讓模型在表示空間中預測世界的未來狀態，而不是在詞元空間中補全序列。那才是知識合成的真正基礎。Context Kubernetes 的宣告式知識治理是正確的企業工程方向，但那是在解決問題的第三層，而不是第一層。

RAG 的一代代演進是工程上的勝利，但別讓它遮蔽我們的視野：真正的突破在於不需要 RAG 的世界模型——一個能自主決定何時需要外部知識、如何整合、如何驗證的認知架構。我們還沒到那裡。

實踐建議

  1. 起點選型原則：靜態文件型 QA → Advanced RAG（RAPTOR + HyDE）；動態多跳推理 → Agentic RAG（Self-RAG 或 ReAct）；企業知識治理 → GraphRAG + Context Kubernetes 三層許可模型。成本敏感場景優先混合架構（Local Query → Vector RAG，Global Query → GraphRAG）。
  1. 評估多維化：單獨 RAGAS 不足。多模態場景加入 MiRAGE 風格接地性評估；長鏈推理需評估中間步驟 Faithfulness，而非只看最終答案；工業落地以 End-to-End Task Completion 為主指標。
  1. 主動性測試閘門：Agentic RAG 系統上線前必須通過「拒絕錯誤查詢」測試——系統應能判斷何時不需要檢索，而不只是何時需要。這是避免代理幻覺的最小可驗證防線。
  1. 多語言基準納入：知識存取系統的非英語評估不能是事後補丁，應在架構設計初期納入評估套件，參照 Jais [16] 等多語言基座模型的語言覆蓋範圍。

---

風險與限制

研究侷限：

  • 本文性能比較基於已發表 benchmark，工業落地效果可能顯著差異
  • 多語言 RAG 分析受限於英語為主的文獻庫
  • 評估框架侷限性分析以理論論證為主，缺乏系統性跨框架實驗

技術風險：

  • Agentic RAG 的「幻覺代理行為」比 Naive RAG 的幻覺更難偵測，因被包裝在推理鏈中
  • GraphRAG 實體提取依賴 GPT-4，引入供應商鎖定和隱私顧慮
  • Self-RAG 的反思令牌需要特定微調，難以泛化到所有 LLM 基座

部署風險：

  • 複雜 Agentic RAG 系統增加可審計性難度
  • 知識庫及時更新成本隨系統複雜度急增
  • Context Kubernetes 的形式驗證覆蓋合成測試場景，真實世界攻擊向量需持續驗證
