企業級 LLM Agent break-glass 緊急覆寫：深度技術研究報告

執行摘要（200-400 字）

企業導入 LLM Agent 時，「正常核准」有時趕不上事故觀測、法遵回覆或產線救援，因而需要受控的 break-glass（緊急覆寫）。本報告彙整 NIST AI RMF、OWASP LLM／GenAI 專案、ISO/IEC 42001 與歐盟 AI 法規入口等來源，指出三項共識：人類監督須內嵌於循環治理、技術防線須同時涵蓋邊界／權限／紀錄、例外操作必須文件化並可稽核。三角佐證顯示，緊急覆寫若繞過與日常相同的觀測平面，便與「過度代理」「不透明工具鏈」等風險同構。工程上宜以理由碼、核准者標識與 run／phase 識別碼，將事後補簽接到 JSONL 或排程狀態等日誌平面；台灣在地落地則宜準雙語程序，使國際稽核與對外說明一致。以下章節依來源內嵌引用，並標註尚待現場驗證的工程假設。

研究背景與動機

生成式 AI 代理人能呼叫工具、讀寫資料與觸發外部副作用，使得「暫時放寬限制」不再是純 IT 帳號議題，而涉及模型行為、工具邊界與資料保護 [來源：OWASP Top 10 for Large Language Model Applications, 2024]。相對地，組織亦被要求建立可追溯的監督與紀錄脈絡 [來源：EU regulatory framework for artificial intelligence, 2024]、[來源：ISO/IEC 42001:2023]。本研究聚焦 break-glass：如何在緊急情境下授權偏离標準 Harness，同時保留可審計的證據鏈，並與本專案多 Phase PowerShell 管線（task_key、phase、run_id）對齊。

核心技術解析（每個主張內嵌引用）

治理面，NIST AI RMF 以 Govern–Map–Measure–Manage 架構風險，並強調利害關係人、透明度與人類監督，可作為 break-glass 政策上位的框架 [來源：NIST AI Risk Management Framework（AI RMF 1.0）, 2023]。威脅面，OWASP 將過度代理、敏感資料外洩、不安全外掛等列為高優先議題，意味任何「緊急能力提升」都必須重新評估同一威脅模型，而非僅切換旗標 [來源：OWASP Top 10 for Large Language Model Applications, 2024]。生成式專案站則把可觀測性、測試案例與日誌需求放在同一敘事，利於將 break-glass 對應到 SOC／DevSecOps 事件嚴重度 [來源：OWASP Gen AI Security Project, 2024]。

管理系統面，ISO/IEC 42001 鼓勵將 AI 使用情境納入文件化控制與稽核軌跡；break-glass 作為例外路徑，應走入風險接受或變更流程並保留紀錄，而非個人默契 [來源：ISO/IEC 42001:2023, 2023]。法規敘事面，歐盟框架說明強調高風險系統的人類監督、技術文件與上市後監控；緊急覆寫若改變行為邊界，必須能回溯核准與理由，以支撐第三方檢視 [來源：EU regulatory framework for artificial intelligence, 2024]。中文政策脈絡方面，ai.gov.tw 提供信任與責任論述背景，便於對外溝通「為何仍有人因監督」，但不宜取代細部控制設計 [來源：行政院科技發展會議／台灣 AI 行動方案入口（ai.gov.tw）, 2024]。

關鍵洞見（含三角佐證狀態 ✅/⚠️）

  1. 人類監督＋文件化例外 是跨多邊框架的共識，break-glass 應以核准矩陣與時間盒體現，而非無記錄覆寫。已驗證✅（NIST、EU 入口、ISO 42001 等三類以上來源）。
  2. 觀測平面不可因緊急而關閉：緊急跳過某道閘門時，仍應產生高解析度事件日誌。已驗證✅（OWASP LLM Top 10、GenAI、NIST）。
  3. 事後補簽須與 run／session 可連結，以利防濫用調查；理想上與 append-only JSONL 或排程狀態檔並陳。已驗證✅（ISO、EU、OWASP 對紀錄／監控之要求）。
  4. Harness 層級欄位命名（task_key、phase、run_id、reason_code） 屬工程化最佳實務，標準文件支持「可追溯」但未規定欄位。待驗證⚠️。
  5. 跨法域敘事 需同時滿足國際客戶稽核與在地利害關係人溝通。待驗證⚠️（政策入口與法規細節不等價）。

實務應用建議（3-5 點）

  1. 定義 break-glass 理由碼表與最大有效期；超時自動收回權限並觸發審計工單。
  2. 將核准與補簽事件寫入結構化日誌（含 approver、理由、關聯 run_id），與現有 JSONL Hooks 或排程狀態並列查詢。
  3. 對照 OWASP 條目重新評估「緊急模式」下工具白名單，避免過度代理路徑被常態化。
  4. 以 ISO AIMS 思維將例外納入變更與風險登錄，保留可供外部稽核摘引的文件版本。
  5. 對外文件採中英對照摘要，政策引用僅作背景，技術控制仍引 NIST／OWASP／ISO。

與現有知識的連結

本專案連續記憶已涵蓋 operator override 雙人覆核、理由碼與 trace 對齊等主題；本報告將 break-glass 置於同一治理光谱的「時間壓力極端」，強調事後補簽與 Harness 索引的不可拆性。與先前「委派式核准」「動態信任衰減」研究並讀時，可共用同一套结构化核准事件模型，僅調整時效與風險權重。
