n8n 深度研究報告

概述

n8n（n8n-io/n8n）是面向技術團隊的 fair-code 工作流自動化平台，結合視覺化節點編排與 JavaScript/Python 程式碼節點，支援 400+ 整合與原生 AI／LangChain 能力。與本知識庫已深入的 Activepieces（MIT、線性 Pieces、MCP 一體化）不同，n8n 的強項在於 成熟節點圖引擎、企業級錯誤處理、70+ AI 節點與 LangChain ReAct Agent，適合作為 daily-digest-prompt 管線外層的「可觀測、可重試」編排層候選。

技術架構

  • 語言與前端：TypeScript（約 91%）+ Vue 7%，單體 monorepo，預設分支 master。
  • 執行模型：每個 workflow 由 Trigger（Webhook、Cron、手動等）啟動，資料以 JSON item 在節點間流動；支援 IF / Switch / Merge 分支與子 workflow。
  • 程式碼擴充：Code 節點可寫 JS/Python、引入 npm 套件；自訂節點透過 n8n-nodes-starter 模板開發。
  • AI 子系統：內建 LangChain 叢集節點（Agent、Memory、Vector Store、Output Parser 等），Tools Agent 預設採 ReAct（推理→選工具→觀察→迴圈）。
  • MCP 雙向：MCP Client Tool 消費外部 MCP 工具；MCP Server Trigger／實例級 MCP Server 將 workflow 暴露給 Claude Desktop、Cursor 等客戶端。
  • 授權：Sustainable Use License（fair-code，非 OSI 開源）；企業版另授權 SSO、進階權限、air-gapped。

工作流核心能力

| 能力 | 說明 |

|------|------|

| 觸發器 | Webhook、Schedule（Cron）、Email、App 事件、子 workflow 呼叫 |

| 整合 | 400+ 內建節點（SaaS、DB、HTTP、通訊） |

| 除錯 | 單步重跑、mock 資料、執行日誌、輸入輸出並列檢視 |

| 錯誤處理 | Error Workflow、重試策略、fallback 路由（優於多數同類工具） |

| 範本 | 900+ 社群 workflow 範本可匯入 |

| 佇列模式 | Redis queue mode 將執行與主實例分離，利於高併發 |

AI 整合機制

  1. LangChain 視覺化：LLM、Embedding、Vector DB、Memory（Postgres/Redis 等）、Chains、Agents 皆以子節點掛在 Agent 下。
  2. Tools Agent：連接 Calculator、HTTP、Google Sheets、Slack 等既有節點作為工具，無需另寫 Python LangChain。
  3. Context Pipeline（企業實務）：Webhook → Code 組裝 Context Object → 注入 System Prompt → Agent → Output Parser 驗證 JSON → Memory 持久化。
  4. 與 LangGraph 分工：複雜循環推理、長期多 Agent 狀態建議在 LangGraph 實作，n8n 透過 HTTP/MCP 呼叫其邊界；簡單單 Agent 流程用 n8n 內建節點即可。
  5. AI 評估：內建評估節點可對 workflow 做回歸測試（利於 prompt/工具變更驗證）。

部署與維運

  • 快速啟動：npx n8n 或 Docker docker.n8n.io/n8nio/n8n（預設 5678）。
  • 生產：PostgreSQL 持久化 + 可選 Redis（queue）；n8n-hosting repo 提供 K8s/Docker Compose 範例。
  • 雲端：n8n Cloud 按 workflow 執行次數計費；自架無執行上限。
  • 安全：加密憑證庫、RBAC、SSO（企業）、可 air-gapped 自架。
  • 可觀測：執行歷史、節點級日誌；可外接 Langfuse 等（本專案另有 Langfuse 研究缺口）。

優缺點評估

優點：

  • GitHub 生態最大（約 19 萬 stars）、整合與範本最豐富
  • 錯誤處理與程式碼節點適合生產級自動化
  • LangChain + MCP 雙向，與 Cursor/Claude Code 策略一致
  • 自架資料主權、queue 模式可擴展

缺點 / 限制：

  • Sustainable Use License 限制部分商業再分發場景（不如 Activepieces MIT）
  • 學習曲線高於線性 builder；複雜 Agent 圖可讀性不如程式碼
  • 高併發需額外 Redis/PostgreSQL 維運
  • 與本專案「機械優先 PS > Python」哲學重疊時，應避免用 n8n 取代已有 CLI 工具鏈
