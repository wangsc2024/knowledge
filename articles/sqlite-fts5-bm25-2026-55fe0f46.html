SQLite FTS5 全文搜尋引擎完整指南

技術概述

SQLite FTS5（Full-Text Search 5）是 SQLite 內建的全文搜尋擴充模組，透過倒排索引（Inverted Index）實現高效的文字檢索。相較於 LIKE '%term%' 的全表掃描，FTS5 可在 50K+ 筆資料中實現毫秒級搜尋。搭配 sqlite-vec 向量擴充，可建構完整的混合檢索（Hybrid Search）RAG 架構，所有功能封裝在單一 SQLite 檔案中，無需外部服務。

---

一、FTS5 核心概念與建置

1.1 建立 FTS5 虛擬表


-- 基本建立
CREATE VIRTUAL TABLE docs USING fts5(
  title,
  content,
  tokenize = 'porter unicode61'
);

-- 外部內容表模式（推薦：節省儲存空間）
CREATE VIRTUAL TABLE fts_docs USING fts5(
  title,
  content,
  content='documents',
  content_rowid='id'
);

外部內容表是 FTS5 的重要最佳實踐：FTS5 只儲存倒排索引，實際文字從基底表讀取，避免資料重複儲存。

1.2 同步機制（Trigger-Based Synchronization）

外部內容表必須透過觸發器保持同步，否則會出現靜默腐壞（silent corruption）：


-- 插入觸發器
CREATE TRIGGER docs_ai AFTER INSERT ON documents BEGIN
  INSERT INTO fts_docs(rowid, title, content)
  VALUES (new.id, new.title, new.content);
END;

-- 刪除觸發器
CREATE TRIGGER docs_ad AFTER DELETE ON documents BEGIN
  INSERT INTO fts_docs(fts_docs, rowid, title, content)
  VALUES ('delete', old.id, old.title, old.content);
END;

-- 更新觸發器
CREATE TRIGGER docs_au AFTER UPDATE ON documents BEGIN
  INSERT INTO fts_docs(fts_docs, rowid, title, content)
  VALUES ('delete', old.id, old.title, old.content);
  INSERT INTO fts_docs(rowid, title, content)
  VALUES (new.id, new.title, new.content);
END;

關鍵提醒：三個觸發器（INSERT、UPDATE、DELETE）必須在上線前全部建立，缺少任何一個都會導致索引漸進失準。

二、BM25 排序機制

2.1 內建 BM25 函數

FTS5 內建 BM25 排序演算法，參數 k1=1.2、b=0.75 為硬編碼值：


SELECT *, bm25(fts_docs) as score
FROM fts_docs
WHERE fts_docs MATCH 'search query'
ORDER BY score;  -- 注意：分數為負數，越小越相關

重要行為：BM25 分數為負數，越小（越負）代表越相關。使用 ORDER BY score（升序）即可取得最佳結果。

2.2 欄位權重調整

透過 bm25() 函數的參數自訂各欄位權重：


-- 標題權重 10.0，內容權重 1.0
SELECT *, bm25(fts_docs, 10.0, 1.0) as score
FROM fts_docs
WHERE fts_docs MATCH 'query'
ORDER BY score;

調校建議：準備 20-50 組「黃金查詢」（golden queries），檢查 Top 3 結果是否符合人類預期，逐步調整權重。

2.3 查詢語法


-- 布林運算
WHERE fts_docs MATCH 'term1 AND term2'
WHERE fts_docs MATCH 'term1 OR term2'
WHERE fts_docs MATCH 'term1 NOT term2'

-- 片語搜尋
WHERE fts_docs MATCH '"exact phrase"'

-- 鄰近搜尋
WHERE fts_docs MATCH 'NEAR(term1 term2, 10)'

-- 前綴搜尋
WHERE fts_docs MATCH 'pre*'

-- 欄位限定
WHERE fts_docs MATCH 'title:query'

2.4 結果高亮


SELECT highlight(fts_docs, 0, '<b>', '</b>') as highlighted_title,
       snippet(fts_docs, 1, '<b>', '</b>', '...', 64) as content_snippet
FROM fts_docs
WHERE fts_docs MATCH 'query';

三、中文（CJK）搜尋解決方案

3.1 問題根因

FTS5 內建的 unicode61 分詞器依賴空白字元分詞，而中文沒有詞間空白，導致整段中文被視為單一 token，無法搜尋。

3.2 解決方案對比

| 方案 | 精確度 | 效能 | 部署難度 | 適用場景 |

|------|--------|------|---------|----------|

| Trigram 分詞器 | 中 | 中 | 低（內建） | 通用子字串匹配 |

| Simple 分詞器（cppjieba） | 高 | 高 | 中（需編譯） | 中文專業搜尋 |

| 預處理分詞 | 高 | 高 | 中 | Node.js/Python 應用 |

| ICU 分詞器 | 高 | 高 | 高（需 ICU 庫） | 多語言專業搜尋 |

3.3 Trigram 分詞器（最簡方案）


CREATE VIRTUAL TABLE fts_docs USING fts5(
  title,
  content,
  tokenize = 'trigram'
);

將文字切成重疊的三字元片段，支援子字串匹配。CJK 單字元即可構成完整詞，trigram 會產生冗餘索引但功能可用。

3.4 預處理分詞（Node.js 推薦方案）

在應用層用 jieba 或 nodejieba 先斷詞，再存入 FTS5：


// Node.js 預處理範例
const nodejieba = require('nodejieba');

function tokenizeForFTS(text) {
  const words = nodejieba.cut(text);
  return words.join(' ');  // 加空白分隔
}

// 寫入時
const segmented = tokenizeForFTS('台灣人工智慧發展趨勢');
// => '台灣 人工智慧 發展 趨勢'
db.run('INSERT INTO fts_docs(title) VALUES (?)', [segmented]);

// 查詢時也需要斷詞
const query = tokenizeForFTS('人工智慧');
db.all('SELECT * FROM fts_docs WHERE fts_docs MATCH ?', [query]);

3.5 Simple 分詞器（C++ 擴充）

wangfenjin/simple 專案提供基於 cppjieba 的 FTS5 分詞器擴充，支援中文和拼音搜尋，有 Python、Go、Node.js 綁定。

四、向量混合檢索架構（Hybrid Search）

4.1 架構概覽


文件 → FTS5 倒排索引（關鍵字匹配）
    → sqlite-vec 向量索引（語義相似度）
         ↓
    Reciprocal Rank Fusion（RRF）合併
         ↓
    最終排序結果

4.2 向量表建立（sqlite-vec）


CREATE VIRTUAL TABLE vec_docs USING vec0(
  doc_id INTEGER PRIMARY KEY,
  embedding float[768]
);

-- 插入嵌入向量
INSERT INTO vec_docs(doc_id, embedding)
  SELECT id, generate_embedding(content) FROM documents;

4.3 Reciprocal Rank Fusion（RRF）演算法

RRF 是合併多個排序列表的標準演算法，公式：


RRF_score = Σ 1 / (k + rank_i)

其中 k 為常數（通常 60），防止排名第一的結果過度主導。

4.4 完整 RRF 混合查詢


WITH vec_matches AS (
  SELECT doc_id,
    row_number() OVER (ORDER BY distance) as rank_number
  FROM vec_docs
  WHERE embedding MATCH :query_embedding AND k = :top_k
),
fts_matches AS (
  SELECT rowid,
    row_number() OVER (ORDER BY rank) as rank_number
  FROM fts_docs
  WHERE fts_docs MATCH :query_text LIMIT :top_k
)
SELECT documents.*,
  COALESCE(1.0 / (60 + fts_matches.rank_number), 0.0) * :weight_fts +
  COALESCE(1.0 / (60 + vec_matches.rank_number), 0.0) * :weight_vec
  AS combined_score
FROM fts_matches
FULL OUTER JOIN vec_matches
  ON vec_matches.doc_id = fts_matches.rowid
JOIN documents ON documents.id =
  COALESCE(fts_matches.rowid, vec_matches.doc_id)
ORDER BY combined_score DESC;

4.5 三種混合策略

| 策略 | SQL 模式 | 適用場景 |

|------|---------|----------|

| RRF 融合 | FULL OUTER JOIN + RRF 公式 | 通用 RAG 檢索 |

| 關鍵字優先 | FTS5 篩選 + 向量重排序 | 精確度優先（如郵件搜尋） |

| 語義優先 | 向量篩選 + BM25 重排序 | 語義理解優先（如問答系統） |

4.6 權重調整建議

  • weight_fts=1.0, weight_vec=1.0：平衡（預設）
  • weight_fts=1.5, weight_vec=1.0：偏向精確匹配
  • weight_fts=1.0, weight_vec=1.5：偏向語義理解
  • 根據實際查詢結果反覆調整

5.1 索引最佳化


-- 建立索引後執行優化
INSERT INTO fts_docs(fts_docs) VALUES('optimize');

-- 增量合併（大量插入後）
INSERT INTO fts_docs(fts_docs, rank) VALUES('merge', 500);

CREATE VIRTUAL TABLE fts_docs USING fts5(
  title, content,
  prefix = '2 3'  -- 建立 2、3 字元前綴索引
);

僅在需要自動完成功能時啟用，會增加索引大小。

5.3 效能數據

  • FTS5 在 50K+ 筆資料中可實現毫秒級搜尋
  • sqlite-vec 向量搜尋在數萬筆中約 100-500ms
  • 完整 RRF 混合查詢約 370ms（包含嵌入生成）
  • 單一 SQLite 檔案，約 100MB 記憶體即可運行

六、常見陷阱

  1. 靜默腐壞：外部內容表缺少同步觸發器，不會報錯但搜尋結果漸進失準
  2. BM25 分數方向：分數為負數，ORDER BY score（升序）才是正確排序
  3. 中文分詞：unicode61 不支援 CJK，必須使用 trigram 或預處理分詞
  4. Stop Words：過度索引停用詞會膨脹索引且降低搜尋品質
  5. 嵌入快取：混合查詢中每次呼叫 embedding 函數有成本，應快取常用查詢
  6. FULL OUTER JOIN：SQLite 3.39.0+ 才支援，舊版需用 UNION ALL 模擬
