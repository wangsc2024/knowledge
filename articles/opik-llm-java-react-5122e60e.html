一、專案概述

Opik 是 Comet 推出的開源 LLM 工程平台，核心定位不是「再做一個 Agent 框架」，而是把 追蹤（tracing）、評估（evaluation）、線上監控（monitoring）、優化（optimization） 串成同一條工程閉環。它要解決的不是模型能不能呼叫，而是 AI 系統上線後如何持續回答三個問題：

  1. 這次輸出是怎麼產生的？
  2. 它好不好、是否退化？
  3. 我能不能用資料與規則自動把它變更好？

因此 Opik 的價值主張很清楚：讓 RAG、聊天機器人、程式碼助手與 agentic workflow 不再只靠人工「憑感覺」驗收，而是能以 trace、dataset、metric、experiment 與 optimizer 形成可重現的改進迴路。

二、技術架構分析

2.1 架構分層

根據官方 Platform Architecture 與 GitHub repo 結構，Opik 採 service-oriented architecture：

  • 後端服務：Java 21 + Dropwizard 4，提供 REST API 與核心資料寫入/查詢能力。
  • 前端應用：TypeScript + React，由 Nginx 提供靜態資源與反向代理。
  • 分析資料層：ClickHouse 負責高吞吐 trace、feedback、dataset、experiment 等分析型資料。
  • 交易資料層：MySQL 負責專案、設定、feedback definition 等 ACID 型資料。
  • 協調與快取：Redis 處理 cache、rate limiting、distributed lock 與 online evaluation stream。
  • 觀測標準：後端以 OpenTelemetry 做 vendor-neutral instrumentation。
  • SDK / 接入層：提供 Python、TypeScript、Ruby（透過 OTel）等入口，並有大量框架整合。

這個分工是合理的。AI tracing 與 evaluation 的事件流本質上偏向 append-heavy、查詢聚合密集，不適合把所有資料都塞在傳統 transactional DB；Opik 把分析查詢與交易資料拆開，是典型的「OLTP/OLAP 職責分離」。

2.2 我看到的關鍵設計決策

第一個關鍵決策是 ClickHouse + MySQL 雙資料庫。這代表團隊並沒有追求單庫簡潔，而是接受系統複雜度，換取 trace/experiment 查詢效能與資料一致性邊界的清楚切分。對 observability 產品來說，這比單純 ORM 驅動的 CRUD 系統成熟很多。

第二個關鍵決策是 以 OpenTelemetry 為觀測骨架。這使 Opik 不只是自家 SDK 封閉平台，而能和現有 telemetry stack 接軌，也讓未來接 Grafana/Prometheus 或企業內部 observability 平台更自然。

第三個關鍵決策是 monorepo + 多模組實作。GitHub repo 可直接看到 apps、deployment、sdks、tests_end_to_end、tests_load 等目錄，說明它不是單純 UI 包裝，而是把產品、部署與測試視為同一個工程系統。這對自託管產品尤其重要，因為 deployability 本身就是產品能力的一部分。

2.3 與同類工具相比的差異

根據官方文件與比較文章，Opik 的差異化不在「只有 tracing」，而是在三件事：

  • evaluation 與 experiment 內建：可以把 dataset、metric、LLM-as-a-judge、heuristic evaluator 放進同一工作流。
  • online evaluation rules：不是只做離線實驗，而是能在 production trace 上做規則與判分。
  • Agent Optimizer：把 prompt/agent optimization 直接建立在既有 trace + evaluation 基礎上，形成「先觀測，再評估，最後自動優化」的閉環。

若把 Langfuse 視為偏 observability/prompt management，LangSmith 視為偏閉環開發體驗，Opik 更像是把 ML experiment tracking 思維 帶進 LLM 工程，這是它最不一樣的產品哲學。

三、功能特色

3.1 主要能力

  • 完整 tracing：LLM calls、conversation、tool usage、agent activity、context assembly 都能追蹤。
  • evaluation：支援 LLM-as-a-judge、hallucination、moderation、RAG 指標如 answer relevance / context precision。
  • experiment / dataset：能把應用評估從零散 sample 檢查提升成可重跑實驗。
  • CI 整合：官方直接提供 PyTest integration，可把評估接入測試流程。
  • production monitoring：支援 dashboard、feedback score、token usage 與 online evaluation rules。
  • optimization：Agent Optimizer 支援 prompt 與 agent 優化，並透過 LiteLLM 面向多模型供應商。
  • self-host：可用 Docker Compose 本地跑，也可用 Kubernetes + Helm 做正式部署。

3.2 為何這些能力組合有工程價值

很多團隊的 AI 工程鏈其實是分裂的：trace 在 A 工具、資料集在 B 工具、評估腳本在 repo、優化靠人工。Opik 把這些放在同一平台後，最實際的好處不是「功能變多」，而是 資料模型一致。同一筆 trace 可回連到 prompt、dataset、experiment 與線上規則，這對定位 regression 非常有用。

四、快速上手方式

4.1 Python SDK 快速開始


from opik.integrations.openai import track_openai
from openai import OpenAI

client = OpenAI()
openai_client = track_openai(client)

completion = openai_client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "請用兩句話介紹 Opik。"}]
)

print(completion.choices[0].message.content)

對應的基本安裝步驟：


pip install opik
opik configure

若要本機自託管，可依官方 README：


git clone https://github.com/comet-ml/opik.git
cd opik
powershell -ExecutionPolicy ByPass -c ".\opik.ps1"

Windows 環境也可用 profile 啟動，例如只啟動基礎設施或 backend。

4.2 實際導入順序建議

  1. 先只接 tracing，不要一次把 evaluation 和 optimizer 全開。
  2. 收集一週真實 trace 後，挑 20-50 筆高價值案例建立 dataset。
  3. 再補 hallucination / answer relevance / custom rubric。
  4. 最後才接 online evaluation rules 與 optimizer，避免指標基礎不穩就自動調參。

六、潛在應用

6.1 對 `daily-digest-prompt`

Opik 很適合補上目前工作流中的「執行後可觀測性」。

  • 把每次 Web research、KB search、匯入、結果 JSON 產生視為一條 trace，定位哪個 phase 最容易失敗。
  • 對研究報告建立 rubric，例如「是否含架構分析」「是否含應用建議」「是否超過 500 字」，透過 evaluation 自動評分。
  • 對不同 prompt 模板做 experiment，比較哪個模板在研究深度與 token 成本上更好。
  • 若未來要做多 agent 研究管線，可用 online evaluation rules 監測 hallucination 或來源不足的報告。

6.2 對 game / 互動應用

若你的 game 有 NPC 對話、劇情生成或玩家行為摘要，Opik 可做：

  • 追蹤每次 NPC 回應的 prompt、context、latency、cost。
  • 以玩家回饋或人工標註建立 dataset，評估角色一致性。
  • 針對安全與世界觀違規建立 online rules。
  • 對提示詞版本做 optimizer/experiment，比較敘事品質與 token 成本。

七、優缺點評估

優點

  • 架構完整，不只看 trace，還把 evaluation、monitoring、optimization 串成閉環。
  • 自託管友好，Docker 與 Kubernetes 路線清楚，適合資料主權要求高的團隊。
  • 技術選型成熟，Java/React/ClickHouse/MySQL/Redis/OpenTelemetry 都是可長期維運的組合。
  • 與主流框架整合面廣，導入門檻相對低。
  • 產品哲學偏工程化，適合把 LLM 應用當成正式軟體系統營運。

缺點

  • 系統本身不輕，雙資料庫加 Redis 的部署與維運成本高於純 SDK 型工具。
  • 若團隊目前只需要最基本 trace，看起來會比輕量 observability SDK 更重。
  • 相較較早期競品，第三方教學、生態案例與社群心智仍在成長中。
  • optimization 功能很有吸引力，但若 evaluation 基礎沒打穩，容易過早自動化。

八、結論

Opik 值得研究的原因，不是它又多了一個 AI 平台，而是它把 「LLM 系統如何像正式軟體一樣被觀測、測試、優化」 這件事做成了有完整資料層與部署模型的產品。若你要的是 Demo 階段的 prompt playground，它可能偏重；但若你要的是把 AI workflow 納入工程治理、效能監控與品質回歸，Opik 的架構成熟度與方向都很值得借鏡。
