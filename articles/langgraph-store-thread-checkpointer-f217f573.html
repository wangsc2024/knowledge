LangGraph Store 與跨 thread 長期記憶：checkpointer 分際與命名空間設計（2026）

執行摘要（約 280 字）

LangGraph 將「可對話續跑」與「跨工作階段仍可召回」拆成兩條持久化軸：短期／thread 範圍記憶由 checkpointer（BaseCheckpointSaver）將圖狀態以 checkpoint 序列寫入特定 thread_id；長期／跨 thread 記憶則由 BaseStore 實作（如 InMemoryStore、PostgresStore）以「命名空間 tuple + 鍵」儲存 JSON 文件，並可選擇性啟用向量語意檢索 [來源：Memory overview, LangChain Docs, 2026]。官方文件明確指出：僅有 checkpointer 無法在 thread 之間共享資訊，因此需要 Store 介面來承載跨對話的偏好、規則或摘要 [來源：Persistence — Memory store, LangChain Docs, 2026]。生產環境建議以資料庫後端（如 PostgresSaver + PostgresStore）取代記憶體實作，並在首次使用前執行 setup() 遷移 [來源：Add memory — production examples, LangChain Docs, 2026]。本筆記最後對照 daily-digest-prompt：以 task_key／排程回合類比 thread_id，以 RAG 與 context/* 連續記憶類比 Store 命名空間，可避免把「應跨排程保留」的知識誤放進僅屬單次執行軌跡的狀態。

研究背景

今日自動任務履歷中反覆出現 Cursor CLI、備援／quota、Phase 2 等關鍵字，與「多後端執行軌跡可恢復、但使用者級偏好應跨執行保留」的問題同構。LangGraph 的記憶模型將此二分為：對話／回合內（thread）vs 使用者或系統層（namespace） [來源：Memory overview, LangChain Docs, 2026]。

核心概念與原理

1. 短期記憶 = thread 範圍 + checkpoint

圖在編譯時傳入 checkpointer=...，執行時於 config["configurable"]["thread_id"] 下保存狀態快照；thread_id 是儲存與還原 checkpoint 的主鍵 [來源：Persistence — Threads, LangChain Docs, 2026]。InMemorySaver 適合測試；生產環境文件建議改用 Postgres、Redis、MongoDB 等後端 [來源：Add memory — short-term production, LangChain Docs, 2026]。

2. 長期記憶 = Store + 命名空間

長期記憶以 JSON 文件存入 Store：namespace（tuple，常用 user_id／org／應用情境）+ key 識別單筆記憶 [來源：Memory overview — Memory storage, LangChain Docs, 2026]。InMemoryStore 僅供開發；正式環境應改用 PostgresStore／RedisStore 等 [來源：Persistence — Memory store, LangChain Docs, 2026]。

3. checkpointer 與 Store 的協作

官方 Persistence 章節說明：兩者同時編譯時，checkpointer 負責 thread 內狀態，Store 讓節點透過 Runtime.store（或注入模式）讀寫跨 thread 資料 [來源：Persistence — Using in LangGraph, LangChain Docs, 2026]。社群 walkthrough 亦強調混淆兩者會導致「新 thread 後偏好遺失」的常見設計錯誤 [來源：Five Agent Memory Types in LangGraph Part 2, DEV Community, 2026]。

4. 語意搜尋與篩選

InMemoryStore 可設定 index（嵌入與維度）啟用 search(..., query=...)；文件亦描述以 filter 搭配向量相似度排序 [來源：Memory overview — Memory storage, LangChain Docs, 2026]。

最佳實踐與常見陷阱

| 做法 | 說明 |

|------|------|

| 釐清資料生命週期 | 只屬單一對話的放 checkpoint；需跨會話的放 Store [來源：Persistence, LangChain Docs, 2026] |

| 生產持久化 | 使用 PostgresSaver／PostgresStore 等並執行 schema 遷移 [來源：Add memory, LangChain Docs, 2026] |

| 命名空間設計 | 以 (user_id, domain) 等層級避免金鑰衝突 [來源：Memory overview, LangChain Docs, 2026] |

陷阱：把使用者偏好只寫在 graph state（checkpoint）而無 Store——新 thread_id 即遺失 [來源：DEV Community walkthrough, 2026]。

與 daily-digest-prompt 的關聯

  • thread_id：可比對單次 run-todoist-agent-team.ps1 Phase 2 執行或單一 task_key 回合 ID，用於可恢復與續跑。
  • Store 命名空間：可比對 context/continuity/auto-task-*.json、context/research-registry.json 等「跨回合仍要保留」的知識；本專案 RAG 匯入則類比為外部知識庫長期儲存。
  • 配額／備援：多後端切換時，持久化層與命名空間應與後端無關，避免將「執行軌跡」與「使用者／專案級記憶」綁在同一 ephemeral 狀態。

程式碼範例（概念對照）


# 概念：短期 thread + 長期 store（摘自官方文件結構）
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.store.memory import InMemoryStore

checkpointer = InMemorySaver()
store = InMemoryStore()

graph = builder.compile(checkpointer=checkpointer, store=store)
# invoke 時帶 thread_id；跨 thread 讀寫用 runtime.store 與 namespace
