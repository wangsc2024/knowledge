Tool, skill, or subagent? 重構過度膨脹的 Agent（Code with Claude London）

youtuberesearchautoanthropicagentmanaged-agentsskillseval

Agent 分解工作坊 — Anthropic Applied AI 工程師 Will 在 Code with Claude London 分享如何重構過度膨脹的 Agent。

場景：StockPilot 庫存管理 Agent 隨業務需求累積，系統提示增長至 400 行、12 個工具（含 3 個子 Agent 封裝），導致 Eval 評分從 83% 滑落至 62%。根因：上下文污染、政策衝突、子 Agent 與 Orchestrator 通訊斷裂。

解法三步：

  1. Skills（漸進揭露）：將偶爾才需要的策略文件從系統提示移出，改為 Skills 封裝，讓 Claude 按需拉入。系統提示由 400 行壓縮至 15 行。
  2. 工具精簡：以 bash/read/write 等 Claude Code 原語取代自訂工具（12 → 3），讓 Claude 用程式碼推理 CSV/Excel，Token 用量顯著降低。
  3. Claude Managed Agents（CMA）：部署至 CMA，由平台負責基礎設施、擴展與安全，工程師專注 Agent 架構本身。CMA 提供 Callable Agents 原生可觀測性，解決子 Agent 通訊與日誌追蹤痛點。

子 Agent 使用時機：①需要大量 Claude 並行處理（深度研究）；②需要全新視角（如程式碼審查，寫碼與審碼分兩個 Claude 實例）。

結果：重構後 Eval 評分升至 92%，Token 用量、成本、執行時間同步下降。

適用情境：Agent prompt 累積至數百行並出現效能退化；工具數量過多；子 Agent 通訊不穩定。
