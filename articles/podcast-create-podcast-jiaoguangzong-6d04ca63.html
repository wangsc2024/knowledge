podcast_create 與 podcast_jiaoguangzong 守衛問題統一分析

問題摘要

兩個 podcast 自動任務同時出現 partial_success：

  • podcast_create：script_20260406_020335.jsonl，TTS+R2 成功，CLI exit 0
  • podcast_jiaoguangzong：ep47 化儀四教 藥方藥味，44 輪對話，CLI exit 0

共同症狀：result_guard_zh: "無法從 stdout 修復；結果檔未通過守衛（守衛回傳成功但結果檔空白）"

根因分析

核心問題：Windows Python/PS1 I/O 不對稱

cursor_cli_result_guard.py 使用 Python open() 讀檔 → 驗證通過 → exit 0。

但 PS1 的 Get-Content -Raw 和 [System.IO.File]::ReadAllText() 在 guard exit 後仍讀到空值。

兩者使用不同 I/O 路徑：

  • Python: open() → Win32 CreateFileW + ReadFile
  • PS1: .NET StreamReader / FileStream → 可能受 OS file cache layer 影響

觸發條件

  1. Agent 寫入結果檔（Python write_json_atomic 的 os.replace() = MoveFileExW）
  2. Guard (Python) 能立即讀取（同一 I/O 堆疊）
  3. PS1 (.NET) 在 45 次重試（~7.75 秒）內持續讀到空值
  4. 可能的加劇因素：Windows Defender 掃描新建檔案、NTFS journal 刷新延遲

與先前分析差異

先前 KB 筆記（0def5449）僅覆蓋 jiaoguangzong 並標註「podcast_create 不受影響」。

本次確認 podcast_create 同樣受影響（同一 trace_id d5a53ced3117，同批次執行）。

修復方案（統一）

方案 A（推薦，2 行程式碼）：Guard 驗證成功時回寫檔案

修改 tools/cursor_cli_result_guard.py 第 337-342 行：


# 原始碼（第 337-342 行）:
else:
    print(f"[result-guard] {task_key}: valid ({data.get('status')})", file=sys.stderr)
    return finalize(True, [])

# 修復後：
else:
    print(f"[result-guard] {task_key}: valid ({data.get('status')})", file=sys.stderr)
    # Windows I/O race mitigation: 回寫檔案確保 PS1 可讀
    if not dry_run:
        write_json_atomic(result_path, data)
    return finalize(True, [])

原理：Guard 已載入完整 JSON data，用 write_json_atomic 回寫一次，

確保 PS1 讀到的是由同一 Python 進程剛寫入的新鮮檔案。

方案 B（補強）：Guard 輸出 JSON 到 stdout

讓 PS1 從 guard stdout 解析結果，完全繞過磁碟重讀。

需同時修改 Python guard 和 PS1 Start-CursorCliJob。

方案 C（短期緩解）：增加延遲

Post-guard read 前加 1 秒延遲。治標不治本。

影響範圍

  • 所有 cursor_cli 後端任務都可能受影響（非僅 podcast）
  • 方案 A 修復是全域性的，一次修復所有任務
  • 無需修改 PS1（PS1 的 post-guard read 會在回寫後正常讀取）

測試驗證

現有測試 tests/tools/test_cursor_cli_result_guard.py 的 TestGuardResult.test_valid_existing_file 覆蓋此路徑，

但不驗證回寫行為。建議新增測試確認 guard 驗證通過後檔案時間戳更新。
