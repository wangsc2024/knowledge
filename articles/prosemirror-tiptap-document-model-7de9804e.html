一、技術概述

ProseMirror 是一套模組化的富文本編輯器基礎框架，由 Marijn Haverbeke（CodeMirror 作者）開發。它的核心理念是：程式碼對文件擁有完全控制權——文件不是一坨 HTML，而是由 Schema 嚴格約束的自訂資料結構。所有更新都透過單一節點（Transaction）進行，使得狀態管理、協作編輯、撤銷歷史等進階功能成為可能。

Tiptap 是建立在 ProseMirror 之上的高層抽象框架，提供更友善的 API 與擴展系統。它本質上是一個「帶有合理預設的 ProseMirror 配置器」，將 ProseMirror 的模組化元件（schema、plugins、commands 等）封裝成統一的 Extension 介面。Tiptap 是目前最流行的 ProseMirror 封裝層，廣泛用於知識庫、CMS、協作文件等場景。

二、ProseMirror 核心概念

2.1 四大核心模組

ProseMirror 分為四個必要模組，每個模組負責一個獨立關注點：

| 模組 | 職責 |

|------|------|

| prosemirror-model | 定義文件模型（Document Model）——描述編輯器內容的資料結構 |

| prosemirror-state | 管理編輯器完整狀態（文件 + 選區），提供 Transaction 系統 |

| prosemirror-view | 渲染編輯器 UI，處理使用者互動 |

| prosemirror-transform | 文件修改操作的記錄與重播，支撐撤銷歷史與協作編輯 |

附加模組包括：prosemirror-commands（基本編輯命令）、prosemirror-keymap（鍵盤綁定）、prosemirror-history（撤銷歷史）、prosemirror-inputrules（輸入巨集）、prosemirror-collab（協作編輯）。

2.2 Document Model（文件模型）

ProseMirror 文件是一棵節點樹，與瀏覽器 DOM 類似但有關鍵差異：

與 DOM 的核心差異——扁平化行內內容：

在 HTML 中，帶有標記的段落是巢狀樹結構：


<p>This is <strong>strong text with <em>emphasis</em></strong></p>

在 ProseMirror 中，行內內容被建模為扁平序列，標記作為 metadata 附加在節點上：


paragraph
  "This is "                         (無 marks)
  "strong text with "                 (marks: [strong])
  "emphasis"                          (marks: [strong, em])

這種扁平化設計的優勢：

  • 可以用字元偏移量（而非樹路徑）表示段落內的位置
  • 分割或修改內容樣式不需要複雜的樹操作
  • 每個文件只有一種合法表示（相鄰同 marks 的文字節點會自動合併）

Node 資料結構：


Node
  type: NodeType      # 節點類型（名稱、有效屬性、渲染規則等）
  content: Fragment   # 子節點序列
  attrs: Object       # 額外屬性（如圖片的 src、alt）
  marks: [Mark]       # 行內標記陣列（如粗體、連結）

節點屬性判斷：

  • isBlock / isInline：區塊或行內
  • inlineContent：是否包含行內內容
  • isTextblock：包含行內內容的區塊節點（如 paragraph）
  • isLeaf：不允許任何子內容（如 horizontal_rule、image）

2.3 不可變性（Immutability）

ProseMirror 節點是不可變值（Persistent Data Structure），不是有狀態的物件：

  • 節點沒有父節點連結，可以同時出現在多個資料結構中
  • 更新文件會產生新的文件值，原文件保持不變
  • 新舊文件共享未變更的子節點，建立新文件的成本很低
  • 這使得狀態切換是瞬間完成的，也讓協作編輯和高效 DOM 差量更新成為可能

2.4 位置索引系統

ProseMirror 的位置系統用整數表示文件中的任何位置：


0   1 2 3 4    5
 <p> O n e </p>

5            6   7 8 9 10    11   12            13
 <blockquote> <p> T w o <img> </p> </blockquote>

規則：

  • 文件開頭是位置 0
  • 進入/離開非葉節點各計 1 個 token
  • 文字中每個字元計 1 個 token
  • 葉節點（如 image）計 1 個 token

2.5 Schema（模式）

每個 ProseMirror 文件都有一個 Schema，它定義了文件中可以出現哪些節點類型及其巢狀關係。

NodeSpec 定義範例：


const trivialSchema = new Schema({
  nodes: {
    doc: { content: "paragraph+" },     // 至少一個段落
    paragraph: { content: "text*" },     // 零或多個文字
    text: { inline: true },              // 行內節點
  }
})

Content Expression（內容表達式）：

  • "paragraph" — 恰好一個段落
  • "paragraph+" — 一個或多個段落
  • "paragraph*" — 零個或多個段落
  • "(paragraph | blockquote)+" — 一個或多個段落或引用區塊
  • "heading paragraph+" — 先一個標題，再一或多個段落
  • "inline*" — 零或多個行內節點

MarkSpec 定義範例：


const schema = new Schema({
  nodes: { /* ... */ },
  marks: {
    strong: {
      toDOM() { return ["strong", 0] },
      parseDOM: [{ tag: "strong" }, { tag: "b" }, { style: "font-weight=bold" }]
    },
    em: {
      toDOM() { return ["em", 0] },
      parseDOM: [{ tag: "em" }, { tag: "i" }, { style: "font-style=italic" }]
    },
    link: {
      attrs: { href: {} },
      toDOM(mark) { return ["a", { href: mark.attrs.href }, 0] },
      parseDOM: [{ tag: "a[href]", getAttrs(dom) { return { href: dom.href } } }]
    }
  }
})

每個 Schema 必須至少定義：頂層節點類型（預設 doc）和 text 文字類型。

2.6 Transaction 與 State

ProseMirror 的狀態管理採用函式式架構：

  1. 使用者互動或程式操作產生 Transaction
  2. Transaction 描述對 State 的變更
  3. 套用 Transaction 產生新的 State
  4. 新 State 用來更新 View

let state = EditorState.create({ schema })
let view = new EditorView(document.body, {
  state,
  dispatchTransaction(transaction) {
    let newState = view.state.apply(transaction)
    view.updateState(newState)
  }
})

三、Tiptap 擴展架構

3.1 Extension 類型

Tiptap 將 ProseMirror 的所有可擴展點統一為三種 Extension 類型：

| 類型 | 用途 | 對應 ProseMirror 概念 |

|------|------|----------------------|

| Node | 定義區塊或行內節點 | NodeSpec + NodeView |

| Mark | 定義行內標記（粗體、連結等） | MarkSpec |

| Extension | 功能擴展（鍵盤綁定、歷史等） | Plugin |

3.2 Tiptap JSON 格式

Tiptap JSON 是 ProseMirror Node JSON 表示的等價格式，是 Tiptap 儲存與操作內容的標準格式：


{
  "type": "doc",
  "content": [
    {
      "type": "paragraph",
      "content": [
        {
          "type": "text",
          "text": "Hello "
        },
        {
          "type": "text",
          "text": "world",
          "marks": [{ "type": "bold" }]
        }
      ]
    }
  ]
}

JSONContent 型別定義：


type JSONContent = {
  type?: string                           // 節點類型
  attrs?: Record<string, any>             // 節點屬性
  content?: JSONContent[]                 // 子節點
  marks?: { type: string; attrs?: Record<string, any> }[]  // 標記
  text?: string                           // 文字內容（僅 text 節點）
}

3.3 自訂 Extension 開發

Tiptap 的 Extension.create() / Node.create() / Mark.create() 提供統一的擴展定義介面：


import { Node } from '@tiptap/core'

const CustomNode = Node.create({
  name: 'customNode',
  group: 'block',
  content: 'inline*',
  
  addAttributes() {
    return {
      color: { default: null },
    }
  },
  
  parseHTML() {
    return [{ tag: 'div[data-custom]' }]
  },
  
  renderHTML({ HTMLAttributes }) {
    return ['div', { 'data-custom': '', ...HTMLAttributes }, 0]
  },
  
  addCommands() {
    return {
      setCustomNode: () => ({ commands }) => {
        return commands.setNode(this.name)
      },
    }
  },
})

3.4 Editor 初始化


import { Editor } from '@tiptap/core'
import StarterKit from '@tiptap/starter-kit'

const editor = new Editor({
  element: document.querySelector('.editor'),
  extensions: [StarterKit],
  content: '<p>Hello World</p>',  // 可以是 HTML 或 JSON
})

// 取得 JSON 內容
const json = editor.getJSON()
// 取得 HTML 內容
const html = editor.getHTML()

四、Markdown 與 Tiptap JSON 轉換機制

4.1 轉換管線

tiptap-markdown（aguingand/tiptap-markdown）是最常用的 Markdown 支援擴展，其轉換管線為：

Markdown → Tiptap JSON（解析方向）：


Markdown 文字
  → markdown-it 渲染為 HTML
  → DOM 正規化（移除多餘換行、修正區塊嵌套）
  → ProseMirror DOMParser 解析為 ProseMirror Node
  → 等價於 Tiptap JSON

Tiptap JSON → Markdown（序列化方向）：


ProseMirror Node（Tiptap JSON）
  → 遍歷節點樹
  → 每個 Node/Mark 透過 Extension 的 markdown.serialize 方法輸出
  → 組裝為 Markdown 字串

4.2 MarkdownParser 實作要點


class MarkdownParser {
  constructor(editor, { html, linkify, breaks }) {
    this.editor = editor;
    this.md = markdownit({ html, linkify, breaks });
  }
  
  parse(content, { inline } = {}) {
    if (typeof content === 'string') {
      // 1. 讓每個 Extension 設定 markdown-it 外掛
      this.editor.extensionManager.extensions.forEach(extension =>
        getMarkdownSpec(extension)?.parse?.setup?.call(..., this.md)
      );
      // 2. markdown-it 渲染為 HTML
      const renderedHTML = this.md.render(content);
      const element = elementFromString(renderedHTML);
      // 3. 各 Extension 可修改 DOM
      // 4. 正規化 DOM（修正區塊嵌套、移除多餘換行）
      this.normalizeDOM(element, { inline, content });
      return element.innerHTML;  // 回傳 HTML 供 ProseMirror 解析
    }
    return content;  // 已經是 JSON 則直接回傳
  }
}

4.3 常見 Markdown → Tiptap JSON 對應

| Markdown 語法 | Tiptap JSON Node/Mark |

|---------------|----------------------|

| # 標題 | { type: 'heading', attrs: { level: 1 } } |

| 粗體 | { type: 'text', marks: [{ type: 'bold' }] } |

| 斜體 | { type: 'text', marks: [{ type: 'italic' }] } |

| 連結 | { type: 'text', marks: [{ type: 'link', attrs: { href: url } }] } |

| ` 程式碼 | { type: 'text', marks: [{ type: 'code' }] }` |

| - 列表 | { type: 'bulletList', content: [{ type: 'listItem', content: [...] }] } |

| > 引用 | { type: 'blockquote', content: [...] } |

| ` | { type: 'codeBlock', content: [{ type: 'text', text: '...' }] } |

| --- | { type: 'horizontalRule' } |

| !圖片 | { type: 'image', attrs: { src: url, alt: '圖片' } } |

| 表格 | { type: 'table', content: [{ type: 'tableRow', content: [...] }] } |

五、協作編輯（Yjs 整合）

5.1 ProseMirror 協作編輯原理

ProseMirror 的協作編輯基於 OT（Operational Transformation） 概念：

  1. 各客戶端的變更立即套用到本地文件
  2. 變更（Steps）發送到中央權威（Authority）
  3. 權威決定變更的應用順序
  4. 被拒絕的變更需要在其他變更之上 rebase 後重新提交
  5. 所有文件最終收斂到一致狀態

5.2 Yjs 與 Tiptap

Yjs 是最流行的 CRDT 框架，與 Tiptap 的整合透過 @tiptap/extension-collaboration：

  • Yjs 使用 CRDT 而非 OT，無需中央權威
  • 支援離線編輯與自動合併
  • y-prosemirror 提供 ProseMirror 外掛
  • y-websocket / y-webrtc 提供傳輸層

六、最佳實踐與常見陷阱

6.1 Markdown 輸入最佳實踐

  1. 使用標準 Markdown 語法：避免非標準擴展語法（如某些 Wiki 語法），markdown-it 預設不支援
  2. 正確的巢狀結構：列表內的段落需要正確縮排（4 空格或 1 tab）
  3. 避免 HTML 混合：除非 html: true，否則 HTML 標籤會被當作純文字
  4. 空行很重要：段落之間需要空行分隔，否則會被合併為單一段落
  5. 程式碼區塊用三個反引號：且反引號後可加語言標記（如 ```javascript）

6.2 常見陷阱

| 陷阱 | 原因 | 解決方案 |

|------|------|----------|

| 巢狀列表顯示錯誤 | 縮排不一致 | 統一使用 2 或 4 空格縮排 |

| 表格無法解析 | Tiptap 預設不含 Table 擴展 | 需安裝 @tiptap/extension-table |

| 連結屬性丟失 | Schema 未定義 target 等屬性 | 在 Link 擴展中添加自訂屬性 |

| 圖片尺寸不保留 | Schema 未定義 width/height | 自訂 Image Node 添加尺寸屬性 |

| 程式碼區塊語言丟失 | CodeBlock 未配置語言屬性 | 使用 CodeBlockLowlight 擴展 |

| 換行被吃掉 | markdown-it 預設將單換行視為空格 | 設定 breaks: true 或使用雙空格換行 |

6.3 效能注意事項

  • 大型文件（>10000 節點）應考慮虛擬渲染
  • 頻繁更新使用 editor.chain().focus().insertContent().run() 批次操作
  • JSON 序列化/反序列化比 HTML 更高效
  • 避免在 Transaction handler 中進行重量級計算

七、與 daily-digest-prompt 知識庫的應用建議

7.1 當前架構分析

本專案知識庫（localhost:3000）的資料流為：


Agent 產出 Markdown (contentText)
  → POST /api/import
  → 後端自動 Markdown → Tiptap JSON (content)
  → 儲存到資料庫
  → 前端用 Tiptap Editor 渲染

關鍵發現：匯入時只需填 contentText（Markdown），content（Tiptap JSON）由後端自動生成。這意味著 Markdown 的品質直接決定了最終渲染效果。

7.2 Markdown 品質優化建議

  1. 標題層級規範：研究筆記使用 H1 為標題、H2 為主要章節、H3 為子章節，不跳級
  2. 表格格式化：對齊 | 分隔符，確保表格正確解析
  3. 程式碼區塊：始終標注語言（``javascript、``python 等），確保語法高亮
  4. 列表一致性：全部使用 - 或全部使用 *，不混用
  5. 連結格式：使用 文字 而非裸 URL
  6. 避免過長行：超過 200 字元的段落應適當分段
  7. 圖片替代文字：!描述 始終填寫描述

7.3 支援的 Markdown 語法清單

後端會自動將以下語法轉為 Tiptap JSON：

  • 標題（# ## ###）
  • 段落（空行分隔）
  • 粗體（text）
  • 斜體（text）
  • 行內程式碼（code）
  • 程式碼區塊（```）
  • 連結（text）
  • 項目列表（- item）
  • 數字列表（1. item）
  • 引用區塊（> text）
  • 分隔線（---）
  • 表格（| col | col |）
