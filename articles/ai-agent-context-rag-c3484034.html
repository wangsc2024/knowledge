AI Agent 系統中的 Context 分層載入與 RAG 優化最佳實踐（2026）

研究摘要

本研究探討 2026 年 AI Agent 系統在 Context Window 管理和 RAG（Retrieval-Augmented Generation）優化方面的最新技術和最佳實踐。隨著大型語言模型 Context Window 擴展到 1M-10M tokens，優化重點已從「能否放入 Context」轉向「如何以最經濟的方式放入且保持品質」。

一、Context Engineering 核心概念

1.1 定義演進

Context Engineering 正在取代 Prompt Engineering，成為 2026 年 AI 開發的核心技能。

  • Context Window：模型單次推理的 token 預算
  • Context Layer：上游治理架構，決定哪些內容進入 Window
  • Context Engineering：系統性設計和優化 Context 輸入的學科

來源：Context Engineering Guide for AI Teams 2026

1.2 分層認知模型

生產級 AI Agent 採用五層 Context 架構：

| 層級 | 內容 | 載入時機 |

|------|------|----------|

| Meta-Context | Agent 身份、語氣、信心閾值 | 永久載入（系統 Prompt） |

| Operational Context | 當前任務、用戶意圖、可用工具、約束條件 | 每次任務開始 |

| Domain Context | 產業知識、業務規則 | 按需載入（條件觸發） |

| Historical Context | 壓縮的互動記憶 | 滾動窗口（最近 N 次） |

| Environmental Context | 系統狀態、即時數據、時間感知 | 動態注入（每次推理前） |

組織實踐：使用模組化 Context Assembler（如 LangChain 的 custom memory classes）根據任務複雜度僅載入必要的層。

來源：Optimizing AI Agent Framework with Context Engineering

二、Context Window 優化技術

2.1 漸進式載入（Progressive Context Loading）

核心原則：按需擴展 Context，避免預先載入全部內容。

實施步驟：

  1. 基線載入：僅載入 Meta + Operational Context
  2. 相關性評估：根據任務類型判斷需要的 Domain Context
  3. 動態擴展：在執行過程中根據需求追加 Historical / Environmental Context
  4. 自適應裁剪：超過閾值時移除低相關內容

實證效果：減少平均 token 使用 30-40%，同時保持任務成功率。

來源：5 AI Context Window Optimization Techniques

2.2 基於嵌入的檢索聚類

策略：將歷史對話、範例、文檔預先嵌入向量，根據當前任務語義檢索最相關的 exemplars。

技術要點：

  • 使用 FAISS / Chroma / Weaviate 等向量數據庫
  • 設定相似度閾值（如 cosine similarity > 0.75）
  • 僅載入 top-K 最相關項（K=3-5 為最佳平衡點）

2.3 注意力加權剪枝（Attention-Weighted Pruning）

原理：在推理過程中移除低注意力權重的內容。

實作方式：

  • 追蹤 Attention Scores（需模型支援，如 GPT-4 Turbo / Claude 3.5）
  • 在 Session 中期（如第 5 輪對話後）移除 attention < 0.1 的段落
  • 保留高權重的「錨點內容」（如任務描述、關鍵規則）

2.4 相關性衰減（Relevance Decay）

策略：根據時間和主題漂移降低舊內容的優先級。

衰減公式：


relevance_score = base_relevance × (0.95 ^ days_elapsed) × topic_similarity
  • days_elapsed：內容產生後經過的天數
  • topic_similarity：與當前任務的主題相似度（cosine）
  • 衰減後 score < 0.3 的內容自動移除

來源：Fix AI Agents that Miss Critical Details

三、RAG 系統優化策略

3.1 多層快取架構

生產級 RAG 系統採用三層快取：

| 層級 | 匹配方式 | 延遲 | 命中率 | 適用場景 |

|------|---------|------|--------|---------|

| Exact Match Cache | 完全一致的查詢 | <1ms | 15-25% | 高重複性查詢（FAQ、常見指令） |

| Semantic Cache | 語義相似（embedding distance < threshold） | 5-10ms | 30-45% | 相似問題、改寫查詢 |

| Session Context | 對話狀態管理 | 實時 | N/A | 跨輪對話的記憶保持 |

實證效果：在高重複性聊天機器人場景中，三層快取可減少 60-70% 的 LLM 呼叫次數。

來源：LLM Token Optimization: Cut Costs & Latency in 2026

3.2 RAGCache 架構

RAGCache 是專為 RAG 設計的延遲優化服務系統（論文：arXiv:2404.12457）：

核心技術：

  1. Knowledge Tree：將檢索到的知識組織為樹狀結構，跨 GPU 和主機記憶體階層快取中間狀態
  2. 動態推測流水線（Dynamic Speculative Pipelining）：利用串流搜尋行為，重疊檢索與 LLM 生成，降低端到端延遲
  3. 階層記憶體管理：熱點知識放 GPU VRAM，次熱放 Host RAM，冷資料放磁碟

性能提升（vs. vLLM + Faiss）：

  • TTFT（Time to First Token）降低 4 倍
  • 吞吐量提升 2.1 倍

來源：RAGCache: Efficient Knowledge Caching for RAG

3.3 Token 優化技術

#### LLMLingua 壓縮

功能：壓縮 Prompt 同時最小化性能損失，特別適合長檢索上下文。

實作策略：

  • 保留關鍵實體和關係詞
  • 移除冗餘的修飾詞、轉折詞
  • 壓縮率可達 50-70%，準確率損失 < 5%

#### 語義分塊（Semantic Chunking）

原則：按語義邊界切分文檔，而非固定長度。

最佳實踐：

  • 使用 LangChain 的 RecursiveCharacterTextSplitter 搭配語義邊界偵測
  • Chunk size：500-1000 tokens（依模型而定）
  • Overlap：50-100 tokens（保持上下文連續性）

#### 固定 Token 預算

策略：限制檢索結果的總 token 數，強制優先相關性而非數量。

配置範例：


retrieval_budget:
  max_tokens: 4000  # 單次檢索上限
  top_k: 5          # 最多返回 5 個 chunks
  min_relevance: 0.7  # 最低相關度閾值

來源：Techniques for Minimizing LLM Token Usage

3.4 成本優化組合策略

生產級 RAG 系統透過以下組合策略可減少 30-45% 的成本：

  1. Smart Routing：簡單查詢路由到小模型（如 GPT-3.5），複雜查詢才用 GPT-4
  2. Semantic Caching：消除重複推理呼叫
  3. Model Degradation：在非關鍵路徑使用較小模型（如摘要、分類）
  4. Prompt Compression：使用 LLMLingua 或自訂壓縮策略

實測節省：某企業 RAG 系統在導入以上策略後，月度基礎設施成本降低 40-46%。

來源：AI Agent Token Cost Optimization Guide 2026

四、對 Daily Digest Prompt 專案的實施建議

4.1 短期優化（1-2 週可完成）

#### 建議 1：實施三層 Context 載入

目標：降低 auto-task 的平均 token 使用 30%。

實作步驟：

  1. 將 prompts/team/assemble-digest.md 拆分為三層：

- L0-meta.md：Agent 身份、語氣、禁令（永久載入）

- L1-operational.md：當前任務步驟、工具約束（每次載入）

- L2-domain.md：Skill 路由、研究去重規則（按需載入）

  1. 在 run-todoist-agent-team.ps1 Phase 1 實施漸進式載入邏輯
  2. 使用 config/context-layers.yaml 配置載入規則

預期效果：

  • 減少 Phase 2 平均 token 使用 2000-3000 tokens
  • Cache hit rate 從 ~60% 提升至 ~75%

#### 建議 2：強化語義快取

目標：KB 查詢命中率從 15% 提升至 40%。

實作步驟：

  1. 擴展 tools/kb_semantic_cache.py 支援 profile 變體（topK=3/5/10）
  2. 在 knowledge-query Skill 的所有搜尋 API 前強制插入 lookup
  3. 調整相似度閾值：distance < 0.15（當前 0.1 過嚴）
  4. 新增快取預熱：在 Phase 0 預載入高頻查詢（如「今日任務」「最新研究」）

預期效果：

  • KB API 呼叫次數減少 40%
  • Phase 1 執行時間縮短 15-20%

4.2 中期優化（2-4 週）

#### 建議 3：導入 RAGCache 式知識樹

目標：建立專案級的知識快取樹，支援跨 Session 重用。

架構設計：


cache/knowledge-tree/
├── meta/             # Skill、ADR、Config 的靜態知識（7 天 TTL）
├── operational/      # 最近 N 次執行的狀態摘要（1 天 TTL）
├── domain/           # 研究主題、待辦事項分類（3 天 TTL）
└── index.json        # 樹狀索引，記錄節點依賴關係

實作要點：

  • 使用 tools/knowledge_tree_manager.py 管理節點生命週期
  • 在 preamble.md 新增「步驟 -1：載入知識樹」
  • 當 Cache Miss 時才讀取原始檔案並更新樹節點

預期效果：

  • TTFT（首 token 延遲）降低 50%
  • 跨 Session 的重複讀取減少 70%

#### 建議 4：實施 Attention-Weighted 記憶保留

目標：自動識別和保留高價值的歷史 Context。

實作步驟：

  1. 在 context/continuity/*.json 新增 attention_score 欄位
  2. 使用 Claude 的 usage metadata（需 API 支援）或啟發式規則計算：

```python

attention_score = (

0.4 * task_success_rate + # 該記憶是否導致成功

0.3 * reuse_count + # 被引用次數

0.3 * recency_factor # 時間衰減

)

```

  1. 在 max_runs 清理邏輯中，保留 top-K 高分項，而非單純 FIFO

預期效果：

  • 關鍵經驗（如失敗根因）的保留期從 5 次延長至有效保留
  • 減少重複踩坑次數

4.3 長期優化（1-2 月）

#### 建議 5：模型分級路由（Smart Routing）

目標：簡單任務用小模型，減少 40% 成本。

分級策略：

| 任務類型 | 模型選擇 | 理由 |

|---------|---------|------|

| KB 查詢摘要 | Haiku | 結構化輸出，無需深度推理 |

| 研究去重判斷 | Sonnet | 需語義理解，但不需生成長文 |

| 深度研究 | Opus | 需複雜推理和長篇生成 |

| Phase 3 組裝 | Sonnet | 結構化組裝，中等複雜度 |

實作：

  • 擴展 tools/llm_router.py 支援任務類型 → 模型映射
  • 在 config/llm-router.yaml 新增 task_complexity 評分規則
  • 在 run-todoist-agent-team.ps1 Phase 1a 預分類時決定模型

預期效果：

  • 月度 API 成本降低 35-40%
  • 簡單任務延遲降低 50%（Haiku 更快）

#### 建議 6：Context 壓縮管線（LLMLingua 整合）

目標：長 Prompt 壓縮 50% 且準確率損失 < 5%。

整合步驟：

  1. 安裝 llmlingua：uv add llmlingua
  2. 建立 tools/context_compressor.py：

```python

from llmlingua import PromptCompressor

def compress_context(text: str, target_ratio: float = 0.5) -> str:

compressor = PromptCompressor()

return compressor.compress_prompt(

text,

rate=target_ratio,

target_token=-1 # 自動計算

)["compressed_prompt"]

```

  1. 在 Phase 2 的 auto-task prompt 組裝前呼叫壓縮
  2. 設定壓縮閾值：僅當 prompt > 8000 tokens 時啟用

預期效果：

  • 研究類任務的 input tokens 減少 40-50%
  • 整體成本再降 15-20%
