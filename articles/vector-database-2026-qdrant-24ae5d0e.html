Vector Database 生產選型 2026：Qdrant、Weaviate、Milvus 與 pgvector 架構比較、效能 Benchmark 與部署實踐

執行摘要

2026 年向量資料庫市場已進入成熟期，四大主流開源選項各占明確的生態位。pgvector 是 Postgres 生態的輕量整合方案，適合 <1000 萬向量、已有 Postgres 基礎設施、過濾需求簡單的場景 [Qdrant Blog, 2026; instaclustr, 2026]。Qdrant 以 Rust 撰寫，ACORN 算法解決了業界長期存在的 filtered HNSW 效能瓶頸，成為複雜 metadata 過濾場景的首選 [Qdrant Sparse Vectors doc, 2026; vecstore, 2026]。Milvus 2.6 以完全分離的計算/存儲/索引架構支援億級規模，新引入 RaBitQ 1-bit 量化與冷熱分層存儲大幅降低成本 [Milvus 2.6 Blog, 2026]。Weaviate 模組化設計內建 text2vec/img2vec，GraphQL 查詢介面降低 embedding 管理複雜度，在知識圖與多模態應用場景具備獨特優勢 [Weaviate Hybrid Search doc, 2026]。

ANN 算法選型已有明確三層決策：HNSW 適合動態中型規模（<1 億向量），DiskANN 適合超出 RAM 的 SSD 大規模場景，IVF+PQ 適合靜態億級磁碟檢索 [Vectroid, 2026; Milvus ANN Blog, 2026]。混合搜尋（BM25+向量+RRF）已成 2026 年生產 RAG 標準配置，將 recall@10 從 65-78% 提升至 91% [GoPenAI Blog, 2026]。

---

研究背景

向量資料庫是現代 RAG（Retrieval-Augmented Generation）系統的核心基礎設施。隨著 LLM 應用規模從百萬向量擴展至數十億向量，選型決策對系統效能、成本與工程複雜度的影響日益顯著。本報告承接 qdrant-vector-database 系列的 foundation 知識（HNSW/量化/分片基礎），進入 application 階段，聚焦於四大平台的實作比較、選型決策框架與生產部署最佳實踐。

---

核心發現

四大平台生態位分化

四大向量資料庫在 2026 年已形成明確分工 [Firecrawl, 2026; DEV.to, 2026; elest.io, 2026; DataCamp, 2026]：

| 平台 | 語言 | 核心優勢 | 主要使用場景 |

|------|------|---------|------------|

| Qdrant | Rust | ACORN filtered search、最低延遲（p50=4ms） | 複雜 metadata 過濾的高效能 RAG |

| Weaviate | Go | 內建 vectorizer、GraphQL、模組化 | 知識圖、多模態、開發體驗優先 |

| Milvus | Go+C++ | 分離式架構、億級規模、K8s 原生 | 企業大規模部署、多租戶 |

| pgvector | C (Postgres 擴充) | Postgres 原生整合、事務一致性 | 小規模 MVP、已有 Postgres |

三種哲學代表三種工程決策路徑：輕量整合優先（pgvector）、過濾效能優先（Qdrant）、分散式擴展優先（Milvus）。✅ 已驗證（4 來源）

混合搜尋已成生產標準

2026 年所有主流向量資料庫均原生支援混合搜尋 [GoPenAI Blog, 2026; Weaviate Hybrid doc, 2026; Qdrant Sparse Vectors doc, 2026; elest.io, 2026]。混合搜尋將 BM25/SPLADE（稀疏檢索）與 dense embedding 並行執行，透過 RRF（Reciprocal Rank Fusion）融合結果：

  • Recall@10 提升：65-78%（純向量） → 91%（混合+RRF），RRF 融合步驟僅需 6ms 額外延遲 [GoPenAI Blog, 2026] ✅
  • SPLADE vs BM25：SPLADE 在 BEIR benchmark 上超越 BM25，但延遲 80-120ms（GPU endpoint）；BM25 <10ms，更適合延遲敏感場景 [GoPenAI Blog, 2026] ⚠️
  • Weaviate rankedFusion：位置加權評分，NDCG@10 提升 42%（官方數據，待第三方確認）[Weaviate Hybrid Search, 2026] ⚠️

Qdrant ACORN 算法的核心突破

長期以來，向量資料庫在面對選擇性過濾查詢時，傳統做法是先向量搜尋再後處理過濾，導致高選擇性過濾下精度急劇下降。Qdrant ACORN 算法將 metadata filtering 整合進 HNSW 圖遍歷本身，而非作為後處理步驟，即使過濾條件淘汰 99% 候選向量，查詢速度仍能維持 [Qdrant Sparse Vectors doc, 2026; vecstore, 2026; instaclustr, 2026]。✅ 已驗證（3 來源）

pgvector 在同樣場景下退化為 O(n) 全表掃描，1536 維向量配合複雜過濾時效能不可接受 [Qdrant Blog, 2026]。

ANN 算法三層決策框架

[Vectroid, 2026; Milvus ANN Blog, 2026; TigerData; netcrit] ✅ 已驗證（4 來源）

  • 資料規模 < 1 億 + 需動態更新 → HNSW：分層圖結構，支援即時插入，記憶體消耗大，所有平台預設索引
  • 資料規模 > 索引 RAM 容量 → DiskANN：圖索引存 SSD，記憶體僅快取熱點，比 HNSW 吞吐量高
  • 靜態億級 + 磁碟優化 → IVF + PQ：向量集群化後量化壓縮，犧牲約 2-5% recall 換取大幅空間節省

Milvus 2.6 成本革命

Milvus 2.6 帶來三項重要突破 [Milvus 2.6 Blog, 2026; Zilliz PR, 2026]：

  1. RaBitQ 1-bit 量化：72% 記憶體節省，查詢速度 4x 提升（官方聲明，待第三方 Benchmark）⚠️
  2. Hot/Cold 分層存儲：整體成本降低 50% ⚠️
  3. Streaming Node：去除外部消息佇列（Kafka/Pulsar）依賴
  4. BM25 全文搜尋：比 Elasticsearch 快 4 倍（官方聲明）⚠️
  5. 100K Collections/Cluster：大幅提升多租戶能力

pgvector 的適用邊界與遷移決策

✅ 已驗證（3 來源）[Qdrant Blog, 2026; instaclustr, 2026; vecstore, 2026]

維持 pgvector：資料量 <1000 萬向量、已有 Postgres、需事務一致性、過濾條件簡單

觸發遷移至 Qdrant/Milvus：

  1. 資料規模 >1000 萬向量（尤其 ColBERT 多向量場景）
  2. 需要複雜 metadata 過濾（多條件、範圍查詢）
  3. 向量與關聯查詢競爭 Postgres 資源
  4. 混合搜尋需求（pgvector 全掃描效率低）

遷移模式：雙寫（MVP）→ Transactional Outbox（生產）→ CDC Pipeline（高吞吐）

深度分析

各平台 LangChain 整合

三大平台在 LangChain 均有官方 VectorStore 實作 [elest.io, 2026; LangChain Milvus docs, 2026; netjstech, 2026]：

Qdrant + LangChain：


from langchain_community.vectorstores import Qdrant
from qdrant_client import QdrantClient

client = QdrantClient(url="http://localhost:6333")
vectorstore = Qdrant(
    client=client, collection_name="my_collection", embeddings=embeddings
)
# metadata 過濾
results = vectorstore.similarity_search(
    query, k=5, filter={"field": "category", "match": {"value": "tech"}}
)

Milvus + LangChain：


from langchain_milvus import Milvus
vectorstore = Milvus(
    embedding_function=embeddings,
    connection_args={"uri": "http://localhost:19530"},
    collection_name="my_collection",
)

Weaviate + LangChain（內建 vectorizer，省去 embedding 步驟）：


import weaviate
from langchain_community.vectorstores import Weaviate
client = weaviate.Client("http://localhost:8080")
vectorstore = Weaviate(client, "Document", "content")

量化策略比較

| 平台 | 量化方式 | 記憶體節省 | Recall 影響 |

|------|---------|----------|-----------|

| Qdrant | Scalar (int8) / Binary | 4x / 32x | <5% / ~10% |

| Weaviate | Rotational Quantization | 4-8x | <3% |

| Milvus | IVF-based PQ / RaBitQ (v2.6) | 8x / ~7x | 2-5% |

實務應用建議

對 daily-digest-prompt 系統：本系統使用 Qdrant（localhost:3000），hybrid search 已是 BM25+向量+RRF 標準配置，技術選型正確。

一般建議：

  1. 新專案默認選 Qdrant（而非 pgvector-再遷移）：支援 ACORN 過濾、混合搜尋、量化，localhost Docker 模式輕量
  2. 混合搜尋配置：alpha=0.5（balanced）起點，延遲敏感時用 BM25 而非 SPLADE
  3. ANN 索引：<100 萬用 HNSW 預設；100 萬-1 億用 HNSW m=16,ef=128；>1 億上 Milvus+DiskANN 或 IVF+PQ
  4. 量化：>1000 萬向量啟用 Scalar Quantization (int8)，<5% recall 損失換 75% 記憶體節省
  5. Milvus 遷移時機：多租戶(>100 tenant)、需 K8s 擴展、>1 億向量
