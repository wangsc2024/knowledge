Unsloth 生產環境端到端部署管線

概述

Unsloth 不僅是高效的 LLM 微調框架（2-5x 加速、80% VRAM 節省），更提供從訓練到生產部署的完整管線。本筆記聚焦「微調完成後」的端到端部署工作流：模型匯出策略選擇、多平台部署（vLLM/Ollama/SGLang/LM Studio）、LoRA 熱插拔、品質驗證、以及 CI/CD 自動化管線建構。這填補了「知道如何訓練，但不知道如何上線」的知識缺口，是將 PoC 轉化為生產服務的關鍵一環。

一、模型匯出策略（三種路徑）

微調完成後，Unsloth 支援三種匯出格式，適用不同部署場景：

1.1 FP16 合併匯出（推薦用於 vLLM）

將 LoRA adapter 合併回基礎模型，產出完整的 16-bit 權重檔案：


from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="./my-finetuned-lora",
    max_seq_length=2048,
    load_in_4bit=True,
)

# 合併為 16-bit（vLLM 推薦格式）
model.save_pretrained_merged(
    "finetuned_model",
    tokenizer,
    save_method="merged_16bit",
)

# 或直接推送至 HuggingFace Hub
model.push_to_hub_merged(
    "hf_username/my-model",
    tokenizer,
    save_method="merged_16bit",
    token="hf_xxx",
)

關鍵原則：FP16 合併檢查點是「標準高保真檢查點」(canonical high-fidelity checkpoint)，所有後續 GGUF 匯出和進一步微調都應從此檔案出發。

1.2 GGUF 量化匯出（推薦用於 Ollama/llama.cpp/LM Studio）


# 一鍵 GGUF 匯出（推薦）
model.save_pretrained_gguf(
    "my-model-gguf",
    tokenizer,
    quantization_method="q4_k_m",  # 最佳平衡點
)

# 推送至 HuggingFace Hub
model.push_to_hub_gguf(
    "hf_username/my-model-GGUF",
    tokenizer,
    quantization_method="q4_k_m",
)

量化方法速查表：

| 方法 | 每權重位元 | 適用場景 | 說明 |

|------|-----------|---------|------|

| f16 | 16 | 最高精度基準 | 檔案大、推論慢 |

| q8_0 | 8 | 快速匯出驗證 | 精度損失極小 |

| q5_k_m | 5.5 | 高品質推論 | 接近 FP16 品質 |

| q4_k_m | 4.5 | 最佳平衡 | Q6_K 用於一半注意力張量 |

| q4_0 | 4 | ARM CPU 優化 | 適合行動裝置 |

| iq2_xxs | 2.06 | 極致壓縮 | 品質明顯下降 |

Dynamic 2.0 量化：Unsloth 預設啟用動態量化，智能地為不同層選擇不同精度——敏感層保持 6-bit，穩健層降至 4-bit，在相同檔案大小下獲得更好品質。每個模型有專屬的量化方案（Gemma 3 的量化模式與 Llama 4 完全不同）。

1.3 LoRA Adapter 匯出（推薦用於多 adapter 服務）

僅儲存 adapter 權重（約 100MB），部署時載入基礎模型 + adapter：


# 僅儲存 LoRA adapter
model.save_pretrained("finetuned_lora")
tokenizer.save_pretrained("finetuned_lora")

# 或透過內建方法
model.save_pretrained_merged(
    "finetuned_model",
    tokenizer,
    save_method="lora",
)

二、vLLM 生產部署

2.1 基礎部署


# 安裝 vLLM
uv pip install -U vllm --torch-backend=auto

# 啟動 API 伺服器
vllm serve finetuned_model

# 指定完整路徑
vllm serve /mnt/models/finetuned_model

# 使用 HuggingFace Hub 模型
vllm serve unsloth/gpt-oss-120b

2.2 生產級配置


vllm serve finetuned_model \
    --quantization fp8 \
    --kv-cache-dtype fp8 \
    --gpu-memory-utilization 0.8 \
    --max-model-len 65536 \
    --enable-lora \
    --max-loras 4 \
    --max-lora-rank 64 \
    --port 8000

關鍵參數說明：

  • --quantization fp8：執行期 FP8 量化（需 FP8 支援 GPU）
  • --kv-cache-dtype fp8：KV 快取使用 FP8，降低記憶體用量
  • --gpu-memory-utilization：GPU 記憶體使用比例（預設 0.9，OOM 時降至 0.75）
  • --max-model-len：最大上下文長度
  • --enable-lora：啟用 LoRA adapter 動態載入
  • --max-loras：同時載入的 LoRA 數量上限

2.3 LoRA 熱插拔（多 Adapter 服務）

適用場景：一個基礎模型服務多個客製化版本（多租戶、A/B 測試）。


# 啟動伺服器（啟用動態 LoRA 載入）
export VLLM_ALLOW_RUNTIME_LORA_UPDATING=True
vllm serve unsloth/Llama-3.1-8B-Instruct \
    --enable-lora \
    --max-loras 4 \
    --max-lora-rank 64


# 動態載入 LoRA adapter
curl -X POST http://localhost:8000/v1/load_lora_adapter \
    -H "Content-Type: application/json" \
    -d '{"lora_name": "customer-A", "lora_path": "/models/lora-customer-A"}'

# 使用特定 adapter 進行推論
curl -X POST http://localhost:8000/v1/chat/completions \
    -H "Content-Type: application/json" \
    -d '{"model": "customer-A", "messages": [{"role": "user", "content": "Hello!"}]}'

# 卸載 adapter
curl -X POST http://localhost:8000/v1/unload_lora_adapter \
    -H "Content-Type: application/json" \
    -d '{"lora_name": "customer-A"}'

生產注意事項：

  • VLLM_ALLOW_RUNTIME_LORA_UPDATING=True 僅適合開發/測試
  • 生產環境應使用 --lora-modules CLI 預載 adapter
  • 動態載入無法保證跨副本一致性，也無法在副本重啟後自動恢復

三、Ollama 部署

3.1 完整工作流


# 步驟 1：匯出為 GGUF
model.save_pretrained_gguf(
    "my-model-gguf",
    tokenizer,
    quantization_method="q8_0",
)


# 步驟 2：建立 Ollama 模型（使用 Unsloth 自動產生的 Modelfile）
ollama create my-model -f ./my-model-gguf/Modelfile

# 步驟 3：執行推論
ollama run my-model "What is machine learning?"

# 步驟 4：作為 API 服務
ollama serve  # 啟動 API 伺服器（localhost:11434）
curl http://localhost:11434/api/chat -d '{
    "model": "my-model",
    "messages": [{"role": "user", "content": "Hello"}]
}'

3.2 Chat Template 對齊（最常見部署問題）

Unsloth 自動產生的 Modelfile 包含訓練時使用的 chat template。訓練和推論必須使用完全相同的 chat template，否則會產生：

  • 亂碼輸出（gibberish）
  • 無限生成（endless generation）
  • 重複輸出（repeated outputs）

排除步驟：

  1. 確認 Modelfile 中的 TEMPLATE 與訓練時一致
  2. 檢查 eos_token 設定是否正確
  3. 使用 Unsloth 提供的 conversational notebooks 強制套用正確 template

五、端到端 CI/CD 管線範例（Union.ai 實戰）


from union import task, Resources, workflow
from flytekit.extras.accelerators import L4

# 階段 1：微調
@task(requests=Resources(mem="23Gi", gpu="1"), accelerator=L4)
def finetune():
    from unsloth import FastLanguageModel
    from trl import SFTTrainer, SFTConfig
    model, tokenizer = FastLanguageModel.from_pretrained(
        model_name="unsloth/Qwen3-1.7B-unsloth-bnb-4bit",
        max_seq_length=2048, load_in_4bit=True,
    )
    model = FastLanguageModel.get_peft_model(
        model, r=32,
        target_modules=["q_proj","k_proj","v_proj","o_proj",
                        "gate_proj","up_proj","down_proj"],
        lora_alpha=32, use_gradient_checkpointing="unsloth",
    )
    trainer = SFTTrainer(model=model, tokenizer=tokenizer,
        train_dataset=dataset,
        args=SFTConfig(per_device_train_batch_size=2,
            gradient_accumulation_steps=4, learning_rate=2e-4,
            optim="adamw_8bit"),
    )
    trainer.train()
    model.save_pretrained("lora_model")

# 階段 2：轉換為 vLLM 格式
@task(requests=Resources(mem="23Gi", gpu="1"), accelerator=L4)
def convert_vllm(tuned_model):
    from unsloth import FastLanguageModel
    model, tokenizer = FastLanguageModel.from_pretrained(
        model_name=tuned_model, max_seq_length=2048, load_in_4bit=True,
    )
    model.save_pretrained_merged(
        "vllm_model", tokenizer, save_method="merged_16bit",
    )

# 階段 3：部署
from union.app.llm import VLLMApp
app = VLLMApp(
    name="my-tuned-model",
    requests=Resources(mem="23Gi", gpu="1"),
    model_id="my-model", stream_model=True,
    accelerator=GPUAccelerator("nvidia-l40s"),
)

@workflow
def train_and_deploy():
    model = finetune()
    convert_vllm(tuned_model=model)

六、部署平台選擇決策樹


需求分析
├── 多用戶 API 服務 → vLLM（FP16 合併 或 LoRA adapter）
│   ├── 多租戶 → vLLM + LoRA 熱插拔
│   └── 單模型 → vLLM serve（最簡部署）
├── 個人/團隊本地使用 → Ollama（GGUF q4_k_m）
│   ├── GUI 介面 → LM Studio
│   └── CLI + API → Ollama serve
├── 結構化輸出需求 → SGLang
├── 邊緣裝置/行動端 → GGUF q4_0（ARM 優化）
└── 快速驗證 → Unsloth 原生推論（fast_inference=True）

七、常見問題與解決方案

| 問題 | 原因 | 解決方案 |

|------|------|--------|

| 推論輸出亂碼或重複 | Chat template 不匹配 | 使用 Unsloth 自動產生的 Modelfile |

| vLLM OOM | memory_usage 預設值過高 | --gpu-memory-utilization 0.75 |

| GGUF 匯出失敗 | llama.cpp 版本不相容 | 更新至最新版 |

| LoRA 熱插拔不穩定 | 動態載入不保證跨副本一致 | 生產用 --lora-modules 預載 |

| 量化品質評估不準 | 使用 perplexity 評估 | 改用 KL Divergence |
