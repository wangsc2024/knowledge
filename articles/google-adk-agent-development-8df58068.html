Google ADK（Agent Development Kit）深度研究

專案概述

Google ADK（google/adk-python）是 Google 於 2025 年 4 月開源的 Code-First AI Agent 開發工具包，19.2k GitHub Stars，3.3k Forks，bi-weekly 發版（目前 v1.x）。核心設計理念：「把 Agent 開發當作軟體開發」——以程式碼定義 Agent 邏輯、工具與編排，強調可測試性、版本控制與企業級部署。

選題理由：A2A 協議（v1.0，已納入 Linux Foundation）+ 原生 MCP 整合，代表 2026 年 AI Agent 互通標準的最前沿，且有 50+ 企業合作夥伴（Box, Deloitte, Elastic, PayPal, Salesforce, ServiceNow, UiPath）生產採用。[來源:GitHub google/adk-python,2026]

技術架構

三層 Agent 架構

ADK 定義三種 Agent 類型，形成分層體系 [來源:adk.dev 官方文件,2026]：

  1. LLM Agents：直接使用語言模型（Gemini 最優化，但 model-agnostic）進行推理與工具呼叫
  2. Workflow Agents：基於 Graph 的確定性編排，支援 Sequential、Parallel、Loop 三種模式
  3. Custom Agents：用戶自定義 Agent 實作，繼承 ADK 基礎設施

關鍵設計：根 Agent（root agent）透過階層結構委派給專門子 Agent，每層都可混用 LLM 動態決策與 Workflow 靜態結構。[來源:ZenML ADK vs LangGraph 比較,2026]

Context Intelligence（ADK 獨特優勢）

「Context 像原始碼一樣被管理（treated like source code）」

ADK 不做 naive 字串拼接，而是主動：

  • 自動過濾（filter）不必要的歷史 turns
  • 按需摘要（summarization）壓縮長 context
  • Lazy-loading：工具輸出與 artifact 僅在需要時注入 context

效果：Token 效率提升，減少 context window 爆炸問題，對長對話 Agent 尤其有價值。[來源:adk.dev 官方文件,2026]

A2A（Agent2Agent）協議

A2A 是 ADK 最具破壞性的創新之一，現已 v1.0 並納入 Linux Foundation 治理 [來源:Google Cloud Blog Building Connected Agents,2026]：


每個 A2A Agent 在 /.well-known/agent-card.json 發佈能力描述
→ 其他 Agent（甚至跨框架：LangGraph, CrewAI）可動態發現並呼叫

三層生產堆疊：

  • MCP：統一工具接入標準（外部 API/資料庫）
  • ADK：Agent 邏輯與編排
  • A2A：跨框架、跨系統的 Agent 間通訊

MCP 原生整合

ADK 對 MCP（Model Context Protocol）提供一等公民支援，可直接使用任何 MCP 工具，無需額外包裝層。結合 A2A + MCP，ADK Agent 可以：

  1. 透過 MCP 連接任何外部工具/資料源
  2. 透過 A2A 與其他框架的 Agent 通訊
  3. 在 Vertex AI 上托管為企業服務

功能特色與差異化

對比同類框架 [來源:ZenML ADK vs LangGraph,2026][來源:RaftLabs AI agent framework comparison,2026]

| 特性 | Google ADK | LangGraph | CrewAI |

|------|-----------|-----------|-------|

| 核心模型 | 階層式 Agent Tree | 節點圖（State Machine） | 角色驅動 Crew |

| 學習曲線 | 中等 | 高 | 低 |

| 多語言 | Python/TS/Go/Java | Python/JS | Python |

| 觀測性 | OpenTelemetry-first | LangSmith（vendor lock） | 有限 |

| MCP 支援 | 原生 | 需包裝 | 需包裝 |

| A2A 支援 | 原生 | 無 | 無 |

| GCP 整合 | 深度 | 無 | 無 |

| Context 管理 | 智能過濾 | 手動控制 | 基本 |

| 生產部署 | Vertex AI Engine | LangSmith Platform | 自管 |

獨特能力

  1. 多模態原生：透過 Gemini 多模態 API，Agent 可處理圖像、音頻、視頻 [來源:RaftLabs比較,2026]
  2. Session Rewind：可回溯至前一個 invocation 狀態（除錯/補救必備）
  3. Tool Confirmation Flow：Human-in-the-loop 的工具執行確認機制
  4. Built-in Eval Framework：視覺化 debug + trajectory 分析 + 用戶模擬測試
  5. AI-Powered Development：框架本身支援 human 與 AI 雙重撰寫模式

使用方式

快速上手


pip install google-adk

最簡單的 Agent


from google.adk.agents import Agent
from google.adk.tools import google_search

root_agent = Agent(
    name="search_assistant",
    model="gemini-2.5-flash",
    instruction="Answer questions using Google Search when needed.",
    tools=[google_search]
)

Multi-Agent 系統


from google.adk.agents import Agent
from google.adk.agents.workflow import SequentialAgent

# 專門子 Agent
research_agent = Agent(name="researcher", model="gemini-2.5-flash", ...)
writer_agent = Agent(name="writer", model="gemini-2.5-flash", ...)

# 協調器
orchestrator = SequentialAgent(
    name="pipeline",
    sub_agents=[research_agent, writer_agent]
)

潛在應用（daily-digest-prompt 與個人專案）

對本專案的高優先借鏡

  1. Context Intelligence 思想 → prompts/team/ 的 context 注入策略：目前 Phase 2 是直接拼接，可借鏡 ADK 的「lazy-load + 按需摘要」模式，減少 token 消耗（對齊 token-efficiency 目標）
  1. A2A Protocol 意義 → 本專案的 run-todoist-agent-team.ps1 Phase 1→2→3 已是多 Agent 架構，若未來要接入 Vertex AI 或其他 Google 服務，A2A 是現成的跨框架橋接協議
  1. Session Rewind 機制 → 類似本專案的 context/continuity/*.json，但 ADK 做到 session 級精確回溯，可作為 auto-task continuity 系統的靈感來源
  1. Tool Confirmation Flow → 對應本專案的 self-heal-authorization.yaml：ADK 在工具執行前的人工確認機制，與本專案 Tier 升級需人工審批是同一模式
  1. OpenTelemetry-first 觀測性 → 本專案的 tools/trace_analyzer.py 目前是規則式，若要升級到 OTel 標準格式，ADK 的實作是最佳參考

可快速落地的技術

  • ADK SequentialAgent / ParallelAgent 可直接替換本專案現有的 PS1 Phase 串行/並行邏輯，且自帶錯誤補償
  • ADK Eval Framework 可用於評測本專案各 auto-task 的輸出品質（替代現有的手動 done_cert 評分）

優缺點評估

優點

  • 深度 GCP 整合：Vertex AI Agent Engine 提供受管 prod 部署，Cloud Trace 觀測，IAM 認證全包 [來源:adk.dev,2026]
  • A2A + MCP 雙協議：目前唯一同時原生支援兩者的主流框架
  • 多語言：Python/TS/Go/Java，企業多語言環境無縫整合
  • Context Intelligence：Token 效率優勢，適合長對話/複雜任務
  • OpenTelemetry-first：不鎖定 vendor，可接任何觀測後端

缺點

  • 框架較新：v1.x，相對 LangGraph（18.8k stars, 10M+ 月下載）的生態成熟度仍有差距 [來源:ZenML比較,2026]
  • Gemini 偏好性：雖說 model-agnostic，但許多功能與 Gemini 深度整合，使用其他 LLM 體驗有差異 [來源:ZenML比較,2026]
  • GCP 綁定傾向：最佳部署體驗在 Google Cloud，on-premise 或其他雲的優化相對欠缺 [來源:RaftLabs比較,2026]
  • 社群規模：r/agentdevelopmentkit 剛起步，Stack Overflow 解答覆蓋率不如 LangChain
