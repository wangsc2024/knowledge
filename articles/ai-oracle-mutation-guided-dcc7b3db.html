AI 自動測試修復與 Oracle 校準：結合 Mutation-Guided Generation、LLM Test Oracle 與 Multi-Agent Consensus 的優化實務

承接既有筆記：

  • AI 驅動的自動測試生成完整指南 -- 從 LLM 單元測試生成、Mutation Testing 自動化到工業級部署的工程實踐與工具鏈（2026）
  • AI 驅動 CI/CD 優化：構建時間預測、flaky test 偵測與測試選擇實作指南（2026）

技術概述

AI 用於測試的真正瓶頸，已不再是「能不能生成測試」，而是「生成後的測試可不可信、壞掉時怎麼修、assert 是否真的反映預期行為」。最新實務顯示，有效方案不是單靠單次 LLM 產生測試，而是把 mutation-guided generation、oracle correctness 檢查、失敗測試修復與共識式審核串成閉環。這種閉環可把 AI 測試從一次性草稿提升為持續可運作的工程資產，特別適合需要長期維持 CI 穩定性與低誤報率的專案。

核心方法與工具

  • Mutation-guided generation：先故意植入目標型 mutant，再要求模型生成能殺死該 mutant 的測試，而不是只追求 coverage。
  • Oracle calibration：把 assert 生成視為獨立問題，檢查它反映的是「預期行為」還是「目前實作行為」。
  • Multi-agent consensus：用多個推理代理討論 oracle，再由較便宜的模型輸出精簡結論，降低 hallucination。
  • Test repair loop：測試失敗後，不直接重寫全部測試，而是保留 failing signal、diff、stack trace，讓模型只修 prefix、fixture 或 assertion。
  • Mutation / flaky gate：用 mutation score 與 flaky rate 做雙閘門，避免 AI 生成大量脆弱測試。

實際工作流程（step-by-step）

  1. 先定義風險面向，不用直接要求「幫我寫測試」，而是描述想防止的回歸型態，例如權限繞過、空值處理、日期邊界或排程重入。
  2. 針對該風險面向建立小量、具意義的 mutants；重點是模擬真實錯誤，不是大量隨機變異。
  3. 讓 LLM 根據程式碼、既有測試與 mutant 差異生成候選測試，並保留生成 rationale 供後續比對。
  4. 把 assertion 拆出去做 oracle review：要求第二個代理判斷 assert 捕捉的是預期規格還是既有實作，必要時提供規格、命名與註解做校準。
  5. 執行 pytest 或對應測試框架，收集 compile/runtime failure、stack trace、snapshot diff 與 mutation kill 結果。
  6. 若測試失敗，進入 test repair：只把失敗測試、錯誤訊息、相關函式與最小必要上下文交給模型，限制它只能修 fixture / assertion / test setup。
  7. 再跑 mutation testing。若 coverage 提高但 mutation score 沒提升，表示測試可能只是碰到程式碼而沒有真正驗證行為。
  8. 最後才進 CI：將 AI 生成測試標記為 generated 或 candidate，通過 mutation 與 flaky 門檻後再升級為正式回歸測試。

程式碼 / 配置範例


# .github/workflows/ai-test-repair.yml
name: ai-test-repair
on:
  pull_request:
    paths:
      - "tools/**"
      - "hooks/**"
      - "tests/**"

jobs:
  verify-generated-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements-dev.txt
      - run: pytest tests/tools -q --maxfail=1
      - run: mutmut run --paths-to-mutate tools
      - run: python tools/check_generated_tests.py --min-mutation-score 0.60 --max-flaky-rate 0.02


# tools/check_generated_tests.py
from __future__ import annotations

import json
import sys
from pathlib import Path

report = json.loads(Path('artifacts/generated-test-report.json').read_text(encoding='utf-8'))
mutation_score = report['mutation_score']
flaky_rate = report['flaky_rate']

if mutation_score < 0.60:
    raise SystemExit(f'mutation score too low: {mutation_score:.2%}')
if flaky_rate > 0.02:
    raise SystemExit(f'flaky rate too high: {flaky_rate:.2%}')

print('generated tests passed quality gates')


System prompt for oracle review agent
- 你只審查 assertion 是否代表預期規格，不負責重寫整支測試。
- 若 assertion 只是重述當前實作，標記為 implementation-derived oracle。
- 優先使用函式名稱、docstring、ADR、驗收條件與既有回歸案例判斷 expected behaviour。
- 只輸出：risk_level, rationale, suggested_assertion。

效果量化（生產力提升數據，若有）

  • Meta 的 ACH 在 7 個平台、10,795 個 Android Kotlin 類別上產生 9,095 個 mutants 與 571 個測試案例；工程師接受了 73% 的測試，其中 36% 被評為隱私相關，代表 mutation-guided 方式已能在真實產品團隊內被採納。
  • 同一研究中，等價 mutant 偵測代理的 precision / recall 為 0.79 / 0.47；經簡單前處理後可提升到 0.95 / 0.96，說明前置清洗與過濾對 LLM 測試流程非常重要。
  • CANDOR（2025/2026 版）顯示：在 Java 端對端單元測試生成上，coverage 可與 EvoSuite 相當，但 mutation score 更高；在 oracle correctness 上，至少比 TOGLL 高 21.1 個百分點，顯示多代理共識比單模型直接產生 assert 更穩。
  • 另一篇 2024 研究指出，LLM 生成的 oracle 仍常把「實際行為」誤當作「預期行為」；但若程式碼與測試命名清楚，其 fault detection potential 仍優於 EvoSuite。這代表命名品質與規格上下文是效果放大的前置條件。

與本專案的應用場景（daily-digest-prompt 如何受益）

  • 可把 tools/ 與 hooks/ 的 AI 生成測試從單純 coverage 補洞，升級為 mutation-guided hardening，優先保護高風險腳本如結果檔驗證、registry 更新與 shell guard。
  • results/todoist-auto-*.json 的 schema 驗證已存在，下一步可把「錯誤結果是否真的被測試捕捉」改用 mutant 驅動方式驗證，避免只測 happy path。
  • 對 PowerShell / Python 混合流程，可導入 test repair agent：當 CI 因編碼、路徑、時區或 Windows 特性造成測試失敗時，只修最小測試片段，減少整包重生造成的漂移。
  • 可把 AGENTS/Skill/workflow 規格轉為 oracle context，讓生成測試不是只對著程式碼猜行為，而是對著專案規範驗證行為。

風險與注意事項

  • 最大風險不是 coverage 不足，而是錯誤 oracle 把 bug 固化成正確行為。
  • 若沒有規格、註解、命名或 ADR，模型容易用現有實作反推 assert，導致「測到了錯的東西」。
  • test repair 必須限制修改範圍；否則模型會用大改 fixture 或跳過 assertion 的方式掩蓋問題。
  • mutation testing 不能只看總分，還要看 survived mutants 分布；若都集中在權限、時間或例外分支，就表示風險仍高。
  • flaky test 與 self-healing test 要分開治理：前者是環境/時序不穩，後者是模型修補策略；混在一起會讓根因模糊。
