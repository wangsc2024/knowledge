Harness Engineering Fitness Function 驅動的元件效能量化與自動調優策略

研究日期：2026-04-08 | 研究系列：harness-engineering（optimization 階段）
承接既有知識：約束驗證自動化（Policy-as-Code）、回饋迴路與補償面自適應、漸進式降級與韌性模式
本文聚焦：如何以 Fitness Function 量化 Harness 元件效能，並透過自動調優機制持續優化系統

---

一、Architecture Fitness Function 在 Harness 中的角色

1.1 從「治理靠檢查」到「治理靠規則」

Neal Ford 等人在《Building Evolutionary Architectures》中提出的 Fitness Function 概念，核心是將架構決策轉化為可執行、可自動驗證的規格。傳統架構治理依賴人工審查（code review、architecture board），但在 Agent 驅動的系統中，這種方式無法跟上變更速度。

Fitness Function 的定義：任何提供架構特性客觀完整性評估的機制。它將架構意圖從隱性知識轉化為顯性、可測試的程式碼，使「偏移」（drift）在發生的瞬間就被捕捉。

1.2 四維分類框架

| 維度 | 類型 A | 類型 B | Harness 對應 |

|------|--------|--------|-------------|

| 範圍 | 原子型（單一特性） | 整體型（多特性組合） | 原子：單一約束覆蓋率；整體：安全+效能+可維護性綜合 |

| 觸發 | 事件觸發 | 持續監測 | 事件：PR 合併時；持續：entropy_score 定時計算 |

| 結果 | 靜態（通過/失敗） | 動態（依上下文調整閾值） | 靜態：Hook 攔截；動態：依系統負載調整 timeout |

| 執行 | 自動化 | 人工 | 自動：CI/CD Pipeline；人工：月度 Harness Review |

1.3 LLM 作為 Fitness Function 的新維度（2026 趨勢）

2026 年的重要突破：部分架構特性難以用確定性規則表達，但 LLM 擅長這類評估。例如：

  • 程式碼可讀性評分
  • API 設計一致性判定
  • 錯誤訊息品質檢查

這催生了「計算型 + 推論型」雙軌 Fitness Function：確定性工具（linter、type checker）提供快速回饋，LLM 評估層（如本專案的 B2-C quality_gate validate_llm）處理語義級判定。

二、Harness 效能量化的五大核心指標

2.1 AI 放大倍率（AI Amplification Ratio）

定義：AI 等效工作時間 / 人類主動投入時間。

衡量 Harness 是否真正提升了生產力。高放大倍率表示 Harness 成功引導 Agent 產出有效工作，低放大倍率暗示約束過緊或工具配置不當。

2.2 Token 效率（Token Efficiency）

定義：每消耗 token 產出的有用輸出量。

Meta-Harness 論文的關鍵發現：優化後的 Harness 僅用 11.4K context token 就達到 48.6% 準確率，而基線方法用 50.8K token 只達 40.9%。4 倍 token 節省 + 7.7 個百分點提升，證明 Harness 品質對 token 經濟學的決定性影響。

2.3 品質門檻通過率（Quality Gate Pass Rate）

定義：Agent 輸出首次通過所有品質門檻的比例。

追蹤各層門檻的首次通過率：

  • 工具調用驗證（permission boundary violations）
  • 約束檢查（linter、type checker）
  • 測試套件通過率
  • LLM 語義驗證

2.4 錯誤與重試率（Error & Retry Rate）

定義：Agent 輸出被品質門檻拒絕的頻率。

高重試率不僅浪費 token，更暗示 Harness 的「前饋控制」（feedforward control）不足 — 應在 Agent 產出前就提供足夠的約束資訊，而非事後攔截。

2.5 約束覆蓋率（Constraint Coverage）

Martin Fowler 文章的核心洞見：「我們需要一種類似程式碼覆蓋率和突變測試之於測試的方式，來評估 Harness 覆蓋率和品質」。這是 2026 年 Harness Engineering 最重要的未解問題之一。

本專案的 B3-A constraint_coverage.py 正是朝此方向的實踐，計算約束規則對系統行為空間的覆蓋比例。

三、Meta-Harness：端對端自動調優的突破

3.1 核心思路

Meta-Harness（arXiv 2603.28052）提出了一個激進但有效的方法：讓 Agent 自動發現最佳 Harness 配置。

目標函式：找到使底層模型在目標任務分佈上表現最佳的 Harness 配置。

3.2 關鍵設計決策

  1. 完整診斷經驗而非摘要：提案者（proposer）可存取先前候選配置的原始碼、執行軌跡和評估分數（中位數每次迭代讀取 82 個檔案），而非僅讀壓縮摘要。消融實驗顯示：僅提供分數的條件達 41.3% 最佳準確率，完整軌跡存取達 56.7%。
  1. 因果假設先行：proposer 在提出變更前，先對失敗模式形成因果假設，再據此修改 Harness 配置。
  1. Token 預算極致優化：每次評估約 10M token context — 遠超傳統方法，但透過選擇性檢視（41% 原始碼 + 40% 執行軌跡）實現高效利用。

3.3 量化成果

| 任務 | Meta-Harness | 基線 | 提升 |

|------|-------------|------|------|

| 文本分類 | 48.6%（11.4K tokens） | 40.9%（50.8K tokens） | +7.7pt, 4× token 節省 |

| IMO 數學推理 | +4.7pt 平均（跨 5 個模型） | — | 單一 Harness 跨模型泛化 |

| TerminalBench-2 | 37.6% pass rate（#1 排名） | — | Haiku 級模型達頂尖表現 |

四、持續調優的六步迭代框架

4.1 agent-engineering.dev 的六步循環

  1. 施加強制函式（Forcing Functions）：設定「僅 Agent 可操作」的約束，迫使 Harness 必須完善
  2. 宣告式意圖定義：用規格而非命令式腳本描述期望行為
  3. 完整追蹤性的觀測工程：每個 Agent 動作、決策點都留下可查詢的追蹤記錄
  4. 永久性修正工程：每次修復轉化為可重用約束，而非一次性 patch
  5. 混合驗證（規則 + LLM）：確定性檢查處理已知模式，LLM 處理長尾情境
  6. 熵值持續監控：追蹤系統配置複雜度，防止 Harness 本身膨脹失控

4.2 Fitness Function 的實施三步法

根據 continuous-architecture.org 的框架：

Step 1 — 選擇：識別關鍵非功能需求。團隊建議限制在 3-5 個核心 Fitness Function，防止度量過載。

Step 2 — 定義：為每個 Function 設定基線（baseline）、目標（target）、失敗閾值（failure threshold），並指定負責人。

Step 3 — 排程：決定持續量測的可行性與頻率。建立每週審查看板，針對失敗項目制定行動計畫。

五、與本專案 Harness 框架（B0-B5）的映射與優化建議

5.1 現有元件的 Fitness Function 對應

| Harness 層 | 元件 | 建議 Fitness Function | 類型 |

|-----------|------|----------------------|------|

| B0 | backlog_writer | 待辦項目清理率（7 天內從 proposed→accepted 的比例） | 原子/事件觸發 |

| B1 | constraint_evolve | 約束演化速率（每週新嵌入約束數量） | 原子/持續 |

| B2 | quality_gate | 品質門檻首次通過率 | 原子/事件觸發 |

| B2-C | validate_llm | LLM 評估與人工判定的一致率 | 整體/事件觸發 |

| B3-A | constraint_coverage | 約束覆蓋率百分比 | 原子/持續 |

| B3-B | entropy_score | 配置熵分數（低於閾值=健康） | 動態/持續 |

| B4-A | harness_gap_tracker | Gap 解決率（發現→嵌入的轉化比） | 原子/持續 |

| B5 | compensation_surface | 人工介入頻率趨勢（應遞減） | 整體/持續 |

5.2 三項具體優化建議

建議 1：引入 Harness Coverage Score（HCS）

類比程式碼覆蓋率，定義 HCS = 已約束行為空間 / 總行為空間。目前 B3-A constraint_coverage 已有基礎，可擴展為包含 Hook 規則覆蓋、品質門檻覆蓋、錯誤恢復覆蓋三個子維度。

建議 2：採用 Meta-Harness 思路的配置自動調優

利用現有的執行軌跡（logs/structured/ 的 JSONL）作為診斷經驗，讓 LLM 分析失敗模式並提出 Harness 配置調整建議。這與 B4-A Gap Loop 天然互補 — Gap Loop 發現問題，Meta-Harness 思路提出系統性修正。

建議 3：建立 Fitness Function Dashboard

將上述 8 個 Fitness Function 整合到 check-health.ps1 的健康度報告中，每日自動計算並追蹤趨勢。設定 3 個月目標：品質門檻首次通過率 ≥ 80%、約束覆蓋率 ≥ 70%、熵分數 ≤ 配置閾值。

六、關鍵洞見與實踐啟示

6.1 LangChain 案例的啟示

LangChain 團隊僅改 Harness 不換模型，就將 TerminalBench-2.0 表現從 52.8% 提升到 66.5%（+13.7pt）。這證明：Harness 品質是 Agent 效能的最大槓桿點，投資於 Fitness Function 和自動調優的 ROI 遠高於模型升級。

6.2 OpenAI Codex 團隊的度量實踐

OpenAI 內部追蹤「每位工程師每日合併 PR 數」（3.5 PRs/engineer/day）作為 Harness 效能的代理指標。當團隊從 3 人擴展到 7 人時，此指標不降反升 — 說明良好的 Harness 設計具有正向規模效應。

6.3 Martin Fowler 的未解問題

「如何評估 Harness 覆蓋率和品質」被 Martin Fowler 文章明確列為 2026 年最重要的開放性挑戰。Meta-Harness 論文提供了一個方向（端對端自動調優），但完整的「Harness 覆蓋率度量學」仍待建立。本專案的 B3-A/B3-B 元件是這個方向的早期實踐。
