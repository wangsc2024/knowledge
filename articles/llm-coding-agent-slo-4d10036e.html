﻿# 排程驅動下 LLM／Coding Agent 可觀測性：SLO 基線實測、流量形狀與誤報矩陣校準：深度技術研究報告

執行摘要（200-400 字）

本研究聚焦排程驅動的 LLM／Coding Agent 可觀測性。當工作流以固定節律觸發時，單看成功率或即時錯誤率，容易把規律尖峰誤判為異常。綜合 Google SRE、Grafana、OpenTelemetry 與 Langfuse，可得三點結論：SLO 基線須同時納入成功率、尾部延遲、工具失敗率與 token／成本密度；告警應採多視窗、多燃燒率區分短暫尖峰與持續退化；quality gate、hook 與 backend 降級事件必須串成可回放因果鏈，才能支援可審計的自動決策。

研究背景與動機

多數 LLM observability 設計預設流量較為連續，或將尖峰視為偶發事件；但 daily-digest-prompt 這類專案有明顯不同的行為輪廓：排程在整點、半整點或高失敗時段集中觸發，Phase 1、Phase 2、Phase 3 又各有 timeout、retry 與 fallback 邏輯，因此系統的「正常樣貌」本身就帶有週期性脈衝。若仍以單一平均值看待，容易讓告警策略在兩種情境都失真：一是短暫排程尖峰造成假陽性，二是真正的持續退化被週期波動掩蓋。

本研究的核心動機因此有三個。第一，為排程驅動的 Agent 管線定義可操作而非純展示型的 SLI／SLO；第二，釐清多視窗錯誤預算燃燒率在固定節律負載下為何容易產生結構性誤報，並提出可用混淆矩陣校準的方法；第三，評估現有 JSONL、state 檔與 harness quality gate 事件能否整合成可回放、可審計、可用於自動決策的證據鏈。

核心技術解析（每個主張內嵌引用）

一、SLO 基線必須同時觀測結果、過程與成本，而不只是最終成功率。 Google SRE 的方法強調 SLO 與錯誤預算應服務於決策，而非僅做報表，因此若只記錄 task success／failure，便無法回答「失敗是由模型延遲、工具錯誤、還是重試鏈膨脹造成」這類問題 [來源：Google SRE Workbook - Alerting on SLOs, n.d.]。OpenTelemetry 的 GenAI metrics 已把 operation duration、token usage、request count 等訊號納入語意層，Agent spans 又能把 create_agent、invoke_agent、execute_tool 等行為標準化，代表排程型 Agent 系統其實已有成熟欄位可借用 [來源：OpenTelemetry - Semantic conventions for Generative AI metrics, n.d.] [來源：OpenTelemetry - Semantic Conventions for GenAI agent and framework spans, n.d.]。Langfuse 的實務也指出，若沒有 traces、成本、session 與 quality 評分切片，就很難做根因分析與回放 [來源：Langfuse - LLM Observability, n.d.]。這與本 repo 現況相吻合：config/benchmark.yaml 已追蹤 daily_success_rate 與 avg_phase1_duration，config/autonomous-harness.yaml 也將 quality_gate_llm、hook_enforcement 與 degradation_chain 納入 intervention sources，說明基礎訊號其實已在，只是尚未被整合成單一 SLO 視圖。

二、固定節律排程下，單一視窗或單一 burn-rate 告警會系統性誤判。 Google SRE Workbook 明確指出單一燃燒率規則在偵測速度、精準度與重置時間之間難以同時兼顧，因此建議多視窗、多燃燒率組合 [來源：Google SRE Workbook - Alerting on SLOs, n.d.]。Grafana Cloud 的 SLO 文件同樣以快慢兩類燃燒規則搭配短長視窗降低噪音 [來源：Grafana Cloud Documentation - Create SLOs, n.d.]。而 Windows Task Scheduler 的 repetition pattern 則提醒我們，這些尖峰常常不是隨機雜訊，而是排程設計本身的流量形狀 [來源：Microsoft Learn - Trigger.Repetition property, n.d.]。也就是說，如果系統在 09:00、14:00、15:00 固定較易失敗，就不該只把它視為服務不穩，而應把時段視為模型的一部分。這點與 config/timeouts.yaml 中對高失敗時段加 30% timeout buffer 並啟用前置健康檢查的做法互相呼應，顯示本 repo 已開始用時段特徵調整執行策略，但尚未把這些時段訊號正式回寫進 SLO 門檻層。

三、真正可用的 Agent observability 必須能回放「因果鏈」，而不是只保留鬆散日誌。 OpenTelemetry 對 Agent spans 的設計價值，不只是統一命名，更在於它提供跨 agent、tool、error 的可串接事件序列 [來源：OpenTelemetry - Semantic Conventions for GenAI agent and framework spans, n.d.]。若 quality_gate_llm 判定未通過、degradation_chain 觸發備援、或 hook 攔截了某次危險操作，系統應能回答「是哪個任務、哪個 phase、哪個 backend、哪個 tool span 造成這個結果」。Langfuse 對 session、trace、evaluation 的綁定，也正是為了支援這種從結果回到過程的反查 [來源：Langfuse - LLM Observability, n.d.]。本 repo 在 ADR 已指出目前缺乏跨 Phase 因果關聯，因此決定以 OpenTelemetry 語義欄位命名自建輕量級四層可觀測性框架，目標就是讓 Error Budget 能取代主觀判斷 [來源：context/adr-registry.json, 2026]。因此，下一步最有價值的工作不是再新增獨立監控面板，而是把現有 scheduler-state、structured JSONL、quality gate 狀態與 backend 降級事件，統一映射到同一 trace/run_id 模型。

四、告警校準需要引入離線回放與混淆矩陣，而不只靠 dashboard 直覺。 在排程驅動系統中，某些尖峰雖然看似錯誤率升高，但若它們未造成錯誤預算實質快速燃燒、也未導致使用者面或任務面失敗，就不應被視為高優先告警。反之，某些尾部延遲與重試放大，可能在平均成功率仍高時就已損傷系統穩定性。這表示門檻設計應該以歷史回放建立「哪些告警是正確、哪些是假陽性、哪些漏報最昂貴」的矩陣，再用此矩陣調整短窗／長窗 burn-rate 與 phase timeout 門檻 [來源：Google SRE Workbook - Alerting on SLOs, n.d.] [來源：Grafana Cloud Documentation - Create SLOs, n.d.] [來源：Microsoft Learn - Trigger.Repetition property, n.d.]。對本專案而言，state/failed-auto-tasks.json、state/quality-trend.json 與 logs/structured/post_tool_logger.jsonl 已經構成誤報分析的最小資料面，只差把分類標準與回放程序系統化。

關鍵洞見（含三角佐證狀態 ✅/⚠️）

  1. SLO 基線至少應包含成功率、尾部延遲、工具失敗率、token／成本密度四類一級指標（✅，4 來源支持）。Google SRE、兩份 OpenTelemetry 規範與 Langfuse 均指出，若缺少過程與成本訊號，SLO 只能描述結果，不能驅動修復決策。
  2. 排程尖峰是流量形狀，不是單純噪音；因此告警應對應時間結構，而非只對應全域平均（✅，4 來源支持）。Google SRE、Grafana、Windows Task Scheduler 文件與中文 SLO 整理均支持以多視窗、多燃燒率處理固定節律負載。
  3. quality_gate_llm、hook 攔截、backend 降級若未共享同一 trace/run_id，便無法形成完整 SLO 證據鏈（✅，3 來源支持）。OpenTelemetry 與 Langfuse 都支持以事件鏈整合決策訊號；本 repo 的 harness 設定也已顯示這些訊號分散存在。
  4. 離線回放與混淆矩陣應成為門檻調校的正式流程，而非事後檢討補充品（✅，3 來源支持）。固定節律負載下，誤報與漏報成本可被量化，因此可直接回寫告警規則。
  5. 品質分數、prompt 版本與 session 切片很可能是下一階段必要訊號，但跨工具標準化仍不足（⚠️，2 來源支持）。這一點在 Langfuse 與 OTel 生態之間已有方向，但尚未形成穩定跨平台共識。

實務應用建議（3-5 點）

  1. 以 run_id 或 trace_id 為主鍵，將 scheduler-state、quality_gate_llm、degradation_chain、hook 事件與 phase 結果寫入同一可回放事件鏈。
  2. 在 config/benchmark.yaml 的既有成功率與 phase duration 之外，新增工具失敗率與 token／成本密度兩類指標，避免高成功率掩蓋高成本或高重試膨脹。
  3. 把 config/timeouts.yaml 的高失敗時段設定進一步升級為 SLO 規則輸入，讓 09:00、14:00、15:00 這類時段能有差異化門檻，而不是只調 timeout buffer。
  4. 建立小型離線回放程序：重播最近 3 天 structured JSONL 與 state 檔，為每條告警打上 TP／FP／FN 標記，再依錯誤預算消耗與人工負荷調整 MWMB 規則。
  5. 將 OTel GenAI spans／metrics 作為命名規範，而非強制全面導入外部基礎設施；先做欄位對齊，再決定是否外送至 OTLP collector。

與現有知識的連結

本研究與現有知識庫中的 observability 路線高度一致。ADR 已明確指出系統目前存在快取命中率偏低、output_len 長期為 0、缺乏跨 Phase 因果關聯與 SLO 量化基準等問題，因此才決定以 OpenTelemetry 語義欄位建立輕量級四層可觀測性框架。這說明本研究不是另起爐灶，而是在既有方向上向前推進：把理論上的 SLO／burn-rate 方法，對接到本 repo 的實際 scheduler、timeout、quality gate 與 state 資料面。換言之，研究結果可直接作為後續 observability-roadmap、phase retry 調整與 backend 降級策略的設計依據。

未來展望

後續可沿三個方向深化。第一，補做近三至七天的離線回放，將不同時段與不同 task type 的告警結果整理為混淆矩陣，實測哪組 MWMB 門檻最能平衡誤報與漏報。第二，將 quality_gate_llm、hook blocked action 與 backend fallback 對齊到共同的 trace schema，驗證是否真能縮短根因分析時間。第三，持續觀察 OpenTelemetry GenAI 語意與實務工具之間的收斂速度，再決定是否把品質分數與 prompt 版本納入正式 SLO，而非只作輔助觀測欄位。
