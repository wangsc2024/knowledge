一、現況診斷：三大關鍵指標異常

1.1 Hook 攔截率暴增（Critical）

| 指標 | 前值 | 現值 | 變化 |

| block_rate | 0.06% | 6.35% | +100 倍 |

| blocked_count | ~10 | 1,085 | 暴增 |

根因假設：自動任務從 19→27 個（+42%），每個任務約觸發 40-60 次 Bash 呼叫，總呼叫量暴增。pre_bash_guard.py 和 pre_write_guard.py 規則在高頻環境下誤攔截率上升。

二、優化方案（含外部最佳實踐）

方案 A：Hook 攔截率調優（P0）

業界最佳實踐：

  • Splunk 2026 AI Agent Observability 提出「Guard Behavior Dashboard」概念，監控每個 guard 的 run duration 和 failure rate
  • 現代 Agent 系統建議實裝「分級防護」：Level 1（警告）→ Level 2（審計日誌）→ Level 3（攔截）
  • Galileo Signals 引擎自動掃描 production traces 識別 drift 原因

專案具體建議：

  1. 規則分級化：將 config/hook-rules.yaml 的規則從二元（block/allow）改為三級（warn/audit/block）

- warn：僅記錄日誌，不阻斷（適用於低風險操作如 rm temp files）

- audit：記錄日誌 + ntfy 告警，不阻斷（適用於中風險操作）

- block：攔截 + 日誌 + ntfy（僅用於高風險操作如 nul 寫入、敏感路徑）

  1. 白名單機制：為自動任務建立 trusted_patterns 白名單

- todoist-auto-* 產生的 curl/rm/cat 命令可跳過部分檢查

- 白名單需版本控制，變更需 PR review

  1. Guard Behavior Dashboard：在 check-health.ps1 新增 [Hook 攔截分析] 區塊

- 按規則名稱分類統計 block/warn 次數

- 標記 top-5 被攔截的命令模式

- 自動識別誤攔截候選（同一 pattern 被攔截 >10 次/天）

預期效果：block_rate 從 6.35% 降至 <2%（透過將 60% 的攔截降級為 warn）

方案 B：快取智能預熱（P1）

業界最佳實踐（綜合 OneUptime、SparkCo 2026 研究）：

  1. 三層預熱策略：

- 預測性預熱：分析歷史存取日誌，在流量低谷期（如凌晨排程）預載 top-N 熱點

- 事件驅動預熱：資料變化時立即觸發相關快取更新（如新任務匯入→預熱 todoist 快取）

- 業務邏輯預熱：Phase 0 啟動時，根據當前排程時段預載對應 API 快取

  1. 自適應 TTL：

- L1 熱數據：短 TTL（15 分鐘），高頻存取項目

- L2 溫數據：中 TTL（60 分鐘），一般查詢

- L3 冷數據：長 TTL（4 小時），靜態配置

- 根據命中率動態調整：命中率 <30% → TTL ×1.5；命中率 >70% → TTL ×0.8

  1. 語義快取：對知識庫查詢結果實裝向量相似度快取

- 相似度 >0.9 的查詢直接返回快取結果

- 可減少 KB API 呼叫 30-50%

專案具體建議：

  • 在 run-agent-team.ps1 Phase 0 增加 Warm-Cache 函式
  • 讀取 frequency-limits.yaml 取得當前時段的自動任務列表
  • 預載對應的 KB 搜尋快取和 Todoist API 快取
  • cache/status.json 新增 warm_keys 和 warm_at 欄位

預期效果：cache_hit_ratio 從 21% 提升至 40-50%

方案 C：Context Window 預算控制（P1）

業界最佳實踐（綜合 16x Engineer、Agenta、OpenAI Community 2026 研究）：

  1. Token 預算分配策略：

- System Prompt：≤15% 總預算

- User Context（歷史對話+文件）：≤50%

- Working Space（推理空間）：≥35%

- 設置 70% 閾值觸發自動摘要壓縮

  1. 動態摘要觸發（Context Compaction）：

- 當 token 使用達 70% 預算 → 壓縮最舊的 30% 歷史為摘要

- LLMLingua 式 token-level pruning：基於 perplexity 分數裁剪低資訊密度 token

- 保留「錨點」資訊（任務目標、關鍵決策、錯誤記錄）

  1. 多層快取策略：

- 精確匹配快取：亞毫秒級延遲（相同查詢）

- 語義快取：略高延遲（相似查詢），可減少 API 呼叫 73%

- Session 上下文管理：維護對話狀態，避免重複載入

  1. 模型分層選擇：

- 簡單分類/提取任務 → 輕量模型（成本低 15-50 倍）

- 複雜推理/關鍵決策 → 旗艦模型

專案具體建議：

  • SKILL_INDEX.md 按需載入：Phase 2 子 Agent 僅載入 matched Skill，不載入完整索引
  • Prompt 模板壓縮：將 100+ 行模板拆為「核心指令」（30 行）+「參考附件」（按需載入）
  • 限制 Phase 2 並行度：從 N 個並行改為 max(2, N) 批次序列執行
  • context-budget-monitor Skill 增加即時告警：超過 70% 預算時 ntfy 通知

預期效果：avg_io_per_call 從 10,673 降至 <6,000

三、額外發現：自動任務公平性與測試覆蓋

3.1 自動任務公平性

  • 4 個暫停任務（unsloth/creative_game/qa/chatroom，daily_limit=0）造成 fairness=0.555
  • 建議：明確廢棄或啟用，減少混淆；改進 round-robin 跳過禁用任務

3.2 測試覆蓋率缺口

  • tools/ 下 300+ 測試不計入覆蓋率（pyproject.toml source 僅含 hooks/skills）
  • hook_pipeline.py（新增）完全未測試
  • 建議：擴大 source 至 tools，新增 hook_pipeline 測試，目標覆蓋率 60%

3.3 日誌積累

  • logs/ 累積 22MB，state/ 殘留 20+ 個 loop-state-*.json
  • on_stop_alert.py 已有 _cleanup_stale_state_files() 但覆蓋範圍不足
  • 建議：擴展清理範圍至 logs/structured/，增加 logs 7 天 TTL 清理
