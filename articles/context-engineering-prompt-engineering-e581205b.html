Context Engineering — 從 Prompt Engineering 到上下文工程的範式轉移

概述

Context Engineering（上下文工程）是 2026 年 AI 領域最重要的範式轉移之一。由 Andrej Karpathy 推動的這個概念，將焦點從「如何措辭指令」（Prompt Engineering）轉移到「如何策展、結構化並防禦送入 LLM 推論時的所有資訊」。

Karpathy 的核心類比：LLM 是新型作業系統，Context Window 是 RAM（工作記憶體）。正如作業系統需要精密的記憶體管理，AI 應用需要精密的上下文管理。

「Context engineering is the delicate art and science of filling the context window with just the right information for the next step.」— Andrej Karpathy

---

一、Prompt Engineering vs Context Engineering

| 維度 | Prompt Engineering（2023-2024） | Context Engineering（2025-2026） |

|------|------|------|

| 焦點 | 單一指令的措辭優化 | 整體資訊架構的策展與管理 |

| 範圍 | 單次互動 | 跨步驟、跨 session 的資訊流 |

| 挑戰 | 讓模型理解意圖 | 資訊紀律（information discipline） |

| 運作層級 | 指令層 | 架構層（記憶、檢索、工具、壓縮） |

| 核心認知 | 好的 prompt 帶來好結果 | 可靠性來自上下文系統，而非 prompt 巧妙度 |

關鍵轉折：2026 年的挑戰不再是模型能力不足（現代模型已能處理百萬級 token），而是資訊紀律——如何確保每個 token 都有明確用途。

---

二、Agent Context 八大元素

根據 LlamaIndex 與 LangChain 的架構分析，AI Agent 的上下文由八個元素組成：

  1. 系統提示與指令（System Prompt）：策略、語氣、角色定義
  2. 使用者輸入（User Input）：即時請求
  3. 短期記憶 / 對話歷史（Short-term Memory）：當前 session 的上下文
  4. 長期記憶（Long-term Memory）：跨 session 的持久化知識
  5. 知識庫檢索資訊（Retrieved Knowledge）：RAG、向量搜尋、API 呼叫結果
  6. 工具定義與描述（Tool Definitions）：可用工具的 schema
  7. 工具回應（Tool Responses）：工具執行的輸出結果
  8. 結構化輸出與全域狀態（Global State）：跨步驟共享的狀態

---

三、四大核心操作模式（WSCI 框架）

LangChain 提出的四大上下文操作模式，構成 Context Engineering 的實務骨架：

1. Write Context（寫入持久化）

將資訊保存到上下文窗口之外，供後續檢索：

  • Scratchpad（筆記板）：Agent 在複雜推理過程中記錄計畫與發現
  • 長期記憶：Reflexion 與 Generative Agents 先驅了跨 session 的自我反思機制
  • 實務模式：Claude Code 的 CLAUDE.md、Cursor 的 .cursorrules

2. Select Context（選擇檢索）

策略性地拉取相關資訊：

  • 記憶選擇：透過 embedding 或知識圖譜檢索任務相關的事實與程序
  • 工具選擇：對工具描述進行語義搜尋，研究顯示可提升準確率 3 倍（44% 改善）
  • 知識檢索：RAG 仍是核心；生產級 Code Agent 結合 AST 解析、embedding 搜尋、grep 與 re-ranking

3. Compress Context（壓縮精簡）

僅保留關鍵 token：

  • 摘要化：遞迴或層級式摘要，在 Agent 軌跡端點或工具呼叫後執行
  • 裁剪：硬編碼移除舊訊息或使用訓練過的修剪器
  • 關鍵數據：聚焦 300 token 的上下文往往優於散漫的 113,000 token

4. Isolate Context（隔離分治）

跨邊界分割資訊：

  • 多 Agent 分解：子 Agent 各自專責，減少個別上下文負載
  • 沙盒環境：Code Agent 在隔離空間執行，保持 token 密集物件（圖片、音訊）在 LLM 上下文之外
  • Anthropic 研究：多 Agent 方法雖然 token 成本更高，但效能優於單一 Agent

---

四、常見失敗模式

| 失敗模式 | 描述 | 防禦策略 |

|---------|------|----------|

| Context Poisoning（上下文毒化） | 幻覺內容進入上下文並持續影響後續推理 | 來源驗證、事實查核閘門 |

| Context Distraction（上下文干擾） | 過量資訊淹沒關鍵資訊 | 相關性過濾、token 預算管理 |

| Context Confusion（上下文混淆） | 多餘內容影響模型回應方向 | 結構化資訊層級、明確分隔 |

| Context Clash（上下文衝突） | 矛盾資訊同時存在於上下文中 | 優先級規則、衝突偵測 |

| Context Rot（上下文腐朽） | 長對話中品質隨時間退化 | 定期壓縮、摘要替換 |

| Mode Collapse（模式崩塌） | 對齊訓練導致輸出多樣性降低 | 溫度調整、多樣化提示 |

五、2026 年產業實踐案例

Claude Code（Anthropic）

  • 對話達 95% 上下文閾值時自動壓縮（Context Compaction）
  • Opus 4.6 支援 Adaptive Thinking：動態決定推理深度
  • Interleaved Thinking：在工具呼叫之間進行推理
  • CLAUDE.md 作為程序性記憶的持久化載體

LangGraph（LangChain）

  • Checkpointing 用於短期狀態持久化
  • 長期記憶集合支援 embedding 檢索
  • 節點級精細控制：精確管理每個 LLM 步驟接收的上下文

LlamaIndex

  • VectorMemoryBlock / FactExtractionMemoryBlock / StaticMemoryBlock 三層記憶架構
  • 結構化輸出提供壓縮的、schema 驗證的上下文

生產級 Code Agent（Cursor, Windsurf, Claude Code）

  • 規則檔案（.md）作為持久化的指令上下文
  • 動態工具載入（Tool Loadout）減少工具描述的 token 佔用

---

六、實務建議

  1. 從簡單開始：靜態 prompt + 固定工具集，僅在需要時加入動態機制
  2. 逐步測試：每次只添加一個 context engineering 特性
  3. 監控指標：追蹤模型呼叫次數、token 使用量、延遲
  4. 相關性優先：每個 token 都應服務明確目的
  5. 來源可追溯：所有注入的資訊應可追溯到已驗證來源
  6. 壓縮勝於完整：精簡表述優於詳盡但冗長的輸入
  7. 層級控制：系統層（策略/語氣）→ 任務層（邏輯/目標）→ 使用者層（即時請求）
