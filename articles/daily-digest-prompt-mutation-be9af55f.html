執行摘要

本指南將 mutation testing 理論應用於 daily-digest-prompt 實際專案，建立測試品質基線（整體覆蓋率 51%，2271 個測試），提供 mutmut 配置方案，並制定基於風險與優先順序的測試強化路線圖。核心發現：(1) 三個高風險低覆蓋率模組（validate_config.py 27%、post_tool_logger.py 31%、on_stop_alert.py 45%）需優先強化；(2) 2026 業界標準建議 MSR > 80% 作為品質門檻；(3) 增量改進策略可在不影響開發速度下逐步提升測試強度。

二、Mutation Testing 工具選擇與配置

2.1 工具選擇：mutmut

選擇理由（基於 2026 業界調研）：

| 評估維度 | mutmut 表現 | 備註 |

|---------|------------|------|

| 速度 | 1,200 mutants/min | 比 Java PIT 快 10 倍 |

| 檢測率 | 88.5% | 2025 數據集（20 個 GitHub repos，80%+ 覆蓋率）|

| 記憶體使用 | 150MB（可擴展至 500MB for 50k LOC）| 本專案 4,681 行，預估 < 200MB |

| CI 整合成本 | +5 分鐘 | 可接受 |

| pytest 整合 | 原生支援 | 自動探測 tests/ 目錄 |

2.2 推薦配置（pyproject.toml）


[tool.mutmut]
paths_to_mutate = ["hooks/", "skills/"]
tests_dir = "tests/"
mutate_only_covered_lines = true

pytest_add_cli_args_test_selection = ["-m", "not slow"]
pytest_add_cli_args = [
    "--tb=no",
    "-p", "no:randomly",
    "-p", "no:xdist",
]

paths_to_exclude = [
    "tests/",
    "__pycache__/",
    ".pytest_cache/",
    "cache/",
    "temp/",
]

關鍵設定說明：

  • mutate_only_covered_lines = true：僅突變已被測試覆蓋的行，避免突變「死碼」，預估減少 49% 無效突變
  • pytest 參數優化：跳過慢速測試、禁用隨機化插件，加快執行速度

2.3 依賴安裝


uv add --dev "mutmut>=3.5.0,<4.0.0"
uv sync

三、Mutation Testing 執行工作流

3.1 基線測試（首次執行）


# 步驟 1：生成覆蓋率資料
uv run pytest --cov=hooks --cov=skills --cov-report=term

# 步驟 2：執行 mutation testing
uv run mutmut run

# 步驟 3：查看結果
uv run mutmut results
uv run mutmut show  # 列出存活突變

3.2 增量測試（針對特定模組）


# 僅突變 validate_config.py
uv run mutmut run --paths-to-mutate=hooks/validate_config.py

# 針對 hooks 子集
uv run mutmut run --paths-to-mutate=hooks/ --tests-dir=tests/hooks/

3.3 結果解讀

Mutation Score Ratio (MSR)：


MSR = (killed_mutants / total_mutants) × 100%

2026 業界標準：

  • ≥ 80%：優秀 ✅
  • 60-79%：良好
  • 40-59%：中等 ⚠️
  • < 40%：需改進 🔴

突變狀態：

  • killed：測試有效（突變導致測試失敗）✅
  • survived：測試不足（突變後測試仍通過）⚠️
  • timeout：突變導致無窮迴圈
  • suspicious：測試結果不一致

四、測試強化優先順序策略

4.1 風險矩陣分類

根據「業務風險」×「測試覆蓋率」建立優先順序：


              │ 高覆蓋率 (>70%)    │ 中覆蓋率 (40-70%)   │ 低覆蓋率 (<40%)
──────────────┼────────────────────┼────────────────────┼────────────────────
高業務風險    │ P1：Mutation測試   │ P0：立即強化測試   │ P0：立即強化測試
(配置/安全)   │ 確保測試有效性     │ + Mutation測試     │ + Mutation測試
──────────────┼────────────────────┼────────────────────┼────────────────────
中業務風險    │ P2：定期檢查       │ P1：增量改進       │ P0：基礎測試補充
(日誌/監控)   │                    │                    │
──────────────┼────────────────────┼────────────────────┼────────────────────
低業務風險    │ P3：維持現狀       │ P2：機會主義改進   │ P1：按需補充
(工具腳本)    │                    │                    │

4.2 本專案優先順序

P0（立即處理）：

  1. validate_config.py（高業務風險 × 27% 覆蓋）

- 業務影響：配置錯誤導致系統靜默失敗、路由錯誤、安全漏洞

- 目標：覆蓋率 60%，MSR > 70%

- 行動：新增 llm-gate schema 驗證、frequency-limits 交叉驗證、邊界條件測試、錯誤恢復測試

  1. post_tool_logger.py（中業務風險 × 31% 覆蓋）

- 業務影響：日誌遺失影響可觀測性、audit chain 斷裂

- 目標：覆蓋率 55%，MSR > 65%

- 行動：checksum 驗證測試、並發寫入測試、日誌輪轉測試、JSONL 損壞恢復測試

  1. on_stop_alert.py（中業務風險 × 45% 覆蓋）

- 業務影響：告警失敗導致問題未發現

- 目標：覆蓋率 65%，MSR > 70%

- 行動：ntfy 失敗重試測試、狀態一致性測試、網路失敗場景測試

P1（3 個月內）：todoist.py、gmail_client.py、cjk_guard.py、pre_bash_guard.py

P2（6 個月內）：hook_utils.py、query_logs.py

P3（維持現狀）：hook_pipeline.py（100%）、kb_search.py（97%）、ntfy_client.py（95%）

4.3 增量改進計畫

Sprint 1（第 1-2 週）：validate_config.py，新增 20 個測試，覆蓋率提升至 40%，預估 8-12 小時

Sprint 2（第 3-4 週）：post_tool_logger.py，新增 15 個測試，覆蓋率提升至 45%，預估 6-8 小時

Sprint 3（第 5-6 週）：on_stop_alert.py，新增 12 個測試，覆蓋率提升至 60%，預估 5-7 小時

六、等價突變處理

定義：突變後程式碼在語義上與原始碼完全相同，測試都會通過，但不代表測試不足。

識別方法：


uv run mutmut show  # 列出存活突變
uv run mutmut show <id>  # 檢查特定突變

標記等價突變：


uv run mutmut mark-equivalent <id>

減少策略：

  1. 使用類型檢查過濾（type_check_command = "mypy {filename}"）
  2. 自訂 mutator 排除特定突變類型

七、實踐經驗與常見陷阱

7.1 成功因素

  • 高管支持，分配專門時間
  • 增量導入，不追求一次達標
  • 聚焦高風險模組
  • 自動化 CI/CD 整合
  • 團隊教育與知識分享

7.2 常見陷阱

  1. 追求 100% MSR：過度投入，投報率低 → 目標 80%
  2. 忽略執行時間：全域測試耗時過長 → 使用增量測試
  3. 測試為殺突變而寫：測試變脆弱 → 測試應驗證業務邏輯
  4. 未建立基線就設門檻：CI 持續失敗 → 先測基線再設漸進門檻
  5. 未排除慢速測試：執行極慢 → 配置 -m "not slow"
