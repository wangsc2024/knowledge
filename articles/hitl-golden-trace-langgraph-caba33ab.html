HITL 工具核准理由碼與 Golden Trace 重放測試：LangGraph interrupt／MCP 邊界與 JSONL 追溯契約：深度技術研究報告

執行摘要（200-400 字）

本報告接續前期三角佐證結論，聚焦人機協同（HITL）在高風險工具呼叫場景下的「可重放核准契約」。綜合 LangGraph 的 interrupt／persistence、OpenAI Agents SDK 的 interruptions／RunState、Google ADK 的 resumable runtime，以及 MCP 規範與安全最佳實務，共識是可恢復 HITL 必須同時具備穩定執行身分（thread／run／世代）、可序列化狀態與明確工具邊界；MCP Host 承載同意與能力治理，圖編排層承載 pause／resume 控制流 [來源：Model Context Protocol Specification (2025-06-18), 2025][來源：LangGraph: Human-in-the-loop, 2025][來源：LangGraph Interrupts — Python, 2025]。在此基礎上，核准「理由碼」應為可機讀枚舉並落入 JSONL／error-object 單一觀測平面，Golden Trace 則需把工具輸入正規化雜湊、模型／prompt 指紋與核准決策綁定到 tool_call_id，才能驗證相同輸入與相同核准下追蹤欄位一致；細部雜湊契約仍有賴專案實作補齊，屬待驗證延伸 [來源：Human-in-the-loop — OpenAI Agents SDK, 2025][來源：Resume stopped agents — Google ADK, 2025]。

研究背景與動機

多代理與長回合編排普及後，工具呼叫從「一次性函數」變成具備安全與合規含義的作業。模型可能遭 prompt injection 誘發過度代理；若無 gate，高影響力工具會在缺乏稽核軌跡下被觸發 [來源：MCP Security Best Practices, 2025][來源：OWASP Top 10 for Large Language Model Applications, 2024]。研究問題因而分成三條：checkpoint／interrupt 下如何綁定唯一識別與版本；MCP Host 或同構核准層的理由碼如何支援稽核與 policy；Golden Trace 如何驗證「相同輸入＋核准決策」時雜湊、模型指紋與 JSONL 欄位一致。

核心技術解析（每個主張內嵌引用）

可持久化 pause／resume 是 HITL 的工程前提。 LangGraph 以 interrupt() 暫停圖執行並透過 Command(resume=...) 恢復；若無 checkpointer 與 thread_id 等穩定關聯，核准後重入會失去一致語意 [來源：LangGraph Interrupts — Python, 2025][來源：LangGraph Persistence — Python, 2025]。OpenAI Agents SDK 以 needs_approval、RunResult.interruptions 與可序列化 RunState 呈現同一問題的不同抽象 [來源：Human-in-the-loop — OpenAI Agents SDK, 2025]。Google ADK 則從 runtime 角度區分暫停與可恢復執行，利於與決定性工作流對照 [來源：Resume stopped agents — Google ADK, 2025]。

工具邊界與同意掛點落在 MCP Host。 MCP 規範將工具能力曝露在 server，治理與同意流程由 Host 承擔，使最小權限與 policy hook 有固定位置 [來源：Model Context Protocol Specification (2025-06-18), 2025]。安全最佳實務進一步將 prompt injection、tool poisoning、confused deputy 列為核心威脅，要求 consent 與 scope minimization [來源：MCP Security Best Practices, 2025]。OWASP LLM Top 10 亦將過度代理與不當工具使用視為關鍵風險類型，並指出技術控制需搭配審計 [來源：OWASP Top 10 for Large Language Model Applications, 2024]。LangGraph 的 human-in-the-loop 概念文件亦強調節點邊界可暫停等待人類輸入，且需與 checkpoint 語意銜接 [來源：LangGraph: Human-in-the-loop, 2025]。

Golden Trace 與理由碼屬實作契約層。 框架文件多處理 pause／resume API；若要對「同一決策」做重放測試，需把 thread_id、checkpoint 世代、tool_call_id（或等價鍵）與核准理由碼一併寫入結構化日誌，並與 error-object 的 type、retryable 等欄位對齊，才能避免觀測平面分裂。工具輸入正規化雜湊與模型指紋的必備性獲兩方以上來源間接支持，但欄位級標準仍待各專案補強 [來源：LangGraph Interrupts — Python, 2025][來源：Human-in-the-loop — OpenAI Agents SDK, 2025]。

關鍵洞見（含三角佐證狀態 ✅／⚠️）

  • 核准事件應綁定 thread_id、checkpoint 世代、tool_call_id，以對齊 interrupt 後的同一決策。 狀態：已驗證✅；支持來源數：4（LangGraph Interrupts、Persistence、OpenAI Agents SDK、ADK resume）。
  • 理由碼應為可機讀枚舉並可對照 error-object，利於稽核統計與下游 policy。 狀態：已驗證✅；支持來源數：3（MCP Security、OWASP LLM Top 10、OpenAI Agents HITL／治理共識）。
  • MCP Host 核准與 LangGraph 節點 interrupt 可分工：前者管能力集合與同意，後者管圖內流程與恢復點。 狀態：已驗證✅；支持來源數：3（MCP 規範、MCP Security、LangGraph HITL 概念）。
  • Golden Trace 須固定工具輸入雜湊與模型／prompt 指紋，否則重放可能因漂移失敗。 狀態：待驗證⚠️；目前以架構推導與部分框架重入模型支持，仍少公開欄位級規範。
  • JSONL（如 post_tool_logger）宜承載 approval_decision、reason_code、mcp_tool_name，與 phase／task_key 並列。 狀態：已驗證✅（與本儲庫 Hook／錯誤物件治理方向一致）；跨框架標準化程度待驗證⚠️。

實務應用建議（3-5 點）

  1. 核准前後皆寫入同一 JSONL schema：至少含 thread_id、tool_call_id、approval_decision、reason_code、checkpoint_gen（或等價）、mcp_tool_name。
  2. 理由碼與 policy 分層：上游枚舉固定（稽核），下游可映射到風險分數與自動允許／拒絕規則。
  3. Golden Trace 測試案例固定「輸入正規化」：對工具引數做排序鍵與空白規範後再雜湊，避免序列化差異造成假陰性。
  4. MCP 與圖層職責矩陣：Host 管 consent 與 tool allowlist；LangGraph（或同級編排）管 interrupt 節點與 resume payload。
  5. 對照 OWASP LLM 風險清單設計稽核儀表：將過度代理、不當工具使用對應到可觀測指標與告警。

與現有知識的連結

本主題與儲庫內「LangGraph 中斷節點之狀態恢復與工具調用重放」「MCP Host 與 LangGraph Interrupt：JSONL／error-object 追溯對齊」「LLM Agent operator override：理由碼與審計軌跡」等深度研究形成連續譜：前兩者建立 interrupt／Host 分工與追蹤平面；本報告把理由碼與 Golden Trace 收斂為可測契約，與 config/error-types.yaml、hooks/post_tool_logger 的結構化欄位可直接對接。
