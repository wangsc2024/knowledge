Activepieces 應用實踐：自訂 Piece 開發、AI Action 整合與 Production 部署指南（2026-04）

本文為 activepieces-workflow-automation 系列 application 階段，承接 foundation（架構與 pieces framework 概覽），聚焦可執行的開發實戰、AI 整合範例與 production 部署流程。

概述

Activepieces（GitHub: activepieces/activepieces）是以 TypeScript pieces framework 為核心的開源 All-in-One AI 自動化平台。截至 2026 年 4 月：

  • Stars：22,000+｜Forks：3,600+｜Commits：59,379
  • License：MIT（Community Edition）+ 商業授權（Enterprise）
  • 核心定位：「給技術團隊搭建工具，給非技術用戶視覺化使用」—— 介於純 no-code（Zapier）與純 code（LangChain）之間的混合模式

技術架構

Activepieces 採用 pnpm workspace monorepo（Turborepo + Bun），主要子模組：

| 模組 | 職責 |

|------|------|

| packages/pieces | 所有官方 pieces（npm packages） |

| packages/pieces-framework | createAction/createTrigger SDK |

| packages/engine | flow 執行引擎 |

| packages/ee | 企業版商業功能 |

| packages/ui | 視覺化 flow builder（Angular） |

執行引擎採沙箱隔離模式：每個 piece action 在獨立 Worker 中執行，支援 NPM 套件動態載入。

自訂 Piece 開發（TypeScript SDK 實戰）

核心 API

自訂 piece 由兩個核心工廠函式組成：


import { createAction, createTrigger, TriggerStrategy, Property } from '@activepieces/pieces-framework';

// === ACTION 範例：呼叫內部 API ===
export const sendInternalAlert = createAction({
  name: 'sendInternalAlert',
  displayName: 'Send Internal Alert',
  description: '呼叫內部告警 API',
  props: {
    message: Property.ShortText({
      displayName: 'Alert Message',
      required: true,
    }),
    priority: Property.StaticDropdown({
      displayName: 'Priority',
      required: false,
      defaultValue: 'normal',
      options: {
        options: [
          { label: 'Low', value: 'low' },
          { label: 'Normal', value: 'normal' },
          { label: 'High', value: 'high' },
        ],
      },
    }),
  },
  async run(context) {
    const { message, priority } = context.propsValue;
    const response = await fetch('https://internal.example.com/alerts', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ message, priority }),
    });
    return response.json();
  },
});


// === TRIGGER 範例：Polling（定期輪詢）===
export const newTaskCreated = createTrigger({
  name: 'newTaskCreated',
  displayName: 'New Task Created',
  description: '新任務建立時觸發',
  type: TriggerStrategy.POLLING,
  props: {},
  sampleData: { id: '1', title: 'Sample Task', createdAt: Date.now() },
  async test(context) {
    return pollingHelper.test(polling, context);
  },
  async onEnable(context) {
    await pollingHelper.onEnable(polling, { store: context.store });
  },
  async onDisable(context) {
    await pollingHelper.onDisable(polling, { store: context.store });
  },
  async run(context) {
    return pollingHelper.poll(polling, context);
  },
});

TriggerStrategy 選擇

| 策略 | 適用場景 | 延遲 |

|------|---------|------|

| POLLING | REST API 無 webhook 支援 | 5min（預設輪詢間隔） |

| WEBHOOK | 服務支援 event push | 近即時 |

開發流程


# 1. 建立 piece 目錄
npx pieces create my-internal-piece

# 2. 安裝依賴
bun install

# 3. 本機開發（熱重載）
bun run dev --piece=my-internal-piece

# 4. 建置
bun run build --piece=my-internal-piece

# 5. 部署至自託管實例（複製至 dist/packages/pieces/my-internal-piece）

AI Action 整合：LLM 與 MCP 雙軌模式

模式 A：Flow 內嵌 AI Action（直接 LLM 呼叫）

典型工作流範例 — 郵件智能分類自動化：


觸發器：Gmail - 新郵件收到
  ↓
動作 1：OpenAI - 分析郵件意圖
  Prompt: "分析以下郵件並回傳 JSON {intent: string, priority: 1-5}：{{trigger.body}}"
  ↓
動作 2：路由條件（Router）
  分支 A：intent = 'support' → Notion 建立工單
  分支 B：intent = 'sales' → HubSpot 建立 Lead
  分支 C：其他 → Slack 發送通知

Field mapping 語法：{{trigger.subject}}、{{step_1.output.intent}} — 動態引用上游步驟輸出。

模式 B：MCP Server 模式（AI Agent 呼叫 Activepieces）

Activepieces 內建 MCP Server，讓 Claude、Cursor 等 AI 助手透過 MCP 協定直接操控 flow：

啟用步驟：

  1. Settings → MCP Server → Toggle Enable
  2. 複製生成的 Server URL（格式：https://{instance}/api/v1/mcp/{token}/sse）
  3. 加入 MCP 客戶端配置：

// Claude Desktop: ~/Library/Application Support/Claude/claude_desktop_config.json
{
  "mcpServers": {
    "activepieces": {
      "url": "https://your-instance.com/api/v1/mcp/YOUR_TOKEN/sse"
    }
  }
}

MCP 工具分類：

| 類別 | 代表工具 |

|------|----------|

| Discovery | list_flows, list_pieces, get_connections |

| Flow Management | create_flow, publish_flow, enable_flow |

| Flow Building | add_action, update_step, delete_step |

| Tables | create_table, upsert_record, query_records |

| Testing | run_flow, get_run_details |

安全設計：OAuth token 綁定，credential 不透過 MCP 工具暴露（只傳遞 connection_id），所有操作 project-scoped。

「幫我在 Activepieces 建立 Contacts 表，包含 name/email/company 欄位」

部署與維運

Production Docker Compose 最小配置


# docker-compose.yaml（production）
version: '3.8'
services:
  activepieces:
    image: activepieces/activepieces:0.72.0  # 生產環境固定版本
    env_file: .env
    depends_on:
      - postgres
      - redis
    ports:
      - '8080:80'

  postgres:
    image: postgres:15-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: activepieces
      POSTGRES_USER: ${AP_POSTGRES_USER}
      POSTGRES_PASSWORD: ${AP_POSTGRES_PASSWORD}

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

volumes:
  postgres_data:
  redis_data:

必要環境變數：

  • AP_ENCRYPTION_KEY：32 字元隨機金鑰（保護 credentials）
  • AP_JWT_SECRET：JWT 簽名密鑰
  • AP_FRONTEND_URL：對外 URL（影響 webhook 回調地址）

基礎建設需求：

  • 最低 2GB RAM（建議 4GB+ 用於多 flow 並行）
  • PostgreSQL 15+ 與 Redis 7+
  • 正向代理（Traefik/Nginx）處理 SSL 終止

水平擴展（Multi-worker）


  worker:
    image: activepieces/activepieces:0.72.0
    environment:
      AP_EXECUTION_MODE: WORKER
    scale: 3  # 3 個 worker 處理 flow 執行

與 daily-digest-prompt 的連結

  1. MCP 橋接層：daily-digest-prompt 的 tools/*.py 可透過 Activepieces 的 custom piece 包裝成 MCP Server，讓 Claude Code 以自然語言觸發 ntfy 通知、更新 Todoist 任務
  2. Phase Pipeline 視覺化：Phase 0→1→2→3 的執行邏輯可在 Activepieces flow builder 中快速原型驗證，尤其是分支（Router）邏輯
  3. Todoist webhook 觸發：Activepieces 原生支援 Todoist 整合，可替代現有 PowerShell 輪詢模式，改為 webhook 驅動，減少輪詢 token 浪費
  4. 自動任務監控：透過 Tables 功能建立任務執行日誌表，MCP 讓 Claude 直接查詢失敗任務並生成分析報告

優缺點評估

優點：

  • TypeScript pieces SDK 成熟，熱重載大幅提升開發效率
  • MCP 雙向支援（Client + Server）是目前開源工具中整合最完整的
  • MIT 授權允許完全自託管，資料不外洩
  • 280+ pieces，60% 社群貢獻，持續成長中
  • Activepieces 2.0（mid-2025）大幅強化 AI Agent 能力

缺點 / 限制：

  • Angular UI 框架較重，與主流 React 生態系分離
  • 文件品質不穩（社群反映 piece 開發教學仍不足）
  • Worker queue 預設輪詢間隔 5min，近即時需求須改用 webhook
  • 原生排程（cron）功能有限，複雜排程需外部觸發
  • Enterprise 功能（RBAC/SSO）需付費授權，成本不透明
