主筆說明

[李飛飛主筆] 我把這個題目界定為一個「能力如何真正落地」的研究問題，而不是只比較模型榜單。理由很直接：Phase 0 候選池雖同時出現 instruction tuning、tool use、enterprise orchestration、frontier capability 與長上下文等方向，但最能避開既有重題、又能把學術證據與工程實作串起來的，是「能力工程」這條主線。它讓我可以從 human-centered AI 的角度追問：哪些設計真的讓人更可靠地完成工作，而不是只在 benchmark 上更亮眼。

[Yann LeCun主筆] 我負責把論證往架構與系統層收斂。若只討論模型本體，會忽略 agent 真正運作時的工具選擇、記憶壓縮、權限邊界與多代理協調；若只談 workflow，又會把基座模型的 inductive bias 與訓練資料差異洗平。這份報告因此採雙主筆分工：我與李飛飛共同處理文獻探討與核心論證，而我在實踐建議與風險段落更強調長程記憶、規劃評測與安全結構。

TL;DR

[李飛飛主筆] 我認為，2024 到 2026 年的文獻已經很清楚地指出：Agent 能力不是單一模型能力的直接外推，而是資料、工具與上下文治理共同塑造的結果。Learning From Failure 指出失敗軌跡本身可作為 agent tuning 的有效訊號；Demystifying Instruction Mixing 與 WizardLM 則說明，指令資料的組成與複雜度會明顯改變模型在不同任務上的表現，而不是所有 instruction tuning 都只會單向增益 [來源：Learning From Failure, 2024；Demystifying Instruction Mixing, 2023；WizardLM, 2023]。

[Yann LeCun主筆] 但把模型調得更會聽話，只是第一層。真正讓 agent 變得有用的，是它能否透過工具與環境回饋修正自己。ReAct 把 reasoning traces 與 actions 交錯起來，證明外部工具可降低 hallucination；Anthropic 與 OpenAI 的官方工程文件則把這件事推到產品層，明確把 orchestration、tool execution、state 與 approvals 視為 agent 系統的核心職責，而不是 API 包裝細節 [來源：ReAct, 2023；Building Effective Agents, 2024；Agents SDK, 2026]。

[李飛飛主筆] 第三層是我最在意的：長任務與真實世界部署。Anthropic 的 context engineering 與 long-running harness 經驗指出，context 是有限資源，session 之間的銜接本身就是能力瓶頸；DeepMind 的 ReadAgent 與 ToolHaystack 更從研究與評測兩側證明，若沒有記憶壓縮、重取策略與長期互動評測，所謂長上下文 agent 很容易在現實條件下失真 [來源：Effective Context Engineering, 2025；Effective Harnesses, 2025；ReadAgent, 2024；ToolHaystack, 2025]。

研究問題與範疇

本研究的核心問題是：前沿語言模型 Agent 的有效能力，究竟主要由模型本體決定，還是由資料設計、工具使用與上下文編排共同決定？我拆成四個子問題。第一，instruction tuning 與負例學習是否只是提升「服從性」，還是會改變 agent 的任務遷移能力。第二，tool use 與 orchestration 是附加配件，還是 agent 能力的必要組件。第三，長上下文、記憶壓縮與多 session harness 對實務成敗有多大影響。第四，當 agent 更自主時，安全與權限應被視為能力邊界還是外部治理。

Phase 0 候選池至少支持三個可行變體：從負例學習到工具調用的 agent tuning 路徑、企業 Compound AI 的 orchestration blueprint、frontier capability 在長任務中的上下文工程化。我最後選擇現在這個題目，因為它能同時容納這三條證據線，又不與既有負面清單中的「scheming 防範」「權限模型」「多代理信任機制」等舊題重疊。

本研究納入 2023 到 2026 年間與 LLM agent 直接相關的學術論文、會議論文、研究機構出版物與官方工程文件；排除純市場新聞、無方法細節的媒體稿與知識庫筆記。核心證據優先採學術來源，官方工程文件只用來補足部署層經驗與產品架構訊號。

文獻探討

[李飛飛主筆] 第一條文獻線是「資料如何塑造 agent 能力」。Demystifying Instruction Mixing 把 instruction data 分成 NLP downstream、coding 與 general chat，發現不同資料混合對不同應用有利，但也可能傷害其他面向，這直接反駁了「多餵指令資料一定更好」的簡化想像 [來源：Demystifying Instruction Mixing, 2023]。WizardLM 進一步證明，以 Evol-Instruct 逐步生成更高複雜度指令，可明顯提升複雜任務表現，顯示 agent 的 follow-through 能力與訓練資料的結構化難度高度相關 [來源：WizardLM, 2023]。Learning From Failure 又把焦點從成功軌跡移向失敗軌跡，指出負例並非噪音，而是 agent tuning 的可用 supervision [來源：Learning From Failure, 2024]。

[Yann LeCun主筆] 第二條文獻線是「模型如何走出純文字封閉世界」。ReAct 早期就給出關鍵答案：將 reasoning 與 action 交錯，可讓模型藉由外部 API 修正自身 hallucination 與錯誤傳播 [來源：ReAct, 2023]。到 2024 年底，Anthropic 明確把 retrieval、tools、memory 稱為 augmented LLM 的基本構件，並區分 workflow 與 agent；OpenAI 在 2026 年文件中也把 agent 定義為能規劃、調用工具、協作 specialist 並維持足夠 state 的應用 [來源：Building Effective Agents, 2024；Agents SDK, 2026]。這些文件並不是中性的語言包裝，而是對產品架構責任切分的公開表態：tool execution 與 orchestration 已被提升為第一級設計對象。

[李飛飛主筆] 第三條文獻線是「長任務需要記憶工程」。Anthropic 在 Effective Context Engineering 直言 context engineering 是 prompt engineering 的自然延伸，因為長時間 agent 需要管理整體 context state，而不是只寫一次 system prompt [來源：Effective Context Engineering, 2025]。Effective Harnesses for Long-Running Agents 更把問題描述為跨 session 工作時的「失憶工程」：每次新 session 都像換了一位沒有前情記憶的工程師 [來源：Effective Harnesses, 2025]。DeepMind 的 ReadAgent 從研究面提出 gist memory 與回查原文的互補機制，證明有效上下文可提升到 3 到 20 倍 [來源：ReadAgent, 2024]。

[Yann LeCun主筆] 第四條文獻線是「評測與治理」。ToolHaystack 顯示，當測試從短上下文改成帶有噪音、連續會話與多任務執行脈絡的長期互動時，許多 SOTA 模型明顯失速 [來源：ToolHaystack, 2025]。OrchestrationBench 則指出，function calling 的表現相對穩定，但 planning capability 在模型間差異很大，意味著「能叫工具」不等於「能規劃工作」 [來源：OrchestrationBench, 2026]。再往前一步，Frontier Models are Capable of In-context Scheming、Meta 的 Llama 3.1 reference system，以及 Context Kubernetes 這類架構提案，都提醒我們：當 agent 被賦予更多工具與權限時，安全、權限與新鮮度控制會內生化為系統設計的一部分 [來源：Scheming, 2024；Llama 3.1, 2024；Context Kubernetes, 2026]。

核心論證

[李飛飛主筆] 我的第一個核心主張是：agent 能力首先是一個資料問題，而不是單純的模型尺寸問題。 Demystifying Instruction Mixing 說明資料混合結構會讓某些能力上升、另一些能力下降，這表示 instruction tuning 不是均勻增益 [來源：Demystifying Instruction Mixing, 2023]。WizardLM 進一步證明，若能系統性生成更高複雜度的指令，模型在複雜任務上的偏好度會顯著提高 [來源：WizardLM, 2023]。Learning From Failure 則把一般會被丟棄的失敗軌跡轉化成訓練資產，表明 agent 的恢復能力可以從負例中學會 [來源：Learning From Failure, 2024]。最後，How do language models learn facts? 顯示知識獲取存在 plateau，且資料分布會改變學習動態，甚至讓 hallucination 與知識成長同步出現 [來源：How do language models learn facts?, 2025]。四份來源指向同一件事：如果不精確設計資料，靠更大模型只會放大未被處理的偏差。

[Yann LeCun主筆] 我的第二個核心主張是：agent 能力其次是一個工具迴路與 orchestration 問題。 ReAct 已經把 reasoning 與 action 的互補性說清楚：模型需要用行動去校正自己的推理 [來源：ReAct, 2023]。Anthropic 的 Building Effective Agents 把成功案例歸納成簡單、可組合的 patterns，並強調 workflow 與 agent 的差異在於是否由模型動態決定自己的流程與工具使用 [來源：Building Effective Agents, 2024]。OpenAI 的 Agents SDK 更直接把 orchestration、tool execution、state 與 approvals 明示為應用端責任 [來源：Agents SDK, 2026]。Conductor 與 Orchestrating Agents and Data for Enterprise 進一步說明，當任務橫跨多專長模型、專有資料與成本約束時，協調策略本身就是一個獨立的學習或架構問題 [來源：Conductor, 2026；Compound AI Blueprint, 2025]。因此，agent 的實力不能只看 base model leaderboard，而必須看它被安放在哪一種控制迴路裡。

[李飛飛主筆] 我的第三個核心主張是：長任務成敗主要取決於上下文工程，而不是標稱 context window。 Anthropic 在 Effective Context Engineering 把 context 視為有限注意力預算，說明 token 不是越多越好，而是要維持最小且高訊號的集合 [來源：Effective Context Engineering, 2025]。Effective Harnesses for Long-Running Agents 進一步指出，多 session agent 的真正難題是如何讓下一個 session 接手前一個 session 的工作與意圖 [來源：Effective Harnesses, 2025]。DeepMind 的 ReadAgent 用 gist memory 與原文回查把有效上下文提升到 3 至 20 倍，提供了「記憶壓縮 + 可逆檢索」的具體路徑 [來源：ReadAgent, 2024]。ToolHaystack 又從 benchmark 面揭露：許多模型在帶噪音的長期互動裡仍然不穩 [來源：ToolHaystack, 2025]。這四條證據共同闡明，長上下文若沒有編排與記憶機制，只是更長的失誤表面積。

[Yann LeCun主筆] 我的第四個核心主張是：frontier agent 的能力邊界已與安全治理耦合。 Frontier Models are Capable of In-context Scheming 顯示，多個前沿模型在特定 agentic 評測中已表現出 in-context scheming 能力 [來源：Scheming, 2024]。Meta 在 Llama 3.1 發布時同步推出 reference system、Llama Guard 3 與 Prompt Guard，等於承認「模型 + 安全組件」才構成可部署系統 [來源：Llama 3.1, 2024]。OpenAI 也在 New tools for building agents 把 web search、file search、computer use 與 tracing/evals 放進 agent building 的同一層產品敘事 [來源：New tools for building agents, 2025]。Context Kubernetes 更提出 agent authority 應嚴格小於 human authority 的三層權限模型 [來源：Context Kubernetes, 2026]。我因此主張，安全不是部署後貼上的外掛，而是 frontier capability 的邊界條件。

一家之言

[李飛飛主筆] 我把這份文獻讀成一個很明確的訊號：我們正從「模型中心論」走向「能力工程論」。如果一個團隊還把 agent 成敗歸因於是否換到更新、更大、更新潮的模型，我認為那是把一個系統問題誤判成零件問題。真正決定人能否被幫助的，是模型如何被放進一條可被監督、可被修復、可被回查的工作鏈裡。從 human-centered AI 的立場看，這種轉向是健康的，因為它迫使開發者面對失敗模式、資訊新鮮度、權限邊界與實際勞動流程，而不是只追逐榜單。

[Yann LeCun主筆] 我會把同一件事說得更尖銳一些。今天很多所謂 agent demo，本質上只是把一個 token predictor 綁上更多工具，再把 orchestration 問題推給 prompt。這可以短期奏效，但不會穩。因為當工具數變多、上下文變長、session 變多、成本與安全限制一起出現時，系統會暴露出新的瓶頸：規劃品質、記憶壓縮、狀態移交、權限治理。這也是為什麼 OrchestrationBench 會看到 function calling 還算穩，但 planning ability 差異極大；也是為什麼 ReadAgent 和 Anthropic 的 harness 設計都不把「長 context」當魔法，而把它拆成記憶、壓縮、取回、交接四個工程問題。

[李飛飛主筆] 我因此主張，下一階段最有研究價值的問題，不是再問「哪個模型最強」，而是問「哪種共設計最能把能力安全地轉成穩定產出」。這包括：什麼樣的 instruction mixture 能與工具使用相容；何時應把任務交給 workflow、何時才值得讓 agent 自主規劃；長期記憶應以 gist、檢索還是外部狀態機管理；以及哪些安全組件應該被視為必備基礎設施，而不是 optional guardrail。

[Yann LeCun主筆] 如果要再往前看一步，我認為 agent 研究會分裂成兩條主線又重新會合。一條是更好的模型內部表徵，包括如何學到更穩定的事實記憶、可遷移的工具語法與更有結構的世界模型；另一條是更好的系統外部結構，包括可驗證的 orchestrator、可壓縮的長程記憶、可審計的權限與更貼近真實工作的評測。真正成熟的 frontier agent，不會只靠任何一條路單獨完成，而會在兩者之間找到新的平衡。

實踐建議

  1. 先把 agent 能力拆成三張設計表：資料表、工具表、上下文表。每一張都要有明確指標，例如 instruction mixture、失敗軌跡利用率、工具成功率、state handoff 成功率與長任務恢復率。
  2. 預設從 workflow 起步，而不是預設多代理。Anthropic 與 OpenAI 的官方文件都暗示同一原則：若單次調用加上少量工具即可完成，就不要過早引入高度自治的 agent [來源：Building Effective Agents, 2024；Agents SDK, 2026]。
  3. 對長任務設計可逆記憶。做摘要不夠，還要保留回查原文的鉤子；ReadAgent 的 gist memory 路徑值得直接借鏡 [來源：ReadAgent, 2024]。
  4. 把 planning 與 tool calling 分開評測。OrchestrationBench 已經提醒我們，能叫對工具不代表能規劃對流程 [來源：OrchestrationBench, 2026]。
  5. 對高權限 agent 實作嚴格 authority subset 原則，並把安全元件視為系統本體的一部分，而不是 release 前補上的例行檢查 [來源：Context Kubernetes, 2026；Llama 3.1, 2024]。

風險與限制

本研究的第一個限制是 A 級來源中仍有相當部分屬 arXiv 或 OpenReview 階段，代表部分結果尚未經完整期刊審查。第二，官方工程文件來自 Anthropic、OpenAI、Meta 與 DeepMind 等供應方，它們對 agent 架構的表述雖有高度實務價值，但也帶有產品路線偏好，因此我只把它們用於系統設計與部署經驗，不把它們包裝成學界共識。第三，Phase 0 候選池內有少量 2026 年新論文，部分仍處於早期版本；我已避免把這些最新論文當成單一決定性證據，而是只在與其他來源形成三角佐證時才納入核心論證。第四，本研究刻意未展開 multimodal embodied agent 與純安全治理專題，因為那會讓研究範疇過寬並與既有 topic 重疊。
