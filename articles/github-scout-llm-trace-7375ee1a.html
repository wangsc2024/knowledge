GitHub Scout 研究報告：LLM 可觀測性／trace／eval

研究日期：2026-04-21

主題：LLM 可觀測性、OpenTelemetry GenAI、trace 分析、evaluation 框架

專案數量：3 個（Phoenix, Langfuse, OpenLLMetry）

改進提案：4 個（2 個 P1, 2 個 P2）

---

一、專案觀測摘要

1. Phoenix (Arize-ai/phoenix) - 9.4k stars

核心價值：原生 OpenTelemetry instrumentation + Claude Agent SDK 支援

  • 架構模式：

- OpenTelemetry-native trace instrumentation，產出標準 OTEL traces/spans

- 支援 Claude Agent SDK、LangGraph、CrewAI、DSPy 等主流框架

- 內建 trace 可視化 UI，可查看完整 LLM 呼叫鏈路

- 整合 LLM evaluation & benchmarking，支援自訂 evaluators

  • 部署選項：local、Jupyter、Docker、K8s、Cloud (app.phoenix.arize.com)
  • 與本系統的關聯：明確支援 Claude Agent SDK，可直接整合至 claude -p 執行流程

2. Langfuse (langfuse/langfuse) - 25.2k stars（最受歡迎）

核心價值：單一平台整合 observability + prompt mgmt + evals + playground

  • 架構模式：

- Session-based tracing，可追蹤跨多輪對話的 context

- Prompt template 版本控制與 A/B testing

- LLM-as-judge evaluations，自動化品質評估

- Datasets 管理，支援測試集與 benchmark

- Self-hosted 選項（Docker/K8s），資料留在本地

  • 整合範圍：OpenAI、LangChain、LlamaIndex、Vercel AI SDK、AutoGen、CrewAI 等 16+ 框架
  • 與本系統的關聯：session tracking 概念可應用至 run-todoist-agent-team.ps1 的 Phase 1/2/3 追蹤

3. OpenLLMetry (traceloop/openllmetry) - 7k stars

核心價值：純 OpenTelemetry 擴展，可與現有 observability stack 整合

  • 架構模式：

- 基於 OpenTelemetry，輸出標準 OTEL 資料

- 自動 instrumentation，無需手動埋點

- 支援多種 observability backend（Datadog, Honeycomb, New Relic）

- 定義 GenAI semantic conventions（gen_ai.* attributes）

  • 最新版本：0.60.0 (2026-04-19)
  • 與本系統的關聯：semantic conventions 可作為 post_tool_logger 的標準化參考

二、改進提案

P1-1：OpenTelemetry-native trace instrumentation

來源專案：Phoenix

目標檔案：hooks/post_tool_logger.py, config/observability.yaml

風險等級：medium

工作量：medium

問題現狀：

  • 現有 post_tool_logger 已記錄 trace_id/session_id，但缺少 gen_ai.* 標準屬性
  • JSONL 輸出無法被 Phoenix/Langfuse 等業界工具直接消費
  • 難以與 OpenTelemetry 生態整合

改進方案：

  1. 在 post_tool_logger 新增 span_attributes 欄位
  2. 包含 OpenTelemetry GenAI semantic conventions：

- gen_ai.system（anthropic/openai/groq）

- gen_ai.operation.name（chat/completion/embedding）

- gen_ai.request.model（claude-sonnet-4-6）

- gen_ai.request.max_tokens

- gen_ai.response.finish_reason

  1. 參考 GitHub Issue open-telemetry/semantic-conventions#2664

預期效益：

  • JSONL 日誌可被業界工具直接讀取與分析
  • 支援與 Phoenix/Langfuse 整合，獲得 trace 可視化能力
  • 對齊 OpenTelemetry 標準，提升可觀測性成熟度

P1-2：Session-based trace grouping

來源專案：Langfuse

目標檔案：hooks/post_tool_logger.py, tools/trace_analyzer.py

風險等級：low

工作量：low

問題現狀：

  • post_tool_logger 已記錄 session_id，但 trace_analyzer 未按 session 分組
  • 難以追蹤「單次 todoist-team 執行」的整體表現
  • 無法快速定位「哪一次執行」出問題

改進方案：

  1. 將 run-todoist-agent-team.ps1 的單次執行視為一個 session
  2. 所有 Phase 1/2/3 的 tool calls 共享同一 session_id
  3. 在 trace_analyzer 新增 session_summary 功能：

- 彙整單次執行的 token 用量

- tool 呼叫次數與分佈

- 成功率與失敗點

  1. 產出 session-level 診斷報告

預期效益：

  • 快速定位問題執行（「2026-04-21 10:30 那次為何失敗」）
  • 追蹤 session 級別的 token 經濟學
  • 支援跨 Phase 的端到端追蹤

P2-1：Prompt template versioning

來源專案：Langfuse

目標檔案：prompts/team/todoist-auto-*.md, config/prompt-registry.yaml

風險等級：medium

工作量：medium

改進方案：

  1. 建立 config/prompt-registry.yaml 記錄版本歷史
  2. 每次修改 prompt 時記錄：版本號、變更原因、效果驗證
  3. 支援 A/B 測試與成功率對比
  4. 提供 prompt 版本回退機制

P2-2：Auto-instrumentation with minimal code changes

來源專案：OpenLLMetry

目標檔案：tools/llm_router.py, tools/llm_gate.ps1

風險等級：low

工作量：low

改進方案：

  1. 在 Invoke-LlmGate 函式加入 pre/post hook
  2. 自動記錄 model、tokens、latency、error
  3. 減少 prompt 層面的 observability 負擔
