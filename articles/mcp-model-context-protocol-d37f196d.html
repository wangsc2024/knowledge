一、MCP 概述與定位

1.1 什麼是 MCP

Model Context Protocol（MCP）是 Anthropic 於 2024 年 11 月推出的開放標準，旨在標準化 AI 系統（特別是 LLM）與外部工具、資料來源的整合方式。MCP 被比喻為「AI 的 USB-C」，解決了 AI Agent 生態系中的 N×M 整合問題 — 在 MCP 之前，連接 N 個 AI 模型到 M 個外部服務需要建構 N×M 個自定義連接器。

1.2 核心架構

MCP 採用 Client-Server 架構：

  • MCP Host：AI 應用程式（如 Claude Desktop、VS Code、Cursor）
  • MCP Client：Host 內部的協議客戶端，負責與 Server 溝通
  • MCP Server：提供特定功能的服務端，暴露 Tools、Resources、Prompts 三種能力

1.3 三大核心能力

| 能力 | 說明 | 範例 |

|------|------|------|

| Tools | Agent 可呼叫的函數 | 搜尋檔案、執行 SQL、發送訊息 |

| Resources | Agent 可讀取的結構化資料 | 資料庫 schema、檔案內容、API 回應 |

| Prompts | 預定義的提示模板 | 程式碼審查模板、報告生成模板 |

三、技術規範深入分析

3.1 傳輸協議演進

| 版本 | 傳輸方式 | 特點 |

|------|---------|------|

| 初始版 | stdio（標準輸入/輸出） | 本地進程間通訊，最簡單 |

| 2025-03 | Streamable HTTP | 取代 SSE，支援 chunked transfer、serverless 部署（AWS Lambda）、企業防火牆友好 |

| 2025-11 | 非同步操作支援 | 長時間任務（數分鐘到數小時）不阻塞 |

3.2 訊息格式

MCP 使用 JSON-RPC 2.0 作為訊息格式：


{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "search_files",
    "arguments": {
      "query": "MCP implementation",
      "path": "/src"
    }
  }
}

3.3 安全模型

2025-06 規範引入多項安全強化：

  • OAuth 2.1 認證：MCP Server 作為 OAuth Resource Server
  • Resource Indicators：防止 Access Token 跨 Server 重用
  • Elicitation：Server 可安全地向使用者請求額外資訊
  • 安全最佳實踐指南：應對 prompt injection、惡意工具描述、取樣漏洞

已識別的主要安全威脅：

  1. Prompt Injection 攻擊：透過惡意工具描述注入指令
  2. 取樣漏洞：不受信任的 Server 操縱 LLM 行為
  3. Registry 碎片化：安裝受損程式碼的風險
  4. Elicitation 濫用：提取使用者敏感資訊

3.4 Server Discovery 機制

  • .well-known URL：Server 透過標準化 URL 發布元資料
  • MCP Registry（2025-09 預覽）：集中式 Server 目錄 API，類似 npm registry
  • Agent Cards：跨產業標準化的 Agent 能力描述格式

四、Agentic AI Foundation（AAIF）

4.1 成立背景

2025 年 12 月，Anthropic 將 MCP 捐贈給新成立的 Agentic AI Foundation（AAIF），一個 Linux Foundation 下的定向基金。其使命是「確保 Agentic AI 以透明、協作和符合公共利益的方式發展」。

4.2 創始成員

| 角色 | 成員 |

|------|------|

| 共同創辦 | Anthropic、Block、OpenAI |

| 支持者 | Google、Microsoft、AWS、Cloudflare、Bloomberg |

4.3 創始專案

| 專案 | 來源 | 用途 |

|------|------|------|

| MCP | Anthropic | AI-工具互操作標準 |

| goose | Block | 開源 AI Agent 框架 |

| AGENTS.md | OpenAI | Agent 行為規範描述標準 |

4.4 治理模式

捐贈後 MCP 治理模式不變：專案維護者繼續主導，優先社群意見與透明決策。AAIF 確保協議保持「開放、廠商中立」。

五、四大 AI Agent 協議對比

5.1 協議定位矩陣

| 協議 | 開發者 | 解決問題 | 連接方向 |

|------|--------|---------|----------|

| MCP | Anthropic 到 AAIF | Agent 到 工具/資料 | 垂直（上下游整合） |

| A2A | Google | Agent 到 Agent | 水平（跨組織協作） |

| ACP | IBM Research | Agent 到 Agent | 水平（內部協調） |

| ANP | 社群驅動 | Agent 到 Agent（網際網路級） | 水平（去中心化） |

5.2 技術架構對比

| 維度 | MCP | A2A | ACP | ANP |

|------|-----|-----|-----|-----|

| 架構 | Client-Server | Peer-to-Peer | Client-Server | P2P + DNS |

| 傳輸 | JSON-RPC/HTTP | HTTPS + JSON-RPC | REST APIs | HTTPS/DNS |

| 訊息格式 | 結構化 Schema | 標準化 JSON | MIME 多部分 | JSON-LD |

| 串流 | Streamable HTTP | SSE | 原生支援 | 可配置 |

| 身份驗證 | OAuth 2.1 | 密碼學身份驗證 | YAML 元資料 | W3C DIDs |

| 發現機制 | Registry + .well-known | Agent Cards | 被動發現（YAML） | DID + DNS |

5.3 互補而非競爭

這四個協議在三個層次上形成互補：

  1. 網路/發現層：ANP（跨組織 Agent 發現）
  2. 上下文/工具層：MCP（Agent 存取工具和資料）
  3. 通訊/協調層：A2A（跨組織 Agent 委派）、ACP（組織內部 Agent 通訊）

實務上建議採用 組合式協議棧：

  • MCP 負責工具存取（垂直整合）
  • A2A 負責跨組織協作（水平擴展）
  • ACP 負責內部 Agent 編排（內部協調）

六、企業採用模式與最佳實踐

6.1 採用階段

| 階段 | 時間 | 活動 |

|------|------|------|

| 實驗期 | 2024-2025 H1 | 概念驗證、內部工具整合 |

| 試點期 | 2025 H2 | 小規模生產部署、安全評估 |

| 規模化 | 2026+ | 企業全面採用、治理框架建立 |

6.2 典型應用場景

  • 事件管理：跨系統收集資料進行事件響應
  • 客服支援：讀取工單 + 自動分配優先級 + 追蹤系統
  • 安全運維：整合日誌、檔案、安全平台
  • 內部工具標準化：Bloomberg 10,000 工程師案例 — 將內部工具模組化為 MCP Server，工程師貢獻工具，AI 團隊專注建構 Agent

6.3 安全治理要點

  1. 權限最小化：MCP 工具預設最小權限，避免 over-permission
  2. Server 信任驗證：僅使用已驗證的 MCP Server（Registry GA 後可核實）
  3. 資料外洩防護：監控 MCP 通訊中的敏感資料流動
  4. 工具冒充防護：驗證 Server Identity，防止惡意工具注入
  5. 治理隨規模擴展：MCP 成為 AI 系統與企業環境的控制層，需要管理階層可見性與政策決策

七、MCP 路線圖（2025-11 規範後）

根據官方 Roadmap，下一階段重點：

  1. 非同步操作（SEP-1686）：支援長時間任務的 kick-off + polling
  2. 無狀態與可擴展性（SEP-1442）：水平擴展、Session 管理優化
  3. Server Identity：.well-known URL 自我描述
  4. 官方擴展：醫療、金融、教育等垂直產業擴展
  5. SDK 分級系統：依規範符合度、維護響應度分級
  6. Registry GA：從預覽過渡到生產可用
  7. 合規測試套件：自動化驗證 Client/Server/SDK 實作正確性
  8. 參考實作：高品質 Client 與 Server 參考實作，展示認證模式與遠端部署
