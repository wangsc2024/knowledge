靜態網站生成（SSG）框架比較與最佳實踐

技術概述

靜態網站生成（Static Site Generation, SSG）是在建置時預先渲染所有 HTML 頁面的技術，產出的純靜態檔案可透過 CDN 全球分發，實現極低延遲與高安全性。2026 年的 SSG 生態系已從純靜態演進為「靜態優先、動態按需」的混合架構，核心趨勢包括 Islands Architecture（島嶼架構）、Incremental Static Regeneration（ISR 增量靜態再生）、Server Islands（伺服器島嶼）、以及 Edge Rendering（邊緣渲染）。主流框架涵蓋 Astro（JS）、Hugo（Go）、Next.js（React）、Eleventy（JS）、Nuxt（Vue）、Zola（Rust）等，各有不同定位與效能特性。

核心概念與原理

SSG vs SSR vs CSR vs ISR

渲染策略時機優點缺點適用場景
SSG建置時極快載入、高安全性、CDN 友好內容更新需重新建置部落格、文件、行銷頁
SSR請求時即時動態內容伺服器負擔大、TTFB 較慢電商、個人化頁面
CSR瀏覽器豐富互動SEO 弱、首次載入慢SPA、後台管理
ISR建置時 + 按需再生兼具靜態速度與動態更新設定複雜度較高新聞、產品目錄

Islands Architecture（島嶼架構）

由 Astro 率先實作的架構模式。頁面預設為純靜態 HTML，僅將需要互動的區域（「島嶼」）獨立水合（hydrate），其餘部分零 JavaScript。優勢：

  • 頁面初始載入極輕量（零 JS 基線）
  • 每個島嶼獨立管理，不互相阻塞
  • 支援 client:load、client:idle、client:visible 等精細載入策略
  • 可混合使用 React、Vue、Svelte 等不同框架

Server Islands（Astro 5.0 新特性）

在 Islands Architecture 基礎上更進一步：允許特定元件在伺服器端渲染，其餘頁面保持靜態。結合了 SSG 的快取優勢與 SSR 的動態能力，適合「90% 靜態 + 10% 動態」的場景。

Content Layer API（Astro 5.0）

統一的內容資料層，支援 Markdown、MDX、JSON、CMS API 等多種來源。內建快取策略、增量建置、內容驗證，適合大規模內容站點。

主流框架深度比較

1. Astro（推薦：內容型網站首選）

  • 語言/生態：JavaScript/TypeScript，支援 React、Vue、Svelte、Solid 等
  • 建置速度：Vite 驅動，支援增量建置，大型站點表現優秀
  • JS 策略：預設零 JS，僅島嶼元件載入所需 JavaScript
  • 核心特色：Islands Architecture、Server Islands、Content Layer API
  • 適用場景：部落格、文件站、行銷頁、知識庫網站、作品集
  • Stars：50K+（GitHub）

2. Hugo（推薦：純靜態極速建置）

  • 語言/生態：Go 語言，單一執行檔，無 npm 依賴
  • 建置速度：業界最快之一，毫秒級建置數千頁面
  • JS 策略：不處理 JavaScript，純靜態 HTML 輸出
  • 核心特色：Go 模板引擎、內建 Sass/PostCSS、圖片處理、多語系
  • 適用場景：大型文件站（1000+ 頁）、企業網站、技術部落格
  • Stars：78K+（GitHub）

3. Next.js（推薦：需要動態互動的全端場景）

  • 語言/生態：React，最大的前端生態系
  • 建置速度：中等，大型站點需 ISR 或 On-Demand Revalidation
  • JS 策略：React hydration，相比 Astro 載入更多 JS
  • 核心特色：ISR、React Server Components、App Router、Edge Runtime
  • 適用場景：電商、SaaS 產品頁、需要認證/個人化的內容站
  • Stars：130K+（GitHub）

最佳實踐與常見陷阱

最佳實踐

  1. 內容型站點（200+ 頁）選 Astro 或 Hugo

- Astro：需要部分互動（搜尋、篩選、暗黑模式切換） - Hugo：純靜態、追求極致建置速度

  1. 善用 ISR 處理頻繁更新的內容

- Next.js 的 revalidate 參數控制再生頻率 - 避免每次內容更新都觸發完整重建

  1. Headless CMS 整合策略

- 建置時從 CMS 拉取資料（SSG 模式） - Webhook 觸發增量重建（CI/CD 整合） - 常用組合：Astro + Sanity、Hugo + Contentful、Next.js + Hygraph

  1. 圖片最佳化

- Astro: <Image> 元件自動最佳化 - Next.js: next/image 元件 - Hugo: 內建圖片處理管線

  1. 部署至邊緣

- Cloudflare Pages/Workers、Vercel Edge、Netlify Edge - 搭配 CDN 快取策略最大化效能

常見陷阱

  1. 過度使用 Next.js 做純靜態站點：如果不需要 React 互動，選 Astro 或 Hugo 更輕量
  2. Hugo 模板語法學習曲線：Go 模板語法不直覺，複雜佈局需要時間適應
  3. Astro 島嶼過多：每個島嶼獨立載入 JS，過多島嶼反而增加請求數
  4. 忽略增量建置：大型站點不用增量建置，CI/CD 時間會線性成長
  5. SSG 選擇錯誤場景：需要即時個人化或使用者生成內容的頁面不適合 SSG
  6. Gatsby 的生態衰退：儘管 Gatsby 仍可用，但社群活躍度明顯下降，新專案應考慮 Astro

現有使用場景

本專案相關的靜態網站包括：

  1. KB 知識庫靜態網站（266+ 篇文章）：從 RAG 知識庫匯出，部署到 Cloudflare Pages
  2. 禪境遊廊遊戲網站（zen-game-portal）：HTML5 遊戲集合站
  3. 楞嚴經知識網站：佛學研究 RAG 轉靜態網站

架構建議

  • KB 知識庫網站：目前手動生成 HTML → 建議評估 Astro 的 Content Layer API，可直接從 JSON/API 拉取筆記資料，自動生成帶搜尋功能的靜態站點
  • 遊戲網站：純 HTML5 Canvas 遊戲 → Hugo 或 Astro 皆適合，Hugo 建置更快
  • 未來擴展：若需要加入互動式搜尋、標籤過濾等功能，Astro 的 Islands Architecture 可僅在搜尋元件載入 JS，其餘頁面零 JS

潛在優化方向

  1. Astro + Content Layer API：統一知識庫資料源，自動增量更新
  2. Cloudflare Workers + Hugo：毫秒級建置 + 邊緣快取
  3. ISR 模式：頻繁更新的頁面（如每日新增筆記）使用 ISR 避免全站重建

2026 年選型決策樹

需要 SSG 框架？
├─ 純靜態、無互動需求？
│  ├─ 1000+ 頁面 → Hugo（最快建置）
│  ├─ 技術文件 → Docusaurus 或 MkDocs
│  └─ 個人部落格 → Eleventy（最簡單）
├─ 部分互動需求？
│  ├─ 內容為主 → Astro（Islands Architecture）
│  └─ 漸進式 SPA → SvelteKit
├─ 全端應用 + 靜態頁面？
│  ├─ React 生態 → Next.js（ISR + App Router）
│  └─ Vue 生態 → Nuxt
└─ 極致效能？
   ├─ 零水合 → Qwik（Resumability）
   └─ 極速建置 → Zola（Rust）
