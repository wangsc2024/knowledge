  • 3 個 Skill 的 cache-ttl 為 null：multi-agent-team-orchestrator、fairness-rebalancer、pre-mortem

深度審查（pre-mortem）

#### 問題清單

| # | 問題 | 嚴重性 | 影響 |

|---|------|--------|------|

| 1 | frontmatter 缺少 "Use when:" 觸發條件描述 | 高 | 觸發時機不明確，路由邏輯難以判斷 |

| 2 | cache-ttl 為 null（應為 "N/A" 或具體值）| 中 | 格式不一致，驗證腳本可能誤判 |

| 3 | triggers 僅 4 個（少於平均值 8-10 個）| 低 | 覆蓋度不足，可能被遺漏 |

| 4 | 缺少可執行範例（無 Bash curl 示範）| 低 | 實作時需推測，增加出錯風險 |

#### 路由一致性

  • ✅ SKILL_INDEX.md 中有正確觸發關鍵字
  • ✅ 被 prompts/team/todoist-query.md 引用（作為 plan 欄位）
  • ⚠️ routing.yaml 中無明確路由（因為是內部 Skill，由 Phase 1 自動呼叫）

實施修正

修正 1：補充 frontmatter

修改前：


name: pre-mortem
version: 1.0.0
description: 任務執行前自動診斷潛在失敗模式，預防性識別風險
triggers:
  - pre-mortem
  - 預防診斷
  - 失敗預測
  - 風險評估

修改後：


name: pre-mortem
version: 1.1.0
description: |
  任務執行前自動診斷潛在失敗模式，預防性識別風險。
  Use when: 在 Phase 1 規劃階段對即將執行的任務進行風險評估，或手動執行前置診斷時。
cache-ttl: "0min"
triggers:
  - pre-mortem
  - 預防診斷
  - 失敗預測
  - 風險評估
  - 前置檢查
  - 風險分析
  - 任務評估

變更說明：

  1. 版本升級：1.0.0 → 1.1.0（minor，因為加了 "Use when"）
  2. description 加入明確的 "Use when:" 觸發時機描述
  3. 新增 cache-ttl: "0min"（診斷結果不快取）
  4. 增加 3 個 triggers：前置檢查、風險分析、任務評估

修正 2：補充可執行範例

在 Step 1 後加入兩個實戰範例：

範例 1：檢查 KB 服務可用性


curl -s --max-time 2 "http://localhost:3000/api/health" > /dev/null 2>&1 && echo "KB 可用" || echo "KB 不可用"

範例 2：檢查快取狀態


uv run python -c "import json,pathlib; cache=json.loads(pathlib.Path('cache/status.json').read_text()); print('有效' if cache.get('knowledge',{}).get('valid') else '過期')" 2>/dev/null || echo "快取不存在"

修正 3：更新 SKILL_INDEX.md

觸發關鍵字從 4 個增加到 7 個，與 SKILL.md 保持一致。
