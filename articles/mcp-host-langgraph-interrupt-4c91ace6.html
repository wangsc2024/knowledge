MCP Host 與 LangGraph Interrupt：深度技術研究報告

執行摘要（200-400 字）

本報告接續 context/ai-research-plan.json 中已完成三角佐證的分析，將 MCP Host 與 LangGraph Human-in-the-loop／interrupt 放在同一條「可暫停、可恢復、可稽核」治理鏈上檢視。共識面有三：其一，人機協同應內建在編排狀態與 checkpoint 語意，而不是只對最終輸出做人工審閱 [來源：LangGraph — Human-in-the-loop（官方概念文件）, 2024]。其二，MCP 規格在傳輸與能力協商層界定工具與資源邊界，但高風險工具仍須 Host 政策與上層圖編排節點共同治理 [來源：Model Context Protocol Specification 2024-11-05, 2024]。其三，daily-digest-prompt 已透過 JSONL、Phase handoff 與 error-object 契約朝向「單一觀測平面」，適合承接核准與中斷事件而不另立平行 schema [來源：共用前言與 Hook／結果檔契約（CLAUDE.md 節錄脈絡）, 2026]。分歧在於核准欄位應否進入 MCP 訊息本体：規格維持中立，本倉建議以 post_tool_logger 與可選巢狀欄位承載，維持協定解耦 [來源：GitHub Scout：post_tool_logger 與 error-object／trace 聯動（本 repo 計畫稿）, 2026]。待驗證項包括 checkpoint 後端與 session 逾時對「工具副作用是否重放」的組合影響，以及 Windows 排程下長時人因延遲的成本 [來源：連續記憶與暫存 break-glass 延伸觀點, 2026]。

研究背景與動機

自動任務管線在 PowerShell 排程與多 Phase Agent 之間來回，若核准發生在工具邊界（MCP）與圖節點（LangGraph interrupt）兩處卻無共用追溯鍵，審計與 self-heal 會分裂。本研究聚焦三問：控制流分工、既有 JSONL／error-object 承接方式、以及離線／短連線環境下可恢復中斷的落地形狀。

核心技術解析（每個主張內嵌引用）

LangGraph 將「等待人類」建模為圖狀態的中斷與恢復，強調 durable state 而非純粹線上互動 UI；這與「只在最後一刻叫人來改稿」不同，後者難以重放與驗證 [來源：LangGraph — Human-in-the-loop（官方概念文件）, 2024]。

MCP 區分 Host／Client／Server：工具列表與資源宣告發生在會話能力協商之後，協定本身不取代應用層對「誰能在何種情境下呼叫何種工具」之政策 [來源：Model Context Protocol Specification 2024-11-05, 2024]。

本倉 Hook 與結果 JSON 契約將每次工具使用與階段交接留在可 grep 的平面；若在中斷節點寫入 task_key、phase、run_id 與 proposal_id（若適用），則與 GitHub Scout 計畫中之 error-object instance urn 可機讀對齊 [來源：GitHub Scout：post_tool_logger 與 error-object／trace 聯動（本 repo 計畫稿）, 2026][來源：共用前言與 Hook／結果檔契約（CLAUDE.md 節錄脈絡）, 2026]。

緊急覆寫（break-glass）與常態核准在運維上都要保留觀測；此與「可關閉日誌的捷徑」不相容，應在架構層明令禁止 [來源：AI 深度研究 break-glass／MCP 延伸觀點（工作區暫存稿）, 2026]。

關鍵洞見（含三角佐證狀態 ✅/⚠️）

  • *待決狀態快照應與 results/todoist-auto- 與 scheduler-state 的日期／run 語意對齊**（✅；多源：LangGraph 持久化概念、CLAUDE 結果檔規範、Scout 計畫稿、連續記憶共識）。
  • 核准事件映射到 error-object instance 空間，可維持單一 grep 平面（✅；Scout 計畫、error-object schema、CLAUDE）。
  • Host 靜態分級 × 圖節點動態覆寫須共用理由碼枚舉（✅；MCP 協定邊界＋本倉治理敘事＋多 Agent 研究連續記憶）。
  • Checkpoint 存儲與 MCP session 逾時對恢復時副作用是否重放（⚠️；本輪未對各發行預設做實測）。
  • Windows 短週期 CLI Agent 不應長阻塞等人；宜用 ntfy／Todoist 作人力佇列（⚠️；營運假設＋ break-glass 延伸）。

實務應用建議（3-5 點）

  1. 在 LangGraph interrupt 回復路徑寫入與 Phase 2 結果檔相同家族的鍵（agent、task_key、status 草案欄位），並由 post_tool_logger 附掛 trace metadata。
  2. MCP Host 側維護工具風險層級表；圖節點只在「動態情境」覆寫，不改層級表枚舉本身。
  3. 核准通過／拒絕皆產生 JSONL 事件，錯誤型態沿用 config/error-types.yaml 命名空間，避免 ad-hoc 字串。
  4. 排程任務遇人中斷時，優先將「待決摘要」推 ntfy，主進程釋出 worker；恢復時由新一輪 Phase 0 讀取快照。
  5. 與現有 github_scout proposal 對齊 proposal_id，以便 OODA 閉環回溯。

與現有知識的連結

延續連續記憶中 break-glass 與 operator override 研究：理由碼、雙人覆核與 Harness 索引已現成；本主題補上 協定層（MCP）與編排層（LangGraph）交界 的具體掛點，並把「核准事件放哪裡」收斂為掛鉤平面 vs 協定 payload 的工程權衡。

未來展望

建議以最小 PoC 驗證：同一 trace 上 MCP call_tool 與 LangGraph interrupt 恢復後的工具重放行為，並測量 PowerShell 管線在 30–120 秒人因延遲下的排程堆疊成本。
