Agent Harness 自主管理最終報告（Phase 2 自治 Gate 補強）

日期：2026-03-20

專案：daily-digest-prompt

執行摘要

  • 已完成 agent harness 深化研究，並確認本系統使用的自治控制面配置版本為 config/autonomous-harness.yaml 中 version: 1、產生日期 2026-03-20。
  • 已補齊 run-agent-team.ps1 的 Phase 2 assembly 自治 gate，讓 daily-digest harness 可依 supervisor 輸出自動切換 full / degraded / skip 與 retry 次數。
  • 已實際驗證 python tools/autonomous_harness.py --format json 可輸出 daily_digest_assembly_mode=degraded，並以手動腳本驗證 recovery -> skip、retries=0。

架構與工作流重點

  • 控制面：tools/autonomous_harness.py 匯總 run-fsm、scheduler state、failed auto tasks、token budget、heartbeat、resource snapshot。
  • 執行面：run-agent-team.ps1 與 run-todoist-agent-team.ps1 讀取 state/autonomous-runtime.json 套用自治 policy。
  • 可觀測性：輸出 state/autonomous-harness-plan.json、state/autonomous-runtime.json、state/autonomous-agent-registry.json、state/autonomous-resource-snapshot.json。
  • 自我修復：將 restart / self-heal / rebalance 決策寫入 recovery queue，為後續 recovery worker 預留介面。

已落地優化方案

  1. 控制面集中化
  2. 目的：把分散腳本邏輯收斂成單一 supervisor，降低人工判讀成本。
  3. 效益：可統一輸出 runtime policy、recovery queue、registry 與資源快照。
  4. 工具：Python、YAML、JSON state files。
  5. 風險：單點控制面複雜度上升；對策是保留 script fallback 與 state-based degrade mode。
  1. 資源感知與動態調度
  2. 目的：依 CPU、記憶體、GPU、token budget 自動調整併發與封鎖任務。
  3. 效益：降低高壓時的錯誤率與資源爭用。
  4. 工具：PowerShell、typeperf、nvidia-smi、Python supervisor。
  5. 風險：Windows 權限造成量測不穩；對策是加入 typeperf fallback。
  1. Phase 2 自治 Gate
  2. 目的：讓 daily-digest assembly 在 degraded / recovery 狀態不再硬跑完整組裝流程。
  3. 效益：減少高壓/故障時的額外 LLM 成本與連鎖失敗。
  4. 工具：PowerShell orchestration、runtime policy、FSM/span logging。
  5. 風險：過度降級可能影響摘要完整性；對策是保留 full / degraded / skip 分級並記錄 span/status。

驗證結果

  • python tools/autonomous_harness.py --format json：成功。
  • runtime 實際輸出：mode=degraded、daily_digest_assembly_mode=degraded、daily_digest_phase2_retries=0。
  • 手動驗證：在 scheduler-heartbeat stale 情境下，runtime 轉為 recovery，且 daily_digest_assembly_mode=skip。
  • agent status：登入成功，但 agent -p ... --mode=ask --workspace ... 實際回傳 [internal]，故研究與實作採本地腳本正式執行。
  • pytest：受 Windows 暫存資料夾 ACL 限制，tmp_path fixture 初始化失敗；已確認屬環境問題，不是本次邏輯錯誤。

後續方向

  • 新增常駐 autonomous_recovery_worker 消費 recovery queue。
  • 將 done_cert、resource snapshot、recovery outcome 匯總為長期治理時間序列。
  • 若未來需要跨主機調度，可將 supervisor 升級為常駐 control plane service。
