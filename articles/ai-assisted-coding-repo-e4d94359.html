AI-Assisted Coding 優化：repo map 壓縮策略、token 成本治理與多代理協作 KPI（2026）

技術概述

AI-Assisted Coding 在 2026 年的關鍵瓶頸已不是「模型會不會寫程式」，而是「如何在大型程式庫裡給對上下文、控制 token 成本，並讓多代理分工不把延遲和費用炸掉」。OpenAI 對 Codex 的公開說明顯示，coding agent 會在工具呼叫與多輪對話中快速吃掉 context window，因此需要 compaction、prompt caching 與明確的 repository instructions。Anthropic 的 Claude Code 文件則把 CLAUDE.md、路徑層級規則、subagent 與 Plan Mode 納入日常工作流。最新 AGENTS.md 研究進一步提醒：上下文檔不是越長越好，過多要求反而會拉低成功率並增加推理成本。這代表最佳實踐應從「多塞文件」轉為「分層上下文 + 最小必要指令 + 可量化 KPI」。

核心方法與工具

  1. 分層記憶與最小指令面

- 全域規則放在 CLAUDE.md 或 AGENTS.md，只保留架構、慣例、必跑命令、禁令。

- 子目錄規則只在真的讀到該樹時載入，避免把整個 repo 的規則一次塞進 context。

- 指令原則是最小必要：列出 build/test/lint、目錄責任、危險區域，不寫長篇教學。

  1. Repo map + Just-in-Time context

- 先產生結構化 repo map：模組邊界、依賴關係、入口點、測試位置、常用命令。

- 代理只在需要時載入目標檔、相依檔、對應測試與規則檔。

- 以 directory reference、檔案 reference、symbol search 取代整包原始碼灌入。

  1. Token budget policy

- 把 token 視為預算而不是免費資源。

- 為 explore、edit、verify、delegate 設定配額與升級條件。

- 對長對話啟用 compaction；對穩定不變的前綴用 prompt caching。

  1. Subagent / Multi-agent 協作

- 把可平行的工作拆給專用 subagent，例如 code-reviewer、test-writer、debugger。

- 每個 subagent 只拿到任務必需工具與局部上下文，避免 context 汙染。

- 主代理只彙總摘要與結論，不接收全部原始探索輸出。

  1. 量化治理 KPI

- 成功率：task success rate、首次通過率、人工接管率。

- 成本：每任務 input/output tokens、prompt cache hit ratio、每成功變更成本。

- 效能：P50/P95 延遲、探索步數、工具呼叫數、verify 迴圈次數。

- 品質：回歸失敗率、測試通過率、review 接受率、reopen rate。

實際工作流程（step-by-step）

  1. 建立專案級 CLAUDE.md 或 AGENTS.md，只放必要規則：架構邊界、常用命令、禁止事項。
  2. 產生 repo map：列出核心模組、依賴、資料流、測試入口與危險檔案。
  3. 定義 token budget：例如探索 20%、編輯 35%、驗證 35%、緩衝 10%。
  4. 收到任務後先用 Plan Mode 或 read-only 階段確認要改哪些模組，不直接全庫掃描。
  5. 僅載入當前任務需要的檔案、相鄰相依與對應測試；超出預算時先 compact 再繼續。
  6. 可平行的工作交給 subagent：例如一個代理整理影響面，另一個代理補測試。
  7. 主代理整合 patch 後執行 verify：lint、unit test、必要時 integration test。
  8. 記錄 KPI：本次任務 token、延遲、快取命中、verify 次數、是否人工修正。
  9. 每週回顧高成本任務，縮短或重寫規則檔，刪除無效 repo context。

程式碼 / 配置範例

1. 最小必要的 `CLAUDE.md`


# Project Memory
- Architecture: `apps/api` 提供 REST API，`packages/domain` 放核心規則，`apps/web` 只呼叫公開 service。
- Commands:
  - test: `pnpm test --filter @acme/api`
  - lint: `pnpm lint`
  - typecheck: `pnpm typecheck`
- Rules:
  - 修改 API handler 時，必須同步檢查 `packages/domain` 與 `apps/web/src/lib/api.ts`
  - 不要直接修改 `packages/domain/generated/*`
  - PR 前至少跑受影響模組的單元測試

2. Repo map 與 token budget 設定


context_policy:
  repo_map:
    include:
      - apps/api
      - packages/domain
      - tests
    summarize:
      - entrypoints
      - public_interfaces
      - test_locations
      - forbidden_paths
  token_budget:
    explore_pct: 20
    edit_pct: 35
    verify_pct: 35
    reserve_pct: 10
  compaction:
    trigger_tokens: 120000
    keep:
      - current_goal
      - changed_files
      - failing_tests
      - open_questions
  delegation:
    enable_subagents: true
    max_parallel_agents: 2

3. 任務遙測事件


{
  "task": "refactor-auth-session",
  "success": true,
  "input_tokens": 18400,
  "output_tokens": 5200,
  "cache_hit_ratio": 0.41,
  "tool_calls": 17,
  "verify_loops": 3,
  "latency_ms_p95": 48600,
  "human_edits_after_agent": 2
}

效果量化

  • OpenAI 在 2026-01-23 的 Codex agent loop 技術文指出，conversation 會因工具呼叫與歷史訊息快速膨脹，因此 CLI 內建 auto compaction 與 prompt caching 穩定策略；這代表大型 repo 場景中，context 管理已是 agent 的核心責任，而不是附屬優化。
  • OpenAI 在 2025-05-16 的 Codex 發布文建議把任務切成多個 well-scoped agents 同時執行，並指出 Codex mini 在 API 定價上提供 prompt caching 折扣，可直接把「快取命中率」視為成本治理指標之一。
  • Anthropic Claude Code 文件明確把 CLAUDE.md、subagent、自動委派、Plan Mode、測試驗證列為日常工作流，代表一線工具已把「分層記憶 + 專用代理 + verify」視為標準做法。
  • 2026-02 的 AGENTS.md 研究則指出：自動或過度冗長的 repository context 常使成功率下降，且推理成本上升超過 20%。因此 repo context 的最佳化目標不是寫更多，而是寫得更少、更精準。
推論：若團隊已經有穩定測試與清楚模組邊界，最值得先追的 KPI 通常不是更大的 context window，而是 cache_hit_ratio、tokens per successful task、verify_loops per merged task 與 human_edits_after_agent。

與本專案的應用場景（daily-digest-prompt 如何受益）

  1. templates/、skills/、workflows/ 很多，適合建立 repo map，讓代理先知道哪裡是規範、哪裡是執行邏輯。
  2. 可把 skills/knowledge-query、results/、context/ 的讀寫規則濃縮到專案級記憶檔，避免每次研究任務都重複探索。
  3. 適合將研究型流程拆成兩個 subagent：

- kb-strategist：只負責查 KB、做去重與系列判斷。

- web-researcher：只負責搜來源與提煉量化數據。

  1. 可新增任務遙測欄位：每次自動任務記錄 token、快取命中、KB 搜尋次數、verify 迴圈，做週期回顧。
  2. 對 context/research-registry.json、context/research-series.json 這類大檔，可優先導入摘要視圖或局部索引，避免主代理整檔載入。

風險與注意事項

  • Context 檔過長會讓代理更聽話，但不一定更有效；要定期刪掉過期規則。
  • 多代理若沒有明確邊界，可能把 token 成本與整體延遲倍增。
  • 把 compaction 做成黑箱摘要會遺失驗證證據；至少要保留目標、變更檔、失敗測試與待決策事項。
  • KPI 不能只看 token 下降，還要一起看成功率與人工返工率，不然容易壓低成本卻惡化品質。
  • prompt caching、repo map、subagent 都建立在「專案結構穩定」前提上；若 repo 持續大改，索引與規則要跟著更新。
