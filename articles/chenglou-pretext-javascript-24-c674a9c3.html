二、解決的核心問題：瀏覽器文字測量的 30 年痛點

傳統做法的問題

在瀏覽器中測量文字尺寸（如 getBoundingClientRect()、offsetHeight）會觸發 Layout Reflow——這是瀏覽器最昂貴的操作之一。每次讀取元素幾何屬性時，瀏覽器必須重新計算整個頁面佈局。在虛擬滾動（virtual scrolling）場景中，這個問題特別嚴重：需要預測數百行文字的高度，而每次 DOM 測量都會導致效能瓶頸。

Pretext 的解法

Pretext 完全繞過 DOM 測量，改用 Canvas API 的 measureText() 直接查詢瀏覽器字型引擎。Canvas 測量不觸發 reflow，因為它只需要字型資料，不涉及佈局計算。

三、核心架構：二階段分離設計

Pretext 的核心洞察是：文字測量與排版計算有不同的成本特性。測量（分段、Canvas 呼叫）昂貴但不頻繁；排版（寬度累加、斷行）便宜但頻繁。因此架構分為兩個階段：

Phase 1：prepare()（昂貴階段，執行一次）


輸入文字 + 字型
  ↓
analyzeText() → Intl.Segmenter 分段
  ↓
CJK 偵測 → 逐字拆分或標點合併
  ↓
measureAnalysis() → canvas.measureText() + 快取查詢
  ↓
computeSegmentLevels() → 雙向文字嵌入層級
  ↓
PreparedText: {widths[], kinds[], segLevels, breakableWidths[]}

效能：500 段文字約 17-19ms

Phase 2：layout()（便宜階段，重複執行）

純算術運算，零 DOM/Canvas 存取。走訪快取的分段寬度，累加至溢出時斷行。

效能：同樣 500 段文字僅需 0.09-0.10ms（相比 DOM 測量快約 500-1500 倍）

快取系統

三個模組級單例在多次 prepare() 呼叫間共享：

  1. 分段度量快取：Map>
  2. Emoji 修正快取：針對 macOS Chrome/Firefox 的寬度膨脹修正
  3. 引擎特徵檔：瀏覽器特定常數（如 lineFitEpsilon）

四、API 設計

主要函式

| 函式 | 用途 | 回傳 |

|------|------|------|

| prepare(text, font) | 文字分析與測量 | PreparedText（不透明控制代碼） |

| layout(prepared, maxWidth, lineHeight) | 高度預測 | {lineCount, height} |

| layoutWithLines(prepared, maxWidth, lineHeight) | 取得每行文字 | 行陣列含文字/座標 |

| layoutNextLine(prepared, cursor, maxWidth) | 動態寬度迭代 | 單行結果或 null |

| walkLineRanges(prepared, maxWidth, lineHeight, cb) | 零分配迭代 | 透過回呼返回 |

使用範例


import { prepare, layout } from '@chenglou/pretext'

// 基本高度測量
const prepared = prepare('AGI 春天到了. بدأت الرحلة 🚀', '16px Inter')
const { height, lineCount } = layout(prepared, textWidth, 20)

// 動態寬度排版（文繞圖）
let cursor = { segmentIndex: 0, graphemeIndex: 0 }
while (true) {
  const width = y < image.bottom ? columnWidth - image.width : columnWidth
  const line = layoutNextLine(prepared, cursor, width)
  if (line === null) break
  ctx.fillText(line.text, 0, y)
  cursor = line.end
  y += 26
}

五、多語系與國際化支援

| 語系類型 | 處理方式 |

|---------|----------|

| 拉丁文/預設 | Intl.Segmenter 詞邊界，標點合併 |

| CJK（中日韓） | 逐字拆分 + 禁則處理（kinsoku shori） |

| RTL（阿拉伯文、希伯來文） | UAX #9 雙向演算法計算嵌入層級 |

| 泰文 | Intl.Segmenter 字典邊界 |

| Emoji | 字素簇 + macOS 特定寬度修正 |

---

六、適用場景

  1. 虛擬滾動：預測行高度而無需掛載 DOM（最核心場景）
  2. Canvas/SVG 渲染：layoutWithLines() 提供每行座標，直接繪製
  3. Shrinkwrap 佈局：walkLineRanges() 找出最緊密的容器寬度
  4. 文繞圖：layoutNextLine() 迭代器支援每行不同寬度

---

七、已知限制

  1. system-ui 字型問題：macOS 上 Canvas 與 DOM 解析為不同光學變體（SF Pro Text vs Display）
  2. 泰文字典不一致：Intl.Segmenter 與 CSS 斷行字典可能不同
  3. lineHeight 需明確指定：不推斷 line-height: normal
  4. CSS 假設固定：硬編碼為 white-space: normal + word-break: normal + overflow-wrap: break-word

---

八、設計影響與靈感

  • Sebastian Markbåge 的 text-layout：架構靈感來源，特別是 Canvas measureText 用於字形成型
  • pdf.js 的 bidi 實作：雙向文字算法的參考實作
  • HarfBuzz 後端：用於無頭環境驗證 100% 詞寬精度

---

九、對本專案的啟示

  1. 二階段分離模式：「昂貴初始化 + 便宜重複查詢」的架構模式可借鏡到知識庫搜尋（向量計算 vs 快取查詢）
  2. 快取策略：三層快取（分段→Emoji→引擎特徵）的精細化設計值得學習
  3. 瀏覽器差異處理：針對不同瀏覽器的修正策略（epsilon 容差、寬度膨脹修正）
  4. 國際化：CJK 禁則處理（kinsoku shori）對處理中文排版的前端專案有參考價值
