Daily-Digest-Prompt Skill 系統可組合性與 Token 經濟學優化建議 — 從靜態索引到動態能力圖譜的演進策略（2026-02-27）

Daily-Digest-Prompt Skill 系統可組合性與 Token 經濟學優化建議

日期：2026-02-27 | 任務：運用 RAG 知識庫檢視專案並提出優化建議
基礎：知識庫 339+ 筆筆記、22 個 Skills、496 個測試、8 大自動任務群組

---

一、現況分析

1.1 Skill 系統現狀

本專案目前採用 Skill-First 策略（ADR-001），擁有 23 個 Skills（17 核心 + 6 工具），透過 SKILL_INDEX.md 提供靜態索引。這套系統在過去 2 週的運作中展現了穩定性，但從 Token 經濟學和可組合性角度分析，存在以下可優化空間：

| 維度 | 現狀 | 業界最佳實踐（2026） | 差距 |

|------|------|---------------------|------|

| 索引載入 | 全量載入 SKILL_INDEX.md（~100 行表格） | 漸進式揭露：L1 元資料 ~100 tokens → L2 運行時 <5K → L3 按需 | 載入過量 |

| Skill 選擇 | 關鍵字 + 標籤匹配 | 語義相似度 + 歷史成功率加權 | 匹配精度可提升 |

| 組合模式 | 手動串聯（prompt 硬編碼鏈式規則） | 動態依賴圖 + 自動組合 | 靈活性不足 |

| Token 預算 | 無顯式預算管理 | 分層預算（system 20% / tools 30% / history 30% / output 20%） | 完全缺失 |

| 快取效益 | API 回應快取（cache-policy.yaml） | 多層快取（精確匹配 + 語義快取 + Session 上下文） | 僅一層 |

1.2 Token 消耗熱點分析

根據 config/benchmark.yaml 定義的 avg_io_per_call <= 5000 目標，以及 post_tool_logger.py 的 JSONL 日誌結構，推估主要 Token 消耗來源：

  1. CLAUDE.md 全文載入：專案 CLAUDE.md 約 400+ 行，每次 session 全量注入 context window
  2. SKILL_INDEX.md 全量讀取：Skill-First 規則要求「啟動第一個動作就是載入 Skill 索引」，但 23 個 Skill 的完整表格在多數任務中只需 3-5 個
  3. 研究註冊表膨脹：research-registry.json 已有 50+ entries，每次研究任務都全量讀取
  4. 模板重複載入：preamble.md 每個 sub-agent 都獨立讀取，團隊模式 5+ 個 agent 產生 5 倍冗餘

二、問題識別（五大瓶頸）

問題 1：靜態索引缺乏上下文感知

現象：SKILL_INDEX.md 是一份靜態 Markdown 表格，所有 23 個 Skill 同等權重呈現。Agent 每次都必須讀取完整索引，即使當前任務只涉及 2-3 個 Skill。

影響：

  • 浪費約 2000-3000 tokens 的上下文空間
  • 觸發關鍵字匹配是字串級別的，缺乏語義理解
  • 新增 Skill 需手動更新索引，維護成本隨 Skill 數量線性增長

問題 2：Skill 組合缺乏形式化描述

現象：Skill 之間的組合關係散落在多處：

  • SKILL_INDEX.md 的 depends-on 註記（如 pingtung-policy-expert depends-on pingtung-news）
  • CLAUDE.md 的文字描述（「pingtung-policy-expert 必搭 pingtung-news」）
  • routing.yaml 的標籤映射（多標籤合併 skills）

影響：

  • 組合規則分散，新 Agent 或新 Skill 整合時容易遺漏
  • 缺乏形式化的依賴圖（dependency graph），無法自動偵測循環依賴或遺漏依賴
  • 模板衝突解決規則（game>code>research>skill>general）硬編碼在 routing.yaml 中

問題 3：Token 預算無顯式管理

現象：目前沒有 Token 預算分配機制。benchmark.yaml 有 avg_io_per_call <= 5000 目標，但這是事後度量而非事前控制。

影響：

  • 研究任務（WebSearch + KB 查詢 + 報告生成）消耗的 token 量遠超一般任務，但使用相同的 timeout 分配
  • 團隊模式中，Phase 1 的 5 個並行 agent 各自獨立管理 context，無全局 token 預算
  • CLAUDE.md 約 400 行 × 每次 session = 固定開銷，但業界建議 system prompt 應控制在總預算的 20% 以內

問題 4：研究註冊表缺乏壓縮機制

現象：research-registry.json 實施 7 天滾動視窗，但高頻研究日（如 2026-02-25 有 20+ entries）導致檔案膨脹。每次研究任務都全量讀取，消耗大量 token。

影響：

  • 7 天累積可達 100+ entries，全量讀取約 5000-8000 tokens
  • 去重判定只需近 3 天的 topic 和 task_type，但讀取了全部 7 天的完整 entry
  • 缺乏摘要/壓縮視圖，每個 entry 含完整 tags 陣列和 kb_note_title

問題 5：快取策略單一維度

現象：cache-policy.yaml 只實作了 API 回應快取（精確匹配 + TTL），缺乏語義快取層。

影響：

  • 相似但不完全相同的查詢無法命中快取（如「AI 深度研究」vs「AI 技術趨勢」）
  • Session 間的上下文無法複用（每次 agent 啟動都是冷啟動）
  • benchmark.yaml 的 cache_hit_ratio >= 40% 目標可能難以達成

---

三、優化建議方案（6 項）

建議 1：實作 Skill 能力圖譜（Skill Capability Graph）

目標：將靜態索引升級為結構化的 JSON 能力圖譜，支援動態查詢和漸進式載入。

方案：


# config/skill-graph.yaml（新增）
version: 1
skills:
  todoist:
    category: core
    capabilities: [task-query, task-create, task-complete]
    inputs: [todoist-api-token]
    outputs: [task-list, task-status]
    depends_on: []
    composable_with: [knowledge-query, ntfy-notify, api-cache]
    avg_tokens: 800  # 典型 SKILL.md 載入 token 數
  
  pingtung-news:
    category: core
    capabilities: [news-query, news-search]
    inputs: [mcp-endpoint]
    outputs: [news-list, news-detail]
    depends_on: []
    composable_with: [pingtung-policy-expert, knowledge-query]
    avg_tokens: 600

效益：

  • Agent 可按 capabilities 精準匹配 Skill，取代字串級關鍵字匹配
  • depends_on 和 composable_with 形式化後，可自動偵測依賴衝突
  • avg_tokens 欄位支援 token 預算估算
  • 漸進式載入：先讀 skill-graph.yaml（~50 行）→ 按需讀取 SKILL.md

預期效益：索引載入 token 減少 40-60%，Skill 匹配準確率提升至 90%+

優先級：P1（高）

建議 2：引入分層 Token 預算管理

目標：為不同類型的 agent session 設定明確的 token 預算分配。

方案：


# config/token-budget.yaml（新增）
version: 1
budgets:
  daily_digest:
    total: 50000  # 估算值，需實測校準
    allocation:
      system_prompt: 20%    # CLAUDE.md + preamble
      skill_loading: 15%    # SKILL.md 按需載入
      api_io: 35%           # API 呼叫 + 回應處理
      output: 20%           # 摘要生成 + 通知
      buffer: 10%           # 重試 + 錯誤處理
  
  todoist_research:
    total: 80000
    allocation:
      system_prompt: 15%
      skill_loading: 10%
      research_io: 40%      # WebSearch + WebFetch + KB
      analysis: 25%         # 深度分析 + 報告撰寫
      buffer: 10%
  
  auto_task:
    total: 30000
    allocation:
      system_prompt: 25%
      skill_loading: 15%
      execution: 40%
      output: 15%
      buffer: 5%

效益：

  • 事前預算分配取代事後度量，防止 token 超支
  • 不同任務類型有差異化的預算策略
  • 搭配 post_tool_logger.py 的實際數據校準預算值

預期效益：token 浪費減少 20-30%，平均回應時間縮短 15%

優先級：P2（中高）

建議 3：研究註冊表壓縮視圖

目標：為高頻讀取的 research-registry.json 提供壓縮摘要，減少 token 消耗。

方案：

在 context/ 目錄新增 research-summary.json，由 self-heal 自動任務每日生成：


{
  "generated_at": "2026-02-27T08:00:00+08:00",
  "summary": {
    "total_entries_7d": 45,
    "by_type": {
      "todoist_research": 12,
      "shurangama": 4,
      "tech_research": 3
    },
    "recent_topics_3d": [
      "深度工作與認知負荷理論",
      "排程效能與資源利用率優化",
      "Circuit Breaker Pattern"
    ],
    "saturated_types": ["todoist_research"]
  }
}

效益：

  • 去重判定只需讀取壓縮摘要（~20 行）而非完整 registry（~200+ 行）
  • saturated_types 預計算後，agent 無需自行遍歷統計
  • 完整 registry 僅在需要寫入新 entry 時才讀取

預期效益：研究任務的去重階段 token 消耗減少 70%

優先級：P1（高）

建議 4：CLAUDE.md 分層載入策略

目標：將 CLAUDE.md 拆分為核心層和擴展層，按需載入。

方案：


CLAUDE.md              → 精簡核心（~100 行）：架構概述 + 禁止行為 + ADR 索引
CLAUDE.detail.md       → 完整細節（~300 行）：執行流程 + Hook 說明 + 常用操作

核心 CLAUDE.md 只保留：

  1. 專案概述（10 行）
  2. Skill-First 策略（15 行）
  3. 文件驅動架構原則（10 行）
  4. 嚴禁行為（10 行）
  5. 架構決策索引 ADR（15 行）
  6. 配置文件速查表（20 行）
  7. 排程配置摘要（10 行）
  8. 引用指針：「完整細節見 CLAUDE.detail.md」

效益：

  • 每次 session 系統提示減少 ~60%（從 400+ 行降至 ~100 行）
  • 自動任務（30K token 預算）節省最多，system prompt 佔比從估計 40% 降至 15%
  • Agent 需要細節時按需讀取 CLAUDE.detail.md

預期效益：每個 session 節省 3000-5000 tokens 的固定開銷

優先級：P2（中高）- 需評估 Claude Code 對分層 CLAUDE.md 的支援程度

建議 5：語義快取層（Semantic Cache）

目標：在現有精確匹配快取之上，新增語義快取層，提升快取命中率。

方案：

擴展 api-cache Skill，新增語義快取邏輯：


# config/cache-policy.yaml 擴展
semantic_cache:
  enabled: true
  similarity_threshold: 0.85  # 語義相似度閾值
  supported_apis:
    - knowledge_query  # KB hybrid search 結果可語義快取
    - hackernews       # HN 新聞查詢結果短期可複用
  ttl_minutes: 30      # 語義快取 TTL 較短
  storage: cache/semantic-cache.json

實作方式：利用知識庫的 hybrid search 本身作為語義比對引擎。查詢前先在 semantic-cache.json 中搜尋相似 query：

  • 相似度 >= 0.85 → 直接返回快取結果
  • 相似度 < 0.85 → 呼叫 API，結果同時寫入精確快取和語義快取

效益：

  • 快取命中率預估從當前 ~30% 提升至 50-60%
  • 研究任務的 KB 去重查詢可複用近似查詢結果
  • 減少重複 API 呼叫（每日 44 次 session × 平均 3 次 KB 查詢 = 132 次，語義快取可減少 ~40 次）

預期效益：API 呼叫量減少 25-30%，benchmark cache_hit_ratio 達標（>= 40%）

優先級：P3（中）- 需要知識庫 API 的語義搜尋能力配合

建議 6：Skill 使用熱力圖與自動退化

目標：基於實際使用數據，動態調整 Skill 載入策略。

擴展 system-insight Skill，新增 Skill 使用熱力圖分析：
