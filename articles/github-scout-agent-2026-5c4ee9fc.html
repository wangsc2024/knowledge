GitHub Scout：Agent 可觀測性與審查閉環（Langfuse、Phoenix、OpenLIT）

專案觀測摘要

  • langfuse/langfuse：約 21.7k stars，2026-02-08 仍有更新。重點是把 tracing、evals、prompt management、datasets 放進同一個工程閉環，適合借鏡到 github_scout 的研究到落實流程。
  • Arize-ai/phoenix：官方頁面與 GitHub repo 都強調 tracing、evaluation、datasets、experiments 的證據鏈，適合作為「提案審查要能回看觀測依據」的參考。
  • openlit/openlit：OpenTelemetry-native，主打低接入成本與標準化語意慣例，提醒我們先把 policy 與欄位契約定清楚，再談 exporter 或 hook 接線。

改進提案

  1. P0 / 低風險：新增 config/github-scout-review-policy.yaml，統一專案證據欄位、proposal 審查欄位、execution contract 最小 metadata 與 results 摘要欄位。
  2. P1 / 中風險：為 backlog 的 implementation plan 補 evidence snapshot 與驗證結果快照，形成可回看審查證據的結構。
  3. P1 / 中風險：規劃 AI observability baseline config，未來再決定是否接 OpenTelemetry / OpenInference exporter。

審查結論

  • 提案 1 可行性 9/10、穩定性 9/10：只改 config/*.yaml 與 prompt/template，不碰 scheduler、PS1 或核心攔截 hooks，且 rollback 明確。
  • 提案 2 可行性 7/10、穩定性 7/10：需要調整 backlog 結構與計畫輸出，屬中風險，先出計畫。
  • 提案 3 可行性 7/10、穩定性 6/10：一旦接到 hook 或 exporter 就可能放大範圍，先保留為計畫。

已落實項目

  • 已新增 config/github-scout-review-policy.yaml。
  • 已更新 templates/auto-tasks/github-scout.md，要求任務先讀取 review policy，再依 policy 補齊 evidence、review 與結果摘要。
  • 已更新 prompts/team/todoist-auto-github_scout.md，強制 team prompt 對齊同一份 policy。

待人工審核項目

  • Review Evidence Snapshot：需決定 backlog entry 是否要正式升級 schema。
  • AI Observability Config Baseline：需決定是否導入 OTEL/OpenInference、exporter 位置與 rollout 範圍。
