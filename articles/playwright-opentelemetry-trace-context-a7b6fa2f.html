Playwright × OpenTelemetry 的瀏覽器遙測整合：事件映射、Trace Context 與治理實務（2026）

技術概述

Playwright 很適合收集「測試如何操作瀏覽器」的高保真資料，但它原生提供的是除錯導向 trace 與測試生命週期 API，不是直接面向 OpenTelemetry 的可觀測性匯出層 [來源：Playwright Tracing, 2026；Playwright Reporter API, 2026]。OpenTelemetry JavaScript 端則提供 traces 與 metrics 的穩定能力，logs 仍在 development，且瀏覽器端 instrumentation 目前仍屬 experimental [來源：OpenTelemetry JavaScript, 2026]。因此，若要在 daily-digest-prompt 一類專案中建立「瀏覽器自動化遙測」，最穩妥的做法不是把 Playwright trace 檔直接當成觀測後端格式，而是把 Playwright 的 hooks、reporter、attachments 與 network/page 事件視為事件來源，再由 Node 端 OTel SDK 建立 spans、metrics 與必要的 logs。這樣才能同時保留測試除錯能力、跨服務 trace 關聯，以及後續警報與成本治理空間 [來源：OpenTelemetry Propagation, 2026；Grafana Trace-based Alerts, 2026；Elastic Synthetic Monitoring, 2026]。

研究背景

今日任務同時碰到 Playwright × OTel、Context Window I/O 與 runtime browser telemetry。KB 目前已有 foundation 層的洞察簡報，但缺少兩個關鍵空白：第一，Playwright 的哪些生命週期與 artefact 最適合轉譯為 OTel spans；第二，synthetic / E2E 資料要如何避免污染正式營運遙測。這份筆記聚焦 mechanism 階段，目標是讓讀者理解整合邊界、資料模型與治理取捨，並能直接推進到 application 階段。

核心發現

1. Playwright 原生 tracing 適合除錯，不應直接替代 OTel 遙測模型

Playwright context.tracing 會捕捉瀏覽器操作、DOM snapshots 與 network activity，但官方明示它不會記錄測試 assertions，並建議優先透過 Playwright Test configuration 啟用 tracing，以取得更完整的失敗除錯脈絡 [來源：Playwright Tracing, 2026]。同時，Reporter API 的 onTestBegin、onStepBegin、onStepEnd、onTestEnd 讓你能在測試執行期間與完成時取得完整 TestResult，而 test.afterEach 與 TestInfo 又能取到測試標題、狀態與附件 [來源：Playwright Reporter API, 2026；Playwright Test, 2026；Playwright TestInfo, 2026]。

實務含義是：Playwright trace zip 應保留做「失敗回放與人工除錯」，OTel 則應承接「跨測試、跨服務、可聚合查詢」的遙測資料。兩者不是二選一，而是雙軌。

2. Playwright × OTel 的穩定整合點在 Node runner，不在瀏覽器 SDK 本身

OpenTelemetry JavaScript 文件指出 traces 與 metrics 已穩定，logs 仍在 development；同一頁也警告 browser client instrumentation 目前是 experimental 且 mostly unspecified [來源：OpenTelemetry JavaScript, 2026]。另外，OTel JS propagation 文件說明多數情況可由 instrumentation libraries 自動傳遞 trace context，但也提供 propagation.inject() / extract() 的手動做法，適合跨進程或自訂協定橋接 [來源：OpenTelemetry Propagation, 2026]。

這代表在 Playwright 場景中，最穩的設計不是把瀏覽器內每個事件都靠實驗中的 browser SDK 處理，而是由 Node 端 test runner 建立父 span，必要時再把 traceparent 注入瀏覽器請求、測試 header 或後端 API 呼叫 [來源：OpenTelemetry Propagation, 2026；Grafana k6 Distributed Tracing, 2023]。Grafana k6 的分散式追蹤整合也是類似思路：由測試工具將 tracing metadata 注入對後端的請求，再把 trace 與測試結果關聯 [來源：Grafana k6 Distributed Tracing, 2023]。

支持來源：

  • OpenTelemetry JavaScript（官方）
  • OpenTelemetry Propagation（官方）
  • Grafana Cloud k6 Distributed Tracing（官方）

3. 瀏覽器層語意模型仍在演進，Web Vitals 可先當次級訊號，不要把整個測試回合壓成單一 span

OpenTelemetry 的 browser semantic conventions 目前仍是 Development，且該章節現階段只正式定義 events；其中最明確的是 browser.web_vital 事件，承載 CLS、FID、INP、LCP 等欄位 [來源：OTel Browser Semantic Conventions, 2026；OTel Browser Events, 2026]。Grafana k6 在 2024-07 宣布 browser module 以 OpenTelemetry 生成 spans/traces，並把 Web Vital metrics、load time 與 executed methods 放進 Browser Timeline [來源：Grafana Browser Timeline, 2024]。這說明產業實踐上，Web Vitals 很適合附掛在 page-level 或 journey-level span/event 上，作為補充訊號，而不是把整個瀏覽器測試只抽象成一個粗粒度 transaction。

4. 真正可操作的資料切法是「journey span → page/action spans → backend correlation」，並把 synthetic 資料獨立治理

Elastic 的 synthetic monitoring 文件把 browser monitors 定義為「在預定間隔、受控環境中，持續執行真實瀏覽器使用者動作」，強調其價值是產生 rich、consistent、repeatable data [來源：Elastic Synthetic Monitoring, 2026]。Grafana trace-based alerts 文件則指出 metrics 能先告訴你有問題，但 traces 才能補上完整路徑與根因線索 [來源：Grafana Trace-based Alerts, 2026]。綜合這兩者，synthetic browser telemetry 最好的結構不是「只存測試結果」，而是：

  1. 一個 test/journey 根 span：代表單次使用者流程。
  2. 多個 page/action 子 spans：對應 goto、click、fill、等待資源等。
  3. 與 backend spans 的 trace context 關聯：用來查某次 UI 失敗其實卡在哪個 API 或 service。
  4. 獨立 tag/resource 標識 synthetic traffic，例如 telemetry.source=synthetic、test.framework=playwright、env.kind=ci。

5. 目前查無 Playwright 官方提供一體化 OTel exporter；最可靠策略是自建薄層橋接

本次查到的 Playwright 官方文件聚焦 trace viewer、test hooks、reporters、attachments 與 runner 生命週期，未發現官方直接提供「輸出 OTel spans 到 Collector」的整合頁面。這不是在宣稱官方絕對不存在相關整合，而是根據本次蒐集結果可做的保守判斷：若專案現在要落地，應預設採用自建薄層橋接，而不是等待一個現成官方 exporter。

核心概念與原理

整體架構可分成四層。

第一層是 Playwright 事件來源層。這層包含 test hooks、custom reporter、TestInfo.attach()、Playwright trace 檔與 page/network 事件。它的責任是提供「測試做了什麼」與「失敗時的 artefact」[來源：Playwright Test, 2026；Playwright Reporter API, 2026；Playwright TestInfo, 2026]。

第二層是 OTel span 建模層。建議把單次測試或單次 synthetic check 當成 root span，將重要互動建成子 span，例如 browser.goto、browser.click、browser.assertion_wait、browser.api_call。若要附加 Web Vitals，可用 browser event 或 span event，而不是全部塞進 attributes [來源：OTel Browser Semantic Conventions, 2026；OTel Browser Events, 2026]。

第三層是 trace context 關聯層。Node runner 端建立 active span 後，可透過 propagation.inject() 將 trace context 序列化，帶進自訂 header、測試 API client，或必要時帶入頁面端請求橋接 [來源：OpenTelemetry Propagation, 2026]。若後端服務也有 OTel，自然就能把前後端流程串成同一條 trace。

第四層是 治理與查詢層。這層重點是區分 synthetic 與 production 資料，避免測試噪音進入正式 SLO；同時藉 trace-based alerts 或以 trace 衍生 metrics 做告警 [來源：Grafana Trace-based Alerts, 2026；Elastic Synthetic Monitoring, 2026]。

最佳實踐與常見陷阱

最佳實踐：

  • 把 Playwright trace 與 OTel traces 分開保存。前者服務除錯，後者服務觀測與關聯分析。
  • 在 beforeEach / afterEach 或 custom reporter 建立與結束 spans，比在測試 body 裡零散埋點更穩定 [來源：Playwright Test, 2026；Playwright Reporter API, 2026]。
  • 對每個 span 加上最少但穩定的屬性，如 test.title、browser.name、page.url.template、test.retry、telemetry.source=synthetic。
  • 用 attachments 保留 screenshot、HAR、trace.zip 路徑，再把檔案位置或摘要寫進 span attributes，避免把二進位內容直接塞進遙測後端 [來源：Playwright TestInfo, 2026]。
  • 只把高價值事件送進 OTel。完整 DOM snapshots 留在 Playwright trace，不要同步成 spans，以免成本暴增。
  • 若要做長期 synthetic monitoring，固定頻率、固定執行環境與固定測試資料，因為 repeatable data 比單次漂亮 trace 更重要 [來源：Elastic Synthetic Monitoring, 2026]。

常見陷阱：

  • 把 Playwright trace viewer 當成 observability backend。它很適合單次回放，不適合做跨 run 聚合。
  • 直接依賴 browser SDK 做全部埋點。由於 browser client instrumentation 仍 experimental，維運風險高 [來源：OpenTelemetry JavaScript, 2026]。
  • 不區分 synthetic traffic 與真實使用者流量，導致 error budget 或 latency 百分位被測試流量污染。
  • 把整個 test case 壓成一個 span，失去 page/action 粒度，後續無法做 root cause 對位。

與本專案的關聯

對 daily-digest-prompt 而言，這項技術最直接的用途不是監控一般網站，而是建立「代理人瀏覽器執行的可觀測性」。如果未來本專案導入 Playwright 來做驗證、登入、資料抓取或 UI smoke test，建議採最小方案：

  1. 只在 Node runner 建 root span 與關鍵 action spans。
  2. 為每次 run 加上 task_key、session_id、agent、todoist_run_id 這類 resource/attributes。
  3. 失敗時附帶 Playwright trace zip、screenshot 與 console/network 摘要。
  4. 在 Collector 或後端查詢時，明確分流 synthetic 與 production。

這樣能把今日已在做的 JSONL / OTLP / observability 工作再往前推一層，讓瀏覽器自動化也能進同一套 trace 分析流程。

程式碼範例


import { test as base } from '@playwright/test';
import { context, trace, SpanStatusCode } from '@opentelemetry/api';

const tracer = trace.getTracer('playwright-runner');

export const test = base.extend({
  page: async ({ page }, use, testInfo) => {
    await tracer.startActiveSpan(`browser.test:${testInfo.title}`, async (span) => {
      span.setAttributes({
        'telemetry.source': 'synthetic',
        'test.framework': 'playwright',
        'test.title': testInfo.title,
        'test.retry': testInfo.retry,
      });

      try {
        await use(page);
        span.setAttribute('test.status', testInfo.status ?? 'unknown');
      } catch (error) {
        span.recordException(error as Error);
        span.setStatus({ code: SpanStatusCode.ERROR });
        throw error;
      } finally {
        const screenshot = await page.screenshot();
        await testInfo.attach('final-screenshot', {
          body: screenshot,
          contentType: 'image/png',
        });
        span.end();
      }
    });
  },
});

上例刻意把 instrumentation 放在 runner/fixture 層，而不是散在每個測試檔內。若後續要細分 action spans，可在 helper 中包 page.goto()、locator.click() 等常用操作。

知識缺口與未來方向

查無 Playwright 官方對「直接輸出 OTel spans 到 Collector」的正式整合頁面，因此下一階段應該驗證兩件事：

  1. 是否以 custom reporter + helper wrappers 就足以覆蓋大多數高價值事件。
  2. 是否需要在瀏覽器頁面內額外掛載輕量 browser instrumentation，只收 Web Vitals 與 navigation timing，而不做全量事件追蹤。

如果要往 application 階段前進，最值得做的是在本專案建一個最小 PoC：Playwright 測試跑一次登入或資料抓取流程，OTel 送到本機 Collector，再驗證查詢、告警與成本。
