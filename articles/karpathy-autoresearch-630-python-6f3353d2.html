Karpathy/autoresearch — AI Agent 驅動的自主 ML 實驗框架

概述

autoresearch 是 Andrej Karpathy（前 Tesla AI 負責人、OpenAI 共同創辦人）於 2026 年 3 月 6 日開源的研究自動化工具。核心理念極其精簡：一個 GPU、一個檔案、一個度量指標。AI Agent 自主修改訓練腳本、執行實驗、評估結果，人類只需撰寫高層研究方向，即可在一夜之間完成約 100 次自主 ML 實驗。

架構設計：三個核心檔案

autoresearch 的設計哲學是極簡主義——整個框架僅由三個關鍵檔案構成：

1. prepare.py（固定基礎設施）

  • 負責 tokenizer 訓練與資料集載入
  • 僅執行一次，不參與實驗迴圈
  • 提供標準化的資料前處理管線

2. train.py（Agent 修改目標）

  • 唯一被 AI Agent 修改的檔案
  • 包含完整 GPT 模型定義、Muon + AdamW 優化器、訓練迴圈
  • 630 行的設計確保完整程式碼可放入現代 LLM 上下文窗口（128K+ tokens）
  • 一切皆可修改：架構、超參數、優化器、batch size

3. program.md（人類研究指令）

  • Markdown 格式的高層研究方向
  • 人類撰寫並迭代優化
  • 引導 Agent 的實驗策略與探索方向

---

自主實驗迴圈（Agentic Loop）

Agent 執行一個持續的自主迴圈：


1. 讀取 program.md → 理解研究方向
2. 修改 train.py → 提出改進假說（如調整學習率、架構深度）
3. 執行訓練 → 固定 5 分鐘時間預算（wall clock，排除啟動/編譯）
4. 評估結果 → 使用 val_bpb 驗證指標
5. 決策 → 保留改進、回退失敗
6. 規劃下一次實驗 → 基於累積結果調整策略

為何是 5 分鐘固定預算？

  • 公平比較：無論 Agent 修改了架構還是超參數，時間預算完全一致
  • 平台無關：不同 GPU 上都能得到可比較的結果
  • 高頻迭代：每小時約 12 次實驗，一夜可完成約 100 次

val_bpb（Validation Bits Per Byte）

  • 驗證集上的 bits per byte，越低越好
  • 詞彙表大小無關：即使更換 tokenizer 或完全替換架構，仍可一致比較
  • 這是唯一的評估指標，確保 Agent 的每次修改都有明確的量化回饋

設計哲學分析

1. 極簡約束驅動創新

autoresearch 刻意限制在 630 行、單 GPU、5 分鐘預算。這些約束不是缺陷，而是設計特徵——它們確保了：

  • 完整程式碼可被 LLM 一次性理解
  • 實驗結果可跨平台比較
  • 進入門檻極低（一張 GPU 即可）

2. 人機分工重定義

  • 人類：設定研究方向（program.md）、解讀結果、規劃大戰略
  • AI Agent：執行「修改→訓練→評估」的重複性迴圈
  • Karpathy 稱之為「迭代的自動化，而非思考的自動化」

3. Git 原生版本控制

  • 每次成功的實驗修改都透過 Git commit 保存
  • 形成完整的實驗歷史追蹤
  • 可回溯任何改進的具體程式碼變更

社群影響與未來展望

爆發式增長

  • 發布一週即突破 30,000 Stars
  • 已出現社群分支（如 jsegov/autoresearch-win-rtx for Windows RTX）
  • 多個教學文章與實踐指南湧現

潛在影響

  1. 研究民主化：單 GPU 即可進行系統性 ML 實驗
  2. Agent-Native 研究範式：AI Agent 不只是工具，而是研究夥伴
  3. 可複製性提升：固定時間預算 + val_bpb 確保結果可比較
  4. 社群驅動的模型改進：Leaderboard 機制鼓勵公開競爭

限制

  • 目前僅針對小型語言模型訓練（nanochat）
  • 需要 NVIDIA GPU（暫不支援 AMD/Apple Silicon 原生）
  • Agent 品質依賴外部 LLM（Claude/Codex）
  • 5 分鐘預算可能限制大型架構探索

對 Daily-Digest 系統的啟示

  1. Agent 自主迭代模式：autoresearch 的「假說→實驗→評估」迴圈可借鏡至自動任務優化
  2. 極簡約束設計：630 行限制確保 LLM 可完全理解，類似本系統的 Skill-First 精簡原則
  3. 固定預算機制：5 分鐘時間盒與本系統的 timeout 機制異曲同工
  4. Git 原生追蹤：每次改進都有版本記錄，與本系統的 research-registry 理念一致
