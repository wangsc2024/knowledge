一、專案概述

Shannon（KeygraphHQ/shannon）是由 Keygraph 團隊開發的開源全自動 AI 滲透測試框架，定位為「Fully autonomous AI hacker to find actual exploits in your web apps」。截至 2026 年 3 月已累積 26,600 Stars、2,654 Forks，採用 AGPL-3.0 授權，是目前 GitHub 上最具影響力的 AI 安全測試開源專案之一。

核心數據

| 指標 | 數值 |

|------|------|

| Stars | 26,600 |

| Forks | 2,654 |

| 語言 | TypeScript |

| 授權 | AGPL-3.0 |

| 建立時間 | 2025-09-27 |

| 核心引擎 | Anthropic Claude Agent SDK |

| XBOW 成功率 | 96.15%（100/104 exploits）|

| 單次測試時間 | 1-1.5 小時 |

| 單次測試成本 | ~$16-50 USD（API 費用）|

核心定位

Shannon 的口號是：「Every Claude (coder) deserves their Shannon.」意即每個用 Claude 寫程式的藍隊（Blue Team），都需要一個 Shannon 作為紅隊（Red Team）。它填補了現代快速開發（vibe coding）與年度滲透測試之間的 364 天安全空窗期。

二、四階段多 Agent 架構（核心技術）

Shannon 模擬人類滲透測試員的方法論，採用精密的多 Agent 架構，結合白箱原始碼分析與黑箱動態漏洞利用：

Phase 1：偵察（Reconnaissance）

建構應用程式的完整攻擊面地圖：

  • 原始碼分析：解析技術棧、框架、依賴關係
  • 基礎設施探測：整合 Nmap、Subfinder、WhatWeb、Schemathesis 等工具
  • 瀏覽器自動化：即時探索應用程式，將程式碼級洞察與實際行為關聯
  • 輸出：所有入口點、API 端點、驗證機制的詳細地圖

Phase 2：漏洞分析（Vulnerability Analysis）-- 並行處理

利用偵察資料，專門化的 Agent 按 OWASP 分類並行運作：

  • 注入（Injection）Agent
  • XSS Agent
  • SSRF Agent
  • 認證/授權 Agent
  • 每個 Agent 執行結構化資料流分析，追蹤使用者輸入到危險 sink 的路徑
  • 輸出：假設可利用路徑清單（hypothesized exploitable paths）

Phase 3：漏洞利用（Exploitation）-- 並行處理

將假設轉化為實際證明：

  • 專用 Exploit Agent 接收假設路徑，嘗試執行真實攻擊
  • 使用瀏覽器自動化、命令列工具和自訂腳本
  • 嚴格策略：No Exploit, No Report -- 無法成功利用的假設視為誤報丟棄
  • 消除傳統掃描器的大量誤報問題

Phase 4：報告（Reporting）

編譯所有經驗證的發現為專業可執行報告：

  • 清理雜訊和 LLM 幻覺產物
  • 僅包含已驗證的漏洞
  • 提供可複製的 Proof-of-Concept（copy-paste 即可重現）
  • 輸出滲透測試等級的最終報告

三、XBOW Benchmark 成績分析

XBOW 是包含 104 個故意設計漏洞應用程式的開源安全基準測試。Shannon 在清理過的無提示版本上取得 96.15% 成功率（白箱模式）。

各漏洞類型成績

| 漏洞類型 | 總挑戰數 | 成功數 | 成功率 |

|----------|---------|--------|--------|

| Broken Authorization | 25 | 25 | 100% |

| SQL Injection | 7 | 7 | 100% |

| Blind SQL Injection | 3 | 3 | 100% |

| SSRF / Misconfiguration | 22 | 21 | 95.45% |

| XSS | 23 | 22 | 95.65% |

| SSTI | 13 | 12 | 92.31% |

| Command Injection | 11 | 10 | 90.91% |

4 個失敗案例的透明分析

  1. XBEN-10（XSS）：正確識別漏洞和字元黑名單，但錯誤評估 JSFuck payload 的限制，放棄了可行的利用路徑。知識缺口在 payload 編碼/混淆庫。
  2. XBEN-22（SSTI/預設憑證/路徑遍歷）：成功利用 3 個漏洞中的 2 個，但將 SSTI 誤分類為偽陽性。分類模型需精煉。
  3. XBEN-34（RFI）：正確找到檔案包含漏洞但將 RFI 誤分類為 LFI，導致使用錯誤的利用技術。
  4. XBEN-82（Command Injection via SSRF）：識別完整攻擊路徑但在分析階段失敗。

與傳統滲透測試的經濟學比較

| 指標 | 傳統滲透測試 | Shannon |

|------|------------|--------|

| 成本 | $10,000+ | ~$16（API 費用）|

| 時間 | 數週到數月 | 1.5 小時以內 |

| 頻率 | 每年 1-2 次 | 每次部署 |

四、技術實作細節

技術棧

  • 核心引擎：Anthropic Claude Agent SDK（推理引擎）
  • 程式語言：TypeScript
  • 工作流編排：Temporal（分散式工作流引擎）
  • 容器化：Docker（完整隔離環境）
  • 整合安全工具：Nmap、Subfinder、WhatWeb、Schemathesis
  • 瀏覽器自動化：用於執行真實 exploit（2FA/TOTP 登入、Google 登入）

原始碼結構


src/
  ai/              # AI Agent 執行引擎
    claude-executor.ts
    message-handlers.ts
    output-formatters.ts
    progress-manager.ts
    router-utils.ts
  audit/            # 審計 Session 管理
    audit-session.ts
    metrics-tracker.ts
    workflow-logger.ts
  temporal/         # Temporal 工作流編排
    activities.ts   # 各 Agent 活動定義
    workflows.ts    # 工作流程定義
    workspaces.ts   # 工作區管理（支援恢復）
  services/         # 外部工具整合
  types/            # TypeScript 型別定義
  utils/            # 工具函式

配置系統

Shannon 支援 YAML 配置檔，可自訂：

  • 認證流程（表單登入、2FA/TOTP、Google OAuth）
  • 測試規則（避免路徑、聚焦路徑）
  • 並行管線數（1-5，預設 5）
  • 重試策略（subscription plan rate limit 處理）

工作區與恢復機制

  • 每次執行建立工作區，進度透過 git commit 檢查點保存
  • 支援中斷後恢復：自動偵測已完成的 Agent，從中斷處繼續
  • URL 比對防護：防止跨目標污染

Router Mode（實驗性）

支援透過 claude-code-router 使用替代 AI 供應商（OpenAI GPT-5.2、Google Gemini 3-flash），但品質因模型而異。

六、產品線與商業模式

| 版本 | 授權 | 適用對象 |

|------|------|--------|

| Shannon Lite | AGPL-3.0（開源） | 安全團隊、獨立研究者 |

| Shannon Pro | 商業授權 | 企業級（CI/CD 整合、LLM 資料流分析引擎）|

Shannon Pro 基於 LLMDFA 論文（arxiv:2402.10754）提供進階的 LLM 驅動資料流分析引擎，用於更深層的漏洞偵測。

Keygraph 的願景是「Rippling for Cybersecurity」-- 從自動化滲透測試擴展到合規自動化（SOC 2、HIPAA）。

七、OWASP Juice Shop 實測成果

Shannon 在業界標準的 OWASP Juice Shop 測試中發現 20+ 個關鍵漏洞：

  • 完整認證繞過：透過注入攻擊竊取整個使用者資料庫
  • 完整權限提升：透過註冊流程繞過建立管理員帳號
  • 系統性授權缺陷（IDOR）：存取和修改任何使用者的私人資料
  • SSRF 漏洞：進行內部網路偵察

同時在 c{api}tal API 和 OWASP crAPI 中也取得類似成果，包括 JWT 演算法混淆攻擊、根級注入攻擊等。

八、對 Daily-Digest-Prompt 系統的啟示

1. 四階段管線 vs 我們的多階段架構

Shannon 的四階段設計（偵察 -> 分析 -> 利用 -> 報告）與我們的 Phase 1（並行擷取）-> Phase 2（組裝）管線有結構性相似。關鍵差異在於 Shannon 的 Phase 2/3 並行化策略：按漏洞類型水平切分並行，而非按資料來源。這提供了一種新的並行化思路：我們可考慮按「任務類型」而非僅「資料來源」進行並行切分。

2. No Exploit, No Report 策略

Shannon 的零誤報策略（假設必須被實際利用才計入報告）啟發我們的品質閘門：

  • 研究任務：可採用「No Evidence, No Claim」原則，所有論述必須有來源佐證
  • Hook 攔截：Shannon 的白箱分析 -> 黑箱驗證模式，類似我們的「規則定義 -> 運行時攔截 -> 日誌記錄」三層防護

3. Temporal 工作流編排

Shannon 使用 Temporal 作為分散式工作流引擎，支援工作區恢復（resume）和進度檢查點（git commit checkpointing）。我們目前使用 PowerShell Start-Job，未來可考慮引入輕量級工作流引擎來提升可靠性和可觀測性。

4. 安全工具整合模式

Shannon 整合 Nmap/Subfinder/WhatWeb 等工具的方式（Docker 容器內統一管理），值得我們的 Skill 系統參考：每個安全工具作為獨立 Skill，統一介面、獨立執行。

5. YAML 配置 + 工作流恢復

與我們的配置驅動架構（config/*.yaml）理念一致。Shannon 的工作區恢復機制啟發我們考慮為長時間運行的任務加入檢查點和恢復能力。

---

九、使用限制與注意事項

  1. 僅限白箱測試：Shannon Lite 需要存取原始碼，不支援純黑箱測試
  2. 禁止用於生產環境：exploit agent 會主動執行攻擊，有資料變異風險
  3. 需合法授權：未經授權的掃描可能違反 CFAA 等法律
  4. LLM 幻覺風險：報告仍需人工驗證
  5. 覆蓋範圍有限：不涵蓋第三方函式庫漏洞、加密弱點等靜態分析項目
