一、技術概述（150 字）

AI 軟體開發系統已從單一模型的 prompt-response 範式演進為多層架構的分散式系統。2026 年，產業焦點從「能否實現」轉向「如何可靠部署」— 89% 組織已實施 Agent 可觀測性，62% 具備詳細追蹤能力。核心架構分為三大支柱：(1) Agent SDK：封裝 Task 抽象、Tool 整合與 Memory 管理的執行框架；(2) 工作流引擎：基於 DAG 或 workflow-as-code 的持久化編排系統；(3) 程式碼生成管線：整合 AST 解析、語義分析與驗證回饋的自動化流程。本文深入三大支柱的實作機制，揭示從工具全景到底層實作的關鍵設計模式。

二、Agent SDK 核心架構設計

2.1 Agent = Model + Harness 架構分解

LangChain 於 2026 年提出的核心論述：Agent = Model + Harness。Harness（代理馬具）涵蓋所有非模型本身的邏輯：系統 prompts、工具執行、編排邏輯、中介軟體 hooks。此架構將「LLM 推理能力」與「任務執行基礎設施」解耦，實現可測試、可組合的系統設計。

Harness 核心元件：

  • System Prompts：角色定義、行為規範、輸出格式約束
  • Tool Execution Layer：工具發現、註冊、呼叫與錯誤處理
  • Orchestration Logic：任務拆解、子任務調度、結果聚合
  • Middleware Hooks：pre-tool / post-tool 攔截器，用於日誌、驗證、預算控管

2.2 Task 抽象模組：從 Goal 到 Execution Plan

Agent SDK 透過 ReAct (Reasoning + Acting) 模式處理任務：


# 偽程式碼：ReAct 任務執行迴圈
def react_loop(goal: str, max_iterations: int = 10):
    state = {"goal": goal, "observations": [], "plan": []}

    for i in range(max_iterations):
        # 1. Thought: LLM 推理下一步
        thought = llm.generate(
            prompt=f"Goal: {goal}\nHistory: {state['observations']}\nNext step?"
        )

        # 2. Action: 選擇工具並執行
        tool_name, tool_args = parse_action(thought)
        result = tool_registry.execute(tool_name, **tool_args)

        # 3. Observation: 記錄結果並更新狀態
        state["observations"].append({
            "thought": thought,
            "action": (tool_name, tool_args),
            "result": result
        })

        # 4. 判斷是否完成（exit condition）
        if is_goal_achieved(state, goal):
            break

    return state["observations"]

Sequential Pipeline 模式：任務按線性順序執行，Agent A 完成後將輸出傳遞給 Agent B，適用於資料處理管線（ETL 流程）。

Delegation 模式：高階 Agent 將複雜任務拆解為子任務並委派給專責 Agent，父 Agent 等待結果後繼續推理（適用於複雜查詢任務）。

Concurrent Execution 模式：多個 Agent 同時執行任務以降低延遲或獲得多樣化視角，最終由「synthesizer」Agent 聚合輸出。

2.3 Tool Integration：MCP 協定與工具註冊機制

Model Context Protocol (MCP) 是 Agent-工具整合的開放標準（類比 AI 領域的「容器」基礎設施），標準化工具的發現、註冊與呼叫流程。

工具註冊範例（LangChain 風格）：


from langchain.tools import Tool

def search_web(query: str) -> str:
    """在網路上搜尋資訊"""
    # 實際呼叫搜尋 API
    return fetch_search_results(query)

# 註冊工具到 Agent
tool = Tool(
    name="WebSearch",
    func=search_web,
    description="搜尋網路資訊，輸入查詢詞回傳結果摘要"
)

agent = create_agent(
    llm=llm,
    tools=[tool, calculator, database_query],  # 多工具註冊
    agent_type="zero-shot-react-description"
)

MCP 整合模式：SDK 透過 MCP 客戶端自動發現可用工具，無需手動註冊每個函式。Agent 可在執行時動態載入符合需求的工具集（如 filesystem 操作、API 呼叫、資料庫查詢）。

2.4 Memory 管理：會話記憶與長期記憶

會話記憶（Conversational Memory）：追蹤當前對話歷史，維持上下文連貫性。可選擇 Redis、Pinecone、Qdrant、Weaviate、Elasticsearch、Postgres 或自建儲存後端。

長期記憶（Long-term Memory）：持久化關鍵資訊（使用者偏好、歷史任務、領域知識），跨會話保留。

記憶架構模式：


# 記憶抽象層（Agent Framework 提供）
memory = ConversationBufferMemory(
    memory_key="chat_history",
    return_messages=True
)

# 外部持久化儲存（for 長時執行任務）
persistent_memory = {
    "task_progress": redis_client.get("task:123:progress"),
    "intermediate_results": postgres.query("SELECT * FROM task_results"),
    "user_preferences": json.load(open("user_profile.json"))
}

# Agent 在推理時整合記憶
context = f"""
User Preferences: {persistent_memory['user_preferences']}
Previous Task Progress: {persistent_memory['task_progress']}
Chat History: {memory.load_memory_variables({})}
"""

關鍵原則：對於跨多次互動或長時執行的編排任務，必須使用外部持久化儲存（而非依賴記憶體內上下文），確保 Agent 在中斷後能恢復工作。

三、工作流編排引擎實作機制

3.1 Temporal：Durable Execution 與 Workflow-as-Code

Temporal 是開源的 Durable Execution 平台，核心理念：應用程式的完整狀態自動保存，即使遇到崩潰或服務中斷也能自動恢復。

核心架構元件：

  1. Frontend Service：API 入口，處理啟動工作流、查詢狀態、發送訊號等客戶端呼叫
  2. History Service：儲存與管理工作流事件歷史，確保執行一致性
  3. Matching Service：將工作流與 Activity 任務有效匹配到可用的 workers，處理任務路由與調度

DAG 執行的 Temporal 方法：

與 Airflow 的外部 DAG 定義不同，Temporal 採用 workflow-as-code：工作流用 Go、Java、Python、TypeScript、C# 或 PHP 等語言撰寫為函式。


# Temporal Workflow 範例（Python）
from temporalio import workflow

@workflow.defn
class DataProcessingWorkflow:
    @workflow.run
    async def run(self, input_data: dict) -> dict:
        # 步驟 1：資料擷取
        raw_data = await workflow.execute_activity(
            fetch_data,
            input_data["source"],
            start_to_close_timeout=timedelta(minutes=5)
        )

        # 步驟 2：資料轉換（並行執行）
        transform_tasks = [
            workflow.execute_activity(transform_chunk, chunk, ...)
            for chunk in split_data(raw_data)
        ]
        transformed = await asyncio.gather(*transform_tasks)

        # 步驟 3：資料載入
        result = await workflow.execute_activity(
            load_to_database,
            transformed,
            start_to_close_timeout=timedelta(minutes=10)
        )

        return result

狀態持久化與恢復機制：

  • 事件溯源（Event Sourcing）：每個決策、計時器、訊號、Activity 呼叫都作為不可變事件記錄
  • Worker 崩潰恢復：若 worker 崩潰，另一個 worker 可透過重播事件歷史從中斷處恢復工作流
  • 確定性執行保證：工作流程式碼必須是確定性的（相同輸入產生相同輸出），避免使用隨機數或當前時間戳記（需透過 workflow API 取得）

2026 年更新：Temporal Nexus（跨 namespace 工作流連接）達到 GA、Multi-Region Replication 提供 99.99% SLA、Temporal Cloud on Google Cloud 上線。

3.2 Airflow：DAG 拓撲排序與調度演算法

Apache Airflow 是市場領導者，普及了 Python 社群的 DAG 概念。DAG 封裝執行工作流所需的一切：排程（Schedule）、任務（Tasks）、任務依賴（Task Dependencies）。

DAG 定義範例：


from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime, timedelta

default_args = {
    'owner': 'data-team',
    'retries': 3,
    'retry_delay': timedelta(minutes=5)
}

with DAG(
    dag_id='data_pipeline',
    default_args=default_args,
    schedule_interval='@daily',
    start_date=datetime(2026, 1, 1),
    catchup=False
) as dag:

    extract = PythonOperator(
        task_id='extract_data',
        python_callable=extract_from_api
    )

    transform = PythonOperator(
        task_id='transform_data',
        python_callable=transform_logic
    )

    load = PythonOperator(
        task_id='load_to_warehouse',
        python_callable=load_to_db
    )

    # 定義依賴關係（DAG 邊）
    extract >> transform >> load

拓撲排序演算法（Airflow Scheduler 核心）：

  1. 建構鄰接表：掃描所有任務的依賴關係，建構 DAG
  2. 入度計算：計算每個任務的前置依賴數量
  3. 初始化佇列：將入度為 0 的任務（無依賴）加入執行佇列
  4. 調度迴圈：

- 從佇列取出任務並執行

- 執行完成後，將所有下游任務的入度減 1

- 若下游任務入度變為 0，加入執行佇列

  1. 循環依賴檢測：若最終有任務入度 > 0，代表存在循環依賴（報錯）

動態任務調度策略：

  • LocalExecutor：在本機行程池中執行任務（適用於單機開發）
  • CeleryExecutor：分散式任務佇列，任務分配到多台 worker 機器（適用於生產環境）
  • KubernetesExecutor：每個任務啟動獨立 Kubernetes Pod（適用於資源隔離需求）

3.3 輕量級 DAG 調度器：Tawazi 與 Dagobah

Tawazi：純 Python 函式並行執行的 DAG 依賴結構。透過裝飾器自動推斷依賴關係，適用於資料科學管線。


from tawazi import dag, xn

@xn
def fetch_data(source):
    return load_from_api(source)

@xn
def clean_data(raw_data):
    return raw_data.dropna()

@xn
def analyze(clean_df):
    return clean_df.describe()

# 自動建構 DAG 並並行執行
@dag
def pipeline(source):
    raw = fetch_data(source)
    clean = clean_data(raw)
    result = analyze(clean)
    return result

pipeline("api.example.com").execute()

Dagobah：基於 Cron 排程的簡單依賴型作業調度器，每個作業啟動一系列按依賴圖定義順序的子行程任務。

四、程式碼生成管線設計

4.1 AST 與 Tree-sitter：從文字到語義

Tree-sitter 是增量式 GLR 解析函式庫，用於程式設計工具（編輯器、linter、語法高亮）。與傳統編譯器的 AST 不同：

| 特性 | AST (Abstract Syntax Tree) | Tree-sitter CST (Concrete Syntax Tree) |

|------|---------------------------|--------------------------------------|

| 表示內容 | 程式的邏輯結構（what it means） | 原始碼的表面結構（what it looks like） |

| 保真度 | 移除多餘符號（括號、分號） | 保留所有原始碼細節（含註解、空白） |

| 用途 | 編譯器、靜態分析 | 編輯器、程式碼格式化、diff 工具 |

| 錯誤處理 | 遇到語法錯誤即停止 | 容錯解析，產生部分樹 |

Tree-sitter 解析流程：

  1. Grammar 定義：為目標語言手寫 Tree-sitter grammar 檔案（如 grammar.js）
  2. 解析產生 CST：輸入原始碼，Tree-sitter 產生完整的具體語法樹
  3. 轉換為通用表示：將 Tree-sitter 節點對應到通用化的語法表示（如統一的 AST 格式）
  4. 語義分析：在通用 AST 上執行型別檢查、符號解析、語義驗證

實際範例（JavaScript 解析）：


from tree_sitter import Language, Parser

# 載入語言 grammar
JS_LANGUAGE = Language('build/languages.so', 'javascript')
parser = Parser()
parser.set_language(JS_LANGUAGE)

# 解析原始碼
code = b"function add(a, b) { return a + b; }"
tree = parser.parse(code)

# 遍歷 CST
root_node = tree.root_node
for child in root_node.children:
    print(f"Node type: {child.type}, Text: {child.text}")

輸出：


Node type: function_declaration, Text: b'function add(a, b) { return a + b; }'
  Node type: identifier, Text: b'add'
  Node type: formal_parameters, Text: b'(a, b)'
  Node type: statement_block, Text: b'{ return a + b; }'

4.2 語義分析與結構化分塊（AST-based Chunking）

挑戰：LLM 的上下文視窗有限（即使擴展到 200K tokens），無法一次載入大型程式碼庫。

解決方案：使用 AST 將程式碼分塊為語義單元（函式、類別、模組），保留結構完整性。

AST Chunking 演算法：

  1. 解析為 AST：用 Tree-sitter 產生完整的語法樹
  2. 識別語義邊界：

- Python：function_definition, class_definition, module

- JavaScript：function_declaration, class_declaration, export_statement

  1. 提取程式碼片段：每個語義單元提取為獨立 chunk，保留行號與檔案路徑
  2. 生成 Embedding：將每個 chunk 透過 CodeT5/CodeBERT 生成向量嵌入
  3. 儲存到向量資料庫：用於 RAG 檢索（如 Pinecone、Qdrant）

RAG 增強的程式碼生成流程：


使用者需求 → 語義搜尋 → 檢索相關程式碼 chunk →
將 chunks 作為上下文 → LLM 生成程式碼 →
AST 驗證（語法正確性） → 單元測試驗證 → 輸出

4.3 驗證回饋循環：Linting + Testing + Iterative Fix

LLMLOOP 框架（2026）：透過迭代回饋循環與多種分析方法增強 LLM 生成的程式碼。

三層驗證閘門：

  1. 編譯檢查：確保所有生成或修改的程式碼可編譯（語法正確）
  2. 靜態分析（Linting）：使用 PMD、ESLint、Pylint 偵測常見缺陷

- 未使用的變數

- 空的 catch 區塊

- 不必要的物件建立

- 違反程式碼風格

  1. 單元測試執行：執行完整測試套件（單元、整合、型別檢查）

Linter-Directed Agents（Factory.ai 實踐）：

Linter 規則直接編碼架構邊界與人體工學到程式碼生成迴圈中。當 Agent 生成的程式碼違反 lint 規則，自動觸發修復循環（self-healing）：


# 偽程式碼：Linter-Directed 生成循環
def generate_with_linting(spec: str, max_iterations: int = 5):
    code = llm.generate(spec)

    for i in range(max_iterations):
        lint_result = run_linter(code)  # 執行 ESLint / Pylint

        if lint_result.passed:
            break  # 無錯誤，生成完成

        # 將 lint 錯誤作為回饋
        fix_prompt = f"""
        Generated code has linting errors:
        {lint_result.errors}

        Fix the code to pass linting rules.
        Original spec: {spec}
        Current code:
        {code}
        """
        code = llm.generate(fix_prompt)  # LLM 修復

    return code

Specification-Driven 工作流（TDD for LLM）：

  1. 鎖定規格：先撰寫需求規格與驗收標準
  2. 生成測試：LLM 根據規格生成測試案例
  3. 生成程式碼：LLM 生成滿足測試的程式碼
  4. 驗證迴圈：執行測試 → 失敗 → 提供錯誤訊息 → LLM 修復 → 重新執行

關鍵洞察：分離驗證與測試流程 — 驗證（編譯 + Linting）可在程式碼變更後快速重跑，無需執行完整的使用者旅程測試套件，使 fix-and-verify 循環更快速。

4.4 產業實踐：SWE-bench 與 Agent-driven Development

SWE-bench：2026 年 AI 程式碼 Agent 的標準基準測試，包含真實 GitHub issue 與對應的正確修復 patch，測量 Agent 的端到端解題能力（理解問題 → 定位程式碼 → 生成修復 → 通過測試）。

頂尖工具表現（2026 年 3 月）：

  • Claude Code（Opus 4.6）：~42% 解決率
  • Devin（Cognition Labs）：~38% 解決率
  • Cursor（Anysphere）：~35% 解決率

Agent-driven 開發流程（Agentic Coding）：

  1. 需求理解：Agent 讀取 issue 描述、現有程式碼與測試
  2. 問題定位：使用 AST 搜尋與語義檢索定位相關程式碼
  3. 修復生成：基於上下文生成 patch
  4. 驗證：執行回歸測試，確保未破壞現有功能
  5. 迭代精煉：若測試失敗，分析錯誤並重新生成

5.2 設計權衡分析

| 設計選擇 | 優點 | 缺點 | 適用場景 |

|---------|------|------|----------|

| Temporal workflow-as-code | 型別安全、IDE 支援、易測試 | 學習曲線陡峭、需部署 Temporal server | 關鍵任務工作流、需強一致性 |

| Airflow DAG | 成熟生態、豐富 operators、視覺化 UI | 無原生 durable execution、DAG 定義繁瑣 | 資料管線、批次處理 |

| LangChain Agent | 快速原型、豐富工具整合、社群活躍 | 產品化挑戰、錯誤處理需自行實作 | 研究專案、PoC 開發 |

| Tree-sitter CST | 完整保真度、容錯解析、增量更新 | 需手寫 grammar、CST 節點繁雜 | 編輯器整合、diff 工具 |

| 傳統編譯器 AST | 語義清晰、適合靜態分析 | 語法錯誤即失敗、語言綁定強 | 程式碼重構、型別檢查 |

5.3 生產級檢查清單

Agent SDK 部署前檢查：

  • [ ] 工具呼叫有 timeout 與 retry 機制
  • [ ] Memory 有 size limit 防止 OOM
  • [ ] 敏感資訊（API key）透過 secrets manager 注入
  • [ ] 可觀測性：每個 Agent 步驟有結構化日誌與 trace ID
  • [ ] 成本控管：Token 使用有預算上限與告警

工作流引擎上線前檢查：

  • [ ] 所有 Activity 有明確的 timeout 設定
  • [ ] 長時執行工作流使用 heartbeat 機制
  • [ ] 狀態持久化在外部儲存（Redis/PostgreSQL）
  • [ ] 有 dead letter queue 處理失敗任務
  • [ ] Multi-region 部署有 failover 機制（Temporal Cloud）

程式碼生成管線上線前檢查：

  • [ ] Linting 規則覆蓋架構約束（import 路徑、命名規範）
  • [ ] 單元測試自動生成並執行
  • [ ] 生成程式碼有 diff review 流程（非自動 merge）
  • [ ] AST 驗證層防止惡意程式碼注入
  • [ ] 有人工審查閘門（critical path 程式碼）

6.1 當前架構對應

daily-digest-prompt 已實作的 mechanism 層元件：

  1. Agent SDK 層：

- Harness：PowerShell 腳本 (run-agent-team.ps1) + Prompt 薄層 (prompts/team/*.md)

- Tool 註冊：skills/SKILL_INDEX.md 作為工具發現機制

  1. 工作流引擎層：

- DAG 編排：3-Phase 模式（Query → Execute → Assemble）

- 狀態持久化：state/scheduler-state.json（PowerShell 獨佔寫入）

- 錯誤恢復：state/autonomous-recovery-queue.json（失敗重試機制）

  1. 程式碼生成管線層：

- Hook 驗證：hooks/pre_bash_guard.py, hooks/pre_write_guard.py（AST 驗證類比）

- Linting：tools/validate_adr.py, hooks/validate_config.py（配置驗證）

- 測試循環：pytest 套件（856 測試，45.71% 覆蓋率）

6.2 可借鑑的優化方向

優化點 1：引入 Temporal 替代 PowerShell FSM

  • 現況：state/run-fsm.json 手動追蹤 Phase 狀態，無自動恢復
  • 改進：用 Temporal workflow-as-code 定義 3-Phase 流程，獲得自動 retry + durable execution

優化點 2：MCP 標準化工具整合

  • 改進：將 Todoist/Gmail/HackerNews API 包裝為 MCP servers，Agent 透過統一協定呼叫

優化點 3：LLM-as-Router 預分類

  • 現況：routing.yaml 靜態規則匹配 Todoist 標籤
  • 改進：用 LLM classifier（Groq Llama 3.3 70B）動態判斷任務類型，提升匹配準確度

八、風險與注意事項

8.1 技術風險

  1. 非確定性執行：LLM 輸出不穩定，相同輸入可能產生不同結果（需多次驗證取最優解）
  2. 工具呼叫失敗鏈：若 20 步驟的 Agent 工作流每步有 5% 失敗率，端到端成功率僅 35.8%（需降至 <1% per step）
  3. Context Window 限制：即使擴展到 200K tokens，仍無法一次載入大型程式碼庫（需 AST chunking）
  4. 記憶體洩漏：長時執行 Agent 若未限制 Memory size，可能 OOM（需實施 size limit + LRU 清理）

8.2 成本風險

  • Token 消耗爆炸：複雜工作流可能累積數百萬 tokens（需預算上限與即時監控）
  • 工具呼叫費用：若整合第三方 API（如搜尋引擎），每次呼叫產生額外成本

8.3 安全風險

  • Prompt Injection：惡意使用者可能透過輸入操控 Agent 行為（需 input sanitization）
  • 工具濫用：Agent 若具備檔案系統或網路存取權限，可能被誘導執行危險操作（需 sandbox 隔離）
  • 資料洩漏：Memory 若儲存敏感資訊（API key、個資），需加密與存取控管
