Qdrant 向量數據庫完整指南 — 核心架構、HNSW 索引、性能調優與生產部署最佳實踐（2026）

技術概述

Qdrant 是用 Rust 構建的高性能開源向量數據庫，採用 HNSW（Hierarchical Navigable Small World）索引算法實現快速相似度搜索。它支援數十億向量的存儲與查詢，透過自定義存儲引擎（Gridstore）、SIMD 硬體加速、io_uring 異步 I/O 與先進量化技術，將內存佔用減少高達 97%。Qdrant 提供靈活的部署選項（本地 Docker、Kubernetes 自架、雲端託管、混合雲），支援 HTTP/gRPC 雙協議，並提供 Python、JavaScript、Rust、Go、.NET、Java 等多語言 SDK。2026 年，Qdrant 獲得 5000 萬美元 B 輪融資，持續演進為 Agentic AI 系統的核心基礎設施，重點強化 4-bit 量化、多租戶擴展、讀寫分離與企業級可觀測性。

一、核心架構設計

1.1 客戶端-服務器架構

Qdrant 採用客戶端-服務器模型，公開兩種協議：

  • HTTP RESTful API：適合人工調試與跨語言整合
  • gRPC：高性能二進制協議，適合高吞吐場景

官方客戶端庫覆蓋主流語言（Python、JavaScript/TypeScript、Rust、Go、.NET、Java），並提供 OpenAPI 規範供自動生成其他語言客戶端。

1.2 數據組織：Collections、Points、Payloads

| 概念 | 說明 |

|------|------|

| Collection | 命名的點集合（類似資料庫中的表），包含向量與元數據 |

| Point | 基本數據單元，由 ID（64-bit 整數或 UUID）、向量、Payload 組成 |

| Payload | 結構化元數據（JSON 格式），可建立索引供過濾查詢 |

| Segment | 底層存儲單元，自動後台優化合併，類似 LSM-tree 的 SSTables |

1.3 分片與副本

在分布式部署中，Collection 分割為 Shards（分片），每個 Shard 保有獨立的 Segments：

  • Automatic Sharding：一致性哈希自動分配點到不同分片
  • User-Defined Sharding (v1.7.0+)：多租戶場景下，按 shard_key 手動分配

每個 Shard 可配置多個 Replicas（副本），提供高可用性與負載均衡。

二、HNSW 索引深度解析

2.1 HNSW 算法原理

HNSW（Hierarchical Navigable Small World）是分層圖結構，透過多層導航加速近似最近鄰搜索：

  • 底層 (Layer 0)：包含全部向量，密集連接
  • 上層 (Layer 1-N)：稀疏節點，長距離跳躍

搜索從頂層開始，逐層下降至底層，最終找到最近鄰。

2.2 Qdrant 的 HNSW 創新

Qdrant 的核心創新是 Payload Indexed HNSW：

  • 傳統做法：先 HNSW 搜索取 top-k，再用 payload 過濾（後過濾，可能遺漏結果）
  • Qdrant 做法：在 HNSW 圖遍歷期間即時應用過濾條件（單次遍歷，精確匹配）

這使得 Qdrant 能在語義搜索階段直接整合結構化過濾，避免傳統雙階段方法的召回損失。

2.3 HNSW 關鍵參數

| 參數 | 作用 | 調優建議 |

|------|------|---------|

| m | 每個節點的連接數 | 預設 16，複雜數據可提升至 32-64（權衡：精度 ↑ 內存 ↑） |

| ef_construct | 建索引時訪問的鄰居數 | 預設 100，密集向量可用 200-500（權衡：索引質量 ↑ 建構時間 ↑） |

| hnsw_ef | 查詢時訪問的鄰居數 | 動態調整（低延遲場景用 64，高召回場景用 256+） |

調優原則：hnsw_ef 應 ≥ limit（返回結果數），且 hnsw_ef ≥ m。

三、存儲引擎與內存管理

3.1 Gridstore 自定義存儲引擎

Qdrant 的 Gridstore 是為向量數據優化的存儲層，特點：

  • 列式存儲：向量維度分開存儲，加速部分維度查詢
  • Delta Encoding：HNSW 圖壓縮技術，僅存儲值差異，減少內存佔用
  • mmap 零拷貝：直接將磁盤文件映射到內存，減少系統調用

3.2 三種內存策略

| 策略 | 配置 | 適用場景 |

|------|------|---------|

| 全內存 | 向量 + HNSW 全在 RAM | 低延遲高吞吐（<10ms P99），成本較高 |

| 量化 + Rescoring | 量化向量在 RAM，原始向量在磁盤 | 推薦：平衡性能與成本（~30ms P99） |

| 全磁盤 | 向量 + HNSW 全在磁盤（SSD/NVMe） | 超大規模數據集，可接受 50-100ms 延遲 |

3.3 量化技術

Qdrant 支援三種量化方式：

  1. Scalar Quantization (int8)：將 float32 壓縮為 int8，內存減少 75%，精度損失 <2%
  2. Product Quantization (PQ)：更激進壓縮，適合超高維向量（>1024）
  3. Binary Quantization：極限壓縮至 1-bit，適合粗排序場景

最佳實踐：


{
  "quantization_config": {
    "scalar": {
      "type": "int8",
      "quantile": 0.99,
      "always_ram": true
    }
  },
  "hnsw_config": {
    "m": 16,
    "ef_construct": 200
  }
}

四、性能調優完整指南

4.1 索引構建優化

批量數據導入：

  1. 初始建立 Collection 時設置 m=0（禁用 HNSW 索引）
  2. 批量插入所有向量（速度提升 3-5x）
  3. 使用 API 觸發一次性索引重建

# 禁用索引
curl -X PUT 'http://localhost:6333/collections/my_collection' \
  -H 'Content-Type: application/json' \
  -d '{"hnsw_config": {"m": 0}}'

# 批量插入後重建
curl -X POST 'http://localhost:6333/collections/my_collection/index'

4.2 磁盤 I/O 優化

Inline Storage (v1.16.0+)：

  • 將向量副本直接嵌入 HNSW 索引文件
  • 減少隨機磁盤讀取，查詢速度提升 20-40%
  • 權衡：存儲空間增加 3-4x

{
  "hnsw_config": {
    "on_disk": true,
    "inline_storage": true
  }
}

4.3 查詢優化

動態調整 hnsw_ef：


from qdrant_client import QdrantClient

client = QdrantClient("localhost", port=6333)

# 低延遲查詢（犧牲少量召回）
results = client.search(
    collection_name="my_collection",
    query_vector=[0.1, 0.2, ...],
    limit=10,
    search_params={"hnsw_ef": 64}
)

# 高召回查詢（可接受更高延遲）
results = client.search(
    collection_name="my_collection",
    query_vector=[0.1, 0.2, ...],
    limit=10,
    search_params={"hnsw_ef": 256}
)

4.4 Segment 配置

| 參數 | 說明 | 推薦值 |

|------|------|--------|

| segment_number | Segment 數量 | 等於 CPU 核心數（最小化延遲） |

| max_segment_size | 單個 Segment 最大點數 | 越大越好吞吐，越小越低延遲（預設 20K） |

低延遲場景：segment_number = CPU核心數，小 Segment

高吞吐場景：較少大 Segment，減少合併開銷

五、分布式部署架構

5.1 集群拓撲設計

生產環境推薦配置：

  • 節點數：≥3 個（允許 1 個節點故障）
  • 副本因子：≥2（每個 Shard 至少 2 個副本）
  • 分片數：2 × 節點數（預留擴展空間）

範例：3 節點集群，預期未來擴展至 12 節點：


{
  "shard_number": 12,
  "replication_factor": 2
}

5.2 分片策略

自動分片（一致性哈希）：


client.create_collection(
    collection_name="distributed_col",
    shard_number=12,
    replication_factor=2
)

用戶定義分片（多租戶場景）：


client.upsert(
    collection_name="multi_tenant",
    points=[
        {
            "id": 1,
            "vector": [0.1, 0.2],
            "payload": {"user_id": "tenant_A"}
        }
    ],
    shard_key_selector="tenant_A"  # 固定分片到特定 Shard
)

5.3 寫入一致性控制


{
  "write_consistency_factor": 2  // 至少 2 個副本確認後才返回成功
}
  • 1：最低延遲，但有數據丟失風險
  • replication_factor：強一致性，最高延遲
  • 推薦：replication_factor / 2 + 1（Quorum 寫入）

5.4 故障恢復

自動恢復流程：

  1. 節點故障 → 集群檢測到副本不足
  2. 從健康副本自動複製數據到存活節點
  3. 達到目標副本數後恢復正常

手動恢復選項：

  • Snapshot 恢復：從全量快照還原
  • WAL Delta 傳輸：僅同步最近變更（適合短暫故障）

六、生產環境最佳實踐

6.1 容量規劃

使用官方 Sizing Calculator（https://cloud.qdrant.io/calculator）：

  • 輸入：向量數量、維度、payload 大小、QPS
  • 輸出：推薦的 CPU、RAM、磁盤配置

經驗公式（int8 量化）：


RAM 需求 = (向量數 × 維度 × 1 byte) + (HNSW 圖 × 0.2 GB / 百萬向量)
磁盤需求 = 向量數 × 維度 × 4 byte × 1.5（包含索引與備份）

6.2 監控指標

| 指標 | 監控重點 | 告警閾值 |

|------|---------|---------|

| P99 Search Latency | 最壞情況延遲 | >100ms |

| Memory Usage | 防止 OOM | >85% |

| Disk IOPS | SSD 性能瓶頸 | <5000 IOPS |

| CPU Utilization | 查詢處理能力 | >80% |

Prometheus 整合：


# qdrant.yaml
telemetry:
  enabled: true
  prometheus_endpoint: /metrics

6.3 備份策略

推薦三層備份：

  1. Snapshots：每日全量快照（本地保留 7 天）
  2. Offsite Backup：每周上傳至 S3/GCS（保留 4 周）
  3. 副本：生產集群本身的 2+ 副本（實時保護）

測試恢復流程：


# 定期演練恢復（至少每月一次）
curl -X POST 'http://localhost:6333/collections/my_collection/snapshots/snapshot_2026_03_17/recover'

6.4 負載測試

混合讀寫測試（80% 讀 / 20% 寫）：


import concurrent.futures
import random

def run_search():
    client.search(...)

def run_upsert():
    client.upsert(...)

with concurrent.futures.ThreadPoolExecutor(max_workers=50) as executor:
    tasks = []
    for _ in range(1000):
        if random.random() < 0.8:
            tasks.append(executor.submit(run_search))
        else:
            tasks.append(executor.submit(run_upsert))

6.5 常見陷阱

| 陷阱 | 後果 | 解決方案 |

|------|------|---------|

| 跳過 Payload 索引 | 過濾查詢速度慢 10-100x | 對常用過濾欄位建立索引 |

| 單節點部署 | 無高可用性 | 至少 3 節點 + 2 副本 |

| 每客戶一個 Collection | 管理複雜度爆炸 | 用 shard_key 實現多租戶 |

| 忽略量化 | 內存成本過高 | 啟用 int8 量化 |

| 混用批量與實時任務 | 互相影響性能 | 分離資源池 |

7.2 混合檢索架構

Qdrant 支援密集向量 + 稀疏向量雙重檢索：

  • 密集向量：語義搜索（如 OpenAI embeddings）
  • 稀疏向量：BM25 關鍵字匹配

client.upsert(
    collection_name="hybrid_search",
    points=[
        {
            "id": 1,
            "vector": {
                "dense": [0.1, 0.2, ...],  # 語義向量
                "sparse": {"indices": [10, 25], "values": [0.8, 0.6]}  # BM25 稀疏向量
            },
            "payload": {"text": "..."}
        }
    ]
)

# 混合搜索
results = client.search(
    collection_name="hybrid_search",
    query_vector={
        "dense": [0.1, 0.2, ...],
        "sparse": {"indices": [10], "values": [1.0]}
    },
    limit=10
)

八、2026 年演進方向

Qdrant 的 2026 路線圖聚焦於：

  1. 4-bit 量化：進一步減少內存佔用（在 int8 基礎上再減 50%）
  2. 讀寫分離：專用只讀副本，降低主節點壓力
  3. 完全可擴展多租戶：動態 Shard 分配，支援數百萬租戶
  4. Block Storage 整合：直接對接 S3/GCS，無需本地磁盤
  5. Agent-Native 檢索：針對 Agentic AI 的工具調用優化

---

九、優缺點評估

優勢

  1. Rust 性能：零成本抽象 + 內存安全，單核性能領先 Python 實現 10-20x
  2. Payload 索引 HNSW：行業唯一在圖遍歷期間過濾的實現
  3. 靈活量化：支援 int8、PQ、Binary 三種量化，可混合使用
  4. 生產就緒：完整的分布式、監控、備份方案
  5. 開源透明：GitHub 21K+ Stars，活躍社群

劣勢

  1. Rust 生態：插件開發門檻較高（相比 Python 系統）
  2. Resharding 成本：自架版本修改 Shard 數需重建 Collection
  3. 稀疏向量支援較新：文檔與案例相對密集向量較少
  4. 托管版價格：Qdrant Cloud 成本高於自架（但省運維）

十、程式碼範例：完整 RAG 管線


from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct
from sentence_transformers import SentenceTransformer

# 1. 初始化
client = QdrantClient(url="http://localhost:6333")
model = SentenceTransformer('all-MiniLM-L6-v2')

# 2. 建立 Collection
client.create_collection(
    collection_name="rag_docs",
    vectors_config=VectorParams(size=384, distance=Distance.COSINE),
    quantization_config={"scalar": {"type": "int8", "always_ram": True}},
    hnsw_config={"m": 16, "ef_construct": 200}
)

# 3. 批量插入文檔
docs = [
    {"id": 1, "text": "Qdrant 是向量數據庫", "source": "doc1.md"},
    {"id": 2, "text": "HNSW 索引提供快速搜索", "source": "doc2.md"},
]

points = [
    PointStruct(
        id=doc["id"],
        vector=model.encode(doc["text"]).tolist(),
        payload={"text": doc["text"], "source": doc["source"]}
    )
    for doc in docs
]

client.upsert(collection_name="rag_docs", points=points)

# 4. 語義搜索 + Payload 過濾
query = "如何優化向量搜索性能？"
query_vector = model.encode(query).tolist()

results = client.search(
    collection_name="rag_docs",
    query_vector=query_vector,
    limit=5,
    query_filter={"must": [{"key": "source", "match": {"value": "doc2.md"}}]},
    search_params={"hnsw_ef": 128}
)

for hit in results:
    print(f"Score: {hit.score}, Text: {hit.payload['text']}")
