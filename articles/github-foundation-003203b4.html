核心內容摘錄：

  • HTTPX 在 Client／AsyncClient 層提供 兩個事件掛鉤：request（請求已組裝完畢、尚未送網路）與 response（已自網路取回、尚未交還呼叫端）。相較於在每個呼叫點手寫日誌，掛鉤把「記錄、監看、追蹤」收斂成客戶端全域策略，適合本專案對 localhost:3000 知識庫 API 與 ntfy.sh 推播這類固定基底 URL、高頻短請求的熱路徑做一致化稽核 [來源:Event Hooks - HTTPX,2026]。與「外層包裝每個 get/post」相比，掛鉤保證所有經由該 Client 實例出去的流量都會經過同一套邏輯；與 OpenTelemetry HTTPXClientInstrumentor 相比，掛鉤偏應用語意（例如寫入 X-Trace-Id、對 4xx 正文取樣），OTel 則偏分散式追蹤 span 生命週期 [來源:opentelemetry-instrumentation-httpx PyPI 專案說明,2026]。
  • 官方文件將 request 定義為「準備完成後、送出前」，response 為「網路返回後、交還前」；掛鉤可讀寫 Request／Response 物件，例如替 request.headers 加上時間戳欄位 [來源:Event Hooks - HTTPX,2026]；encode 倉庫內 Markdown 源與官網敘述一致，便於版本對照 [來源:encode/httpx docs advanced event-hooks.md,2026]。中文技術鏡像（W3Cschool、BookStack）重述同一語意，利於團隊內非英文母語成員對齊「何時可改 header、何時尚未觸網」[來源:httpx 事件挂接_w3cschool,2022][來源:书栈网 httpx v0.28 Event Hooks,2026]。
  • 官方 Note 明確寫：response 掛鉤觸發時，尚未決定是否讀取 body；若要在掛鉤內讀內文，必須呼叫 response.read() 或 Async 版 await response.aread() [來源:Event Hooks - HTTPX,2026][來源:encode/httpx docs advanced event-hooks.md,2026]。這與「回應是 byte stream、只能消費一次」的設計一致：掛鉤若提前消費或関閉串流，呼叫端再讀會得到 StreamClosed 類錯誤 [來源:GitHub Discussion #2364,2022]。
  • 討論 #2224 顯示：把 response.raise_for_status() 放在 response 掛鉤中時，若未先 read()，例外處理區塊讀 e.response.text 會遇到 ResponseNotRead／StreamClosed；解法是在 raise_for_status 前先 read()（Async 則 aread()）[來源:GitHub Discussion #2224,2022]。#2364 進一步指出：掛鉤內丟出例外會觸發 response.close()，導致「先 raise 再在 except 讀 body」模式失效；對巨型串流若一律 aread() 可能不現實，討論建議需要「讀完 body 後才執行的第三種掛鉤」或改寫 Client.send 層級行為 [來源:GitHub Discussion #2364,2022]。
  • AsyncClient 註冊的掛鉤必須是 async 函式，同步函式不會被正確 await，這點在官網第二個 Note 與中文譯介均被強調 [來源:Event Hooks - HTTPX,2026][來源:httpx 事件挂接_w3cschool,2022]。
  • PyPI 套件 opentelemetry-instrumentation-httpx 說明：HTTPXClientInstrumentor 可全域或單一 client 注入；亦支援 request_hook／response_hook／async_request_hook／async_response_hook，在 span 建立後與結束前回呼，參數為 transport 層原始 tuple（method, url, headers, stream, extensions）而非高階 Request 物件 [來源:opentelemetry-instrumentation-httpx PyPI 專案說明,2026]。實務上常見組合是：OTel 產生 trace／metric，httpx event_hooks 寫應用欄位或取樣日誌。
