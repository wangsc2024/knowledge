httpx Event Hooks 與 AsyncClient 可觀測性：request／response 稽核與 KB／ntfy 熱路徑

技術概述

HTTPX 在 Client／AsyncClient 層提供 兩個事件掛鉤：request（請求已組裝完畢、尚未送網路）與 response（已自網路取回、尚未交還呼叫端）。相較於在每個呼叫點手寫日誌，掛鉤把「記錄、監看、追蹤」收斂成客戶端全域策略，適合本專案對 localhost:3000 知識庫 API 與 ntfy.sh 推播這類固定基底 URL、高頻短請求的熱路徑做一致化稽核 [來源:Event Hooks - HTTPX,2026]。與「外層包裝每個 get/post」相比，掛鉤保證所有經由該 Client 實例出去的流量都會經過同一套邏輯；與 OpenTelemetry HTTPXClientInstrumentor 相比，掛鉤偏應用語意（例如寫入 X-Trace-Id、對 4xx 正文取樣），OTel 則偏分散式追蹤 span 生命週期 [來源:opentelemetry-instrumentation-httpx PyPI 專案說明,2026]。

核心概念與原理

1. 兩段生命週期與可修改性

官方文件將 request 定義為「準備完成後、送出前」，response 為「網路返回後、交還前」；掛鉤可讀寫 Request／Response 物件，例如替 request.headers 加上時間戳欄位 [來源:Event Hooks - HTTPX,2026]；encode 倉庫內 Markdown 源與官網敘述一致，便於版本對照 [來源:encode/httpx docs advanced event-hooks.md,2026]。中文技術鏡像（W3Cschool、BookStack）重述同一語意，利於團隊內非英文母語成員對齊「何時可改 header、何時尚未觸網」[來源:httpx 事件挂接_w3cschool,2022][來源:书栈网 httpx v0.28 Event Hooks,2026]。

2. 回應本體與串流：為何預設讀不到內文

官方 Note 明確寫：response 掛鉤觸發時，尚未決定是否讀取 body；若要在掛鉤內讀內文，必須呼叫 response.read() 或 Async 版 await response.aread() [來源:Event Hooks - HTTPX,2026][來源:encode/httpx docs advanced event-hooks.md,2026]。這與「回應是 byte stream、只能消費一次」的設計一致：掛鉤若提前消費或關閉串流，呼叫端再讀會得到 StreamClosed 類錯誤 [來源:GitHub Discussion #2364,2022]。

3. `raise_for_status` 置於掛鉤時的例外語意

討論 #2224 顯示：把 response.raise_for_status() 放在 response 掛鉤中時，若未先 read()，例外處理區塊讀 e.response.text 會遇到 ResponseNotRead／StreamClosed；解法是在 raise_for_status 前先 read()（Async 則 aread()）[來源:GitHub Discussion #2224,2022]。#2364 進一步指出：掛鉤內丟出例外會觸發 response.close()，導致「先 raise 再在 except 讀 body」模式失效；對巨型串流若一律 aread() 可能不現實，討論建議需要「讀完 body 後才執行的第三種掛鉤」或改寫 Client.send 層級行為 [來源:GitHub Discussion #2364,2022]。

4. Async 掛鉤必須為 coroutine

AsyncClient 註冊的掛鉤必須是 async 函式，同步函式不會被正確 await，這點在官網第二個 Note 與中文譯介均被強調 [來源:Event Hooks - HTTPX,2026][來源:httpx 事件挂接_w3cschool,2022]。

5. 與 OpenTelemetry instrumentation 的分工

PyPI 套件 opentelemetry-instrumentation-httpx 說明：HTTPXClientInstrumentor 可全域或單一 client 注入；亦支援 request_hook／response_hook／async_request_hook／async_response_hook，在 span 建立後與結束前回呼，參數為 transport 層原始 tuple（method, url, headers, stream, extensions）而非高階 Request 物件 [來源:opentelemetry-instrumentation-httpx PyPI 專案說明,2026]。實務上常見組合是：OTel 產生 trace／metric，httpx event_hooks 寫應用欄位或取樣日誌。

最佳實踐

  1. 稽核欄位在 request 掛鉤注入：在送往 KB／ntfy 前寫入 X-Task-Key、X-Run-Id（或專案既有 trace id），避免只在少數呼叫點設定而漏網；依據為掛鉤可安全修改 request 物件 [來源:Event Hooks - HTTPX,2026]。反例：在掛鉤內做阻塞型 DNS 反查或同步磁碟寫檔，會拖慢所有請求。
  1. 錯誤本體取樣：aread() 後再 raise_for_status：針對已知小 payload 的 JSON API（本專案 POST /api/import、ntfy JSON），在 response 掛鉤先 await response.aread() 再決定是否 raise_for_status，確保 except 區塊可讀 text／content [來源:GitHub Discussion #2224,2022]。反例：對下載型 URL 盲目 aread() 導致記憶體尖峰。
  1. 掛鉤只做 O(1) 觀測、重試仍交給 RetryTransport／tenacity：延續本專案既有 httpx 研究結論——網路層與 HTTP 狀態碼重試應分層；掛鉤內應避免再發次級 HTTP 請求以免造成遞迴 [來源:KB 筆記「httpx 傳輸層重試與連線池」,2026]。
  1. 單例 AsyncClient + 共用掛鉤：與「長生命週期 AsyncClient」一致，掛鉤註冊在建立 client 時一次完成，避免熱路徑重複組 dict [來源:KB 筆記「Python httpx 非同步 HTTP 客戶端生產實踐」,2026]。
  1. 與 OTel 並存時注意雙重 span：若同時使用 HTTPXClientInstrumentor 與自訂 transport 包裝，需參考社群 PR 討論避免「transport 被重複 instrument」造成重複 span（具體重複條件需對照執行環境與 otel 版本）⚠️待驗證。

陷阱與反例

  1. 掛鉤內 raise_for_status 但未 read：已於 #2224／官方 Note 交叉驗證；除錯時誤以為「掛鉤等同 try/except 外層」會踩雷 [來源:GitHub Discussion #2224,2022][來源:Event Hooks - HTTPX,2026]。
  1. 例外導致 body 永不可讀：#2364 描述掛鉤早退 + close() 使下游無法讀錯誤 HTML／JSON；規避：僅對小回應在掛鉤讀滿，或改在呼叫端用 response.read() 後再檢查 status [來源:GitHub Discussion #2364,2022]。
  1. Async 掛鉤誤用 def：會變成未 await 的 coroutine 或型別錯誤；中文文件亦單獨段落提醒 [來源:httpx 事件挂接_w3cschool,2022][來源:Event Hooks - HTTPX,2026]。

與本專案（daily-digest-prompt）關聯

  • tools/ 共用 KB client：在 skills/knowledge-query 建議的 POST /api/search/hybrid／POST /api/import 路徑上，以 AsyncClient(event_hooks=...) 統一寫入 task_key、耗時（time.perf_counter 差值）到 logs/structured 或既有 JSONL，銜接 post_tool_logger 的 trace 欄位；呼應 continuity 建議的「共用 async KB client」方向 [路徑:context/continuity/auto-task-tech_research.json]。
  • ntfy curl 遷移 httpx 時的稽核：CLAUDE.md 規定 JSON 檔 + curl；若改為 httpx，仍應在掛鉤記錄 priority、topic 摘要長度，對照 config/notification.yaml 3500 字軟上限，便於除錯推播失敗 [路徑:CLAUDE.md ntfy 區塊]。

程式碼範例（官方語意改寫為 async）


import time
import httpx

async def audit_request(request: httpx.Request) -> None:
    request.extensions.setdefault("audit", {})
    request.extensions["audit"]["t0"] = time.perf_counter()

async def audit_response(response: httpx.Response) -> None:
    await response.aread()
    t0 = response.request.extensions.get("audit", {}).get("t0")
    if t0 is not None:
        elapsed_ms = (time.perf_counter() - t0) * 1000
        # 寫入 JSONL：method, url, status_code, elapsed_ms, len(response.content)
    response.raise_for_status()

async def make_client() -> httpx.AsyncClient:
    return httpx.AsyncClient(
        timeout=httpx.Timeout(30.0, connect=5.0),
        event_hooks={"request": [audit_request], "response": [audit_response]},
    )

來源：綜合 [Event Hooks - HTTPX,2026] 與 [GitHub Discussion #2224,2022]（aread 順序）。
