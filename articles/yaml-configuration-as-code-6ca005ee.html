YAML 配置驅動架構完整指南（2026）

技術概述

YAML 配置驅動架構（Configuration-Driven Architecture）是一種將系統行為從程式碼中抽離至結構化配置文件的軟體設計模式。核心理念是「改配置不改程式碼」——透過修改 YAML 檔案即可調整系統行為，無需重新部署或修改邏輯層。此模式廣泛應用於 CI/CD 管線（GitHub Actions）、基礎設施即程式碼（Terraform、Ansible）、API 閘道（Kong）、AI Agent 編排系統等領域。在 2025-2026 年，隨著 LLM Agent 系統的興起，YAML 配置驅動架構更成為 Prompt-as-Code 與 Agent 行為管理的事實標準。

---

核心概念與原理

一、Configuration as Code（CaC）的三大支柱

  1. 聲明式描述（Declarative）：配置描述「期望狀態」而非「操作步驟」。例如 daily_limit: 5 宣告限制，而非 if count >= 5 then skip
  2. 關注點分離（Separation of Concerns）：邏輯層（程式碼）負責「如何做」，配置層（YAML）負責「做什麼/做多少」
  3. 版本控制友善：YAML 是純文字格式，天然適合 Git 差異比對（diff-friendly），變更歷史清晰可追溯

二、YAML 相較於其他配置格式的優劣

| 維度 | YAML | JSON | TOML | HCL |

|------|------|------|------|-----|

| 人類可讀性 | 極佳（縮排結構） | 中等（大括號巢狀） | 佳（INI 風格） | 佳（專為 IaC 設計） |

| 註釋支援 | 是（#） | 否 | 是（#） | 是（// #） |

| 複雜結構 | 佳（錨點 & / 引用 *） | 無 | 有限 | 佳 |

| 生態系統 | 極廣（K8s、CI/CD、Ansible） | 極廣（API、前端） | 中等（Rust/Go） | 中等（Terraform） |

| 常見陷阱 | 縮排敏感、型別推斷 | 冗長、無註釋 | 深巢狀笨拙 | 學習曲線 |

| LLM 理解度 | 極高（Markdown-like） | 高 | 中 | 中 |

三、配置驅動架構的層次模型


Level 0：硬編碼（Magic Numbers in Code）
  ↓ 抽出常量
Level 1：常量集中化（Constants File）
  ↓ 結構化
Level 2：結構化配置（YAML/JSON Config）
  ↓ 加 Schema 驗證
Level 3：型別安全配置（Schema-Validated Config）
  ↓ 加動態載入
Level 4：聲明式管線（Declarative Pipeline）
  ↓ 加版本化演進
Level 5：配置即契約（Config as Contract）

設計模式與最佳實踐

模式 1：配置-邏輯分離（Config-Logic Separation）

核心原則：程式碼是「引擎」，配置是「燃料」。引擎不變，換燃料改行為。

實作範例（daily-digest-prompt 的 config/frequency-limits.yaml）：


tasks:
  shurangama:
    name: "楞嚴經研究"
    daily_limit: 5
    counter_field: "shurangama_count"
    template: "templates/auto-tasks/shurangama-research.md"
    execution_order: 1

程式碼（PowerShell 腳本）讀取 YAML，依 daily_limit 判斷是否執行，依 execution_order 決定順序。新增一個自動任務只需在 YAML 加一段，不改腳本。

效益：

  • 新增/移除任務零程式碼改動
  • 非工程師也能調整任務頻率
  • Git blame 清楚顯示「誰在何時改了哪個任務的頻率」

模式 2：按需載入（Lazy Loading Config）

問題：將所有配置在啟動時載入 Context Window，浪費 LLM Token。

解法：Prompt 是「薄層調度器」，僅在需要時 Read 對應配置。

daily-digest-prompt 實踐：

  • 主 Prompt（~80 行）只定義步驟骨架
  • 每步執行時才讀取 config/pipeline.yaml、config/cache-policy.yaml 等
  • 子 Agent 模板從 templates/sub-agent/ 按需載入

模式 3：Schema 驗證閘門（Schema Validation Gate）

問題：YAML 沒有型別系統，拼錯欄位名稱不會報錯。

解法：獨立的 Schema 驗證工具在部署前檢查。

daily-digest-prompt 實踐：

  • hooks/validate_config.py：7 個 YAML 的 Schema 定義
  • 驗證項目：必要欄位存在、型別正確、交叉引用一致（如 routing.yaml 的標籤對應 SKILL.md 的存在性）
  • 可由 check-health.ps1 呼叫或獨立執行

常見 Schema 驗證策略：

| 策略 | 工具 | 適用場景 |

|------|------|----------|

| JSON Schema | jsonschema (Python) | 通用，YAML 轉 dict 後驗證 |

| Pydantic | pydantic (Python) | 型別安全 + 預設值 |

| yq + jq | CLI | 輕量腳本驗證 |

| CUE | cue lang | 進階約束邏輯 |

| 自訂 Python | 手寫 | 複雜交叉驗證 |

模式 4：配置繼承與覆寫（Config Inheritance）

問題：多環境（dev/staging/prod）需要不同配置。

解法：基礎配置 + 環境覆寫。


# base.yaml
cache_ttl: 30min
max_retries: 3

# prod.yaml (覆寫)
cache_ttl: 60min

daily-digest-prompt 實踐：

  • config/hook-rules.yaml 有 3 個 preset（strict/normal/permissive）
  • 環境變數 HOOK_SECURITY_PRESET 選擇預設集
  • 團隊腳本預設 strict，互動開發可用 normal

模式 5：單一定義處（Single Source of Truth）

原則：每個可配置值只在一處定義。

反模式：


# routing.yaml
max_tasks: 3

# scoring.yaml
max_tasks_per_run: 3  # 同一概念，兩處定義 → 不一致風險

正確做法：一個 YAML 定義，其他引用。或用共用前言（templates/shared/preamble.md）統一規則。

常見陷阱與反模式

陷阱 1：YAML 的「挪威問題」（Norway Problem）

YAML 1.1 會將某些值自動轉型：


country: NO      # 被解析為布林值 false！
version: 1.0     # 被解析為浮點數 1.0，而非字串 "1.0"
port: 0x1A       # 被解析為整數 26
date: 2026-03-04 # 被解析為日期物件

修正：始終用引號包裹可能被誤判的值：country: "NO"、version: "1.0"

陷阱 2：配置膨脹（Config Bloat）

把所有邏輯塞進配置，YAML 變成另一種程式語言。

判斷標準：如果配置需要 if/else 分支或迴圈 → 應放在程式碼層。

陷阱 3：缺乏遷移策略（No Migration Plan）

配置 Schema 改版時，舊格式檔案會壞掉。

最佳實踐：

  • 配置加 version 欄位
  • 程式碼依 version 做向下相容處理
  • 提供 migration 腳本

陷阱 4：機密洩露（Secrets in Config）

絕對禁止在 YAML 中放入 API Token、密碼。

正確做法：環境變數 + 配置引用（token: ${TODOIST_API_TOKEN}）

架構效益量化

  1. 改動範圍最小化：調整自動任務頻率只需改 1 個 YAML 欄位（vs 修改 PS 腳本 + prompt + 多處 hardcode）
  2. LLM Token 節省：薄層 Prompt（~80 行）vs 全配置內嵌 Prompt（~500+ 行），節省 ~80% Token
  3. 一致性保證：validate_config.py 的 7 個 Schema 驗證確保配置合法性
  4. 版本追溯：所有配置 Git 管理，git log config/frequency-limits.yaml 追蹤變更歷史
  5. 分工明確：PowerShell 腳本 = 引擎（穩定），YAML = 燃料（頻繁調整）

演進方向

  1. Schema 驗證 100% 覆蓋：目前 7/12 有 Schema，目標 12/12
  2. 配置差異告警：Git Hook 偵測 YAML 變更並 ntfy 通知
  3. 配置即契約（Level 5）：YAML 變更自動觸發回歸測試
  4. CUE 升級：複雜交叉驗證從 Python 遷移至 CUE 語言

程式碼範例：PyYAML + Schema 驗證


import yaml
from pathlib import Path

def load_config(config_path: str, schema: dict = None) -> dict:
    """載入 YAML 配置並可選 Schema 驗證"""
    path = Path(config_path)
    if not path.exists():
        raise FileNotFoundError(f"Config not found: {config_path}")
    
    with open(path, 'r', encoding='utf-8') as f:
        config = yaml.safe_load(f)
    
    if schema:
        validate_schema(config, schema)
    
    return config

def validate_schema(config: dict, schema: dict):
    """簡易 Schema 驗證：檢查必要欄位與型別"""
    for field, expected_type in schema.get('required_fields', {}).items():
        if field not in config:
            raise ValueError(f"Missing required field: {field}")
        if not isinstance(config[field], expected_type):
            raise TypeError(
                f"Field '{field}' expected {expected_type.__name__}, "
                f"got {type(config[field]).__name__}"
            )

# 使用範例
frequency_schema = {
    'required_fields': {
        'version': int,
        'tracking_file': str,
        'tasks': dict,
    }
}

config = load_config('config/frequency-limits.yaml', frequency_schema)
print(f"Loaded {len(config['tasks'])} auto-tasks")
