LiteLLM — 統一 LLM 閘道與多後端路由深度研究報告

執行摘要

本文主張：LiteLLM（BerriAI/litellm）以 OpenAI 相容 API 將 100+ LLM 供應商收斂為單一代理層，在「多後端切換、成本歸因、故障轉移」上與本專案 config/llm-gate.yaml + frequency-limits.yaml 的設計目標高度重疊，但兩者層級不同——LiteLLM 是 HTTP/SDK 層的供應商抽象，本專案 Gate 是 PowerShell 層的 CLI 編排與品質閘 [來源:TrueFoundry LiteLLM vs LangChain,2026][來源:本專案 config/CLAUDE.md,2026]。截至 2026-06-02（GitHub API），Stars 48,975、Forks 8,540、Open Issues 3,547、pushed_at 2026-06-02，近 90 天持續活躍 [來源:GitHub API BerriAI/litellm,2026-06-02]。

---

一、專案概述

LiteLLM 解決的核心問題是：應用與平台團隊在接入 OpenAI、Anthropic、Azure、Bedrock、Vertex、Ollama、vLLM 等供應商時，不必為每家 API 格式、認證、計價單位重寫客戶端 [來源:BerriAI/litellm README,2026]。目標使用者包括：需要集中管控 API Key 與預算的 ML Platform 團隊、要在多雲/多區部署間做負載均衡的後端工程師、以及希望用單一 base_url 切模型的應用開發者 [來源:docs.litellm.ai Proxy 文件,2026]。

與 LangChain 的差異：LangChain 負責 Agent/Chain/RAG 編排；LiteLLM 負責「呼叫哪個模型、花多少錢、掛了換誰」——生產環境常並用而非二選一 [來源:Markaicode LiteLLM vs LangChain,2026][來源:TrueFoundry,2026]。

二、技術架構

2.1 雙模式：Python SDK vs Proxy Server

| 模式 | 部署形態 | 適用 | 關鍵能力 |

|------|----------|------|----------|

| Python SDK | pip install litellm，程式內 completion() | 單體腳本、快速 PoC | Router 重試/回退、應用層成本追蹤 |

| AI Gateway（Proxy） | litellm CLI 或 Docker，預設 :4000 | 團隊/企業、多語言客戶端 | Virtual Keys、Admin UI、Guardrails、多租戶預算 |

Proxy 對外暴露 /chat/completions、/embeddings、/models 等 OpenAI 相容端點；對內以 config.yaml 的 model_list 映射 model_name（別名）→ litellm_params（真實 provider/model/api_base） [來源:docs.litellm.com.cn quick_start,2026][來源:docs.litellm.ai model_management,2026]。

2.2 路由與可靠性（非僅 README 宣稱）

  • 負載均衡 / 延遲路由：可在同一 model_name 下掛多個 deployment，依延遲或成本策略分流 [來源:腾讯云 LiteLLM 网关教程,2026]。
  • Fallback：主模型失敗時自動切換備援（與本專案 llm-gate 的 quality_tier + allowed_adapters 語意相近，但實作層級在 HTTP 而非 PS1） [來源:HN Show HN liteLLM Proxy,2023][來源:本專案 llm-gate.yaml,2026]。
  • 效能：官方標稱 Proxy 在 1k RPS 下 P95 延遲約 8ms（基準見 docs.litellm.ai/benchmarks）；中文社群文件稱負載測試可達 1.5k+ RPS [來源:litellm README,2026][來源:docs.litellm.com.cn,2026]。

2.3 可觀測性與治理

  • 內建 spend tracking、virtual keys、團隊/預算（Enterprise 擴展 SSO/SCIM）[來源:docs.litellm.ai enterprise,2026]。
  • 可掛 Langfuse、MLflow、Prometheus 等 callback [來源:BerriAI README,2026]。
  • 內建 litellm_content_filter guardrail（regex/關鍵字，pre/post/during_call），與本專案 config/cursor-cli-architecture-guardrails.yaml 的「閘道層政策」概念類似但實作在 Proxy 內 [來源:docs.litellm.ai guardrails,2026]。

2.4 架構分層示意


Client (OpenAI SDK / curl / LangChain)
        │  OpenAI-compatible HTTP
        ▼
┌───────────────────────────────┐

三、功能特色與競品比較

| 維度 | LiteLLM | OpenRouter | Portkey | 本專案 llm-gate.ps1 |

|------|---------|------------|---------|---------------------|

| 自託管 | ✅ OSS Proxy | ❌ 雲端 | 部分 | ✅ 本機 PS1 |

| 供應商數 | 100+ | 多（市集） | 多 | 依 frequency-limits backends |

| 成本追蹤 | ✅ Virtual Key/團隊 | 帳單整合 | ✅ | 日誌 + token 工具鏈 |

| Agent 編排 | ❌（需 LangChain 等） | ❌ | 部分 | ✅ Todoist Team Phase |

| CLI 編排 | ❌ | ❌ | ❌ | ✅ claude/cursor/codex spawn |

| 本地模型 | Ollama/vLLM 一級 | 弱 | 中 | 透過各 backend 腳本 |

差異化亮點：① 單一 OpenAI 格式覆蓋最多供應商；② Proxy 模式讓非 Python 服務零改動接入；③ /model/new 熱新增模型無需重啟 Proxy [來源:docs.litellm.ai model_management,2026]。

劣勢（社群與評測）：生產部署常需 Docker + Postgres + Redis，運維重量高於「三行啟動」的輕量閘道 [來源:DEV Lynkr vs LiteLLM 2026,2026]；程式庫體量大，審計成本高 [來源:DEV LiteLLM got hacked,2026]。

四、使用方式

4.1 快速啟動 Proxy


uv tool install 'litellm[proxy]'
litellm --model gpt-4o

4.2 以 OpenAI SDK 指向 Proxy


from openai import OpenAI
client = OpenAI(api_key="sk-your-virtual-key", base_url="http://127.0.0.1:4000")
resp = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "你好"}],
)
print(resp.choices[0].message.content)

4.3 config.yaml 多部署範例


model_list:
  - model_name: gpt-4o
    litellm_params:
      model: azure/my-deployment
      api_base: https://xxx.openai.azure.com/
      api_key: os.environ/AZURE_API_KEY
  - model_name: local-qwen
    litellm_params:
      model: ollama/qwen3:8b
      api_base: http://localhost:11434

六、潛在應用（daily-digest-prompt）

場景 1：統一 OpenAI 相容後端給 `openrouter_runner`

落地：在 config/frequency-limits.yaml 中將部分 backend 的 HTTP 端點改指向本機 LiteLLM Proxy（http://127.0.0.1:4000），由 Proxy 的 model_list 管理 Groq/OpenRouter/Azure 真實 Key，應用層只持 Virtual Key。步驟：① Docker 部署 ghcr.io/berriai/litellm-database；② 匯入 model_list；③ Invoke-LlmGate 測試 profile 指向 Proxy；④ 記錄 spend 到現有 token 工具鏈。

場景 2：Phase 2 多任務的成本歸因

落地：為每個 task_key 建立 LiteLLM Virtual Key + 預算上限，對齊 config/frequency-limits.yaml 的配額治理。步驟：Admin UI 建 Key → 在 run-todoist-agent-team.ps1 注入環境變數 OPENAI_API_BASE → 比對 state/token-efficiency 與 Proxy 儀表板。

場景 3：本地 vLLM/Ollama 與雲端混合回退

落地：結合已研究的 SGLang/vLLM 推理層，LiteLLM 作為統一入口：本地 deployment 優先，失敗 fallback 至 Claude/OpenRouter。步驟：config.yaml 設定 fallback 鏈 → 與 llm-gate 的 quality_gate_blocked 策略對照測試。

七、優缺點評估

優點

  1. 供應商覆蓋面業界領先，降低「換模型改程式」成本 [來源:BerriAI README,2026]。
  2. Proxy + Virtual Key 適合多租戶與成本中心分攤 [來源:腾讯云教程,2026]。
  3. 與 LangChain/Langfuse 生態整合成熟 [來源:TrueFoundry,2026]。
  4. 支援熱新增模型，利於實驗新模型而不動客戶端 [來源:model_management docs,2026]。

缺點（有來源）

  1. 2026-03 供應鏈攻擊顯示 PyPI 直裝路徑風險，必須 pin 版本並優先 Docker 映像 [來源:HN 47501426,2026]。
  2. 生產拓樸複雜（DB/Redis），小團隊可能 overkill [來源:DEV Lynkr comparison,2026]。
  3. 程式庫規模大、審計成本高，與本專案「機械優先、薄依賴」原則需權衡 [來源:DEV 5 issues 2026,2026]。
  4. 不取代 Agent Team 編排；無法替代 run-todoist-agent-team.ps1 的 Phase/守衛邏輯。

不適用情境：僅用單一 Claude Code CLI、無多供應商切換需求；或必須極簡可審計閘道（<500 行）時，應維持現有 llm-gate.ps1 而非再疊一層 Proxy。
