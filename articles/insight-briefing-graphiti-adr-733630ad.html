深度研究洞察簡報：Graphiti 時態知識圖譜 × ADR Stale Premise 精準度提升

日期：2026-06-29

主題：評估 getzep/graphiti 追蹤 ADR registry invalidated_at，改善 self_heal stale premise 精準度

透鏡：distill-sam-altman（主）、distill-jensen-huang（主）、distill-taleb（對抗）

摘要

DDP 的 self_heal 機制反覆引用已 Deferred 或 Superseded 的 ADR 作為修復前提（stale premise），導致 self_heal 連敗後才升級至 arch_evolution，消耗大量 LLM token。核心發現：write-time invalidation 比 LLM query-time stale 偵測精準（68% vs 55.2%）。最小可行應用路徑：先機械過濾 Deferred/Superseded ADR（零成本），再引入 Graphiti FalkorDB Lite embedded 批次時態追蹤。

關鍵洞見（7 項）

  1. DDP self_heal 的根本問題是 write-time vs query-time 架構選擇。STALE 論文（arXiv 2605.06527）實證：LLM query-time stale 識別精度僅 55.2%；CUPMem write-time adjudication 提升至 68%。
  1. Graphiti 雙時態模型（t_valid/t_invalid/t'_created/t'_expired）與 DDP adr-registry.json 的 created_at/decided_at/review_due/status 有直接對應。
  1. adr-registry.json 中 1281 筆 Deferred、161 筆 Superseded，其 fix_instructions 被 self_heal 引用，是 Type II 傳播衝突的系統性來源。
  1. Graphiti v0.29.2 支援 FalkorDB Lite embedded mode（Python 3.12+），uv add graphiti-core[falkordblite] 零外部基礎設施啟動。
  1. 每個 episode 攝入需 LLM 呼叫，DDP 每日 ADR 異動 <10 筆，批次攝入成本估算 <$0.01/日。
  1. Graphiti 混合檢索 P95 300ms，無 LLM 呼叫，可作為 self_heal Phase 0 零成本前置守衛。
  1. Entity resolution 可消除 ADR 命名漂移導致的 dedup 失效（source_pattern 命名不一致問題）。

多透鏡洞察

Sam Altman 視角

先 ship 再迭代：先以 FalkorDB Lite 嵌入跑 2 週驗證，測量 self_heal 引用 Superseded/Deferred ADR 頻率。批次模式（每日攝入 ADR 異動 diff）是唯一可接受的成本路徑。

Jensen Huang 視角

Workload first：DDP 每日 ADR 異動 <10 筆，批次 TCO 遠低於 real-time。Full-stack：graphiti-core[falkordb] + uv + 日排程，不引入 Neo4j server。

Taleb 對抗視角

Via Negativa：先移除讓 stale premise 存活的機制（依賴 LLM query-time 推理），而非新增複雜度。機械過濾 Deferred/Superseded ADR 是最先做的事。review_due 欄位是火雞安慰劑——沒有機制確認 review 實際發生。

建議行動（Ponytail 決策階梯）

  • 階梯①：YAGNI 驗證——統計 Deferred/Superseded ADR 被引用比例，<5% 機械過濾即可
  • 階梯②：機械過濾——arch_decision_step_h.py pre-flight validate 加入 status in ['Deferred','Superseded','Wontfix'] 過濾，zero cost
  • 階梯⑤：Graphiti MVP——uv add graphiti-core[falkordblite]，tools/graphiti_adr_probe.py，每日批次攝入，kill switch DDP_GRAPHITI_ADR=off
