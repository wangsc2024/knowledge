AI系統開發持續整合software-engineering

AI 驅動的持續整合進階實作：Flaky Test 根因分類、隔離治理與預測式選測（2026）

技術概述

AI 輔助 CI 已從單純的測試重跑，進化成三段式閉環：先用歷史執行資料做預測式選測，縮小每次提交的測試範圍；再用 flaky detection 找出同一 commit 上忽通忽斷的測試；最後把不可靠測試隔離、標記、通知負責團隊，並持續追蹤修復後是否回穩。實務上，這不是單一模型問題，而是「資料蒐集、風險分類、管線策略、團隊責任」一起設計。對 daily-digest-prompt 這類自動化腳本專案，價值在於降低排程假失敗、縮短回歸驗證時間，並把 CI 噪音轉成可治理的品質訊號。

核心方法與工具

1. 預測式選測（Predictive Test Selection）

  • Launchable 以歷史測試結果與程式碼變更做機器學習選測，讓 test runner 只跑高風險子集。
  • 適合 PR 驗證、pre-merge 快速守門、夜間全量測試搭配。
  • 關鍵輸入：commit diff、過往失敗分佈、測試耗時、測試名稱穩定性。

2. Flaky test 偵測

  • Buildkite Test Engine 以「同一 commit SHA 上同一測試同時出現 pass/fail」定義 flaky。
  • 若最近 100 次執行或 7 天內恢復穩定，會移除 flaky 標記。
  • GitLab 的 unhealthy tests 文件則進一步把根因分成 state leak、dataset-specific、random input、unreliable DOM selector、datetime-sensitive、unstable infrastructure、improper synchronization 等類別。

3. 隔離與修復治理

  • Buildkite 建議把 flaky test 轉為 muted 或 skipped；muted 較佳，因為仍保留執行資料。
  • 自動動作可包含：加標籤、Slack 通知、建立 Linear/Jira issue、恢復穩定後自動解除隔離。
  • Google 的大規模經驗說明：只靠 retry 不夠，還要有 quarantine、變動監控與 flakiness level 追蹤。

實際工作流程（step-by-step）

  1. 收集測試資料

在 CI 上傳每次測試結果、commit SHA、測試名稱、耗時、重跑次數。

  1. 建立快速路徑

PR pipeline 先跑預測式選測子集；主幹或夜間排程跑全量測試補足 coverage。

  1. 偵測 flaky

針對同一 commit 若同一測試同時 pass/fail，立即標記為 flaky candidate。

  1. 做根因分類

依 GitLab 分類規則分到 state leak、同步問題、隨機輸入、時間敏感、基礎設施不穩等桶。

  1. 套用隔離政策

高干擾測試先 mute，保留遙測；極高成本且已知無法快速修復者才 skip。

  1. 建立責任閉環

自動通知 owner、建立 issue、附上最近失敗樣本、seed、log、變更 commit。

  1. 驗證恢復

當最近 7 天或一定執行次數內恢復一致結果，自動移除 flaky 標記並重新納入 gate。

程式碼 / 配置範例

範例 1：GitHub Actions 中先跑快速子集，再做 flaky 分流


name: ci
on:
  pull_request:
  push:
    branches: [main]

jobs:
  test-fast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install -r requirements-dev.txt
      - name: Run selected tests
        run: pytest tests/unit tests/integration -q --maxfail=1 --junitxml=reports/junit.xml
      - name: Upload results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: junit-fast
          path: reports/junit.xml

  flaky-triage:
    if: always()
    needs: [test-fast]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4
        with:
          name: junit-fast
          path: reports
      - run: python scripts/flaky_triage.py reports/junit.xml

範例 2：本機簡化版 flaky 分析器


from collections import defaultdict
import xml.etree.ElementTree as ET

def summarize_junit(path: str):
    tree = ET.parse(path)
    seen = defaultdict(set)
    for case in tree.iterfind('.//testcase'):
        key = f"{case.get('classname')}::{case.get('name')}"
        status = 'failed' if list(case) else 'passed'
        seen[key].add(status)
    return {k: sorted(v) for k, v in seen.items() if len(v) > 1}

if __name__ == '__main__':
    suspects = summarize_junit('reports/junit.xml')
    for test_name, states in suspects.items():
        print(f"FLAKY {test_name} => {states}")

效果量化

  • Google 公開表示，其整體測試執行中約 1.5% 會出現 flaky 結果，接近 16% 的測試具某種程度 flakiness，而 pass→fail 轉換中約 84% 最後被證實與 flaky 有關。這說明若不做治理，CI 告警大多是噪音。
  • Buildkite 文件指出，test splitting 可把 4 個 agent 的總執行時間從 10 分鐘降到約 4 分鐘；其 flaky quarantine 又能避免重試造成的額外浪費。
  • Launchable 的做法不直接保證覆蓋率最大化，而是優先把「最可能抓到變更風險」的測試拉進前景，實務上適合把 PR 等待時間壓到可接受區間。

與本專案的應用場景（daily-digest-prompt 如何受益）

  • 對 PowerShell / Python 驗證腳本建立 junit 或 JSONL 結果輸出，累積每次自動任務的穩定性資料。
  • 把 results/*.json 驗證、prompt lint、設定檔檢查拆成快速守門集與夜間完整集，降低每次修改的等待時間。
  • 對偶發失敗的排程任務建立 muted-but-observed 名單，避免單次網路抖動或本機服務未啟動時污染品質趨勢。
  • 將 flaky 類型對映到專案語境：API 連線不穩歸 infrastructure、時間依賴歸 datetime-sensitive、順序問題歸 state leak / order dependent。

風險與注意事項

  • 只做 retry 會掩蓋真實缺陷，不能把「重跑後過了」當成修復。
  • skip 會失去觀測資料，除非成本極高，否則先用 mute 比較穩健。
  • 預測式選測需要足夠歷史資料；資料品質差時，先把它當加速層，而不是唯一 gate。
  • flaky 分類需要把 commit、seed、測試檔、執行環境版本一起存下來，否則根因追不回去。
