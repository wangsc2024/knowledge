MoE Router 機制深度解析：Gating Network、Top-K 選擇與負載均衡策略

系列關聯：承接「MoE（Mixture of Experts）混合專家架構 — 2025-2026 年最新進展與大規模部署實踐」（foundation），本文進入 mechanism 階段，深入探討 Router 演算法的數學原理與實作細節。

一、Router 架構：簡約而不簡單

1.1 核心設計

Router（也稱 Gating Network）是 MoE 的流量控制核心，決定每個 token 應該由哪些專家處理。令人驚訝的是，現代 MoE 的 Router 設計極為簡約：一個不帶 bias 的線性層。

數學表達：


G(x) = xW_g

其中：

  • x：輸入 token 的 hidden state（維度 d）
  • W_g：Router 權重矩陣（d × n，n 為專家數量）
  • 輸出：n 個專家的路由分數

1.2 為什麼是單層線性，而非多層 MLP？

四大理由支持這個反直覺的選擇：

  1. 計算效率：Router 對每個 token 執行，必須極度輕量
  2. 輸入已足夠豐富：x 來自 post-attention 層，已包含完整上下文資訊
  3. 訓練穩定性：更深的 Router 容易導致梯度不穩定和專家崩潰（Expert Collapse）
  4. 可解釋性：W_g 的每一列可視為專家的「原型向量」，便於分析專家特化模式

輸入來源：x 取自 Attention 層之後、經過 LayerNorm 的 hidden state，確保 Router 能基於完整上下文做決策。

二、Top-K 路由：從分數到稀疏激活

2.1 三步驟路由流程

現代 MoE 使用 Noisy Top-K Gating，分為三個步驟：

#### Step 1：添加可調噪聲


H(x)_i = (x · W_g)_i + StandardNormal() × Softplus((x · W_noise)_i)
  • W_noise：可學習的噪聲權重矩陣
  • 噪聲有助於探索（Exploration），防止過早收斂到次優路由策略

#### Step 2：保留 Top-K 值


KeepTopK(v, k)_i = {
  v_i,     if v_i is in top k elements
  -∞,      otherwise
}

將非 Top-K 的專家分數設為 -∞，確保 Softmax 後權重為 0。

#### Step 3：Softmax 歸一化


G(x) = Softmax(KeepTopK(H(x), k))

最終得到 k 個專家的權重，總和為 1.0。

2.2 溫度縮放（Temperature Scaling）

可引入溫度參數 τ 控制路由決策的「尖銳度」：


G(x) = Softmax(KeepTopK(H(x), k) / τ)
  • 低溫（τ < 1）：強化差異，產生更果斷的路由（一個專家獨大）
  • 高溫（τ > 1）：平滑分布，鼓勵探索多個專家

2.3 常見 K 值選擇

| K 值 | 代表模型 | 特點 |

|------|---------|------|

| K=1 | Switch Transformer | 極致稀疏，每 token 僅激活 1 個專家 |

| K=2 | Mixtral 8x7B | 業界主流，平衡品質與效率 |

| K=4-8 | GShard、GLaM | 高品質場景，犧牲部分稀疏性 |

三、負載均衡：防止專家崩潰

3.1 問題本質

Router 透過梯度學習最優路由，但存在根本性盲點：未被選中的專家收不到梯度訊號，無法知道自己是否能做得更好。這導致「富者愈富」的馬太效應，最終僅少數專家被使用（Expert Collapse）。

3.2 傳統方案：Auxiliary Loss

引入輔助損失函數鼓勵均勻分配：


L_total = L_task + α × L_balance

Load Balancing Loss（Switch Transformer）：


L_balance = N × Σ(f_i × P_i)

其中：

  • N：專家數量
  • f_i：專家 i 處理的 token 比例
  • P_i：專家 i 被路由到的平均機率

缺點：α 難以調優，且引入「幹擾梯度」，與主任務梯度衝突，損害模型性能。

3.3 前沿方案：Loss-Free Balancing（2024）

核心思路：在 Top-K 選擇之前動態調整專家偏差（bias），無需額外損失函數。

演算法：

  1. 追蹤每個專家的近期負載（如最近 1000 個 token）
  2. 計算負載比例 load_i / avg_load
  3. 動態更新偏差：

```

bias_i = bias_i - η × (load_i / avg_load - 1)

```

- 若專家 i 過載（load_i > avg）→ 降低 bias，減少被選機率

- 若專家 i 閒置（load_i < avg）→ 提升 bias，增加被選機率

  1. 修正路由分數：

```

H_corrected(x)_i = H(x)_i + bias_i

```

實驗結果（arXiv:2408.15664）：

  • 在 3B 參數模型、200B token 訓練規模測試
  • 性能優於傳統 Auxiliary Loss
  • 負載分佈標準差降低 40%
  • 無幹擾梯度，訓練更穩定

DeepSeek-V3 實踐：採用此策略，實現無輔助損失的負載均衡，是當前（2026 年）業界最先進的方案之一。

四、Expert Capacity：容量上限與 Token Dropping

4.1 容量公式

為防止某個專家收到過多 token（超出記憶體／算力承載），設定 Expert Capacity：


Capacity = (tokens_per_batch / num_experts) × capacity_factor

參數說明：

  • capacity_factor（CF）：緩衝倍數，推薦 1.0 - 1.25
  • CF = 1.0：無緩衝，可能丟棄 token（Switch Transformer 實測品質仍優）
  • CF = 1.25：25% 緩衝，平衡品質與通訊成本

範例計算：


Batch size = 2048 tokens
Num experts = 8
CF = 1.25

Capacity = (2048 / 8) × 1.25 = 320 tokens/expert

4.2 Token Dropping 機制

當 Top-K 選中的所有專家都已達容量上限時，overflow token 的處理方式：

| 策略 | 實現 | 影響 |

|------|------|------|

| 殘差跳過 | 直接送入下一層（via residual connection） | 資訊保留，但專家未處理 |

| 直接丟棄 | 該 token 不參與此層計算 | 潛在資訊損失 |

實驗發現（ST-MoE 論文）：即使 11% token 被丟棄，微調（Fine-tuning）時模型品質未顯著下降 → Token Dropping 可能起到正則化作用，防止過擬合。

五、訓練動態與專家特化

5.1 梯度流動特性

  • Router 梯度：連續可微，透過 Softmax 權重傳遞
  • 專家梯度：僅被選中的 K 個專家收到梯度
  • 未選中專家：完全無梯度訊號 → 依賴負載均衡機制「強制輪換」

5.2 自發湧現的專家特化模式

實證研究發現，即使未人為設計，Router 也會學到有意義的分工模式：

| 層級 | 特化類型 | 範例 |

|------|---------|------|

| 早期層 | 句法特徵 | 詞性、句法結構 |

| 中層 | 語義領域 | 科技、醫學、法律等主題 |

| 深層 | 推理模式 | 因果推理、類比推理 |

觀察：越深的層，路由決策越「果斷」（Softmax 輸出更尖銳），專家分工更明確。

六、實作要點與配置建議

6.1 PyTorch 偽代碼（Top-2 Router）


class MoERouter(nn.Module):
    def __init__(self, d_model, num_experts, k=2):
        super().__init__()
        self.gate = nn.Linear(d_model, num_experts, bias=False)
        self.k = k
        
    def forward(self, x):
        # x: [batch, seq_len, d_model]
        logits = self.gate(x)  # [batch, seq_len, num_experts]
        
        # Top-K selection
        top_k_logits, top_k_indices = torch.topk(logits, self.k, dim=-1)
        
        # Softmax only on top-k
        weights = F.softmax(top_k_logits, dim=-1)
        
        return top_k_indices, weights  # [batch, seq_len, k]

6.2 關鍵超參數配置

| 參數 | 推薦值 | 說明 |

|------|--------|------|

| k | 2 | Top-K 專家數（業界主流） |

| capacity_factor | 1.0 - 1.25 | 容量緩衝倍數 |

| noise_std | 0.1 - 0.5 | 探索噪聲標準差 |

| balance_alpha | 0.01 - 0.1 | Auxiliary Loss 權重（若使用） |

| bias_lr | 0.001 | Loss-Free Balancing 的偏差學習率 |

6.3 DeepSeek-V3 配置參考（2026 年 SOTA）

  • 專家架構：Shared Experts（始終激活）+ Routed Experts（Top-K 選擇）
  • 細粒度專家：將 N 個專家擴展為 mN 個，每個專家維度縮減至 1/m，激活 m 個
  • Node-Limited Routing：限制每個 token 最多路由至 M 個計算節點（降低通訊成本）
  • Loss-Free Balancing：動態偏差調整，無 Auxiliary Loss

七、總結：Router 設計的藝術與科學

核心要點

  1. 架構簡約性：單層線性足以，過度複雜反而有害
  2. Top-K 稀疏性：K=2 是當前最佳實踐
  3. 負載均衡：Loss-Free Balancing 代表 2024-2026 年最新進展，優於傳統 Auxiliary Loss
  4. 容量控制：Expert Capacity 與 Token Dropping 是實務必需，低 CF 值（1.0-1.25）已足夠
  5. 專家特化：自發湧現，無需人為設計

未來方向

  • Expert Choice Routing：由專家選 token，而非 token 選專家（Google 2022 提出，仍在探索）
  • 多粒度專家：同時存在粗粒度（通用）和細粒度（專業）專家
  • 動態 K 值：根據任務複雜度自適應調整激活專家數
