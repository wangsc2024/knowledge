KV Cache 優化技術全景 — 2026 年 LLM 推理效率核心瓶頸的三層解法

概述

KV Cache（Key-Value Cache）是 LLM 推理的核心優化機制，透過儲存中間 Key-Value 對避免重複計算注意力權重，將計算複雜度從 O(n²) 降至 O(n)。然而隨著上下文長度爆發（1M tokens 已成標配），KV Cache 的記憶體佔用成為推理效率的首要瓶頸。2025-2026 年湧現大量前沿研究，形成 Token 層級 → Model 層級 → System 層級 的三層優化體系。

---

一、Token 層級優化（最活躍的研究方向）

1.1 KV Cache 選擇策略

靜態選擇（預判保留）：

  • SnapKV（NeurIPS 2024）：利用觀察窗口中的注意力模式，在生成前預判哪些 token 重要，一次性裁剪
  • 優點：零額外開銷；缺點：無法適應生成過程中的動態需求

動態選擇（永久驅逐）：

  • H2O（NeurIPS 2023）：Heavy-Hitter Oracle，追蹤累積注意力分數，驅逐低分 token
  • Keyformer（MLSys 2024）：結合 Gumbel-Softmax 採樣的 token 選擇
  • 缺點：被驅逐的 token 永遠無法恢復，可能丟失長距離依賴

動態選擇（無永久驅逐）：

  • InfLLM（2024）：將非活躍 token 移至 CPU/磁碟，需要時回調，支援無限上下文
  • Quest（ICML 2024）：Query-aware 稀疏注意力，按需選擇相關 KV 對

1.2 預算分配策略

層級預算分配（Layer-wise Budget）：

  • PyramidKV（2024）：金字塔式分配，底層保留多、頂層保留少，模擬資訊漏斗
  • SimLayerKV（2024）：識別「懶惰層」（注意力高度集中），大幅壓縮其 KV Cache
  • Entropy-Guided Caching（2026）：計算每層注意力權重的熵值，高熵層（注意力分散）分配更大預算，低熵層（sink-like 模式）縮減預算。使用聚合注意力分數識別共享重要 token 集合，保持多頭注意力完整性

頭級預算分配（Head-wise Budget）：

  • DuoAttention（2024）：區分「檢索頭」（需要完整 KV）和「串流頭」（只需近期 token）
  • HeadKV（2024）：按頭級別壓縮，整合推理能力評估

1.3 語義保留壓縮：ChunkKV（NeurIPS 2025 亮點）

核心創新：以語義區塊（chunk，預設 10 tokens）而非單獨 token 作為壓縮基本單位

技術細節：

  • 將連續 token 分組為固定大小的 chunk
  • 區塊重要性分數 = 區塊內所有 token 注意力分數之和：A_i = Σ(chunk i 內各 token 注意力分數)
  • 整個 chunk 要麼保留、要麼丟棄，保持主謂賓等語言結構完整

層間索引重用（Layer-wise Index Reuse）：

  • 相鄰 Transformer 層共享相同的保留索引
  • 重用深度為 2 時，KV Cache 壓縮時間減少 20%
  • 吞吐量提升 26.5%，延遲降低 20.7%

基準測試成績（10% 壓縮比）：

| 任務 | 模型 | ChunkKV | SnapKV | H2O |

|------|------|---------|--------|-----|

| LongBench | LLaMA-3-8B | -2.29% | -3.16% | -10.61% |

| LongBench | Qwen2-7B | +0.42% | -0.64% | -5.28% |

| NIAH (128) | LLaMA-3.1-8B | 73.8% | 58.9% | 47.9% |

| NIAH (128) | Mistral-7B | 99.8% | 91.6% | 88.2% |

| GSM8K | LLaMA-3-8B | 65.7% | 50.3% | 45.0% |

| GSM8K | LLaMA-3.1-8B | 77.6% | 68.8% | 64.0% |

1.4 合併策略

層內合併（Intra-layer）：

  • CaM（ICML 2024）：合併相似 KV 對而非丟棄
  • ZeroMerge（2025）：零參數合併，無需額外訓練
  • Dynamic Memory Compression（ICML 2024）：回溯式加速推理

跨層合併（Cross-layer）：

  • MiniCache（2024）：在深度維度壓縮
  • KVSharer（2024）：基於層間相異度的跨層共享

1.5 量化

  • FP8 量化：記憶體佔用減少 75%（LLaMA-3.1-8B），可儲存 3 倍 token
  • QJL（2024）：1-bit 量化的 Johnson-Lindenstrauss 變換
  • vLLM v0.11.2 支援 kv_cache_dtype="fp8_e5m2" + calculate_kv_scales=True
  • 校準方法：無校準（scale=1.0）、隨機 token 校準、資料集校準（最準確）

二、Model 層級優化

2.1 注意力架構改進

  • MQA（Multi-Query Attention）：所有頭共享一組 KV，記憶體大幅減少
  • GQA（Grouped-Query Attention）：折衷方案，分組共享 KV，LLaMA-2/3 採用
  • MLA（Multi-head Latent Attention）：DeepSeek-V2 採用，KV 投影到低維潛在空間

2.2 非 Transformer 架構

  • Mamba、RWKV 等線性注意力架構天然避免 KV Cache 問題
  • 混合架構（Transformer + SSM）在長上下文場景表現優異

---

三、System 層級優化

3.1 記憶體管理（vLLM PagedAttention）

  • Block Pool：預分配固定大小 GPU 記憶體區塊，雙向鏈結串列，O(1) 分配/驅逐
  • Free Block Queue：支援 LRU 和參考計數驅逐策略
  • Logical Block Tables：維護 hash key → block ID 映射，實現前綴共享和跨請求重用
  • LRU vs FIFO 權衡：LRU 增加 15% 記憶體開銷，但減少 30% cache miss

3.2 多層級動態儲存（MTDS）

  • 將 KV Cache 從 GPU VRAM 卸載到 CPU/NVMe 分層儲存
  • 選擇性重用機制：僅回調高重要性的 KV 對
  • 適用於記憶體受限環境的長上下文推理

3.3 企業級持久化（Pure Storage KVA）

  • NFS 後端：推理速度提升最高 20 倍
  • S3 後端：推理速度提升最高 6 倍
  • 支援跨 session 的 KV Cache 重用

3.4 生產效能基準

| 模型 | 後端 | 速度 | 延遲降低 |

|------|------|------|----------|

| Qwen 7B | NFS | 22,000 tokens/s | 95% |

| LLaMA-3.1 70B | S3 | 3,600 tokens/s | 85% |

| LLaMA-3.1 8B | FP8 | 15,000 tokens/s | 80% |

測試環境：NVIDIA A100 GPU、batch size 16、序列長度 8,192、8 節點多 GPU

四、實務部署建議

4.1 方案選擇決策樹

  1. 記憶體充足 + 短上下文（<8K）→ 標準 KV Cache + FP8 量化即可
  2. 記憶體受限 + 中等上下文（8-32K）→ SnapKV/ChunkKV 靜態壓縮
  3. 長上下文（32K-1M）→ InfLLM（CPU 卸載）+ PyramidKV 層級預算 + FP8
  4. 企業多用戶→ vLLM PagedAttention + Prefix Caching + LRU 驅逐
  5. 超長上下文 + 多輪對話→ MTDS + LoopServe 雙階段自適應

4.2 常見陷阱

  • 過度壓縮會破壞長距離推理（GSM8K 等數學任務最敏感）
  • Token-level 方法會切斷語義單元（ChunkKV 的 chunk-level 方案解決此問題）
  • LRU 驅逐策略在高併發下的記憶體開銷不可忽視（+15%）
  • FP8 量化需 CUDA 12.1+，低版本 GPU 不支援

---

五、2026 年趨勢展望

  1. 自適應動態策略：結合 entropy/variance/kurtosis 等統計量，per-layer per-head 動態調整
  2. 硬體-軟體協同設計：KV Cache 友好的加速器架構（如 Groq TSP 的 SRAM 優化）
  3. 混合方法整合：量化 + 選擇 + 合併的多技術融合成為標配
  4. Diffusion LM 的 KV Cache：EntropyCache（2026 年 3 月）將熵引導策略擴展到擴散語言模型
