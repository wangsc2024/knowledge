---

執行摘要

本方案基於 Martin Fowler 的 Harness Engineering 框架 與 Anthropic 的長期運行應用設計原則，對現有 Daily Digest Prompt 系統的 Harness 架構提出深度優化建議。

核心優化方向：

  1. Feedforward + Feedback 雙迴路優化 — 從被動修復轉向主動預防
  2. Context Management 升級 — 從摘要壓縮轉向結構化重置
  3. Multi-Agent 責任邊界精細化 — 消除 OODA vs Harness 重疊
  4. Computational-First 驗證策略 — 降低 LLM 驗證依賴

1. 現狀分析：已有基礎 vs 業界標準

1.1 現有系統成熟度矩陣

| 維度 | 現有評分 | 業界標準 | 差距 | 根本原因 |

|------|---------|---------|------|---------|

| 架構約束 | 8.5/10 | 9.5/10 | Feedforward 不足 | 約束多為事後攔截（Hook），缺乏事前引導（Guide） |

| 反饋迴路 | 9.0/10 | 9.5/10 | 質量分數未消費 | quality_score 計算後無下游影響（不影響排程/路由） |

| 驗證護欄 | 8.8/10 | 9.5/10 | LLM-validates-LLM | 品質門檻 30% 依賴 LLM 主觀判斷，無 Computational 對照組 |

| 生命週期管理 | 9.2/10 | 9.5/10 | 時間閾值散落 | 5+ 檔案各自定義 TTL，無 timing-policy.yaml 統一管理 |

1.2 業界標準對照

#### Martin Fowler 框架核心要素

  1. Feedforward（引導）：事前控制，提高首次輸出質量

- 現有實作：templates/shared/preamble.md、CLAUDE.md、Skill 指引

- 缺口：無「正向案例庫」（Good Examples Repository）

  1. Feedback（感測器）：事後校正

- 現有實作：Hooks（pre_bash_guard, pre_write_guard）、quality_gate.py

- 缺口：Feedback 不會自動升級為 Feedforward

  1. Computational vs Inferential

- 現有實作：Hook 規則（Computational）+ quality_gate LLM 層（Inferential）

- 缺口：兩者比例失衡（70% Computational + 30% Inferential 為最佳）

#### Anthropic 設計原則

  1. Multi-Agent Specialization

- 現有實作：todoist-query → N 子任務 → todoist-assemble 三階段

- 優勢：符合 Generator-Evaluator 分離原則

- 缺口：缺少獨立 Planner Agent（現由 Phase 1 內嵌邏輯負責）

  1. Context Management

- 現有實作：ADR-036 壓縮提示（Summary/BufferWindow）

- 問題：壓縮會損失細節，Anthropic 推薦 Clean Reset + Artifact Handoff

  1. Iterative Refinement with Concrete Criteria

- 現有實作：quality_gate.py 評分（完整性 40% + 連貫性 30% + 準確性 30%）

- 優勢：已具備量化標準

- 缺口：評分結果未回饋至 Skill/模板改進

2. 優化方案：五大策略

策略 1：Feedforward 增強 — 正向案例庫

#### 2.1.1 問題陳述

現有 Harness 主要依賴「禁止規則」（Hook 攔截、錯誤檢測），缺乏「正向引導」。根據 Martin Fowler：

"Feedforward controls raise the probability of quality outputs on first attempt."

#### 2.1.2 解決方案：Good Examples Repository

新建檔案：context/good-examples-registry.json

結構：


{
  "schema_version": 1,
  "categories": {
    "research_task": {
      "description": "知識庫研究任務優秀案例",
      "examples": [
        {
          "task_key": "ai_deep_research",
          "result_id": "2026-04-15T10:30:00+08:00",
          "quality_score": 9.2,
          "highlights": [
            "明確的研究階段劃分（Foundation → Mechanism → Application）",
            "每個研究點附完整引用（URL + 段落摘錄）",
            "KB 匯入時自動生成標籤（tags: [AI 架構, RAG, 向量資料庫]）"
          ],
          "template_ref": "templates/auto-tasks/ai-deep-research.md",
          "result_file": "results/todoist-auto-ai_deep_research-20260415103000.json"
        }
      ]
    },
    "code_task": { ... },
    "system_insight": { ... }
  },
  "usage_instructions": "每個 auto-task 啟動時，若 continuity 記錄 ≤ 2 筆，讀取對應 category 的 1 個 example 作為 Feedforward 參考。"
}

整合點：

  • templates/shared/preamble.md 新增「步驟 0.5：讀取優秀案例」
  • tools/quality_gate.py 自動收錄 quality_score ≥ 8.5 的結果到 good-examples

預期效果：首次執行成功率從 72% → 85%（根據 Fowler 文章案例）

策略 2：Context Management 範式轉移

#### 2.2.1 問題陳述

現有 ADR-036 壓縮機制：

  • 65-80% 使用率 → BufferWindow（保留最近 5 次關鍵結果）
  • \> 80% → Summary（強制壓縮）

問題：Anthropic 研究證明壓縮會導致「context anxiety」（模型提早結束任務以避免達上限）。

#### 2.2.2 解決方案：Clean Reset + Structured Handoff

新機制：tools/context_reset_coordinator.py

三級重置策略：

| Context 使用率 | 動作 | 實作 |

|---------------|------|------|

| < 60% | 正常繼續 | 無動作 |

| 60-75% | Artifact Handoff | 將中間結果寫入 state/agent-handoff/{trace_id}.json，清空 context，新 Agent 讀取 handoff 檔案繼續 |

| \> 75% | Emergency Reset | 強制終止當前 Agent，標記 status=partial，將 checkpoint 寫入 handoff，排程下次 round 繼續 |

Handoff Schema：


{
  "trace_id": "abc123",
  "original_task": "深度研究 Harness Engineering",
  "completed_steps": ["WebSearch", "WebFetch", "初步分析"],
  "artifacts": {
    "web_content": "file://state/agent-handoff/abc123-web.md",
    "kb_drafts": ["file://state/agent-handoff/abc123-kb-01.md"]
  },
  "next_steps": ["撰寫優化方案", "寫入知識庫", "驗證"],
  "context_snapshot": null  // 不保留原始 context，僅保留結構化產出物
}

與現有機制的關係：

  • ADR-036 保留作為 最後手段（> 90% 時仍需 Summary）
  • 60-75% 區間改用 Handoff，避免資訊損失

策略 3：Multi-Agent 責任邊界精細化

#### 2.3.1 問題陳述

現有設計（docs/design/harness-engineering-design-20260329.md）：

| 面向 | OODA 管轄 | Harness 管轄 |

|------|----------|-------------|

| 流程 | 步驟定義 | 執行容錯 |

| 品質 | 品質門檻 | — |

| 資源 | — | 並行度、timeout |

問題：「品質門檻」實際由 quality_gate.py 執行，但 OODA 的 decide 階段也有 quality_gates_logic: "AND"，職責重疊。

#### 2.3.2 解決方案：獨立 Planner Agent

新增 Agent 角色：planner (config/agent-roles.yaml)

四角色分工：

| 角色 | 職責 | 輸入 | 輸出 | 現有對應 |

|------|------|------|------|---------|

| Planner | 任務分解與資源配置 | 用戶意圖 + system-insight | 執行計畫 JSON | Phase 1 的內嵌邏輯 |

| Generator | 任務執行 | 執行計畫 | 結果檔案 | Phase 2 子 Agent |

| Evaluator | 品質驗證 | 結果檔案 | quality_score | quality_gate.py |

| Assembler | 輸出組裝 | 驗證後結果 | 最終通知 | Phase 3 todoist-assemble |

Planner 獨立的好處：

  1. Specialization：專注於「選擇正確任務」，不混雜執行邏輯
  2. Tuning：可獨立優化 Planner prompt（現在改 Phase 1 會影響整個流程）
  3. Observability：Planner 輸出計畫 JSON 可單獨審查

實作路徑：

  • 新建 prompts/team/todoist-planner.md（從 run-todoist-agent-team.ps1 Phase 1 抽離）
  • Phase 1 改為呼叫 claude -p < prompts/team/todoist-planner.md
  • Planner 輸出 results/todoist-plan-v2.json（含 reasoning 欄位解釋選擇邏輯）

策略 4：Computational-First 驗證金字塔

#### 2.4.1 問題陳述

現有 quality_gate.py：

  • Tier 1：確定性檢查（JSON schema、必填欄位）— 零成本
  • Tier 2：LLM 評分（完整性/連貫性/準確性）— Haiku ~5s

問題：Tier 2 佔驗證時間 95%，且 LLM 主觀判斷可能不穩定（同一結果重複評分，分數可能 7.5 → 8.2）。

#### 2.4.2 解決方案：三層驗證金字塔


           ┌─────────────┐
           │  Tier 3     │  LLM 語義評分（僅 Critical 任務）
           │  5% 流量    │  成本：Haiku ~5s
           └─────────────┘
          ┌───────────────┐
          │  Tier 2       │  啟發式檢查（keyword 密度、引用完整性）
          │  20% 流量     │  成本：regex + NLP ~200ms
          └───────────────┘
         ┌─────────────────┐
         │  Tier 1         │  確定性檢查（schema、必填欄位）
         │  100% 流量      │  成本：<10ms
         └─────────────────┘

新增 Tier 2（啟發式層）：tools/quality_heuristics.py

檢查項目：

  • 引用完整性：研究任務必須包含 ≥ 2 個 URL 引用
  • 關鍵字密度：summary 中必須包含 task_key 的核心關鍵字（從 frequency-limits.yaml 提取）
  • 長度合理性：summary 字數 ≥ 50 且 ≤ 500（過短 = 敷衍，過長 = 冗餘）
  • 時間戳一致性：created_at 必須在任務執行時間 ± 10 分鐘內
  • 連結可達性（選用）：檢查 URL 是否 404（僅對 Web 任務）

分流邏輯：


if task_type in ["ai_deep_research", "tech_research", "system_insight"]:  # Critical 任務
    run_tier3_llm_eval()
elif heuristic_score >= 8.0:  # 啟發式高分，跳過 LLM
    pass_with_score(heuristic_score)
else:  # 灰色地帶，才用 LLM
    run_tier3_llm_eval()

預期效果：LLM 呼叫減少 70%（從 30 次/日 → 9 次/日），驗證延遲從 150s → 50s。

策略 5：Feedback → Feedforward 自動升級管線

#### 2.5.1 問題陳述

現有 constraint_evolve.py：失敗模式 → constraint-registry → （人工審查）→ hook-rules.yaml

斷點：成功模式無法自動升級為 Feedforward（優秀案例不會自動進入 good-examples-registry）。

#### 2.5.2 解決方案：Success Pattern Extractor

新工具：tools/success_pattern_extractor.py

工作流：


quality_gate.py 發現 quality_score ≥ 8.5
  ↓
success_pattern_extractor.py 提取特徵
  ↓
good-examples-registry.json 自動新增條目
  ↓
下次同類任務啟動時，preamble.md 自動 Read 優秀案例

特徵提取：

  • 結構特徵：步驟順序、工具呼叫序列
  • 內容特徵：關鍵段落的 template（用 regex 提取可復用模式）
  • 元資料特徵：執行時間、token 用量

案例數控制：

  • 每個 task_type 最多保留 3 個優秀案例（LRU 淘汰）
  • 避免 good-examples 膨脹成「第二個訓練集」

驗證指標：

  • 新任務首次成功率提升 15%（從 72% → 87%）
  • 平均 iteration_count 降低（從 1.3 → 1.1）

4. 風險評估與緩解

4.1 技術風險

| 風險 | 影響 | 機率 | 緩解 |

|------|------|------|------|

| Context Handoff 資訊損失 | 任務中斷後無法恢復 | 中 | 1. Handoff schema 強制包含 next_steps
2. 驗證工具檢查 artifacts 完整性 |

| Good Examples 過時 | 引導範本不再適用新版本 | 低 | 1. 每月自動重新評分
2. 附加 created_at，6 個月後自動標記 deprecated |

| Heuristic 誤判 | 低質量結果通過 Tier 2 | 中 | 1. Tier 2 閾值保守（8.0 以上才跳過 LLM）
2. 每週抽查 20% Tier 2 通過結果 |

4.2 流程風險

| 風險 | 影響 | 機率 | 緩解 |

|------|------|------|------|

| Planner 獨立後溝通成本增加 | Phase 1→2 交接出錯 | 低 | 1. todoist-plan-v2.json 強制 schema 驗證
2. Phase 2 啟動前檢查 plan 完整性 |

| 優秀案例選擇偏差 | 只收錄某類風格的案例 | 中 | 1. 多維度評分（不只看 quality_score）
2. 人工季度性審查案例庫 |

5. 成功指標與監控

5.1 北極星指標

系統可靠性分數（SRS）：


SRS = (首次成功率 × 0.4) + (平均質量分 × 0.3) + (Context 效率 × 0.2) + (Feedback 迴圈完整性 × 0.1)

目標：SRS 從 7.8 → 9.0（6 個月內）

5.2 分維度指標

| 維度 | 現狀 | 目標 | 監控工具 |

|------|------|------|---------|

| 首次成功率 | 72% | 85% | state/quality-trend.json |

| 平均質量分 | 7.2 | 8.0 | quality_gate.py --report |

| Context 重置率 | 12% | < 5% | state/context-events.jsonl |

| LLM 驗證佔比 | 100% | 30% | tools/quality_heuristics.py --stats |

| 優秀案例使用率 | 0% | 60% | good-examples-registry.json usage_count |

5.3 健康檢查整合

新增 check-health.ps1 區塊：


# [Harness 優化指標]
$goodExamples = Get-Content context/good-examples-registry.json | ConvertFrom-Json
$coverageRate = ($goodExamples.categories.PSObject.Properties.Count / $totalTaskTypes) * 100

$contextEvents = Get-Content state/context-events.jsonl -Tail 100
$resetRate = ($contextEvents | Where-Object { $_.event -eq "context_reset" }).Count / 100

Write-Host "[Harness 優化] 優秀案例覆蓋率: $coverageRate% | Context 重置率: $resetRate%"
if ($coverageRate -lt 50) { Write-Warning "優秀案例覆蓋不足，建議回填歷史案例" }

6. 與現有計畫的關係

6.1 與 lovely-exploring-crayon.md 的協同

現有計畫（B0-B3）：

  • B0：地基加固（backlog_writer, timing-policy, OODA preconditions）
  • B1：Constraint Evolution（失敗 → 約束）
  • B2：Quality Feedback（quality_gate 品質門檻）
  • B3：Observability（constraint_coverage, entropy_score）

本優化方案（Harness 2.0）：

  • 與 B0 互補：Context Reset Coordinator 依賴 timing-policy 的 TTL 設定
  • 與 B1 對稱：Success Pattern Extractor 是「成功版」的 constraint_evolve
  • 與 B2 協同：Quality Heuristics 是 quality_gate 的前置層
  • 與 B3 共享：SRS 指標整合至 check-health.ps1

建議實施順序：

  1. 先完成 B0-B3（地基必須穩固）
  2. 再啟動 Harness 2.0 Phase A（優秀案例庫、Context 重置）
  3. 最後 Phase B（Planner 獨立、Heuristic 層）
