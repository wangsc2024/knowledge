Context JSON 與 YAML Registry 雙軌同步策略 — SSOT、漂移偵測與 Agent 專案治理（2026-04-01）

研究範圍說明：本筆記優先採用本倉庫內文件交叉驗證（ARCHITECTURE、memory-policy、dedup-policy、preamble、ADR 摘錄）。本次執行環境無法對外 WebFetch，公開標準書目附於文末供讀者自行開啟核對。

執行摘要

在 Agent 驅動的程式庫中，常同時存在兩類「真相」：人類可編輯的宣告式組態（多為 YAML），以及執行過程累積的脈絡／狀態（多為 JSON，位於 context/、state/）。若缺乏單一真相來源（SSOT）與機器可驗證的同步閘門，兩軌會出現保留期不一致、去重規則分歧、或模板敘述與實際政策脫節等漂移。對 daily-digest-prompt 而言，專案已透過 config/memory-policy.yaml 統一各記憶層用途與保留策略 [來源：memory-policy.yaml, 2026]；config/dedup-policy.yaml 與 context/research-registry.json 則承載研究去重的宣告與執行狀態 [來源：dedup-policy.yaml, 2026]；架構上明確區分「Prompt 薄層／config 數據／模板按需載入」 [來源：ARCHITECTURE.md, 2026]。建議的實務組合是：政策與 TTL 以 YAML 為準、狀態與索引以 JSON 為準、且寫入路徑只讀 YAML 政策；並沿用既有 config/schemas 與 workflows 清單，對結果檔與關鍵 context 做語義／Schema 驗證，將漂移暴露於 CI 而非執行期。

技術概述

daily-digest-prompt 採文件驅動架構：管線、路由、去重、記憶治理均可調整於 config/.yaml 與 context/.json，避免硬寫在 prompt [來源：ARCHITECTURE.md, 2026]。其中 memory-policy.yaml 明確定義 continuity、research_registry、research_series 等層的 storage 與保留規則 [來源：memory-policy.yaml, 2026]；dedup-policy.yaml 則指定 registry 檔路徑、topic 冷卻與飽和閾值 [來源：dedup-policy.yaml, 2026]。此二分與「宣告 vs 狀態』分離一致，是雙軌治理的基礎。

核心概念與原理

1. 宣告軌（YAML）與狀態軌（JSON）的角色分工

  • YAML 軌適合表達人類可讀、可版控的政策：保留天數、冷卻天數、topK、類別關鍵字等地域規則 [來源：dedup-policy.yaml, 2026]。
  • JSON 軌適合表達程式讀寫的索引與歷史：topics_index、entries、執行日誌聚合欄位等（見研究註冊表結構與模板對 registry 的讀寫要求）[來源：dedup-policy.yaml + preamble research-registry 流程, 2026]。
  • 架構文件將大量「可變邏輯」列於 config，並以 workflows 驗證 config 與檔案系統的一致性 [來源：ARCHITECTURE.md, 2026]。

2. 單一真相來源（SSOT）與「連續記憶」對齊

  • preamble.md 規定自動任務連續記憶預設保留筆數以 memory-policy.yaml -> layers.continuity.retention_runs 為準；若 continuity 檔案中 max_runs 與政策不一致，優先沿用檔案實值並在維護時校正 [來源：preamble.md, 2026]。
  • 同一文件亦載明 research_registry 等記憶層應先讀 memory-policy.yaml，新增規則時以該檔為唯一真相來源 [來源：preamble.md + memory-policy.yaml, 2026]。
  • 此三路（preamble／continuity 檔／memory-policy）並陳的目的，是防止「prompt 描述的保留期」與「政策 YAML」漂移；維運上應定期 diff 或用小型腳本斷言一致性。

3. 漂移偵測與機器驗證

  • 專案已具備多種 results／config schema 與驗證 workflow（例如 config/schemas/results-auto-task-schema.json、workflows/results-validation-checklist.md 等目錄列舉）— 代表團隊已將「可驗證性」視為治理手段 [來源：ARCHITECTURE.md, 2026]。
  • ADR 記錄中亦可見對 JSON schema／DONE_CERT 驗證 的架構決策（Guardrails 與輸出層驗證議題），與「把不一致提早變成可觀測錯誤」方向一致 [來源：adr-registry.json 摘錄 ADR-20260224-002, 2026]。

4. 與外部標準的類比（書面參考，非本次即時摘錄）

  • Twelve-Factor App 的 Config 因子強調組態與程式嚴格分離，並以環境或檔案注入；概念上可對照本專案「config 與 prompt 分離」 [來源：12factor.net/config，公開文件]。
  • JSON Schema 常用於驗證設定與交換檔格式；可對照本專案 config/schemas 與結果 JSON 驗證需求 [來源：json-schema.org，公開規格]。
  • GitOps／期望狀態（desired state） 強調版本化宣告與實際狀態對照；可類比「YAML 政策 + registry 現況」的對帳思路 [來源：CNCF GitOps 白皮書／廠商介紹頁，請讀者自行選擇權威讀本]。

最佳實踐與常見陷阱

最佳實踐

  1. 改保留期／冷卻／topK 只改 dedup-policy 或 memory-policy，再讓工具與 prompt 引用同一鍵；避免魔法數字散落在模板中 [來源：dedup-policy.yaml + preamble, 2026]。
  2. 狀態檔寫入前 Read 政策檔，寫入後驗證 JSON schema（結果檔 middleware／workflow 已提示）[來源：preamble.md 結果驗證 + ARCHITECTURE schemas, 2026]。
  3. 對 continuity 與 research_registry 類高頻寫入檔，啟動 短腳本斷言：topics_index 日期格式、entries TTL 與 retention_days 一致 [來源：dedup-policy.yaml retention_days, 2026]。

常見陷阱

  • 雙源並存未指定優先序：例如同時在 prompt 與 policy 寫保留 5 筆與 7 天，下一代維運者無所適從—應遵循 preamble 的「policy 優先、檔案實值需較正」規則 [來源：preamble.md, 2026]。
  • 把業務狀態寫進 YAML：導致版控噪音與合併衝突；應讓 YAML 保持規則、JSON 保持實例。
  • 跳過 schema 驗證：Phase 3 組裝才發現欄位缺失；應務必在結果檔寫入前驗證 [來源：preamble.md, 2026]。

與 daily-digest-prompt 的關聯

  • 研究任務：tech_research 模板要求更新 context/research-registry.json 的 topics_index 與 entries，參數來自 dedup-policy.yaml [來源：任務規格 + dedup-policy, 2026]。
  • 連續記憶：context/continuity/auto-task-tech_research.json 的 max_runs 應與 memory-policy.yaml 對齊檢查 [來源：memory-policy.yaml + preamble, 2026]。
  • 下一步工程化：可在 check-health.ps1 或 preflight_gate.py 路徑加入「policy ↔ 首則 continuity 檔」欄位比對，將漂移變成健康分扣點。

程式碼範例（PowerShell：載入政策後再組結果檔欄位）


# 概念示例：先讀 memory-policy 再決定 continuity 保留上限（實際欄位名請對照真實 YAML）
$mp = pwsh -NoProfile -Command "@'
$(Get-Content ..\config\memory-policy.yaml -Raw)
'@ | ConvertFrom-Yaml"
# 若無 ConvertFrom-Yaml，可改以 uv run python 讀取 yaml.safe_load
$retention = $mp.layers.continuity.retention_runs
# 寫入 results JSON 時附带 policy_retention_runs=$retention 供 Phase 3 稽核
