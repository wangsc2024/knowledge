Daily-Digest-Prompt 跨 Phase 資料契約與防禦性編排模式優化建議

結果檔案 Schema 治理、命名一致性修復與中間驗證閘門設計

研究日期：2026-03-07

類型：專案優化 / 深度思維 / 知識庫

適用對象：daily-digest-prompt 專案

---

一、概述

本文運用知識庫中既有的 FSM 設計模式、品質閘門、YAML 配置驅動架構、PowerShell 並行程式設計等研究成果，針對 daily-digest-prompt 專案的跨 Phase 資料契約進行深度檢視。重點分析 Phase 間結果檔案的 Schema 治理現況，識別出命名不一致、欄位定義分歧、中間驗證缺失等系統性問題，並提出具體的防禦性編排改善方案。

核心問題：專案歷史上至少出現 3 次「結果檔案缺失」的根因修復（2026-02-18 檔案命名修正、2026-02-24 動態掃描修復、hyphen→underscore 正規化修復），這些問題表面上是個案 bug，實質上反映了跨 Phase 資料契約缺乏形式化定義與強制驗證的系統性缺陷。

---

二、現況診斷：五大結構性問題

2.1 自動任務結果檔案命名不一致（嚴重度：HIGH）

15 個自動任務中，僅 2 個（tech_research、chatroom_optimize）的結果檔案命名與 YAML key 完全一致。其餘 13 個存在 hyphen/underscore 混用：

| YAML Key | Prompt 指定的輸出檔名 | 預期格式 | 差異 |

|----------|---------------------|---------|------|

| ai_deep_research | todoist-auto-ai-deep-research.json | todoist-auto-ai_deep_research.json | hyphen |

| ai_github_research | todoist-auto-ai-github.json | todoist-auto-ai_github_research.json | 截斷+hyphen |

| log_audit | todoist-auto-logaudit.json | todoist-auto-log_audit.json | 無分隔符 |

| git_push | todoist-auto-gitpush.json | todoist-auto-git_push.json | 無分隔符 |

| creative_game_optimize | todoist-auto-creative-game.json | todoist-auto-creative_game_optimize.json | 完全不同 |

| self_heal | todoist-auto-self-heal.json | todoist-auto-self_heal.json | hyphen |

| system_insight | todoist-auto-system-insight.json | todoist-auto-system_insight.json | hyphen |

| github_scout | todoist-auto-github-scout.json | todoist-auto-github_scout.json | hyphen |

影響：PowerShell 編排腳本的 $keyAliases 表（15+ 條映射）本質上是在 patch 這個命名混亂。每次新增自動任務，都必須手動維護別名表，否則 Phase 3 找不到結果檔案。

2.2 結果 JSON 欄位定義分歧（嚴重度：HIGH）

Phase 3（todoist-assemble.md）期望所有自動任務結果包含 type 欄位以匹配 frequency-limits.yaml 的 counter_field，但實際上：

| 任務類型 | 使用欄位 | Phase 3 期望 |

|---------|---------|------------|

| 研究類（shurangama 等） | type | type 正確 |

| system_insight | task_key | type 不符 |

| self_heal | task_key | type 不符 |

| git_push | knowledge_sync | type 不符 |

後果：Phase 3 讀取 type 欄位失敗 → 無法正確更新頻率計數器 → round-robin 公平性受損。

2.3 Phase 2→3 缺乏中間驗證閘門（嚴重度：MEDIUM-HIGH）

目前 Phase 2 完成後直接啟動 Phase 3，沒有中間步驟驗證結果檔案是否存在且格式正確。結果檔案缺失只在 Phase 3 組裝時才被發現。

理想流程應為：Phase 2 完成 → 驗證閘門（檔案存在 + JSON 可解析 + 必要欄位存在）→ Phase 3 啟動。

知識庫關聯：已有「自動化品質閘門」筆記提出 Ratchet 模式，可直接應用於 Phase 間驗證。

2.4 結果 JSON 無 Schema 驗證（嚴重度：MEDIUM）

hooks/validate_config.py 已對 7 個 YAML 配置做 Schema 驗證，但結果 JSON 完全無驗證。config/schemas/ 目錄存在，但不含任何 result-*.schema.json。

配置檔案（靜態、低頻變更）有完整驗證，而結果檔案（動態、每次執行都生成）反而無驗證 — 這是「燈下黑」現象。

2.5 DONE_CERT 軟性解析（嚴重度：LOW-MEDIUM）

Phase 2 Agent 輸出的 ===DONE_CERT_BEGIN=== JSON ===DONE_CERT_END=== 標記被 Phase 3 讀取，但無 JSON 格式驗證、無必要欄位檢查，Agent 超時可能導致截斷的 DONE_CERT。

三、優化建議：四層防禦性架構

3.1 第一層：命名契約標準化（P0 — 立即修復）

原則：結果檔案名稱必須與 frequency-limits.yaml 的 key 完全一致（底線格式）。

具體動作：

  1. 遍歷所有 15 個 prompts/team/todoist-auto-*.md，將輸出檔名統一為 results/todoist-auto-{yaml_key}.json
  2. 在 specs/interface-spec/phase-results.md 新增「結果檔案命名規則」段落
  3. 刪除 $keyAliases 別名表（根因消除後不再需要 patch）
  4. 新增 validate_config.py --check-result-names 子命令，比對 prompt 中的輸出路徑與 YAML key

預期效果：消除 13 個命名不一致問題，防止未來新增自動任務時重複犯錯。

3.2 第二層：結果 JSON Schema 標準化（P0 — 與命名一起修復）

定義通用基礎 Schema（所有自動任務必須包含）：


{
  "required": ["agent", "status", "type", "content", "summary"],
  "properties": {
    "agent": { "type": "string" },
    "status": { "enum": ["success", "partial", "failed"] },
    "type": { "type": "string", "description": "必須與 frequency-limits.yaml key 一致" },
    "content": { "type": "string" },
    "summary": { "type": "string" },
    "error": { "type": ["string", "null"] },
    "duration_seconds": { "type": "number" },
    "done_cert": { "type": "object" }
  }
}

動作：將 Schema 存為 config/schemas/auto-task-result.schema.json，更新所有 prompt 統一使用 type 欄位。

3.3 第三層：Phase 間驗證閘門（P1 — 短期內實施）

在 run-todoist-agent-team.ps1 的 Phase 2→3 之間插入 Test-PhaseResults 驗證函數：

  • 讀取 todoist-plan.json 取得預期的結果檔案列表
  • 逐一檢查檔案存在性、JSON 可解析性、必要欄位（type, status）
  • 所有通過 → Phase 3 正常啟動
  • 部分缺失但 >=50% 存在 → Phase 3 啟動，通知標記降級
  • 全部缺失 → 跳過 Phase 3，直接發送告警

借鏡品質閘門 Ratchet 模式：設定基線成功率並逐步提升。

3.4 第四層：契約文件化與自動驗證（P2 — 中期實施）

強化 specs/interface-spec/phase-results.md：

  1. 為每個 Phase 的輸入/輸出加上「契約版本號」
  2. 加入 breaking change 管理：修改結果 Schema 時必須同步更新消費端
  3. 新增 validate_config.py --check-contracts 自動化契約一致性檢查

六、深度思維反思

6.1 為何這個問題長期未被根治？

根本原因是漸進式有機成長的副作用。專案從最初 3 個自動任務成長到 19 個，每次新增都是解決當下需求，沒有回頭統一契約。$keyAliases 別名表就是典型的「每次 patch 一個映射」的累積結果。

這與知識庫中「熵增與複雜度治理」的研究一致 — 系統不主動治理就會自然走向熵增。P0-A（CLAUDE.md 精簡）和 P1-B（自動任務一致性驗證）已解決了配置層的熵增，但資料契約層的熵增仍在持續。

6.2 防禦性編排 vs. 防禦性程式設計

傳統防禦性程式設計著重「輸入驗證」，但在 LLM Agent 編排系統中，更重要的是輸出契約驗證 — 因為 Agent 的輸出不可完全預測。Phase 間驗證閘門本質上是在 LLM 產出物上建立「型別檢查」機制，這與 TypeScript 對 JavaScript 的關係異曲同工。

6.3 契約驅動開發（Contract-Driven Development）的適用性

本建議的核心理念是將「隱式契約」（散落在 prompt 中的格式期望）轉化為「顯式契約」（Schema 定義 + 自動驗證）。這與 API 設計中的 OpenAPI/Swagger 規範、微服務中的 Consumer-Driven Contract Testing 思路一致，但應用場景從服務間通訊轉移到 Agent Phase 間通訊。
