Phoenix 的 tracing schema、eval pipeline 與 OpenTelemetry/OpenInference 整合機制

技術概述

Phoenix 是一個偏向 AI observability 與 evaluation layer 的開源平台，核心能力不是 durable workflow，而是把應用執行過程轉成 trace、再把 trace 與 dataset、experiment、prompt 版本管理串成一條可反覆驗證的改進回路。[來源：What is Arize Phoenix?, 2026][來源：Arize Phoenix | Phoenix, 2025][來源：GitHub - Arize-ai/phoenix: AI Observability & Evaluation, 2026] 對 daily-digest-prompt 而言，它最有價值的不是「取代現有排程」，而是補上自動任務 tracing、prompt/agent 評估、失敗案例回放與實驗對照這一層。

研究背景

今日已完成任務裡，和本題直接相關的技術訊號包括 Phoenix AI Pipeline 觀測平台研究、auto_task_fairness Emergency 修復、Skill/Workflow/Tools 使用統計檢討、系統洞察 與 constraint-pipeline-automator Skill 鑄造。這些任務共同指向一個需求：系統已經有大量自動任務、JSONL 日誌與治理規則，但仍缺一個能把 tracing、evaluation、prompt 迭代與 dataset 實驗放在同一工作台的觀測層。

另外，KB 既有 Phoenix 筆記停留在 foundation 階段，內容偏概覽，尚未把 tracing schema、semantic translation 與 eval pipeline 的內部資料流講清楚，因此本次研究定位為 mechanism 階段延伸。

核心發現

1. Phoenix 的 tracing 路徑本質上是「instrumentation -> exporter -> collector/UI」，其中 collector 直接就是 Phoenix 伺服器

Phoenix 官方 tracing 說明把資料流拆成 instrumentor、exporter、collector 三段；應用先被手動或自動 instrument，span 再由 exporter 送到 collector，而 Phoenix 伺服器本身同時扮演 collector 與 UI。[來源：How Tracing Works - Phoenix, 2026][來源：What is my Phoenix Endpoint? - Phoenix, 2026][來源：GitHub - Arize-ai/phoenix: AI Observability & Evaluation, 2026] 官方文件也明確指出 Phoenix 目前支援 OTLP over HTTP，預設可透過 http://localhost:6006/v1/traces 收 trace。[來源：How Tracing Works - Phoenix, 2026][來源：What is my Phoenix Endpoint? - Phoenix, 2026][來源：Environments - Phoenix, 2026]

對本專案的意義是：現有 JSONL 不是 Phoenix 的原生輸入，真正穩定的接法應該是把 JSONL 事件先整理成 OpenTelemetry/OpenInference span，再經 OTLP 匯入，而不是直接把 JSONL 當 Phoenix 資料模型。

2. Phoenix 在 schema 層選擇 OpenInference 作為標準語彙，並靠 translation span processors 吃下其他 tracing 慣例

Phoenix 官方明講它用 OpenInference semantic conventions 作為標準格式；若資料源來自 OpenLIT、OpenLLMetry 或 OpenTelemetry GenAI，必須先透過 span processors 做語意轉譯，才能在 Phoenix UI 裡一致呈現。[來源：Translating Semantic Conventions - Phoenix, 2026][來源：OpenInference Specification | openinference, 2026][來源：Openinference Semantic Conventions | Arize Docs, 2025]

OpenInference 本身定義了 span kind 與屬性語彙，例如 openinference.span.kind、llm.model_name、llm.input_messages、tool.name、retrieval.documents 等，目的是讓不同模型、框架與供應商輸出的 AI trace 可以落到共同欄位。[來源：OpenInference Specification | openinference, 2026][來源：Openinference Semantic Conventions | Arize Docs, 2025][來源：OpenInference Java - Phoenix, 2026] 這代表 Phoenix 的「觀測效果」很大程度取決於你是否願意在接入端先遵守 OpenInference 的資料契約。

3. Phoenix 的 eval pipeline 不是單純打分，而是把 dataset、task、evaluator、experiment 與 trace 綁成同一條 evidence chain

Phoenix 的 datasets/experiments 文件把資料流拆得很清楚：dataset 由一組 examples 組成，task 是你要測的函式或工作流，evaluator 用來評分 task 輸出，最後 experiment 會把同一批輸入上的執行結果與評分一起保存。[來源：Overview: Datasets & Experiments - Phoenix, 2026][來源：How to: Experiments - Phoenix, 2026][來源：Run Experiments - Phoenix, 2026]

在 SDK 流程中，run_experiment 會把 task 與 evaluators 套到 dataset 上；在 UI 流程中，Prompt Playground 可直接拿 dataset 跑 prompt 變體，再把結果存成 experiment。[來源：Run Experiments - Phoenix, 2026][來源：Prompt Playground - Phoenix, 2026][來源：Iterate on Your Prompts - Phoenix, 2026] 這種設計的重點不是「評分函式存在」，而是每次改 prompt、模型、參數或工具設定後，都能在同一輸入集合上重跑並留痕，讓品質改進從感覺變成可追溯的比較。

4. Phoenix 的 LLM evaluator 以結構化輸出與 traceability 為優先，避免自由文字評估難以對照與回放

Phoenix 官方 evaluation 文件指出，LLM evaluators 同時支援 client-side 與 server-side 兩種模式；而在輸出解析上，它傾向用 tool calling/function calling 來產生結構化 judgment，而不是解析自由文字。[來源：Evaluation - Phoenix, 2026][來源：LLM Evaluators - Phoenix, 2026][來源：Run Experiments - Phoenix, 2026]

更重要的是，官方也說 evaluator 呼叫本身會產生 OpenTelemetry trace，並記錄 prompt、模型回應、tool calls、延遲與 token usage，讓 evaluator 本身也能被觀測與除錯。[來源：LLM Evaluators - Phoenix, 2026][來源：Evaluation - Phoenix, 2026][來源：What is Arize Phoenix?, 2026] 這對本專案很關鍵，因為 quality gate、system_insight 或未來的自動評分器，不該只是回傳一個分數，而要能追蹤「為什麼這次分數長這樣」。

5. Phoenix 自架的門檻不高，但 production 化仍涉及資料庫、端點與權限治理

Phoenix 官方文件明示可免費 self-host，且資料可以完全留在自家基礎設施中，甚至 air-gapped。[來源：Self-Hosting - Phoenix, 2026][來源：License - Phoenix, 2026][來源：Home - Phoenix, 2026] 部署上支援 terminal、container、Docker、Kubernetes、Helm、CloudFormation 等多種方式；架構上預設用 SQLite，production 則建議切 PostgreSQL。[來源：Self-Hosting - Phoenix, 2026][來源：Self-Hosting | Arize Phoenix, 2025][來源：Environments - Phoenix, 2026]

因此 Phoenix 雖然不是很重的平臺，但若要接到 daily-digest-prompt，至少要先決定三件事：trace 要送到哪個 collector endpoint、資料要先落 SQLite 還是 Postgres、哪些 prompt/eval traces 屬於敏感資料需要遮罩或縮減保留期。

最佳實踐與常見陷阱

Phoenix 導入時，最穩健的順序不是先把所有自動任務都接上，而是先選一條高價值、可重放、已有失敗樣本的工作流做試點，例如 todoist-auto-tech_research 或 system_insight。先把 trace schema 對齊，再加 evaluator，最後才加 Playground/experiment；這樣能把故障面壓到最小。[來源：How Tracing Works - Phoenix, 2026][來源：Run Experiments - Phoenix, 2026][來源：Evaluation - Phoenix, 2026]

另一個最佳實踐是把 Phoenix 看成「觀測與實驗層」，不是取代既有排程器或 durable executor。官方首頁與 README 都把 Phoenix 定位成 observability、evaluation、prompt engineering、datasets/experiments，而不是 workflow orchestration。[來源：What is Arize Phoenix?, 2026][來源：Arize Phoenix | Phoenix, 2025][來源：GitHub - Arize-ai/phoenix: AI Observability & Evaluation, 2026] 若把它誤當成 Temporal、n8n 這類工作流引擎，導入後很容易在責任邊界上混亂。

常見陷阱之一是直接餵入非標準 trace 欄位，期待 UI 自動理解。Phoenix 已經明確說明不同 semantic conventions 需要翻譯到 OpenInference；若來源不做正規化，UI 雖然可能收得到 span，卻不一定能在 Agent、Tool、Retriever、Evaluator 等維度正確展開。[來源：Translating Semantic Conventions - Phoenix, 2026][來源：OpenInference Specification | openinference, 2026][來源：Openinference Semantic Conventions | Arize Docs, 2025]

另一個陷阱是把 evaluator 當黑箱。Phoenix 雖提供高階封裝，但官方同時強調 evaluator traces、prompt versioning、tagging 與 structured output，這表示 evaluator 本身也應被納入觀測、回放與版本治理。[來源：LLM Evaluators - Phoenix, 2026][來源：Evaluation - Phoenix, 2026][來源：Prompt Playground - Phoenix, 2026]

與本專案的關聯

daily-digest-prompt 已經具備 JSONL 結構化日誌、phase-based 自動任務、KB 匯入、品質閘與研究型工作流；Phoenix 最適合扮演這些能力之上的觀測與評估中介層。

實作上可分三步：

  1. 先把 results/.json 與 logs/structured/.jsonl 的關鍵事件映射成 OpenInference spans，至少包含 task_key、phase、status、tool_calls、prompt_version、latency、error.type。
  2. 針對 todoist-auto-tech_research、ai_deep_research、system_insight 建立 dataset 與 evaluator，驗證摘要品質、引用完整性、結果格式與 KB 匯入成功率。
  3. 把 Prompt Playground 用在高波動的研究 prompt 與品質閘 prompt，讓新版本 prompt 可以在同一批歷史案例上比較，而不是直接進 production。

若只做一步，我會優先做「JSONL -> OpenInference span 映射」，因為這是後續 tracing、experiment、evaluator 全部成立的前提。

程式碼範例


from phoenix.otel import register
from openinference.instrumentation.openai import OpenAIInstrumentor
from opentelemetry import trace

# Phoenix collector 預設可吃 OTLP traces
tracer_provider = register(endpoint="http://127.0.0.1:6006/v1/traces")
OpenAIInstrumentor().instrument(tracer_provider=tracer_provider)
tracer = trace.get_tracer("daily-digest")

with tracer.start_as_current_span("todoist-auto-tech_research") as span:
    span.set_attribute("openinference.span.kind", "CHAIN")
    span.set_attribute("input.value", "研究 Phoenix tracing schema")
    span.set_attribute("metadata", "{\"task_key\":\"tech_research\",\"phase\":\"phase2\"}")

這段示意碼的重點不是 OpenAI，而是先確認本專案可以用 OpenTelemetry/OpenInference 形式把任務執行送進 Phoenix；後續再替換成更貼近 daily-digest-prompt 的自訂 instrumentor。
