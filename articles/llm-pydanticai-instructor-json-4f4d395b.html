LLM 結構化輸出驗證管線實作指南 — PydanticAI、Instructor 與 JSON Schema 的雙層護欄

技術概述

今天完成的 Todoist 任務大量依賴「研究結果必須寫成固定 JSON、再匯入 RAG 知識庫」這種流程，因此真正的關鍵不是單純讓模型吐出 JSON，而是建立一條可驗證、可重試、可觀測的結構化輸出管線。最新實務已明顯分成兩層：第一層用模型或 SDK 的原生 Structured Outputs / Native Output 盡量把輸出限制在 schema 內；第二層再由 PydanticAI 或 Instructor 以型別、驗證器與重試機制補上應用端護欄，避免格式漂移、欄位缺漏與靜默失敗。

核心概念與原理

1. 三層責任分工

  • JSON Schema：定義資料契約，描述欄位、型別、必要欄位與額外欄位限制。
  • 模型原生 structured output：盡量在生成階段就讓輸出符合 schema，降低後處理成本。
  • PydanticAI / Instructor：把 schema 轉成程式語言型別，執行驗證、重試、例外處理與執行觀測。

2. 為什麼只靠 JSON mode 不夠

OpenAI 官方已明確區分 JSON mode 與 Structured Outputs：前者只保證輸出是合法 JSON，後者才保證符合指定 schema。這代表對 results/*.json、KB 匯入 payload、跨 Phase 中繼檔來說，JSON mode 仍然不夠安全，因為它無法阻止模型漏欄位、亂填 enum 或夾帶多餘欄位。

3. PydanticAI 的角色

PydanticAI 的 output_type 會直接把 Pydantic 型別轉成 JSON Schema，並在 agent 執行完成時回傳已驗證的結構化結果。它預設偏向 agent runtime 思維，適合有工具呼叫、依賴注入、結果 usage 追蹤與後續 evaluation 的流程。

4. Instructor 的角色

Instructor 更偏向 extraction layer。它會把 response_model 注入到各家 LLM client，當驗證失敗時，把錯誤原因回灌到提示上下文再重試。這種模式特別適合「把長文字研究摘要轉成固定結果 JSON」或「把自由文本轉成 KB 匯入 payload」。

5. 雙層護欄模式

建議把結構化輸出分成：

  1. 生成時護欄：能用原生 structured output 就先用，例如 OpenAI json_schema 或 PydanticAI NativeOutput。
  2. 應用時護欄：永遠再跑一次 Pydantic 驗證；若失敗，保留原始 completion、錯誤訊息與重試次數。
  3. 流程護欄：把輸出寫入檔案前再檢查必要欄位，例如 status、summary、kb_imported、done_cert。

最佳實踐與常見陷阱

最佳實踐

  • schema 一律設 additionalProperties: false，避免模型偷偷加欄位。
  • 讓 Pydantic 型別成為單一真實來源，再由型別自動導出 JSON Schema，避免 schema 與程式型別分叉。
  • 對高風險欄位加語意驗證，例如 URL 格式、日期格式、狀態列舉值、字數下限。
  • 把重試次數、最後一次 raw response、驗證錯誤一起記錄到 log，否則很難除錯。
  • 對不同模型能力做分級：支援原生 schema 的模型走 strict/native；不支援者退回 prompt + validation + retry。

常見陷阱

  • 只驗證 JSON 可解析，卻不驗證業務語意。
  • 直接手寫 JSON Schema 與 Pydantic model，久了兩邊漂移。
  • 把 union schema 做得太複雜，導致模型選錯分支。
  • streaming 時只看 partial JSON，不做最終完整驗證。
  • 驗證失敗後直接覆寫錯誤，而不是保留最後一次 completion，讓問題無法重現。

與本專案的關聯

daily-digest-prompt 至少有四個位置適合導入這種管線：

  • results/*.json：所有 agent 結果檔都可用統一 Pydantic model 先驗證，再寫檔。
  • KB 匯入：contentText、tags、source、title 可先做 schema 驗證，避免匯入失敗。
  • 研究摘要：把 Web 研究結果轉成固定結構，再交給 Markdown 組裝器。
  • 跨 Phase 中繼檔：例如 kb-research-brief.json、registry entry，都適合用同一套模型層保護。

如果未來把本專案的研究 agent 收斂成標準輸出契約，最實際的做法是：

  1. 先定義 ResearchResult, KbImportNote, RegistryEntry 三組 Pydantic model。
  2. 對支援 strict schema 的供應商，優先走原生 structured output。
  3. 對不支援者，使用 Instructor 做 extraction + retry。
  4. 寫檔前統一執行最後一次本地驗證。

程式碼範例


from typing import Literal

import instructor
from openai import OpenAI
from pydantic import BaseModel, Field, ValidationError
from pydantic_ai import Agent, NativeOutput


class ResearchResult(BaseModel):
    agent: Literal["todoist-auto-tech_research"]
    status: Literal["success", "partial", "failed"]
    topic: str = Field(min_length=5)
    kb_imported: bool
    summary: str = Field(min_length=20)


# 第一層：若模型支援原生 structured output，先用 NativeOutput
agent = Agent(
    "openai:gpt-4.1",
    output_type=NativeOutput(ResearchResult, name="research_result"),
)

native_result = agent.run_sync("請輸出研究結果摘要").output


# 第二層：再用 Instructor 做應用端驗證與必要重試
client = instructor.from_provider("openai/gpt-4.1-mini", max_retries=2)

validated = client.create(
    response_model=ResearchResult,
    messages=[
        {
            "role": "user",
            "content": f"把這段內容整理成研究結果 JSON：{native_result.model_dump_json()}",
        }
    ],
)

try:
    payload = validated.model_dump(mode="json")
except ValidationError as exc:
    # 這裡應記錄 raw completion、validation errors、retry count
    raise RuntimeError("研究結果結構驗證失敗") from exc

研究結論

若本專案只選一個最有投資報酬率的方向，不是再加更多 prompt，而是把所有關鍵結果檔改成「schema-first」。模型端負責盡量貼齊 schema，應用端負責最終驗證、重試與保留失敗證據。這樣才能把研究、KB 匯入、結果檔案與後續自動化接起來，而不是把錯誤留到下游才爆炸。
