改進建議

P0 - 高價值低成本（可直接落實）

#### 1. 自動失敗恢復機制（LangGraph）

來源：langchain-ai/langgraph

目標檔案：

  • tools/autonomous_harness.py
  • config/autonomous-harness.yaml

描述：

引入 LangGraph 的持久化執行與自動恢復策略。目前系統只有重試一次（run-*.ps1），缺乏從中斷點恢復的能力。可在 autonomous_harness.py 新增 checkpoint 機制，記錄每個步驟的狀態，失敗時從最後成功的 checkpoint 恢復，而非從頭重跑。

預期效益：

  • 減少重複執行開銷
  • 提升長期運行任務的可靠性
  • 降低失敗率（目前 daily_success_rate=87.5%，目標 95%）

---

#### 2. API-first 架構與統一 API 層（Dify）

來源：langgenius/dify

目標檔案：

  • bot/routes.js
  • 新建 api/unified_gateway.py

描述：

參考 Dify 的 API-first 設計，建立統一 API 閘道層。目前 Todoist/Gmail/Pingtung News 等外部服務呼叫散落各處，缺乏統一管理。建議新建 api/unified_gateway.py，提供 /api/todoist、/api/gmail、/api/news 等端點，內部整合 api-cache、llm-router、budget_guard，外部提供一致介面。

預期效益：

  • 統一錯誤處理與重試邏輯
  • 集中式快取與預算管理
  • 簡化外部服務整合

---

P1 - 高價值中成本（需計畫執行）

#### 3. Monorepo + Workspace 架構（n8n）

來源：n8n-io/n8n

目標檔案：

  • 新建 pnpm-workspace.yaml
  • 重組專案結構為 packages/core、packages/skills、packages/hooks、packages/bot

描述：

參考 n8n 的 monorepo 架構（pnpm workspace），將專案重組為子模組。當前平面結構（27 個 skills、13 個 tools、8 個 hooks 混雜根目錄）缺乏模組邊界，導致依賴關係不清晰。monorepo 可強制模組化，每個 package 有獨立 pyproject.toml。

預期效益：

  • 清晰的模組邊界與依賴管理
  • 減少配置膨脹風險
  • 支援獨立版本控制

---

#### 4. Graph-based Workflow Orchestration（LangGraph）

來源：langchain-ai/langgraph

目標檔案：

  • 新建 config/workflow-graph.yaml
  • 新建 tools/workflow_executor.py

描述：

引入 LangGraph 的圖式編排機制。目前 run-agent-team.ps1 是線性 Phase 0→1→2，無法處理條件分支、並行任務、動態路由。建議新增 workflow-graph.yaml 定義節點（fetch/analyze/assemble）與邊（條件/並行），workflow_executor.py 解析圖並執行，支援 branching、subgraphs、human-in-the-loop。

預期效益：

  • 支援複雜工作流編排
  • 提升任務並行度
  • 減少硬編碼邏輯

---

#### 5. 聲明式 Workflow 定義（Dify）

來源：langgenius/dify

目標檔案：

  • config/pipeline.yaml
  • 新建 templates/workflows/*.yaml

描述：

參考 Dify 的聲明式 workflow 模型，將複雜任務編排從 prompt 中抽離到 YAML 配置。當前 todoist-auto-*.md 包含大量執行邏輯（if/else、迴圈），難以維護。建議新增 templates/workflows/ 目錄，用 YAML 定義步驟、條件、循環，prompt 僅負責呼叫 workflow_executor.py。

預期效益：

  • 降低 prompt 複雜度
  • 提升可維護性
  • 支援 workflow 版本控制

---

P2 - 中價值中成本

#### 6. 可觀測性整合（Dify）

來源：langgenius/dify

目標檔案：

  • 新建 tools/observability_bridge.py
  • 新建 config/observability.yaml

描述：

參考 Dify 整合 Langfuse、Opik、Arize Phoenix 等可觀測性平台。目前 post_tool_logger.py 只寫 JSONL，缺乏視覺化 dashboard 和 trace 分析。建議新增 observability_bridge.py，將 Hook 日誌轉發到 Langfuse（開源 LLM observability），提供 token 使用、latency、error rate 的即時監控。

預期效益：

  • 視覺化監控 dashboard
  • Trace 分析與根因定位
  • 成本追蹤與優化
