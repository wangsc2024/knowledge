Test-Time Compute 應用實踐指南 — OpenAI o1、DeepSeek R1、Gemini 與 Claude 實作範例與最佳實踐（2026）

本文為 test-time-compute-reasoning 系列第 application 階段，承接 foundation 階段的 API 控制面知識，提供實作程式碼、部署方法、成本優化與最佳實踐。

執行摘要（200-400 字）

Test-Time Compute（測試時計算擴展）是 2026 年 AI 領域的重大範式轉移，從預訓練時計算擴展到推理時計算擴展。主流模型（OpenAI o1、DeepSeek R1、Gemini 3.1 Pro、Claude Opus/Sonnet 4.6）均已支援此技術，透過在推理階段投入更多計算資源，顯著提升複雜任務（數學、程式設計、推理）的準確性。本文提供四大主流模型的實作指南，涵蓋 API 使用、部署方法、成本優化與最佳實踐。

核心發現：

  • OpenAI o1-preview 定價為每百萬 input tokens $15、output tokens $60，適合高精度推理任務 [DataCamp, 2026]
  • DeepSeek R1 提供 1.5B 至 70B 參數的開源版本，32B 和 70B 版本保留大部分推理能力且成本更低 [SitePoint, 2026]
  • Gemini 3.1 Pro 支援三個 thinking levels（Low/Medium/High），為當前最便宜的前沿模型 [Zemith, 2026]
  • Claude Opus 4.6 和 Sonnet 4.6 採用 adaptive thinking，讓模型動態決定思考深度，取代固定的 budget_tokens [Apiyi, 2026]
  • 研究顯示，最佳 reasoning token 預算約為 4K，超過此值可能因「過度思考」而降低性能 [Medium, Adnan Masood, 2026]

一、OpenAI o1 系列實作指南

1.1 Access Requirements 與定價

OpenAI o1 模型需要 Tier 5 使用等級才能透過 API 存取，當前速率限制為每分鐘 20 個請求 [DataCamp, 2026]。定價結構如下：

| 模型 | Input Tokens ($/1M) | Output Tokens ($/1M) | 適用場景 |

|------|---------------------|----------------------|----------|

| o1-preview | $15.00 | $60.00 | 高精度推理任務（數學、程式設計） |

| o1-mini | $3.00 | $12.00 | 平衡成本與性能的中等複雜度任務 |

| o1-pro | - | - | 最高精度版本（API 文件已發布） |

[來源：DataCamp, OpenAI API Documentation, 2026]

1.2 API 使用範例（Python）

OpenAI o1 模型目前缺少 function calling、streaming 和 system messages 功能，但 OpenAI 正在積極新增這些功能 [DataCamp, 2026]。


import openai

# 基礎 API 呼叫
response = openai.ChatCompletion.create(
    model="o1-preview",
    messages=[
        {"role": "user", "content": "Solve this complex math problem: ..."}
    ]
)

# 存取輸出
print(response.choices[0].message.content)

# 注意：reasoning tokens 不會顯示在輸出中，但會計入成本
# reasoning tokens 是模型內部思考過程使用的 tokens

[來源：DataCamp, Apidog, 2026]

1.3 Reasoning Tokens 成本控制

在 o1 模型中，reasoning tokens 指模型在思考過程中內部使用的 tokens。這些 tokens 會消耗計算資源並計入總成本，但不會顯示給終端用戶 [DataCamp, 2026]。

成本優化建議：

  • 對於簡單任務，使用 o1-mini 而非 o1-preview（成本降低 80%）
  • 監控 reasoning tokens 使用量（透過 API 回應的 usage 欄位）
  • 對於批量任務，考慮使用 batch API（未來功能）

二、DeepSeek R1 開源實作與部署

2.1 模型變體與硬體需求

DeepSeek R1 提供 1.5B、7B、8B、14B、32B 和 70B 參數的蒸餾版本，32B 和 70B 版本保留大部分推理能力且成本更低 [SitePoint, 2026; Daily.dev, 2026]。

| 模型大小 | 最低 VRAM | 推薦 RAM | 量化格式 | 適用硬體 |

|----------|-----------|----------|----------|----------|

| 7B | 4GB | 8GB | Q4_K_M (GGUF) | 消費級 GPU（RTX 3060） |

| 14B | 8GB | 16GB | Q4_K_M (GGUF) | RTX 3090 / 4070 Ti |

| 32B | 20GB | 32GB | Q4_K_M / AWQ | RTX 4090 / A100 40GB |

| 70B | 40GB+ | 64GB+ | AWQ | A100 80GB / Multi-GPU |

[來源：SitePoint, DEV Community, 2026]

2.2 本地部署（Ollama）

Ollama 提供最簡單的部署路徑，適合快速實驗和原型開發 [SitePoint, 2026]。


# 步驟 1：安裝 Ollama（一行指令）
curl -fsSL https://ollama.com/install.sh | sh

# 步驟 2：下載模型
ollama pull deepseek-r1:14b

# 步驟 3：執行推理
ollama run deepseek-r1:14b "Solve: What is the sum of prime numbers between 1 and 100?"

[來源：SitePoint, Compute Market, 2026]

2.3 生產部署（vLLM + Docker）

vLLM 提供更高吞吐量和更好的資源利用率，適合生產環境 [SitePoint, 2026]。


# docker-compose.yml
version: '3.8'
services:
  deepseek-r1:
    image: vllm/vllm-openai:latest
    runtime: nvidia
    environment:
      - MODEL_NAME=deepseek-ai/DeepSeek-R1-Distill-Qwen-32B
      - TENSOR_PARALLEL_SIZE=2  # 多 GPU 並行
      - MAX_MODEL_LEN=32768     # 上下文長度
      - GPU_MEMORY_UTILIZATION=0.9
    ports:
      - "8000:8000"
    volumes:
      - ./models:/models


# 啟動服務
docker-compose up -d

# API 呼叫（OpenAI 相容）
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "deepseek-ai/DeepSeek-R1-Distill-Qwen-32B",
    "messages": [{"role": "user", "content": "Explain quantum entanglement"}]
  }'

[來源：SitePoint, DigitalOcean, Portworx, 2026]

三、Gemini 與 Claude Thinking Modes

3.1 Gemini 3.1 Pro Thinking Levels

重要更新：Gemini 2.0 系列已於 2026 年 6 月 1 日終止服務，目前主流為 Gemini 3.1 Pro（2026 年 2 月 19 日預覽版發布）[Zemith, 2026]。

Gemini 3.1 Pro 支援三個 thinking levels（Low/Medium/High）：


import google.generativeai as genai

# 配置 thinking level
model = genai.GenerativeModel('gemini-3.1-pro')
response = model.generate_content(
    "Solve this complex coding problem: ...",
    generation_config={
        'thinking_level': 'High',  # Low / Medium / High
        'max_output_tokens': 65536  # 64K output window
    }
)

print(response.text)

Thinking Level 選擇指南：

  • Low：簡單問答、事實查詢
  • Medium：一般推理任務、程式碼生成
  • High：複雜數學、多步驟推理、系統設計

[來源：Zemith, SurePrompts, Google AI Docs, 2026]

3.2 Claude Adaptive Thinking（取代 Extended Thinking）

Claude Opus 4.6 和 Sonnet 4.6 引入 adaptive thinking，取代舊版的 budget_tokens 參數，讓模型動態決定思考深度 [Apiyi, 2026]。


import anthropic

client = anthropic.Anthropic(api_key="your-api-key")

# 使用 adaptive thinking（推薦）
response = client.messages.create(
    model="claude-opus-4.6",
    max_tokens=8192,
    thinking={
        "type": "adaptive"  # 讓 Claude 動態決定思考深度
    },
    messages=[
        {"role": "user", "content": "Design a distributed cache system"}
    ]
)

# 舊版 budget_tokens（已棄用）
# thinking={"type": "extended", "budget_tokens": 5000}  # ❌ 不再推薦

print(response.content)

Adaptive Thinking 四大優勢：

  1. 動態資源分配：簡單問題少思考，複雜問題深入思考
  2. 成本優化：避免固定 budget 導致的資源浪費
  3. 可見推理過程：可看到模型思考步驟（非僅結論）
  4. Interleaved Thinking：支援交錯思考與輸出（Sonnet 4.6 獨有）

[來源：Apiyi, APIScout, Claude API Docs, 2026]

四、Test-Time Compute 最佳實踐與成本優化

4.1 Compute-Optimal 策略

研究顯示，縮放推理計算比縮放模型參數更具成本效益 [arXiv:2408.03314, 2026]。關鍵發現：

  • 更小模型 + 進階推理演算法 = Pareto 最優：在成本與性能之間達到最佳平衡
  • Self-Consistent Sampling > Verification Methods：對於大多數實際推理預算，自一致性採樣比驗證方法更具計算效率
  • 最佳 Reasoning Token 預算 ≈ 4K：超過此值可能因「過度思考」而降低性能 [Medium, Adnan Masood, 2026]

[來源：arXiv, OpenReview, Hugging Face Blog, 2026]

4.2 批次處理優化

將多個推理任務分組為批次可顯著降低成本：


# 範例：批次處理數學問題
problems = [
    "Solve: 2x + 3 = 7",
    "Solve: x^2 - 5x + 6 = 0",
    "Solve: 3x + 2y = 8, x - y = 2"
]

# 並行處理（而非序列）
import asyncio

async def solve_batch(problems):
    tasks = [solve_with_o1(p) for p in problems]
    return await asyncio.gather(*tasks)

results = asyncio.run(solve_batch(problems))

效益：

  • 啟用多個推理任務的並行 token 生成
  • 減少處理多個測試案例時的冗餘計算成本

[來源：Hugging Face Blog, Redis Blog, 2026]

4.3 基礎設施優化三層模型

推理優化在三個層級運作 [Redis, 2026]：

  1. 模型層級：量化、蒸餾、剪枝
  2. 服務層級：快取、批次處理、動態路由
  3. 基礎設施層級：GPU 選擇、容器化、負載平衡

優先順序：從最高影響、最低努力的優化開始，通常是快取和服務策略。

六、知識缺口與未來方向

當前限制

  1. OpenAI o1 功能缺口：尚未支援 function calling、streaming、system messages [DataCamp, 2026]
  2. 最佳化研究不足：模型類型和問題難度對性能的影響尚不明確，無單一策略普遍優於其他策略 [OpenReview, 2026]
  3. 過度思考問題：超過 4K reasoning tokens 可能降低性能，但缺乏動態調整機制 [Medium, 2026]

未來研究方向（優化階段）

  • Process Reward Models (PRM) / Outcome Reward Models (ORM)：訓練方法深入研究
  • Tree Search 演算法優化：提升多步驟推理效率
  • 推理 Token 經濟學：成本與效能的精細化平衡模型
  • 動態思考預算分配：根據問題複雜度自動調整 reasoning tokens
