Conventional Commits 完整指南

技術概述

Conventional Commits 是一套輕量級的 Git 提交訊息規範，建立在 SemVer（語義化版本）之上，透過結構化的提交訊息格式，讓自動化工具能解析提交歷史、自動產生 CHANGELOG、自動決定版本號遞增策略。該規範已成為 2025-2026 年開源社群與企業專案的事實標準，被 Angular、Vue.js、Electron 等數千個專案採用。

---

核心規範（v1.0.0）

提交訊息結構


<type>[optional scope]: <description>

[optional body]

[optional footer(s)]

必要元素

  1. type（類型）：描述此次提交的性質
  2. description（描述）：簡潔摘要，緊接在冒號和空格之後
  3. scope（範圍）：可選，用括號包裹的名詞，描述影響的程式碼區段

標準類型對照表

| 類型 | 語義化版本 | 說明 | 範例 |

|------|-----------|------|------|

| feat | MINOR | 新增功能 | feat(hooks): 新增 behavior_tracker |

| fix | PATCH | 修復錯誤 | fix(guard): 修正 MinGW 路徑正規化 |

| docs | — | 文件變更 | docs: 更新 CLAUDE.md 架構段落 |

| style | — | 格式調整（不影響邏輯） | style: 統一縮排為 2 空格 |

| refactor | — | 重構（非新功能、非修復） | refactor(logger): 抽取共用模組 |

| perf | PATCH | 效能改善 | perf: 正則快取降低編譯開銷 |

| test | — | 測試相關 | test(guardian): 新增 43 個測試 |

| build | — | 建置系統變更 | build: 升級 pyyaml>=6.0 |

| ci | — | CI/CD 配置 | ci: 新增 pre-commit hook |

| chore | — | 雜項維護 | chore: 清理過期 loop-state |

破壞性變更（Breaking Changes）

兩種標記方式（任選或並用）：

  1. 感嘆號標記：feat!: 移除舊版 API 支援
  2. Footer 標記：
    3. 
feat(api): 重構認證流程

BREAKING CHANGE: Bearer token 格式從 v1 改為 JWT

破壞性變更觸發 MAJOR 版本遞增（無論 type 為何）。

Footer 慣例

遵循 Git Trailer 格式：


fix(parser): 修正 UTF-8 解析錯誤

Refs: #123
Reviewed-by: Alice
Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
  • Footer token 使用連字號替代空格（Reviewed-by 非 Reviewed by）
  • BREAKING CHANGE 是唯一例外，可用空格

自動化工具鏈生態系統

一、提交訊息生成與驗證

#### Commitizen（互動式提交嚮導）


# Python 版本安裝
pip install commitizen

# 互動式提交
cz commit

# 自動版本遞增
cz bump

# 產生 CHANGELOG
cz changelog

配置範例（pyproject.toml）：


[tool.commitizen]
name = "cz_conventional_commits"
version = "1.2.3"
tag_format = "v$version"
update_changelog_on_bump = true
major_version_zero = true

#### Commitlint（提交訊息驗證）


# 安裝
npm install --save-dev @commitlint/{cli,config-conventional}

# 配置 commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'scope-enum': [2, 'always', ['hooks', 'config', 'skills', 'tests']],
    'subject-max-length': [2, 'always', 72]
  }
};

搭配 Husky Git Hooks 在 commit 前自動驗證。

二、CHANGELOG 與版本管理

#### semantic-release（全自動發佈）


# .releaserc.yml
branches: ['main']
plugins:
  - '@semantic-release/commit-analyzer'
  - '@semantic-release/release-notes-generator'
  - '@semantic-release/changelog'
  - '@semantic-release/git'
  - '@semantic-release/github'

運作流程：

  1. 分析 feat / fix / BREAKING CHANGE 提交
  2. 自動決定版本號（MAJOR.MINOR.PATCH）
  3. 產生 Release Notes
  4. 建立 Git Tag + GitHub Release
  5. 發佈套件（npm/PyPI）

#### git-cliff（Rust 高效能 CHANGELOG 生成器）


# cliff.toml
[changelog]
header = "# Changelog\n"
body = """
{% for group, commits in commits | group_by(attribute="group") %}
## {{ group | upper_first }}
{% for commit in commits %}
- {{ commit.message | upper_first }} ({{ commit.id | truncate(length=7, end="") }})
{% endfor %}
{% endfor %}
"""
trim = true

[git]
conventional_commits = true
filter_unconventional = true
commit_parsers = [
  { message = "^feat", group = "Features" },
  { message = "^fix", group = "Bug Fixes" },
  { message = "^perf", group = "Performance" },
  { message = "^refactor", group = "Refactoring" },
  { message = "^docs", group = "Documentation" },
  { message = "^chore", group = "Miscellaneous" },
]

2025 新功能：

  • 統計變數：提交數量、時間跨度、conventional commit 比例
  • 路徑過濾：include_paths / exclude_paths 支援 monorepo
  • 子目錄自動偵測：自動找到 Git 根目錄和配置檔

#### Cocogitto（Rust 全方位工具箱）


# 安裝
cargo install cocogitto

# 驗證所有提交
cog check

# 自動版本遞增
cog bump --auto

# 產生 CHANGELOG
cog changelog

原生支援 monorepo 的多套件版本管理。

三、AI 輔助 Commit 訊息生成

#### gcm.js（Gemini 驅動）


# 使用流程
git add .
gcm  # 自動分析 staged diff → 產生 Conventional Commit

技術特點：

  • 使用 Gemini 2.5 Flash（速度快、成本低）
  • 智慧摘要：95,733 字元 diff 壓縮至 1,199 字元
  • 同時產生：commit message + branch name + PR description
  • 處理時間 < 1 秒（小型 diff）

#### aicommits（OpenAI 驅動）


npx aicommits  # 分析 git diff → 產生 commit message

#### Claude Code 內建

本專案的 git-smart-commit Skill 就是 AI 輔助 Conventional Commits 的實例：

  1. 取得 git diff --staged 變更
  2. 依目錄/檔案模式自動分群（9 種 scope 規則）
  3. 產生符合 Conventional Commits 的訊息
  4. 自動加上 Co-Authored-By footer

Monorepo Scope 策略

Scope 命名慣例

| 策略 | 適用場景 | 範例 |

|------|---------|------|

| 套件名稱 | npm monorepo | feat(ui-button): 新增 disabled 狀態 |

| 目錄路徑 | 一般 monorepo | fix(hooks): 修正 guard 規則 |

| 功能模組 | 功能導向 | refactor(auth): 重構 JWT 驗證 |

| 無 scope | 跨模組變更 | chore: 更新 CI 配置 |

自動 Scope 偵測


// prepare-commit-msg hook 範例
const changedFiles = execSync('git diff --cached --name-only').toString();
const scope = detectScope(changedFiles);  // hooks/ → hooks, config/ → config

Semantic Versioning 對映規則


提交類型          → 版本遞增
─────────────────────────────
BREAKING CHANGE   → MAJOR (x.0.0)
feat              → MINOR (0.x.0)
fix / perf        → PATCH (0.0.x)
docs / style / refactor / test / chore / ci / build → 無版本變更

CMU SEI 的輕量實作（15 行 Python）：


import subprocess, re

msg = subprocess.check_output(['git', 'log', '-1', '--format=%s']).decode()
tag = subprocess.check_output(['git', 'describe', '--tags', '--abbrev=0']).decode().strip()
major, minor, patch = map(int, tag.lstrip('v').split('.'))

if 'BREAKING CHANGE' in msg or msg.startswith('major'):
    major += 1; minor = 0; patch = 0
elif msg.startswith('feat'):
    minor += 1; patch = 0
else:
    patch += 1

new_tag = f'v{major}.{minor}.{patch}'

常見陷阱與最佳實踐

陷阱

  1. scope 爆炸：過多的 scope 讓歷史難讀 → 限制 scope 清單（commitlint scope-enum）
  2. type 誤用：refactor 改變了行為應是 fix 或 feat → type 決定版本遞增，須精確
  3. description 過長：超過 72 字元在 git log --oneline 會被截斷 → 詳情放 body
  4. 忽略 body：複雜變更只有一行描述 → body 解釋「為什麼」而非「做了什麼」
  5. 破壞性變更遺漏：API 改變但未標記 ! 或 BREAKING CHANGE → 用戶升級時出錯

最佳實踐

  1. 原子提交：一個 commit 只做一件事，避免混合 feat + fix
  2. 先 scope 後 type：先確定影響範圍，再判斷變更性質
  3. 自動化驗證：commitlint + Husky 在 commit 時攔截不合規訊息
  4. CI 雙重檢查：PR 合併前用 CI job 驗證所有提交訊息
  5. Squash Merge：每個 PR 壓縮為一個 conventional commit，保持主線清潔
  6. AI 輔助 + 人工審查：AI 生成初稿，人工確認 type 和 scope 正確性

與 daily-digest-prompt 專案的關聯

現有整合

  1. git-smart-commit Skill：專案已有 AI 輔助 Conventional Commits 的 Skill（skills/git-smart-commit/SKILL.md），定義了 9 種目錄→scope 映射規則

可改進方向

  1. git-cliff 整合：引入 git-cliff 自動產生 CHANGELOG.md，追蹤專案演進歷史
  2. commitlint 驗證：在 Git pre-commit hook 加入提交訊息格式驗證
  3. scope 標準化：將 9 種 scope 規則寫入 commitlint 配置，確保一致性
  4. 版本管理：目前 SKILL.md 有 version 欄位但無自動遞增機制，可考慮引入語義版本自動化
  5. CHANGELOG 自動化：每次 git-push 自動任務執行時，順便更新 CHANGELOG
