一、專案概述

Composio（ComposioHQ/composio）是由 Composio 團隊開發的開源 AI Agent 工具整合平台，定位為「讓 AI Agent 從意圖轉化為行動」的基礎設施層。截至 2026 年 3 月已累積 27,300+ Stars、4,500+ Forks，採用 MIT License。

核心數據

| 指標 | 數值 |

|------|------|

| GitHub Stars | 27,300+ |

| Forks | 4,500+ |

| 支援工具包 | 1,000+ |

| 支援 AI 框架 | 22+（TS 10 + Python 12） |

| 授權 | MIT License |

| 語言 | TypeScript + Python（pnpm monorepo） |

| API 版本 | v3（backend.composio.dev/api/v3） |

核心價值主張

Composio 解決的核心問題是「AI Agent 如何可靠地與真實世界互動」——包含認證管理、工具發現、安全執行和上下文管理四大面向。傳統做法需要為每個外部服務手動整合 OAuth、處理 API 差異、管理 token 刷新，Composio 將這些複雜性封裝為統一的 SDK 層。

二、系統架構深度分析

2.1 三層架構

Composio 採用經典三層架構，但以 Hub-and-Spoke 模式 連接多個 AI 框架：


┌─────────────────────────────────────────┐
│           開發者層（Developer Layer）       │
│  TypeScript SDK (10 providers)           │
│  Python SDK (12 providers)               │
└────────────────┬────────────────────────┘
                 │ OpenAPI-based unified schema
┌────────────────▼────────────────────────┐
│           平台核心（Platform Core）        │
│  Backend API → Tool Router → MCP Server │
│  Auth System → File Service → Triggers  │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│        外部服務（500+ Integrations）       │
│  GitHub, Gmail, Slack, Notion, etc.      │
└─────────────────────────────────────────┘

2.2 Hub-and-Spoke Provider Pattern（核心設計模式）

這是 Composio 最值得借鑑的架構決策。核心 SDK 作為 Hub，各 AI 框架的 Provider 作為 Spoke：

BaseProvider 介面（~100 行即可新增框架）：

  • wrapTools()：將統一 OpenAPI schema 轉為框架特定格式
  • wrapTool()：單一工具轉換
  • handleToolCalls()：處理 LLM 回應（非 Agentic 模式）

各框架的格式轉換對照：

| 框架 | 工具格式 |

|------|----------|

| OpenAI | { type: "function", function: { name, description, parameters } } |

| Anthropic | { name, description, input_schema } |

| LangChain | Tool objects with _call() methods |

| Vercel AI SDK | CoreTool with Zod schemas |

| MCP | Standard Model Context Protocol format |

架構價值：新增框架支援不需修改核心邏輯，只需實作 BaseProvider 介面（約 100 行程式碼），實現了真正的 Open-Closed Principle。

2.3 工具執行管線（Tool Execution Pipeline）

Composio 區分兩種執行模式：

非 Agentic 模式（OpenAI、Anthropic、Gemini、Vercel）：

  1. 開發者呼叫 composio.tools.get(userId, {toolkits})
  2. LLM 分析格式化工具並生成 tool_calls
  3. 開發者顯式呼叫 provider.handleToolCalls(userId, response)
  4. 結果手動注入回 Agent 繼續推理

Agentic 模式（LangChain、CrewAI、LlamaIndex、OpenAI Agents）：

  1. 工具透過 composio.tools.get() 註冊至框架
  2. 框架自主決定工具執行，無需開發者介入
  3. Provider 將 executeFn 閉包嵌入工具，含 session context
  4. 框架自動以結果繼續推理循環

後端處理管線增強：

  • 版本驗證：禁止生產環境使用 "latest" 版本
  • Modifier 系統：schema/beforeExecute/afterExecute 三階段修改器
  • 檔案處理：S3 presigned URLs 透明處理上傳/下載
  • 大型回應：伺服器端處理，防止 context overflow

2.4 五層認證階層（Authentication Hierarchy）

| 優先級 | 方法 | 適用場景 |

|--------|------|----------|

| 1 | 顯式 connectedAccounts 映射 | 多帳號用戶（工作+個人） |

| 2 | 自訂 authConfigs | 生產應用的自訂 OAuth |

| 3 | Tool Router 自動建立 | 透明 OAuth app 建立 |

| 4 | Composio 託管認證 | 快速原型開發 |

| 5 | 錯誤狀態 | 無可用認證機制 |

安全模型：所有 token 伺服器端儲存（永不暴露給客戶端），user_id 範圍隔離所有帳號，Tool Router session 使用 presigned URLs 嵌入憑證。

2.5 MCP（Model Context Protocol）整合

Composio 在 mcp.composio.dev 提供 MCP 伺服器，實現：

  • 500+ 工具包的標準化 MCP 端點
  • 自動工具發現與 schema 暴露
  • Session 繼承 Tool Router 的認證上下文
  • 解耦工具提供者與特定 LLM 實作

關鍵洞察：MCP 方法將工具提供者從特定 LLM 實作中解耦，使 Composio 工具包可跨多個模型提供者運作，無需自訂 provider 程式碼。

三、Agent Orchestrator 深度分析

ComposioHQ 另開源了 agent-orchestrator（3,900+ Stars），專為並行 AI 編碼 Agent 設計的編排平台。

3.1 八插槽插件系統

| 插槽 | 預設 | 替代方案 |

|------|------|----------|

| Runtime | tmux | docker, k8s, process |

| Agent | claude-code | codex, aider, opencode |

| Workspace | worktree | clone |

| Tracker | github | linear |

| SCM | github | — |

| Notifier | desktop | slack, composio, webhook |

| Terminal | iterm2 | web |

| Lifecycle | 核心 | — |

3.2 事件驅動反應架構（取代集中式狀態機）

傳統多 Agent 系統使用集中式 Planner-Executor 模式，Agent Orchestrator 則採用事件驅動反應鏈：


Issue → Workspace 建立 → Agent 執行 → PR 建立
→ CI 監控 → 反應觸發 → 可選重啟 Agent

核心設計決策：不做集中式 planning，而是讓每個 Agent 獨立處理完整任務，系統僅在事件發生時（CI 失敗、review 留言）被動反應。這使系統可擴展到 30 個並行 Agent 而無集中瓶頸。

3.3 CI 失敗自修復回饋迴圈

| 組件 | 功能 |

|------|------|

| Tracker plugin | 監聽 GitHub/Linear 狀態變更 |

| Reaction engine | 將事件匹配到設定的反應 |

| Re-spawn action | 自動建立帶失敗上下文的新 Agent session |

| Agent ingestion | Agent 讀取 CI 輸出、診斷問題、推送修復 |

實際成效：41 次 CI 失敗跨 9 個分支，全部由 Agent 自行修正，整體 CI 成功率 84.6%。

3.4 三層程式碼審查機制

| 審查者 | 數量 | 角色 |

|--------|------|------|

| Cursor Bugbot（自動化） | 377 次 | 行內靜態分析 |

| AI Agents | 316 次 | 基於 bot 留言自我修正 |

| 人類 | 13 次 | 僅在需要判斷時介入 |

700 則自動化程式碼審查留言中，Agent 立即修復 ~68%、有意保留 ~7%、延後至未來 PR ~4%。

3.6 Activity Detection via JSONL

關鍵技術：Agent Orchestrator 直接讀取 Claude Code 的 JSONL session 檔案來偵測 Agent 狀態（生成中/等待中/閒置/完成），而非依賴 Agent 自我報告。原因：「Agent 會說謊，或至少會混淆自己的狀態」。

四、對 Daily-Digest-Prompt 系統的七大借鑑啟示

啟示 1：Hub-and-Spoke Provider Pattern → Skill Provider 抽象

Composio 做法：核心 SDK 定義統一 schema，各框架 Provider 僅需 ~100 行實作。

借鑑：目前 skills/ 下各 SKILL.md 是文件驅動，可考慮為每個 Skill 定義標準化介面（input schema + output schema + execute），使新增 Skill 更系統化。

啟示 2：事件驅動反應 → FSM 增強

Composio 做法：webhook 事件觸發反應鏈，無集中式 planner。

借鑑：目前 state/run-fsm.json 是集中式狀態機。可增加事件驅動層——當 Todoist 任務完成、知識庫更新、CI 結果返回時，自動觸發下游反應，減少輪詢式排程的延遲。

啟示 3：八插槽插件架構 → 可插拔組件

Composio 做法：8 個可替換插槽（Runtime/Agent/Workspace/Tracker/Notifier 等）。

借鑑：目前通知固定用 ntfy、排程固定用 Windows Task Scheduler。可抽象出 Notifier/Scheduler 介面，未來支援 Slack/Line/Telegram 等替代方案，提高系統適應性。

啟示 4：JSONL Activity Detection → Hook 日誌增強

Composio 做法：讀取 JSONL 檔偵測 Agent 狀態。

借鑑：hooks/post_tool_logger.py 已產出 JSONL 結構化日誌。可增加即時狀態推斷——根據最近 N 條工具呼叫記錄判斷 Agent 是否卡住（loop detection 已有 LoopDetector），進一步實現自動殭屍偵測。

啟示 5：CI 自修復迴圈 → 自動任務錯誤恢復

Composio 做法：CI 失敗自動注入上下文至新 Agent session，84.6% 成功率。

借鑑：目前 run-agent-team.ps1 的殭屍偵測（running>2h→timeout）可進化為「失敗上下文注入」——將上次失敗的錯誤訊息、日誌摘要注入重試 session，提高自動恢復成功率。

啟示 6：五層認證階層 → API 降級策略增強

Composio 做法：5 層優先級，從顯式指定到託管認證的漸進式降級。

借鑑：目前 config/cache-policy.yaml 有快取降級，circuit-breaker-utils.ps1 有斷路器。可借鑑 Composio 的分層策略，為每個 API 設定多個認證方式（主 token → 備用 token → 降級為快取），提高系統韌性。

啟示 7：MCP 標準化 → 工具介面統一

Composio 做法：MCP 伺服器提供 500+ 工具的標準化端點。

借鑑：目前 Skill 呼叫各自使用不同的 curl/API 格式。可考慮為常用 API（Todoist、ntfy、知識庫）包裝為 MCP 端點，使工具呼叫格式統一、可測試性提升，並與 Claude Code 的 MCP 整合原生相容。
