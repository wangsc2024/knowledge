今日任務涉及技術：RAG 知識庫、Vite 靜態站同步、OpenClaw 六角架構、Mechanistic Interpretability、論證映射、測試與品質閘門

本次研究主題：Vite 靜態知識庫同步管線（因知識庫尚無此主題的深入研究）

技術概述

Vite 靜態知識庫同步管線的核心，不只是把 Markdown 或 JSON 丟給 vite build，而是把「資料同步」、「build-time 載入」、「本機預覽驗證」與「邊緣部署」拆成可觀測的四段流程。對 daily-digest 這類知識密集型專案來說，先由同步腳本把知識庫內容整理成穩定資料檔，再交給 Vite 或 VitePress 在建置期讀取，最後以 dist/ 為唯一發布產物，能同時降低執行期依賴、提升 SEO 與部署穩定性。2025 年後 Vite 7 與 Cloudflare Vite Plugin GA 也讓這條路線更清楚：靜態站可維持簡單，若未來需要 API 或 Workers 能力，再沿同一工具鏈逐步擴展。

核心概念與原理

1. 四段式責任分離

  1. 資料同步層：由 sync_knowledge.py 這類腳本向 KB API 取資料，做過濾、去敏感、增量更新，輸出到 public/data/ 或其他建置期可讀目錄。
  2. 建置層：Vite 的標準產物是 dist/，應把它視為唯一可部署輸出，而不是直接部署原始資料目錄。
  3. 載入層：若站點偏內容導向，可用 VitePress data loader，在 build-time 將遠端或本地資料轉成 JSON，避免把抓取邏輯帶進瀏覽器。
  4. 部署層：Cloudflare Pages 適合純靜態發布；若未來要把搜尋 API、權限邏輯或預處理放到邊緣，可再轉向 Workers + @cloudflare/vite-plugin。

2. Build-time data 的邊界

VitePress 官方明確指出，data loader 只在 build-time 執行，結果會序列化成 JSON 放進最終 bundle。這代表：

  • 適合放「索引、摘要、前言資料、文章中繼資料」。
  • 不適合直接塞入過大的全文資料，否則會放大 client bundle。
  • 若資料來源是本地檔案，應搭配 watch 或 glob，讓內容變更能觸發快速更新。

3. Vite 7 對此題的意義

Vite 7 已成為目前官方主線，要求 Node.js 20.19+ 或 22.12+，且延續 Environment API。這對知識站專案的實務意義有兩點：

  • 如果本機或 CI 還停在舊版 Node，建置腳本可能先在 runtime 相容性上出問題。
  • 若未來要把靜態站升級成「靜態站 + Workers API」混合模式，Vite 7 的環境模型已經是正確演進方向。

4. Cloudflare 發布限制不能忽略

Cloudflare Workers Static Assets 目前對每個 Worker 版本的靜態檔有數量與單檔大小限制。知識庫網站如果未來筆記數持續成長，就必須留意：

  • Free / Paid 的檔案數上限差異。
  • 單檔大小限制。
  • _headers、_redirects 的規則上限。
  • 若要使用提高後的檔案數限制，Wrangler 版本也有最低要求。

這些限制不是理論問題，而是知識站規模擴大後會直接撞到的部署風險。

最佳實踐與常見陷阱

最佳實踐

  • 把同步腳本和前端建置解耦：python sync_knowledge.py 成功後才進 npm run build，失敗時能準確定位問題在資料還是前端。
  • 把 vite preview 放進驗證步驟：官方明確說 vite preview 是用來本機驗證 dist/，不是正式伺服器；但它非常適合作為 CI 或人工檢查點。
  • 只把必要索引放進前端：完整正文保留在靜態頁面或分頁資料檔，前端僅載入搜尋索引、標題、標籤、摘要。
  • 預留部署平台轉向空間：今天可以用 Pages；明天若需要邊緣 API，不必重寫整個工具鏈，只需沿 Vite/Cloudflare 路徑升級。
  • 將資產規模納入監控：筆記篇數、JSON 檔大小、單次 build 時間，都應納入同步腳本的輸出報表。

常見陷阱

  • 把 KB API 直接綁在前端 runtime：這會讓網站失去真正的靜態優勢，也增加不可用風險。
  • 把全文索引一次塞進單一 JSON：初期方便，但後期很容易造成 bundle 膨脹與部署限制問題。
  • 只測 npm run dev 不測 npm run build / vite preview：dev server 可用，不代表靜態輸出與路徑都正確。
  • 忽略 base 或部署路徑差異：若未來從 custom domain 換到 repo path 或不同平台，靜態資源路徑可能直接壞掉。
  • 把 Pages 與 Workers 當成完全一樣：兩者都能服務靜態內容，但能力邊界與限制並不完全相同。

與本專案的關聯

本專案在 templates/auto-tasks/git-push.md 已把知識站流程拆成三步：先執行 python sync_knowledge.py，再 npm run build，最後做隱私檢查與推送。這其實就是一條非常典型的 Vite 靜態知識庫同步管線。

對 daily-digest-prompt 的直接應用建議如下：

  • 將 sync_knowledge.py 的輸出分成 notes-index.json、tags.json、recent.json 等小檔，而不是一個超大總檔。
  • 在 build 後增加 vite preview 或等效 smoke test，避免「同步成功但站點壞掉」的情況。
  • 若知識站未來要加入站內搜尋 API、受保護內容或邊緣快取策略，可優先評估 Cloudflare Vite Plugin + Workers，而不必直接改換另一套前端框架。
  • 將靜態資產數量、最大 JSON 檔大小與建置耗時寫入同步日誌，提前觀察是否接近平台限制。

程式碼範例


// notes.data.ts
// 在 build-time 讀取同步腳本產出的 JSON，前端只拿輕量索引
import fs from 'node:fs'
import path from 'node:path'
import { defineLoader } from 'vitepress'

type NoteIndex = {
  id: string
  title: string
  tags: string[]
  summary: string
  updatedAt: string
}

declare const data: NoteIndex[]
export { data }

export default defineLoader({
  watch: ['../public/data/notes-index.json'],
  async load() {
    const file = path.resolve(process.cwd(), 'public/data/notes-index.json')
    return JSON.parse(fs.readFileSync(file, 'utf-8')) as NoteIndex[]
  }
})


{
  "scripts": {
    "sync": "python sync_knowledge.py",
    "build": "vite build",
    "preview": "vite preview --port 4173",
    "publish": "npm run sync && npm run build"
  }
}

這種做法的重點是：同步腳本負責資料正確性，Vite/VitePress 負責產物生成，部署工具只處理 dist/。責任清楚，問題才容易追。
