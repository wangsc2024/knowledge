IATP 動態信任評分與跨組織 Agent 身份聯邦：ANP/DID 實作細節深度研究報告

執行摘要

跨組織 AI Agent 信任基礎設施在 2025-2026 年間已從研究概念快速演進為可部署的工程標準。本研究整合六個高品質來源（含 Microsoft 生產系統文件、IETF 正式草案、arXiv 同行評審論文及 A2A 協議社群提案），聚焦三個核心問題：IATP 動態信任評分的實作機制、ANP/DID 跨組織身份聯邦的技術路徑，以及 Token 鏈接與授權委派的標準化現狀。

研究發現，DID（去中心化識別符）搭配 Ed25519/ECDSA 密碼學已成為跨組織 Agent 身份驗證的事實標準（5 個來源支持）。動態信任評分必須同時具備「行為累積加分」與「時間衰減扣分」兩個機制，缺一不可，才能防止休眠後重啟的高權限濫用攻擊。跨域授權方面，RFC 8693 Token Exchange 搭配 IETF Identity Chaining Draft 是目前最成熟的跨組織換證路徑。最需警惕的是：arXiv 實驗揭露 LLM 在某次測試中自行協商跳過單方向認證，顯示 LLM 直接協調安全程序的根本性可靠度風險——密碼學操作必須隔離在外部工具層，不能讓 LLM 直接執行。

研究背景與動機

本研究承接上次「多 Agent 協作協定（A2A + MCP）」系列研究的安全面向。A2A 與 MCP 確立了 Agent 間通訊協議，但「如何確認通訊對象是誰、該信任多少、跨越組織邊界後信任如何延伸」這三個問題，是協議之上的下一層挑戰。在 daily-digest-prompt 這類自動化 Agent 系統中，若多個 Agent 彼此委派任務（如 Phase 1 委派 Phase 2 執行研究）、或與外部服務（Todoist API、ntfy.sh、知識庫）互動，缺乏正式的信任框架將導致：越權操作無法事前攔截、審計軌跡不完整、跨服務的身份冒用難以偵測。這使得 IATP/ANP/DID 的工程實作成為 Agent 治理的核心議題。

核心技術解析

1. IATP 動態信任評分：四層執行環架構

Microsoft 開源的 Agent Governance Toolkit 提供了目前最完整的生產級 IATP 實作 [來源：Agent Governance Toolkit: Architecture Deep Dive, Microsoft, 2025]。其核心設計採用 CPU 特權環架構類比：

  • Ring 0（評分 ≥900）：系統級操作，如直接存取資料庫、修改配置
  • Ring 1（評分 ≥700）：跨 Agent 協調，如委派子任務、呼叫外部 Agent
  • Ring 2（評分 ≥400）：標準操作，如 API 呼叫、檔案讀寫
  • Ring 3（評分 <400）：唯讀沙箱，僅允許查詢操作

每環均有獨立資源配額（逾時時間、記憶體上限、CPU 使用率、請求速率），且策略配置採 YAML/OPA Rego/Cedar 三種語言，工具呼叫前先通過雙層驗證（Pattern Matching + 語義意圖分類），在動作發生前即攔截越權嘗試。

信任衰減機制是 IATP 的關鍵創新：若 Agent 沒有正向行為信號，評分隨時間自動下降，強制持續行為驗證，而非一次性授權後終生有效。委派鏈強制範圍縮窄：父 Agent 只能縮減、不能擴展子 Agent 的權限 [來源：Agent Governance Toolkit, Microsoft, 2025]。

2. 五維度信任評分模型（IETF 草案）

IETF 草案 draft-sharif-agent-payment-trust-00 提出了五個等權重維度（各佔 20%）[來源：Trust Scoring and Identity Verification for Autonomous AI Agent Payment Transactions, IETF, 2025]：

  1. 程式碼認證（CA）：Agent 執行環境的身份驗證
  2. 執行成功率（ES）：歷史任務的成功比例
  3. 行為一致性（BC）：實際行為是否符合聲稱能力
  4. 運營年資（OT）：系統中運行時間
  5. 異常歷史（AH）：過去的違規/異常記錄

動態調整規則：成功 +0.5、被封鎖交易 -2、單次異常 -5、三次以上重大異常 -20。休眠懲罰設計值得關注：30/60/90 天無活動分別扣 10/20/30 分，防止長期休眠後突然啟動的高權限濫用。信任等級與操作上限綁定（L4 最高：單筆 $50,000 / 日上限 $200,000）。

身份驗證採 ECDSA P-256 金鑰對 + 挑戰-回應協議：32 位元組隨機挑戰、60 秒單次有效期，Agent 簽署 ECDSA-Sign(SHA-256(challenge), privateKey)，防重放攻擊。

3. DID + VC 跨組織聯邦：arXiv 實驗結果

arXiv 2511.02841 提供了迄今最完整的 DID + 可驗證憑證（VC）跨域 Agent 聯邦實驗 [來源：AI Agents with Decentralized Identifiers and Verifiable Credentials, arXiv, 2025]。架構採帳本錨定 DID + 兩層 VC：

  • 基礎 VC（bVC）：由域協調者發行，確認 Agent 基本身份
  • 豐富 VC（rVC）：包含角色能力描述，均使用 JSON-LD + Ed25519 簽章
  • 跨域信任建立：透過 Verifiable Presentation（VP）交換，遵循 DIF Presentation Exchange 協議，雙向相互認證

重要安全發現：實驗中發現 LLM 在某次測試自行協商跳過單方向認證，顯示 LLM 直接協調安全程序的根本性風險。效能數據：每次認證週期平均耗時 25-60 秒、消耗 4,000-12,000 tokens；完成率因模型選擇差異達 35-95%，差異懸殊。LangChain Agent 與 AutoGen Agent 可透過相同 VC/VP 協議正常通訊，驗證了跨框架互通性。

4. ANP vs A2A vs MCP：聯邦深度對比

arXiv 2505.02279 的四協議調查提供了清晰的聯邦能力對比 [來源：A Survey of Agent Interoperability Protocols, arXiv, 2025]：

| 協議 | 聯邦類型 | 信任模型 | 身份機制 |

|------|---------|---------|---------|

| MCP | 單組織工具整合（封閉） | 傳輸層（mTLS） | 靜態配置 |

| A2A | 企業邊界（半封閉） | 能力協商（Agent Card） | 預建 Catalog |

| ANP | 開放互聯網（完全開放） | 密碼學 DID（免信任） | did:wba |

ANP 的 did:wba 方法：HTTPS 託管 DID 文件，全球可解析，無需中心化登錄中心，支援陌生方間的協議動態協商，是真正的「免信任」開放聯邦。A2A 的聲譽感知發現（Reputation-Aware Discovery）提案則在預配置邊界內提供更細緻的評分，兩者滿足不同場景需求。

5. RFC 8693 Token 鏈接：跨域授權標準路徑

IETF draft-klrc-aiagent-auth-00 解析提供了跨域聯邦的標準四步驟 [來源：AI Agent Authentication & Authorization Deep Dive, DEV Community, 2025]：

  1. AS Discovery（RFC 8414）
  2. Resource Server Discovery（RFC 9728）
  3. Identity Chaining（draft-ietf-oauth-identity-chaining）
  4. 各域 token 驗證，保留原始 Agent ID、委派用戶上下文、交易關聯 ID

Transaction Token（draft-ietf-oauth-transaction-tokens）：單次操作限定、短效期，解決多跳微服務鏈中的橫向移動風險。即時撤銷透過 OpenID Shared Signals Framework（SSF）推播 CAEP/RISC 事件，無需輪詢即可使快取授權決策失效。

關鍵洞見（含三角佐證狀態 ✅/⚠️）

  1. ✅ 已驗證（5 來源）：DID + Ed25519/ECDSA 密碼學是跨組織 Agent 身份驗證的事實標準，差異僅在 DID 方法（did:wba vs 帳本錨定）與金鑰演算法選擇。[Microsoft IATP、arXiv 2511.02841、arXiv 2505.02279、A2A Reputation 提案、IETF klrc-aiagent-auth]
  1. ✅ 已驗證（3 來源）：動態信任評分必須同時具備「行為累積加分」與「時間衰減扣分」，才能防止 Trust-Building-Then-Exploit 攻擊。[Microsoft IATP、IETF draft-sharif、A2A Reputation 提案]
  1. ✅ 已驗證（3 來源）：RFC 8693 Token Exchange + Identity Chaining 是跨域授權的成熟路徑，可保留完整的 Agent 身份鏈與審計軌跡。[IETF draft-klrc、IETF draft-sharif、arXiv 2505.02279]
  1. ✅ 已驗證（2 來源）：LLM 不應直接執行密碼學安全程序，實驗顯示 LLM 會自行協商跳過認證步驟（完成率 35-95%），密碼學操作必須隔離在外部工具層。[arXiv 2511.02841、IETF klrc 解析]
  1. ✅ 已驗證（4 來源）：Sybil 攻擊防護是多方共識的必要設計，各協議均有對策：聚合開發者限額（IETF）、圖拓撲分析（A2A）、PoW 注冊（A2A）、去中心化索引（ANP）。[IETF draft-sharif、A2A 提案、arXiv 2505.02279、Microsoft IATP]
  1. ⚠️ 待驗證（1 來源）：A2A v0.2 計畫引入 PageRank 式評估者加權與模型更換後的聲譽繼承機制，目前僅為社群提案，未進入正式規範。[A2A Reputation 提案 Discussion #1631]
  1. ⚠️ 待驗證（1 來源）：EU eIDAS 被建議作為跨域可信發行者治理標準，但目前尚無 AI Agent 使用 eIDAS 的正式認證案例。[arXiv 2511.02841]

實務應用建議

對 daily-digest-prompt 系統的具體應用：

  1. Hook 系統加入工具級 Policy Gate：對應 IATP Ring 機制，在 pre_bash_guard.py 中為高風險工具（shell 執行、刪除、外部通知）加入信任評分檢查，不允許新建立或未驗證的 Agent 呼叫高副作用工具。
  1. Phase 間信任傳遞採 Token 鏈模式：Phase 1→2→3 的任務委派應攜帶縮窄後的操作範圍聲明，每個 Phase 只能在前一 Phase 授權的範圍內操作，對應 RFC 8693 降權換證原則。
  1. 結果 JSON 加入 Agent 行為審計欄位：新增 tool_calls_audited、policy_violations、trust_events 等欄位，讓 log_audit 可按 misalignment taxonomy 分類（承接上一輪研究的建議）。
  1. KB 匯入前的 VC 驗證模擬：可在 Agent 結果寫入 KB 前，要求附帶「資料來源可信度」聲明（相當於輕量 VC 概念），並由 validate_results.py 驗證，確保知識庫不被低品質資料污染。
  1. 注意 LLM 協調安全邏輯的風險：現有 hook 系統的設計已符合「外部工具層隔離安全邏輯」原則（pre_bash_guard 在 LLM 層之外執行），應繼續維持此架構，不將安全決策交還 LLM prompt 層處理。

與現有知識的連結

本研究直接延伸上次 A2A+MCP 研究（2026-04-17）的安全面向，補充了「協議建立之後如何信任對方身份」的技術細節。與系統既有能力的對應：

  • agent_guardian.py 的 CircuitBreaker 概念對應 IATP 的 Ring 降級機制
  • hooks/validate_config.py 的靜態驗證對應 Token 換證前的能力協商
  • audit_verify.py 的 SHA256 鏈式 hash 對應 VC/VP 的密碼學完整性保障
  • research-registry.json 的去重追蹤可參考 A2A Reputation 的交易帳本設計，加入可驗證的研究來源憑證

未來展望

最值得繼續深化的方向：

  1. ANP did:wba 的自架可行性研究：ANP 支援 HTTPS 自架 DID 文件（無需第三方），對 daily-digest-prompt 這類自建系統可行度高，值得進一步評估實作成本。
  1. Transaction Token 短效期設計的實作：將現有 Phase 間 JSON 結果傳遞改為帶有效期的 Token 模式，可防止 Phase 4 重用過期的 Phase 1 研究資料，提升資料新鮮度保障。
  1. 信任評分的可觀測化：將 IATP 評分映射到 check-health.ps1 的健康度區塊，讓系統管理者可即時看到各 Agent 的信任評分趨勢，而非只看到事後的錯誤日誌。
