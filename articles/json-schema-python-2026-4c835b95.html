一、技術概述

JSON Schema 是一個用於描述和驗證 JSON 資料結構的宣告式語言，提供一致的方式定義資料的型別、格式、約束條件與結構關係。其核心價值在於：

  1. 資料契約標準化：在前後端、微服務、API 閘道之間建立明確的資料契約
  2. 自動化驗證：在資料進入系統前即時攔截格式錯誤，降低除錯成本
  3. 文件化：Schema 本身即為可執行的文件，確保規格與實作同步
  4. 跨語言支援：JavaScript、Python、Go、Java 等語言皆有標準實作庫

在 2026 年，JSON Schema 已廣泛應用於 LLM 結構化輸出驗證（Instructor、PydanticAI）、API 規格定義（OpenAPI）、配置檔驗證（Kubernetes CRD）等場景。

二、規範演進：Draft 7 → 2019-09 → 2020-12

2.1 版本歷史與里程碑

| 版本 | 發布時間 | 主要變化 |

|------|---------|---------|

| Draft 4 | 2013 | 初代穩定版本，確立核心關鍵字 |

| Draft 6 | 2017 | 新增 const、propertyNames、if/then/else |

| Draft 7 | 2018 | 新增 readOnly、writeOnly、contentMediaType |

| Draft 2019-09 | 2019 | 引入 $recursiveRef、unevaluatedProperties |

| Draft 2020-12 | 2020 | 重新設計陣列/元組處理、動態引用重構 |

2.2 Draft 2020-12 的關鍵改進

根據 JSON Schema 官方 release notes，2020-12 針對 Draft 7 的三大痛點進行了改進：

#### （1）陣列與元組處理重新設計

問題：Draft 7 的 items 關鍵字同時用於定義「陣列」（所有元素相同型別）與「元組」（固定位置不同型別），導致初學者經常誤用。


// Draft 7：混淆的 items 用法
{
  "items": {"type": "string"}  // 陣列（所有元素都是 string）
}
{
  "items": [{"type": "string"}, {"type": "number"}]  // 元組（第 1 個 string、第 2 個 number）
}

改進：2020-12 拆分為 prefixItems（元組前綴）與 items（陣列所有元素）。


// Draft 2020-12：語義明確
{
  "prefixItems": [{"type": "string"}, {"type": "number"}],  // 前 2 個元素型別
  "items": {"type": "boolean"}  // 第 3 個之後的元素型別
}

#### （2）動態引用機制重構

問題：Draft 2019-09 的 $recursiveRef 只能處理遞迴結構，語義受限。

改進：2020-12 引入 $dynamicRef / $dynamicAnchor，支援更廣泛的動態解析場景（非遞迴的動態引用、具名錨點）。

#### （3）詞彙（Vocabulary）模組化

改進：將 format 拆分為 format-annotation（純註解）與 format-assertion（斷言驗證），將 unevaluatedProperties / unevaluatedItems 抽離為獨立詞彙，讓實作庫可選擇性支援。

2.3 向後相容性與遷移建議

  • Draft 7 → 2020-12 遷移：大多數 schema 可直接使用，僅需調整 items + additionalItems 為 prefixItems + items
  • 實作庫支援狀況（2026）：jsonschema 4.26+、fastjsonschema 2.20+、ajv 8.12+ 皆完整支援 2020-12

三、Python 驗證庫比較矩陣

3.1 四大主流驗證庫特性對比

| 面向 | jsonschema | fastjsonschema | Pydantic | 適用場景 |

|------|-----------|---------------|----------|---------|

| 性能 | 基準線（42ms） | 極快（2ms） | 快（12ms） | 高吞吐量選 fastjsonschema/Pydantic |

| Draft 支援 | 4/6/7/2019-09/2020-12 | 4/6/7 | N/A（生成 Draft 2020-12） | 需最新規範選 jsonschema |

| 定義方式 | JSON Schema | JSON Schema | Python 類別（type hints） | Python 原生選 Pydantic |

| 錯誤訊息 | 詳細、可自訂 | 簡單 | 豐富、結構化 | 需友善錯誤選 Pydantic/jsonschema |

| 生態系統 | OpenAPI、Swagger | 輕量獨立 | FastAPI、LLM 整合 | Web 框架選 Pydantic |

| 編譯機制 | 無 | 程式碼生成 | BaseModel 編譯 | 需預編譯選 fastjsonschema |

3.2 性能實測數據（2026 年基準）

根據 dasroot.net 的 2026 年基準測試（10,000 筆資料驗證）：


jsonschema v4.20:     42ms  (基準線)
fastjsonschema v2.20:  2ms  (21x 加速)
Pydantic v2.5:        12ms  (3.5x 加速)

效能差異原因：

  • fastjsonschema：將 schema 編譯為 Python 程式碼，執行期零解析開銷
  • Pydantic：利用 Rust 底層（pydantic-core）+ 類別編譯快取
  • jsonschema：純 Python 實作，每次驗證都需遍歷 schema 樹

3.3 使用場景決策樹


是否需要跨語言相容？
├─ 是 → jsonschema（標準 JSON Schema，可與其他語言共享）
└─ 否 → 是否已使用 FastAPI/LiteLLM？
    ├─ 是 → Pydantic（原生整合，自動生成 schema）
    └─ 否 → 是否為高頻驗證（> 1000 次/秒）？
        ├─ 是 → fastjsonschema（預編譯加速）
        └─ 否 → jsonschema（完整 Draft 支援）

3.4 其他 Python 驗證工具簡介

  • marshmallow：物件序列化/反序列化框架，驗證是附帶功能，適合 ORM 整合
  • cerberus：輕量驗證庫，自訂 schema 格式（非 JSON Schema），適合簡單驗證
  • attrs：資料類別定義工具（類似 dataclasses），搭配 cattrs 驗證，適合型別安全場景

四、錯誤處理最佳實踐

4.1 jsonschema 的錯誤資訊結構

ValidationError 提供豐富的除錯資訊：


from jsonschema import validate, ValidationError

try:
    validate(instance={"age": "not a number"}, schema={"type": "object", "properties": {"age": {"type": "integer"}}})
except ValidationError as e:
    print(e.message)       # "不是有效的整數"
    print(e.path)          # deque(['age'])：指向錯誤欄位
    print(e.schema_path)   # deque(['properties', 'age', 'type'])：指向 schema 中的規則
    print(e.instance)      # "not a number"：實際的錯誤值
    print(e.validator)     # "type"：失敗的關鍵字

4.2 處理多重錯誤：best_match() 函數

當一份資料違反多個規則時，best_match() 可找出「最相關」的錯誤：


from jsonschema import Draft7Validator
from jsonschema.exceptions import best_match

schema = {
    "type": "object",
    "properties": {
        "name": {"type": "string"},
        "age": {"type": "integer", "minimum": 0}
    },
    "required": ["name", "age"]
}

instance = {"name": 123, "age": -5}  # 兩個錯誤

validator = Draft7Validator(schema)
errors = list(validator.iter_errors(instance))
print(len(errors))  # 2

best_error = best_match(errors)
print(best_error.message)  # "123 不是 'string' 型別"（優先顯示型別錯誤）

選擇邏輯：best_match() 優先回傳「路徑較淺」的錯誤，但對 anyOf / oneOf 則回傳「最深」的錯誤（因為這些關鍵字只需一個分支通過）。

4.3 巢狀錯誤導航：ErrorTree

對於複雜的巢狀結構，ErrorTree 提供類似字典的查詢介面：


from jsonschema import Draft7Validator, ErrorTree

schema = {
    "type": "array",
    "items": {"type": "object", "properties": {"id": {"type": "integer"}}}
}

instance = [{"id": "not int"}, {"id": 2}, {"id": "also not int"}]

validator = Draft7Validator(schema)
tree = ErrorTree(validator.iter_errors(instance))

if 0 in tree:  # 檢查索引 0 是否有錯誤
    print(tree[0].errors)  # {'type': ValidationError(...)}

4.4 自訂錯誤訊息策略

方法 1：包裝 ValidationError


def friendly_validate(instance, schema):
    try:
        validate(instance, schema)
    except ValidationError as e:
        field = '.'.join(str(p) for p in e.path)
        raise ValueError(f"欄位 '{field}' 驗證失敗：{e.message}（預期型別：{e.validator_value}）")

方法 2：使用 Pydantic 的 @field_validator


from pydantic import BaseModel, field_validator, ValidationError

class User(BaseModel):
    age: int

    @field_validator('age')
    @classmethod
    def check_age(cls, v):
        if v < 0:
            raise ValueError('年齡必須為正數')
        return v

try:
    User(age=-5)
except ValidationError as e:
    print(e.errors()[0]['msg'])  # "年齡必須為正數"

五、CI/CD 整合與自動化測試

5.1 pre-commit Hook 驗證範例


# .pre-commit-config.yaml
repos:
  - repo: local
    hooks:
      - id: validate-config-schemas
        name: Validate YAML configs against JSON Schema
        entry: python scripts/validate_all_configs.py
        language: python
        pass_filenames: false


# scripts/validate_all_configs.py
import json, yaml
from pathlib import Path
from jsonschema import validate, ValidationError

config_dir = Path("config")
schema_file = config_dir / "schemas" / "config-schema.json"

with open(schema_file) as f:
    schema = json.load(f)

for yaml_file in config_dir.glob("*.yaml"):
    with open(yaml_file) as f:
        data = yaml.safe_load(f)
    try:
        validate(data, schema)
        print(f"✅ {yaml_file.name}")
    except ValidationError as e:
        print(f"❌ {yaml_file.name}: {e.message}")
        exit(1)

5.2 pytest 參數化測試


import pytest
from jsonschema import validate, ValidationError

@pytest.mark.parametrize("valid_data", [
    {"name": "Alice", "age": 30},
    {"name": "Bob", "age": 25}
])
def test_valid_schemas(valid_data, user_schema):
    validate(valid_data, user_schema)  # 不應拋出錯誤

@pytest.mark.parametrize("invalid_data,expected_error", [
    ({"name": 123, "age": 30}, "不是 'string' 型別"),
    ({"name": "Alice"}, "'age' 是必填欄位")
])
def test_invalid_schemas(invalid_data, user_schema, expected_error):
    with pytest.raises(ValidationError, match=expected_error):
        validate(invalid_data, user_schema)

6.2 可借鏡的優化方向

根據本文研究，以下改進可提升專案的配置驗證品質：

#### （1）Schema 外部化（參考 Open WebUI 的 Valves 模式）

將 YAML 配置的 schema 定義為獨立的 JSON Schema 檔案：


// config/schemas/frequency-limits-schema.json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "required": ["version", "auto_tasks"],
  "properties": {
    "version": {"type": "integer"},
    "auto_tasks": {
      "type": "object",
      "patternProperties": {
        "^[a-z_]+$": {
          "type": "object",
          "required": ["plan_key", "group", "daily_limit"],
          "properties": {
            "plan_key": {"type": "string"},
            "group": {"type": "string"},
            "daily_limit": {"type": "integer", "minimum": 0}
          }
        }
      }
    }
  }
}

#### （2）使用 jsonschema 驗證，改善錯誤訊息


from jsonschema import validate, ValidationError, best_match

def validate_config_file(yaml_path, schema_path):
    with open(yaml_path) as f:
        data = yaml.safe_load(f)
    with open(schema_path) as f:
        schema = json.load(f)

    try:
        validate(data, schema)
    except ValidationError:
        errors = list(Draft7Validator(schema).iter_errors(data))
        best_error = best_match(errors)
        field_path = '.'.join(str(p) for p in best_error.path)
        raise ValueError(
            f"❌ {yaml_path.name} 驗證失敗\n"
            f"  欄位：{field_path}\n"
            f"  錯誤：{best_error.message}\n"
            f"  預期：{best_error.validator} = {best_error.validator_value}"
        )

#### （3）結果 JSON Schema 的格式漂移偵測

workflow-forge 已產出 config/schemas/results-auto-task-schema.json，可用於驗證所有 results/todoist-auto-*.json 是否符合規範：


# tools/check_results_schema.py
from pathlib import Path
from jsonschema import validate

schema_file = Path("config/schemas/results-auto-task-schema.json")
results_dir = Path("results")

with open(schema_file) as f:
    schema = json.load(f)

for result_file in results_dir.glob("todoist-auto-*.json"):
    with open(result_file) as f:
        data = json.load(f)
    validate(data, schema)  # 格式漂移時會拋出錯誤

七、程式碼範例

7.1 三大驗證庫使用範例

#### jsonschema（標準實作）


from jsonschema import validate, ValidationError

schema = {
    "type": "object",
    "properties": {
        "name": {"type": "string", "minLength": 1},
        "email": {"type": "string", "format": "email"}
    },
    "required": ["name", "email"]
}

# 驗證成功
validate({"name": "Alice", "email": "alice@example.com"}, schema)

# 驗證失敗
try:
    validate({"name": "", "email": "invalid"}, schema)
except ValidationError as e:
    print(e.message)  # "'invalid' is not a 'email'"

#### fastjsonschema（性能優先）


import fastjsonschema

schema = {"type": "object", "properties": {"age": {"type": "integer"}}}

# 一次編譯，多次使用
validator = fastjsonschema.compile(schema)

validator({"age": 30})  # 通過
validator({"age": "30"})  # 拋出 JsonSchemaValueException

#### Pydantic（Python 原生）


from pydantic import BaseModel, EmailStr, field_validator

class User(BaseModel):
    name: str
    email: EmailStr
    age: int

    @field_validator('age')
    @classmethod
    def check_age(cls, v):
        if v < 0:
            raise ValueError('年齡必須為正數')
        return v

# 自動驗證
user = User(name="Alice", email="alice@example.com", age=30)

# 自動生成 JSON Schema
print(user.model_json_schema())
