一、專案概述

Letta（前身為 MemGPT）是由加州大學伯克利分校（UC Berkeley）Sky Computing Lab 孵化、現已獨立商業化的有狀態 Agent 平台。其核心命題是：「Agent 不只使用記憶，Agent 活在記憶裡面」。與 LangChain、LangGraph 等以工具鏈為中心的框架不同，Letta 採用操作系統（OS）啟發的記憶模型，讓 Agent 像作業系統一樣主動管理哪些資料留在「RAM」（上下文視窗）、哪些頁入頁出「磁碟」（外部向量儲存）。

這一設計使 Letta 能維持跨越 500+ 輪互動的長期任務一致性，遠超傳統 RAG 基線在約 50 輪後資訊碎裂的瓶頸 [來源：TokenMix Blog, 2026]。MemGPT 的最活躍開發現已遷至 Letta，MemGPT 套件仍作為開源參考實作保留。截至 2026 年 4 月，Letta Code 在 Terminal-Bench 上榮登「#1 模型無關開源 Coding Agent」，標誌著 Letta 從記憶框架演進為完整 Agent 平台。

二、技術架構

2.1 三層記憶體系（OS 啟發）

Letta 的核心差異化在於三層記憶體系 [來源：Vectorize.io, 2026]：

| 層次 | 類比 | 說明 |

|------|------|------|

| Core Memory | RAM | 永遠在 context window 中；儲存使用者 profile、當前任務狀態；Agent 可直接讀寫（self-edit） |

| Recall Memory | 磁碟快取 | 可搜尋的歷史對話（向量 + 全文 + 混合搜尋）；由 Agent 主動 tool call 讀取 |

| Archival Memory | 冷儲存 | 長期語意向量庫；Agent 透過 tool call 主動 query；容量近乎無限 |

這種分層讓 Agent 突破 context window 固定上限，實現「記憶按需頁入、上下文保持精簡」。Agent 不是被動接收塞進 prompt 的資料，而是主動決策哪些記憶值得保留、何時檢索 [來源：DEV Community, 2026]。

2.2 V1 Agent Loop — 從 MemGPT 到原生推理架構

2025-2026 年間 Letta 完成最重要的架構轉型 [來源：Letta 官方部落格「Rearchitecting Letta's Agent Loop」, 2026]：

MemGPT 舊架構問題：

  • 一切動作（含回訊息）都透過 tool call 執行（send_message tool）
  • 注入特殊參數 thinking（思維鏈）與 request_heartbeat（繼續執行訊號）
  • Agent 預設終止，除非 request_heartbeat=true
  • 強制 prompt engineering 替代模型原生能力

Letta V1 新架構：

  • 依賴模型原生 reasoning tokens 與 assistant message 直接生成
  • 廢棄 heartbeat 機制與 send_message tool
  • Agent 保持「in-distribution」：與現代 LLM post-training pattern 對齊
  • 支援 OpenAI Responses API encrypted reasoning（Claude Extended Thinking 等）

設計動機：「現代 LLM 已針對 agentic pattern 進行 post-training，Agent 架構應跟隨模型訓練分佈，而非對抗它。」

演進路徑：ReAct（格式化 prompt 推理）→ MemGPT（tool-based 有狀態）→ Letta V1（原生模型能力）

2.3 Context Repositories 與 Skill Learning（2026 新特性）

[來源：Letta Code 官方部落格, 2026]

  1. Context Repositories：對 Agent 記憶的 git-based 版本化管理；程式化控制 context 更新；提供完整記憶歷史回溯能力
  2. Skill Learning：Agent 完成任務後能自動從對話提取「技能」存為 .md 檔（如「生成 DB migration」、「建立 PostHog dashboard」）；可加入 git 版本管理；後續相似任務中自動復用

這兩個特性讓 Letta 朝向「會自我改善的 Agent 平台」演進。

三、功能特色與競品比較

與 Mem0、Zep 的核心差異

| 面向 | Letta | Mem0 | Zep |

|------|-------|------|-----|

| 角色 | 完整 Agent Runtime | 輕量記憶 SDK | 記憶即服務 |

| 記憶控制 | Agent 主動 self-edit | 系統被動萃取 | 研究級混合架構 |

| 整合方式 | 整個 Runtime | 插入現有框架 | 插入現有框架 |

| 鎖定程度 | 高（重寫 Agent Loop） | 低 | 中 |

| GitHub Stars | ~22.4k | ~48k | N/A |

| LongMemEval | 未公布 | 49.0% | 未公布 |

[來源：Vectorize.io 2026；TokenMix Blog 2026；DEV Community 2026]

與 LangGraph（LangMem）的比較

LangGraph 2025 年推出 LangMem SDK，加入 episodic/semantic/procedural 三類記憶。最大差異：LangMem 有procedural memory（Agent 自我更新 system instruction），Letta 目前沒有。但 LangMem 與 LangChain 緊密耦合，非 LangGraph 用戶採用代價高 [來源：Vectorize.io, 2026]。

Letta Code vs Claude Code、Cursor

Letta Code 差異化：跨 session 代碼庫記憶。/init 命令讓 Agent 分析本機代碼庫後重寫 system prompt；/search 讓 Agent 查詢過去對話的向量記憶。Terminal-Bench 上 Letta Code 達「可對比 Gemini CLI、Claude Code、Codex」的水準，最大優勢是模型無關性（model-agnostic） [來源：Letta Code 官方部落格, 2026]。

四、使用方式

快速啟動（CLI 方式）


# 安裝 Letta Code CLI
npm install -g @letta-ai/letta-code

# 初始化代碼庫記憶
letta-code /init

# 搜尋歷史對話記憶
letta-code /search "DB migration pattern"

Python SDK 方式


from letta import create_client

client = create_client()

# 建立有狀態 Agent（記憶自動持久化跨 session）
agent = client.create_agent(
    name="my-stateful-agent",
    memory_blocks=[
        {"label": "human", "value": "使用者偏好 Python，目前開發 FastAPI 後端"},
        {"label": "persona", "value": "我是一個記憶管理型助手"}
    ]
)

# 跨 session 互動（Core Memory 自動持久）
response = client.send_message(
    agent_id=agent.id,
    message="繼續上次討論的 API 設計"
)

部署選項

  • 自架（Self-hosted）：pip install letta，SQLite/Postgres 後端，免費，Apache-2.0
  • Managed Cloud：$20-$200/月，含 ADE（Agent Development Environment）視覺化除錯工具

六、潛在應用（daily-digest-prompt 視角）

場景 A：自動任務 Agent 跨輪記憶升級（P1，本月可導入）

問題：context/continuity/auto-task-*.json 格式簡單（max 5 runs），跨任務類型知識無法共享。

Letta 落地：每個 task_type 對應一個 Letta Agent（持久化 Agent ID 存 config/letta-agents.yaml）；Core Memory 儲存任務模式摘要；Archival Memory 自動索引所有研究報告 KB note，跨任務語意查詢。Agent 可查詢「3 個月前研究的 LangGraph 架構的具體細節」而非依賴 KB 隨機搜尋命中。

場景 B：Letta Code 整合至代碼審查（P2）

letta-code /init 指向 D:\Source\daily-digest-prompt，讓 Agent 建立代碼庫記憶；arch_evolution、harness_review 任務可直接查詢歷史代碼修改記憶，避免重複建立上下文。落地步驟：npm install -g @letta-ai/letta-code → /init → 修改 run-system-audit-team.ps1 在 Phase 1 前呼叫 Letta Code search。

場景 C：研究連續性無限擴展（P2）

目前 continuity 上限 5 runs。用 Letta Archival Memory 替換，支援無限期追蹤所有研究歷史與語意查詢（「之前研究過哪些 Agent 框架的記憶架構？」），比 JSON 陣列更具探索性。

七、優缺點評估

優點

  1. 最完整的記憶管理自主性：Agent 主動控制記憶頁入/頁出，比被動向量注入更透明可控 [來源：DEV Community, 2026]
  2. 長期連續性（500+ rounds）：在長對話任務一致性遠超 RAG baseline（後者 ~50 輪後資訊碎裂）[來源：TokenMix Blog, 2026]
  3. V1 架構對齊現代 LLM：原生 reasoning token 支援，相容 Claude Extended Thinking、OpenAI Responses API [來源：Letta 官方部落格, 2026]
  4. Skill Learning 自我改善：技能以 git 版本化管理，首個 OSS 框架展示「週運行後任務效能可測量提升」

缺點

  1. 高鎖定成本：採用 Letta 等於採用整個 Agent Runtime，遷移需重寫 Agent Loop（社群估計 2-6 週）[來源：Vectorize.io, 2026]
  2. 本機 LLM 支援脆弱：GitHub Issues 回報 90% 非 OpenAI/Anthropic 模型錯誤率；小型模型難以完成記憶 self-management [來源：社群評測, 2026]
  3. 記憶 self-edit 破壞 prompt caching：每次 Core Memory 更新使 Anthropic/OpenAI prompt cache 失效，帶來額外延遲與費用 [來源：社群評測, 2026]
  4. Python-first 限制：其他語言只能透過 REST API 存取，JS/TS 生態整合受限

不適用情境

  • 短對話場景（分鐘至小時級），三層記憶開銷不合算
  • 需要快速生產部署的團隊（建議先用 Mem0 或 LangMem）
  • 本機 Ollama 用戶（模型能力不足時 memory self-edit 失敗率過高）
