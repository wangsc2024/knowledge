OpenClaw — 開源個人 AI 助手閘道框架深度技術解析

一、專案概述與爆發式增長

OpenClaw 是由 PSPDFKit 創辦人 Peter Steinberger 於 2025 年 11 月以「Clawdbot」之名發佈的開源個人 AI 助手，2026 年 1 月爆發式增長，從 9,000 Stars 在數天內飆升至 60,000+，截至 2026 年 3 月已突破 302,000 Stars，成為 GitHub 史上增長最快的開源專案之一。2026 年 2 月 14 日，Steinberger 宣布加入 OpenAI，OpenClaw 轉交開源基金會維運。

核心定位：不是聊天機器人，而是一個「本地 AI 閘道」（Local AI Gateway），在使用者自己的裝置上運行，將 AI 模型連接到 50+ 整合服務，透過使用者已在使用的通訊管道（WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等 20+ 平台）提供 AI 助手服務。

關鍵數據：

  • 單週 200 萬訪客
  • 50+ 整合服務
  • 20+ 通訊平台支援
  • 5,400+ 社群技能（Skills）
  • Node.js 22+ 技術棧

---

二、Hub-and-Spoke 核心架構

OpenClaw 採用中心輻射式（Hub-and-Spoke）架構，以單一 Gateway 作為控制平面：

2.1 Gateway（閘道層）

  • 預設綁定 127.0.0.1:18789（僅本地迴路，安全優先）
  • Node.js 22+ 實作，所有客戶端透過 WebSocket 連線
  • 連接介面層（通訊平台）與智能層（Agent 執行時）的橋樑
  • 支援 CLI、Web UI、行動 App、通訊平台適配器

2.2 Agent Runtime（代理執行時）

每次對話輪次執行四個階段：

  1. Session Resolution：將訊息映射到安全邊界（主 Session vs DM/群組 Session）
  2. Context Assembly：載入對話歷史、從工作空間檔案動態組裝系統提示、語義搜尋記憶
  3. Model Streaming：呼叫配置的 LLM 提供者，攔截工具呼叫並執行
  4. State Persistence：將更新後的 Session 狀態持久化到磁碟

2.3 系統提示架構（多源組合）

提示詞按優先順序從多個來源組合：

  • AGENTS.md（核心指令）→ SOUL.md（人格特質）→ TOOLS.md（慣例）
  • 動態上下文：Session 歷史、相關技能、記憶搜尋結果
  • 工具定義：從內建和外掛工具自動生成

---

三、MCP 自擴展技能系統（Self-Extending Skills）

這是 OpenClaw 最具創新性的設計：

3.1 技能即 MCP 伺服器

  • ClawHub（技能市場）上的每個技能都是一個 MCP（Model Context Protocol）伺服器
  • 啟用技能時，OpenClaw 連接該 MCP 伺服器，使其工具可供 AI Agent 呼叫
  • 原生支援 @modelcontextprotocol/sdk@1.25.3，無需自訂包裝器

3.2 智慧技能注入（Skill Injection）

  • 非全量注入：不會將所有技能盲目注入每個提示
  • 選擇性注入：執行時僅注入與當前對話輪次相關的技能
  • 目的：避免提示膨脹導致模型效能下降
  • 這與 Daily-Digest-Prompt 的 Skill-First 策略高度呼應

3.3 自我擴展能力

  • OpenClaw 能夠自行撰寫新技能，有效地擴展自身能力而無需人工介入
  • 5,400+ 社群技能已在 awesome-openclaw-skills 倉庫中分類整理

---

四、記憶系統（Hybrid Search）

4.1 儲存架構

  • SQLite + 向量嵌入：~/.openclaw/memory/.sqlite
  • 混合搜尋：向量相似度（語義匹配）+ BM25 關鍵字相關性
  • 與 Daily-Digest-Prompt 的知識庫（Trilium + hybrid search）設計理念一致

4.2 記憶檔案

  • MEMORY.md：穩定事實（僅主 Session）
  • YYYY-MM-DD.md：每日筆記
  • 自動重新索引：檔案變更時自動觸發
  • 嵌入提供者優先順序：本地模型 > OpenAI > Gemini

4.3 Session 管理

  • Append-only 事件日誌：支援分支以恢復歷史
  • 自動壓縮：摘要較舊的對話輪次以尊重模型上下文限制
  • 記憶沖刷（Memory Flush）：在壓縮前將重要細節提升到記憶檔案

五、分層安全模型

Cisco AI 威脅研究團隊稱 OpenClaw「能力上突破性」但「安全上是噩夢」，其安全設計值得深入分析：

5.1 六層防禦

  1. 網路層：預設綁定本地迴路；遠端存取透過 SSH 隧道或 Tailscale
  2. 認證層：Token 或密碼認證（非本地迴路連線）
  3. 裝置配對層：新裝置需密碼學挑戰-回應；本地迴路自動批准
  4. 通道存取控制：DM 配對碼審批、白名單、群組提及要求
  5. 工具沙箱層：Docker 容器隔離（主 Session 原生執行，DM/群組 Session 預設容器化）
  6. 提示注入防禦：上下文隔離分離使用者輸入與系統指令

5.2 工具策略優先順序（逐層收窄）

Tool Profile → Provider Profile → Global Policy → Provider Policy → Agent Policy → Group Policy → Sandbox Policy

5.3 安全隱患

  • 技能倉庫缺乏嚴格的惡意提交審核
  • 廣泛的系統權限（郵件、日曆、通訊平台）帶來攻擊面
  • 錯誤配置或暴露的實例構成隱私風險

---

六、多 Agent 路由與 Canvas/A2UI

6.1 多 Agent 路由

  • 不同通道/群組可路由到獨立 Agent 實例
  • 每個實例有獨立工作空間、模型和行為
  • 例如：Discord 用 Claude Sonnet，Telegram DM 用 GPT-4，各有不同沙箱策略

6.2 Canvas（Agent-to-UI）

  • 獨立伺服器程序（port 18793）
  • Agent 生成帶 A2UI 屬性的 HTML，創建互動元素
  • 使用者互動觸發事件 → Canvas 轉派為工具呼叫 → Agent 處理
  • 崩潰隔離：Canvas 崩潰不影響 Gateway

6.3 Session 工具（跨 Agent 通訊）

  • sessions_list：發現活躍 Agent
  • sessions_send：向另一個 Session 發送訊息
  • sessions_spawn：程式化創建新 Session

---

七、通道適配器設計

每個通訊平台實作標準介面，負責四項職責：

  1. 認證：QR 碼、Token、原生整合
  2. 入站解析：文字、媒體、執行緒、元資料
  3. 存取控制：白名單、配對、群組策略
  4. 出站格式化：Markdown 方言、大小限制、媒體上傳

技術選型：WhatsApp（Baileys）、Telegram（grammY）、Discord（discord.js）、Signal、iMessage 等。

八、可借鏡的技術項目（對 Daily-Digest-Prompt 的啟示）

8.1 智慧技能注入（高優先）

  • 現狀：DDP 的 Skill-First 策略要求主動讀取 SKILL_INDEX.md
  • 借鏡：OpenClaw 的「選擇性注入」機制更優雅 — 根據當前上下文自動判定需要哪些技能
  • 行動建議：在 tools/llm_classifier.py 加入技能相關性評分，僅注入 top-N 相關技能

8.2 記憶沖刷機制（中優先）

  • 現狀：DDP 的 digest-memory.json 和 continuity/ 是手動設計的記憶格式
  • 借鏡：OpenClaw 的 Memory Flush 在壓縮 Session 前自動提取重要資訊到持久記憶
  • 行動建議：在 on_stop_alert.py 加入自動摘要提取，將關鍵發現寫入 continuity

8.3 分層安全模型（中優先）

  • 現狀：DDP 有 Hook 機器強制層（pre_bash_guard、pre_write_guard）
  • 借鏡：OpenClaw 的六層防禦 + 工具策略優先順序鏈 是更系統化的安全架構
  • 行動建議：在 config/hook-rules.yaml 加入策略優先順序概念

8.4 多 Agent 路由（低優先但有啟發性）

  • 現狀：DDP 的 run-todoist-agent-team.ps1 已有 3 階段並行
  • 借鏡：OpenClaw 的「不同通道路由到不同 Agent 實例」概念可應用於「不同任務類型路由到不同 LLM」
  • 關聯：這與 DDP 已有的 tools/llm_router.py 設計方向一致

8.5 Append-Only Session 日誌（低優先）

  • 現狀：DDP 用 JSONL 日誌（post_tool_logger.py）
  • 借鏡：OpenClaw 的 append-only + 分支 + 自動壓縮是更成熟的日誌架構
  • 行動建議：未來 JSONL 日誌可考慮加入壓縮摘要機制

九、技術對比總結

| 維度 | OpenClaw | Daily-Digest-Prompt |

|------|----------|--------------------|

| 架構模式 | Hub-and-Spoke 閘道 | PowerShell Phase 管線 |

| 技能管理 | MCP 自動發現 + 選擇性注入 | SKILL_INDEX.md + 手動讀取 |

| 記憶系統 | SQLite + 向量嵌入 + Memory Flush | JSON 檔案 + 知識庫 API |

| 安全模型 | 六層防禦 + 策略鏈 | Hook 機器強制層 |

| 多 Agent | 通道路由 + Session 工具 | Team 模式 + 子 Agent |

| 部署 | 本地/VPS/Docker/Fly.io | Windows Task Scheduler |

十、結論

OpenClaw 的爆發式增長並非偶然 — 它精準解決了「AI 助手碎片化」問題：一個統一閘道連接所有通訊管道，本地運行保護隱私，MCP 協議實現無限擴展。其 Hub-and-Spoke 架構、智慧技能注入和分層安全模型都是值得借鏡的工程設計。對於 Daily-Digest-Prompt 而言，最具價值的借鏡是「選擇性技能注入」和「記憶沖刷」兩個機制，可直接提升現有系統的效率和記憶品質。
