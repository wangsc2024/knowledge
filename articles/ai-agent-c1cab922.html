AI Agent 部署安全的三角困境：透明度、自主性與可驗證性

執行摘要

本文主張：當前 AI agent 部署呈現三角困境——透明度缺口、多 agent 系統的攻擊面擴張、與可驗證性需求之間的結構性張力。基於 The 2025 AI Agent Index 對 30 個已部署系統的分析 [The 2025 AI Agent Index, 2026]，多數開發者在安全評估、社會影響與決策邏輯上提供甚少資訊，形成問責真空。同時，多 agent 架構將單點攻擊放大為橫向傳播風險——Cornell Tech 與以色列理工學院研究顯示，首個 GenAI 蠕蟲（Morris II）可觸發級聯式間接 prompt injection，感染率達 48% [Multi-Agent Infection Chains, Medium 2026]。產業信心與實際防護的落差尤其顯著：82% 高管認為現有政策足夠，但 80.9% 技術團隊已進入全面部署階段 [Lasso Security, 2026]。

本研究論證此三維困境需要分層應對策略：透明度需區分商業機密與安全必要資訊（如攻擊面揭露、權限範疇）、攻擊面緩解需透過架構隔離（Model Context Protocol 的 Micro-MCP 模式、Auton 的 constraint manifold formalism）與即時防禦管線、可驗證性則需依場景動態調整（金融交易採 Blueprint First 確定性執行，通用助理保留適應彈性）。本文闡明：透明度與可驗證性並非零和遊戲，而是需要透過技術手段（MCP 標準化、policy projection）與治理機制（審計追蹤、RBAC）共同達成的平衡。

---

研究問題與範疇

本研究聚焦三個核心問題：

  1. 透明度缺口的具體表現：當前 AI agent 系統在哪些維度缺乏透明度？此缺口對使用者信任、監管問責、研究驗證各有何影響？
  2. 多 agent 系統的攻擊傳播機制：相較於單一 LLM，multi-agent 架構如何放大攻擊面？橫向感染的技術成因與防禦邊界在哪？
  3. 可驗證性與彈性的權衡：確定性執行框架（如 Blueprint First）如何在結構化場景提升可靠性？此權衡在金融、醫療、通用助理等應用中如何排序？

研究範疇界定：本文不涉及「AI agent 是否應完全自主」的哲學辯論，而聚焦於已部署系統的實證風險與可行緩解措施。不窮舉所有框架功能比較，僅分析安全相關設計（LangGraph、CrewAI、Auton、Blueprint First）。技術細節止於架構原則，不深入具體 exploit payload。

---

文獻探討

透明度現狀與產業落差

The 2025 AI Agent Index 針對 30 個 state-of-the-art agentic systems 進行系統性分析，發現「most developers share little information about safety, evaluations, and societal impacts」[Staufer et al., 2026]。該研究透過公開資訊與開發者通信收集數據，揭示透明度水準高度不一致——部分系統公開評估基準與安全措施，其他則將決策邏輯、訓練數據、存取控制列為商業機密。文獻指出透明度對治理與知情部署的價值遠超過文件化已公開資訊的風險 [The 2025 AI Agent Index, arXiv 2602.17753]。

產業調查進一步揭示信心與實際防護的危險落差。Lasso Security 2026 年企業調查顯示，82% 高管表示現有政策足以防範未授權 agent 行為，但技術團隊中 80.9% 已越過規劃階段進入測試或全面部署 [Lasso Security, 2026]。此落差反映兩個層面問題：高層對 agent 執行層漏洞（execution layer vulnerabilities）的認知不足，以及「shadow AI」現象——許多 agent 未經安全審查即部署，連接到安全團隊從未盤點的工具 [Recorded Future, 2026]。

多 agent 系統的攻擊面擴張

相較於單一 LLM 的垂直攻擊（使用者 → LLM → 工具），multi-agent 系統引入橫向傳播路徑。Cornell Tech、以色列理工學院與 Intuit 的聯合研究發表首個針對 GenAI 生態的 zero-click 蠕蟲 Morris II，演示 adversarial self-replicating prompt 可觸發級聯式間接 prompt injection，攻擊在整個 agent 網路中擴散 [Multi-Agent Infection Chains, Medium 2026]。

arXiv 2410.07283 正式定義「Prompt Infection」為「malicious prompts self-replicate across interconnected agents, behaving much like a computer virus」[Prompt Infection, 2025]。攻擊機制包含三階段：(1) 初始感染——惡意 prompt 嵌入 agent 的記憶或工具輸出；(2) 橫向移動——感染 agent 與其他 agent 交換數據時，將惡意指令注入對方的 context；(3) 協調執行——受感染 agent 群體協調資料竊取或指令傳遞，「even when agents do not publicly share all communications」[Prompt Infection Abstract, 2025]。

Christian Schneider 的安全分析指出，「lateral movement spreads the attack across users, devices, connected services, or other agents in multi-agent architectures」，且「by the time you detect the injection, the agent may have already executed multiple tool calls, persisted malicious data, and propagated to other systems」[Christian Schneider Blog, 2026]。此時間差使得傳統「偵測後阻斷」策略失效。

確定性執行與可驗證性框架

針對 LLM 的非確定性（non-determinism）在結構化操作情境下造成的風險，Blueprint First, Model Second 框架提出認知與執行分離架構。該研究主張「expert-defined operational procedure is first codified into a source code-based Execution Blueprint, which is then executed by a deterministic engine」，LLM 僅用於有界的複雜子任務，不決定 workflow 路徑 [Qiu et al., 2025]。此方法在 tau-bench（測試複雜 user-tool-rule 場景）取得 +10.1 percentage points 的提升，同時顯著改善執行效率 [Blueprint First, arXiv 2508.02721]。

Auton Agentic AI Framework 引入 constraint manifold formalism，透過 policy projection 在決策層強制安全邊界，而非事後過濾（post-hoc filtering）[Auton Framework, arXiv 2602.23720]。此方法避免「生成後再檢查」的時間窗口漏洞，將安全約束編碼為策略空間的幾何限制。

然而確定性執行必然犧牲彈性——當遇到未預見的邊緣案例時，嚴格遵循預定義藍圖的 agent 無法即時適應。文獻指出此權衡在不同應用場景有不同最優解：金融交易、醫療診斷等高風險領域優先可驗證性；客服、內容創作等場景則需保留創造性與適應能力。

標準化隔離機制：Model Context Protocol (MCP)

Model Context Protocol 於 2026 年成為 unifying standard，允許工具無縫跨框架運作，減少 vendor lock-in [OpenAgents Blog, 2026]。MCP Security Analysis（GitHub cosai-oasis/ws4-secure-design-agentic-systems）定義十二個核心威脅類別與近四十項具體威脅，要求部署方採用 zero-trust architectures、hardware-based isolation（如 trusted execution environments）、rigorous supply chain vetting、與持續監控 [MCP Security, GitHub 2026]。

Micro-MCP 架構模式進一步強化隔離：將多個 single-purpose MCP servers 配置於輕量級 gateway 後方，「bringing security isolation, independent deployability, and unified access to tools/resources/prompts」[Micro-MCP, GitHub 2026]。此設計確保單一服務的妥協或過載不會級聯影響其他服務，實現 blast radius containment。

---

核心論證

論證一：透明度缺口非技術問題，而是治理與激勵結構失衡

當前透明度不足並非開發者技術能力限制，而是缺乏外部壓力與統一標準。The 2025 AI Agent Index 揭示，30 個系統中透明度差異巨大——部分公開完整評估與安全措施，其他則幾乎無揭露 [Staufer et al., 2026]。此分化反映：(1) 監管真空——歐盟 AI Act 雖於 2026 年 8 月 2 日生效，但執行細則仍在制定中；(2) 競爭壓力——率先揭露攻擊面可能被競爭對手利用；(3) 問責恐懼——詳細記錄決策邏輯可能在事故後成為法律證據。

本文論證：透明度需求應區分「商業機密」（如模型訓練細節、私有數據集）與「安全必要資訊」（如權限範疇、攻擊面文件、評估基準）。後者應成為強制揭露項，類比食品標籤（營養成分強制標示，但配方可保密）。標準化揭露框架（如 The AI Agent Index 的評估維度）可降低個別開發者的相對風險。

論證二：多 agent 架構將攻擊從「點」放大為「網」

單一 LLM 的攻擊面主要在 user input → model → tool execution 的垂直路徑。多 agent 系統引入三個新維度：(1) 共享記憶（shared memory）——感染 agent 可將惡意 prompt 寫入共用資料庫，下個讀取該記憶的 agent 被感染；(2) 訊息傳遞（message passing）——agent 間通信時，惡意指令嵌入訊息內容或 metadata；(3) 工具存取（tool access）——受感染 agent 可透過共享工具（如檔案系統、API）傳播攻擊 [Prompt Infection, arXiv 2410.07283]。

Morris II 蠕蟲實證此風險：adversarial self-replicating prompt 在 agent 網路中級聯傳播，無需單點持續注入 [Multi-Agent Infection Chains, Medium 2026]。Parallax 架構雖透過認知-執行分離、adversarial validation、資訊流控制、與可逆執行阻擋 98.9% 攻擊（280 測試案例），但此成功率基於已知攻擊模式——zero-day 攻擊與新型社會工程手法仍可突破 [Parallax Abstract, 2026]。

本文論證：多 agent 系統的攻擊面擴張不可避免，但可透過架構隔離緩解。Micro-MCP 的 single-purpose servers + gateway 模式實現 blast radius containment；LLM Tagging 機制（標記訊息來源與信任等級）「when combined with existing safeguards, significantly mitigates infection spread」[Prompt Infection, 2025]；Multi-Agent Defense Pipeline 透過 specialized LLM agents in coordinated pipelines 即時偵測與中和 prompt injection [Multi-Agent Defense, arXiv 2509.14285]。防禦策略需從「反應式阻斷」轉向「預防式隔離」。

論證三：可驗證性與彈性的權衡需依應用場景動態調整

Blueprint First 框架在 tau-bench 的 +10.1% 提升證實確定性執行的價值，但此設計明確犧牲彈性——「prioritizes procedural fidelity and predictable execution in structured operational environments」，必然「restricts flexibility in favor of auditability and compliance」[Qiu et al., 2025]。當遇到未預見情境時，遵循固定藍圖的 agent 無法即時調整策略。

本文論證：可驗證性需求並非絕對，而是風險容忍度的函數。三類應用場景：

  1. 高風險/低彈性需求（金融交易、醫療診斷、法規遵循）：Blueprint First + Auton constraint manifold formalism 為最優解。每次執行需可審計、可解釋、可回溯，犧牲適應性換取零容忍錯誤。
  2. 中風險/平衡需求（企業客服、SOP 自動化）：Agent-S 的三 LLM + Global Action Repository 模式，允許 agent 在 SOP 定義範圍內動態選擇行動，同時保留 fault tolerance 機制（決定何時重試或諮詢外部知識）[Agent-S, arXiv 2503.15520]。
  3. 低風險/高彈性需求（內容創作、研究助理、brainstorming）：CrewAI 的 role-based collaboration + hierarchical memory isolation，優先創造性與快速迭代，透過 memory isolation 而非執行確定性保障安全 [CrewAI v1.12, 2025]。

LangGraph 與 CrewAI 的設計哲學分野體現此權衡：LangGraph 的 enterprise plan 強調「self-hosted deployment, SSO, and RBAC」，適合「strict audit requirements, human-in-the-loop checkpoints」[OpenAgents Blog, 2026]；CrewAI 聚焦「rapid prototyping and role-based collaboration」，將安全邊界下推至 memory 層而非 execution 層 [PE Collective Framework Comparison, 2026]。

一家之言

基於前文證據，本文提出三項獨立判斷：

判斷一：透明度與安全的關係並非單調正相關

傳統觀點認為「更多透明度 = 更高安全性」，但過度透明可能暴露攻擊面。例如，詳細揭露 agent 的工具權限與 API 端點，可能讓攻擊者精準設計 prompt injection payload。本文主張：最佳透明度是「對問責方（監管者、使用者、研究者）充分、對攻擊者模糊」的資訊不對稱。實務上可透過分級揭露機制——公開揭露安全評估結果與權限範疇（「此 agent 可讀寫檔案系統」），但技術細節（具體 API 端點、認證機制）僅向經認證稽核方揭露。

判斷二：多 agent 系統的橫向感染風險被嚴重低估

當前產業對 agent 安全的關注仍集中在單點防禦（input sanitization, output filtering）。但 Morris II 蠕蟲與 Prompt Infection 研究揭示，multi-agent 架構的真正風險在於時間與空間的擴散——單一感染點可在偵測前傳播至 48% 的 co-running agents，且透過共享記憶或工具持久化，使攻擊在 agent 重啟後仍存在。本文主張：防禦策略需從「邊界防護」升級為「縱深防禦 + 爆炸半徑限制」——每個 agent 視為潛在感染源，透過 Micro-MCP 的 single-purpose servers 隔離、LLM Tagging 的信任傳播追蹤、與即時異常偵測（Multi-Agent Defense Pipeline）形成多層防線。

判斷三：可驗證性框架的採用門檻在於開發者體驗而非技術可行性

Blueprint First 與 Auton 證明確定性執行技術上可行且效能優越（+10.1% on tau-bench, 98.9% attack blocking）。但產業採用緩慢的真正原因是：(1) 需要領域專家預先定義完整 operational blueprint，增加初期開發成本；(2) 開發者習慣「prompt → 測試 → 調整」的快速迭代，確定性框架要求「設計 → 編碼 → 測試」的瀑布式流程；(3) 缺乏標準化工具鏈——從需求到藍圖再到執行引擎的轉換仍需大量手工。本文主張：可驗證性框架的推廣需要「低程式碼藍圖設計工具」+ 「自動驗證管線」+ 「漸進式遷移路徑」（允許部分 workflow 先確定化，其餘保留彈性）。FROAV 的 visual workflow orchestration + evaluation framework 是此方向的早期嘗試 [FROAV, arXiv 2601.07504]。

實踐建議

基於核心論證與獨立判斷，本文提出分層實踐建議：

對 AI agent 開發者

  1. 實施分級透明度揭露：參考 The AI Agent Index 的評估維度（safety evaluations, access controls, decision logic），將「問責必要資訊」列為強制文件化項目。使用標準化模板（如 Model Cards, System Cards）降低揭露成本。
  2. 預設架構隔離：即使初期為單一 agent，仍應採用 MCP 標準化工具接口，為未來多 agent 擴展預留隔離邊界。Micro-MCP 模式可作為架構藍圖。
  3. 依應用場景選擇框架：高風險場景（金融、醫療）優先 Blueprint First 或 LangGraph（強審計追蹤）；快速迭代場景（研究、創作）選用 CrewAI 或 LightAgent（最小依賴）；SOP 自動化場景考慮 Agent-S（平衡確定性與容錯）。

對企業部署方

  1. 消弭高管與技術層的認知落差：建立 agent 風險儀表板，將「shadow AI 數量」「未審查工具連接數」「橫向傳播路徑」等指標可視化，使高管理解執行層漏洞的實際規模。
  2. 採用縱深防禦策略：(a) 邊界層：input sanitization + output filtering；(b) 執行層：Micro-MCP 隔離 + LLM Tagging；(c) 監控層：Multi-Agent Defense Pipeline 即時偵測；(d) 審計層：完整執行追蹤與可回溯性。
  3. 漸進式可驗證性遷移：先識別高風險 workflow（涉及金融交易、個資存取、系統修改），將其轉換為 Blueprint First 模式；其餘 workflow 保留彈性，避免全面確定化造成的開發僵化。

對監管者與標準化組織

  1. 強制安全必要資訊揭露：將「權限範疇」「攻擊面文件」「評估基準」列為 agent 系統的強制標籤項（類比食品營養標籤），但允許技術實作細節保密。
  2. 建立 agent 安全認證體系：參考 ISO 27001（資訊安全）與 SOC 2（服務組織控制），制定 agent-specific 認證標準，涵蓋「透明度等級」「隔離機制」「可驗證性」三維度。
  3. 推動 MCP 標準化與最佳實踐：將 MCP Security Analysis 的十二威脅類別與防禦措施編碼為可執行檢查清單，提供參考實作（如 Micro-MCP）與測試套件。

風險與限制

本研究存在以下限制：

  1. 數據時效性：The 2025 AI Agent Index 分析的 30 個系統為 2026 年初快照，agent 生態快速演進可能使部分發現過時。
  2. 攻擊面樣本偏誤：Parallax、Prompt Infection 等研究基於已知攻擊模式，zero-day 攻擊與新型社會工程手法的風險可能被低估。
  3. 可驗證性權衡的定量化不足：Blueprint First 在 tau-bench 的 +10.1% 提升與「犧牲彈性」的程度缺乏量化對比，不同應用場景的最優權衡點仍需更多實證。
  4. 產業預測的不確定性：「2026 年底 40% 企業應用將嵌入 AI agents」的預測來自單一來源 [The 2025 AI Agent Index]，實際採用率受監管、技術成熟度、經濟環境多重因素影響。
  5. 非技術手段的探討不足：本文聚焦技術解方（架構隔離、確定性執行），但治理機制（法規遵循、使用者教育、組織文化）對安全的影響同等重要，未在本研究範疇內深入分析。
