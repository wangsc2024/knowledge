ntfy 推播通知系統完整指南（2026）

技術概述

ntfy（發音為 "notify"）是一個基於 HTTP 的 pub-sub 推播通知服務，支援透過簡單的 PUT/POST 請求從任何裝置發送通知到手機或桌面。其核心設計理念是「零依賴、零配置」——不需要註冊帳號、不需要安裝 SDK，只要能發 HTTP 請求就能推播。ntfy 在 GitHub 擁有超過 20K stars，是開源自架通知系統的事實標準。

本專案（daily-digest-prompt）的 ntfy 整合點包括：每日摘要推播（assemble-digest.md）、Todoist 任務完成通知（todoist-assemble.md）、Session 結束告警（on_stop_alert.py）、以及健康監控（check-health.ps1），topic 為 wangsc2025。

---

核心概念與原理

架構模型

ntfy 採用 pub-sub 模型，以 topic（主題）作為消息路由的最小單位。發布者透過 HTTP POST/PUT 將消息發送到特定 topic URL，所有訂閱該 topic 的客戶端即時收到通知。


發布者 (curl/script) → ntfy server (topic) → 訂閱者 (手機/桌面/webhook)

傳輸協定

  • HTTP POST/PUT：最常用，body 為消息內容，headers 攜帶元資料
  • WebSocket：持久連接，適合即時訂閱
  • Server-Sent Events (SSE)：瀏覽器原生支持的事件流
  • JSON 輪詢：定期拉取新消息

HTTP API 完整參數（進階功能）

基礎發布


# 最簡形式
curl -d "Message text" ntfy.sh/mytopic

# JSON 格式（推薦，支援完整功能）
curl -H "Content-Type: application/json" -d @payload.json ntfy.sh/

全部 Headers 參數一覽

| Header | 別名 | 用途 | 值範例 |

|--------|------|------|--------|

| X-Title | Title, ti, t | 通知標題 | 伺服器告警 |

| X-Priority | Priority, prio, p | 優先級 1-5 | 4（高） |

| X-Tags | Tags, tag, ta | 標籤+表情符號 | warning,server1 |

| X-Click | Click | 點擊開啟 URL | https://example.com |

| X-Icon | Icon | 通知圖示 URL | https://example.com/icon.png |

| X-Attach | Attach, a | 外部附件 URL | https://example.com/file.pdf |

| X-Filename | Filename, File, f | 附件檔名 | report.pdf |

| X-Email | Email | 同時寄 email | user@example.com |

| X-Delay | Delay, At, In | 排程發送 | tomorrow, 10am |

| X-Markdown | Markdown, md | Markdown 渲染 | yes |

| X-Actions | Actions | 互動按鈕 | 見下方 |

優先級系統（Priority）

| 等級 | 名稱 | 行為 |

|------|------|------|

| 1 | min | 無聲音、無振動、不顯示 |

| 2 | low | 無聲音、無振動 |

| 3 | default | 聲音+振動 |

| 4 | high | 聲音+振動+彈出 |

| 5 | max/urgent | 持續聲音+振動+強制彈出 |

Tags 與 Emoji 映射

X-Tags 中使用 emoji shortcode 會自動轉換：

  • warning → ⚠️、rotating_light → 🚨、tada → 🎉
  • white_check_mark → ✅、x → ❌、hourglass → ⏳
  • 非 emoji 的 tag 會顯示在通知下方作為分類標籤

Markdown 支援

設定 X-Markdown: yes 或 Content-Type: text/markdown 後支援：

  • 粗體 (text)、斜體 (text)
  • 連結、!圖片
  • 程式碼區塊、標題、列表、引用區塊

Action Buttons 互動通知

每則通知最多可加入 3 個 Action Button，支援四種動作類型：

1. View（開啟 URL）


curl -H "Actions: view, Open Dashboard, https://dashboard.example.com, clear=true" \
  -d "System alert" ntfy.sh/mytopic

2. HTTP（發送 HTTP 請求）


curl -H "Actions: http, Restart Service, https://api.example.com/restart, method=POST, body={\"service\":\"nginx\"}" \
  -d "Service down" ntfy.sh/mytopic

支援自訂 method、headers、body，可直接從通知觸發 API 呼叫。

3. Broadcast（Android Intent）


curl -H "Actions: broadcast, Take Photo, extras.cmd=photo, extras.camera=front" \
  -d "Motion detected" ntfy.sh/mytopic

僅限 Android，可觸發 Tasker、Macrodroid 等自動化 app。

4. Copy（複製到剪貼簿）


curl -H "Actions: copy, Copy Code, ABC-123-XYZ" \
  -d "Your verification code" ntfy.sh/mytopic

JSON 格式 Actions


{
  "topic": "mytopic",
  "message": "Build failed",
  "actions": [
    {"action": "view", "label": "View Logs", "url": "https://ci.example.com/logs"},
    {"action": "http", "label": "Retry", "url": "https://ci.example.com/retry", "method": "POST"},
    {"action": "copy", "label": "Copy ID", "value": "build-12345"}
  ]
}

模板引擎（Templates）

內建模板

ntfy 內建三個 webhook 模板，可自動解析 JSON payload：

  • github — GitHub webhooks（PR、Issue、Comment）
  • grafana — Grafana 告警 webhooks
  • alertmanager — Prometheus Alertmanager webhooks

使用方式：?template=github

自訂模板

放置 .yml 檔案在模板目錄（預設 /etc/ntfy/templates），使用 Go template 語法：


title: "{{.status}} alert"
message: "Type: {{.type}}, Value: {{.percent}}%"
priority: "{{ if gt .percent 90.0 }}5{{ else }}3{{ end }}"

這允許根據 payload 內容動態設定優先級——例如 CPU 使用率超過 90% 時自動升級為 urgent。

行內模板

無需預定義模板檔案，直接在 URL 中使用 Go template：


curl -d '{"level":"severe","desc":"Disk full"}' \
  'ntfy.sh/mytopic?tpl=yes&t={{.level}}+Error&m={{.desc}}'

排程發送（Scheduled Delivery）

支援延遲發送通知，適合提醒、倒數計時場景：


# 30 分鐘後
curl -H "In: 30m" -d "Reminder" ntfy.sh/mytopic

# 明天早上 10 點
curl -H "At: tomorrow, 10am" -d "Meeting reminder" ntfy.sh/mytopic

# Unix 時間戳
curl -H "At: 1639194738" -d "Scheduled" ntfy.sh/mytopic
  • 最短延遲：10 秒
  • 最長延遲：3 天
  • 可透過 DELETE 請求取消排程

附件系統（Attachments）

本地檔案上傳


curl -T screenshot.png -H "Filename: screenshot.png" ntfy.sh/mytopic
  • 最大單檔：15 MB（ntfy.sh 公共服務）
  • 每位訪客配額：100 MB
  • 附件保留：3 小時

外部 URL 附件


curl -H "Attach: https://example.com/report.pdf" \
  -H "Filename: monthly-report.pdf" \
  -d "Report ready" ntfy.sh/mytopic

外部附件無大小限制，不佔用伺服器儲存。

自架部署最佳實踐

Docker Compose 部署


version: '3'
services:
  ntfy:
    image: binwiederhier/ntfy
    container_name: ntfy
    command: serve
    environment:
      - TZ=Asia/Taipei
      - NTFY_BASE_URL=https://ntfy.yourdomain.com
      - NTFY_BEHIND_PROXY=true
      - NTFY_CACHE_FILE=/var/cache/ntfy/cache.db
      - NTFY_AUTH_FILE=/var/lib/ntfy/user.db
      - NTFY_AUTH_DEFAULT_ACCESS=deny-all
      - NTFY_ATTACHMENT_CACHE_DIR=/var/cache/ntfy/attachments
      - NTFY_UPSTREAM_BASE_URL=https://ntfy.sh
    volumes:
      - ./ntfy-cache:/var/cache/ntfy
      - ./ntfy-data:/var/lib/ntfy
      - ./server.yml:/etc/ntfy/server.yml:ro
    ports:
      - "8080:80"
    healthcheck:
      test: ["CMD", "wget", "-q", "--tries=1", "http://localhost:80/v1/health", "-O", "/dev/null"]
      interval: 60s
      timeout: 10s
      retries: 3
    restart: unless-stopped

關鍵配置說明

| 配置 | 說明 |

|------|------|

| NTFY_BASE_URL | 伺服器公開 URL（必填，iOS 推送需要） |

| NTFY_BEHIND_PROXY | 反向代理後方時設為 true |

| NTFY_UPSTREAM_BASE_URL | iOS 推送必須設為 https://ntfy.sh |

| NTFY_AUTH_DEFAULT_ACCESS | 建議 deny-all，僅授權用戶可存取 |

| NTFY_CACHE_FILE | 消息快取資料庫路徑 |

| NTFY_ATTACHMENT_CACHE_DIR | 附件暫存目錄 |

認證與存取控制


# 建立管理員
ntfy user add --role=admin admin_user

# 建立唯讀用戶（僅訂閱）
ntfy user add reader_user
ntfy access reader_user 'alerts-*' read-only

# 建立唯寫用戶（僅發布，適合 CI/CD）
ntfy user add ci_bot
ntfy access ci_bot 'build-*' write-only

速率限制

自架服務可自訂速率限制（server.yml）：

  • visitor-request-limit-burst: 單次爆發上限
  • visitor-request-limit-replenish: 恢復速率
  • visitor-attachment-total-size-limit: 附件總量限制

自動化整合模式

1. CI/CD 管線通知


# GitHub Actions
- name: Notify build status
  if: always()
  run: |
    STATUS=${{ job.status }}
    EMOJI=$([ "$STATUS" = "success" ] && echo "white_check_mark" || echo "x")
    curl -H "Title: Build $STATUS" -H "Tags: $EMOJI,github" \
      -H "Priority: $([ '$STATUS' = 'failure' ] && echo '4' || echo '3')" \
      -d "Commit: ${{ github.sha }}" ntfy.sh/builds

2. 監控告警（Uptime Kuma）

Uptime Kuma 原生支援 ntfy，設定步驟：

  1. 在 Uptime Kuma 新增 ntfy 通知供應商
  2. 填入 ntfy 伺服器 URL 和 topic
  3. 可設定不同優先級對應不同服務等級

3. 智慧家庭（Home Assistant）


# configuration.yaml
notify:
  - name: ntfy_alerts
    platform: rest
    resource: https://ntfy.sh/home-alerts
    method: POST_JSON
    headers:
      Authorization: "Bearer tk_your_token"
    data:
      priority: 4
      tags: ["house"]

4. Cron Job 完成通知


# 備份完成通知
0 2 * * * /usr/local/bin/backup.sh && curl -d "Backup completed" ntfy.sh/backups || curl -H "Priority: 5" -H "Tags: x" -d "Backup FAILED" ntfy.sh/backups

5. 長時間命令完成通知


# Bash alias
alias alert='ntfy_send() { curl -d "Command finished: $1" ntfy.sh/terminal; }; ntfy_send'

# 使用
make build && alert "Build success" || alert "Build failed"

與本專案的關聯（daily-digest-prompt 應用）

現行整合點

本專案目前使用 ntfy 的場景：

  1. 每日摘要推播（assemble-digest.md）：每日 3 次（08:00, 11:15, 21:15）推送完整摘要
  2. Todoist 任務完成通知（todoist-assemble.md）：Phase 3 組裝後推播結果
  3. Session 結束告警（on_stop_alert.py）：異常時自動 ntfy warning/critical
  4. 通知配置集中化（config/notification.yaml）：topic、標籤、模板統一管理

可應用的進階功能

  1. Action Buttons 應用：

- 摘要通知加入「查看詳情」按鈕（view action 開啟知識庫 URL）

- 錯誤告警加入「重新執行」按鈕（http action 觸發排程腳本）

- Todoist 完成通知加入「開啟 Todoist」按鈕

  1. 動態優先級：

- 正常摘要 priority=3，包含錯誤時自動升級 priority=4

- blocked 次數 ≥3 時 priority=5（目前 on_stop_alert.py 已實作類似邏輯）

  1. Markdown 格式化：

- 摘要內容改用 Markdown 渲染，提升可讀性（粗體標題、列表、連結）

  1. 模板引擎：

- 自訂 daily-digest 模板，標準化每日摘要格式

- 自訂 todoist-result 模板，標準化任務完成通知

  1. 圖示系統：

- 不同通知類型使用不同 icon（摘要用日曆圖示、錯誤用警告圖示）

Windows 環境注意事項

本專案在 Windows 環境下使用 ntfy 的經驗總結：


# 正確做法：用 Write 工具建立 JSON 檔，再用 curl 發送
curl -s -X POST "https://ntfy.sh/topic" \
  -H "Content-Type: application/json; charset=utf-8" \
  -d @notification.json

# 錯誤做法：inline JSON 會因 Windows shell 轉義問題導致亂碼
curl -d '{"message":"中文"}' ntfy.sh/topic  # ← 可能亂碼

關鍵要點：

  • 必須加 charset=utf-8：確保中文正確顯示
  • 必須用檔案發送：Windows Bash 的 JSON 轉義不可靠
  • 發送後刪除暫存檔：避免累積
