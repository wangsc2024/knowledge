LangGraph put_writes 機制深度研究：並行 Superstep 原子性容錯與失敗恢復架構實作指南（2026）

技術概述

LangGraph 的 Checkpointing 系統採用雙軌寫入策略：put() 儲存完整狀態快照（checkpoint boundary），put_writes() 儲存節點中間輸出作為 pending writes 暫存。此設計賦予並行 superstep 原子性容錯能力——當多個節點並行執行時，若其中一個失敗，已完成節點的輸出透過 put_writes() 保留在暫存層；恢復時只有失敗分支重試，成功節點不重新執行。這是 daily-digest-prompt Phase 2 並行自動任務設計的核心機制，可將「全部失敗重跑」升級為「精準只重試失敗節點」。

---

核心概念與原理

1. put vs put_writes 雙軌設計

put(config, checkpoint, metadata, new_versions)：持久化完整狀態快照。將反序列化的通道值直接寫入 channel_values，並記錄對應的版本識別符。此方法代表 checkpoint boundary 的權威狀態。

put_writes(config, writes, task_id, task_path)：儲存節點任務的中間輸出，格式為 (task_id, channel, value) tuple list，與主 checkpoint 解耦，保留在獨立的 writes table 中，直到下一個執行週期才應用。

這種分離使引擎可以在不改動已提交 checkpoint 的情況下，支援中斷恢復與人工介入（Human-in-the-Loop）。[來源：DeepWiki langchain-ai/langgraph checkpointing-architecture, 2025]

2. Pending Writes 生命週期（二階段流程）


執行期 (Recording Phase)
├── 節點 A 執行 → put_writes([('task-A', 'result_a', value_a)])
├── 節點 B 執行 → put_writes([('task-B', 'result_b', value_b)])
└── 節點 C 失敗 → put_writes([(-1, 'error', exception_info)])  # ERROR=-1

恢復期 (Application Phase)
├── get_tuple() 呼叫 → pending_writes=[A的結果, B的結果]
├── Pregel loop 透過 reducer 應用 pending writes 到前進狀態
└── 只有節點 C 需要重試，A 和 B 的輸出直接複用

特殊寫入索引常數（sentinel values）：

  • ERROR = -1：節點錯誤寫入
  • SCHEDULED = -2：任務排程標記
  • INTERRUPT = -3：執行暫停信號（HITL 使用）
  • RESUME = -4：繼續執行值

這些常數讓引擎不需要名稱空間衝突即可區分操作型 metadata 與使用者狀態。[來源：DeepWiki langchain-ai/langgraph checkpointing-architecture, 2025]

3. 並行 Superstep 的原子性容錯機制

LangGraph 採用 Pregel superstep 執行模型：一個 superstep 是圖的單次「時鐘滴答」，所有該步驟排程的節點並發執行，全部完成後才進入下一步。

容錯保證（3個獨立來源確認）：

  1. 若並行節點中一個失敗，整個 superstep 從圖的外部視角看是「失敗」
  2. 但 LangGraph 內部已透過 put_writes() 保存所有成功節點的輸出
  3. 恢復時（重新呼叫 graph.invoke() 或 resume checkpoint），成功節點的輸出直接復用，只有失敗節點重新執行

# 範例：並行查詢場景
results: Annotated[list, operator.add]  # append-only reducer 必須

# superstep N: Wikipedia 成功, ArXiv 失敗
# → put_writes 保存 Wikipedia 結果
# → ArXiv 的 ERROR 寫入到 put_writes

# 恢復後
# → Wikipedia 節點不重執行（pending_writes 有其輸出）
# → 只有 ArXiv 節點重試

[來源 1：LangChain Persistence 官方文檔, 2025] [來源 2：LangChain Forum fan-out best practices, 2025] [來源 3：AI Practitioner Substack, 2025]

4. Channel Versioning 機制

每個通道攜帶版本識別符（str/int/float）：

  • channel_versions：checkpoint 中各通道當前版本
  • versions_seen：各節點已處理的版本狀態
  • 節點只在上游通道版本更新時才執行（Pregel 可達性判斷）

這確保並行場景中無論哪個節點失敗，版本號都能精準指出哪些節點需要重試。[來源：DeepWiki langchain-ai/langgraph checkpointing-architecture, 2025]

Fan-out/Fan-in 最佳實踐與陷阱

正確模式

必須使用 Reducer 定義合併規則：


from typing import Annotated
import operator

class ParallelState(TypedDict):
    task_results: Annotated[list, operator.add]  # append-only
    errors: Annotated[list, operator.add]

# 工作節點各自寫入 task_results，不互相覆蓋

避免交叉分支變異：讓並行工作節點寫入專用 key，由下游聚合節點合成，禁止多個並行節點覆寫同一標量值。否則會觸發 INVALID_CONCURRENT_GRAPH_UPDATE 錯誤。

Send API 動態 Fan-out（map-reduce 模式）：


from langgraph.types import Send

def routing_node(state):
    # 根據運行時狀態動態創建任務數量（2到N個）
    return [Send("worker_node", {"item": item}) for item in state["items"]]

RetryPolicy 節點級重試


from langgraph.pregel import RetryPolicy

graph.add_node(
    "api_caller",
    api_caller_fn,
    retry=RetryPolicy(max_attempts=3, initial_interval=1.0, backoff_factor=2.0)
)

並發控制


graph.compile(
    checkpointer=saver,

PostgresSaver 生產配置（2026 標準）


from langgraph.checkpoint.postgres import PostgresSaver
from psycopg_pool import ConnectionPool

DB_URI = "postgresql://user:pass@host:5432/langgraph?sslmode=require"

pool = ConnectionPool(
    conninfo=DB_URI,
    min_size=5,
    max_size=20,  # workers × max_size 需 < Postgres max_connections
    kwargs={
        "autocommit": True,          # 避免 LLM 呼叫期間持有長事務鎖
        "prepare_threshold": 0,      # PgBouncer 相容，禁用 prepared statements
    }
)

with pool.connection() as conn:
    saver = PostgresSaver(conn)
    saver.setup()  # 冪等，每次啟動可安全呼叫（自動建 schema）

graph = workflow.compile(checkpointer=saver)

三個關鍵設定的原因（來源：rapidclaw.dev 2026 deep dive）：

  1. autocommit=True：LLM 呼叫通常需要數秒，若在事務中等待會造成 Postgres 行鎖長期佔用，影響其他 worker
  2. prepare_threshold=0：PgBouncer 等連接池工具不支援 prepared statements，此設定確保相容性
  3. Schema 自動建立：不需要手動執行 migration，部署時自動處理

Async 高吞吐版本：

Temporal vs LangGraph 決策矩陣（容錯邊界）

| 場景 | 風險 | 建議架構 |

|------|------|----------|

| 單一唯讀任務 < 30s | 低 | LangGraph 足夠 |

| 3+ 工具呼叫或 API 重試 | 中高 | LangGraph + Temporal |

| 多小時/天的審批等待 | 高 | LangGraph + Temporal |

| 生產副作用（部署、支付） | 高 | LangGraph + Temporal + Governance Gate |

關鍵認知：「Durable execution（Temporal 強項）和 Governance（LangGraph 推理強項）是不同的控制維度，應分開處理，而非相互取代。」

[來源：cordum.io Temporal vs LangGraph 2026]

與本專案（daily-digest-prompt）的關聯

現況

daily-digest-prompt 的 Phase 2 並行執行多個 auto-task，目前採用「補寫 result file + 腳本補償」策略：

  • 若某 auto-task 失敗，Phase 3 讀到 status=failed 的佔坑 JSON
  • 此為「script-compensation 派」模式（retry + fallback + 補寫 result file）

可升級路徑

近期（低成本）：為 Phase 2 並行節點加入 RetryPolicy（節點級重試），比目前 Phase 3 組裝時才發現失敗更早恢復。

中期：引入 SqliteSaver（本機開發）→ 升級 PostgresSaver（生產），讓每個 auto-task 執行有 checkpoint，失敗後從 put_writes 保留的成功節點恢復，不重跑。

Put_writes 與結果檔的對應：

  • 目前 results/todoist-auto-{task_key}.json 扮演的角色 = LangGraph 的 put_writes 暫存
  • 兩者思路相同：Phase 3 組裝時只用「已成功的結果」，失敗的任務不影響其他任務輸出
  • 本專案已隱式實作 put_writes 語義，正式引入 LangGraph checkpointer 可使此機制更加健壯

程式碼範例（本專案適用）


# 在 LangGraph workflow 中，Phase 2 並行 auto-tasks
class DigestState(TypedDict):
    task_results: Annotated[list[dict], operator.add]  # 各任務結果累積
    failed_tasks: Annotated[list[str], operator.add]   # 失敗任務記錄

async def run_auto_task(state: DigestState, task_config: dict):
    """每個 auto-task 的執行節點"""
    try:
        result = await invoke_task(task_config)
        return {"task_results": [result]}
    except Exception as e:
        return {"failed_tasks": [task_config["task_key"]],
                "task_results": [{"status": "failed", "task_key": task_config["task_key"]}]}

# 啟用 checkpointer：任何節點失敗後，成功節點透過 put_writes 保留，不重跑
graph = workflow.compile(checkpointer=SqliteSaver.from_conn_string(":memory:"))
