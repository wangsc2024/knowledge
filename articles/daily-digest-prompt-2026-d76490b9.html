Daily-Digest-Prompt 2026-03-13 深度優化建議

執行可靠性危機、程式碼品質債務與配置膨脹趨勢的三維治理方案

研究日期：2026-03-13

類型：專案優化 / 深度思維 / 知識庫

適用對象：daily-digest-prompt 專案

方法：RAG 知識庫交叉比對 + 三路並行 Agent 靜態分析

---

一、分析方法論

本次分析採用三路並行子 Agent 策略：

  1. Config Health Agent：分析 10 個配置/狀態檔案的一致性與趨勢
  2. Test Coverage Agent：評估 956 個測試的覆蓋分佈與品質模式
  3. Code Quality Agent：掃描 ~20,000 LOC 的程式碼味道與架構問題

同時交叉比對 KB 內已有的 4 篇相關優化研究（優化日報 03-10、Skill 依賴圖、知識活化分析、狀態檔案治理），確保本報告聚焦尚未覆蓋的新發現。

---

二、面向一：執行可靠性危機（CRITICAL）

2.1 Phase 2 卡死問題

發現：run-fsm.json 中有 13 個 incomplete 的執行紀錄，全部卡在 Phase 1 完成但 Phase 2/3 從未啟動。最近 24 小時內無任何 Phase 3 完成紀錄。

影響：

  • Todoist 任務執行實際上處於半癱瘓狀態
  • 自動任務的研究結果無法組裝與通知
  • 使用者無法感知任務完成（ntfy 通知依賴 Phase 3）

建議：

  • 在 run-todoist-agent-team.ps1 的 Phase 1→2 銜接處加入診斷日誌
  • 檢查 Phase 2 的 timeout 機制是否過於激進（部分任務 timeout 僅 600s）
  • 加入 Phase 2 啟動確認（Phase 1 完成後 30 秒內未見 Phase 2 啟動 → 觸發告警）

2.2 今日異常 Phase 失敗率

發現：failure-stats.json 顯示 2026-03-13 已累計 10 次 phase 失敗，遠高於歷史日均 1-4 次。

建議：

  • 立即檢查 logs/structured/ 中今日的 JSONL 日誌
  • 比對 state/scheduler-heartbeat.json 確認排程器是否正常
  • 考慮加入 circuit breaker：連續 5 次失敗 → 暫停自動排程 + ntfy 告警

2.3 Token 使用量劇烈波動

發現：Token 使用從基線 2.3M（03-06）飆升至 30.5M（03-09），暴增 13 倍後又回落至 3.9M（03-12）。groq_calls 和 claude_calls 欄位尚未有數據填充。

根因推測：

  • 某個研究任務觸發了過深的遞迴搜尋
  • LLM Router 的 Groq 降級路徑未生效，全部 fallback 到 Claude

建議：

  • 在 tools/budget_guard.py 加入日粒度 Token 追蹤（非僅累計）
  • 填充 groq_calls/claude_calls 指標（schema v2 已定義但未實作記錄邏輯）
  • 設定每日 Token 軟上限（如 10M），超過時降級為 Groq-only 模式

三、面向二：程式碼品質債務（HIGH）

3.1 端點硬編碼散落

發現：localhost:3000（KB API）硬編碼於 5+ 個 Python 檔案中，localhost:3002（Groq Relay）硬編碼於 2 個檔案中。tools/config_loader.py 已存在但未被普遍使用。

建議：

  • 統一通過 config_loader.py 或 config/llm-router.yaml 載入端點
  • 設定環境變數 fallback：KB_API_URL、GROQ_RELAY_URL
  • 預估工時：2 小時

3.2 validate_config.py 巨型單體

發現：1,500+ LOC、880 語句、12% 測試覆蓋率。該檔案承擔 4 個職責：

  1. Schema 驗證
  2. Skill 品質評分
  3. 配置遷移
  4. DAG 圖生成

建議：拆分為 4 個模組：

  • hooks/config_validator.py（核心驗證）
  • tools/skill_quality_scorer.py（品質評分）
  • tools/config_migrator.py（遷移腳本）
  • tools/config_dag_generator.py（DAG 生成）

3.3 print() 汙染生產程式碼

發現：70+ 個 print() 呼叫散落於 hooks 和 tools 中，特別是 validate_config.py（60+）和 query_logs.py（40+）。這些輸出不受結構化日誌系統管控。

建議：

  • 引入 logging 模組，設定 StreamHandler + FileHandler
  • 用 logger.info/warning/error 替換 print()
  • 預估工時：3 小時

3.4 裸 Exception 捕獲

發現：8+ 個 except Exception: 散落在 hook_utils.py、on_stop_alert.py、behavior_tracker.py 中，可能隱藏真正的錯誤。

建議：逐一替換為具體異常類型（json.JSONDecodeError、FileNotFoundError、yaml.YAMLError 等），並記錄被捕獲的異常。

四、面向三：測試覆蓋與品質缺口（MEDIUM）

4.1 覆蓋率分佈失衡

| 模組 | 覆蓋率 | 語句數 | 評估 |

|------|--------|--------|------|

| hook_pipeline.py | 100% | — | 優秀 |

| agent_guardian.py | 83% | — | 良好 |

| validate_config.py | 12% | 880 | 危險 |

| on_stop_alert.py | 46% | 592 | 不足 |

| post_tool_logger.py | 41% | 311 | 不足 |

建議：優先提升 validate_config.py 至 60%（加 schema 違規、型別錯誤、空 YAML 邊界測試），可新增 ~50 個測試案例。

4.2 PowerShell 零測試

發現：18 個主要 .ps1 腳本（~10,000 LOC）完全沒有自動化測試。所有驗證依賴手動執行。

建議：

  • 為核心腳本加入 -DryRun 模式
  • 用 Pester（PowerShell 測試框架）撰寫 FSM 狀態轉換測試
  • 優先目標：run-todoist-agent-team.ps1（最複雜，3 階段並行）

4.3 工具模組測試空白

發現：compose_video、generate_podcast_audio、generate_report_pdf、md_to_docx 等媒體/文件工具完全無測試。

建議：用 mock 模式測試輸入驗證與輸出格式，不需要真實 FFmpeg/LibreOffice。

五、面向四：配置膨脹趨勢（MEDIUM）

5.1 YAML 成長速度警告

發現：config-metrics.json 顯示 3 天內：

  • YAML 總行數：3,052 → 3,402（+11%）
  • YAML 檔案數：22 → 25（+14%）
  • Hook 程式碼：4,617 → 4,840 行（+5%）

按此速率，一個月後 YAML 將超過 5,000 行，認知負擔顯著增加。

5.2 配置合併建議

| 現有（分散） | 建議合併為 | 理由 |

|-------------|-----------|------|

| health-scoring.yaml + benchmark.yaml | system-metrics.yaml | 同屬系統度量 |

| audit-scoring.yaml + slo.yaml | audit-slo.yaml | 同屬審計標準 |

| notification.yaml + ntfy 相關 | notification.yaml（已整合） | 確認不重複 |

5.3 frequency-limits.yaml 品質問題

  • 重複條目：ai_smart_city 在 codex_exec 規則中出現兩次（行 158、161）
  • execution_order 不連續：github_scout 的 order=23 超出文件範圍（1-20）
  • 註解不一致：宣稱 22 個任務但實際只有 14 個啟用

八、結論

本專案的架構基礎穩固（Skill-First + 文件驅動 + Hook 強制層），但執行層面存在可靠性危機。Phase 2 卡死和失敗率飆升是最緊迫的問題——系統在「設計完善但執行不穩」的狀態。建議立即投入 P0 診斷，在穩定執行後再處理程式碼品質和配置膨脹等中期議題。

一句話總結：架構好但跑不穩——先修管線再提品質。
