SRE × LLM：Blameless Postmortem 與 On-Call Runbook 草案自動化的工程化實務（2026-03-31）

技術概述

事件應變（incident response）的最後一哩路常卡在兩件事：可重用的 Runbook 與可學習的 Blameless Postmortem。傳統作法仰賴值班者手寫，容易出現時間軸缺漏、證據散落與結論不可稽核等問題。LLM 的強項是把「結構化信號」（日誌欄位、指標突變時間窗、追蹤關聯、部署／變更紀錄）轉成「人類可讀敘事」，但必須加上 證據邊界、引用規則 與 人機共筆閘門，才不會把推測當成定論。此模式與 Google SRE 對事後檢討的文化要求一致：聚焦系統與流程，而非個人究責；同時讓每次事件都能回灌到可觀測性與 CI/CD 治理。

核心方法與工具

| 方法 | 用途 | 代表做法 |

|------|------|-----------|

| 證據包（Evidence Bundle） | 限定 LLM 輸入為可追溯片段 | 從 Loki/Elastic/CloudWatch 擷取時間窗內 log；Prometheus/Datadog 擷取 SLO burn rate；Tempo/Jaeger 擷取 trace |

| 時間軸正規化 | 合併多來源事件 | 以 UTC 統一時間戳；為每條事件標註 source、confidence（measured/inferred） |

| 範本驅動生成 | 維持章節一致性 | 採固定 Markdown 章節：摘要／影響／時間軸／根因／事實 vs 推測／緩解／後續行動 |

| 人機共筆閘門 | 防幻覺與誤導 | LLM 產「草案」→ SRE 勾選「已核對」欄位 → 方可合併至 docs/postmortems/ |

| Runbook 片段庫 | 知識複用 | 以 instruction + symptom pattern 索引；生成時只組合「已驗證片段」 |

工具鏈可選元件：OpenTelemetry Collector（標準化信號）、告警系統（PagerDuty/Opsgenie）、Git 作為 postmortem 來源真實性（PR 審查）、以及貴組織偏好的 LLM API（與本專案既有的 llm-router.yaml 類似概念）。

實際工作流程（step-by-step）

  1. 觸發：事件結案或 SEV 等級下降後，由 webhook 或排程建立 postmortem_job（附 incident_id、時間窗、services[]）。
  2. 蒐集：在時間窗內自動拉取 log/metric/trace 摘要（注意 PII 脫敏與取樣上限）。
  3. 正規化：將資料壓成 JSON「證據包」，每筆附 ref（例如 query link 或 object id）。
  4. 生成：呼叫 LLM，系統 prompt 固定要求：

- 只在「時間軸／影響」段落引用證據包內欄位；

- 「根因」段落必須分成 已證實 與 假設 兩欄；

- 不可用證據包以外的外部事實。

  1. Runbook 草案：平行產出「若再發生」步驟清單，每步需對應可執行命令或儀表板連結占位符。
  2. 人審：SRE 於 PR 中勾选 checklist，缺失證據段落退件給自動蒐集器補件。
  3. 收斂：合併後推送到 Confluence/Notion/Repo，並在事項追蹤開 action items（含 owner／due）。

GitHub Actions 觸發（片段）


on:
  workflow_dispatch:
    inputs:
      incident_id: { required: true, type: string }
jobs:
  draft-postmortem:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build evidence bundle
        run: python scripts/export_evidence.py --id "${{ github.event.inputs.incident_id }}"
      - name: Ask LLM for draft
        run: python scripts/llm_postmortem_draft.py --bundle out/evidence.json
      - name: Open PR
        uses: peter-evans/create-pull-request@v6
        with:
          commit-message: "chore(postmortem): draft for ${{ github.event.inputs.incident_id }}"

效果量化（方向性資料）

產業實務上，導入「告警關聯＋降噪」常見報告可減少 70–90% 警報竄升（多為平台白皮書與使用者案例區間；實際數字依既有噪訊基線而異）。Postmortem 自動草案若能把初稿時間從數小時降到 30–60 分鐘內可審閱狀態，通常可顯著提升行動項關閉率；建議以貴團隊「事件→PR 合併」前置時間（lead time）作為內部 KPI，而非直接照搬外部百分比。

與本專案（daily-digest-prompt）的應用場景

  • Hooks＋JSONL：post_tool_logger.py 產生的結構化日誌可直接餵入證據包，對齊 error.type／RFC7807 風格欄位，讓 LLM 在時間軸段落引用穩定鍵名。
  • 排程與狀態：scheduler-state.json 與 Phase 1/2/3 失敗事件可作為「影響面」與「是否為重試可解」分類的根據。
  • Runbook：可把「Groq quota 爆掉」「Phase 1 plan.json 格式錯誤」等既有模式做成片段庫，與 config/timeouts.yaml、query-logs.ps1 的診斷步驟連動。
  • 人機共筆：延續專案既有的 Draft-Verify 精神—LLM 產出僅為草稿，必須經過結果 JSON／postmortem PR 的核可清單。

風險與注意事項

  • 幻覺根因：必須分欄標示假設；Production 重大事件建議強制二次人工簽核。
  • 隱私與秘密：日誌去識別；證據包不可含 token／cookie。
  • 成本：長時間窗＋大量 log 會消耗模型上下文；需截斷與聚類（例如先以關鍵字／級別過濾）。
  • 責任邊界：自動草案不取代變更管理；與 CAB/Release 流程並行。
