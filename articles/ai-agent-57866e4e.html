AI Agent 架構設計 深度研究報告

取用日期：2026-05-03（Asia/Taipei）

執行摘要

原理：LLM 作認知核心，協調層管理「推理—工具—觀察」迴圈與終止；OpenAI 文件區分 LLM 與程式編排 [來源：OpenAI Agents SDK, 2025]。資策會整理 Google 白皮書之四元件（模型、工具、協調、部署）與分級 [來源：資策會數轉院, 2025]。進展：arXiv:2604.18133 對照傳統 MAS 與 LFM 型 MAS 之架構遷移 [來源：Wang et al., 2026]；arXiv:2601.13243 指出多代理結構複雜度與正確率非單調關係 [來源：Li et al., 2026]。案例：Anthropic 示範極簡鷹架於 SWE-bench Verified 達 49%，強調評的是 agent 系統 [來源：Anthropic, 2025]。挑戰：幻覺／工具迴圈與上下文暴衝需預算與斷路器 [來源：masturbyte, 2026]；SWE-bench 對非修 bug 場景預測力弱 [來源：Groundy, 2026]。結論：四層架構＋鷹架版本治理＋成本實測＋內部標竿，四者並重。

研究背景與動機

企業已普遍導入生成式 AI，但「聊天建議」與「端到端完成工作」之間存在鴻溝。Agent 架構把 LLM 放入可觀測、可限制、可重試的控制迴路中，並連結企業 API 與知識庫；若缺乏架構視角，易淪為一次性腳本或無法稽核的黑箱。研究缺口在於：學術調查（LMAS 與傳統 MAS 比較、推理範式評估）與 產業落地敘事（編排、KPI、治理）之間需要可操作的橋接語彙——本報告以三角佐證彙整兩邊觀點，並對齊本專案既有的 Agent 管線（Phase 腳本、結果 JSON、ntfy／KB 整合）脈絡。

核心技術解析

子題一：單代理迴圈與工具介面

ReAct 類模式透過反覆「思考→選工具→讀觀察」讓中間推理可追溯，降低純 CoT 在缺資料時的胡說風險；產業文章亦將其與 reflection、planning 並列為 agentic design pattern 家族。工具 schema 與錯誤訊息設計直接影響成功率——Anthropic 明確將大量心力放在 Bash／Editor 工具描述與防呆（例如強制絕對路徑）以減少模型誤解 [來源：Anthropic SWE-bench Sonnet 研究文章, 2025]。

子題二：多代理編排與狀態

OpenAI 文件將編排分為 LLM 決策 與 程式碼決策：前者適合開放式任務，後者適合需可預測成本／延遲的流程；並建議投資在提示、eval、專一代理而非萬能代理 [來源：OpenAI Agents SDK, 2025]。Wang 等之 arXiv 調查則從學理整理 LMAS 的 角色定義、感知層級、規劃、記憶、執行 五大模組，以及合作／競爭／混合架構與 MCP、A2A 等通訊協定之角色 [來源：arXiv:2604.18133, 2026]——與資策會文章對 MCP／A2A 的「互操作性支柱」敘述互相呼應 [來源：資策會數轉院, 2025]。

子題三：評測、基準飽和與「代理」邊界

SWE-bench Verified 以 500 筆人類審核可解題之 GitHub issue 為子集，採 FAIL_TO_PASS／PASS_TO_PASS 全過才計分；Groundy 文章提醒：高分僅代表在該 Docker 化、單次提交設定下之修補能力，不等於程式品質、資安或長期維護 [來源：Groundy, 2026]。Anthropic 亦指出長回合 token 消耗、隱藏測試與「模型以為已修好」等實務痛點 [來源：Anthropic, 2025]。arXiv:2601.13243 則從方法學補刀：在固定模型下比較單模 CoT 與多代理工作流，複雜結構不必然提升推理表現 [來源：arXiv:2601.13243, 2026]——與「別為了多代理而多代理」的工程直覺一致。

關鍵洞見

  1. ✅ 四層解剖（模型／工具／協調／部署）是跨文檔共識（Google 白皮書路線圖、資策會整理、OpenAI 編排文件均相容）；支持來源數：≥3。
  2. ✅ 「Agent = 模型 + 鷹架」在 SWE-bench 敘述中為官方立論；同模型不同鷹架可造成大幅分數差 [來源：Anthropic, 2025；Groundy, 2026]；支持來源數：≥3。
  3. ✅ 多代理與 LFM 帶來語意層協調與角色分工，但經典 MAS 在可證明控制與低階運動仍不可替代 [來源：arXiv:2604.18133, 2026]；支持來源數：≥2。
  4. ⚠️ 企業預測數字（如某年內建 Agent 之應用比例）多屬趨勢敘事，實際比例依調查定義而異，宜當情境假設而非硬 KPI；支持來源偏產業報導 [來源：CIO Taiwan, 2026]；支持來源數：1–2（標註待外部獨立驗證）。
  5. ✅ 生產故障模式（迴圈、暴衝、注入、狀態競爭）已有系統化清單與緩解 [來源：masturbyte, 2026]；支持來源數：≥2（與 OpenAI「監控與 eval」建議合流）。
  6. ⚠️ 封閉式基準高分 vs. 開放式語意能力 之間仍有方法裂縫；arXiv:2601.13243 提出 MIMeBench 作為補充軸 [來源：arXiv:2601.13243, 2026]；支持來源數：1（前沿論文，落地工具鏈仍演進中）。

（反方觀點摘要）產業可質疑：學術調查偏一般化框架，未必涵蓋貴司技術棧；學界可質疑：產業文章引用第三方預測時常缺乏可重現數據欄位——故規劃上應 內部標竿測試 與 學術指標 並行。

實務應用建議

  1. 先畫編排邊界：為每一條使用者旅程標註「LLM 決策」與「程式決策」區段；高風險動作（刪除、對外付款）鎖在程式決策區並加人審。依據：OpenAI 文件之 tradeoff 論述 [來源：OpenAI Agents SDK, 2025]。風險：過度程式化會失去彈性，需每季檢討分界。
  2. 實施工具預算與斷路器：限制每請求工具呼叫次數、相同參數重複呼叫、工具輸出 token；超標即降級為部分結果或人工作業。依據：masturbyte 故障模式與範例程式 [來源：masturbyte, 2026]。風險：閾值過緊可能誤殺合理長任務。
  3. 建立「鷹架版本＋模型版本」雙維度 registries：任何 SWE 或內部基準分數必須附 agent 版本 與 prompt／工具 schema 雜湊，避免無法重現的數字治理。依據：Anthropic、Groundy [來源：2025–2026]。
  4. 多代理採「小隊＋窄提示」而非單體巨提示：與 OpenAI「專一代理」及 Wang 等「角色定義」一致 [來源：OpenAI, 2025；arXiv:2604.18133, 2026]。風險：代理間通訊成本上升，需監控 MCP／A2A 流量與延遲。
  5. 以 arXiv:2601.13243 為鑑：在引入複雜 MAS 前先跑 成本—正確率 曲線，確認相對於「強 CoT 單模」的邊際效益。風險：實驗工時；長期則降低架構債。

未來展望

仍未完全解決之問題包括：（1）長 horizon 任務上，可靠性是否隨 token 預算單調改善（產業與 METR 類敘事指出高原現象，待與貴司任務長度對照）；（2）多代理通訊的標準化（MCP／A2A）在跨廠商治理上的成熟度；（3）開放式語意評測與封閉式工單評測的統一儀表板。下次可深化方向：「Agent 編排＋本專案 ntfy／排程／結果 JSON 守衛」一體化 SLO 設計（約 15 字內可寫為：排程與結果守衛對齊）。
