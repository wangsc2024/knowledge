Unsloth 資料集準備與格式化最佳實踐

概述

Unsloth 微調框架支援多種資料集格式，從單輪指令（Alpaca）到多輪對話（ShareGPT/ChatML），再到偏好最佳化（DPO/ORPO/KTO）。正確的資料準備是微調成功的關鍵：品質遠比數量重要，格式一致性決定訓練效率。本筆記涵蓋四大核心面向：(1) 四種基礎資料格式的結構與選型，(2) standardize_sharegpt 與 to_sharegpt 格式轉換工具鏈，(3) DPO/ORPO/KTO 偏好資料集的建構方法，(4) 自定義資料管線的端到端實作流程。這是 Unsloth 微調知識庫中「訓練之前」的關鍵一環。

---

一、四種基礎資料格式

Unsloth 支援四種主要資料類型，適用於不同的微調場景：

1.1 Raw Corpus（原始語料）

連續文本，用於繼續預訓練（Continued Pre-training），無結構要求：


{"text": "這是一段長篇文章的內容，用於讓模型學習特定領域知識..."}

適用場景：領域適應（如醫療、法律、金融語料）、語言適應（繁中語料注入）。

1.2 Alpaca 格式（單輪指令）

最簡單的指令微調格式，三個欄位：


{
  "instruction": "將以下句子翻譯為英文",
  "input": "今天天氣很好",
  "output": "The weather is nice today."
}

注意：input 欄位為可選。若無額外上下文，可留空字串或省略。

適用場景：指令遵循、翻譯、摘要等單輪任務。

1.3 ShareGPT 格式（多輪對話）

使用 from/value 鍵值對，角色在 human 和 gpt 之間交替：


{
  "conversations": [
    {"from": "human", "value": "什麼是量子計算？"},
    {"from": "gpt", "value": "量子計算是利用量子力學原理進行資訊處理的計算範式..."},
    {"from": "human", "value": "它和傳統計算有什麼差異？"},
    {"from": "gpt", "value": "主要差異在於計算單元：傳統計算用位元（0或1），量子計算用量子位元..."}
  ]
}

適用場景：聊天機器人、多輪問答、角色扮演。

1.4 ChatML 格式（OpenAI 標準）

使用 role/content 鍵值對，是 Hugging Face 和 OpenAI 的標準格式：


{
  "messages": [
    {"role": "system", "content": "你是一位有幫助的助手。"},
    {"role": "user", "content": "解釋機器學習"},
    {"role": "assistant", "content": "機器學習是人工智慧的子領域..."}
  ]
}

適用場景：與 ChatML 模板的模型（如 Qwen、Phi）配合使用。

格式選型指南

| 格式 | 多輪 | System Prompt | 生態系統 | 推薦模型 |

|------|------|---------------|---------|----------|

| Alpaca | 否 | 否 | 經典 | 通用 |

| ShareGPT | 是 | 可選 | 社群主流 | Llama, Gemma |

| ChatML | 是 | 是 | OpenAI/HF 標準 | Qwen, Phi, Mistral |

| Raw Corpus | N/A | N/A | 預訓練 | 任意基座模型 |

二、格式轉換工具鏈

Unsloth 提供兩個核心函數處理資料格式轉換：

2.1 `standardize_sharegpt()` — ShareGPT → ChatML

將 ShareGPT 格式（from/value）轉換為 ChatML 格式（role/content）：


from unsloth.chat_templates import standardize_sharegpt
from datasets import load_dataset

# 載入 ShareGPT 格式資料集
dataset = load_dataset("mlabonne/FineTome-100k", split="train")

# 轉換為標準 ChatML 格式
dataset = standardize_sharegpt(dataset)
print(dataset[0])  # 確認格式已轉換

轉換對映：

  • from: "human" → role: "user"
  • from: "gpt" → role: "assistant"
  • value: "..." → content: "..."

2.2 `to_sharegpt()` — 多欄位 → 對話格式

將含有多個欄位的結構化資料集（如 CSV/表格）合併為 ShareGPT 格式：


from unsloth.chat_templates import to_sharegpt
from datasets import load_dataset

# 載入含多欄位的資料集（如 Titanic）
dataset = load_dataset("HuggingFaceTB/titanic-survival", split="train")

# 定義合併模板
dataset = to_sharegpt(
    dataset,
    merged_prompt = """\
以下是一位乘客的資料：
[[該乘客從 {Embarked} 登船。]]
[[年齡為 {Age} 歲。]]
[[票價為 {Fare} 元。]]
性別：{Sex}，艙等：{Pclass}
請預測此乘客的生還機率。""",
    output_column_name = "Survived",
    conversation_extension = 3,  # 隨機合併 3 筆成多輪
)

關鍵參數說明：

  • {column_name}：花括號包圍的欄位名，自動替換為實際值
  • [[optional text]]：雙方括號包圍的可選區段，若欄位為空自動跳過
  • output_column_name：指定作為回應的欄位
  • conversation_extension：將多筆單輪合併為多輪對話的筆數

2.3 `get_chat_template()` — 套用模型特定模板


from unsloth.chat_templates import get_chat_template, CHAT_TEMPLATES

# 查看所有支援的模板
print(list(CHAT_TEMPLATES.keys()))
# ['llama-3', 'llama-3.1', 'llama-3.2', 'mistral', 'gemma-2', 
#  'gemma-3', 'qwen-2.5', 'phi-3', 'phi-4', 'chatml', 'alpaca', ...]

# 套用模板到 tokenizer
tokenizer = get_chat_template(
    tokenizer,
    chat_template = "llama-3.1",  # 選擇與模型匹配的模板
)

重要：使用 Unsloth 的 get_chat_template 而非 tokenizer 原生方法，因為 Unsloth 版本經過「徹底檢查與修正」。

三、偏好資料集建構（DPO/ORPO/KTO）

3.1 DPO 資料集格式

Direct Preference Optimization 需要三欄式偏好資料：


{
  "prompt": "如何有效學習程式設計？",
  "chosen": "有效學習程式設計建議從基礎開始，選擇一門語言深入學習，透過實際專案練習，並善用社群資源如 Stack Overflow 和 GitHub。建立每日練習習慣，從小專案到大專案循序漸進。",
  "rejected": "就上網找教學看一看就好了。"
}

推薦資料集：

  • argilla/ultrafeedback-binarized-preferences（60k 筆）
  • HuggingFaceH4/ultrafeedback_binarized
  • 自建：用 GPT-4 或 Claude 生成 chosen/rejected 對

3.2 ORPO 實戰配置

ORPO 將 SFT 和偏好對齊合併為單一訓練階段，無需參考模型：


from unsloth import FastLanguageModel
from trl import ORPOConfig, ORPOTrainer

# 載入模型
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/Llama-3.2-3B-bnb-4bit",
    max_seq_length = 2048,
    load_in_4bit = True,
)

# 配置 LoRA
model = FastLanguageModel.get_peft_model(
    model,
    r = 64,
    target_modules = ["q_proj", "k_proj", "v_proj", "o_proj",
                      "gate_proj", "up_proj", "down_proj"],
    lora_alpha = 64,
    lora_dropout = 0,
    use_gradient_checkpointing = "unsloth",
)

# ORPO 訓練配置
orpo_args = ORPOConfig(
    learning_rate = 8e-6,
    beta = 0.1,  # 偏好最佳化強度
    per_device_train_batch_size = 4,
    gradient_accumulation_steps = 2,
    num_train_epochs = 3,
    max_seq_length = 2048,
    lr_scheduler_type = "cosine_with_restarts",
    optim = "paged_adamw_8bit",
    logging_steps = 1,
    output_dir = "outputs",
)

# 啟動訓練
orpo_trainer = ORPOTrainer(
    model = model,
    args = orpo_args,
    train_dataset = dataset,
    tokenizer = tokenizer,
)
orpo_trainer.train()

3.3 KTO 資料集格式

Kahneman-Tversky Optimization 只需二元標記（好/壞），不需配對：


[
  {"prompt": "解釋 RESTful API", "completion": "RESTful API 是...", "label": true},
  {"prompt": "解釋 RESTful API", "completion": "不知道", "label": false}
]

KTO 優勢：資料收集成本低於 DPO（不需配對），適合大規模弱標註場景。

3.4 偏好資料集建構策略

| 方法 | 成本 | 品質 | 適用規模 |

|------|------|------|----------|

| 人工標註 | 高 | 最高 | < 5k |

| GPT-4/Claude 生成 | 中 | 高 | 5k-50k |

| 現有排行榜資料 | 低 | 中 | > 50k |

| LLM-as-Judge 自動評分 | 低 | 中-高 | 任意 |

四、端到端自定義資料管線

4.1 完整流程範例


from unsloth import FastLanguageModel
from unsloth.chat_templates import (
    get_chat_template, 
    standardize_sharegpt,
    to_sharegpt
)
from datasets import load_dataset

# Step 1: 載入模型與 tokenizer
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name = "unsloth/Qwen2.5-7B-bnb-4bit",
    max_seq_length = 4096,
    load_in_4bit = True,
)

# Step 2: 套用 Chat Template
tokenizer = get_chat_template(
    tokenizer,
    chat_template = "qwen-2.5",
)

# Step 3: 載入並轉換資料集
dataset = load_dataset("your_org/custom_dataset", split="train")

# 3a: 若為 ShareGPT 格式 → 標準化
dataset = standardize_sharegpt(dataset)

# 3b: 若為多欄位 CSV → 合併為對話格式
# dataset = to_sharegpt(dataset, merged_prompt="...", output_column_name="answer")

# Step 4: 格式化為訓練輸入
def formatting_prompts_func(examples):
    convos = examples["conversations"]
    texts = [
        tokenizer.apply_chat_template(
            convo, 
            tokenize=False, 
            add_generation_prompt=False
        ) for convo in convos
    ]
    return {"text": texts}

dataset = dataset.map(formatting_prompts_func, batched=True)

# Step 5: 檢查格式化結果
print(dataset[0]["text"][:500])

4.2 合併多資料集

Unsloth 建議在訓練前統一合併資料集，而非依序微調：


from datasets import concatenate_datasets, load_dataset

# 載入多個資料集
ds1 = load_dataset("dataset_a", split="train")
ds2 = load_dataset("dataset_b", split="train")

# 分別標準化格式
ds1 = standardize_sharegpt(ds1)
ds2 = standardize_sharegpt(ds2)

# 分別套用格式化
ds1 = ds1.map(formatting_prompts_func, batched=True)
ds2 = ds2.map(formatting_prompts_func, batched=True)

# 合併
combined = concatenate_datasets([ds1, ds2])
combined = combined.shuffle(seed=42)

五、資料品質檢查清單

5.1 訓練前驗證

  • [ ] 格式一致性：所有樣本使用相同的鍵值結構
  • [ ] 編碼正確：UTF-8，無亂碼或不可見字元
  • [ ] 長度分佈：檢查 token 長度，移除超過 max_seq_length 的樣本
  • [ ] 角色交替：對話格式中 user/assistant 嚴格交替
  • [ ] 空值處理：移除或填補空白回應
  • [ ] 去重：移除完全相同的樣本（hash 比對）

5.2 資料集規模建議

| 規模 | 品質 | 預期效果 |

|------|------|----------|

| 100 筆 | 高品質手工 | 基本能力展示 |

| 1,000 筆 | 混合（手工+合成） | 強效能 |

| 10,000+ 筆 | 多來源清洗 | 生產級 |

| 50,000+ 筆 | 大規模標註 | 接近 SOTA |

5.3 常見陷阱

  1. 模板不匹配：訓練用 ChatML 模板但推論用 Llama 模板 → 效能驟降
  2. 遺忘 System Prompt：DPO 訓練時忘記加入 system prompt → 偏好學習不穩定
  3. 序列截斷：max_seq_length 設太小，重要回應被截斷 → 品質下降
  4. 多資料集衝突：不同來源的回應風格衝突 → 建議統一後處理
  5. ShareGPT 角色錯亂：第一條訊息不是 human → 訓練崩潰

六、自定義 Chat Template

Unsloth 允許建立完全自定義的 Chat Template：


from unsloth.chat_templates import get_chat_template

# 自定義模板（Jinja2 格式）
custom_template = (
    "{% for message in messages %}"
    "{% if message['role'] == 'user' %}"
    "[用戶] {{ message['content'] }}\n"
    "{% elif message['role'] == 'assistant' %}"
    "[助手] {{ message['content'] }}{{ eos_token }}\n"
    "{% endif %}"
    "{% endfor %}",
    "eos_token"  # EOS token 對映
)

tokenizer = get_chat_template(
    tokenizer,
    chat_template = custom_template,
)

注意：自定義模板必須包含 EOS token 對映，否則模型無法正確學習回應結束位置。
