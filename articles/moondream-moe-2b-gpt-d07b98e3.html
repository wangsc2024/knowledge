一、專案概述

Moondream（vikhyat/moondream）是由 M87 Labs 的 Vikhyat Korrapati 開發的開源視覺語言模型（VLM）家族，核心理念是「Tiny Vision Language Model That Kicks Ass and Runs Anywhere」。截至 2026 年 2 月已累積 9,400+ Stars、729 Forks、25 位貢獻者，採用 Python（95.8%）開發。

核心價值

  • 極致效率：9B 總參數但僅 2B 活躍（MoE），推理速度媲美 2B 密集模型
  • 無所不在：從 Raspberry Pi、手機到雲端 GPU 皆可運行
  • 超越預期：在多項基準測試中擊敗 GPT-5 與 Gemini 2.5-Flash
  • Apache 2.0（Moondream 2）/ BSL 1.1（Moondream 3）：研究與個人專案可自由使用

模型家族

| 模型 | 參數量 | 活躍參數 | 定位 | 授權 |

|------|--------|---------|------|------|

| Moondream 0.5B | 500M | 500M | 邊緣裝置蒸餾目標 | Apache 2.0 |

| Moondream 2B | 1.8B | 1.8B | 通用 VLM（密集模型） | Apache 2.0 |

| Moondream 3 Preview | 9B | 2B | 前沿推理（MoE） | BSL 1.1 |

二、技術架構（核心）

2.1 MoE（Mixture-of-Experts）架構設計

Moondream 3 是視覺語言模型中首批採用 MoE 的小型模型之一：

  • 24 層 Transformer

- 前 4 層：密集層（所有參數活躍，學習通用表徵）

- 後 20 層：MoE 層（64 個專家，每個 token 激活 8 個）

  • 隱藏維度：2048
  • FFN 設計：GeGLU 架構，inner/gate dimension 1024
  • 路由策略：學習型路由器（learned router），訓練初期加 load-balancing loss + router orthogonality loss，後期訓練時關閉 load balancing 以防災難性遺忘

2.2 視覺編碼器

  • SigLIP-based：使用 Sigmoid Loss for Image-Language Pre-training
  • 多裁切通道串接（Multi-crop Channel Concatenation）：Token 效率的高解析度處理
  • 影像編碼可快取（encode_image()），對同一影像進行多次推理時僅需編碼一次

2.3 上下文窗口

  • 32K tokens（從 Moondream 2 的 2K 大幅擴展 16 倍）
  • 訓練時直接在預訓練階段交錯長上下文樣本（4096 預設長度）
  • 實作「學習溫度縮放」（Learned Temperature Scaling），靈感來自 YaRN 方法

2.4 Tokenizer

  • 自研 SuperBPE 分詞器（starmie-v1），比標準 BPE 更高效
  • 支援多語言文字處理

2.5 注意力機制

  • Multi-headed Attention + 位置相關 + 資料相關的學習溫度縮放
  • 推理時支援 FlexAttention + .compile() 優化

五、快速上手

5.1 安裝


pip install transformers torch pillow

5.2 基本使用


import torch
from transformers import AutoModelForCausalLM
from PIL import Image

# 載入模型
moondream = AutoModelForCausalLM.from_pretrained(
    "moondream/moondream3-preview",
    trust_remote_code=True,
    dtype=torch.bfloat16,
    device_map={"":"cuda"},
)
moondream.compile()  # FlexAttention 優化

# 視覺問答
image = Image.open("photo.jpg")
result = moondream.query(image=image, question="What's in this image?")
print(result["answer"])

# 物件偵測
result = moondream.detect(image, "person")
for obj in result["objects"]:
    print(f"({obj['x_min']:.2f}, {obj['y_min']:.2f}) - ({obj['x_max']:.2f}, {obj['y_max']:.2f})")

# 影像編碼快取（多次查詢同一影像）
encoded = moondream.encode_image(image)
for q in ["How many people?", "What time of day?"]:
    r = moondream.query(image=encoded, question=q, reasoning=False)
    print(r["answer"])

八、優缺點評估

優點

  1. 極致效率比：2B 活躍參數達到千億級模型的水準，堪稱 VLM 領域的「效能之王」
  2. 真正的邊緣部署：不只是「可以跑」，而是「跑得好」— Raspberry Pi 可用
  3. MoE 架構創新：64 專家 / 8 激活的細粒度路由，在小型 VLM 中首創
  4. 完整功能集：Query/Caption/Point/Detect 四合一，無需多模型切換
  5. 開發者友好：Transformers 原生支援，3 行程式碼即可使用
  6. 影像編碼快取：對同一影像多次查詢時大幅節省推理時間

缺點

  1. Moondream 3 授權限制：BSL 1.1 對商業 API 託管有限制（Moondream 2 仍為 Apache 2.0）
  2. Preview 階段：推理程式碼未完全優化，實際速度低於預期
  3. 生態系統較小：相比 LLaVA 等主流 VLM，社群插件與整合較少
  4. 長上下文能力待驗證：32K 窗口剛推出，實戰經驗不足
  5. GPU 記憶體需求：Moondream 3 仍需 CUDA GPU（bfloat16），純 CPU 推理困難
  6. 單一維護者風險：核心開發主要由 Vikhyat 一人驅動

九、關鍵技術決策分析

為什麼選擇 MoE 而非更大的密集模型？

Moondream 3 的設計哲學是「用架構創新換取效率」而非「用規模換取能力」。透過 Drop Upcycling 從 2B 密集模型初始化 9B MoE 模型，保留了已學習的知識同時引入專家化分工。這比從零訓練 9B 密集模型節省了大量計算資源，且推理時僅使用 2B 參數。

為什麼前 4 層用密集層？

前幾層處理低層次視覺特徵（邊緣、紋理、形狀），這些特徵具有高度通用性，所有 token 都需要。只有在高層語義理解時，不同類型的問題（偵測 vs 問答 vs OCR）才需要不同的專家處理。

訓練策略的巧妙之處

先用 load-balancing loss 確保專家被均勻利用（避免死亡專家），再在後期關閉 load balancing 讓模型自然學習最優路由。這個兩階段策略是 MoE 訓練的最佳實踐。
