Cloudflare Workers 完整指南 — V8 Isolate 架構、儲存整合與 Edge Computing 最佳實踐

技術概述（200 字）

Cloudflare Workers 是基於 V8 Isolate 的無伺服器邊緣計算平台，在全球 300+ 邊緣節點執行程式碼，冷啟動低於 1ms（對比 AWS Lambda 的 100-1000ms）。與傳統容器化 Serverless 不同，Workers 在同一 OS 進程內運行多個隔離的 V8 Isolate，大幅降低啟動成本。平台實現 WinterCG 標準（fetch、Request/Response、URL、Headers、Crypto API），支援 JavaScript/TypeScript、Python（Pyodide→WASM）、Rust（workers-rs）與 WebAssembly。搭配 R2 物件儲存、D1 SQLite 資料庫、KV 全球鍵值對與 Durable Objects 有狀態協調器，構成完整的邊緣全端應用開發平台。

---

一、核心架構：V8 Isolate 模型

1.1 與傳統 Serverless 的根本差異

| 特性 | 傳統 Serverless (Lambda) | Cloudflare Workers |

|------|------------------------|--------------------|

| 隔離單位 | Container / microVM | V8 Isolate |

| 冷啟動 | 100-1000ms | <1ms |

| 記憶體上限 | 3-10 GB | 128 MB/Isolate |

| CPU 時間 | 15 分鐘 | 5 分鐘（付費版） |

| 全球分布 | 區域性 | 300+ 節點自動分布 |

| 計費方式 | 按 wall time | 按 CPU time（I/O 不計）|

1.2 V8 Isolate 安全模型

每個請求在獨立的 V8 Isolate 中執行，Isolate 之間無法存取彼此的記憶體空間。同一進程可運行數千個 Isolate，共享 OS 層資源但保持完全隔離。這種設計使 Workers 的資源效率遠高於容器方案。

1.3 Runtime 環境

  • 實現 WinterCG 標準（非 Node.js API）
  • 支援 fetch()、Request、Response、URL、Headers、crypto
  • ctx.waitUntil(promise) — 背景任務（回應後繼續執行）
  • ctx.passThroughOnException() — 異常降級至 origin

二、四大儲存方案

2.1 Workers KV — 全球鍵值對

  • 一致性：最終一致（~60 秒全球傳播）
  • 適用場景：Feature flags、API 快取、靜態配置
  • 讀取延遲：<10ms（邊緣命中）
  • 寫入限制：每秒 1 次/key
  • 值大小：最大 25 MB

2.2 R2 — 物件儲存（S3 相容）

  • 零出口費用：與 S3 的核心差異化
  • API：env.MY_BUCKET.get(key)、put(key, body)、delete(key)
  • 最大物件：5 TB（單次上傳 5 GB，分片上傳至 5 TB）
  • 適用場景：媒體檔案（Podcast MP3、影片 MP4）、備份、大型資產
  • 與 Workers 綁定：wrangler.toml 設定 [[r2_buckets]]

// R2 上傳範例
export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    const key = url.pathname.slice(1);
    
    if (request.method === 'PUT') {
      // 驗證 Bearer Token
      const auth = request.headers.get('Authorization');
      if (auth !== `Bearer ${env.AUTH_SECRET}`) {
        return new Response('Unauthorized', { status: 401 });
      }
      await env.MY_BUCKET.put(key, request.body);
      return new Response(`Uploaded ${key}`);
    }
    
    if (request.method === 'GET') {
      const object = await env.MY_BUCKET.get(key);
      if (!object) return new Response('Not Found', { status: 404 });
      return new Response(object.body, {
        headers: { 'Content-Type': object.httpMetadata?.contentType || 'application/octet-stream' }
      });
    }
  }
};

2.3 D1 — Serverless SQLite

  • 無需連線池或 VPC 設定
  • 直接綁定：Worker 內直接操作 SQL
  • 遷移：Wrangler 管理 schema 演進
  • 適用場景：結構化元資料（Podcast 集數、使用者資料）

2.4 Durable Objects — 有狀態協調器

  • 強一致性：全球唯一實例，序列化請求處理
  • 儲存：10 GB 持久化（SQLite-backed）
  • WebSocket Hibernation：長連線不消耗 CPU
  • 適用場景：聊天室、協作編輯、速率限制、遊戲狀態

四、Podcast 媒體管線實踐（podcast-it 案例）

4.1 架構設計

實際案例 podcast-it 展示如何用 Workers + R2 + D1 建構 Podcast 自動化管線：


文章/部落格內容
  ↓ POST /api/episodes
Worker（OpenAI 生成旁白腳本）
  ↓
OpenAI TTS API（文字→語音）
  ↓
R2 儲存音訊（episode-slug.wav）
  ↓
D1 儲存元資料（slug, transcript, status, timestamps）
  ↓
GET /api/episodes/:slug（客戶端輪詢）

4.2 關鍵設計決策

  1. 儲存分離：R2 存音訊二進位、D1 存元資料 JSON — 避免 base64 編碼效能瓶頸
  2. 非同步處理：POST 立即回傳 202，背景 ctx.waitUntil() 繼續處理
  3. 狀態機：pending → processing → complete，客戶端輪詢 status

五、最佳實踐與常見陷阱

最佳實踐

  1. 並行 I/O：永遠用 Promise.all() 而非逐一 await
  2. 背景處理：ctx.waitUntil() 處理日誌、快取預熱、分析
  3. Cache API：在邊緣快取計算結果，減少重複執行
  4. Hono 框架：2026 標準路由框架，<14KB，Express 風格語法
  5. Wrangler 本地開發：完整模擬 KV/D1/R2/Durable Objects
  6. Secret 管理：wrangler secret put 管理敏感資訊，不放 wrangler.toml

常見陷阱

  1. 128 MB 記憶體限制：不適合大型資料處理，需分批或串流
  2. CPU 時間 ≠ wall time：I/O 等待不計入，但迴圈運算會快速耗盡額度
  3. KV 最終一致：寫入後全球讀取可能延遲 60 秒
  4. 6 個同時連線：fetch/KV/R2/Cache 共享連線池
  5. Vitest + D1 的相容性問題：watch mode 不支援，需手動建立 mock
  6. 大檔案上傳：單次 PUT 上限 5 GB，需分片上傳 API

六、與 daily-digest-prompt 專案的關聯

6.1 現有整合

本專案已建立 workers/ 目錄（Cloudflare Workers 應用），搭配：

  • tools/upload-podcast.ps1 — 上傳 Podcast 音訊至 R2
  • tools/delete-podcast-r2.ps1 — 刪除 R2 上的 Podcast 檔案
  • config/podcast.yaml — Podcast 管線配置
  • config/media-pipeline.yaml — 媒體管線配置

6.2 潛在應用

  1. Podcast RSS Worker：Worker 從 R2 讀取音訊 + D1 讀取元資料 → 動態生成 RSS feed
  2. Podcast Index API：workers/podcast-index/ 可作為 Podcast 目錄的邊緣 API
  3. 快取代理：Worker 快取外部 API（Todoist/ntfy/HN）回應，降低 origin 負載
  4. 靜態站點 CDN：R2 + Workers 提供知識庫 Vite 網站的全球 CDN

6.3 PowerShell 與 Workers 的互操作


# 從 PowerShell 上傳至 R2 via Worker API
