機械過濾 Deferred/Superseded ADR 防止 self_heal stale premise（Ponytail 階梯②）

研究日期：2026-06-30

主題分類：AI Agent 架構與治理 × 軟體工程方法論

透鏡組合：Sam Altman（主）/ Jensen Huang（主）/ Taleb（逆向）

執行摘要

本次研究聚焦「如何防止 AI agent（尤其是 self_heal）在引用架構決策記錄（ADR）時誤用已過期的 Deferred 或 Superseded 狀態決策」。研究發現 DDP 系統中 1,281 筆 Deferred ADR（佔全體 2,197 筆的 58.3%）是 stale premise 的最高風險源，但現有機制已具備三層防線：registry staleness check、necessity-first filtering（Ponytail 階梯②）、confirmation section fitness function。最大缺口是在 prompt-injection 時尚無自動排除 Deferred/Superseded ADR 的機制，需補齊 arch_decision_step_h.py pre-flight validate 步驟。

核心洞見（7 項）

  1. 58.3% ADR 處於 Deferred 狀態（1,281/2,197）——最大 stale premise 風險源；self_heal 若使用全量 registry = 78% 噪訊比
  2. Confirmation Section = 可執行 Pre-Flight Guardrail（已有機制但未接 agent RAG 過濾條件）
  3. Loop Contract 三段 mode（report_only → advise → enforce）阻止 Deferred 自動執行，但不阻止被引用為依據
  4. Ponytail 階梯② = 必要性前置過濾：Accepted = 可用；Deferred/Superseded/Wontfix = 只讀參考
  5. ADR-090 季度 slimming 退役標準（90 天零觸發）可移植為 ADR freshness metric
  6. 業界 OPA Policy-as-Code + CI 閘門 = 架構治理閉環（MADR 4.0、arXiv:2504.08207）
  7. 三閘門過濾架構：Registry Staleness Check → Necessity Validation → Confirmation Status Verification

建議行動 P0


STALE_STATUSES = {"Deferred", "Superseded", "Wontfix"}
valid_adrs = [a for a in adr_registry if a.get("status") not in STALE_STATUSES or a.get("review_due", "9999-99-99") > today_str]

加入 arch_decision_step_h.py pre-flight，一行消除 78% 噪訊比。

多透鏡洞察摘要

  • Sam Altman（主）：閘門 1 可一行實作，不等完美 OPA 基礎設施；反模式：用 LLM 判 stale
  • Jensen Huang（主）：adr-registry.json 是「GPU driver」，已存在，問題是 agent 未正確調用；建議做 infrastructure service tools/adr_preflight.py
  • Taleb（逆向）：1,281 筆 Deferred 1% 誤用 = 每天 12.8 筆錯誤決策；寧 false negative 不 false positive；fail-safe 方向應全量排除而非全量包含
