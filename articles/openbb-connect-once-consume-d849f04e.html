一、專案概述

OpenBB（OpenBB-finance/OpenBB）是一個開源金融資料平台，定位為「分析師、量化交易者和 AI Agent 的統一資料基礎設施」。其核心理念是 Connect Once, Consume Everywhere——將分散在 100+ 資料提供商的金融資料正規化後，透過 Python SDK、Web Workspace、Excel Add-in、MCP Server、REST API 等多種介面統一對外提供。

解決的問題

金融資料領域長期由 Bloomberg Terminal 壟斷（年費 $20,000+/人），中小型機構和個人分析師難以承擔。即使使用多個低成本 API，也面臨：

  • 資料碎片化：每個 API 回傳格式、欄位命名、時間戳格式都不同
  • 供應商鎖定：換一個資料源就要重寫整個資料層
  • 重複工程：每個下游應用（Python、Excel、Web）都要重新建立連接
  • AI 整合困難：LLM/Agent 需要結構化、一致的資料介面

OpenBB 的 Open Data Platform 正是解決這些問題的「中間層」。

二、技術架構：三層模組化設計

OpenBB Platform 採用經典的三層架構，核心設計決策是資料取得與資料消費完全解耦。

2.1 核心層（openbb-core）

基於 FastAPI + Pydantic 構建，負責三件事：

  1. 識別供應商：根據查詢參數選擇對應的資料提供商
  2. 發送請求：調用供應商的 Extract 方法取得原始資料
  3. 標準化回傳：將資料封裝在統一的 Pydantic Model 中

2.2 路由層（Routers）

中間層定義 API 端點，按金融領域分類：

  • openbb-equity：股票資料
  • openbb-economy：總經指標
  • openbb-crypto：加密貨幣
  • openbb-forex：外匯
  • openbb-options：選擇權

支援巢狀路由器（nested routers），100+ 個端點涵蓋主要金融資料類別。

2.3 供應商層（Providers）

最外層負責與外部資料源通訊。每個 Provider 實作 TET（Transform-Extract-Transform）模式：


[Input Transform] -> [Extract] -> [Output Transform]
      |                 |               |
  驗證參數          呼叫外部 API      標準化輸出
  套用預設值        取得原始資料       欄位統一
  參數轉換                            格式正規化

這個 TET 模式是 OpenBB 最精妙的設計之一：

  • Input Transform：將統一的查詢參數轉換為各供應商的特定格式
  • Extract：實際呼叫 REST API/資料庫/本地檔案取得資料
  • Output Transform：將各供應商回傳的不同格式標準化為統一 Schema

2.4 消費層（多介面統一）

同一套 API 透過 OpenAPI 規格自動生成多種客戶端：

| 介面 | 目標用戶 | 技術實現 |

|------|---------|----------|

| Python SDK | 量化分析師 | pip install openbb，動態函數生成 |

| OpenBB Workspace | 金融分析師 | Web 應用 + 視覺化儀表板 |

| Excel Add-in | 傳統分析師 | 試算表整合 |

| MCP Server | AI Agent | Model Context Protocol 整合 |

| REST API | 開發者 | FastAPI 本地伺服器 |

三、功能特色：差異化優勢

3.1 AI-Native 設計

OpenBB 是金融資料平台中最早全面擁抱 AI Agent 的專案：

OpenBB Copilot：內建的 AI 助手，支援自然語言查詢金融資料。使用者可以直接問「Show me Tesla's revenue trend」而非撰寫程式碼。

MCP 整合：透過 Model Context Protocol，任何 AI Agent（Claude、自訂 Agent）都可以直接調用 OpenBB 的資料端點，無需客製工程。MCP 伺服器將 OpenBB 的每個資料端點轉換為 AI 可調用的 Tool。

PydanticAI 適配器：官方提供 OpenBBAIAdapter 和 OpenBBAIEventStream，讓 PydanticAI Agent 可以原生對話 OpenBB。使用了延遲執行模式（Deferred Execution Pattern）：

  1. Agent 呼叫 widget tool，觸發 CallDeferred
  2. 適配器發送 get_widget_data SSE 事件
  3. 前端取得資料後發送新 POST 請求
  4. Agent 恢復執行，匹配 tool_call_id 取回結果

3.2 Local-First 安全模型

  • 所有 API 金鑰和憑證僅儲存在本地
  • 不收集任何遙測資料
  • 可完全離線使用（配合本地資料源）

3.3 擴展機制

  • Provider 插件系統：任何人都可以撰寫新的 Provider，遵循 TET 模式即可
  • 私有擴展：企業可開發內部供應商擴展，不需要貢獻到開源倉庫
  • Router 擴展：可新增自訂端點和金融指標計算

3.4 生產級功能

  • 自動快取（避免重複 API 呼叫）
  • 智慧速率限制
  • 完整錯誤處理與重試邏輯
  • 非同步支援（高吞吐量資料管線）
  • FastAPI 伺服器即時啟動

四、使用方式：快速上手

基礎安裝


pip install openbb

Python SDK 查詢範例


from openbb import obb

# 查詢股票歷史價格
data = obb.equity.price.historical(symbol="AAPL", provider="polygon")
print(data.to_df())

# 查詢總經指標
gdp = obb.economy.gdp.nominal(provider="oecd")

# 查詢加密貨幣
btc = obb.crypto.price.historical(symbol="BTC-USD", provider="polygon")

MCP 整合（AI Agent 使用）


claude mcp add --transport http openbb-docs-mcp [smithery.ai endpoint]

六、潛在應用：如何應用在自己的專案中

6.1 daily-digest-prompt 整合可能

  1. TET 模式借鑑：OpenBB 的 Transform-Extract-Transform 模式可以啟發 daily-digest-prompt 的 API 快取層設計。目前 api-cache Skill 做的是簡單的 TTL 快取，若加入 Input/Output Transform 正規化，可以更優雅地處理多 API 回應格式差異。
  1. MCP Server 靈感：OpenBB 將每個資料端點轉換為 MCP Tool 的做法，可以應用在知識庫 API 上——將 localhost:3000 的各個端點包裝成 MCP Tools，讓 Agent 更自然地調用。
  1. Provider 插件模式：OpenBB 的供應商擴展機制可以啟發 Skill 系統的進化方向。目前 Skill 是靜態的 Markdown 指引，若採用類似 Provider 的程式化模式，可以實現更精確的資料取得流程。
  1. Deferred Execution Pattern：延遲執行模式適合處理需要多輪資料取得的複雜任務，例如 Todoist 團隊模式中 Phase 2 到 Phase 3 的結果傳遞。

七、優缺點評估

優點

  1. 架構優雅：TET 模式 + 多介面統一消費是教科書級的解耦設計
  2. AI-First：業界少有的金融平台原生支援 MCP + AI Copilot
  3. 免費替代 $20K Bloomberg：對中小機構和個人分析師是顛覆性的
  4. Local-First 隱私：金融資料敏感，本地優先是正確的安全選擇
  5. 社群龐大：500+ Contributors，62K Stars 證明市場需求真實
  6. Python 生態：100% Python，與 pandas/numpy/sklearn 無縫整合
  7. 擴展性強：Provider 插件系統讓任何人都能接入自訂資料源

缺點

  1. 資料品質差距：與 Bloomberg 的即時、高品質機構數據仍有差距
  2. 效能問題：長時間跨度查詢速度慢，無內建快取優化
  3. 學習曲線：100+ 端點的 API 設計需要時間熟悉
  4. AGPL 授權限制：企業嵌入閉源產品需商業授權
  5. 資料延遲：部分免費資料源有 15 分鐘或更長的延遲
  6. 不適合高頻交易：延遲和吞吐量無法滿足 HFT 需求
