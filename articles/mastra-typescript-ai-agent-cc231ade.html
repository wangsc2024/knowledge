---

一、專案概述

Mastra 是由 Gatsby.js 原班人馬打造的 TypeScript 原生 AI Agent 框架，定位為「從原型到生產的一站式 Agent 開發平台」。框架涵蓋 Agent 定義、記憶管理、工作流編排、RAG 管線、評估系統與部署方案，以 TypeScript 型別安全 為核心設計原則。

核心定位

  • 非 LangChain 替代品：Mastra 不是「另一個 LLM 鏈式調用庫」，而是完整的 Agent 應用框架
  • TypeScript-First：所有 API 都有完整型別推導，IDE 自動補全覆蓋率極高
  • 宣告式 Agent 定義：Agent 行為透過結構化配置定義，非命令式程式碼
  • 多框架整合：原生支援 Next.js、React、Astro、Express、SvelteKit、Hono

---

二、架構核心：四大基元（Primitives）

Mastra 將所有 AI 操作抽象為四大基元：

| 基元 | 職責 | 類比 |

|------|------|------|

| Agents | 自主推理與工具調用，開放式任務解決 | 決策者 |

| Workflows | 確定性多步驟流程，圖式控制流 | 指揮棒 |

| RAG | 文件處理→分塊→嵌入→向量檢索 | 知識庫 |

| Evals | 品質與準確度評估，內建追蹤 | 品管員 |

2.1 Agent 系統設計

Agent 使用 LLM 推理目標、選擇工具、迭代執行直到產生最終答案。關鍵特性：

  • 工具綁定：Agent 可綁定任意 TypeScript 函數作為工具
  • 對話歷史：自動管理 conversation history
  • Human-in-the-Loop：Agent 可 suspend 並等待使用者輸入後 resume（利用 Storage 持久化執行狀態）

2.2 工作流引擎

圖式工作流引擎提供直觀的鏈式語法：


workflow
  .then(stepA)
  .branch([
    [condition1, stepB],
    [condition2, stepC]
  ])
  .parallel([stepD, stepE])

支援巢狀工作流（Nested Workflows）、觸發線（Tripwires）傳播、以及不定期暫停恢復。

三、核心創新：Observational Memory（觀測式記憶）

這是 Mastra 最具創新性的設計，也是與其他框架的最大差異化。

3.1 運作機制

傳統 RAG 記憶將完整對話歷史存入向量資料庫，Token 成本隨對話長度線性增長。Mastra 的 Observational Memory 採用三層壓縮架構：

  1. Raw Messages（原始訊息）→ 正常對話紀錄
  2. Observations（觀測日誌）→ 從對話中萃取的密集資訊摘要，壓縮比 5-40x
  3. Reflections（反思凝煉）→ 當 Observations 過長時，進一步凝煉為高階摘要

3.2 記憶範圍（Scoping）

  • Thread Scope：綁定特定對話線程，記住「這次對話」的脈絡
  • Resource Scope：綁定特定資源（使用者、專案），跨線程共享記憶

3.3 檢索策略

記憶檢索支援三種模式：

  • Recency（時間近因）：最近的記憶優先
  • Semantic Similarity（語義相似度）：與當前查詢語義最近的記憶
  • Thread-based（對話線程）：特定線程的完整記憶

3.4 為什麼這很重要？

| 維度 | 傳統 RAG 記憶 | Observational Memory |

|------|-------------|---------------------|

| Token 成本 | 線性增長 | 4-10x 節省 |

| 長對話表現 | 退化（上下文溢出） | 穩定（自動壓縮） |

| 跨對話連貫性 | 依賴外部狀態 | 內建 Resource Scope |

| 實作複雜度 | 需自建壓縮邏輯 | 框架內建 |

四、Agent Network（vNext）— 動態路由編排

4.1 核心概念

Agent Network 是 Mastra 的智慧編排層，讓 AI 系統動態決定如何使用 Agents、Workflows 和 Tools，而非遵循預定義的執行路徑。

4.2 兩種運作模式

模式 1：非結構化輸入 → 結構化執行

  • 功能：智慧路由器
  • 範例：使用者說「告訴我巴黎的事」→ 系統識別需要城市查詢工作流 → 自動填入 { city: "Paris" } → 執行

模式 2：非確定性多步驟工作流

  • 功能：動態執行計畫
  • 範例：「研究法國最大城市的天氣並寫成部落格」→ LLM 自動規劃：研究 Agent → 城市工作流 × 3 → 天氣 API → 生成文章

4.3 記憶驅動的路由決策

  • 對話感知：利用上下文選擇最適基元
  • 任務歷史：追蹤進度防止重複執行
  • 完成偵測：識別目標何時達成

4.4 API 介面

| 方法 | 用途 | 適用場景 |

|------|------|----------|

| .generate() | 單任務路由 | 聊天介面、一問一答 |

| .loop() | 多基元自動編排 | 複雜多步驟任務 |

4.5 與傳統工作流的關鍵差異

底層仍用 Mastra Workflows（非純工具調用），這保證了可靠性。清晰的基元描述（description）直接影響路由品質 — 更好的文件 = 更精確的路由。

---

五、Supervisor Pattern — 多 Agent 協調

2026-02-26 新增的一等公民模式：

  • 委派鉤子（Delegation Hooks）：控制任務如何分配給子 Agent
  • 迭代監控（Iteration Monitoring）：追蹤每個 Agent 的執行輪次
  • 完成度評分（Completion Scoring）：評估任務是否達標
  • 記憶隔離（Memory Isolation）：防止並行 Agent 間的狀態污染
  • 工具審批傳播（Tool Approval Propagation）：父 Agent 的工具權限向下傳遞
  • 上下文過濾（Context Filtering）：每個 Agent 只看到需要的資訊
  • bail 機制：允許提前退出無法完成的任務

六、對 Agent 系統設計的六大借鑑啟示

啟示 1：記憶壓縮比 RAG 更重要

Mastra 的 Observational Memory 證明：壓縮比檢索更關鍵。與其把所有歷史丟進向量資料庫，不如在寫入前先壓縮為觀測日誌。

可借鏡：Daily-Digest-Prompt 的 digest-memory.json 可引入「觀測層」— 每次寫入前先萃取核心觀測，而非追加完整紀錄。這能解決記憶檔案無限膨脹問題。

啟示 2：動態路由 vs 靜態路由的共存

Mastra 的 Agent Network 使用 LLM 動態路由，但底層仍用 Workflows 保證確定性。這種「頂層動態 + 底層確定」的雙層架構值得學習。

可借鏡：我們的 config/routing.yaml 靜態路由可加一層 LLM 動態路由（類似現有 tools/llm_classifier.py），但關鍵操作仍走靜態路徑。

啟示 3：Supervisor Pattern 的記憶隔離

Mastra 的多 Agent 協調特別強調記憶隔離 — 並行 Agent 不共享狀態，防止污染。我們的 Team 模式目前透過 results/*.json 做隔離，概念一致但可更精細化。

可借鏡：加入「上下文過濾」機制 — 每個子 Agent 的 prompt 只包含必要資訊，而非完整 context。

啟示 4：Human-in-the-Loop 的持久化暫停

Mastra 允許 Agent 或 Workflow 暫停並無限期等待使用者輸入，透過 Storage 持久化執行狀態。這對長時間運行的任務極為實用。

可借鏡：Todoist Agent 在需要使用者確認時可 suspend，而非自行猜測並繼續。

啟示 5：Eval 內建而非事後補做

Mastra 將 Evals 作為四大基元之一，與 Agent、Workflow、RAG 並列。這意味著品質評估是架構的一等公民，而非外掛。

可借鏡：我們的 done_cert 機制已有此概念，但可擴展為更系統化的 Eval Pipeline。

啟示 6：描述品質決定路由精度

Mastra Agent Network 明確指出：基元描述的品質直接影響路由決策的精度。更好的文件 = 更精確的 LLM 路由。

可借鏡：我們的 SKILL.md 的 description 欄位不只是文件，它是 LLM Router 的「輸入特徵」。應以「路由精度」為目標優化描述，而非以「人類可讀性」。
