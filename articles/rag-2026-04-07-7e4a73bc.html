優化摘要

二、效能瓶頸分析（5 維度）

2.1 Token 統計 FileLock 競爭（P1-中等）

現況：post_tool_logger.py 使用 10 秒 timeout 的 FileLock 保護 state/token-usage.json 的 read-modify-write 操作。團隊模式下最多 5 路 Phase 並行，存在鎖競爭風險。

量化影響：在 5 路並行場景下，最壞情況鎖等待可達 10 秒 × 4 = 40 秒累計阻塞，但實際觀測中尚未觸發（每筆寫入 <50ms）。

建議：改用指數退避（100ms → 200ms → 400ms → 800ms），搭配 jitter 隨機偏移，將最壞情況從固定 10 秒降至 ~2 秒。

2.2 JSONL 逐筆追加 I/O（P2-低）

現況：每次 Hook 觸發都直接 append 一筆 JSONL 記錄到磁碟，無批次緩衝。在 NTFS 上每次追加涉及檔案系統 metadata 更新。

量化影響：當前寫入密度（2–3 筆/秒）下影響可忽略。但若未來 Hook 數量增至 20+，峰值可能達 10 筆/秒，NTFS 上可能出現可觀測的延遲。

建議：引入 QueueHandler 非同步日誌寫入模式，主線程 append 到記憶體 queue，背景線程每 100ms 或 50 筆批次 flush。參考 Python 標準庫 logging.handlers.QueueHandler 實作。

2.3 hook-rules.yaml 規則膨脹（P2-低）

現況：54 條規則（20 bash + 8 write + 4 read + 33 benign patterns），版本已至 v8。新增規則無需改 Python（配置外部化），但每條規則的正則需逐一比對。

量化影響：正則快取（LRU 512 條）有效緩解，首次編譯後同一 session 零開銷。但 benign_output_patterns（33 條）在每次 PostToolUse 都要全掃。

建議：將 benign_output_patterns 編譯為單一聯合正則（pattern1|pattern2|...），O(33) 降為 O(1) 比對。

2.4 Pipeline 中介軟體數量（P3-觀察）

現況：hook_pipeline.py 的 DeerFlow 式短路機制設計精良，O(n) 執行且早期返回。目前僅 3 層中介軟體（trace → schema → allow），距離效能瓶頸甚遠。

業界對標：Microsoft Agent Framework 1.0 建議 Agent/Function/Chat 三層；業界生產環境常見 8 層（PII 偵測、prompt injection 攔截、快取、斷路器等）。本專案已在 pre_bash_guard/pre_write_guard 層實現安全篩查，pipeline 層僅作 schema 驗證，架構分工合理。

2.5 OpenTelemetry 整合成熟度（P3-觀察）

現況：10% 採樣率，靜默失敗，otel-config.yaml 配置完善。但目前僅 post_tool_logger 埋點，缺乏跨 Hook 的 trace_id 串連。

建議：在 hook_pipeline.py 的 log_trace_middleware 中注入 OTEL span，使同一次 Agent run 的所有 Hook 事件可透過 trace_id 串連查詢。

四、8 項優化建議與實施優先級

P0（立即可做，收益高）

建議 1：benign_output_patterns 聯合正則

  • 將 33 條 benign patterns 編譯為單一 re.compile("|".join(patterns))
  • 預期效果：PostToolUse 事件比對從 O(33) 降為 O(1)
  • 修改範圍：hook_utils.py 的正則快取初始化
  • 風險：低（純效能優化，行為不變）

建議 2：FileLock 指數退避

  • 替換固定 10 秒 timeout 為指數退避（100ms 起步、最大 2 秒、含 jitter）
  • 修改範圍：hook_utils.py FileLock 類
  • 預期效果：團隊模式下鎖等待時間降低 80%

P1（短期，收益中）

建議 3：非同步日誌寫入（QueueHandler）

  • 引入 Python 標準庫 QueueHandler + QueueListener
  • 主線程 append 到 queue，背景線程批次 flush
  • 修改範圍：post_tool_logger.py 日誌寫入路徑
  • 風險：中（需處理程序異常退出時的 flush）

建議 4：跨 Hook trace_id 串連

  • 在 hook_pipeline.py 的 trace middleware 中注入環境變數 HOOK_TRACE_ID
  • 所有下游 Hook 讀取並寫入 JSONL 的 hook_trace_id 欄位
  • 修改範圍：hook_pipeline.py + post_tool_logger.py
  • 預期效果：可觀測性提升，支援「某次 Agent run 的所有 Hook 事件」查詢

P2（中期，改善體驗）

建議 5：query_logs.py 流式讀取

  • 當 JSONL 檔超過 10MB 時，改用逐行 generator + 分區掃描
  • 避免全檔載入記憶體
  • 修改範圍：query_logs.py 主查詢函式

建議 6：OTEL 動態採樣

  • 依 AGENT_PHASE 環境變數動態調整採樣率
  • Phase 2（並行執行）採樣 30%，Phase 3（組裝）採樣 5%
  • 修改範圍：config/otel-config.yaml + post_tool_logger.py

P3（長期，架構演進）

建議 7：validate_config.py 模組拆分

  • 2,055 行為專案最大 Hook 模組，建議拆為 3 個子模組
  • validate_schema.py（schema 驗證）+ validate_auto_tasks.py（自動任務一致性）+ validate_config_core.py（核心配置）
  • 降低認知負擔，便於獨立測試

建議 8：Rust 替代評估（prek）

  • prek 是 2026 年新興的 Rust 實作 pre-commit 替代品
  • 已被 CPython/FastAPI/Ruff 採用，3–5x 加速
  • 本專案 Hook 模型不同於 git pre-commit，但可評估 Rust 級效能的日誌寫入器

五、業界對標（2025–2026 技術趨勢）

5.1 Agent 中介軟體三層模型（Microsoft Agent Framework 1.0）

  • Agent 層：安全篩查、審計日誌、內容政策
  • Function 層：參數驗證、執行計時、工具預算
  • Chat 層：PII 過濾、Token 限制、快取

本專案的 pre_bash_guard/pre_write_guard 對應 Agent 層，hook_pipeline 對應 Function 層，post_tool_logger 對應 Chat 層，架構分層已暗合此模型。

5.2 結構化日誌最佳實踐（structlog + JSON）

  • structlog 可降低 40–70% 日誌格式化開銷
  • dictConfig 集中配置，避免模組級散落設定
  • 本專案的 JSONL + checksum 鏈已超越基本 JSON 日誌，具備完整性驗證

5.3 洋蔥式中介軟體管線


Request → [安全] → [快取] → [日誌] → [Tool] → [日誌] → [快取] → [安全] → Response

每層可獨立短路，before/after 對稱執行。本專案的 PreToolUse → PostToolUse 雙向 Hook 已實現此對稱模式。
