研究類型：AI 技術研究（ai_research）

一、技術概述

AI Agent 記憶架構（Memory Architecture）是 2026 年 AI Agent 開發的核心基礎設施議題。隨著 Agent 從單輪對話工具進化為可持久運行的自主系統，如何讓 Agent「記住」跨 Session 的上下文、使用者偏好、任務歷史成為關鍵挑戰。ICLR 2026 Workshop「MemAgents: Memory for LLM-Based Agentic Systems」標誌著此領域已進入學術主流。

傳統 LLM 受限於 Context Window，即使擴展至 1M tokens（如 Gemini 3.1 Pro），仍無法涵蓋所有歷史互動。記憶架構的核心思路是將資訊分層管理——類似作業系統的 RAM 與磁碟層級——讓 Agent 按需檢索、動態更新、自主管理記憶。

---

二、記憶類型分類

2.1 四種核心記憶類型

| 記憶類型 | 定義 | 類比 | 典型用途 |

|----------|------|------|----------|

| 工作記憶（Working Memory） | 當前 Session 的即時上下文 | RAM | 對話歷史、當前任務狀態 |

| 情節記憶（Episodic Memory） | 特定事件與互動的時序記錄 | 日記 | 「上次使用者提到偏好深色主題」 |

| 語義記憶（Semantic Memory） | 領域知識與事實性資訊 | 百科全書 | 規則、政策、技術文件 |

| 程序記憶（Procedural Memory） | 行為模式與操作步驟 | 肌肉記憶 | 常用工作流、工具使用偏好 |

2.2 偏好記憶（Preference Memory）

AWS AgentCore 等平台額外定義「偏好記憶」，專門追蹤個別使用者的個人化偏好（如語言、格式、通知頻率），與語義記憶的「通用事實」區分。

2.3 總結記憶（Summarization Memory）

將冗長的互動歷史壓縮為精煉摘要，解決 Context Window 限制。Mem0 與 Letta 均支援自動摘要，將過期的工作記憶降級為壓縮的情節記憶。

三、主要框架與架構比較

3.1 Letta（原 MemGPT）— LLM-as-OS 典範

核心理念：將 LLM 視為作業系統，Agent 自主管理記憶的讀寫、換入換出。

架構設計：

  • In-Context Memory：系統指令 + 可讀寫記憶區塊（core_memory） + 當前對話
  • Out-of-Context Memory：長期對話歷史（recall_memory）+ 外部知識（archival_memory）
  • 記憶工具：core_memory_append、core_memory_replace、recall_search、archival_insert
  • 特色：記憶為一級公民（first-class citizen），開發者可直接檢視和編輯記憶區塊

效能：LoCoMo 基準達 74.0%（GPT-4o mini），超越 Mem0 的 68.5%

3.2 Mem0 — 生產就緒的記憶層

核心理念：提供可擴展的記憶中心架構，動態從對話中萃取、整合、檢索顯著資訊。

架構設計：

  • 記憶萃取管線：對話 → NLU 解析 → 事實/偏好/情節抽取 → 向量化儲存
  • Graph Memory 變體：利用圖結構捕捉實體間的複雜關係（如「使用者 A → 負責 → 專案 B → 使用 → 技術 C」）
  • 記憶整合：自動偵測衝突記憶並合併（如偏好更新時覆蓋舊值）
  • 多記憶類型：工作記憶、事實記憶、情節記憶、語義記憶

效能：研究顯示持久記憶系統可提升 26% 回應準確度

3.3 Zep — 時序情節記憶專家

  • 強調情節與時序記憶，將互動結構化為有意義的序列（非扁平日誌）
  • 接近人類對話記憶的運作方式
  • 適合需要「記住對話脈絡」的場景

3.4 AWS AgentCore Long-Term Memory

  • 三種記憶類型：語義記憶、情節記憶、偏好記憶
  • 自動記憶萃取與檢索
  • 與 AWS Bedrock Agent 深度整合
  • 企業級可擴展性與合規性

3.5 框架對比摘要

| 框架 | 核心方法 | 記憶類型 | LoCoMo 分數 | 部署模式 | 特色 |

|------|----------|----------|-------------|----------|------|

| Letta | LLM-as-OS | In/Out-Context | 74.0% | 自架/雲端 | Agent 自主管理記憶 |

| Mem0 | 記憶中心 | 4 種 + Graph | 68.5% | SaaS/自架 | 最快投產路徑 |

| Zep | 時序情節 | 情節/語義 | — | SaaS/自架 | 時序脈絡最強 |

| AgentCore | 雲端託管 | 3 種 | — | AWS 整合 | 企業級擴展 |

四、Graph Memory — 關係型記憶的突破

4.1 為什麼需要 Graph Memory

傳統向量資料庫以「點」儲存記憶，缺乏實體間的關係建模能力。Graph Memory 以知識圖譜形式儲存，能回答「使用者的專案用了哪些技術？」這類多跳推理問題。

4.2 架構設計


對話輸入 → 實體抽取（NER） → 關係萃取（Relation Extraction）
    → 圖節點/邊建立 → 圖資料庫（Neo4j/NebulaGraph）
    → 圖查詢（Cypher/GQL） → 上下文注入 → LLM 生成

4.3 應用場景

  • 客服 Agent：記住客戶的所有產品、問題歷史、偏好的關聯
  • 研究 Agent：追蹤論文、作者、概念之間的引用與延伸關係
  • 專案管理 Agent：維護團隊、任務、里程碑之間的依賴圖

五、LoCoMo 基準評測

5.1 基準設計

LoCoMo（Long Conversation Memory）是評估 Agent 長對話記憶能力的 QA 基準。每個測試樣本包含兩位虛構說話者的帶時間戳對話，要求系統回答關於說話者或對話事實的問題。

5.2 關鍵發現

  1. Agent 能力比檢索機制更重要：「Agent 是否『記住』某事，取決於它能否在需要時成功檢索到正確資訊」
  2. 熟悉工具勝過專用工具：使用檔案系統操作（grep、search_files）的 Letta Agent 表現優於使用專用記憶工具的系統
  3. 多跳查詢能力關鍵：允許 Agent 自主生成多次查詢並精煉搜尋，效果遠勝單次檢索
  4. 簡單方法有效：「專門為單跳檢索設計的記憶工具，不如讓 Agent 自主搜尋資料有效」

六、上下文工程（Context Engineering）最佳實踐

6.1 現代 Agent 循環


觸發 → 規劃 → 工具呼叫 → 記憶存取 → 輸出

取代傳統線性「感知→決策→行動」流程，Agent 在規劃階段即可拉取歷史記憶，執行中動態調整。

6.2 設計原則

  1. 記憶分層：工作記憶（快但有限）→ 短期記憶（Session 級）→ 長期記憶（持久化）
  2. 按需檢索：不預載所有記憶，用語義搜尋按需拉取相關片段
  3. 自動整合：定期壓縮、去重、更新過時記憶
  4. 透明可檢視：記憶內容對開發者可見可編輯，便於除錯
  5. 多 Agent 共享：團隊模式下的 Agent 可共享記憶庫，避免上下文斷裂

6.3 反模式

  • 將所有歷史塞入 Context Window（超限後崩潰）
  • 僅依賴向量搜尋而忽略結構化記憶（缺乏關係推理）
  • 記憶只寫不讀（寫入但從不實際使用）
  • 缺乏記憶過期與清理機制（資訊膨脹）

七、2026 前沿趨勢

7.1 ICLR 2026 MemAgents Workshop

三大研究軸線：

  1. 記憶架構與表徵：情節、語義、工作記憶及其與外部儲存的介面
  2. 系統與評估：資料結構、檢索/整合管線、非 i.i.d. 長期能力基準
  3. 應用整合：記憶在多 Agent 協作、個人化、持續學習中的角色

7.2 Gartner 預測

到 2027 年，組織使用小型任務專用 AI 模型的頻率將是通用 LLM 的 3 倍。記憶架構讓小模型透過外部記憶補償參數量不足，成為 Edge AI 的關鍵賦能技術。

7.3 新興方向

  • Neuro-symbolic Memory：結合神經網路檢索與符號推理的混合記憶系統
  • Memory Consolidation：模仿人類睡眠期間的記憶整合機制，在 Agent 空閒時自動重組記憶
  • Federated Memory：跨裝置、跨平台的分散式記憶共享
  • Continual Learning：Agent 透過持久記憶實現增量學習，無需重新訓練

---

八、對 Daily Digest Agent 系統的啟示

本專案的 digest-memory.json 跨次記憶機制屬於簡化的「語義記憶 + 情節記憶」混合：

  • 追蹤連續天數（情節記憶）
  • 記錄待辦完成率（語義記憶）
  • 跨 Session 持久化（長期記憶）

可借鏡的優化方向：

  1. 引入圖結構追蹤任務間的關聯關係
  2. 記憶自動摘要與壓縮（避免 JSON 膨脹）
  3. 記憶品質評分機制（清理低價值記憶）
