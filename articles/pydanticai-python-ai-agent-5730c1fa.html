一、專案概述

PydanticAI（pydantic/pydantic-ai）是由 Pydantic 團隊開發的 Python AI Agent 框架，目標是將 FastAPI 的開發體驗帶入生成式 AI 應用領域。Pydantic 是 Python 生態系統中最核心的驗證庫之一，被 OpenAI SDK、Anthropic SDK、Google ADK、LangChain、LlamaIndex、AutoGPT、Transformers、CrewAI 等主流 AI 框架依賴。PydanticAI 正是站在這個巨人肩膀上，將型別安全與資料驗證的哲學延伸到 Agent 開發。

定位

PydanticAI 明確拒絕「炒作驅動」的 GenAI 開發模式，強調「構建 GenAI 應用仍然是工程問題」。它瞄準的是需要將 AI Agent 部署到生產環境的工程師，而非快速原型或展示型 Demo。

二、核心技術架構分析

2.1 泛型型別化 Agent 系統

PydanticAI 的核心是一個泛型 Agent 類別，使用 Python 型別參數化：


Agent[DependencyType, OutputType]

例如：Agent[SupportDependencies, SupportOutput]。這個設計讓 IDE 和型別檢查器能在編譯期捕捉錯誤，而非等到執行期才發現。在 v1 開發過程中，型別安全機制捕捉了 23 個本來會進入生產的 bug。

2.2 依賴注入（Dependency Injection）

PydanticAI 的依賴注入系統是其最具特色的設計之一。透過 RunContext[DepsType] 參數，開發者可以將上下文資訊（如資料庫連線、API 客戶端、用戶資訊）傳遞給系統提示、工具函式和輸出驗證器，且全程型別安全。


@agent.tool
async def get_user_data(ctx: RunContext[AppDeps], user_id: str) -> str:
    db = ctx.deps.database  # 型別安全存取
    return await db.query(user_id)

設計哲學：不發明神秘的「魔法」機制，而是延續 Python 既有的最佳實踐（如 FastAPI 的 Depends 模式），讓依賴「可理解、易測試、便於部署到生產環境」。

2.3 結構化輸出與自動修正

PydanticAI 利用 Pydantic 模型生成 JSON Schema，告知 LLM 應以何種格式返回資料，並在執行結束時進行驗證。若驗證失敗，框架會自動觸發「反思與自我修正」（reflection and self-correction）——將錯誤資訊回傳給 LLM，要求重新輸出正確格式的結果。


class FlightResult(BaseModel):
    airline: str
    flight_number: str
    departure: datetime
    price: float

agent = Agent('openai:gpt-4o', output_type=FlightResult)

2.4 工具系統（Tool Calling）

使用 @agent.tool 裝飾器註冊工具函式。工具的 docstring 自動成為 LLM 可見的工具描述，參數 docstring 生成 schema 細節。支援：

  • Human-in-the-Loop 工具審批：可標記特定工具呼叫需要人工核准才能執行
  • 延遲工具（Deferred Tools）：非同步執行的工具呼叫
  • MCP 整合：透過 Model Context Protocol 存取外部工具和資料源
  • 內建工具庫：常用工具的預建模組

2.5 Graph 工作流支援（Beta）

PydanticAI 提供型別提示驅動的 Graph 定義能力，用於複雜應用中標準控制流會退化為義大利麵式程式碼的場景。支援步驟（steps）、聚合/歸約（joins/reducers）、決策（decisions）和並行執行（parallel execution）。

2.6 持久化執行（Durable Execution）

與 Temporal、DBOS、Prefect 等持久化工作流引擎整合，讓 Agent 能夠：

  • 在 API 暫時性故障和應用程式錯誤後恢復進度
  • 從崩潰的中間點精確恢復（而非從頭開始）
  • 處理長時間運行的非同步工作流
  • 支援 human-in-the-loop 流程的生產級可靠性

TemporalAgent 包裝器自動將所有 I/O 操作（模型請求、工具呼叫、MCP 伺服器通訊）卸載到 Temporal Activities，同時保持編排邏輯在確定性工作流中執行。

三、模型與協議支援

3.1 模型無關設計

PydanticAI 支援幾乎所有主流 LLM 提供者：

| 類別 | 支援清單 |

|------|----------|

| 模型 | OpenAI、Anthropic、Gemini、DeepSeek、Grok、Cohere、Mistral、Perplexity |

| 雲端平台 | Azure AI Foundry、Amazon Bedrock、Google Vertex AI |

| 自託管 | Ollama、LiteLLM |

| 推論服務 | Groq、OpenRouter、Together AI、Fireworks AI、Cerebras、SambaNova |

| 其他 | Hugging Face、GitHub、Heroku、Vercel、Nebius、OVHcloud、Alibaba Cloud、Outlines |

模型透過字串標識符設定（如 'anthropic:claude-sonnet-4-5-20250929'），或透過 PydanticAI Gateway 統一介面存取。

3.2 MCP 整合

PydanticAI 整合 Model Context Protocol（MCP），使 Agent 能存取外部工具和資料源，擴展 Agent 的能力邊界。

3.3 A2A 協議支援

PydanticAI 支援 Google 的 Agent2Agent（A2A）協議——一個開放標準，讓不同框架和供應商構建的 AI Agent 能夠互相通訊和互操作。Pydantic 團隊建立了 FastA2A 庫，使 A2A 協議在 Python 中的實作更簡便。

四、可觀測性與品質保證

4.1 OpenTelemetry 原生整合

PydanticAI 發射符合語義慣例（semantic convention）的 OpenTelemetry 資料，避免供應商鎖定。透過 Pydantic Logfire 平台提供：

  • 即時除錯與追蹤
  • 基於評估（Evals）的效能監控
  • 行為追蹤與成本監控
  • Agent 決策路徑可視化

4.2 Pydantic Evals

內建全面的評估框架，支援系統性測試和評估 Agent 系統的效能與準確度，並透過 Logfire 監控效能隨時間的變化。

4.3 品質保證標準

  • 100% 測試覆蓋率
  • 文件中的範例全部有單元測試：防止過時範例浪費開發者時間
  • API 穩定性承諾：v1 至少 6 個月不破壞 API，v2 發布後仍持續 v1 安全修復至少 6 個月

五、可借鑑的技術項目（對 Daily-Digest-Prompt 系統）

5.1 型別安全的依賴注入模式（高優先級）

PydanticAI 做法：透過 RunContext[DepsType] 將依賴傳遞給工具和系統提示，全程型別安全。

Daily-Digest 借鑑：目前系統的 Hooks 和 Skills 之間的資料傳遞依賴隱式約定（如 results/*.json 的 JSON 結構）。可以借鑑 PydanticAI 的模式，為 Hook 和 Skill 之間建立顯式的資料契約（Pydantic Schema），在寫入前驗證結構正確性。

具體實作方向：

  • 為 results/*.json 定義 Pydantic Schema（如 PhaseResult、AutoTaskResult）
  • Hook 中使用 pydantic.TypeAdapter 驗證輸出
  • 減少因 JSON 結構不一致導致的 Phase 3 組裝失敗

5.2 結構化輸出驗證與自動重試（高優先級）

PydanticAI 做法：LLM 輸出不符合 Schema 時，自動將驗證錯誤反饋給 LLM 要求修正。

Daily-Digest 借鑑：目前 Todoist 團隊模式中，Phase 1 的 LLM 輸出格式偶爾不符預期（如連字號 vs 底線 key），導致 Phase 2/3 失敗。可以在 Phase 1 輸出後加入結構化驗證層，若格式不正確則自動要求 LLM 重新輸出。

具體實作方向：

  • todoist-query.md 的 plan_key 加入正則驗證（僅允許底線，禁止連字號）
  • run-todoist-agent-team.ps1 的 Phase 1 結果讀取後加入 JSON Schema 驗證
  • 驗證失敗自動重試一次（已有重試機制，只需加入驗證條件）

5.3 持久化執行模式（中優先級）

PydanticAI 做法：透過 Temporal 整合，Agent 崩潰後能從中間點恢復，而非從頭開始。

Daily-Digest 借鑑：目前系統的 Phase 失敗重試是完整重跑整個 Phase。可以在 scheduler-state.json 中記錄更細粒度的進度檢查點（checkpoint），Phase 2 的 N 個並行任務中若只有 1 個失敗，只重試該任務而非全部。

具體實作方向：

  • Phase 2 結果記錄至 state/phase2-checkpoint.json
  • 重試時讀取 checkpoint，跳過已成功的任務
  • 預估可節省 30-50% 的重試時間

5.4 Human-in-the-Loop 工具審批（中優先級）

PydanticAI 做法：特定工具呼叫可標記為需要人工核准，支援條件式審批（基於參數、對話歷史或用戶偏好）。

Daily-Digest 借鑑：目前 Hook 系統是「全攔截或全放行」的二元決策。可以借鑑 PydanticAI 的分級審批模式，對高風險操作（如刪除大量檔案、修改核心配置）推送 ntfy 通知等待用戶確認，而非直接攔截。

5.5 OpenTelemetry 可觀測性標準化（低優先級）

PydanticAI 做法：發射符合語義慣例的 OpenTelemetry 資料。

Daily-Digest 借鑑：目前的 JSONL 日誌是自定義格式。長期可考慮採用 OTLP（OpenTelemetry Line Protocol）格式，使日誌能接入 Grafana、Jaeger 等標準觀測工具。但短期內自定義 JSONL 已滿足需求，此項為低優先級。

核心洞察

PydanticAI 和 LangChain 代表了 AI Agent 開發的兩種根本不同的哲學：

  • LangChain 用「管線」思維，將文件載入、分割、嵌入、檢索、提示鏈、多步驟工作流串聯成大型系統
  • PydanticAI 用「型別與函式」思維，將 AI 視為一般 Python 服務——定義 Schema、描述需求、讓模型填入

當核心挑戰是可靠性（結構化預測輸出、填表、做決策、呼叫工具）時，PydanticAI 更自然。當核心挑戰是編排（構建完整 AI 工作流）時，LangChain 更合適。

七、與 Daily-Digest-Prompt 系統的對比分析

| 面向 | Daily-Digest-Prompt | PydanticAI | 差異分析 |

|------|-------------------|-----------|----------|

| Agent 定義 | Markdown Prompt + YAML 配置 | Python 型別化 Agent 類別 | Daily-Digest 採文件驅動，LLM 更易理解；PydanticAI 更適合程式化場景 |

| 依賴管理 | 隱式（results/*.json 約定） | 顯式型別化 DI | PydanticAI 更嚴格，Daily-Digest 更靈活但易出錯 |

| 輸出驗證 | 品質閘門（quality-gate.md） | Pydantic Schema + 自動重試 | 可借鑑 Schema 驗證概念強化品質閘門 |

| 持久化 | PowerShell 重試機制 | Temporal 工作流恢復 | Daily-Digest 整體重試，PydanticAI 精確恢復 |

| 可觀測性 | JSONL 自定義日誌 | OpenTelemetry 標準 | 自定義已足夠，長期可考慮標準化 |

| 工具系統 | Hook 攔截 + Skill 指引 | 型別化工具註冊 | 兩者各有優勢，Hook 更安全，工具系統更靈活 |

| 多 Agent | Team 模式 + Phase 並行 | A2A 協議互操作 | Daily-Digest 的 Phase 模式更適合批次處理 |

八、結論與建議

8.1 核心洞察

PydanticAI 的成功驗證了一個重要趨勢：AI Agent 框架正從「什麼都有的工具箱」走向「做好一件事的專家工具」。與 LangChain 的 1,000+ 整合相比，PydanticAI 選擇在型別安全、結構化輸出和生產可靠性三個面向做到極致，而非追求功能覆蓋的廣度。

8.2 對本專案的三大啟示

  1. Schema-First 設計：為 Phase 間的資料傳遞定義正式 Schema，將「格式不一致」類型的 bug 從執行期移到建構期
  2. 細粒度恢復：將粗粒度的「整體重試」升級為「檢查點恢復」，提升系統效率
  3. 驗證 + 自動修正閉環：在品質閘門中加入「驗證失敗 → 反饋 → 重新生成」的自動修正機制

8.3 不建議直接採用的部分

  • PydanticAI 作為框架替代：Daily-Digest 的文件驅動架構（Prompt + YAML）更適合 CLI Agent 場景，不需要改用程式化 Agent 框架
  • OpenTelemetry 全面遷移：現階段自定義 JSONL 已滿足需求，遷移成本不值得
  • A2A 協議：Daily-Digest 的 Phase 並行模式已經夠用，不需要引入 Agent 互操作協議
