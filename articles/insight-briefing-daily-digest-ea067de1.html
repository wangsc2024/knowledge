Daily-Digest-Prompt 系統韌性提升：從失敗模式分析到自適應降級的綜合策略

日期：2026-04-10

研究類型：跨領域系統洞察

資料來源：system-insight.json + improvement-backlog.json + 知識庫既有研究

---

摘要

Daily-Digest-Prompt 系統當前面臨多維度韌性挑戰：排程成功率僅 65%（目標 90%）、9 個高失敗時段呈明顯聚集、快取命中率 30.2%（目標 40%）、9/11 任務結果檔缺失。本簡報整合系統洞察數據與既有診斷報告，識別五大核心問題並提出三層韌性策略：即時修復層（診斷函數實作）、適應層（時段感知降級）、演化層（公平性調度優化）。預期可將成功率提升至 85%+ 並建立長期自適應能力。

---

關鍵洞見

1. 失敗模式呈現明顯時段聚集（新發現）

數據：

  • 高失敗時段：1, 4, 5, 6, 7, 8, 11, 21, 22（9 個時段，佔全日 37.5%）
  • 早晨 4-8 時連續 5 小時失敗率顯著高於其他時段
  • 成功率從理想的 90% 降至實際的 65%（差距 25%）

洞察：

失敗時段的聚集性暗示非隨機因素主導，可能原因包括：

  • 後端模型在特定時段的健康狀態變化（例如 API rate limit 重置時間）
  • 特定時段的任務類型分布（例如早晨集中執行研究類任務）
  • Timeout 設定與實際執行時間的不匹配（特定任務在該時段耗時更長）

證據鏈：

  • system-insight.json high_failure_hours: [1, 4, 5, 6, 7, 8, 11, 21, 22]
  • system-insight.json alerts.daily_success_rate: 0.65 < 0.9

建議行動：

  1. 短期：執行 query-logs.ps1 -Mode errors 並按小時分組，識別失敗任務類型
  2. 中期：實施時段感知的 timeout 動態調整（4-8 時 +30%）
  3. 長期：建立時段健康度監控，失敗率 >40% 時自動切換 backend

5. 系統缺少跨時段的韌性降級機制（架構級缺口）

數據：

  • 9 個高失敗時段仍使用相同的 timeout 與 backend 配置
  • 無時段感知的自適應機制
  • timeouts.yaml 為靜態配置，未依時段或歷史失敗率動態調整

洞察：

當前系統採用「一刀切」配置：所有時段使用相同 timeout、相同 backend 優先級、相同重試策略。但失敗時段的聚集性表明環境變異性（例如 API健康度、網路延遲），需要上下文感知的降級策略。

理論支撐：

  • Harness Engineering 原則：系統應具備「漸進式降級」與「自適應調整」能力
  • 類比 Circuit Breaker 模式：連續失敗應觸發快速失敗或切換備選路徑

建議行動：

  1. 時段感知 timeout：在 timeouts.yaml 新增 hour_multipliers，4-8 時 timeout × 1.3
  2. Backend fallback 順序調整：失敗時段優先使用穩定性更高的 backend（可能犧牲成本）
  3. 失敗率斷路器：某時段連續 3 次失敗率 >50%，該時段後續任務自動跳過或延遲至下一時段
