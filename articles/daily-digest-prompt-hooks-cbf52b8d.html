Daily Digest Prompt 專案優化報告

優化背景

透過 4 路並行深度分析（Hooks 程式碼品質、配置檔一致性、測試覆蓋率缺口、PowerShell 腳本品質），識別 4 個高優先級優化項目並全數實施完成。

優化項目

1. routing.yaml JSON Schema 驗證修正

  • 問題：config/schemas/routing.schema.json 的 labelMapping 定義設了 additionalProperties: false，但 ^創意 和 ^遊戲研究 映射加了 description 欄位，導致 test_all_configs_valid 測試失敗
  • 修正：在 labelMapping 的 properties 中新增 description 為可選屬性（type: string, minLength: 1）
  • 影響：1 個測試從失敗轉為通過

2. API_SOURCE_PATTERNS 重複定義消除

  • 問題：hooks/post_tool_logger.py（第 30-37 行）和 hooks/agent_guardian.py（第 174-181 行）各自定義了完全相同的 API_SOURCE_PATTERNS 字典（6 個 API 來源映射），違反 DRY 原則
  • 修正：將 API_SOURCE_PATTERNS 移至 hooks/hook_utils.py 共用模組，兩個消費者改為 from hook_utils import API_SOURCE_PATTERNS
  • 防禦性設計：agent_guardian.py 使用 try/except ImportError 保留 fallback 定義，確保獨立執行時仍可運作
  • 影響：未來只需在 hook_utils.py 一處更新 API 來源映射

3. 正則快取機制（Hot Path 效能提升）

  • 問題：pre_bash_guard.py 和 pre_read_guard.py 在每次工具呼叫時重新編譯所有正則表達式。以 bash guard 為例，13 條規則每條含 1-7 個 pattern，每次呼叫最多編譯 40+ 個正則
  • 修正：在 hooks/hook_utils.py 新增 get_compiled_regex(pattern, flags) 函數，使用模組層級 _compiled_regex_cache 字典快取已編譯的正則
  • 整合：pre_bash_guard.py 和 pre_read_guard.py 的 hot path 改用 get_compiled_regex(p, re_flags).search(command) 替代原本的 re.search(p, command, re_flags)
  • 影響：首次呼叫時編譯並快取，後續呼叫直接使用快取，在高頻排程場景（每小時 2 次 Todoist + 3 次每日摘要）顯著減少 CPU 開銷

4. CJK Guard 測試套件新增

  • 問題：hooks/cjk_guard.py（273 行）是偵測日文漢字混入繁體中文的重要守護工具，但完全沒有測試覆蓋
  • 修正：新增 tests/hooks/test_cjk_guard.py，包含 35 個測試案例，分 5 個類別

- TestCJKCorrections（7 個）：映射表完整性、CORRECTIONS 過濾邏輯、重複 key 合併

- TestDetectIssues（11 個）：偵測精確度、行/列定位、context 擷取

- TestFixText（7 個）：修正邏輯、計數正確性、混合文字處理

- TestConstants（2 個）：SCAN_EXTENSIONS、EXCLUDED_FILES 常量

- TestPostFix（8 個）：stdin JSON 處理、工具類型過濾、檔案類型過濾

  • 發現：CJK_CORRECTIONS 中 (0x8FBA, 0x908A) 重複出現兩次（第 38、41 行），但因 dict 覆蓋不影響功能

測試結果

  • 修改前：461 個測試，460 通過 / 1 失敗
  • 修改後：496 個測試，496 通過 / 0 失敗（+35 新增，+1 修復）

架構洞察

  • 正則快取適用場景：任何 PreToolUse hook 在每次工具呼叫時執行，高頻場景（每日 44 次 Agent 執行，每次 20-100 個工具呼叫）下快取效益明顯
  • 共用模組模式：hook_utils.py 已從最初的 5 個函數（2026-02-15）成長為包含 INJECTION_PATTERNS、API_SOURCE_PATTERNS、正則快取的核心共用層
  • Schema 寬鬆原則：配置檔 Schema 應預留可選擴展欄位（如 description），避免對人類可讀性註釋過度限制
