AI 驅動 CI/CD 智能化：Flaky Test 偵測、構建失敗預測與自癒管線（2026）

技術概述

2026 年 CI/CD 正從「失敗後人工排查」轉向 預測式 + 自癒式管線：以歷史測試執行資料訓練 ML 模型偵測 flaky test（同一代碼下間歇性 pass/fail），並以 LLM Agent 對失敗 log 做 RCA 與自動修復 PR。Google 報告 flaky test 消耗約 16% CI 算力；ML 偵測準確率 85–92%（閾值法僅 60–70%）。Frontiers 2026 提出 SAPAL 框架與 PHI/TSS/FPC 三項可靠性指標，將管線 KPI 從吞吐轉向決策品質。

核心方法與工具

1. Flaky Test 預測（特徵工程 + 梯度提升）

| 訊號 | 說明 |

|------|------|

| pass-rate variance | 30-run 滾動窗口內通過率變異 |

| execution-time std | 執行時間標準差（競態/超時指標）|

| retry pass pattern | 立即重跑即通過 |

| async/network density | 測試碼中 async/HTTP 呼叫密度 |

| co-failure correlation | 與其他測試共失模式 |

模型：LightGBM / XGBoost 分類器；進階方案 DeepFlaky 結合 CodeBERT 語意嵌入。

平台：Harness CI（自動 quarantine）、Semaphore（統計不穩定層）、Desplega.ai（Playwright Reporter 整合）。

2. 構建失敗預測

  • 輸入：commit 變更檔案、歷史 build log、依賴圖、執行時長
  • 輸出：PR 級失敗機率 → 動態測試選擇（僅跑高風險用例，案例報告 40% build 時間縮短）
  • 指標：Pipeline Health Index (PHI)、Failure Prediction Confidence (FPC)

3. LLM 自癒管線

| 框架 | 能力 | 量化 |

|------|------|------|

| LogSage (ByteDance) | log 預處理 + RAG 修復 + tool-calling | RCA F1 +38pp；1.07M 執行部署 |

| GitHub Copilot SDK | JSON-RPC agent loop、自訂 tools、HITL | 2026-01 GA |

| Semaphore + Copilot Agent | CI 失敗 → 自動 PR | 雲端隔離修復 |

實際工作流程（step-by-step）

Phase A：Flaky 偵測管線（MLOps 最小版）

  1. 收集：CI reporter 輸出 JUnit/JSON（test name, status, duration, retry, commit SHA）
  2. 特徵：每測試 30-run 滾動聚合 → CSV feature matrix
  3. 訓練：LightGBM，flaky=1 若同 SHA 下 pass/fail 交替
  4. 推論：自訂 Playwright reporter 於 onTestEnd 附加 flakiness_score
  5. 政策：score > 0.7 → quarantine（不阻 merge）；score 0.4–0.7 → 標記 review
  6. 重訓：cron 每日夜間重跑 pipeline

Phase B：自癒 CI（GitHub Actions + Copilot SDK）

  1. CI job 失敗 → workflow_run 觸發 heal workflow
  2. Agent 讀取 log + diff + 相關源碼
  3. 生成修復 patch（path-scope 限制 + 測試必須通過）
  4. HITL：PR 需人工 approve 後 merge
  5. 記錄 fix 至 RAG 知識庫供 LogSage 式重用

程式碼/配置範例

Playwright Flaky Reporter（TypeScript 骨架）


import type { Reporter, TestCase, TestResult } from '@playwright/test/reporter';
import flakyScores from './flaky-scores.json'; // 每日 ML 產出

export default class FlakyReporter implements Reporter {
  onTestEnd(test: TestCase, result: TestResult) {
    const score = flakyScores[test.title] ?? 0;
    if (score > 0.7) {
      console.warn(`[FLAKY-RISK ${score.toFixed(2)}] ${test.title}`);
    }
    // 附加 CI annotation
    process.env.CI && console.log(`::warning title=Flaky Risk::${test.title} score=${score}`);
  }
}

Python 特徵提取（30-run 窗口）


import pandas as pd
from pathlib import Path

runs = pd.read_json('ci-test-runs.jsonl', lines=True)
window = runs.groupby('test_id').tail(30)
features = window.groupby('test_id').agg(
    pass_rate=('status', lambda s: (s == 'passed').mean()),
    pass_var=('status', lambda s: s.eq('passed').rolling(5).mean().std()),
    duration_std=('duration_ms', 'std'),
    fail_count=('status', lambda s: (s == 'failed').sum()),
).reset_index()
features.to_parquet('flaky-features.parquet')

GitHub Actions 自癒觸發


on:
  workflow_run:
    workflows: [CI]
    types: [completed]

jobs:
  auto-heal:
    if: ${{ github.event.workflow_run.conclusion == 'failure' }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Dispatch heal agent
        env:
          COPILOT_TOKEN: ${{ secrets.COPILOT_TOKEN }}
        run: |
          uv run python tools/heal_ci_agent.py \
            --run-id ${{ github.event.workflow_run.id }} \
            --max-files 5 \
            --require-hitl

與本專案應用場景

本 repo（daily-digest-prompt）已有：

  • pre-commit pytest 即時閘門（SKIP_TEST_HOOK=1 bypass）→ 可匯出 pytest JSON report 餵 flaky 模型
  • 448+ CLI 工具 + PS1 管線 → 失敗叢聚於 state/failed-auto-tasks.json，適合 build failure 特徵
  • planner_executor 取代 LangGraph → Agent 自癒可复用 PE job 結構 + results/todoist-auto-*.json 作訓練標籤
  • validate_results.py 閘門 → flaky 等價物是 result_file_missing 間歇失敗；可對 task_key 計算 pass-rate variance

建議落地順序：

  1. 從 logs/structured/post_tool_logger.jsonl 提取 task 執行 pass/fail/duration
  2. 對 tests/ 下 pytest 用例做 30-run 特徵（CI nightly）
  3. score > 0.7 的測試加 @pytest.mark.quarantine
  4. Phase 2 失敗時觸發 Cursor CLI heal（已有 run-todoist-agent-team.ps1 架構）

風險與注意事項

  • False negative 成本高：quarantine 漏判會持續污染 CI；建議保守 threshold + 人工 audit
  • 資料漂移：重構後特徵失效 → 必須 cron 重訓
  • LLM 自癒安全：必須 path-scope + HITL + 測試全綠才允許 merge；禁止 agent 直接 push main
  • 不可取代確定性測試：smoke test 與 main branch 全量 suite 仍須保留
  • 隱私：log 可能含 secret → 餵模型前須 redact
