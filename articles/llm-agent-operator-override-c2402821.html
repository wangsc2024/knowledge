LLM Agent operator override：雙人覆核、理由碼與審計軌跡的工程化

執行摘要（200-400 字）

本報告彙整 NIST AI RMF、OWASP LLM Top 10、LangGraph interrupts、MCP 架構，以及本儲存庫內之 Operator Override Registry 計畫與 github-scout-review-policy，探討在 LLM Agent 生產系統中，人類監督如何必須落在可暫停、可恢復的執行路徑上，而非僅對模型輸出事後審閱。威脅模型核心包含過度代理、工具／外掛邊界遭繞過，以及敏感資料經工具鏈外洩；對應控制則強調四眼原則、可機讀理由碼、與 run_id／trace_id 綁定的審計軌跡。工程上，LangGraph 的 interrupt／resume 與 MCP Host 側最小權限，提供將「核准 payload」寫回狀態的著力點；本 repo 則提出集中式 registry 承載 override_id、scope、default_mode 與 rollback 參照，使 daily-digest 類多 Phase PowerShell JSON handoff 能以最小切片納入治理。分歧面在於集中式 registry 與分散 policy-as-code 的成本，以及 break-glass 即時性與雙人延遲的張力，需以補簽與加強日誌制度化。讀者讀畢應能設計一組可審計、可回放、且與現有 harness 元資料對齊的 operator override 事件模型。

研究背景與動機

生成式 AI 進入工具呼叫與多步編排之後，單一提示詞已無法表徵實際風險：模型可能發起高權限 MCP 呼叫、寫入版本庫，或經由插件擴張行動空間。國際框架（NIST AI RMF）與應用層指引（OWASP LLM Top 10）均要求治理、映射、度量與管理閉環，並點名人類監督與實質介入 [來源：NIST AI Risk Management Framework（AI RMF）, 2023]。在此前提下，operator override 不再是「關掉安全開關」的口語，而必須是可版本化、可回放、可問責的結構化事件；否則無法滿足內外部稽核，也無法在事故後還原「當時哪一組 policy 生效」。

核心技術解析

（一）威脅模型與工具邊界

OWASP 將過度代理、不安全外掛／工具設計與敏感資料外洩列為高頻風險；威脅建模需涵蓋供應鏈、提示注入與工具濫用 [來源：OWASP Top 10 for Large Language Model Applications, 2023]。据三角佐證，工具邊界與 Host 信任根 亦為 MCP 架構文件重點：override 若擴張可用工具，必須同步留下可觀測證據，避免 silent escalation [來源：Model Context Protocol — Architecture, 2024]。

（二）執行期人機協同（HIL）

LangGraph 的 interrupts 允許在節點邊界暫停，待核准後再恢復，並與 checkpoint 搭配支援重播 [來源：LangGraph：Interrupts（Human-in-the-loop）, 2024]。這與「只在最後審稿」不同：核准發生在狀態轉移，理由碼與核准者可作為 resume payload 的一部分寫回——滿足「可機讀」與「與圖狀態一致」兩項審計需求。

（三）可機讀治理欄位與本儲存庫對齊

本 repo 計畫書主張集中式 Operator Override Registry，承載 override_id、scope、effective_from、rollback_ref，並從 trace_id 映射當次生效集合 [來源：GitHub Scout 高風險計畫：集中式 Operator Override Registry（本儲存庫計畫書）, 2026]。github-scout-review-policy 則要求 operator_override 具 override_entry、default_mode、rollback_strategy，並與 execution_contract 的 human_review_required、run_metadata 對齊 [來源：github_scout 審查政策（operator_override 結構化欄位）, 2026]。NIST.SP.1270 對 TEVV 與文件義務的論述，可類比為「多角色覆核與第三方可讀性」的制度包裝 [來源：NIST.SP.1270 — Towards a Standard for Identifying and Managing Bias in AI, 2020]。

關鍵洞見（含三角佐證狀態 ✅／⚠️）

  • 核准須內建於可暫停編排路徑（已驗證✅）：NIST、OWASP、LangGraph、MCP 四方在「監督落在執行路徑」上形成共識，而非僅輸出層審查。
  • 工具邊界與過度代理為 override 威脅核心（已驗證✅）：覆寫預設不得弱化最低基線，且須可追蹤 [來源：OWASP；MCP；本 repo policy]。
  • 審計軌跡最小欄位：誰、何時、範圍、模式、理由碼、run／trace、rollback（已驗證✅）：與 registry 計畫及 LangGraph payload 模型一致。
  • 集中式 registry 與分散 policy-as-code 的取捨（分歧）：一方利於跨 run 版本化審計，一方常僅在 Git／CI 層管理，執行期關聯較弱。
  • TEVV 制度化類比雙人簽核（待驗證⚠️）：文獻支持利害關係人與 TEVV，但與「雙票電子簽核」的一一對應仍需組織內制度落地。
  • 跨模型差分與 registry 版本結合（待驗證⚠️）：屬與前序 continuity 主題的延伸假設，尚待實作驗證。

實務應用建議（3-5 點）

  1. 在 PowerShell 多 Phase JSON handoff 中，為每個 Phase 輸出帶上 task_key、phase、run_id 與當次生效之 override_id 列表，與 post_tool_logger／scheduler-state 並陳。
  2. 理由碼 採「限定枚舉＋自由文字附註」，枚舉供儀表板聚合，文字承載情境；與 LangGraph interrupt resume 結構對齊。
  3. MCP Host 側若因 override 擴張工具允許列表，應記錄擴張前後指紋與工單編號，降低 confused deputy 與 silent escalation。
  4. 將 registry dry-run／review-only 預設與自動審查分數閘門並用，降低誤放行。
  5. 對 break-glass 預先定義「先放行、後補簽」與 ticket 自動開立，並加強該時段之日誌留存與抽樣審計。

與現有知識的連結

本研究與知識庫中 多 Agent 委派式核准、高風險工具授權模擬、動態信任衰減與跨模型差分核准 等主題銜接：共通主軸為「核准內建於狀態機＋可追溯 trace」。daily-digest 管線已具 JSON 交接與 Hook 日誌，適合作為最小治理切片落地點。

未來展望

後續可將 Operator Override Registry 與 autonomous-harness 的 human_review_gates／guarded_operations 建立 metadata 索引（計畫書已提示），並對 跨模型回放 下核准差異做版本化標記；另需實證評估四眼延遲對 SLO 的影響與 break-glass 比例。
