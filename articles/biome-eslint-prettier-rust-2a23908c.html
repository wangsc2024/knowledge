一、專案概述

Biome（biomejs/biome）是一款以 Rust 撰寫的高效能 Web 開發工具鏈，定位為「A performant toolchain for web projects」。專案前身為 Rome（Facebook 孵化），2023 年 8 月 fork 後以 Biome 名義重生並發布 v1.0，截至 2026 年 3 月已累積 23,900+ Stars、881 Forks，被 96,100+ 專案 使用，月下載量超過 1,500 萬次，採用 MIT + Apache 2.0 雙授權。

二、核心能力與效能基準

2.1 統一工具鏈理念

Biome 的核心主張是一個工具取代整個 JS/TS 工具鏈：

| 傳統方案 | Biome 替代 |

|----------|------------|

| ESLint（Linter） | biome lint |

| Prettier（Formatter） | biome format |

| eslint-config-* | 內建 423+ 規則 |

| .eslintrc + .prettierrc + .editorconfig | 單一 biome.json |

| 127+ npm 依賴 | 單一原生二進制 |

2.2 效能基準（Benchmark）

| 操作 | ESLint/Prettier | Biome | 加速倍數 |

|------|----------------|-------|----------|

| Lint 10,000 檔案 | 45.2 秒 | 0.8 秒 | 56x |

| Format 10,000 檔案 | 12.1 秒 | 0.3 秒 | 40x |

| Prettier 相容度 | — | 97% | — |

| Lint 規則數 | 依插件 | 423+ | — |

效能優勢來源：

  1. 原生二進制執行：無 JavaScript runtime 開銷
  2. 多執行緒並行：Rust 的 rayon 並行處理
  3. 增量解析：僅重新解析變動區塊
  4. 高效快取：結構共享的 Green Tree 避免重複分配

2.3 支援語言

  • JavaScript / TypeScript / JSX / TSX
  • JSON / JSONC
  • CSS
  • GraphQL
  • Vue / Svelte / Astro（v2.x 新增解析支援）
  • HTML（2026 穩定化中）
  • YAML（解析器穩定化中）

三、架構深度解析

3.1 Red-Green Tree 語法樹架構

Biome 的語法表示層建立在 biome_rowan（fork 自 rust-analyzer 的 rowan 庫）之上，採用 Red-Green Tree 模式：


源碼 → Parser → Green Tree（不可變、結構共享）
                      ↓
               Red Tree（可變游標導航）
                      ↓
              型別安全 AST 包裝器
  • Green Tree：不可變、結構共享的底層表示，記憶體效率極高
  • Red Tree：提供可變的游標（cursor）導航，無需 clone 整棵樹
  • 型別安全包裝器：每種語言定義具型別的節點列舉（如 JsArrowFunctionExpression）
  • TextSize / TextRange：精確的位置追蹤，支援 LSP 定位

關鍵設計決策：採用 CST（Concrete Syntax Tree） 而非 AST，保留所有原始碼資訊（空白、註解、分號），實現「Full Fidelity」無損表示。

3.2 Parser 架構

每種語言有獨立的解析器基礎設施：

| Crate | 職責 |

|-------|------|

| biome_js_parser | JavaScript/TypeScript 解析器 |

| biome_js_syntax | 型別化 AST 節點定義 |

| biome_js_factory | 節點建構工廠 |

| biome_parser | 共用解析器基礎設施 |

解析器發射 parse events → TreeBuilder 轉換為 Green Tree → 包裝為型別化 Red Tree 節點。

3.3 Formatter 管線


CST 節點 → Format trait 實作 → FormatElement IR → Printer → 格式化文字
  1. 語言專屬格式化器：實作 Format trait（如 biome_js_formatter）
  2. FormatElement IR：抽象表示格式化決策（縮排、換行、群組）
  3. Printer：將 IR 轉換為最終文字，可配置行寬
  4. 97% Prettier 相容：透過 Prettier 測試套件驗證

3.4 Linter / 分析管線

  • biome_analyze：通用規則引擎，提供執行框架
  • 語言專屬分析器：如 biome_js_analyze 實作具體規則
  • Rule trait：每條規則實作 Rule，含元資料與診斷
  • 8 大規則分類：a11y、complexity、correctness、nursery、performance、security、style、suspicious

3.5 語義分析（JavaScript/TypeScript）

| 模組 | 能力 |

|------|------|

| biome_js_semantic | Scope 層次、Binding 解析、閉包分析 |

| biome_js_type_info | 型別推論引擎、型別層次追蹤 |

| biome_module_graph | 跨檔案依賴追蹤、模組匯出/匯入 |

Type-Aware Linting（v2.0 里程碑）：Biome 是第一個不依賴 TypeScript 編譯器就能執行 type-aware lint 規則的工具，透過自建推論引擎實現，由 Vercel 贊助開發。

3.6 Workspace 架構

biome_service::Workspace 是中央調度點：

  • Trait-based 設計：多種實作（WorkspaceServer 管理狀態）
  • 維護：documents（開啟檔案追蹤）、projects（巢狀專案管理）、module_graph（跨檔依賴快取）、settings（合併配置與覆寫）
  • 統一介面供 CLI、LSP、JavaScript API 使用

3.7 使用者介面層

| 介面 | 技術 | 用途 |

|------|------|------|

| biome_cli | Rust binary | CLI（Check/Write/CI 模式） |

| biome_lsp | tower-lsp | LSP 伺服器（即時診斷/格式化/Code Actions） |

| biome_wasm | wasm-bindgen | WebAssembly 綁定 |

| @biomejs/js-api | TypeScript | WASM 的 TS 包裝 |

五、遷移指南要點

5.1 從 ESLint + Prettier 遷移


# 1. 安裝 Biome
npm install --save-dev --save-exact @biomejs/biome

# 2. 初始化配置
npx @biomejs/biome init

# 3. 自動遷移 ESLint 配置
npx @biomejs/biome migrate eslint --write

# 4. 自動遷移 Prettier 配置
npx @biomejs/biome migrate prettier --write

# 5. 執行檢查
npx @biomejs/biome check --write .

5.2 遷移注意事項

  • ESLint 規則相容度約 80%+（2026 年持續提升）
  • 需要 type information 的規則（如 @typescript-eslint/no-floating-promises）：Biome v2 的 type-aware 已覆蓋部分，但仍建議保留 tsc --noEmit 在 CI
  • 自定義 ESLint 插件無法自動遷移，需手動轉換或等待對應 Biome 規則

六、與本專案（Daily Digest Prompt）的關聯啟示

6.1 Rust-Based 工具鏈趨勢

Biome 與 Oxc、Rolldown、Turbopack 共同代表 2026 年前端工具鏈「Rust 化」浪潮。核心驅動力：

  • JavaScript 工具用 JavaScript 寫 → 受限於單執行緒、GC、冷啟動
  • Rust 工具 → 原生多執行緒、零成本抽象、可編譯為 WASM

6.2 架構借鑑

| Biome 設計 | DDP 可借鑑 |

|------------|----------|

| Red-Green Tree（不可變 + 可變游標） | 類似 digest-memory.json 的不可變歷史 + 可變 session 狀態 |

| Trait-based Workspace | 類似 Skill-First 的 trait-based 行為分派 |

| 語言專屬 crate + 共用基礎設施 | 類似 templates/sub-agent/ 專屬模板 + shared/ 共用前言 |

| 8 分類規則系統 | 類似 hook-rules.yaml 的分類規則管理 |

七、結論與評價

優勢

  • 驚人效能：40-56x 速度提升，CI 時間大幅縮短
  • 統一配置：一個 biome.json 取代多個 config 檔案
  • 零依賴：單一二進制，無 node_modules 膨脹
  • Type-Aware 創新：不依賴 tsc 的型別推論，開創先例
  • Cross-Language Vision：跨語言 lint 規則是革命性方向

挑戰

  • ESLint 插件生態系統龐大（2,800+），完全替代仍需時間
  • SCSS/HTML 等語言支援仍在穩定化中
  • 社群規模（24K Stars）vs ESLint（25K+ Stars）仍在追趕

評分：4.5/5

Biome 是 2026 年前端工具鏈最重要的革新之一。對於追求極致 CI/CD 效能、厭倦 ESLint 配置地獄的團隊，Biome 已是成熟且可靠的選擇。
