---

執行摘要

DeepEval 是一個開源 LLM 評測框架，由 Confident AI 開發，專注於將評測整合到 CI/CD 管線中 GitHub - confident-ai/deepeval。其核心價值在於「類似 Pytest 的 LLM 評測體驗」，提供 50+ 研究支持的評測指標，涵蓋 RAG、Agent、Chatbot、多輪對話與安全性等使用場景 Quick Introduction | DeepEval。與競品相比，DeepEval 的 pytest 整合度最高，適合需要在 CI/CD 管線中阻擋劣質部署的工程團隊 EVAL #006: LLM Evaluation Tools — RAGAS vs DeepEval vs Braintrust vs LangSmith vs Arize Phoenix。

關鍵發現：

  • pytest 原生整合：可寫成 def test_hallucination() 形式，在 CI 中自動執行，失敗即阻擋部署
  • 50+ 開箱即用指標：涵蓋通用（G-Eval, DAG）、RAG（Faithfulness, Answer Relevancy）、Agent（Task Completion, Tool Correctness）、安全性（Hallucination, Bias, Toxicity）
  • 六層架構：分離評測邏輯、資料管理、指標計算、模型抽象、可觀測性與入口點
  • 黑盒 + 白盒雙模式：可端到端評測整個應用，或透過 @observe 裝飾器追蹤內部元件

---

研究背景

問題脈絡

LLM 應用的品質閘門機制目前依賴手動測試或簡單的單元測試，缺乏針對 LLM 特性（幻覺、偏見、語義一致性）的評測框架 Build an LLM Evaluation Framework: Metrics, Methods & Tools | Codecademy。傳統軟體測試的「assert 期望值 == 實際值」無法適用於 LLM 的非確定性輸出，需要新的評測範式。

DeepEval 的解決方案

DeepEval 將評測範式從「精確比對」轉為「語義評分」，並將評測流程標準化為 pytest 風格的單元測試，使 LLM 評測能無縫整合到現有的 CI/CD 工作流 Using DeepEval for Large Language Model (LLM) Evaluation in Python | Codecademy。

---

核心發現（含三角佐證標記）

1. 專案定位：Pytest for LLMs

DeepEval 的核心定位是「LLM 評測的 Pytest」，將評測視為單元測試的一部分 GitHub - confident-ai/deepeval。這種定位經過三角佐證 ✅：

  • 來源 1（官方）：GitHub README 明確標示為 "The LLM Evaluation Framework"，安裝僅需 pip install -U deepeval GitHub
  • 來源 2（文件）：官方文件描述為 "Pytest for LLMs"，強調與 pytest 的整合 Quick Introduction
  • 來源 3（評測）：獨立評測文章稱其為 "engineering-native approach"，最適合需要 CI/CD 整合的團隊 EVAL #006

2. 指標體系：50+ 研究支持的 Metrics

DeepEval 提供 50+ 開箱即用的評測指標，涵蓋多種使用場景 ✅（三角佐證）：

  • 來源 1：官方文件列出 50+ LLM-evaluated metrics，並強調「research backing」Introduction to LLM Metrics
  • 來源 2：教學文章確認 "50+ out-of-the-box metrics covering RAG, conversational, red-teaming, agentic, and multimodal use cases" RAG Evaluation Quickstart
  • 來源 3：DeepWiki 系統架構文件詳細說明指標分類與演算法設計 System Architecture | DeepWiki

指標分類：

  • 通用指標：G-Eval（基於研究的 LLM-as-a-judge）、DAG（圖基礎確定性指標）
  • RAG 指標：Answer Relevancy（答案與問題的相關性）、Faithfulness（答案是否忠於檢索上下文）、Contextual Recall（檢索覆蓋度）、Contextual Precision（檢索準確度）
  • Agent 指標：Task Completion（任務完成度）、Tool Correctness（工具使用正確性）、Goal Accuracy（目標達成準確性）、Step Efficiency（步驟效率）
  • 安全性指標：Hallucination（幻覺偵測）、Bias（偏見檢測）、Toxicity（毒性檢測）、JSON Correctness（JSON 格式正確性）

3. 架構設計：六層模組化架構

DeepEval 採用六層架構設計，清晰分離關注點 ✅（雙來源佐證）：

  • 來源 1：DeepWiki 系統架構文件明確列出六層：Entry Points, Evaluation Engine, Data Management, Metrics, Model Abstraction, Observability System Architecture
  • 來源 2：官方文件說明核心元件（Metrics Architecture, Evaluation Engine, Model Abstraction, Tracing）DeepEval LLM evaluation framework architecture

六層架構說明：

  1. Entry Points：pytest 整合介面（@pytest.mark.parametrize）、命令列工具（deepeval test run）
  2. Evaluation Engine：測試用例執行、指標計算、結果聚合
  3. Data Management：測試用例（LLMTestCase）、黃金資料集（Golden）、評測資料集（EvaluationDataset）
  4. Metrics：50+ 指標的演算法實作，支援 LLM-as-a-judge、DAG、equation-based 等多種評分方式
  5. Model Abstraction：統一 LLM 介面，支援 6+ 原生 provider（OpenAI, Anthropic, Azure, Cohere, Groq, Ollama）+ 自訂實作
  6. Observability：透過 @observe 裝飾器追蹤執行流程，整合 OpenTelemetry，支援元件級評測

4. 競品對比：DeepEval vs RAGAS vs LangSmith

三個框架各有優勢，選擇依團隊需求而定 ✅（三角佐證）：

  • 來源 1：dev.to 評測文章提供詳細對比，建議「需要 CI/CD 整合 → DeepEval；零預算 → RAGAS；已用 LangChain → LangSmith」EVAL #006
  • 來源 2：Braintrust 文章確認 DeepEval 的 pytest 整合是「最佳 CI/CD 整合」DeepEval alternatives
  • 來源 3：ZenML 部落格比較 8 個 DeepEval 替代方案，確認其在工程化評測的獨特定位 8 Best DeepEval Alternatives

對比總結：

| 框架 | 核心優勢 | 最適合 |

|------|---------|--------|

| DeepEval | Pytest 整合、50+ 指標、CI/CD 原生 | 需要在 CI 中阻擋劣質部署的工程團隊 |

| RAGAS | 專注 RAG 評測、零成本（本機執行）| 預算有限、只需 RAG 評測的團隊 |

| LangSmith | LangChain 整合、追蹤 + 評測一體 | 已使用 LangChain 生態的團隊 |

深度分析

RAG 評測實踐

DeepEval 的 RAG 評測流程包含檢索器（Retriever）與生成器（Generator）的分離評測 RAG Evaluation Guide。

基本範例（Python）：


from deepeval import evaluate
from deepeval.metrics import AnswerRelevancyMetric, FaithfulnessMetric
from deepeval.test_case import LLMTestCase

# 定義測試用例
test_case = LLMTestCase(
    input="什麼是 RAG？",
    actual_output="RAG 是 Retrieval-Augmented Generation 的縮寫",
    retrieval_context=["RAG combines retrieval with generation..."]
)

# 定義指標
answer_relevancy = AnswerRelevancyMetric(threshold=0.7)
faithfulness = FaithfulnessMetric(threshold=0.7)

# 執行評測
evaluate([test_case], [answer_relevancy, faithfulness])

多輪對話評測：


from deepeval.test_case import ConversationalTestCase, Turn
from deepeval.metrics import TurnRelevancy, TurnFaithfulness

# 定義對話回合
turns = [
    Turn(user_input="你好", llm_output="你好！有什麼可以幫到你的？"),
    Turn(user_input="什麼是 RAG？", llm_output="RAG 是...", retrieval_context=["..."])
]

test_case = ConversationalTestCase(turns=turns)
evaluate([test_case], [TurnRelevancy(), TurnFaithfulness()])

CI/CD 整合

DeepEval 最大的差異化在於 pytest 整合，可直接在 CI 管線中執行：


# test_rag.py
import pytest
from deepeval import assert_test
from deepeval.metrics import HallucinationMetric

@pytest.mark.parametrize("test_case", generate_test_cases())
def test_hallucination(test_case):
    metric = HallucinationMetric(threshold=0.8)
    assert_test(test_case, [metric])

GitHub Actions 範例：


name: LLM Evaluation
on: [push, pull_request]

jobs:
  evaluate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install dependencies
        run: pip install deepeval pytest
      - name: Run LLM tests
        run: deepeval test run test_rag.py

Confident AI 雲端平台

DeepEval 提供雲端平台（Confident AI）用於結果視覺化、A/B 實驗與團隊協作 Quick Introduction。但核心框架完全開源，可本機執行。

快速上手步驟

  1. 安裝：pip install -U deepeval（需 Python ≥ 3.9）
  2. 設定 API Key：export OPENAI_API_KEY="..."（或用本機模型如 Ollama）
  3. 建立測試：寫 pytest 風格的測試檔（參考上方範例）
  4. 執行評測：deepeval test run test_rag.py
  5. 整合 CI：加入 GitHub Actions workflow

優缺點評估

優點

  • 工程化評測體驗：pytest 整合是業界最佳，CI/CD 整合無痛 EVAL #006
  • 全面的指標覆蓋：50+ 指標涵蓋 RAG/Agent/Chatbot/安全性，無需自己實作 Introduction to LLM Metrics
  • 黑盒 + 白盒雙模式：既可端到端評測，又可深入元件級追蹤
  • 開源 + 雲端雙軌：核心框架完全開源，需要協作時可選 Confident AI 平台

缺點

  • LLM-as-a-judge 成本：50+ 指標大多依賴 LLM 評分，評測成本隨測試用例數量線性增長（來源：官方文件說明所有指標使用 LLM-as-a-judge G-Eval）
  • 本機執行受限：部分指標（如 G-Eval）需要強大的 LLM，本機模型（如 Llama 7B）可能無法達到足夠的評測品質（來源：社群討論反映小型模型作為 judge 的可靠性問題）
  • 學習曲線：50+ 指標需要理解各自的適用場景與閾值設定（來源：官方文件建議每個指標需調整 threshold Metrics Introduction）

---

知識缺口與未來方向

當前限制

  • 指標可解釋性：LLM-as-a-judge 的評分過程不透明，難以解釋為何某個測試失敗
  • 多語言支援：官方文件未明確說明中文等非英語語言的評測效果
  • 自訂指標門檻：需手動調整每個指標的 threshold，缺少自動最佳化機制

下一階段研究方向（mechanism 階段）

  1. G-Eval 演算法深度解析：理解 chain-of-thought 評分的內部機制
  2. DAG 指標實作細節：樹狀圖決策流程的演算法設計
  3. Model Abstraction 層源碼分析：如何統一不同 LLM provider 的介面
  4. Tracing 系統整合：@observe 裝飾器如何與 OpenTelemetry 整合
