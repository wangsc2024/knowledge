MCP taint propagation 2026：跨工具資料流標記、政策引擎與執行隔離

執行摘要

本報告回答三個研究問題：第一，MCP 之所以會出現「跨工具 taint propagation」，而不僅是單一 tool poisoning，是因為模型上下文在多個 MCP server 與多個 tool 之間形成共享狀態：任何來自不可信 descriptor、tool output 或遠端資源的片段，都可能在後續推理中被轉寫成另一個工具的參數或呼叫序列，造成信任邊界「橫向移動」而非僅垂直注入 [來源：Securing the Model Context Protocol: Defending LLMs Against Tool Poisoning and Adversarial Attacks, 2025]。第二，2025–2026 的官方規範與多篇安全論文，在授權、metadata 驗證與 runtime guardrail 上已形成部分共識：OAuth 2.1、RFC 8707 的 resource parameter、audience 驗證、禁止 token passthrough、短生命週期 token 與 PKCE 等，被寫入 MCP Authorization 與 Security Best Practices 文件 [來源：Authorization - Model Context Protocol (2025-11-25), 2025]；研究社群則補上 capability attestation、雙向取樣缺乏來源認證、以及多 server 信任傳播等協定層缺口 [來源：Breaking the Protocol: Security Analysis of the Model Context Protocol Specification and Prompt Injection Vulnerabilities in Tool-Integrated LLM Agents, 2026]。第三，在生產環境落地時，應把 capability 分類、OAuth scope 最小化、resource／audience 綁定、runtime monitoring（含參數可見性與決策追蹤）、以及 STDIO／本機行程隔離 串成一條可操作防線，而不是只依賴「使用者按一次同意」 [來源：Security Best Practices - Model Context Protocol, 2025]。

核心結論共五點：（1）taint 的單位應以「資料流進入 agent state 的通道」界定，而非僅以 tool 名稱界定。（2）OAuth 的 resource／audience 驗證是把信任域收斂在單一 MCP server 的結構性控制點。（3）Tool poisoning、shadowing、rug pull 等語意層攻擊，需要 signing、semantic vetting 與 runtime heuristics 並行 [來源：Securing the Model Context Protocol: Defending LLMs Against Tool Poisoning and Adversarial Attacks, 2025]。（4）STDIO 啟動模型若將 command／args 暴露給不可信輸入，會把整條 agent 供應鏈拉高到 RCE 等級 [來源：MCP STDIO Command Injection: Full Vulnerability Advisory, 2026]。（5）中文產業與監管敘事已把「代理式 AI 最小權限」視為合規底線，與 MCP 工具鏈風險敘事互相強化 [來源：OpenClaw養龍蝦AI｜私隱公署：授予代理式AI任務所需最小權限 - HK01, 2026]。

研究背景與動機

當 LLM 透過 MCP 同時掛載檔案系統、瀏覽器、企業 API 與本機命令列時，單一惡意工具已不足以描述真實威脅模型：模型會在單一對話狀態中累積多來源證據，並在後續步驟「合法地」呼叫另一個看起來無害的工具，卻把先前被污染的語境轉成具副作用的行為。官方文件已將 confused deputy、本機 MCP server 被攻破、以及 scope 膨脹列為核心風險 [來源：Security Best Practices - Model Context Protocol, 2025]，但規範層仍大量依賴 client 與操作者的實作品質；研究論文則指出協定層仍缺乏完整的 capability attestation 與訊息來源鑑別 [來源：Breaking the Protocol: Security Analysis of the Model Context Protocol Specification and Prompt Injection Vulnerabilities in Tool-Integrated LLM Agents, 2026]。因此，本研究聚焦「taint 如何跨工具傳播」以及「政策引擎＋執行隔離」如何填補規範與實務之間的縫隙，並對照本專案既有的 MCP 安全深度研究筆記，形成可延續的知識鏈。

核心技術解析

子題一：協定層信任邊界與 OAuth 綁定

MCP Authorization 要求 client 使用 RFC 8707 的 resource parameter，server 必須驗證 token audience，且不得把 access token 原封轉傳給下游 [來源：Authorization - Model Context Protocol (2025-06-18), 2025]。2025-11-25 版更明確限制 token 不得出現在 query string、要求短生命週期與 refresh rotation [來源：Authorization - Model Context Protocol (2025-11-25), 2025]。這些條款在工程上把「哪一個 MCP server 有權代表使用者行動」寫成可驗證的不變量，直接壓縮 taint 外溢到其他 trust domain 的機率。Security Best Practices 亦將 scope 最小化、逐步升權與危險模式揭露列為必要操作文化 [來源：Security Best Practices - Model Context Protocol, 2025]。

子題二：Tool poisoning 與多工具語意攻擊面（學術視角）

ArXiv 2512.06556 將攻擊分類為 tool poisoning、shadowing、rug pull，並提出 manifest signing、semantic vetting、runtime heuristic guardrails 的分層防禦 [來源：Securing the Model Context Protocol: Defending LLMs Against Tool Poisoning and Adversarial Attacks, 2025]。ArXiv 2601.17549 進一步以大量情境實驗指出，在 MCP 架構下攻擊成功率相對非 MCP 整合顯著放大，並以 MCPSec 將成功率自 52.8% 降至 12.4% 且額外延遲中位數約 8.3ms [來源：Breaking the Protocol: Security Analysis of the Model Context Protocol Specification and Prompt Injection Vulnerabilities in Tool-Integrated LLM Agents, 2026]。ArXiv 2603.22489 以 STRIDE／DREAD 檢視五個 MCP 元件後，認定 tool poisoning 是 client 端最關鍵風險，並建議 static metadata analysis、decision path tracking 與行為異常偵測 [來源：Model Context Protocol Threat Modeling and Analyzing Vulnerabilities to Prompt Injection with Tool Poisoning, 2026]。三篇論文共同支持：若沒有執行期可觀測性，靜態同意無法對抗語意層組合攻擊。

子題三：STDIO、供應鏈與「組態即命令」風險（官方＋產業）

官方安全最佳實踐已提醒 one-click 本機 server 設定必須揭露即將執行的命令，並建議沙箱與最小權限 [來源：Security Best Practices - Model Context Protocol, 2025]。OX Security 的 advisory 說明 STDIO configuration-to-command execution 路徑如何把不可信輸入映射到實際行程參數，並指出純輸入過濾仍可能被 args 層繞過 [來源：MCP STDIO Command Injection: Full Vulnerability Advisory, 2026]。中文轉述報導亦反映社群已將此類問題理解為「架構性」而非單一產品臭蟲 [來源：Anthropic MCP 協議被指存在設計缺陷，伺服器可被誘導執行任意代碼 - IT之家, 2026]。此子題與子題一、二交會：即使 OAuth 正確，若本機啟動鏈可被篡改，taint 仍可直接變成任意程式執行。

關鍵洞見

  1. ✅（支持來源數 5）真正要標記與稽核的是進入 agent state 的 taint 流，而非僅標記「某 tool 是否惡意」；反方觀點可能認為過細標記會拖慢互動，但證據顯示缺乏決策追蹤時使用者無法審計模型的跨步組合 [來源：Model Context Protocol Threat Modeling and Analyzing Vulnerabilities to Prompt Injection with Tool Poisoning, 2026]。
  2. ✅（4）政策引擎的最小單位應是 capability＋scope＋data sink（寄信、寫檔、執行 shell），而不是 tool 字串；與 OAuth incremental consent 方向一致 [來源：Authorization - Model Context Protocol (2025-11-25), 2025]。
  3. ✅（4）resource parameter 與 audience validation 是限制 taint 在單一 server trust domain 的關鍵控制；忽略時等於允許 confused deputy 跨資源誤用 [來源：Authorization - Model Context Protocol (2025-06-18), 2025]。
  4. ✅（4）若只做 pre-approval，仍可能被 rug pull 或執行期組合行為繞過，因此核准必須延伸到 runtime [來源：Securing the Model Context Protocol: Defending LLMs Against Tool Poisoning and Adversarial Attacks, 2025]。
  5. ✅（3）多數現行客戶端缺乏 parameter visibility 與 decision trace，使 tainted context 難以被使用者或稽核者還原；此與論文對 client 實作落差的實證一致 [來源：Model Context Protocol Threat Modeling and Analyzing Vulnerabilities to Prompt Injection with Tool Poisoning, 2026]。
  6. ⚠️（2）MCPSec、manifest signing 與 runtime anomaly detection 可能收斂為標準化路線，但目前仍屬研究導向，尚未內建於 MCP 官方規範文本；採納前需評估營運成本與誤判率 [來源：Breaking the Protocol: Security Analysis of the Model Context Protocol Specification and Prompt Injection Vulnerabilities in Tool-Integrated LLM Agents, 2026]。

實務應用建議

  1. 建立 capability 分級表：以「讀取／寫入／對外通訊／執行程式」四象限為軸，為每個 MCP server 與 tool 對應分級；落地步驟為盤點現有 MCP 清單→對照 OAuth scope→標示資料 sink；風險為分級過粗導致過度放行，需每季重審。
  2. 強制 resource／audience 測試：在 CI 以負向測試確認 client 不會在錯誤 audience 下送出 token；風險為與舊版 IdP 相容性，可採 feature flag。
  3. 執行期「參數可見性」介面：在呼叫高風險工具前，顯示解析後的完整參數與來源訊息片段；風險為 UX 摩擦，可用風險分數閾值觸發。
  4. STDIO 啟動硬化：預設拒絕可變更的 command template、改以 allowlist 行程與唯讀組態目錄；參考官方「揭露即將執行的命令」要求 [來源：Security Best Practices - Model Context Protocol, 2025]。
  5. 與企業治理對齊：把代理式 AI 的最小權限原則寫入內部政策，對照隱私與資安稽核問題清單 [來源：OpenClaw養龍蝦AI｜私隱公署：授予代理式AI任務所需最小權限 - HK01, 2026]。

與現有知識的連結

本專案 context/research-registry.json 已登錄「MCP 安全 2026：Tool Poisoning、Remote MCP 信任邊界與 Runtime 防禦」等條目；本次主題在其上往橫向延伸，把論述重心從「單一工具是否可信」轉為「多工具共享狀態下的 taint 傳播與政策引擎」。與 daily-digest-prompt 的 Agent 管線亦相呼應：自動任務大量依賴工具呼叫與結果 JSON，若未來引入 MCP 作為外掛能力，應沿用本報告的分層控制敘事，避免把工具授權簡化成單一布林值。

未來展望

仍未完全解決的問題包括：官方是否會提出訊息層級的 provenance／taint 標準欄位；MCPSec 類中介在高效能場景的部署形態；以及中文原生技術規範與實務案例仍偏少，需補強在地化證據鏈。下次可深化的方向是「具體 policy engine schema 與現有 OAuth client library 的整合測試矩陣」。
