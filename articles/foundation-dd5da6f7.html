結果檔案缺失診斷框架：根因分類與系統性修復策略（foundation）

研究日期：2026-04-08

系列：result-file-diagnostic-recovery（foundation 階段）

摘要

過去 24 小時，9/11 自動任務結果檔被標記為 result_file_missing_or_invalid，導致 Phase 3 組裝為 partial_success。本研究建立診斷框架，提煉四種根因分類（backend 執行失敗、檔案覆寫、診斷欄位未補全、JSON 損壞）與對應修復策略，為後續 mechanism 階段奠定基礎。

關鍵洞見

1. 診斷流程已標準化，但實作未完整整合

workflow-forge 已產出完整診斷流程（6 步驟、4 根因），但 run-todoist-agent-team.ps1 尚未實作 Invoke-ResultFileDiagnostic 函數。Gap：workflow 定義與執行邏輯之間存在實作缺口。

2. 四種根因分類清晰，可直接應用

  • backend_execution_failed（backend 未產出）
  • file_overwritten（檔案被覆寫）
  • diagnostic_not_applied（診斷欄位未補全）
  • json_corruption（JSON 損壞）

每種根因都有明確判定條件、證據清單、修復策略。

3. 修復策略分層，但缺少自動化

Tier 1（自動補全）僅針對 schema_incomplete，其他三種根因缺少自動化修復邏輯。建議引入 Write-FileAtomic、backend 停用閾值、Harness Gap Loop 整合。

4. 診斷欄位規範完善，但填寫時機需優化

cli_exit_note_zh、result_guard_zh 規範明確，但占位/Repair 路徑可能跳過補全。建議統一補全點，確保所有路徑都填寫。

5. 追蹤機制設計完整，可作為長期觀測基礎

state/result-file-diagnostics.json 保留 7 天，可機器解析與趨勢分析。建議優先級：P0 實作基礎追蹤、P1 整合 check-health、P2 Dashboard、P3 預測性診斷。

建議行動（三層優化策略）

短期（1-2 週）

  • 實作 Invoke-ResultFileDiagnostic 函數
  • Phase 2 後自動觸發，診斷結果寫入 state/result-file-diagnostics.json
  • 預期：診斷覆蓋率 100%，根因定位準確率 >90%

中期（1 個月）

  • 實作 Write-FileAtomic（原子性檔案寫入）
  • backend 失敗率 >60% 時自動停用
  • 統一診斷欄位補全邏輯
  • 預期：JSON 損壞率降至 <1%

長期（2-3 個月）

  • 整合 Harness Gap Loop
  • Dashboard 可視化
  • 預測性診斷
  • 預期：partial_success 率降至 <5%

後續研究方向（mechanism 階段）

  1. Cursor CLI 與 PowerShell 交接機制深入分析
  2. backend 失敗模式分析
  3. 占位與 Repair 路徑優化
