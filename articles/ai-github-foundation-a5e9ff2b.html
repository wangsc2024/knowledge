  • 研究日期：2026-04-25 研究類型：ai_workflow_github 自動任務 子領域：No-code 工作流 GitHub：FlowiseAI/Flowise | Stars：52.2k | Forks：24.2k | License：Apache 2.0 | 最新版本：v3.1.2（2026-04-14）
  • Flowise 是一個以 TypeScript 開發的開源 AI Agent 視覺化建構平台。核心理念是「Build AI Agents, Visually」，讓技術與非技術人員都能透過拖放畫布，把向量資料庫、LLM、工具節點串接成可部署的 RAG Pipeline 或 Agent Workflow，而無需撰寫應用程式碼。
  • 相較 n8n 的「通用自動化 + AI 疊加」定位，Flowise 從一開始就是 AI-first：每個節點都是 LangChain 生態系的直接視覺對應，最短 15 分鐘可跑起一個附有 RAG 的聊天機器人原型。2026 年 4 月發布的 v3.1.2 持續迭代 AgentFlow V2 架構，已成為 No-code AI 工作流領域的代表性標竿之一。
  • Flowise 採用 PNPM Workspace + Turborepo 管理的 Monorepo，按照 components → server → ui 的建構順序串接：
  • | Package | 技術 | 職責 | |---------|------|------| | packages/server | Express.js | REST API + 執行引擎 | | packages/ui | React + ReactFlow | 拖放畫布前端 | | packages/components | TypeScript | 1000+ AI 提供者整合 | | packages/agentflow | React（模組化） | AgentFlow V2 有狀態工作流 UI | | packages/api-documentation | Swagger | API 文件 |
  • 語言組成：TypeScript 59.5%、JavaScript 28.6%，Runtime 需求 Node.js ≥18.15.0。
  • NodesPool 類別在伺服器啟動時，自動掃描並載入 flowise-components Package 中的所有節點實作（每個節點實作 INode 介面）。執行時依工作流圖的依賴順序，逐節點初始化、注入憑證、呼叫 AI 提供者，最終透過 SSE（Server-Sent Events）串流結果。這種設計讓新增自訂節點等同於新增一個 TypeScript 類別，無需修改執行引擎。
  • Chatflow（DAG 型）：使用廣度優先遍歷（buildChatflow()），無狀態、單輪或多輪對話，適合 RAG Q&A、文件問答、客服機器人。
  • Agentflow（有狀態序列型）：使用 executeAgentFlow() 序列執行，支援共享狀態（$flow.state）、分支、迴圈，適合多步驟 Agent 任務、Human-in-the-Loop 流程。
  • V2 是 Flowise 最重要的架構演進，放棄依賴外部框架（LangChain Agent），改為原生開發 14 種專用節點：
  • - Start：工作流入口，初始化 $flow.state - LLM：呼叫語言模型 - Agent：具備工具呼叫能力的 AI 代理 - Tool：外部工具（API/函數/MCP） - Retriever：知識庫檢索 - HTTP：REST API 呼叫 - Condition：條件分支 - Loop / Iteration：迴圈與迭代 - Human Input：暫停等待人類輸入（HITL） - Direct Reply：直接回覆不經 LLM - Custom Function：JavaScript 自訂邏輯 - Execute Flow：呼叫子 Chatflow/AgentFlow（子工作流）
  • $flow.state 共享狀態機制：工作流執行期間的 Key-Value 儲存庫，必須在 Start 節點初始化。所有節點可跨分支讀寫，實現跨步驟資料持久化——類似本系統的 state/*.json 模式，但在視覺層直接操作。
  • 多 Agent 協作（Supervisor/Worker 模式）：Supervisor Agent 可分解任務並委派給多個 Worker Agent，每個 Agent 可存取完整對話歷史，是視覺化工具中少數支援原生多 Agent 協調的實作。

  • 這種分層讓「本地開發」與「千台雲端 Agent 部署」共用同一套代碼庫，僅切換 Workspace 實作即可[來源：arxiv 2511.03690]。

  • 事件層次結構分三層： - Event（基底，所有事件的父類） - LLMConvertibleEvent（LLM 可見，含 ActionEvent 和 ObservationBaseEvent） - 內部書記事件（不進入 LLM context）

  • 工具定義兩種模式： - 直接實例化（無狀態工具，如 finish、think） - 子類工廠模式（有狀態工具，如 execute_bash、file_editor，透過 @classmethod create() 工廠方法配置 session 狀態）
