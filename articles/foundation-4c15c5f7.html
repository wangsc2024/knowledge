核心內容摘錄：

  • 本報告回答三個研究問題：第一，MCP 之所以會出現「跨工具 taint propagation」，而不僅是單一 tool poisoning，是因為模型上下文在多個 MCP server 與多個 tool 之間形成共享狀態：任何來自不可信 descriptor、tool output 或遠端資源的片段，都可能在後續推理中被轉寫成另一個工具的參數或呼叫序列，造成信任邊界「橫向移動」而非僅垂直注入 [來源：Securing the Model Context Protocol: Defending LLMs Against Tool Poisoning and Adversarial Attacks, 2025]。第二，2025–2026 的官方規範與多篇安全論文，在授權、metadata 驗證與 runtime guardrail 上已形成部分共識：OAuth 2.1、RFC 8707 的 resource parameter、audience 驗證、禁止 token passthrough、短生命週期 token 與 PKCE 等，被寫入 MCP Authorization 與 Security Best Practices 文件 [來源：Authorization - Model Context Protocol (2025-11-25), 2025]；研究社群則補上 capability attestation、雙向取樣缺乏來源認證、以及多 server 信任傳播等協定層缺口 [來源：Breaking the Protocol: Security Analysis of the Model Context Protocol Specification and Prompt Injection Vulnerabilities in Tool-Integrated LLM Agents, 2026]。第三，在生產環境落地時，應把 capability 分類、OAuth scope 最小化、resource／audience 綁定、runtime monitoring（含參數可見性與決策追蹤）、以及 STDIO／本機行程隔離 串成一條可操作防線，而不是只依賴「使用者按一次同意」 [來源：Security Best Practices - Model Context Protocol, 2025]。
  • 核心結論共五點：（1）taint 的單位應以「資料流進入 agent state 的通道」界定，而非僅以 tool 名稱界定。（2）OAuth 的 resource／audience 驗證是把信任域收斂在單一 MCP server 的結構性控制點。（3）Tool poisoning、shadowing、rug pull 等語意層攻擊，需要 signing、semantic vetting 與 runtime heuristics 並行 [來源：Securing the Model Context Protocol: Defending LLMs Against Tool Poisoning and Adversarial Attacks, 2025]。（4）STDIO 啟動模型若將 command／args 暴露給不可信輸入，會把整條 agent 供應鏈拉高到 RCE 等級 [來源：MCP STDIO Command Injection: Full Vulnerability Advisory, 2026]。（5）中文產業與監管敘事已把「代理式 AI 最小權限」視為合規底線，與 MCP 工具鏈風險敘事互相強化 [來源：OpenClaw養龍蝦AI｜私隱公署：授予代理式AI任務所需最小權限 - HK01, 2026]。
  • 當 LLM 透過 MCP 同時掛載檔案系統、瀏覽器、企業 API 與本機命令列時，單一惡意工具已不足以描述真實威脅模型：模型會在單一對話狀態中累積多來源證據，並在後續步驟「合法地」呼叫另一個看起來無害的工具，卻把先前被污染的語境轉成具副作用的行為。官方文件已將 confused deputy、本機 MCP server 被攻破、以及 scope 膨脹列為核心風險 [來源：Security Best Practices - Model Context Protocol, 2025]，但規範層仍大量依賴 client 與操作者的實作品質；研究論文則指出協定層仍缺乏完整的 capability attestation 與訊息來源鑑別 [來源：Breaking the Protocol: Security Analysis of the Model Context Protocol Specification and Prompt Injection Vulnerabilities in Tool-Integrated LLM Agents, 2026]。因此，本研究聚焦「taint 如何跨工具傳播」以及「政策引擎＋執行隔離」如何填補規範與實務之間的縫隙，並對照本專案既有的 MCP 安全深度研究筆記，形成可延續的知識鏈。

  • - 過濾等效變異使 reward signal 保持乾淨 - 使用 Group Relative Policy Optimization（GRPO）建立 compact but aggressive 測試套件 - 增量 reward 機制：只在新測試殺死前輪 surviving mutant 時給正獎勵，防止重複覆蓋
  • - 記憶式執行：記住已執行結果，PR CI 只重跑受影響變更，CI 時間 -70% - selective mutation（Python idioms）： - mutate_only_covered_lines = true：只 mutate coverage.py 追蹤到的覆蓋行 - 靜態分析標記等效變異（2025+ 版本），減少人工 review 負擔 - CI 整合：--mutation-score-threshold + --no-rebuild 在增量模式下自動跳過低影響檔案

  • ### Nightly 層（Full + AI 反饋） `` 1. mutmut run --all（全量，parallelize with -j 4） 2. surviving mutants → feed to Claude/LLM："以下 N 個 mutant 存活，針對每個生成補強測試" 3. AI 生成測試 → 人工 review queue（或 auto-accept if confidence > 0.8） 4. MSR 趨勢寫入 state/mutation-trend.json ``
  • ### Flaky Test 防護 `` 1. LLM-generated tests 標記 @pytest.mark.llm_generated 2. 執行 3 次取中位數（detect non-determinism） 3. FlakyLens 過濾：避免 Thread.sleep / await 等 keyword-fixated 誤分類 4. flaky 率 > 15% 時，停止 auto-accept 進入人工 review ``

  • - Speculative Decoding 透過 draft model 預先生成候選 tokens 並由 target model 批次驗證，可顯著降低 LLM 推理延遲與成本。本研究綜合 vLLM、SGLang、TensorRT-LLM 官方文件與生產實務案例，揭示三大核心發現：（1）高併發場景下 Speculative Decoding 仍可提升效能，打破「僅適用低 QPS」假設；（2）draft token acceptance rate 是評估效益的関鍵指標，需動態調整 draft tokens 數量以維持穩定增益；（3）生產部署必須使用與實際工作負載匹配的資料集進行 benchmark，通用基準不足以指導調參。研究同時揭示學術界發現但工業界文件未提及的 output equivalence 問題，以及針對 coding agents 等 decode-heavy 工作負載的專屬調參策略（6-8 draft heads vs. 一般語言模型的 3-4）。對於 daily-digest 類自動化任務，建議優先採用 EAGLE 或 N-Gram 方法，並建立 acceptance rate 監控機制以確保長期效益。
  • - LLM 推理的計算成本主要來自 autoregressive decoding 過程中的序列依賴性——每個 token 必須等待前一個 token 生成完成。Speculative Decoding 透過引入輕量 draft model 平行生成多個候選 tokens，再由 target model 一次性驗證，可將多次序列推理合併為單次批次計算，從而降低延遲與成本。
  • - 然而，何時開啟 Speculative Decoding 才能真正降低 TTFT（Time To First Token）、TPOT（Time Per Output Token）與單位 token 成本？不同推理框架（vLLM、SGLang、TensorRT-LLM）的 scheduler、lookahead、draft token window 設計有何差異？如何為特定工作負載（如 coding agents、daily-digest）建立可重現的 batch profiling 與調參流程？這些問題直接影響生產環境的部署決策，但工業界文件與學術研究之間存在明顯資訊落差。
