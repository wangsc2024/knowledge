GitHub 靈感蒐集 foundation 深化研究

研究狀態：Foundation 深化階段 (Phase 2)
核心目標：整合 GitHub Scout 系列既有證據，建立 Agent 工具使用（Tool Use）與安全護欄（Guardrails）的技術基底。
技術棧關聯：LangGraph, MCP (Model Context Protocol), OODA Workflow, SSRF Protection.

1. 研究背景與技術動機

本研究旨在解決 github_scout 系列在自動化任務（如 ai_deep_research, tech_research）運行過程中，因缺乏統一安全規範與工具邊界定義所導致的風險。隨著 AI Agent 從實驗室邁向生產環境，如何建立「機制優先 (Mechanism-first)」的防禦架構成為核心議題。

本研究採用 OODA 循環 邏輯組織：

  • Observe/Orient: 透過 GitHub Scout 觀測開源工具的護欄實踐。
  • Decide/Act: 提煉可落地的 Playbook 並轉化為 SKILL.md 配置。

2. 核心技術分析：Agent 護欄機制 (Guardrails)

透過對 8 筆關鍵證據的深度解構，本研究提煉出三種主流的 Agent 工具護欄模式：

2.1 模式一：YAML 驅動的 Fail-Closed 規則 (以 BodAIGuard 為例)

針對高風險環境，透過預定義規則集強制攔截非法行為。

| 維度 | 技術規格 | 實作細節 |

| :--- | :--- | :--- |

| 核心機制 | Fail-Closed (預設拒絕) | 僅允許白名單內的指令執行，不符合規則即阻斷。 |

| 防禦層級 | 四層覆蓋 | Input $\rightarrow$ Planning $\rightarrow$ Tool Execution $\rightarrow$ Output。 |

| 關鍵防禦 | SSRF Protection | 攔截對 169.254.169.254 (Cloud Metadata) 的存取。 |

| 執行模式 | 多樣化 Proxy | 支援 HTTP Proxy, MCP stdio proxy, REST API。 |

2.2 模式二：Pre-ToolUse Hook 攔截 (以 anywhere-agents 為例)

透過腳本在工具調用前進行動態檢查，具備高度可攜性。

  • 實作路徑：利用 scripts/guard.py 作為 Hook 點。
  • 配置標準：透過 CLAUDE.md 或 AGENTS.md 聲明邊界。
  • 優勢：可跨 Claude Code 與 Codex 環境，實現行動攔截（Action Interception）。

2.3 模式三：基於 Passport 的身份授權 (以 aport 為例)

將權限與身份（Identity）深度綁定，實現細粒度控制。

  • 核心概念：Passport = Identity + Capabilities + Limits。
  • 技術特性：具備確定性（Deterministic），難以透過 Prompt Injection 繞過。
  • 落地挑戰：需修改核心攔截層（如 pre_bash_guard.py），實作成本較高。

3. 實踐應用 Playbook (Implementation Roadmap)

根據研究結果，建議按以下優先級執行優化：

P1：基礎安全加固 (Immediate Action)

  • [ ] SSRF 防禦實施：在 config/hook-rules.yaml 新增 ssrf-cloud-metadata 規則，攔截雲端元數據端點存取。
  • [ ] 觸發詞同步：更新 skills/github-scout/SKILL.md 至 v2.4.0，納入 fail-closed guardrail 等關鍵術語，提升 Agent 對安全指令的識別度。

P2：架構優化 (Mid-term Plan)

  • [ ] 工具授權模型：評估導入 Passport-based 授權機制，將權限從「全域」轉向「單一任務 (Per-task)」。
  • [ ] 邊界聲明標準化：建立統一的 AGENTS.md 規範，確保所有開發者與 Agent 共享相同的工具邊界認知。

4. 風險與權衡 (Risks & Trade-offs)

| 風險項目 | 描述 | 緩解策略 |

| :--- | :--- | :--- |

| 過度攔截 (False Positives) | 過嚴的 Guardrails 可能導致 Agent 無法執行合法任務。 | 引入 confirm 模式，針對中風險行為要求人工介入。 |

| 性能開銷 (Latency) | 多層次 Hook 與 Proxy 會增加 Tool Call 的延遲。 | 採用輕量化 YAML 規則引擎，減少複雜的 LLM 判斷。 |

| 繞過風險 (Bypass) | Prompt Injection 可能試圖欺騙 Hook 邏輯。 | 採用 Deterministic 邏輯，將安全檢查移出 LLM 控制範圍。 |

5. 延伸研究方向

  1. MCP (Model Context Protocol) 深度整合：研究如何將護欄邏輯直接內建於 MCP Server 層。
  2. AI-Ready 文件管線：探索 OpenAPI 與 llms.txt 在 Agent 知識獲取中的自動化對齊。
  3. 動態護欄演化：研究如何利用 Agent 自我觀察（Self-observation）來自動優化 hook-rules.yaml。
