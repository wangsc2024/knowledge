Claude Code Hooks 進階實戰指南 — 17 事件生命週期、三層 Handler 與企業級安全治理模式

主題概述

Claude Code Hooks 在 2026 年 2 月已從最初的 7 個生命週期事件擴展到 17 個事件，Handler 類型從純 command 擴展到 command / prompt / agent 三層，新增 async 非同步執行、PreToolUse 輸入修改（v2.0.10+）、Skill/Agent frontmatter 內嵌 Hooks、Plugin hooks 支援等進階功能。本筆記聚焦於既有 KB 研究未涵蓋的 Hooks 深度技術細節，包括完整事件生命週期、三層 Handler 設計模式、updatedInput 透明沙盒化、企業級治理架構，以及生產環境最佳實踐。

---

一、完整生命週期：17 個事件

截至 2026 年 2 月，Claude Code Hooks 支援以下 17 個生命週期事件：

| 事件 | 觸發時機 | 可阻擋？ | 支援 Matcher？ |

|------|---------|---------|---------------|

| SessionStart | 新 session 或恢復 | 否 | startup/resume/clear/compact |

| UserPromptSubmit | 用戶提交 prompt | 是 | 否 |

| PreToolUse | 工具呼叫執行前 | 是 | 工具名（Bash/Edit/Write 等） |

| PermissionRequest | 顯示權限對話框 | 是 | 工具名 |

| PostToolUse | 工具成功執行後 | 否 | 工具名 |

| PostToolUseFailure | 工具執行失敗後 | 否 | 工具名 |

| Notification | 系統通知 | 否 | permission_prompt/idle_prompt 等 |

| SubagentStart | 子 Agent 啟動 | 否 | Agent 類型 |

| SubagentStop | 子 Agent 結束 | 是 | Agent 類型 |

| Stop | 主 Agent 停止 | 是 | 否 |

| TeammateIdle | Team 成員閒置 | 是 | 否 |

| TaskCompleted | 任務標記完成 | 是 | 否 |

| ConfigChange | 設定檔變更 | 是 | 設定來源 |

| WorktreeCreate | Worktree 建立 | 是 | 否 |

| WorktreeRemove | Worktree 移除 | 否 | 否 |

| PreCompact | Context 壓縮前 | 否 | manual/auto |

| SessionEnd | Session 結束 | 否 | 結束原因 |

關鍵新增事件（2026 年新增）

  1. TeammateIdle：Agent Teams 中隊友閒置時觸發，可用 exit 2 阻止閒置並提供反饋
  2. TaskCompleted：任務完成前品質閘門，exit 2 可阻止標記完成
  3. ConfigChange：設定檔變更審計（policy_settings 無法阻擋）
  4. WorktreeCreate/Remove：支援非 Git VCS（SVN、Perforce）的 worktree 隔離
  5. PreCompact：壓縮前保存完整對話記錄
  6. SessionEnd：Session 結束清理（reason: clear/logout/prompt_input_exit）
  7. SubagentStart/SubagentStop：子 Agent 生命週期監控

二、三層 Handler 設計模式

2.1 Command Handler（傳統）


{
  "type": "command",
  "command": ".claude/hooks/guard.sh",
  "timeout": 600,
  "async": false
}
  • exit 0：允許，stdout 解析為 JSON
  • exit 2：阻擋，stderr 作為錯誤訊息回饋給 Claude
  • 其他 exit code：非阻擋錯誤，stderr 顯示在 verbose 模式

2.2 Prompt Handler（LLM 評估）


{
  "type": "prompt",
  "prompt": "評估是否所有任務完成：$ARGUMENTS",
  "model": "haiku",
  "timeout": 30
}
  • 使用快速模型（預設 Haiku）進行單輪評估
  • 回傳 {"ok": true} 允許，{"ok": false, "reason": "..." } 阻擋
  • 適用場景：語義層面的判斷（如「所有任務是否完成」）

2.3 Agent Handler（多輪驗證）


{
  "type": "agent",
  "prompt": "驗證所有單元測試通過。執行測試套件並檢查結果。$ARGUMENTS",
  "timeout": 120
}
  • 生成具備 Read/Grep/Glob 工具存取權的子 Agent
  • 最多 50 輪對話，深度檢查程式碼和測試結果
  • 適用場景：需要實際檢查檔案或執行測試的品質閘門

Handler 適用事件對照

| 支援全部三種 | 僅支援 command |

|-------------|---------------|

| PreToolUse, PostToolUse, PostToolUseFailure | SessionStart, SessionEnd |

| PermissionRequest, Stop, SubagentStop | SubagentStart, Notification |

| UserPromptSubmit, TaskCompleted | TeammateIdle, PreCompact |

| — | ConfigChange, WorktreeCreate/Remove |

三、PreToolUse 輸入修改（v2.0.10+ 重大突破）

3.1 updatedInput 機制

v2.0.10 起，PreToolUse hooks 不僅能阻擋，還能修改工具輸入參數後繼續執行：


{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow",
    "updatedInput": {
      "command": "npm run lint --fix"
    }
  }
}
  • Claude 看到的是自己的原始指令，實際執行的是修改後的版本
  • 必須搭配 permissionDecision 使用（allow 或 ask）

3.2 透明沙盒化應用場景

  1. 自動加 dry-run 旗標：資料庫遷移指令自動加上 --dry-run
  2. 機密消毒：攔截含 API key 的 curl 指令，替換為環境變數引用
  3. 路徑修正：MinGW 雙斜線路徑自動正規化
  4. 團隊規範強制：commit message 格式自動修正（加前綴、簽名）
  5. 依賴自動安裝：偵測 import 錯誤時自動加 pip install 前綴

3.3 PermissionRequest 的 updatedPermissions

PermissionRequest hook 除了 allow/deny，還可修改權限規則：


{
  "hookSpecificOutput": {
    "hookEventName": "PermissionRequest",
    "decision": {
      "behavior": "allow",
      "updatedInput": { "command": "npm run lint" },
      "updatedPermissions": [
        { "type": "toolAlwaysAllow", "tool": "Bash" }
      ]
    }
  }
}

四、企業級安全治理架構

4.1 四層設定檔層級

| 層級 | 檔案位置 | 範圍 | 可共享 |

|------|---------|------|-------|

| 組織策略 | Managed Policy | 全組織 | 管理員控制 |

| 使用者 | ~/.claude/settings.json | 個人所有專案 | 否 |

| 專案 | .claude/settings.json | 單一專案 | 可 commit |

| 本地 | .claude/settings.local.json | 單一專案 | 否（gitignored） |

  • 企業管理員可設 allowManagedHooksOnly 阻擋使用者/專案/Plugin hooks
  • disableAllHooks 在使用者層無法覆蓋管理策略層的 hooks
  • policy_settings 的 ConfigChange 無法被阻擋（確保企業策略生效）

4.2 Claude Code Security（2026-02-20 發布）

  • Enterprise/Team 客戶限定研究預覽
  • 已發現 500+ 零日漏洞
  • 治理模式：「Claude 發現 → Claude 建議 → 人類決策」
  • 多階段驗證：結果需經信心評分和嚴重性分級

4.3 生產環境 Hook 安全最佳實踐

  1. 驗證並消毒輸入：永遠不要盲信 stdin 資料
  2. Shell 變數加引號：使用 "$VAR" 而非 $VAR
  3. 阻擋路徑遍歷：檢查 .. 路徑逃逸
  4. 使用絕對路徑：搭配 $CLAUDE_PROJECT_DIR 環境變數
  5. 跳過敏感檔案：.env/.git/金鑰等
  6. 防禦性錯誤處理：JSON 解析失敗預設 exit 0（允許），避免 hook 故障阻斷工作流
  7. 漸進式導入：先 warn 觀察 1-2 週 → 確認無誤再改為 block

五、Async 非同步 Hooks

5.1 設定方式


{
  "type": "command",
  "command": "./run-tests.sh",
  "async": true,
  "timeout": 300
}
  • 僅 command 類型支援 async
  • Hook 在背景執行，Claude 繼續工作不被阻塞
  • 完成後結果在下一個對話輪次傳遞（systemMessage/additionalContext）
  • 限制：無法阻擋操作、無法回傳 decision

5.2 典型應用

  • PostToolUse 編輯後自動跑測試，結果非同步回報
  • 部署腳本、CI/CD 觸發（不阻塞開發流程）

六、Skill/Agent Frontmatter Hooks

Hooks 可直接定義在 Skill 或 Agent 的 YAML frontmatter 中：


---
name: secure-operations
description: 帶安全檢查的操作
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/security-check.sh"
---
  • Hooks 生命週期與 Skill/Agent 綁定（啟用時生效，結束時清理）
  • 支援 once: true 一次性 hook（僅在 Skill 中使用）
  • Agent 的 Stop hooks 自動轉換為 SubagentStop

---

七、MCP 工具 Hook 整合

  • MCP 工具命名格式：mcp____
  • 可用正則匹配特定 MCP 伺服器或工具群組：

- mcp__memory__.* 匹配所有 memory server 工具

- mcp__.__write. 匹配任何含 write 的 MCP 工具

  • PostToolUse 支援 updatedMCPToolOutput 修改 MCP 工具輸出

八、實戰模式參考

8.1 四層 Guard 架構（社群最佳實踐）

  1. Safety Guards：阻擋 rm -rf、git reset --hard、--force 等破壞性操作
  2. Context Injection：注入 repo metadata（正確的 GitHub org/owner）
  3. Workflow Rules：TDD 提醒、GPG 簽名、Worktree 偵測
  4. Skill Suggestion：依檔案模式自動建議 Skill（.test.go → TDD Skill）

8.2 本專案實作對照

本專案（daily-digest-prompt）已實作完整的 Hook 強制層：

  • pre_bash_guard.py：13 條 Bash 規則（含子 shell 繞過、base64 編碼外洩偵測）
  • pre_write_guard.py：4 條寫入規則
  • pre_read_guard.py：3 條讀取規則
  • post_tool_logger.py：JSONL 結構化日誌 + 自動標籤分類
  • on_stop_alert.py：Session 結束自動告警
  • 外部化規則：config/hook-rules.yaml（20 條規則、3 個 preset）
  • 三個 preset：strict / normal / permissive

九、SessionStart 環境變數持久化

CLAUDE_ENV_FILE 環境變數允許 SessionStart hook 設定持久化環境變數：


if [ -n "$CLAUDE_ENV_FILE" ]; then
  echo 'export NODE_ENV=production' >> "$CLAUDE_ENV_FILE"
  echo 'export DEBUG_LOG=true' >> "$CLAUDE_ENV_FILE"
fi

這些變數在整個 session 中所有後續 Bash 指令可用。僅 SessionStart 支援此功能。

十、除錯與疑難排解

  • claude --debug：顯示 hook 執行細節（匹配結果、exit code、輸出）
  • Ctrl+O：切換 verbose 模式查看 hook 進度
  • 設定檔變更不立即生效：Claude Code 啟動時快照 hooks 設定，防止 mid-session 修改
  • 外部修改 hooks 時會警告並要求在 /hooks 選單中審查
