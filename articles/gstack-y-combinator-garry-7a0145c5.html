一、專案概述

gstack 是 Y Combinator 總裁暨 CEO Garry Tan 於 2026 年 3 月 12 日開源的 Claude Code 技能包（Skill Pack），將單一 AI 助手轉化為一支虛擬軟體開發團隊。專案在發布後 48 小時內突破 10,000 Stars，至今已超過 23,000 Stars、2,200+ Forks，成為 2026 年成長最快的開發者工具之一。

核心數據

  • GitHub Stars：23,000+（截至 2026-03-27）
  • Forks：2,200+
  • 授權：MIT
  • 語言組成：79.6% TypeScript、18.3% Go
  • 版本：0.4.1
  • 創建者：Garry Tan（garrytan）

核心理念

Tan 的核心洞見是：不同開發階段需要不同的認知模式。正如他所說：「Planning is not review. Review is not shipping.」gstack 不把 AI 當萬能工具，而是為每個階段提供明確的「認知齒輪」——讓使用者在每個時刻指定模型應操作的專家模式。

二、九大角色（Slash Commands）

| 指令 | 角色 | 職責 |

|------|------|------|

| /plan-ceo-review | 創辦人/CEO | 從使用者角度重新思考問題；識別功能需求中隱藏的 10 星級產品 |

| /plan-eng-review | 工程經理/技術主管 | 鎖定架構、資料流，產出序列圖、狀態圖、元件圖、資料流圖 |

| /review | 偏執型資深工程師 | 結構性審計：N+1 查詢、競態條件、信任違規、缺少索引 |

| /ship | 發布工程師 | 同步 main、跑測試、解決 review、推送分支、開 PR |

| /browse | QA 工程師 | 登入、導航、截圖、讀取 console 錯誤；60 秒完成完整 QA |

| /qa | QA 主管 + 修復工程師 | Diff-aware 系統測試，find-fix-verify 循環，產出健康分數（0-100）|

| /qa-only | QA 報告員 | 僅報告測試結果，不修改程式碼 |

| /setup-browser-cookies | Session 管理員 | 從真實瀏覽器匯入 cookies 進行認證測試 |

| /retro | 工程經理 | 分析 commits、速度、測試比率；提供團隊回顧與 JSON 快照 |

三、技術架構

瀏覽器自動化堆疊

  • 持久化 Chromium daemon（Playwright 驅動）
  • 首次啟動 ~3 秒，後續指令 100-200ms 延遲
  • Cookies、tabs、localStorage 在指令間持久化
  • 30 分鐘閒置自動關閉
  • 比 Chrome MCP 工具快 20 倍，零 context 膨脹

編譯二進位

  • 58MB 可執行檔（Bun 建構）
  • 原生 SQLite 用於 cookie 解密
  • 執行時無 node_modules
  • 內建 HTTP server（localhost-only 綁定）

元素定址

  • 使用無障礙樹參照（@e1, @e2）取代 CSS 選擇器
  • 外部用 Playwright Locators，避免 DOM 變異
  • 相容 CSP 限制網站、React hydration、Shadow DOM

安全性

  • 每 session Bearer token 認證
  • Cookie 值不以明文寫入磁碟
  • Keychain 存取需使用者明確授權
  • 唯讀資料庫存取
  • 硬編碼 registry 防 shell injection

---

四、並行執行（Conductor Agent）

gstack 的 Conductor 機制可同時執行最多 10 個 Claude Code session，每個擁有：

  • 隔離的工作區
  • 獨立的 Chromium 實例
  • 唯一 port 分配（10000-60000）
  • 獨立的 cookie/tab 狀態
  • 無 port 衝突或共享狀態

---

五、工作流程

自然開發循環依序流動：

  1. 產品驗證（CEO Review）→ 2. 架構設計（Eng Review）→ 3. 實作 → 4. 結構審查（Review）→ 5. 自動化發布（Ship）→ 6. 視覺 QA（Browse/QA）→ 7. 團隊回顧（Retro）

生產力數據

Garry Tan 使用此設定，在 50 天期間平均每週產出 10,000 行程式碼和 100 個 Pull Requests。

六、安裝方式

需求：Claude Code、Git、Bun v1.0+

git clone https://github.com/garrytan/gstack.git ~/.claude/skills/gstack \

# 團隊設定（選用）

七、社群反應與爭議

正面

  • 48 小時破萬 Stars，Product Hunt 趨勢榜
  • 早期使用者回報成功發現 XSS 安全漏洞
  • 某 CTO 稱其為「god mode」
  • Claude 自身評價：「一個成熟、有主見的系統，由真正重度使用的人打造」

爭議

  1. 實質 vs 炒作：批評者認為 gstack 本質上「只是一堆 prompts」，許多開發者早已私下建立類似系統
  2. 影響力優勢：質疑若非 Tan 的 YC CEO 身份，專案能否純靠技術優勢獲得相同關注
  3. 專業疑慮：「god mode」見證引發批評——為何關鍵安全漏洞需要 AI 工具才能發現？

八、與本專案的關聯分析

gstack 的設計哲學與本專案（daily-digest-prompt）有高度共鳴：

| 面向 | gstack | daily-digest-prompt |

|------|--------|--------------------|

| 角色分離 | 9 個 slash command 角色 | Sub-Agent 專責分工 |

| 工作流程強制 | CEO→Eng→Review→Ship | Phase 0→1→2→3 |

| 品質門檻 | QA 健康分數 0-100 | done_cert quality_score 1-5 |

| 並行執行 | Conductor（10 sessions）| Agent Team（TeamCreate）|

| Skill 驅動 | Markdown skill 檔案 | SKILL.md + SKILL_INDEX |

可借鑑之處

  1. /retro 回顧機制：自動分析 commits 與速度指標，可強化本專案的 system-audit
  2. 瀏覽器 QA 整合：Playwright 持久化 daemon 概念，可用於 webapp-testing Skill
  3. 元素定址（@e1）：無障礙樹參照比 CSS 選擇器更穩定，值得在爬蟲任務中採用

---

九、結論

gstack 代表了 2026 年 AI 輔助開發的一個重要趨勢：從「AI 寫程式碼」進化到「AI 模擬完整開發團隊」。它的核心創新不在於單一技術突破，而在於將專業軟體開發流程（產品思維→架構設計→實作→審查→發布→QA→回顧）系統化地編碼為可組合的 Skill 集合。爭議的存在也反映了業界對 AI 工具「是否只是精美包裝的 prompts」這一根本問題的持續辯論。
