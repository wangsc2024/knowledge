JSONL 結構化日誌與可觀測性工程完整指南

技術概述

JSONL（JSON Lines）是一種以行為單位的結構化日誌格式，每行為一個完整的 JSON 物件。相較於傳統純文字日誌，JSONL 同時提供機器可解析性與人類可讀性，已成為 2025-2026 年現代應用程式監控與可觀測性的產業標準。主要平台如 Elasticsearch、Splunk、Datadog、Fluentd、Logstash 均原生支援 JSONL 格式。其行分隔特性使其特別適合即時串流處理與分析，無需緩衝整個檔案即可逐行解析。

核心概念與原理

結構化日誌的三大支柱

  1. 機器可讀格式：JSON 的 key-value 結構使自動化處理成為可能，不再需要複雜的正則表達式解析
  2. 上下文豐富化（Contextual Enrichment）：每筆日誌攜帶足夠的上下文屬性，支援多維度切片分析
  3. 統一 Schema：跨服務維持一致的欄位命名與結構，避免查詢邏輯碎片化

JSONL vs 其他日誌格式

格式機器解析人類可讀串流友善多行安全查詢效率
純文字差（需正則）佳佳差差
JSON（整檔）佳中差（需完整檔案）佳佳
JSONL佳中佳佳佳
CSV中中佳差中
Syslog中差佳佳中

Schema 設計最佳實踐

必要欄位

{
  "ts": "2026-02-23T10:30:45+08:00",
  "level": "info",
  "msg": "API call completed",
  "sid": "abc12345",
  "service": "daily-digest",
  "host": "WIN-DESKTOP"
}
欄位說明格式建議
ts時間戳ISO 8601，包含時區（建議 UTC 或本地帶 offset）
level日誌等級debug/info/warn/error/fatal
msg人類可讀描述簡潔明確
sid相關性 ID（Session/Correlation ID）UUID 或短 hash
service服務名稱全系統一致
host主機名稱自動注入

擴展欄位設計原則

  1. 保持淺層結構：最多 2-3 層巢狀，深層巢狀嚴重影響查詢效能
  2. 一致的命名慣例：全系統統一使用 snake_case（如 user_id），不混用 camelCase
  3. 欄位前綴分群：用前綴區分類別（如 http_method、http_status、db_query）
  4. 避免動態 key 名：不要用可變值作為 key（如 {"user_123": "data"}），改用 {"user_id": 123, "data": "..."}

錯誤日誌結構化

{
  "ts": "2026-02-23T10:30:45+08:00",

查詢模式與分析技巧

命令列工具（本地分析）

# jq — JSONL 的瑞士刀
# 篩選 error 等級
cat logs/structured/*.jsonl | jq -c 'select(.level == "error")'

# 統計各等級數量
cat logs/structured/*.jsonl | jq -r '.level' | sort | uniq -c | sort -rn

# 篩選特定 session
cat logs/structured/*.jsonl | jq -c 'select(.sid == "abc12345")'

# 篩選含特定標籤的記錄
cat logs/structured/*.jsonl | jq -c 'select(.tags | arrays | index("api-call"))'

# 時間範圍篩選
cat logs/structured/*.jsonl | jq -c 'select(.ts >= "2026-02-23T08:00" and .ts <= "2026-02-23T12:00")'

Python 分析腳本

import json
from collections import Counter
from pathlib import Path

def analyze_jsonl(log_dir: str, days: int = 7) -> dict:
    """分析 JSONL 日誌的基本統計"""
    stats = Counter()
    errors = []
    
    for f in sorted(Path(log_dir).glob("*.jsonl")):
        for line in f.open(encoding="utf-8"):
            try:
                entry = json.loads(line)
            except json.JSONDecodeError:
                continue
            
            stats[entry.get("level", "unknown")] += 1
            
            if entry.get("has_error") or entry.get("level") == "error":
                errors.append(entry)
            
            # 標籤統計
            for tag in entry.get("tags", []):
                stats[f"tag:{tag}"] += 1
    
    return {
        "total": sum(v for k, v in stats.items() if k in ("info", "warn", "error", "debug")),
        "by_level": dict(stats),
        "error_count": len(errors),
        "recent_errors": errors[-5:]
    }

PowerShell 分析（Windows 環境）

# 讀取 JSONL 並篩選
$logs = Get-Content "logs/structured/*.jsonl" | ForEach-Object {
    try { $_ | ConvertFrom-Json } catch { $null }
} | Where-Object { $_ -ne $null }

# 統計各等級
$logs | Group-Object level | Select-Object Name, Count | Sort-Object Count -Descending

# 篩選攔截事件
$logs | Where-Object { $_.tags -contains 'blocked' } | Format-Table ts, tool, summary

日誌輪轉與保留策略

分層保留（Tiered Retention）

層級保留天數存取速度說明
Hot（熱存）7-30 天即時原始 JSONL，未壓縮
Warm（溫存）7-90 天快速gzip 壓縮，合併索引
Cold（冷存）30-365 天中等歸檔壓縮
Delete>90-365 天—合規要求後刪除

大小觸發輪轉

import os
import gzip
import shutil

def rotate_if_needed(log_path: str, max_size_mb: int = 50):
    """日誌超過閾值時自動輪轉壓縮"""
    if os.path.getsize(log_path) > max_size_mb * 1024 * 1024:
        archive = f"{log_path}.{datetime.now():%Y%m%d_%H%M%S}.gz"
        with open(log_path, 'rb') as f_in:
            with gzip.open(archive, 'wb') as f_out:
                shutil.copyfileobj(f_in, f_out)
        open(log_path, 'w').close()  # 清空原檔

可觀測性整合：三大支柱

1. 日誌（Logs）

結構化 JSONL 日誌是事件紀錄的基礎，提供離散事件的詳細上下文。

2. 指標（Metrics）

從日誌衍生的聚合數據（如每分鐘錯誤率、API 回應時間 P95），適合趨勢監控與告警。

3. 追蹤（Traces）

透過 Correlation ID（相關性 ID）串聯跨服務的請求流程，形成完整的請求鏈路圖。

相關性 ID 實踐

import uuid
from contextvars import ContextVar

# 每個請求生成唯一 ID
correlation_id = ContextVar('correlation_id', default='')

def new_request():
    cid = str(uuid.uuid4())[:8]
    correlation_id.set(cid)
    return cid

def log_entry(level, msg, **extra):
    return json.dumps({
        "ts": datetime.now().isoformat(),
        "level": level,
        "msg": msg,
        "sid": correlation_id.get(),
        **extra
    }, ensure_ascii=False)

安全與效能

敏感資料防護

  • 絕不記錄：密碼、Token、信用卡號、API Key
  • 遮罩處理：PII 資料使用遮罩（如 email → u***@example.com）
  • 靜態加密：日誌存檔時使用加密
  • 存取控制：日誌目錄設定適當的檔案權限

效能優化

  • 非同步寫入：使用背景執行緒或 buffer 批次寫入，避免阻塞主流程
  • 採樣策略：高流量的常規操作可設定 1/10 或 1/100 採樣率，錯誤事件保持 100% 記錄
  • gzip 壓縮：JSONL 壓縮率通常可達 85-95%
  • 現代框架效能：結構化日誌庫的開銷通常僅 0.1-0.5ms/條

與 Daily-Digest-Prompt 專案的關聯

現有實作

本專案的 post_tool_logger.py 已實作完整的 JSONL 結構化日誌系統，具備：

  • 自動標籤分類（api-call、cache-read、skill-read、blocked、error 等）
  • 50MB 緊急輪轉機制
  • Session ID（sid）追蹤
  • query_logs.py CLI 查詢工具

可優化方向

  1. Schema 版本化：在 JSONL 中加入 schema_version 欄位，支援未來格式演進
  2. 壓縮歸檔：目前保留 7 天原始檔，可加入 gzip 壓縮的 warm 層延長至 30 天
  3. 衍生指標：從 JSONL 自動衍生每日指標（API 呼叫次數、快取命中率、平均延遲），存入 state/daily-metrics.json
  4. 健康評分增強：query-logs.ps1 -Mode health-score 可結合日誌趨勢分析（如連續 3 天錯誤率上升觸發預警）
  5. 欄位索引：對高頻查詢欄位（如 tags、sid、level）建立本地索引加速搜尋

實際應用範例

本專案每日 44 次 Agent 執行，每次 20-100 次工具呼叫，產生約 900-4400 條 JSONL 記錄。透過結構化日誌，可以：

  • 追蹤單次 Session 的完整工具呼叫鏈
  • 統計各 API 的呼叫頻率與快取命中率
  • 識別 Hook 攔截模式（誤判 vs 合理攔截）
  • 偵測 Agent 迴圈行為（LoopDetector 整合）
  • 自動產生 Session 健康報告
