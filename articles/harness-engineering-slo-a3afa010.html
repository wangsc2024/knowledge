Harness Engineering 生產實測指標與自動化約束校準

研究階段：harness-engineering 系列 post-synthesis 延伸
承接既有筆記：foundation（CI/CD 智能化）→ application（約束演化管線）→ optimization（fitness function 與漂移偵測）→ synthesis（統一治理框架）
本文聚焦：如何用量化指標度量 Harness 有效性、自動化約束校準、以及 SLO 驅動的 Agent 治理

---

一、執行摘要

Harness Engineering 已從概念定義階段進入量化度量階段。2026 年的產業趨勢顯示三大演進方向：（1）從定性約束走向 SLO/SLI 驅動的量化治理；（2）從靜態閾值走向自動校準的漸進式品質門檻；（3）從人工監控走向 OpenTelemetry 原生的可觀測性堆疊。本報告整合 6 個獨立來源，提出「量化約束校準四層模型」，為 daily-digest-prompt 系統的 Harness 實施提供具體的度量與自動化建議。

---

二、研究背景：定性到量化的轉折點

既有 Harness Engineering 研究（本系列前四階段）已建立完整的理論框架：約束即賦能、反饋迴路、自癒機制、fitness function。但一個關鍵缺口始終存在——如何量化 Harness 的有效性？ [Epsilla Blog, 2026] 指出「在驗證功能正確性方面仍存在顯著差距」，這正是本次研究要填補的空白。

---

三、核心發現：量化約束校準四層模型

3.1 第一層：漸進式品質門檻（Progressive Quality Gates）

Galileo 的 Agent 評估框架提出環境分級的品質閾值 [Galileo, 2026]：

| 環境 | 任務成功率要求 | 安全保證 |

|------|-------------|--------|

| 開發 | ≥70% | 基本安全檢查 |

| 預備 | ≥85% | 完整安全驗證 |

| 生產 | ≥95% | 特定安全保證 + 自動回滾 |

這個漸進模型解決了「一刀切」閾值的問題——約束的嚴格程度隨部署階段自動調整。Canary 部署期間會持續比較錯誤率、延遲、使用者滿意度和工具使用模式，24-48 小時內自動回滾退化版本 [Galileo, 2026]。

已驗證✅（3+ 來源佐證）：漸進式門檻概念見於 Galileo、Fiddler AI、agent-engineering.dev。

3.2 第二層：Eval-to-Guardrail 生命週期

Galileo 提出的「eval-to-guardrail lifecycle」是量化治理的核心機制 [Galileo, 2026]：

  1. 評估發現階段：透過持續監控識別失敗模式，自動擷取問題互動並匿名化加入回歸測試集
  2. 護欄部署階段：三層防護模型（模型層、治理層、執行層），主動攔截不安全輸出
  3. 持續回饋迴圈：生產失敗自動轉化為永久品質改進，防止問題復發

三種觸發機制：

  • Commit 觸發：程式碼變更時啟動評估，品質門檻未過則阻擋部署
  • 排程觸發：每日/每週執行，偵測模型漂移與 API 變更
  • 事件觸發：錯誤率飆升或異常自動觸發深度評估

已驗證✅（3+ 來源佐證）：eval-to-guardrail 生命週期見於 Galileo、Guardrails AI、Fiddler AI。

3.3 第三層：OpenTelemetry 原生可觀測性

Guardrails AI 已原生整合 OpenTelemetry，實現標準化的約束監控 [Guardrails AI, 2026]：


guard = Guard.from_string(
  validators=[ValidLength(min=1, max=10, on_fail="exception")],
  tracer=default_otel_collector_tracer("petname_guard")
)

可追蹤的四大指標類別：

  • Guard/Validator 延遲：單一 Guard 及各 Validator 的處理時間
  • 驗證通過/失敗率：跨所有 Guard 的統計
  • Guard 成功率：整體準確度趨勢
  • LLM 延遲：跨模型效能比較

搭配 Grafana 預設儀表板，可視化 Guard 執行時長、驗證成功/失敗率、延遲指標。生產環境的效能基準：護欄決策延遲 <100ms，吞吐量需支持每秒數百個 prompt 評估 [Fiddler AI, 2026]。

已驗證✅（3+ 來源佐證）：OTEL 整合見於 Guardrails AI、Fiddler AI、agent-engineering.dev（可觀測性為核心組件）。

3.4 第四層：多維度 Fitness Function 評分

Galileo 的三層 Rubric 結構提供了系統化的 Fitness Function 框架 [Galileo, 2026]：

  • 7 個主要維度 → 分解為 25 個子維度 → 細化為 130 個可量測項目
  • 雙時間維度：部署前驗證（建立基線）+ 生產持續監控（追蹤漂移）
  • 軌跡指標：trajectory_exact_match、trajectory_precision、trajectory_recall（評估推理路徑）
  • 結果指標：任務成功率、回應品質、延遲要求

關鍵產業數據：

  • 企業級 AI 部署中，Agent 單次執行成功率約 60%，但跨 8 次執行後降至 25% [Galileo, 2026]
  • LLM-as-Judge 目標：與人類評估者的 Spearman 相關係數 ≥0.80 [Galileo, 2026]
  • 進階框架可達到 Spearman 相關係數 7.5% 的改善 [Galileo, 2026]

四、產業現況與量化佐證

4.1 採用率數據

根據 LangChain 2026 State of Agent Engineering 調查 [LangChain, 2026]：

  • 89% 的受訪者已實施 Agent 可觀測性（超越 evals 採用率 52%）
  • 57% 的受訪者已有 Agent 在生產環境運行
  • 大型企業在採用率上領先

4.2 可靠性提升

  • 遵循 Harness Engineering 實踐的團隊實現 2-5 倍可靠性提升 [agent-engineering.dev, 2026]
  • OpenAI Codex 團隊產出超過 100 萬行程式碼，零人工介入 [agent-engineering.dev, 2026]
  • LangChain Deep Agents 透過 Harness 改進（非模型變更），Terminal Bench 2.0 分數從 52.8% 提升至 66.5%（Top 30 → Top 5）[agent-engineering.dev, 2026]

五、對 Daily-Digest-Prompt 系統的實務應用建議

基於本次研究發現，提出以下優化方案：

5.1 P0：建立約束 SLI/SLO 定義

在 config/benchmark.yaml 新增 constraint_slo 區塊：

| SLI 指標 | SLO 目標 | 量測方式 |

|---------|---------|--------|

| Hook 攔截準確率 | ≥95% | blocked_count / (blocked + false_negative) |

| Guard 延遲 P95 | <200ms | post_tool_logger.py 時間戳差值 |

| 約束驗證通過率 | ≥90% | 驗證通過數 / 總驗證數 |

| 自癒成功率 | ≥70% | 自動修復數 / 偵測異常數 |

5.2 P1：實施漸進式品質門檻

將 Galileo 的 70/85/95 模型適配至本系統：

  • 開發（手動執行）：任務成功率 ≥70%，允許 partial 結果
  • 排程（自動執行）：任務成功率 ≥85%，partial 計入警告
  • 關鍵路徑（摘要組裝）：成功率 ≥95%，失敗自動重試

5.3 P2：OTEL 式約束監控

現有 post_tool_logger.py JSONL 日誌已具備基礎，可擴展為：

  • 新增 guard_duration_ms 欄位追蹤各 Hook 執行時間
  • 新增 validation_result 欄位（pass/fail/warn）
  • 定期產出 Guard 成功率趨勢報告（整合至 check-health.ps1）

5.4 P3：Eval-to-Constraint 自動化管線

將 context/research-registry.json 的失敗模式自動轉化為約束規則：

  • 失敗模式出現 ≥3 次 → 自動建議新增 Hook 規則
  • 新規則加入 config/hook-rules.yaml 後自動進入 Canary 階段
  • 24 小時無誤報 → 正式啟用

六、知識缺口與未來方向

  1. 自動閾值校準演算法：目前產業多採用靜態閾值 + 人工調整，尚無成熟的自動校準開源方案
  2. 跨 Agent 約束一致性：多 Agent 團隊中如何確保約束策略一致，仍缺少標準化框架
  3. 約束成本-效益量化：增加約束帶來的可靠性提升 vs. 延遲/Token 開銷的精確權衡模型
