Harness Engineering 生產級可靠度優化：Reliability Surface R(k,ε,λ)、Chaos 注入與級聯防禦對本專案 B0–B5 管線的對照藍圖（2026-04-10）

執行摘要

本筆記在既有 KB 對 Harness Engineering、Fitness Function 與 B3 儀表三角 的累積上，補上兩條 2025–2026 年產業與學術共識的優化軸線：（1）以 Reliability Surface R(k,ε,λ) 將「多次取樣一致性、輸入擾動穩健度、工具／API 故障注入下的容錯」收斂成可量測曲面，作為 Harness 的動態適應度函數；（2）以 Agent 專用 Chaos（工具逾時、速率限制、串流截斷、schema 漂移等）與 行為不變式（invariants） 做持續驗證，對齊本專案 config/autonomous-harness.yaml 的 execution_contracts、checkpoint 與 retry／冪等 設計。並以 OpenTelemetry 優先的可觀測性 與 分類器級聯（含 shadow／probe） 對照治理堆疊三層（行為、營運、風險）之工程化落地。

研究背景

  • Agent harness 被描述為包覆 LLM 執行的控制面：管理生命週期、脈絡、工具權限與安全邊界，與「僅組裝 Agent 的框架」不同 [來源：htek.dev, 2026]。
  • 產業敘事強調：多步驟成功率會複合衰減（例如單步 95% 在長鏈上快速惡化），Harness 透過驗證迴路、退避重試、checkpoint-resume 等拉高端到端可靠度 [來源：Harness Engineering 部落格, 2026]。
  • Agentic AI 可觀測性 主張以 OTel 統一收集 prompt／工具／檢索／成本／策略決策，作為合規與治理的控制面 [來源：Arthur.ai Column, 2026-04]。

核心發現（含三角佐證標記）

1. 可靠度曲面 R(k,ε,λ) 可作為 Harness 的「多維 Fitness」✅

ReliabilityBench 提出 R(k,ε,λ)：k 次取樣一致性、ε 級輸入擾動穩健度、λ 強度下的混沌故障容錯，並給出系統化故障類型（逾時、限流、部分回應、schema 漂移等）[來源：arXiv 2601.06112, 2026]。此與演化式架構中「Chaos 作為動態 Fitness Function」的敘事一致 [來源：LinkedIn／Atul Mistry 對 Fitness Function 與 Chaos 的整理, 2025–2026]。

2. Agent Chaos 工具鏈補足「基礎設施 Chaos」無法涵蓋的語意／工具失敗 ✅

Flakestorm 明確定位為生產 Agent 的混沌工程：對工具與 LLM 注入失敗並檢查行為契約 [來源：GitHub flakestorm/flakestorm]。agent-chaos 亦提供 LLM／tool 多類注入與 fuzz 空間 [來源：GitHub deepankarm/agent-chaos]。兩者與 ReliabilityBench 的故障分類可互相參照 ✅。

3. 治理堆疊應以「程式化閘門＋稽核軌跡」實作，而非僅提示詞 ✅

Harness 層需同時涵蓋行為治理（工具／資料）、營運治理（步數、token、重試）與結構化稽核軌跡 [來源：Harness Engineering 治理專文, 2026]。Constitutional Classifiers++ 展示 兩階段級聯（輕量 probe 全量篩選 → 較重分類器／ensemble 複判）與 shadow deployment 校準拒答率與成本 [來源：Anthropic Research；arXiv 2601.04603 / OpenReview, 2025–2026] —— 可作為高風險輸出閘門的參考模式 ✅。

深度分析：對照本專案 `autonomous-harness.yaml` 與 B0–B5

| 外部機制 | 本專案對應（示意） | 優化含義 |

|----------|-------------------|----------|

| R(k,ε,λ) 持續量測 | B2 確定性／LLM 品質門檻、B3 儀表（熵、覆蓋率、飽和） | 將「單次 eval」擴展為 k／扰動／故障強度 三維報表，與現有 quality_gate、entropy_score 並列 |

| Tool／LLM Chaos | execution_contracts.retry_policy、checkpoint、Phase2 備援 | 在 staging 對 run-*-team.ps1 與 Cursor CLI 路徑注入延遲／429／部分 JSON，驗證 never_retry_without_review 與冪等鍵是否生效 |

| OTel-first 追蹤 | hooks/post_tool_logger.py、JSONL→OTLP 方向 | 將「每次工具決策、guard 結果、token／延遲」對齊 決策可溯源（provenance） 欄位，便於 B4 Gap 與 SLO 燃燒率連動 |

| 分類器級聯 | Hooks pre_*_guard、LLM Gate、高風險路徑 human_review_gates | 低成本的規則／探針先擋，僅升級可疑 session 到較重檢查，避免全量延遲 |

實務應用建議（可排入 backlog）

  1. 定義 3 條 Harness SLO 曲面切片：例如「Todoist Phase2 完成率 ×（模擬 429 強度 λ）×（prompt 輕微改寫 ε）」，每週在離線矩陣上回歸；對齊 ReliabilityBench 方法論 [2601.06112]。
  2. 選一條最小 Chaos profile：從 agent-chaos／Flakestorm 任選與本專案腳本相容的注入點（例如對 mock invoke-llm 或測試 harness），驗證 checkpoint.auto_cleanup_on_success 與重試預算。
  3. B3「開放問題」閉環：延續 KB 筆記「B3 儀表三角」— 為熵、覆蓋率、飽和三指標各設 上下閾＋告警路由（ntfy／日誌），並記錄與 harness_gap_tracker 的對應欄位。
  4. 高風險變更走級聯：對 pre_bash_guard／run-*.ps1 等 human_review_gates 已列路徑，參考 Constitutional Classifiers++ 的「probe 先篩、再升級」以降低全量檢查成本 [Anthropic；2601.04603]。

知識缺口與未來方向

  • ReliabilityBench 實驗多於 工具型 Agent 網域任務；與本專案「長時排程＋多 JSON 結果檔」的差異需自訂 狀態 oracle（例如 results/*.json schema＋cli_exit_note_zh）。
  • Chaos 與生產排程並行時需嚴格 環境隔離，避免污染 state/（ADR-006：仍由 PowerShell 獨佔寫入）。
