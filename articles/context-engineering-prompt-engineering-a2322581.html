Context Engineering（上下文工程）完整指南

從 Prompt Engineering 演進到生產級 AI Agent 的上下文設計方法論

---

一、概述與定義

2026 年，AI 應用開發的核心挑戰已從「如何寫好一句 Prompt」轉變為「如何設計整個上下文生態系統」。Context Engineering（上下文工程）是一門新興的工程學科，專注於動態策展模型的上下文視窗，確保 Agent 在正確的時間看到正確的資訊。

1.1 定義

Context Engineering 是刻意設計和管理模型在生成回應時所看到或知道的一切的實踐。

上下文（Context）包含所有提供給 LLM 的 Token：系統提示、工具描述、外部資料、對話歷史、記憶檢索結果。工程挑戰在於在 LLM 的有限注意力預算內，最大化每個 Token 的效用。

1.2 與 Prompt Engineering 的關鍵差異

| 面向 | Prompt Engineering | Context Engineering |

|------|-------------------|--------------------|

| 範圍 | 專注於撰寫提示文字 | 涵蓋所有輸入：提示、系統訊息、檢索資料、工具、歷史 |

| 性質 | 相對靜態的模板 | 動態、運行時組裝 |

| 焦點 | 如何措辭問題 | 提供什麼資訊搭配問題 |

| 整合 | 純文字互動 | 整合工具、API、記憶系統 |

| 角色 | 任何人可做的日常技巧 | 正式的工程學科 |

核心轉變：「效能提升越來越不是來自更好的模型，而是來自更聰明的上下文」。

---

二、Context Rot（上下文腐化）— 為什麼更大的上下文視窗不是解方

Transformer 架構中，每個 Token 都會注意到所有其他 Token，產生 n² 的成對關係。這造成：

  1. 注意力稀釋：上下文越長，注意力容量被拉薄
  2. 訓練經驗不足：模型對超長序列的訓練經驗有限
  3. 檢索精度下降：位置編碼內插技術有幫助但仍有退化
  4. 記憶衰減：類似人類工作記憶的有限容量
Needle-in-a-Haystack 基準測試已量化了 Context Rot 現象：隨著上下文 Token 數增加，模型準確召回資訊的能力下降。

關鍵啟示：上下文視窗是稀缺資源，需要刻意的架構設計，而非簡單地擴大視窗大小。

三、四大核心策略（LangChain WSCI 框架）

3.1 Write（寫入）— 持久化上下文到外部

將資訊持久化到上下文視窗之外，供後續檢索：

  • Scratchpad（便箋本）：Agent 透過工具呼叫或狀態欄位做筆記
  • 長期記憶：跨 Session 持久化的使用者偏好和學習事實
  • 自動生成記憶：從 Agent 回饋中合成的記憶

實踐案例：Claude 玩 Pokemon 時，Agent 在上千步操作中維護精確的計數，映射已探索區域、記住成就，完全不需要提示記憶結構。

3.2 Select（選取）— 拉入相關資訊

將相關資訊拉入上下文視窗：

  • 讀取便箋本內容：透過工具呼叫或狀態暴露
  • 記憶檢索：使用 Embedding 和知識圖譜
  • RAG 工具選擇：語意搜尋減少工具描述重疊（準確度提升 3 倍）

Just-In-Time 上下文檢索（Anthropic 提出）：Agent 維護輕量級識別符（檔案路徑、查詢、連結），在運行時動態檢索。Claude Code 的 CLAUDE.md 檔案預載 + glob/grep 即時導覽就是此混合模式的典範。

3.3 Compress（壓縮）— 保留必要 Token

只維護任務完成所需的最小 Token 集合：

  • 上下文摘要：遞迴或分層壓縮 Agent 軌跡
  • 上下文修剪：啟發式移除舊訊息
  • 工具呼叫後處理：為 Token 效率優化特定工具的輸出

壓縮實作原則：

  1. 先最大化召回率（捕獲所有相關資訊）
  2. 迭代提升精確度（消除多餘內容）
  3. 工具呼叫/結果是低風險的壓縮目標

3.4 Isolate（隔離）— 分離上下文管理複雜性

透過分割上下文來管理複雜性：

  • 多 Agent 架構：專門化的子 Agent 處理聚焦任務
  • 沙箱環境：程式碼執行的隔離
  • 狀態 Schema 設計：隔離上下文不暴露給 LLM

階層委派模式：Supervisor Agent 分解任務並委派給專門的 Worker Agent，Supervisor 維護全域上下文，Worker 只接收與子任務相關的局部上下文。

四、四大上下文失敗模式（必須防範）

| 失敗模式 | 描述 | 防範策略 |

|---------|------|----------|

| Context Poisoning | 幻覺資訊透過重用在上下文中累積 | 驗證外部檢索內容、定期清理 |

| Context Distraction | 過多歷史導致過度依賴過去行為 | 摘要壓縮、相關性過濾 |

| Context Confusion | 無關工具或文件擠壓決策空間 | 動態工具選擇、語意路由 |

| Context Clash | 矛盾資訊產生衝突假設 | 來源優先級、衝突偵測 |

五、生產級架構模式

5.1 上下文引擎（Context Engine）

中介層架構位於使用者和 LLM 之間：


[使用者輸入] -> [查詢解析] -> [即時資料檢索] -> [資訊聚合] -> [最終 Prompt 建構] -> [LLM 推理]

5.2 六大支柱框架（Weaviate）

  1. Agents — 編排決策、協調資訊流
  2. Query Augmentation — 精煉使用者輸入
  3. Retrieval — 連接外部知識源
  4. Prompting — 以清晰指令引導推理
  5. Memory — 跨互動保存歷史
  6. Tools — 透過 API 和外部系統啟用行動

5.3 分層上下文設計


[系統指令] -> [背景知識] -> [規則與限制] -> [工具描述] -> [期望格式] -> [對話歷史]

5.4 記憶階層架構

| 記憶類型 | 功能 | 實作方式 |

|---------|------|----------|

| 短期記憶 | 當前對話上下文 | Context Window 原生 |

| 工作記憶 | 多步驟任務暫存 | State Schema / Scratchpad |

| 長期記憶 | 跨 Session 持久化 | Vector Store / KV Store |

| 情節記憶 | 特定互動回憶 | Graph Memory / Mem0 |

六、Agentic RAG — 上下文工程的進階應用

6.1 從傳統 RAG 到 Agentic RAG

傳統 RAG 是單次 retrieve-generate 管線。Agentic RAG 升級為具有規劃、反思和自我修正能力的智能 Agent 架構：

  • 忠實度提升 42%（相比傳統 RAG，在多步驟企業知識問答中）
  • Thought-Action-Observation 循環：Agent 觀察結果並調整策略
  • 自動集合路由：LLM Agent 自動選擇適當的資料來源

6.2 MCP（Model Context Protocol）標準化

將 M x N 整合問題（M 個應用 x N 個工具需要客製程式碼）轉化為 M+N 架構，透過通用 JSON-RPC 通訊標準化工具整合。

七、長期任務管理策略（Long-Horizon Tasks）

當 Agent 操作超出上下文視窗限制時：

7.1 Compaction（壓縮）

在接近上下文限制時摘要對話內容，保留架構決策和未解決問題，丟棄冗餘輸出。

7.2 結構化筆記（Structured Note-Taking）

Agent 在上下文視窗外寫入持久筆記，需要時拉回：

  • 持久記憶，最小開銷
  • 跨複雜任務的進度追蹤
  • 跨工具呼叫維護關鍵依賴

7.3 子 Agent 架構（Sub-Agent）

專門化子 Agent 以乾淨的上下文視窗處理聚焦任務，廣泛探索後返回濃縮摘要（通常 1,000-2,000 Token）。在複雜研究任務上，子 Agent 架構顯著優於單一 Agent 系統。

八、生產最佳實踐（Anthropic + LangChain 綜合）

8.1 設計原則

  1. 從最小開始：以最佳模型測試最小 Prompt，根據失敗模式迭代添加
  2. 視上下文為有限資源：每個新 Token 消耗注意力預算，無情地策展
  3. Append-Only 設計：修改先前上下文會破壞 KV-Cache，成本倍增
  4. 漸進式揭露：讓 Agent 透過互動逐步發現上下文

8.2 工具設計原則

  • 自包含且對錯誤魯棒
  • 功能明確、最小重疊
  • 返回 Token 高效的資訊
  • 描述性強的輸入參數
  • 使用語意搜尋動態選擇工具（LangGraph Bigtool）

8.3 系統提示的黃金區間

避免兩個極端：

  • 過度脆弱：複雜硬編碼邏輯，隨時間碎片化
  • 過度模糊：高層指引缺乏具體行為訊號

有效系統提示使用 XML 標籤或 Markdown 標題組織為不同區段。

8.4 監控與可觀測性

  • 追蹤 Token 使用量，識別瓶頸
  • 實施評估測試以衡量上下文工程影響
  • 使用 LangSmith 等工具進行 Agent 追蹤
  • 建立回饋循環持續優化
